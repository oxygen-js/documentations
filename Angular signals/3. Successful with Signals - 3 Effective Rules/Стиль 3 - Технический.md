Бесспорно, что сигналы (Signals) определят будущее Angular. На первый взгляд они очень просты в использовании: сеттеры принимают новые значения, геттеры возвращают текущие значения, а шаблоны, равно как и эффекты (Effects), уведомляются об изменениях.

Может возникнуть соблазн обращаться с сигналами как с обычными переменными. В принципе это работает и может быть практичным вариантом при миграции существующего кода. Однако в этом случае преимущества сигналов и реактивных систем реализуются лишь в ограниченной мере. Кроме того, имеется ряд подводных камней, а код оказывается не таким понятным и, следовательно, не таким сопровождаемым, каким мог бы быть.

Чтобы не допускать подобных ситуаций, в этой статье я хотел бы дать вам три простых эвристических правила, которые позволяют использовать сигналы максимально идиоматично.

## Исходный пример с пространством для улучшения

Три обещанных правила я хотел бы обсудить на примере простого Angular-приложения:

![Пример приложения](https://www.angulararchitects.io/wp-content/uploads/2024/04/dessert-app.png)

Первая рассматриваемая реализация не является реактивной и также оставляет некоторое пространство для улучшения:

```typescript
@Component([…])
export class DessertsComponent implements OnInit {
  #dessertService = inject(DessertService);
  #ratingService = inject(RatingService);
  […]

  originalName = '';
  englishName = '';
  loading = false;

  desserts: Dessert[] = [];

  ngOnInit(): void {
    this.search();
  }

  search(): void {
    const filter: DessertFilter = {
      originalName: this.originalName,
      englishName: this.englishName,
    };

    this.loading = true;

    this.#dessertService.find(filter).subscribe({
      next: (desserts) => {
        this.desserts = desserts;
        this.loading = false;
      },
      error: (error) => { […] },
    });
  }

  toRated(desserts: Dessert[], ratings: DessertIdToRatingMap): Dessert[] {
    return desserts.map((d) =>
      ratings[d.id] ? { ...d, rating: ratings[d.id] } : d,
    );
  }

  loadRatings(): void {
    this.loading = true;

    this.#ratingService.loadExpertRatings().subscribe({
      next: (ratings) => {
        const rated = this.toRated(this.desserts, ratings);
        this.desserts = rated;
        this.loading = false;
      },
      error: (error) => { […] },
    });
  }
  […]
}
```

Поскольку привязываемые свойства не являются ни Observable, ни сигналами, стратегию обнаружения изменений `OnPush` для повышения производительности привязки данных применить нельзя. При более внимательном рассмотрении мы также замечаем, что метод `loadRatings` обновляет массив `desserts`, хотя его фактическая задача — загрузка рейтингов — не имеет к этому отношения.

Кроме того, разработчики обязаны помнить, что после любых изменений рейтингов массив `desserts` также необходимо модифицировать. Именно это и может приводить к трудно сопровождаемому коду и скрытым ошибкам — особенно когда `desserts` и `ratings` изменяются в разных точках кода. Ситуация усложняется ещё сильнее, когда в этих вычислениях приходится учитывать дополнительные структуры данных. Первое из представленных здесь эвристических правил решает эту проблему.

## Правило 1: выводите состояние синхронно везде, где это возможно

Перечисленные ранее недостатки можно компенсировать с помощью сигналов. Поскольку введение сигналов делает компонент реактивным, теперь можно активировать `OnPush`. Кроме того, компонент может синхронно выводить своё состояние из отдельных сигналов с помощью `computed` (вычисляемый сигнал, пересчитывается лениво при чтении и только при изменении зависимостей):

```typescript
@Component({
  […],
  changeDetection: ChangeDetectionStrategy.OnPush,
})
export class DessertsComponent implements OnInit {
  #dessertService = inject(DessertService);
  #ratingService = inject(RatingService);

  originalName = signal('');
  englishName = signal('');
  loading = signal(false);

  desserts = signal<Dessert[]>([]);
  ratings = signal<DessertIdToRatingMap>({});
  ratedDesserts = computed(() => this.toRated(this.desserts(), this.ratings()));

  […]

  loadRatings(): void {
    this.loading.set(true);

    this.#ratingService.loadExpertRatings().subscribe({
      next: (ratings) => {
        this.ratings.set(ratings);
        this.loading.set(false);
      },
      error: (error) => { […] }
    });

  […]
}
```

Это существенно повышает понятность кода: метод `loadRatings` просто загружает рейтинги и помещает их в сигнал. Вычисляемый (computed) сигнал `ratedDesserts` берёт на себя слияние `desserts` и `ratings`. Независимо от того, когда и где приложение обновляет `desserts` или `ratings`, сигнал `ratedDesserts` всегда актуален (зависимости отслеживаются автоматически по факту чтения внутри `computed`).

При применении этого паттерна важно учитывать, что `computed` способен выводить состояние только синхронно. Для асинхронного вывода Angular предоставляет [Resource API](https://www.angulararchitects.io/en/blog/asynchronous-resources-with-angulars-new-resource-api/).

## Правило 2: избегайте эффектов для распространения состояния

Эффекты (Effects) — правильный выбор, когда представление значений недостижимо через привязку данных. Однако при использовании их для распространения изменений они несут ряд подводных камней. В следующих разделах я разбираю это подробнее.

### Правильное использование эффектов

В большинстве случаев сигналы привязываются в шаблоне. Однако случается, что желаемую форму вывода невозможно получить через привязку данных. Пример — вывод сигнала в консоль для целей отладки. Другой пример — тосты (toasts), которые активируются через сервисы и предназначены для представления значения сигнала. Для этих случаев Angular предоставляет эффекты:

```typescript
[…]
constructor() {
  effect(() => {
    console.log('originalName', this.originalName());
    console.log('englishName', this.englishName());
  });

  effect(() => {
    this.#toastService.show(this.desserts().length + ' desserts loaded!');
  });
}
[…]
```

### Сигналы свободны от глитчей (glitch-free)

При написании кода, как в предыдущем разделе, необходимо осознавать, что сигналы свободны от глитчей (glitch-free — отсутствие наблюдаемых промежуточных/несогласованных значений). Это означает, что если изменить сигнал несколько раз подряд (в пределах одного кадра стека, stack frame), потребитель — например, эффект — увидит только последнее изменение:

```typescript
@Component([...])
export class AboutComponent {

  constructor() {
    const signal1 = signal('A');
    const signal2 = signal('B');

    effect(() => {
      console.log('signal1', signal1());
      console.log('signal2', signal2());
    });

    signal1.set('C');
    signal1.set('D');

    signal1.set('E');

    signal2.set('F');
  }
}
```

В этом случае в консоли мы увидим только значения `E` и `F`. Промежуточные значения пропускаются.

Это показывает, что сигналы предназначены не для моделирования событий, а для данных, которые мы хотим привязать к представлению. В последнем случае нам нужно только текущее значение, тогда как привязка промежуточных значений была бы контрпродуктивной. По этой причине эффекты, показанные в предыдущем разделе, инициируются только один раз, даже если изменений несколько подряд.

В случаях, когда нужно выразить события, путь лежит через Observable, поскольку они по своей конструкции не дают гарантии отсутствия глитчей.

### Проблемное использование эффектов

Даже при использовании эффектов сигналы применяются прежде всего для доставки нужных данных в представление. Тем не менее теоретически эффекты можно было бы использовать и для передачи состояния другим сигналам:

```typescript
effect(() => {
    this.originalName.set(this.englishName());
});
```

Однако у таких подходов есть несколько недостатков, поэтому Angular по умолчанию запрещает запись в сигналы внутри эффектов:

![Сообщение об ошибке при попытке установить сигнал в эффекте](https://www.angulararchitects.io/wp-content/uploads/2024/04/signal-write-error.png)

Один из этих недостатков — возможность возникновения неуправляемых каскадов изменений, а с ними — трудно сопровождаемого кода и циклических зависимостей. Поскольку эффекты неявно регистрируются (подписываются) на всех используемых сигналах, связанные с этим проблемы могут быть незаметны с первого взгляда. Если вы всё же хотите использовать эффекты для записи, можно заставить Angular закрыть на это глаза, установив `allowSignalWrites`:

```typescript
// Всеми силами старайтесь этого избегать
effect(() => {
    this.originalName.set(this.englishName());
  },
  { allowSignalWrites: true },
);
```

> Прикладной код должен использовать `allowSignalWrites` только как крайнее средство.

Общий консенсус в сообществе состоит в том, что прикладной код должен использовать `allowSignalWrites` только как крайнее средство. С другой стороны, такие библиотеки, как NGRX, используют эту опцию внутри себя. Но в этом случае за её корректное применение отвечают авторы библиотеки, поэтому прикладным разработчикам беспокоиться об этом не нужно.

Важно также учитывать, что эффект регистрируется и на сигналах внутри вызываемых методов. Например, следующий эффект инициируется, когда сигналы изменяются внутри `search`:

```typescript
// Всеми силами старайтесь этого избегать
effect(() => {
    this.search();
  },
  { allowSignalWrites: true },
);
```

Это ведёт к дальнейшему росту сложности. По крайней мере, эту проблему можно смягчить встроенными средствами:

```typescript
// Всеми силами старайтесь этого избегать
effect(() => {
    const originalName = this.originalName(); 
    const englishName = this.englishName();
    untracked(() => {
        this.load(originalName, englishName);
    })
  }
);
```

Функция `untracked` предотвращает распространение текущего реактивного контекста на вызываемый метод `search` (чтения сигналов внутри `untracked` не регистрируются как зависимости эффекта). Теперь Angular и сам использует этот паттерн в [отдельных случаях](https://github.com/angular/angular/pull/54614). Пример — генерация событий в дочерних компонентах, чтобы обработчик события не выполнялся в реактивном контексте кода, инициировавшего это событие. Среди других популярных библиотек, использующих эту технику, — NGRX, NGRX Signal Store и ngextensions.

### Стратегии предотвращения эффектов с записью в сигналы

Эффектов, распространяющих данные через запись в сигналы, во многих случаях можно избежать с помощью следующих подходов:

* Последовательный вывод состояния с помощью `computed` (см. правило 1 выше) или [Resource API](https://www.angulararchitects.io/en/blog/asynchronous-resources-with-angulars-new-resource-api/);
* Прямое использование событий, которые вызвали изменение сигнала.

Вместо вызова `search` в эффекте, как показано выше, приложение могло бы использовать событие изменения полей ввода для фильтров поиска. В качестве источника таких действий также можно использовать Observable. Метод `search` мог бы, например, запускаться через Observable `valueChanges` у `FormGroup`. В случаях, когда у вас есть только сигналы, их можно преобразовать в Observable с помощью RxJS Interop, предоставляемого Angular:

```typescript
@Component([…])
export class DessertsComponent {
  #dessertService = inject(DessertService);
  #ratingService = inject(RatingService);
  #toastService = inject(ToastService);

  originalName = signal('');
  englishName = signal('Cake');
  loading = signal(false);

  originalName$ = toObservable(this.originalName);
  englishName$ = toObservable(this.englishName);

  desserts$ = combineLatest({
    originalName: this.originalName$,
    englishName: this.englishName$,
  }).pipe(
    filter((c) => c.originalName.length >= 3 || c.englishName.length >= 3),
    debounceTime(300),
    tap(() => this.loading.set(true)),
    switchMap((c) => this.findDesserts(c)),
    tap(() => this.loading.set(false)),
  );

  desserts = toSignal(this.desserts$, {
    initialValue: [],
  });

  ratings = signal<DessertIdToRatingMap>({});
  ratedDesserts = computed(() => this.toRated(this.desserts(), this.ratings()));

  findDesserts(c: DessertFilter): Observable<Dessert[]> {
    return this.#dessertService.find(c).pipe(
      catchError((error) => {
        this.#toastService.show('Error loading desserts!');
        console.error(error);
        return of([]);
      }),
    );
  }
  […]
}
```

> Операторы уплощения (flattening operators) из RxJS дают гарантии при перекрывающихся асинхронных действиях и тем самым предотвращают состояния гонки (race conditions).

Использование Observable даёт здесь несколько преимуществ:

* В отличие от сигналов, Observable также пригодны для запуска асинхронных действий.
* Функция `toObservable` снимает текущий реактивный контекст с помощью `untracked`.
* RxJS поставляется с множеством мощных операторов, таких как `debounceTime`.
* Операторы уплощения из RxJS дают гарантии при перекрывающихся асинхронных действиях и тем самым предотвращают состояния гонки. В показанном примере `switchMap` обеспечивает, что при перекрытии поисковых запросов используется только результат последнего, а все остальные отменяются (предыдущие подписки отписываются).

Во многих случаях можно возразить, что вместо преобразования сигнала в Observable уместнее напрямую использовать событие, которое привело к изменению сигнала, как предложено выше. С другой стороны, по мере того как API Angular всё активнее перенимают подходы на основе сигналов, их прямое использование, вероятно, станет удобнее и интуитивнее. Поэтому это, по-видимому, серая зона, в которой нужно помнить о последствиях, в частности связанных с гарантией отсутствия глитчей у сигналов. Я планирую разобрать эту тему подробнее в одной из следующих статей.

## Правило 3: хранилища упрощают реактивный поток данных

Хранилища (stores) — такие как классический NGRX Store или лёгкий NGRX Signal Store — не только берут на себя управление состоянием (state management), но и помогают поддерживать управляемость реактивного потока данных:

![Однонаправленный поток данных с хранилищем](https://www.angulararchitects.io/wp-content/uploads/2024/04/store.png)

Приложение передаёт хранилищу своё намерение (intention) в составе события. Здесь я использую термин «намерение» в абстрактном, технологически нейтральном смысле, тем более что разные хранилища реализуют этот аспект по-разному. С Redux — а значит, и при использовании классического NGRX store — приложение отправляет в хранилище action, который тот передаёт редьюсеру (Reducer) и эффектам. Для лёгких хранилищ вроде NGRX Signal Store приложение вместо этого делегирует вызов методу, предоставляемому хранилищем.

> Перенос асинхронных операций в хранилище также компенсирует тот факт, что сигналы на данный момент рассчитаны только на синхронные действия.

Затем хранилище приступает к работе и инициирует синхронные или асинхронные операции. Если приложение использует для этого RxJS, состояний гонки можно избежать с помощью операторов уплощения, как упоминалось выше. Перенос асинхронных операций в хранилище также компенсирует тот факт, что сигналы на данный момент рассчитаны только на синхронные действия.

Результат этих операций приводит к изменению состояния, которым управляет хранилище. Это состояние можно выразить сигналами, которые могут быть отображены в другие сигналы посредством вычислений (computing, см. правило 1). Такие отображения могут происходить как в хранилище, так и в компоненте (или ином потребителе хранилища). Это зависит от того, насколько локальны или глобальны само хранилище и выводимые из него данные.

В сухом остатке: последовательное применение этого подхода поддерживает так называемый однонаправленный поток данных (unidirectional data flow), который делает поведение системы более понятным. Следующий листинг демонстрирует это с точки зрения компонента, опирающегося на NGRX Signal Store:

```typescript
@Component([…])
export class DessertsComponent {
  #store = inject(DessertStore);

  originalName = this.#store.filter.originalName;
  englishName = this.#store.filter.englishName;

  ratedDesserts = this.#store.ratedDesserts;
  loading = this.#store.loading;

  constructor() {
    this.#store.loadDesserts();
  }

  search(): void {
    this.#store.loadDesserts();
  }

  loadRatings(): void {
    this.#store.loadRatings();
  }

  updateRating(id: number, rating: number): void {
    this.#store.updateRating(id, rating);
  }

  updateFilter(filter: DessertFilter): void {
    this.#store.updateFilter(filter);
  }
}
```

Поскольку компоненту достаточно лишь делегировать вызовы хранилищу, он получается очень простым.

## Резюме

Чтобы по-настоящему задействовать преимущества сигналов, приложение должно быть спроектировано как реактивная система. Это, помимо прочего, означает, что запись значений избегается в пользу вывода значений из уже существующих. Это упрощает программный код, тем более что выведенные значения автоматически поддерживаются в актуальном состоянии.

В настоящее время сигналы прежде всего подходят для доставки данных в представление. Эффекты используются, когда для этого необходимы вызовы API, например при отображении тоста. Текущая реализация сигналов не предназначена для запуска асинхронных действий. Вместо этого путь лежит через классические события или Observable. Хранилища, способные работать и с асинхронными операциями, помогают установить однонаправленный поток данных и сделать реактивные приложения более управляемыми.
