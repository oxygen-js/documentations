Не вызывает сомнений, что сигналы (Signals) определят будущее Angular. На первый взгляд, пользоваться ими очень просто: сеттеры принимают новые значения, геттеры выдают текущие значения, а шаблоны, как и эффекты (Effects), уведомляются об изменениях.

Теперь может возникнуть соблазн обращаться с сигналами как с обычными переменными. В принципе это работает и может быть практичным вариантом при миграции существующего кода. Однако в этом случае преимущества сигналов и реактивных систем проявляются лишь в ограниченной мере. Кроме того, имеется ряд подводных камней, а код оказывается не таким понятным и, следовательно, не таким сопровождаемым, каким мог бы быть.

Чтобы предотвратить подобные ситуации, в этой статье я хотел бы дать вам три простых эмпирических правила, которые позволяют использовать сигналы максимально идиоматично.

## Исходный пример с некоторым пространством для улучшения

Три обещанных правила я хотел бы обсудить на примере простого Angular-приложения:

![Пример приложения](https://www.angulararchitects.io/wp-content/uploads/2024/04/dessert-app.png)

Первая рассматриваемая реализация не является реактивной и также оставляет некоторое пространство для улучшения:

```typescript
@Component([…])
export class DessertsComponent implements OnInit {
  #dessertService = inject(DessertService);
  #ratingService = inject(RatingService);
  […]

  originalName = '';
  englishName = '';
  loading = false;

  desserts: Dessert[] = [];

  ngOnInit(): void {
    this.search();
  }

  search(): void {
    const filter: DessertFilter = {
      originalName: this.originalName,
      englishName: this.englishName,
    };

    this.loading = true;

    this.#dessertService.find(filter).subscribe({
      next: (desserts) => {
        this.desserts = desserts;
        this.loading = false;
      },
      error: (error) => { […] },
    });
  }

  toRated(desserts: Dessert[], ratings: DessertIdToRatingMap): Dessert[] {
    return desserts.map((d) =>
      ratings[d.id] ? { ...d, rating: ratings[d.id] } : d,
    );
  }

  loadRatings(): void {
    this.loading = true;

    this.#ratingService.loadExpertRatings().subscribe({
      next: (ratings) => {
        const rated = this.toRated(this.desserts, ratings);
        this.desserts = rated;
        this.loading = false;
      },
      error: (error) => { […] },
    });
  }
  […]
}
```

Поскольку привязываемые свойства не являются ни Observable, ни сигналами, стратегию `OnPush` для повышения производительности привязки данных использовать нельзя. При более внимательном рассмотрении мы также замечаем, что метод `loadRatings` обновляет массив `desserts`, хотя его фактическая задача — загрузка рейтингов — не имеет к этому никакого отношения.

Кроме того, разработчики должны помнить, что после любых изменений рейтингов массив `desserts` тоже необходимо изменить. Именно это и может приводить к трудно сопровождаемому коду и скрытым ошибкам — особенно когда `desserts` и `ratings` меняются в разных местах. Всё становится ещё сложнее, когда в этих вычислениях приходится учитывать дополнительные структуры данных. Первое из представленных здесь эмпирических правил решает эту проблему.

## Правило 1: выводите состояние синхронно везде, где это возможно

Упомянутые ранее недостатки можно компенсировать с помощью сигналов. Поскольку введение сигналов делает компонент реактивным, теперь можно активировать `OnPush`. Кроме того, компонент может синхронно выводить своё состояние из отдельных сигналов с помощью `computed`:

```typescript
@Component({
  […],
  changeDetection: ChangeDetectionStrategy.OnPush,
})
export class DessertsComponent implements OnInit {
  #dessertService = inject(DessertService);
  #ratingService = inject(RatingService);

  originalName = signal('');
  englishName = signal('');
  loading = signal(false);

  desserts = signal<Dessert[]>([]);
  ratings = signal<DessertIdToRatingMap>({});
  ratedDesserts = computed(() => this.toRated(this.desserts(), this.ratings()));

  […]

  loadRatings(): void {
    this.loading.set(true);

    this.#ratingService.loadExpertRatings().subscribe({
      next: (ratings) => {
        this.ratings.set(ratings);
        this.loading.set(false);
      },
      error: (error) => { […] }
    });

  […]
}
```

Это делает код намного понятнее: метод `loadRatings` просто загружает рейтинги и помещает их в сигнал. Вычисляемый (computed) сигнал `ratedDesserts` берёт на себя объединение `desserts` и `ratings`. Независимо от того, когда и где приложение обновляет `desserts` или `ratings`, сигнал `ratedDesserts` всегда актуален.

При применении этого паттерна важно учитывать, что `computed` может выводить состояние только синхронным образом. Для асинхронного вывода Angular предоставляет [Resource API](https://www.angulararchitects.io/en/blog/asynchronous-resources-with-angulars-new-resource-api/).

## Правило 2: избегайте эффектов для распространения состояния

Эффекты (Effects) — правильный выбор, когда представление значений не может быть достигнуто через привязку данных. Однако они привносят ряд подводных камней при использовании для распространения изменений. В следующих разделах я останавливаюсь на этом подробнее.

### Правильное использование эффектов

В большинстве случаев сигналы привязываются в шаблоне. Однако случается, что желаемую форму вывода невозможно получить через привязку данных. Пример этого — вывод сигнала в консоль для целей отладки. Другой пример — тосты (toasts), которые могут активироваться через сервисы и предназначены для представления значения сигнала. Для этих случаев Angular предоставляет эффекты:

```typescript
[…]
constructor() {
  effect(() => {
    console.log('originalName', this.originalName());
    console.log('englishName', this.englishName());
  });

  effect(() => {
    this.#toastService.show(this.desserts().length + ' desserts loaded!');
  });
}
[…]
```

### Сигналы свободны от глитчей (glitch-free)

При написании кода, как в предыдущем разделе, нам нужно осознавать, что сигналы свободны от глитчей (glitch-free). Это означает, что если вы меняете сигнал несколько раз подряд (в пределах одного кадра стека), потребитель — например, эффект — увидит только последнее изменение:

```typescript
@Component([...])
export class AboutComponent {

  constructor() {
    const signal1 = signal('A');
    const signal2 = signal('B');

    effect(() => {
      console.log('signal1', signal1());
      console.log('signal2', signal2());
    });

    signal1.set('C');
    signal1.set('D');

    signal1.set('E');

    signal2.set('F');
  }
}
```

В этом случае мы увидим в консоли только значения `E` и `F`. Промежуточные значения пропускаются.

Это показывает, что сигналы предназначены не для моделирования событий, а для данных, которые мы хотим привязать к представлению. В последнем случае нам нужно лишь текущее значение, тогда как привязка промежуточных значений была бы контрпродуктивной. По этой причине эффекты, показанные в предыдущем разделе, срабатывают только один раз, даже если изменений несколько подряд.

В случаях, когда мы хотим выразить события, путь лежит через Observable, поскольку они по своей конструкции не имеют этой гарантии отсутствия глитчей.

### Проблемное использование эффектов

Даже при использовании эффектов сигналы применяются прежде всего для того, чтобы доставить нужные данные в представление. Тем не менее, теоретически эффекты можно было бы использовать и для передачи состояния другим сигналам:

```typescript
effect(() => {
    this.originalName.set(this.englishName());
});
```

Однако у таких подходов есть несколько недостатков, поэтому Angular по умолчанию запрещает запись сигналов внутри эффектов:

![Сообщение об ошибке при попытке установить сигнал в эффекте](https://www.angulararchitects.io/wp-content/uploads/2024/04/signal-write-error.png)

Один из этих недостатков состоит в том, что могут возникнуть неуправляемые каскады изменений, а вместе с ними — трудно сопровождаемый код и циклические зависимости. Поскольку эффекты неявно регистрируются на всех используемых сигналах, связанные с этим проблемы могут быть даже незаметны с первого взгляда. Если вы всё же хотите использовать эффекты для записи, вы можете заставить Angular закрыть на это глаза, установив `allowSignalWrites`:

```typescript
// Всеми силами старайтесь этого избегать
effect(() => {
    this.originalName.set(this.englishName());
  },
  { allowSignalWrites: true },
);
```

> Прикладной код должен использовать `allowSignalWrites` только в крайнем случае.

Общий консенсус в сообществе состоит в том, что прикладной код должен использовать `allowSignalWrites` только в крайнем случае. С другой стороны, такие библиотеки, как NGRX, используют эту опцию внутри себя. Однако в этом случае за её корректное использование отвечают авторы библиотеки, поэтому прикладным разработчикам не нужно об этом беспокоиться.

Важно также учитывать, что эффект регистрируется и на сигналах в вызываемых методах. Например, следующий эффект срабатывает, когда сигналы изменяются внутри `search`:

```typescript
// Всеми силами старайтесь этого избегать
effect(() => {
    this.search();
  },
  { allowSignalWrites: true },
);
```

Это ведёт к дальнейшему росту сложности. По крайней мере, эту проблему можно смягчить встроенными средствами:

```typescript
// Всеми силами старайтесь этого избегать
effect(() => {
    const originalName = this.originalName(); 
    const englishName = this.englishName();
    untracked(() => {
        this.load(originalName, englishName);
    })
  }
);
```

Функция `untracked` предотвращает «перетекание» текущего реактивного контекста в вызываемый метод `search`. Теперь Angular и сам использует этот паттерн в [отдельных случаях](https://github.com/angular/angular/pull/54614). Пример этого — генерация событий в дочерних компонентах, чтобы обработчик события не выполнялся в реактивном контексте кода, который это событие вызвал. Среди других популярных библиотек, использующих эту технику, — NGRX, NGRX Signal Store и ngextensions.

### Стратегии предотвращения эффектов с записью в сигналы

Эффектов, которые распространяют данные через запись в сигналы, во многих случаях можно избежать с помощью следующих подходов:

* Последовательный вывод состояния с помощью `computed` (см. правило 1 выше) или [Resource API](https://www.angulararchitects.io/en/blog/asynchronous-resources-with-angulars-new-resource-api/);
* Прямое использование событий, которые вызвали изменение сигнала.

Вместо того чтобы вызывать `search` в эффекте, как показано выше, приложение могло бы вместо этого использовать событие изменения полей ввода для фильтров поиска. В качестве источника таких действий также можно использовать Observable. Метод `search` мог бы, например, запускаться через Observable `valueChanges` у `FormGroup`. В случаях, когда у вас есть только сигналы, их можно преобразовать в Observable с помощью RxJS Interop, предлагаемого Angular:

```typescript
@Component([…])
export class DessertsComponent {
  #dessertService = inject(DessertService);
  #ratingService = inject(RatingService);
  #toastService = inject(ToastService);

  originalName = signal('');
  englishName = signal('Cake');
  loading = signal(false);

  originalName$ = toObservable(this.originalName);
  englishName$ = toObservable(this.englishName);

  desserts$ = combineLatest({
    originalName: this.originalName$,
    englishName: this.englishName$,
  }).pipe(
    filter((c) => c.originalName.length >= 3 || c.englishName.length >= 3),
    debounceTime(300),
    tap(() => this.loading.set(true)),
    switchMap((c) => this.findDesserts(c)),
    tap(() => this.loading.set(false)),
  );

  desserts = toSignal(this.desserts$, {
    initialValue: [],
  });

  ratings = signal<DessertIdToRatingMap>({});
  ratedDesserts = computed(() => this.toRated(this.desserts(), this.ratings()));

  findDesserts(c: DessertFilter): Observable<Dessert[]> {
    return this.#dessertService.find(c).pipe(
      catchError((error) => {
        this.#toastService.show('Error loading desserts!');
        console.error(error);
        return of([]);
      }),
    );
  }
  […]
}
```

> Операторы уплощения (flattening operators), предлагаемые RxJS, дают гарантии при перекрывающихся асинхронных действиях и тем самым предотвращают состояния гонки (race conditions).

Использование Observable имеет в этом месте несколько преимуществ:

* В отличие от сигналов, Observable также подходят для запуска асинхронных действий.
* Функция `toObservable` снимает текущий реактивный контекст с помощью `untracked`.
* RxJS поставляется с множеством мощных операторов, таких как `debounceTime`.
* Операторы уплощения, предлагаемые RxJS, дают гарантии при перекрывающихся асинхронных действиях и тем самым предотвращают состояния гонки. В показанном примере `switchMap` обеспечивает, что при перекрытии поисковых запросов используется только результат последнего, а все остальные отменяются.

Во многих случаях можно было бы возразить, что вместо преобразования сигнала в Observable уместнее напрямую использовать событие, которое привело к изменению сигнала, как предложено выше. С другой стороны, по мере того как API Angular всё активнее перенимают подходы, основанные на сигналах, их прямое использование, вероятно, станет удобнее и интуитивнее. Поэтому это, по-видимому, серая зона, в которой нам нужно помнить о последствиях, таких как те, что связаны с гарантией отсутствия глитчей у сигналов. Я планирую обсудить эту тему подробнее в одной из следующих статей.

## Правило 3: хранилища упрощают реактивный поток данных

Хранилища (stores), такие как классический NGRX Store или лёгкий NGRX Signal Store, не только берут на себя управление состоянием (state management), но и помогают сохранять управляемость реактивного потока данных:

![Однонаправленный поток данных с хранилищем](https://www.angulararchitects.io/wp-content/uploads/2024/04/store.png)

Приложение передаёт своё намерение в хранилище в составе события. Здесь я использую термин «намерение» в абстрактном, технологически нейтральном смысле, тем более что разные хранилища реализуют этот аспект по-разному. С Redux, а значит, и при использовании классического NGRX store, приложение отправляет в хранилище action, который оно передаёт редьюсеру (Reducer) и эффектам. Для лёгких хранилищ вроде NGRX Signal Store приложение вместо этого делегирует вызов методу, предлагаемому хранилищем.

> Перенос асинхронных операций в хранилище также компенсирует тот факт, что сигналы в настоящее время рассчитаны только на синхронные действия.

Затем хранилище принимается за дело и инициирует синхронные или асинхронные операции. Если приложение использует для этого RxJS, состояний гонки можно избежать с помощью операторов уплощения, как упоминалось выше. Перенос асинхронных операций в хранилище также компенсирует тот факт, что сигналы в настоящее время рассчитаны только на синхронные действия.

Результат этих операций приводит к изменению состояния, которым управляет хранилище. Это состояние можно выразить сигналами, которые могут быть отображены в другие сигналы с помощью вычислений (см. правило 1). Такие отображения могут происходить как в хранилище, так и в компоненте (или в другом потребителе хранилища). Это зависит от того, насколько локальны или глобальны само хранилище и выводимые данные.

В итоге последовательное применение этого подхода поддерживает так называемый однонаправленный поток данных (unidirectional data flow), который делает поведение системы более понятным. Следующий листинг демонстрирует это с точки зрения компонента, опирающегося на NGRX Signal Store:

```typescript
@Component([…])
export class DessertsComponent {
  #store = inject(DessertStore);

  originalName = this.#store.filter.originalName;
  englishName = this.#store.filter.englishName;

  ratedDesserts = this.#store.ratedDesserts;
  loading = this.#store.loading;

  constructor() {
    this.#store.loadDesserts();
  }

  search(): void {
    this.#store.loadDesserts();
  }

  loadRatings(): void {
    this.#store.loadRatings();
  }

  updateRating(id: number, rating: number): void {
    this.#store.updateRating(id, rating);
  }

  updateFilter(filter: DessertFilter): void {
    this.#store.updateFilter(filter);
  }
}
```

Поскольку компоненту достаточно лишь делегировать вызовы хранилищу, он получается очень простым.

## Резюме

Чтобы по-настоящему воспользоваться преимуществами сигналов, приложение должно быть спроектировано как реактивная система. Это, помимо прочего, означает, что запись значений избегается в пользу вывода значений из уже существующих. Это упрощает программный код, тем более что выведенные значения автоматически поддерживаются в актуальном состоянии.

В настоящее время сигналы прежде всего подходят для доставки данных в представление. Эффекты используются, когда для этого необходимы вызовы API, например при отображении тоста. Текущая реализация сигналов не предназначена для запуска асинхронных действий. Вместо этого путь лежит через классические события или Observable. Хранилища, которые умеют работать и с асинхронными операциями, помогают выстроить однонаправленный поток данных и сделать реактивные приложения более управляемыми.
