Никто не спорит с тем, что именно сигналы (Signals) определят будущее Angular. На первый взгляд пользоваться ими совсем просто: сеттеры принимают новые значения, геттеры отдают текущие, а шаблоны и эффекты (Effects) узнают об изменениях.

Возникает соблазн обращаться с сигналами как с обычными переменными. В принципе так и можно сделать — это вполне разумный вариант при миграции старого кода. Но в этом случае преимущества сигналов и реактивных систем раскрываются лишь отчасти. Вдобавок появляются подводные камни, а сам код выходит не таким прозрачным и, следовательно, не таким удобным в сопровождении, каким он мог бы быть.

Чтобы такого не случалось, в этой статье я хочу дать вам три простых эмпирических правила, которые позволят использовать сигналы настолько идиоматично, насколько это возможно.

## Стартовый пример, который можно улучшить

Три обещанных правила я разберу на примере простого Angular-приложения:

![Пример приложения](https://www.angulararchitects.io/wp-content/uploads/2024/04/dessert-app.png)

Первая реализация, которую мы рассмотрим, не является реактивной и тоже оставляет простор для улучшений:

```typescript
@Component([…])
export class DessertsComponent implements OnInit {
  #dessertService = inject(DessertService);
  #ratingService = inject(RatingService);
  […]

  originalName = '';
  englishName = '';
  loading = false;

  desserts: Dessert[] = [];

  ngOnInit(): void {
    this.search();
  }

  search(): void {
    const filter: DessertFilter = {
      originalName: this.originalName,
      englishName: this.englishName,
    };

    this.loading = true;

    this.#dessertService.find(filter).subscribe({
      next: (desserts) => {
        this.desserts = desserts;
        this.loading = false;
      },
      error: (error) => { […] },
    });
  }

  toRated(desserts: Dessert[], ratings: DessertIdToRatingMap): Dessert[] {
    return desserts.map((d) =>
      ratings[d.id] ? { ...d, rating: ratings[d.id] } : d,
    );
  }

  loadRatings(): void {
    this.loading = true;

    this.#ratingService.loadExpertRatings().subscribe({
      next: (ratings) => {
        const rated = this.toRated(this.desserts, ratings);
        this.desserts = rated;
        this.loading = false;
      },
      error: (error) => { […] },
    });
  }
  […]
}
```

Поскольку привязываемые свойства не являются ни Observable, ни сигналами, стратегию `OnPush` для ускорения привязки данных применить нельзя. Присмотревшись повнимательнее, мы замечаем ещё кое-что: метод `loadRatings` обновляет массив `desserts`, хотя его настоящая задача — загрузка рейтингов — никак с этим не связана.

К тому же разработчик должен помнить, что после любых изменений рейтингов нужно ещё и поправить массив `desserts`. Именно из-за подобных вещей код становится трудным в сопровождении и обрастает скрытыми багами — особенно когда `desserts` и `ratings` меняются в разных местах. А если в этих вычислениях участвуют ещё какие-то структуры данных, всё становится только запутаннее. Первое правило как раз и решает эту проблему.

## Правило 1: выводите состояние синхронно, где только возможно

Перечисленные недостатки можно устранить с помощью сигналов. Поскольку их появление делает компонент реактивным, теперь можно включить `OnPush`. Кроме того, компонент может синхронно выводить своё состояние из отдельных сигналов с помощью `computed`:

```typescript
@Component({
  […],
  changeDetection: ChangeDetectionStrategy.OnPush,
})
export class DessertsComponent implements OnInit {
  #dessertService = inject(DessertService);
  #ratingService = inject(RatingService);

  originalName = signal('');
  englishName = signal('');
  loading = signal(false);

  desserts = signal<Dessert[]>([]);
  ratings = signal<DessertIdToRatingMap>({});
  ratedDesserts = computed(() => this.toRated(this.desserts(), this.ratings()));

  […]

  loadRatings(): void {
    this.loading.set(true);

    this.#ratingService.loadExpertRatings().subscribe({
      next: (ratings) => {
        this.ratings.set(ratings);
        this.loading.set(false);
      },
      error: (error) => { […] }
    });

  […]
}
```

Код стал заметно прозрачнее: метод `loadRatings` просто загружает рейтинги и кладёт их в сигнал. Вычисляемый (computed) сигнал `ratedDesserts` берёт на себя объединение `desserts` и `ratings`. Когда бы и где бы приложение ни обновило `desserts` или `ratings`, сигнал `ratedDesserts` всегда остаётся актуальным.

Применяя этот подход, важно помнить: `computed` умеет выводить состояние только синхронно. Для асинхронного вывода Angular предлагает [Resource API](https://www.angulararchitects.io/en/blog/asynchronous-resources-with-angulars-new-resource-api/).

## Правило 2: не используйте эффекты для распространения состояния

Эффекты (Effects) — правильный выбор, когда отобразить значения через привязку данных невозможно. Но при использовании их для распространения изменений они таят немало ловушек. В следующих разделах я расскажу об этом подробнее.

### Как правильно использовать эффекты

В большинстве случаев сигналы привязываются в шаблоне. Однако бывает, что нужную форму вывода через привязку данных получить нельзя. Пример — вывод сигнала в консоль для отладки. Другой пример — тосты (toasts), которые активируются через сервисы и должны показать значение сигнала. Для таких случаев Angular и предоставляет эффекты:

```typescript
[…]
constructor() {
  effect(() => {
    console.log('originalName', this.originalName());
    console.log('englishName', this.englishName());
  });

  effect(() => {
    this.#toastService.show(this.desserts().length + ' desserts loaded!');
  });
}
[…]
```

### Сигналы свободны от «глитчей» (glitch-free)

Когда пишешь код вроде показанного выше, нужно помнить: сигналы свободны от «глитчей» (glitch-free). Это значит, что если вы меняете сигнал несколько раз подряд (в пределах одного кадра стека), потребитель — например, эффект — увидит только последнее изменение:

```typescript
@Component([...])
export class AboutComponent {

  constructor() {
    const signal1 = signal('A');
    const signal2 = signal('B');

    effect(() => {
      console.log('signal1', signal1());
      console.log('signal2', signal2());
    });

    signal1.set('C');
    signal1.set('D');

    signal1.set('E');

    signal2.set('F');
  }
}
```

В этом случае в консоли мы увидим только значения `E` и `F`. Промежуточные значения пропускаются.

Это показывает, что сигналы предназначены не для моделирования событий, а для данных, которые мы хотим привязать к представлению. А в этом случае нам нужно лишь текущее значение, и привязка промежуточных была бы только во вред. Именно поэтому эффекты из предыдущего раздела срабатывают лишь один раз, даже если изменений было несколько подряд.

Когда же нам нужно выразить именно события, на сцену выходят Observable: по своей природе они не дают такой гарантии отсутствия «глитчей».

### Проблемное использование эффектов

Даже при работе с эффектами сигналы используются в первую очередь для того, чтобы доставить нужные данные в представление. Теоретически, однако, эффекты можно было бы применить и для передачи состояния другим сигналам:

```typescript
effect(() => {
    this.originalName.set(this.englishName());
});
```

Но у такого подхода есть несколько недостатков, поэтому по умолчанию Angular запрещает запись в сигналы внутри эффектов:

![Сообщение об ошибке при попытке установить сигнал внутри эффекта](https://www.angulararchitects.io/wp-content/uploads/2024/04/signal-write-error.png)

Один из недостатков в том, что могут возникнуть неуправляемые каскады изменений, а с ними — трудно сопровождаемый код и циклические зависимости. Поскольку эффекты неявно регистрируются на всех используемых сигналах, связанные с этим проблемы можно даже не заметить с первого взгляда. Если вам всё же хочется применять эффекты для записи, можно «уговорить» Angular закрыть на это глаза, выставив `allowSignalWrites`:

```typescript
// Изо всех сил старайтесь этого избегать
effect(() => {
    this.originalName.set(this.englishName());
  },
  { allowSignalWrites: true },
);
```

> Прикладной код должен использовать `allowSignalWrites` только как крайнее средство.

В сообществе сложился общий консенсус: в прикладном коде `allowSignalWrites` стоит применять лишь как крайнюю меру. С другой стороны, такие библиотеки, как NGRX, используют эту опцию у себя внутри. Но в этом случае за её корректное применение отвечают авторы библиотеки, так что прикладным разработчикам об этом беспокоиться не нужно.

Важно также помнить, что эффект регистрируется и на сигналах внутри вызываемых методов. Например, следующий эффект сработает, когда изменятся сигналы внутри `search`:

```typescript
// Изо всех сил старайтесь этого избегать
effect(() => {
    this.search();
  },
  { allowSignalWrites: true },
);
```

Это ещё сильнее повышает сложность. По крайней мере, эту проблему можно немного смягчить встроенными средствами:

```typescript
// Изо всех сил старайтесь этого избегать
effect(() => {
    const originalName = this.originalName(); 
    const englishName = this.englishName();
    untracked(() => {
        this.load(originalName, englishName);
    })
  }
);
```

Функция `untracked` не даёт текущему реактивному контексту «перетечь» в вызываемый метод `search`. Кстати, теперь Angular и сам применяет этот приём в [отдельных случаях](https://github.com/angular/angular/pull/54614). Пример — генерация событий в дочерних компонентах: так обработчик события не выполняется в реактивном контексте кода, который это событие вызвал. Эту же технику используют и другие популярные библиотеки: NGRX, NGRX Signal Store, ngextensions.

### Как избежать эффектов с записью в сигналы

Эффектов, которые распространяют данные через запись в сигналы, во многих случаях можно избежать следующими способами:

* Последовательный вывод состояния с помощью `computed` (см. правило 1 выше) или [Resource API](https://www.angulararchitects.io/en/blog/asynchronous-resources-with-angulars-new-resource-api/);
* Прямое использование событий, которые и привели к изменению сигнала.

Вместо того чтобы вызывать `search` внутри эффекта, как показано выше, приложение могло бы воспользоваться событием изменения полей ввода для фильтров поиска. В качестве источника таких действий подойдут и Observable. Метод `search` мог бы, к примеру, запускаться через Observable `valueChanges` у `FormGroup`. А если у вас на руках только сигналы, их можно превратить в Observable с помощью RxJS Interop, который предлагает Angular:

```typescript
@Component([…])
export class DessertsComponent {
  #dessertService = inject(DessertService);
  #ratingService = inject(RatingService);
  #toastService = inject(ToastService);

  originalName = signal('');
  englishName = signal('Cake');
  loading = signal(false);

  originalName$ = toObservable(this.originalName);
  englishName$ = toObservable(this.englishName);

  desserts$ = combineLatest({
    originalName: this.originalName$,
    englishName: this.englishName$,
  }).pipe(
    filter((c) => c.originalName.length >= 3 || c.englishName.length >= 3),
    debounceTime(300),
    tap(() => this.loading.set(true)),
    switchMap((c) => this.findDesserts(c)),
    tap(() => this.loading.set(false)),
  );

  desserts = toSignal(this.desserts$, {
    initialValue: [],
  });

  ratings = signal<DessertIdToRatingMap>({});
  ratedDesserts = computed(() => this.toRated(this.desserts(), this.ratings()));

  findDesserts(c: DessertFilter): Observable<Dessert[]> {
    return this.#dessertService.find(c).pipe(
      catchError((error) => {
        this.#toastService.show('Error loading desserts!');
        console.error(error);
        return of([]);
      }),
    );
  }
  […]
}
```

> Операторы уплощения (flattening operators) из RxJS дают гарантии при перекрывающихся асинхронных действиях и тем самым предотвращают состояния гонки (race conditions).

У использования Observable здесь есть несколько преимуществ:

* В отличие от сигналов, Observable отлично подходят для запуска асинхронных действий.
* Функция `toObservable` снимает текущий реактивный контекст через `untracked`.
* RxJS приносит с собой массу мощных операторов вроде `debounceTime`.
* Операторы уплощения из RxJS дают гарантии при перекрывающихся асинхронных действиях и тем самым предотвращают состояния гонки. В показанном примере `switchMap` гарантирует, что при перекрывающихся поисковых запросах используется только результат последнего, а все остальные отменяются.

Во многих случаях можно возразить, что вместо превращения сигнала в Observable уместнее напрямую использовать событие, которое привело к изменению сигнала, — как и предлагалось выше. С другой стороны, по мере того как API Angular всё активнее переходят на подходы, основанные на сигналах, использовать их напрямую станет, вероятно, удобнее и интуитивнее. Так что это похоже на «серую зону», где нужно держать в уме последствия — например, связанные с гарантией отсутствия «глитчей» у сигналов. Эту тему я планирую разобрать подробнее в одной из следующих статей.

## Правило 3: хранилища упрощают реактивный поток данных

Хранилища (stores) — будь то классический NGRX Store или лёгкий NGRX Signal Store — берут на себя не только управление состоянием (state management), но и помогают держать реактивный поток данных под контролем:

![Однонаправленный поток данных с хранилищем](https://www.angulararchitects.io/wp-content/uploads/2024/04/store.png)

Приложение передаёт хранилищу своё намерение в составе события. Здесь я использую термин «намерение» в абстрактном, не привязанном к конкретной технологии смысле, тем более что разные хранилища реализуют этот аспект по-разному. С Redux — а значит, и с классическим NGRX store — приложение отправляет в хранилище action, который тот направляет редьюсеру (Reducer) и эффектам. В случае лёгких хранилищ вроде NGRX Signal Store приложение вместо этого делегирует вызов методу, который предоставляет само хранилище.

> Перенос асинхронных операций в хранилище заодно компенсирует и то, что сигналы пока рассчитаны только на синхронные действия.

Дальше хранилище принимается за дело и запускает синхронные или асинхронные операции. Если приложение использует для этого RxJS, состояний гонки можно избежать с помощью операторов уплощения, о которых говорилось выше. Перенос асинхронных операций в хранилище заодно компенсирует и то, что сигналы пока рассчитаны только на синхронные действия.

Результат этих операций приводит к изменению состояния, которым управляет хранилище. Это состояние можно выразить сигналами, а их, в свою очередь, отобразить в другие сигналы с помощью вычислений (см. правило 1). Такие отображения могут происходить как в самом хранилище, так и в компоненте (или другом потребителе хранилища). Зависит это от того, насколько локальны или глобальны само хранилище и выводимые из него данные.

Итог: последовательное применение такого подхода поддерживает так называемый однонаправленный поток данных (unidirectional data flow), благодаря которому поведение системы становится понятнее. Следующий листинг демонстрирует это с точки зрения компонента, опирающегося на NGRX Signal Store:

```typescript
@Component([…])
export class DessertsComponent {
  #store = inject(DessertStore);

  originalName = this.#store.filter.originalName;
  englishName = this.#store.filter.englishName;

  ratedDesserts = this.#store.ratedDesserts;
  loading = this.#store.loading;

  constructor() {
    this.#store.loadDesserts();
  }

  search(): void {
    this.#store.loadDesserts();
  }

  loadRatings(): void {
    this.#store.loadRatings();
  }

  updateRating(id: number, rating: number): void {
    this.#store.updateRating(id, rating);
  }

  updateFilter(filter: DessertFilter): void {
    this.#store.updateFilter(filter);
  }
}
```

Поскольку компоненту достаточно лишь делегировать вызовы хранилищу, он получается очень простым.

## Итоги

Чтобы по-настоящему раскрыть преимущества сигналов, приложение нужно проектировать как реактивную систему. Это, помимо прочего, означает, что запись значений отходит на второй план в пользу их вывода из уже существующих. Так программный код упрощается, тем более что выведенные значения автоматически поддерживаются в актуальном состоянии.

Сейчас сигналы прежде всего подходят для доставки данных в представление. Эффекты применяются, когда для этого нужны вызовы API — например, при показе тоста. Текущая реализация сигналов не предназначена для запуска асинхронных действий. Вместо них на сцену выходят классические события или Observable. А хранилища, которые умеют работать и с асинхронными операциями, помогают выстроить однонаправленный поток данных и делают реактивные приложения более управляемыми.
