Новые сигналы (Signals) в Angular — это простой реактивный «строительный блок». Но, как это часто бывает, дьявол кроется в деталях. В этой статье я поделюсь двумя советами, которые помогут вам использовать сигналы проще и увереннее.

## Отправная теория: однонаправленный поток данных с сигналами

Подход к построению однонаправленного потока данных, о котором мы говорили в прошлой статье, послужит нам отправной точкой для двух советов, которые я разберу здесь:

![Однонаправленный поток данных со стором](https://www.angulararchitects.io/wp-content/uploads/2024/05/store.png)

Обработчики UI-событий делегируют работу стору (store). Я использую для этого абстрактный термин _намерение_ (intention), потому что в разных сторах это реализовано совершенно по-разному: в основанном на Redux NGRX Store отправляются (dispatch) экшены (Actions), а вот в легковесном NGRX Signal Store компонент просто вызывает метод, который предоставляет стор.

Дальше стор выполняет синхронные или асинхронные задачи. Обычно они приводят к изменению состояния (state), которое приложение доставляет во вьюхи отдельных компонентов через сигналы. В рамках этого потока данных состояние можно проецировать на вью-модели (View Models) с помощью _computed_ — то есть на структуры данных, которые представляют взгляд отдельных сценариев использования на состояние.

В основе этого подхода лежит тот факт, что сигналы в первую очередь подходят для того, чтобы синхронно информировать вью о данных и их изменениях. Для асинхронных задач и для представления событий они подходят хуже. Во-первых, они не дают простого способа справиться с перекрывающимися асинхронными запросами и вытекающими из них состояниями гонки (race conditions). Кроме того, они не умеют напрямую представлять состояния ошибок. Во-вторых, сигналы игнорируют промежуточные состояния, возникающие, когда значения меняются одно за другим. Это желательное свойство называется «отсутствием глитчей» (glitch-free).

Например, если сигнал меняется с 1 на 2, а сразу после этого с 2 на 3, потребитель получит уведомление только о тройке. Это полезно ещё и для производительности привязки данных (data binding): ведь обновление промежуточными результатами привело бы к лишней работе.

## Совет 1: сигналы прекрасно сочетаются с RxJS

Сигналы намеренно сделаны простыми. Поэтому возможностей у них меньше, чем у RxJS, который закрепился в мире Angular уже много лет назад. Благодаря интеропу с RxJS (RxJS interop), который привносит Angular, вы можете объединить лучшее из обоих миров. Следующий листинг это демонстрирует:

```typescript
@Component({
  selector: 'app-desserts',
  standalone: true,
  imports: [DessertCardComponent, FormsModule, JsonPipe],
  templateUrl: './desserts.component.html',
  styleUrl: './desserts.component.css',
  changeDetection: ChangeDetectionStrategy.OnPush,
})
export class DessertsComponent {
  #dessertService = inject(DessertService);
  #ratingService = inject(RatingService);
  #toastService = inject(ToastService);

  originalName = signal('');
  englishName = signal('Cake');
  loading = signal(false);

  ratings = signal<DessertIdToRatingMap>({});
  ratedDesserts = computed(() => this.toRated(this.desserts(), this.ratings()));

  originalName$ = toObservable(this.originalName);
  englishName$ = toObservable(this.englishName);

  desserts$ = combineLatest({
    originalName: this.originalName$,
    englishName: this.englishName$,
  }).pipe(
    filter((c) => c.originalName.length >= 3 || c.englishName.length >= 3),
    debounceTime(300),
    tap(() => this.loading.set(true)),
    switchMap((c) =>
      this.#dessertService.find(c).pipe(
        catchError((error) => {
          this.#toastService.show('Error loading desserts!');
          console.error(error);
          return of([]);
        }),
      ),
    ),
    tap(() => this.loading.set(false)),
  );

  desserts = toSignal(this.desserts$, {
    initialValue: [],
  });

  […]
}
```

Этот пример конвертирует сигналы _from_ и _to_ в Observable-ы и на этой основе реализует typeahead (поиск с подсказками по мере ввода). Для этого используются операторы _filter_, _debounceTime_ и _switchMap_ из RxJS. Последний к тому же предотвращает состояния гонки от перекрывающихся запросов: он учитывает только самый свежий запрос, а все остальные ещё выполняющиеся запросы отменяются.

В конце получившийся Observable снова конвертируется в сигнал, чтобы приложение могло дальше работать с новым Signals API. Разумеется, из соображений производительности не стоит слишком часто прыгать между этими двумя мирами.

В отличие от схемы выше, здесь стор не используется. И _намерение_, и асинхронное действие происходят прямо внутри реактивного потока данных. Если вынести этот поток данных в сервис, который раздаёт загруженные данные (например, через _shareReplay_), то такой сервис уже можно рассматривать как простой стор. Но, как и на схеме выше, компонент делегирует выполнение асинхронной задачи куда-то наружу и получает результирующее состояние через сигналы.

### RxJS внутри сторов

RxJS часто используется и в самих сторах. Например, традиционный NGRX store применяет RxJS для экшенов (Actions) и эффектов (Effects). В качестве альтернативы эффектам NGRX Signal Store предлагает реактивные методы, которые можно описать через _rxMethod_:

```typescript
export const DessertStore = signalStore(
  { providedIn: 'root' },
  withState({
    filter: {
      originalName: '',
      englishName: 'Cake',
    },
    loading: false,
    ratings: {} as DessertIdToRatingMap,
    desserts: [] as Dessert[],
  }),
  […]
  withMethods(
    (
      store,
      dessertService = inject(DessertService),
      toastService = inject(ToastService),
    ) => ({

      […]
      loadDessertsByFilter: rxMethod<DessertFilter>(
        pipe(
          filter(
            (f) => f.originalName.length >= 3 || f.englishName.length >= 3,
          ),
          debounceTime(300),
          tap(() => patchState(store, { loading: true })),
          switchMap((f) =>
            dessertService.find(f).pipe(
              tapResponse({
                next: (desserts) => {
                  patchState(store, { desserts, loading: false });
                },
                error: (error) => {
                  toastService.show('Error loading desserts!');
                  console.error(error);
                  patchState(store, { loading: false });
                },
              }),
            ),
          ),
        ),
      ),
    }),
  ),
  withHooks({
    onInit(store) {
      const filter = store.filter;
      store.loadDessertsByFilter(filter);
    },
  }),
);
```

В этом примере в сторе настраивается реактивный метод _loadDessertsByFilter_. Поскольку он описан как _rxMethod<DessertFilter>_, он принимает _Observable<DessertFilter>_. Значения этого Observable проходят через заданный pipe. А так как _rxMethod_ автоматически подписывается на этот Observable, код приложения должен забрать результат потока данных через _tap_ или _tapResponse_. Последний — это оператор из пакета _ngrx/operators_, объединяющий функциональность _tap_, _catchError_ и _finalize_.

Потребитель реактивного метода может передать соответствующий Observable, а может — сигнал или конкретное значение. Например, показанный хук _onInit_ передаёт сигнал _filter_. Это значит, что все значения, которые этот сигнал постепенно получает, проходят через pipe в _loadDessertsByFilter_, соблюдая упомянутое выше свойство glitch-free.

Что интересно, _rxMethod_ по задумке можно использовать и вне Signal Store. Например, компонент может с его помощью настроить себе реактивный метод.

## Совет 2: как избежать состояний гонки

Перекрывающиеся асинхронные операции обычно приводят к нежелательным состояниям гонки. Скажем, если пользователь быстро ищет два разных десерта подряд, оба результата могут отобразиться один за другим. Один из них лишь мелькнёт на мгновение, прежде чем его сменит другой. Из-за асинхронной природы порядок результатов вовсе не обязан совпадать с порядком запросов.

Чтобы избежать такого запутанного поведения, RxJS предоставляет различные «уплощающие» операторы (flattening operators):

* _switchMap_
* _mergeMap_
* _concatMap_
* _exhaustMap_

Эти операторы отличаются тем, как они обходятся с перекрывающимися запросами. Уже упомянутый выше _switchMap_ при нескольких перекрывающихся запросах всегда возвращает результат только последнего поискового запроса. Любые запросы, которые могли уже выполняться, отменяются при поступлении нового. Именно такое поведение пользователи интуитивно ожидают от поисковых фильтров.

Операторы _mergeMap_ и _concatMap_ выполняют все запросы: первый — параллельно, второй — последовательно. А оператор _exhaustMap_ игнорирует новые запросы, пока выполняется текущий. Эти возможности — ещё один повод использовать RxJS, как и обсуждавшиеся ранее RxJS interop и _rxMethod_.

Ещё одна стратегия, которую часто применяют дополнительно или вместо этого, — это флаг, сообщающий, общается ли приложение с бэкендом прямо сейчас:

```typescript
loadRatings(): void {
  patchState(store, { loading: true });

  ratingService.loadExpertRatings().subscribe({
    next: (ratings) => {
      patchState(store, { ratings, loading: false });
    },
    error: (error) => {
      patchState(store, { loading: false });
      toastService.show('Error loading ratings!');
      console.error(error);
    },
  });
},
```

В зависимости от значения этого флага приложение может показывать индикатор загрузки или блокировать соответствующую кнопку. Последнее, конечно, контрпродуктивно при сильно реактивном UI или вовсе невозможно, если у приложения нет явной кнопки.

## Итог

RxJS и сигналы прекрасно работают вместе, а RxJS interop, который предлагает Angular, даёт нам лучшее из обоих миров. Я бы рекомендовал использовать RxJS для представления событий. Для обработки асинхронных задач хорошо подходят RxJS или сторы, которые тоже могут быть построены на RxJS. А вот синхронную доставку полученных данных во вью лучше доверить сигналам. Вместе RxJS, сторы и сигналы образуют строительные блоки для построения однонаправленных потоков данных.

Кроме того, «уплощающие» операторы RxJS — это элегантное решение для предотвращения состояний гонки. В качестве альтернативы или в дополнение можно также использовать флаги, чтобы обозначить, выполняется ли сейчас запрос к бэкенду.
