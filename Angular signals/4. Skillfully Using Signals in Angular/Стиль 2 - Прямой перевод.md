Новые сигналы (Signals) в Angular представляют собой простой реактивный строительный блок. Однако, как это часто бывает, дьявол кроется в деталях. В этой статье я приведу два совета, которые помогут вам использовать сигналы более прямолинейно.

## Направляющая теория: однонаправленный поток данных с сигналами

Подход к установлению однонаправленного потока данных, представленный в прошлой статье, служит направляющей теорией для двух представленных здесь советов:

![Однонаправленный поток данных со стором](https://www.angulararchitects.io/wp-content/uploads/2024/05/store.png)

Обработчики UI-событий делегируют стору (store). Я использую для этого абстрактный термин _намерение_ (intention), так как разные сторы имеют совершенно разные реализации этого: в основанном на Redux NGRX Store отправляются (dispatch) экшены (Actions); в легковесном же NGRX Signal Store компонент вызывает метод, предлагаемый стором.

Затем стор выполняет синхронные или асинхронные задачи. Обычно они приводят к изменению состояния (state), которое приложение транспортирует во вью отдельных компонентов посредством сигналов. В рамках этого потока данных состояние можно проецировать на вью-модели (View Models) с помощью _computed_, то есть на структуры данных, которые представляют взгляд отдельных сценариев использования на состояние.

Этот подход основан на том факте, что сигналы в первую очередь подходят для синхронного информирования вью о данных и изменениях данных. Они менее подходят для асинхронных задач и для представления событий: во-первых, они не предлагают простого способа справиться с перекрывающимися асинхронными запросами и вытекающими из них состояниями гонки (race conditions). Кроме того, они не могут напрямую представлять состояния ошибок. Во-вторых, сигналы игнорируют возникающие промежуточные состояния, когда изменения значений происходят непосредственно друг за другом. Это желательное свойство называется glitch-free (свободный от глитчей).

Например, если сигнал меняется с 1 на 2 и сразу после этого с 2 на 3, потребитель получает уведомление только о 3. Это также выгодно для производительности привязки данных (data binding), тем более что обновление промежуточными результатами привело бы к ненужным задачам.

## Совет 1: сигналы гармонируют с RxJS

Сигналы намеренно сохранены простыми. Поэтому они предлагают меньше возможностей, чем RxJS, который закрепился в мире Angular уже много лет. Благодаря интеропу с RxJS (RxJS interop), который привносит Angular, вы можете объединить лучшее из обоих миров. Следующий листинг это демонстрирует:

```typescript
@Component({
  selector: 'app-desserts',
  standalone: true,
  imports: [DessertCardComponent, FormsModule, JsonPipe],
  templateUrl: './desserts.component.html',
  styleUrl: './desserts.component.css',
  changeDetection: ChangeDetectionStrategy.OnPush,
})
export class DessertsComponent {
  #dessertService = inject(DessertService);
  #ratingService = inject(RatingService);
  #toastService = inject(ToastService);

  originalName = signal('');
  englishName = signal('Cake');
  loading = signal(false);

  ratings = signal<DessertIdToRatingMap>({});
  ratedDesserts = computed(() => this.toRated(this.desserts(), this.ratings()));

  originalName$ = toObservable(this.originalName);
  englishName$ = toObservable(this.englishName);

  desserts$ = combineLatest({
    originalName: this.originalName$,
    englishName: this.englishName$,
  }).pipe(
    filter((c) => c.originalName.length >= 3 || c.englishName.length >= 3),
    debounceTime(300),
    tap(() => this.loading.set(true)),
    switchMap((c) =>
      this.#dessertService.find(c).pipe(
        catchError((error) => {
          this.#toastService.show('Error loading desserts!');
          console.error(error);
          return of([]);
        }),
      ),
    ),
    tap(() => this.loading.set(false)),
  );

  desserts = toSignal(this.desserts$, {
    initialValue: [],
  });

  […]
}
```

Этот пример конвертирует сигналы _from_ и _to_ в Observable-ы и на этой основе реализует typeahead. Для этого он использует операторы _filter_, _debounceTime_ и _switchMap_, предлагаемые RxJS. Последний также предотвращает состояния гонки, вытекающие из перекрывающихся запросов, учитывая только самый недавний запрос. Все остальные ещё выполняющиеся запросы отменяются.

В конце получившийся Observable конвертируется в сигнал, чтобы приложение могло продолжить работу с новым Signals API. Разумеется, по соображениям производительности приложению не следует слишком часто переключаться между двумя мирами.

В отличие от рисунка выше, здесь стор не используется. И _намерение_, и асинхронное действие происходят в реактивном потоке данных. Если вы переместите поток данных в сервис, который раздаёт загруженные данные, например, с помощью _shareReplay_, вы уже могли бы рассматривать этот сервис как простой стор. Однако, как и на обсуждаемом рисунке, компонент делегирует выполнение асинхронной задачи куда-то ещё и получает результирующее состояние посредством сигналов.

### RxJS в сторах

RxJS также часто используется в сторах. Например, традиционный NGRX store использует RxJS для экшенов (Actions) и эффектов (Effects). В качестве альтернативы эффектам NGRX Signal Store предлагает реактивные методы, которые можно определить с помощью _rxMethod_:

```typescript
export const DessertStore = signalStore(
  { providedIn: 'root' },
  withState({
    filter: {
      originalName: '',
      englishName: 'Cake',
    },
    loading: false,
    ratings: {} as DessertIdToRatingMap,
    desserts: [] as Dessert[],
  }),
  […]
  withMethods(
    (
      store,
      dessertService = inject(DessertService),
      toastService = inject(ToastService),
    ) => ({

      […]
      loadDessertsByFilter: rxMethod<DessertFilter>(
        pipe(
          filter(
            (f) => f.originalName.length >= 3 || f.englishName.length >= 3,
          ),
          debounceTime(300),
          tap(() => patchState(store, { loading: true })),
          switchMap((f) =>
            dessertService.find(f).pipe(
              tapResponse({
                next: (desserts) => {
                  patchState(store, { desserts, loading: false });
                },
                error: (error) => {
                  toastService.show('Error loading desserts!');
                  console.error(error);
                  patchState(store, { loading: false });
                },
              }),
            ),
          ),
        ),
      ),
    }),
  ),
  withHooks({
    onInit(store) {
      const filter = store.filter;
      store.loadDessertsByFilter(filter);
    },
  }),
);
```

Этот пример настраивает в сторе реактивный метод _loadDessertsByFilter_. Поскольку он определён с помощью _rxMethod<DessertFilter>_, он принимает _Observable<DessertFilter>_. Значения этого Observable проходят через указанный pipe. Поскольку _rxMethod_ автоматически подписывается на этот Observable, код приложения должен получить результат потока данных с помощью _tap_ или _tapResponse_. Последний — это оператор из пакета _ngrx/operators_, объединяющий функциональность _tap_, _catchError_ и _finalize_.

Потребитель реактивного метода может передать соответствующий Observable, но также сигнал или конкретное значение. Например, показанный хук _onInit_ передаёт сигнал _filter_. Это означает, что все значения, которые сигнал постепенно получает, проходят через pipe в _loadDessertsByFilter_, соблюдая упомянутое выше свойство glitch-free.

Что интересно, _rxMethod_ по задумке можно использовать и вне Signal Store. Например, компонент мог бы использовать его для настройки реактивного метода.

## Совет 2: избегание состояний гонки

Перекрывающиеся асинхронные операции обычно приводят к нежелательным состояниям гонки. Например, если пользователь ищет два разных десерта в быстрой последовательности, оба результата могут быть отображены один за другим. Один из двух лишь кратко мелькает, прежде чем другой его заменяет. Из-за асинхронной природы порядок результатов не обязан совпадать с порядком запросов.

Чтобы предотвратить это запутанное поведение, RxJS предоставляет различные уплощающие операторы (flattening operators):

* _switchMap_
* _mergeMap_
* _concatMap_
* _exhaustMap_

Эти операторы различаются тем, как они обрабатывают перекрывающиеся запросы. Уже обсуждавшийся выше _switchMap_ при наличии нескольких перекрывающихся запросов всегда возвращает только результат последнего поискового запроса. Любые запросы, которые могут уже выполняться, отменяются при поступлении нового запроса. Именно такое поведение пользователи интуитивно ожидают при работе с поисковыми фильтрами.

Операторы _mergeMap_ и _concatMap_ выполняют все запросы: первый — параллельно, а второй — последовательно. Оператор _exhaustMap_ игнорирует дальнейшие запросы, пока один выполняется. Эти возможности — ещё одна причина использовать RxJS, как и обсуждавшиеся ранее RxJS interop и _rxMethod_.

Ещё одна стратегия, которая часто используется дополнительно или в качестве альтернативы, — это флаг, который информирует, общается ли приложение в данный момент с бэкендом:

```typescript
loadRatings(): void {
  patchState(store, { loading: true });

  ratingService.loadExpertRatings().subscribe({
    next: (ratings) => {
      patchState(store, { ratings, loading: false });
    },
    error: (error) => {
      patchState(store, { loading: false });
      toastService.show('Error loading ratings!');
      console.error(error);
    },
  });
},
```

В зависимости от значения этого флага приложение могло бы отображать индикатор загрузки или блокировать соответствующую кнопку. Последнее, конечно, контрпродуктивно при сильно реактивном UI или вовсе невозможно, если у приложения нет явной кнопки.

## Резюме

RxJS и сигналы прекрасно работают вместе, и RxJS interop, предлагаемый Angular, даёт нам лучшее из обоих миров. Я бы рекомендовал использовать RxJS для представления событий. Для обработки асинхронных задач хорошо подходят RxJS или сторы, которые могут быть основаны на RxJS. Однако сигналы должны взять на себя синхронную транспортировку полученных данных во вью. Вместе RxJS, сторы и сигналы являются строительными блоками для установления однонаправленных потоков данных.

Кроме того, уплощающие операторы в RxJS — это элегантные решения для предотвращения состояний гонки. В качестве альтернативы или в дополнение можно также использовать флаги, чтобы указать, выполняется ли в данный момент запрос на бэкенде.
