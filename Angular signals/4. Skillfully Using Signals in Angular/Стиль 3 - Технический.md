Новые сигналы (Signals) в Angular — это простой реактивный примитив (building block). Однако, как часто бывает, дьявол кроется в деталях. В этой статье я приведу два совета, которые помогут использовать сигналы более прямолинейно.

## Направляющая теория: однонаправленный поток данных на сигналах

Подход к организации однонаправленного потока данных (unidirectional data flow), представленный в предыдущей статье, служит направляющей теорией для двух приведённых здесь рекомендаций:

![Однонаправленный поток данных со стором](https://www.angulararchitects.io/wp-content/uploads/2024/05/store.png)

Обработчики UI-событий делегируют работу стору (store). Для этого я использую абстрактный термин _намерение_ (intention), поскольку в разных сторах он реализуется совершенно по-разному: в основанном на Redux NGRX Store диспетчеризуются (dispatch) экшены (Actions); в легковесном же NGRX Signal Store компонент вызывает метод, предоставляемый стором.

Затем стор выполняет синхронные или асинхронные задачи. Обычно они приводят к изменению состояния (state), которое приложение доставляет (transport) во вью отдельных компонентов через сигналы. В рамках этого потока данных состояние можно проецировать на вью-модели (View Models) с помощью _computed_, то есть на структуры данных, представляющие проекцию состояния под конкретные сценарии использования.

Этот подход опирается на тот факт, что сигналы в первую очередь пригодны для синхронного информирования вью о данных и их изменениях. Они менее пригодны для асинхронных задач и для представления событий. Во-первых, они не предоставляют простого способа обработать перекрывающиеся асинхронные запросы и вытекающие из них состояния гонки (race conditions). Кроме того, они не способны напрямую представлять состояния ошибок. Во-вторых, сигналы игнорируют возникающие промежуточные состояния при изменениях значений, следующих непосредственно одно за другим. Это желательное свойство называется glitch-free (отсутствие промежуточных «глитч»-уведомлений).

Например, если сигнал меняется с 1 на 2 и сразу после этого с 2 на 3, потребитель получает уведомление только о значении 3. Это также выгодно для производительности привязки данных (data binding), тем более что обновление промежуточными результатами породило бы лишние задачи (повторные вычисления и перерисовки).

## Совет 1: сигналы гармонируют с RxJS

Сигналы намеренно оставлены простыми. Поэтому они предоставляют меньше возможностей, чем RxJS, давно закрепившийся в экосистеме Angular. Благодаря интеропу с RxJS (RxJS interop), который предоставляет Angular, можно объединить лучшее из обоих миров. Это демонстрирует следующий листинг:

```typescript
@Component({
  selector: 'app-desserts',
  standalone: true,
  imports: [DessertCardComponent, FormsModule, JsonPipe],
  templateUrl: './desserts.component.html',
  styleUrl: './desserts.component.css',
  changeDetection: ChangeDetectionStrategy.OnPush,
})
export class DessertsComponent {
  #dessertService = inject(DessertService);
  #ratingService = inject(RatingService);
  #toastService = inject(ToastService);

  originalName = signal('');
  englishName = signal('Cake');
  loading = signal(false);

  ratings = signal<DessertIdToRatingMap>({});
  ratedDesserts = computed(() => this.toRated(this.desserts(), this.ratings()));

  originalName$ = toObservable(this.originalName);
  englishName$ = toObservable(this.englishName);

  desserts$ = combineLatest({
    originalName: this.originalName$,
    englishName: this.englishName$,
  }).pipe(
    filter((c) => c.originalName.length >= 3 || c.englishName.length >= 3),
    debounceTime(300),
    tap(() => this.loading.set(true)),
    switchMap((c) =>
      this.#dessertService.find(c).pipe(
        catchError((error) => {
          this.#toastService.show('Error loading desserts!');
          console.error(error);
          return of([]);
        }),
      ),
    ),
    tap(() => this.loading.set(false)),
  );

  desserts = toSignal(this.desserts$, {
    initialValue: [],
  });

  […]
}
```

Этот пример конвертирует сигналы _from_ и _to_ в Observable-ы (через `toObservable`/`toSignal`) и на этой основе реализует typeahead (автодополнение по мере ввода). Для этого применяются операторы _filter_, _debounceTime_ и _switchMap_ из RxJS. Последний к тому же предотвращает состояния гонки от перекрывающихся запросов, учитывая только самый последний запрос: все остальные ещё выполняющиеся запросы отменяются (отписка от внутреннего Observable).

В завершение получившийся Observable конвертируется обратно в сигнал, чтобы приложение могло продолжить работу с новым Signals API. Разумеется, из соображений производительности не следует слишком часто переключаться между двумя мирами.

В отличие от схемы выше, здесь стор не используется. И _намерение_, и асинхронное действие выполняются внутри реактивного потока данных. Если вынести поток данных в сервис, раздающий загруженные данные (например, через _shareReplay_ — мультикаст с буферизацией последнего значения), то такой сервис уже можно рассматривать как простой стор. Тем не менее, как и на обсуждаемой схеме, компонент делегирует выполнение асинхронной задачи вовне и получает результирующее состояние через сигналы.

### RxJS в сторах

RxJS также часто используется внутри сторов. Например, традиционный NGRX store применяет RxJS для экшенов (Actions) и эффектов (Effects). В качестве альтернативы эффектам NGRX Signal Store предлагает реактивные методы, определяемые с помощью _rxMethod_:

```typescript
export const DessertStore = signalStore(
  { providedIn: 'root' },
  withState({
    filter: {
      originalName: '',
      englishName: 'Cake',
    },
    loading: false,
    ratings: {} as DessertIdToRatingMap,
    desserts: [] as Dessert[],
  }),
  […]
  withMethods(
    (
      store,
      dessertService = inject(DessertService),
      toastService = inject(ToastService),
    ) => ({

      […]
      loadDessertsByFilter: rxMethod<DessertFilter>(
        pipe(
          filter(
            (f) => f.originalName.length >= 3 || f.englishName.length >= 3,
          ),
          debounceTime(300),
          tap(() => patchState(store, { loading: true })),
          switchMap((f) =>
            dessertService.find(f).pipe(
              tapResponse({
                next: (desserts) => {
                  patchState(store, { desserts, loading: false });
                },
                error: (error) => {
                  toastService.show('Error loading desserts!');
                  console.error(error);
                  patchState(store, { loading: false });
                },
              }),
            ),
          ),
        ),
      ),
    }),
  ),
  withHooks({
    onInit(store) {
      const filter = store.filter;
      store.loadDessertsByFilter(filter);
    },
  }),
);
```

Этот пример настраивает в сторе реактивный метод _loadDessertsByFilter_. Поскольку он определён как _rxMethod<DessertFilter>_, он принимает _Observable<DessertFilter>_. Значения этого Observable проходят через заданный pipe. Так как _rxMethod_ автоматически подписывается на этот Observable (и управляет жизненным циклом подписки), код приложения должен забрать результат потока данных через _tap_ или _tapResponse_. Последний — это оператор из пакета _ngrx/operators_, объединяющий функциональность _tap_, _catchError_ и _finalize_.

Потребитель реактивного метода может передать соответствующий Observable, а также сигнал или конкретное значение. Например, показанный хук _onInit_ передаёт сигнал _filter_. Это означает, что все значения, которые сигнал последовательно получает, проходят через pipe в _loadDessertsByFilter_, при этом соблюдается упомянутое выше свойство glitch-free.

Что интересно, _rxMethod_ по задумке можно использовать и вне Signal Store. Например, компонент может с его помощью настроить реактивный метод.

## Совет 2: предотвращение состояний гонки

Перекрывающиеся асинхронные операции обычно приводят к нежелательным состояниям гонки. Например, если пользователь ищет два разных десерта в быстрой последовательности, оба результата могут отобразиться один за другим. Один из двух лишь кратко мелькает, прежде чем его заменяет другой. В силу асинхронной природы порядок результатов не обязан совпадать с порядком запросов.

Чтобы предотвратить такое сбивающее с толку поведение, RxJS предоставляет различные уплощающие операторы (flattening operators):

* _switchMap_
* _mergeMap_
* _concatMap_
* _exhaustMap_

Эти операторы различаются тем, как они обрабатывают перекрывающиеся запросы. Уже обсуждавшийся выше _switchMap_ при нескольких перекрывающихся запросах всегда возвращает результат только последнего поискового запроса. Любые уже выполняющиеся запросы отменяются при поступлении нового. Именно такого поведения пользователи интуитивно ожидают при работе с поисковыми фильтрами.

Операторы _mergeMap_ и _concatMap_ выполняют все запросы: первый — параллельно, второй — последовательно. Оператор _exhaustMap_ игнорирует новые запросы, пока выполняется текущий. Эти возможности — ещё одна причина использовать RxJS, наряду с рассмотренными ранее RxJS interop и _rxMethod_.

Ещё одна стратегия, часто применяемая дополнительно или как альтернатива, — флаг, сигнализирующий о том, взаимодействует ли приложение с бэкендом в текущий момент:

```typescript
loadRatings(): void {
  patchState(store, { loading: true });

  ratingService.loadExpertRatings().subscribe({
    next: (ratings) => {
      patchState(store, { ratings, loading: false });
    },
    error: (error) => {
      patchState(store, { loading: false });
      toastService.show('Error loading ratings!');
      console.error(error);
    },
  });
},
```

В зависимости от значения этого флага приложение может отображать индикатор загрузки или блокировать соответствующую кнопку. Последнее, разумеется, контрпродуктивно при сильно реактивном UI или вовсе невозможно, если у приложения нет явной кнопки.

## Резюме

RxJS и сигналы прекрасно работают вместе, а RxJS interop, предоставляемый Angular, даёт нам лучшее из обоих миров. Я бы рекомендовал использовать RxJS для представления событий. Для обработки асинхронных задач хорошо подходят RxJS или сторы, которые также могут быть построены на RxJS. Однако синхронную доставку полученных данных во вью следует возложить на сигналы. Вместе RxJS, сторы и сигналы образуют строительные блоки для организации однонаправленных потоков данных.

Кроме того, уплощающие операторы RxJS — это элегантное решение для предотвращения состояний гонки. В качестве альтернативы или в дополнение можно также использовать флаги, чтобы обозначить, выполняется ли в текущий момент запрос к бэкенду.
