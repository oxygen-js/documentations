С появлением сигналов (Signals) у Angular появился новый реактивный строительный блок, который заметно упрощает управление состоянием. Его лёгкий и понятный API отлично подойдёт разработчикам, которым нужен простой путь в реактивное программирование. Многие типовые задачи теперь решаются буквально парой строк кода.

При этом RxJS остаётся мощным инструментом для более сложных сценариев — потоков, отмены операций и запутанных схем движения данных. И хорошая новость в том, что сигналы и RxJS прекрасно дружат друг с другом. Благодаря этой совместимости вы можете по необходимости сочетать простоту одного и гибкость другого.

Сигналы также отлично работают в связке со стратегией обнаружения изменений `OnPush` и изначально проектировались с прицелом на будущий zone-less-режим (обнаружение изменений без Zone.js). Более того, в перспективе они вполне могут стать фундаментом для ещё более точечных механизмов обнаружения изменений.

Эта серия статей рассказывает, как эффективно работать с сигналами Angular — от азов до продвинутых паттернов. В первой части мы разберём базовые строительные блоки: `signal`, `computed`, `effect` и `untracked`.

## Как пользоваться сигналами

Чтобы использовать сигналы в привязке данных, свойства, которые мы хотим связать, объявляются как сигналы:

```typescript
@Component([…])
export class FlightSearchComponent {

  private flightService = inject(FlightService);

  from = signal('Hamburg');
  to = signal('Graz');
  flights = signal<Flight[]>([]);

  […]

}
```

Здесь стоит обратить внимание: у сигнала всегда есть значение. Поэтому в функцию `signal` обязательно передаётся значение по умолчанию. А если тип данных по нему не выводится автоматически, то в примере он задаётся явно через параметр типа.

Чтобы прочитать значение сигнала, используется его геттер. Технически это значит, что сигнал вызывается как обычная функция:

```typescript
async search(): Promise<void> {
  if (!this.from() || !this.to()) {
    return;
  }
  const flights = await this.flightService.findAsPromise(this.from(), this.to());
  this.flights.set(flights);
}
```

А чтобы записать значение, у сигнала есть явный сеттер — метод `set`. В нашем примере сеттер используется для того, чтобы сохранить загруженные рейсы. Тот же геттер применяется и для привязки данных в шаблоне:

```html
<div *ngIf="flights().length > 0">
  {{flights().length}} flights found!
</div>

<div class="row">
  <div *ngFor="let f of flights()">
    <flight-card [item]="f" />
  </div>
</div>
```

Раньше вызовы методов в шаблонах считались дурным тоном — в первую очередь из-за того, что они могли проседать по производительности. Но к простым операциям вроде геттеров это, как правило, не относится. К тому же шаблон выступает здесь в роли потребителя (consumer), а значит, его можно уведомлять об изменениях.

Кстати, двусторонние привязки (Two-Way-Bindings) в Angular напрямую поддерживают сигналы:

```html
<form #form="ngForm">
  <div class="form-group">
    <label>From:</label>
    <input [(ngModel)]="from" name="from" class="form-control">
  </div>

  <div class="form-group">
    <label>To:</label>
    <input [(ngModel)]="to" name="to" class="form-control">
  </div>

  <div class="form-group">
    <button class="btn btn-default" (click)="search()">Search</button>
    <button class="btn btn-default" (click)="delay()">Delay</button>
  </div>
</form>
```

В одной из будущих версий команда Angular адаптирует под сигналы и работу с формами.

## Как обновлять сигналы

Помимо уже показанного сеттера, у сигналов есть и метод `update` — он позволяет получить новое значение на основе текущего:

```typescript
this.flights.update(f => {
  const flight = f[0];
  const date = addMinutes(flight.date, 15);
  const updated = {...flight, date};

  return [
    updated,
    ...f.slice(1)
  ];
});
```

## Значения сигналов должны быть неизменяемыми

По умолчанию предполагается, что значение сигнала неизменяемо (immutable). Именно поэтому в предыдущем разделе просто поменять дату рейса было бы недостаточно. Вместо этого нам приходится клонировать изменяемые части, чтобы они получили новую ссылку на объект.

Сравнивая эти ссылки, стратегия обнаружения изменений `OnPush` в Angular может эффективно определить, какие части объекта, хранящегося в сигнале, изменились. В предыдущем разделе новую ссылку на объект получают массив и первый рейс. Остальные рейсы не менялись, поэтому они просто копируются через `slice` — и сохраняют свою прежнюю ссылку на объект.

## Вычисляемые значения, побочные эффекты и гарантии

Некоторые значения выводятся из уже существующих. Для этого в Angular есть вычисляемые сигналы (computed signals):

```typescript
flightRoute = computed(() => this.from() + ' to ' + this.to());
```

Такой сигнал доступен только для чтения и одновременно выступает и потребителем (consumer), и поставщиком (producer). Как потребитель, он считывает значения используемых сигналов — здесь это `from` и `to` — и узнаёт об их изменениях. Как поставщик, он возвращает вычисленное значение.

Если же сигналы нужно потреблять программно, на помощь приходит функция `effect`:

```typescript
constructor() {
    effect(() => {
        console.log('from:', this.from());
        console.log('route:', this.flightRoute());
    });
}
```

Функция `effect` выполняет переданную ей лямбда-выражение и регистрирует себя как потребителя для всех использованных в ней сигналов. Как только один из этих сигналов меняется, эффект запускается снова.

Учтите, что обычно сигналы потребляются через привязку данных. Однако иногда нам нужен именно вызов функции или метода, чтобы показать значение сигнала пользователю. Примеры — логирование или показ всплывающего уведомления (toast). Подробнее об идеях, стоящих за эффектами, можно почитать [здесь](https://www.angulararchitects.io/blog/when-not-to-use-effects-in-angular-and-what-to-do-instead/).

## Нужен контекст внедрения зависимостей

Некоторые строительные блоки для сигналов — например, `effect` — можно использовать только в контексте внедрения зависимостей (injection context). А это везде, где разрешён `inject`: в конструкторе, в значениях полей класса по умолчанию и в фабриках провайдеров. Кроме того, можно воспользоваться функцией `runInInjectionContext`, чтобы выполнить код в таком контексте.

Поэтому эффект, который мы выше настроили в конструкторе, выдал бы ошибку, если бы оказался в `ngOnInit` или в любом другом методе:

```typescript
ngOnInit(): void {
    // Эффекты здесь использовать нельзя.
    // Поэтому этот код упадёт с ошибкой:
    effect(() => {
        console.log('route:', this.flightRoute());
    });
}
```

В этом случае вы получите вот такую ошибку:

> ERROR Error: NG0203: effect() can only be used within an injection context such as a constructor, a factory function,

Техническая причина в том, что эффекты используют `inject`, чтобы получить текущий `DestroyRef`. Этот сервис, доступный начиная с Angular 16, помогает узнать о времени жизни текущего строительного блока — например, текущего компонента или сервиса. С помощью `DestroyRef` эффект «отписывается», когда этот блок собирается уничтожаться.

Поэтому эффекты обычно настраивают в конструкторе, как мы показали в прошлом разделе. Но если вам всё-таки нужно настроить эффект где-то ещё, на помощь придёт функция `runInInjectionContext`. Правда, ей нужна ссылка на `Injector`:

```typescript
injector = inject(Injector);

ngOnInit(): void {
  runInInjectionContext(this.injector, () => {
    effect(() => {
      console.log('route:', this.flightRoute());
    });
  });
}
```

## Свойство «отсутствия сбоев» (glitch-free)

Если сигнал меняется несколько раз подряд или если один за другим меняются сразу несколько сигналов, могут возникнуть нежелательные промежуточные результаты. Представим, что мы меняем поисковый фильтр `Hamburg - Graz` на `London - Paris`:

```typescript
setTimeout(() => {
  this.from.set('London');
  this.to.set('Paris');
}, 2000);
```

Здесь сразу после установки `from` в `London` могло бы на мгновение «промелькнуть» состояние `London - Graz`. Как и многие другие реализации сигналов, реализация в Angular предотвращает подобные ситуации. В [README команды Angular](https://github.com/angular/angular/blob/71d5cdae195f916e345d977f1f23f9490e09482e/packages/core/src/signals/README.md), где заодно объясняется используемый push/pull-алгоритм, эту полезную гарантию называют «glitch-free» (без сбоев).

## Сигналы и обнаружение изменений

Подобно тому как Observable, привязанный к шаблону через пайп `async`, запускает обнаружение изменений, привязанный сигнал делает то же самое. Причём это работает и в более эффективном режиме `OnPush`:

```typescript
@Component({
  [...]
  changeDetection: ChangeDetectionStrategy.OnPush
})
export class FlightSearchComponent { [...] } 

[...]

@Component({
  [...]
  changeDetection: ChangeDetectionStrategy.OnPush
})
export class FlightCardComponent { [...] } 
```

Однако, чтобы Angular в режиме `OnPush` смог разобраться, на какие дочерние компоненты ему нужно обратить внимание, придётся использовать неизменяемые значения (Immutables), о которых мы говорили выше.

## Интеграция с RxJS (RxJS Interop)

Надо признать, что на первый взгляд сигналы очень похожи на механизм, который Angular использует уже давно, — на Observable из RxJS. Однако сигналы намеренно сделаны проще.

Если же вам нужна вся мощь RxJS и его операторов, сигналы можно преобразовать в Observable. В пространстве имён `@angular/core/rxjs-interop` есть функция `toObservable`, которая превращает сигнал в Observable, и функция `toSignal` — для обратного преобразования. Они позволяют сочетать простоту сигналов с мощью RxJS.

В следующем листинге показано использование этих двух методов: мы расширяем наш пример до type-ahead-поиска с дебаунсингом (debounce):

```typescript
@Component([...])
export class FlightSearchComponent {
  private flightService = inject(FlightService);

  from = signal('Hamburg');
  to = signal('Graz');
  basket = signal<Record<number, boolean>>({ 1: true });
  flightRoute = computed(() => this.from() + ' to ' + this.to());

  from$ = toObservable(this.from);
  to$ = toObservable(this.to);

  flights$ = combineLatest({ from: this.from$, to: this.to$ }).pipe(
    filter(c => c.from.length >= 3 && c.to.length >= 3),
    debounceTime(300),
    switchMap(c => this.flightService.find(c.from, c.to))
  );

  flights = toSignal(this.flights$, {
    initialValue: []
  });
}
```

В этом примере сигналы `from` и `to` преобразуются в Observable `from$` и `to$`, а затем объединяются с помощью `combineLatest`. Как только одно из значений меняется, сначала выполняются фильтрация и дебаунсинг, и только потом `switchMap` запускает запрос к бэкенду. Один из плюсов «уплощающих» (flattening) операторов вроде `switchMap` — гарантии в отношении асинхронности. Именно они помогают избегать состояний гонки (race conditions).

Параметр `initialValue`, передаваемый в `toSignal`, нужен потому, что у сигналов всегда есть значение. А вот Observable может вообще не испустить ни одного значения. Если вы уверены, что у вашего Observable есть начальное значение — например, потому что это `BehaviorSubject` или потому что вы используете оператор `startsWith`, — можно вместо этого выставить `requireSync` в `true`:

```typescript
flights = toSignal(this.flights, { 
    requireSync: true
});
```

Если же вы не задали ни `initialValue`, ни `requireSync`, то тип возвращаемого сигнала будет дополнительно включать тип `undefined`, допуская начальное значение `undefined`. В нашем примере это дало бы `Signal<Flight[] | undefinied>` вместо `Signal<Flight[]>`. А значит, в вашем коде придётся ещё и проверять на `undefined`.

## Заключение

Сигналы делают реактивность в Angular легковеснее. Типовые сценарии становятся простыми, а для более сложных мы всегда можем «перекинуть мостик» к RxJS благодаря first-class-интеграции между ними.

Команда Angular остаётся верна себе: сигналы не спрятаны где-то в недрах фреймворка или за прокси, а сделаны явными. Поэтому разработчик всегда знает, с какой именно структурой данных он на самом деле имеет дело. К тому же сигналы — это всего лишь опция. Никому не нужно переписывать старый код, и сочетание традиционного обнаружения изменений с обнаружением на основе сигналов вполне возможно.
