# Сигналы в Angular: строительные блоки

С введением сигналов (Signals) Angular предоставляет новый реактивный строительный блок, который существенно упрощает управление состоянием. Его легковесный и доступный API идеален для разработчиков, которым нужен простой путь в реактивное программирование. Многие распространённые сценарии использования теперь можно реализовать с минимальным количеством кода.

RxJS остаётся мощным решением для продвинутых сценариев, таких как потоки, отмена и сложный поток данных. Хорошая новость: сигналы и RxJS интегрируются бесшовно. Эта совместимость позволяет разработчикам сочетать простоту и гибкость по мере необходимости.

Сигналы также работают рука об руку со стратегией обнаружения изменений `OnPush` в Angular и спроектированы для поддержки грядущего обнаружения изменений без зон (zone-less). Более того, в будущем они могут послужить основой для ещё более точечных механизмов обнаружения изменений.

Эта серия статей исследует, как эффективно использовать сигналы Angular — от основ до продвинутых паттернов. Эта первая часть охватывает основополагающие строительные блоки: `signal`, `computed`, `effect` и `untracked`.

## Использование сигналов

Для использования сигналов в привязке данных свойства, которые нужно привязать, выражаются как сигналы:

```typescript
@Component([…])
export class FlightSearchComponent {

  private flightService = inject(FlightService);

  from = signal('Hamburg');
  to = signal('Graz');
  flights = signal<Flight[]>([]);

  […]

}
```

Здесь следует отметить, что у сигнала всегда есть значение. Поэтому в функцию `signal` необходимо передать значение по умолчанию. Если тип данных невозможно вывести из него, в примере он указывается явно через параметр типа.

Геттер сигнала используется для чтения его значения. Технически это означает, что сигнал вызывается как функция:

```typescript
async search(): Promise<void> {
  if (!this.from() || !this.to()) {
    return;
  }
  const flights = await this.flightService.findAsPromise(this.from(), this.to());
  this.flights.set(flights);
}
```

Чтобы установить значение, сигнал предоставляет явный сеттер в виде метода `set`. В показанном примере сеттер используется для того, чтобы сохранить загруженные рейсы. Геттер также используется для привязки данных в шаблоне:

```html
<div *ngIf="flights().length > 0">
  {{flights().length}} flights found!
</div>

<div class="row">
  <div *ngFor="let f of flights()">
    <flight-card [item]="f" />
  </div>
</div>
```

В прошлом вызовы методов в шаблонах не приветствовались, особенно потому, что они могли приводить к узким местам в производительности. Однако это в целом не относится к несложным процедурам, таким как геттеры. Кроме того, шаблон выступает здесь как потребитель (consumer), и в этом качестве он может быть уведомлён об изменениях.

Между тем двусторонние привязки (Two-Way-Bindings) в Angular напрямую поддерживают сигналы:

```html
<form #form="ngForm">
  <div class="form-group">
    <label>From:</label>
    <input [(ngModel)]="from" name="from" class="form-control">
  </div>

  <div class="form-group">
    <label>To:</label>
    <input [(ngModel)]="to" name="to" class="form-control">
  </div>

  <div class="form-group">
    <button class="btn btn-default" (click)="search()">Search</button>
    <button class="btn btn-default" (click)="delay()">Delay</button>
  </div>
</form>
```

В одной из будущих версий команда Angular адаптирует работу с формами под сигналы.

## Обновление сигналов

В дополнение к показанному ранее сеттеру сигналы также предоставляют метод `update` для проецирования текущего значения в новое:

```typescript
this.flights.update(f => {
  const flight = f[0];
  const date = addMinutes(flight.date, 15);
  const updated = {...flight, date};

  return [
    updated,
    ...f.slice(1)
  ];
});
```

## Значения сигналов должны быть неизменяемыми

По умолчанию предполагается, что значение сигнала неизменяемо (immutable). По этой причине простого обновления даты рейса из предыдущего раздела было бы недостаточно. Вместо этого мы должны клонировать изменённые части, чтобы они получили новую ссылку на объект.

Сравнивая эти ссылки, стратегия обнаружения изменений `OnPush` в Angular может эффективно определить изменённые части объекта, управляемого сигналом. В предыдущем разделе массив и первый рейс получают новую ссылку на объект. Остальные рейсы не изменяются и поэтому просто копируются с помощью `slice`. В результате они сохраняют свою ссылку на объект.

## Вычисляемые значения, побочные эффекты и гарантии (assertions)

Некоторые значения выводятся из существующих значений. Для этого Angular предоставляет вычисляемые сигналы (calculated signals):

```typescript
flightRoute = computed(() => this.from() + ' to ' + this.to());
```

Такой сигнал доступен только для чтения и выступает одновременно как потребитель (consumer) и как поставщик (producer). Как потребитель, он получает значения используемых сигналов — здесь `from` и `to` — и информируется об изменениях. Как поставщик, он возвращает вычисленное значение.

Если вы хотите потреблять сигналы программно, можно использовать функцию `effect`:

```typescript
constructor() {
    effect(() => {
        console.log('from:', this.from());
        console.log('route:', this.flightRoute());
    });
}
```

Функция `effect` выполняет переданное лямбда-выражение и регистрирует себя как потребителя для используемых сигналов. Когда один из этих сигналов изменяется, эффект запускается снова.

Обратите внимание, что обычно сигналы потребляются через привязку данных. Однако в некоторых случаях нам нужен вызов функции или метода, чтобы представить значение сигнала пользователю. Примерами являются логирование или отображение всплывающего сообщения (toast). Более [подробно об идеях, стоящих за эффектами, можно прочитать здесь](https://www.angulararchitects.io/blog/when-not-to-use-effects-in-angular-and-what-to-do-instead/).

## Необходим контекст внедрения зависимостей (Injection Context)

Несколько строительных блоков для сигналов, таких как `effect`, могут использоваться только в контексте внедрения зависимостей (injection context). Это везде, где разрешён `inject`: в конструкторе, в качестве значений полей класса по умолчанию и в фабриках провайдеров. Также можно использовать функцию `runInInjectionContext`, чтобы выполнить код в контексте внедрения зависимостей.

Поэтому эффект, настроенный в конструкторе выше, завершился бы неудачей, если бы был помещён в `ngOnInit` или в другой метод:

```typescript
ngOnInit(): void {
    // Эффекты здесь использовать нельзя.
    // Поэтому это завершится неудачей:
    effect(() => {
        console.log('route:', this.flightRoute());
    });
}
```

В этом случае вы получите следующую ошибку:

> ERROR Error: NG0203: effect() can only be used within an injection context such as a constructor, a factory function,

Техническая причина в том, что эффекты используют `inject`, чтобы получить текущий `DestroyRef`. Этот сервис, доступный начиная с Angular 16, помогает узнать о времени жизни текущего строительного блока, например текущего компонента или сервиса. Эффект использует `DestroyRef`, чтобы «отписаться» от самого себя, когда этот строительный блок собирается быть уничтоженным.

По этой причине вы обычно настраиваете свои эффекты в конструкторе, как показано в прошлом разделе. Если вы действительно хотите настроить эффект где-то ещё, можно воспользоваться функцией `runInInjectionContext`. Однако ей нужна ссылка на `Injector`:

```typescript
injector = inject(Injector);

ngOnInit(): void {
  runInInjectionContext(this.injector, () => {
    effect(() => {
      console.log('route:', this.flightRoute());
    });
  });
}
```

## Свойство отсутствия сбоев (Glitch-Free)

Если сигнал изменяется несколько раз подряд или если несколько сигналов изменяются один за другим, могут возникнуть нежелательные промежуточные результаты. Представим, что мы меняем поисковый фильтр `Hamburg - Graz` на `London - Paris`:

```typescript
setTimeout(() => {
  this.from.set('London');
  this.to.set('Paris');
}, 2000);
```

Здесь `London - Graz` могло бы появиться сразу после установки `from` в `London`. Как и многие другие реализации сигналов, реализация Angular предотвращает такие случаи. В [README](https://github.com/angular/angular/blob/71d5cdae195f916e345d977f1f23f9490e09482e/packages/core/src/signals/README.md) команды Angular, в котором также объясняется используемый push/pull-алгоритм, эту желательную гарантию называют «glitch-free» (без сбоев).

## Сигналы и обнаружение изменений

Подобно Observable, привязанному к шаблону с помощью пайпа `async`, привязанный сигнал запускает обнаружение изменений. Это также работает для более эффективного режима `OnPush`:

```typescript
@Component({
  [...]
  changeDetection: ChangeDetectionStrategy.OnPush
})
export class FlightSearchComponent { [...] } 

[...]

@Component({
  [...]
  changeDetection: ChangeDetectionStrategy.OnPush
})
export class FlightCardComponent { [...] } 
```

Однако, чтобы помочь Angular в режиме `OnPush` также узнать о дочерних компонентах, на которые нужно обратить внимание, вы должны использовать неизменяемые значения (Immutables), как обсуждалось выше.

## Интеграция с RxJS (RxJS Interop)

Надо признать, что на первый взгляд сигналы очень похожи на механизм, который Angular использует уже давно, а именно на Observable из RxJS. Однако сигналы намеренно сделаны проще.

Если же вам нужна мощь RxJS и его операторов, вы можете преобразовать их в Observable. Пространство имён `@angular/core/rxjs-interop` содержит функцию `toObservable`, преобразующую сигнал в Observable, и функцию `toSignal` для обратного преобразования. Они позволяют использовать простоту сигналов с мощью RxJS.

Следующий листинг иллюстрирует использование этих двух методов, расширяя показанный пример до type-ahead-поиска с дебаунсингом (debounce):

```typescript
@Component([...])
export class FlightSearchComponent {
  private flightService = inject(FlightService);

  from = signal('Hamburg');
  to = signal('Graz');
  basket = signal<Record<number, boolean>>({ 1: true });
  flightRoute = computed(() => this.from() + ' to ' + this.to());

  from$ = toObservable(this.from);
  to$ = toObservable(this.to);

  flights$ = combineLatest({ from: this.from$, to: this.to$ }).pipe(
    filter(c => c.from.length >= 3 && c.to.length >= 3),
    debounceTime(300),
    switchMap(c => this.flightService.find(c.from, c.to))
  );

  flights = toSignal(this.flights$, {
    initialValue: []
  });
}
```

В примере сигналы `from` и `to` преобразуются в Observable `from$` и `to$` и объединяются с помощью `combineLatest`. Как только одно из значений изменяется, происходят фильтрация и дебаунсинг, прежде чем `switchMap` запускает запрос к бэкенду. Одно из преимуществ, которое дают «уплощающие» (flattening) операторы вроде `switchMap`, — это гарантии в отношении асинхронности. Эти гарантии помогают избежать состояний гонки (race conditions).

Значение `initialValue`, передаваемое в `toSignal`, необходимо потому, что у сигналов всегда есть значение. Напротив, Observable может вообще не испустить никакого значения. Если вы уверены, что у вашего Observable есть начальное значение, например потому что это `BehaviorSubject` или из-за использования оператора `startsWith`, вы также можете установить `requireSync` в `true`:

```typescript
flights = toSignal(this.flights, { 
    requireSync: true
});
```

Если вы не задаёте ни `initialValue`, ни `requireSync`, тип возвращаемого сигнала также поддерживает тип `undefined`, допуская начальное значение `undefined`. В показанном примере это привело бы к `Signal<Flight[] | undefinied>` вместо `Signal<Flight[]>`. Следовательно, ваш код также должен проверять на `undefined`.

## Заключение

Сигналы делают реактивность в Angular более легковесной. Они делают обычные сценарии использования простыми, а для более сложных сценариев мы можем перейти к RxJS с помощью first-class-реализации интеграции.

Команда Angular остаётся верна себе: сигналы не спрятаны во внутренней структуре или за прокси, а сделаны явными. Поэтому разработчики всегда знают, с какой структурой данных они на самом деле имеют дело. Кроме того, сигналы — это всего лишь опция. Никому не нужно менять унаследованный код, и сочетание традиционного обнаружения изменений и обнаружения изменений на основе сигналов будет возможно.
