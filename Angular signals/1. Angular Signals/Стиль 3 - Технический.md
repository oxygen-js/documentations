# Сигналы (Signals) в Angular: строительные блоки

Angular переходит на реактивный механизм под названием Signals (сигналы), чтобы сделать обнаружение изменений (change detection — процесс синхронизации модели с DOM) более легковесным и при этом более мощным.

С введением сигналов (Signals) Angular предоставляет новый реактивный строительный блок, который существенно упрощает управление состоянием (state management). Его легковесный и доступный API идеально подходит разработчикам, которым нужен простой вход в реактивное программирование. Многие распространённые сценарии теперь реализуются с минимальным объёмом кода.

RxJS остаётся мощным решением для продвинутых сценариев — таких как потоки (streams), отмена операций (cancelation) и сложные потоки данных. Хорошая новость: сигналы и RxJS интегрируются бесшовно. Эта совместимость (interoperability) позволяет разработчикам по мере необходимости сочетать простоту и гибкость.

Сигналы также работают в тесной связке со стратегией обнаружения изменений `OnPush` в Angular (когда проверка компонента запускается только при изменении ссылок на входные данные либо при событиях) и спроектированы с прицелом на грядущий режим обнаружения изменений без зон (zone-less, без Zone.js). Более того, в будущем они могут стать основой для ещё более гранулярных (точечных, на уровне отдельных узлов) механизмов обнаружения изменений.

Эта серия статей рассматривает, как эффективно использовать сигналы Angular — от основ до продвинутых паттернов. Эта первая часть охватывает основополагающие строительные блоки: `signal`, `computed`, `effect` и `untracked`.

## Использование сигналов

Чтобы использовать сигналы в привязке данных (data binding), свойства, подлежащие привязке, объявляются как сигналы:

```typescript
@Component([…])
export class FlightSearchComponent {

  private flightService = inject(FlightService);

  from = signal('Hamburg');
  to = signal('Graz');
  flights = signal<Flight[]>([]);

  […]

}
```

Здесь следует отметить, что сигнал всегда имеет значение. Поэтому в функцию `signal` необходимо передать значение по умолчанию. Если из него невозможно вывести тип данных, в примере он указывается явно через параметр типа (type parameter дженерика).

Для чтения значения сигнала используется его геттер (getter). Технически это означает, что сигнал вызывается как функция:

```typescript
async search(): Promise<void> {
  if (!this.from() || !this.to()) {
    return;
  }
  const flights = await this.flightService.findAsPromise(this.from(), this.to());
  this.flights.set(flights);
}
```

Для установки значения сигнал предоставляет явный сеттер (setter) в виде метода `set`. В показанном примере сеттер используется, чтобы сохранить загруженные рейсы. Геттер также применяется для привязки данных в шаблоне:

```html
<div *ngIf="flights().length > 0">
  {{flights().length}} flights found!
</div>

<div class="row">
  <div *ngFor="let f of flights()">
    <flight-card [item]="f" />
  </div>
</div>
```

В прошлом вызовы методов в шаблонах не приветствовались — прежде всего потому, что они могли приводить к узким местам в производительности (метод вызывается при каждом цикле обнаружения изменений). Однако к несложным процедурам вроде геттеров это, как правило, не относится. Кроме того, шаблон выступает здесь как потребитель (consumer) сигнала, а значит, может быть уведомлён об изменениях (и перерисован адресно).

При этом двусторонние привязки (Two-Way-Bindings) в Angular напрямую поддерживают сигналы:

```html
<form #form="ngForm">
  <div class="form-group">
    <label>From:</label>
    <input [(ngModel)]="from" name="from" class="form-control">
  </div>

  <div class="form-group">
    <label>To:</label>
    <input [(ngModel)]="to" name="to" class="form-control">
  </div>

  <div class="form-group">
    <button class="btn btn-default" (click)="search()">Search</button>
    <button class="btn btn-default" (click)="delay()">Delay</button>
  </div>
</form>
```

В одной из будущих версий команда Angular адаптирует под сигналы и механизм работы с формами.

## Обновление сигналов

Помимо показанного ранее сеттера, сигналы предоставляют также метод `update` для проецирования текущего значения в новое (на вход подаётся функция, преобразующая прежнее значение в новое):

```typescript
this.flights.update(f => {
  const flight = f[0];
  const date = addMinutes(flight.date, 15);
  const updated = {...flight, date};

  return [
    updated,
    ...f.slice(1)
  ];
});
```

## Значения сигналов должны быть неизменяемыми (immutable)

По умолчанию предполагается, что значение сигнала неизменяемо (immutable). По этой причине простого изменения даты рейса из предыдущего раздела было бы недостаточно. Вместо этого мы должны клонировать изменённые части, чтобы они получили новую ссылку на объект (object reference).

Сравнивая эти ссылки (поверхностное сравнение по ссылке, а не по содержимому), стратегия обнаружения изменений `OnPush` в Angular может эффективно определить, какие части объекта, управляемого сигналом, изменились. В предыдущем разделе новую ссылку на объект получают массив и первый рейс. Остальные рейсы не изменяются и поэтому просто копируются через `slice` — в результате они сохраняют свою прежнюю ссылку на объект (что и позволяет `OnPush` пропустить их повторную проверку).

## Вычисляемые значения, побочные эффекты и гарантии

Некоторые значения являются производными от уже существующих. Для этого Angular предоставляет вычисляемые сигналы (computed / calculated signals):

```typescript
flightRoute = computed(() => this.from() + ' to ' + this.to());
```

Такой сигнал доступен только для чтения (read-only) и выступает одновременно как потребитель (consumer) и как поставщик (producer). Как потребитель, он считывает значения используемых сигналов — здесь `from` и `to` — и уведомляется об их изменениях (зависимости отслеживаются автоматически при первом обращении). Как поставщик, он возвращает вычисленное значение (вычисление производится лениво и кешируется до следующего изменения зависимостей).

Если требуется потреблять сигналы программно (а не через привязку в шаблоне), можно использовать функцию `effect`:

```typescript
constructor() {
    effect(() => {
        console.log('from:', this.from());
        console.log('route:', this.flightRoute());
    });
}
```

Функция `effect` выполняет переданное лямбда-выражение и регистрирует себя как потребителя для использованных в нём сигналов. Когда один из этих сигналов изменяется, эффект запускается повторно (планируется к выполнению в рамках цикла обнаружения изменений).

Обратите внимание, что обычно сигналы потребляются через привязку данных. Однако в некоторых случаях нам нужен именно вызов функции или метода, чтобы представить значение сигнала пользователю. Примеры — логирование или показ всплывающего уведомления (toast). Подробнее об идеях, лежащих в основе эффектов, можно [прочитать здесь](https://www.angulararchitects.io/blog/when-not-to-use-effects-in-angular-and-what-to-do-instead/).

## Требуется контекст внедрения зависимостей (Injection Context)

Ряд строительных блоков для сигналов, таких как `effect`, можно использовать только в контексте внедрения зависимостей (injection context). Это везде, где допустим `inject`: в конструкторе, в качестве значений полей класса по умолчанию и в фабриках провайдеров (provider factories). Кроме того, для выполнения кода в контексте внедрения зависимостей можно использовать функцию `runInInjectionContext`.

Следовательно, эффект, настроенный в конструкторе выше, завершился бы ошибкой, будучи помещённым в `ngOnInit` или в другой метод:

```typescript
ngOnInit(): void {
    // Эффекты здесь использовать нельзя.
    // Поэтому это завершится ошибкой:
    effect(() => {
        console.log('route:', this.flightRoute());
    });
}
```

В этом случае вы получите следующую ошибку:

> ERROR Error: NG0203: effect() can only be used within an injection context such as a constructor, a factory function,

Техническая причина в том, что эффекты используют `inject`, чтобы получить текущий `DestroyRef`. Этот сервис, доступный начиная с Angular 16, помогает узнать о времени жизни текущего строительного блока — например, текущего компонента или сервиса. Эффект использует `DestroyRef`, чтобы «отписаться» от самого себя (освободить ресурсы и прекратить реагировать на сигналы), когда этот строительный блок собирается быть уничтоженным.

По этой причине эффекты обычно настраивают в конструкторе, как показано в прошлом разделе. Если же вам действительно нужно настроить эффект в другом месте, можно воспользоваться функцией `runInInjectionContext`. Однако ей требуется ссылка на `Injector`:

```typescript
injector = inject(Injector);

ngOnInit(): void {
  runInInjectionContext(this.injector, () => {
    effect(() => {
      console.log('route:', this.flightRoute());
    });
  });
}
```

## Свойство отсутствия сбоев (Glitch-Free)

Если сигнал изменяется несколько раз подряд или если несколько сигналов изменяются один за другим, могут возникнуть нежелательные промежуточные результаты (несогласованные «глитчи»). Представим, что мы меняем поисковый фильтр `Hamburg - Graz` на `London - Paris`:

```typescript
setTimeout(() => {
  this.from.set('London');
  this.to.set('Paris');
}, 2000);
```

Здесь сразу после установки `from` в `London` могло бы возникнуть промежуточное состояние `London - Graz`. Как и многие другие реализации сигналов, реализация Angular предотвращает подобные случаи. В [README](https://github.com/angular/angular/blob/71d5cdae195f916e345d977f1f23f9490e09482e/packages/core/src/signals/README.md) команды Angular, где также объясняется применяемый push/pull-алгоритм (изменения «проталкиваются» как уведомление о неактуальности, а фактический пересчёт значений выполняется «по требованию» при чтении), эту желательную гарантию называют «glitch-free» (отсутствие промежуточных несогласованных состояний).

## Сигналы и обнаружение изменений

Подобно Observable, привязанному к шаблону через пайп `async`, привязанный сигнал запускает обнаружение изменений. Это работает и в более эффективном режиме `OnPush`:

```typescript
@Component({
  [...]
  changeDetection: ChangeDetectionStrategy.OnPush
})
export class FlightSearchComponent { [...] } 

[...]

@Component({
  [...]
  changeDetection: ChangeDetectionStrategy.OnPush
})
export class FlightCardComponent { [...] } 
```

Однако, чтобы помочь Angular в режиме `OnPush` также определить, какие дочерние компоненты подлежат проверке, необходимо использовать неизменяемые значения (Immutables), как обсуждалось выше (поскольку `OnPush` ориентируется на изменение ссылок на входные данные).

## Интеграция с RxJS (RxJS Interop)

Надо признать, что на первый взгляд сигналы очень похожи на механизм, который Angular использует уже давно, — на Observable из RxJS. Однако сигналы намеренно сделаны проще.

Если же вам нужна мощь RxJS и его операторов, сигналы можно преобразовать в Observable. Пространство имён `@angular/core/rxjs-interop` содержит функцию `toObservable`, преобразующую сигнал в Observable, и функцию `toSignal` для обратного преобразования. Они позволяют сочетать простоту сигналов с мощью RxJS.

Следующий листинг иллюстрирует использование этих двух методов, расширяя показанный пример до type-ahead-поиска с подавлением частоты событий (debounce):

```typescript
@Component([...])
export class FlightSearchComponent {
  private flightService = inject(FlightService);

  from = signal('Hamburg');
  to = signal('Graz');
  basket = signal<Record<number, boolean>>({ 1: true });
  flightRoute = computed(() => this.from() + ' to ' + this.to());

  from$ = toObservable(this.from);
  to$ = toObservable(this.to);

  flights$ = combineLatest({ from: this.from$, to: this.to$ }).pipe(
    filter(c => c.from.length >= 3 && c.to.length >= 3),
    debounceTime(300),
    switchMap(c => this.flightService.find(c.from, c.to))
  );

  flights = toSignal(this.flights$, {
    initialValue: []
  });
}
```

В примере сигналы `from` и `to` преобразуются в Observable `from$` и `to$` и объединяются с помощью `combineLatest`. Как только одно из значений изменяется, сначала выполняются фильтрация и подавление частоты (debounce), и лишь затем `switchMap` инициирует запрос к бэкенду. Одно из преимуществ «уплощающих» (flattening) операторов вроде `switchMap` — это гарантии в отношении асинхронности (`switchMap` отменяет предыдущий незавершённый внутренний поток при поступлении нового значения). Эти гарантии помогают избежать состояний гонки (race conditions).

Значение `initialValue`, передаваемое в `toSignal`, необходимо потому, что сигналы всегда имеют значение. Observable же, напротив, может вообще не испустить ни одного значения. Если вы уверены, что у вашего Observable есть начальное значение — например, потому что это `BehaviorSubject` или из-за использования оператора `startsWith`, — можно вместо этого установить `requireSync` в `true`:

```typescript
flights = toSignal(this.flights, { 
    requireSync: true
});
```

Если же вы не задаёте ни `initialValue`, ни `requireSync`, тип возвращаемого сигнала будет дополнительно включать тип `undefined`, допуская начальное значение `undefined`. В показанном примере это привело бы к типу `Signal<Flight[] | undefinied>` вместо `Signal<Flight[]>`. Следовательно, ваш код должен дополнительно проверять значение на `undefined`.

## Заключение

Сигналы делают реактивность в Angular более легковесной. Они упрощают типовые сценарии, а для более сложных мы можем перейти к RxJS благодаря first-class-реализации интеграции (interop).

Команда Angular остаётся верна себе: сигналы не спрятаны во внутренней структуре фреймворка или за прокси, а сделаны явными. Поэтому разработчики всегда знают, с какой именно структурой данных они на самом деле имеют дело. Кроме того, сигналы — это всего лишь опция. Никому не нужно переписывать унаследованный (legacy) код, и сочетание традиционного обнаружения изменений с обнаружением на основе сигналов будет возможным.
