## Основной сценарий применения эффектов

Эффекты (effects) предназначены в первую очередь для отрисовки того, что невозможно отрисовать средствами привязки данных (data binding).

Ниже несколько примеров, которые также есть в [документации](https://angular.dev/guide/signals#effects):

1) Логирование  
2) Отрисовка на Canvas  
3) Нестандартное (custom) поведение DOM

Поскольку привязка данных — предпочтительный механизм отображения данных пользователю, понятно, почему документация указывает, что эффекты нужны лишь в отдельных случаях.

Хороший пример к пункту 2 — превосходная библиотека [Angular Three](https://angular-threejs.netlify.app) от [Chau Tran](https://x.com/Nartc1410), служащая мостом между Angular и Three.js.

Пункт 3 касается поведения DOM, которое невозможно выразить синтаксисом шаблона. Пример — отображение `SnackBar` через императивный API, как в Angular Material.

## Помните про авто-отслеживание

Помните, что для `computed` и `effects` Angular применяет неявное отслеживание (авто-отслеживание, auto-tracking) — то есть зависимости определяются по фактическому чтению сигналов в момент выполнения. В примере ниже эффект будет отслеживать сигнал `error`, хотя тот не читается напрямую в функции эффекта, а читается внутри вызываемого метода `logError` (отслеживание распространяется на весь стек вызовов внутри реактивного контекста).

```
effect(() => {
  this.logError();
});
:
logError(): void {
  const error = this.error();
  if (error) {
    console.error(error);
  }
}
```

Это также подчёркивает, что текущая реализация эффектов в Angular рассчитана прежде всего на рендеринг. Сигналы, прочитанные во время рендеринга, отслеживаются; при их изменении рендеринг запускается повторно. Это проектное решение разобрал [Alex Rickabaugh](https://x.com/synalx) из команды Angular в этом [issue на GitHub](https://github.com/angular/angular/issues/56155#issuecomment-2177202036).

## Когда НЕ применять эффекты?

[Документация](https://angular.dev/guide/signals#effects) также явно указывает: НЕ применяйте эффекты для распространения состояния (propagating state). Здесь возможен ряд проблем, в частности циклические обновления (circular updates). Добавлю от себя: поведение авто-отслеживания быстро приводит к коду, который тяжело сопровождать. Это также обсуждается в упомянутом [issue на GitHub](https://github.com/angular/angular/issues/56155#issuecomment-2177202036).

Кроме того, эффекты делают код более императивным и менее декларативным — декларативность понимается в смысле реактивного программирования. О том, как повысить декларативность кода, рассказывает мой коллега по GDE [Mike F. Pearson](https://x.com/mfpears), автор [StateAdapt](https://state-adapt.github.io/angular).

Ещё один момент: сигналы свободны от «глитчей» (glitch-free) — то есть промежуточные несогласованные состояния не наблюдаются. Если вы меняете сигнал несколько раз подряд (в пределах одного кадра стека, stack frame), наблюдается только последнее изменение. Это удобно для рендеринга, но также показывает, что сигналы не подходят для представления событий (промежуточные значения теряются).

## Эффекты для реактивных помощников

Хотя в документации это не упомянуто, эффекты также используются «под капотом» для построения реактивных помощников (reactive helpers). Примеры — `toObservable` в Angular, `rxMethod` в NGRX Signal Store, а также многочисленные помощники в библиотеке [ngxtension](https://ngxtension.netlify.app/) от [Chau Tran](https://x.com/Nartc1410) и [Enea Jahollari](https://x.com/Enea_Jahollari).

К этой теме я вернусь в конце этой короткой статьи.

## Реагирование на изменения сигналов

До сих пор мы обсуждали, когда применять эффекты и когда их следует избегать. Однако остаётся вопрос, который обычно возникает довольно быстро: как реагировать на изменяющиеся сигналы?

Есть несколько ответов:

1) Выводите состояние через `computed` или Resource API  
2) Используйте событие, стоящее за изменением сигнала  
3) Используйте RxJS  
4) Используйте реактивный помощник

`computed` — правильный выбор, когда нужное значение выводится из существующих сигналов _синхронно_. Для асинхронного вывода Angular предоставляет [Resource API](https://www.angulararchitects.io/en/blog/asynchronous-resources-with-angulars-new-resource-api/).

В случае пункта 2 вы не отслеживаете изменение сигнала, а реагируете на событие, стоящее за этим изменением:

![](https://www.angulararchitects.io/wp-content/uploads/2024/09/events-behind.png)

Использование события, стоящего за изменением, устраняет упомянутые выше проблемы — например, сложности авто-отслеживания или циклические обновления.

Под пунктом 3 я подразумеваю применение RxJS — вместо сигналов или в дополнение к ним. Во втором случае переход между двумя моделями выполняется через RxJS-интероп Angular, предоставляющий функции `toObservable` и `toSignal`. RxJS позволяет выстроить сквозной (end-to-end) реактивный поток — например, от действия пользователя до вывода. Также учитывайте, что уплощающие операторы RxJS (flattening operators), такие как `switchMap`, дают гарантии относительно перекрывающихся запросов (overlapping requests) и тем самым помогают избежать состояний гонки (race conditions). Эту тему разобрал мой коллега по GDE [Jan-Niklas Wortmann](https://github.com/niklas-wortmann) в [докладе про реактивность в Angular-приложениях](https://www.youtube.com/watch?v=p2JGWnUmdNw) на ng-conf 2024.

Примеры реактивных помощников из пункта 4 — `rxMethod` в [NGRX Signal Store](https://ngrx.io/guide/signals/signal-store) или `deriveAsync` в [ngxtension](https://ngxtension.netlify.app/). Метод, настроенный через `rxMethod`, может принять сигнал и подключить его к pipe на базе RxJS:

```
this.store.rxLoad(this.id);
```

Ниже простой Signal Store, реализующий `rxLoad`:

```
export const DessertDetailStore = signalStore(
    { providedIn: 'root' },
    withState({
        dessert: initDessert,
        loading: false
    }),
    withMethods((
        store,
        dessertService = inject(DessertService)
    ) => ({
        rxLoad: rxMethod<number>(pipe(
            tap(() => patchState(store, { loading: true })),
            switchMap((id) => dessertService.findById(id)),
            tap((dessert) => patchState(store, { dessert, loading: false })),
        )),
    }))
);
```

Ради простоты этот пример не содержит обработки ошибок.

## За и против explicitEffect

Ещё один часто обсуждаемый помощник, который также упоминает Alex Rickabaugh в [указанном issue на GitHub](https://github.com/angular/angular/issues/56155#issuecomment-2177202036), — это `explicitEffect`. Его реализацию предоставляет библиотека [ngxtension](https://ngxtension.netlify.app/). По сути, это комбинация `effect` и `untracked`, ограничивающая отслеживание лишь несколькими явно перечисленными сигналами:

```
explicitEffect(this.id, (id) => {
  this.store.load(id);
});
```

В этом случае отслеживается только сигнал `id`. Явный эффект (explicit effect) — простой способ смягчить недостатки авто-отслеживания при использовании эффектов не для рендеринга, как описано выше. Однако он не компенсирует прочие уже рассмотренные проблемы, сопутствующие применению эффектов для таких задач. Например, вы всё равно можете получить циклические обновления и код, который трудно анализировать (hard to reason about). Кроме того, такой код императивен и, следовательно, не соответствует реактивной парадигме, в которой мы стремимся к декларативному коду и реактивной цепочке, связывающей входы и выходы.

Если вы придерживаетесь реактивной парадигмы, то будете избегать `explicitEffect`. Вместо него вы, скорее всего, выберете RxJS или помощники вроде `rxMethod`, множество декларативных помощников в [ngxtension](https://ngxtension.netlify.app/) либо Angular [Resource API](https://www.angulararchitects.io/en/blog/asynchronous-resources-with-angulars-new-resource-api/). Это и есть рекомендуемый подход. Если же он вам не подходит, то и вы, и ваша команда должны осознавать последствия применения такого рода эффектов.

## Резюме

Эффекты предназначены прежде всего для отрисовки того, что нельзя отрисовать через привязку данных. Примеры — логирование, отрисовка на canvas или нестандартное поведение DOM, такое как отображение `SnackBar` через императивный API, как в Angular Material. Чтобы реагировать на изменение сигнала, отдавайте предпочтение `computed`, если нужное значение выводится синхронно, либо Resource API — для асинхронного вывода. Также используйте события, стоящие за изменениями сигналов, RxJS или реактивные помощники, чтобы выстроить сквозную реактивную цепочку.
