## Главный сценарий для эффектов

Эффекты (effects) нужны прежде всего для того, чтобы отрисовывать то, что невозможно вывести через привязку данных (data binding).

Вот несколько примеров — их вы найдёте и в [документации](https://angular.dev/guide/signals#effects):

1) Логирование  
2) Рисование на Canvas  
3) Нестандартное поведение DOM

Поскольку привязка данных — это основной и предпочтительный способ показывать данные пользователю, становится понятно, почему документация говорит: эффекты нужны лишь время от времени.

Отличный пример для пункта 2 — замечательная библиотека [Angular Three](https://angular-threejs.netlify.app) от [Chau Tran](https://x.com/Nartc1410), которая связывает Angular и Three.js.

Пункт 3 — про поведение DOM, которое нельзя выразить синтаксисом шаблона. Например, показ `SnackBar` через императивный API — такой, как в Angular Material.

## Не забывайте про авто-отслеживание

Помните, что для `computed` и `effects` Angular использует неявное отслеживание (auto-tracking). В примере ниже эффект будет отслеживать сигнал `error`, хотя тот напрямую в функции эффекта не используется — он вызывается внутри метода `logError`.

```
effect(() => {
  this.logError();
});
:
logError(): void {
  const error = this.error();
  if (error) {
    console.error(error);
  }
}
```

Это лишний раз подчёркивает, что текущая реализация эффектов в Angular рассчитана в первую очередь на рендеринг. Сигналы, к которым обращаются во время рендеринга, отслеживаются. Меняете их — рендеринг запускается снова. Это проектное решение подробно разобрал [Alex Rickabaugh](https://x.com/synalx) из команды Angular в этом [issue на GitHub](https://github.com/angular/angular/issues/56155#issuecomment-2177202036).

## Когда НЕ стоит использовать эффекты?

В [документации](https://angular.dev/guide/signals#effects) прямо сказано: НЕ используйте эффекты для распространения состояния (propagating state). Здесь возможен целый ряд проблем — например, циклические обновления. От себя добавлю: поведение авто-отслеживания легко приводит к коду, который потом тяжело поддерживать. Об этом тоже говорится в упомянутом [issue на GitHub](https://github.com/angular/angular/issues/56155#issuecomment-2177202036).

Более того, эффекты делают ваш код более императивным и менее декларативным — декларативность здесь понимается в духе реактивного программирования. Если хотите узнать, как сделать код декларативнее, вам поможет мой коллега по GDE [Mike F. Pearson](https://x.com/mfpears), автор [StateAdapt](https://state-adapt.github.io/angular).

Ещё один момент, о котором стоит помнить: сигналы свободны от «глитчей» (glitch-free). Если вы меняете сигнал несколько раз подряд (в рамках одного кадра стека), будет видно только последнее изменение. Для рендеринга это удобно, но это же показывает, что для представления событий сигналы не подходят.

## Эффекты для реактивных помощников

Хотя в документации об этом не говорится, эффекты под капотом используются и для построения реактивных помощников (reactive helpers). Примеры — `toObservable` в Angular, `rxMethod` в NGRX Signal Store, а также множество помощников из библиотеки [ngxtension](https://ngxtension.netlify.app/) от [Chau Tran](https://x.com/Nartc1410) и [Enea Jahollari](https://x.com/Enea_Jahollari).

К этой теме я ещё вернусь в конце этой короткой статьи.

## Как реагировать на изменения сигналов

До сих пор мы обсуждали, когда эффекты уместны, а когда их лучше избегать. Но остаётся вопрос, который обычно возникает довольно быстро: как же тогда реагировать на изменения сигналов?

Есть несколько ответов:

1) Выводите состояние через `computed` или Resource API  
2) Используйте событие, стоящее за изменением сигнала  
3) Используйте RxJS  
4) Используйте реактивный помощник

`computed` — это правильный выбор, когда нужное значение можно вывести из существующих сигналов _синхронно_. Для асинхронного вывода Angular предлагает [Resource API](https://www.angulararchitects.io/en/blog/asynchronous-resources-with-angulars-new-resource-api/).

В случае пункта 2 вы не отслеживаете изменение сигнала, а реагируете на событие, которое за ним стоит:

![](https://www.angulararchitects.io/wp-content/uploads/2024/09/events-behind.png)

Использование события, стоящего за изменением, снимает упомянутые выше проблемы — вроде сложностей с авто-отслеживанием или циклическими обновлениями.

Под пунктом 3 я имею в виду использование RxJS — вместо сигналов или вместе с ними. Во втором случае вы переключаетесь между двумя мирами с помощью RxJS-интеропа Angular, который даёт функции вроде `toObservable` и `toSignal`. RxJS позволяет выстроить реактивный поток от начала до конца — например, от действия пользователя до результата. И учтите: уплощающие операторы RxJS, такие как `switchMap`, дают гарантии относительно перекрывающихся запросов и тем самым помогают избежать гонок (race conditions). Эту тему разобрал мой коллега по GDE [Jan-Niklas Wortmann](https://github.com/niklas-wortmann) в своём [докладе про реактивность в Angular-приложениях](https://www.youtube.com/watch?v=p2JGWnUmdNw) на ng-conf 2024.

Примеры реактивных помощников из пункта 4 — это `rxMethod` в [NGRX Signal Store](https://ngrx.io/guide/signals/signal-store) или `deriveAsync` в [ngxtension](https://ngxtension.netlify.app/). Метод, созданный через `rxMethod`, может принять сигнал и подключить его к pipe на основе RxJS:

```
this.store.rxLoad(this.id);
```

Вот простой Signal Store, реализующий `rxLoad`:

```
export const DessertDetailStore = signalStore(
    { providedIn: 'root' },
    withState({
        dessert: initDessert,
        loading: false
    }),
    withMethods((
        store,
        dessertService = inject(DessertService)
    ) => ({
        rxLoad: rxMethod<number>(pipe(
            tap(() => patchState(store, { loading: true })),
            switchMap((id) => dessertService.findById(id)),
            tap((dessert) => patchState(store, { dessert, loading: false })),
        )),
    }))
);
```

Ради простоты в этом примере нет обработки ошибок.

## За и против explicitEffect

Ещё один часто обсуждаемый помощник, о котором тоже говорит Alex Rickabaugh в [упомянутом issue на GitHub](https://github.com/angular/angular/issues/56155#issuecomment-2177202036), — это `explicitEffect`. Реализацию предоставляет библиотека [ngxtension](https://ngxtension.netlify.app/). По сути, это комбинация `effect` и `untracked`, ограничивающая отслеживание лишь несколькими явно указанными сигналами:

```
explicitEffect(this.id, (id) => {
  this.store.load(id);
});
```

Здесь отслеживается только сигнал `id`. Явный эффект (explicit effect) — это простой способ смягчить минусы авто-отслеживания, когда вы используете эффекты не для рендеринга, а для чего-то другого, как описано выше. Однако он не избавляет от прочих уже упомянутых проблем, которые возникают при таком применении эффектов. Например, вы всё равно можете столкнуться с циклическими обновлениями и кодом, в котором трудно разобраться. К тому же такой код императивен и, значит, не вписывается в реактивную парадигму, где мы стремимся к декларативному коду и реактивной цепочке, соединяющей входы и выходы.

Если вы приверженец реактивной парадигмы, то `explicitEffect` вы будете избегать. Вместо него вы, скорее всего, выберете RxJS или помощники вроде `rxMethod`, множество декларативных помощников из [ngxtension](https://ngxtension.netlify.app/) или Angular [Resource API](https://www.angulararchitects.io/en/blog/asynchronous-resources-with-angulars-new-resource-api/). Это и есть рекомендуемый подход. Если же он вам не подходит, то и вы, и ваши коллеги должны понимать, к каким последствиям приводит использование такого рода эффектов.

## Итоги

Эффекты нужны прежде всего для того, чтобы отрисовывать то, что нельзя вывести через привязку данных. Примеры — логирование, рисование на canvas или нестандартное поведение DOM, например показ `SnackBar` через императивный API вроде того, что есть в Angular Material. Чтобы реагировать на изменение сигнала, отдавайте предпочтение `computed`, если нужное значение можно вывести синхронно, или Resource API — для асинхронного вывода. Также используйте события, стоящие за изменениями сигналов, RxJS или реактивные помощники, чтобы выстроить сквозную реактивную цепочку.
