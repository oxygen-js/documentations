## Основной сценарий использования эффектов

Эффекты (effects) предназначены прежде всего для отрисовки того, что нельзя отрисовать с помощью привязки данных (data binding).

Вот несколько примеров, которые вы также можете найти в [документации](https://angular.dev/guide/signals#effects):

1) Логирование  
2) Рисование на Canvas  
3) Нестандартное поведение DOM

Поскольку привязка данных является предпочтительным способом отображения данных пользователю, очевидно, почему документация говорит нам, что эффекты нужны лишь иногда.

Хороший пример для пункта 2 — это превосходная библиотека [Angular Three](https://angular-threejs.netlify.app) от [Chau Tran](https://x.com/Nartc1410), соединяющая Angular и Three.js.

Пункт 3 — про поведение DOM, которое нельзя выразить с помощью синтаксиса шаблона. Пример — показ `SnackBar` с императивным API, например того, что есть в Angular Material.

## Помните об авто-отслеживании

Пожалуйста, помните, что Angular использует неявное отслеживание (авто-отслеживание, auto-tracking) для `computed` и `effects`. В следующем примере эффект будет отслеживать сигнал `error`, хотя тот не используется напрямую в функции эффекта, а используется внутри вызываемого метода `logError`.

```
effect(() => {
  this.logError();
});
:
logError(): void {
  const error = this.error();
  if (error) {
    console.error(error);
  }
}
```

Это также подчёркивает, что текущая реализация эффектов в Angular предназначена прежде всего для рендеринга. Сигналы, к которым обращаются во время рендеринга, отслеживаются. Если вы их меняете, рендеринг запускается снова. [Alex Rickabaugh](https://x.com/synalx) из Angular обсудил это проектное решение в этом [issue на GitHub](https://github.com/angular/angular/issues/56155#issuecomment-2177202036).

## Когда НЕ использовать эффекты?

В [документации](https://angular.dev/guide/signals#effects) также прямо упоминается, что НЕ следует использовать эффекты для распространения состояния (propagating state). Здесь возможен ряд проблем, таких как циклические обновления. Я бы добавил: поведение авто-отслеживания может быстро привести к коду, который становится трудно поддерживать. Это также обсуждается в упомянутом [issue на GitHub](https://github.com/angular/angular/issues/56155#issuecomment-2177202036).

Кроме того, эффекты делают ваш код более императивным и менее декларативным. Под последним понимается декларативность в смысле реактивного программирования. Если вы хотите узнать, как сделать ваш код более декларативным, вам поможет мой коллега по GDE [Mike F. Pearson](https://x.com/mfpears), автор [StateAdapt](https://state-adapt.github.io/angular).

Ещё одна вещь, о которой нужно помнить: сигналы свободны от «глитчей» (glitch-free). Если вы меняете сигнал несколько раз подряд (в пределах одного кадра стека), видно только последнее изменение. Это подходит для рендеринга, но также показывает, что сигналы не подходят для представления событий.

## Эффекты для реактивных помощников

Хотя в документации это не упоминается, эффекты также используются под капотом для построения реактивных помощников (reactive helpers). Пример — это `toObservable` в Angular, `rxMethod` в NGRX Signal Store и многие помощники в библиотеке [ngxtension](https://ngxtension.netlify.app/) от [Chau Tran](https://x.com/Nartc1410) и [Enea Jahollari](https://x.com/Enea_Jahollari).

Я вернусь к этой теме в конце этой короткой статьи.

## Реагирование на изменения сигналов

До сих пор мы обсуждали, когда использовать эффекты и когда их следует избегать. Однако остаётся вопрос, который обычно возникает довольно быстро: как реагировать на изменяющиеся сигналы?

Есть несколько ответов:

1) Выводите ваше состояние через `computed` или Resource API  
2) Используйте событие, стоящее за изменением сигнала  
3) Используйте RxJS  
4) Используйте реактивный помощник

Использование `computed` — это правильный путь, когда нужное значение можно вывести из существующих сигналов _синхронно_. Для асинхронного вывода Angular предоставляет [Resource API](https://www.angulararchitects.io/en/blog/asynchronous-resources-with-angulars-new-resource-api/).

В случае пункта 2 вы не отслеживаете изменение сигнала, а реагируете на событие, стоящее за ним:

![](https://www.angulararchitects.io/wp-content/uploads/2024/09/events-behind.png)

Использование события, стоящего за изменением, смягчает упомянутые выше проблемы, такие как проблемы с авто-отслеживанием или циклическими обновлениями.

Под пунктом 3 я подразумеваю использование RxJS вместо сигналов или в дополнение к ним. В последнем случае вы выполняете преобразование между двумя мирами с помощью RxJS-интеропа Angular, предоставляющего функции вроде `toObservable` и `toSignal`. RxJS позволяет иметь реактивный поток от начала до конца, например, от действия пользователя до вывода. Также имейте в виду, что уплощающие операторы RxJS, такие как `switchMap`, дают гарантии в отношении перекрывающихся запросов и, следовательно, помогают избежать состояний гонки (race conditions). Мой коллега по GDE [Jan-Niklas Wortmann](https://github.com/niklas-wortmann) обсудил эту тему в своём [докладе про реактивность в Angular-приложениях](https://www.youtube.com/watch?v=p2JGWnUmdNw) на ng-conf 2024.

Примеры реактивных помощников, упомянутые в пункте 4, — это `rxMethod` в [NGRX Signal Store](https://ngrx.io/guide/signals/signal-store) или `deriveAsync` в [ngxtension](https://ngxtension.netlify.app/). Метод, настроенный с помощью `rxMethod`, может принять сигнал и подключить его к pipe на основе RxJS:

```
this.store.rxLoad(this.id);
```

Вот простой Signal Store, реализующий `rxLoad`:

```
export const DessertDetailStore = signalStore(
    { providedIn: 'root' },
    withState({
        dessert: initDessert,
        loading: false
    }),
    withMethods((
        store,
        dessertService = inject(DessertService)
    ) => ({
        rxLoad: rxMethod<number>(pipe(
            tap(() => patchState(store, { loading: true })),
            switchMap((id) => dessertService.findById(id)),
            tap((dessert) => patchState(store, { dessert, loading: false })),
        )),
    }))
);
```

Ради простоты этот пример не содержит обработки ошибок.

## За и против explicitEffect

Ещё один часто обсуждаемый помощник, который также обсуждается Alex Rickabaugh в [упомянутом issue на GitHub](https://github.com/angular/angular/issues/56155#issuecomment-2177202036), — это `explicitEffect`. Библиотека [ngxtension](https://ngxtension.netlify.app/) предоставляет его реализацию. По сути, это комбинация `effect` и `untracked`, ограничивающая отслеживание несколькими явно указанными сигналами:

```
explicitEffect(this.id, (id) => {
  this.store.load(id);
});
```

В этом случае отслеживается только сигнал `id`. Явный эффект (explicit effect) — это простой способ смягчить недостатки авто-отслеживания при использовании эффектов для чего-то помимо рендеринга, как обсуждалось выше. Однако он не компенсирует другие уже обсуждённые проблемы, которые возникают при использовании эффектов для таких задач. Например, вы можете прийти к циклическим обновлениям и коду, в котором трудно разобраться. Кроме того, такой код императивен и, следовательно, не соответствует реактивной парадигме, в которой мы стремимся иметь декларативный код и реактивную цепочку, соединяющую входы и выходы.

Если вы отдаёте предпочтение реактивной парадигме, вы будете избегать `explicitEffect`. Вместо этого вы, скорее всего, выберете RxJS или помощники вроде `rxMethod`, множество декларативных помощников в [ngxtension](https://ngxtension.netlify.app/) или Angular [Resource API](https://www.angulararchitects.io/en/blog/asynchronous-resources-with-angulars-new-resource-api/). Это также рекомендуемый подход. Если же это не ваш случай, то вы и ваши коллеги должны осознавать последствия, которые возникают при использовании такого рода эффектов.

## Резюме

Эффекты предназначены прежде всего для отрисовки того, что нельзя отрисовать через привязку данных. Примеры — это логирование, рисование на canvas или нестандартное поведение DOM, такое как показ `SnackBar` с императивным API, например того, что есть в Angular Material. Чтобы реагировать на изменение сигнала, вам следует отдавать предпочтение `computed`, если нужное значение можно вывести синхронно, или Resource API — для асинхронного вывода. Также используйте события, стоящие за изменениями сигналов, RxJS или реактивные помощники, чтобы установить сквозную реактивную цепочку.
