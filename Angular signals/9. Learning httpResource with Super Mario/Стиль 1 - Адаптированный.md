## Изучаем httpResource на примере Super Mario

В Angular 19.2 появился новый `httpResource`, который загружает данные прямо внутри реактивного потока. В этой статье мы посмотрим, как пользоваться этой — пока ещё экспериментальной — возможностью. Для примера соберём небольшое приложение, в котором можно листать уровни в стиле классической игры Super Mario.

Каждый уровень собирается из тайлов (плиток), и у нас есть четыре стиля оформления на выбор — Overworld (наземный мир), Underground (подземелье), Underwater (под водой) и Castle (замок):

📂 [Исходный код](https://github.com/manfredsteyer/mario.git) содержит тот самый компонент, который через `httpResource` загружает файлы уровней (JSON) и тайлы для их отрисовки. За саму отрисовку и анимацию уровней отвечает очень простой «движок» («engine»), который в этой статье мы рассматриваем как чёрный ящик.

## Файлы уровней

Каждый уровень описывается JSON-файлом, который задаёт, какие тайлы — например, пол или облако — и в каких координатах нужно показать:

```json
{
  "levelId": 1,
  "backgroundColor": "#9494ff",
  "items": [
    { "tileKey": "floor", "col": 0, "row": 13, [...] },
    { "tileKey": "cloud", "col": 12, "row": 1, [...] },
    [...]
  ]
}
```

Координаты задают положение внутри матрицы из блоков размером 16×16 пикселей. Помимо самих файлов уровней, есть ещё `overview.json` — он сообщает, какие уровни вообще доступны:

```json
{
  "levels": [
 {
      "levelKey": "01",
      "title": "Level 1"
 },
 {
      "levelKey": "02",
      "title": "Level 2"
 },
 [...]
 ]
}
```

## Загрузка JSON с помощью ngResource

Чтобы абстрагировать доступ к данным, удобно спрятать создание `httpResource` внутри сервиса:

```typescript
@Injectable({ providedIn: 'root' })
export class LevelLoader {
  getLevelOverviewResource(): HttpResourceRef<LevelOverview> {
    return httpResource<LevelOverview>(() => '/levels/overview.json', {
      defaultValue: initLevelOverview,
    });
  }

  getLevelResource(levelKey: () => string | undefined): HttpResourceRef<Level> {
    return httpResource<Level>(() => !levelKey() ? undefined : /levels/${levelKey()}.json, {
      defaultValue: initLevel,
    });
  }

 [...]
}
```

Объект с настройками, который передаётся вторым параметром, позволяет задать значение по умолчанию (`defaultValue`) — оно используется до того, как ресурс (resource) загрузится.

Чтобы указать, какой именно уровень загрузить, `getLevelResource` ожидает сигнал (signal), возвращающий ключ уровня, из которого сервис уже выводит имя файла. Этот сигнал доступен только для чтения и описан общей абстракцией `() => string | undefined`.

Обратите внимание: URL, который передаётся в `getLevelResource`, представлен **лямбда-выражением.** Это нужно для того, чтобы ресурс **автоматически пересчитывал** URL каждый раз, когда меняется входящий в него сигнал `levelKey`. Под капотом `httpResource` использует это лямбда-выражение, чтобы построить вычисляемый сигнал (Computed Signal), играющий роль триггера: как только этот триггер меняется, ресурс загружает URL заново.

Чтобы `httpResource` не сработал, выражение должно вернуть `undefined`. Это пригодится, когда приложению нужно отложить загрузку, пока не появятся нужные параметры — например, тот самый `levelKey`.

## Больше возможностей с явным HttpResourceRequest

Если нужно больше контроля над отправляемым HTTP-запросом (request), вместо URL можно передать объект `HttpResourceRequest`:

```typescript
getLevelResource(levelKey: () => string) {
  return httpResource<Level>(
    () => ({
      url: /levels/${levelKey()}.json,
      method: "GET",
      headers: {
        accept: "application/json",
      },
      params: {
        levelId: levelKey(),
      },
      reportProgress: false,
      body: null,
      transferCache: false,
      withCredentials: false,
    }),
    { defaultValue: initLevel }
  );
}
```

Этот `HttpResourceRequest` тоже можно представить лямбда-выражением, на основе которого `httpResource` внутри себя построит вычисляемый сигнал (Computed Signal).

Важно помнить: `httpResource` предназначен **только для получения данных,** даже несмотря на то, что мы можем задать метод (HTTP-глагол) отличный от `GET` и передать `body` в качестве полезной нагрузки. Эта возможность нужна, чтобы работать с веб-API, которые не вписываются в привычную семантику HTTP-глаголов. По умолчанию переданное `body` сериализуется в JSON. Но в целом этот механизм не предназначен для записи данных обратно на сервер — такую задачу `httpResource` не решает. К тому же автоматическое применение семантики `switchMap` отлично подходит для чтения данных, но почти наверняка окажется неверным поведением при записи данных на сервер.

С помощью опции `reportProgress` вызывающий код может запросить информацию о ходе текущей операции. Это удобно при скачивании больших файлов — пример этого приведён ниже.

## Разбор и валидация полученных данных

По умолчанию `httpResource` ожидает JSON, соответствующий указанному параметру типа. То есть после разбора полученного JSON он просто применяет приведение типов (type assertion), чтобы TypeScript считал данные нужным типом.

Однако в этот процесс можно вклиниться и добавить собственную логику: проверить полученное «сырое» значение и преобразовать его к типу, заданному параметром типа. Для этого в объекте настроек задают функцию `parse`:

```typescript
getLevelResourceAlternative(levelKey: () => string) {
  return httpResource<Level>(() => /levels/${levelKey()}.json, {
    defaultValue: initLevel,
    parse: (raw) => {
      return toLevel(raw);
    },
  });
}
```

`httpResource` превращает полученный JSON в объект типа `unknown` и передаёт его в `parse`. В нашем примере используется простая самописная функция `toLevel`. Но `parse` можно использовать и для того, чтобы подключить ресурс к специализированной библиотеке валидации схем — например, к [Zod](https://zod.dev/).

## Загрузка данных не только в формате JSON

По умолчанию `httpResource` ждёт JSON, преобразованный в JavaScript-объект, но вызывающий код может запросить и другие представления данных:

* `httpResource.text` возвращает обычный текст;
* `httpResource.blob` возвращает загруженные данные в виде Blob;
* `httpResource.arrayBuffer` возвращает загруженные данные в виде ArrayBuffer.

Чтобы это продемонстрировать, в примере мы запрашиваем картинку со всеми возможными тайлами в виде Blob. Из этого Blob можно «вырезать» нужные тайлы для выбранного стиля уровня.

На следующей картинке показан фрагмент такой карты тайлов. Видно, что пример легко переключается между стилями, просто смещаясь по горизонтали или по вертикали:

![Фрагмент карты тайлов, используемой в примере](https://www.angulararchitects.io/wp-content/uploads/2025/02/tile-map.png)

Эта карта тайлов взята [отсюда](https://www.spriters-resource.com/nes/supermariobros/sheet/52571). Для её загрузки сервис `TilesMapLoader` делегирует работу `httpResource.blob`:

```typescript
@Injectable({ providedIn: "root" })
export class TilesMapLoader {
  getTilesMapResource(): HttpResourceRef<Blob | undefined> {
    return httpResource.blob(() => {
      url: "/tiles.png",
      reportProgress: true,
    });
  }
}
```

Этот ресурс также запрашивает информацию о прогрессе, которую пример подхватывает и показывает слева от выпадающих списков.

## Замечание на полях: под капотом работает HttpClient, поэтому доступны интерсепторы

Под капотом новый `httpResource` сейчас использует `HttpClient`. Значит, нам нужно предоставить HttpClient — обычно это делается через `provideHttpClient` на этапе начальной загрузки (bootstrapping). Как следствие, `httpResource` автоматически использует и зарегистрированные `HttpInterceptor`-ы.

При этом стоит помнить: использование `HttpClient` — это деталь реализации, и в будущем её вполне могут заменить чем-то другим.

## Собираем всё вместе: реактивный поток

Теперь, когда у нас есть фабрики для всех `httpResource`-ов, можно описать наш реактивный граф:

![Реактивный поток ngMario](https://www.angulararchitects.io/wp-content/uploads/2025/06/graph-scaled.png)

Сигналы `levelKey`, `style` и `animation` представляют ввод пользователя. Первые два соответствуют выпадающим спискам наверху. Сигнал `animation` содержит `boolean` и сообщает, запущена ли анимация — её включает кнопка `Toggle Animation` (см. скриншот выше).

`tilesResource` в верхней средней части — это классический ресурс, который выводит отдельные тайлы для выбранного стиля. По сути, он просто делегирует работу функции из игрового «движка», который мы тут рассматриваем как чёрный ящик.

Отрисовка запускается эффектом (effect), потому что нарисовать уровень напрямую через привязку данных (data binding) не получится. Эффект рисует или анимирует уровень на холсте (canvas), полученном как `viewChild` на основе сигналов. Этот эффект срабатывает всякий раз, когда меняется `level` (его даёт `levelResource`), `style`, флаг `animation` или сам `canvas`.

Сигнал `tilesMapProgress` использует информацию о прогрессе из `tilesMapResource`, чтобы показать, какая часть карты тайлов уже скачана. Для загрузки списка доступных уровней пример использует `levelOverviewResource`, который напрямую не связан с описанным выше реактивным графом.

В следующем листинге показана реализация этого реактивного потока в виде полей компонента `LevelComponent`:

```typescript
export class LevelComponent implements OnDestroy {
  private tilesMapLoader = inject(TilesMapLoader);
  private levelLoader = inject(LevelLoader);

  canvas = viewChild<ElementRef<HTMLCanvasElement>>("canvas");

  levelKey = linkedSignal<string | undefined>(() => this.getFirstLevelKey());
  style = signal<Style>("overworld");
  animation = signal(false);

  tilesMapResource = this.tilesMapLoader.getTilesMapResource();
  levelResource = this.levelLoader.getLevelResource(this.levelKey);
  levelOverviewResource = this.levelLoader.getLevelOverviewResource();

  tilesResource = createTilesResource(this.tilesMapResource, this.style);

  tilesMapProgress = computed(() =>
    calcProgress(this.tilesMapResource.progress())
  );

  constructor() {
    [...]
    effect(() => {
      this.render();
    });
  }

  reload() {
    this.tilesMapResource.reload();
    this.levelResource.reload();
  }

  private getFirstLevelKey(): string | undefined {
    return this.levelOverviewResource.value()?.levels?.[0]?.levelKey;
  }

  [...]
}
```

Использование `linkedSignal` для `levelKey` позволяет нам взять первый уровень в качестве значения по умолчанию, как только список уровней будет загружен. Вспомогательная функция `getFirstLevelKey` как раз и возвращает этот первый уровень из `levelOverviewResource`.

Эффект отрисовки, по сути, забирает упомянутые значения и делегирует работу функциям «движка» — `animateLevel` или `rederLevel`:

```typescript
private render() {
  const tiles = this.tilesResource.value();
  const level = this.levelResource.value();
  const canvas = this.canvas()?.nativeElement;
  const animation = this.animation();

  if (!tiles || !canvas) {
    return;
  }

  if (animation) {
    animateLevel({
      canvas,
      level,
      tiles,
    });
  } else {
    renderLevel({
      canvas,
      level,
      tiles,
    });
  }
}
```

## Ресурсы и недостающие параметры

`tilesResource`, показанный на схеме выше, по сути делегирует работу асинхронной функции `extractTiles`, которую тоже предоставляет игровой «движок»:

```typescript
function createTilesResource(
  tilesMapResource: HttpResourceRef<Blob | undefined>,
  style: () => Style
) {
  const tilesMap = tilesMapResource.value();

  // undefined не даёт ресурсу запуститься
  const params = computed(() =>
    !tilesMap
      ? undefined
      : {
          tilesMap: tilesMap,
          style: style(),
        }
  );

  return resource({
    params,
    loader: (loaderParams) => {
      const { tilesMap, style } = loaderParams.params;
      return extractTiles(tilesMap, style);
    },
  });
}
```

В этом простом ресурсе есть одна важная деталь: пока карта тайлов не загрузилась, `value` у `tilesMapResource` равно `undefined`. Но без `tilesMap` мы не можем вызвать `extractTiles` и, соответственно, не можем запустить `tilesResource`. Сигнал `params` учитывает это: в таком случае он тоже возвращает `undefined`, чтобы **загрузчик (loader) не вызывался.**

## Показ прогресса

Выше мы настроили `tilesMapResource` так, чтобы он сообщал о ходе загрузки через сигнал `progress`. С помощью вычисляемого сигнала в `LevelComponent` эта информация превращается в строку:

```typescript
function calcProgress(progress: HttpProgressEvent | undefined): string {
  if (!progress) {
    return "-";
  }

  if (progress.total) {
    const percent = Math.round((progress.loaded / progress.total) * 100);
    return percent + "%";
  }

  const kb = Math.round(progress.loaded / 1024);
  return kb + " KB";
}
```

Если сервер сообщает общий размер файла, функция считает процент загрузки. Если нет — просто показывает количество скачанных килобайт. До начала загрузки информации о прогрессе ещё нет, и в этом случае возвращается прочерк.

Чтобы проверить эту возможность, можно искусственно ограничить скорость сети в инструментах разработчика браузера и нажать кнопку `reload`, которая вызовет метод `reload` ресурса.

## Статус, заголовки, ошибка и не только

Если приложению нужно знать код статуса или полученные заголовки, `httpResource` предоставляет для этого отдельные сигналы:

```typescript
console.log("status", this.levelOverviewResource.status());
console.log("statusCode", this.levelOverviewResource.statusCode());
console.log("headers", this.levelOverviewResource.headers()?.keys());
```

Помимо этого, `httpResource` даёт всё то, что мы уже знаем по обычным ресурсам: сигнал `error`, сообщающий о возможной ошибке, и возможность обновить `value`, который трактуется как локальная рабочая копия. Это также означает, что значение не записывается на сервер автоматически — эту задачу по-прежнему нужно решать классическим способом.

## Заключение

Новый `httpResource` — это ещё один кирпичик, дополняющий новую модель реактивности в Angular. Он позволяет загружать данные прямо внутри реактивного графа. Сейчас под капотом он использует `HttpClient` как деталь реализации, которую со временем могут заменить другим решением.

Хотя `HTTP resource` и позволяет получать данные с использованием HTTP-глаголов не только GET (например, через вызовы POST), он не предназначен для записи данных обратно на сервер. Эту задачу по-прежнему нужно решать традиционным способом.
