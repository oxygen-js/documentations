## Изучаем httpResource на примере Super Mario

Angular 19.2 поставляется с новым `httpResource`, который загружает данные внутри реактивного потока. В этой статье показано, как использовать эту — на текущий момент экспериментальную — возможность, реализовав простое приложение, прокручивающее уровни в стиле традиционной игры Super Mario.

Каждый уровень состоит из тайлов, которые доступны в 4 различных стилях — Overworld, Underground, Underwater и Castle — и которые мы можем свободно выбирать:

📂 [Исходный код](https://github.com/manfredsteyer/mario.git) предоставляет показанный компонент, который отвечает за загрузку файлов уровней (JSON) и тайлов для отрисовки уровней через `httpResource`. Для рендеринга и анимации уровней код также **содержит** очень простой **«движок» («engine»)**, который в этой статье рассматривается как **чёрный ящик**.

## Файлы уровней

Различные уровни представлены JSON-файлами, определяющими, какие тайлы — например, пол или облако — показываются в каких координатах:

```json
{
  "levelId": 1,
  "backgroundColor": "#9494ff",
  "items": [
    { "tileKey": "floor", "col": 0, "row": 13, [...] },
    { "tileKey": "cloud", "col": 12, "row": 1, [...] },
    [...]
  ]
}
```

Координаты определяют положение внутри матрицы из блоков размером 16x16 пикселей. Помимо этих файлов уровней, существует `overview.json`, сообщающий о доступных уровнях:

```json
{
  "levels": [
 {
      "levelKey": "01",
      "title": "Level 1"
 },
 {
      "levelKey": "02",
      "title": "Level 2"
 },
 [...]
 ]
}
```

## Загрузка JSON с помощью ngResource

Чтобы абстрагировать доступ к данным, скрыв создание `httpResource` в сервисе, является хорошей идеей:

```typescript
@Injectable({ providedIn: 'root' })
export class LevelLoader {
  getLevelOverviewResource(): HttpResourceRef<LevelOverview> {
    return httpResource<LevelOverview>(() => '/levels/overview.json', {
      defaultValue: initLevelOverview,
    });
  }

  getLevelResource(levelKey: () => string | undefined): HttpResourceRef<Level> {
    return httpResource<Level>(() => !levelKey() ? undefined : /levels/${levelKey()}.json, {
      defaultValue: initLevel,
    });
  }

 [...]
}
```

Объект опций, принимаемый вторым параметром, позволяет задать значение по умолчанию (default value), которое используется до того, как ресурс (resource) будет загружен.

Чтобы определить, какой уровень загружать, `getLevelResource` ожидает сигнал (Signal), возвращающий ключ уровня, из которого сервис выводит имя файла. Этот доступный только для чтения сигнал представлен общей абстракцией `() => string | undefined`.

Обратите внимание, что URL, передаваемый в `getLevelResource`, представлен **лямбда-выражением.** Это необходимо, чтобы заставить ресурс **автоматически пересчитывать** URL, когда меняется входящий в него сигнал `levelKey`. Под капотом `httpResource` использует это лямбда-выражение для построения вычисляемого сигнала (Computed Signal), выступающего в роли триггера: каждый раз, когда этот триггер меняется, ресурс загружает URL.

Чтобы предотвратить срабатывание `httpResource`, это выражение должно вернуть `undefined`. Это необходимо, когда приложению нужно отложить загрузку до тех пор, пока не станут доступны конкретные параметры, такие как `levelKey`.

## Больше опций с явным HttpResourceRequest

Чтобы получить больше контроля над отправляемым HTTP-запросом (request), вызывающая сторона может передать `HttpResourceRequest` вместо URL:

```typescript
getLevelResource(levelKey: () => string) {
  return httpResource<Level>(
    () => ({
      url: /levels/${levelKey()}.json,
      method: "GET",
      headers: {
        accept: "application/json",
      },
      params: {
        levelId: levelKey(),
      },
      reportProgress: false,
      body: null,
      transferCache: false,
      withCredentials: false,
    }),
    { defaultValue: initLevel }
  );
}
```

Также этот `HttpResourceRequest` может быть представлен лямбда-выражением, с помощью которого `httpResource` внутренне строит вычисляемый сигнал (Computed Signal).

Важно отметить, что `httpResource` предназначен **только для получения данных,** даже несмотря на то, что мы можем определить метод (HTTP-глагол) помимо `GET` и `body`, передаваемое как полезная нагрузка. Эта опция позволяет работать с веб-API, которые не согласуются с семантикой HTTP-глаголов. По умолчанию переданное `body` преобразуется в JSON. Однако эта опция не предназначена для записи данных обратно на сервер. Эта задача не покрывается `httpResource`. Кроме того, хотя автоматическое использование семантики `switchMap` подходит для получения данных, оно, весьма вероятно, будет неправильным поведением при записи данных обратно на сервер.

Используя опцию `reportProgress`, вызывающая сторона может запросить информацию о ходе текущей операции. Это удобно при скачивании больших файлов. Раздел ниже приводит пример этого.

## Разбор и валидация полученных данных

По умолчанию `httpResource` ожидает JSON, соответствующий предоставленному параметру типа. Поэтому после разбора полученного JSON он просто использует приведение типов (type assertion), чтобы TypeScript предполагал указанный тип.

Однако в этот процесс можно вклиниться, чтобы предоставить собственную логику для валидации полученного сырого значения и преобразования его к типу, заданному параметром типа. Для этого вызывающая сторона может определить функцию `parse` в объекте опций:

```typescript
getLevelResourceAlternative(levelKey: () => string) {
  return httpResource<Level>(() => /levels/${levelKey()}.json, {
    defaultValue: initLevel,
    parse: (raw) => {
      return toLevel(raw);
    },
  });
}
```

`httpResource` преобразует полученный JSON в объект типа `unknown`, который передаётся в `parse`. В нашем примере используется простая написанная вручную функция `toLevel`. Однако `parse` также можно использовать для подключения ресурса к библиотеке, специализирующейся на валидации схем, такой как [Zod](https://zod.dev/).

## Загрузка данных помимо JSON

Хотя `httpResource` по умолчанию ожидает JSON, преобразованный в JavaScript-объект, вызывающая сторона также может запросить другие представления данных:

* `httpResource.text` возвращает простой текст
* `httpResource.blob` возвращает полученные данные в виде Blob
* `httpResource.arrayBuffer` возвращает полученные данные в виде ArrayBuffer

Чтобы это продемонстрировать, пример запрашивает изображение со всеми возможными тайлами в виде Blob. Из этого Blob можно вывести нужные тайлы для выбранного стиля уровня.

Следующее изображение показывает фрагмент этой карты тайлов и демонстрирует, что пример может легко переключаться между различными стилями, используя горизонтальное или вертикальное смещение:

![Фрагмент карты тайлов, используемой в примере](https://www.angulararchitects.io/wp-content/uploads/2025/02/tile-map.png)

Эта карта тайлов была взята [отсюда](https://www.spriters-resource.com/nes/supermariobros/sheet/52571). Для загрузки карты тайлов `TilesMapLoader` делегирует работу `httpResource.blob`:

```typescript
@Injectable({ providedIn: "root" })
export class TilesMapLoader {
  getTilesMapResource(): HttpResourceRef<Blob | undefined> {
    return httpResource.blob(() => {
      url: "/tiles.png",
      reportProgress: true,
    });
  }
}
```

Этот ресурс также запрашивает информацию о прогрессе, которую пример подхватывает для отображения информации о прогрессе слева от выпадающих списков.

## Замечание на полях: HttpClient под капотом позволяет использовать интерсепторы

Под капотом новый `httpResource` в настоящее время использует `HttpClient`. Поэтому нам нужно предоставить HttpClient, что обычно делается с помощью `provideHttpClient` во время начальной загрузки (bootstrapping). Как следствие, `httpResource` также автоматически задействует зарегистрированные `HttpInterceptor`-ы.

Однако использование `HttpClient` — это деталь реализации, которая со временем может быть заменена другой реализацией.

## Собираем всё вместе: реактивный поток

Теперь, когда у нас есть фабрики для всех `httpResource`-ов, мы можем определить наш реактивный граф:

![Реактивный поток ngMario](https://www.angulararchitects.io/wp-content/uploads/2025/06/graph-scaled.png)

Сигналы `levelKey`, `style` и `animation` представляют ввод пользователя. Первые два соответствуют выпадающим спискам наверху. Сигнал `animation` содержит `boolean`, сообщающий, была ли запущена анимация нажатием кнопки `Toggle Animation` (см. скриншот выше).

`tilesResource` в верхней средней части — это классический ресурс, который выводит отдельные тайлы для выбранного стиля. Для этого он, по сути, делегирует работу функции, предоставляемой игровым «движком», рассматриваемым здесь как чёрный ящик.

Рендеринг вызывается эффектом (effect), поскольку мы не можем напрямую нарисовать уровень с помощью привязки данных (data binding). Он рисует или анимирует уровень на холсте (canvas), предоставленном как основанный на сигналах `viewChild`. Этот эффект вызывается всякий раз, когда меняется `level` (предоставляемый `levelResource`), `style`, флаг `animation` или `canvas`.

Сигнал `tilesMapProgress` использует информацию о прогрессе, предоставляемую `tilesMapResource`, чтобы указать, какая часть карты тайлов уже скачана. Для загрузки списка доступных уровней пример использует `levelOverviewResource`, который напрямую не связан с обсуждённым до сих пор реактивным графом.

Следующий листинг показывает реализацию этого реактивного потока в форме членов, используемых `LevelComponent`:

```typescript
export class LevelComponent implements OnDestroy {
  private tilesMapLoader = inject(TilesMapLoader);
  private levelLoader = inject(LevelLoader);

  canvas = viewChild<ElementRef<HTMLCanvasElement>>("canvas");

  levelKey = linkedSignal<string | undefined>(() => this.getFirstLevelKey());
  style = signal<Style>("overworld");
  animation = signal(false);

  tilesMapResource = this.tilesMapLoader.getTilesMapResource();
  levelResource = this.levelLoader.getLevelResource(this.levelKey);
  levelOverviewResource = this.levelLoader.getLevelOverviewResource();

  tilesResource = createTilesResource(this.tilesMapResource, this.style);

  tilesMapProgress = computed(() =>
    calcProgress(this.tilesMapResource.progress())
  );

  constructor() {
    [...]
    effect(() => {
      this.render();
    });
  }

  reload() {
    this.tilesMapResource.reload();
    this.levelResource.reload();
  }

  private getFirstLevelKey(): string | undefined {
    return this.levelOverviewResource.value()?.levels?.[0]?.levelKey;
  }

  [...]
}
```

Использование `linkedSignal` для `levelKey` позволяет нам использовать первый уровень как значение по умолчанию, как только список уровней будет загружен. Вспомогательная функция `getFirstLevelKey` возвращает этот уровень из `levelOverviewResource`.

Эффект рендеринга, по сути, извлекает упомянутые значения и делегирует работу функциям «движка» `animateLevel` или `rederLevel`:

```typescript
private render() {
  const tiles = this.tilesResource.value();
  const level = this.levelResource.value();
  const canvas = this.canvas()?.nativeElement;
  const animation = this.animation();

  if (!tiles || !canvas) {
    return;
  }

  if (animation) {
    animateLevel({
      canvas,
      level,
      tiles,
    });
  } else {
    renderLevel({
      canvas,
      level,
      tiles,
    });
  }
}
```

## Ресурсы и недостающие параметры

`tilesResource`, отображённый на диаграмме выше, по сути делегирует работу асинхронной функции `extractTiles`, которая также предоставляется игровым «движком»:

```typescript
function createTilesResource(
  tilesMapResource: HttpResourceRef<Blob | undefined>,
  style: () => Style
) {
  const tilesMap = tilesMapResource.value();

  // undefined предотвращает срабатывание ресурса
  const params = computed(() =>
    !tilesMap
      ? undefined
      : {
          tilesMap: tilesMap,
          style: style(),
        }
  );

  return resource({
    params,
    loader: (loaderParams) => {
      const { tilesMap, style } = loaderParams.params;
      return extractTiles(tilesMap, style);
    },
  });
}
```

Этот простой ресурс содержит примечательную деталь: пока карта тайлов не загружена, `value` у `tilesMapResource` равно `undefined`. Однако без `tilesMap` мы не можем вызвать `extractTiles` и, следовательно, запустить `tilesResource`. Это учитывается сигналом `params`: в этом случае он также возвращает `undefined`, чтобы **загрузчик (loader) не вызывался.**

## Отображение прогресса

Выше `tilesMapResource` был настроен на предоставление информации о ходе загрузки через свой сигнал `progress`. С помощью вычисляемого сигнала в `LevelComponent` она проецируется в строку:

```typescript
function calcProgress(progress: HttpProgressEvent | undefined): string {
  if (!progress) {
    return "-";
  }

  if (progress.total) {
    const percent = Math.round((progress.loaded / progress.total) * 100);
    return percent + "%";
  }

  const kb = Math.round(progress.loaded / 1024);
  return kb + " KB";
}
```

Если сервер предоставляет общий размер файла, эта функция вычисляет скачанный процент. В противном случае она просто отображает количество скачанных килобайт. Перед началом загрузки информации о прогрессе нет. В этом случае возвращается прочерк.

Чтобы протестировать эту возможность, можно ограничить сетевое соединение браузера в инструментах разработчика и нажать кнопку `reload`, чтобы вызвать метод `reload` ресурса.

## Статус, заголовки, ошибка и не только

На случай, если приложению нужно знать код статуса или полученные заголовки, `httpResource` предоставляет соответствующие сигналы:

```typescript
console.log("status", this.levelOverviewResource.status());
console.log("statusCode", this.levelOverviewResource.statusCode());
console.log("headers", this.levelOverviewResource.headers()?.keys());
```

Помимо этого, `httpResource` предоставляет всё, что мы уже знаем по обычным ресурсам, например сигнал `error`, сообщающий о возможной ошибке, или возможность обновить `value`, который трактуется как локальная рабочая копия. Это также означает, что значение не записывается на сервер автоматически. Эту задачу по-прежнему нужно реализовывать классическим способом.

## Заключение

Новый `httpResource` — это ещё один строительный блок, дополняющий новую историю реактивности в Angular. Он позволяет загружать данные внутри реактивного графа. В настоящее время он использует `HttpClient` как деталь реализации, которая со временем может быть заменена другим решением.

Хотя `HTTP resource` также позволяет получать данные, используя HTTP-глаголы помимо GET, например с помощью вызовов POST, он не предназначен для записи данных обратно на сервер. Эту задачу по-прежнему нужно выполнять традиционным способом.
