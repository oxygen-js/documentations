# Реализация Angular Signals

Этот каталог содержит код реактивного примитива Angular — реализацию концепции «сигнала» (signal). Сигнал — это значение, которое является «реактивным», то есть способно уведомлять заинтересованных потребителей о своём изменении. Существует множество различных реализаций этой концепции с разными подходами к тому, как подписываться на такие уведомления и распространять их, как устроены очистка/отписка, как отслеживаются зависимости и т. д. Этот документ описывает алгоритм, лежащий в основе нашей конкретной реализации паттерна «сигнал».

## Концептуальная поверхность

Angular Signals — это функции без аргументов (`() => T`). При вызове они возвращают текущее значение сигнала. Вызов сигнала не запускает побочных эффектов, хотя может лениво пересчитывать промежуточные значения (ленивая мемоизация).

Отдельные контексты (например, выражения в шаблонах) могут быть _реактивными_. В таких контекстах вызов сигнала возвращает значение, но также регистрирует сигнал как зависимость данного контекста. Владелец контекста будет уведомлён, если любая из его сигнальных зависимостей выдаст новое значение (обычно это приводит к повторному выполнению этих выражений для получения новых значений).

Этот механизм контекста и функции-геттера позволяет отслеживать сигнальные зависимости контекста _автоматически_ и _неявно_. Пользователям не нужно объявлять массивы зависимостей, и набор зависимостей конкретного контекста не обязан оставаться неизменным от выполнения к выполнению.

### Устанавливаемые сигналы: `signal()`

Функция `signal()` создаёт особый тип сигнала, известный как `SettableSignal`. Помимо того что это функция-геттер, `SettableSignal` имеет дополнительный API для изменения значения сигнала (вместе с уведомлением всех зависимых об изменении). Сюда входят операция `.set` для замены значения сигнала, `.update` для вычисления нового значения и `.mutate` для выполнения внутренней мутации текущего значения. Они доступны как функции на самом геттере сигнала.

```typescript
const counter = signal(0);

counter.set(2);
counter.update(count => count + 1);
```

Значение сигнала также можно обновить на месте с помощью специального метода `.mutate`:

```typescript
const todoList = signal<Todo[]>([]);

todoList.mutate(list => {
    list.push({title: 'One more task', completed: false});
});
```

#### Равенство

При создании сигнала можно опционально указать функцию-компаратор равенства. Компаратор используется, чтобы решить, является ли вновь переданное значение таким же или отличным по сравнению с текущим значением сигнала.

Если функция равенства определяет, что 2 значения равны, она:
* блокирует обновление значения сигнала;
* пропускает распространение изменения.

### Декларативные производные значения: `computed()`

`computed()` создаёт мемоизирующий сигнал, который вычисляет своё значение из значений некоторого количества входных сигналов.

```typescript
const counter = signal(0);

// Автоматически обновляется при изменении `counter`:
const isEven = computed(() => counter() % 2 === 0);
```

Поскольку функция вычисления, используемая для создания `computed`, выполняется в реактивном контексте, любые сигналы, прочитанные этой функцией, будут отслеживаться как зависимости, а значение вычисляемого сигнала будет пересчитываться при изменении любой из этих зависимостей.

Аналогично сигналам, для `computed` можно (опционально) указать функцию-компаратор равенства.

### Побочные эффекты: `effect()`

`effect()` планирует и запускает функцию с побочным эффектом внутри реактивного контекста. Сигнальные зависимости этой функции захватываются, и побочный эффект повторно выполняется при каждом изменении любой из его зависимостей.

```typescript
const counter = signal(0);
effect(() => console.log('The counter is:', counter()));
// The counter is: 0

counter.set(1);
// The counter is: 1
```

Эффекты не выполняются синхронно с вызовом set (см. раздел о выполнении без сбоев ниже), а планируются и разрешаются фреймворком. Точное время выполнения эффектов не определено.

## Producer и Consumer

Внутренне реализация сигналов определена в терминах двух абстракций — `Producer` и `Consumer` — это интерфейсы, реализуемые различными частями системы реактивности. `Producer` представляет значения, способные доставлять уведомления об изменениях, такие как различные разновидности `Signal`. `Consumer` представляет реактивный контекст, который может зависеть от некоторого количества `Producer`. Иными словами, `Producer` производят реактивность, а `Consumer` её потребляют.

Реализации `Producer` и `Consumer` предоставляют экземпляры структур данных, используемых библиотекой сигналов, и взаимодействуют с библиотекой через вызовы вспомогательных функций.

Некоторые сущности являются одновременно и `Producer`, _и_ `Consumer`. Например, производные выражения `computed` потребляют другие сигналы, чтобы производить новые реактивные значения.

### Граф зависимостей

И `Producer`, и `Consumer` отслеживают рёбра зависимостей (`Edge`) друг к другу. `Producer` знают, какие `Consumer` зависят от их значения, а `Consumer` знают обо всех `Producer`, от которых они зависят. Эти ссылки всегда двунаправленные.

Важной особенностью дизайна Angular Signals является то, что рёбра зависимостей (`Edge`) отслеживаются с помощью слабых ссылок (`WeakRef`). В любой момент может оказаться, что `Consumer` вышел из области видимости и был собран сборщиком мусора, даже если на него всё ещё ссылается `Producer` (или наоборот). Это устраняет необходимость в явных операциях очистки, которые удаляли бы эти рёбра зависимостей для сигналов, выходящих «из области видимости». В результате управление жизненным циклом сигналов значительно упрощается, и нет шанса на утечки памяти из-за отслеживания зависимостей.

Чтобы упростить отслеживание рёбер (`Edge`) через `WeakRef`, и `Producer`, и `Consumer` имеют числовые идентификаторы, генерируемые при их создании. Эти идентификаторы используются как ключи `Map` вместо отслеживаемых объектов `Producer` или `Consumer`, которые вместо этого хранятся в `Edge` как `WeakRef`.

В различные моменты во время чтения или записи значений сигналов эти `WeakRef` разыменовываются. Если ссылка оказывается `undefined` (то есть другая сторона ребра зависимости была освобождена сборщиком мусора), то ребро зависимости (`Edge`) может быть очищено.

## Свойство «Glitch Free» (без сбоев)

Рассмотрим следующую конфигурацию:

```typescript
const counter = signal(0);
const evenOrOdd = computed(() => counter() % 2 === 0 ? 'even' : 'odd');
effect(() => console.log(counter() + ' is ' + evenOrOdd()));

counter.set(1);
```

Когда эффект создаётся впервые, он выведет «0 is even», как и ожидается, и зафиксирует, что и `counter`, и `evenOrOdd` являются зависимостями логирующего эффекта.

Когда `counter` устанавливается в `1`, это инвалидирует и `evenOrOdd`, и логирующий эффект. Однако если бы `counter.set()` перебирал зависимости `counter` и сначала запускал логирующий эффект, прежде чем уведомить `evenOrOdd` об изменении, мы могли бы увидеть несогласованное сообщение в лог «1 is even». В конце концов `evenOrOdd` был бы уведомлён, что снова запустило бы логирующий эффект, выведя корректное сообщение «1 is odd».

В этой ситуации наблюдение логирующим эффектом несогласованного состояния «1 is even» называется _сбоем_ (glitch). Важной целью проектирования реактивной системы является предотвращение наблюдения таких промежуточных состояний и обеспечение _выполнения без сбоев_ (glitch-free execution).

### Алгоритм Push/Pull

Angular Signals гарантирует выполнение без сбоев, разделяя обновления графа `Producer`/`Consumer` на две фазы. Первая фаза выполняется немедленно (eagerly) при изменении значения `Producer`. Это уведомление об изменении распространяется по графу, оповещая `Consumer`, которые зависят от данного `Producer`, о потенциальном обновлении. Некоторые из этих `Consumer` могут быть производными значениями и, следовательно, тоже `Producer`, которые инвалидируют свои кэшированные значения, а затем продолжают распространение уведомления об изменении к собственным `Consumer`, и так далее. Другие `Consumer` могут быть эффектами, которые планируют себя для повторного выполнения.

Что критически важно, во время этой первой фазы не запускаются никакие побочные эффекты и не выполняется пересчёт промежуточных или производных значений — только инвалидация кэшированных значений. Это позволяет уведомлению об изменении достичь всех затронутых узлов графа без возможности наблюдения промежуточных или сбойных состояний.

После того как это распространение изменения завершилось (синхронно), может начаться вторая фаза. В этой второй фазе значения сигналов могут быть прочитаны приложением или фреймворком, что вызывает пересчёт всех необходимых производных значений, которые ранее были инвалидированы.

Мы называем это алгоритмом «push/pull»: «загрязнённость» (dirtiness) немедленно _проталкивается_ (push) по графу при изменении исходного сигнала, но пересчёт выполняется лениво, только когда значения _извлекаются_ (pull) путём чтения их сигналов.

## Динамическое отслеживание зависимостей

Когда выполняется операция реактивного контекста (например, функция побочного эффекта `effect`), сигналы, которые она читает, отслеживаются как зависимости. Однако это может быть не один и тот же набор сигналов от выполнения к выполнению. Например, этот вычисляемый сигнал:

```typescript
const dynamic = computed(() => useA() ? dataA() : dataB());
```

читает либо `dataA`, либо `dataB` в зависимости от значения сигнала `useA`. В любой данный момент он будет иметь набор зависимостей либо `[useA, dataA]`, либо `[useA, dataB]`, и он никогда не может зависеть от `dataA` и `dataB` одновременно.

Потенциальные зависимости реактивного контекста неограниченны. Сигналы могут храниться в переменных или других структурах данных и время от времени заменяться другими сигналами. Таким образом, реализация сигналов должна справляться с потенциальными изменениями в наборе зависимостей `Consumer` при каждом выполнении.

Наивный подход состоял бы в том, чтобы просто удалять все старые рёбра зависимостей перед повторным выполнением реактивной операции или помечать их все как устаревшие заранее и удалять те, которые не были прочитаны. Это концептуально просто, но вычислительно затратно, особенно для реактивных контекстов с практически неизменным набором зависимостей.

### Версионирование рёбер зависимостей

Вместо этого наша реализация использует более лёгкий подход к инвалидации зависимостей, который опирается на монотонный счётчик версий, поддерживаемый `Consumer` и называемый `trackingVersion`. Перед выполнением реактивной операции `Consumer` его `trackingVersion` инкрементируется. Когда сигнал читается, `trackingVersion` `Consumer` сохраняется в ребре зависимости (`Edge`), где он доступен для `Producer`.

Когда у `Producer` появляется обновлённое значение, он перебирает свои исходящие рёбра ко всем заинтересованным `Consumer`, чтобы уведомить их об изменении. В этот момент `Producer` может проверить, является ли зависимость актуальной или устаревшей, сравнив текущий `trackingVersion` `Consumer` с тем, что сохранён в ребре зависимости (`Edge`). Несовпадение означает, что зависимости `Consumer` изменились и больше не включают данный `Producer`, поэтому этот `Consumer` не уведомляется, а устаревшее ребро вместо этого удаляется.

## Семантика равенства

`Producer` могут лениво производить своё значение (например, `computed`, который пересчитывает значение, только когда его извлекают). Однако `Producer` также может применять проверку равенства к производимым значениям и определить, что вновь вычисленное значение семантически «равно» предыдущему. В этом случае `Consumer`, зависящие от этого значения, не должны выполняться повторно. Например, следующий эффект:

```typescript
const counter = signal(0);
const isEven = computed(() => counter() % 2 === 0);
effect(() => console.log(isEven() ? 'even!' : 'odd!'));
```

должен запуститься, если `counter` обновляется до `1`, так как значение `isEven` переключается с `true` на `false`. Но если затем `counter` устанавливается в `3`, `isEven` пересчитает то же самое значение: `false`. Следовательно, логирующий эффект не должен запускаться.

Это сложное для гарантирования свойство в нашей реализации, потому что значения не пересчитываются во время фазы push распространения изменения. `isEven` инвалидируется при изменении `counter`, что приводит к инвалидации и планированию логирующего `effect`. Наивно, `isEven` не пересчитывался бы до тех пор, пока логирующий эффект фактически не запустится и не попытается прочитать его значение, что слишком поздно, чтобы заметить, что он вообще не должен был запускаться.

### Версионирование значений

Чтобы решить эту проблему, наша реализация использует технику, аналогичную отслеживанию устаревания зависимостей. `Producer` отслеживают монотонно возрастающий `valueVersion`, представляющий семантическую идентичность их значения. `valueVersion` инкрементируется, когда `Producer` производит семантически новое значение. Текущий `valueVersion` сохраняется в структуре ребра зависимости (`Edge`), когда `Consumer` читает из `Producer`.

Перед тем как `Consumer` запускают свои реактивные операции (например, функцию побочного эффекта для `effect` или пересчёт для `computed`), они опрашивают свои зависимости и запрашивают обновление `valueVersion`, если необходимо. Для `computed` это вызовет пересчёт значения и последующую проверку равенства, если значение устарело (что делает этот опрос рекурсивным процессом, так как `computed` сам является `Consumer`, который будет опрашивать собственные `Producer`). Если этот пересчёт производит семантически изменённое значение, `valueVersion` инкрементируется.

Затем `Consumer` может сравнить `valueVersion` нового значения с тем, что закэширован в его ребре зависимости (`Edge`), чтобы определить, действительно ли изменилась эта конкретная зависимость. Делая это для всех `Producer`, `Consumer` может определить, что если все `valueVersion` совпадают, то _фактического_ изменения ни одной зависимости не произошло, и он может пропустить реакцию на это изменение (например, пропустить запуск функции побочного эффекта).

## Примитив `Watch`

`Watch` — это примитив, используемый для построения различных типов эффектов. `Watch` — это `Consumer`, которые запускают функции с побочными эффектами в своём реактивном контексте, но где планирование побочного эффекта делегируется реализатору. `Watch` будет вызывать эту операцию планирования, когда получит уведомление о том, что он устарел.
