## Взгляд со стороны потребителя

В нашем примере Angular-клиент инициализирует чат с помощью фабрики _chatConnection_. Она предоставляет структуру, представляющую отдельные сообщения чата как потоковый ресурс, а также предлагает дополнительные сигналы и метод _send_:

```typescript
@Component([…])
export class ChatResourceComponent {
  ResourceStatus = ResourceStatus;

  userName = signal('');
  chat = chatConnection('ws://localhost:6502', this.userName);
  messages = computed(() => this.chat.resource.value() ?? []);

  userNameInField = linkedSignal(() => this.chat.acceptedUserName());
  currentMessage = signal<string>('');

  send() {
    this.chat.send(this.currentMessage());
    this.currentMessage.set('');
  }

  join() {
    this.userName.set(this.userNameInField());
  }
}
```

Сигнал _userName_, передаваемый в _chatConnection_, запускает ресурс и устанавливает соединение с сервером чата. Сигналы _userNameInField_ и _currentMessages_ привязаны к полям ввода. Чтобы установить соединение, функция _join_ записывает значение _userNameInField_ в сигнал _userName_.

Сервер может скорректировать запрошенное имя пользователя, чтобы гарантировать его уникальность. Поэтому _userNameInField_ реализован через _linkedSignal_. Этот связанный сигнал (Linked Signal) перезаписывает локальное значение исправленным значением от сервера, которое содержится в сигнале _acceptedUserName_.

Когда пользователь хочет отправить сообщение, приложение передаёт _currentMessage_ в метод _send_. Затем оно сбрасывает _currentMessage_, чтобы пользователь мог ввести следующее сообщение.

## Свойство отсутствия «глитчей» (glitch-free) как вызов для потоковых ресурсов

Чат отличается от таймера из предыдущей статьи в одном существенном моменте: если таймер интересовало только самое свежее сообщение из потока, то чату важны все полученные на данный момент сообщения — или хотя бы все сообщения за определённое временное окно. Иначе говоря, таймер — это объект, который меняется со временем. Чат же — это набор событий с сообщениями, которые приложение должно учитывать в их совокупности.

Привычный нам взгляд на таймер хорошо ложится на сигналы, тем более что сигналы всегда отражают самое последнее значение. Свойство отсутствия «глитчей» (glitch-free), при котором ненужные промежуточные значения игнорируются, тоже здесь уместно: если бы ресурс изменил таймер с 0 на 1, с 1 на 2 и с 2 на 3 в рамках одной задачи (task), потребитель ресурса увидел бы только значение 3.

Однако в событийном сценарии — таком, как чат, — подобное поведение было бы фатальным. Тут отдельные сообщения просто потерялись бы. Проверить это легко: пусть сервер чата отправит одно и то же сообщение несколько раз подряд одному и тому же клиенту. В реализации, которая учитывает только последнее полученное сообщение, большая часть этих повторений будет потеряна.

Это наблюдение показывает, что «в отличие от Observable в RxJS, сигналы (Signals) плохо подходят для представления событий или сообщений!» Перед нами встаёт вызов при использовании потоковых ресурсов, но, к счастью, его можно преодолеть, просто сменив угол зрения: если позволить ресурсу представлять не только текущее значение, но и все полученные на данный момент сообщения — или хотя бы все сообщения в интересующем нас временном окне, — то эту совокупность сообщений можно снова рассматривать как значение, которое меняется со временем.

Иными словами, нам нужно перенести сбор и управление отдельными сообщениями внутрь ресурса. Такой взгляд хорошо согласуется и с наблюдением из предыдущей статьи: в мире сигналов используются крупнозернистые строительные блоки, заточенные под конкретный сценарий использования.

## Сообщения и протокол

В примере определено несколько типов для отдельных сообщений, которыми клиент обменивается с сервером чата:

```typescript
export type ChatRequest =
  | {
      type: 'username';
      id: number;
      name: string;
    }
  | {
      type: 'message';
      id: number;
      text: string;
    };

export type ChatResponse =
  | {
      type: 'id';
      id: number;
    }
  | {
      type: 'username';
      id: number;
      name: string;
    }
  | {
      type: 'message';
      id: number;
      name: string;
      text: string;
    };
```

Протокол между клиентом и сервером выглядит так:

1.  Клиент устанавливает соединение по веб-сокету.
2.  Сервер отвечает _ChatResponse_ типа _id_, через который клиент получает уникальный идентификатор сессии.
3.  Клиент отправляет _ChatRequest_ типа _username_, чтобы сообщить имя пользователя.
4.  Сервер подтверждает это имя с помощью _ChatResponse_ типа _username_. Если желаемое имя уже занято, через это сообщение клиент получает исправленное имя, которое сервер формирует, добавляя число к предпочтительному варианту.
5.  Клиент отправляет _ChatRequest_ со своими сообщениями (тип _message_). Сервер раздаёт это сообщение всем клиентам в виде _ChatResponse_ типа _message_.

## Представление чата

Соединение с чатом в примере представлено типом _ChatConnection_:

```typescript
export type SendFn = (message: string) => void;

export type ChatConnection = {
  resource: ResourceRef<ChatResponse[] | undefined>;
  connected: () => boolean;
  acceptedUserName: () => string;
  send: SendFn;
};
```

Сердце _ChatConnection_ — это ресурс, который представляет все полученные на данный момент сообщения в виде массива. Он также предоставляет дополнительную информацию о состоянии в виде сигналов. Сюда входят статус соединения (_connected_) и имя пользователя (_acceptedUserName_), при необходимости исправленное сервером.

Метод _send_ позволяет чату отправлять сообщения на сервер. Реализация преобразует переданную строку в _ChatRequest_ типа _message_.

## Фабрика для чата

Фабрика для чата устроена похоже на ту, что мы рассматривали для таймера. Однако сперва она подготавливает несколько дополнительных переменных:

```typescript
export function chatConnection(
  websocketUrl: string,
  userName: () => string
): ChatConnection {

  let connection: WebSocket;

  const connected = signal(false);
  const id = signal(0);
  const acceptedUserName = signal('');

  const params = computed(() => ({
    userName: userName(),
  }));

  const chatResource = resource({
    params,
    stream: async (loaderParams) => {
      // инициализируем соединение по веб-сокету
      // обрабатываем и собираем сообщения
      […]
    },
  });

  const send: SendFn = (message: string) => {
    const request: ChatRequest = {
      type: 'message',
      id: id(),
      text: message,
    };
    connection.send(JSON.stringify(request));
  };

  return {
    connected,
    resource: chatResource,
    acceptedUserName,
    send,
  };
}
```

Среди этих переменных — _connection_, которая отражает соединение по веб-сокету, установленное потоковым загрузчиком (Streaming Loader), уже упомянутые сигналы _connected_ и _acceptedUserName_, а также сигнал _id_, представляющий текущую сессию пользователя. В итоге фабрика возвращает часть этих сигналов, созданный потоковый ресурс и метод _send_ в виде _ChatConnection_. Сигнал _id_ она наружу не отдаёт, так как это деталь реализации.

Потоковый загрузчик ресурса сначала создаёт массив _messages_, в который собирает полученные сообщения чата:

```typescript
const chatResource = resource({
  request,
  stream: async (loaderParams) => {
    const userName = loaderParams.params.userName;

    let messages: ChatResponse[] = [];

    // 1. Создаём сигнал, представляющий поток
    const resultSignal = signal<ResourceStreamItem<ChatResponse>>({
      value: messages,
    });

    if (!userName) {
      return resultSignal;
    }

    // 2. Настраиваем асинхронную логику, обновляющую сигнал
    connection = new WebSocket(websocketUrl, 'json');

    connection.addEventListener('open', (event) => {
      console.log('[open]');
      connected.set(true);
    });

    connection.addEventListener('message', (event) => {
      const value = JSON.parse(event.data) as ChatResponse;
      console.log('[message]', value);

      if (value.type === 'id') {
        id.set(value.id);
        sendUserName(value.id, userName, connection);
      }

      if (value.type === 'username' && value.id == id()) {
        acceptedUserName.set(value.name);
      }

      if (value.type === 'message' || value.type === 'username') {
        messages = [...messages, value];
        resultSignal.set({ value: messages });
      }
    });

    connection.addEventListener('error', (event) => {
      const error = new Error(Error with websocket connection);
      console.log('[event]', event);
      resultSignal.set({ error });
    });

    // 3. Настраиваем обработчик очистки, запускаемый по AbortSignal
    params.abortSignal.addEventListener('abort', () => {
      console.log('clean up!');
      connection.close();
      connected.set(false);
      id.set(0);
      acceptedUserName.set('');
    });

    // 4. Возвращаем сигнал
    return resultSignal;
  },
});

function sendUserName(id: number, userName: string, connection: WebSocket) {
  const message: ChatRequest = {
    type: 'username',
    id: id,
    name: userName,
  };
  connection.send(JSON.stringify(message));
}
```

Остальная часть реализации повторяет четыре пункта, которые мы разобрали в предыдущей статье. Подготовленный сигнал отражает поток и содержит управляемый массив с полученными сообщениями.

Клиент получает информацию от сервера через настроенные обработчики событий. Событие _message_ занимается описанным выше протоколом. Оно добавляет все полученные сообщения в массив messages. Делать это нужно иммутабельно, то есть через поверхностную копию (shallow copy), чтобы сигнал заметил изменение.

Событие _open_ устанавливает _connected_ в _true_, а событие _error_ записывает текущую ошибку в поток. Событие _abort_ от AbortSignal закрывает соединение по веб-сокету и сбрасывает управляемые свойства. В конце потоковый загрузчик, как обычно, возвращает сигнал, отражающий поток.

## Заключение

Сигналы в Angular всегда представляют только текущее значение. Промежуточные значения для изменений, которые следуют друг за другом в рамках одной задачи (task), игнорируются (свойство glitch-free). Это хорошо подходит для меняющихся состояний — например, счётчика, — но может приводить к потере данных в потоках событий, таких как чат, который должен отображать все полученные сообщения.

Решение — собирать полученные сообщения (в интересующем нас временном окне) внутри ресурса и публиковать получившийся массив через ресурс. Таким образом, этот массив соответствует значению, которое меняется со временем.

И хотя это интересный паттерн, никто, разумеется, не мешает нам использовать другие подходы — например, RxJS, который напрямую представляет поток событий.
