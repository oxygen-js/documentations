## Взгляд со стороны потребителя

В нашем примере Angular-клиент инициализирует чат с помощью фабрики _chatConnection_. Она предоставляет структуру, представляющую отдельные сообщения чата как потоковый ресурс, а также дополнительные сигналы и метод _send_:

```typescript
@Component([…])
export class ChatResourceComponent {
  ResourceStatus = ResourceStatus;

  userName = signal('');
  chat = chatConnection('ws://localhost:6502', this.userName);
  messages = computed(() => this.chat.resource.value() ?? []);

  userNameInField = linkedSignal(() => this.chat.acceptedUserName());
  currentMessage = signal<string>('');

  send() {
    this.chat.send(this.currentMessage());
    this.currentMessage.set('');
  }

  join() {
    this.userName.set(this.userNameInField());
  }
}
```

Сигнал _userName_, передаваемый в _chatConnection_, триггерит ресурс и устанавливает соединение с сервером чата (изменение reactive-параметра приводит к перезапуску загрузчика ресурса). Сигналы _userNameInField_ и _currentMessages_ привязаны к полям ввода. Чтобы установить соединение, функция _join_ записывает значение _userNameInField_ в сигнал _userName_.

Сервер может скорректировать запрошенное имя пользователя, чтобы гарантировать уникальность. Поэтому _userNameInField_ реализован через _linkedSignal_ (записываемый сигнал с вычисляемым значением по умолчанию). Этот связанный сигнал (Linked Signal) перезаписывает локальное значение исправленным значением от сервера, которое содержится в сигнале _acceptedUserName_.

Когда пользователь хочет отправить сообщение, приложение передаёт _currentMessage_ в метод _send_. Затем оно сбрасывает _currentMessage_, чтобы пользователь мог ввести следующее сообщение.

## Свойство отсутствия глитчей (glitch-free) как вызов для потоковых ресурсов

Пример чата отличается от таймера, рассмотренного в предыдущей статье, в одном существенном аспекте: если таймер интересовало только самое последнее сообщение из потока (stream), то чату важны все полученные на данный момент сообщения — или, по крайней мере, все сообщения из определённого временного окна. Иначе говоря, таймер представляет объект, изменяющийся во времени. Чат же, напротив, представляет несколько событий с сообщениями, которые приложение должно учитывать в их совокупности.

Принятый здесь взгляд на таймер хорошо согласуется с сигналами, тем более что сигналы всегда отражают самое последнее значение. Свойство отсутствия глитчей (glitch-free), при котором ненужные промежуточные значения игнорируются, тоже здесь уместно: если бы ресурс изменил таймер с 0 на 1, с 1 на 2 и с 2 на 3 в рамках одной задачи (task — единая синхронная единица выполнения в event loop), потребитель ресурса увидел бы только значение 3.

Однако при событийном использовании — таком, как чат, — подобное поведение было бы фатальным. Здесь отдельные сообщения оказались бы потеряны. Это поведение легко протестировать, заставив сервер чата отправить одно и то же сообщение несколько раз подряд одному и тому же клиенту. Большинство этих повторений было бы потеряно в реализации, которая учитывает только последнее полученное сообщение.

Это рассуждение показывает, что «в отличие от Observable в RxJS, сигналы (Signals) плохо подходят для представления событий или сообщений!» Это ставит перед нами вызов при использовании потоковых ресурсов, который, к счастью, преодолевается простой сменой угла зрения: если позволить ресурсу представлять не только текущее значение, но и все полученные на данный момент сообщения — или, по крайней мере, все сообщения в интересующем нас временном окне, — то эту совокупность сообщений можно снова рассматривать как значение, изменяющееся во времени.

Иными словами, сбор отдельных сообщений и управление ими нужно перенести внутрь ресурса. Этот взгляд хорошо согласуется и с наблюдением из предыдущей статьи: в мире сигналов используются специфичные для сценария и, следовательно, крупнозернистые (coarse-grained) строительные блоки.

## Сообщения и протокол

В примере определено несколько типов для отдельных сообщений, которыми клиент обменивается с сервером чата:

```typescript
export type ChatRequest =
  | {
      type: 'username';
      id: number;
      name: string;
    }
  | {
      type: 'message';
      id: number;
      text: string;
    };

export type ChatResponse =
  | {
      type: 'id';
      id: number;
    }
  | {
      type: 'username';
      id: number;
      name: string;
    }
  | {
      type: 'message';
      id: number;
      name: string;
      text: string;
    };
```

Протокол между клиентом и сервером следующий:

1.  Клиент устанавливает соединение по WebSocket.
2.  Сервер отвечает _ChatResponse_ типа _id_, через который клиент получает уникальный идентификатор сессии.
3.  Клиент отправляет _ChatRequest_ типа _username_, чтобы передать имя пользователя.
4.  Сервер подтверждает это имя с помощью _ChatResponse_ типа _username_. Если желаемое имя уже занято, через это сообщение клиент получает скорректированное имя, которое сервер формирует, добавляя число к предпочтительному варианту.
5.  Клиент отправляет _ChatRequest_ со своими сообщениями (тип _message_). Сервер распространяет это сообщение всем клиентам как _ChatResponse_ типа _message_.

## Представление чата

Соединение с чатом представлено в примере типом _ChatConnection_:

```typescript
export type SendFn = (message: string) => void;

export type ChatConnection = {
  resource: ResourceRef<ChatResponse[] | undefined>;
  connected: () => boolean;
  acceptedUserName: () => string;
  send: SendFn;
};
```

Ядро _ChatConnection_ — это ресурс, представляющий все полученные на данный момент сообщения в виде массива. Он также предоставляет дополнительную информацию о состоянии в виде сигналов. Сюда входят статус соединения (_connected_) и имя пользователя (_acceptedUserName_), при необходимости скорректированное сервером.

Метод _send_ позволяет чату отправлять сообщения на сервер. Реализация отображает (maps) переданную строку в _ChatRequest_ типа _message_.

## Фабрика для чата

Фабрика для чата структурно аналогична той, что рассматривалась ранее для таймера. Однако сначала она подготавливает несколько дополнительных переменных:

```typescript
export function chatConnection(
  websocketUrl: string,
  userName: () => string
): ChatConnection {

  let connection: WebSocket;

  const connected = signal(false);
  const id = signal(0);
  const acceptedUserName = signal('');

  const params = computed(() => ({
    userName: userName(),
  }));

  const chatResource = resource({
    params,
    stream: async (loaderParams) => {
      // инициализация соединения по WebSocket
      // обработка и накопление сообщений
      […]
    },
  });

  const send: SendFn = (message: string) => {
    const request: ChatRequest = {
      type: 'message',
      id: id(),
      text: message,
    };
    connection.send(JSON.stringify(request));
  };

  return {
    connected,
    resource: chatResource,
    acceptedUserName,
    send,
  };
}
```

Среди этих переменных — _connection_, отражающая соединение по WebSocket, установленное потоковым загрузчиком (Streaming Loader); упомянутые ранее сигналы _connected_ и _acceptedUserName_; а также сигнал _id_, представляющий текущую сессию пользователя. В итоге фабрика возвращает часть этих сигналов, созданный потоковый ресурс и метод _send_ как _ChatConnection_. Сигнал _id_ наружу не публикуется, поскольку это деталь реализации.

Потоковый загрузчик ресурса сначала создаёт массив _messages_, в который накапливает полученные сообщения чата:

```typescript
const chatResource = resource({
  request,
  stream: async (loaderParams) => {
    const userName = loaderParams.params.userName;

    let messages: ChatResponse[] = [];

    // 1. Создаём сигнал, представляющий поток (stream)
    const resultSignal = signal<ResourceStreamItem<ChatResponse>>({
      value: messages,
    });

    if (!userName) {
      return resultSignal;
    }

    // 2. Настраиваем асинхронную логику, обновляющую сигнал
    connection = new WebSocket(websocketUrl, 'json');

    connection.addEventListener('open', (event) => {
      console.log('[open]');
      connected.set(true);
    });

    connection.addEventListener('message', (event) => {
      const value = JSON.parse(event.data) as ChatResponse;
      console.log('[message]', value);

      if (value.type === 'id') {
        id.set(value.id);
        sendUserName(value.id, userName, connection);
      }

      if (value.type === 'username' && value.id == id()) {
        acceptedUserName.set(value.name);
      }

      if (value.type === 'message' || value.type === 'username') {
        messages = [...messages, value];
        resultSignal.set({ value: messages });
      }
    });

    connection.addEventListener('error', (event) => {
      const error = new Error(Error with websocket connection);
      console.log('[event]', event);
      resultSignal.set({ error });
    });

    // 3. Настраиваем обработчик очистки (clean-up), запускаемый по AbortSignal
    params.abortSignal.addEventListener('abort', () => {
      console.log('clean up!');
      connection.close();
      connected.set(false);
      id.set(0);
      acceptedUserName.set('');
    });

    // 4. Возвращаем сигнал
    return resultSignal;
  },
});

function sendUserName(id: number, userName: string, connection: WebSocket) {
  const message: ChatRequest = {
    type: 'username',
    id: id,
    name: userName,
  };
  connection.send(JSON.stringify(message));
}
```

Остальная часть реализации следует четырём пунктам, рассмотренным в предыдущей статье. Подготовленный сигнал отражает поток (stream) и содержит управляемый массив с полученными сообщениями.

Клиент получает информацию от сервера через настроенные обработчики событий (event handlers). Событие _message_ реализует описанный выше протокол. Оно добавляет все полученные сообщения в массив messages. Делать это нужно иммутабельно, то есть через поверхностную копию (shallow copy), чтобы сигнал зафиксировал изменение (сравнение по ссылке: in-place мутация массива не была бы замечена системой реактивности).

Событие _open_ устанавливает _connected_ в _true_, а событие _error_ записывает текущую ошибку в поток. Событие _abort_ объекта _AbortSignal_ закрывает соединение по WebSocket и сбрасывает управляемые свойства. В конце потоковый загрузчик, как обычно, возвращает сигнал, отражающий поток.

## Заключение

Сигналы в Angular всегда представляют только текущее значение. Промежуточные значения для изменений, следующих друг за другом в рамках одной задачи (task), игнорируются (свойство Glitch Free). Это хорошо подходит для изменяющихся состояний, таких как счётчик, но может приводить к потере данных в потоках событий — например, в чате, который должен отображать все полученные сообщения.

Решение состоит в том, чтобы накапливать полученные сообщения (во временном окне, которое нас интересует) внутри ресурса и публиковать получившийся массив через ресурс. Таким образом, этот массив соответствует значению, изменяющемуся во времени.

Хотя это интересный паттерн, никто, разумеется, не мешает нам использовать иные подходы — например, RxJS, который напрямую представляет поток событий.
