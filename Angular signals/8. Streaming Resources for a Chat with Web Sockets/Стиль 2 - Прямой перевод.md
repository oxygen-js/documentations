## Взгляд со стороны потребителя

В нашем примере Angular-клиент инициализирует чат с помощью фабрики _chatConnection_. Она предоставляет структуру, представляющую отдельные сообщения чата как потоковый ресурс, и предлагает дополнительные сигналы и метод _send_:

```typescript
@Component([…])
export class ChatResourceComponent {
  ResourceStatus = ResourceStatus;

  userName = signal('');
  chat = chatConnection('ws://localhost:6502', this.userName);
  messages = computed(() => this.chat.resource.value() ?? []);

  userNameInField = linkedSignal(() => this.chat.acceptedUserName());
  currentMessage = signal<string>('');

  send() {
    this.chat.send(this.currentMessage());
    this.currentMessage.set('');
  }

  join() {
    this.userName.set(this.userNameInField());
  }
}
```

Сигнал _userName_, передаваемый в _chatConnection_, запускает ресурс и устанавливает соединение с сервером чата. Сигналы _userNameInField_ и _currentMessages_ привязаны к полям ввода. Чтобы установить соединение, функция _join_ записывает значение _userNameInField_ в сигнал _userName_.

Сервер может скорректировать запрошенное имя пользователя, чтобы обеспечить уникальность. Поэтому _userNameInField_ реализован с помощью _linkedSignal_. Этот связанный сигнал (Linked Signal) перезаписывает локальное значение исправленным значением от сервера, которое содержится в сигнале _acceptedUserName_.

Когда пользователь хочет отправить сообщение, приложение передаёт _currentMessage_ в метод _send_. Затем оно сбрасывает _currentMessage_, чтобы пользователь мог ввести следующее сообщение.

## Свойство отсутствия глитчей (Glitch-Free) как вызов для потоковых ресурсов

Пример чата отличается от таймера, обсуждавшегося в предыдущей статье, в одном существенном аспекте: в то время как таймер интересовало только самое последнее сообщение из потока, чат интересуют все полученные на данный момент сообщения или, по крайней мере, все сообщения из определённого временного окна. Иными словами, таймер представляет объект, который меняется со временем. В противоположность этому, чат представляет несколько событий с сообщениями, которые приложение должно учитывать в их совокупности.

То, как мы здесь рассматриваем таймер, хорошо согласуется с сигналами, тем более что сигналы всегда отражают самое последнее значение. Свойство отсутствия глитчей (glitch-free), которое игнорирует ненужные промежуточные значения, также подходит здесь: если бы ресурс изменил таймер с 0 на 1, с 1 на 2 и с 2 на 3 в рамках одной задачи (task), потребитель ресурса увидел бы только значение 3.

Однако такое поведение было бы фатальным при событийном использовании, таком как чат. Здесь отдельные сообщения были бы потеряны. Это поведение легко протестировать, заставив сервер чата отправить одно и то же сообщение несколько раз подряд одному и тому же клиенту. Большинство этих повторений были бы потеряны в реализации, которая учитывает только последнее полученное сообщение.

Это рассуждение показывает, что «в отличие от Observable в RxJS, сигналы (Signals) плохо подходят для представления событий или сообщений!» Это ставит перед нами вызов при использовании потоковых ресурсов, который, к счастью, можно преодолеть, просто изменив нашу точку зрения: если мы позволим ресурсу представлять не только текущее значение, но и все полученные на данный момент сообщения или, по крайней мере, все сообщения во временном окне, которое нас интересует, мы снова можем рассматривать эту совокупность сообщений как значение, которое меняется со временем.

Иными словами, нам нужно перенести сбор отдельных сообщений и управление ими в ресурс. Этот взгляд также хорошо согласуется с наблюдением из предыдущей статьи о том, что в мире сигналов используются специфичные для сценария использования и, следовательно, крупнозернистые строительные блоки.

## Сообщения и протокол

В примере определено несколько типов для отдельных сообщений, которыми клиент обменивается с сервером чата:

```typescript
export type ChatRequest =
  | {
      type: 'username';
      id: number;
      name: string;
    }
  | {
      type: 'message';
      id: number;
      text: string;
    };

export type ChatResponse =
  | {
      type: 'id';
      id: number;
    }
  | {
      type: 'username';
      id: number;
      name: string;
    }
  | {
      type: 'message';
      id: number;
      name: string;
      text: string;
    };
```

Протокол между клиентом и сервером следующий:

1.  Клиент устанавливает соединение по веб-сокету.
2.  Сервер отвечает _ChatResponse_ типа _id_, через который клиент получает уникальный идентификатор сессии.
3.  Клиент отправляет _ChatRequest_ типа _username_, чтобы сообщить имя пользователя.
4.  Сервер подтверждает это имя пользователя с помощью _ChatResponse_ типа _username_. Если желаемое имя пользователя уже занято, клиент получает через это сообщение исправленное имя пользователя, которое сервер создаёт, добавляя число к предпочтительному.
5.  Клиент отправляет _ChatRequest_ со своими сообщениями (тип _message_). Сервер распространяет это сообщение всем клиентам как _ChatResponse_ типа _message_.

## Представление чата

Соединение с чатом представлено в примере типом _ChatConnection_:

```typescript
export type SendFn = (message: string) => void;

export type ChatConnection = {
  resource: ResourceRef<ChatResponse[] | undefined>;
  connected: () => boolean;
  acceptedUserName: () => string;
  send: SendFn;
};
```

Сердцем _ChatConnection_ является ресурс, который представляет все полученные на данный момент сообщения в виде массива. Он также предоставляет дополнительную информацию о состоянии в виде сигналов. Сюда входят статус соединения (_connected_) и имя пользователя (_acceptedUserName_), при необходимости исправленное сервером.

Метод _send_ позволяет чату отправлять сообщения на сервер. Реализация преобразует переданную строку в _ChatRequest_ типа _message_.

## Фабрика для чата

Фабрика для чата структурирована аналогично той, что обсуждалась ранее для таймера. Однако сначала она подготавливает несколько дополнительных переменных:

```typescript
export function chatConnection(
  websocketUrl: string,
  userName: () => string
): ChatConnection {

  let connection: WebSocket;

  const connected = signal(false);
  const id = signal(0);
  const acceptedUserName = signal('');

  const params = computed(() => ({
    userName: userName(),
  }));

  const chatResource = resource({
    params,
    stream: async (loaderParams) => {
      // инициализация соединения по веб-сокету
      // обработка и сбор сообщений
      […]
    },
  });

  const send: SendFn = (message: string) => {
    const request: ChatRequest = {
      type: 'message',
      id: id(),
      text: message,
    };
    connection.send(JSON.stringify(request));
  };

  return {
    connected,
    resource: chatResource,
    acceptedUserName,
    send,
  };
}
```

Эти переменные включают _connection_, которая отражает соединение по веб-сокету, установленное потоковым загрузчиком (Streaming Loader), ранее упомянутые сигналы _connected_ и _acceptedUserName_, а также сигнал _id_, представляющий текущую сессию пользователя. В итоге фабрика возвращает несколько из этих сигналов, созданный потоковый ресурс и метод _send_ как _ChatConnection_. Она не публикует сигнал _id_, так как это деталь реализации.

Потоковый загрузчик ресурса сначала создаёт массив _messages_, в который он собирает полученные сообщения чата:

```typescript
const chatResource = resource({
  request,
  stream: async (loaderParams) => {
    const userName = loaderParams.params.userName;

    let messages: ChatResponse[] = [];

    // 1. Создаём сигнал, представляющий поток
    const resultSignal = signal<ResourceStreamItem<ChatResponse>>({
      value: messages,
    });

    if (!userName) {
      return resultSignal;
    }

    // 2. Настраиваем асинхронную логику, обновляющую сигнал
    connection = new WebSocket(websocketUrl, 'json');

    connection.addEventListener('open', (event) => {
      console.log('[open]');
      connected.set(true);
    });

    connection.addEventListener('message', (event) => {
      const value = JSON.parse(event.data) as ChatResponse;
      console.log('[message]', value);

      if (value.type === 'id') {
        id.set(value.id);
        sendUserName(value.id, userName, connection);
      }

      if (value.type === 'username' && value.id == id()) {
        acceptedUserName.set(value.name);
      }

      if (value.type === 'message' || value.type === 'username') {
        messages = [...messages, value];
        resultSignal.set({ value: messages });
      }
    });

    connection.addEventListener('error', (event) => {
      const error = new Error(Error with websocket connection);
      console.log('[event]', event);
      resultSignal.set({ error });
    });

    // 3. Настраиваем обработчик очистки, запускаемый по AbortSignal
    params.abortSignal.addEventListener('abort', () => {
      console.log('clean up!');
      connection.close();
      connected.set(false);
      id.set(0);
      acceptedUserName.set('');
    });

    // 4. Возвращаем сигнал
    return resultSignal;
  },
});

function sendUserName(id: number, userName: string, connection: WebSocket) {
  const message: ChatRequest = {
    type: 'username',
    id: id,
    name: userName,
  };
  connection.send(JSON.stringify(message));
}
```

Остальная часть реализации следует четырём пунктам, обсуждавшимся в предыдущей статье. Подготовленный сигнал отражает поток и содержит управляемый массив с полученными сообщениями.

Клиент получает информацию от сервера через настроенные обработчики событий. Событие _message_ заботится об описанном выше протоколе. Оно добавляет все полученные сообщения в массив messages. Это должно делаться иммутабельно, то есть с использованием поверхностной копии (shallow copy), чтобы сигнал воспринял изменение.

Событие _open_ устанавливает _connected_ в _true_, а событие _error_ записывает текущую ошибку в поток. Событие _abort_ от _AbortSignal_ закрывает соединение по веб-сокету и сбрасывает управляемые свойства. В конце потоковый загрузчик, как обычно, возвращает сигнал, отражающий поток.

## Заключение

Сигналы в Angular всегда представляют только текущее значение. Промежуточные значения для изменений, следующих друг за другом в рамках одной задачи (task), игнорируются (свойство Glitch Free). Это хорошо подходит для меняющихся состояний, таких как счётчик, но может приводить к потере данных в потоках событий — таких как чат, который должен отображать все полученные сообщения.

Решение состоит в том, чтобы собирать полученные сообщения (во временном окне, которое нас интересует) в ресурсе и публиковать получающийся массив через ресурс. Этот массив, таким образом, соответствует значению, которое меняется со временем.

Хотя это интересный паттерн, никто, разумеется, не мешает нам использовать другие подходы, такие как RxJS, которые напрямую представляют поток событий.
