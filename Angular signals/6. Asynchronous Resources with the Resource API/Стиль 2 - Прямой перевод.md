Экспериментальный Resource API, представленный в Angular 19 и переработанный в Angular 20, позволяет асинхронно загружать ресурсы с использованием сигналов (Signals). Типичный сценарий использования `resource` — это загрузка данных. В этой статье я показываю, как построить типичный CRUD-сценарий с использованием нового API `resource`.

## Первые шаги с Resource API

У каждого `resource` есть функция `loader`, которая возвращает `Promise` с загруженными данными. Этот `loader` срабатывает, когда ресурс инициализирован. Опционально у ресурса может быть сигнал `params`, предоставляющий параметры (критерии поиска) для `loader`. Каждый раз, когда сигнал `params` изменяется, загрузчик срабатывает снова:

```typescript
@Component([...])
export class DessertsComponent {
  #dessertService = inject(DessertService);

  [...]

  // Критерии для поиска
  originalName = signal('');
  englishName = signal('');

  // Объединяем критерии в computed-сигнал
  dessertsCriteria = computed(() => ({
    originalName: this.originalName(),
    englishName: this.englishName(),
  }));

  // Определяем resource с params (= критериями поиска) и loader.
  // Каждый раз, когда params изменяются, срабатывает loader
  dessertsResource = resource({
    params: this.dessertsCriteria,
    loader: (loaderParams) => {
      return this.#dessertService.findPromise(loaderParams.params);
    },
    defaultValue: []
  });

  // изначально ресурсы имеют значение undefined
  desserts = this.dessertsResource.value;

  loading = this.dessertsResource.isLoading;
  error = this.dessertsResource.error;

  // Реактивный поток продолжается...
  ratings = signal<DessertIdToRatingMap>({});
  ratedDesserts = computed(() => this.toRated(this.desserts(), this.ratings()));

  [...]
}
```

В этом примере `dessertsResource` использует значения в сигналах `originalName` и `englishName` в качестве параметров. Загрузчик срабатывает изначально и при изменении этих сигналов. Объект `param`, передаваемый в `loader`, содержит текущие критерии поиска из сигнала `params`.

Результат `resource` находится в его сигнале `value`. По умолчанию сигнал value ресурса равен `undefined`, пока загрузчик асинхронно не вернёт первое значение. Чтобы предотвратить это, в показанном примере для `defaultValue` задан пустой массив. Булев `isLoaded` информирует о состоянии загрузки, а `error` — это сигнал с возможной ошибкой, возникшей во время загрузки.

Computed-сигнал `ratedDesserts` объединяет полученные десерты с возможно загруженными оценками. Это показывает, что у нас есть чёткий реактивный поток, который простирается от пользовательского ввода через загрузку ресурса до проекции ресурса в модель, привязанную в представлении. Одно действие реактивно ведёт к другому.

## Важно: загрузчики не отслеживаются!

Важно отметить, что хотя `params` отслеживаются, `loader` — нет. Это означает, что изменение сигнала `params` запускает новое выполнение `loader`, но изменение сигнала (`Signal`), используемого внутри `loader`, **не запускает**.

Это связано с тем, что автоматическое отслеживание работало бы только в первой части `loader`, которая выполняется синхронно. Всё, что выполняется после первой асинхронной операции, например код после `await` или в обработчике `.then`, не может быть подвергнуто автоматическому отслеживанию Angular. Чтобы избежать таких запутанных ситуаций, когда часть загрузчика обрабатывается иначе, весь загрузчик объявлен неотслеживаемым (untracked).

## Гонки запросов (Race Conditions)

Во многих веб-приложениях пользователь может легко запустить пересекающиеся запросы. Особенно это касается реактивного UI, где простое изменение фильтра может запустить ещё одну операцию загрузки:

![Пересекающиеся запросы](https://www.angulararchitects.io/wp-content/uploads/2024/10/overlapping.png)

В этом случае пользователь ожидал бы получить результаты только для «Ice Cream Pancakes», хотя незадолго до этого у него были другие фильтры. Было бы довольно запутанно, если бы на короткое время промелькнули результаты для обычных «Pancakes» и «Sacher Cake». Было бы ещё более запутанно, если бы первый поиск занял чуть больше времени:

![Пересекающиеся запросы](https://www.angulararchitects.io/wp-content/uploads/2024/10/race-condition.png)

В этом случае нежелательные промежуточные результаты промелькнули бы в порядке, который не соответствует порядку запросов. Это называется гонкой запросов (race condition).

## Resource API имеет семантику switchMap

В Angular мы обычно используем RxJS и его оператор `switchMap`, который отменяет все пересекающиеся запросы, кроме последнего, и тем самым предотвращает такие ситуации. Хорошая новость в том, что `resource` использует то же поведение. Однако по умолчанию он не может отменить предыдущий запрос. Вместо этого он просто игнорирует его результат.

Если вы действительно хотите отменить предыдущий запрос, когда планируется другой, вам нужно учитывать `AbortSignal`, который Resource API передаёт в `loader`:

```typescript
[...]
dessertsResource = resource({
  params: this.dessertsCriteria,
  loader: (loaderParams) => {
    return this.#dessertService.findPromise(
      loaderParams.params,
      loaderParams.abortSignal,
    );
  },
  defaultValue: []
});
[...]
```

Хотя он называется `AbortSignal`, это не сигнал в смысле примитивов реактивности Angular, а JavaScript-API, предоставляемое всеми современными браузерами и поддерживаемое несколькими API, такими как Fetch API. Хотя `HttpClient` в Angular уже способен использовать `fetch` под капотом, его API не позволяет передавать `AbortSignal`. Чтобы это компенсировать, я написал простую вспомогательную функцию `toPromise`, которая преобразует `Observable` в `Promise` с учётом `AbortSignal`. Вы найдёте эту реализацию, вдохновлённую функцией `rxResource`, упомянутой ниже, в предоставленном демо-репозитории.

Хотя семантика `switchMap` может быть наиболее распространённой для загрузки данных, RxJS предоставляет и другие так называемые операторы сведения (flattening operators), по-разному работающие с пересекающимися запросами. Однако для рассматриваемых операций `resource` всегда использует семантику `switchMap`. Помимо этого, метод `reload`, рассматриваемый ниже, всегда использует семантику `exhaustMap`.

Это показывает, что команда Angular не хочет конкурировать с существующими библиотеками. Цель — предоставить строительные блоки с разумной логикой по умолчанию, покрывающей большинство сценариев использования. Для всего, что выходит за эти рамки, мы можем реализовать собственные решения или добавить проверенные библиотеки, такие как RxJS.

## Дебаунсинг (Debouncing)

Обычно, когда запросы запускаются сразу после того, как пользователь изменяет фильтр, нам нужен некоторый дебаунсинг: велика вероятность, что пользователь меняет дальнейшие фильтры, и поэтому мы хотим подождать некоторое время, чтобы не запускать несколько ненужных запросов подряд.

Простой способ добиться дебаунсинга — это добавить задержку в загрузчик. Как следствие, во время дебаунсинга ресурс уже находится в состоянии загрузки. Однако в хорошей дискуссии с некоторыми из моих коллег по Angular GDE и членами команды Angular ([Rainer Hahnekamp](https://bsky.app/profile/rainerhahnekamp.bsky.social), [Deborah Kurata](https://bsky.app/profile/deborahkurata.bsky.social), [Sander Elias](https://bsky.app/profile/sanderelias.nl), [Matthieu Riegler](https://bsky.app/profile/jeanmeche.com) и [Alex Rickabaugh](https://bsky.app/profile/synalx.bsky.social), если назвать лишь некоторых) мы пришли к выводу, что дебаунсинг события, ведущего к новому запросу, может быть более подходящим. В пользу этого есть несколько аргументов. Один из аргументов состоит в том, что дебаунсинг лежащего в основе события не даёт ресурсу перейти в состояние загрузки до завершения дебаунсинга. Другой — в том, что это позволяет нам перенести ресурс в стор (store), который используется независимо от того, дебаунсится ли ввод или нет.

Дебаунсинг лежащего в основе события прост при использовании Reactive Forms. В этом случае мы можем дебаунсить Observable `valueChanges`. В случаях, когда мы используем Template-driven Forms с Two-Way Bindings, нам нужен помощник, дебаунсящий привязанный сигнал. Для этого я написал простой помощник `debounceSignal`:

```typescript
#debouncedCriteria = debounceSignal(this.#dessertsCriteria, 300);

#dessertsResource = resource({
  params: this.#debouncedCriteria,
  loader: (loaderParams) => {
    return this.#dessertService.findPromise(loaderParams.params, param.abortSignal);
  },
  defaultValue: []
});
```

Для простой реализации debounceSignal мы можем воспользоваться интеропом Angular с RxJS:

```typescript
import { Signal } from "@angular/core";
import { toObservable, toSignal } from "@angular/core/rxjs-interop";
import { debounceTime } from "rxjs";

export function debounceSignal<T>(source: Signal<T>, timeMsec: number): Signal<T | undefined> {
    return toSignal(toObservable(source).pipe(debounceTime(timeMsec)));
}
```

Обратите внимание, что результирующий сигнал будет представлять `undefined`, пока не будет выполнен первый дебаунсинг. Сигнал `params`, возвращающий `undefined`, не даёт ресурсу запустить загрузчик, что во многих случаях может быть предполагаемым поведением. В случаях, когда вы хотите пропустить начальный дебаунсинг, в качестве значения по умолчанию можно использовать начальное значение сигнала:

```typescript
export function debounceSignal<T>(source: Signal<T>, timeMsec: number): Signal<T> {
    return toSignal(toObservable(source).pipe(debounceTime(timeMsec)), {
        initialValue: source()
    });
}
```

Разумеется, как всегда, мы также можем предоставить реализацию, не использующую RxJS. Это может быть интересно в средне- и долгосрочной перспективе, когда RxJS станет необязательным в Angular. В любом случае, я бы рассматривал такую функциональность как инфраструктурный код, которому место не в коде приложения, а в самом фреймворке, во вспомогательной библиотеке или хотя бы в служебном модуле, который приложение трактует как «чёрный ящик». Посмотрим, предоставят ли предстоящие формы на основе сигналов (Signal-based Forms) хорошее решение для таких сценариев использования.

## Перезагрузка и ручная загрузка

Использованный до сих пор `resource` автоматически запускал `loader` после своей инициализации и после каждого изменения сигнала `params`. Это было удобно в рассматриваемом примере. Однако это не то поведение, которое нужно всегда.

Если вы хотите отключить начальное выполнение загрузчика, просто убедитесь, что ваш сигнал `params` возвращает `undefined`:

```typescript
type Requested = undefined | true;

[...]

#ratingsRequested = signal<Requested>(undefined);

[...]

#ratingsResource = resource({
  params: this.#ratingsRequested,
  loader: () => {
    return this.#ratingService.loadExpertRatingsPromise();
  },
  defaultValue: {}
});

readonly ratings = this.#ratingsResource.value;

[...]
```

Как упоминалось выше, сигнал `params`, возвращающий `undefined`, по замыслу не запускает загрузчик. В текущем случае у нас нет параметров, влияющих на то, какие оценки загружать, поэтому я просто обхожусь значением `true` или `undefined`.

Чтобы запустить загрузчик в первый раз, мы теперь можем установить сигнал `params` в `true`. Чтобы запустить загрузчик снова позже, приложение вызывает метод `reload` у ресурса:

```typescript
loadRatings(): void {
  this.#ratingsRequested.set(true);
  this.#ratingsResource.reload();
}
```

Метод `reload` предотвращает пересекающиеся запросы, немедленно возвращаясь, если уже выполняется один. Используя термины из RxJS, это можно описать как семантику `exhaustMap`. По этой причине нам не нужно различение случаев, первый это запрос или последующий. Это, в свою очередь, упрощает реализацию `loadRatings`.

Признаю, что использование типа `undefined | true` может ощущаться немного странно. Однако это позволяет использовать Resource так, как и задумано: всё дело в реактивной загрузке данных. Этот пример ощущался бы более естественным, если бы у ресурса был настоящий параметр, например `expertId`.

## Интерлюдия 1: linkedSignal для обновлений

Пользователь может менять оценки. Обычно это нормально, потому что ресурс можно обновлять вручную методами `set` и `update`, используемыми в разделе ниже. Однако `value` у `resource` по умолчанию равно `undefined`. По крайней мере с тех пор, как Angular-приложения по умолчанию используют строгий режим (strict mode) TypeScript, я стараюсь избегать `null` и `undefined` в пользу объекта по умолчанию, также известного как Null Object. До сих пор я использовал `computed`, чтобы проецировать `null` и `undefined` в соответствующий Null Object:

```typescript
ratings = computed(() => this.ratingsResource.value() ?? {});
```

К сожалению, computed-сигнал доступен только для чтения. Спасением является функция `linkedSignal`. Она предоставляет computed-сигнал, который можно менять:

```typescript
ratings = linkedSignal(() => this.ratingsResource.value() ?? {});
```

Если сигналы, используемые в вычислении, меняются, вычисление запускается заново, и напрямую присвоенное значение перезаписывается. Для полноты картины хочу отметить, что вычисление происходит лениво: если значение никому не нужно, повторное вычисление не происходит.

Эта особенность делает `linkedSignal` идеальным выбором для форм: мы можем менять локальную копию и по требованию отправлять её обратно для сохранения.

## Обработка ошибок

Обработка ошибок также встроена в Resource API. Когда загрузчик выбрасывает исключение или отклоняет возвращаемый `Promise`, `resource` переходит в состояние `error`. В этом состоянии сигнал `error` предоставляет выброшенное значение или значение, переданное в reject-функцию `Promise`.

Однако `resource` всё равно продолжит работать: если кто-то вызовет метод `reload` или если сигнал `params` изменится, `loader` выполнится снова. Если `loader` отработает успешно, ресурс очистит сигнал `error` и перейдёт в состояние `resolved`.

Это сопоставимо с сегодняшним поведением по умолчанию у Effects в NgRx, где некоторая дополнительная логика под капотом не даёт используемому RxJS-пайплайну остановиться в случае ошибки.

Однако начиная с Angular 20 вам не разрешается обращаться к `value` ресурса, если есть ошибка. Поэтому в шаблоне сначала нужно проверить состояние `error`:

```typescript
@if(!error()) {
  @for (dessert of ratedDesserts(); track dessert.id) {
    <app-dessert-card
      [dessert]="dessert"
      (ratingChange)="updateRating(dessert.id, $event)"
    ></app-dessert-card>
  }

}
@else {
  <b>Error loading desserts!</b>
}
```

В этом случае `ratedDesserts` выводится из значения ресурса с помощью `computed`. По этой причине и здесь нам приходится проверять состояние ошибки.

## rxResource для интеропа с RxJS

Если у вас уже есть Observable, вам не нужно преобразовывать его в `Promise`. Вместо этого вы можете использовать API `rxResource`. Он работает как `resource`, но ожидается, что загрузчик вернёт `Observable`. Кроме того, загрузчик определяется через свойство streaming, потому что `rxResource` — это так называемый потоковый ресурс, который может выдавать несколько значений со временем.

```typescript
[...]

dessertsResource = rxResource({
  params: this.dessertsCriteria,
  stream: (loaderParams) => {
    return timer(300).pipe(switchMap(() => this.#dessertService.find(loaderParams.params)));
  },
  defaultValue: []
});

[...]

ratingsResource = rxResource({
  params: this.#ratingsRequested,
  stream: () => {
    return this.#ratingService.loadExpertRatings()
  },
  defaultValue: {}
});

[...]
```

Кроме того, загрузчику не нужен `AbortSignal`, поскольку Observable в целом можно прервать (неявно или явно отписавшись).

## Resource в сервисах и сторах

В показанных примерах `resource` использовался напрямую в компоненте. Это было сделано в первую очередь ради простоты. Часто вы будете использовать ресурсы в сервисах или сторах. Одна из причин этого — управление состоянием: загруженные данные переживут момент, когда Angular уничтожит рассматриваемый компонент (например, из-за переключения на другой маршрут), так что их можно будет использовать позже экземпляром того же или другого компонента.

Вторая причина в том, что сторы помогают упорядочить ваш реактивный поток данных:

![](https://www.angulararchitects.io/wp-content/uploads/2024/10/store.png)

В результате получается так называемый однонаправленный поток данных: компоненты отправляют в стор намерения (intentions). Я использую термин «намерение» в абстрактном смысле, потому что то, как оно выражается, зависит от стора. При использовании Redux-стора это будет диспетчеризованный action; в более лёгком сторе вроде NGRX Signal Store это может быть просто вызванный метод.

Выраженное намерение заставляет стор выполнить некоторые задачи, в результате которых в сигналы записываются новые значения. Эти сигналы можно проецировать через `computed` и передавать данные вниз, в представление компонента.

Очень простой стор — это сервис, предоставляющий сигналы. То есть мы можем перенести код, который мы до сих пор обсуждали, в сервис:

```typescript
type Requested = undefined | true;

@Injectable({ providedIn: 'root' })
export class DessertStore {
  #dessertService = inject(DessertService);
  #ratingService = inject(RatingService);

  private #ratingsRequested = signal<Requested>(undefined);

  readonly originalName = signal('');
  readonly englishName = signal('');

  #dessertsCriteria = computed(() => ({
    originalName: this.originalName(),
    englishName: this.englishName(),
  }));

  #dessertsResource = resource({
    params: this.#dessertsCriteria,
    loader: debounce((loaderParams) => {
      return this.#dessertService.findPromise(loaderParams.params, loaderParams.abortSignal);
    }),
    defaultValue: []
  });

  #ratingsResource = resource({
    params: this.#ratingsRequested,
    loader: () => {
      return this.#ratingService.loadExpertRatingsPromise();
    },
    defaultValue: {}
  });

  readonly loading = computed(() => this.#ratingsResource.isLoading() || this.#dessertsResource.isLoading());

  readonly desserts = this.#dessertsResource.value;
  readonly ratings = this.#ratingsResource.value;
  readonly ratedDesserts = computed(() => toRated(this.desserts(), this.ratings()));
  readonly error = computed(() => getErrorMessage(this.dessertsResource.error() || this.ratingsResource.error()));

  loadRatings(): void {
    this.#ratingsRequested.set(true);
    this.#ratingsResource.reload();
  }

  updateRating(id: number, rating: number): void {
    this.#ratingsResource.update((ratings) => ({
      ...ratings,
      [id]: rating,
    }));
  }
}
```

Здесь мы видим компонент, потребляющий стор:

```typescript
@Component([...])
export class DessertsComponent {
  #store = inject(DessertStore);

  originalName = this.#store.originalName;
  englishName = this.#store.englishName;
  loading = this.#store.loading;
  error = this.#store.error;

  ratedDesserts = this.#store.ratedDesserts;

  loadRatings(): void {
    this.#store.loadRatings();
  }

  updateRating(id: number, rating: number): void {
    this.#store.updateRating(id, rating);
  }
}
```

В этом примере однонаправленный поток данных виден довольно ясно: компонент отправляет наверх свои намерения, вызывая методы, и получает (обновлённые) данные через сигналы, привязанные в шаблоне.

## Обновление ресурсов

Ресурсы не только для чтения. Мы также можем обновлять их новым значением. Хотя обратную запись изменённых значений нужно выполнять вручную, локальное обновление ресурса гарантирует, что новое значение станет частью нашего реактивного потока. Это означает, что оно будет отображено и спроецировано и станет основой для дальнейших обновлений.

В следующем примере показан простой сервис, выступающий в роли стора для страницы с подробностями. Обратите особое внимание на метод `save`:

```typescript
@Injectable({ providedIn: 'root' })
export class DessertDetailStore {
  #dessertService = inject(DessertService);
  #id = signal<number | undefined>(undefined);

  #dessertResource = resource({
    params: computed(() => ({ id: this.#id() })),
    loader: async (loaderParams) => {
      const id = loaderParams.params.id;
      if (id) {
        const result = await this.#dessertService.findPromiseById(id);
        return result ?? initDessert;
      }
      else {
        return Promise.resolve(initDessert);
      }
    },
    defaultValue: initDessert
  });

  readonly loading = this.#dessertResource.isLoading;
  readonly error = this.#dessertResource.error;

  #saving = signal(false);

  load(id: number): void {
    this.#id.set(id);
  }

  save(dessert: Dessert): void {
    try {
      this.#saving.set(true);
      console.log('saving', dessert);
      // Здесь был бы ваш HTTP-вызов
      [...]
      this.#dessertResource.set(dessert);
    }
    finally {
      this.#saving.set(false);
    }    
  }
}
```

Метод `save` получает обновлённый десерт и записывает его обратно на сервер. Соответствующий HTTP-вызов здесь обозначен только комментарием. После выполнения этого вызова `dessertResource` обновляется новым значением с помощью `set`. В качестве альтернативы есть также метод `update`, который позволяет спроецировать текущее значение в новое. Если в процессе обновления ресурса таким локальным значением идёт процесс загрузки, этот процесс загрузки прерывается.

В примере выше `resource` был приватным для сервиса. Поэтому сервис может гарантировать, что ресурс обновляется только надлежащим образом. Если вы хотите выставить весь `resource` наружу, вы можете использовать его метод `asReadonly`, чтобы помешать потребителям менять управляемые данные:

```typescript
readonly dessert = dessertResource.asReadonly();
```

## Интерлюдия 2: linkedSignal для формы с подробностями

При привязке загруженного `dessert` к шаблонной форме (template-driven form) нам нужны записываемые (writable) сигналы. Однако сторы обычно предоставляют сигналы только для чтения. Также сигнал `value`, предоставляемый `resource`, доступен только для чтения, хотя его можно менять напрямую через методы `set` и `update` ресурса.

Как уже упоминалось выше, это хороший случай для `linkedSignal`:

```typescript
@Component({
  selector: 'app-dessert-detail',
  standalone: true,
  imports: [JsonPipe, RouterLink, FormsModule],
  templateUrl: './dessert-detail.component.html',
  styleUrl: './dessert-detail.component.css'
})
export class DessertDetailComponent implements OnChanges {
  store = inject(DessertDetailStore);

  id = input.required({
    transform: numberAttribute
  });

  loadedDessert = this.store.dessert;
  loading = this.store.loading;
  error = this.store.error;

  dessert = {
    originalName: linkedSignal(() => this.loadedDessert().originalName),
    englishName: linkedSignal(() => this.loadedDessert().englishName),
    kcal: linkedSignal(() => this.loadedDessert().kcal)
  };

  ngOnChanges(): void {
    const id = this.id();
    this.store.load(id);
  }

  save(): void {
    const dessert = {
      ...this.loadedDessert(),
      originalName: this.dessert.originalName(),
      englishName: this.dessert.englishName(),
      kcal: this.dessert.kcal(),    
    };

    this.store.save(dessert);
  }
}
```

Здесь каждое свойство, которое мы хотим привязать к полю формы, представлено связанным сигналом (linked Signal). Они обновляются всякий раз, когда меняется их источник. Тем не менее их можно обновлять локальным значением. Метод `save` берёт это локальное значение и записывает его обратно в стор, который делегирует работу `resource`.

В результате мы можем напрямую привязать ngModel к нашим сигналам:

```html
<form>
  <div>
    <label for="englishName"> English Name </label>
    <input name="englishName" [(ngModel)]="dessert.englishName" />
  </div>
  <div>
    <label for="originalName"> OriginalName </label>
    <input name="originalName" [(ngModel)]="dessert.originalName" />
  </div>
  <div>
    <label for="kcal"> kcal </label>
    <input name="kcal" [(ngModel)]="dessert.kcal" />
  </div>
</form>
```

Эти связанные сигналы можно рассматривать как аналог объектов `FormControl` в реактивных формах. Конечно, объекты `FormControl` предоставляют больше возможностей, например регистрацию валидаторов. Но, и это моё личное впечатление, это показывает, что с сигналами оба подхода не так уж далеки друг от друга.

Хотя в этом примере я решил использовать шаблонные формы, вы вполне можете использовать и реактивные формы. В этом случае вы, возможно, захотите использовать `effect` для связывания сигналов, полученных из стора, с вашими `FormControl`.

## Бонус: вспомогательные функции для упрощения

При реализации показанного здесь демо-приложения я также написал несколько реактивных помощников, упрощающих работу с сигналами и ресурсами. Вы найдёте их в предоставленном демо-репозитории в ветке `07c-final`. Один из них — `deepLink`: он берёт сигнал с объектом и преобразует все его свойства в связанные сигналы:

```typescript
dessert = deepLink(this.loadedDessert);
```

Это избавляет нас от вызова `linkedSignal` по одному разу на каждое свойство, как в примере из предыдущего раздела.

Также я хотел не дать индикатору загрузки появляться при быстрых операциях загрузки. Поэтому мой помощник `debounceTrue` использует ресурс, чтобы дебаунсить переход из `false` в `true`:

```typescript
loading = debounceTrue(() => this.ratingsResource.isLoading() || this.dessertsResource.isLoading(), 500);
```

Он не дебаунсит переход из `true` в `false`, потому что обычно мы хотим, чтобы индикатор загрузки исчезал немедленно, как только загруженный ресурс становится доступен.

## Заключение

Новый Resource API действительно является недостающим звеном в истории сигналов в Angular: он даёт нам официальный способ загружать асинхронные ресурсы напрямую внутри реактивного потока. Также он берёт на себя гонки запросов и содержит базовую обработку ошибок.

С этим новым API у нас есть простая в использовании встроенная возможность для очень распространённых сценариев. Для более продвинутых сценариев, таких как работа с несколькими параллельными потоками данных, мы можем переключиться на что-то более мощное, например RxJS. Благодаря интеропу с RxJS и `rxResource` оба мира можно соединить.
