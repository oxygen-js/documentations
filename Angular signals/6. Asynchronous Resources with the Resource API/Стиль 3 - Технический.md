Экспериментальный Resource API, представленный в Angular 19 и переработанный в Angular 20, обеспечивает асинхронную загрузку ресурсов на базе сигналов (Signals). Типичный сценарий применения `resource` — загрузка данных. В этой статье я демонстрирую построение типового CRUD-сценария с использованием нового API `resource`.

## Первые шаги с Resource API

Каждый `resource` имеет функцию-загрузчик `loader`, возвращающую `Promise` с загруженными данными. Этот `loader` инициируется после инициализации ресурса. Опционально ресурс может иметь сигнал `params`, предоставляющий параметры (критерии поиска) для `loader`. При каждом изменении сигнала `params` загрузчик инициируется повторно:

```typescript
@Component([...])
export class DessertsComponent {
  #dessertService = inject(DessertService);

  [...]

  // Критерии для поиска
  originalName = signal('');
  englishName = signal('');

  // Объединяем критерии в computed-сигнал
  dessertsCriteria = computed(() => ({
    originalName: this.originalName(),
    englishName: this.englishName(),
  }));

  // Определяем resource с params (= критериями поиска) и loader.
  // При каждом изменении params инициируется loader
  dessertsResource = resource({
    params: this.dessertsCriteria,
    loader: (loaderParams) => {
      return this.#dessertService.findPromise(loaderParams.params);
    },
    defaultValue: []
  });

  // изначально ресурсы имеют значение undefined
  desserts = this.dessertsResource.value;

  loading = this.dessertsResource.isLoading;
  error = this.dessertsResource.error;

  // Реактивный поток продолжается...
  ratings = signal<DessertIdToRatingMap>({});
  ratedDesserts = computed(() => this.toRated(this.desserts(), this.ratings()));

  [...]
}
```

В этом примере `dessertsResource` использует значения сигналов `originalName` и `englishName` в качестве параметров. Загрузчик инициируется при инициализации и при изменении этих сигналов. Объект `param`, передаваемый в `loader`, содержит текущие критерии поиска из сигнала `params`.

Результат работы `resource` находится в его сигнале `value`. По умолчанию сигнал value ресурса равен `undefined` до тех пор, пока загрузчик асинхронно не вернёт первое значение. Чтобы исключить это, в показанном примере для `defaultValue` задан пустой массив. Булев сигнал `isLoaded` информирует о состоянии загрузки, а `error` — это сигнал с возможной ошибкой, возникшей в процессе загрузки.

Computed-сигнал `ratedDesserts` комбинирует полученные десерты с возможно загруженными оценками (ratings). Это показывает наличие чёткого реактивного потока, простирающегося от пользовательского ввода через загрузку ресурса до проекции ресурса в модель, привязанную в представлении (view). Одно действие реактивно влечёт за собой следующее.

## Важно: загрузчики не отслеживаются (untracked)!

Важно отметить: хотя `params` отслеживаются (tracked), `loader` — нет. Это означает, что изменение сигнала `params` инициирует новое выполнение `loader`, тогда как изменение сигнала (`Signal`), используемого внутри `loader`, — **не инициирует**.

Причина в том, что автоотслеживание (auto-tracking) работало бы только для первой части `loader`, выполняемой синхронно. Всё, что выполняется после первой асинхронной операции (например, код после `await` или в обработчике `.then`), не может попасть под автоотслеживание Angular. Чтобы избежать запутанных ситуаций, когда одна часть загрузчика обрабатывается иначе, весь загрузчик объявлен неотслеживаемым (untracked).

## Гонки запросов (Race Conditions)

Во многих веб-приложениях пользователь может легко инициировать пересекающиеся (overlapping) запросы. Особенно это характерно для реактивного UI, где простое изменение фильтра уже способно инициировать дополнительную операцию загрузки:

![Пересекающиеся запросы](https://www.angulararchitects.io/wp-content/uploads/2024/10/overlapping.png)

В этом случае пользователь ожидал бы получить результаты только для «Ice Cream Pancakes», хотя незадолго до этого фильтры были другими. Было бы весьма сбивающим с толку, если бы на краткое время промелькнули результаты для обычных «Pancakes» и «Sacher Cake». Ещё более запутывало бы, если бы первый поиск занял несколько больше времени:

![Пересекающиеся запросы](https://www.angulararchitects.io/wp-content/uploads/2024/10/race-condition.png)

В этом случае нежелательные промежуточные результаты промелькнули бы в порядке, не соответствующем порядку запросов. Это называется гонкой запросов (race condition).

## Resource API имеет семантику switchMap

В Angular мы обычно задействуем RxJS и его оператор `switchMap`, который отменяет все пересекающиеся запросы, кроме последнего, и тем самым предотвращает подобные ситуации. Хорошая новость: `resource` использует то же поведение. Однако по умолчанию он не может отменить предыдущий запрос — вместо этого он просто игнорирует его результат.

Если вам действительно требуется отменять предыдущий запрос при планировании нового, необходимо учитывать `AbortSignal`, который Resource API передаёт в `loader`:

```typescript
[...]
dessertsResource = resource({
  params: this.dessertsCriteria,
  loader: (loaderParams) => {
    return this.#dessertService.findPromise(
      loaderParams.params,
      loaderParams.abortSignal,
    );
  },
  defaultValue: []
});
[...]
```

Хотя он называется `AbortSignal`, это не сигнал в смысле примитивов реактивности Angular, а JavaScript-API, предоставляемое всеми современными браузерами и поддерживаемое рядом API, в том числе Fetch API. Хотя `HttpClient` в Angular уже способен использовать `fetch` под капотом, его API не позволяет передавать `AbortSignal`. Чтобы это компенсировать, я написал простую вспомогательную функцию `toPromise`, которая преобразует `Observable` в `Promise` с учётом `AbortSignal`. Эта реализация, вдохновлённая функцией `rxResource` (упомянута ниже), доступна в предоставленном демо-репозитории.

Хотя семантика `switchMap`, по-видимому, наиболее распространена для загрузки данных, RxJS предоставляет и другие так называемые операторы сведения (flattening operators), по-разному обрабатывающие пересекающиеся запросы. Однако для рассматриваемых операций `resource` всегда использует семантику `switchMap`. Помимо этого, метод `reload`, рассматриваемый ниже, всегда использует семантику `exhaustMap`.

Это демонстрирует, что команда Angular не стремится конкурировать с существующими библиотеками. Цель — предоставить строительные блоки с разумной логикой по умолчанию, покрывающей большинство сценариев. Для всего, что выходит за эти рамки, можно реализовать собственные решения либо подключить проверенные библиотеки, такие как RxJS.

## Дебаунсинг (Debouncing)

Как правило, при инициировании запросов непосредственно после изменения пользователем фильтра требуется определённый дебаунсинг (debouncing): велика вероятность, что пользователь изменит и другие фильтры, поэтому стоит выждать некоторое время, чтобы не инициировать несколько лишних запросов подряд.

Простой способ обеспечить дебаунсинг — добавить задержку в загрузчик. Как следствие, во время дебаунсинга ресурс уже находится в состоянии загрузки. Тем не менее в продуктивной дискуссии с рядом коллег по Angular GDE и членами команды Angular ([Rainer Hahnekamp](https://bsky.app/profile/rainerhahnekamp.bsky.social), [Deborah Kurata](https://bsky.app/profile/deborahkurata.bsky.social), [Sander Elias](https://bsky.app/profile/sanderelias.nl), [Matthieu Riegler](https://bsky.app/profile/jeanmeche.com) и [Alex Rickabaugh](https://bsky.app/profile/synalx.bsky.social), если назвать лишь некоторых) мы пришли к выводу, что дебаунсинг события, приводящего к новому запросу, может быть более уместен. В пользу этого есть несколько аргументов. Один из них: дебаунсинг лежащего в основе события предотвращает переход ресурса в состояние загрузки до завершения дебаунсинга. Другой: это позволяет вынести ресурс в стор (store), используемый независимо от того, дебаунсится ли ввод.

Дебаунсинг лежащего в основе события прост при использовании Reactive Forms: в этом случае можно дебаунсить Observable `valueChanges`. В случаях с Template-driven Forms и two-way bindings требуется помощник, дебаунсящий привязанный сигнал. Для этого я написал простой помощник `debounceSignal`:

```typescript
#debouncedCriteria = debounceSignal(this.#dessertsCriteria, 300);

#dessertsResource = resource({
  params: this.#debouncedCriteria,
  loader: (loaderParams) => {
    return this.#dessertService.findPromise(loaderParams.params, param.abortSignal);
  },
  defaultValue: []
});
```

Для простой реализации debounceSignal можно задействовать интероп Angular с RxJS:

```typescript
import { Signal } from "@angular/core";
import { toObservable, toSignal } from "@angular/core/rxjs-interop";
import { debounceTime } from "rxjs";

export function debounceSignal<T>(source: Signal<T>, timeMsec: number): Signal<T | undefined> {
    return toSignal(toObservable(source).pipe(debounceTime(timeMsec)));
}
```

Обратите внимание: результирующий сигнал будет представлять `undefined` до завершения первого дебаунсинга. Сигнал `params`, возвращающий `undefined`, предотвращает инициирование загрузчика ресурсом (то есть `loader` не запускается), что во многих случаях и является предполагаемым поведением. Если требуется пропустить начальный дебаунсинг, в качестве значения по умолчанию можно использовать начальное значение сигнала:

```typescript
export function debounceSignal<T>(source: Signal<T>, timeMsec: number): Signal<T> {
    return toSignal(toObservable(source).pipe(debounceTime(timeMsec)), {
        initialValue: source()
    });
}
```

Разумеется, как и всегда, можно предоставить реализацию без RxJS. Это может представлять интерес в средне- и долгосрочной перспективе, когда RxJS станет необязательным в Angular. В любом случае, подобную функциональность я бы рассматривал как инфраструктурный код, которому место не в коде приложения, а в самом фреймворке, во вспомогательной библиотеке или хотя бы в служебном модуле, трактуемом приложением как «чёрный ящик». Посмотрим, предложат ли предстоящие формы на основе сигналов (Signal-based Forms) удачное решение для подобных сценариев.

## Перезагрузка (reload) и ручная загрузка

Использованный до сих пор `resource` автоматически инициировал `loader` после инициализации и после каждого изменения сигнала `params`. В рассматриваемом примере это было удобно. Однако такое поведение требуется не всегда.

Чтобы отключить начальное выполнение загрузчика, достаточно обеспечить, чтобы сигнал `params` возвращал `undefined`:

```typescript
type Requested = undefined | true;

[...]

#ratingsRequested = signal<Requested>(undefined);

[...]

#ratingsResource = resource({
  params: this.#ratingsRequested,
  loader: () => {
    return this.#ratingService.loadExpertRatingsPromise();
  },
  defaultValue: {}
});

readonly ratings = this.#ratingsResource.value;

[...]
```

Как отмечено выше, сигнал `params`, возвращающий `undefined`, по замыслу не инициирует загрузчик. В текущем случае параметров, влияющих на то, какие оценки загружать, нет, поэтому я обхожусь значением `true` или `undefined`.

Чтобы инициировать загрузчик в первый раз, теперь можно установить сигнал `params` в `true`. Для повторного инициирования загрузчика впоследствии приложение вызывает метод `reload` ресурса:

```typescript
loadRatings(): void {
  this.#ratingsRequested.set(true);
  this.#ratingsResource.reload();
}
```

Метод `reload` предотвращает пересекающиеся запросы, немедленно завершаясь без действия, если запрос уже выполняется. В терминах RxJS это описывается как семантика `exhaustMap`. По этой причине различение случаев (первый запрос или последующий) не требуется, что, в свою очередь, упрощает реализацию `loadRatings`.

Признаю, что использование типа `undefined | true` может показаться несколько неестественным. Однако это позволяет применять Resource так, как и задумано: суть в реактивной загрузке данных. Этот пример выглядел бы более естественно при наличии у ресурса реального параметра, например `expertId`.

## Интерлюдия 1: linkedSignal для обновлений

Пользователь может изменять оценки. Обычно это допустимо, поскольку ресурс можно обновлять вручную методами `set` и `update`, рассматриваемыми в разделе ниже. Однако сигнал `value` у `resource` по умолчанию равен `undefined`. По крайней мере с тех пор, как Angular-приложения по умолчанию используют строгий режим (strict mode) TypeScript, я стараюсь избегать `null` и `undefined` в пользу объекта по умолчанию, также известного как Null Object. До сих пор я использовал `computed` для проекции `null` и `undefined` в соответствующий Null Object:

```typescript
ratings = computed(() => this.ratingsResource.value() ?? {});
```

К сожалению, computed-сигнал доступен только для чтения (read only). Решением является функция `linkedSignal`. Она предоставляет computed-сигнал, который при этом допускает изменение:

```typescript
ratings = linkedSignal(() => this.ratingsResource.value() ?? {});
```

Если сигналы, используемые в вычислении, изменяются, вычисление инициируется повторно, и напрямую присвоенное значение перезаписывается. Для полноты отмечу, что вычисление выполняется лениво (lazily): если значение никому не требуется, повторное вычисление не выполняется.

Эта характеристика делает `linkedSignal` идеально подходящим для форм: мы можем изменять локальную копию и по требованию отправлять её обратно для сохранения.

## Обработка ошибок

Обработка ошибок также встроена в Resource API. Когда загрузчик выбрасывает исключение или отклоняет (reject) возвращаемый `Promise`, `resource` переходит в состояние `error`. В этом состоянии сигнал `error` предоставляет выброшенное значение или значение, переданное в reject-функцию `Promise`.

Однако `resource` продолжит работу: при вызове метода `reload` или при изменении сигнала `params` `loader` выполняется повторно. Если `loader` завершается успешно, ресурс очищает сигнал `error` и переходит в состояние `resolved`.

Это сопоставимо с текущим поведением по умолчанию у Effects в NgRx, где дополнительная внутренняя логика предотвращает остановку используемого RxJS-пайплайна в случае ошибки.

Однако начиная с Angular 20 доступ к `value` ресурса при наличии ошибки запрещён. Поэтому в шаблоне сначала необходимо проверить состояние `error`:

```typescript
@if(!error()) {
  @for (dessert of ratedDesserts(); track dessert.id) {
    <app-dessert-card
      [dessert]="dessert"
      (ratingChange)="updateRating(dessert.id, $event)"
    ></app-dessert-card>
  }

}
@else {
  <b>Error loading desserts!</b>
}
```

В этом случае `ratedDesserts` выводится из значения ресурса посредством `computed`. По этой причине и здесь необходимо проверять состояние ошибки.

## rxResource для интеропа с RxJS

Если у вас уже есть Observable, преобразовывать его в `Promise` не требуется. Вместо этого можно использовать API `rxResource`. Он работает аналогично `resource`, но ожидается, что загрузчик вернёт `Observable`. Кроме того, загрузчик определяется через свойство streaming, поскольку `rxResource` — это так называемый потоковый ресурс (streaming resource), способный эмитировать несколько значений во времени.

```typescript
[...]

dessertsResource = rxResource({
  params: this.dessertsCriteria,
  stream: (loaderParams) => {
    return timer(300).pipe(switchMap(() => this.#dessertService.find(loaderParams.params)));
  },
  defaultValue: []
});

[...]

ratingsResource = rxResource({
  params: this.#ratingsRequested,
  stream: () => {
    return this.#ratingService.loadExpertRatings()
  },
  defaultValue: {}
});

[...]
```

Кроме того, загрузчику не требуется `AbortSignal`, поскольку Observable в целом можно прервать (отписавшись неявно или явно).

## Resource в сервисах и сторах

В показанных примерах `resource` использовался непосредственно в компоненте. Это было сделано прежде всего ради простоты. Зачастую же ресурсы используются в сервисах или сторах. Одна из причин — управление состоянием (state management): загруженные данные переживут момент уничтожения текущего компонента (например, при переключении на другой маршрут), так что их сможет использовать позднее экземпляр того же или другого компонента.

Вторая причина в том, что сторы помогают упорядочить реактивный поток данных:

![](https://www.angulararchitects.io/wp-content/uploads/2024/10/store.png)

В результате получается так называемый однонаправленный поток данных (unidirectional data flow): компоненты отправляют в стор намерения (intentions). Я использую термин «намерение» абстрактно, поскольку способ его выражения зависит от стора. При использовании Redux-стора это диспетчеризованный action; в более лёгком сторе, таком как NGRX Signal Store, это может быть просто вызванный метод.

Выраженное намерение заставляет стор выполнить определённые задачи, в результате которых в сигналы помещаются новые значения. Эти сигналы можно проецировать посредством `computed` и передавать данные вниз, в представление компонента.

Очень простой стор — это сервис, предоставляющий сигналы. То есть рассмотренный до сих пор код можно перенести в сервис:

```typescript
type Requested = undefined | true;

@Injectable({ providedIn: 'root' })
export class DessertStore {
  #dessertService = inject(DessertService);
  #ratingService = inject(RatingService);

  private #ratingsRequested = signal<Requested>(undefined);

  readonly originalName = signal('');
  readonly englishName = signal('');

  #dessertsCriteria = computed(() => ({
    originalName: this.originalName(),
    englishName: this.englishName(),
  }));

  #dessertsResource = resource({
    params: this.#dessertsCriteria,
    loader: debounce((loaderParams) => {
      return this.#dessertService.findPromise(loaderParams.params, loaderParams.abortSignal);
    }),
    defaultValue: []
  });

  #ratingsResource = resource({
    params: this.#ratingsRequested,
    loader: () => {
      return this.#ratingService.loadExpertRatingsPromise();
    },
    defaultValue: {}
  });

  readonly loading = computed(() => this.#ratingsResource.isLoading() || this.#dessertsResource.isLoading());

  readonly desserts = this.#dessertsResource.value;
  readonly ratings = this.#ratingsResource.value;
  readonly ratedDesserts = computed(() => toRated(this.desserts(), this.ratings()));
  readonly error = computed(() => getErrorMessage(this.dessertsResource.error() || this.ratingsResource.error()));

  loadRatings(): void {
    this.#ratingsRequested.set(true);
    this.#ratingsResource.reload();
  }

  updateRating(id: number, rating: number): void {
    this.#ratingsResource.update((ratings) => ({
      ...ratings,
      [id]: rating,
    }));
  }
}
```

Здесь представлен компонент, потребляющий стор:

```typescript
@Component([...])
export class DessertsComponent {
  #store = inject(DessertStore);

  originalName = this.#store.originalName;
  englishName = this.#store.englishName;
  loading = this.#store.loading;
  error = this.#store.error;

  ratedDesserts = this.#store.ratedDesserts;

  loadRatings(): void {
    this.#store.loadRatings();
  }

  updateRating(id: number, rating: number): void {
    this.#store.updateRating(id, rating);
  }
}
```

В этом примере однонаправленный поток данных виден достаточно отчётливо: компонент отправляет наверх свои намерения, вызывая методы, и получает (обновлённые) данные через сигналы, привязанные в шаблоне.

## Обновление ресурсов

Ресурсы не являются исключительно read-only. Их можно обновлять новым значением. Хотя обратную запись изменённых значений приходится выполнять вручную, локальное обновление ресурса гарантирует, что новое значение станет частью реактивного потока. То есть оно будет отображено и спроецировано и станет основой для дальнейших обновлений.

В следующем примере показан простой сервис, выступающий стором для представления с деталями. Особое внимание обратите на метод `save`:

```typescript
@Injectable({ providedIn: 'root' })
export class DessertDetailStore {
  #dessertService = inject(DessertService);
  #id = signal<number | undefined>(undefined);

  #dessertResource = resource({
    params: computed(() => ({ id: this.#id() })),
    loader: async (loaderParams) => {
      const id = loaderParams.params.id;
      if (id) {
        const result = await this.#dessertService.findPromiseById(id);
        return result ?? initDessert;
      }
      else {
        return Promise.resolve(initDessert);
      }
    },
    defaultValue: initDessert
  });

  readonly loading = this.#dessertResource.isLoading;
  readonly error = this.#dessertResource.error;

  #saving = signal(false);

  load(id: number): void {
    this.#id.set(id);
  }

  save(dessert: Dessert): void {
    try {
      this.#saving.set(true);
      console.log('saving', dessert);
      // Здесь был бы ваш HTTP-вызов
      [...]
      this.#dessertResource.set(dessert);
    }
    finally {
      this.#saving.set(false);
    }    
  }
}
```

Метод `save` получает обновлённый десерт и записывает его обратно на сервер. Соответствующий HTTP-вызов здесь обозначен только комментарием. После выполнения этого вызова `dessertResource` обновляется новым значением посредством `set`. В качестве альтернативы существует метод `update`, позволяющий спроецировать текущее значение в новое. Если во время обновления ресурса таким локальным значением выполняется процесс загрузки, этот процесс прерывается.

В примере выше `resource` был приватным для сервиса. Благодаря этому сервис может гарантировать корректность обновления ресурса. Если требуется выставить весь `resource` наружу, можно использовать его метод `asReadonly`, чтобы воспрепятствовать изменению управляемых данных потребителями:

```typescript
readonly dessert = dessertResource.asReadonly();
```

## Интерлюдия 2: linkedSignal для формы с деталями

При привязке загруженного `dessert` к шаблонной форме (template-driven form) требуются записываемые (writable) сигналы. Однако сторы обычно предоставляют сигналы только для чтения. Также сигнал `value`, предоставляемый `resource`, доступен только для чтения, хотя его можно изменять напрямую через методы `set` и `update` ресурса.

Как уже отмечено выше, это удачный случай для `linkedSignal`:

```typescript
@Component({
  selector: 'app-dessert-detail',
  standalone: true,
  imports: [JsonPipe, RouterLink, FormsModule],
  templateUrl: './dessert-detail.component.html',
  styleUrl: './dessert-detail.component.css'
})
export class DessertDetailComponent implements OnChanges {
  store = inject(DessertDetailStore);

  id = input.required({
    transform: numberAttribute
  });

  loadedDessert = this.store.dessert;
  loading = this.store.loading;
  error = this.store.error;

  dessert = {
    originalName: linkedSignal(() => this.loadedDessert().originalName),
    englishName: linkedSignal(() => this.loadedDessert().englishName),
    kcal: linkedSignal(() => this.loadedDessert().kcal)
  };

  ngOnChanges(): void {
    const id = this.id();
    this.store.load(id);
  }

  save(): void {
    const dessert = {
      ...this.loadedDessert(),
      originalName: this.dessert.originalName(),
      englishName: this.dessert.englishName(),
      kcal: this.dessert.kcal(),    
    };

    this.store.save(dessert);
  }
}
```

Здесь каждое свойство, привязываемое к полю формы, представлено связанным сигналом (linked Signal). Они обновляются всякий раз при изменении их источника. При этом их можно обновлять локальным значением. Метод `save` берёт это локальное значение и записывает его обратно в стор, который делегирует работу `resource`.

В результате можно напрямую привязать ngModel к нашим сигналам:

```html
<form>
  <div>
    <label for="englishName"> English Name </label>
    <input name="englishName" [(ngModel)]="dessert.englishName" />
  </div>
  <div>
    <label for="originalName"> OriginalName </label>
    <input name="originalName" [(ngModel)]="dessert.originalName" />
  </div>
  <div>
    <label for="kcal"> kcal </label>
    <input name="kcal" [(ngModel)]="dessert.kcal" />
  </div>
</form>
```

Эти связанные сигналы можно рассматривать как аналог объектов `FormControl` из реактивных форм. Разумеется, объекты `FormControl` предоставляют больше возможностей, например регистрацию валидаторов. Но (это моё личное впечатление) это показывает, что с сигналами оба подхода не так уж далеки друг от друга.

Хотя в этом примере я выбрал шаблонные формы, вполне можно задействовать и реактивные формы. В этом случае может потребоваться `effect` для связывания сигналов, полученных из стора, с вашими `FormControl`.

## Бонус: вспомогательные функции для упрощения

При реализации показанного здесь демо-приложения я также написал ряд реактивных помощников, упрощающих работу с сигналами и ресурсами. Они доступны в предоставленном демо-репозитории в ветке `07c-final`. Один из них — `deepLink`: он берёт сигнал с объектом и преобразует все его свойства в связанные сигналы:

```typescript
dessert = deepLink(this.loadedDessert);
```

Это избавляет от необходимости вызывать `linkedSignal` для каждого свойства по отдельности, как в примере из предыдущего раздела.

Также я хотел предотвратить появление индикатора загрузки при быстрых операциях загрузки. Поэтому мой помощник `debounceTrue` использует ресурс для дебаунсинга перехода из `false` в `true`:

```typescript
loading = debounceTrue(() => this.ratingsResource.isLoading() || this.dessertsResource.isLoading(), 500);
```

Переход из `true` в `false` он не дебаунсит, поскольку обычно требуется, чтобы индикатор загрузки исчезал немедленно, как только загруженный ресурс становится доступен.

## Заключение

Новый Resource API действительно является недостающим звеном в истории сигналов в Angular: он даёт официальный способ загрузки асинхронных ресурсов непосредственно внутри реактивного потока. Кроме того, он берёт на себя обработку гонок запросов и содержит базовую обработку ошибок.

С этим новым API мы получаем простую в использовании встроенную возможность для весьма распространённых сценариев. Для более продвинутых сценариев, например работы с несколькими параллельными потоками данных, можно переключиться на что-то более мощное, такое как RxJS. Благодаря интеропу с RxJS и `rxResource` оба мира могут быть соединены.
