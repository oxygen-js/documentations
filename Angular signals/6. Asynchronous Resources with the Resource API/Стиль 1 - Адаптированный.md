Экспериментальный Resource API, который появился в Angular 19 и был переработан в Angular 20, позволяет асинхронно загружать ресурсы с помощью сигналов (Signals). Типичный сценарий применения `resource` — загрузка данных. В этой статье я покажу, как собрать классический CRUD-сценарий на новом API `resource`.

## Первые шаги с Resource API

У каждого `resource` есть функция-загрузчик `loader`, которая возвращает `Promise` с загруженными данными. Этот `loader` срабатывает, как только ресурс инициализирован. По желанию у ресурса может быть сигнал `params` с параметрами (критериями поиска) для `loader`. Каждый раз, когда сигнал `params` меняется, загрузчик запускается снова:

```typescript
@Component([...])
export class DessertsComponent {
  #dessertService = inject(DessertService);

  [...]

  // Критерии поиска
  originalName = signal('');
  englishName = signal('');

  // Объединяем критерии в computed-сигнал
  dessertsCriteria = computed(() => ({
    originalName: this.originalName(),
    englishName: this.englishName(),
  }));

  // Определяем resource с params (= критериями поиска) и loader.
  // Каждый раз при изменении params срабатывает loader
  dessertsResource = resource({
    params: this.dessertsCriteria,
    loader: (loaderParams) => {
      return this.#dessertService.findPromise(loaderParams.params);
    },
    defaultValue: []
  });

  // изначально ресурсы имеют значение undefined
  desserts = this.dessertsResource.value;

  loading = this.dessertsResource.isLoading;
  error = this.dessertsResource.error;

  // Реактивный поток продолжается...
  ratings = signal<DessertIdToRatingMap>({});
  ratedDesserts = computed(() => this.toRated(this.desserts(), this.ratings()));

  [...]
}
```

В этом примере `dessertsResource` использует значения сигналов `originalName` и `englishName` в качестве параметров. Загрузчик срабатывает сразу при инициализации и при каждом изменении этих сигналов. Объект `param`, который передаётся в `loader`, содержит текущие критерии поиска из сигнала `params`.

Результат работы `resource` находится в его сигнале `value`. По умолчанию сигнал value ресурса равен `undefined`, пока загрузчик асинхронно не вернёт первое значение. Чтобы этого избежать, в примере выше для `defaultValue` задан пустой массив. Булев сигнал `isLoaded` сообщает о состоянии загрузки, а `error` — это сигнал с возможной ошибкой, возникшей во время загрузки.

Computed-сигнал `ratedDesserts` объединяет полученные десерты с оценками, которые, возможно, были загружены. Это показывает, что у нас выстраивается понятный реактивный поток: от ввода пользователя — через загрузку ресурса — к проекции ресурса в модель, привязанную в шаблоне. Одно действие реактивно влечёт за собой следующее.

## Важно: загрузчики не отслеживаются!

Важно понимать: хотя `params` отслеживаются, сам `loader` — нет. Это значит, что изменение сигнала `params` запускает новое выполнение `loader`, а вот изменение какого-либо сигнала (`Signal`), использованного внутри `loader`, — **нет**.

Дело в том, что автоматическое отслеживание сработало бы только для первой части `loader`, которая выполняется синхронно. Всё, что выполняется после первой асинхронной операции, например код после `await` или внутри обработчика `.then`, под автоматическое отслеживание Angular попасть уже не может. Чтобы не создавать запутанных ситуаций, когда одна часть загрузчика обрабатывается иначе, чем другая, весь загрузчик объявлен неотслеживаемым (untracked).

## Гонки запросов (race conditions)

Во многих веб-приложениях пользователь легко может запустить пересекающиеся запросы. Особенно это касается реактивного UI, где простое изменение фильтра уже может запустить новую загрузку:

![Пересекающиеся запросы](https://www.angulararchitects.io/wp-content/uploads/2024/10/overlapping.png)

В этом случае пользователь ожидает увидеть результаты только для «Ice Cream Pancakes», хотя незадолго до этого фильтры были другими. Было бы довольно странно, если бы на мгновение мелькнули результаты для обычных «Pancakes» и «Sacher Cake». Ещё сильнее запутало бы, если первый поиск занял бы чуть больше времени:

![Пересекающиеся запросы](https://www.angulararchitects.io/wp-content/uploads/2024/10/race-condition.png)

В этом случае нежелательные промежуточные результаты мелькнули бы в порядке, который не соответствует порядку запросов. Это и называется гонкой запросов (race condition).

## Resource API ведёт себя как switchMap

В Angular мы обычно полагаемся на RxJS и его оператор `switchMap`, который отменяет все пересекающиеся запросы, кроме последнего, и тем самым предотвращает подобные ситуации. Хорошая новость в том, что `resource` ведёт себя точно так же. Правда, по умолчанию он не может отменить предыдущий запрос — он просто игнорирует его результат.

Если же вы действительно хотите отменять предыдущий запрос, когда планируется новый, нужно учитывать `AbortSignal`, который Resource API передаёт в `loader`:

```typescript
[...]
dessertsResource = resource({
  params: this.dessertsCriteria,
  loader: (loaderParams) => {
    return this.#dessertService.findPromise(
      loaderParams.params,
      loaderParams.abortSignal,
    );
  },
  defaultValue: []
});
[...]
```

Хотя он называется `AbortSignal`, это не сигнал в смысле примитивов реактивности Angular, а JavaScript-API, который есть во всех современных браузерах и поддерживается рядом интерфейсов, например Fetch API. Хотя `HttpClient` в Angular уже умеет использовать `fetch` под капотом, его API не позволяет передать `AbortSignal`. Чтобы это компенсировать, я написал простую вспомогательную функцию `toPromise`, которая преобразует `Observable` в `Promise` с учётом `AbortSignal`. Эта реализация, вдохновлённая функцией `rxResource` (о ней ниже), есть в демо-репозитории.

Хотя семантика `switchMap`, пожалуй, наиболее распространена для загрузки данных, RxJS предлагает и другие так называемые операторы сведения (flattening operators), которые по-разному обрабатывают пересекающиеся запросы. Но для описанных операций `resource` всегда использует семантику `switchMap`. Кроме того, метод `reload`, о котором речь пойдёт ниже, всегда использует семантику `exhaustMap`.

Это показывает, что команда Angular не пытается конкурировать с существующими библиотеками. Цель — дать строительные блоки с разумной логикой по умолчанию, покрывающей большинство сценариев. Для всего, что выходит за эти рамки, мы можем реализовать собственные решения или подключить проверенные библиотеки вроде RxJS.

## Дебаунсинг

Обычно, когда запросы запускаются сразу после того, как пользователь изменил фильтр, хочется добавить немного дебаунсинга (debouncing): велика вероятность, что пользователь поменяет ещё пару фильтров, поэтому стоит немного подождать, чтобы не запускать несколько лишних запросов подряд.

Самый простой способ добиться дебаунсинга — добавить задержку в загрузчик. Но как следствие, во время дебаунсинга ресурс уже будет находиться в состоянии загрузки. Впрочем, в хорошей дискуссии с коллегами из числа Angular GDE и членами команды Angular ([Rainer Hahnekamp](https://bsky.app/profile/rainerhahnekamp.bsky.social), [Deborah Kurata](https://bsky.app/profile/deborahkurata.bsky.social), [Sander Elias](https://bsky.app/profile/sanderelias.nl), [Matthieu Riegler](https://bsky.app/profile/jeanmeche.com) и [Alex Rickabaugh](https://bsky.app/profile/synalx.bsky.social), если назвать лишь некоторых) мы пришли к выводу, что лучше дебаунсить само событие, которое приводит к новому запросу. В пользу этого есть несколько аргументов. Один из них: дебаунсинг исходного события не даёт ресурсу перейти в состояние загрузки до того, как дебаунсинг завершится. Другой — это позволяет вынести ресурс в стор (store), который используется независимо от того, дебаунсится ввод или нет.

Дебаунсить исходное событие просто, если вы используете Reactive Forms: в этом случае можно дебаунсить Observable `valueChanges`. А вот когда мы используем Template-driven Forms с two-way bindings, нам нужен помощник, который дебаунсит привязанный сигнал. Для этого я написал простой помощник `debounceSignal`:

```typescript
#debouncedCriteria = debounceSignal(this.#dessertsCriteria, 300);

#dessertsResource = resource({
  params: this.#debouncedCriteria,
  loader: (loaderParams) => {
    return this.#dessertService.findPromise(loaderParams.params, param.abortSignal);
  },
  defaultValue: []
});
```

Для простой реализации debounceSignal можно воспользоваться интеропом Angular с RxJS:

```typescript
import { Signal } from "@angular/core";
import { toObservable, toSignal } from "@angular/core/rxjs-interop";
import { debounceTime } from "rxjs";

export function debounceSignal<T>(source: Signal<T>, timeMsec: number): Signal<T | undefined> {
    return toSignal(toObservable(source).pipe(debounceTime(timeMsec)));
}
```

Учтите, что итоговый сигнал будет равен `undefined`, пока не отработает первый дебаунсинг. Сигнал `params`, возвращающий `undefined`, не даёт ресурсу запустить загрузчик — и во многих случаях именно такое поведение и нужно. Если же вы хотите пропустить начальный дебаунсинг, в качестве значения по умолчанию можно взять начальное значение сигнала:

```typescript
export function debounceSignal<T>(source: Signal<T>, timeMsec: number): Signal<T> {
    return toSignal(toObservable(source).pipe(debounceTime(timeMsec)), {
        initialValue: source()
    });
}
```

Конечно, как всегда, можно сделать и реализацию без RxJS. Это может быть интересно в средне- и долгосрочной перспективе, когда RxJS в Angular станет необязательным. Но в любом случае подобный функционал я бы рассматривал как инфраструктурный код, которому место не в коде приложения, а в самом фреймворке, в библиотеке-помощнике или хотя бы в служебном модуле, который приложение воспринимает как «чёрный ящик». Посмотрим, предложат ли будущие формы на сигналах (Signal-based Forms) удобное решение для таких сценариев.

## Перезагрузка и ручная загрузка

Использованный до сих пор `resource` автоматически запускал `loader` после инициализации и после каждого изменения сигнала `params`. В нашем примере это было удобно. Но это не то поведение, которое нужно всегда.

Если вы хотите отключить начальное выполнение загрузчика, просто сделайте так, чтобы сигнал `params` возвращал `undefined`:

```typescript
type Requested = undefined | true;

[...]

#ratingsRequested = signal<Requested>(undefined);

[...]

#ratingsResource = resource({
  params: this.#ratingsRequested,
  loader: () => {
    return this.#ratingService.loadExpertRatingsPromise();
  },
  defaultValue: {}
});

readonly ratings = this.#ratingsResource.value;

[...]
```

Как уже говорилось выше, сигнал `params`, возвращающий `undefined`, специально не запускает загрузчик. В нашем случае нет параметров, влияющих на то, какие оценки загружать, поэтому я просто обхожусь значениями `true` или `undefined`.

Чтобы запустить загрузчик в первый раз, теперь можно установить сигнал `params` в `true`. А чтобы запустить загрузчик ещё раз позже, приложение вызывает метод `reload` у ресурса:

```typescript
loadRatings(): void {
  this.#ratingsRequested.set(true);
  this.#ratingsResource.reload();
}
```

Метод `reload` предотвращает пересекающиеся запросы: если запрос уже выполняется, он сразу же завершается без действия. На языке RxJS это можно описать как семантику `exhaustMap`. Благодаря этому нам не нужно различать, первый это запрос или последующий. А это, в свою очередь, упрощает реализацию `loadRatings`.

Признаю, использование типа `undefined | true` может показаться немного странным. Но это позволяет применять Resource так, как и задумано: всё крутится вокруг реактивной загрузки данных. Этот пример выглядел бы естественнее, будь у ресурса настоящий параметр, например `expertId`.

## Интерлюдия 1: linkedSignal для обновлений

Пользователь может менять оценки. Обычно это нормально, потому что ресурс можно обновлять вручную методами `set` и `update`, которые рассматриваются ниже. Однако у `resource` сигнал `value` по умолчанию равен `undefined`. По крайней мере с тех пор, как Angular-приложения по умолчанию используют строгий режим (strict mode) TypeScript, я стараюсь избегать `null` и `undefined` в пользу объекта по умолчанию, также известного как Null Object. До сих пор я использовал `computed`, чтобы проецировать `null` и `undefined` в соответствующий Null Object:

```typescript
ratings = computed(() => this.ratingsResource.value() ?? {});
```

К сожалению, computed-сигнал доступен только для чтения. Спасает функция `linkedSignal`. Она даёт computed-сигнал, который при этом можно менять:

```typescript
ratings = linkedSignal(() => this.ratingsResource.value() ?? {});
```

Если сигналы, использованные в вычислении, меняются, вычисление запускается заново, и присвоенное напрямую значение перезаписывается. Для полноты картины упомяну, что вычисление происходит лениво: если значение никому не нужно, повторного вычисления не будет.

Эта особенность делает `linkedSignal` идеальным выбором для форм: мы можем менять локальную копию и по требованию отправлять её обратно для сохранения.

## Обработка ошибок

Обработка ошибок тоже встроена в Resource API. Когда загрузчик выбрасывает исключение или отклоняет возвращаемый `Promise`, `resource` переходит в состояние `error`. В этом состоянии сигнал `error` предоставляет выброшенное значение или значение, переданное в reject-функцию `Promise`.

Тем не менее `resource` продолжит работать: если кто-то вызовет метод `reload` или если сигнал `params` изменится, `loader` выполнится снова. Если `loader` отработает успешно, ресурс очистит сигнал `error` и перейдёт в состояние `resolved`.

Это сопоставимо с нынешним поведением по умолчанию у Effects в NgRx, где дополнительная логика под капотом не даёт используемому RxJS-пайплайну остановиться при ошибке.

Однако начиная с Angular 20 нельзя обращаться к `value` ресурса при наличии ошибки. Поэтому в шаблоне сначала нужно проверить состояние `error`:

```typescript
@if(!error()) {
  @for (dessert of ratedDesserts(); track dessert.id) {
    <app-dessert-card
      [dessert]="dessert"
      (ratingChange)="updateRating(dessert.id, $event)"
    ></app-dessert-card>
  }

}
@else {
  <b>Error loading desserts!</b>
}
```

В этом случае `ratedDesserts` выводится из значения ресурса через `computed`. Поэтому и здесь нам приходится проверять состояние ошибки.

## rxResource для интеропа с RxJS

Если у вас уже есть Observable, преобразовывать его в `Promise` не нужно. Вместо этого можно воспользоваться API `rxResource`. Он работает как `resource`, но загрузчик должен возвращать `Observable`. Кроме того, загрузчик определяется через свойство streaming, потому что `rxResource` — это так называемый потоковый ресурс, который может выдавать несколько значений со временем.

```typescript
[...]

dessertsResource = rxResource({
  params: this.dessertsCriteria,
  stream: (loaderParams) => {
    return timer(300).pipe(switchMap(() => this.#dessertService.find(loaderParams.params)));
  },
  defaultValue: []
});

[...]

ratingsResource = rxResource({
  params: this.#ratingsRequested,
  stream: () => {
    return this.#ratingService.loadExpertRatings()
  },
  defaultValue: {}
});

[...]
```

Кроме того, загрузчику не нужен `AbortSignal`, поскольку Observable в принципе можно прервать (неявно или явно отписавшись).

## Resource в сервисах и сторах

В показанных примерах `resource` использовался прямо в компоненте. Это было сделано в первую очередь для простоты. Зачастую же вы будете использовать ресурсы в сервисах или сторах. Одна из причин — управление состоянием: загруженные данные переживут момент, когда Angular уничтожит текущий компонент (например, из-за перехода на другой маршрут), и их можно будет использовать позже в экземпляре того же или другого компонента.

Вторая причина в том, что сторы помогают упорядочить ваш реактивный поток данных:

![](https://www.angulararchitects.io/wp-content/uploads/2024/10/store.png)

В результате получается так называемый однонаправленный поток данных: компоненты отправляют в стор намерения (intentions). Я использую термин «намерение» в абстрактном смысле, потому что то, как оно выражается, зависит от стора. В Redux-сторе это будет диспетчеризованный action; в более лёгком сторе вроде NGRX Signal Store это может быть просто вызванный метод.

Выраженное намерение заставляет стор выполнить некоторые задачи, в результате которых в сигналы записываются новые значения. Эти сигналы можно спроецировать через `computed` и передать данные вниз, в шаблон компонента.

Очень простой стор — это сервис, предоставляющий сигналы. То есть мы можем перенести рассмотренный код в сервис:

```typescript
type Requested = undefined | true;

@Injectable({ providedIn: 'root' })
export class DessertStore {
  #dessertService = inject(DessertService);
  #ratingService = inject(RatingService);

  private #ratingsRequested = signal<Requested>(undefined);

  readonly originalName = signal('');
  readonly englishName = signal('');

  #dessertsCriteria = computed(() => ({
    originalName: this.originalName(),
    englishName: this.englishName(),
  }));

  #dessertsResource = resource({
    params: this.#dessertsCriteria,
    loader: debounce((loaderParams) => {
      return this.#dessertService.findPromise(loaderParams.params, loaderParams.abortSignal);
    }),
    defaultValue: []
  });

  #ratingsResource = resource({
    params: this.#ratingsRequested,
    loader: () => {
      return this.#ratingService.loadExpertRatingsPromise();
    },
    defaultValue: {}
  });

  readonly loading = computed(() => this.#ratingsResource.isLoading() || this.#dessertsResource.isLoading());

  readonly desserts = this.#dessertsResource.value;
  readonly ratings = this.#ratingsResource.value;
  readonly ratedDesserts = computed(() => toRated(this.desserts(), this.ratings()));
  readonly error = computed(() => getErrorMessage(this.dessertsResource.error() || this.ratingsResource.error()));

  loadRatings(): void {
    this.#ratingsRequested.set(true);
    this.#ratingsResource.reload();
  }

  updateRating(id: number, rating: number): void {
    this.#ratingsResource.update((ratings) => ({
      ...ratings,
      [id]: rating,
    }));
  }
}
```

А вот компонент, который потребляет этот стор:

```typescript
@Component([...])
export class DessertsComponent {
  #store = inject(DessertStore);

  originalName = this.#store.originalName;
  englishName = this.#store.englishName;
  loading = this.#store.loading;
  error = this.#store.error;

  ratedDesserts = this.#store.ratedDesserts;

  loadRatings(): void {
    this.#store.loadRatings();
  }

  updateRating(id: number, rating: number): void {
    this.#store.updateRating(id, rating);
  }
}
```

В этом примере однонаправленный поток данных виден довольно ясно: компонент отправляет наверх свои намерения, вызывая методы, и получает (обновлённые) данные через сигналы, привязанные в шаблоне.

## Обновление ресурсов

Ресурсы не только для чтения. Мы также можем обновлять их новым значением. Хотя обратную запись изменённых значений приходится делать вручную, локальное обновление ресурса гарантирует, что новое значение станет частью нашего реактивного потока. То есть оно будет отображено и спроецировано и станет основой для дальнейших обновлений.

В следующем примере показан простой сервис, выступающий в роли стора для страницы с подробностями. Обратите особое внимание на метод `save`:

```typescript
@Injectable({ providedIn: 'root' })
export class DessertDetailStore {
  #dessertService = inject(DessertService);
  #id = signal<number | undefined>(undefined);

  #dessertResource = resource({
    params: computed(() => ({ id: this.#id() })),
    loader: async (loaderParams) => {
      const id = loaderParams.params.id;
      if (id) {
        const result = await this.#dessertService.findPromiseById(id);
        return result ?? initDessert;
      }
      else {
        return Promise.resolve(initDessert);
      }
    },
    defaultValue: initDessert
  });

  readonly loading = this.#dessertResource.isLoading;
  readonly error = this.#dessertResource.error;

  #saving = signal(false);

  load(id: number): void {
    this.#id.set(id);
  }

  save(dessert: Dessert): void {
    try {
      this.#saving.set(true);
      console.log('saving', dessert);
      // Здесь был бы ваш HTTP-вызов
      [...]
      this.#dessertResource.set(dessert);
    }
    finally {
      this.#saving.set(false);
    }    
  }
}
```

Метод `save` получает обновлённый десерт и записывает его обратно на сервер. Соответствующий HTTP-вызов здесь обозначен лишь комментарием. После выполнения этого вызова `dessertResource` обновляется новым значением через `set`. Как альтернатива, есть и метод `update`, который позволяет спроецировать текущее значение в новое. Если во время обновления ресурса таким локальным значением идёт процесс загрузки, этот процесс прерывается.

В примере выше `resource` был приватным внутри сервиса. Благодаря этому сервис может гарантировать, что ресурс обновляется только корректно. Если же вы хотите выставить весь `resource` наружу, можно воспользоваться его методом `asReadonly`, чтобы потребители не могли менять управляемые данные:

```typescript
readonly dessert = dessertResource.asReadonly();
```

## Интерлюдия 2: linkedSignal для формы с подробностями

Когда мы привязываем загруженный `dessert` к шаблонной форме (template-driven form), нам нужны записываемые (writable) сигналы. Но сторы обычно предоставляют сигналы только для чтения. Да и сигнал `value`, предоставляемый `resource`, доступен только для чтения, хотя его можно менять напрямую через методы `set` и `update` ресурса.

Как уже упоминалось выше, это отличный случай для `linkedSignal`:

```typescript
@Component({
  selector: 'app-dessert-detail',
  standalone: true,
  imports: [JsonPipe, RouterLink, FormsModule],
  templateUrl: './dessert-detail.component.html',
  styleUrl: './dessert-detail.component.css'
})
export class DessertDetailComponent implements OnChanges {
  store = inject(DessertDetailStore);

  id = input.required({
    transform: numberAttribute
  });

  loadedDessert = this.store.dessert;
  loading = this.store.loading;
  error = this.store.error;

  dessert = {
    originalName: linkedSignal(() => this.loadedDessert().originalName),
    englishName: linkedSignal(() => this.loadedDessert().englishName),
    kcal: linkedSignal(() => this.loadedDessert().kcal)
  };

  ngOnChanges(): void {
    const id = this.id();
    this.store.load(id);
  }

  save(): void {
    const dessert = {
      ...this.loadedDessert(),
      originalName: this.dessert.originalName(),
      englishName: this.dessert.englishName(),
      kcal: this.dessert.kcal(),    
    };

    this.store.save(dessert);
  }
}
```

Здесь каждое свойство, которое мы хотим привязать к полю формы, представлено связанным сигналом (linked Signal). Они обновляются всякий раз, когда меняется их источник. И при этом их можно обновлять локальным значением. Метод `save` берёт это локальное значение и записывает его обратно в стор, который делегирует работу `resource`.

В результате мы можем напрямую привязать ngModel к нашим сигналам:

```html
<form>
  <div>
    <label for="englishName"> English Name </label>
    <input name="englishName" [(ngModel)]="dessert.englishName" />
  </div>
  <div>
    <label for="originalName"> OriginalName </label>
    <input name="originalName" [(ngModel)]="dessert.originalName" />
  </div>
  <div>
    <label for="kcal"> kcal </label>
    <input name="kcal" [(ngModel)]="dessert.kcal" />
  </div>
</form>
```

Эти связанные сигналы можно воспринимать как аналог объектов `FormControl` из реактивных форм. Конечно, объекты `FormControl` дают больше возможностей, например регистрацию валидаторов. Но — и это моё личное впечатление — это показывает, что с сигналами оба подхода не так уж далеки друг от друга.

Хотя в этом примере я выбрал шаблонные формы, вы вполне можете использовать и реактивные формы. В этом случае вам, возможно, захочется применить `effect`, чтобы связать сигналы, полученные из стора, с вашими `FormControl`.

## Бонус: вспомогательные функции для упрощения

Реализуя показанное здесь демо-приложение, я также написал несколько реактивных помощников, упрощающих работу с сигналами и ресурсами. Вы найдёте их в демо-репозитории в ветке `07c-final`. Один из них — `deepLink`: он берёт сигнал с объектом и превращает все его свойства в связанные сигналы:

```typescript
dessert = deepLink(this.loadedDessert);
```

Это избавляет нас от необходимости вызывать `linkedSignal` отдельно для каждого свойства, как в примере из предыдущего раздела.

Кроме того, я хотел, чтобы индикатор загрузки не появлялся при быстрых операциях загрузки. Поэтому мой помощник `debounceTrue` использует ресурс, чтобы дебаунсить переход из `false` в `true`:

```typescript
loading = debounceTrue(() => this.ratingsResource.isLoading() || this.dessertsResource.isLoading(), 500);
```

При этом он не дебаунсит переход из `true` в `false`, потому что обычно мы хотим, чтобы индикатор загрузки исчезал сразу же, как только загруженный ресурс становится доступен.

## Заключение

Новый Resource API действительно оказался недостающим звеном в истории сигналов в Angular: он даёт нам официальный способ загружать асинхронные ресурсы прямо внутри реактивного потока. Кроме того, он берёт на себя гонки запросов и содержит базовую обработку ошибок.

С этим новым API у нас появляется простая в использовании встроенная возможность для очень распространённых сценариев. Для более продвинутых случаев, например для работы с несколькими параллельными потоками данных, мы можем переключиться на что-то более мощное вроде RxJS. Благодаря интеропу с RxJS и `rxResource` оба мира можно соединить.
