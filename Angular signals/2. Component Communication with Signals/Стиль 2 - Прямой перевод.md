## Входные сигналы (Input Signals)

Входные сигналы позволяют получать данные через связывания свойств (Property Bindings) в виде сигналов. Статья демонстрирует это на примере простого `OptionComponent`.

### Определение входного сигнала

```typescript
@Component({
  selector: 'app-option',
  standalone: true,
  imports: [],
  template: `
    <div class="option">
      {{ label() }}
    </div>
  `,
  styles: [...]
})
export class OptionComponent {
  label = input.required<string>();
}
```

Функция `input` подхватывается компилятором Angular, который генерирует исходный код для связываний свойств. `InputSignal` всегда доступен только для чтения и может использоваться как `Signal`.

### Привязка к входному сигналу

```typescript
<app-option label="Option #1">
<app-option [label]="myStringProperty">
<app-option [label]="mySignalProperty()">
```

### Вычисляемые сигналы (Computed Signals) и эффекты (Effects) как замена хуков жизненного цикла

```typescript
markDownTitle = computed(() => '# ' + this.label())

constructor() {
  effect(() => {
    console.log('label updated', this.label());
    console.log('markdown', this.markDownTitle());
  });
}
```

### Опции для входных сигналов

| Исходный код | Описание |
|---|---|
| `label = input();` | Необязательное свойство, представленное `InputSignal<string \| undefined>` |
| `label = input('Hello');` | Необязательное свойство, представленное `InputSignal<string>` с начальным значением `Hello` |
| `label = input<string \| undefined>('Hello');` | Необязательное свойство, представленное `InputSignal<string \| undefined>` с начальным значением `Hello` |

### Обязательные входы (Required Inputs) не могут иметь значения по умолчанию!

По определению `input.required` не может иметь значения по умолчанию. Если попытаться прочитать значение обязательного входа до того, как оно было привязано, Angular выбрасывает исключение. Вы не можете обратиться к нему напрямую в конструкторе. Вместо этого можно использовать `ngOnInit` или `ngOnChanges`. Использование входов внутри `computed` или `effect` всегда безопасно:

```typescript
@Component([...])
export class OptionComponent implements OnInit, OnChanges {
  label = input.required<string>();

  // безопасно
  markDownTitle = computed(() => '# ' + this.label())

  constructor() {
    // это вызвало бы исключение,
    // так как данные ещё не были привязаны
    console.log('label', this.label);

    effect(() => {
        // безопасно
        console.log('label', this.label);
    })
  }

  ngOnInit() {
    // безопасно
    console.log('label', this.label);
  }

  ngOnChanges() {
    // безопасно
    console.log('label', this.label);
  }
}
```

### Псевдонимы (Aliases) для входных сигналов

И `input`, и `input.require` также принимают объект параметров, который позволяет определить `alias` (псевдоним):

```typescript
label = input.required({ alias: "title" });
```

В этом случае вызывающая сторона должна привязываться к имени свойства, определённому псевдонимом:

```typescript
<app-option title="Option #1">
  <app-option [title]="myStringProperty">
    <app-option [title]="mySignalProperty()"></app-option>
  </app-option>
</app-option>
```

В большинстве случаев следует избегать использования псевдонимов, так как они создают ненужную косвенность.

### Преобразователи (Transformers) для входных сигналов

Преобразователи позволяют преобразовывать значение, переданное через связывание свойства:

```typescript
@Component({
  selector: 'app-option',
  standalone: true,
  imports: [],
  template: `
    <div class="option">
      {{ label() }} @if (featured()) { ⭐ }
    </div>
  `,
  styles: [...]
})
export class OptionComponent {
  label = input.required<string>();
  featured = input.required({
    transform: booleanAttribute
  })
}
```

Этот преобразователь конвертирует строки в булевы значения:

```typescript
<app-option label="Option #1" featured="true"></app-option>
```

Кроме того, если атрибут присутствует, но значение ему не присвоено, предполагается `true`:

```typescript
<app-option label="Option #1" featured></app-option>
```

Тип этого сигнала — `InputSignal<boolean, unknown>`. Первый параметр типа (`boolean`) представляет значение, полученное от преобразователя; второй (`unknown`) — это значение, привязанное в шаблоне вызывающей стороны и переданное в преобразователь.

Для пользовательского преобразователя:

```typescript
function boolTranformer(value: unknown): boolean {
  return value !== "no";
}
```

Затем зарегистрируйте эту функцию в вашем `input`:

```typescript
@Component([...])
export class OptionComponent {
  label = input.required<string>();
  featured = input.required({
    transform: boolTranformer
  })
}
```

## Двустороннее связывание данных с модельными сигналами (Model Signals)

Входные сигналы доступны только для чтения. Если вы хотите передать сигнал, который может быть обновлён вызываемым компонентом, вам нужно настроить так называемый модельный сигнал (Model Signal). Потребитель может использовать этот компонент:

```typescript
<app-tabbed-pane [(current)]="current">
  <app-tab title="1st tab"> Lorem, ipsum dolor sit amet ... </app-tab>
  <app-tab title="2nd tab"> Sammas ergo gemma, ipsum dolor ... </app-tab>
  <app-tab title="3nd tab"> Gemma ham ipsum dolor sit ... </app-tab>
</app-tabbed-pane>

<p class="current-info">Current: {{ current() }}</p>
```

Чтобы это стало возможным, `TabbedPaneComponent` должен предоставить модельный сигнал с помощью `model`:

```typescript
@Component([...])
export class TabbedPaneComponent {
  current = model(0);
  [...]
}
```

Здесь `0` — начальное значение. Опции аналогичны опциям для input: `model.required` определяет обязательное свойство, и вы можете задать псевдоним через объект опций. Однако преобразователь определить нельзя.

Если этот компонент обновляет модельный сигнал, новое значение распространяется вверх к сигналу, привязанному в шаблоне:

```typescript
current.set(1);
```

### Двустороннее связывание данных как комбинация входа и выхода

Двусторонние связывания на основе сигналов можно определить с помощью (доступного только для чтения) входа (Input) и соответствующего выхода (Output). Имя выхода должно быть именем входа с суффиксом `Change`. Следовательно, для `current` нам нужно определить `currentChange`:

```typescript
@Component([...])
export class TabbedPaneComponent {
  current = input(0);
  currentChange = output<number>();
}
```

Для настройки выхода используется новый output API. Чтобы инициировать событие, приложение должно вызвать метод `emit` выхода:

```typescript
<button [...] (click)="currentChange.emit($index)">{{tab.title()}}</button>
```

## Запросы к содержимому (Content Queries) с сигналами

`TabbedPaneComponent` также позволяет нам продемонстрировать ещё одну возможность: запросы к содержимому (Content Queries), которые получают доступ к спроецированным компонентам или директивам.

`TabbedPaneComponent` получает несколько `TabComponent`. Они проецируются в представление `TabbedPaneComponent`. Однако в каждый момент времени мы хотим отображать только один из них. Следовательно, `TabbedPaneComponent` нужно получить программный доступ к своим `TabComponent`. Это можно сделать с помощью новой функции `contentChildren`:

```typescript
@Component({
  selector: 'app-tabbed-pane',
  standalone: true,
  imports: [],
  template: `
    <div class="pane">
      <div class="nav" role="group">
        @for(tab of tabs(); track tab) {
        <button
            [class.secondary]="tab !== currentTab()"
            (click)="activate($index)">
                {{tab.title()}}
        </button>
        }
      </div>
      <article>
        <ng-content></ng-content>
      </article>
    </div>
  `,
  styles: [...]
})
export class TabbedPaneComponent {
  current = model(0);
  tabs = contentChildren(TabComponent);
  currentTab = computed(() => this.tabs()[this.current()]);

  activate(active: number): void {
    this.current.set(active);
  }
}
```

Функция `contentChildren` является аналогом традиционного декоратора @ContentChildren. Поскольку `TabComponent` был передан как так называемый локатор (locator), она возвращает сигнал с массивом, содержащим все спроецированные `TabComponent`.

Наличие спроецированных узлов в виде сигнала позволяет нам реактивно проецировать их с помощью `computed`. Показанный пример использует эту возможность, чтобы вывести сигнал `currentTab`.

Спроецированный `TabComponent` использует этот сигнал, чтобы выяснить, должен ли он быть видимым:

```typescript
@Component({
  selector: "app-tab",
  standalone: true,
  imports: [],
  template: `
    @if(visible()) {
    <div class="tab">
      <h2>{{ title() }}</h2>
      <ng-content></ng-content>
    </div>
    }
  `,
})
export class TabComponent {
  pane = inject(TabbedPaneComponent);

  title = input.required<string>();

  visible = computed(() => this.pane.currentTab() === this);
}
```

Для этого нам нужно знать, что мы можем получить всех родителей, расположенных в DOM, через внедрение зависимостей (dependency injection). Сигнал `visible` выводится из сигнала `currentTab`.

Эта процедура обычна в реактивном мире: вместо императивной установки значений они декларативно выводятся из других значений.

### Запросы к содержимому для потомков (Descendants)

По умолчанию запрос к содержимому раскрывает только прямых дочерних элементов содержимого. «Внуки», такие как третья вкладка ниже, игнорируются:

```typescript
<app-tabbed-pane [(current)]="current">
  <app-tab title="1st tab"> Lorem, ipsum dolor sit amet ... </app-tab>
  <app-tab title="2nd tab"> Sammas ergo gemma, ipsum dolor ... </app-tab>

  <div class="danger-zone">
    <app-tab title="3nd tab">
      Here, you can delete the whole internet!
    </app-tab>
  </div>
</app-tabbed-pane>
```

Чтобы получить доступ и к таким узлам, мы можем установить опцию `descendants` в `true`:

```typescript
tabs = contentChildren(TabComponent, { descendants: true });
```

## Output API

Ради симметричности API в Angular 17.3 был представлен новый output API. Теперь для определения события, предоставляемого компонентом, используется функция `output`. Подобно новому input API, компилятор Angular подхватывает вызов `output` и генерирует соответствующий код. Метод `emit` возвращённого `OutputEmitterRef` используется для инициирования события:

```typescript
@Component([...])
export class TabbedPaneComponent {
  current = model(0);
  tabs = contentChildren(TabComponent);
  currentTab = computed(() => this.tabs()[this.current()]);

  tabActivated = output<TabActivatedEvent>();

  activate(active: number): void {
    const previous = this.current();
    this.current.set(active);
    this.tabActivated.emit({ previous, active });
  }
}
```

### Предоставление Observable в качестве выходов

Помимо этого простого способа настройки выходов, вы можете использовать Observable в качестве источника для выхода. Для этого в слое интеграции с RxJS (RxJS interop) есть функция `outputFromObservable`:

```typescript
import { 
    outputFromObservable, 
    toObservable 
} from '@angular/core/rxjs-interop';
[...]

@Component([...])
export class TabbedPaneComponent {
  current = model(0);
  tabs = contentChildren(TabComponent);
  currentTab = computed(() => this.tabs()[this.current()]);

  tabChanged$ = toObservable(this.current).pipe(
    scan(
      (acc, active) => ({ active, previous: acc.active }),
      { active: -1, previous: -1 }
    ),
    skip(1),
  );

  tabChanged = outputFromObservable(this.tabChanged$);

  activate(active: number): void {
    this.current.set(active);
  }

}
```

Функция `outputFromObservable` преобразует Observable в `OutputEmitterRef`. В показанном примере оператор `scan` запоминает предыдущую активированную вкладку, а `skip` гарантирует, что при начальной установке `current` событие не будет испущено. Последнее обеспечивает паритет возможностей с показанным ранее примером.

## Запросы к представлению (View Queries) с сигналами

В то время как запрос к содержимому (Content Query) возвращает спроецированные узлы, запрос к представлению (View Query) возвращает узлы из собственного представления. Это узлы, найденные в шаблоне соответствующего компонента. В большинстве случаев использование связывания данных вместо этого является предпочтительным решением. Однако в некоторых ситуациях требуется получить программный доступ к дочернему элементу представления.

Чтобы продемонстрировать, как запрашивать дочерние элементы представления, в статье используется простая форма для установки имени пользователя и пароля. Оба поля `input` помечены как `required`. Если при нажатии `Save` валидация не проходит, фокус должен получить первое поле с ошибкой валидации. Для этого нам нужен доступ как к директиве `NgForm`, которую `FormModule` добавляет к нашему тегу `form`, так и к DOM-узлам, представляющим поля `input`:

```typescript
@Component({
  selector: "app-form",
  standalone: true,
  imports: [FormsModule, JsonPipe],
  template: `
    <h1>Form Demo</h1>
    <form autocomplete="off">
      <input
        [(ngModel)]="userName"
        placeholder="User Name"
        name="userName"
        #userNameCtrl
        required
      />
      <input
        [(ngModel)]="password"
        placeholder="Password"
        type="password"
        name="password"
        #passwordCtrl
        required
      />
      <button (click)="save()">Save</button>
    </form>
  `,
  styles: `
    form {
      max-width: 600px;
    }
  `,
})
export class FormDemoComponent {
  form = viewChild.required(NgForm);

  userNameCtrl =
    viewChild.required<ElementRef<HTMLInputElement>>("userNameCtrl");
  passwordCtrl =
    viewChild.required<ElementRef<HTMLInputElement>>("passwordCtrl");

  userName = signal("");
  password = signal("");

  save(): void {
    const form = this.form();

    if (form.controls["userName"].invalid) {
      this.userNameCtrl().nativeElement.focus();
      return;
    }

    if (form.controls["password"].invalid) {
      this.passwordCtrl().nativeElement.focus();
      return;
    }

    console.log("save", this.userName(), this.password());
  }
}
```

Оба запроса выполняются с помощью функции `viewChild`. В первом случае пример передаёт тип `NgForm` в качестве локатора. Однако простой поиск полей по типу не работает, так как дочерних элементов с этим типом может быть несколько. По этой причине входы помечены хэндлами (`#userName` и `#password`), и имя соответствующего хэндла передаётся как локатор.

Дочерние элементы представления могут быть представлены разными типами: типом соответствующего компонента или директивы, `ElementRef`, представляющим его DOM-узел, или `ViewContainerRef`. Последний используется в следующем разделе.

Желаемый тип можно указать с помощью опции `read`, использованной в предыдущем примере.

## Запросы и ViewContainerRef

Бывают ситуации, когда вам нужно динамически добавить компонент в заглушку (placeholder). Примеры — модальные диалоги или всплывающие уведомления (toasts). Простой способ достичь этого — использование директивы `*ngComponentOutlet`. Более гибкий способ — запросить `ViewContainerRef` заглушки.

Контейнер представления (View Container) можно представить как невидимый контейнер вокруг каждого компонента и куска статического HTML. Получив доступ к нему, вы можете добавлять дополнительные компоненты или шаблоны.

Чтобы продемонстрировать это, в статье используется простой пример, показывающий всплывающее уведомление (toast). Пример использует `ng-container` в качестве заглушки:

```typescript
@Component({
  selector: 'app-dynamic',
  standalone: true,
  imports: [],
  template: `
    <h2>Toast Demo</h2>
    <button (click)="show()">Show Toast</button>
    <ng-container #placeholder></ng-container>
  `,
  styles: [...]
})
export class ToastDemoComponent {
  counter = 0;
  placeholder = viewChild.required('placeholder', { read: ViewContainerRef });

  show() {
    const ref = this.placeholder()?.createComponent(ToastComponent);
    this.counter++;
    ref?.setInput('label', 'Message #' + this.counter);
    setTimeout(() => ref?.destroy(), 2000);
  }

}
```

Свойство `read` ясно даёт понять, что мы хотим прочитать не компонент-заглушку, а его `ViewContainerRef`. Метод `createComponent` создаёт экземпляр и добавляет `ToastComponent`. Возвращённый `ComponentRef` используется для установки свойства `label` нового компонента. Для этого используется его метод `setInput`. Через две секунды метод `destroy` снова удаляет уведомление.

Для простоты компонент `ToastComponent` здесь был зашит жёстко. В более универсальных реальных сценариях используемый компонент может настраиваться, например, вызовом метода сервиса, который принимает тип компонента и уведомляет другой компонент, добавляющий компонент этого типа в заглушку.

### Программная настройка выхода (Output)

В предыдущем примере вызывался `setInput` для присвоения значения входу `title` компонента `ToastComponent`. Здесь в статье обсуждается, как определять обработчики событий для таких динамически добавленных компонентов.

Предположим, `ToastComponent` показывает ссылку подтверждения. При клике по этой ссылке он испускает событие `confirmed`:

```typescript
@Component([...])
export class ToastComponent {
  label = input.required<string>();
  confirmed = output<string>();

  confirm(): void {
    this.confirmed.emit(this.label());
  }
}
```

Чтобы настроить обработчик этого события, мы можем напрямую использовать свойство `instance` возвращённого `ComponentRef`. Оно указывает на добавленный экземпляр компонента и, следовательно, предоставляет доступ ко всем его свойствам:

```typescript
@Component([...])
export class ToastDemoComponent {
  counter = 0;
  placeholder = viewChild.required('placeholder', { read: ViewContainerRef });

  show() {
    const ref = this.placeholder()?.createComponent(ToastComponent);
    this.counter++;
    const title = 'Message #' + this.counter;
    ref.setInput('label', title);

    // Обработчик события для выхода confirm
    ref.instance.confirmed.subscribe(title => {
      ref?.destroy();
      console.log('confirmed', title);
    });

    setTimeout(() => ref?.destroy(), 5000);
  }

}
```

Метод `subscribe` у `OutputEmitterRef` позволяет определить обработчик события. В нашем случае он просто удаляет уведомление с помощью `destroy` и логирует полученную строку в консоль.

Однако этот пример содержит небольшой косметический изъян. Независимо от того, кликает ли пользователь по ссылке подтверждения, пример вызывает `destroy` через 5 секунд. Следовательно, уведомление может быть удалено дважды: один раз после подтверждения и ещё раз после отображения его в течение 5 секунд.

К счастью, двойное уничтожение компонента не приводит к ошибке. Чтобы решить эту проблему, мы могли бы ввести флаг `destroyed`. В следующем разделе показан более мощный подход: потребление выходов как Observable.

### Потребление выходов как Observable

Хотя `OutputEmitterRef` предоставляет метод `subscribe`, он не является Observable. Однако исходный `EventEmitter`, использовавшийся вместе с декоратором @Output, им был. Чтобы вернуть все возможности, связанные с выходами на основе Observable, вы можете использовать функцию `outputToObservable`, которая является частью слоя интеграции с RxJS:

```typescript
import { outputToObservable } from '@angular/core/rxjs-interop';
[...]

@Component([...])
export class ToastDemoComponent {
  counter = 0;
  placeholder = viewChild.required('placeholder', { read: ViewContainerRef });

  show() {
    const ref = this.placeholder()?.createComponent(ToastComponent);
    this.counter++;
    const title = 'Message #' + this.counter;
    ref.setInput('label', title);

    const confirmed$ = outputToObservable(ref.instance.confirmed)
      .pipe(map(title => ({ trigger: 'confirmed', title })));

    const timer$ = timer(5000);
      .pipe(map(() => ({ trigger: 'timeout', title })));

    race(confirmed$, timer$).subscribe(action => {
      ref?.destroy();
      console.log('action', action);
    });

  }

}
```

Функция `outputToObservable` преобразует `OutputEmitterRef` в Observable. Показанный пример использует её, чтобы выразить как событие подтверждения, так и пятисекундный таймаут в виде Observable. Оператор `race` гарантирует, что будет использован только тот Observable, который первым выдаст значение.

Observable, возвращаемый `outputToObservable`, завершается, когда Angular уничтожает компонент этого выхода. По этой причине нет необходимости отписываться вручную.

## Паритет возможностей между запросами к содержимому и к представлению

До сих пор статья работала с `contentChildren` для запроса нескольких спроецированных дочерних элементов и с `viewChild` для получения одного узла в представлении. Однако оба понятия обладают паритетом возможностей: например, существуют также функции `contentChild` и `viewChildren`.

Кроме того, все опции, использованные выше для запросов к представлению или содержимому, такие как использование хэндлов в качестве локаторов или использование свойства `read`, работают для обоих видов запросов.

## Заключение

Несколько новых функций заменяют декораторы свойств и помогают настроить концепции связывания данных. Эти функции подхватываются компилятором Angular, который генерирует соответствующий код.

Функция `input` определяет входы (Inputs) для связываний свойств, `model` определяет входы для двустороннего связывания данных, а `contentChild(ren)` и `viewChild(ren)` отвечают за запросы к содержимому и к представлению. Использование этих функций приводит к сигналам, которые можно проецировать с помощью `computed` и использовать внутри эффектов.
