## Входные сигналы (Input Signals)

Входные сигналы (Input Signals) позволяют получать данные через связывания свойств (Property Bindings) в форме сигналов. В статье это демонстрируется на простом `OptionComponent`.

### Определение входного сигнала

```typescript
@Component({
  selector: 'app-option',
  standalone: true,
  imports: [],
  template: `
    <div class="option">
      {{ label() }}
    </div>
  `,
  styles: [...]
})
export class OptionComponent {
  label = input.required<string>();
}
```

Вызов функции `input` подхватывается компилятором Angular, который генерирует исходный код для связываний свойств. `InputSignal` всегда доступен только для чтения (read-only) и используется так же, как `Signal` (вызывается как функция-геттер).

### Привязка к входному сигналу

```typescript
<app-option label="Option #1">
<app-option [label]="myStringProperty">
<app-option [label]="mySignalProperty()">
```

### Вычисляемые сигналы (computed) и эффекты (effect) как замена хуков жизненного цикла

```typescript
markDownTitle = computed(() => '# ' + this.label())

constructor() {
  effect(() => {
    console.log('label updated', this.label());
    console.log('markdown', this.markDownTitle());
  });
}
```

### Опции входных сигналов

| Исходный код | Описание |
|---|---|
| `label = input();` | Необязательное (optional) свойство, представленное `InputSignal<string \| undefined>` |
| `label = input('Hello');` | Необязательное свойство, представленное `InputSignal<string>` с начальным значением `Hello` |
| `label = input<string \| undefined>('Hello');` | Необязательное свойство, представленное `InputSignal<string \| undefined>` с начальным значением `Hello` |

### Обязательные входы (required inputs) не могут иметь значения по умолчанию!

По определению `input.required` не может иметь значения по умолчанию (default value). При попытке прочитать значение обязательного входа до того, как к нему выполнена привязка, Angular выбрасывает исключение. Прямое обращение в конструкторе недопустимо (на этом этапе данные ещё не привязаны). Вместо этого следует использовать `ngOnInit` или `ngOnChanges`. Доступ к входам внутри `computed` или `effect` безопасен всегда (реактивный контекст читает актуальное значение в момент выполнения):

```typescript
@Component([...])
export class OptionComponent implements OnInit, OnChanges {
  label = input.required<string>();

  // безопасно
  markDownTitle = computed(() => '# ' + this.label())

  constructor() {
    // это вызвало бы исключение,
    // так как данные ещё не были привязаны
    console.log('label', this.label);

    effect(() => {
        // безопасно
        console.log('label', this.label);
    })
  }

  ngOnInit() {
    // безопасно
    console.log('label', this.label);
  }

  ngOnChanges() {
    // безопасно
    console.log('label', this.label);
  }
}
```

### Псевдонимы (aliases) входных сигналов

И `input`, и `input.require` принимают объект параметров, в котором можно задать `alias` (псевдоним — публичное имя свойства, отличное от имени поля класса):

```typescript
label = input.required({ alias: "title" });
```

В этом случае вызывающая сторона привязывается к имени свойства, заданному псевдонимом:

```typescript
<app-option title="Option #1">
  <app-option [title]="myStringProperty">
    <app-option [title]="mySignalProperty()"></app-option>
  </app-option>
</app-option>
```

В большинстве случаев использования псевдонимов следует избегать, поскольку они вводят лишний уровень косвенности (indirection).

### Преобразователи (transformers) входных сигналов

Преобразователи позволяют трансформировать значение, переданное через связывание свойства (преобразование применяется при записи входящего значения):

```typescript
@Component({
  selector: 'app-option',
  standalone: true,
  imports: [],
  template: `
    <div class="option">
      {{ label() }} @if (featured()) { ⭐ }
    </div>
  `,
  styles: [...]
})
export class OptionComponent {
  label = input.required<string>();
  featured = input.required({
    transform: booleanAttribute
  })
}
```

Этот преобразователь конвертирует строки в булевы значения:

```typescript
<app-option label="Option #1" featured="true"></app-option>
```

Кроме того, если атрибут присутствует, но значение ему не присвоено, предполагается `true`:

```typescript
<app-option label="Option #1" featured></app-option>
```

Тип данного сигнала — `InputSignal<boolean, unknown>`. Первый параметр типа (`boolean`) — это значение, полученное на выходе преобразователя (тип, который читает потребитель сигнала); второй (`unknown`) — это значение, привязанное в шаблоне вызывающей стороны и переданное на вход преобразователя (тип записи).

Для пользовательского преобразователя:

```typescript
function boolTranformer(value: unknown): boolean {
  return value !== "no";
}
```

Затем зарегистрируйте эту функцию в `input`:

```typescript
@Component([...])
export class OptionComponent {
  label = input.required<string>();
  featured = input.required({
    transform: boolTranformer
  })
}
```

## Двустороннее связывание данных через модельные сигналы (Model Signals)

Входные сигналы доступны только для чтения. Если требуется передать сигнал, который может быть обновлён вызываемым компонентом, необходимо настроить так называемый модельный сигнал (Model Signal — записываемый сигнал, синхронизируемый в обе стороны). Потребитель может использовать компонент так:

```typescript
<app-tabbed-pane [(current)]="current">
  <app-tab title="1st tab"> Lorem, ipsum dolor sit amet ... </app-tab>
  <app-tab title="2nd tab"> Sammas ergo gemma, ipsum dolor ... </app-tab>
  <app-tab title="3nd tab"> Gemma ham ipsum dolor sit ... </app-tab>
</app-tabbed-pane>

<p class="current-info">Current: {{ current() }}</p>
```

Чтобы это работало, `TabbedPaneComponent` должен предоставить модельный сигнал с помощью `model`:

```typescript
@Component([...])
export class TabbedPaneComponent {
  current = model(0);
  [...]
}
```

Здесь `0` — начальное значение. Опции аналогичны опциям `input`: `model.required` определяет обязательное свойство, а через объект опций можно задать псевдоним. Однако преобразователь (transform) для `model` определить нельзя.

При обновлении модельного сигнала компонентом новое значение распространяется вверх к сигналу, привязанному в шаблоне (синхронизация изнутри наружу):

```typescript
current.set(1);
```

### Двустороннее связывание как комбинация входа (input) и выхода (output)

Двусторонние связывания на основе сигналов можно определить парой из (read-only) входа (Input) и соответствующего выхода (Output). Имя выхода должно совпадать с именем входа плюс суффикс `Change` (соглашение Angular о банан-в-коробке `[(...)]`). Следовательно, для `current` определяется `currentChange`:

```typescript
@Component([...])
export class TabbedPaneComponent {
  current = input(0);
  currentChange = output<number>();
}
```

Для настройки выхода используется новый output API. Для инициирования события приложение вызывает метод `emit` этого выхода:

```typescript
<button [...] (click)="currentChange.emit($index)">{{tab.title()}}</button>
```

## Запросы к содержимому (Content Queries) на сигналах

`TabbedPaneComponent` позволяет продемонстрировать ещё одну возможность: запросы к содержимому (Content Queries), которые предоставляют доступ к спроецированным (projected) компонентам или директивам.

В `TabbedPaneComponent` передаётся несколько `TabComponent`. Они проецируются в представление `TabbedPaneComponent` (через `ng-content`). При этом одновременно должен отображаться лишь один из них. Следовательно, `TabbedPaneComponent` требуется программный доступ к своим `TabComponent`. Это реализуется новой функцией `contentChildren`:

```typescript
@Component({
  selector: 'app-tabbed-pane',
  standalone: true,
  imports: [],
  template: `
    <div class="pane">
      <div class="nav" role="group">
        @for(tab of tabs(); track tab) {
        <button
            [class.secondary]="tab !== currentTab()"
            (click)="activate($index)">
                {{tab.title()}}
        </button>
        }
      </div>
      <article>
        <ng-content></ng-content>
      </article>
    </div>
  `,
  styles: [...]
})
export class TabbedPaneComponent {
  current = model(0);
  tabs = contentChildren(TabComponent);
  currentTab = computed(() => this.tabs()[this.current()]);

  activate(active: number): void {
    this.current.set(active);
  }
}
```

Функция `contentChildren` — это аналог традиционного декоратора @ContentChildren. Так как `TabComponent` передан в качестве локатора (locator), функция возвращает сигнал с массивом всех спроецированных `TabComponent`.

Представление спроецированных узлов в виде сигнала позволяет реактивно отображать их через `computed`. В примере на этой основе выводится сигнал `currentTab` (производное значение, пересчитываемое при изменении набора вкладок или текущего индекса).

Спроецированный `TabComponent` использует этот сигнал, чтобы определить собственную видимость:

```typescript
@Component({
  selector: "app-tab",
  standalone: true,
  imports: [],
  template: `
    @if(visible()) {
    <div class="tab">
      <h2>{{ title() }}</h2>
      <ng-content></ng-content>
    </div>
    }
  `,
})
export class TabComponent {
  pane = inject(TabbedPaneComponent);

  title = input.required<string>();

  visible = computed(() => this.pane.currentTab() === this);
}
```

Здесь существенно, что всех родителей по иерархии DOM можно получить через внедрение зависимостей (dependency injection — `inject(TabbedPaneComponent)`). Сигнал `visible` выводится из сигнала `currentTab`.

Этот приём типичен для реактивной модели: значения не присваиваются императивно, а декларативно выводятся из других значений.

### Запросы к содержимому, включающие потомков (descendants)

По умолчанию запрос к содержимому раскрывает только прямые дочерние элементы содержимого (direct content children). «Внуки» (вложенные глубже одного уровня) — например, третья вкладка ниже — игнорируются:

```typescript
<app-tabbed-pane [(current)]="current">
  <app-tab title="1st tab"> Lorem, ipsum dolor sit amet ... </app-tab>
  <app-tab title="2nd tab"> Sammas ergo gemma, ipsum dolor ... </app-tab>

  <div class="danger-zone">
    <app-tab title="3nd tab">
      Here, you can delete the whole internet!
    </app-tab>
  </div>
</app-tabbed-pane>
```

Чтобы получить доступ и к таким узлам, опцию `descendants` устанавливают в `true`:

```typescript
tabs = contentChildren(TabComponent, { descendants: true });
```

## Output API

Ради симметричности API в Angular 17.3 введён новый output API. Для определения события, предоставляемого компонентом, теперь используется функция `output`. Аналогично новому input API, компилятор Angular подхватывает вызов `output` и генерирует соответствующий код. Для инициирования события используется метод `emit` возвращаемого `OutputEmitterRef`:

```typescript
@Component([...])
export class TabbedPaneComponent {
  current = model(0);
  tabs = contentChildren(TabComponent);
  currentTab = computed(() => this.tabs()[this.current()]);

  tabActivated = output<TabActivatedEvent>();

  activate(active: number): void {
    const previous = this.current();
    this.current.set(active);
    this.tabActivated.emit({ previous, active });
  }
}
```

### Предоставление Observable в качестве выходов

Помимо этого простого способа настройки выходов, источником выхода может выступать Observable. Для этого в слое интеграции с RxJS (RxJS interop) предусмотрена функция `outputFromObservable`:

```typescript
import { 
    outputFromObservable, 
    toObservable 
} from '@angular/core/rxjs-interop';
[...]

@Component([...])
export class TabbedPaneComponent {
  current = model(0);
  tabs = contentChildren(TabComponent);
  currentTab = computed(() => this.tabs()[this.current()]);

  tabChanged$ = toObservable(this.current).pipe(
    scan(
      (acc, active) => ({ active, previous: acc.active }),
      { active: -1, previous: -1 }
    ),
    skip(1),
  );

  tabChanged = outputFromObservable(this.tabChanged$);

  activate(active: number): void {
    this.current.set(active);
  }

}
```

Функция `outputFromObservable` преобразует Observable в `OutputEmitterRef`. В примере оператор `scan` сохраняет предыдущую активированную вкладку (накопление состояния), а `skip` гарантирует, что при начальной установке `current` событие не эмитируется. Последнее обеспечивает паритет поведения с показанным ранее примером.

## Запросы к представлению (View Queries) на сигналах

Если запрос к содержимому (Content Query) возвращает спроецированные узлы, то запрос к представлению (View Query) возвращает узлы из собственного представления компонента — узлы, найденные в его шаблоне. В большинстве случаев предпочтительнее использовать связывание данных. Однако в некоторых ситуациях программный доступ к дочернему элементу представления необходим.

Для демонстрации запроса дочерних элементов представления в статье используется простая форма для ввода имени пользователя и пароля. Оба поля `input` помечены как `required`. Если при нажатии `Save` валидация не проходит, фокус должен получить первое поле с ошибкой валидации. Для этого требуется доступ как к директиве `NgForm`, которую `FormModule` добавляет к тегу `form`, так и к DOM-узлам полей `input`:

```typescript
@Component({
  selector: "app-form",
  standalone: true,
  imports: [FormsModule, JsonPipe],
  template: `
    <h1>Form Demo</h1>
    <form autocomplete="off">
      <input
        [(ngModel)]="userName"
        placeholder="User Name"
        name="userName"
        #userNameCtrl
        required
      />
      <input
        [(ngModel)]="password"
        placeholder="Password"
        type="password"
        name="password"
        #passwordCtrl
        required
      />
      <button (click)="save()">Save</button>
    </form>
  `,
  styles: `
    form {
      max-width: 600px;
    }
  `,
})
export class FormDemoComponent {
  form = viewChild.required(NgForm);

  userNameCtrl =
    viewChild.required<ElementRef<HTMLInputElement>>("userNameCtrl");
  passwordCtrl =
    viewChild.required<ElementRef<HTMLInputElement>>("passwordCtrl");

  userName = signal("");
  password = signal("");

  save(): void {
    const form = this.form();

    if (form.controls["userName"].invalid) {
      this.userNameCtrl().nativeElement.focus();
      return;
    }

    if (form.controls["password"].invalid) {
      this.passwordCtrl().nativeElement.focus();
      return;
    }

    console.log("save", this.userName(), this.password());
  }
}
```

Оба запроса выполняются функцией `viewChild`. В первом случае в качестве локатора передаётся тип `NgForm`. Однако поиск самих полей по типу не работает, поскольку дочерних элементов этого типа может быть несколько. Поэтому поля помечены ссылками-хэндлами шаблона (template reference variables — `#userName` и `#password`), и в локатор передаётся имя соответствующего хэндла.

Дочерние элементы представления могут быть представлены разными типами: типом соответствующего компонента или директивы, `ElementRef` (обёртка над DOM-узлом) или `ViewContainerRef`. Последний используется в следующем разделе.

Требуемый тип указывается опцией `read`, как в предыдущем примере.

## Запросы и ViewContainerRef

Бывают ситуации, когда компонент нужно динамически добавить в заглушку (placeholder). Примеры — модальные диалоги или всплывающие уведомления (toasts). Простой способ — директива `*ngComponentOutlet`. Более гибкий способ — запрос `ViewContainerRef` заглушки.

Контейнер представления (View Container) можно рассматривать как невидимый контейнер вокруг каждого компонента и каждого фрагмента статического HTML. Получив ссылку на него, можно добавлять в него дополнительные компоненты или шаблоны.

Для демонстрации в статье используется простой пример с всплывающим уведомлением (toast). В роли заглушки выступает `ng-container`:

```typescript
@Component({
  selector: 'app-dynamic',
  standalone: true,
  imports: [],
  template: `
    <h2>Toast Demo</h2>
    <button (click)="show()">Show Toast</button>
    <ng-container #placeholder></ng-container>
  `,
  styles: [...]
})
export class ToastDemoComponent {
  counter = 0;
  placeholder = viewChild.required('placeholder', { read: ViewContainerRef });

  show() {
    const ref = this.placeholder()?.createComponent(ToastComponent);
    this.counter++;
    ref?.setInput('label', 'Message #' + this.counter);
    setTimeout(() => ref?.destroy(), 2000);
  }

}
```

Свойство `read` явно указывает, что читается не компонент-заглушка, а его `ViewContainerRef`. Метод `createComponent` инстанцирует и добавляет `ToastComponent`. Возвращаемый `ComponentRef` используется для установки свойства `label` нового компонента — через его метод `setInput`. Через две секунды метод `destroy` удаляет уведомление.

Для простоты компонент `ToastComponent` здесь зашит жёстко. В более универсальных реальных сценариях тип используемого компонента может конфигурироваться — например, вызовом метода сервиса, который принимает тип компонента и уведомляет другой компонент, добавляющий компонент этого типа в заглушку.

### Программная настройка выхода (Output)

В предыдущем примере вызывался `setInput` для присвоения значения входу `title` компонента `ToastComponent`. Далее обсуждается, как определять обработчики событий для таких динамически добавленных компонентов.

Допустим, `ToastComponent` отображает ссылку подтверждения. При клике по ней он эмитирует событие `confirmed`:

```typescript
@Component([...])
export class ToastComponent {
  label = input.required<string>();
  confirmed = output<string>();

  confirm(): void {
    this.confirmed.emit(this.label());
  }
}
```

Чтобы настроить обработчик этого события, можно напрямую использовать свойство `instance` возвращённого `ComponentRef`. Оно указывает на добавленный экземпляр компонента и, следовательно, предоставляет доступ ко всем его свойствам:

```typescript
@Component([...])
export class ToastDemoComponent {
  counter = 0;
  placeholder = viewChild.required('placeholder', { read: ViewContainerRef });

  show() {
    const ref = this.placeholder()?.createComponent(ToastComponent);
    this.counter++;
    const title = 'Message #' + this.counter;
    ref.setInput('label', title);

    // Обработчик события для выхода confirm
    ref.instance.confirmed.subscribe(title => {
      ref?.destroy();
      console.log('confirmed', title);
    });

    setTimeout(() => ref?.destroy(), 5000);
  }

}
```

Метод `subscribe` у `OutputEmitterRef` позволяет определить обработчик события. В данном случае он удаляет уведомление через `destroy` и логирует полученную строку в консоль.

Однако в этом примере есть небольшой косметический дефект. Независимо от того, кликнул ли пользователь по ссылке подтверждения, через 5 секунд вызывается `destroy`. Следовательно, уведомление может быть удалено дважды: один раз после подтверждения и ещё раз после отображения в течение 5 секунд.

К счастью, повторное уничтожение компонента не приводит к ошибке. Для решения проблемы можно ввести флаг `destroyed`. В следующем разделе показан более мощный подход — потребление выходов в виде Observable.

### Потребление выходов в виде Observable

Хотя `OutputEmitterRef` предоставляет метод `subscribe`, сам он не является Observable. В то же время прежний `EventEmitter`, использовавшийся с декоратором @Output, им был. Чтобы вернуть все возможности выходов на основе Observable, можно использовать функцию `outputToObservable` из слоя интеграции с RxJS:

```typescript
import { outputToObservable } from '@angular/core/rxjs-interop';
[...]

@Component([...])
export class ToastDemoComponent {
  counter = 0;
  placeholder = viewChild.required('placeholder', { read: ViewContainerRef });

  show() {
    const ref = this.placeholder()?.createComponent(ToastComponent);
    this.counter++;
    const title = 'Message #' + this.counter;
    ref.setInput('label', title);

    const confirmed$ = outputToObservable(ref.instance.confirmed)
      .pipe(map(title => ({ trigger: 'confirmed', title })));

    const timer$ = timer(5000);
      .pipe(map(() => ({ trigger: 'timeout', title })));

    race(confirmed$, timer$).subscribe(action => {
      ref?.destroy();
      console.log('action', action);
    });

  }

}
```

Функция `outputToObservable` преобразует `OutputEmitterRef` в Observable. В примере она используется, чтобы выразить и событие подтверждения, и пятисекундный таймаут в виде Observable. Оператор `race` гарантирует, что будет использован только тот Observable, который первым выдаст значение.

Observable, возвращаемый `outputToObservable`, завершается (completes), когда Angular уничтожает компонент данного выхода. Поэтому отписываться вручную не требуется.

## Паритет возможностей запросов к содержимому и к представлению

До этого момента в статье использовались `contentChildren` для запроса нескольких спроецированных дочерних элементов и `viewChild` для получения одного узла представления. Однако оба механизма обладают паритетом возможностей: существуют также функции `contentChild` и `viewChildren`.

Кроме того, все опции, применённые выше для запросов к представлению или содержимому, — например, использование хэндлов в качестве локаторов или свойства `read`, — работают для обоих видов запросов.

## Заключение

Несколько новых функций заменяют декораторы свойств и помогают настроить механизмы связывания данных. Эти функции подхватываются компилятором Angular, который генерирует соответствующий код.

Функция `input` определяет входы (Inputs) для связываний свойств, `model` определяет входы для двустороннего связывания данных, а `contentChild(ren)` и `viewChild(ren)` отвечают за запросы к содержимому и к представлению. Использование этих функций даёт сигналы, которые можно проецировать через `computed` и использовать внутри эффектов.
