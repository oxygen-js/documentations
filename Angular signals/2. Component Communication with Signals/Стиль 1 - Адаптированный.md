## Входные сигналы (Input Signals)

Входные сигналы позволяют получать данные через связывание свойств (Property Bindings) в виде сигналов. Покажем это на примере простого `OptionComponent`.

### Объявляем входной сигнал

```typescript
@Component({
  selector: 'app-option',
  standalone: true,
  imports: [],
  template: `
    <div class="option">
      {{ label() }}
    </div>
  `,
  styles: [...]
})
export class OptionComponent {
  label = input.required<string>();
}
```

Вызов функции `input` подхватывает компилятор Angular и генерирует код для связывания свойств. Полученный `InputSignal` всегда доступен только для чтения, и с ним можно работать так же, как с обычным `Signal`.

### Привязываемся к входному сигналу

```typescript
<app-option label="Option #1">
<app-option [label]="myStringProperty">
<app-option [label]="mySignalProperty()">
```

### Вычисляемые сигналы (computed) и эффекты (effect) вместо хуков жизненного цикла

```typescript
markDownTitle = computed(() => '# ' + this.label())

constructor() {
  effect(() => {
    console.log('label updated', this.label());
    console.log('markdown', this.markDownTitle());
  });
}
```

### Варианты объявления входных сигналов

| Исходный код | Описание |
|---|---|
| `label = input();` | Необязательное свойство, представленное `InputSignal<string \| undefined>` |
| `label = input('Hello');` | Необязательное свойство, представленное `InputSignal<string>` с начальным значением `Hello` |
| `label = input<string \| undefined>('Hello');` | Необязательное свойство, представленное `InputSignal<string \| undefined>` с начальным значением `Hello` |

### У обязательных входов не может быть значения по умолчанию!

По определению `input.required` не может иметь значения по умолчанию. Если попытаться прочитать значение обязательного входа до того, как к нему что-то привязали, Angular выбросит исключение. Напрямую обратиться к нему в конструкторе нельзя. Вместо этого можно воспользоваться `ngOnInit` или `ngOnChanges`. А вот использовать входы внутри `computed` или `effect` безопасно всегда:

```typescript
@Component([...])
export class OptionComponent implements OnInit, OnChanges {
  label = input.required<string>();

  // безопасно
  markDownTitle = computed(() => '# ' + this.label())

  constructor() {
    // это вызовет исключение,
    // так как данные ещё не привязаны
    console.log('label', this.label);

    effect(() => {
        // безопасно
        console.log('label', this.label);
    })
  }

  ngOnInit() {
    // безопасно
    console.log('label', this.label);
  }

  ngOnChanges() {
    // безопасно
    console.log('label', this.label);
  }
}
```

### Псевдонимы (aliases) для входных сигналов

И `input`, и `input.require` принимают объект параметров, в котором можно задать `alias` (псевдоним):

```typescript
label = input.required({ alias: "title" });
```

В этом случае вызывающая сторона должна привязываться к имени свойства, заданному псевдонимом:

```typescript
<app-option title="Option #1">
  <app-option [title]="myStringProperty">
    <app-option [title]="mySignalProperty()"></app-option>
  </app-option>
</app-option>
```

В большинстве случаев псевдонимов лучше избегать: они создают лишний слой косвенности.

### Преобразователи (transformers) для входных сигналов

Преобразователи позволяют менять значение, переданное через связывание свойства:

```typescript
@Component({
  selector: 'app-option',
  standalone: true,
  imports: [],
  template: `
    <div class="option">
      {{ label() }} @if (featured()) { ⭐ }
    </div>
  `,
  styles: [...]
})
export class OptionComponent {
  label = input.required<string>();
  featured = input.required({
    transform: booleanAttribute
  })
}
```

Этот преобразователь конвертирует строки в булевы значения:

```typescript
<app-option label="Option #1" featured="true"></app-option>
```

Кроме того, если атрибут присутствует, но значение ему не присвоено, подразумевается `true`:

```typescript
<app-option label="Option #1" featured></app-option>
```

Тип такого сигнала — `InputSignal<boolean, unknown>`. Первый параметр типа (`boolean`) — это значение, которое возвращает преобразователь; второй (`unknown`) — значение, привязанное в шаблоне вызывающей стороны и переданное в преобразователь.

Свой собственный преобразователь может выглядеть так:

```typescript
function boolTranformer(value: unknown): boolean {
  return value !== "no";
}
```

Затем регистрируем эту функцию в `input`:

```typescript
@Component([...])
export class OptionComponent {
  label = input.required<string>();
  featured = input.required({
    transform: boolTranformer
  })
}
```

## Двустороннее связывание данных через модельные сигналы (Model Signals)

Входные сигналы доступны только для чтения. Если нужно передать сигнал, который вызываемый компонент сможет обновлять, понадобится так называемый модельный сигнал (Model Signal). Тогда потребитель сможет использовать компонент вот так:

```typescript
<app-tabbed-pane [(current)]="current">
  <app-tab title="1st tab"> Lorem, ipsum dolor sit amet ... </app-tab>
  <app-tab title="2nd tab"> Sammas ergo gemma, ipsum dolor ... </app-tab>
  <app-tab title="3nd tab"> Gemma ham ipsum dolor sit ... </app-tab>
</app-tabbed-pane>

<p class="current-info">Current: {{ current() }}</p>
```

Чтобы это работало, `TabbedPaneComponent` должен предоставить модельный сигнал с помощью функции `model`:

```typescript
@Component([...])
export class TabbedPaneComponent {
  current = model(0);
  [...]
}
```

Здесь `0` — начальное значение. Возможности похожи на те, что есть у `input`: `model.required` объявляет обязательное свойство, а через объект параметров можно задать псевдоним. А вот преобразователь задать нельзя.

Если компонент обновит модельный сигнал, новое значение поднимется наверх — к сигналу, привязанному в шаблоне:

```typescript
current.set(1);
```

### Двустороннее связывание как комбинация входа и выхода

Двустороннее связывание на сигналах можно собрать из (доступного только для чтения) входа (Input) и соответствующего выхода (Output). Имя выхода должно совпадать с именем входа плюс суффикс `Change`. Значит, для `current` нужно объявить `currentChange`:

```typescript
@Component([...])
export class TabbedPaneComponent {
  current = input(0);
  currentChange = output<number>();
}
```

Для создания выхода используется новый API `output`. Чтобы инициировать событие, приложение вызывает метод `emit` этого выхода:

```typescript
<button [...] (click)="currentChange.emit($index)">{{tab.title()}}</button>
```

## Запросы к спроецированному содержимому (Content Queries) на сигналах

`TabbedPaneComponent` помогает показать ещё одну возможность — запросы к содержимому (Content Queries), которые дают доступ к спроецированным компонентам или директивам.

В `TabbedPaneComponent` передаётся несколько `TabComponent`. Они проецируются (project) в представление `TabbedPaneComponent`. Но в один момент времени мы хотим показывать только один из них. Поэтому `TabbedPaneComponent` нужен программный доступ к своим `TabComponent`. Это можно сделать с помощью новой функции `contentChildren`:

```typescript
@Component({
  selector: 'app-tabbed-pane',
  standalone: true,
  imports: [],
  template: `
    <div class="pane">
      <div class="nav" role="group">
        @for(tab of tabs(); track tab) {
        <button
            [class.secondary]="tab !== currentTab()"
            (click)="activate($index)">
                {{tab.title()}}
        </button>
        }
      </div>
      <article>
        <ng-content></ng-content>
      </article>
    </div>
  `,
  styles: [...]
})
export class TabbedPaneComponent {
  current = model(0);
  tabs = contentChildren(TabComponent);
  currentTab = computed(() => this.tabs()[this.current()]);

  activate(active: number): void {
    this.current.set(active);
  }
}
```

Функция `contentChildren` — это аналог традиционного декоратора @ContentChildren. Поскольку в неё передан `TabComponent` (так называемый локатор, locator), она возвращает сигнал с массивом всех спроецированных `TabComponent`.

То, что спроецированные узлы доступны как сигнал, позволяет реактивно преобразовывать их через `computed`. В примере на этой основе выводится сигнал `currentTab`.

Сам спроецированный `TabComponent` использует этот сигнал, чтобы понять, должен ли он быть видимым:

```typescript
@Component({
  selector: "app-tab",
  standalone: true,
  imports: [],
  template: `
    @if(visible()) {
    <div class="tab">
      <h2>{{ title() }}</h2>
      <ng-content></ng-content>
    </div>
    }
  `,
})
export class TabComponent {
  pane = inject(TabbedPaneComponent);

  title = input.required<string>();

  visible = computed(() => this.pane.currentTab() === this);
}
```

Здесь стоит помнить, что всех родителей в DOM можно получить через внедрение зависимостей (dependency injection). Сигнал `visible` выводится из сигнала `currentTab`.

Такой подход типичен для реактивного мира: вместо того чтобы императивно присваивать значения, их декларативно выводят из других значений.

### Запросы к содержимому, включающие потомков (descendants)

По умолчанию запрос к содержимому раскрывает только прямых дочерних элементов. «Внуки» — например, третья вкладка ниже — игнорируются:

```typescript
<app-tabbed-pane [(current)]="current">
  <app-tab title="1st tab"> Lorem, ipsum dolor sit amet ... </app-tab>
  <app-tab title="2nd tab"> Sammas ergo gemma, ipsum dolor ... </app-tab>

  <div class="danger-zone">
    <app-tab title="3nd tab">
      Here, you can delete the whole internet!
    </app-tab>
  </div>
</app-tabbed-pane>
```

Чтобы дотянуться и до таких узлов, можно выставить опцию `descendants` в `true`:

```typescript
tabs = contentChildren(TabComponent, { descendants: true });
```

## API выходов (Output API)

Ради симметричности API в Angular 17.3 появился новый API выходов. Теперь для объявления события, которое предоставляет компонент, используется функция `output`. Как и в случае с новым API входов, компилятор Angular подхватывает вызов `output` и генерирует соответствующий код. Чтобы инициировать событие, используют метод `emit` возвращённого `OutputEmitterRef`:

```typescript
@Component([...])
export class TabbedPaneComponent {
  current = model(0);
  tabs = contentChildren(TabComponent);
  currentTab = computed(() => this.tabs()[this.current()]);

  tabActivated = output<TabActivatedEvent>();

  activate(active: number): void {
    const previous = this.current();
    this.current.set(active);
    this.tabActivated.emit({ previous, active });
  }
}
```

### Observable в качестве источника для выхода

Помимо такого простого способа объявить выход, в качестве его источника можно взять Observable. Для этого в слое интеграции с RxJS (RxJS interop) есть функция `outputFromObservable`:

```typescript
import { 
    outputFromObservable, 
    toObservable 
} from '@angular/core/rxjs-interop';
[...]

@Component([...])
export class TabbedPaneComponent {
  current = model(0);
  tabs = contentChildren(TabComponent);
  currentTab = computed(() => this.tabs()[this.current()]);

  tabChanged$ = toObservable(this.current).pipe(
    scan(
      (acc, active) => ({ active, previous: acc.active }),
      { active: -1, previous: -1 }
    ),
    skip(1),
  );

  tabChanged = outputFromObservable(this.tabChanged$);

  activate(active: number): void {
    this.current.set(active);
  }

}
```

Функция `outputFromObservable` превращает Observable в `OutputEmitterRef`. В примере оператор `scan` запоминает предыдущую активированную вкладку, а `skip` гарантирует, что при начальной установке `current` событие не отправится. Последнее обеспечивает паритет возможностей с показанным ранее примером.

## Запросы к собственному представлению (View Queries) на сигналах

Если запрос к содержимому (Content Query) возвращает спроецированные узлы, то запрос к представлению (View Query) возвращает узлы из собственного представления компонента — то есть узлы, найденные в его шаблоне. В большинстве случаев лучше предпочесть связывание данных. Но иногда программный доступ к дочернему элементу представления всё же нужен.

Чтобы показать, как запрашивать дочерние элементы представления, в статье используется простая форма для ввода имени пользователя и пароля. Оба поля `input` помечены как `required`. Если при нажатии `Save` проверка не проходит, фокус должен встать на первое поле с ошибкой валидации. Для этого нам нужен доступ как к директиве `NgForm`, которую `FormModule` добавляет к тегу `form`, так и к DOM-узлам, представляющим поля `input`:

```typescript
@Component({
  selector: "app-form",
  standalone: true,
  imports: [FormsModule, JsonPipe],
  template: `
    <h1>Form Demo</h1>
    <form autocomplete="off">
      <input
        [(ngModel)]="userName"
        placeholder="User Name"
        name="userName"
        #userNameCtrl
        required
      />
      <input
        [(ngModel)]="password"
        placeholder="Password"
        type="password"
        name="password"
        #passwordCtrl
        required
      />
      <button (click)="save()">Save</button>
    </form>
  `,
  styles: `
    form {
      max-width: 600px;
    }
  `,
})
export class FormDemoComponent {
  form = viewChild.required(NgForm);

  userNameCtrl =
    viewChild.required<ElementRef<HTMLInputElement>>("userNameCtrl");
  passwordCtrl =
    viewChild.required<ElementRef<HTMLInputElement>>("passwordCtrl");

  userName = signal("");
  password = signal("");

  save(): void {
    const form = this.form();

    if (form.controls["userName"].invalid) {
      this.userNameCtrl().nativeElement.focus();
      return;
    }

    if (form.controls["password"].invalid) {
      this.passwordCtrl().nativeElement.focus();
      return;
    }

    console.log("save", this.userName(), this.password());
  }
}
```

Оба запроса делаются через функцию `viewChild`. В первом случае в качестве локатора передаётся тип `NgForm`. А вот искать сами поля по типу не получится: дочерних элементов с этим типом может быть несколько. Поэтому поля помечены ссылками-хэндлами (`#userName` и `#password`), и в локатор передаётся имя соответствующего хэндла.

Дочерние элементы представления могут быть представлены разными типами: типом самого компонента или директивы, `ElementRef`, представляющим его DOM-узел, или `ViewContainerRef`. Последний нам понадобится в следующем разделе.

Нужный тип можно указать через опцию `read`, как в предыдущем примере.

## Запросы и ViewContainerRef

Бывают ситуации, когда компонент нужно добавить в заглушку (placeholder) динамически. Примеры — модальные диалоги или всплывающие уведомления (toasts). Простой способ — директива `*ngComponentOutlet`. Более гибкий — запросить `ViewContainerRef` у заглушки.

Контейнер представления (View Container) можно представить как невидимую обёртку вокруг каждого компонента и каждого куска статического HTML. Получив доступ к нему, можно добавлять туда новые компоненты или шаблоны.

В качестве иллюстрации в статье показывается простой пример с всплывающим уведомлением (toast). В роли заглушки выступает `ng-container`:

```typescript
@Component({
  selector: 'app-dynamic',
  standalone: true,
  imports: [],
  template: `
    <h2>Toast Demo</h2>
    <button (click)="show()">Show Toast</button>
    <ng-container #placeholder></ng-container>
  `,
  styles: [...]
})
export class ToastDemoComponent {
  counter = 0;
  placeholder = viewChild.required('placeholder', { read: ViewContainerRef });

  show() {
    const ref = this.placeholder()?.createComponent(ToastComponent);
    this.counter++;
    ref?.setInput('label', 'Message #' + this.counter);
    setTimeout(() => ref?.destroy(), 2000);
  }

}
```

Свойство `read` ясно даёт понять, что мы хотим прочитать не сам компонент-заглушку, а его `ViewContainerRef`. Метод `createComponent` создаёт и добавляет `ToastComponent`. Возвращённый `ComponentRef` используется, чтобы задать свойство `label` нового компонента — для этого вызывается его метод `setInput`. Через две секунды метод `destroy` снова убирает уведомление.

Для простоты компонент `ToastComponent` здесь зашит жёстко. В более универсальных реальных сценариях тип используемого компонента можно настраивать — например, вызовом метода сервиса, который принимает тип компонента и уведомляет другой компонент, добавляющий компонент этого типа в заглушку.

### Программная настройка выхода (Output)

В предыдущем примере мы вызывали `setInput`, чтобы присвоить значение входу `title` компонента `ToastComponent`. Теперь обсудим, как задавать обработчики событий для таких динамически добавленных компонентов.

Допустим, `ToastComponent` показывает ссылку подтверждения. При клике по этой ссылке он испускает событие `confirmed`:

```typescript
@Component([...])
export class ToastComponent {
  label = input.required<string>();
  confirmed = output<string>();

  confirm(): void {
    this.confirmed.emit(this.label());
  }
}
```

Чтобы настроить обработчик этого события, можно напрямую воспользоваться свойством `instance` возвращённого `ComponentRef`. Оно указывает на добавленный экземпляр компонента и даёт доступ ко всем его свойствам:

```typescript
@Component([...])
export class ToastDemoComponent {
  counter = 0;
  placeholder = viewChild.required('placeholder', { read: ViewContainerRef });

  show() {
    const ref = this.placeholder()?.createComponent(ToastComponent);
    this.counter++;
    const title = 'Message #' + this.counter;
    ref.setInput('label', title);

    // Обработчик события для выхода confirm
    ref.instance.confirmed.subscribe(title => {
      ref?.destroy();
      console.log('confirmed', title);
    });

    setTimeout(() => ref?.destroy(), 5000);
  }

}
```

Метод `subscribe` у `OutputEmitterRef` позволяет задать обработчик события. В нашем случае он просто убирает уведомление через `destroy` и выводит полученную строку в консоль.

Однако у этого примера есть маленький косметический изъян. Независимо от того, кликнул ли пользователь по ссылке подтверждения, через 5 секунд вызывается `destroy`. Значит, уведомление может быть удалено дважды: один раз после подтверждения и ещё раз — после того как оно провисело 5 секунд.

К счастью, повторное уничтожение компонента ошибки не вызывает. Решить проблему можно, заведя флаг `destroyed`. В следующем разделе показан более мощный подход — обработка выходов как Observable.

### Обработка выходов как Observable

Хотя у `OutputEmitterRef` есть метод `subscribe`, сам он не является Observable. А вот прежний `EventEmitter`, использовавшийся вместе с декоратором @Output, им был. Чтобы вернуть все возможности выходов на базе Observable, можно воспользоваться функцией `outputToObservable` из слоя интеграции с RxJS:

```typescript
import { outputToObservable } from '@angular/core/rxjs-interop';
[...]

@Component([...])
export class ToastDemoComponent {
  counter = 0;
  placeholder = viewChild.required('placeholder', { read: ViewContainerRef });

  show() {
    const ref = this.placeholder()?.createComponent(ToastComponent);
    this.counter++;
    const title = 'Message #' + this.counter;
    ref.setInput('label', title);

    const confirmed$ = outputToObservable(ref.instance.confirmed)
      .pipe(map(title => ({ trigger: 'confirmed', title })));

    const timer$ = timer(5000);
      .pipe(map(() => ({ trigger: 'timeout', title })));

    race(confirmed$, timer$).subscribe(action => {
      ref?.destroy();
      console.log('action', action);
    });

  }

}
```

Функция `outputToObservable` превращает `OutputEmitterRef` в Observable. В примере она используется, чтобы выразить и событие подтверждения, и пятисекундный таймаут как Observable. Оператор `race` гарантирует, что будет использован только тот Observable, который первым выдаст значение.

Observable, возвращаемый `outputToObservable`, завершается, когда Angular уничтожает компонент этого выхода. Поэтому отписываться вручную не нужно.

## Паритет возможностей между запросами к содержимому и к представлению

До сих пор мы работали с `contentChildren`, чтобы запросить несколько спроецированных дочерних элементов, и с `viewChild`, чтобы получить один узел из представления. Но у обоих понятий есть паритет возможностей: например, существуют ещё и функции `contentChild`, и `viewChildren`.

Кроме того, все опции, которые мы использовали выше для запросов к представлению или содержимому — например, хэндлы в качестве локаторов или свойство `read`, — работают для обоих видов запросов.

## Заключение

Несколько новых функций заменяют декораторы свойств и помогают настроить механизмы связывания данных. Эти функции подхватывает компилятор Angular и генерирует соответствующий код.

Функция `input` объявляет входы для связывания свойств, `model` объявляет входы для двустороннего связывания данных, а `contentChild(ren)` и `viewChild(ren)` отвечают за запросы к содержимому и к представлению. Использование этих функций даёт сигналы, которые можно преобразовывать через `computed` и использовать внутри эффектов.
