# Streaming-ресурсы в Angular — детали и семантика

## Пример приложения

В качестве примера здесь возьмём «Hello World» реактивного программирования — простой таймер, который каждую секунду увеличивает счётчик.

Прелесть этого элементарного примера в том, что он уже содержит достаточно материала, чтобы разобрать небольшие, но важные нюансы потоковых ресурсов (Streaming Resources). Вариацию этого же примера мы потом используем, чтобы показать совместную работу с RxJS.

## Анатомия потокового ресурса

В отличие от обычных ресурсов, загрузчик (Loader) потокового ресурса всегда возвращает следующую структуру:

```
PromiseLike<Signal<ResourceStreamItem<T>>>
```

Это Promise с сигналом (Signal), который хранит `ResourceStreamItem` — то есть либо последнее опубликованное значение, либо ошибку. Angular определяет этот тип так:

```
type ResourceStreamItem<T> = {
    value: T;
} | {
    error: Error;
};
```

Поскольку сигнал со временем принимает несколько значений или состояний ошибки, он, по сути, и представляет собой поток данных.

Чтобы создать потоковый ресурс, в коде используется функция *resource* — та же самая, что и для обычных ресурсов. Однако вместо обычного загрузчика ей передаётся потоковый загрузчик (Streaming Loader) через свойство *stream*:

```
const myResource = resource({
  params: params,
  stream: async (loaderParams) => {

    // 1. Создаём Signal, представляющий поток
    const result = signal<StreamItem<number>>({ 
      value: 4711 
    });

    // 2. Настраиваем асинхронную логику, обновляющую Signal
    […]

    // 3. Настраиваем обработчик очистки, запускаемый по AbortSignal
    loaderParams.abortSignal.addEventListener('abort', () => {
      […]
    });

    // 4. Возвращаем Signal
    return result;
  },
});
```

Благодаря ключевому слову *async* нам не нужно явно создавать Promise, который отдаёт сигнал. Загрузчик можно условно разбить на четыре части — они отмечены комментариями в примере выше:

1.  Сначала потоковый загрузчик создаёт новый сигнал, представляющий поток данных. Этому сигналу задаётся начальное значение.
2.  Затем загрузчик запускает асинхронную операцию, которая со временем выдаёт несколько результатов. Эти результаты публикуются по одному через сигнал.
3.  Загрузчик также отвечает за логику очистки (cleanup), которая завершает лежащую в основе асинхронную операцию, когда её значения больше не нужны.
4.  В самом конце потоковый загрузчик возвращает сигнал.

Для логики очистки потоковый загрузчик использует *AbortSignal*, который предоставляет Resource API: его можно найти в переданном объекте параметров. Любопытно, что эти пункты более или менее совпадают с типичным подходом при прямом использовании конструктора Observable в мире RxJS.

## Семантика switch-map при переключении между потоками

Заданная логика очистки срабатывает всякий раз, когда приложению больше не нужен текущий поток данных. Причин для этого две. Первая — когда Angular уничтожает строительный блок, в котором живёт ресурс. Представьте компонент с ресурсом. При уходе с него роутер уничтожает этот компонент, а вместе с ним Angular уничтожает и ресурс.

Вторая причина возникает, когда меняется сигнал *params* ресурса. Каждое изменение запускает потоковый загрузчик, который выдаёт новый поток. При переходе между этими потоками Resource API использует ту же семантику, что и *switchMap* в RxJS.

Это означает, что ресурс всегда потребляет только самый свежий поток. Такой подход обычно и применяется при загрузке данных. Как это в целом принято для сигналов в Angular, цель — предложить простые концепции для стандартных случаев. Для более сложных сценариев приложение может воспользоваться библиотеками вроде RxJS.

## Простой таймер на основе ресурса

Теперь, когда мы разобрали базовую структуру потокового ресурса, я хочу показать его конкретное применение на первом примере — таймере, который с заданным интервалом увеличивает число. В следующем листинге этот *timerResource* показан с точки зрения потребителя:

```
@Component([…])
export class TimerResourceComponent {

  startValue = signal(0);
  timer = timerResource(1000, this.startValue);

  forward(): void {
    this.startValue.update((v) => nextSegment(v));
  }
}

function nextSegment(currentValue: number): number {
  return Math.floor(currentValue / 100) * 100 + 100;
}
```

Таймер возвращает поток, который начинает считать с заданного начального значения. Первый параметр *timerResource* — это желаемый интервал в миллисекундах. Начальное значение задаётся сигналом *startValue*. Всякий раз, когда оно меняется, таймер переключается на новый поток. Чтобы это продемонстрировать, метод *forward* перепрыгивает к следующей полной сотне: например, с 17 на 100 или со 123 на 200.

## Фабрика для потокового ресурса

Чтобы упростить использование потокового ресурса для таймера, функция *timerResource* — это просто фабрика:

```
export function timerResource(
  timeout: number,
  startValue: () => number
): ResourceRef<number | undefined> {

  const params = computed(() => ({
    startValue: startValue(),
  }));

  const result = resource({
    params: params,
    stream: async (loaderParams) => {
      const counter = loaderParams.params.startValue;
      […]
    }
  });

  return result;
}
```

Эта фабрика получает желаемый интервал (*timeout*) и сигнал с начальным значением. Для этого сигнала *timerResource* интересует только геттер. Поэтому сигнал типизируется как *() => number*.

Возвращаемое значение — *ResourceRef <number | undefined>*. Параметр типа соответствует значениям в потоке. Поскольку у ресурса нет *defaultValue*, он автоматически стартует со значения *undefined*.

Вычисляемый сигнал (computed) *params* представляет все параметры, которые запускают загрузчик. Так как в нашем примере единственным триггером выступает *startValue*, этот computed-сигнал кажется немного лишним посредником. Тем не менее я предпочитаю придерживаться такого подхода — особенно потому, что он позволяет позже расширить набор триггеров и делает имя *startValue* доступным внутри (потокового) загрузчика: в нашем случае загрузчик может получить текущее значение через *param.params.startValue*.

## Потоковый загрузчик для таймера

Структура потокового загрузчика соответствует четырём разделам, о которых мы говорили выше:

```
const result = resource({
  params: params,
  stream: async (loaderParams) => {
    let counter = loaderParams.params.startValue;

    // 1. Создаём Signal, представляющий поток
    const resultSignal = signal<StreamItem<number>>({
      value: loaderParams.params.startValue,
    });

    // 2. Настраиваем асинхронную логику, обновляющую Signal
    const ref = setInterval(() => {
      counter++;
      console.log('tick', counter);

      if (counter === 7 || counter === 13) {
        resultSignal.set({ error: new Error('bad luck!') });
      } else {
        resultSignal.set({ value: counter });
      }
    }, timeout);

    // 3. Настраиваем обработчик очистки, запускаемый по AbortSignal
    loaderParams.abortSignal.addEventListener('abort', () => {
      console.log('clean up!');
      clearInterval(ref);
    });

    // 4. Возвращаем Signal
    return resultSignal;
  },
});
```

Загрузчик реализует асинхронный подсчёт с помощью традиционной JavaScript-функции *setInterval*. Чтобы продемонстрировать поведение в состоянии ошибки, таймер сообщает об ошибке для значений 7 и 13. Так что и суеверные пользователи тоже не остаются обделёнными.

Из-за асинхронного поведения *setInterval* шаг 4 происходит раньше первого выполнения колбэка из шага 2. Это значит, что загрузчик сначала возвращает сигнал с его начальным значением, и только потом сигнал постепенно получает новые значения.

## Пробуем потоковый ресурс в деле

При запуске примера сначала видно, как счётчик увеличивается каждую секунду. Вместо значений 7 и 13 он сообщает об ошибке. Однако, в отличие от RxJS, такая ошибка не завершает поток. Как только потоковый загрузчик возвращает новое значение, ресурс снова делает его доступным.

На следующем изображении показан вывод загрузчика в консоль:

Оно подчёркивает, что потоковый ресурс использует только самый новый поток, из чего и вытекает семантика switch-map.

Незадолго до каждого *clean up!* пользователь вызывал функцию forward. Это приводит к новому значению в *startValue*, что снова запускает загрузчик. Ресурс уведомляет *AbortSignal* старого потока, а его обработчик *abort* этот поток завершает. С этого момента ресурс использует новый поток, предоставленный загрузчиком, который продолжает счёт со следующей сотни.

## Совместная работа с RxJS и Observable

Вместе с Resource API в Angular 19 появился и *rxResource*, который позволяет навести мост к миру RxJS. Начиная с Angular 19.2, *rxResource* всегда является потоковым ресурсом. Это значит, что *rxResource* постепенно отдаёт значения Observable, возвращённого функцией *stream*. Чтобы продемонстрировать это поведение, в следующем листинге показана вариация ранее рассмотренного таймера — уже на основе *rxResource*:

```
export function timerResource(
  timeout: number,
  startValue: () => number
): ResourceRef<number | undefined> {

  const params = computed(() => ({
    startValue: startValue(),
  }));

  return rxResource({
    params: params,
    stream: (loaderParams) => {
      const startValue = loaderParams.params.startValue;
      return interval(timeout).pipe(
        map((v) => v + startValue + 1),
        startWith(startValue),
        tap((v) => console.log('counter', v)),
        switchMap((value) => {
          if (value === 7 || value === 13) {
            return throwError(() => new Error('bad luck'));
          }
          return [value];
        })
      );
    },
  });
}
```

Эта реализация получилась немного короче — благодаря множеству операторов, которые предлагает RxJS. В принципе, аналогичные вспомогательные функции можно было бы сделать и для сигналов и ресурсов. Однако мне кажется, что тренд в мире сигналов движется в сторону более специфичных под конкретные сценарии и потому более крупноблочных строительных кирпичиков вроде *timerResource*. Время покажет, оправдается ли это впечатление.

## Важные детали при работе с потоками в rxResource

Критическое различие между поведением Observable и потоковых ресурсов проявляется при возникновении ошибки. Именно здесь сталкиваются семантики Observable и ресурсов. Observable автоматически закрывается при первой же необработанной ошибке. По этой причине Observable внутри *rxResource* не восстанавливается после первой ошибки, которая в нашем примере возникает вместо значения 7.

Тем не менее весь *rxResource* в целом может выйти из состояния ошибки, переключившись на новый поток. Для этого, например, должен измениться сигнал *params*. Он снова запускает загрузчик, который возвращает новый Observable.

Как обычно, при переключении на новые Observable применяется семантика switch-map. Это значит, что *rxResource* в каждый момент времени использует только самый свежий Observable. Предыдущие он закрывает, отписываясь (*unsubscribing*) от них.

## Резюме

Потоковые ресурсы представляют потоки данных без опоры на RxJS. Асинхронный потоковый загрузчик обновляет сигнал, представляющий поток. Сигнал *params* ресурса запускает потоковый загрузчик, который выдаёт новый поток данных и завершает старый — это соответствует семантике switch-map, знакомой по RxJS.

*rxResource*, наводящий мост к RxJS, теперь по умолчанию является потоковым ресурсом. В отличие от Observable, ресурс, столкнувшийся с необработанной ошибкой, не закрывается, а может продолжать публиковать значения. Однако, поскольку *rxResource* построен на Observable, текущий поток не может восстановиться из состояния ошибки. Зато возможно переключение на новый поток, когда приложение снова запускает потоковый загрузчик.
