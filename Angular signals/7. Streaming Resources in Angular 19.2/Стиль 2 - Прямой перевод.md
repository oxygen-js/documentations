# Streaming Resources в Angular — детали и семантика

## Пример приложения

Используемый здесь пример — это «Hello World» реактивного программирования: простой таймер, который каждую секунду считает вверх.

Прелесть этого элементарного примера в том, что он уже содержит много материала для обсуждения небольших, но важных нюансов потоковых ресурсов (Streaming Resources). Вариация этого примера также используется для демонстрации взаимодействия с RxJS.

## Анатомия потокового ресурса

В отличие от обычных ресурсов, загрузчик (Loader) потокового ресурса всегда возвращает следующую структуру:

```
PromiseLike<Signal<ResourceStreamItem<T>>>
```

Это Promise с сигналом (Signal), хранящим `ResourceStreamItem`, который представляет либо последнее опубликованное значение, либо ошибку. Angular определяет его следующим образом:

```
type ResourceStreamItem<T> = {
    value: T;
} | {
    error: Error;
};
```

Поскольку сигнал со временем принимает несколько значений или состояний ошибки, он представляет поток данных.

В программном коде для создания потокового ресурса используется функция *resource*, которая применяется и для обычных ресурсов. Однако вместо загрузчика этой функции передаётся потоковый загрузчик (Streaming Loader) через свойство *stream*:

```
const myResource = resource({
  params: params,
  stream: async (loaderParams) => {

    // 1. Создать Signal, представляющий поток
    const result = signal<StreamItem<number>>({ 
      value: 4711 
    });

    // 2. Настроить асинхронную логику, обновляющую Signal
    […]

    // 3. Настроить обработчик очистки, запускаемый по AbortSignal
    loaderParams.abortSignal.addEventListener('abort', () => {
      […]
    });

    // 4. Вернуть Signal
    return result;
  },
});
```

Благодаря ключевому слову *async* Promise, предоставляющий сигнал, не нужно создавать явно. Загрузчик можно разделить на четыре части, которые отмечены комментариями в показанном примере:

1.  Сначала потоковый загрузчик создаёт новый сигнал, представляющий поток данных. Этому сигналу задаётся начальное значение.
2.  Потоковый загрузчик запускает асинхронную операцию, которая со временем производит несколько результатов. Он публикует эти результаты один за другим через сигнал.
3.  Потоковый загрузчик также отвечает за предоставление логики очистки, которая завершает лежащую в основе асинхронную операцию, когда её значения больше не нужны.
4.  В конце потоковый загрузчик возвращает сигнал.

Для предоставления логики очистки потоковый загрузчик использует *AbortSignal*, предоставляемый Resource API, который можно найти в переданном объекте параметров. Интересно, что эти пункты более или менее соответствуют типичному подходу при прямом использовании конструктора Observable в мире RxJS.

## Семантика switch-map при переходе между потоками

Определённая логика очистки выполняется всякий раз, когда приложению больше не нужен текущий поток данных. Для этого есть две причины. Первая возникает, когда Angular уничтожает строительный блок, в котором размещается ресурс. Представим компонент с ресурсом. При уходе роутер уничтожает этот компонент, и Angular также уничтожает ресурс.

Вторая причина возникает при изменении сигнала *params* ресурса. Каждое изменение запускает потоковый загрузчик, который затем доставляет новый поток. При переходе между этими потоками Resource API использует ту же семантику, что и *switchMap* в RxJS.

Это означает, что ресурс всегда потребляет только самый свежий поток. Такой подход обычно используется при загрузке данных. Как это обычно бывает с сигналами в Angular, цель состоит в том, чтобы предложить простые концепции для стандартных случаев. Для более сложных сценариев приложение может использовать библиотеки, такие как RxJS.

## Простой таймер на основе ресурса

Теперь, когда мы обсудили базовую структуру потокового ресурса, я хотел бы продемонстрировать его конкретное использование на первом примере: таймере, который с определённым интервалом считает число вверх. Следующий листинг показывает этот *timerResource* с точки зрения потребителя:

```
@Component([…])
export class TimerResourceComponent {

  startValue = signal(0);
  timer = timerResource(1000, this.startValue);

  forward(): void {
    this.startValue.update((v) => nextSegment(v));
  }
}

function nextSegment(currentValue: number): number {
  return Math.floor(currentValue / 100) * 100 + 100;
}
```

Таймер возвращает поток, который начинает считать с заданного начального значения. Первый параметр *timerResource* представляет желаемый интервал в миллисекундах. Начальное значение определяется сигналом *startValue*. Всякий раз, когда оно меняется, таймер переключается на новый поток. Чтобы продемонстрировать это, метод *forward* переходит к следующей полной сотне, например, с 17 на 100 или со 123 на 200.

## Фабрика для потокового ресурса

Чтобы упростить использование потокового ресурса для таймера, функция *timerResource* представляет собой просто фабрику:

```
export function timerResource(
  timeout: number,
  startValue: () => number
): ResourceRef<number | undefined> {

  const params = computed(() => ({
    startValue: startValue(),
  }));

  const result = resource({
    params: params,
    stream: async (loaderParams) => {
      const counter = loaderParams.params.startValue;
      […]
    }
  });

  return result;
}
```

Эта фабрика получает желаемый интервал (*timeout*) и сигнал со значением по умолчанию. Для этого сигнала *timerResource* интересует только геттер. Поэтому он типизирует сигнал как *() => number*.

Возвращаемое значение — *ResourceRef <number | undefined>*. Параметр типа соответствует значениям в потоке. Поскольку у ресурса нет *defaultValue*, ресурс автоматически начинает со значения *undefined*.

Вычисляемый сигнал (computed) *params* представляет все параметры, которые запускают загрузчик. Поскольку в обсуждаемом примере в качестве триггера используется только *startValue*, вычисляемый сигнал немного похож на посредника без назначения. Тем не менее мне нравится придерживаться этого подхода, особенно потому, что он позволяет позже расширить триггер и делает имя *startValue* доступным внутри (потокового) загрузчика: в обсуждаемом случае загрузчик может получить текущее значение через *param.params.startValue*.

## Потоковый загрузчик для таймера

Структура потокового загрузчика соответствует четырём разделам, обсуждённым выше:

```
const result = resource({
  params: params,
  stream: async (loaderParams) => {
    let counter = loaderParams.params.startValue;

    // 1. Создать Signal, представляющий поток
    const resultSignal = signal<StreamItem<number>>({
      value: loaderParams.params.startValue,
    });

    // 2. Настроить асинхронную логику, обновляющую Signal
    const ref = setInterval(() => {
      counter++;
      console.log('tick', counter);

      if (counter === 7 || counter === 13) {
        resultSignal.set({ error: new Error('bad luck!') });
      } else {
        resultSignal.set({ value: counter });
      }
    }, timeout);

    // 3. Настроить обработчик очистки, запускаемый по AbortSignal
    loaderParams.abortSignal.addEventListener('abort', () => {
      console.log('clean up!');
      clearInterval(ref);
    });

    // 4. Вернуть Signal
    return resultSignal;
  },
});
```

Загрузчик реализует асинхронную операцию подсчёта с помощью традиционной JavaScript-функции *setInterval*. Чтобы продемонстрировать поведение состояний ошибки, таймер сообщает об ошибке для значений 7 и 13. Тем самым удовлетворяются и потребности суеверных пользователей.

Из-за асинхронного поведения *setInterval* шаг 4 происходит до первого выполнения колбэка в шаге 2. Это означает, что загрузчик сначала возвращает сигнал с его начальным значением, и только затем сигнал постепенно получает новые значения.

## Опробование потокового ресурса

При опробовании примера можно сначала увидеть, как пример увеличивает счётчик каждую секунду. Вместо значений 7 и 13 он сообщает об ошибке. Однако, в отличие от RxJS, такая ошибка не завершает поток. Как только потоковый загрузчик возвращает новое значение, ресурс делает его доступным.

Оно подчёркивает, что потоковый ресурс использует только самый новый поток, что приводит к семантике switch-map.

Незадолго до каждого *clean up!* пользователь вызывал функцию forward. Это приводит к новому значению в *startValue*, и это обстоятельство снова запускает загрузчик. Ресурс уведомляет *AbortSignal* старого потока, а его обработчик *abort* завершает этот поток. С этого момента ресурс использует новый поток, предоставленный загрузчиком, который продолжает со следующей сотни.

## Взаимодействие с RxJS и Observable

Вместе с Resource API в Angular 19 также появился *rxResource*, который позволяет навести мост к миру RxJS. Начиная с Angular 19.2, *rxResource* всегда является потоковым ресурсом. Это означает, что *rxResource* постепенно доставляет значения Observable, возвращённого функцией *stream*. Чтобы продемонстрировать это поведение, следующий листинг показывает вариацию ранее обсуждённого таймера на основе *rxResource*:

```
export function timerResource(
  timeout: number,
  startValue: () => number
): ResourceRef<number | undefined> {

  const params = computed(() => ({
    startValue: startValue(),
  }));

  return rxResource({
    params: params,
    stream: (loaderParams) => {
      const startValue = loaderParams.params.startValue;
      return interval(timeout).pipe(
        map((v) => v + startValue + 1),
        startWith(startValue),
        tap((v) => console.log('counter', v)),
        switchMap((value) => {
          if (value === 7 || value === 13) {
            return throwError(() => new Error('bad luck'));
          }
          return [value];
        })
      );
    },
  });
}
```

Эта реализация несколько короче благодаря многочисленным операторам, которые предлагает RxJS. В принципе, аналогичные вспомогательные функции можно было бы предоставить и для сигналов и ресурсов. Однако мне кажется, что тренд в мире сигналов движется в сторону более специфичных под конкретный сценарий и потому более крупноблочных строительных блоков, таких как *timerResource*. Время покажет, окажется ли это впечатление верным.

## Важные детали при использовании потоков с rxResource

Критическое различие между поведением Observable и потоковых ресурсов раскрывается при возникновении ошибки. Именно здесь сталкиваются семантики Observable и ресурсов. Observable автоматически закрывается при первой необработанной ошибке. По этой причине Observable в *rxResource* не восстанавливается после первой ошибки, которая в нашем примере возникает вместо значения 7.

Однако весь *rxResource* в целом может восстановиться из состояния ошибки, переключившись на новый поток. Для этого, например, должен измениться сигнал *params*. Он снова запускает загрузчик, который возвращает новый Observable.

Как обычно, при переключении на новые Observable используется семантика switch-map. Это означает, что *rxResource* в каждый момент времени использует только самый свежий Observable. Предшественников он закрывает, отписываясь (*unsubscribing*) от них.

## Резюме

Потоковые ресурсы представляют потоки данных, не полагаясь на RxJS. Асинхронный потоковый загрузчик обновляет сигнал, представляющий поток. Сигнал *params* ресурса запускает потоковый загрузчик, который доставляет новый поток данных и завершает старый — это соответствует семантике switch-map, известной из RxJS.

*rxResource*, который наводит мост к RxJS, теперь по умолчанию является потоковым ресурсом. В отличие от Observable, ресурс, столкнувшийся с необработанной ошибкой, не закрывается, а может продолжать публиковать значения. Однако, поскольку *rxResource* основан на Observable, текущий поток не может восстановиться из состояния ошибки. Тем не менее возможно переключение на новый поток, когда приложение снова запускает потоковый загрузчик.
