## Пример приложения

В качестве примера здесь взят «Hello World» реактивного программирования: простой таймер, инкрементирующий значение раз в секунду.

Ценность этого элементарного примера в том, что он уже содержит достаточно материала для разбора небольших, но важных нюансов потоковых ресурсов (Streaming Resources). Вариация этого же примера используется и для демонстрации интероперабельности (interop) с RxJS.

## Анатомия потокового ресурса

В отличие от обычных (non-streaming) ресурсов, загрузчик (Loader) потокового ресурса всегда возвращает следующую структуру:

```
PromiseLike<Signal<ResourceStreamItem<T>>>
```

Это Promise, содержащий сигнал (Signal), который хранит `ResourceStreamItem` — то есть либо последнее опубликованное значение, либо ошибку. Angular определяет этот тип так:

```
type ResourceStreamItem<T> = {
    value: T;
} | {
    error: Error;
};
```

Поскольку сигнал со временем принимает несколько значений (или состояний ошибки), он представляет собой поток данных (data stream).

Для создания потокового ресурса в коде используется функция *resource* — та же, что и для обычных ресурсов. Однако вместо обычного загрузчика ей передаётся потоковый загрузчик (Streaming Loader) через свойство *stream*:

```
const myResource = resource({
  params: params,
  stream: async (loaderParams) => {

    // 1. Создаём Signal, представляющий поток
    const result = signal<StreamItem<number>>({ 
      value: 4711 
    });

    // 2. Настраиваем асинхронную логику, обновляющую Signal
    […]

    // 3. Настраиваем обработчик очистки, запускаемый по AbortSignal
    loaderParams.abortSignal.addEventListener('abort', () => {
      […]
    });

    // 4. Возвращаем Signal
    return result;
  },
});
```

Благодаря ключевому слову *async* Promise, предоставляющий сигнал, не требуется создавать явно (его оборачивает сам async-механизм). Загрузчик можно разбить на четыре части, отмеченные комментариями в приведённом примере:

1.  Сначала потоковый загрузчик создаёт новый сигнал, представляющий поток данных. Сигналу присваивается начальное значение.
2.  Потоковый загрузчик запускает асинхронную операцию, которая со временем порождает несколько результатов, и публикует их по одному через сигнал.
3.  Потоковый загрузчик также отвечает за логику очистки (cleanup), которая завершает лежащую в основе асинхронную операцию, когда её значения больше не требуются.
4.  В завершение потоковый загрузчик возвращает сигнал.

Для логики очистки потоковый загрузчик использует *AbortSignal*, предоставляемый Resource API через переданный объект параметров. Примечательно, что эти пункты в целом соответствуют типичному подходу при прямом использовании конструктора Observable в экосистеме RxJS.

## Семантика switch-map при переходе между потоками

Заданная логика очистки выполняется всякий раз, когда приложению больше не нужен текущий поток данных. Для этого есть две причины. Первая возникает, когда Angular уничтожает строительный блок (компонент, директиву и т. п.), в котором размещён ресурс. Рассмотрим компонент с ресурсом: при уходе с маршрута роутер уничтожает компонент, и Angular уничтожает связанный с ним ресурс.

Вторая причина возникает при изменении сигнала *params* ресурса. Каждое изменение запускает потоковый загрузчик, который затем доставляет новый поток. При переходе между потоками Resource API применяет ту же семантику, что и *switchMap* в RxJS (отписка от предыдущего источника при подписке на новый).

Это означает, что ресурс всегда потребляет только самый последний (most recent) поток. Такой подход обычно и применяется при загрузке данных. Как принято для сигналов в Angular, цель — предоставить простые концепции для стандартных случаев. Для более сложных сценариев приложение может задействовать библиотеки, такие как RxJS.

## Простой таймер на основе ресурса

Разобрав базовую структуру потокового ресурса, продемонстрирую его конкретное применение на первом примере — таймере, инкрементирующем число с заданным интервалом. Следующий листинг показывает *timerResource* со стороны потребителя (consumer):

```
@Component([…])
export class TimerResourceComponent {

  startValue = signal(0);
  timer = timerResource(1000, this.startValue);

  forward(): void {
    this.startValue.update((v) => nextSegment(v));
  }
}

function nextSegment(currentValue: number): number {
  return Math.floor(currentValue / 100) * 100 + 100;
}
```

Таймер возвращает поток, начинающий счёт с заданного начального значения. Первый параметр *timerResource* задаёт желаемый интервал в миллисекундах. Начальное значение определяется сигналом *startValue*. При каждом его изменении таймер переключается на новый поток. Для демонстрации метод *forward* перескакивает к следующей полной сотне, например, с 17 на 100 или со 123 на 200.

## Фабрика для потокового ресурса

Для упрощения использования потокового ресурса в таймере функция *timerResource* реализована как фабрика (factory):

```
export function timerResource(
  timeout: number,
  startValue: () => number
): ResourceRef<number | undefined> {

  const params = computed(() => ({
    startValue: startValue(),
  }));

  const result = resource({
    params: params,
    stream: async (loaderParams) => {
      const counter = loaderParams.params.startValue;
      […]
    }
  });

  return result;
}
```

Фабрика получает желаемый интервал (*timeout*) и сигнал с начальным значением. Для этого сигнала *timerResource* нужен только геттер, поэтому сигнал типизируется как *() => number*.

Возвращаемое значение — *ResourceRef <number | undefined>*. Параметр типа соответствует значениям в потоке. Поскольку у ресурса отсутствует *defaultValue*, ресурс автоматически стартует со значения *undefined*.

Вычисляемый сигнал (computed) *params* агрегирует все параметры, запускающие загрузчик. Так как в рассматриваемом примере единственным триггером выступает *startValue*, computed-сигнал кажется лишним посредником. Тем не менее я придерживаюсь этого подхода: он позволяет позже расширить набор триггеров и делает имя *startValue* доступным внутри (потокового) загрузчика — в данном случае загрузчик получает текущее значение через *param.params.startValue*.

## Потоковый загрузчик для таймера

Структура потокового загрузчика соответствует четырём разделам, рассмотренным выше:

```
const result = resource({
  params: params,
  stream: async (loaderParams) => {
    let counter = loaderParams.params.startValue;

    // 1. Создаём Signal, представляющий поток
    const resultSignal = signal<StreamItem<number>>({
      value: loaderParams.params.startValue,
    });

    // 2. Настраиваем асинхронную логику, обновляющую Signal
    const ref = setInterval(() => {
      counter++;
      console.log('tick', counter);

      if (counter === 7 || counter === 13) {
        resultSignal.set({ error: new Error('bad luck!') });
      } else {
        resultSignal.set({ value: counter });
      }
    }, timeout);

    // 3. Настраиваем обработчик очистки, запускаемый по AbortSignal
    loaderParams.abortSignal.addEventListener('abort', () => {
      console.log('clean up!');
      clearInterval(ref);
    });

    // 4. Возвращаем Signal
    return resultSignal;
  },
});
```

Загрузчик реализует асинхронную операцию счёта через традиционную JavaScript-функцию *setInterval*. Для демонстрации поведения состояний ошибки таймер сообщает об ошибке для значений 7 и 13 — так удовлетворяются и потребности суеверных пользователей.

В силу асинхронного поведения *setInterval* шаг 4 выполняется раньше первого вызова колбэка из шага 2. Таким образом, загрузчик сначала возвращает сигнал с его начальным значением, и лишь затем сигнал постепенно получает новые значения.

## Проверка потокового ресурса на практике

При запуске примера сначала видно, как счётчик инкрементируется каждую секунду. Вместо значений 7 и 13 он сообщает об ошибке. При этом, в отличие от RxJS, такая ошибка не завершает поток: как только потоковый загрузчик возвращает новое значение, ресурс делает его доступным.

Оно подчёркивает, что потоковый ресурс использует только самый новый поток, что и даёт семантику switch-map.

Незадолго до каждого *clean up!* пользователь вызывал функцию forward. Это приводит к новому значению *startValue*, что повторно запускает загрузчик. Ресурс уведомляет *AbortSignal* старого потока, а его обработчик *abort* завершает этот поток. С этого момента ресурс использует новый поток, предоставленный загрузчиком, который продолжает счёт со следующей сотни.

## Интероперабельность с RxJS и Observable

Вместе с Resource API в Angular 19 появился *rxResource*, обеспечивающий мост к экосистеме RxJS. Начиная с Angular 19.2, *rxResource* всегда является потоковым ресурсом. Это означает, что *rxResource* постепенно доставляет значения Observable, возвращённого функцией *stream*. Для демонстрации этого поведения следующий листинг показывает вариацию ранее рассмотренного таймера на основе *rxResource*:

```
export function timerResource(
  timeout: number,
  startValue: () => number
): ResourceRef<number | undefined> {

  const params = computed(() => ({
    startValue: startValue(),
  }));

  return rxResource({
    params: params,
    stream: (loaderParams) => {
      const startValue = loaderParams.params.startValue;
      return interval(timeout).pipe(
        map((v) => v + startValue + 1),
        startWith(startValue),
        tap((v) => console.log('counter', v)),
        switchMap((value) => {
          if (value === 7 || value === 13) {
            return throwError(() => new Error('bad luck'));
          }
          return [value];
        })
      );
    },
  });
}
```

Эта реализация несколько короче — благодаря множеству операторов (operators), предоставляемых RxJS. В принципе, аналогичные вспомогательные функции можно было бы предоставить и для сигналов и ресурсов. Однако, на мой взгляд, тренд в экосистеме сигналов смещается в сторону более специализированных под конкретный сценарий и потому более крупноблочных (coarse-grained) строительных блоков, таких как *timerResource*. Время покажет, подтвердится ли это наблюдение.

## Важные детали при использовании потоков с rxResource

Критическое различие в поведении Observable и потоковых ресурсов проявляется при возникновении ошибки — именно здесь семантики Observable и ресурсов вступают в конфликт. Observable автоматически закрывается при первой же необработанной ошибке. По этой причине Observable внутри *rxResource* не восстанавливается после первой ошибки, которая в нашем примере возникает вместо значения 7.

Тем не менее *rxResource* в целом может выйти из состояния ошибки за счёт переключения на новый поток. Для этого, например, должен измениться сигнал *params*: он повторно запускает загрузчик, который возвращает новый Observable.

Как обычно, при переключении на новые Observable применяется семантика switch-map. Это означает, что *rxResource* в каждый момент времени использует только самый последний Observable, а предшественников закрывает через отписку (*unsubscribing*).

## Резюме

Потоковые ресурсы представляют потоки данных без опоры на RxJS. Асинхронный потоковый загрузчик обновляет сигнал, представляющий поток. Сигнал *params* ресурса запускает потоковый загрузчик, который доставляет новый поток данных и завершает старый — это соответствует семантике switch-map, известной из RxJS.

*rxResource*, обеспечивающий мост к RxJS, теперь по умолчанию является потоковым ресурсом. В отличие от Observable, ресурс, столкнувшийся с необработанной ошибкой, не закрывается, а может продолжать публиковать значения. Однако, поскольку *rxResource* основан на Observable, текущий поток не может восстановиться из состояния ошибки. При этом возможно переключение на новый поток, когда приложение повторно запускает потоковый загрузчик.
