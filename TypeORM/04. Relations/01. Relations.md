# Связи

## Что такое связи?

Связи помогают легко работать со связанными сущностями.
Есть несколько типов связей:

- [«один к одному»](<02. One-to-one relations.md>) — через `@OneToOne`
- [«многие к одному»](<03. Many-to-one - one-to-many relations.md>) — через `@ManyToOne`
- [«один ко многим»](<03. Many-to-one - one-to-many relations.md>) — через `@OneToMany`
- [«многие ко многим»](<04. Many-to-many relations.md>) — через `@ManyToMany`

## Опции связей

Для связей можно задать несколько опций:

- `eager: boolean` (по умолчанию `false`) — если установлено в true, связь всегда будет загружаться вместе с основной сущностью при использовании методов `find*` или `QueryBuilder` для этой сущности
- `cascade: boolean | ("insert" | "update")[]` (по умолчанию `false`) — если установлено в true, связанный объект будет вставлен и обновлён в базе данных. Можно также указать массив [опций каскада](#опции-каскада).
- `onDelete: "RESTRICT"|"CASCADE"|"SET NULL"` (по умолчанию `RESTRICT`) — определяет поведение внешнего ключа при удалении объекта, на который он ссылается
- `deferrable: "INITIALLY DEFERRED"|"INITIALLY IMMEDIATE"` — если задано, ограничения внешнего ключа становятся отложенными (то есть проверяются в момент фиксации транзакции). Для связей «многие ко многим» это применяется к обоим внешним ключам join-таблицы. Поддерживается в PostgreSQL, better-sqlite3 и SAP HANA.
- `nullable: boolean` (по умолчанию `true`) — указывает, допускает ли колонка этой связи NULL. По умолчанию допускает. Для связей `ManyToOne` и владеющей стороны `OneToOne` установка `nullable: false` также заставляет TypeORM использовать `INNER JOIN` вместо `LEFT JOIN` при загрузке связи, поскольку связанная сущность гарантированно существует.
- `orphanedRowAction: "nullify" | "delete" | "soft-delete" | "disable"` (по умолчанию `nullify`) — определяет, что произойдёт с дочерними записями, которые всё ещё есть в базе данных, если родительская сущность сохраняется (с включённым каскадом) без них.
    - _nullify_ — обнулит ключ связи. Если колонка внешнего ключа не допускает NULL, осиротевшая строка будет вместо этого удалена, поскольку её нельзя установить в `null`.
    - _delete_ — удалит эти дочерние записи из базы данных.
    - _soft-delete_ — пометит дочерние записи как мягко удалённые.
    - _disable_ — оставит связь нетронутой. Для удаления придётся использовать собственный репозиторий.

## Каскады

Пример каскадов:

```typescript
import { Entity, PrimaryGeneratedColumn, Column, ManyToMany } from "typeorm"
import { Question } from "./Question"

@Entity()
export class Category {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    name: string

    @ManyToMany((type) => Question, (question) => question.categories)
    questions: Question[]
}
```

```typescript
import {
    Entity,
    PrimaryGeneratedColumn,
    Column,
    ManyToMany,
    JoinTable,
} from "typeorm"
import { Category } from "./Category"

@Entity()
export class Question {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    title: string

    @Column()
    text: string

    @ManyToMany((type) => Category, (category) => category.questions, {
        cascade: true,
    })
    @JoinTable()
    categories: Category[]
}
```

```typescript
const category1 = new Category()
category1.name = "ORMs"

const category2 = new Category()
category2.name = "Programming"

const question = new Question()
question.title = "How to ask questions?"
question.text = "Where can I ask TypeORM-related questions?"
question.categories = [category1, category2]
await dataSource.manager.save(question)
```

Как видно из этого примера, мы не вызывали `save` для `category1` и `category2`.
Они будут вставлены автоматически, потому что мы установили `cascade` в true.

Помните: большая сила требует большой ответственности.
Каскады могут показаться хорошим и простым способом работы со связями,
но они также способны привести к ошибкам и проблемам безопасности, когда в базу данных сохраняется нежелательный объект.
Кроме того, они делают сохранение новых объектов в базу менее явным.

### Опции каскада

Опцию `cascade` можно задать как `boolean` или как массив опций каскада `("insert" | "update" | "remove" | "soft-remove" | "recover")[]`.

По умолчанию она равна `false`, то есть каскады отключены. Значение `cascade: true` включает полный каскад. Можно также указать конкретные опции, передав массив.

Например:

```typescript
@Entity(Post)
export class Post {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    title: string

    @Column()
    text: string

    // Полный каскад для categories.
    @ManyToMany((type) => PostCategory, {
        cascade: true,
    })
    @JoinTable()
    categories: PostCategory[]

    // Каскад insert здесь означает, что если в этой связи задан новый экземпляр PostDetails,
    // он будет автоматически вставлен в базу при сохранении этой сущности Post
    @ManyToMany((type) => PostDetails, (details) => details.posts, {
        cascade: ["insert"],
    })
    @JoinTable()
    details: PostDetails[]

    // Каскад update здесь означает, что если есть изменения в существующем PostImage,
    // они будут автоматически сохранены в базу при сохранении этой сущности Post
    @ManyToMany((type) => PostImage, (image) => image.posts, {
        cascade: ["update"],
    })
    @JoinTable()
    images: PostImage[]

    // Каскад insert и update здесь означает, что новые экземпляры PostInformation
    // или изменения существующих будут автоматически вставлены или обновлены
    // при сохранении этой сущности Post
    @ManyToMany((type) => PostInformation, (information) => information.posts, {
        cascade: ["insert", "update"],
    })
    @JoinTable()
    informations: PostInformation[]
}
```

> **Примечание** Каскадное удаление. При использовании `cascade: ["remove"]` или `cascade: true` вызов `manager.remove(entity)` также удалит связанные сущности, загруженные в экземпляр сущности. TypeORM обходит только те связи, которые заполнены в объекте: если связь не загружена, её дочерние записи каскадно удалены не будут. Не забудьте загрузить связи перед удалением:
>
> ```typescript
> const post = await manager.findOne(Post, {
>     where: { id: 1 },
>     relations: { categories: true },
> })
> await manager.remove(post) // categories тоже будут удалены
> ```

## Опции `@JoinColumn`

`@JoinColumn` не только определяет, какая сторона связи содержит join-колонку с внешним ключом,
но и позволяет настроить имя join-колонки и имя колонки, на которую она ссылается.

Когда мы указываем `@JoinColumn`, в базе данных автоматически создаётся колонка с именем `propertyName + referencedColumnName`.
Например:

```typescript
@ManyToOne(type => Category)
@JoinColumn() // этот декоратор необязателен для @ManyToOne, но обязателен для @OneToOne
category: Category;
```

Этот код создаст в базе данных колонку `categoryId`.
Если вы хотите изменить это имя, можно указать собственное имя join-колонки:

```typescript
@ManyToOne(type => Category)
@JoinColumn({ name: "cat_id" })
category: Category;
```

Join-колонки всегда ссылаются на какие-то другие колонки (через внешний ключ).
По умолчанию связь всегда ссылается на первичную колонку связанной сущности.
Если вы хотите создать связь с другими колонками связанной сущности —
их также можно указать в `@JoinColumn`:

```typescript
@ManyToOne(type => Category)
@JoinColumn({ referencedColumnName: "name" })
category: Category;
```

Теперь связь ссылается на `name` сущности `Category`, а не на `id`.
Имя колонки для этой связи станет `categoryName`.

Можно объединять и несколько колонок. Учтите, что по умолчанию они не ссылаются на первичную колонку связанной сущности: имя колонки, на которую идёт ссылка, нужно указать явно.

```typescript
@ManyToOne(type => Category)
@JoinColumn([
    { name: "category_id", referencedColumnName: "id" },
    { name: "locale_id", referencedColumnName: "locale_id" }
])
category: Category;
```

> **Примечание** При использовании составных `@JoinColumn` или `@JoinTable` TypeORM автоматически сортирует колонки внешнего ключа в соответствии с порядком первичного ключа связанной сущности. Это обеспечивает совместимость с базами данных вроде MySQL, MSSQL и SAP HANA, которые требуют, чтобы колонки FK ссылались на колонки PK в порядке индекса.

## Опции `@JoinTable`

`@JoinTable` используется для связей `many-to-many` и описывает join-колонки «связующей» (junction) таблицы.
Junction-таблица — это отдельная специальная таблица, которую TypeORM создаёт автоматически, с колонками, ссылающимися на связанные сущности.
Имена колонок внутри junction-таблиц и колонок, на которые они ссылаются, можно менять с помощью `@JoinColumn`.
Можно также изменить имя генерируемой «связующей» таблицы.

```typescript
@ManyToMany(type => Category)
@JoinTable({
    name: "question_categories", // имя junction-таблицы для этой связи
    joinColumn: {
        name: "question",
        referencedColumnName: "id"
    },
    inverseJoinColumn: {
        name: "category",
        referencedColumnName: "id"
    }
})
categories: Category[];
```

Если у целевой таблицы составной первичный ключ,
то в `@JoinTable` нужно передать массив свойств.
