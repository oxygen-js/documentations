# Postgres / CockroachDB

TypeORM поддерживает драйверы для PostgreSQL, CockroachDB и Amazon Aurora Postgres.

Базы данных, совместимые с PostgreSQL, также можно использовать с TypeORM через тип источника данных `postgres`.

Чтобы начать работу с YugabyteDB, обратитесь к [их документации по ORM](https://docs.yugabyte.com/stable/drivers-orms/nodejs/typeorm/). Учтите, что часть возможностей Postgres [не поддерживается](https://docs.yugabyte.com/stable/develop/postgresql-compatibility/#unsupported-postgresql-features) в YugabyteDB, поэтому некоторая функциональность TypeORM может быть ограничена.

## Установка

```shell
npm install pg
```

Для поддержки потоковой передачи:

```shell
npm install pg-query-stream
```

## Опции источника данных

Общие опции источника данных описаны в разделе [Опции источника данных](<../02. Data Source/02. Data Source Options.md>). Для подключения к соответствующим базам данных используйте тип источника данных `postgres`, `cockroachdb` или `aurora-postgres`.

- `url` — URL подключения. Учтите, что остальные опции источника данных переопределяют параметры, заданные в URL.

- `host` — хост базы данных.

- `port` — порт хоста базы данных. Порт Postgres по умолчанию — `5432`.

- `username` — имя пользователя базы данных.

- `password` — пароль базы данных.

- `database` — имя базы данных.

- `schema` — имя схемы. По умолчанию «public».

- `connectTimeoutMS` — время в миллисекундах до истечения таймаута при первоначальном подключении к серверу postgres. Если значение `undefined` или `0`, таймаута нет. По умолчанию `undefined`.

- `ssl` — объект с параметрами ssl. См. [TLS/SSL](https://node-postgres.com/features/ssl).

- `uuidExtension` — расширение Postgres, используемое при генерации UUID. По умолчанию `uuid-ossp`. Можно заменить на `pgcrypto`, если расширение `uuid-ossp` недоступно.

- `poolErrorHandler` — функция, вызываемая, когда нижележащий пул порождает событие `'error'`. Принимает единственный параметр (экземпляр ошибки); по умолчанию пишет в лог с уровнем `warn`.

- `maxTransactionRetries` — максимальное количество повторов транзакции при ошибке 40001. По умолчанию 5.

- `logNotifications` — логическое значение, определяющее, включать ли [уведомительные сообщения](https://www.postgresql.org/docs/current/plpgsql-errors-and-messages.html) сервера postgres и [события уведомлений](https://www.postgresql.org/docs/current/sql-notify.html) в логи клиента с уровнем `info` (по умолчанию: `false`).

- `installExtensions` — логическое значение, управляющее тем, устанавливать ли необходимые расширения postgres автоматически (по умолчанию: `true`)

- `extensions` — список дополнительных расширений Postgres, которые нужно установить в базе данных (по умолчанию: `undefined`)

- `applicationName` — строка, видимая в статистике и логах; помогает соотнести приложение с соединением (по умолчанию: `undefined`)

- `parseInt8` — логическое значение, включающее разбор 64-битных целых чисел (int8) как чисел JavaScript. По умолчанию значения `int8` (bigint) возвращаются строками, чтобы избежать переполнения. Числа в JavaScript соответствуют IEEE-754 и теряют точность за пределами максимального безопасного целого (`Number.MAX_SAFE_INTEGER = +2^53`). Если вам нужен полный 64-битный диапазон, лучше работать с возвращаемыми строками или преобразовывать их в нативный `bigint`, а не использовать эту опцию.

Дополнительные опции можно добавить в объект `extra` — они передаются напрямую в клиентскую библиотеку. Подробнее — в документации `pg` по [Pool](https://node-postgres.com/apis/pool#new-pool) и [Client](https://node-postgres.com/apis/client#new-client).

## Типы колонок

### Типы колонок для `postgres`

`int`, `int2`, `int4`, `int8`, `smallint`, `integer`, `bigint`, `decimal`, `numeric`, `real`, `float`, `float4`, `float8`, `double precision`, `money`, `character varying`, `varchar`, `character`, `char`, `text`, `citext`, `hstore`, `bytea`, `bit`, `varbit`, `bit varying`, `timetz`, `timestamptz`, `timestamp`, `timestamp without time zone`, `timestamp with time zone`, `date`, `time`, `time without time zone`, `time with time zone`, `interval`, `bool`, `boolean`, `enum`, `point`, `line`, `lseg`, `box`, `path`, `polygon`, `circle`, `cidr`, `inet`, `macaddr`, `macaddr8`, `tsvector`, `tsquery`, `uuid`, `xml`, `json`, `jsonb`, `jsonpath`, `int4range`, `int8range`, `numrange`, `tsrange`, `tstzrange`, `daterange`, `int4multirange`, `int8multirange`, `nummultirange`, `tsmultirange`, `tstzmultirange`, `multidaterange`, `geometry`, `geography`, `cube`, `ltree`, `vector`, `halfvec`.

### Типы колонок для `cockroachdb`

`array`, `bool`, `boolean`, `bytes`, `bytea`, `blob`, `date`, `numeric`, `decimal`, `dec`, `float`, `float4`, `float8`, `double precision`, `real`, `inet`, `int`, `integer`, `int2`, `int8`, `int64`, `smallint`, `bigint`, `interval`, `string`, `character varying`, `character`, `char`, `char varying`, `varchar`, `text`, `time`, `time without time zone`, `timestamp`, `timestamptz`, `timestamp without time zone`, `timestamp with time zone`, `json`, `jsonb`, `uuid`

Примечание: CockroachDB возвращает все числовые типы данных как `string`. Однако, если не указывать тип колонки и объявить свойство как `number`, ORM применит `parseInt` и преобразует строку в число.

### Векторные колонки

Векторные колонки можно использовать для поиска по сходству с помощью векторных операторов PostgreSQL:

```typescript
// Расстояние L2 (евклидово) — <->
const results = await dataSource.sql`
    SELECT id, embedding
    FROM post
    ORDER BY embedding <-> ${"[1,2,3]"}
    LIMIT 5`

// Косинусное расстояние — <=>
const results = await dataSource.sql`
    SELECT id, embedding
    FROM post
    ORDER BY embedding <=> ${"[1,2,3]"}
    LIMIT 5`

// Скалярное произведение — <#>
const results = await dataSource.sql`
    SELECT id, embedding
    FROM post
    ORDER BY embedding <#> ${"[1,2,3]"}
    LIMIT 5`
```

### Пространственные колонки

Поддержка PostgreSQL и CockroachDB в TypeORM использует [GeoJSON](http://geojson.org/) как формат обмена, поэтому колонки геометрии следует помечать типом `object` или `Geometry` (либо его подклассами, например `Point`), предварительно импортировав [типы `geojson`](https://www.npmjs.com/package/@types/geojson) или воспользовавшись встроенными в TypeORM типами GeoJSON:

```typescript
import {
    Entity,
    PrimaryColumn,
    Column,
    Point,
    LineString,
    MultiPoint
} from "typeorm"

@Entity()
export class Thing {
    @PrimaryColumn()
    id: number

    @Column("geometry")
    point: Point

    @Column("geometry")
    linestring: LineString

    @Column("geometry", {
        spatialFeatureType: "MultiPoint",
        srid: 4326,
    })
    multiPointWithSRID: MultiPoint
}

...

const thing = new Thing()
thing.point = {
    type: "Point",
    coordinates: [116.443987, 39.920843],
}
thing.linestring = {
    type: "LineString",
    coordinates: [
        [-87.623177, 41.881832],
        [-90.199402, 38.627003],
        [-82.446732, 38.413651],
        [-87.623177, 41.881832],
    ],
}
thing.multiPointWithSRID = {
    type: "MultiPoint",
    coordinates: [
        [100.0, 0.0],
        [101.0, 1.0],
    ],
}
```

TypeORM старается поступать правильно, но не всегда возможно определить,
когда вставляемое значение или результат функции PostGIS следует трактовать
как геометрию. Поэтому вам может понадобиться писать код вроде приведённого
ниже, где значения преобразуются из GeoJSON в `geometry` PostGIS и обратно
в GeoJSON как `json`:

```typescript
import { Point } from "typeorm"

const origin: Point = {
    type: "Point",
    coordinates: [0, 0],
}

await dataSource.manager
    .createQueryBuilder(Thing, "thing")
    // преобразуем GeoJSON в виде строки в геометрию с SRID, совпадающим
    // со спецификацией таблицы
    .where(
        "ST_Distance(geom, ST_SetSRID(ST_GeomFromGeoJSON(:origin), ST_SRID(geom))) > 0",
    )
    .orderBy(
        "ST_Distance(geom, ST_SetSRID(ST_GeomFromGeoJSON(:origin), ST_SRID(geom)))",
        "ASC",
    )
    .setParameters({
        // сериализуем GeoJSON в строку
        origin: JSON.stringify(origin),
    })
    .getMany()

await dataSource.manager
    .createQueryBuilder(Thing, "thing")
    // преобразуем результат-геометрию в GeoJSON, трактуемый как JSON (чтобы
    // TypeORM знал, что его нужно десериализовать)
    .select("ST_AsGeoJSON(ST_Buffer(geom, 0.1))::json geom")
    .from("thing")
    .getMany()
```
