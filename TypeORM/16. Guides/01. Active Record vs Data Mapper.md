# Active Record против Data Mapper

## Что такое паттерн Active Record?

В TypeORM можно использовать оба паттерна — Active Record и Data Mapper.

При подходе Active Record все методы запросов вы объявляете внутри самой модели, а сохраняете, удаляете и загружаете объекты с помощью методов модели.

Проще говоря, Active Record — это подход, при котором доступ к базе данных осуществляется внутри моделей.
Подробнее о паттерне Active Record можно почитать в [Википедии](https://en.wikipedia.org/wiki/Active_record_pattern).

Пример:

```typescript
import { BaseEntity, Entity, PrimaryGeneratedColumn, Column } from "typeorm"

@Entity()
export class User extends BaseEntity {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    firstName: string

    @Column()
    lastName: string

    @Column()
    isActive: boolean
}
```

Все active-record сущности должны наследоваться от класса `BaseEntity`, который предоставляет методы для работы с сущностью.
Пример работы с такой сущностью:

```typescript
// пример сохранения AR-сущности
const user = new User()
user.firstName = "Timber"
user.lastName = "Saw"
user.isActive = true
await user.save()

// пример удаления AR-сущности
await user.remove()

// пример загрузки AR-сущностей
const users = await User.find({ skip: 2, take: 5 })
const newUsers = await User.findBy({ isActive: true })
const timber = await User.findOneBy({ firstName: "Timber", lastName: "Saw" })
```

У `BaseEntity` есть большинство методов стандартного `Repository`.
В большинстве случаев с active-record сущностями вам не понадобятся ни `Repository`, ни `EntityManager`.

Допустим, теперь нам нужна функция, возвращающая пользователей по имени и фамилии.
Такие функции можно создать как статический метод класса `User`:

```typescript
import { BaseEntity, Entity, PrimaryGeneratedColumn, Column } from "typeorm"

@Entity()
export class User extends BaseEntity {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    firstName: string

    @Column()
    lastName: string

    @Column()
    isActive: boolean

    static findByName(firstName: string, lastName: string) {
        return this.createQueryBuilder("user")
            .where("user.firstName = :firstName", { firstName })
            .andWhere("user.lastName = :lastName", { lastName })
            .getMany()
    }
}
```

И пользоваться им так же, как остальными методами:

```typescript
const timber = await User.findByName("Timber", "Saw")
```

## Что такое паттерн Data Mapper?

В TypeORM можно использовать оба паттерна — Active Record и Data Mapper.

При подходе Data Mapper все методы запросов вы объявляете в отдельных классах — «репозиториях», и сохраняете, удаляете и загружаете объекты через репозитории.
В data mapper сущности предельно «глупые»: они лишь описывают свои свойства и могут иметь несколько простейших методов.

Проще говоря, data mapper — это подход, при котором доступ к базе данных осуществляется в репозиториях, а не в моделях.
Подробнее о data mapper можно почитать в [Википедии](https://en.wikipedia.org/wiki/Data_mapper_pattern).

Пример:

```typescript
import { Entity, PrimaryGeneratedColumn, Column } from "typeorm"

@Entity()
export class User {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    firstName: string

    @Column()
    lastName: string

    @Column()
    isActive: boolean
}
```

Пример работы с такой сущностью:

```typescript
const userRepository = dataSource.getRepository(User)

// пример сохранения DM-сущности
const user = new User()
user.firstName = "Timber"
user.lastName = "Saw"
user.isActive = true
await userRepository.save(user)

// пример удаления DM-сущности
await userRepository.remove(user)

// пример загрузки DM-сущностей
const users = await userRepository.find({ skip: 2, take: 5 })
const newUsers = await userRepository.findBy({ isActive: true })
const timber = await userRepository.findOneBy({
    firstName: "Timber",
    lastName: "Saw",
})
```

Чтобы расширить стандартный репозиторий собственными методами, используйте [паттерн пользовательского репозитория](<../07. Working with Entity Manager/04. Custom repositories.md>).

## Какой подход выбрать?

Решать вам.
У обеих стратегий есть свои плюсы и минусы.

При разработке ПО всегда стоит помнить о том, как мы будем поддерживать наши приложения.
Подход `Data Mapper` помогает с поддерживаемостью и потому эффективнее в крупных приложениях.
Подход `Active Record` помогает сохранять простоту, что хорошо работает в небольших приложениях.
