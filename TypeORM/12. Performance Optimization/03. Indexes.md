# Использование индексов

Индексы — это структуры данных, позволяющие базе данных находить строки без полного сканирования таблицы. Без индекса запрос вынужден просматривать каждую строку в поисках совпадений. С индексом база данных может сразу перейти к нужным строкам, резко сокращая количество просматриваемых строк.

Индексы ускоряют операции чтения, но добавляют накладные расходы при записи (вставке, обновлении, удалении), поскольку индекс необходимо поддерживать в актуальном состоянии. Поэтому создавать их следует обдуманно и только там, где это действительно нужно.

Хорошие кандидаты на индексирование — колонки, часто используемые в условиях `WHERE`, в соединениях и при сортировке. Если фильтрация идёт более чем по одной колонке, может иметь смысл создать [составной индекс](<../05. Indexes.md#индексы-по-нескольким-колонкам>), но учитывайте, что порядок колонок важен: наиболее селективные (с наибольшим числом уникальных значений) колонки нужно ставить первыми.

## Анализ производительности запросов

Большинство баз данных позволяют посмотреть, как выполняется запрос, — это называется планом выполнения. Понимание такого плана помогает выявить недостающие индексы и оптимизировать построение запроса, сравнивая разные способы получить одни и те же данные.

#### CockroachDB

Используйте `EXPLAIN ANALYZE` (синтаксис тот же, что и в PostgreSQL):

```sql
EXPLAIN ANALYZE SELECT * FROM "user" WHERE "firstName" = 'Timber';
```

Обращайте внимание на `full scan` (индекс не используется) в противовес `scan` с конкретным именем индекса.

> **Информация** Подробнее — [Indexes](https://www.cockroachlabs.com/docs/stable/indexes.html)
> - [EXPLAIN](https://www.cockroachlabs.com/docs/stable/explain.html)

#### Google Spanner

Используйте [визуализатор плана запроса](https://cloud.google.com/spanner/docs/query-execution-plans#query_plan_visualizer) в Google Cloud Console либо выполните запрос в режиме `PLAN`/`PROFILE` через CLI `gcloud`:

```bash
gcloud spanner databases execute-sql DATABASE_ID \
    --instance=INSTANCE_ID \
    --query-mode=PROFILE \
    --sql="SELECT * FROM user WHERE firstName = 'Timber'"
```

В возвращённом плане обращайте внимание на `Table Scan` (индекс не используется) в противовес `Index Scan`.

> **Информация** Подробнее — [Secondary Indexes](https://cloud.google.com/spanner/docs/secondary-indexes)
> - [Query Execution Plans](https://cloud.google.com/spanner/docs/query-execution-plans)

#### MariaDB

Для просмотра плана выполнения запроса используйте `EXPLAIN` или `ANALYZE`:

```sql
ANALYZE SELECT * FROM user WHERE firstName = 'Timber';
```

Смотрите на колонку `type`: значение `ALL` означает полное сканирование таблицы, а `ref`, `range` или `index` говорят об использовании индекса. Колонка `key` показывает, какой индекс был выбран.

> **Информация** Подробнее — [Optimization and Indexes](https://mariadb.com/kb/en/optimization-and-indexes/)
> - [EXPLAIN](https://mariadb.com/kb/en/explain/)
> - [ANALYZE Statement](https://mariadb.com/kb/en/analyze-statement/)

#### MongoDB

Чтобы увидеть план выполнения, вызовите для запроса метод `explain()`:

```javascript
db.user.find({ firstName: "Timber" }).explain("executionStats")
```

Обращайте внимание на `COLLSCAN` (сканирование коллекции — индекс не используется) в противовес `IXSCAN` (сканирование по индексу).

> **Информация** Подробнее — [Indexes](https://www.mongodb.com/docs/manual/indexes/)
> - [Explain Results](https://www.mongodb.com/docs/manual/reference/explain-results/)

#### MySQL

Для просмотра плана выполнения запроса используйте `EXPLAIN`:

```sql
EXPLAIN SELECT * FROM user WHERE firstName = 'Timber';
```

Смотрите на колонку `type`: значение `ALL` означает полное сканирование таблицы, а `ref`, `range` или `index` говорят об использовании индекса. Колонка `key` показывает, какой индекс был выбран.

> **Совет** Aurora MySQL Это относится и к Amazon Aurora MySQL, которая использует тот же движок запросов и тот же синтаксис EXPLAIN. Рекомендации, специфичные для Aurora, см. в [Aurora MySQL tuning](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/AuroraMySQL.Managing.Tuning.html).

> **Информация** Подробнее — [Optimization and Indexes](https://dev.mysql.com/doc/refman/en/optimization-indexes.html)
> - [EXPLAIN Output Format](https://dev.mysql.com/doc/refman/en/explain-output.html)

#### Oracle

Для просмотра плана выполнения используйте `EXPLAIN PLAN`:

```sql
EXPLAIN PLAN FOR SELECT * FROM "user" WHERE "firstName" = 'Timber';
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

Обращайте внимание на `TABLE ACCESS FULL` (индекс не используется) в противовес `INDEX RANGE SCAN` или `INDEX UNIQUE SCAN`.

> **Информация** Подробнее — [Indexes and Index-Organized Tables](https://docs.oracle.com/en/database/oracle/oracle-database/23/cncpt/indexes-and-index-organized-tables.html)
> - [Generating and Displaying Execution Plans](https://docs.oracle.com/en/database/oracle/oracle-database/23/tgsql/generating-and-displaying-execution-plans.html)

#### PostgreSQL

Используйте `EXPLAIN ANALYZE`, чтобы увидеть, как PostgreSQL выполняет запрос и использует ли он индексы:

```sql
EXPLAIN ANALYZE SELECT * FROM "user" WHERE "firstName" = 'Timber';
```

Обращайте внимание на `Seq Scan` (полное сканирование таблицы — индекс не используется) в противовес `Index Scan` или `Index Only Scan` (индекс используется).

> **Совет** Aurora PostgreSQL Это относится и к Amazon Aurora PostgreSQL, которая использует тот же движок запросов и тот же синтаксис EXPLAIN. Кроме того, в Aurora есть [управление планами запросов](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/AuroraPostgreSQL.Optimize.html) для фиксации и контроля планов выполнения.

> **Информация** Подробнее — [Indexes](https://www.postgresql.org/docs/current/indexes.html)
> - [EXPLAIN](https://www.postgresql.org/docs/current/sql-explain.html)
> - [Using EXPLAIN](https://www.postgresql.org/docs/current/using-explain.html)

#### SAP HANA

Для просмотра плана выполнения используйте `EXPLAIN PLAN FOR`:

```sql
EXPLAIN PLAN FOR SELECT * FROM "user" WHERE "firstName" = 'Timber';
SELECT * FROM EXPLAIN_PLAN_TABLE;
```

В колонке оператора обращайте внимание на `TABLE SCAN` (индекс не используется) в противовес `INDEX SCAN` или `INDEX SEEK`.

> **Информация** Подробнее — [Indexes](https://help.sap.com/docs/hana-cloud-database/sap-hana-cloud-sap-hana-database-sql-reference-guide/indexes)
> - [EXPLAIN PLAN](https://help.sap.com/docs/hana-cloud-database/sap-hana-cloud-sap-hana-database-sql-reference-guide/explain-plan-statement-data-manipulation)

#### Microsoft SQL Server

Для анализа запросов используйте план выполнения. В SQL Server Management Studio нажмите `Ctrl+M`, чтобы включить фактический план выполнения, и запустите запрос. Программно:

```sql
SET STATISTICS IO ON;
SELECT * FROM [user] WHERE firstName = 'Timber';
SET STATISTICS IO OFF;
```

В плане выполнения обращайте внимание на `Table Scan` (индекс не используется) в противовес `Index Seek` или `Index Scan`.

> **Информация** Подробнее — [Indexes](https://learn.microsoft.com/en-us/sql/relational-databases/indexes/indexes)
> - [Execution Plans](https://learn.microsoft.com/en-us/sql/relational-databases/performance/execution-plans)

#### SQLite

Чтобы увидеть, как SQLite разбирает запрос, используйте `EXPLAIN QUERY PLAN`:

```sql
EXPLAIN QUERY PLAN SELECT * FROM user WHERE firstName = 'Timber';
```

В выводе обращайте внимание на `SCAN` (индекс не используется) в противовес `SEARCH` (индекс используется).

> **Информация** Подробнее — [Query Planning](https://www.sqlite.org/queryplanner.html)
> - [EXPLAIN QUERY PLAN](https://www.sqlite.org/eqp.html)

## Определение индексов в TypeORM

TypeORM поддерживает создание индексов по колонкам таблицы с помощью декоратора `@Index`.

#### Простой индекс

```typescript
import { Entity, PrimaryGeneratedColumn, Column, Index } from "typeorm"

@Entity()
export class User {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    @Index()
    email: string

    @Column()
    firstName: string
}
```

#### Составной индекс

```typescript
import { Entity, PrimaryGeneratedColumn, Column, Index } from "typeorm"

@Entity()
@Index(["lastName", "firstName"]) // наиболее селективная колонка идёт первой
export class User {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    firstName: string

    @Column()
    lastName: string
}
```

#### Join-колонка

```typescript
import {
    Entity,
    PrimaryGeneratedColumn,
    Column,
    ManyToOne,
    Index,
} from "typeorm"
import { User } from "./User"

@Entity()
export class Post {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    title: string

    @ManyToOne(() => User)
    @Index() // индексируйте колонки внешних ключей, чтобы ускорить соединения
    author: User
}
```

Подробный обзор различных типов индексов (уникальных, пространственных, полнотекстовых, конкурентных и других) — в руководстве [Индексы](<../05. Indexes.md>).

Правильная индексация зачастую даёт наибольший прирост производительности из всех возможных: начните с анализа самых медленных запросов с помощью описанных выше средств просмотра плана выполнения, а затем добавьте индексы там, где они дадут наибольший эффект.
