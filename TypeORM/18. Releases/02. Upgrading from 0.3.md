# Обновление с 0.3 до 1.0

Это руководство по переходу с версии `0.3.x` на `1.0`.

## Автоматическое обновление

Пакет `@typeorm/codemod` умеет автоматизировать большинство обратно несовместимых изменений, описанных в этом руководстве:

```bash
npx @typeorm/codemod v1 src/
```

Он изменит код на месте — используйте `--dry`, чтобы посмотреть изменения без записи на диск. Codemod берёт на себя переименования импортов, замены API, синтаксис опций поиска, обновление зависимостей и многое другое. Всё, что автоматизировать не удалось, помечается комментариями `TODO` для ручной правки.

Полное описание использования и опций см. в [README codemod](https://github.com/typeorm/typeorm/tree/master/packages/codemod).

## Требования к платформе

### Node.js 20+

Минимальная цель компиляции JavaScript теперь — `ES2023`, а для неё требуется **Node.js 20 или новее**. Если вы используете более старую версию Node.js, обновите её перед обновлением TypeORM.

### `Buffer` заменён на `Uint8Array` на не-Node платформах

Браузерный полифил `Buffer` удалён. На платформах, отличных от Node (браузер, Deno, Bun), двоичные данные теперь представлены типом `Uint8Array`. Пользователей Node.js это не затрагивает — в Node `Buffer` наследуется от `Uint8Array` и продолжает работать как прежде.

## Изменения в драйверах

### MySQL / MariaDB

#### Опция `connectorPackage` удалена

Опция `connectorPackage` удалена вместе с поддержкой старого клиента `mysql`. Единственный поддерживаемый клиент базы данных — `mysql2`, TypeORM попытается загрузить его по умолчанию. Если в вашем проекте использовался `mysql`, просто замените его на `mysql2`.

#### Значение `legacySpatialSupport` по умолчанию изменено на `false`

Опция `legacySpatialSupport` теперь по умолчанию равна `false`, то есть TypeORM использует соответствующие стандарту пространственные функции `ST_GeomFromText` и `ST_AsText`, появившиеся в MySQL 5.7 и обязательные в MySQL 8.0+. Устаревшие функции `GeomFromText` и `AsText` были удалены в MySQL 8.0.

Если вы работаете с MySQL 5.6 или более ранней версией и используете пространственные типы, задайте `legacySpatialSupport: true` явно:

```typescript
new DataSource({
    type: "mysql",
    legacySpatialSupport: true,
    // ...
})
```

#### Опции колонок `width` и `zerofill` удалены

В MySQL 8.0.17 ширина отображения для целочисленных типов и атрибут `ZEROFILL` были объявлены устаревшими, а в MySQL 8.4 — удалены полностью. TypeORM больше не поддерживает опции колонок `width` и `zerofill`. Если вы их использовали, удалите их из описаний колонок:

```typescript
// Было
@Column({ type: "int", width: 9, zerofill: true })
postCode: number

// Стало
@Column({ type: "int" })
postCode: number
```

Если вам нужно форматирование с ведущими нулями, реализуйте его на уровне приложения через `String.prototype.padStart()` или функцией MySQL `LPAD()` в сыром запросе. Опция `unsigned` для целочисленных типов этим изменением **не** затронута.

#### Сигнатура `QueryBuilder.useIndex` изменена

При указании подсказки запроса `USE INDEX` метод `QueryBuilder.useIndex()` больше не принимает сырой SQL — только имя индекса или массив имён индексов.

```typescript
// Было
qb.useIndex("index1, `index#2`")

// Стало
qb.useIndex(["index1", "index#2"])
```

### SQLite

Поддержка пакета `sqlite3` прекращена. Используйте вместо него `better-sqlite3`:

```typescript
// Было
new DataSource({
    type: "sqlite",
    database: "db.sqlite",
})

// Стало
new DataSource({
    type: "better-sqlite3",
    database: "db.sqlite",
})
```

#### Опция `flags` удалена

Пакет `sqlite3` принимал флаги открытия уровня C (`OPEN_URI`, `OPEN_SHAREDCACHE` и т. п.). `better-sqlite3` этого не поддерживает — используйте специальные опции:

- `readonly` для режима «только чтение»
- `enableWAL` для режима журналирования WAL

#### Опция `busyTimeout` переименована в `timeout`

В пакете `sqlite3` для настройки busy timeout использовалась опция `busyTimeout`. В `better-sqlite3` вместо неё используется `timeout` (по умолчанию 5000 мс):

```typescript
// Было
new DataSource({
    type: "sqlite",
    database: "db.sqlite",
    busyTimeout: 2000,
})

// Стало
new DataSource({
    type: "better-sqlite3",
    database: "db.sqlite",
    timeout: 2000,
})
```

### MongoDB

Теперь TypeORM требует **драйвер `mongodb` для Node.js версии 7 или новее** (`^7.0.0`). Поддержка драйвера `mongodb` v5/v6 прекращена.

#### Устаревшие опции подключения удалены

Следующие опции подключения к MongoDB были удалены:

| Удалённая опция         | Что делать                                                     |
| ----------------------- | -------------------------------------------------------------- |
| `appname`               | Используйте `appName` (camelCase)                              |
| `fsync`                 | Используйте `writeConcern: { journal: true }`                  |
| `j`                     | Используйте `writeConcern: { journal: true }`                  |
| `keepAlive`             | Удалите — включено всегда, начиная с MongoDB Driver v6.0       |
| `keepAliveInitialDelay` | Удалите — не настраивается, начиная с MongoDB Driver v6.0      |
| `ssl`                   | Используйте `tls`                                              |
| `sslCA`                 | Используйте `tlsCAFile`                                        |
| `sslCRL`                | Удалите — в современном драйвере замены нет                    |
| `sslCert`               | Используйте `tlsCertificateKeyFile`                            |
| `sslKey`                | Используйте `tlsCertificateKeyFile`                            |
| `sslPass`               | Используйте `tlsCertificateKeyFilePassword`                    |
| `sslValidate`           | Используйте `tlsAllowInvalidCertificates` (обратное значение)  |
| `useNewUrlParser`       | Удалите — ничего не делает, начиная с MongoDB Driver v4.0      |
| `useUnifiedTopology`    | Удалите — ничего не делает, начиная с MongoDB Driver v4.0      |
| `w`                     | Используйте `writeConcern: { w: 1 }`                           |
| `wtimeout`              | Используйте `writeConcern: { wtimeoutMS: 2500 }`               |
| `wtimeoutMS`            | Используйте `writeConcern: { wtimeoutMS: 2500 }`               |

#### Метод `stats()` удалён

Метод `stats()` удалён из `MongoQueryRunner`, `MongoEntityManager` и `MongoRepository`. Лежащая в его основе команда `collStats` объявлена устаревшей в MongoDB server 6.2, а метод `Collection.stats()` удалён в MongoDB Driver v7.

Используйте вместо него стадию агрегации [`$collStats`](https://www.mongodb.com/docs/manual/reference/operator/aggregation/collStats/). Обратите внимание, что форма ответа отличается: свойства вроде `count`, `size` и `storageSize` вложены в `storageStats`, а не лежат на верхнем уровне.

```typescript
// Было
const stats = await mongoRepository.stats()
console.log(stats.count)
console.log(stats.size)
console.log(stats.totalIndexSize)

// Стало — используем стадию агрегации $collStats
const [stats] = await dataSource.mongoManager
    .aggregate(MyEntity, [{ $collStats: { storageStats: {} } }])
    .toArray()
console.log(stats.storageStats.count)
console.log(stats.storageStats.size)
console.log(stats.storageStats.totalIndexSize)
```

#### Глобальные функции `getMongoRepository` и `getMongoManager` удалены

Устаревшие глобальные функции `getMongoRepository()` и `getMongoManager()` удалены. Используйте соответствующие методы экземпляров `DataSource` или `EntityManager`:

```typescript
// Было
import { getMongoManager, getMongoRepository } from "typeorm"

const manager = getMongoManager()
const repository = getMongoRepository(User)

// Стало
const manager = dataSource.mongoManager
const repository = dataSource.getMongoRepository(User)
```

#### Типы

Внутренние типы MongoDB (`ObjectId` и т. п.) больше не реэкспортируются из `typeorm`. Импортируйте их напрямую из `mongodb`:

```typescript
// Было
import { ObjectId } from "typeorm"

// Стало
import { ObjectId } from "mongodb"
```

### MS SQL Server

#### Опция подключения `domain` удалена

Устаревшая опция `domain` в `SqlServerConnectionCredentialsOptions` удалена. Используйте опцию `authentication` с типом NTLM:

```typescript
// Было
new DataSource({
    type: "mssql",
    domain: "MYDOMAIN",
    username: "user",
    password: "pass",
    // ...
})

// Стало
new DataSource({
    type: "mssql",
    authentication: {
        type: "ntlm",
        options: {
            domain: "MYDOMAIN",
            userName: "user",
            password: "pass",
        },
    },
    // ...
})
```

#### `options.isolation` и `options.connectionIsolationLevel`

Опция `options.isolation` в `SqlServerDataSourceOptions` переименована в `options.isolationLevel`, поскольку изначальное имя было некорректным. Также обратите внимание, что формат значения изменился с `READ_COMMITTED` на `READ COMMITTED` (подчёркивание заменено пробелом) — ради соответствия формату, принятому в TypeORM повсеместно. Обновите опции вашего DataSource:

```typescript
// Было
new DataSource({
    type: "mssql",
    options: {
        isolation: "READ_COMMITTED",
        connectionIsolationLevel: "READ_COMMITTED",
        // ...
    },
    // ...
})

// Стало
new DataSource({
    type: "mssql",
    options: {
        isolationLevel: "READ COMMITTED",
        connectionIsolationLevel: "READ COMMITTED",
        // ...
    },
    // ...
})
```

### Oracle

`LegacyOracleNamingStrategy` больше не экспортируется из основного пакета TypeORM. Если он вам всё ещё нужен, импортируйте его из `@typeorm/legacy-naming-strategies`.

```typescript
// Было
import { LegacyOracleNamingStrategy } from "typeorm"

// Стало
import { LegacyOracleNamingStrategy } from "@typeorm/legacy-naming-strategies"
```

### SAP HANA

Несколько устаревших псевдонимов опций подключения SAP HANA были удалены.

- `hanaClientDriver` удалён. Используйте `driver`.
- `pool.max` удалён. Используйте `pool.maxConnectedOrPooled`.
- `pool.requestTimeout` удалён. Используйте `pool.maxWaitTimeoutIfPoolExhausted`.
- `pool.idleTimeout` удалён. Используйте `pool.maxPooledIdleTime` (в секундах).
- `pool.min`, `pool.maxWaitingRequests` и `pool.checkInterval` удалены без замены.

Также обратите внимание на изменения поведения по умолчанию в настройке пула:

- `pool.maxPooledIdleTime` теперь по умолчанию равен `30` секундам и больше не берёт значение из `pool.idleTimeout`.
- `pool.maxWaitTimeoutIfPoolExhausted` теперь по умолчанию равен `0` и больше не берёт значение из `pool.requestTimeout`.

### Expo

Минимальная поддерживаемая версия Expo SDK — 52, в ней используется современный асинхронный API SQLite. TypeORM теперь загружает `expo-sqlite` автоматически, поэтому опция `driver` больше не нужна:

```typescript
// Было
new DataSource({
    type: "expo",
    driver: require("expo-sqlite"),
    database: "db.sqlite",
})

// Стало
new DataSource({
    type: "expo",
    database: "db.sqlite",
})
```

### Redis (кэш)

Удалена поддержка устаревших (v3) клиентов Redis в `RedisQueryResultCache`. Обновитесь до **клиента Redis версии 4 или новее** (`redis`, `ioredis`):

```typescript
// Было — redis v3
import { createClient } from "redis"
const client = createClient()

// Стало — redis v4+
import { createClient } from "redis"
const client = createClient()
await client.connect()
```

## Источник данных

### `Connection` → `DataSource`

`DataSource` пришёл на смену `Connection` ещё в v0.3. Теперь обратно совместимый псевдоним удалён:

```typescript
// Было
import { Connection, ConnectionOptions } from "typeorm"

const connection = await createConnection(options)
await connection.close()

// Стало
import { DataSource, DataSourceOptions } from "typeorm"

const dataSource = new DataSource(options)
await dataSource.initialize()
await dataSource.destroy()
```

Повсеместно применяются следующие переименования:

| Было                                 | Стало                      |
| ------------------------------------ | -------------------------- |
| `Connection`                         | `DataSource`               |
| `ConnectionOptions`                  | `DataSourceOptions`        |
| `BaseConnectionOptions`              | `BaseDataSourceOptions`    |
| `MysqlConnectionOptions`             | `MysqlDataSourceOptions`   |
| _(та же схема для всех драйверов)_   |                            |
| `connection.connect()`               | `dataSource.initialize()`  |
| `connection.close()`                 | `dataSource.destroy()`     |
| `connection.isConnected`             | `dataSource.isInitialized` |

### Свойство `name` удалено

Устаревшее свойство `name` у `DataSource` и `BaseDataSourceOptions` удалено. Именованные соединения объявлены устаревшими в v0.3, когда был удалён `ConnectionManager`. Если вы использовали `name` для идентификации соединений, управляйте экземплярами `DataSource` напрямую.

Примечание: код, читающий `dataSource.name`, теперь получит `undefined` вместо `"default"`. Если это значение используется в логировании или логике мультитенантности, обновите его.

### Свойство `.connection` в разных классах теперь называется `.dataSource`

Свойство `connection` в классах `Driver`, `QueryRunner`, `EntityManager`, `QueryBuilder`, `EntityMetadata` и `*Event` переименовано в `dataSource`. Для `EntityManager` это изменение анонсировали ещё в 0.3, но фактически не реализовали. Чтобы упростить переход, добавлен устаревший геттер, возвращающий то же значение, что и `dataSource`.

### Разное

Класс `ConnectionManager` удалён. Если вы использовали его для управления несколькими соединениями, создавайте экземпляры `DataSource` и управляйте ими напрямую.

`ConnectionOptionsReader` также упрощён: метод `all()` переименован в `get()` (возвращает все конфигурации массивом), а старые методы `get(name)` и `has(name)` удалены. Теперь он ищет файл `ormconfig` в `process.cwd()`, а не по пути приложения. Изменить место поиска можно опцией `root`.

```typescript
const reader = new ConnectionOptionsReader({ root: "/path/to/config/" })

// когда в вашем ormconfig один источник данных
const [options] = await reader.get()

// когда из нескольких источников данных нужна конкретная конфигурация
const allOptions = await reader.get()
const postgresOptions = allOptions.find((o) => o.type === "postgres")
```

### Глобальные функции-помощники удалены

Следующие устаревшие глобальные функции удалены:

- `createConnection` / `createConnections`
- `getConnection`
- `getConnectionManager`
- `getConnectionOptions`
- `getManager`
- `getSqljsManager`
- `getRepository`
- `getTreeRepository`
- `createQueryBuilder`

Используйте эквивалентные методы вашего экземпляра `DataSource`:

```typescript
// Было
const repo = getRepository(User)
const qb = createQueryBuilder("user")

// Стало
const repo = dataSource.getRepository(User)
const qb = dataSource.createQueryBuilder("user")
```

### Конфигурация через переменные окружения удалена

Устаревший класс `ConnectionOptionsEnvReader` и возможность настраивать соединения через `TYPEORM_CONNECTION`, `TYPEORM_URL` и прочие переменные окружения `TYPEORM_*` удалены. Формат файла `ormconfig.env` также больше не поддерживается. TypeORM больше не подгружает `.env`-файлы автоматически и не зависит от `dotenv`.

Используйте вместо этого конфигурационный файл на TypeScript или JavaScript:

```typescript
// ormconfig.ts
export default {
    type: process.env.DB_TYPE,
    url: process.env.DB_URL,
    // ...
}
```

## Изменения в поведении

### Значение `invalidWhereValuesBehavior` по умолчанию изменено на `throw`

**Это существенное изменение поведения, которое может сломать существующие приложения во время выполнения.**

Поведение по умолчанию для значений null и undefined в условиях where изменилось. Раньше такие значения молча игнорировались (свойство пропускалось). Теперь оба **по умолчанию приводят к ошибке**.

Это изменение предотвращает трудноуловимые ошибки, когда запрос вида `findBy({ id: undefined })` вместо падения молча возвращал все строки.

```typescript
// v0.3: молча возвращает все посты (null игнорируется)
// v1.0: выбрасывает TypeORMError
await repository.find({ where: { text: null } })

// v0.3: молча возвращает все посты (undefined игнорируется)
// v1.0: выбрасывает TypeORMError
await repository.find({ where: { text: undefined } })
```

Чтобы искать по значению null, используйте оператор `IsNull()`:

```typescript
import { IsNull } from "typeorm"

await repository.find({ where: { text: IsNull() } })
```

Чтобы вернуть прежнее поведение, задайте `invalidWhereValuesBehavior` в опциях источника данных:

```typescript
new DataSource({
    // ...
    invalidWhereValuesBehavior: {
        null: "ignore",
        undefined: "ignore",
    },
})
```

Эта настройка охватывает все высокоуровневые API — операции поиска, изменяющие методы репозитория и менеджера, а также `queryBuilder.setFindOptions()` (единственный метод QueryBuilder, которого она касается). Остальные методы QueryBuilder (`.where()`, `.andWhere()`, `.orWhere()`) **не** затронуты: значения null и undefined проходят через них как есть. Подробности см. в разделе [Обработка null и undefined](<../02. Data Source/05. Handling null and undefined values in where conditions.md>).

### Хеширование

В TypeORM v1 алгоритм хеширования SHA1 применяется непосредственно к входным данным. В предыдущих версиях (v0.3) входные данные сначала кодировались функцией [encodeURIComponent](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/encodeURIComponent).

Если вы используете кэш результатов запросов TypeORM, существующие записи кэша после обновления станут недействительными, поскольку хеш-функция выдаёт другой результат. Кэш будет перестроен автоматически — вы можете заметить кратковременный рост промахов кэша.

Если имена таблиц и колонок содержат специальные символы, можно воспользоваться `@typeorm/legacy-naming-strategies`, чтобы избежать изменений в базе данных:

```typescript
import { NamingStrategyV03 } from "@typeorm/legacy-naming-strategies"

const dataSource = new DataSource({
    ...
    namingStrategy: new NamingStrategyV03(),
    ...
})
```

### Glob-шаблоны

Glob-шаблоны (используемые при поиске файлов сущностей и миграций) теперь обрабатываются библиотекой `tinyglobby` вместо `glob`. Для большинства проектов это прозрачная замена.

### `orphanedRowAction: "nullify"` с внешними ключами, не допускающими NULL

Когда `orphanedRowAction` равен `"nullify"` (значение по умолчанию), а колонка внешнего ключа не допускает NULL, осиротевшие дочерние записи теперь **удаляются**, а не приводят к нарушению ограничения базы данных. Раньше TypeORM пытался записать во внешний ключ `null`, что для колонок с NOT NULL заканчивалось ошибкой.

Это работает только тогда, когда связь загружена в экземпляр сущности. TypeORM не загружает связи автоматически — он обходит только те значения связей, которые уже заполнены в объекте. Чтобы осиротевшие дочерние записи были обработаны, загрузите связь до вызова `remove` или `save`:

```typescript
const parent = await manager.findOne(Parent, {
    where: { id: 1 },
    relations: { children: true },
})
await manager.remove(parent)
```

Если связь не загружена (то есть свойство равно `undefined`), TypeORM не обнаружит и не удалит осиротевшие дочерние записи, что может привести к нарушению ограничений внешнего ключа.

Если вы полагались на ошибку как на защиту от случайного удаления дочерних записей, задайте у связи `orphanedRowAction: "disable"`, чтобы сохранить прежнее поведение.

### Каскадное удаление теперь работает для связей «один ко многим»

Вызов `manager.remove(entity)` с `cascade: true` или `cascade: ["remove"]` для связи «один ко многим» теперь корректно удаляет дочерние сущности перед родительской. Раньше дочерние сущности каскадно не удалялись, из-за чего DELETE падал с нарушением ограничения внешнего ключа. Кроме того, каскадные операции теперь ограничены соответствующим типом операции: например, `save()` следует только каскадам `insert`/`update` и больше не обходит связи, помеченные только `remove`.

### Строки join-таблиц «многие ко многим» и мягко удалённые сущности

Вызов `recover()` для мягко удалённой сущности со связями «многие ко многим» больше не приводит к нарушению уникальности ключа. `softRemove` не трогает строки join-таблицы, но раньше TypeORM не видел их при загрузке сущности для восстановления и пытался вставить дубликаты.

Как побочный эффект, сравнение join-записей «многие ко многим» при `save()` теперь учитывает и мягко удалённые связанные сущности. Если вы явно задаёте массив связи, исключив из него мягко удалённую сущность, её строка в join-таблице будет удалена:

```typescript
// photo2 был мягко удалён независимо, но его строка в join-таблице существует
user.manyToManyPhotos = [photo1] // photo2 исключён
await manager.save(user) // строка join-таблицы для photo2 теперь удалена
```

Это работает только при явно заданном свойстве связи. Если оно равно `undefined`, сравнение не выполняется и строки join-таблицы остаются нетронутыми.

### Логгер

`FileLogger` теперь передаёт работу с путями нижележащей платформе (например, NodeJS), а не вычисляет путь относительно корня приложения. Если приложение запускается не из своей корневой папки, вы можете указать абсолютный путь (или путь относительно `process.cwd()`):

```typescript
const dataSource = new DataSource({
    logger: new FileLogger("all", { logPath: "/path/to/file.log" }),
})
```

## Колонки

### Опция `readonly` удалена

Устаревшая опция колонки `readonly` удалена. Используйте вместо неё опцию `update` — учтите, что она принимает **противоположное** значение:

```typescript
// Было
@Column({ readonly: true })
authorName: string

// Стало
@Column({ update: false })
authorName: string
```

### `unsigned` в `ColumnNumericOptions` удалено

Устаревшее свойство `unsigned` в `ColumnNumericOptions` (использовалось с перегрузками типов колонок decimal/float, например `@Column("decimal", { unsigned: true })`) удалено, поскольку MySQL объявил `UNSIGNED` устаревшим для нецелочисленных числовых типов. Опция `unsigned` в `ColumnOptions` для целочисленных типов **не** затронута.

## Связи

### `nullable: false` теперь использует INNER JOIN

Связи, помеченные `nullable: false`, при загрузке через `relations`, жадную загрузку или опции поиска теперь используют `INNER JOIN` вместо `LEFT JOIN`. Это касается только тех типов связей, которые владеют join-колонкой (`ManyToOne` и владеющая сторона `OneToOne`).

Семантически это верно, поскольку внешний ключ, не допускающий NULL, гарантирует существование связанной сущности, и позволяет оптимизатору базы данных строить более эффективные планы запросов.

**Потенциально ломающее изменение:** если в вашей базе есть строки, нарушающие ограничение `NOT NULL` (например, осиротевшие внешние ключи, или `nullable: false` был задан, тогда как колонка в БД фактически допускает NULL), такие строки будут исключены из результатов запроса. Проверьте целостность данных или при необходимости измените связь на `nullable: true`.

```typescript
// INNER JOIN — связанная сущность гарантированно существует
@ManyToOne(() => User, { nullable: false })
author: User

// LEFT JOIN — связанной сущности может не быть (по умолчанию)
@ManyToOne(() => User)
optionalEditor: User
```

Связи `OneToMany`, `ManyToMany` и обратная сторона `OneToOne` всегда используют `LEFT JOIN` независимо от настройки `nullable`, поскольку у этих типов связей нет join-колонки в текущей таблице.

**Исключение для мягкого удаления:** если у связанной сущности есть `@DeleteDateColumn`, применяется `LEFT JOIN` даже для связей с `nullable: false` (если не задано `withDeleted: true`). Это не даёт мягко удалённым связанным сущностям отфильтровывать свои родительские строки.

## Репозиторий

### `findOneById`

Устаревший метод `findOneById` удалён из `EntityManager`, `Repository`, `BaseEntity`, `MongoEntityManager` и `MongoRepository`. Используйте вместо него `findOneBy`:

```typescript
// Было
const user = await manager.findOneById(User, 1)
const user = await repository.findOneById(1)
const user = await User.findOneById(1)

// Стало
const user = await manager.findOneBy(User, { id: 1 })
const user = await repository.findOneBy({ id: 1 })
const user = await User.findOneBy({ id: 1 })
```

Для сущностей MongoDB с `@ObjectIdColumn()` метод `findOneBy` работает точно так же — TypeORM автоматически преобразует имя свойства в `_id`.

### `findByIds` удалён

Устаревший метод `findByIds` удалён из `EntityManager`, `Repository` и `BaseEntity`. Используйте `findBy` вместе с оператором `In`:

```typescript
// Было
const users = await repository.findByIds([1, 2, 3])

// Стало
import { In } from "typeorm"

const users = await repository.findBy({ id: In([1, 2, 3]) })
```

### `exist` переименован в `exists`

Устаревший метод `Repository.exist()` удалён. Используйте `exists()` — поведение идентично:

```typescript
// Было
const hasUsers = await userRepository.exist({ where: { isActive: true } })

// Стало
const hasUsers = await userRepository.exists({ where: { isActive: true } })
```

### `AbstractRepository`, `@EntityRepository` и `getCustomRepository` удалены

Класс `AbstractRepository`, декоратор `@EntityRepository` и метод `getCustomRepository()` удалены. Они были объявлены устаревшими в v0.3 в пользу `Repository.extend()`:

```typescript
// Было
@EntityRepository(User)
class UserRepository extends AbstractRepository<User> {
    findByName(name: string) {
        return this.repository.findOneBy({ name })
    }
}
const userRepo = dataSource.getCustomRepository(UserRepository)

// Стало
const UserRepository = dataSource.getRepository(User).extend({
    findByName(name: string) {
        return this.findOneBy({ name })
    },
})
```

Также удалены следующие классы ошибок: `CustomRepositoryDoesNotHaveEntityError`, `CustomRepositoryCannotInheritRepositoryError`, `CustomRepositoryNotFoundError`.

### Декоратор `@RelationCount` и `loadRelationCountAndMap` удалены

Декоратор `@RelationCount` и метод `SelectQueryBuilder.loadRelationCountAndMap()` удалены. Используйте вместо них `@VirtualColumn` или подзапрос в query builder:

```typescript
// Было
@RelationCount((post: Post) => post.categories)
categoryCount: number

// Стало — используем @VirtualColumn с подзапросом
// Замените имя join-таблицы и имена колонок в соответствии с вашей схемой
@VirtualColumn({
    query: (alias) =>
        `SELECT COUNT(*) FROM post_categories_category WHERE postId = ${alias}.id`,
})
categoryCount: number
```

## Опции поиска

### Опция `join` удалена

Устаревшее свойство `join` в `FindOneOptions` и `FindManyOptions` удалено вместе с интерфейсом `JoinOptions`.

#### `leftJoinAndSelect` → `relations`

Если вы использовали `leftJoinAndSelect`, замените его объектным синтаксисом `relations` — `relations` всегда выполняет LEFT JOIN с выборкой, что эквивалентно:

```typescript
// Было
const posts = await repository.find({
    join: {
        alias: "post",
        leftJoinAndSelect: {
            categories: "post.categories",
            author: "post.author",
        },
    },
})

// Стало
const posts = await repository.find({
    relations: { categories: true, author: true },
})
```

#### Все остальные виды join → QueryBuilder

Опция `relations` поддерживает только LEFT JOIN с выборкой. Если вы использовали `innerJoinAndSelect`, `innerJoin` или `leftJoin` (без выборки), переходите на API QueryBuilder:

```typescript
// Было — innerJoinAndSelect
const posts = await repository.find({
    join: {
        alias: "post",
        innerJoinAndSelect: {
            categories: "post.categories",
        },
    },
})

// Стало — QueryBuilder с innerJoinAndSelect
const posts = await repository
    .createQueryBuilder("post")
    .innerJoinAndSelect("post.categories", "categories")
    .getMany()

// Было — leftJoin (без выборки)
const posts = await repository.find({
    join: {
        alias: "post",
        leftJoin: {
            categories: "post.categories",
        },
    },
    where: { categories: { isRemoved: false } },
})

// Стало — QueryBuilder с leftJoin
const posts = await repository
    .createQueryBuilder("post")
    .leftJoin("post.categories", "categories")
    .where("categories.isRemoved = :isRemoved", { isRemoved: false })
    .getMany()
```

На практике это различие важно. Например, PostgreSQL и CockroachDB не допускают `FOR UPDATE` на стороне внешнего соединения, допускающей NULL, поэтому запросам, сочетающим блокировки с присоединёнными связями, могут понадобиться INNER JOIN:

```typescript
// Было — innerJoinAndSelect + блокировка
const post = await repository.findOne({
    join: {
        alias: "post",
        innerJoinAndSelect: {
            categories: "post.categories",
        },
    },
    lock: { mode: "pessimistic_write", tables: ["category"] },
})

// Стало — QueryBuilder с innerJoinAndSelect + блокировкой
const post = await repository
    .createQueryBuilder("post")
    .innerJoinAndSelect("post.categories", "categories")
    .setLock("pessimistic_write", undefined, ["categories"])
    .getOne()
```

#### Блокировки с вложенными связями → QueryBuilder

Опцию `relations` нельзя использовать вместе с пессимистичными блокировками присоединённых таблиц, поскольку `relations` всегда применяет LEFT JOIN, а PostgreSQL/CockroachDB отклоняют `FOR UPDATE` на стороне внешнего соединения, допускающей NULL. Используйте QueryBuilder с `innerJoinAndSelect`:

```typescript
// Было — вложенные связи + блокировка через опции поиска
const post = await repository.findOne({
    where: { id: 1 },
    join: {
        alias: "post",
        innerJoinAndSelect: {
            categories: "post.categories",
            images: "categories.images",
        },
    },
    lock: { mode: "pessimistic_write", tables: ["images"] },
})

// Стало — QueryBuilder с innerJoinAndSelect + блокировкой
const post = await repository
    .createQueryBuilder("post")
    .innerJoinAndSelect("post.categories", "categories")
    .innerJoinAndSelect("categories.images", "images")
    .where("post.id = :id", { id: 1 })
    .setLock("pessimistic_write", undefined, ["images"])
    .getOne()
```

Обратите внимание, что блокировка _основной_ таблицы по-прежнему работает с `relations` — QueryBuilder с inner join нужен только для блокировки _присоединённых_ таблиц.

### Строковый вариант `select` удалён

Устаревший синтаксис `select` в виде массива строк в опциях поиска удалён. Используйте объектный синтаксис:

```typescript
// Было
const users = await repository.find({
    select: ["id", "name"],
})

// Стало
const users = await repository.find({
    select: { id: true, name: true },
})
```

Удалённый тип — `FindOptionsSelectByString`.

### Строковый вариант `relations` удалён

Устаревший синтаксис `relations` в виде массива строк в опциях поиска удалён. Используйте объектный синтаксис:

```typescript
// Было
const users = await repository.find({
    relations: ["profile", "posts"],
})

// Стало
const users = await repository.find({
    relations: { profile: true, posts: true },
})
```

Удалённый тип — `FindOptionsRelationByString`.

## QueryBuilder

### `printSql` удалён

Метод `printSql()` у query builder'ов удалён. Он был избыточен: при включённом логировании запросов все выполненные запросы и так автоматически логируются настроенным логгером. Чтобы посмотреть сгенерированный SQL, используйте `getSql()` или `getQueryAndParameters()`:

```typescript
// Было
const users = await dataSource
    .getRepository(User)
    .createQueryBuilder("user")
    .where("user.id = :id", { id: 1 })
    .printSql()
    .getMany()

// Стало — смотрим SQL перед выполнением
const qb = dataSource
    .getRepository(User)
    .createQueryBuilder("user")
    .where("user.id = :id", { id: 1 })

console.log(qb.getSql())
// или: const [sql, params] = qb.getQueryAndParameters()

const users = await qb.getMany()
```

Чтобы автоматически логировать все выполняемые запросы, включите логирование запросов в DataSource:

```typescript
new DataSource({
    // ...
    logging: ["query"],
})
```

### `onConflict` удалён

Метод `onConflict()` у `InsertQueryBuilder` удалён. Он принимал сырые SQL-строки, специфичные для конкретного драйвера, и легко приводил к ошибкам. Используйте вместо него `orIgnore()` или `orUpdate()`:

```typescript
// Было
await dataSource
    .createQueryBuilder()
    .insert()
    .into(Post)
    .values(post)
    .onConflict(`("id") DO NOTHING`)
    .execute()

// Стало
await dataSource
    .createQueryBuilder()
    .insert()
    .into(Post)
    .values(post)
    .orIgnore()
    .execute()

// Было
await dataSource
    .createQueryBuilder()
    .insert()
    .into(Post)
    .values(post)
    .onConflict(`("id") DO UPDATE SET "title" = :title`)
    .setParameter("title", post.title)
    .execute()

// Стало
await dataSource
    .createQueryBuilder()
    .insert()
    .into(Post)
    .values(post)
    .orUpdate(["title"], ["id"])
    .execute()
```

### Объектная перегрузка `orUpdate` удалена

Объектная перегрузка `orUpdate()`, принимавшая `{ columns?, overwrite?, conflict_target? }`, удалена. Используйте сигнатуру с массивами:

```typescript
// Было
.orUpdate({ conflict_target: ["date"], overwrite: ["title"] })

// Стало
.orUpdate(["title"], ["date"])
```

### `setNativeParameters` удалён

```typescript
// Было
qb.setNativeParameters({ key: "value" })

// Стало
qb.setParameters({ key: "value" })
```

### Псевдоним типа `WhereExpression` удалён

```typescript
// Было
import { WhereExpression } from "typeorm"

// Стало
import { WhereExpressionBuilder } from "typeorm"
```

### `replacePropertyNames` удалён

Устаревший защищённый метод `replacePropertyNames()` у QueryBuilder удалён. Если вы переопределяли его в собственном подклассе, это переопределение больше не вызывается.

### Устаревшие режимы блокировки удалены

```typescript
// Было
.setLock("pessimistic_partial_write")

// Стало
.setLock("pessimistic_write")
.setOnLocked("skip_locked")

// Было
.setLock("pessimistic_write_or_fail")

// Стало
.setLock("pessimistic_write")
.setOnLocked("nowait")
```

То же самое относится к опциям поиска:

```typescript
// Было
{ lock: { mode: "pessimistic_partial_write" } }

// Стало
{ lock: { mode: "pessimistic_write", onLocked: "skip_locked" } }

// Было
{ lock: { mode: "pessimistic_write_or_fail" } }

// Стало
{ lock: { mode: "pessimistic_write", onLocked: "nowait" } }
```

## Миграции

### `getAllMigrations` удалён

Устаревший метод `getAllMigrations()` удалён из `MigrationExecutor`. Используйте `getPendingMigrations()` или `getExecutedMigrations()`, либо обращайтесь напрямую к `dataSource.migrations` за списком зарегистрированных классов миграций:

```typescript
// Было
const migrations = await migrationExecutor.getAllMigrations()

// Стало — в зависимости от того, что вам нужно
const pending = await migrationExecutor.getPendingMigrations()
const executed = await migrationExecutor.getExecutedMigrations()
const registered = dataSource.migrations
```

### `QueryRunner.loadedTables` и `loadedViews` удалены

```typescript
// Было
const tables = queryRunner.loadedTables
const views = queryRunner.loadedViews

// Стало
const tables = await queryRunner.getTables()
const views = await queryRunner.getViews()
```

Примечание: замены — это асинхронные методы, а не синхронные свойства.

## Система контейнеров

Устаревшая интеграция с IoC-контейнером удалена: `useContainer()`, `getFromContainer()`, `ContainerInterface`, `ContainedType` и `UseContainerOptions`.

В TypeORM больше нет встроенной поддержки IoC-контейнеров. Пакеты `typeorm-typedi-extensions` и `typeorm-routing-controllers-extensions` также больше не совместимы. В разделах ниже описано, как выполнить переход в зависимости от вашей конфигурации.

### Подписчики и миграции с зависимостями

TypeORM всегда создаёт подписчиков и миграции внутри себя, используя конструктор без аргументов, поэтому передать готовые экземпляры невозможно. Если вашим миграциям нужен доступ к сервисам, используйте внутри самой миграции `DataSource` (доступен через `queryRunner.dataSource`):

```typescript
// Было
import { useContainer } from "typeorm"
import { Container } from "typedi"
useContainer(Container)

// Стало — доступ к зависимостям через DataSource внутри миграции
export class MyMigration1234 implements MigrationInterface {
    public async up(queryRunner: QueryRunner): Promise<void> {
        const repo = queryRunner.dataSource.getRepository(User)
        // ...
    }
}
```

### Доступ к репозиториям и entity manager

Если раньше вы использовали `typeorm-typedi-extensions`, чтобы внедрять `EntityManager` или репозитории в свои сервисы, обращайтесь напрямую к `DataSource`:

```typescript
// Было (с typeorm-typedi-extensions)
import { InjectManager, InjectRepository } from "typeorm-typedi-extensions"

class UserService {
    @InjectManager()
    private manager: EntityManager

    @InjectRepository(User)
    private userRepository: Repository<User>
}

// Стало — получаем из экземпляра DataSource
class UserService {
    private manager: EntityManager
    private userRepository: Repository<User>

    constructor(dataSource: DataSource) {
        this.manager = dataSource.manager
        this.userRepository = dataSource.getRepository(User)
    }
}
```

### Использование с DI-фреймворком

Если вы используете DI-фреймворк, зарегистрируйте `DataSource` (или его репозитории) как провайдеры в своём контейнере:

```typescript
// пример для typedi
import { DataSource } from "typeorm"
import { Container } from "typedi"

const dataSource = new DataSource({/* ... */})
await dataSource.initialize()
Container.set(DataSource, dataSource)
Container.set("UserRepository", dataSource.getRepository(User))
```

### NestJS

Пользователей NestJS это не затрагивает: у пакета `@nestjs/typeorm` собственная интеграция, не зависящая от удалённой системы контейнеров TypeORM. Однако `@nestjs/typeorm` v10 и v11.0.0 пытаются зарегистрировать удалённый класс `Connection` и падают при запуске. Для совместимости с TypeORM v1 обновитесь до `@nestjs/typeorm` v11.0.1 или новее. Инструмент `@typeorm/codemod` поднимет версию автоматически.

## Прочие удаления внутренних API

Следующие внутренние API удалены. Это касается вас, только если вы писали собственные драйверы, расширяли QueryBuilder или использовали низкоуровневые API метаданных:

| Удалено                                        | Замена                                                       |
| ---------------------------------------------- | ------------------------------------------------------------ |
| `EntityMetadata.createPropertyPath()` (static) | Удалён, публичной замены нет                                 |
| `EntityMetadata.getValueMap()` — параметр `options` | Уберите третий аргумент — он никогда не работал         |
| `DriverUtils.buildColumnAlias()`               | Используйте `DriverUtils.buildAlias()`                       |
| `Broadcaster.broadcastLoadEventsForAll()`      | Замены нет — используйте отдельных подписчиков на события    |
| `QueryExpressionMap.nativeParameters`          | Используйте `QueryExpressionMap.parameters`                  |
| `RdbmsSchemaBuilder.renameTables()`            | Удалён                                                       |
