# Сущности

## Что такое сущность?

Сущность — это класс, который отображается на таблицу базы данных (или на коллекцию, если вы используете MongoDB).
Чтобы создать сущность, объявите новый класс и пометьте его `@Entity()`:

```typescript
import { Entity, PrimaryGeneratedColumn, Column } from "typeorm"

@Entity()
export class User {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    firstName: string

    @Column()
    lastName: string

    @Column()
    isActive: boolean
}
```

Это создаст следующую таблицу в базе данных:

```text
+-------------+--------------+----------------------------+
|                          user                           |
+-------------+--------------+----------------------------+
| id          | int          | PRIMARY KEY AUTO_INCREMENT |
| firstName   | varchar(255) |                            |
| lastName    | varchar(255) |                            |
| isActive    | boolean      |                            |
+-------------+--------------+----------------------------+
```

Базовые сущности состоят из колонок и связей.
Каждая сущность **ОБЯЗАНА** иметь первичную колонку (или колонку `ObjectId`, если вы используете MongoDB).

Каждая сущность должна быть зарегистрирована в опциях источника данных:

```typescript
import { DataSource } from "typeorm"
import { User } from "./entities/User"

const myDataSource = new DataSource({
    type: "mysql",
    host: "localhost",
    port: 3306,
    username: "test",
    password: "test",
    database: "test",
    entities: [User],
})
```

Либо можно указать целый каталог со всеми сущностями внутри — и все они будут загружены:

```typescript
import { DataSource } from "typeorm"

const dataSource = new DataSource({
    type: "mysql",
    host: "localhost",
    port: 3306,
    username: "test",
    password: "test",
    database: "test",
    entities: [__dirname + "/entities/**/*{.js,.ts}"],
})
```

Если для сущности `User` нужно другое имя таблицы, укажите его в `@Entity`: `@Entity("my_users")`.
Если нужно задать общий префикс для всех таблиц базы данных в вашем приложении, укажите `entityPrefix` в опциях источника данных.

Если у сущности есть конструктор, все его аргументы **должны быть необязательными**. ORM создаёт экземпляры классов сущностей при загрузке из базы данных и ничего не знает об аргументах вашего конструктора.

Подробнее о параметрах `@Entity` — в [справочнике по декораторам](<../17. Help/03. Decorator reference.md>).

## Колонки сущности

Раз таблицы базы данных состоят из колонок, то и ваши сущности должны состоять из колонок.
Каждое свойство класса сущности, помеченное `@Column`, будет отображено на колонку таблицы базы данных.

### Первичные колонки

У каждой сущности должна быть хотя бы одна первичная колонка.
Есть несколько видов первичных колонок:

- `@PrimaryColumn()` создаёт первичную колонку, принимающую любое значение любого типа. Тип колонки можно указать явно. Если тип не указан, он будет выведен из типа свойства. Пример ниже создаст `id` типа `int`, значение которого вы обязаны присвоить вручную перед сохранением.

```typescript
import { Entity, PrimaryColumn } from "typeorm"

@Entity()
export class User {
    @PrimaryColumn()
    id: number
}
```

- `@PrimaryGeneratedColumn()` создаёт первичную колонку, значение которой генерируется автоматически с автоинкрементом. Будет создана колонка `int` с `auto-increment`/`serial`/`sequence`/`identity` (в зависимости от базы данных и заданной конфигурации). Присваивать значение вручную перед сохранением не нужно — оно сгенерируется автоматически.

```typescript
import { Entity, PrimaryGeneratedColumn } from "typeorm"

@Entity()
export class User {
    @PrimaryGeneratedColumn()
    id: number
}
```

- `@PrimaryGeneratedColumn("uuid")` создаёт первичную колонку, значение которой генерируется автоматически как `uuid`. Uuid — это уникальный строковый идентификатор. Присваивать значение вручную перед сохранением не нужно — оно сгенерируется автоматически.

```typescript
import { Entity, PrimaryGeneratedColumn } from "typeorm"

@Entity()
export class User {
    @PrimaryGeneratedColumn("uuid")
    id: string
}
```

Первичные колонки могут быть и составными:

```typescript
import { Entity, PrimaryColumn } from "typeorm"

@Entity()
export class User {
    @PrimaryColumn()
    firstName: string

    @PrimaryColumn()
    lastName: string
}
```

Когда вы сохраняете сущности через `save`, TypeORM всегда пытается найти в базе данных сущность с заданным идентификатором (или идентификаторами).
Если идентификатор найден, соответствующая строка в базе данных будет обновлена.
Если строки с таким идентификатором нет, будет вставлена новая строка.

Чтобы найти сущность по идентификатору, используйте `manager.findOneBy` или `repository.findOneBy`. Пример:

```typescript
// поиск по идентификатору с одним первичным ключом
const person = await dataSource.manager.findOneBy(Person, { id: 1 })
const person = await dataSource.getRepository(Person).findOneBy({ id: 1 })

// поиск по идентификатору с составным первичным ключом
const user = await dataSource.manager.findOneBy(User, {
    firstName: "Timber",
    lastName: "Saw",
})
const user = await dataSource.getRepository(User).findOneBy({
    firstName: "Timber",
    lastName: "Saw",
})
```

### Специальные колонки

Есть несколько специальных типов колонок с дополнительной функциональностью:

- `@CreateDateColumn` — специальная колонка, в которую автоматически записывается дата вставки сущности.
  Задавать эту колонку не нужно — она заполняется автоматически.

- `@UpdateDateColumn` — специальная колонка, в которую автоматически записывается время обновления сущности
  при каждом вызове `save` у entity manager или репозитория, а также при операциях `upsert`, если происходит обновление.
  Задавать эту колонку не нужно — она заполняется автоматически.

- `@DeleteDateColumn` — специальная колонка, в которую автоматически записывается время удаления сущности при каждом вызове мягкого удаления у entity manager или репозитория. Задавать эту колонку не нужно — она заполняется автоматически. Если `@DeleteDateColumn` задана, областью выборки по умолчанию становятся «не удалённые» записи.

- `@VersionColumn` — специальная колонка, в которую автоматически записывается версия сущности (инкрементируемое число)
  при каждом вызове `save` у entity manager или репозитория, а также при операциях `upsert`, если происходит обновление.
  Задавать эту колонку не нужно — она заполняется автоматически.

## Типы колонок

TypeORM поддерживает все наиболее распространённые типы колонок, доступные в базах данных.
Типы колонок специфичны для конкретной СУБД — это даёт больше гибкости в том, как будет выглядеть схема вашей базы данных.

Тип колонки можно указать первым параметром `@Column`
или в опциях колонки `@Column`, например:

```typescript
@Column("int")
```

или

```typescript
@Column({ type: "int" })
```

Если нужно задать дополнительные параметры типа, сделайте это через опции колонки.
Например:

```typescript
@Column("varchar", { length: 200 })
```

> **Примечание**
>
> О типе `bigint`: тип колонки `bigint`, применяемый в SQL-базах данных, не помещается в обычный тип `number`, поэтому свойство отображается в `string`.

### Тип колонки `enum`

Тип колонки `enum` поддерживается в `postgres` и `mysql`. Определить такую колонку можно по-разному:

С использованием enum из TypeScript:

```typescript
export enum UserRole {
    ADMIN = "admin",
    EDITOR = "editor",
    GHOST = "ghost",
}

@Entity()
export class User {
    @PrimaryGeneratedColumn()
    id: number

    @Column({
        type: "enum",
        enum: UserRole,
        default: UserRole.GHOST,
    })
    role: UserRole
}
```

> **Примечание**
>
> Поддерживаются строковые, числовые и смешанные enum.

С использованием массива значений enum:

```typescript
export type UserRoleType = "admin" | "editor" | "ghost",

@Entity()
export class User {

    @PrimaryGeneratedColumn()
    id: number;

    @Column({
        type: "enum",
        enum: ["admin", "editor", "ghost"],
        default: "ghost"
    })
    role: UserRoleType
}
```

### Тип колонки `simple-array`

Есть специальный тип колонки `simple-array`, который позволяет хранить массив примитивных значений в одной строковой колонке.
Все значения разделяются запятой. Например:

```typescript
@Entity()
export class User {
    @PrimaryGeneratedColumn()
    id: number

    @Column("simple-array")
    names: string[]
}
```

```typescript
const user = new User()
user.names = ["Alexander", "Alex", "Sasha", "Shurik"]
```

Будет сохранено в одной колонке базы данных как значение `Alexander,Alex,Sasha,Shurik`.
При загрузке данных из базы имена вернутся массивом — ровно в том виде, в котором вы их сохранили.

Учтите: в записываемых значениях **НЕ ДОЛЖНО** быть запятых.

### Тип колонки `simple-json`

Есть специальный тип колонки `simple-json`, который позволяет хранить любые значения, представимые через JSON.stringify.
Очень удобно, когда в вашей базе данных нет типа json, а объект нужно сохранять и загружать
без лишних хлопот.
Например:

```typescript
@Entity()
export class User {
    @PrimaryGeneratedColumn()
    id: number

    @Column("simple-json")
    profile: { name: string; nickname: string }
}
```

```typescript
const user = new User()
user.profile = { name: "John", nickname: "Malkovich" }
```

Будет сохранено в одной колонке базы данных как значение `{"name":"John","nickname":"Malkovich"}`.
При загрузке данных из базы вы получите обратно свой объект/массив/примитив через JSON.parse

### Колонки с генерируемыми значениями

Колонку с генерируемым значением можно создать с помощью декоратора `@Generated`. Например:

```typescript
@Entity()
export class User {
    @PrimaryColumn()
    id: number

    @Column()
    @Generated("uuid")
    uuid: string
}
```

Значение `uuid` будет сгенерировано автоматически и сохранено в базу данных.

Помимо "uuid" есть также типы генерации "increment", "identity" (только Postgres 10+) и "rowid" (только CockroachDB), однако на некоторых
платформах баз данных этот способ генерации имеет ограничения (например, в некоторых базах данных может быть только одна колонка с инкрементом,
а в некоторых требуется, чтобы инкрементируемая колонка была первичным ключом).

### Векторные колонки

Векторные колонки поддерживаются в MariaDB/MySQL, Microsoft SQL Server, PostgreSQL (через расширение [`pgvector`](https://github.com/pgvector/pgvector)) и SAP HANA Cloud. Они позволяют хранить и запрашивать векторные представления (embeddings) для поиска по сходству и задач машинного обучения.

TypeORM поддерживает типы колонок `vector` и `halfvec` в разных базах данных:

- `vector` — хранит векторы как 4-байтовые числа с плавающей точкой (одинарная точность)
    - MariaDB/MySQL: нативный тип `vector`
    - Microsoft SQL Server: нативный тип `vector`
    - PostgreSQL: тип `vector`, доступен через расширение `pgvector`
    - SAP HANA Cloud: псевдоним для типа `real_vector`
- `halfvec` — хранит векторы как 2-байтовые числа с плавающей точкой (половинная точность) для экономии памяти
    - PostgreSQL: тип `halfvec`, доступен через расширение `pgvector`
    - SAP HANA Cloud: псевдоним для типа `half_vector`

Количество измерений вектора задаётся опцией `length`:

```typescript
@Entity()
export class Post {
    @PrimaryGeneratedColumn()
    id: number

    // вектор без указания размерности
    @Column("vector")
    embedding: number[] | Buffer

    // вектор с 3 измерениями: vector(3)
    @Column("vector", { length: 3 })
    embedding_3d: number[] | Buffer

    // вектор половинной точности с 4 измерениями: halfvec(4) (работает только в PostgreSQL и SAP HANA)
    @Column("halfvec", { length: 4 })
    halfvec_embedding: number[] | Buffer
}
```

> **Примечание**
>
> - **MariaDB/MySQL**: векторы поддерживаются начиная с MariaDB 11.7 и MySQL 9
> - **Microsoft SQL Server**: поддержка типа vector требует SQL Server 2025 (17.x) или новее.
> - **PostgreSQL**: для векторных колонок требуется установленное расширение `pgvector`. Расширение предоставляет векторные типы данных и операторы сходства.
> - **SAP HANA**: для векторных колонок требуется SAP HANA Cloud (2024Q1+) и поддерживаемая версия `@sap/hana-client`.

### Пространственные колонки

Microsoft SQLServer, MySQL/MariaDB, PostgreSQL/CockroachDB и SAP HANA поддерживают пространственные колонки. Поддержка в TypeORM для каждой из этих СУБД немного различается — в частности, потому что различаются имена колонок.

MS SQL, MySQL/MariaDB и SAP HANA используют геометрию в формате [well-known text
(WKT)](https://en.wikipedia.org/wiki/Well-known_text), поэтому колонки с геометрией
следует помечать типом `string`.

```typescript
import { Entity, PrimaryColumn, Column } from "typeorm"

@Entity()
export class Thing {
    @PrimaryColumn()
    id: number

    @Column("point")
    point: string

    @Column("linestring")
    linestring: string
}

...

const thing = new Thing()
thing.point = "POINT(1 1)"
thing.linestring = "LINESTRING(0 0,1 1,2 2)"
```

Для Postgres/CockroachDB см. [типы данных Postgis](<../13. Drivers/06. Postgres - CockroachDB.md#пространственные-колонки>)

## Опции колонок

Опции колонок задают дополнительные параметры для колонок вашей сущности.
Указать их можно в `@Column`:

```typescript
@Column({
    type: "varchar",
    length: 150,
    unique: true,
    // ...
})
name: string;
```

Список доступных опций в `ColumnOptions`:

- `type: ColumnType` — тип колонки. Один из типов, перечисленных [выше](#типы-колонок).
- `name: string` — имя колонки в таблице базы данных.
  По умолчанию имя колонки генерируется из имени свойства.
  Его можно изменить, указав своё имя.

- `length: number` — длина типа колонки. Например, чтобы получить тип `varchar(150)`, укажите тип колонки и опцию длины.
- `onUpdate: string` — триггер `ON UPDATE`. Используется только в [MySQL](https://dev.mysql.com/doc/refman/5.7/en/timestamp-initialization.html).
- `nullable: boolean` — делает колонку `NULL` или `NOT NULL` в базе данных. По умолчанию — `nullable: false`.
- `update: boolean` — указывает, обновляется ли значение колонки операцией «save». Если `false`, записать это значение можно будет только при первой вставке объекта. Значение по умолчанию — `true`.
- `insert: boolean` — указывает, задаётся ли значение колонки при первой вставке объекта. Значение по умолчанию — `true`.
- `select: boolean` — определяет, скрывать ли эту колонку по умолчанию при выполнении запросов. Если задано `false`, данные колонки не попадут в результат обычного запроса. По умолчанию — `select: true`
- `default: string` — добавляет значение `DEFAULT` для колонки на уровне базы данных.
- `primary: boolean` — помечает колонку как первичную. То же самое, что использовать `@PrimaryColumn`.
- `unique: boolean` — помечает колонку как уникальную (создаёт ограничение уникальности).
- `comment: string` — комментарий к колонке в базе данных. Поддерживается не всеми типами баз данных.
- `precision: number` — точность для колонки decimal (точного числового типа; применяется только к таким колонкам) — максимальное
  количество цифр, сохраняемых для значений. Используется в некоторых типах колонок.
- `scale: number` — масштаб для колонки decimal (точного числового типа; применяется только к таким колонкам) — количество цифр справа от десятичной точки; не должен превышать precision. Используется в некоторых типах колонок.
- `unsigned: boolean` — добавляет атрибут `UNSIGNED` числовой колонке. Используется только в MySQL.
- `charset: string` — задаёт кодировку колонки. Поддерживается не всеми типами баз данных.
- `collation: string` — задаёт правило сортировки (collation) колонки.
- `enum: string[]|AnyEnum` — используется в колонках типа `enum` для перечисления допустимых значений. Можно указать массив значений или класс enum.
- `enumName: string` — задаёт имя используемого enum.
- `asExpression: string` — выражение генерируемой колонки. Поддерживается в PostgreSQL/CockroachDB, MySQL/MariaDB, Oracle, SAP HANA, Spanner, SQLite и SQL Server.
- `generatedType: "VIRTUAL"|"STORED"` — тип генерируемой колонки. Поддерживается в PostgreSQL (только `STORED`), CockroachDB, MySQL/MariaDB, Oracle (только `VIRTUAL`), Spanner (только `STORED`), SQLite и SQL Server (только `STORED`, внутренне отображается в `PERSISTED`).
- `hstoreType: "object"|"string"` — тип возвращаемого значения для колонки `HSTORE`. Возвращает значение строкой или объектом. Используется только в [Postgres](https://www.postgresql.org/docs/9.6/static/hstore.html).
- `array: boolean` — используется для типов колонок postgres и cockroachdb, которые могут быть массивами (например, int[])
- `transformer: { from(value: DatabaseType): EntityType, to(value: EntityType): DatabaseType }` — используется для преобразования свойств произвольного типа `EntityType` в поддерживаемый базой данных тип `DatabaseType`. Массивы трансформеров также поддерживаются: при записи они применяются в прямом порядке, при чтении — в обратном. Например, `[lowercase, encrypt]` при записи сначала приведёт строку к нижнему регистру, а затем зашифрует её; при чтении — расшифрует и больше ничего делать не будет.
- `utc: boolean` — указывает, следует ли сохранять и считывать значения дат в часовом поясе UTC вместо локального. Применяется только к типу колонки `date`. Значение по умолчанию — `false` (используется локальный часовой пояс для обратной совместимости).

Примечание: большинство этих опций специфичны для реляционных СУБД и недоступны в `MongoDB`.

## Наследование сущностей

Наследование сущностей позволяет сократить дублирование в коде.

Например, у вас есть сущности `Photo`, `Question`, `Post`:

```typescript
@Entity()
export class Photo {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    title: string

    @Column()
    description: string

    @Column()
    size: string
}

@Entity()
export class Question {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    title: string

    @Column()
    description: string

    @Column()
    answersCount: number
}

@Entity()
export class Post {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    title: string

    @Column()
    description: string

    @Column()
    viewCount: number
}
```

Как видите, у всех этих сущностей есть общие колонки: `id`, `title`, `description`. Чтобы сократить дублирование и получить более удачную абстракцию, создадим для них базовый класс `Content`:

```typescript
export abstract class Content {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    title: string

    @Column()
    description: string
}
@Entity()
export class Photo extends Content {
    @Column()
    size: string
}

@Entity()
export class Question extends Content {
    @Column()
    answersCount: number
}

@Entity()
export class Post extends Content {
    @Column()
    viewCount: number
}
```

Все колонки (а также связи, встраиваемые сущности и т. д.) родительских сущностей (родитель, в свою очередь, тоже может наследовать другую сущность)
будут унаследованы и созданы в итоговых сущностях.

## Древовидные сущности

TypeORM поддерживает паттерны Adjacency list (список смежности) и Closure table (таблица замыканий) для хранения древовидных структур.

### Adjacency list

Adjacency list — это простая модель со ссылкой на саму себя.
Преимущество этого подхода — простота,
недостаток — из-за ограничений на join нельзя загрузить большое дерево целиком.
Пример:

```typescript
import {
    Entity,
    Column,
    PrimaryGeneratedColumn,
    ManyToOne,
    OneToMany,
} from "typeorm"

@Entity()
export class Category {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    name: string

    @Column()
    description: string

    @ManyToOne((type) => Category, (category) => category.children)
    parent: Category

    @OneToMany((type) => Category, (category) => category.parent)
    children: Category[]
}
```

### Closure table

Closure table особым образом хранит связи между родителем и потомком в отдельной таблице.
Этот подход эффективен и при чтении, и при записи.
Подробнее о closure table — в [этой отличной презентации Билла Карвина](https://www.slideshare.net/billkarwin/models-for-hierarchical-data).
Пример:

```typescript
import {
    Entity,
    Tree,
    Column,
    PrimaryGeneratedColumn,
    TreeChildren,
    TreeParent,
    TreeLevelColumn,
} from "typeorm"

@Entity()
@Tree("closure-table")
export class Category {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    name: string

    @Column()
    description: string

    @TreeChildren()
    children: Category[]

    @TreeParent()
    parent: Category

    @TreeLevelColumn()
    level: number
}
```
