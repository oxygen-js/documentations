# Настройка

Прежде чем работать с миграциями, нужно правильно настроить опции [DataSource](<../02. Data Source/01. DataSource.md>):

```ts
export default new DataSource({
    // базовая настройка
    synchronize: false,
    migrations: [__dirname + "/migrations/**/*{.js,.ts}"],

    // необязательное
    migrationsRun: false,
    migrationsTableName: "migrations",
    migrationsTransactionMode: "all",

    // другие опции...
})
```

## `synchronise`

Отключение автоматической синхронизации схемы обязательно для работы с миграциями. Иначе в них не было бы смысла.

## `migrations`

Задаёт список миграций, которые должен загрузить TypeORM. Принимает как классы миграций, так и каталоги, из которых их загружать.

Проще всего указать каталог, где лежат файлы миграций (glob-шаблоны поддерживаются):

```ts
migrations: [__dirname + "/migrations/**/*{.js,.ts}"]
```

Указание сразу обоих расширений, `.js` и `.ts`, позволит запускать миграции как в разработке, так и из скомпилированного в JavaScript кода в продакшене (например, из Docker-образа).

Как вариант, можно указать конкретные классы, чтобы получить более тонкий контроль:

```ts
import FirstMigration from "./migrations/TIMESTAMP-first-migration"
import SecondMigration from "./migrations/TIMESTAMP-second-migration"

export default new DataSource({
    migrations: [FirstMigration, SecondMigration],
})
```

но это требует больше ручной работы и легко ошибиться.

- `migrationsRun` — указывает, нужно ли автоматически запускать [миграции](<01. How migrations work.md>) при каждом старте приложения.

## Необязательные настройки

### `migrationsRun`

Указывает, нужно ли автоматически запускать миграции при каждом старте приложения. По умолчанию: `false`

### `migrationsTableName`

Возможно, вы захотите задать имя таблицы, в которой будет храниться информация о выполненных миграциях. По умолчанию она называется `'migrations'`.

```ts
migrationsTableName: "some_custom_migrations_table"
```

### `migrationsTransactionMode`

Управляет режимом транзакций при выполнении миграций. Возможные значения:

- `all` (_по умолчанию_) — оборачивает весь запуск миграций в одну транзакцию
- `none`
- `each`
