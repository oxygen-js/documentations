# Генерация

TypeORM умеет автоматически генерировать файлы миграций на основе изменений, внесённых в сущности, сравнивая их с текущей схемой базы данных на сервере.

Автоматическая генерация создаёт новый файл миграции и записывает в него все SQL-запросы, которые нужно выполнить для обновления базы данных. Если изменений не обнаружено, команда завершится с кодом `1`.

Допустим, у вас есть сущность `Post` с колонкой `title`, и вы переименовали `title` в `name`.

Сгенерировать миграцию можно следующей командой:

```shell
typeorm migration:generate -d <path/to/datasource> <migration-name>
```

Значение аргумента `-d` должно указывать путь, по которому определён экземпляр [DataSource](<../02. Data Source/01. DataSource.md>).

Как вариант, имя можно задать параметром `--name`

```shell
typeorm migration:generate -- -d <path/to/datasource> --name=<migration-name>
```

или использовать полный путь:

```shell
typeorm migration:generate -d <path/to/datasource> <path/to/migrations>/<migration-name>
```

Если в качестве имени вы указали `post-refactoring`, будет создан новый файл `{TIMESTAMP}-post-refactoring.ts` со следующим содержимым:

```typescript
import { MigrationInterface, QueryRunner } from "typeorm"

export class PostRefactoringTIMESTAMP implements MigrationInterface {
    async up(queryRunner: QueryRunner): Promise<void> {
        await queryRunner.query(
            `ALTER TABLE "post" ALTER COLUMN "title" RENAME TO "name"`,
        )
    }

    async down(queryRunner: QueryRunner): Promise<void> {
        await queryRunner.query(
            `ALTER TABLE "post" ALTER COLUMN "name" RENAME TO "title"`,
        )
    }
}
```

Кроме того, миграции можно выводить в виде файлов JavaScript с помощью флага `o` (сокращение от `--outputJs`). Это удобно для проектов на чистом JavaScript, где дополнительные пакеты TypeScript не установлены. Эта команда создаст новый файл миграции `{TIMESTAMP}-PostRefactoring.js` со следующим содержимым:

```javascript
/**
 * @typedef {import('typeorm').MigrationInterface} MigrationInterface
 * @typedef {import('typeorm').QueryRunner} QueryRunner
 */

/**
 * @class
 * @implements {MigrationInterface}
 */
module.exports = class PostRefactoringTIMESTAMP {
    /**
     * @param {QueryRunner} queryRunner
     */
    async up(queryRunner) {
        await queryRunner.query(
            `ALTER TABLE "post" ALTER COLUMN "title" RENAME TO "name"`,
        )
    }

    /**
     * @param {QueryRunner} queryRunner
     */
    async down(queryRunner) {
        await queryRunner.query(
            `ALTER TABLE "post" ALTER COLUMN "name" RENAME TO "title"`,
        )
    }
}
```

По умолчанию с флагом `o` (сокращение от `--outputJs`) генерируется код JavaScript в формате CommonJS, но с флагом `esm` можно сгенерировать и код в формате ESM. Это удобно для проектов на JavaScript, использующих ESM:

```javascript
/**
 * @typedef {import('typeorm').MigrationInterface} MigrationInterface
 * @typedef {import('typeorm').QueryRunner} QueryRunner
 */

/**
 * @class
 * @implements {MigrationInterface}
 */
export class PostRefactoringTIMESTAMP {
    /**
     * @param {QueryRunner} queryRunner
     */
    async up(queryRunner) {
        await queryRunner.query(
            `ALTER TABLE "post" ALTER COLUMN "title" RENAME TO "name"`,
        )
    }

    /**
     * @param {QueryRunner} queryRunner
     */
    async down(queryRunner) {
        await queryRunner.query(
            `ALTER TABLE "post" ALTER COLUMN "name" RENAME TO "title"`,
        )
    }
}
```

Как видите, писать запросы самостоятельно не нужно.

Общее правило: генерируйте миграции после **каждого** изменения моделей. Чтобы сгенерированные запросы миграции форматировались в несколько строк, используйте флаг `p` (сокращение от `--pretty`).
