# FAQ

## Как обновить схему базы данных?

Одна из главных задач TypeORM — поддерживать таблицы базы данных в соответствии с вашими сущностями.
Добиться этого можно двумя способами:

- Указать `synchronize: true` в опциях источника данных:

    ```typescript
    import { DataSource } from "typeorm"

    const myDataSource = new DataSource({
        // ...
        synchronize: true,
    })
    ```

    Эта опция автоматически синхронизирует таблицы базы данных с указанными сущностями при каждом запуске кода.
    В разработке она незаменима, но в продакшене её лучше не включать.

- Воспользоваться инструментами командной строки и запустить синхронизацию схемы вручную:

    ```shell
    typeorm schema:sync
    ```

    Эта команда выполнит синхронизацию схемы.

Синхронизация схемы работает очень быстро.
Если вы подумываете отключить опцию synchronize во время разработки из-за проблем с производительностью,
сначала проверьте, насколько она быстра.

## Как изменить имя колонки в базе данных?

По умолчанию имена колонок формируются из имён свойств.
Изменить это можно, просто задав опцию колонки `name`:

```typescript
@Column({ name: "is_active" })
isActive: boolean;
```

## Как задать значением по умолчанию какую-нибудь функцию, например `NOW()`?

Опция колонки `default` поддерживает функции.
Если вы передадите функцию, возвращающую строку,
эта строка будет использована как значение по умолчанию без экранирования.
Например:

```typescript
@Column({ default: () => "NOW()" })
date: Date;
```

## Как выполнять валидацию?

Валидация не входит в TypeORM, поскольку это отдельный процесс,
не имеющий прямого отношения к тому, чем занимается TypeORM.
Если вам нужна валидация, воспользуйтесь [class-validator](https://github.com/pleerock/class-validator) — он отлично работает вместе с TypeORM.

## Что означает «владеющая сторона» в связях и зачем нужны `@JoinColumn` и `@JoinTable`?

Начнём со связи `one-to-one`.
Допустим, у нас есть две сущности: `User` и `Photo`:

```typescript
@Entity()
export class User {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    name: string

    @OneToOne()
    photo: Photo
}
```

```typescript
@Entity()
export class Photo {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    url: string

    @OneToOne()
    user: User
}
```

В этом примере нет `@JoinColumn`, и это неверно.
Почему? Потому что для настоящей связи нужно создать колонку в базе данных.
Нужно создать колонку `userId` в `photo` либо `photoId` в `user`.
Но какую именно колонку создавать — `userId` или `photoId`?
TypeORM не может решить это за вас.
Чтобы принять решение, нужно поставить `@JoinColumn` на одной из сторон.
Если вы поставите `@JoinColumn` в `Photo`, в таблице `photo` будет создана колонка `userId`.
Если вы поставите `@JoinColumn` в `User`, в таблице `user` будет создана колонка `photoId`.
Сторона с `@JoinColumn` называется «владеющей стороной связи».
Другая сторона связи, без `@JoinColumn`, называется «обратной (не владеющей) стороной связи».

То же самое и в связи `@ManyToMany`. Владеющую сторону связи вы обозначаете с помощью `@JoinTable`.

В связях `@ManyToOne` и `@OneToMany` декоратор `@JoinColumn` не нужен, потому что
эти два декоратора различаются, и колонка связи появится в той таблице, где стоит декоратор `@ManyToOne`.

Декораторы `@JoinColumn` и `@JoinTable` можно также использовать для задания дополнительных
настроек join-колонки или связующей таблицы — например, имени join-колонки или имени связующей таблицы.

## Как добавить дополнительные колонки в таблицу связи «многие ко многим» (связующую таблицу)?

Добавить дополнительные колонки в таблицу, созданную связью «многие ко многим», невозможно.
Вам придётся создать отдельную сущность и связать её с целевыми сущностями двумя связями «многие к одному»
(результат будет такой же, как при создании таблицы «многие ко многим»),
а дополнительные колонки добавить уже в неё. Подробнее об этом читайте в разделе [Связи «многие ко многим»](<../04. Relations/04. Many-to-many relations.md#связи-многие-ко-многим-с-дополнительными-свойствами>).

## Как быть с опцией компилятора TypeScript outDir?

Если вы используете опцию компилятора `outDir`, не забудьте скопировать в выходной каталог ассеты и ресурсы, которые использует приложение.
Либо позаботьтесь о том, чтобы пути к этим ассетам были указаны верно.

Важно знать: когда вы удаляете или перемещаете сущности, старые сущности остаются нетронутыми в выходном каталоге.
Например, вы создали сущность `Post` и переименовали её в `Blog` —
файла `Post.ts` в проекте больше нет. Однако `Post.js` остаётся в выходном каталоге.
И когда TypeORM читает сущности из выходного каталога, он видит две сущности — `Post` и `Blog`.
Это может стать источником ошибок.
Поэтому при удалении и перемещении сущностей с включённым `outDir` настоятельно рекомендуется удалить выходной каталог и заново скомпилировать проект.

## Как использовать TypeORM с ts-node?

Компиляции файлов при каждом запуске можно избежать с помощью [ts-node](https://github.com/TypeStrong/ts-node).
Если вы используете ts-node, укажите в опциях источника данных сущности с расширением `ts`:

```typescript
{
    entities: [__dirname + "/entities/**/*{.js,.ts}"],
    subscribers: [__dirname + "/subscribers/**/*{.js,.ts}"]
}
```

Кроме того, если вы компилируете js-файлы в тот же каталог, где лежат файлы TypeScript,
обязательно используйте опцию компилятора `outDir`, чтобы избежать
[этой проблемы](https://github.com/TypeStrong/ts-node/issues/432).

А если вы хотите пользоваться CLI через ts-node, запускайте TypeORM так:

```shell
npx typeorm-ts-node-commonjs schema:sync
```

Для ESM-проектов используйте вместо этого:

```shell
npx typeorm-ts-node-esm schema:sync
```

## Как использовать Webpack для бэкенда?

Webpack выдаёт предупреждения о том, что считает отсутствующими выражениями require, — это require для всех драйверов, поддерживаемых TypeORM. Чтобы подавить эти предупреждения для неиспользуемых драйверов, отредактируйте конфигурационный файл webpack.

```javascript
const FilterWarningsPlugin = require('webpack-filter-warnings-plugin');

module.exports = {
    ...
    plugins: [
        // игнорируем ненужные драйверы. Это полный список всех драйверов — уберите из него те, которые собираетесь использовать.
        new FilterWarningsPlugin({
            exclude: [/mongodb/, /mssql/, /mysql2/, /oracledb/, /pg/, /pg-native/, /pg-query-stream/, /react-native-sqlite-storage/, /redis/, /sql.js/, /typeorm-aurora-data-api-driver/]
        })
    ]
};
```

### Сборка файлов миграций

По умолчанию Webpack пытается собрать всё в один файл. Это может создать проблемы, если в проекте есть файлы миграций, которые должны выполняться после развёртывания собранного кода в продакшене. Чтобы TypeORM смог распознать и выполнить все ваши [миграции](<../06. Migrations/01. How migrations work.md>), для файлов миграций может понадобиться «объектный синтаксис» настройки `entry`.

```javascript
const { globSync } = require("node:fs")
const path = require("node:path")

module.exports = {
    // ... здесь ваши настройки webpack...
    // Динамически строим отображение `{ [name]: sourceFileName }` для опции `entry`
    // замените `src/db/migrations` на относительный путь к вашему каталогу миграций
    entry: globSync(path.resolve("src/db/migrations/*.ts")).reduce(
        (entries, filename) => {
            const migrationName = path.basename(filename, ".ts")
            return Object.assign({}, entries, {
                [migrationName]: filename,
            })
        },
        {},
    ),
    resolve: {
        // предполагается, что все файлы миграций написаны на TypeScript
        extensions: [".ts"],
    },
    output: {
        // замените `path` на каталог, куда вы хотите класть скомпилированные файлы миграций.
        path: __dirname + "/dist/db/migrations",
        // это важно — для файлов миграций нам нужен UMD (Universal Module Definition).
        libraryTarget: "umd",
        filename: "[name].js",
    },
}
```

Кроме того, начиная с Webpack 4 при использовании `mode: 'production'` файлы по умолчанию оптимизируются, и в том числе имена в коде искажаются ради уменьшения размера файлов. Это ломает [миграции](<../06. Migrations/01. How migrations work.md>), поскольку TypeORM определяет по их именам, какие из них уже были выполнены. Минификацию можно полностью отключить, добавив:

```javascript
module.exports = {
    // ... здесь другие настройки Webpack
    optimization: {
        minimize: false,
    },
}
```

Либо, если вы используете `UglifyJsPlugin`, можно запретить ему менять имена классов и функций:

```javascript
const UglifyJsPlugin = require("uglifyjs-webpack-plugin")

module.exports = {
    // ... здесь другие настройки Webpack
    optimization: {
        minimizer: [
            new UglifyJsPlugin({
                uglifyOptions: {
                    keep_classnames: true,
                    keep_fnames: true,
                },
            }),
        ],
    },
}
```

Наконец, убедитесь, что скомпилированные файлы миграций указаны в опциях источника данных:

```javascript
// Настройки TypeORM
module.exports = {
    // ...
    migrations: [__dirname + "/migrations/**/*{.js,.ts}"],
}
```

## Как использовать TypeORM в ESM-проектах?

Обязательно добавьте `"type": "module"` в `package.json` вашего проекта, чтобы TypeORM знал, что файлы нужно подключать через `import( ... )`.

Чтобы избежать проблем с циклическими зависимостями при импорте, используйте тип-обёртку `Relation` при объявлении типов связей в сущностях:

```typescript
@Entity()
export class User {
    @OneToOne(() => Profile, (profile) => profile.user)
    profile: Relation<Profile>
}
```

Так тип свойства не сохраняется в метаданных свойства в скомпилированном коде, что снимает проблему циклических зависимостей.

Поскольку тип колонки уже задан декоратором `@OneToOne`, дополнительные метаданные о типе, сохраняемые TypeScript, всё равно не используются.

> **Важно** не используйте `Relation` для колонок, не являющихся связями
