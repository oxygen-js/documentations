# Справочник по декораторам

## Декораторы сущностей

#### `@Entity`

Помечает вашу модель как сущность. Сущность — это класс, который превращается в таблицу базы данных.
Имя таблицы можно указать в сущности:

```typescript
@Entity("users")
export class User {}
```

Этот код создаст таблицу с именем "users".

Также можно указать некоторые дополнительные опции сущности:

- `name` — имя таблицы. Если не указано, имя таблицы формируется из имени класса сущности.
- `database` — имя базы данных на выбранном сервере СУБД.
- `schema` — имя схемы.
- `comment` — комментарий к таблице базы данных. Поддерживается не всеми типами баз данных. На данный момент поддерживается в MySQL, MariaDB, PostgreSQL, SAP HANA.
- `engine` — движок базы данных, задаваемый при создании таблицы (работает только в некоторых базах данных).
- `synchronize` — сущности, помеченные значением `false`, пропускаются при обновлении схемы.
- `orderBy` — задаёт сортировку сущностей по умолчанию при использовании операций `find` и `QueryBuilder`.

Пример:

```typescript
@Entity({
    name: "users",
    engine: "MyISAM",
    database: "example_dev",
    schema: "schema_with_best_tables",
    comment: "This is users table",
    synchronize: false,
    orderBy: {
        name: "ASC",
        id: "DESC",
    },
})
export class User {}
```

Подробнее о [сущностях](<../03. Entity/01. Entities.md>).

#### `@ViewEntity`

Сущность-представление — это класс, отображаемый на представление (view) базы данных.

`@ViewEntity()` принимает следующие опции:

- `name` — имя представления. Если не указано, имя представления формируется из имени класса сущности.
- `database` — имя базы данных на выбранном сервере СУБД.
- `schema` — имя схемы.
- `expression` — определение представления. **Обязательный параметр**.

`expression` может быть строкой с правильно экранированными колонками и таблицами — в зависимости от используемой базы данных (в примере — postgres):

```typescript
@ViewEntity({
    expression: `
        SELECT "post"."id" "id", "post"."name" AS "name", "category"."name" AS "categoryName"
        FROM "post" "post"
        LEFT JOIN "category" "category" ON "post"."categoryId" = "category"."id"
    `,
})
export class PostCategory {}
```

либо экземпляром QueryBuilder

```typescript
@ViewEntity({
    expression: (dataSource: DataSource) =>
        dataSource
            .createQueryBuilder()
            .select("post.id", "id")
            .addSelect("post.name", "name")
            .addSelect("category.name", "categoryName")
            .from(Post, "post")
            .leftJoin(Category, "category", "category.id = post.categoryId"),
})
export class PostCategory {}
```

**Примечание:** привязка параметров не поддерживается из-за ограничений драйверов. Используйте вместо неё литеральные параметры.

```typescript
@ViewEntity({
    expression: (dataSource: DataSource) =>
        dataSource
            .createQueryBuilder()
            .select("post.id", "id")
            .addSelect("post.name", "name")
            .addSelect("category.name", "categoryName")
            .from(Post, "post")
            .leftJoin(Category, "category", "category.id = post.categoryId")
            .where("category.name = :name", { name: "Cars" }) // <-- так неправильно
            .where("category.name = 'Cars'"), // <-- а так правильно
})
export class PostCategory {}
```

## Декораторы колонок

#### `@Column`

Помечает свойство сущности как колонку таблицы.
Пример:

```typescript
@Entity("users")
export class User {
    @Column({ primary: true })
    id: number

    @Column({ type: "varchar", length: 200, unique: true })
    firstName: string

    @Column({ nullable: true })
    lastName: string

    @Column({ default: false })
    isActive: boolean
}
```

`@Column` принимает несколько опций, которые вы можете использовать:

- `type: ColumnType` — тип колонки. Один из [поддерживаемых типов колонок](<../03. Entity/01. Entities.md#типы-колонок>).
- `name: string` — имя колонки в таблице базы данных.
  По умолчанию имя колонки формируется из имени свойства.
  Вы можете изменить его, задав собственное имя.
- `length: string|number` — длина типа колонки. Например, чтобы создать тип `varchar(150)`,
  вы указываете тип колонки и опцию length.
- `width: number` — отображаемая ширина типа колонки. Используется только для [целочисленных типов MySQL](https://dev.mysql.com/doc/refman/5.7/en/integer-types.html). _Устарело_ в новых версиях MySQL, будет удалено из TypeORM в одной из следующих версий.
- `onUpdate: string` — триггер `ON UPDATE`. Используется только в [MySQL](https://dev.mysql.com/doc/refman/5.7/en/timestamp-initialization.html).
- `nullable: boolean` — определяет, может ли колонка принимать значение `NULL` или всегда должна быть `NOT NULL`. По умолчанию колонка имеет `nullable: false`.
- `update: boolean` — указывает, обновляется ли значение колонки операцией «save». Если задано false, записать это значение вы сможете только при первой вставке объекта.
  Значение по умолчанию — `true`.
- `insert: boolean` — указывает, задаётся ли значение колонки при первой вставке объекта. Значение по умолчанию — `true`.
- `select: boolean` — определяет, скрывать ли эту колонку по умолчанию при выполнении запросов. Если задано `false`, данные колонки не будут попадать в результат обычного запроса. По умолчанию колонка имеет `select: true`
- `default: string` — добавляет значение `DEFAULT` для колонки на уровне базы данных.
- `primary: boolean` — помечает колонку как первичную. То же самое, что использование `@PrimaryColumn`.
- `unique: boolean` — помечает колонку как уникальную (создаёт ограничение уникальности). Значение по умолчанию — false.
- `comment: string` — комментарий к колонке базы данных. Поддерживается не всеми типами баз данных.
- `precision: number` — точность для колонки decimal (точного числового типа; применимо только к decimal-колонкам) — максимальное
  количество цифр, хранимых в значениях. Используется в некоторых типах колонок.
- `scale: number` — масштаб для колонки decimal (точного числового типа; применимо только к decimal-колонкам),
  то есть количество цифр справа от десятичной точки; не должен превышать precision.
  Используется в некоторых типах колонок.
- `zerofill: boolean` — добавляет атрибут `ZEROFILL` числовой колонке. Используется только в MySQL. Если задано `true`, MySQL автоматически добавляет к этой колонке атрибут `UNSIGNED`. _Устарело_ в новых версиях MySQL, будет удалено из TypeORM в одной из следующих версий. Используйте символьную колонку и функцию `LPAD`, как рекомендует MySQL.
- `unsigned: boolean` — добавляет атрибут `UNSIGNED` числовой колонке. Используется только в MySQL.
- `charset: string` — задаёт кодировку колонки. Поддерживается не всеми типами баз данных.
- `collation: string` — задаёт правило сортировки колонки.
- `enum: string[]|AnyEnum` — используется в типе колонки `enum` для указания списка допустимых значений перечисления.
  Вы можете указать массив значений или класс перечисления.
- `enumName: string` — имя генерируемого типа enum. Если не указано, TypeORM сформирует имя типа enum из имён сущности и колонки, поэтому эта опция необходима, если вы намерены использовать один и тот же тип enum в разных таблицах.
- `primaryKeyConstraintName: string` — имя ограничения первичного ключа. Если не указано, имя ограничения формируется из имени таблицы и имён участвующих колонок.
- `asExpression: string` — выражение генерируемой колонки. Поддерживается в PostgreSQL/CockroachDB, MySQL/MariaDB, Oracle, SAP HANA, Spanner, SQLite и SQL Server.
- `generatedType: "VIRTUAL"|"STORED"` — тип генерируемой колонки. Поддерживается в PostgreSQL (только `STORED`), CockroachDB, MySQL/MariaDB, Oracle (только `VIRTUAL`), Spanner (только `STORED`), SQLite и SQL Server (только `STORED`, внутренне отображается на `PERSISTED`).
- `hstoreType: "object"|"string"` — тип возвращаемого значения колонки `HSTORE`. Возвращает значение как строку или как объект. Используется только в [Postgres](https://www.postgresql.org/docs/9.6/static/hstore.html).
- `array: boolean` — используется для тех типов колонок postgres и cockroachdb, которые могут быть массивами (например, int[]).
- `transformer: ValueTransformer|ValueTransformer[]` — задаёт преобразователь значений (или массив преобразователей), применяемый к этой колонке при чтении из базы данных и записи в неё. В случае массива преобразователи применяются в прямом порядке при переходе от entityValue к databaseValue и в обратном порядке при переходе от databaseValue к entityValue.
- `spatialFeatureType: string` — необязательный тип объекта (`Point`, `Polygon`, `LineString`, `Geometry`), используемый как ограничение пространственной колонки. Если не указан, поведение будет таким же, как при указании `Geometry`. Используется только в PostgreSQL и CockroachDB.
- `srid: number` — необязательный [идентификатор пространственной привязки](https://postgis.net/docs/using_postgis_dbmanagement.html#spatial_ref_sys), используемый как ограничение пространственной колонки. Если не указан, по умолчанию принимается `0`. Стандартным географическим координатам (широта/долгота в датуме WGS84) соответствует [EPSG 4326](http://spatialreference.org/ref/epsg/wgs-84/). Используется только в PostgreSQL и CockroachDB.

Подробнее о [колонках сущности](<../03. Entity/01. Entities.md#колонки-сущности>).

#### `@PrimaryColumn`

Помечает свойство сущности как первичную колонку таблицы.
То же, что декоратор `@Column`, но с опцией `primary`, установленной в true.

Пример:

```typescript
@Entity()
export class User {
    @PrimaryColumn()
    id: number
}
```

`@PrimaryColumn()` поддерживает пользовательское имя ограничения первичного ключа:

```typescript
@Entity()
export class User {
    @PrimaryColumn({ primaryKeyConstraintName: "pk_user_id" })
    id: number
}
```

> **Примечание** при использовании `primaryKeyConstraintName` с несколькими первичными ключами имя ограничения должно быть одинаковым для всех первичных колонок.

Подробнее о [колонках сущности](<../03. Entity/01. Entities.md#колонки-сущности>).

#### `@PrimaryGeneratedColumn`

Помечает свойство сущности как первичную колонку таблицы с генерируемым значением.
Создаваемая колонка является первичной, а её значение генерируется автоматически.
Пример:

```typescript
@Entity()
export class User {
    @PrimaryGeneratedColumn()
    id: number
}
```

`@PrimaryGeneratedColumn()` поддерживает пользовательское имя ограничения первичного ключа:

```typescript
@Entity()
export class User {
    @PrimaryGeneratedColumn({ primaryKeyConstraintName: "pk_user_id" })
    id: number
}
```

Существует четыре стратегии генерации:

- `increment` — использует AUTO_INCREMENT / SERIAL / SEQUENCE (в зависимости от типа базы данных) для генерации возрастающего числа.
- `identity` — только для [PostgreSQL 10+](https://www.postgresql.org/docs/13/sql-createtable.html). Версии Postgres старше 10 поддерживают соответствующую стандарту SQL колонку **IDENTITY**. Если в качестве стратегии генерации указан `identity`, колонка будет создана с использованием `GENERATED [ALWAYS|BY DEFAULT] AS IDENTITY`
- `uuid` — генерирует уникальную строку `uuid`.
- `rowid` — только для [CockroachDB](https://www.cockroachlabs.com/docs/stable/serial.html). Значение генерируется автоматически с помощью функции `unique_rowid()`.
  Она формирует 64-битное целое число из текущей метки времени и идентификатора узла, выполняющего операцию `INSERT` или `UPSERT`.
    > **Примечание**: свойство со стратегией генерации `rowid` должно иметь тип данных `string`

Стратегия генерации по умолчанию — `increment`; чтобы сменить её на другую, просто передайте её первым аргументом декоратора:

```typescript
@Entity()
export class User {
    @PrimaryGeneratedColumn("uuid")
    id: string
}
```

Подробнее о [колонках сущности](<../03. Entity/01. Entities.md#колонки-сущности>).

#### `@ObjectIdColumn`

Помечает свойство сущности как `ObjectId`.
Этот декоратор используется только в MongoDB.
Каждая сущность в MongoDB должна иметь колонку `ObjectId`.
Пример:

```typescript
@Entity()
export class User {
    @ObjectIdColumn()
    id: ObjectId
}
```

Подробнее о [MongoDB](<../13. Drivers/03. MongoDB.md>).

#### `@CreateDateColumn`

Особая колонка, которой автоматически присваивается время вставки сущности.
Записывать значение в эту колонку не нужно — оно будет установлено автоматически.
Пример:

```typescript
@Entity()
export class User {
    @CreateDateColumn()
    createdDate: Date
}
```

#### `@UpdateDateColumn`

Особая колонка, которой автоматически присваивается время обновления сущности
при каждом вызове `save` из entity manager или репозитория.
Записывать значение в эту колонку не нужно — оно будет установлено автоматически.

Эта колонка также автоматически обновляется в ходе операций `upsert`, когда из-за конфликта происходит обновление.

```typescript
@Entity()
export class User {
    @UpdateDateColumn()
    updatedDate: Date
}
```

#### `@DeleteDateColumn`

Особая колонка, которой автоматически присваивается время удаления сущности при каждом вызове мягкого удаления из entity manager или репозитория. Задавать эту колонку не нужно — она будет заполнена автоматически.

Собственный механизм мягкого удаления TypeORM использует глобальные области видимости, чтобы извлекать из базы данных только «неудалённые» сущности.

Если задан @DeleteDateColumn, областью видимости по умолчанию станет «неудалённые».

```typescript
@Entity()
export class User {
    @DeleteDateColumn()
    deletedDate: Date
}
```

#### `@VersionColumn`

Особая колонка, которой автоматически присваивается версия сущности (возрастающее число)
при каждом вызове `save` из entity manager или репозитория.
Записывать значение в эту колонку не нужно — оно будет установлено автоматически.

Эта колонка также автоматически обновляется в ходе операций `upsert`, когда из-за конфликта происходит обновление.

```typescript
@Entity()
export class User {
    @VersionColumn()
    version: number
}
```

#### `@Generated`

Помечает колонку как содержащую генерируемое значение. Например:

```typescript
@Entity()
export class User {
    @Column()
    @Generated("uuid")
    uuid: string
}
```

Значение будет сгенерировано только один раз, перед вставкой сущности в базу данных.

#### `@VirtualColumn`

Особая колонка, которая никогда не сохраняется в базу данных и потому ведёт себя как свойство только для чтения.
При каждом вызове `find` или `findOne` из entity manager значение пересчитывается на основе функции запроса, переданной в декоратор VirtualColumn. Аргумент alias, передаваемый в запрос, ссылается на точный псевдоним сущности в генерируемом «под капотом» запросе.

```typescript
@Entity({ name: "companies", alias: "COMP" })
export class Company extends BaseEntity {
    @PrimaryColumn("varchar", { length: 50 })
    name: string

    @VirtualColumn({
        query: (alias) =>
            `SELECT COUNT("name") FROM "employees" WHERE "companyName" = ${alias}.name`,
    })
    totalEmployeesCount: number

    @OneToMany((type) => Employee, (employee) => employee.company)
    employees: Employee[]
}

@Entity({ name: "employees" })
export class Employee extends BaseEntity {
    @PrimaryColumn("varchar", { length: 50 })
    name: string

    @ManyToOne((type) => Company, (company) => company.employees)
    company: Company
}
```

## Декораторы связей

#### `@OneToOne`

Связь «один к одному» — это связь, при которой A содержит только один экземпляр B, а B содержит только один экземпляр A.
Возьмём для примера сущности `User` и `Profile`.
У пользователя может быть только один профиль, и один профиль принадлежит только одному пользователю.
Пример:

```typescript
import { Entity, OneToOne, JoinColumn } from "typeorm"
import { Profile } from "./Profile"

@Entity()
export class User {
    @OneToOne((type) => Profile, (profile) => profile.user)
    @JoinColumn()
    profile: Profile
}
```

Подробнее о [связях «один к одному»](<../04. Relations/02. One-to-one relations.md>).

#### `@ManyToOne`

Связь «многие к одному» / «один ко многим» — это связь, при которой A содержит несколько экземпляров B, а B содержит только один экземпляр A.
Возьмём для примера сущности `User` и `Photo`.
У пользователя может быть несколько фотографий, но каждая фотография принадлежит только одному пользователю.
Пример:

```typescript
import { Entity, PrimaryGeneratedColumn, Column, ManyToOne } from "typeorm"
import { User } from "./User"

@Entity()
export class Photo {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    url: string

    @ManyToOne((type) => User, (user) => user.photos)
    user: User
}
```

Подробнее о [связях «многие к одному» / «один ко многим»](<../04. Relations/03. Many-to-one - one-to-many relations.md>).

#### `@OneToMany`

Связь «многие к одному» / «один ко многим» — это связь, при которой A содержит несколько экземпляров B, а B содержит только один экземпляр A.
Возьмём для примера сущности `User` и `Photo`.
У пользователя может быть несколько фотографий, но каждая фотография принадлежит только одному пользователю.
Пример:

```typescript
import { Entity, PrimaryGeneratedColumn, Column, OneToMany } from "typeorm"
import { Photo } from "./Photo"

@Entity()
export class User {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    name: string

    @OneToMany((type) => Photo, (photo) => photo.user)
    photos: Photo[]
}
```

Подробнее о [связях «многие к одному» / «один ко многим»](<../04. Relations/03. Many-to-one - one-to-many relations.md>).

#### `@ManyToMany`

Связь «многие ко многим» — это связь, при которой A содержит несколько экземпляров B, а B содержит несколько экземпляров A.
Возьмём для примера сущности `Question` и `Category`.
У вопроса может быть несколько категорий, и в каждой категории может быть несколько вопросов.
Пример:

```typescript
import {
    Entity,
    PrimaryGeneratedColumn,
    Column,
    ManyToMany,
    JoinTable,
} from "typeorm"
import { Category } from "./Category"

@Entity()
export class Question {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    title: string

    @Column()
    text: string

    @ManyToMany((type) => Category)
    @JoinTable()
    categories: Category[]
}
```

Подробнее о [связях «многие ко многим»](<../04. Relations/04. Many-to-many relations.md>).

#### `@JoinColumn`

Определяет, какая сторона связи содержит join-колонку с внешним ключом, и
позволяет настроить имя join-колонки, имя колонки, на которую она ссылается, и имя внешнего ключа.
Пример:

```typescript
@Entity()
export class Post {
    @ManyToOne((type) => Category)
    @JoinColumn({
        name: "cat_id",
        referencedColumnName: "name",
        foreignKeyConstraintName: "fk_cat_id",
    })
    category: Category
}
```

#### `@JoinTable`

Используется для связей `many-to-many` и описывает join-колонки «связующей» (junction) таблицы.
Связующая таблица — это особая отдельная таблица, автоматически создаваемая TypeORM, с колонками, ссылающимися на связанные сущности.
Вы можете изменить имя генерируемой «связующей» таблицы, имена колонок внутри неё, колонки, на которые они
ссылаются, — с помощью атрибутов `joinColumn` и `inverseJoinColumn`, — а также имена создаваемых внешних ключей.
Кроме того, вы можете задать параметр `synchronize` в false, чтобы пропустить обновление схемы (так же, как в @Entity)

Пример:

```typescript
@Entity()
export class Post {
    @ManyToMany((type) => Category)
    @JoinTable({
        name: "question_categories",
        joinColumn: {
            name: "question",
            referencedColumnName: "id",
            foreignKeyConstraintName: "fk_question_categories_questionId",
        },
        inverseJoinColumn: {
            name: "category",
            referencedColumnName: "id",
            foreignKeyConstraintName: "fk_question_categories_categoryId",
        },
        synchronize: false,
    })
    categories: Category[]
}
```

Если у целевой таблицы составной первичный ключ,
в декоратор `@JoinTable` необходимо передать массив свойств.

#### `@RelationId`

Загружает id (или несколько id) указанных связей в свойства.
Например, если у вас в сущности `Post` есть связь `category` типа «многие к одному»,
вы можете получить id категории, пометив новое свойство декоратором `@RelationId`.
Пример:

```typescript
@Entity()
export class Post {
    @ManyToOne((type) => Category)
    category: Category

    @RelationId((post: Post) => post.category) // нужно указать целевую связь
    categoryId: number
}
```

Эта возможность работает со всеми видами связей, включая `many-to-many`:

```typescript
@Entity()
export class Post {
    @ManyToMany((type) => Category)
    categories: Category[]

    @RelationId((post: Post) => post.categories)
    categoryIds: number[]
}
```

Id связи используется только для представления данных.
Сама связь не добавляется, не удаляется и не изменяется при изменении этого значения.

## Декораторы подписчиков и слушателей

#### `@AfterLoad`

Вы можете определить в сущности метод с любым именем и пометить его декоратором `@AfterLoad` —
тогда TypeORM будет вызывать его каждый раз, когда сущность
загружается через `QueryBuilder` или методы find репозитория/менеджера.
Пример:

```typescript
@Entity()
export class Post {
    @AfterLoad()
    updateCounters() {
        if (this.likesCount === undefined) this.likesCount = 0
    }
}
```

#### `@EventSubscriber`

Помечает класс как подписчика на события, который может слушать события конкретной сущности или события любых сущностей.
События вызываются через `QueryBuilder` и методы репозитория/менеджера.
Пример:

```typescript
@EventSubscriber()
export class PostSubscriber implements EntitySubscriberInterface<Post> {
    /**
     * Указывает, что этот подписчик слушает только события Post.
     */
    listenTo() {
        return Post
    }

    /**
     * Вызывается перед вставкой post.
     */
    beforeInsert(event: InsertEvent<Post>) {
        console.log(`BEFORE POST INSERTED: `, event.entity)
    }
}
```

Вы можете реализовать любой метод из `EntitySubscriberInterface`.
Чтобы слушать любую сущность, просто опустите метод `listenTo` и используйте `any`:

```typescript
@EventSubscriber()
export class PostSubscriber implements EntitySubscriberInterface {
    /**
     * Вызывается перед вставкой сущности.
     */
    beforeInsert(event: InsertEvent<any>) {
        console.log(`BEFORE ENTITY INSERTED: `, event.entity)
    }
}
```

## Прочие декораторы

#### `@Index`

Этот декоратор позволяет создать индекс базы данных для одной или нескольких колонок.
Он также позволяет пометить колонку или колонки как уникальные.
Декоратор можно применять как к колонкам, так и к самой сущности.
Применяйте его к колонке, когда нужен индекс по одной колонке,
и к сущности, когда требуется один индекс по нескольким колонкам.
Примеры:

```typescript
@Entity()
export class User {
    @Index()
    @Column()
    firstName: string

    @Index({ unique: true })
    @Column()
    lastName: string
}
```

```typescript
@Entity()
@Index(["firstName", "lastName"])
@Index(["lastName", "middleName"])
@Index(["firstName", "lastName", "middleName"], { unique: true })
export class User {
    @Column()
    firstName: string

    @Column()
    lastName: string

    @Column()
    middleName: string
}
```

Подробнее об [индексах](<../05. Indexes.md>).

#### `@Unique`

Этот декоратор позволяет создать в базе данных ограничение уникальности для одной или нескольких колонок.
Этот декоратор можно применять только к самой сущности.
В качестве аргументов необходимо указывать имена полей сущности (а не имена колонок базы данных).

Примеры:

```typescript
@Entity()
@Unique(["firstName"])
@Unique(["lastName", "middleName"])
@Unique("UQ_NAMES", ["firstName", "lastName", "middleName"])
export class User {
    @Column({ name: "first_name" })
    firstName: string

    @Column({ name: "last_name" })
    lastName: string

    @Column({ name: "middle_name" })
    middleName: string
}
```

> **Примечание** MySQL хранит ограничения уникальности как уникальные индексы

#### `@Check`

Этот декоратор позволяет создать в базе данных ограничение-проверку (check constraint) для одной или нескольких колонок.
Этот декоратор можно применять только к самой сущности.

Примеры:

```typescript
@Entity()
@Check(`"firstName" <> 'John' AND "lastName" <> 'Doe'`)
@Check(`"age" > 18`)
export class User {
    @Column()
    firstName: string

    @Column()
    lastName: string

    @Column()
    age: number
}
```

> **Примечание** MySQL не поддерживает ограничения-проверки.

#### `@Exclusion`

Этот декоратор позволяет создать в базе данных ограничение исключения (exclusion constraint) для одной или нескольких колонок.
Этот декоратор можно применять только к самой сущности.

Примеры:

```typescript
@Entity()
@Exclusion(`USING gist ("room" WITH =, tsrange("from", "to") WITH &&)`)
export class RoomBooking {
    @Column()
    room: string

    @Column()
    from: Date

    @Column()
    to: Date
}
```

> **Примечание** ограничения исключения поддерживает только PostgreSQL.

#### `@ForeignKey`

Этот декоратор позволяет создать в базе данных внешний ключ для одной или нескольких колонок.
Декоратор можно применять как к колонкам, так и к самой сущности.
Применяйте его к колонке, когда нужен внешний ключ по одной колонке,
и к сущности, когда требуется один внешний ключ по нескольким колонкам.

> **Примечание** **не используйте этот декоратор со связями.** Внешние ключи создаются автоматически для связей,
> которые вы определяете с помощью [декораторов связей](#декораторы-связей) (`@ManyToOne`, `@OneToOne` и т. д.).
> Декоратор `@ForeignKey` следует использовать только для создания внешних ключей в базе данных, когда
> вы не хотите определять соответствующую связь между сущностями.

Примеры:

```typescript
@Entity("orders")
@ForeignKey(() => City, ["cityId", "countryCode"], ["id", "countryCode"])
export class Order {
    @PrimaryColumn()
    id: number

    @Column("uuid", { name: "user_uuid" })
    @ForeignKey<User>("User", "uuid", { name: "FK_user_uuid" })
    userUuid: string

    @Column({ length: 2 })
    @ForeignKey(() => Country, "code")
    countryCode: string

    @Column()
    @ForeignKey("cities")
    cityId: number

    @Column()
    dispatchCountryCode: string

    @ManyToOne(() => Country)
    dispatchCountry: Country

    @Column()
    dispatchCityId: number

    @ManyToOne(() => City)
    dispatchCity: City
}
```

```typescript
@Entity("cities")
@Unique(["id", "countryCode"])
export class City {
    @PrimaryColumn()
    id: number

    @Column({ length: 2 })
    @ForeignKey("countries", { onDelete: "CASCADE", onUpdate: "CASCADE" })
    countryCode: string

    @Column()
    name: string
}
```

```typescript
@Entity("countries")
export class Country {
    @PrimaryColumn({ length: 2 })
    code: string

    @Column()
    name: string
}
```

```typescript
@Entity("users")
export class User {
    @PrimaryColumn({ name: "ref" })
    id: number

    @Column("uuid", { unique: true })
    uuid: string
}
```
