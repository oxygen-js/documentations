# Несколько источников данных, баз данных, схем и настройка репликации

## Использование нескольких источников данных

Чтобы использовать несколько источников данных, подключённых к разным базам, просто создайте несколько экземпляров DataSource:

```typescript
import { DataSource } from "typeorm"

const db1DataSource = new DataSource({
    type: "mysql",
    host: "localhost",
    port: 3306,
    username: "root",
    password: "admin",
    database: "db1",
    entities: [__dirname + "/entities/*{.js,.ts}"],
    synchronize: true,
})

const db2DataSource = new DataSource({
    type: "mysql",
    host: "localhost",
    port: 3306,
    username: "root",
    password: "admin",
    database: "db2",
    entities: [__dirname + "/entities/*{.js,.ts}"],
    synchronize: true,
})
```

## Использование нескольких баз данных в одном источнике данных

Чтобы работать с несколькими базами данных в одном источнике данных,
можно указать имя базы для каждой сущности:

```typescript
import { Entity, PrimaryGeneratedColumn, Column } from "typeorm"

@Entity({ database: "secondDB" })
export class User {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    firstName: string

    @Column()
    lastName: string
}
```

```typescript
import { Entity, PrimaryGeneratedColumn, Column } from "typeorm"

@Entity({ database: "thirdDB" })
export class Photo {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    url: string
}
```

Сущность `User` будет создана в базе данных `secondDB`, а сущность `Photo` — в базе данных `thirdDB`.
Все остальные сущности будут созданы в базе данных по умолчанию, указанной в опциях источника данных.

Если нужно выбрать данные из другой базы данных, достаточно указать сущность:

```typescript
const users = await dataSource
    .createQueryBuilder()
    .select()
    .from(User, "user")
    .addFrom(Photo, "photo")
    .andWhere("photo.userId = user.id")
    .getMany() // userId не является внешним ключом, так как это межбазовый запрос
```

Этот код сформирует следующий SQL-запрос (в зависимости от типа базы данных):

```sql
SELECT * FROM "secondDB"."user" "user", "thirdDB"."photo" "photo"
    WHERE "photo"."userId" = "user"."id"
```

Вместо сущности можно указать путь к таблице:

```typescript
const users = await dataSource
    .createQueryBuilder()
    .select()
    .from("secondDB.user", "user")
    .addFrom("thirdDB.photo", "photo")
    .andWhere("photo.userId = user.id")
    .getMany() // userId не является внешним ключом, так как это межбазовый запрос
```

Эта возможность поддерживается только в базах данных mysql и mssql.

## Использование нескольких схем в одном источнике данных

Чтобы использовать несколько схем в своих приложениях, просто задайте `schema` для каждой сущности:

```typescript
import { Entity, PrimaryGeneratedColumn, Column } from "typeorm"

@Entity({ schema: "secondSchema" })
export class User {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    firstName: string

    @Column()
    lastName: string
}
```

```typescript
import { Entity, PrimaryGeneratedColumn, Column } from "typeorm"

@Entity({ schema: "thirdSchema" })
export class Photo {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    url: string
}
```

Сущность `User` будет создана в схеме `secondSchema`, а сущность `Photo` — в схеме `thirdSchema`.
Все остальные сущности будут созданы в базе данных по умолчанию, указанной в опциях источника данных.

Если нужно выбрать данные из другой схемы, достаточно указать сущность:

```typescript
const users = await dataSource
    .createQueryBuilder()
    .select()
    .from(User, "user")
    .addFrom(Photo, "photo")
    .andWhere("photo.userId = user.id")
    .getMany()
```

Этот код сформирует следующий SQL-запрос (в зависимости от типа базы данных):

```sql
SELECT * FROM "secondSchema"."user" "user", "thirdSchema"."photo" "photo"
    WHERE "photo"."userId" = "user"."id"
```

Вместо сущности можно указать путь к таблице:

```typescript
const users = await dataSource
    .createQueryBuilder()
    .select()
    .from("secondSchema.user", "user") // в mssql можно указать даже базу данных: secondDB.secondSchema.user
    .addFrom("thirdSchema.photo", "photo") // в mssql можно указать даже базу данных: thirdDB.thirdSchema.photo
    .andWhere("photo.userId = user.id")
    .getMany()
```

Эта возможность поддерживается только в базах данных postgres и mssql.
В mssql также можно сочетать схемы и базы данных, например:

```typescript
import { Entity, PrimaryGeneratedColumn, Column } from "typeorm"

@Entity({ database: "secondDB", schema: "public" })
export class User {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    firstName: string

    @Column()
    lastName: string
}
```

## Репликация

С помощью TypeORM можно настроить репликацию с разделением чтения и записи.
Пример опций репликации:

```typescript
const datasource = new DataSource({
    type: "mysql",
    logging: true,
    replication: {
        master: {
            host: "server1",
            port: 3306,
            username: "test",
            password: "test",
            database: "test",
        },
        slaves: [
            {
                host: "server2",
                port: 3306,
                username: "test",
                password: "test",
                database: "test",
            },
            {
                host: "server3",
                port: 3306,
                username: "test",
                password: "test",
                database: "test",
            },
        ],
    },
})
```

Если реплики-slave заданы, TypeORM по умолчанию начнёт отправлять на них все запросы, которые возможно.

- все запросы, выполняемые методами `find` или через `SelectQueryBuilder`, будут использовать случайный экземпляр `slave`
- все запросы на запись, выполняемые через `update`, `create`, `InsertQueryBuilder`, `UpdateQueryBuilder` и т. п., будут использовать экземпляр `master`
- все «сырые» запросы, выполняемые через `.query()`, будут использовать экземпляр `master`
- все операции обновления схемы выполняются на экземпляре `master`

### Явный выбор адресата запроса

По умолчанию TypeORM отправляет все запросы на чтение случайной реплике-slave, а все записи — на master. Это значит, что как только вы впервые добавите настройки `replication` в конфигурацию, все существующие query runner'ы для чтения, у которых режим репликации не задан явно, начнут обращаться к slave. Для масштабируемости это хорошо, но если какие-то из этих запросов _обязаны_ возвращать актуальные данные, при создании query runner нужно явно передавать режим репликации.

Если вы хотите явно использовать `master` для запросов на чтение, передайте явный `ReplicationMode` при создании `QueryRunner`:

```typescript
const masterQueryRunner = dataSource.createQueryRunner("master")
try {
    const postsFromMaster = await dataSource
        .createQueryBuilder(Post, "post", masterQueryRunner) // QueryRunner можно передать как необязательный аргумент query builder
        .setQueryRunner(masterQueryRunner) // либо использовать setQueryRunner, который задаёт или переопределяет QueryRunner у query builder
        .getMany()
} finally {
    await masterQueryRunner.release()
}
```

Если вы хотите использовать slave для «сырых» запросов, при создании query runner передайте `slave` в качестве режима репликации:

```typescript
const slaveQueryRunner = dataSource.createQueryRunner("slave")
try {
    const userFromSlave = await slaveQueryRunner.query(
        "SELECT * FROM users WHERE id = $1",
        [userId],
        slaveQueryRunner,
    )
} finally {
    return slaveQueryRunner.release()
}
```

**Примечание**: созданные вручную экземпляры `QueryRunner` необходимо освобождать самостоятельно. Если этого не делать, они будут удерживать соединение, изъятое из пула, и другие запросы не смогут его использовать.

### Изменение адресата по умолчанию для чтения

Если вы не хотите, чтобы все операции чтения по умолчанию уходили на экземпляр `slave`, можно изменить адресата запросов на чтение по умолчанию, передав `defaultMode: "master"` в опциях репликации:

```typescript
const datasource = new DataSource({
    type: "mysql",
    logging: true,
    replication: {
        // задаём master в качестве адресата по умолчанию для запросов на чтение
        defaultMode: "master",
        master: {
            host: "server1",
            port: 3306,
            username: "test",
            password: "test",
            database: "test",
        },
        slaves: [
            {
                host: "server2",
                port: 3306,
                username: "test",
                password: "test",
                database: "test",
            },
        ],
    },
})
```

В этом режиме по умолчанию ни один запрос не уйдёт на реплики для чтения, и отправлять запросы на них придётся явно — вызовами `.createQueryRunner("slave")`.

Если вы впервые добавляете опции репликации в существующее приложение, это хороший вариант: поведение не изменится сразу, а переходить на реплики для чтения можно постепенно, query runner за query runner'ом.

### Поддерживаемые драйверы

Репликация поддерживается драйверами подключения MySQL, PostgreSQL, SQL Server, Cockroach, Oracle и Spanner.

Репликация в MySQL поддерживает дополнительные опции конфигурации:

```typescript
{
  replication: {
    master: {
      host: "server1",
      port: 3306,
      username: "test",
      password: "test",
      database: "test"
    },
    slaves: [{
      host: "server2",
      port: 3306,
      username: "test",
      password: "test",
      database: "test"
    }, {
      host: "server3",
      port: 3306,
      username: "test",
      password: "test",
      database: "test"
    }],

    /**
    * Если true, PoolCluster будет пытаться переподключиться при сбое соединения. (По умолчанию: true)
    */
    canRetry: true,

    /**
     * При сбое соединения счётчик ошибок узла errorCount увеличивается.
     * Когда errorCount превышает removeNodeErrorCount, узел удаляется из PoolCluster. (По умолчанию: 5)
     */
    removeNodeErrorCount: 5,

    /**
     * Если соединение не удалось, задаёт количество миллисекунд до следующей попытки подключения.
     * Если задано 0, узел вместо этого будет удалён и больше не будет использоваться. (По умолчанию: 0)
     */
     restoreNodeTimeout: 0,

    /**
     * Определяет, как выбираются slave-узлы:
     * RR: выбирать по очереди (Round-Robin).
     * RANDOM: выбирать узел случайным образом.
     * ORDER: безусловно выбирать первый доступный узел.
     */
    selector: "RR"
  }
}
```
