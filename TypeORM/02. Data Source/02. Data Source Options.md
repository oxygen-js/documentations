# Опции источника данных

## Что такое DataSourceOptions?

`DataSourceOptions` — это конфигурация источника данных, которую вы передаёте при создании нового экземпляра `DataSource`.
У разных СУБД есть свои специфичные опции.

## Общие опции источника данных

- `type` — тип СУБД. Вы обязаны указать, какой движок базы данных используете.
  Возможные значения:
  "mysql", "postgres", "cockroachdb", "sap", "spanner", "mariadb", "cordova", "react-native", "nativescript", "sqljs", "oracle", "mssql", "mongodb", "aurora-mysql", "aurora-postgres", "expo", "better-sqlite3", "capacitor".
  Эта опция **обязательна**.

- `extra` — дополнительные опции, передаваемые нижележащему драйверу.
  Используйте её, если хотите передать дополнительные настройки драйверу базы данных.

- `entities` — сущности или схемы сущностей, которые нужно загрузить и использовать для этого источника данных.
  Принимает классы сущностей, классы схем сущностей и пути к каталогам, из которых выполняется загрузка.
  Каталоги поддерживают glob-шаблоны.
  Пример: `entities: [Post, Category, "entities/*.js", "modules/**/entities/*.js"]`.
  Подробнее о [сущностях](<../03. Entity/01. Entities.md>).
  Подробнее о [схемах сущностей](<../03. Entity/06. Separating Entity Definition.md>).

- `subscribers` — подписчики, которые нужно загрузить и использовать для этого источника данных.
  Принимает как классы сущностей, так и каталоги, из которых выполняется загрузка.
  Каталоги поддерживают glob-шаблоны.
  Пример: `subscribers: [PostSubscriber, AppSubscriber, "subscribers/*.js", "modules/**/subscribers/*.js"]`.
  Подробнее о [подписчиках](<../11. Listeners and Subscribers.md>).

- `logging` — указывает, включено ли логирование.
  Если задано `true`, будет включено логирование запросов и ошибок.
  Можно также указать, какие типы логирования включить, например `["query", "error", "schema"]`.
  Подробнее о [логировании](<../14. Logging.md>).

- `logger` — логгер, используемый для логирования. Возможные значения: "advanced-console", "formatted-console", "simple-console" и "file".
  По умолчанию — "advanced-console". Также можно указать класс логгера, реализующий интерфейс `Logger`.
  Подробнее о [логировании](<../14. Logging.md>).

- `maxQueryExecutionTime` — если время выполнения запроса превысит заданное максимальное время выполнения (в миллисекундах),
  логгер запишет этот запрос в лог.

- `poolSize` — задаёт максимальное количество активных соединений в пуле.

- `namingStrategy` — стратегия именования таблиц и колонок в базе данных.

- `entityPrefix` — добавляет указанную строку в качестве префикса ко всем таблицам (или коллекциям) этого источника данных.

- `entitySkipConstructor` — указывает, должен ли TypeORM пропускать конструкторы при десериализации сущностей
  из базы данных. Учтите: если конструктор не вызывается, приватные свойства и свойства со значениями по умолчанию
  будут работать не так, как ожидается.

- `dropSchema` — удаляет схему при каждой инициализации источника данных.
  Будьте осторожны с этой опцией и не используйте её в production — иначе вы потеряете все production-данные.
  Эта опция полезна при отладке и разработке.

- `synchronize` — указывает, должна ли схема базы данных создаваться автоматически при каждом запуске приложения.
  Будьте осторожны с этой опцией и не используйте её в production — иначе вы можете потерять production-данные.
  Эта опция полезна при отладке и разработке.
  В качестве альтернативы можно использовать CLI и выполнять команду schema:sync.
  Учтите, что для базы данных MongoDB схема не создаётся, поскольку MongoDB не имеет схемы.
  Вместо этого синхронизация выполняется только созданием индексов.

- `migrations` — [миграции](<../06. Migrations/01. How migrations work.md>), которые нужно загрузить и использовать для этого источника данных

- `migrationsRun` — указывает, должны ли [миграции](<../06. Migrations/01. How migrations work.md>) автоматически выполняться при каждом запуске приложения.

- `migrationsTableName` — имя таблицы в базе данных, которая будет содержать информацию о выполненных [миграциях](<../06. Migrations/01. How migrations work.md>).

- `migrationsTransactionMode` — управляет режимом транзакций при выполнении [миграций](<../06. Migrations/01. How migrations work.md>).

- `metadataTableName` — имя таблицы в базе данных, которая будет содержать информацию о метаданных таблиц.
  По умолчанию эта таблица называется "typeorm_metadata".

- `cache` — включает кэширование результатов сущностей. Здесь же можно настроить тип кэша и другие его опции.
  Подробнее о кэшировании читайте [здесь](<../08. Query Builder/06. Caching queries.md>).

- `isolateWhereStatements` — включает изоляцию выражений where, автоматически оборачивая каждое условие where в скобки.
  Например, `.where("user.firstName = :search OR user.lastName = :search")` превращается в `WHERE (user.firstName = ? OR user.lastName = ?)` вместо `WHERE user.firstName = ? OR user.lastName = ?`

- `invalidWhereValuesBehavior` — управляет обработкой значений null и undefined в условиях where для высокоуровневых операций (операции поиска, методы репозитория, методы EntityManager). Не влияет напрямую на `.where()` у QueryBuilder.
    - варианты поведения для `null`:
        - `'ignore'` — пропускать свойства со значением null
        - `'sql-null'` — преобразовывать null в SQL NULL
        - `'throw'` (по умолчанию) — выбрасывать ошибку
    - варианты поведения для `undefined`:
        - `'ignore'` — пропускать свойства со значением undefined
        - `'throw'` (по умолчанию) — выбрасывать ошибку

    Пример: `invalidWhereValuesBehavior: { null: 'sql-null', undefined: 'ignore' }`.

    Подробнее об [обработке null и undefined](<05. Handling null and undefined values in where conditions.md>).

## Пример опций источника данных

Небольшой пример опций источника данных для mysql:

```typescript
{
    host: "localhost",
    port: 3306,
    username: "test",
    password: "test",
    database: "test",
    logging: true,
    synchronize: true,
    entities: [__dirname + "/entities/**/*{.js,.ts}"],
    subscribers: [__dirname + "/subscribers/**/*{.js,.ts}"],
    entitySchemas: [__dirname + "/schemas/**/*.json"],
    migrations: [__dirname + "/migrations/**/*{.js,.ts}"]
}
```
