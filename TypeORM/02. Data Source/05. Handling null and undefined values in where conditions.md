# Обработка значений null и undefined в условиях where

В условиях «WHERE» значения `null` и `undefined` не являются строго допустимыми значениями в TypeORM.

Передача заведомо известного значения `null` запрещается TypeScript на этапе компиляции (если в tsconfig.json включена опция `strictNullChecks`). По умолчанию значения `null` и `undefined`, встреченные во время выполнения, приводят к выбросу ошибки.

Способ обработки значений `null` и `undefined` можно настроить через опцию конфигурации `invalidWhereValuesBehavior` в опциях источника данных. Это распространяется на высокоуровневые операции: операции поиска, методы репозитория и методы EntityManager (update, delete, softDelete, restore).

> **Предупреждение** Эта настройка **не** влияет на методы QueryBuilder `.where()`, `.andWhere()` и `.orWhere()`. QueryBuilder — низкоуровневый API, где значения null/undefined передаются как есть. Для явной обработки null в QueryBuilder используйте оператор `IsNull()` или параметризованные условия.

## Поведение по умолчанию

По умолчанию TypeORM выбрасывает ошибку, когда в условиях where встречаются значения `null` или `undefined`. Это предотвращает неожиданные результаты и помогает обнаруживать потенциальные ошибки на раннем этапе:

```typescript
// Both queries will throw an error
const posts1 = await repository.find({
    where: {
        text: null,
    },
})
// Error: Null value encountered in property 'text' of a where condition.

const posts2 = await repository.find({
    where: {
        text: undefined,
    },
})
// Error: Undefined value encountered in property 'text' of a where condition.
```

Чтобы найти совпадения со значениями null в условиях where, используйте оператор `IsNull` (подробности см. в [Опциях поиска](<../07. Working with Entity Manager/03. Find Options.md>)):

```typescript
const posts = await repository.find({
    where: {
        text: IsNull(),
    },
})
```

## Конфигурация

Настроить обработку значений null и undefined можно с помощью опции `invalidWhereValuesBehavior` в конфигурации источника данных:

```typescript
const dataSource = new DataSource({
    // ... other options
    invalidWhereValuesBehavior: {
        null: "ignore" | "sql-null" | "throw",
        undefined: "ignore" | "throw",
    },
})
```

### Варианты поведения для null

Поведение для `null` можно задать одним из трёх значений:

#### `'ignore'`

Значения JavaScript `null` в условиях where игнорируются, и свойство пропускается:

```typescript
const dataSource = new DataSource({
    // ... other options
    invalidWhereValuesBehavior: {
        null: "ignore",
    },
})

// This will return all posts, ignoring the text property
const posts = await repository.find({
    where: {
        text: null,
    },
})
```

#### `'sql-null'`

Значения JavaScript `null` преобразуются в SQL-условия `NULL`:

```typescript
const dataSource = new DataSource({
    // ... other options
    invalidWhereValuesBehavior: {
        null: "sql-null",
    },
})

// This will only return posts where the text column is NULL in the database
const posts = await repository.find({
    where: {
        text: null,
    },
})
```

#### `'throw'` (по умолчанию)

Значения JavaScript `null` приводят к выбросу TypeORMError:

```typescript
const dataSource = new DataSource({
    // ... other options
    invalidWhereValuesBehavior: {
        null: "throw",
    },
})

// This will throw an error
const posts = await repository.find({
    where: {
        text: null,
    },
})
// Error: Null value encountered in property 'text' of a where condition.
// To match with SQL NULL, the IsNull() operator must be used.
// Set 'invalidWhereValuesBehavior.null' to 'ignore' or 'sql-null' in data source options to skip or handle null values.
```

### Варианты поведения для undefined

Поведение для `undefined` можно задать одним из двух значений:

#### `'ignore'`

Значения JavaScript `undefined` в условиях where игнорируются, и свойство пропускается:

```typescript
const dataSource = new DataSource({
    // ... other options
    invalidWhereValuesBehavior: {
        undefined: "ignore",
    },
})

// This will return all posts, ignoring the text property
const posts = await repository.find({
    where: {
        text: undefined,
    },
})
```

#### `'throw'` (по умолчанию)

Значения JavaScript `undefined` приводят к выбросу TypeORMError:

```typescript
const dataSource = new DataSource({
    // ... other options
    invalidWhereValuesBehavior: {
        undefined: "throw",
    },
})

// This will throw an error
const posts = await repository.find({
    where: {
        text: undefined,
    },
})
// Error: Undefined value encountered in property 'text' of a where condition.
// Set 'invalidWhereValuesBehavior.undefined' to 'ignore' in data source options to skip properties with undefined values.
```

Обратите внимание: это относится только к явно заданным значениям `undefined`, а не к пропущенным свойствам.

## Совместное использование обеих опций

Оба варианта поведения можно настроить независимо друг от друга для полного контроля:

```typescript
const dataSource = new DataSource({
    // ... other options
    invalidWhereValuesBehavior: {
        null: "sql-null",
        undefined: "throw",
    },
})
```

Такая конфигурация будет:

1. Преобразовывать значения JavaScript `null` в SQL `NULL` в условиях where
2. Выбрасывать ошибку, если встретится любое значение `undefined`
3. По-прежнему игнорировать свойства, которые не указаны в выражении where

Такое сочетание полезно, когда вы хотите:

- Явно указывать поиск значений NULL в базе данных
- Отлавливать потенциальные ошибки программирования, при которых значения undefined могут просочиться в ваши запросы

## Поддерживаемые операции

Конфигурация `invalidWhereValuesBehavior` применяется к высокоуровневым операциям TypeORM, но не к прямому методу `.where()` у QueryBuilder:

### Операции поиска

```typescript
// Repository.find() / findOne() / findBy() / findOneBy()
await repository.find({ where: { text: null } }) // Respects invalidWhereValuesBehavior

// EntityManager.find() / findOne() / findBy() / findOneBy()
await manager.find(Post, { where: { text: null } }) // Respects invalidWhereValuesBehavior
```

### Методы Repository и EntityManager

```typescript
// Repository.update()
await repository.update({ text: null }, { title: "Updated" }) // Respects invalidWhereValuesBehavior

// Repository.delete()
await repository.delete({ text: null }) // Respects invalidWhereValuesBehavior

// EntityManager.update()
await manager.update(Post, { text: null }, { title: "Updated" }) // Respects invalidWhereValuesBehavior

// EntityManager.delete()
await manager.delete(Post, { text: null }) // Respects invalidWhereValuesBehavior

// EntityManager.softDelete()
await manager.softDelete(Post, { text: null }) // Respects invalidWhereValuesBehavior
```

> **Предупреждение** **Пустые критерии отклоняются.** Методы `update`, `delete`, `softDelete` и `restore` требуют непустых критериев — пустое условие превратится в `WHERE 1=1` и затронет **все** строки. Поскольку режим `"ignore"` вырезает свойства со значениями `null`/`undefined`, критерий, у которого вырезаны **все** ключи, становится пустым. В этом случае операция отклоняется вместо выполнения записи без фильтра:
>
> ```typescript
> const dataSource = new DataSource({
>     // ... other options
>     invalidWhereValuesBehavior: { null: "ignore", undefined: "ignore" },
> })
>
> // { text: null } becomes {} after stripping -> rejected; the table is NOT emptied
> await manager.delete(Post, { text: null })
> // Error: Empty criteria(s) are not allowed for the delete method.
> ```
>
> Если вы намеренно хотите затронуть все строки, используйте специальные методы `updateAll()` / `deleteAll()`.

### Нормализуются только критерии в виде простого объекта

`invalidWhereValuesBehavior` нормализует только **простые объекты `FindOptionsWhere`**. Любое другое значение критерия, переданное в `update` / `delete` / `softDelete` / `restore` — **экземпляр класса сущности**, `FindOperator`, массив, `Date`, `Buffer` или примитивный `id`, — передаётся без изменений и **не** проверяется. Поэтому экземпляр сущности, у которого допускающая NULL колонка равна `null`, превратится в `col = NULL` (не совпадёт ни с чем), а не вызовет ошибку и не будет преобразован. Если вам нужна обработка null, передайте простой объект с оператором `IsNull()` (например, `{ text: IsNull() }`).

> Глубокая проверка критериев в виде экземпляра сущности требует метаданных сущности и выходит за рамки этого раздела; используйте простой объект `FindOptionsWhere`, если хотите, чтобы применялось `invalidWhereValuesBehavior`.

### QueryBuilder с setFindOptions

```typescript
// setFindOptions goes through the find-options path, so it respects the setting
await dataSource
    .createQueryBuilder(Post, "post")
    .setFindOptions({ where: { text: null } }) // Respects invalidWhereValuesBehavior
    .getMany()
```

### Не затрагивается: `.where()` у QueryBuilder

Методы QueryBuilder `.where()`, `.andWhere()` и `.orWhere()` — низкоуровневые API, и эта настройка на них **не** влияет. Значения null и undefined передаются как есть:

```typescript
// This does NOT respect invalidWhereValuesBehavior — null passes through as-is
await dataSource
    .createQueryBuilder()
    .update(Post)
    .set({ title: "Updated" })
    .where({ text: null })
    .execute()
```

## Как ведут себя null и undefined в `.where()` у QueryBuilder

Поскольку QueryBuilder — низкоуровневый API, значения null и undefined **не** проверяются и не преобразуются. Понимание их поведения важно, чтобы избежать неожиданных результатов.

### `null` в `.where()` у QueryBuilder

Когда `null` передаётся в качестве значения в объектной форме `.where()`, генерируется SQL-проверка на равенство с `NULL`:

```typescript
await dataSource
    .createQueryBuilder(Post, "post")
    .where({ text: null })
    .getMany()
// Generates: WHERE post.text = NULL
```

В SQL выражение `column = NULL` **всегда ложно** — ничто не равно NULL. Такой запрос **вернёт ноль результатов**, что почти наверняка не соответствует вашим намерениям. Чтобы найти совпадения со значениями NULL, используйте оператор `IsNull()`:

```typescript
import { IsNull } from "typeorm"

await dataSource
    .createQueryBuilder(Post, "post")
    .where({ text: IsNull() })
    .getMany()
// Generates: WHERE post.text IS NULL
```

Либо используйте строковое условие:

```typescript
await dataSource
    .createQueryBuilder(Post, "post")
    .where("post.text IS NULL")
    .getMany()
```

### `undefined` в `.where()` у QueryBuilder

Когда в качестве значения передаётся `undefined`, поведение то же самое — генерируется `WHERE column = NULL`, что всегда ложно:

```typescript
await dataSource
    .createQueryBuilder(Post, "post")
    .where({ text: undefined })
    .getMany()
// Generates: WHERE post.text = NULL
// Returns: zero results
```

### Сводная таблица

| Значение                             | Высокоуровневый API (find/репозиторий/менеджер) | QueryBuilder `.where()`           |
| ------------------------------------ | ---------------------------------------- | --------------------------------- |
| `null` с `"ignore"`                  | Свойство пропускается — без фильтра       | `WHERE col = NULL` — ноль результатов |
| `null` с `"sql-null"`                | `WHERE col IS NULL`                      | `WHERE col = NULL` — ноль результатов |
| `null` с `"throw"` (по умолчанию)    | Выбрасывает ошибку                       | `WHERE col = NULL` — ноль результатов |
| `undefined` с `"ignore"`             | Свойство пропускается — без фильтра       | `WHERE col = NULL` — ноль результатов |
| `undefined` с `"throw"` (по умолчанию) | Выбрасывает ошибку                     | `WHERE col = NULL` — ноль результатов |
| `IsNull()`                           | `WHERE col IS NULL`                      | `WHERE col IS NULL`               |

> **Совет** Всегда используйте `IsNull()`, когда нужно найти совпадения со значениями SQL NULL, независимо от того, каким API вы пользуетесь. Он корректно работает и в высокоуровневом контексте, и в QueryBuilder.
