# Начало работы

TypeORM — это [ORM](https://en.wikipedia.org/wiki/Object-relational_mapping),
который работает на платформах Node.js, Browser, Cordova, Ionic, React Native, NativeScript, Expo и Electron
и может использоваться с TypeScript и JavaScript (ES2021).

Его цель — всегда поддерживать новейшие возможности JavaScript и давать дополнительные средства,
которые помогают разрабатывать любые приложения, работающие с базами данных: от
небольших программ с парой таблиц до крупных корпоративных систем
с несколькими базами данных.

TypeORM поддерживает больше баз данных, чем любой другой JS/TS ORM: [Google Spanner](<13. Drivers/01. Google Spanner.md>), [Microsoft SqlServer](<13. Drivers/02. Microsoft SQLServer.md>), [MongoDB](<13. Drivers/03. MongoDB.md>), [MySQL/MariaDB](<13. Drivers/04. MySQL - MariaDB.md>), [Oracle](<13. Drivers/05. Oracle.md>), [Postgres](<13. Drivers/06. Postgres - CockroachDB.md>), [SAP HANA](<13. Drivers/07. SAP HANA.md>) и [SQLite](<13. Drivers/08. SQLite.md>), а также производные от них базы данных и различные драйверы.

TypeORM поддерживает оба паттерна — [Active Record](<16. Guides/01. Active Record vs Data Mapper.md#что-такое-паттерн-active-record>) и [Data Mapper](<16. Guides/01. Active Record vs Data Mapper.md#что-такое-паттерн-data-mapper>),
в отличие от всех существующих сегодня JavaScript-ORM.
Это значит, что вы можете максимально продуктивно писать качественные,
слабо связанные, масштабируемые и удобные в сопровождении приложения.

На TypeORM сильно повлияли другие ORM, такие как [Hibernate](http://hibernate.org/orm/),
[Doctrine](http://www.doctrine-project.org/) и [Entity Framework](https://www.asp.net/entity-framework).

С TypeORM ваши модели выглядят так:

```typescript
import { Entity, PrimaryGeneratedColumn, Column } from "typeorm"

@Entity()
export class User {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    firstName: string

    @Column()
    lastName: string

    @Column()
    age: number
}
```

А ваша доменная логика — так:

```typescript
const userRepository = AppDataSource.getRepository(User)

const user = new User()
user.firstName = "Timber"
user.lastName = "Saw"
user.age = 25
await userRepository.save(user)

const allUsers = await userRepository.find()
const firstUser = await userRepository.findOneBy({
    id: 1,
}) // поиск по id
const timber = await userRepository.findOneBy({
    firstName: "Timber",
    lastName: "Saw",
}) // поиск по firstName и lastName

await userRepository.remove(timber)
```

Если же вам ближе реализация `ActiveRecord`, вы можете использовать и её:

```typescript
import { Entity, PrimaryGeneratedColumn, Column, BaseEntity } from "typeorm"

@Entity()
export class User extends BaseEntity {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    firstName: string

    @Column()
    lastName: string

    @Column()
    age: number
}
```

Тогда ваша доменная логика будет выглядеть так:

```typescript
const user = new User()
user.firstName = "Timber"
user.lastName = "Saw"
user.age = 25
await user.save()

const allUsers = await User.find()
const firstUser = await User.findOneBy({
    id: 1,
})
const timber = await User.findOneBy({
    firstName: "Timber",
    lastName: "Saw",
})

await timber.remove()
```

## Установка

1. Установите npm-пакет:

    `npm install typeorm`

2. Понадобится установить шим `reflect-metadata`:

    `npm install reflect-metadata`

    и импортировать его где-нибудь в глобальной области вашего приложения (например, в `app.ts`):

    `import "reflect-metadata"`

3. Возможно, потребуется установить типы для node:

    `npm install @types/node --save-dev`

4. Установите драйвер базы данных: см. документацию по каждому конкретному драйверу — [mongodb](<13. Drivers/03. MongoDB.md#установка>), [mssql](<13. Drivers/02. Microsoft SQLServer.md#установка>), [mysql/mariadb](<13. Drivers/04. MySQL - MariaDB.md#установка>), [oracle](<13. Drivers/05. Oracle.md#установка>), [postgres](<13. Drivers/06. Postgres - CockroachDB.md#установка>), [sap](<13. Drivers/07. SAP HANA.md#установка>), [spanner](<13. Drivers/01. Google Spanner.md#установка>), [sqlite](<13. Drivers/08. SQLite.md#установка>).

### Настройка TypeScript

Также убедитесь, что вы используете TypeScript версии **4.5** или выше
и что в `tsconfig.json` включены следующие настройки:

```json
"emitDecoratorMetadata": true,
"experimentalDecorators": true,
```

## Быстрый старт

Быстрее всего начать работу с TypeORM можно с помощью его CLI-команд, генерирующих стартовый проект.
Быстрый старт работает, только если вы используете TypeORM в приложении на Node.js.
Если вы работаете на других платформах, переходите к [пошаговому руководству](#пошаговое-руководство).

Чтобы создать новый проект через CLI, выполните команду:

```shell
npx typeorm init --name MyProject --database postgres
```

Здесь `name` — имя вашего проекта, а `database` — база данных, которую вы будете использовать.
Значение `database` может быть одним из следующих: `mysql`, `mariadb`, `postgres`, `cockroachdb`, `sqlite`, `mssql`, `sap`, `spanner`, `oracle`, `mongodb`,
`cordova`, `react-native`, `expo`, `nativescript`.

Эта команда создаст новый проект в каталоге `MyProject` со следующими файлами:

```text
MyProject
├── src                   // место для вашего кода на TypeScript
│   ├── entities          // место хранения ваших сущностей (моделей базы данных)
│   │   └── User.ts       // пример сущности
│   ├── migrations        // место хранения ваших миграций
│   ├── data-source.ts    // источник данных и вся конфигурация подключения
│   └── index.ts          // точка входа вашего приложения
├── .gitignore            // стандартный файл gitignore
├── package.json          // зависимости node-модулей
├── README.md             // простой файл readme
└── tsconfig.json         // параметры компилятора TypeScript
```

> `typeorm init` можно запустить и в существующем node-проекте, но будьте осторожны — команда может перезаписать некоторые уже имеющиеся у вас файлы.

Следующий шаг — установить зависимости нового проекта:

```shell
cd MyProject
npm install
```

После установки всех зависимостей отредактируйте файл `data-source.ts` и укажите в нём свои параметры подключения к базе данных:

```ts
export const AppDataSource = new DataSource({
    type: "postgres",
    host: "localhost",
    port: 5432,
    username: "test",
    password: "test",
    database: "test",
    synchronize: true,
    logging: true,
    entities: [Post, Category],
    subscribers: [],
    migrations: [],
})
```

Как правило, вам достаточно настроить лишь опции
`host`, `username`, `password`, `database` и, возможно, `port`.

Когда конфигурация готова и все node-модули установлены, можно запускать приложение:

```shell
npm start
```

Вот и всё: приложение должно успешно запуститься и добавить нового пользователя в базу данных.
Дальше вы можете продолжать работу над этим проектом, подключать нужные вам модули
и создавать новые сущности.

> Чтобы сгенерировать ESM-проект, выполните команду
> `npx typeorm init --name MyProject --database postgres --module esm`.

> Чтобы сгенерировать более продвинутый проект с установленным express, выполните команду
> `npx typeorm init --name MyProject --database mysql --express`.

> Чтобы сгенерировать файл docker-compose, выполните команду
> `npx typeorm init --name MyProject --database postgres --docker`.

## Пошаговое руководство

Чего вы ждёте от ORM?
Прежде всего — что он сам создаст таблицы в базе данных
и позволит искать, вставлять, обновлять и удалять данные, избавив от мучительного
написания множества плохо поддерживаемых SQL-запросов.
Это руководство покажет, как настроить TypeORM с нуля и заставить его делать то, чего вы ждёте от ORM.

### Создание модели

Работа с базой данных начинается с создания таблиц.
Как сказать TypeORM, что нужно создать таблицу в базе данных?
Ответ — через модели.
Модели в вашем приложении и есть таблицы вашей базы данных.

Например, у вас есть модель `Photo`:

```typescript
export class Photo {
    id: number
    name: string
    description: string
    filename: string
    views: number
    isPublished: boolean
}
```

И вы хотите хранить фотографии в базе данных.
Чтобы что-то хранить в базе данных, вам сначала нужна таблица,
а таблицы создаются из ваших моделей.
Но не из всех моделей, а только из тех, которые вы объявите _сущностями_.

### Создание сущности

_Сущность_ — это ваша модель, помеченная декоратором `@Entity`.
Для таких моделей будет создана таблица в базе данных.
С сущностями вы работаете в TypeORM повсюду.
Их можно загружать, вставлять, обновлять, удалять и выполнять с ними другие операции.

Сделаем нашу модель `Photo` сущностью:

```typescript
import { Entity } from "typeorm"

@Entity()
export class Photo {
    id: number
    name: string
    description: string
    filename: string
    views: number
    isPublished: boolean
}
```

Теперь для сущности `Photo` будет создана таблица в базе данных, и мы сможем работать с ней в любом месте приложения.
Таблицу мы создали — но какая же таблица без колонок?
Добавим в неё несколько колонок.

### Добавление колонок в таблицу

Чтобы добавить колонки в базу данных, пометьте декоратором `@Column`
те свойства сущности, которые должны стать колонками.

```typescript
import { Entity, Column } from "typeorm"

@Entity()
export class Photo {
    @Column()
    id: number

    @Column()
    name: string

    @Column()
    description: string

    @Column()
    filename: string

    @Column()
    views: number

    @Column()
    isPublished: boolean
}
```

Теперь в таблицу `photo` будут добавлены колонки `id`, `name`, `description`, `filename`, `views` и `isPublished`.
Типы колонок в базе данных выводятся из типов свойств, которые вы использовали: например,
`number` превратится в `integer`, `string` — в `varchar`, `boolean` — в `bool` и т. д.
Но вы можете использовать любой тип колонки, поддерживаемый вашей СУБД, явно указав его в декораторе `@Column`.

Мы сгенерировали таблицу с колонками, но осталась одна деталь.
В каждой таблице базы данных должна быть колонка с первичным ключом.

### Создание колонки первичного ключа

Каждая сущность **обязана** иметь хотя бы одну колонку первичного ключа.
Это требование, обойти которое нельзя.
Чтобы сделать колонку первичным ключом, используйте декоратор `@PrimaryColumn`.

```typescript
import { Entity, Column, PrimaryColumn } from "typeorm"

@Entity()
export class Photo {
    @PrimaryColumn()
    id: number

    @Column()
    name: string

    @Column()
    description: string

    @Column()
    filename: string

    @Column()
    views: number

    @Column()
    isPublished: boolean
}
```

### Создание автогенерируемой колонки

Допустим, теперь вы хотите, чтобы значение колонки id генерировалось автоматически (это известно как auto-increment / sequence / serial / generated identity column).
Для этого замените декоратор `@PrimaryColumn` на `@PrimaryGeneratedColumn`:

```typescript
import { Entity, Column, PrimaryGeneratedColumn } from "typeorm"

@Entity()
export class Photo {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    name: string

    @Column()
    description: string

    @Column()
    filename: string

    @Column()
    views: number

    @Column()
    isPublished: boolean
}
```

### Типы данных колонок

Теперь приведём в порядок типы данных. По умолчанию строка отображается на тип вроде varchar(255) (в зависимости от типа СУБД).
Число отображается на целочисленный тип (тоже в зависимости от типа СУБД).
Но мы не хотим ограничивать все наши колонки varchar-ами и целыми числами.
Зададим правильные типы данных:

```typescript
import { Entity, Column, PrimaryGeneratedColumn } from "typeorm"

@Entity()
export class Photo {
    @PrimaryGeneratedColumn()
    id: number

    @Column({
        length: 100,
    })
    name: string

    @Column("text")
    description: string

    @Column()
    filename: string

    @Column("double")
    views: number

    @Column()
    isPublished: boolean
}
```

Типы колонок зависят от конкретной СУБД.
Вы можете задать любой тип колонки, поддерживаемый вашей базой данных.
Подробнее о поддерживаемых типах колонок можно прочитать [здесь](<03. Entity/01. Entities.md#типы-колонок>).

### Создание нового `DataSource`

Теперь, когда сущность создана, создадим файл `index.ts` и настроим в нём наш `DataSource`:

```typescript
import "reflect-metadata"
import { DataSource } from "typeorm"
import { Photo } from "./entity/Photo"

const AppDataSource = new DataSource({
    type: "postgres",
    host: "localhost",
    port: 5432,
    username: "root",
    password: "admin",
    database: "test",
    entities: [Photo],
    synchronize: true,
    logging: false,
})

// чтобы установить первоначальное соединение с базой данных, зарегистрировать все сущности
// и «синхронизировать» схему базы данных, вызовите метод "initialize()" только что созданного
// источника данных один раз при старте вашего приложения
try {
    await AppDataSource.initialize()
} catch (error) {
    console.log(error)
}
```

В этом примере мы используем Postgres, но вы можете взять любую другую поддерживаемую базу данных.
Чтобы переключиться на другую СУБД, поменяйте в опциях `type` на используемый вами тип базы данных:
`mysql`, `mariadb`, `postgres`, `cockroachdb`, `sqlite`, `mssql`, `oracle`, `sap`, `spanner`, `cordova`, `nativescript`, `react-native`,
`expo` или `mongodb`.
Не забудьте также указать собственные значения host, port, username, password и database.

Мы добавили нашу сущность Photo в список сущностей этого источника данных.
Каждая сущность, используемая в соединении, должна быть указана в этом списке.

Опция `synchronize` гарантирует, что ваши сущности будут синхронизироваться с базой данных при каждом запуске приложения.

### Запуск приложения

Теперь, если вы запустите свой `index.ts`, будет установлено соединение с базой данных и создана таблица для ваших фотографий.

```text
+-------------+--------------+----------------------------+
|                         photo                           |
+-------------+--------------+----------------------------+
| id          | int          | PRIMARY KEY AUTO_INCREMENT |
| name        | varchar(100) |                            |
| description | text         |                            |
| filename    | varchar(255) |                            |
| views       | int          |                            |
| isPublished | boolean      |                            |
+-------------+--------------+----------------------------+
```

### Создание фотографии и вставка её в базу данных

Теперь создадим новую фотографию и сохраним её в базе данных:

```typescript
import { Photo } from "./entity/Photo"
import { AppDataSource } from "./index"

const photo = new Photo()
photo.name = "Me and Bears"
photo.description = "I am near polar bears"
photo.filename = "photo-with-bears.jpg"
photo.views = 1
photo.isPublished = true

await AppDataSource.manager.save(photo)
console.log("Photo has been saved. Photo id is", photo.id)
```

Как только сущность сохранена, она получает сгенерированный id.
Метод `save` возвращает экземпляр того же объекта, который вы ему передали.
Это не новая копия объекта: метод изменяет его «id» и возвращает его же.

### Использование Entity Manager

Мы только что создали новую фотографию и сохранили её в базе данных.
Для сохранения мы использовали `EntityManager`.
С помощью entity manager вы можете работать с любой сущностью в вашем приложении.
Например, загрузим сохранённую сущность:

```typescript
import { Photo } from "./entity/Photo"
import { AppDataSource } from "./index"

const savedPhotos = await AppDataSource.manager.find(Photo)
console.log("All photos from the db: ", savedPhotos)
```

`savedPhotos` будет массивом объектов Photo с данными, загруженными из базы данных.

Подробнее об [EntityManager](<07. Working with Entity Manager/01. EntityManager.md>).

### Использование репозиториев

Теперь отрефакторим наш код и будем использовать `Repository` вместо `EntityManager`.
У каждой сущности есть свой репозиторий, который выполняет все операции с этой сущностью.
Когда вы много работаете с сущностями, репозитории удобнее, чем EntityManager:

```typescript
import { Photo } from "./entity/Photo"
import { AppDataSource } from "./index"

const photo = new Photo()
photo.name = "Me and Bears"
photo.description = "I am near polar bears"
photo.filename = "photo-with-bears.jpg"
photo.views = 1
photo.isPublished = true

const photoRepository = AppDataSource.getRepository(Photo)

await photoRepository.save(photo)
console.log("Photo has been saved")

const savedPhotos = await photoRepository.find()
console.log("All photos from the db: ", savedPhotos)
```

Подробнее о Repository читайте [здесь](<07. Working with Entity Manager/02. Repository.md>).

### Загрузка из базы данных

Попробуем другие операции загрузки с помощью репозитория:

```typescript
import { Photo } from "./entity/Photo"
import { AppDataSource } from "./index"

const photoRepository = AppDataSource.getRepository(Photo)
const allPhotos = await photoRepository.find()
console.log("All photos from the db: ", allPhotos)

const firstPhoto = await photoRepository.findOneBy({
    id: 1,
})
console.log("First photo from the db: ", firstPhoto)

const meAndBearsPhoto = await photoRepository.findOneBy({
    name: "Me and Bears",
})
console.log("Me and Bears photo from the db: ", meAndBearsPhoto)

const allViewedPhotos = await photoRepository.findBy({ views: 1 })
console.log("All viewed photos: ", allViewedPhotos)

const allPublishedPhotos = await photoRepository.findBy({ isPublished: true })
console.log("All published photos: ", allPublishedPhotos)

const [photos, photosCount] = await photoRepository.findAndCount()
console.log("All photos: ", photos)
console.log("Photos count: ", photosCount)
```

### Обновление в базе данных

Теперь загрузим из базы данных одну фотографию, изменим её и сохраним:

```typescript
import { Photo } from "./entity/Photo"
import { AppDataSource } from "./index"

const photoRepository = AppDataSource.getRepository(Photo)
const photoToUpdate = await photoRepository.findOneBy({
    id: 1,
})
photoToUpdate.name = "Me, my friends and polar bears"
await photoRepository.save(photoToUpdate)
```

Теперь фотография с `id = 1` будет обновлена в базе данных.

### Удаление из базы данных

Теперь удалим нашу фотографию из базы данных:

```typescript
import { Photo } from "./entity/Photo"
import { AppDataSource } from "./index"

const photoRepository = AppDataSource.getRepository(Photo)
const photoToRemove = await photoRepository.findOneBy({
    id: 1,
})
await photoRepository.remove(photoToRemove)
```

Теперь фотография с `id = 1` будет удалена из базы данных.

### Создание связи «один к одному»

Создадим связь «один к одному» с другим классом.
Создадим новый класс в `PhotoMetadata.ts`. Этот класс PhotoMetadata будет содержать дополнительную метаинформацию о нашей фотографии:

```typescript
import {
    Entity,
    Column,
    PrimaryGeneratedColumn,
    OneToOne,
    JoinColumn,
} from "typeorm"
import { Photo } from "./Photo"

@Entity()
export class PhotoMetadata {
    @PrimaryGeneratedColumn()
    id: number

    @Column("int")
    height: number

    @Column("int")
    width: number

    @Column()
    orientation: string

    @Column()
    compressed: boolean

    @Column()
    comment: string

    @OneToOne(() => Photo)
    @JoinColumn()
    photo: Photo
}
```

Здесь мы используем новый декоратор `@OneToOne`. Он позволяет создать связь «один к одному» между двумя сущностями. Мы также добавляем декоратор `@JoinColumn`, который указывает, что эта сторона связи будет владеть связью.
Связи могут быть однонаправленными или двунаправленными.
Владеющей может быть только одна сторона связи.
Декоратор `@JoinColumn` обязателен на владеющей стороне связи.

Если запустить приложение, вы увидите вновь созданную таблицу, и в ней будет колонка с внешним ключом для связи с фотографией:

```text
+-------------+--------------+----------------------------+
|                     photo_metadata                      |
+-------------+--------------+----------------------------+
| id          | int          | PRIMARY KEY AUTO_INCREMENT |
| height      | int          |                            |
| width       | int          |                            |
| comment     | varchar(255) |                            |
| compressed  | boolean      |                            |
| orientation | varchar(255) |                            |
| photoId     | int          | FOREIGN KEY                |
+-------------+--------------+----------------------------+
```

### Сохранение связи «один к одному»

Теперь сохраним фотографию и её метаданные и свяжем их друг с другом.

```typescript
import { Photo } from "./entity/Photo"
import { PhotoMetadata } from "./entity/PhotoMetadata"

// Создаём фотографию
const photo = new Photo()
photo.name = "Me and Bears"
photo.description = "I am near polar bears"
photo.filename = "photo-with-bears.jpg"
photo.views = 1
photo.isPublished = true

// Создаём метаданные фотографии
const metadata = new PhotoMetadata()
metadata.height = 640
metadata.width = 480
metadata.compressed = true
metadata.comment = "cybershoot"
metadata.orientation = "portrait"
metadata.photo = photo // так мы их связываем

// Получаем репозитории сущностей
const photoRepository = AppDataSource.getRepository(Photo)
const metadataRepository = AppDataSource.getRepository(PhotoMetadata)

// Сначала нужно сохранить фотографию
await photoRepository.save(photo)

// Фотография сохранена. Теперь нужно сохранить её метаданные
await metadataRepository.save(metadata)

// Готово
console.log(
    "Metadata is saved, and the relation between metadata and photo is created in the database too",
)
```

### Обратная сторона связи

Связи могут быть однонаправленными или двунаправленными.
Сейчас наша связь между PhotoMetadata и Photo однонаправленная.
Владелец связи — PhotoMetadata, а Photo ничего не знает о PhotoMetadata.
Из-за этого обращаться к PhotoMetadata со стороны Photo неудобно.
Чтобы это исправить, нужно добавить обратную связь и сделать связь между PhotoMetadata и Photo двунаправленной.
Изменим наши сущности:

```typescript
import {
    Entity,
    Column,
    PrimaryGeneratedColumn,
    OneToOne,
    JoinColumn,
} from "typeorm"
import { Photo } from "./Photo"

@Entity()
export class PhotoMetadata {
    /* ... остальные колонки */

    @OneToOne(() => Photo, (photo) => photo.metadata)
    @JoinColumn()
    photo: Photo
}
```

```typescript
import { Entity, Column, PrimaryGeneratedColumn, OneToOne } from "typeorm"
import { PhotoMetadata } from "./PhotoMetadata"

@Entity()
export class Photo {
    /* ... остальные колонки */

    @OneToOne(() => PhotoMetadata, (photoMetadata) => photoMetadata.photo)
    metadata: PhotoMetadata
}
```

`photo => photo.metadata` — это функция, возвращающая имя обратной стороны связи.
Так мы показываем, что свойство metadata класса Photo — это место, где в классе Photo хранится PhotoMetadata.
Вместо функции, возвращающей свойство photo, в декоратор `@OneToOne` можно было бы передать строку, например `"metadata"`.
Но мы выбрали подход с функцией, чтобы упростить последующий рефакторинг.

Обратите внимание: декоратор `@JoinColumn` следует использовать только на одной стороне связи.
Та сторона, на которую вы поставите этот декоратор, и будет владеющей стороной связи.
Владеющая сторона связи содержит в базе данных колонку с внешним ключом.

### Связи в ESM-проектах

Если в вашем TypeScript-проекте используется ESM, свойства связей следует оборачивать в тип-обёртку `Relation`, чтобы избежать проблем с циклическими зависимостями.
Изменим наши сущности:

```typescript
import {
    Entity,
    Column,
    PrimaryGeneratedColumn,
    OneToOne,
    JoinColumn,
    Relation,
} from "typeorm"
import { Photo } from "./Photo"

@Entity()
export class PhotoMetadata {
    /* ... остальные колонки */

    @OneToOne(() => Photo, (photo) => photo.metadata)
    @JoinColumn()
    photo: Relation<Photo>
}
```

```typescript
import {
    Entity,
    Column,
    PrimaryGeneratedColumn,
    OneToOne,
    Relation,
} from "typeorm"
import { PhotoMetadata } from "./PhotoMetadata"

@Entity()
export class Photo {
    /* ... остальные колонки */

    @OneToOne(() => PhotoMetadata, (photoMetadata) => photoMetadata.photo)
    metadata: Relation<PhotoMetadata>
}
```

### Загрузка объектов вместе с их связями

Теперь загрузим нашу фотографию и её метаданные одним запросом.
Сделать это можно двумя способами — через методы `find*` или через возможности `QueryBuilder`.
Начнём с методов `find*`.
Методы `find*` позволяют передать объект, соответствующий интерфейсу `FindOneOptions` / `FindManyOptions`.

```typescript
import { Photo } from "./entity/Photo"
import { PhotoMetadata } from "./entity/PhotoMetadata"
import { AppDataSource } from "./index"

const photoRepository = AppDataSource.getRepository(Photo)
const photos = await photoRepository.find({
    relations: {
        metadata: true,
    },
})
```

Здесь photos будет массивом фотографий из базы данных, и каждая фотография будет содержать свои метаданные.
Подробнее об опциях поиска — в [этом разделе документации](<07. Working with Entity Manager/03. Find Options.md>).

Опции поиска хороши и предельно просты, но если вам нужен более сложный запрос, используйте `QueryBuilder`.
`QueryBuilder` позволяет элегантно строить более сложные запросы:

```typescript
import { Photo } from "./entity/Photo"
import { PhotoMetadata } from "./entity/PhotoMetadata"
import { AppDataSource } from "./index"

const photos = await AppDataSource.getRepository(Photo)
    .createQueryBuilder("photo")
    .innerJoinAndSelect("photo.metadata", "metadata")
    .getMany()
```

`QueryBuilder` позволяет создавать и выполнять SQL-запросы практически любой сложности.
Работая с `QueryBuilder`, думайте так, будто составляете SQL-запрос.
В этом примере «photo» и «metadata» — это псевдонимы (алиасы), присвоенные выбираемым фотографиям.
Псевдонимы используются для обращения к колонкам и свойствам выбранных данных.

### Автоматическое сохранение связанных объектов с помощью каскадов

Мы можем задать в наших связях опции каскадов — для случаев, когда связанный объект должен сохраняться вместе с основным.
Немного изменим декоратор `@OneToOne` у нашей фотографии:

```typescript
export class Photo {
    // ... остальные колонки

    @OneToOne(() => PhotoMetadata, (metadata) => metadata.photo, {
        cascade: true,
    })
    metadata: PhotoMetadata
}
```

Использование `cascade` избавляет нас от необходимости сохранять фотографии и объекты метаданных по отдельности.
Теперь можно просто сохранить объект фотографии, и объект метаданных сохранится автоматически благодаря опциям каскада.

```typescript
import { AppDataSource } from "./index"

// создаём объект фотографии
const photo = new Photo()
photo.name = "Me and Bears"
photo.description = "I am near polar bears"
photo.filename = "photo-with-bears.jpg"
photo.isPublished = true

// создаём объект метаданных фотографии
const metadata = new PhotoMetadata()
metadata.height = 640
metadata.width = 480
metadata.compressed = true
metadata.comment = "cybershoot"
metadata.orientation = "portrait"

photo.metadata = metadata // так мы их связываем

// получаем репозиторий
const photoRepository = AppDataSource.getRepository(Photo)

// сохранение фотографии сохраняет и метаданные
await photoRepository.save(photo)

console.log("Photo is saved, photo metadata is saved too.")
```

Обратите внимание: теперь мы задаём свойство `metadata` у фотографии, а не свойство `photo` у метаданных, как раньше. Механизм `cascade` работает, только если вы связываете фотографию с её метаданными со стороны фотографии. Если задать связь со стороны метаданных, они не будут сохранены автоматически.

### Создание связи «многие к одному» / «один ко многим»

Создадим связь «многие к одному» / «один ко многим».
Допустим, у фотографии есть один автор, а у каждого автора может быть много фотографий.
Сначала создадим класс `Author`:

```typescript
import {
    Entity,
    Column,
    PrimaryGeneratedColumn,
    OneToMany,
    JoinColumn,
} from "typeorm"
import { Photo } from "./Photo"

@Entity()
export class Author {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    name: string

    @OneToMany(() => Photo, (photo) => photo.author) // примечание: свойство author в классе Photo мы создадим ниже
    photos: Photo[]
}
```

`Author` содержит обратную сторону связи.
`OneToMany` — это всегда обратная сторона связи, и она не может существовать без `ManyToOne` на другой стороне.

Теперь добавим владеющую сторону связи в сущность Photo:

```typescript
import { Entity, Column, PrimaryGeneratedColumn, ManyToOne } from "typeorm"
import { PhotoMetadata } from "./PhotoMetadata"
import { Author } from "./Author"

@Entity()
export class Photo {
    /* ... остальные колонки */

    @ManyToOne(() => Author, (author) => author.photos)
    author: Author
}
```

В связях «многие к одному» / «один ко многим» владеющая сторона — всегда «многие к одному».
Это значит, что id связанного объекта будет хранить класс, использующий `@ManyToOne`.

После запуска приложения ORM создаст таблицу `author`:

```text
+-------------+--------------+----------------------------+
|                          author                         |
+-------------+--------------+----------------------------+
| id          | int          | PRIMARY KEY AUTO_INCREMENT |
| name        | varchar(255) |                            |
+-------------+--------------+----------------------------+
```

Он также изменит таблицу `photo`, добавив новую колонку `author` и создав для неё внешний ключ:

```text
+-------------+--------------+----------------------------+
|                         photo                           |
+-------------+--------------+----------------------------+
| id          | int          | PRIMARY KEY AUTO_INCREMENT |
| name        | varchar(255) |                            |
| description | varchar(255) |                            |
| filename    | varchar(255) |                            |
| isPublished | boolean      |                            |
| authorId    | int          | FOREIGN KEY                |
+-------------+--------------+----------------------------+
```

### Создание связи «многие ко многим»

Создадим связь «многие ко многим».
Допустим, фотография может входить во многие альбомы, а каждый альбом может содержать много фотографий.
Создадим класс `Album`:

```typescript
import {
    Entity,
    PrimaryGeneratedColumn,
    Column,
    ManyToMany,
    JoinTable,
} from "typeorm"

@Entity()
export class Album {
    @PrimaryGeneratedColumn()
    id: number

    @Column()
    name: string

    @ManyToMany(() => Photo, (photo) => photo.albums)
    @JoinTable()
    photos: Photo[]
}
```

Декоратор `@JoinTable` обязателен: он указывает, что это владеющая сторона связи.

Теперь добавим обратную сторону нашей связи в класс `Photo`:

```typescript
export class Photo {
    // ... остальные колонки

    @ManyToMany(() => Album, (album) => album.photos)
    albums: Album[]
}
```

После запуска приложения ORM создаст _связующую таблицу_ **album_photos_photo_albums**:

```text
+-------------+--------------+----------------------------+
|                album_photos_photo_albums                |
+-------------+--------------+----------------------------+
| album_id    | int          | PRIMARY KEY FOREIGN KEY    |
| photo_id    | int          | PRIMARY KEY FOREIGN KEY    |
+-------------+--------------+----------------------------+
```

Не забудьте зарегистрировать класс `Album` в соединении ORM:

```typescript
const options: DataSourceOptions = {
    // ... остальные опции
    entities: [Photo, PhotoMetadata, Author, Album],
}
```

Теперь вставим альбомы и фотографии в нашу базу данных:

```typescript
import { AppDataSource } from "./index"

// создаём несколько альбомов
const album1 = new Album()
album1.name = "Bears"
await AppDataSource.manager.save(album1)

const album2 = new Album()
album2.name = "Me"
await AppDataSource.manager.save(album2)

// создаём несколько фотографий
const photo = new Photo()
photo.name = "Me and Bears"
photo.description = "I am near polar bears"
photo.filename = "photo-with-bears.jpg"
photo.views = 1
photo.isPublished = true
photo.albums = [album1, album2]
await AppDataSource.manager.save(photo)

// теперь наша фотография сохранена, и к ней прикреплены альбомы
// а теперь загрузим их:
const loadedPhoto = await AppDataSource.getRepository(Photo).findOne({
    where: {
        id: 1,
    },
    relations: {
        albums: true,
    },
})
```

`loadedPhoto` будет равна:

```typescript
{
    id: 1,
    name: "Me and Bears",
    description: "I am near polar bears",
    filename: "photo-with-bears.jpg",
    albums: [{
        id: 1,
        name: "Bears"
    }, {
        id: 2,
        name: "Me"
    }]
}
```

### Использование QueryBuilder

С помощью QueryBuilder можно строить SQL-запросы практически любой сложности. Например, так:

```typescript
const photos = await AppDataSource.getRepository(Photo)
    .createQueryBuilder("photo") // Первый аргумент — псевдоним. Псевдоним обозначает то, что вы выбираете, — фотографии. Его нужно указать обязательно.
    .innerJoinAndSelect("photo.metadata", "metadata")
    .leftJoinAndSelect("photo.albums", "album")
    .where("photo.isPublished = true")
    .andWhere("(photo.name = :photoName OR photo.name = :bearName)")
    .orderBy("photo.id", "DESC")
    .skip(5)
    .take(10)
    .setParameters({ photoName: "My", bearName: "Mishka" })
    .getMany()
```

Этот запрос выбирает все опубликованные фотографии с именами «My» или «Mishka».
Он начнёт выборку с позиции 5 (смещение постраничной выборки)
и вернёт только 10 результатов (ограничение постраничной выборки).
Результат будет отсортирован по id по убыванию.
Альбомы фотографий будут присоединены левым соединением, а их метаданные — внутренним.

Query builder вам придётся использовать в приложении часто.
Подробнее о QueryBuilder читайте [здесь](<08. Query Builder/01. Select using Query Builder.md>).
