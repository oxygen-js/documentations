# Транзакции

## Создание и использование транзакций

Транзакции создаются с помощью `DataSource` или `EntityManager`.
Примеры:

```typescript
await myDataSource.transaction(async (transactionalEntityManager) => {
    // выполняем запросы через transactionalEntityManager
})
```

или

```typescript
await myDataSource.manager.transaction(async (transactionalEntityManager) => {
    // выполняем запросы через transactionalEntityManager
})
```

Всё, что должно выполняться в транзакции, необходимо выполнять внутри колбэка:

```typescript
await myDataSource.manager.transaction(async (transactionalEntityManager) => {
    await transactionalEntityManager.save(users)
    await transactionalEntityManager.save(photos)
    // ...
})
```

Самое важное ограничение при работе в транзакции — **ВСЕГДА** использовать переданный экземпляр entity manager,
в этом примере `transactionalEntityManager`. НЕ ИСПОЛЬЗУЙТЕ ГЛОБАЛЬНЫЙ ENTITY MANAGER.
Все операции **ДОЛЖНЫ** выполняться через переданный транзакционный entity manager.

## Указание уровня изоляции

Уровень изоляции транзакции можно задать, передав его первым параметром:

```typescript
await myDataSource.manager.transaction(
    "SERIALIZABLE",
    (transactionalEntityManager) => {},
)
```

## Поддерживаемые уровни изоляции

Реализации уровней изоляции _не_ одинаковы во всех базах данных. Каждый драйвер объявляет, какие уровни он поддерживает, и TypeORM выбросит ошибку, если запросить неподдерживаемый уровень.

#### CockroachDB

- `READ COMMITTED` — требует кластерной настройки `sql.txn.read_committed_isolation.enabled`, которая в свежих версиях включена по умолчанию
- `REPEATABLE READ` — требует кластерной настройки `sql.txn.repeatable_read_isolation.enabled`, которая по умолчанию отключена (появилась в v24.3.0)
- `SERIALIZABLE` (по умолчанию)

CockroachDB отображает запросы более слабых уровней изоляции SQL на более строгие. Поведение при подмене зависит от соответствующих кластерных настроек. Например, когда вы запрашиваете `READ UNCOMMITTED`, `READ COMMITTED` или `REPEATABLE READ` через SQL (скажем, `BEGIN TRANSACTION ISOLATION LEVEL READ UNCOMMITTED`), CockroachDB принимает синтаксис, но может выполнить транзакцию на более строгом уровне:

- `READ COMMITTED` → выполняется как:
    - `SERIALIZABLE`, если `sql.txn.read_committed_isolation.enabled` равно false и `sql.txn.repeatable_read_isolation.enabled` равно false
    - `REPEATABLE READ`, если `sql.txn.repeatable_read_isolation.enabled` равно true и `sql.txn.read_committed_isolation.enabled` равно false
- `READ UNCOMMITTED` → выполняется как:
    - `SERIALIZABLE`, если `sql.txn.read_committed_isolation.enabled` равно false и `sql.txn.repeatable_read_isolation.enabled` равно false
    - `READ COMMITTED`, если `sql.txn.read_committed_isolation.enabled` равно true
    - `REPEATABLE READ`, если `sql.txn.repeatable_read_isolation.enabled` равно true и `sql.txn.read_committed_isolation.enabled` равно false
- `REPEATABLE READ` → выполняется как `SERIALIZABLE`, если `sql.txn.repeatable_read_isolation.enabled` равно false

> **Информация** Подробнее — [CockroachDB Read Committed](https://www.cockroachlabs.com/docs/stable/read-committed)
> - [CockroachDB Repeatable Read](https://www.cockroachlabs.com/docs/releases/v24.3.html#:~:text=sql.txn.repeatable_read_isolation.enabled%3A%20defaults%20to%20false.%20When%20set%20to%20true%2C%20the%20following%20statements%20configure%20transactions%20to%20run%20under%20REPEATABLE%20READ%20isolation%2C%20rather%20than%20being%20automatically%20interpreted%20as%20SERIALIZABLE)
> - [CockroachDB Transactions](https://www.cockroachlabs.com/docs/stable/transactions)

Он обеспечивает изоляцию на основе снимков (snapshot isolation) и допускает аномалии write-skew — если это важно, используйте `SELECT ... FOR UPDATE`.

#### Microsoft SQL Server

- `READ UNCOMMITTED`
- `READ COMMITTED`
- `REPEATABLE READ`
- `SERIALIZABLE`
- `SNAPSHOT`

> **Предупреждение** Опции, специфичные для драйвера SQL Server также поддерживает специфичные для драйвера настройки `options.isolationLevel` и `options.connectionIsolationLevel`, но на них распространяется [ограничение пула в вышестоящей библиотеке](<13. Drivers/02. Microsoft SQLServer.md#пул-соединений-не-сбрасывает-уровень-изоляции>). Описанная на этой странице опция верхнего уровня `isolationLevel` этому не подвержена, поскольку применяется явно для каждой транзакции.

> **Информация** Подробнее — [SQL Server SET TRANSACTION ISOLATION LEVEL](https://learn.microsoft.com/en-us/sql/t-sql/statements/set-transaction-isolation-level-transact-sql)

#### MySQL

- `READ UNCOMMITTED`
- `READ COMMITTED`
- `REPEATABLE READ`
- `SERIALIZABLE`

> **Предупреждение** Aurora MySQL через RDS Data API **Уровни изоляции не поддерживаются драйвером `aurora-mysql`.** Запрос любого уровня приводит к ошибке валидации.
>
> Это ограничение транспорта, а не движка. Сам движок Aurora MySQL [полностью поддерживает](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/AuroraMySQL.Reference.IsolationLevels.html) стандартный набор уровней, но драйвер `aurora-mysql` общается с кластером через не хранящий состояние [RDS Data API](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/data-api.html), а не через постоянное соединение по протоколу MySQL, и Data API не даёт способа связать уровень изоляции с транзакцией:
>
> - [`BeginTransaction`](https://docs.aws.amazon.com/rdsdataservice/latest/APIReference/API_BeginTransaction.html) принимает только `resourceArn`, `secretArn`, `database` и `schema` — параметра изоляции нет.
> - Data API управляет пулом бэкенд-соединений непрозрачно. Запрос `SET TRANSACTION ISOLATION LEVEL ...`, отправленный отдельным `ExecuteStatement` перед `BeginTransaction`, не имеет гарантированной привязки к той бэкенд-сессии, в которой будет выполняться транзакция, поэтому настройка молча теряется.
> - [Многооператорный SQL не поддерживается](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/data-api.troubleshooting.html), так что отправить `SET TRANSACTION ...; START TRANSACTION;` одним вызовом тоже нельзя.
> - MySQL отклоняет `SET TRANSACTION ISOLATION LEVEL` внутри уже начатой транзакции с [ошибкой 1568](https://dev.mysql.com/doc/refman/8.0/en/set-transaction.html), поэтому подход, применяемый для Aurora PostgreSQL (отправка `SET` первой инструкцией внутри начатой транзакции), в MySQL недоступен.

> **Совет** Aurora MySQL через драйвер `mysql` Если для кластера Aurora MySQL нужны уровни изоляции на уровне отдельных транзакций, используйте вместо `aurora-mysql` стандартный драйвер `mysql`, указывающий на writer-эндпоинт кластера. Этот путь работает через обычное соединение по протоколу MySQL (посредством `mysql2`) и поддерживает весь перечисленный выше набор уровней изоляции.

> **Информация** Подробнее — [MySQL SET TRANSACTION](https://dev.mysql.com/doc/refman/8.0/en/set-transaction.html)

## Уровень изоляции по умолчанию

Уровень изоляции по умолчанию для всех транзакций настраивается через опцию `isolationLevel` в параметрах DataSource:

```typescript
const dataSource = new DataSource({
    type: "postgres",
    isolationLevel: "SERIALIZABLE",
    // ...
})
```

Если он задан, все транзакции, запущенные без явного указания уровня изоляции, будут использовать это значение. Уровень изоляции, явно переданный в `transaction()` или `startTransaction()`, переопределит значение по умолчанию.

## Использование `QueryRunner` для создания и управления состоянием одного соединения с базой данных

`QueryRunner` предоставляет одно соединение с базой данных.
Транзакции организуются с помощью query runner'ов.
Одна транзакция может быть установлена только на одном query runner.
Вы можете вручную создать экземпляр query runner и использовать его для ручного управления состоянием транзакции.
Пример:

```typescript
// создаём новый query runner
const queryRunner = dataSource.createQueryRunner()

// устанавливаем реальное соединение с базой данных через наш новый query runner
await queryRunner.connect()

// теперь мы можем выполнять на query runner любые запросы, например:
await queryRunner.query("SELECT * FROM users")

// также нам доступен entity manager, работающий с соединением, созданным query runner'ом:
const users = await queryRunner.manager.find(User)

// а теперь откроем новую транзакцию:
await queryRunner.startTransaction()

try {
    // выполняем в этой транзакции какие-то операции:
    await queryRunner.manager.save(user1)
    await queryRunner.manager.save(user2)
    await queryRunner.manager.save(photos)

    // фиксируем транзакцию:
    await queryRunner.commitTransaction()
} catch (err) {
    // раз возникли ошибки, откатываем сделанные изменения
    await queryRunner.rollbackTransaction()
} finally {
    // созданный вручную query runner необходимо освободить:
    await queryRunner.release()
}
```

Управлять транзакциями в `QueryRunner` можно тремя методами:

- `startTransaction` — начинает новую транзакцию внутри экземпляра query runner.
- `commitTransaction` — фиксирует все изменения, сделанные через экземпляр query runner.
- `rollbackTransaction` — откатывает все изменения, сделанные через экземпляр query runner.
