# Введение в асинхронный JavaScript

В этой статье мы объясним, что такое асинхронное программирование, зачем оно нужно, и вкратце обсудим некоторые способы, которыми асинхронные функции исторически реализовывались в JavaScript.

| | |
|---|---|
| **Предварительные требования:** | Уверенное понимание [основ JavaScript](https://developer.mozilla.org/en-US/docs/Learn_web_development/Core/Scripting). |
| **Результаты обучения:** | <ul><li>Получить представление о том, что такое асинхронный JavaScript, чем он отличается от синхронного и зачем он нужен.</li><li>Что такое синхронное программирование и почему оно порой оказывается проблемным.</li><li>Как асинхронное программирование стремится решить эти проблемы.</li><li>Обработчики событий и функции-колбэки, а также их связь с асинхронным программированием.</li></ul> |

Асинхронное программирование — это приём, позволяющий вашей программе запустить потенциально долго выполняющуюся задачу и при этом сохранять способность реагировать на другие события, пока эта задача выполняется, вместо того чтобы ждать её завершения. Когда задача завершится, программа получит результат.

Многие функции, предоставляемые браузерами, — особенно самые интересные — могут выполняться довольно долго и поэтому являются асинхронными. Например:

- HTTP-запросы с помощью [`fetch()`](https://developer.mozilla.org/en-US/docs/Web/API/Window/fetch)
- Доступ к камере или микрофону пользователя с помощью [`getUserMedia()`](https://developer.mozilla.org/en-US/docs/Web/API/MediaDevices/getUserMedia)
- Запрос к пользователю выбрать файлы с помощью [`showOpenFilePicker()`](https://developer.mozilla.org/en-US/docs/Web/API/Window/showOpenFilePicker)

Так что, даже если вам не так уж часто придётся _реализовывать_ собственные асинхронные функции, вам почти наверняка понадобится уметь правильно их _использовать_.

В этой статье мы начнём с рассмотрения проблемы долго выполняющихся синхронных функций, из-за которой асинхронное программирование становится необходимостью.

## Синхронное программирование

Рассмотрим следующий код:

```js
const name = "Miriam";
const greeting = `Hello, my name is ${name}!`;
console.log(greeting);
// "Hello, my name is Miriam!"
```

Этот код:

1. Объявляет строку `name`.
2. Объявляет другую строку `greeting`, которая использует `name`.
3. Выводит приветствие в консоль JavaScript.

Здесь стоит отметить, что браузер, по сути, проходит по программе строка за строкой, в том порядке, в котором мы её написали. В каждой точке браузер ждёт, пока строка завершит свою работу, прежде чем перейти к следующей. Он вынужден так делать, потому что каждая строка зависит от работы, выполненной в предыдущих строках.

Это делает данную программу **синхронной**. Она осталась бы синхронной, даже если бы мы вызвали отдельную функцию, вот так:

```js
function makeGreeting(name) {
  return `Hello, my name is ${name}!`;
}

const name = "Miriam";
const greeting = makeGreeting(name);
console.log(greeting);
// "Hello, my name is Miriam!"
```

Здесь `makeGreeting()` — **синхронная функция**, потому что вызывающий код вынужден ждать, пока функция завершит свою работу и вернёт значение, прежде чем продолжить.

## Долго выполняющаяся синхронная функция

А что, если синхронная функция выполняется долго?

Приведённая ниже программа использует очень неэффективный алгоритм для генерации нескольких больших простых чисел, когда пользователь нажимает кнопку «Generate primes» («Сгенерировать простые числа»). Чем больше простых чисел укажет пользователь, тем дольше будет длиться операция.

```html
<label for="quota">Number of primes:</label>
<input type="text" id="quota" name="quota" value="1000000" />

<button id="generate">Generate primes</button>
<button id="reload">Reload</button>

<div id="output"></div>
```

```js
const MAX_PRIME = 1000000;

function isPrime(n) {
  for (let i = 2; i <= Math.sqrt(n); i++) {
    if (n % i === 0) {
      return false;
    }
  }
  return n > 1;
}

const random = (max) => Math.floor(Math.random() * max);

function generatePrimes(quota) {
  const primes = [];
  while (primes.length < quota) {
    const candidate = random(MAX_PRIME);
    if (isPrime(candidate)) {
      primes.push(candidate);
    }
  }
  return primes;
}

const quota = document.querySelector("#quota");
const output = document.querySelector("#output");

document.querySelector("#generate").addEventListener("click", () => {
  const primes = generatePrimes(quota.value);
  output.textContent = `Finished generating ${quota.value} primes!`;
});

document.querySelector("#reload").addEventListener("click", () => {
  document.location.reload();
});
```

> _В оригинале статьи здесь встроен интерактивный пример: соберите приведённый выше код или запустите его в онлайн-редакторе._

Попробуйте нажать «Generate primes». В зависимости от быстродействия вашего компьютера пройдёт, вероятно, несколько секунд, прежде чем программа покажет сообщение «Finished!» («Готово!»).

## В чём беда долго выполняющихся синхронных функций

Следующий пример точно такой же, как предыдущий, только мы добавили текстовое поле для ввода. На этот раз нажмите «Generate primes» и сразу же попробуйте что-нибудь напечатать в текстовом поле.

Вы обнаружите, что, пока выполняется наша функция `generatePrimes()`, программа полностью неотзывчива: вы не можете ничего напечатать, ни на что нажать и вообще ничего сделать.

```html hidden
<label for="quota">Number of primes:</label>
<input type="text" id="quota" name="quota" value="1000000" />

<button id="generate">Generate primes</button>
<button id="reload">Reload</button>

<textarea id="user-input" rows="5" cols="62">
Try typing in here immediately after pressing "Generate primes"
</textarea>

<div id="output"></div>
```

```css hidden
textarea {
  display: block;
  margin: 1rem 0;
}
```

```js hidden
const MAX_PRIME = 1000000;

function isPrime(n) {
  for (let i = 2; i <= Math.sqrt(n); i++) {
    if (n % i === 0) {
      return false;
    }
  }
  return n > 1;
}

const random = (max) => Math.floor(Math.random() * max);

function generatePrimes(quota) {
  const primes = [];
  while (primes.length < quota) {
    const candidate = random(MAX_PRIME);
    if (isPrime(candidate)) {
      primes.push(candidate);
    }
  }
  return primes;
}

const quota = document.querySelector("#quota");
const output = document.querySelector("#output");

document.querySelector("#generate").addEventListener("click", () => {
  const primes = generatePrimes(quota.value);
  output.textContent = `Finished generating ${quota.value} primes!`;
});

document.querySelector("#reload").addEventListener("click", () => {
  document.location.reload();
});
```

> _В оригинале статьи здесь встроен интерактивный пример с текстовым полем для проверки отзывчивости страницы._

Причина в том, что эта JavaScript-программа _однопоточная_. Поток (thread) — это последовательность инструкций, которым следует программа. Поскольку программа состоит из одного потока, она может делать только одну вещь за раз: то есть, если она ждёт возврата из нашего долго выполняющегося синхронного вызова, она не может делать ничего другого.

Нам нужен способ, чтобы наша программа могла:

1. Запустить долго выполняющуюся операцию, вызвав функцию.
2. Сделать так, чтобы эта функция запустила операцию и немедленно вернула управление, чтобы наша программа могла и дальше реагировать на другие события.
3. Сделать так, чтобы функция выполняла операцию способом, который не блокирует основной поток, — например, запустив новый поток.
4. Уведомить нас о результате операции, когда та в конце концов завершится.

Именно это и позволяют делать асинхронные функции. Остальная часть этого модуля объясняет, как они реализованы в JavaScript.

## Обработчики событий

Только что приведённое описание асинхронных функций может напомнить вам обработчики событий — и если так, то вы правы. Обработчики событий, по сути, представляют собой разновидность асинхронного программирования: вы предоставляете функцию (обработчик событий), которая будет вызвана не сразу, а всякий раз, когда произойдёт событие. Если «событие» — это «асинхронная операция завершилась», то это событие может использоваться, чтобы уведомить вызывающий код о результате вызова асинхронной функции.

Некоторые ранние асинхронные API использовали события именно таким образом. API [`XMLHttpRequest`](https://developer.mozilla.org/en-US/docs/Web/API/XMLHttpRequest) позволяет делать HTTP-запросы к удалённому серверу с помощью JavaScript. Поскольку это может занять много времени, это асинхронный API, и вы получаете уведомления о ходе выполнения и об окончательном завершении запроса, прикрепляя обработчики событий (слушатели) к объекту `XMLHttpRequest`.

Следующий пример показывает это в действии. Нажмите «Click to start request» («Нажмите, чтобы начать запрос»), чтобы отправить запрос. Мы создаём новый [`XMLHttpRequest`](https://developer.mozilla.org/en-US/docs/Web/API/XMLHttpRequest) и слушаем его событие [`loadend`](https://developer.mozilla.org/en-US/docs/Web/API/XMLHttpRequestEventTarget/loadend_event). Обработчик выводит сообщение «Finished!» вместе с кодом статуса.

После добавления обработчика событий мы отправляем запрос. Обратите внимание, что после этого мы можем вывести «Started XHR request»: то есть наша программа может продолжать работу, пока запрос выполняется, а наш обработчик событий будет вызван, когда запрос завершится.

```html
<button id="xhr">Click to start request</button>
<button id="reload">Reload</button>

<pre class="event-log"></pre>
```

```css hidden
pre {
  display: block;
  margin: 1rem 0;
}
```

```js
const log = document.querySelector(".event-log");

document.querySelector("#xhr").addEventListener("click", () => {
  log.textContent = "";

  const xhr = new XMLHttpRequest();

  xhr.addEventListener("loadend", () => {
    log.textContent = `${log.textContent}Finished with status: ${xhr.status}`;
  });

  xhr.open(
    "GET",
    "https://raw.githubusercontent.com/mdn/content/main/files/en-us/_wikihistory.json",
  );
  xhr.send();
  log.textContent = `${log.textContent}Started XHR request\n`;
});

document.querySelector("#reload").addEventListener("click", () => {
  log.textContent = "";
  document.location.reload();
});
```

> _В оригинале статьи здесь встроен интерактивный пример._

Это [обработчик событий](https://developer.mozilla.org/en-US/docs/Learn_web_development/Core/Scripting/Events) — точно такой же, как обработчики пользовательских действий вроде нажатия кнопки. Однако на этот раз событием является изменение состояния объекта.

## Колбэки

Обработчик событий — это частный случай колбэка. Колбэк — это просто функция, которая передаётся в другую функцию с расчётом на то, что она будет вызвана в подходящий момент. Как мы только что видели, раньше колбэки были основным способом реализации асинхронных функций в JavaScript.

Однако код на основе колбэков может стать трудным для понимания, когда сам колбэк должен вызывать функции, принимающие колбэк. Это распространённая ситуация, если вам нужно выполнить операцию, которая разбивается на последовательность асинхронных функций. Например, рассмотрим следующее:

```js
function doStep1(init) {
  return init + 1;
}

function doStep2(init) {
  return init + 2;
}

function doStep3(init) {
  return init + 3;
}

function doOperation() {
  let result = 0;
  result = doStep1(result);
  result = doStep2(result);
  result = doStep3(result);
  console.log(`result: ${result}`);
}

doOperation();
```

Здесь у нас есть одна операция, разбитая на три шага, где каждый шаг зависит от предыдущего. В нашем примере первый шаг прибавляет к входному значению 1, второй — 2, а третий — 3. Начав с входного значения 0, мы получаем в итоге 6 (0 + 1 + 2 + 3). Как синхронная программа это выглядит очень просто. Но что, если бы мы реализовали эти шаги с помощью колбэков?

```js
function doStep1(init, callback) {
  const result = init + 1;
  callback(result);
}

function doStep2(init, callback) {
  const result = init + 2;
  callback(result);
}

function doStep3(init, callback) {
  const result = init + 3;
  callback(result);
}

function doOperation() {
  doStep1(0, (result1) => {
    doStep2(result1, (result2) => {
      doStep3(result2, (result3) => {
        console.log(`result: ${result3}`);
      });
    });
  });
}

doOperation();
```

Поскольку нам приходится вызывать колбэки внутри колбэков, мы получаем глубоко вложенную функцию `doOperation()`, которую гораздо труднее читать и отлаживать. Это иногда называют «адом колбэков» (callback hell) или «пирамидой судьбы» (pyramid of doom) — потому что отступы образуют фигуру, похожую на лежащую на боку пирамиду.

Когда мы так вкладываем колбэки, обработка ошибок тоже может стать очень сложной: часто приходится обрабатывать ошибки на каждом уровне «пирамиды», вместо того чтобы обрабатывать их лишь однажды на верхнем уровне.

По этим причинам большинство современных асинхронных API не используют колбэки. Вместо этого фундаментом асинхронного программирования в JavaScript является [`Promise`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise) (промис), и именно ему посвящена следующая статья.
