# Как использовать промисы

**Промисы (promises)** — это фундамент асинхронного программирования в современном JavaScript. Промис — это объект, возвращаемый асинхронной функцией, который представляет текущее состояние операции. В момент, когда промис возвращается вызывающему коду, операция часто ещё не завершена, но объект промиса предоставляет методы для обработки её будущего успеха или неудачи.

В [предыдущей статье](<01. Introducing asynchronous JavaScript.md>) мы говорили об использовании колбэков для реализации асинхронных функций. При таком подходе вы вызываете асинхронную функцию, передавая ей свою функцию-колбэк. Функция немедленно возвращает управление и вызывает ваш колбэк, когда операция завершится.

В случае API на основе промисов асинхронная функция запускает операцию и возвращает объект [`Promise`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise). Затем вы можете прикрепить обработчики к этому объекту промиса, и они будут выполнены, когда операция завершится успехом или неудачей.

## Использование API fetch()

> [!NOTE]
> В этой статье мы будем изучать промисы, копируя примеры кода со страницы в консоль JavaScript вашего браузера. Чтобы это настроить:
>
> 1. откройте вкладку браузера и перейдите на <https://example.org>;
> 2. на этой вкладке откройте консоль JavaScript в [инструментах разработчика вашего браузера](https://developer.mozilla.org/en-US/docs/Learn_web_development/Howto/Tools_and_setup/What_are_browser_developer_tools);
> 3. когда мы показываем пример, скопируйте его в консоль. Вам придётся перезагружать страницу каждый раз перед вводом нового примера, иначе консоль будет жаловаться, что вы повторно объявили `fetchPromise`.

В этом примере мы загрузим JSON-файл с <https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/products.json> и выведем в лог некоторую информацию о нём.

Для этого мы сделаем **HTTP-запрос** к серверу. В HTTP-запросе мы отправляем удалённому серверу сообщение с запросом, а он присылает нам обратно ответ. В данном случае мы отправим запрос, чтобы получить JSON-файл с сервера. Помните, как в прошлой статье мы делали HTTP-запросы с помощью API [`XMLHttpRequest`](https://developer.mozilla.org/en-US/docs/Web/API/XMLHttpRequest)? Так вот, в этой статье мы воспользуемся API [`fetch()`](https://developer.mozilla.org/en-US/docs/Web/API/Window/fetch) — современной заменой `XMLHttpRequest` на основе промисов.

Скопируйте это в консоль JavaScript вашего браузера:

```js
const fetchPromise = fetch(
  "https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/products.json",
);

console.log(fetchPromise);

fetchPromise.then((response) => {
  console.log(`Received response: ${response.status}`);
});

console.log("Started request…");
```

Здесь мы:

1. вызываем API `fetch()` и присваиваем возвращаемое значение переменной `fetchPromise`;
2. сразу же после этого выводим в лог переменную `fetchPromise`. Это должно вывести что-то вроде: `Promise { <state>: "pending" }`, что говорит нам о том, что у нас есть объект `Promise` и у него есть свойство `state` со значением `"pending"`. Состояние `"pending"` означает, что операция fetch всё ещё выполняется;
3. передаём функцию-обработчик в метод **`then()`** промиса. Когда (и если) операция fetch завершится успешно, промис вызовет наш обработчик, передав в него объект [`Response`](https://developer.mozilla.org/en-US/docs/Web/API/Response), который содержит ответ сервера;
4. выводим в лог сообщение о том, что мы начали запрос.

Полный вывод должен быть примерно таким:

```plain
Promise { <state>: "pending" }
Started request…
Received response: 200
```

Обратите внимание, что `Started request…` выводится _до_ того, как мы получаем ответ. В отличие от синхронной функции, `fetch()` возвращает управление, пока запрос ещё выполняется, что позволяет нашей программе оставаться отзывчивой. Ответ показывает [код статуса](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Status) `200` (OK), то есть наш запрос выполнен успешно.

Наверное, это очень похоже на пример из прошлой статьи, где мы добавляли обработчики событий к объекту [`XMLHttpRequest`](https://developer.mozilla.org/en-US/docs/Web/API/XMLHttpRequest). Только вместо этого мы передаём обработчик в метод `then()` возвращённого промиса.

## Цепочки промисов

С API `fetch()`, получив объект `Response`, нужно вызвать ещё одну функцию, чтобы получить данные ответа. В данном случае мы хотим получить данные ответа в виде JSON, поэтому вызовем метод [`json()`](https://developer.mozilla.org/en-US/docs/Web/API/Response/json) объекта `Response`. Оказывается, `json()` тоже асинхронный. Так что это случай, когда нам приходится вызывать две асинхронные функции подряд.

Попробуйте это:

```js
const fetchPromise = fetch(
  "https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/products.json",
);

fetchPromise.then((response) => {
  const jsonPromise = response.json();
  jsonPromise.then((data) => {
    console.log(data[0].name);
  });
});
```

В этом примере, как и раньше, мы добавляем обработчик `then()` к промису, возвращённому `fetch()`. Но на этот раз наш обработчик вызывает `response.json()`, а затем передаёт новый обработчик `then()` в промис, возвращённый `response.json()`.

Это должно вывести в лог «baked beans» (название первого продукта в списке из «products.json»).

Но постойте! Помните прошлую статью, где мы говорили, что при вызове колбэка внутри другого колбэка мы получаем всё более вложенные уровни кода? И что этот «ад колбэков» делает наш код трудным для понимания? Разве это не то же самое, только с вызовами `then()`?

Так и есть, конечно. Но изящная особенность промисов в том, что `then()` сам возвращает новый промис, который выполняется значением, возвращаемым функцией-колбэком (при условии, что она отработала успешно). Это означает, что мы можем (и определённо должны) переписать приведённый выше код так:

```js
const fetchPromise = fetch(
  "https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/products.json",
);

fetchPromise
  .then((response) => response.json())
  .then((data) => {
    console.log(data[0].name);
  });
```

Вместо того чтобы вызывать второй `then()` внутри обработчика первого `then()`, мы можем _вернуть_ промис, возвращаемый `json()`, и вызвать второй `then()` уже на этом возвращённом значении. Это называется **цепочкой промисов (promise chaining)** и означает, что мы можем избежать всё возрастающих уровней отступов, когда нам нужно выполнять последовательные вызовы асинхронных функций.

Прежде чем перейти к следующему шагу, добавим ещё одну деталь. Нам нужно проверить, что сервер принял запрос и смог его обработать, прежде чем пытаться прочитать ответ. Сделаем это, проверив код статуса в ответе и выбросив ошибку, если он не был «OK»:

```js
const fetchPromise = fetch(
  "https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/products.json",
);

fetchPromise
  .then((response) => {
    if (!response.ok) {
      throw new Error(`HTTP error: ${response.status}`);
    }
    return response.json();
  })
  .then((data) => {
    console.log(data[0].name);
  });
```

## Перехват ошибок

Это подводит нас к последней части: как обрабатывать ошибки? API `fetch()` может выбросить ошибку по множеству причин (например, из-за отсутствия сетевого соединения или из-за того, что URL был как-то некорректно сформирован), и мы сами выбрасываем ошибку, если сервер вернул ошибку.

В прошлой статье мы видели, что обработка ошибок может стать очень трудной при вложенных колбэках, заставляя нас обрабатывать ошибки на каждом уровне вложенности.

Для поддержки обработки ошибок объекты `Promise` предоставляют метод [`catch()`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise/catch). Он очень похож на `then()`: вы вызываете его и передаёте функцию-обработчик. Однако, в то время как обработчик, переданный в `then()`, вызывается при _успехе_ асинхронной операции, обработчик, переданный в `catch()`, вызывается при её _неудаче_.

Если добавить `catch()` в конец цепочки промисов, то он будет вызван при неудаче любого из вызовов асинхронных функций. Таким образом, вы можете реализовать операцию как несколько последовательных вызовов асинхронных функций и иметь единое место для обработки всех ошибок.

Попробуйте эту версию нашего кода с `fetch()`. Мы добавили обработчик ошибок с помощью `catch()`, а также изменили URL так, чтобы запрос завершился неудачей.

```js
const fetchPromise = fetch(
  "bad-scheme://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/products.json",
);

fetchPromise
  .then((response) => {
    if (!response.ok) {
      throw new Error(`HTTP error: ${response.status}`);
    }
    return response.json();
  })
  .then((data) => {
    console.log(data[0].name);
  })
  .catch((error) => {
    console.error(`Could not get products: ${error}`);
  });
```

Попробуйте запустить эту версию: вы должны увидеть ошибку, выведенную нашим обработчиком `catch()`.

## Терминология промисов

С промисами связана довольно специфическая терминология, в которой стоит разобраться.

Во-первых, промис может находиться в одном из трёх состояний:

- **pending (ожидание)**: начальное состояние. Операция ещё не завершилась (ни успехом, ни неудачей).
- **fulfilled (выполнен)**: операция завершилась успешно. Это момент, когда вызывается обработчик `.then()` промиса.
- **rejected (отклонён)**: операция завершилась неудачей. Это момент, когда вызывается обработчик `.catch()` промиса.

Заметьте, что́ именно означает здесь «успех» или «неудача», зависит от конкретного API. Например, `fetch()` отклоняет возвращённый промис, если (среди прочих причин) сетевая ошибка помешала отправить запрос, но выполняет промис, если сервер прислал ответ, — даже если этот ответ был ошибкой вроде [404 Not Found](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Status/404).

Для описания состояния промиса мы также используем ещё несколько терминов:

- **completed (завершён)**: промис больше не находится в состоянии ожидания; он либо выполнен, либо отклонён.
- **resolved (улажен)**: промис завершён, либо он был «привязан» к следованию состоянию другого промиса. Это более продвинутая концепция, актуальная, когда один промис зависит от другого.

Статья [«Let's talk about how to talk about promises»](https://thenewtoys.dev/blog/2021/02/08/lets-talk-about-how-to-talk-about-promises/) даёт отличное объяснение деталей этой терминологии.

## Комбинирование нескольких промисов

Цепочка промисов — это то, что вам нужно, когда операция состоит из нескольких асинхронных функций и требуется, чтобы каждая из них завершилась перед запуском следующей. Но есть и другие способы комбинировать вызовы асинхронных функций, и API `Promise` предоставляет для них некоторые вспомогательные средства.

Иногда вам нужно, чтобы все промисы были выполнены, но они не зависят друг от друга. В таком случае гораздо эффективнее запустить их все вместе, а затем получить уведомление, когда все они выполнятся. Здесь вам понадобится метод [`Promise.all()`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise/all). Он принимает массив промисов и возвращает один промис.

Промис, возвращаемый `Promise.all()`:

- выполняется, когда и если выполнены _все_ промисы в массиве. В этом случае обработчик `then()` вызывается с массивом всех ответов в том же порядке, в котором промисы были переданы в `all()`;
- отклоняется, когда и если отклонён _любой_ из промисов в массиве. В этом случае обработчик `catch()` вызывается с ошибкой, выброшенной отклонённым промисом.

Например:

```js
const fetchPromise1 = fetch(
  "https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/products.json",
);
const fetchPromise2 = fetch(
  "https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/not-found",
);
const fetchPromise3 = fetch(
  "https://mdn.github.io/learning-area/javascript/oojs/json/superheroes.json",
);

Promise.all([fetchPromise1, fetchPromise2, fetchPromise3])
  .then((responses) => {
    for (const response of responses) {
      console.log(`${response.url}: ${response.status}`);
    }
  })
  .catch((error) => {
    console.error(`Failed to fetch: ${error}`);
  });
```

Здесь мы делаем три запроса `fetch()` к трём разным URL. Если все они успешны, мы выведем в лог статус ответа каждого. Если хотя бы один из них завершится неудачей, мы выведем в лог сообщение о неудаче.

С предоставленными нами URL все запросы должны быть выполнены, хотя для второго сервер вернёт `404` (Not Found) вместо `200` (OK), потому что запрошенный файл не существует. Так что вывод должен быть таким:

```plain
https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/products.json: 200
https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/not-found: 404
https://mdn.github.io/learning-area/javascript/oojs/json/superheroes.json: 200
```

Если мы попробуем тот же код с некорректно сформированным URL, вот так:

```js
const fetchPromise1 = fetch(
  "https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/products.json",
);
const fetchPromise2 = fetch(
  "https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/not-found",
);
const fetchPromise3 = fetch(
  "bad-scheme://mdn.github.io/learning-area/javascript/oojs/json/superheroes.json",
);

Promise.all([fetchPromise1, fetchPromise2, fetchPromise3])
  .then((responses) => {
    for (const response of responses) {
      console.log(`${response.url}: ${response.status}`);
    }
  })
  .catch((error) => {
    console.error(`Failed to fetch: ${error}`);
  });
```

то мы можем ожидать, что сработает обработчик `catch()`, и увидим что-то вроде:

```plain
Failed to fetch: TypeError: Failed to fetch
```

Иногда вам может понадобиться, чтобы был выполнен любой один промис из набора, и неважно, какой именно. В этом случае вам нужен [`Promise.any()`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise/any). Он похож на `Promise.all()`, за исключением того, что выполняется, как только выполнен любой из промисов в массиве, или отклоняется, если отклонены все они:

```js
const fetchPromise1 = fetch(
  "https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/products.json",
);
const fetchPromise2 = fetch(
  "https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/not-found",
);
const fetchPromise3 = fetch(
  "https://mdn.github.io/learning-area/javascript/oojs/json/superheroes.json",
);

Promise.any([fetchPromise1, fetchPromise2, fetchPromise3])
  .then((response) => {
    console.log(`${response.url}: ${response.status}`);
  })
  .catch((error) => {
    console.error(`Failed to fetch: ${error}`);
  });
```

Обратите внимание, что в этом случае мы не можем предсказать, какой запрос fetch завершится первым.

Это лишь два из дополнительных методов `Promise` для комбинирования нескольких промисов. Чтобы узнать об остальных, смотрите [справочную документацию по `Promise`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise).

## async и await

Ключевое слово [`async`](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Statements/async_function) даёт вам более простой способ работать с асинхронным кодом на основе промисов. Добавление `async` в начало функции делает её асинхронной функцией:

```js
async function myFunction() {
  // Это асинхронная функция
}
```

Внутри асинхронной функции вы можете использовать ключевое слово `await` перед вызовом функции, возвращающей промис. Это заставляет код в этой точке ждать, пока промис не будет улажен, после чего значение выполненного промиса воспринимается как возвращаемое значение, а значение отклонённого промиса — выбрасывается как ошибка.

Это позволяет писать код, который использует асинхронные функции, но выглядит как синхронный. Например, мы могли бы с его помощью переписать наш пример с fetch:

```js
async function fetchProducts() {
  try {
    // после этой строки наша функция будет ждать, пока не уладится вызов `fetch()`
    // вызов `fetch()` либо вернёт Response, либо выбросит ошибку
    const response = await fetch(
      "https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/products.json",
    );
    if (!response.ok) {
      throw new Error(`HTTP error: ${response.status}`);
    }
    // после этой строки наша функция будет ждать, пока не уладится вызов `response.json()`
    // вызов `response.json()` либо вернёт разобранный JSON-объект, либо выбросит ошибку
    const data = await response.json();
    console.log(data[0].name);
  } catch (error) {
    console.error(`Could not get products: ${error}`);
  }
}

fetchProducts();
```

Здесь мы вызываем `await fetch()`, и вместо `Promise` наш вызывающий код получает обратно полностью готовый объект `Response`, точно так же, как если бы `fetch()` был синхронной функцией!

Мы можем даже использовать блок `try...catch` для обработки ошибок — ровно так же, как если бы код был синхронным.

Учтите, впрочем, что асинхронные функции всегда возвращают промис, поэтому вы не можете сделать что-то вроде:

```js
async function fetchProducts() {
  try {
    const response = await fetch(
      "https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/products.json",
    );
    if (!response.ok) {
      throw new Error(`HTTP error: ${response.status}`);
    }
    const data = await response.json();
    return data;
  } catch (error) {
    console.error(`Could not get products: ${error}`);
  }
}

const promise = fetchProducts();
console.log(promise[0].name); // "promise" — это объект Promise, поэтому это не сработает
```

Вместо этого вам нужно было бы сделать что-то вроде:

```js
async function fetchProducts() {
  const response = await fetch(
    "https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/products.json",
  );
  if (!response.ok) {
    throw new Error(`HTTP error: ${response.status}`);
  }
  const data = await response.json();
  return data;
}

const promise = fetchProducts();
promise
  .then((data) => {
    console.log(data[0].name);
  })
  .catch((error) => {
    console.error(`Could not get products: ${error}`);
  });
```

Здесь мы вернули `try...catch` обратно в виде обработчика `catch` на возвращаемом промисе. Это означает, что нашему обработчику `then` не нужно иметь дело со случаем, когда ошибка была перехвачена внутри функции `fetchProducts`, из-за чего `data` оказалась бы `undefined`. Обрабатывайте ошибки как последний шаг вашей цепочки промисов.

Также обратите внимание, что использовать `await` можно только внутри `async`-функции, если только ваш код не находится в [модуле JavaScript](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Modules). Это значит, что в обычном скрипте так сделать нельзя:

```js
try {
  // использование await вне async-функции допускается только в модуле
  const response = await fetch(
    "https://mdn.github.io/learning-area/javascript/apis/fetching-data/can-store/products.json",
  );
  if (!response.ok) {
    throw new Error(`HTTP error: ${response.status}`);
  }
  const data = await response.json();
  console.log(data[0].name);
} catch (error) {
  console.error(`Could not get products: ${error}`);
  throw error;
}
```

Вы, вероятно, будете часто использовать `async`-функции там, где иначе применили бы цепочки промисов, — они делают работу с промисами гораздо более интуитивной.

Имейте в виду, что, как и цепочка промисов, `await` заставляет асинхронные операции выполняться последовательно, одну за другой. Это необходимо, если результат следующей операции зависит от результата предыдущей, но, если это не так, что-то вроде `Promise.all()` будет производительнее.

## Итоги

Промисы — это фундамент асинхронного программирования в современном JavaScript. Они облегчают выражение последовательностей асинхронных операций и рассуждение о них без глубоко вложенных колбэков и поддерживают стиль обработки ошибок, схожий с синхронной инструкцией `try...catch`.

Ключевые слова `async` и `await` упрощают построение операции из последовательности идущих подряд вызовов асинхронных функций, избавляя от необходимости создавать явные цепочки промисов и позволяя писать код, который выглядит совсем как синхронный.

Промисы работают в последних версиях всех современных браузеров; единственное место, где поддержка промисов будет проблемой, — это Opera Mini, а также IE11 и более ранние версии.

Мы затронули не все возможности промисов в этой статье, только самые интересные и полезные. По мере того как вы будете узнавать больше о промисах, вам будут встречаться и другие возможности и приёмы.

Многие современные веб-API основаны на промисах, включая [WebRTC](https://developer.mozilla.org/en-US/docs/Web/API/WebRTC_API), [Web Audio API](https://developer.mozilla.org/en-US/docs/Web/API/Web_Audio_API), [Media Capture and Streams API](https://developer.mozilla.org/en-US/docs/Web/API/Media_Capture_and_Streams_API) и многие другие.
