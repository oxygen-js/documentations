# Введение в RxJS

Совсем новичок в RxJS? В этой статье мы устроим ускоренный курс по всем
основным концепциям, которые вам нужно будет усвоить, чтобы начать
осваивать RxJS и продуктивно с ним работать. Держитесь крепче — начинаем!

## Что такое Observable?

Observable представляет поток, или источник данных, которые могут поступать с
течением времени. Observable можно создать почти из чего угодно, но самый
распространённый сценарий использования в RxJS — из событий. Это может быть что
угодно: движения мыши, нажатия кнопок, ввод в текстовое поле или даже смена
маршрута. Проще всего создать observable с помощью встроенных функций создания.
Например, мы можем использовать вспомогательную функцию
[`fromEvent`](<../02. Operators/03. Creation/06. fromEvent.md>), чтобы создать
observable из событий щелчка мыши:

```js
// импортируем оператор fromEvent
import { fromEvent } from 'rxjs';

// получаем ссылку на кнопку
const button = document.getElementById('myButton');

// создаём observable из щелчков по кнопке
const myObservable = fromEvent(button, 'click');
```

На этом этапе у нас есть observable, но он ничего не делает. **Это потому,
что observable являются холодными, то есть не активируют производителя (например,
не подключают слушатель событий), пока не появится...**

## Подписка

Подписки — это то, что приводит всё в движение. Можете представить это как кран:
у вас есть поток воды, готовый к использованию (observable), нужно лишь, чтобы
кто-то повернул ручку. В случае с observable эта роль принадлежит `subscriber`.

Чтобы создать подписку, вы вызываете метод `subscribe`, передавая функцию
(или объект), также известный как `observer`. Именно здесь вы можете решить, как
**реагировать** (-тивное программирование) на каждое событие. Давайте разберём,
что происходит в предыдущем сценарии при создании подписки:

```js
// импортируем оператор fromEvent
import { fromEvent } from 'rxjs';

// получаем ссылку на кнопку
const button = document.getElementById('myButton');

// создаём observable из щелчков по кнопке
const myObservable = fromEvent(button, 'click');

// пока что просто логируем событие на каждый щелчок
const subscription = myObservable.subscribe(event => console.log(event));
```

В примере выше вызов `myObservable.subscribe()`:

1. Установит слушатель событий на нашей кнопке для событий щелчка.
2. Будет вызывать функцию, которую мы передали в метод subscribe (observer), на
   каждое событие щелчка.
3. Вернёт объект подписки с методом `unsubscribe`, который содержит логику
   очистки, например удаление соответствующих слушателей событий.

Метод subscribe также принимает объект-отображение для обработки случая ошибки
или завершения. Вероятно, вы будете использовать это не так часто, как простую
передачу функции, но об этом полезно знать на случай необходимости:

```js
// вместо функции передадим объект с методами next, error и complete
const subscription = myObservable.subscribe({
  // при успешных эмиссиях
  next: event => console.log(event),
  // при ошибках
  error: error => console.log(error),
  // вызывается один раз при завершении
  complete: () => console.log('complete!')
});
```

Важно отметить, что каждая подписка создаёт новый контекст выполнения. Это
означает, что повторный вызов `subscribe` создаст новый слушатель событий:

```js
// addEventListener вызван
const subscription = myObservable.subscribe(event => console.log(event));

// addEventListener вызван снова!
const secondSubscription = myObservable.subscribe(event => console.log(event));

// очистка через unsubscribe
subscription.unsubscribe();
secondSubscription.unsubscribe();
```

По умолчанию подписка создаёт разговор один на один, односторонний, между
observable и observer. Это как если бы ваш начальник (observable) кричал
(испускал) на вас (observer) за то, что вы влили неудачный PR. Это также
известно как **одноадресная передача** (unicasting). Если вам ближе сценарий
конференции — один observable, много observers, — вы примените другой подход,
включающий **многоадресную передачу** (multicasting) с помощью `Subjects` (явно
или за кулисами). Подробнее об этом — в одной из будущих статей!

Стоит отметить, что когда мы говорим об источнике Observable, испускающем данные
подписчикам, — это модель, основанная на проталкивании (push). Источник не знает
и не заботится о том, что подписчики делают с данными, — он просто проталкивает
их дальше по цепочке.

Работа с потоком событий — это хорошо, но само по себе это не так уж полезно.
**А тем, что делает RxJS «lodash-ем для событий», являются его...**

## Операторы

Операторы предлагают способ манипулировать значениями из источника, возвращая
observable преобразованных значений. Многие операторы RxJS покажутся вам
знакомыми, если вы привыкли к методам `Array` в JavaScript. Например, если вы
хотите преобразовать испущенные значения из источника-observable, вы можете
использовать [`map`](<../02. Operators/07. Transformation/11. map.md>):

```js
import { of } from 'rxjs';
import { map } from 'rxjs/operators';
/*
 *  'of' позволяет доставлять значения по порядку
 *  В данном случае он испустит 1,2,3,4,5 по очереди.
 */
const dataSource = of(1, 2, 3, 4, 5);

// подписываемся на наш источник-observable
const subscription = dataSource
  .pipe(
    // прибавляем 1 к каждому испущенному значению
    map(value => value + 1)
  )
  // лог: 2, 3, 4, 5, 6
  .subscribe(value => console.log(value));
```

Или, если вы хотите отфильтровать определённые значения, вы можете использовать
[`filter`](<../02. Operators/06. Filtering/08. filter.md>):

```js
import { of } from 'rxjs';
import { filter } from 'rxjs/operators';

const dataSource = of(1, 2, 3, 4, 5);

// подписываемся на наш источник-observable
const subscription = dataSource
  .pipe(
    // принимаем только значения 2 или больше
    filter(value => value >= 2)
  )
  // лог: 2, 3, 4, 5
  .subscribe(value => console.log(value));
```

На практике, если у вас есть задача, которую нужно решить, то более чем вероятно,
что **для неё найдётся оператор**. И хотя огромное число операторов может пугать
в начале вашего путешествия по RxJS, вы можете сузить его до небольшой горстки (и
мы это сделаем), чтобы начать работать эффективно. Со временем вы научитесь ценить
гибкость библиотеки операторов, когда неизбежно возникнут малопонятные сценарии.

**Возможно, вы заметили в примере выше, что операторы существуют внутри...**

## Pipe

Функция `pipe` — это сборочная линия, ведущая от вашего источника данных-observable
через ваши операторы. Подобно тому как сырьё на фабрике проходит через ряд
остановок, прежде чем стать готовым продуктом, исходные данные могут проходить
через `pipe`-линию операторов, где вы можете манипулировать данными, фильтровать
и преобразовывать их под свой сценарий использования. Нередко в цепочке observable
используется 5 (или более) операторов, заключённых в функцию `pipe`.

Например, решение для автодополнения (typeahead), построенное на observable,
может использовать группу операторов, чтобы оптимизировать как процесс запроса,
так и процесс отображения:

```js
// observable значений из текстового поля, pipe связывает операторы в цепочку
inputValue
  .pipe(
    // ждём паузу в 200 мс
    debounceTime(200),
    // если значение то же самое, игнорируем
    distinctUntilChanged(),
    // если во время активного запроса приходит обновлённое значение, отменяем предыдущий запрос и 'переключаемся' на новый observable
    switchMap(searchTerm => typeaheadApi.search(searchTerm))
  )
  // создаём подписку
  .subscribe(results => {
    // обновляем dom
  });
```

**Но как узнать, какой оператор подходит для вашего сценария? Хорошая новость в
том, что...**

## Операторы можно сгруппировать в общие категории

Первая остановка при поиске нужного оператора — найти подходящую категорию. Нужно
отфильтровать данные из источника? Загляните к операторам
[`фильтрации`](<../02. Operators/06. Filtering/00. Filtering.md>). Пытаетесь выследить
баг или отладить поток данных через ваш поток observable? Есть операторы
[`утилиты`](<../02. Operators/08. Utility/00. Utility.md>), которые справятся с этой задачей.
**Категории операторов включают...**

### [Операторы создания](<../02. Operators/03. Creation/00. Creation.md>)

Эти операторы позволяют создавать observable почти из чего угодно. От общих до
специфических сценариев использования — вы вольны превратить в поток что угодно.

Например, предположим, мы создаём индикатор прогресса по мере того, как
пользователь прокручивает статью. Мы могли бы превратить событие прокрутки в
поток, воспользовавшись оператором
[`fromEvent`](<../02. Operators/03. Creation/06. fromEvent.md>):

```js
fromEvent(scrollContainerElement, 'scroll')
  .pipe(
    // стратегии очистки вроде этой мы обсудим в будущей статье
    takeUntil(userLeavesArticle)
  )
  .subscribe(event => {
    // вычисляем и обновляем DOM
  });
```

Наиболее часто используемые операторы создания —
[`of`](<../02. Operators/03. Creation/09. of.md>), [`from`](<../02. Operators/03. Creation/05. from.md>)
и [`fromEvent`](<../02. Operators/03. Creation/06. fromEvent.md>).

### [Операторы комбинирования](<../02. Operators/01. Combination/00. Combination.md>)

Операторы комбинирования позволяют объединять информацию из нескольких
observable. Порядок, время и структура испущенных значений — вот основные
различия между этими операторами.

Например, мы можем объединить обновления из нескольких источников данных, чтобы
выполнить вычисление:

```js
// дай мне последнее испущенное значение из каждого источника всякий раз, когда любой из источников испускает
combineLatest(sourceOne, sourceTwo).subscribe(
  ([latestValueFromSourceOne, latestValueFromSourceTwo]) => {
    // выполняем вычисление
  }
);
```

Наиболее часто используемые операторы комбинирования —
[`combineLatest`](<../02. Operators/01. Combination/02. combineLatest.md>),
[`concat`](<../02. Operators/01. Combination/03. concat.md>),
[`merge`](<../02. Operators/01. Combination/07. merge.md>),
[`startWith`](<../02. Operators/01. Combination/11. startWith.md>) и
[`withLatestFrom`](<../02. Operators/01. Combination/12. withLatestFrom.md>).

### [Операторы обработки ошибок](<../02. Operators/04. Error Handling/00. Error Handling.md>)

Операторы обработки ошибок предоставляют эффективные способы аккуратно
обрабатывать ошибки и выполнять повторы, если они возникнут.

Например, мы можем использовать
[`catchError`](<../02. Operators/04. Error Handling/01. catch - catchError.md>), чтобы
подстраховаться на случай неудавшихся сетевых запросов:

```js
source
  .pipe(
    mergeMap(value => {
      return makeRequest(value).pipe(
        catchError(handleErrorByReturningObservable)
      );
    })
  )
  .subscribe(value => {
    // предпринимаем действие
  });
```

Наиболее часто используемый оператор обработки ошибок —
[`catchError`](<../02. Operators/04. Error Handling/01. catch - catchError.md>).

### [Операторы фильтрации](<../02. Operators/06. Filtering/00. Filtering.md>)

Операторы фильтрации предоставляют приёмы для принятия — или отклонения — значений
из источника-observable и для борьбы с обратным давлением (backpressure), то есть
накоплением значений внутри потока.

Например, мы можем использовать оператор [`take`](<../02. Operators/06. Filtering/18. take.md>),
чтобы захватить только первые `5` испущенных значений из источника:

```js
source.pipe(take(5)).subscribe(value => {
  // предпринимаем действие
});
```

Наиболее часто используемые операторы фильтрации —
[`debounceTime`](<../02. Operators/06. Filtering/04. debounceTime.md>),
[`distinctUntilChanged`](<../02. Operators/06. Filtering/06. distinctUntilChanged.md>),
[`filter`](<../02. Operators/06. Filtering/08. filter.md>),
[`take`](<../02. Operators/06. Filtering/18. take.md>) и
[`takeUntil`](<../02. Operators/06. Filtering/20. takeUntil.md>).

### [Операторы многоадресной передачи](<../02. Operators/05. Multicasting/00. Multicasting.md>)

В RxJS observable по умолчанию холодные, или одноадресные (unicast, один источник
на подписчика). Эти операторы могут сделать observable горячим, или многоадресным
(multicast), позволяя разделять побочные эффекты между несколькими подписчиками.

Например, мы можем захотеть, чтобы поздние подписчики разделяли и получали
последнее испущенное значение из активного источника:

```js
const source = data.pipe(shareReplay());

const firstSubscriber = source.subscribe(value => {
  // выполняем некоторое действие
});

// некоторое время спустя...

// второй подписчик получает при подписке последнее испущенное значение, разделяет контекст выполнения с 'firstSubscriber'
const secondSubscriber = source.subscribe(value => {
  // выполняем некоторое действие
});
```

Наиболее часто используемый оператор многоадресной передачи —
[`shareReplay`](<../02. Operators/05. Multicasting/04. shareReplay.md>).

### [Операторы преобразования](<../02. Operators/07. Transformation/00. Transformation.md>)

Преобразование значений по мере их прохождения через цепочку операторов — частая
задача. Эти операторы предоставляют приёмы преобразования почти для любого
сценария использования, с которым вы столкнётесь.

Например, мы можем захотеть накапливать объект состояния из источника с течением
времени, подобно [Redux](https://redux.js.org/):

```js
source
  .pipe(
    scan((accumulatedState, currentState) => {
      return { ...accumulatedState, ...currentState };
    })
  )
  .subscribe();
```

Наиболее часто используемые операторы преобразования —
[`concatMap`](<../02. Operators/07. Transformation/06. concatMap.md>),
[`map`](<../02. Operators/07. Transformation/11. map.md>),
[`mergeMap`](<../02. Operators/07. Transformation/13. mergeMap - flatMap.md>),
[`scan`](<../02. Operators/07. Transformation/18. scan.md>) и
[`switchMap`](<../02. Operators/07. Transformation/19. switchMap.md>).

## Операторы имеют общие модели поведения

Хотя операторы можно сгруппировать в общие категории, операторы внутри одной
категории зачастую разделяют общее поведение. Распознавая это общее поведение, вы
можете начать выстраивать в уме
[дерево _«выбери свой оператор»_](https://rxjs-dev.firebaseapp.com/operator-decision-tree).

**Например, большое число операторов можно сгруппировать в...**

#### Операторы, которые уплощают

Или, иными словами, операторы, которые управляют подпиской на внутренний
observable, испуская его значения в единый источник-observable. Один
распространённый сценарий использования уплощающих операторов — обработка
HTTP-запросов в API на основе observable или promise, но это лишь верхушка
айсберга:

```js
fromEvent(button, 'click')
  .pipe(
    mergeMap(value => {
      // этой 'внутренней' подпиской управляет mergeMap, а значение ответа испускается observer
      return makeHttpRequest(value);
    })
  )
  .subscribe(response => {
    // делаем что-нибудь
  });
```

**Затем мы можем разделить уплощающие операторы по общим моделям поведения,
таким как...**

#### Операторы, которые `switch`

Как выключатель света, операторы на основе `switch` выключают (отписываются)
текущий observable и включают новый observable при эмиссиях из источника.
Операторы switch полезны в ситуациях, когда вы не хотите (или вам не нужно) иметь
больше одного активного observable одновременно:

```js
inputValueChanges
  // важно только последнее значение, если приходит новое значение — отменяем предыдущий запрос / observable
  .pipe(
    // делаем GET-запрос за данными
    switchMap(requestObservable)
  )
  .subscribe();
```

Операторы на основе switch включают `switchAll`,
[`switchMap`](<../02. Operators/07. Transformation/19. switchMap.md>) и
[`switchMapTo`](<../02. Operators/07. Transformation/20. switchMapTo.md>).

#### Операторы, которые `concat`

Сравнимо с очередью к банкомату: следующая транзакция не может начаться, пока не
завершится предыдущая. В терминах observable в каждый момент времени будет
происходить только одна подписка, по порядку, запускаемая завершением
предыдущей. Они полезны в ситуациях, где важен порядок выполнения:

```js
concat(
  firstObservable,
  // начнётся, когда 'firstObservable` завершится
  secondObservable,
  // начнётся, когда 'secondObservable` завершится
  thirdObservable
).subscribe(values => {
  // предпринимаем действие
});
```

Операторы на основе concat включают [`concat`](<../02. Operators/01. Combination/03. concat.md>),
[`concatAll`](<../02. Operators/01. Combination/04. concatAll.md>),
[`concatMap`](<../02. Operators/07. Transformation/06. concatMap.md>) и
[`concatMapTo`](<../02. Operators/07. Transformation/07. concatMapTo.md>).

#### Операторы, которые `merge`

Как полоса для въезда на автомагистраль, операторы на основе merge поддерживают
несколько активных observable, вливающихся в одну полосу в порядке живой
очереди. Операторы merge полезны в ситуациях, когда вы хотите запускать действие
при возникновении события из одного из многих источников:

```js
merge(firstObservable, secondObservable)
  // любые эмиссии из первого или второго observable по мере их возникновения
  .pipe(mergeMap(saveActivity))
  .subscribe();
```

Операторы на основе merge включают [`merge`](<../02. Operators/01. Combination/07. merge.md>),
[`mergeMap`](<../02. Operators/07. Transformation/13. mergeMap - flatMap.md>), `mergeMapTo` и
[`mergeAll`](<../02. Operators/01. Combination/08. mergeAll.md>).

## Другие сходства между операторами

Есть также операторы, которые преследуют схожую цель, но предлагают гибкость в
своих триггерах. Например, для отписки от observable после выполнения
определённого условия мы могли бы использовать:

1. [`take`](<../02. Operators/06. Filtering/18. take.md>), когда мы знаем, что нам когда-либо нужно только `n`
   значений.
2. [`takeLast`](<../02. Operators/06. Filtering/19. takeLast.md>), когда вам нужны только последние
   `n` значений.
3. [`takeWhile`](<../02. Operators/06. Filtering/21. takeWhile.md>), когда у нас есть предикатное
   выражение, которое можно передать.
4. [`takeUntil`](<../02. Operators/06. Filtering/20. takeUntil.md>), когда мы хотим, чтобы
   источник оставался активным лишь до тех пор, пока не испустит другой источник.

Хотя число операторов RxJS поначалу может казаться ошеломляющим, эти общие
модели поведения и паттерны способны довольно быстро навести мост при изучении
RxJS.

## Что мне это даёт?

По мере того как вы будете всё лучше осваивать программирование на основе
проталкивания (push) через Observable, вы сможете начать моделировать всё
асинхронное поведение в ваших приложениях через потоки observable. Это открывает
простые решения и гибкость для заметно сложного поведения.

Например, предположим, мы хотели бы сделать запрос, сохраняющий активность
пользователя, когда он ответил на вопрос викторины. Наша первоначальная реализация
могла бы использовать оператор
[`mergeMap`](<../02. Operators/07. Transformation/13. mergeMap - flatMap.md>), который отправляет
запрос на сохранение на каждое событие:

```js
const formEvents = fromEvent(formField, 'click');
const subscription = formEvents
  .pipe(
    map(convertToAppropriateValue),
    mergeMap(saveRequest)
  )
  .subscribe();
```

Позже выясняется, что нам нужно обеспечить порядок этих сохранений. Вооружившись
знанием о поведении операторов из вышесказанного, вместо реализации сложной
системы очередей мы можем просто заменить оператор
[`mergeMap`](<../02. Operators/07. Transformation/13. mergeMap - flatMap.md>) на
[`concatMap`](<../02. Operators/07. Transformation/06. concatMap.md>) и залить наши изменения:

```js
const formEvents = fromEvent(formField, 'click');
const subscription = formEvents
  .pipe(
    map(convertToAppropriateValue),
    // теперь следующий запрос не начнётся, пока не завершится предыдущий
    concatMap(saveRequest)
  )
  .subscribe();
```

Изменив одно слово, мы теперь ставим наши запросы событий в очередь, и это лишь
верхушка айсберга того, что возможно!
