# throwError

```typescript
throwError(errorOrErrorFactory: (() => any) | any): Observable<never>
```

**Создаёт Observable, который немедленно испускает уведомление об ошибке при подписке.**

---

💡 **Когда использовать throwError, а когда просто выбрасывать ошибку?**

В большинстве случаев внутри колбэков операторов (таких как `map`, `tap`, `mergeMap`) можно просто использовать нативный оператор `throw` из JavaScript, поскольку RxJS оборачивает их в блоки try-catch. Используйте `throwError()` именно тогда, когда нужно **вернуть Observable, который завершается с ошибкой** — в частности, в операторах вроде `switchMap`, `mergeMap` или `concatMap`, где в качестве возвращаемого значения ожидается Observable.

💡 **Рекомендуется фабричная функция**

Начиная с RxJS 7+, передавайте фабричную функцию `() => error`, а не саму ошибку напрямую. Так ошибка создаётся в момент подписки, что даёт более качественные трассировки стека: `throwError(() => new Error('message'))`.

---

## Зачем нужен throwError?

Представьте `throwError` как ваш «генератор сигнала об ошибке» — он создаёт Observable, который не делает ничего, кроме как немедленно посылает сигнал об ошибке. Это как специальная тревожная кнопка: когда вы её нажимаете, она не испускает никаких значений и не завершается нормально, а лишь запускает путь ошибки.

Вы обратитесь к `throwError`, когда работаете в конвейерах Observable, где нужно вернуть Observable, но что-то пошло не так и вы хотите распространить эту ошибку дальше по потоку. Например, при [условном вызове API](#пример-2-условная-ошибка-в-switchmap), где вы проверяете входные данные перед выполнением запроса, или при [реализации логики повторных попыток](#пример-3-использование-throwerror-со-стратегией-повторных-попыток), где нужно сигнализировать о конкретных сбоях.

Вот ключевая мысль: хотя нативный оператор `throw` из JavaScript прекрасно работает внутри операторов вроде `map` или `tap`, `throwError()` по-настоящему раскрывается там, где операторы ожидают, что вы вернёте Observable — например, в `switchMap`, `mergeMap` или внутри собственной функции создания. В таких случаях простое выбрасывание ошибки разорвёт цепочку; вместо этого нужно вернуть Observable с ошибкой. Это разница между тем, чтобы выбросить ошибку в коде, и тем, чтобы сконструировать Observable, представляющий состояние ошибки.

## Примеры

### Пример 1: Базовое испускание ошибки

[StackBlitz](https://stackblitz.com/edit/typescript-5d3stz?file=index.ts&devtoolsheight=100)

```typescript
import { throwError } from 'rxjs';

// Создаём observable, который немедленно испускает ошибку
const error$ = throwError(() => new Error('Something went wrong!'));

// Подписываемся, чтобы увидеть ошибку
error$.subscribe({
  next: val => console.log('Next:', val), // Не будет вызвано
  error: err => console.error('Error caught:', err.message),
  complete: () => console.log('Complete!') // Не будет вызвано
});

// Вывод: "Error caught: Something went wrong!"
```

### Пример 2: Условная ошибка в switchMap

[StackBlitz](https://stackblitz.com/edit/typescript-oulg71yj?file=index.ts)

```typescript
import { of, throwError } from 'rxjs';
import { mergeMap, catchError } from 'rxjs/operators';

interface User {
  id: number;
  name: string;
}

// Имитируем получение данных пользователя
function fetchUser(id: number) {
  return of({ id, name: `User ${id}` });
}

// Проверяем ID пользователя перед получением — обрабатываем ошибки для каждого элемента
of(0, 5, -1, 10)
  .pipe(
    mergeMap((id) => {
      // Создаём исходный observable на основе проверки
      const source$ =
        id <= 0
          ? throwError(() => new Error(`Invalid user ID: ${id}`))
          : fetchUser(id);

      // Обрабатываем ошибки для каждого элемента по отдельности
      return source$.pipe(
        catchError((err) => {
          console.error('Caught:', err());
          // Предоставляем запасного пользователя только для этого элемента
          return of({ id: 0, name: 'Guest User' } as User);
        })
      );
    })
  )
  .subscribe((user) => console.log('User:', user.name));

/* Вывод:
   Caught: Invalid user ID: 0
   User: Guest User
   User: User 5
   Caught: Invalid user ID: -1
   User: Guest User
   User: User 10
*/
```

### Пример 3: Использование throwError со стратегией повторных попыток

[StackBlitz](https://stackblitz.com/edit/typescript-gtwzvj3b?file=index.ts)

```typescript
import { of, throwError, timer } from 'rxjs';
import { mergeMap, retry, tap } from 'rxjs/operators';

let attemptCount = 0;

// Имитируем ненадёжный вызов API
function unreliableApiCall() {
  attemptCount++;
  console.log(`API call attempt #${attemptCount}`);
  
  // Первые 2 попытки завершаются неудачей, 3-я успешна
  return attemptCount < 3
    ? throwError(() => new Error('Network timeout'))
    : of({ data: 'Success!' });
}

// Пробуем вызвать API с логикой повторных попыток
of(null).pipe(
  mergeMap(() => unreliableApiCall()),
  retry(2) // Повторить до 2 раз при ошибке
).subscribe({
  next: result => console.log('Result:', result.data),
  error: err => console.error('Final error:', err.message)
});

/* Вывод:
   API call attempt #1
   API call attempt #2
   API call attempt #3
   Result: Success!
*/
```
