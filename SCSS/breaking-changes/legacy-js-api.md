# Ломающее изменение: устаревший JS API

Изначально Dart Sass использовал API, основанный на API Node Sass, но в Dart Sass 1.45.0 заменил его новым современным API. Устаревший (legacy) JS API теперь официально признан устаревшим и будет удалён в Dart Sass 2.0.0.

## Миграция использования

### Точки входа

В устаревшем JS API было две точки входа для компиляции Sass: `render` и `renderSync`, которые принимали объект опций, включающий либо `file` (для компиляции файла), либо `data` (для компиляции строки). В современном API их четыре: `compile` и `compileAsync` — для компиляции файла, и `compileString` и `compileStringAsync` — для компиляции строки. Эти функции принимают путь или исходную строку в качестве первого аргумента, а затем объект со всеми остальными опциями в качестве второго аргумента. В отличие от `render`, который использовал callback, `compileAsync` и `compileStringAsync` вместо этого возвращают promise.

Подробнее см. в [документации по использованию].

[документации по использованию]: https://sass-lang.com/documentation/js-api/#md:usage

### Импортеры

Импортеры в устаревшем API представляли собой единственную функцию, которая принимала URL правила зависимости и URL содержащей его таблицы стилей (а также callback `done` для асинхронных импортеров) и возвращала объект с либо путём `file` на диске, либо содержимым (`contents`) загружаемой таблицы стилей.

В современном API вместо этого используется [`Importer`] с двумя методами: `canonicalize`, который принимает URL правила и возвращает каноническую форму этого URL, и `load`, который принимает канонический URL и возвращает объект с содержимым загруженной таблицы стилей. Такое разделение гарантирует, что один и тот же модуль загружается только один раз и что относительные URL работают согласованно. У асинхронных импортеров оба этих метода возвращают promise.

Также существует специальный [`FileImporter`], который перенаправляет все загрузки на существующие файлы на диске; его следует использовать при миграции с устаревших импортеров, возвращавших `file` вместо `contents`.

[`Importer`]: https://sass-lang.com/documentation/js-api/interfaces/Importer/
[`ImporterResult`]: https://sass-lang.com/documentation/js-api/interfaces/ImporterResult/
[`FileImporter`]: https://sass-lang.com/documentation/js-api/interfaces/FileImporter/

### Пользовательские функции

В устаревшем JS API пользовательские функции принимали отдельный JS-аргумент для каждого аргумента Sass, а для асинхронных пользовательских функций дополнительно использовался callback `done`. В современном API пользовательские функции вместо этого принимают единственный JS-аргумент, содержащий список всех аргументов Sass, а асинхронные пользовательские функции возвращают promise.

Современный API также использует гораздо более надёжный класс [`Value`], который поддерживает все типы значений Sass, проверки типов, а также простой доступ к элементам карт (map) и списков.

[`Value`]: https://sass-lang.com/documentation/js-api/classes/Value/

### Сборщики

Если вы используете сборщик или другой инструмент, который вызывает API Sass, а не работает с ним напрямую, вам может потребоваться изменить конфигурацию, передаваемую этому инструменту, чтобы указать ему использовать современный API.

Webpack должен уже использовать современный API по умолчанию, но если вы получаете предупреждения, установите `api` в значение `"modern"` или `"modern-compiler"`. Подробнее см. в [документации Webpack].

Vite 6 использует современный API по умолчанию. Предыдущие версии Vite всё ещё используют устаревший API, однако начиная с Vite 5.4 вы можете переключиться, установив `api` в значение `"modern"` или `"modern-compiler"`. Подробнее см. в [документации Vite].

Для остальных инструментов уточняйте в их документации или трекере задач информацию о поддержке современного API Sass.

[документации Webpack]: https://webpack.js.org/loaders/sass-loader/#api
[документации Vite]: https://vitejs.dev/config/shared-options.html#css-preprocessoroptions

## Отключение предупреждений

Хотя устаревший JS API был помечен как устаревший ещё в Dart Sass 1.45.0, одновременно с выходом современного API, мы начали выводить предупреждения о его использовании начиная с Dart Sass 1.79.0. Если вы пока не можете перейти на современный API, но хотите на время отключить предупреждения, вы можете передать `legacy-js-api` в опции `silenceDeprecations`:

```js
const sass = require('sass');

const result = sass.renderSync({
  silenceDeprecations: ['legacy-js-api'],
  ...
});
```

Это отключит предупреждения на время, но устаревший API будет полностью удалён в Dart Sass 2.0.0, поэтому вам всё равно стоит запланировать переход на новый API в ближайшее время.
