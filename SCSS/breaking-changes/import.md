# Ломающее изменение: @import и глобальные встроенные функции

Изначально Sass использовал правила `@import` для загрузки других файлов в единое глобальное пространство имён, причём все встроенные функции также были доступны глобально. Теперь, когда система модулей (правила `@use` и `@forward`) существует уже несколько лет, мы объявляем устаревшими как правила `@import` в Sass, так и глобальные встроенные функции.

`@import` создаёт множество проблем: он требует вручную оформлять сущности Sass через пространства имён, чтобы избежать конфликтов, замедляет компиляцию при многократном импорте одного и того же файла и сильно затрудняет как людям, так и инструментам понимание того, откуда взялась та или иная переменная, миксин или функция.

Система модулей решает эти проблемы и поднимает модульность Sass до уровня лучших практик других современных языков, но мы не можем получить от неё полную пользу, пока `@import` остаётся частью языка.

Начиная с Dart Sass 1.80.0 `@import` объявлен устаревшим. Кроме того, мы также объявляем устаревшими глобальные версии встроенных функций Sass, доступных в модулях `sass:`.

## Переходный период

**Совместимость:** Dart Sass: начиная с 1.80.0 · LibSass: ✗ · Ruby Sass: ✗

Правила `@import` в Sass и вызовы глобальных встроенных функций теперь выдают предупреждения об устаревании. Хотя Dart Sass 2.0.0 выйдет уже скоро и принесёт различные небольшие ломающие изменения, мы не планируем удалять правила `@import` в Sass или глобальные встроенные функции до выхода Dart Sass 3.0.0, который появится не раньше чем через два года после Dart Sass 1.80.0.

В конечном итоге все правила `@import` будут трактоваться как [обычные CSS-импорты `@import`], вероятно, после промежуточного периода, в течение которого всё, что раньше было импортом Sass `@import`, будет вызывать ошибку.

[обычные CSS-импорты `@import`]: https://sass-lang.com/documentation/at-rules/import/#plain-css-imports

## Автоматическая миграция

Вы можете использовать [Sass migrator][] для автоматического обновления ваших таблиц стилей, чтобы они использовали систему модулей.

[Sass migrator]: https://github.com/sass/migrator#readme

```shellsession
$ npm install -g sass-migrator
$ sass-migrator module --migrate-deps your-entrypoint.scss
```

Если вы хотите отказаться от глобальных встроенных функций, но ещё не готовы полностью перевести свои правила `@import`, вы можете передать флаг `--built-in-only`, чтобы перевести только функции, оставив правила `@import` как есть.

## Рецепты миграции

### Вложенные импорты

Хотя `@import` можно использовать внутри CSS-правил, `@use` обязательно должен быть написан на верхнем уровне файла (это потому, что CSS каждого модуля, подключённого через `@use`, включается в вывод лишь один раз, поэтому не имело бы смысла разрешать его как во вложенном контексте, так и на верхнем уровне). Есть два способа перевести вложенные импорты `@import` на систему модулей:

1. Рекомендуемый способ, требующий немного больше усилий на старте, — обернуть весь CSS, генерируемый вашими вложенными модулями, в [миксины] и подключать эти миксины через `@include` во вложенном контексте. Это соответствует тому, как работает большинство других языков программирования, где каждый файл определяет функцию или класс, вызываемый файлами, которые его используют, и это чётко показывает, как именно вы ожидаете использовать этот файл. Это также упрощает добавление конфигурации, поскольку вы можете просто передавать параметры или даже [блоки `@content`] в миксин.

2. Более прямой вариант перевода — использовать [миксин `meta.load-css()`], чтобы напрямую загрузить CSS модуля там, где вы хотите его использовать. Это подходит, когда у вас нет возможности изменить загружаемый файл, чтобы обернуть его в миксин. Обратите внимание, что `meta.load-css()` полностью компилирует CSS ещё до какой-либо вложенности, поэтому любые [родительские селекторы] не «увидят» правила за пределами вызова `meta.load-css()`.

[миксины]: https://sass-lang.com/documentation/at-rules/mixin/
[блоки `@content`]: https://sass-lang.com/documentation/at-rules/mixin/#content-blocks
[миксин `meta.load-css()`]: https://sass-lang.com/documentation/modules/meta/#load-css
[родительские селекторы]: https://sass-lang.com/documentation/style-rules/parent-selector/

### Настраиваемые темы

Один из паттернов, который иногда используют с `@import`, — иметь библиотеку компонентов, состоящую из партиалов, которые используют одни и те же переменные без явной их загрузки, а затем несколько разных «тематических» точек входа, определяющих разные значения этих переменных для разных визуальных тем. Они могут либо определять переменные напрямую, либо переопределять значения по умолчанию базового партиала темы. Упрощённый пример:

**SCSS**

```scss
// components/_button.scss
button {
  color: $text-color;
  background-color: $background-color;
}
```
```scss
// _theme.scss
$text-color: black !default;
$background-color: white !default;
```
```scss
// dark.scss
$text-color: white;
$background-color: black;
@import "theme";

@import "components/button";
// More components are usually imported here.
```

**Sass**

```sass
// components/_button.scss
button
  color: $text-color
  background-color: $background-color
```
```sass
// _theme.scss
$text-color: black
$background-color: white
```
```sass
// dark.scss
$text-color: white
$background-color: black
@import "theme"

@import "components/button"
// More components are usually imported here.
```

**Результат CSS**

```css
button {
  color: white;
  background-color: black;
}
```

В системе модулей партиалы компонентов должны явно ссылаться на переменные, к которым обращаются. Но это не означает, что такое тематизирование перестаёт работать! Поскольку многократное подключение одного и того же модуля через `@use` всегда использует одну и ту же конфигурацию, если вы настроите её один раз в точке входа, все остальные подключения увидят эту же конфигурацию:

**SCSS**

```scss
// components/_button.scss
@use "../theme";

button {
  color: theme.$text-color;
  background-color: theme.$background-color;
}
```
```scss
// _theme.scss
$text-color: black !default;
$background-color: white !default;
```
```scss
// dark.scss
@use "theme" with (
  $text-color: white,
  $background-color: black,
);

@use "components/button";
// More components are usually imported here.
```

**Sass**

```sass
// components/_button.scss
@use "../theme"

button
  color: theme.$text-color
  background-color: theme.$background-color
```
```sass
// _theme.scss
$text-color: black !default
$background-color: white !default
```
```sass
// dark.scss
@use "theme" with (
  $text-color: white,
  $background-color: black,
)

@use "components/button"
// More components are usually imported here.
```

**Результат CSS**

```css
button {
  color: white;
  background-color: black;
}
```

## Могу ли я отключить предупреждения?

Sass предоставляет мощный набор опций для управления тем, какие предупреждения об устаревании вы видите и когда.

### Краткий и подробный режим

По умолчанию Sass работает в кратком режиме, в котором каждый тип предупреждения об устаревании выводится не более пяти раз, после чего дальнейшие предупреждения такого типа подавляются. Это помогает пользователям узнавать о предстоящем ломающем изменении, не заваливая консоль избыточным шумом.

Если же запустить Sass в подробном режиме, он выведет *каждое* встреченное предупреждение об устаревании. Это может быть полезно для отслеживания оставшейся работы по исправлению устареваний. Включить подробный режим можно с помощью

[флага `--verbose`] в командной строке, либо

[опции `verbose`] в JavaScript API.

[флага `--verbose`]: https://sass-lang.com/documentation/cli/dart-sass/#verbose
[опции `verbose`]: https://sass-lang.com/documentation/js-api/interfaces/Options/#verbose

> ⚠️ **Внимание**
>
> При запуске через [JS API] Sass не делится никакой информацией между компиляциями, поэтому по умолчанию он выводит по пять предупреждений для *каждой* компилируемой таблицы стилей. Однако это можно исправить, написав (или попросив автора вашего любимого фреймворкового Sass-плагина написать) [пользовательский `Logger`], который выводит только пять ошибок на каждое устаревание и может использоваться совместно в нескольких компиляциях.
>
> [JS API]: https://sass-lang.com/documentation/js-api/
> [пользовательский `Logger`]: https://sass-lang.com/documentation/js-api/interfaces/Logger-1/

### Отключение предупреждений для зависимостей

Иногда предупреждения об устаревании возникают в ваших зависимостях, и вы никак не можете на это повлиять. Вы можете отключить предупреждения об устаревании из зависимостей, при этом по-прежнему выводя их для вашего приложения, с помощью

[флага `--quiet-deps`] в командной строке, либо

[опции `quietDeps`] в JavaScript API.

[флага `--quiet-deps`]: https://sass-lang.com/documentation/cli/dart-sass/#quiet-deps
[опции `quietDeps`]: https://sass-lang.com/documentation/js-api/interfaces/Options/#quietDeps

Для целей этого флага «зависимостью» считается любая таблица стилей, загрузка которой не является просто цепочкой относительных загрузок из точки входа. Это означает всё, что загружается через путь загрузки, и большинство таблиц стилей, загружаемых через пользовательские импортеры.

### Отключение конкретных устареваний

Если вы знаете, что конкретное устаревание не создаёт для вас проблем, вы можете отключить предупреждения именно для него с помощью

[флага `--silence-deprecation`] в командной строке, либо

[опции `silenceDeprecations`] в JavaScript API.

[флага `--silence-deprecation`]: https://sass-lang.com/documentation/cli/dart-sass/#silence-deprecation
[опции `silenceDeprecations`]: https://sass-lang.com/documentation/js-api/interfaces/Options/#silenceDeprecations

Примечание: хотя устаревания для `@import` и глобальных встроенных функций выпускаются вместе и мы ожидаем, что обе возможности будут удалены одновременно (в Dart Sass 3.0.0), с точки зрения API они считаются двумя отдельными устареваниями. Если вы хотите отключить предупреждения об устаревании как для `@import`, так и для глобальных встроенных функций, вам нужно передать в `--silence-deprecation`/`silenceDeprecations` оба значения: `import` и `global-builtin`.
