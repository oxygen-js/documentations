# Ломающее изменение: имена функций

Это небольшое ломающее изменение запрещает пользовательские функции, имена которых совпадают с именами обычных CSS-функций, имеющих особый синтаксис, даже если они написаны не в нижнем регистре. Оно также убирает особый разбор вызовов функций там, где такой особый синтаксис больше не нужен.

### Имена пользовательских функций

Это изменение запретит пользователям определять функции с именами `URL`, `EXPRESSION`
или `ELEMENT`, а также с теми же именами в любой комбинации строчных и заглавных букв.

Полностью строчный вариант этих имён уже был запрещён. Однако имена функций в CSS чувствительны к регистру, поэтому для полной совместимости с CSS Sass должен применять особые правила разбора даже к вариантам с заглавными буквами. Это означает, что пользовательские функции не могут использовать эти имена, поскольку особый разбор должен происходить до того, как языку становится известно, какие пользовательские функции существуют.

> 💡 **Занятный факт**
>
> В рамках этого изменения были смягчены и некоторые другие ограничения на допустимые имена функций. Функции с префиксами в стиле вендорных (vendor-style), чьи имена заканчиваются на `-url`, `-expression`, `-and`, `-or` и `-not`, больше не будут вызывать ошибок.

### Вызовы функций

Это изменение также уберёт особый разбор для функции `expression()` и функций `progid:...()` с вендорными префиксами. Ни один браузер на самом деле никогда не поддерживал версии этих функций с вендорными префиксами, поэтому особый разбор для них не нужен. Функции `expression()` с вендорными префиксами, такие как `-foo-expression()`, будут разбираться как обычные функции Sass, а функции `progid:...()` с вендорными префиксами, такие как `-foo-progid:bar()`, будут вызывать синтаксические ошибки, поскольку это в целом не является допустимым синтаксисом CSS.

### Этап 1

**Совместимость:** Dart Sass: начиная с 1.98.0 · LibSass: ✗ · Ruby Sass: ✗


В настоящее время Dart Sass выводит предупреждение об устаревании, если вы определяете функцию `URL`, `EXPRESSION` или `ELEMENT`, а также с теми же именами в любой комбинации строчных и заглавных букв. (Полностью строчные имена были запрещены ещё до этого изменения и остаются запрещёнными.)

Вызовы функций с такими именами по-прежнему разбираются как обычные вызовы функций Sass: вызывается пользовательская функция, если она существует, либо, в противном случае, происходит откат к обычной CSS-функции с обычным разбором аргументов SassScript.

Dart Sass также выводит предупреждение об устаревании для вызовов функций `expression()` с вендорными префиксами, но только если такие функции будут вести себя иначе при разборе как обычные функции Sass. Сюда относятся вызовы, аргументы которых не являются синтаксически корректными выражениями SassScript, либо аргументы которых имели бы значение в SassScript, отличное от их значения в обычном CSS, — но не вызовы, аргументы которых обрабатываются одинаково в обоих случаях. В любом случае всегда безопасно использовать интерполяцию.

Dart Sass выводит предупреждение об устаревании для всех вызовов `progid:...()` с вендорными префиксами.

## Как отключить предупреждения?

Sass предоставляет широкий набор опций для управления тем, какие предупреждения об устаревании вы видите и когда.

### Краткий и подробный режим

По умолчанию Sass работает в кратком режиме (terse mode), в котором каждый вид предупреждения об устаревании выводится не более пяти раз, после чего дальнейшие предупреждения подавляются. Это помогает пользователям узнавать о предстоящих ломающих изменениях, не создавая при этом излишний шум в консоли.

Если же запустить Sass в подробном режиме (verbose mode), он будет выводить *каждое* встреченное предупреждение об устаревании. Это может быть полезно для отслеживания того, сколько работы осталось для устранения устареваний. Включить подробный режим можно с помощью

[флага `--verbose`] в командной строке, или

[опции `verbose`] в JavaScript API.

[флага `--verbose`]: https://sass-lang.com/documentation/cli/dart-sass/#verbose
[опции `verbose`]: https://sass-lang.com/documentation/js-api/interfaces/Options/#verbose

> ⚠️ **Внимание**
>
> При запуске из [JS API] Sass не сохраняет никакой информации между отдельными компиляциями, поэтому по умолчанию для *каждой* компилируемой таблицы стилей будет выведено до пяти предупреждений. Однако это можно исправить, написав (или попросив автора вашего любимого фреймворкового плагина Sass написать) [пользовательский `Logger`], который выводит не более пяти ошибок на каждое устаревание и может использоваться совместно для нескольких компиляций.

[JS API]: https://sass-lang.com/documentation/js-api/
[пользовательский `Logger`]: https://sass-lang.com/documentation/js-api/interfaces/Logger-1/

### Отключение предупреждений в зависимостях

Иногда в ваших зависимостях появляются предупреждения об устаревании, с которыми вы ничего не можете поделать. Вы можете отключить предупреждения об устаревании из зависимостей, продолжая при этом выводить их для вашего приложения, с помощью

[флага `--quiet-deps`] в командной строке, или

[опции `quietDeps`] в JavaScript API.

[флага `--quiet-deps`]: https://sass-lang.com/documentation/cli/dart-sass/#quiet-deps
[опции `quietDeps`]: https://sass-lang.com/documentation/js-api/interfaces/Options/#quietDeps

Для целей этого флага «зависимостью» считается любая таблица стилей, которая не является просто цепочкой относительных загрузок из входной таблицы стилей. Это означает всё, что приходит из пути загрузки, а также большинство таблиц стилей, загружаемых через пользовательские импортеры (custom importers).

### Отключение отдельных предупреждений

Если вы знаете, что конкретное устаревание не представляет для вас проблемы, вы можете отключить предупреждения именно для него с помощью

[флага `--silence-deprecation`] в командной строке, или

[опции `silenceDeprecations`] в JavaScript API.

[флага `--silence-deprecation`]: https://sass-lang.com/documentation/cli/dart-sass/#silence-deprecation
[опции `silenceDeprecations`]: https://sass-lang.com/documentation/js-api/interfaces/Options/#silenceDeprecations


### Этап 2

**Совместимость:** Dart Sass: ✗ · LibSass: ✗ · Ruby Sass: ✗


В одном из будущих релизов Dart Sass будет выдавать ошибку для пользовательских функций с именами `URL`, `EXPRESSION` или `ELEMENT`, а также с теми же именами в любой комбинации строчных и заглавных букв. Вызовы функций с такими именами будут разбираться как [специальные функции], а не как обычный SassScript.

Вызовы функций `expression()` с вендорными префиксами будут разбираться как обычный SassScript, а вызовы функций `progid:...()` с вендорными префиксами будут приводить к ошибкам.

[специальные функции]: https://sass-lang.com/documentation/syntax/special-functions
