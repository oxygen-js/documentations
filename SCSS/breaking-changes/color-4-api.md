# Ломающее изменение: JS API для цветов

Некоторые аспекты JS API для работы с цветом, спроектированные исходя из
предположения, что все цвета взаимно совместимы, больше не имеют смысла теперь,
когда Sass поддерживает все цветовые пространства CSS Color 4.

Подобно тому как некоторые аспекты [функций Sass для работы с цветом объявляются
устаревшими] в связи с добавлением поддержки [CSS Color 4], отдельные части JS
API для работы с цветами тоже объявляются устаревшими.

[функций Sass для работы с цветом объявляются устаревшими]: https://sass-lang.com/documentation/breaking-changes/color-functions
[CSS Color 4]: https://developer.mozilla.org/en-US/blog/css-color-module-level-4/

### `color.change()` теперь требует `space` для изменений между пространствами

Раньше [метод `color.change()`] просто принимал набор имён каналов из
пространств RGB, HSL или HWB. Пока эти каналы не смешивались между
пространствами (например, если одновременно менялись `red` и `hue`), Sass мог
определить, какое пространство имелось в виду.

[метод `color.change()`]: https://sass-lang.com/documentation/js-api/classes/SassColor/#change

С появлением Color 4 цветовые пространства больше нельзя однозначно определить
только по именам каналов. Во многих пространствах есть каналы `red`, `green` и
`blue` с разными диапазонами; во многих пространствах есть каналы `hue`,
которые дают совершенно разные цветовые круги. Чтобы устранить эту
неоднозначность, `color.change()` теперь принимает параметр `space`, явно
указывающий имя цветового пространства, в котором нужно выполнить
преобразование:

```js
const color = new sass.SassColor({red: 0x66, green: 0x33, blue: 0x99});
color.change({hue: 270, space: "okclh"});
```

Указывать цветовое пространство обязательно, если рассматриваемый цвет не
находится в [устаревшем цветовом пространстве] или если вы меняете канал вроде
chroma, который существует только в неустаревших цветовых пространствах. Это
всегда необязательно, если вы меняете канал, существующий в собственном
пространстве цвета, поэтому `color.change({red: 0.8})` всегда относится к
нативному каналу red любого цвета с каналами `red`, `green` и `blue`.

[устаревшем цветовом пространстве]: https://sass-lang.com/documentation/values/colors#legacy-color-spaces

Для обратной совместимости, если вы меняете устаревшие каналы для устаревшего
цвета, Sass по-прежнему будет автоматически преобразовывать цвет за вас. Однако
такое поведение объявлено устаревшим. Для надёжности вам следует _всегда_
передавать параметр `space`, если только вы не уверены, что цвет уже находится
в том цветовом пространстве, канал которого вы хотите изменить.

### Значения канала `null`

Одно из главных изменений в CSS Color 4 — новая концепция [«отсутствующих»
каналов]. Например, у `hsl(none 0% 40%)` отсутствует hue, который в большинстве
случаев трактуется как 0, но не участвует в интерполяции цвета, поэтому у
градиента с таким цветом не появится фантомного красного оттенка посередине.
При построении цветов Sass представляет отсутствующие значения как значение
`null`.

[«отсутствующих» каналов]: https://developer.mozilla.org/en-US/docs/Web/CSS/color_value#missing_color_components

До добавления поддержки CSS Color 4 типы TypeScript в JS API Sass запрещали
использование `null` везде, где это было бы уместно. Однако сам код обрабатывал
`null` так же, как `undefined`, и мы не хотим нарушать совместимость с обычным
JavaScript-кодом, который полагался на такое поведение. Пока что значение
`null` трактуется как `undefined` и выдаёт предупреждение об устаревании при
создании нового [устаревшего цвета] или при вызове `color.change()` для
устаревшего цвета. В обоих случаях, если вы явно передадите параметр `space`,
вы перейдёте на новое поведение, и `null` будет трактоваться как отсутствующий
канал.

[устаревшего цвета]: https://sass-lang.com/documentation/values/colors#legacy-color-spaces

## Переходный период

**Совместимость:** Dart Sass: начиная с 1.79.0 · LibSass: ✗ · Ruby Sass: ✗


Сначала мы будем выдавать предупреждения об устаревании при всех использованиях
этих API, которые планируется изменить. В Dart Sass 2.0.0 ломающие изменения
вступят в силу полностью, и старое поведение перестанет работать как прежде.

## Можно ли отключить предупреждения?

Sass предоставляет широкий набор параметров для управления тем, какие
предупреждения об устаревании вы видите и когда.

### Лаконичный и подробный режим

По умолчанию Sass работает в лаконичном режиме, в котором он выводит каждый тип
предупреждения об устаревании только пять раз, после чего дальнейшие
предупреждения подавляются. Это помогает пользователям узнавать о предстоящих
ломающих изменениях, не создавая при этом избыточного шума в консоли.

Если вместо этого запустить Sass в подробном режиме, он будет выводить *каждое*
встреченное предупреждение об устаревании. Это может быть полезно для
отслеживания оставшейся работы при исправлении устареваний. Включить подробный
режим можно с помощью

[параметра `verbose`] в JavaScript API.

[флага `--verbose`]: https://sass-lang.com/documentation/cli/dart-sass/#verbose
[параметра `verbose`]: https://sass-lang.com/documentation/js-api/interfaces/Options/#verbose

> ⚠️ **Внимание**
>
> При запуске через [JS API] Sass не передаёт никакой информации между
> компиляциями, поэтому по умолчанию он выводит по пять предупреждений для
> *каждой компилируемой таблицы стилей*. Однако это можно исправить, написав
> (или попросив автора вашего любимого Sass-плагина для фреймворка написать)
> [пользовательский `Logger`], который выводит только пять ошибок на каждое
> устаревание и может использоваться совместно для нескольких компиляций.

[JS API]: https://sass-lang.com/documentation/js-api/
[пользовательский `Logger`]: https://sass-lang.com/documentation/js-api/interfaces/Logger-1/

### Отключение предупреждений для зависимостей

Иногда в ваших зависимостях есть предупреждения об устаревании, с которыми вы
ничего не можете поделать. Вы можете отключить предупреждения об устаревании
для зависимостей, сохранив их вывод для вашего приложения, с помощью

[параметра `quietDeps`] в JavaScript API.

[флага `--quiet-deps`]: https://sass-lang.com/documentation/cli/dart-sass/#quiet-deps
[параметра `quietDeps`]: https://sass-lang.com/documentation/js-api/interfaces/Options/#quietDeps

Для целей этого флага «зависимостью» считается любая таблица стилей, которая не
является просто цепочкой относительных загрузок от точки входа. Это означает
всё, что загружается из пути загрузки, а также большинство таблиц стилей,
загружаемых через пользовательские загрузчики (importers).

### Отключение отдельных предупреждений об устаревании

Если вы знаете, что конкретное устаревание не представляет для вас проблемы, вы
можете отключить предупреждения для этого конкретного устаревания с помощью

[параметра `silenceDeprecations`] в JavaScript API.

[флага `--silence-deprecation`]: https://sass-lang.com/documentation/cli/dart-sass/#silence-deprecation
[параметра `silenceDeprecations`]: https://sass-lang.com/documentation/js-api/interfaces/Options/#silenceDeprecations
