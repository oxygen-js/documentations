# Environment API для сред выполнения

> **Информация: релиз-кандидат**
>
> API окружений в целом находится на стадии релиз-кандидата (release candidate). Мы будем поддерживать стабильность этих API между мажорными релизами, чтобы экосистема могла экспериментировать с ними и строить на их основе. Однако учтите, что [некоторые конкретные API](https://vite.dev/changes/#considering) всё ещё считаются экспериментальными.
>
> Мы планируем стабилизировать эти новые API (с возможными несовместимыми изменениями) в одном из будущих мажорных релизов, как только у нижестоящих проектов будет время поэкспериментировать с новыми возможностями и проверить их.
>
> Ресурсы:
>
> - [Обсуждение обратной связи](https://github.com/vitejs/vite/discussions/16358), где мы собираем отзывы о новых API.
> - [PR с Environment API](https://github.com/vitejs/vite/pull/16471), где новые API были реализованы и прошли ревью.
>
> Пожалуйста, поделитесь с нами своими отзывами.

Эта страница предназначена для поставщиков сред выполнения (runtime providers) — авторов, которые интегрируют JavaScript-среду выполнения с Vite. Среда выполнения здесь — это JavaScript-движок, в котором выполняется преобразованный код: например, Node.js, браузер, workerd от Cloudflare или поток-воркер (Worker thread). Поставщик среды выполнения упаковывает интеграцию для одной из таких сред выполнения, чтобы авторам фреймворков и конечным пользователям (разработчикам, создающим приложение) не приходилось настраивать её самостоятельно.

## Фабрики окружений

Фабрики окружений предназначены для реализации поставщиками сред выполнения, а не конечными пользователями. Фабрики окружений возвращают `EnvironmentOptions` для самого распространённого случая — использования целевой среды выполнения как в окружении разработки, так и в окружении сборки. Также можно задать опции окружения по умолчанию, чтобы пользователю не приходилось делать это самому.

```ts
function createWorkerdEnvironment(
  userConfig: EnvironmentOptions,
): EnvironmentOptions {
  return mergeConfig(
    {
      resolve: {
        conditions: [
          /*...*/
        ],
      },
      dev: {
        createEnvironment(name, config) {
          return createWorkerdDevEnvironment(name, config, {
            hot: true,
            transport: customHotChannel(),
          })
        },
      },
      build: {
        createEnvironment(name, config) {
          return createWorkerdBuildEnvironment(name, config)
        },
      },
    },
    userConfig,
  )
}
```

Тогда файл конфигурации можно написать так:

```js
import { createWorkerdEnvironment } from 'vite-environment-workerd'

export default {
  environments: {
    ssr: createWorkerdEnvironment({
      build: {
        outDir: '/dist/ssr',
      },
    }),
    rsc: createWorkerdEnvironment({
      build: {
        outDir: '/dist/rsc',
      },
    }),
  },
}
```

а фреймворки могут использовать окружение со средой выполнения workerd для выполнения SSR так:

```js
const ssrEnvironment = server.environments.ssr
```

## Создание новой фабрики окружения

Dev-сервер Vite по умолчанию предоставляет два окружения: окружение `client` и окружение `ssr`. Окружение `client` по умолчанию является браузерным, а раннер модулей реализуется за счёт импорта виртуального модуля `/@vite/client` в клиентские приложения. Окружение SSR по умолчанию работает в той же среде выполнения Node, что и сервер Vite, и позволяет использовать серверы приложений для рендеринга запросов во время разработки с полной поддержкой HMR.

Преобразованный исходный код называется модулем, а связи между модулями, обрабатываемыми в каждом окружении, хранятся в графе модулей. Преобразованный код этих модулей отправляется на выполнение в среды выполнения, связанные с каждым окружением. Когда модуль вычисляется в среде выполнения, запрашиваются импортируемые им модули, что запускает обработку соответствующего участка графа модулей.

Раннер модулей Vite (Vite Module Runner) позволяет запускать любой код, предварительно обработав его плагинами Vite. Он отличается от `server.ssrLoadModule` тем, что реализация раннера отделена от сервера. Это позволяет авторам библиотек и фреймворков реализовать собственный слой коммуникации между сервером Vite и раннером. Браузер взаимодействует со своим окружением через серверный WebSocket и с помощью HTTP-запросов. Раннер модулей для Node может напрямую вызывать функции для обработки модулей, так как работает в том же процессе. Другие окружения могут запускать модули, подключаясь к JS-среде выполнения вроде workerd или к потоку-воркеру (Worker Thread), как это делает Vitest.

Одна из целей этой возможности — предоставить настраиваемый API для обработки и запуска кода. Пользователи могут создавать новые фабрики окружений, используя предоставляемые примитивы.

```ts
import { DevEnvironment, HotChannel } from 'vite'

function createWorkerdDevEnvironment(
  name: string,
  config: ResolvedConfig,
  context: DevEnvironmentContext
) {
  const connection = /* ... */
  const transport: HotChannel = {
    on: (listener) => { connection.on('message', listener) },
    send: (data) => connection.send(data),
  }

  const workerdDevEnvironment = new DevEnvironment(name, config, {
    options: {
      resolve: { conditions: ['custom'] },
      ...context.options,
    },
    hot: true,
    transport,
  })
  return workerdDevEnvironment
}
```

По умолчанию к транспортам `HotChannel` применяются ограничения `server.fs`, то есть отдаваться могут только файлы из разрешённых директорий. Если ваш транспорт не доступен по сети (например, он взаимодействует через потоки-воркеры или внутрипроцессные вызовы), вы можете задать `skipFsCheck: true` в `HotChannel`, чтобы обойти эти ограничения.

Существует [несколько уровней коммуникации для `DevEnvironment`](https://vite.dev/guide/api-environment-frameworks#devenvironment-communication-levels). Чтобы фреймворкам было проще писать код, независимый от среды выполнения, мы рекомендуем реализовывать наиболее гибкий из возможных уровней коммуникации.

## `ModuleRunner`

Раннер модулей создаётся в целевой среде выполнения. Все API в следующем разделе импортируются из `vite/module-runner`, если не указано иное. Эта точка входа для экспорта сделана максимально лёгкой и экспортирует лишь минимум, необходимый для создания раннеров модулей.

**Сигнатура типа:**

```ts
export class ModuleRunner {
  constructor(
    public options: ModuleRunnerOptions,
    public evaluator: ModuleEvaluator = new ESModulesEvaluator(),
    private debug?: ModuleRunnerDebugger,
  ) {}
  /**
   * URL для выполнения.
   * Принимает путь к файлу, путь на сервере или id относительно корня.
   */
  public async import<T = any>(url: string): Promise<T>
  /**
   * Очищает все кэши, включая слушателей HMR.
   */
  public clearCache(): void
  /**
   * Очищает все кэши, удаляет всех слушателей HMR, сбрасывает поддержку карт исходников.
   * Этот метод не разрывает HMR-соединение.
   */
  public async close(): Promise<void>
  /**
   * Возвращает `true`, если раннер был закрыт вызовом `close()`.
   */
  public isClosed(): boolean
}
```

Вычислитель модулей (module evaluator) в `ModuleRunner` отвечает за выполнение кода. Vite «из коробки» экспортирует `ESModulesEvaluator`, который использует `new AsyncFunction` для вычисления кода. Вы можете предоставить собственную реализацию, если ваша JavaScript-среда выполнения не поддерживает небезопасное вычисление (unsafe evaluation).

Раннер модулей предоставляет метод `import`. Когда сервер Vite инициирует HMR-событие `full-reload`, все затронутые модули будут выполнены заново. Учтите, что при этом раннер модулей не обновляет объект `exports` (он перезаписывает его); вам потребуется снова вызвать `import` или заново получить модуль из `evaluatedModules`, если вы полагаетесь на актуальность объекта `exports`.

**Пример использования:**

```js
import {
  ModuleRunner,
  ESModulesEvaluator,
  createNodeImportMeta,
} from 'vite/module-runner'
import { transport } from './rpc-implementation.js'

const moduleRunner = new ModuleRunner(
  {
    transport,
    createImportMeta: createNodeImportMeta, // если раннер модулей работает в Node.js
  },
  new ESModulesEvaluator(),
)

await moduleRunner.import('/src/entry-point.js')
```

## `ModuleRunnerOptions`

```ts
interface ModuleRunnerOptions {
  /**
   * Набор методов для взаимодействия с сервером.
   */
  transport: ModuleRunnerTransport
  /**
   * Настраивает способ разрешения карт исходников.
   * Предпочитает `node`, если доступен `process.setSourceMapsEnabled`.
   * В противном случае по умолчанию используется `prepareStackTrace`,
   * который переопределяет метод `Error.prepareStackTrace`.
   * Вы можете передать объект, чтобы настроить, как разрешаются содержимое
   * файлов и карты исходников для файлов, которые не были обработаны Vite.
   */
  sourcemapInterceptor?:
    | false
    | 'node'
    | 'prepareStackTrace'
    | InterceptorOptions
  /**
   * Отключает HMR или настраивает опции HMR.
   *
   * @default true
   */
  hmr?: boolean | ModuleRunnerHmr
  /**
   * Пользовательский кэш модулей. Если не указан, для каждого экземпляра
   * раннера модулей создаётся отдельный кэш модулей.
   */
  evaluatedModules?: EvaluatedModules
}
```

## `ModuleEvaluator`

**Сигнатура типа:**

```ts
export interface ModuleEvaluator {
  /**
   * Количество строк-префиксов в преобразованном коде.
   */
  startOffset?: number
  /**
   * Вычисляет код, преобразованный Vite.
   * @param context Контекст функции
   * @param code Преобразованный код
   * @param id ID, который использовался для получения модуля
   */
  runInlinedModule(
    context: ModuleRunnerContext,
    code: string,
    id: string,
  ): Promise<any>
  /**
   * вычисляет вынесенный (externalized) модуль.
   * @param file URL файла внешнего модуля
   */
  runExternalModule(file: string): Promise<any>
}
```

Vite по умолчанию экспортирует `ESModulesEvaluator`, реализующий этот интерфейс. Он использует `new AsyncFunction` для вычисления кода, поэтому если код содержит встроенную карту исходников, она должна иметь [смещение в 2 строки](https://tc39.es/ecma262/#sec-createdynamicfunction), чтобы учесть добавленные новые строки. `ESModulesEvaluator` делает это автоматически. Пользовательские вычислители дополнительных строк не добавляют.

## `ModuleRunnerTransport`

**Сигнатура типа:**

```ts
interface ModuleRunnerTransport {
  connect?(handlers: ModuleRunnerTransportHandlers): Promise<void> | void
  disconnect?(): Promise<void> | void
  send?(data: HotPayload): Promise<void> | void
  invoke?(data: HotPayload): Promise<{ result: any } | { error: any }>
  timeout?: number
}
```

Транспортный объект, который взаимодействует с окружением через RPC или путём прямого вызова функции. Если метод `invoke` не реализован, требуется реализовать методы `send` и `connect`. Vite сконструирует `invoke` внутренне.

Вам нужно связать его с экземпляром `HotChannel` на сервере, как в этом примере, где раннер модулей создаётся в потоке-воркере:

**worker.js**

```js
import { parentPort } from 'node:worker_threads'
import { fileURLToPath } from 'node:url'
import {
  ESModulesEvaluator,
  ModuleRunner,
  createNodeImportMeta,
} from 'vite/module-runner'

/** @type {import('vite/module-runner').ModuleRunnerTransport} */
const transport = {
  connect({ onMessage, onDisconnection }) {
    parentPort.on('message', onMessage)
    parentPort.on('close', onDisconnection)
  },
  send(data) {
    parentPort.postMessage(data)
  },
}

const runner = new ModuleRunner(
  {
    transport,
    createImportMeta: createNodeImportMeta,
  },
  new ESModulesEvaluator(),
)
```

**server.js**

```js
import { BroadcastChannel } from 'node:worker_threads'
import { createServer, RemoteEnvironmentTransport, DevEnvironment } from 'vite'

function createWorkerEnvironment(name, config, context) {
  const worker = new Worker('./worker.js')
  const handlerToWorkerListener = new WeakMap()
  const client = {
    send(payload: HotPayload) {
      worker.postMessage(payload)
    },
  }

  const workerHotChannel = {
    // Сообщения из потоков-воркеров не передаются по сети, пропускаем проверки server.fs
    skipFsCheck: true,
    send: (data) => worker.postMessage(data),
    on: (event, handler) => {
      // клиент уже подключён
      if (event === 'vite:client:connect') return
      if (event === 'vite:client:disconnect') {
        const listener = () => {
          handler(undefined, client)
        }
        handlerToWorkerListener.set(handler, listener)
        worker.on('exit', listener)
        return
      }

      const listener = (value) => {
        if (value.type === 'custom' && value.event === event) {
          handler(value.data, client)
        }
      }
      handlerToWorkerListener.set(handler, listener)
      worker.on('message', listener)
    },
    off: (event, handler) => {
      if (event === 'vite:client:connect') return
      if (event === 'vite:client:disconnect') {
        const listener = handlerToWorkerListener.get(handler)
        if (listener) {
          worker.off('exit', listener)
          handlerToWorkerListener.delete(handler)
        }
        return
      }

      const listener = handlerToWorkerListener.get(handler)
      if (listener) {
        worker.off('message', listener)
        handlerToWorkerListener.delete(handler)
      }
    },
  }

  return new DevEnvironment(name, config, {
    transport: workerHotChannel,
  })
}

await createServer({
  environments: {
    worker: {
      dev: {
        createEnvironment: createWorkerEnvironment,
      },
    },
  },
})
```

Обязательно реализуйте события `vite:client:connect` / `vite:client:disconnect` в методах `on` / `off`, если эти методы присутствуют. Событие `vite:client:connect` должно генерироваться при установлении соединения, а событие `vite:client:disconnect` — при его закрытии. Объект `HotChannelClient`, передаваемый обработчику событий, должен иметь одну и ту же ссылку для одного и того же соединения.

Другой пример, использующий HTTP-запрос для взаимодействия между раннером и сервером:

```ts
import { ESModulesEvaluator, ModuleRunner } from 'vite/module-runner'

export const runner = new ModuleRunner(
  {
    transport: {
      async invoke(data) {
        const response = await fetch(`http://my-vite-server/invoke`, {
          method: 'POST',
          body: JSON.stringify(data),
        })
        return response.json()
      },
    },
    hmr: false, // отключаем HMR, так как HMR требует transport.connect
  },
  new ESModulesEvaluator(),
)

await runner.import('/entry.js')
```

В этом случае можно использовать метод `handleInvoke` из `NormalizedHotChannel`:

```ts
const customEnvironment = new DevEnvironment(name, config, context)

server.onRequest((request: Request) => {
  const url = new URL(request.url)
  if (url.pathname === '/invoke') {
    const payload = (await request.json()) as HotPayload
    const result = customEnvironment.hot.handleInvoke(payload)
    return new Response(JSON.stringify(result))
  }
  return Response.error()
})
```

Но учтите, что для поддержки HMR требуются методы `send` и `connect`. Метод `send` обычно вызывается при срабатывании пользовательского события (например, `import.meta.hot.send("my-event")`).

Для окружения SSR, работающего в том же процессе Node.js, что и сервер Vite, Vite экспортирует готовый `HotChannel` — `createServerHotChannel`:

```js
import { createServerHotChannel, DevEnvironment } from 'vite'

new DevEnvironment(name, config, {
  hot: true,
  transport: createServerHotChannel(),
})
```
