# Плагины для окружений

> **Информация: Release Candidate**
>
> API окружений в целом находится на стадии release candidate. Мы будем поддерживать стабильность этих API между мажорными релизами, чтобы дать экосистеме возможность экспериментировать с ними и строить на их основе. Однако учтите, что [некоторые конкретные API](https://vite.dev/changes/#considering) всё ещё считаются экспериментальными.
>
> Мы планируем стабилизировать эти новые API (с возможными ломающими изменениями) в одном из будущих мажорных релизов, после того как проекты-потребители получат время на эксперименты с новыми возможностями и их проверку.

>
> - [Обсуждение обратной связи](https://github.com/vitejs/vite/discussions/16358), где мы собираем отзывы о новых API.

## Доступ к текущему окружению в хуках

Поскольку до Vite 6 было всего два окружения (`client` и `ssr`), булева переменная `ssr` была достаточной для определения текущего окружения в API Vite. Хуки плагинов получали булево значение `ssr` в последнем параметре опций, а некоторые API ожидали необязательный последний параметр `ssr`, чтобы правильно связать модули с нужным окружением (например, `server.moduleGraph.getModuleByUrl(url, { ssr })`).

С появлением настраиваемых окружений у нас теперь есть единообразный способ доступа к их опциям и экземплярам в плагинах. Хуки плагинов теперь предоставляют `this.environment` в своём контексте, а API, которые ранее ожидали булево значение `ssr`, теперь привязаны к соответствующему окружению (например, `environment.moduleGraph.getModuleByUrl(url)`).

У сервера Vite есть общий конвейер плагинов, но при обработке модуля она всегда выполняется в контексте определённого окружения. Экземпляр `environment` доступен в контексте плагина.

Плагин может использовать экземпляр `environment`, чтобы изменять способ обработки модуля в зависимости от конфигурации окружения (к которой можно обратиться через `environment.config`).

```ts
  transform(code, id) {
    console.log(this.environment.config.resolve.conditions)
  }
```

## Регистрация новых окружений с помощью хуков

Плагины могут добавлять новые окружения в хуке `config`. Например, [поддержка RSC](https://vite.dev/plugins/#vitejs-plugin-rsc) использует дополнительное окружение, чтобы иметь отдельный граф модулей с условием `react-server`:

```ts
  config(config: UserConfig) {
    return {
      environments: {
        rsc: {
          resolve: {
            conditions: ['react-server', ...defaultServerConditions],
          },
        },
      },
    }
  }
```

Пустого объекта достаточно для регистрации окружения — при этом используются значения по умолчанию из конфигурации окружения корневого уровня.

## Настройка окружения с помощью хуков

Пока выполняется хук `config`, полный список окружений ещё не известен, и на окружения могут влиять как значения по умолчанию из конфигурации окружения корневого уровня, так и явно заданные значения через запись `config.environments`.
Плагины должны задавать значения по умолчанию с помощью хука `config`. Чтобы настроить каждое окружение, они могут использовать новый хук `configEnvironment`. Этот хук вызывается для каждого окружения с его частично разрешённой конфигурацией, включая разрешение окончательных значений по умолчанию.

```ts
  configEnvironment(name: string, options: EnvironmentOptions) {
    // добавить условие "workerd" к окружению rsc
    if (name === 'rsc') {
      return {
        resolve: {
          conditions: ['workerd'],
        },
      }
    }
  }
```

## Хук `hotUpdate`

- **Тип:** `(this: { environment: DevEnvironment }, options: HotUpdateOptions) => Array<EnvironmentModuleNode> | void | Promise<Array<EnvironmentModuleNode> | void>`
- **Вид:** `async`, `sequential`
- **См. также:** [HMR API](https://vite.dev/guide/api-hmr)

Хук `hotUpdate` позволяет плагинам выполнять собственную обработку HMR-обновлений для конкретного окружения. При изменении файла алгоритм HMR запускается для каждого окружения последовательно, в порядке, заданном в `server.environments`, поэтому хук `hotUpdate` будет вызван несколько раз. Хук получает объект контекста со следующей сигнатурой:

```ts
interface HotUpdateOptions {
  type: 'create' | 'update' | 'delete'
  file: string
  timestamp: number
  modules: Array<EnvironmentModuleNode>
  read: () => string | Promise<string>
  server: ViteDevServer
}
```

- `this.environment` — это окружение выполнения модулей, в котором в данный момент обрабатывается обновление файла.

- `modules` — это массив модулей в этом окружении, на которые влияет изменённый файл. Это массив, потому что один файл может отображаться на несколько обслуживаемых модулей (например, Vue SFC).

- `read` — это асинхронная функция чтения, которая возвращает содержимое файла. Она предоставляется потому, что на некоторых системах колбэк изменения файла может сработать слишком быстро, до того как редактор завершит обновление файла, и прямой вызов `fs.readFile` вернёт пустое содержимое. Переданная функция чтения нормализует это поведение.

Хук может:

- Отфильтровать и сузить список затронутых модулей, чтобы сделать HMR более точным.

- Вернуть пустой массив и выполнить полную перезагрузку:

  ```js
  hotUpdate({ modules, timestamp }) {
    if (this.environment.name !== 'client')
      return

    // Инвалидировать модули вручную
    const invalidatedModules = new Set()
    for (const mod of modules) {
      this.environment.moduleGraph.invalidateModule(
        mod,
        invalidatedModules,
        timestamp,
        true
      )
    }
    this.environment.hot.send({ type: 'full-reload' })
    return []
  }
  ```

- Вернуть пустой массив и выполнить полностью собственную обработку HMR, отправляя клиенту пользовательские события:

  ```js
  hotUpdate() {
    if (this.environment.name !== 'client')
      return

    this.environment.hot.send({
      type: 'custom',
      event: 'special-update',
      data: {}
    })
    return []
  }
  ```

  Код на стороне клиента должен зарегистрировать соответствующий обработчик с помощью [HMR API](https://vite.dev/guide/api-hmr) (он может быть внедрён хуком `transform` того же плагина):

  ```js
  if (import.meta.hot) {
    import.meta.hot.on('special-update', (data) => {
      // выполнить собственное обновление
    })
  }
  ```

## Состояние плагина для каждого окружения

Поскольку один и тот же экземпляр плагина используется для разных окружений, состояние плагина нужно связывать с ключом `this.environment`. Это тот же паттерн, который экосистема уже использовала для хранения состояния о модулях, применяя булево значение `ssr` в качестве ключа, чтобы не смешивать состояние client- и ssr-модулей. Для раздельного хранения состояния каждого окружения можно использовать `Map<Environment, State>`. Обратите внимание, что ради обратной совместимости `buildStart` и `buildEnd` без флага `perEnvironmentStartEndDuringDev: true` вызываются только для окружения client. То же самое касается `watchChange` и флага `perEnvironmentWatchChangeDuringDev: true`.

```js
function PerEnvironmentCountTransformedModulesPlugin() {
  const state = new Map<Environment, { count: number }>()
  return {
    name: 'count-transformed-modules',
    perEnvironmentStartEndDuringDev: true,
    buildStart() {
      state.set(this.environment, { count: 0 })
    },
    transform(id) {
      state.get(this.environment).count++
    },
    buildEnd() {
      console.log(this.environment.name, state.get(this.environment).count)
    }
  }
}
```

## Плагины для отдельных окружений

Плагин может определить, к каким окружениям он должен применяться, с помощью функции `applyToEnvironment`.

```js
const UnoCssPlugin = () => {
  // общее глобальное состояние
  return {
    buildStart() {
      // инициализировать состояние для каждого окружения через WeakMap<Environment,Data>
      // используя this.environment
    },
    configureServer() {
      // использовать глобальные хуки как обычно
    },
    applyToEnvironment(environment) {
      // вернуть true, если этот плагин должен быть активен в данном окружении,
      // или вернуть новый плагин, чтобы заменить его.
      // если хук не используется, плагин активен во всех окружениях
    },
    resolveId(id, importer) {
      // вызывается только для окружений, к которым применяется этот плагин
    },
  }
}
```

Если плагин не учитывает окружения и имеет состояние, не привязанное к текущему окружению, хук `applyToEnvironment` позволяет легко сделать его работающим для каждого окружения отдельно.

```js
import { nonShareablePlugin } from 'non-shareable-plugin'

export default defineConfig({
  plugins: [
    {
      name: 'per-environment-plugin',
      applyToEnvironment(environment) {
        return nonShareablePlugin({ outputName: environment.name })
      },
    },
  ],
})
```

Vite экспортирует вспомогательную функцию `perEnvironmentPlugin`, чтобы упростить такие случаи, когда никакие другие хуки не требуются:

```js
import { nonShareablePlugin } from 'non-shareable-plugin'

export default defineConfig({
  plugins: [
    perEnvironmentPlugin('per-environment-plugin', (environment) =>
      nonShareablePlugin({ outputName: environment.name }),
    ),
  ],
})
```

Хук `applyToEnvironment` вызывается на этапе конфигурации, в настоящее время после `configResolved` — из-за того что проекты в экосистеме модифицируют плагины внутри него. В будущем разрешение плагинов окружений может быть перенесено на этап до `configResolved`.

## Взаимодействие приложения и плагина

`environment.hot` позволяет плагинам взаимодействовать с кодом на стороне приложения для конкретного окружения. Это эквивалент [возможности взаимодействия клиента и сервера](https://vite.dev/guide/api-plugin#client-server-communication), но с поддержкой окружений, отличных от окружения client.

> **Внимание: Примечание**
>
> Обратите внимание, что эта возможность доступна только для окружений, поддерживающих HMR.

### Управление экземплярами приложения

Учтите, что в одном и том же окружении может выполняться несколько экземпляров приложения. Например, если у вас открыто несколько вкладок в браузере, каждая вкладка — это отдельный экземпляр приложения с отдельным подключением к серверу.

Когда устанавливается новое подключение, на экземпляре `hot` окружения генерируется событие `vite:client:connect`. Когда подключение закрывается, генерируется событие `vite:client:disconnect`.

Каждый обработчик события получает `NormalizedHotChannelClient` в качестве второго аргумента. Клиент — это объект с методом `send`, который можно использовать для отправки сообщений конкретному экземпляру приложения. Ссылка на клиента всегда одна и та же для одного подключения, поэтому её можно сохранить, чтобы отслеживать подключение.

### Пример использования

Сторона плагина:

```js
configureServer(server) {
  server.environments.ssr.hot.on('my:greetings', (data, client) => {
    // сделать что-то с данными
    // и при необходимости отправить ответ этому экземпляру приложения
    client.send('my:foo:reply', `Hello from server! You said: ${data}`)
  })

  // разослать сообщение всем экземплярам приложения
  server.environments.ssr.hot.send('my:foo', 'Hello from server!')
}
```

Сторона приложения устроена так же, как в возможности взаимодействия клиента и сервера. Вы можете использовать объект `import.meta.hot` для отправки сообщений плагину.

## Окружение в хуках сборки

Точно так же, как во время разработки, хуки плагинов получают экземпляр окружения и во время сборки, заменяя булево значение `ssr`.
Это также работает для `renderChunk`, `generateBundle` и других хуков, специфичных для сборки.

## Общие плагины во время сборки

До Vite 6 конвейеры плагинов работали по-разному во время разработки и во время сборки:

- **Во время разработки:** плагины общие.
- **Во время сборки:** плагины изолированы для каждого окружения (в разных процессах: `vite build`, затем `vite build --ssr`).

Это вынуждало фреймворки разделять состояние между сборкой `client` и сборкой `ssr` через файлы манифеста, записываемые в файловую систему. В Vite 6 мы теперь собираем все окружения в одном процессе, поэтому способ работы конвейера плагинов и взаимодействия между окружениями можно согласовать с режимом разработки.

В одном из будущих мажорных релизов мы могли бы достичь полного согласования:

- **Как во время разработки, так и во время сборки:** плагины общие, с [фильтрацией по окружениям](#per-environment-plugins).

Также во время сборки будет один общий экземпляр `ResolvedConfig`, что позволит кэшировать на уровне всего процесса сборки приложения так же, как мы делали это с `WeakMap<ResolvedConfig, CachedData>` во время разработки.

Для Vite 6 нам нужно сделать шаг поменьше, чтобы сохранить обратную совместимость. Плагины экосистемы в настоящее время используют `config.build` вместо `environment.config.build` для доступа к конфигурации, поэтому нам нужно по умолчанию создавать новый `ResolvedConfig` для каждого окружения. Проект может согласиться на использование общей полной конфигурации и общего конвейера плагинов, установив `builder.sharedConfigBuild` в `true`.

Поначалу эта опция будет работать лишь для небольшого подмножества проектов, поэтому авторы плагинов могут согласиться на то, чтобы конкретный плагин был общим, установив флаг `sharedDuringBuild` в `true`. Это позволяет легко разделять состояние как для обычных плагинов:

```js
function myPlugin() {
  // Разделять состояние между всеми окружениями в режиме разработки и сборки
  const sharedState = ...
  return {
    name: 'shared-plugin',
    transform(code, id) { ... },

    // Согласиться на единый экземпляр для всех окружений
    sharedDuringBuild: true,
  }
}
```
