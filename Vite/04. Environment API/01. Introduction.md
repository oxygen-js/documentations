# Environment API: введение

> **Информация: Release Candidate**
>
> Environment API в целом находится на стадии кандидата в релиз. Мы будем поддерживать стабильность этих API между мажорными релизами, чтобы экосистема могла экспериментировать с ними и строить на их основе. Однако учтите, что [некоторые конкретные API](https://vite.dev/changes/#considering) по-прежнему считаются экспериментальными.
>
> Мы планируем стабилизировать эти новые API (с возможными обратно несовместимыми изменениями) в одном из будущих мажорных релизов, как только у нижестоящих проектов будет время поэкспериментировать с новыми возможностями и проверить их.
>
> Ресурсы:
>
> - [Обсуждение для обратной связи](https://github.com/vitejs/vite/discussions/16358), где мы собираем отзывы о новых API.
> - [PR с Environment API](https://github.com/vitejs/vite/pull/16471), где новые API были реализованы и прошли ревью.
>
> Пожалуйста, поделитесь с нами своей обратной связью.

## Формализация окружений

Vite 6 формализует концепцию окружений (Environments). Вплоть до Vite 5 существовало два неявных окружения (`client` и, опционально, `ssr`). Новый Environment API позволяет пользователям и авторам фреймворков создавать столько окружений, сколько нужно, чтобы отразить то, как их приложения работают в продакшене. Эта новая возможность потребовала большой внутренней переработки, но много усилий было вложено в обратную совместимость. Изначальная цель Vite 6 — как можно более плавно перевести экосистему на новый мажор, отложив внедрение этих API до тех пор, пока достаточное число пользователей не мигрирует, а авторы фреймворков и плагинов не проверят новый дизайн.

## Сокращение разрыва между сборкой и разработкой

Для простого SPA/MPA в конфигурацию не добавляется никаких новых API, связанных с окружениями. Внутри себя Vite применит опции к окружению `client`, но при конфигурировании Vite знать об этой концепции не обязательно. Конфигурация и поведение из Vite 5 должны здесь работать без изменений.

Когда мы переходим к типичному приложению с серверным рендерингом (SSR), у нас будет два окружения:

- `client`: запускает приложение в браузере.
- `ssr`: запускает приложение в node (или в другой серверной среде выполнения), которая рендерит страницы перед их отправкой в браузер.

В режиме разработки Vite выполняет серверный код в том же процессе Node, что и dev-сервер Vite, что даёт близкое приближение к продакшн-окружению. Однако серверы также могут работать в других JS-средах выполнения, таких как [workerd от Cloudflare](https://github.com/cloudflare/workerd), у которых иные ограничения. Современные приложения также могут работать более чем в двух окружениях, например: браузер, node-сервер и edge-сервер. Vite 5 не позволял корректно представить эти окружения.

Vite 6 позволяет пользователям настраивать своё приложение во время сборки и разработки так, чтобы отобразить все его окружения. Во время разработки один dev-сервер Vite теперь можно использовать для одновременного запуска кода в нескольких разных окружениях. Исходный код приложения по-прежнему преобразуется dev-сервером Vite. Поверх общих HTTP-сервера, промежуточных обработчиков (middleware), разрешённой конфигурации и конвейера плагинов у dev-сервера Vite теперь есть набор независимых dev-окружений. Каждое из них настроено так, чтобы максимально соответствовать продакшн-окружению, и подключено к dev-среде выполнения, где выполняется код (для workerd серверный код теперь может выполняться локально в miniflare). В клиенте код импортирует и выполняет браузер. В других окружениях преобразованный код загружает и вычисляет module runner.

## Конфигурация окружений

Для SPA/MPA конфигурация будет выглядеть похоже на Vite 5. Внутри себя эти опции используются для настройки окружения `client`.

```js
export default defineConfig({
  build: {
    sourcemap: false,
  },
  optimizeDeps: {
    include: ['lib'],
  },
})
```

Это важно, потому что мы хотим сохранить доступность Vite и не выставлять новые концепции наружу, пока в них нет необходимости.

Если приложение состоит из нескольких окружений, то эти окружения можно настроить явно с помощью опции конфигурации `environments`.

```js
export default {
  build: {
    sourcemap: false,
  },
  optimizeDeps: {
    include: ['lib'],
  },
  environments: {
    server: {},
    edge: {
      resolve: {
        noExternal: true,
      },
    },
  },
}
```

Если явно не задокументировано иное, окружение наследует настроенные опции конфигурации верхнего уровня (например, новые окружения `server` и `edge` унаследуют опцию `build.sourcemap: false`). Небольшое число опций верхнего уровня, таких как `optimizeDeps`, применяется только к окружению `client`, поскольку они плохо работают, когда применяются по умолчанию к серверным окружениям. Такие опции помечены значком в [справочнике](https://vite.dev/config/). Окружение `client` также можно настроить явно через `environments.client`, но мы рекомендуем делать это с помощью опций верхнего уровня, чтобы конфигурация клиента оставалась неизменной при добавлении новых окружений.

Интерфейс `EnvironmentOptions` предоставляет все опции для каждого окружения. Есть опции окружения, которые применяются как к `build`, так и к `dev`, например `resolve`. А есть `DevEnvironmentOptions` и `BuildEnvironmentOptions` — для опций, специфичных для разработки и сборки (например, `dev.warmup` или `build.outDir`). Некоторые опции, такие как `optimizeDeps`, применяются только к разработке, но для обратной совместимости остаются на верхнем уровне, а не вложены в `dev`.

```ts
interface EnvironmentOptions {
  define?: Record<string, any>
  resolve?: EnvironmentResolveOptions
  optimizeDeps: DepOptimizationOptions
  consumer?: 'client' | 'server'
  dev: DevOptions
  build: BuildOptions
}
```

Интерфейс `UserConfig` расширяет интерфейс `EnvironmentOptions`, позволяя настраивать клиент и значения по умолчанию для других окружений, конфигурируемых через опцию `environments`. Окружение `client` и серверное окружение с именем `ssr` всегда присутствуют во время разработки. Это обеспечивает обратную совместимость с `server.ssrLoadModule(url)` и `server.moduleGraph`. Во время сборки окружение `client` присутствует всегда, а окружение `ssr` присутствует только если оно настроено явно (через `environments.ssr` или, для обратной совместимости, `build.ssr`). Приложению не обязательно использовать имя `ssr` для своего SSR-окружения — оно может назвать его, например, `server`.

```ts
interface UserConfig extends EnvironmentOptions {
  environments: Record<string, EnvironmentOptions>
  // прочие опции
}
```

Обратите внимание, что свойство верхнего уровня `ssr` будет объявлено устаревшим, как только Environment API станет стабильным. Эта опция играет ту же роль, что и `environments`, но для окружения `ssr` по умолчанию и позволяла конфигурировать лишь небольшой набор опций.

## Пользовательские экземпляры окружений

Доступны низкоуровневые API конфигурации, чтобы провайдеры сред выполнения могли предоставлять окружения с правильными значениями по умолчанию для своих сред. Такие окружения также могут порождать другие процессы или потоки для выполнения модулей во время разработки в среде, более близкой к продакшн-окружению.

Например, [плагин Cloudflare для Vite](https://developers.cloudflare.com/workers/vite-plugin/) использует Environment API для запуска кода в среде выполнения Cloudflare Workers (`workerd`) во время разработки.

```js
import { customEnvironment } from 'vite-environment-provider'

export default {
  build: {
    outDir: '/dist/client',
  },
  environments: {
    ssr: customEnvironment({
      build: {
        outDir: '/dist/ssr',
      },
    }),
  },
}
```

## Обратная совместимость

Текущий серверный API Vite пока не объявлен устаревшим и обратно совместим с Vite 5.

`server.moduleGraph` возвращает смешанное представление графов модулей клиента и ssr. Из всех его методов будут возвращаться обратно совместимые смешанные узлы модулей. Та же схема используется для узлов модулей, передаваемых в `handleHotUpdate`.

Мы пока не рекомендуем переходить на Environment API. Мы стремимся к тому, чтобы значительная часть пользовательской базы приняла Vite 6, прежде чем это произойдёт, чтобы плагинам не приходилось поддерживать две версии. Ознакомьтесь с разделом о будущих обратно несовместимых изменениях, чтобы узнать о предстоящих устареваниях и пути обновления:

- [`this.environment` в хуках](https://vite.dev/changes/this-environment-in-hooks)
- [Хук плагина HMR `hotUpdate`](https://vite.dev/changes/hotupdate-hook)
- [Переход на API для отдельных окружений](https://vite.dev/changes/per-environment-apis)
- [SSR с использованием API `ModuleRunner`](https://vite.dev/changes/ssr-using-modulerunner)
- [Общие плагины во время сборки](https://vite.dev/changes/shared-plugins-during-build)

## Целевая аудитория

Это руководство даёт базовые концепции об окружениях для конечных пользователей.

Авторам плагинов доступен более согласованный API для взаимодействия с конфигурацией текущего окружения. Если вы строите поверх Vite, руководство [Environment API: руководство для плагинов](https://vite.dev/guide/api-environment-plugins) описывает то, как расширенные API плагинов помогают поддерживать несколько пользовательских окружений.

Фреймворки могут решить предоставлять окружения на разных уровнях. Если вы автор фреймворка, продолжите чтение руководства [Environment API: руководство для фреймворков](https://vite.dev/guide/api-environment-frameworks), чтобы узнать о программной стороне Environment API.

Для провайдеров сред выполнения руководство [Environment API: руководство по средам выполнения](https://vite.dev/guide/api-environment-runtimes) объясняет, как предлагать пользовательские окружения для использования фреймворками и пользователями.
