# Environment API для фреймворков

> **Release Candidate**
>
> Environment API в целом находится в фазе release candidate. Мы будем поддерживать стабильность этих API между мажорными релизами, чтобы экосистема могла экспериментировать с ними и строить на их основе. Однако учтите, что [некоторые конкретные API](https://vite.dev/changes/#considering) всё ещё считаются экспериментальными.
>
> Мы планируем стабилизировать эти новые API (с возможными обратно несовместимыми изменениями) в одном из будущих мажорных релизов, после того как зависимые проекты успеют поэкспериментировать с новыми возможностями и проверить их.
>
> Ресурсы:
>
> - [Обсуждение для обратной связи](https://github.com/vitejs/vite/discussions/16358), где мы собираем отзывы о новых API.
> - [PR с Environment API](https://github.com/vitejs/vite/pull/16471), где новые API были реализованы и прошли ревью.

## Уровни коммуникации DevEnvironment

Поскольку окружения могут выполняться в разных средах выполнения (runtime), коммуникация с окружением может иметь ограничения в зависимости от среды выполнения. Чтобы фреймворки могли легко писать код, не зависящий от конкретной среды выполнения, Environment API предоставляет три вида уровней коммуникации.

### `RunnableDevEnvironment`

`RunnableDevEnvironment` — это окружение, которое может обмениваться произвольными значениями JavaScript с кодом вашего приложения. Импорт модуля возвращает его настоящие, «живые» экспорты (функции, экземпляры классов и любые другие значения), поэтому фреймворки могут запускать свои серверные точки входа напрямую. Неявное окружение `ssr` и другие неклиентские окружения по умолчанию используют `RunnableDevEnvironment` во время разработки. Вы можете защитить доступ к runner с помощью функции `isRunnableDevEnvironment`.

Его `runner` — это `ModuleRunner`. Через него вы импортируете модули с помощью `runner.import(url)`, который загружает, преобразует и вычисляет модуль из графа модулей Vite (аргумент `url` принимает путь к файлу, серверный путь или id относительно корня) и возвращает созданный экземпляр модуля с полной поддержкой HMR. Это современная замена `server.ssrLoadModule`, поэтому фреймворки могут перейти на него, чтобы включить HMR для своей SSR-разработки.

> **Почему он может обмениваться произвольными значениями**
>
> `RunnableDevEnvironment` вычисляет модули в той же среде выполнения, что и сервер Vite, поэтому значения пересекают границу внутри одного процесса, а не сериализуются. Именно это отличает его от [`FetchableDevEnvironment`](#fetchabledevenvironment), который может обмениваться данными только через сериализованные объекты `Request`/`Response` посредством Fetch API. Как следствие, использование `RunnableDevEnvironment` требует, чтобы среда выполнения runner совпадала с той, в которой работает сервер Vite.

```ts
export class RunnableDevEnvironment extends DevEnvironment {
  public readonly runner: ModuleRunner
}

class ModuleRunner {
  /**
   * URL для выполнения.
   * Принимает путь к файлу, серверный путь или id относительно корня.
   * Возвращает созданный экземпляр модуля (как в ssrLoadModule)
   */
  public async import(url: string): Promise<Record<string, any>>
  /**
   * Другие методы ModuleRunner...
   */
}

if (isRunnableDevEnvironment(server.environments.ssr)) {
  await server.environments.ssr.runner.import('/entry-point.js')
}
```

> **Внимание**
>
> `runner` вычисляется лениво, только при первом обращении к нему. Учтите, что Vite включает поддержку карт исходников (source map) при создании `runner`, вызывая `process.setSourceMapsEnabled` или переопределяя `Error.prepareStackTrace`, если он недоступен.

Имея сервер Vite, настроенный в режиме middleware, как описано в [руководстве по настройке SSR](https://vite.dev/guide/ssr#setting-up-the-dev-server), давайте реализуем SSR-middleware с помощью Environment API. Помните, что окружение не обязательно должно называться `ssr`, поэтому в этом примере мы назовём его `server`. Обработка ошибок опущена.

```js
import fs from 'node:fs'
import path from 'node:path'
import { createServer } from 'vite'

const viteServer = await createServer({
  server: { middlewareMode: true },
  appType: 'custom',
  environments: {
    server: {
      // по умолчанию модули выполняются в том же процессе, что и сервер vite
    },
  },
})

// В TypeScript вам может понадобиться привести это к RunnableDevEnvironment
// или использовать isRunnableDevEnvironment для защиты доступа к runner
const serverEnvironment = viteServer.environments.server

app.use('*', async (req, res, next) => {
  const url = req.originalUrl

  // 1. Читаем index.html
  const indexHtmlPath = path.resolve(import.meta.dirname, 'index.html')
  let template = fs.readFileSync(indexHtmlPath, 'utf-8')

  // 2. Применяем HTML-преобразования Vite. Это внедряет клиент Vite HMR,
  //    а также применяет HTML-преобразования из плагинов Vite, например
  //    глобальные преамбулы из @vitejs/plugin-react
  template = await viteServer.transformIndexHtml(url, template)

  // 3. Загружаем серверную точку входа. import(url) автоматически преобразует
  //    исходный код ESM для использования в Node.js! Бандлинг не требуется,
  //    и при этом обеспечивается полная поддержка HMR.
  const { render } = await serverEnvironment.runner.import(
    '/src/entry-server.js',
  )

  // 4. Рендерим HTML приложения. Предполагается, что экспортируемая из
  //    entry-server.js функция `render` вызывает соответствующие SSR API
  //    фреймворка, например ReactDOMServer.renderToString()
  const appHtml = await render(url)

  // 5. Внедряем отрендеренный приложением HTML в шаблон.
  const html = template.replace(`<!--ssr-outlet-->`, appHtml)

  // 6. Отправляем отрендеренный HTML обратно.
  res.status(200).set({ 'Content-Type': 'text/html' }).end(html)
})
```

При использовании окружений, поддерживающих HMR (таких как `RunnableDevEnvironment`), для оптимального поведения вам следует добавить `import.meta.hot.accept()` в файл серверной точки входа. Без этого изменения серверных файлов будут инвалидировать весь граф серверных модулей:

```js
// src/entry-server.js
export function render(...) { ... }

if (import.meta.hot) {
  import.meta.hot.accept()
}
```

### `FetchableDevEnvironment`

> **Информация**
>
> Мы ждём отзывов о [предложении `FetchableDevEnvironment`](https://github.com/vitejs/vite/discussions/18191).

`FetchableDevEnvironment` — это окружение, которое может обмениваться данными со своей средой выполнения через интерфейс [Fetch API](https://developer.mozilla.org/en-US/docs/Web/API/Window/fetch). Поскольку `RunnableDevEnvironment` возможно реализовать лишь в ограниченном наборе сред выполнения, мы рекомендуем использовать `FetchableDevEnvironment` вместо `RunnableDevEnvironment`.

Частая причина обратиться к нему — фреймворк, который хочет поддерживать среду выполнения, не способную запускать Vite напрямую (например, Cloudflare Workers). `RunnableDevEnvironment` там использовать нельзя, поскольку он требует, чтобы runner разделял среду выполнения сервера Vite для передачи значений внутри одного процесса. Стандартизация на Fetch API позволяет фреймворку сохранять единый путь обработки запросов во всех целевых средах выполнения: его dev-middleware перенаправляет каждый входящий запрос браузера как `Request` и отправляет возвращённый `Response` обратно в браузер, отражая то, как приложение обрабатывает запросы в продакшене.

Это окружение предоставляет стандартизированный способ обработки запросов через метод `handleRequest`:

```ts
import {
  createServer,
  createFetchableDevEnvironment,
  isFetchableDevEnvironment,
} from 'vite'

const server = await createServer({
  server: { middlewareMode: true },
  appType: 'custom',
  environments: {
    custom: {
      dev: {
        createEnvironment(name, config) {
          return createFetchableDevEnvironment(name, config, {
            handleRequest(request: Request): Promise<Response> | Response {
              // обрабатываем Request и возвращаем Response
            },
          })
        },
      },
    },
  },
})

// Теперь любой потребитель environment API может вызвать `dispatchFetch`
if (isFetchableDevEnvironment(server.environments.custom)) {
  const response: Response = await server.environments.custom.dispatchFetch(
    new Request('http://example.com/request-to-handle'),
  )
}
```

> **Внимание**
>
> Vite проверяет входные и выходные данные метода `dispatchFetch`: запрос должен быть экземпляром глобального класса `Request`, а ответ — экземпляром глобального класса `Response`. Если это не так, Vite выбросит `TypeError`.
>
> Обратите внимание: хотя `FetchableDevEnvironment` реализован как класс, команда Vite считает это деталью реализации, и он может измениться в любой момент.

### «Сырой» `DevEnvironment`

Если окружение не реализует интерфейсы `RunnableDevEnvironment` или `FetchableDevEnvironment`, вам нужно настроить коммуникацию вручную.

Если ваш код может выполняться в той же среде выполнения, что и пользовательские модули (то есть не полагается на специфичные для Node.js API), вы можете использовать виртуальный модуль. Такой подход избавляет от необходимости обращаться к значению из кода, использующего API Vite.

```ts
// код, использующий API Vite
import { createServer } from 'vite'

const server = createServer({
  plugins: [
    // плагин, обрабатывающий `virtual:entrypoint`
    {
      name: 'virtual-module',
      /* реализация плагина */
    },
  ],
})
const ssrEnvironment = server.environment.ssr
const input = {}

// используйте функции, предоставляемые каждой фабрикой окружений, которая запускает код
// проверьте у каждой фабрики окружений, что именно она предоставляет
if (ssrEnvironment instanceof CustomDevEnvironment) {
  ssrEnvironment.runEntrypoint('virtual:entrypoint')
} else {
  throw new Error(`Unsupported runtime for ${ssrEnvironment.name}`)
}

// -------------------------------------
// virtual:entrypoint
const { createHandler } = await import('./entrypoint.js')
const handler = createHandler(input)
const response = handler(new Request('http://example.com/'))

// -------------------------------------
// ./entrypoint.js
export function createHandler(input) {
  return function handler(req) {
    return new Response('hello')
  }
}
```

Например, чтобы вызвать `transformIndexHtml` для пользовательского модуля, можно использовать следующий плагин:

```ts
function vitePluginVirtualIndexHtml(): Plugin {
  let server: ViteDevServer | undefined
  return {
    name: vitePluginVirtualIndexHtml.name,
    configureServer(server_) {
      server = server_
    },
    resolveId(source) {
      return source === 'virtual:index-html' ? '\0' + source : undefined
    },
    async load(id) {
      if (id === '\0' + 'virtual:index-html') {
        let html: string
        if (server) {
          this.addWatchFile('index.html')
          html = fs.readFileSync('index.html', 'utf-8')
          html = await server.transformIndexHtml('/', html)
        } else {
          html = fs.readFileSync('dist/client/index.html', 'utf-8')
        }
        return `export default ${JSON.stringify(html)}`
      }
      return
    },
  }
}
```

Если вашему коду нужны API Node.js, вы можете использовать `hot.send`, чтобы из пользовательских модулей общаться с кодом, использующим API Vite. Однако учтите, что этот подход может работать иначе после сборки.

```ts
// код, использующий API Vite
import { createServer } from 'vite'

const server = createServer({
  plugins: [
    // плагин, обрабатывающий `virtual:entrypoint`
    {
      name: 'virtual-module',
      /* реализация плагина */
    },
  ],
})
const ssrEnvironment = server.environment.ssr
const input = {}

// используйте функции, предоставляемые каждой фабрикой окружений, которая запускает код
// проверьте у каждой фабрики окружений, что именно она предоставляет
if (ssrEnvironment instanceof RunnableDevEnvironment) {
  ssrEnvironment.runner.import('virtual:entrypoint')
} else if (ssrEnvironment instanceof CustomDevEnvironment) {
  ssrEnvironment.runEntrypoint('virtual:entrypoint')
} else {
  throw new Error(`Unsupported runtime for ${ssrEnvironment.name}`)
}

const req = new Request('http://example.com/')

const uniqueId = 'a-unique-id'
ssrEnvironment.send('request', serialize({ req, uniqueId }))
const response = await new Promise((resolve) => {
  ssrEnvironment.on('response', (data) => {
    data = deserialize(data)
    if (data.uniqueId === uniqueId) {
      resolve(data.res)
    }
  })
})

// -------------------------------------
// virtual:entrypoint
const { createHandler } = await import('./entrypoint.js')
const handler = createHandler(input)

import.meta.hot.on('request', (data) => {
  const { req, uniqueId } = deserialize(data)
  const res = handler(req)
  import.meta.hot.send('response', serialize({ res: res, uniqueId }))
})

const response = handler(new Request('http://example.com/'))

// -------------------------------------
// ./entrypoint.js
export function createHandler(input) {
  return function handler(req) {
    return new Response('hello')
  }
}
```

## Окружения во время сборки

В CLI вызовы `vite build` и `vite build --ssr` по-прежнему будут собирать только клиентское и только ssr-окружение для обратной совместимости.

Когда задана опция `builder` (даже пустым объектом `{}`, что и делает `vite build --app`), `vite build` вместо этого переключается на сборку всего приложения целиком. Это станет поведением по умолчанию в одном из будущих мажорных релизов. В этом режиме Vite создаёт экземпляр `ViteBuilder` (эквивалент `ViteDevServer` для времени сборки) и использует его для сборки всех настроенных окружений для продакшена. По умолчанию окружения собираются последовательно, следуя порядку в объекте `environments`.

### Настройка сборки приложения с помощью `builder.buildApp`

Фреймворк или пользователь может управлять тем, как собираются окружения, через опцию `builder.buildApp`. Она получает экземпляр `ViteBuilder` (в примере ниже он назван `builder`) и отвечает за сборку каждого окружения; например, чтобы собрать некоторые из них параллельно:

**vite.config.js**

```js
import { defineConfig } from 'vite'

export default defineConfig({
  builder: {
    buildApp: async (builder) => {
      const environments = Object.values(builder.environments)
      await Promise.all(
        environments.map((environment) => builder.build(environment)),
      )
    },
  },
})
```

### Хук плагина `buildApp`

Помимо опции конфигурации `builder.buildApp`, плагины могут определить хук `buildApp`, чтобы участвовать в сборке приложения. Опция конфигурации и хуки плагинов выполняются в определённом порядке: сначала выполняются хуки с порядком `'pre'` или `null`, затем настроенный `builder.buildApp`, затем хуки с порядком `'post'`. Внутри хука `environment.isBuilt` сообщает, было ли окружение уже собрано, что позволяет плагину избежать его повторной сборки.

### Программная сборка с помощью `createBuilder`

Чтобы запустить сборку приложения из собственного кода, используйте `createBuilder` вместо отдельной функции `build`. `createBuilder` — это эквивалент `createServer` для времени сборки: он разрешает конфигурацию и возвращает `ViteBuilder`, метод `buildApp` которого собирает каждое настроенное окружение. Вы также можете собрать одно окружение с помощью `builder.build(environment)`.

**build.js**

```js
import { createBuilder } from 'vite'

const builder = await createBuilder()
await builder.buildApp()
```

`createBuilder` заменяет отдельную функцию `build` для сборок, учитывающих окружения. `build` по-прежнему работает как простая точка входа для устаревших сборок только клиента и только ssr, описанных выше, но не может собирать произвольные окружения. Вызов `builder.buildApp()` — это программный эквивалент `vite build --app`.

## Код, не зависящий от окружения

В большинстве случаев текущий экземпляр `environment` будет доступен как часть контекста выполняемого кода, поэтому необходимость обращаться к ним через `server.environments` должна возникать редко. Например, внутри хуков плагинов окружение предоставляется как часть `PluginContext`, поэтому к нему можно обратиться через `this.environment`. См. [Environment API для плагинов](https://vite.dev/guide/api-environment-plugins), чтобы узнать, как создавать плагины, учитывающие окружения.
