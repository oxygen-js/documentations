# Plugin API

Плагины Vite расширяют интерфейс плагинов Rolldown несколькими дополнительными опциями, специфичными для Vite. Благодаря этому вы можете написать плагин Vite один раз, и он будет работать как в режиме разработки, так и при сборке.

**Перед чтением следующих разделов рекомендуется сначала ознакомиться с [документацией по плагинам Rolldown](https://rolldown.rs/apis/plugin-api).**

## Создание плагина

Vite стремится предоставлять устоявшиеся паттерны «из коробки», поэтому перед созданием нового плагина убедитесь, что ваш случай ещё не покрыт, заглянув в [руководство по возможностям](https://vite.dev/guide/features). Также ознакомьтесь с доступными плагинами сообщества — как в виде [совместимого плагина Rollup](https://github.com/rollup/awesome), так и в виде [плагинов, специфичных для Vite](https://github.com/vitejs/awesome-vite#plugins).

При создании плагина вы можете встроить его прямо в свой `vite.config.js`. Создавать для него отдельный пакет не обязательно. Как только вы увидите, что плагин оказался полезен в ваших проектах, подумайте о том, чтобы поделиться им и помочь другим [в экосистеме](https://chat.vite.dev).

> **Совет**
> При изучении, отладке или создании плагинов мы рекомендуем подключить в проект [vite-plugin-inspect](https://github.com/antfu/vite-plugin-inspect). Он позволяет исследовать промежуточное состояние плагинов Vite. После установки вы можете зайти на `localhost:5173/__inspect/`, чтобы изучить модули и стек преобразований вашего проекта. Инструкции по установке смотрите в [документации vite-plugin-inspect](https://github.com/antfu/vite-plugin-inspect).
>
> ![vite-plugin-inspect](https://vite.dev/images/vite-plugin-inspect.webp)

## Соглашения

Если плагин не использует специфичные для Vite хуки и может быть реализован как [совместимый плагин Rolldown](#rolldown-plugin-compatibility), то рекомендуется следовать [соглашениям по именованию плагинов Rolldown](https://rolldown.rs/apis/plugin-api#conventions).

- Плагины Rolldown должны иметь понятное имя с префиксом `rolldown-plugin-`.
- Добавьте ключевые слова `rolldown-plugin` и `vite-plugin` в поле `keywords` файла package.json.

Это позволит использовать плагин также в проектах на чистом Rolldown или Rollup.

Для плагинов, работающих только с Vite:

- Плагины Vite должны иметь понятное имя с префиксом `vite-plugin-`.
- Добавьте ключевое слово `vite-plugin` в поле `keywords` файла package.json.
- Включите в документацию плагина раздел с объяснением, почему это плагин только для Vite (например, потому что он использует специфичные для Vite хуки плагинов).

Если ваш плагин предназначен для работы только с определённым фреймворком, его имя должно включаться в состав префикса:

- префикс `vite-plugin-vue-` для плагинов Vue
- префикс `vite-plugin-react-` для плагинов React
- префикс `vite-plugin-svelte-` для плагинов Svelte

См. также [соглашение о виртуальных модулях](#virtual-modules-convention).

## Конфигурация плагинов

Пользователи добавляют плагины в `devDependencies` проекта и настраивают их через опцию-массив `plugins`.

**`vite.config.js`**

```js
import vitePlugin from 'vite-plugin-feature'
import rollupPlugin from 'rollup-plugin-feature'

export default defineConfig({
  plugins: [vitePlugin(), rollupPlugin()],
})
```

Ложные (falsy) значения плагинов будут проигнорированы, что можно использовать для простого включения или отключения плагинов.

`plugins` также принимает пресеты, включающие несколько плагинов в виде одного элемента. Это удобно для сложных возможностей (например, интеграции с фреймворком), которые реализованы с помощью нескольких плагинов. Массив будет развёрнут (flattened) внутренне.

```js
// framework-plugin
import frameworkRefresh from 'vite-plugin-framework-refresh'
import frameworkDevtools from 'vite-plugin-framework-devtools'

export default function framework(config) {
  return [frameworkRefresh(config), frameworkDevTools(config)]
}
```

**`vite.config.js`**

```js
import { defineConfig } from 'vite'
import framework from 'vite-plugin-framework'

export default defineConfig({
  plugins: [framework()],
})
```

## Простые примеры

> **Совет**
> Общепринятое соглашение — оформлять плагин Vite/Rolldown/Rollup в виде фабричной функции, которая возвращает сам объект плагина. Функция может принимать опции, что позволяет пользователям настраивать поведение плагина.

### Преобразование пользовательских типов файлов

```js
const fileRegex = /\.(my-file-ext)$/

export default function myPlugin() {
  return {
    name: 'transform-file',

    transform: {
      filter: {
        id: fileRegex,
      },
      handler(src, id) {
        return {
          code: compileFileToJS(src),
          map: null, // предоставьте source map, если она доступна
        }
      },
    },
  }
}
```

## Соглашение о виртуальных модулях

Виртуальные модули — это удобная схема, которая позволяет передавать информацию времени сборки в исходные файлы с помощью обычного синтаксиса импорта ESM.

```js
import { exactRegex } from '@rolldown/pluginutils'

export default function myPlugin() {
  const virtualModuleId = 'virtual:my-module'
  const resolvedVirtualModuleId = '\0' + virtualModuleId

  return {
    name: 'my-plugin', // обязательно; будет отображаться в предупреждениях и ошибках
    resolveId: {
      filter: { id: exactRegex(virtualModuleId) },
      handler() {
        return resolvedVirtualModuleId
      },
    },
    load: {
      filter: { id: exactRegex(resolvedVirtualModuleId) },
      handler() {
        return `export const msg = "from virtual module"`
      },
    },
  }
}
```

Это позволяет импортировать модуль в JavaScript:

```js
import { msg } from 'virtual:my-module'

console.log(msg)
```

Виртуальные модули в Vite (а также в Rolldown / Rollup) по соглашению имеют префикс `virtual:` в пути, видимом пользователю. По возможности имя плагина следует использовать в качестве пространства имён, чтобы избежать коллизий с другими плагинами в экосистеме. Например, `vite-plugin-posts` может предложить пользователям импортировать виртуальные модули `virtual:posts` или `virtual:posts/helpers`, чтобы получить информацию времени сборки. Внутренне плагины, использующие виртуальные модули, при разрешении id должны добавлять к идентификатору модуля префикс `\0` — это соглашение из экосистемы rollup. Это не даёт другим плагинам пытаться обрабатывать данный id (например, выполнять разрешение по правилам node), а базовые возможности, такие как карты исходников (sourcemaps), могут использовать эту информацию, чтобы отличать виртуальные модули от обычных файлов. `\0` не является допустимым символом в URL импортов, поэтому при анализе импортов мы вынуждены его заменять. Виртуальный id вида `\0{id}` в браузере во время разработки в итоге кодируется как `/@id/__x00__{id}`. Перед входом в конвейер плагинов id декодируется обратно, поэтому код хуков плагинов этого не видит.

Обратите внимание, что модули, непосредственно производные от реального файла — как, например, модуль скрипта в однофайловом компоненте (Single File Component, таком как .vue или .svelte SFC), — не обязаны следовать этому соглашению. SFC при обработке обычно порождают набор подмодулей, но код в них можно сопоставить обратно с файловой системой. Использование `\0` для таких подмодулей помешало бы корректной работе карт исходников (sourcemaps).

## Универсальные хуки

Во время разработки dev-сервер Vite создаёт контейнер плагинов, который вызывает [хуки сборки Rolldown](https://rolldown.rs/apis/plugin-api#build-hooks) точно так же, как это делает Rolldown.

Следующие хуки вызываются один раз при запуске сервера:

- [`options`](https://rolldown.rs/reference/Interface.Plugin#options)
- [`buildStart`](https://rolldown.rs/reference/Interface.Plugin#buildstart)

Следующие хуки вызываются при каждом входящем запросе модуля:

- [`resolveId`](https://rolldown.rs/reference/Interface.Plugin#resolveid)
- [`load`](https://rolldown.rs/reference/Interface.Plugin#load)
- [`transform`](https://rolldown.rs/reference/Interface.Plugin#transform)

У этих хуков также есть расширенный параметр `options` с дополнительными свойствами, специфичными для Vite. Подробнее об этом можно прочитать в [документации по SSR](https://vite.dev/guide/ssr#ssr-specific-plugin-logic).

В некоторых вызовах `resolveId` значение `importer` может быть абсолютным путём к обобщённому `index.html` в корне, поскольку из-за паттерна несобранного dev-сервера Vite не всегда возможно определить фактический импортирующий модуль. Для импортов, обрабатываемых в конвейере разрешения Vite, импортирующий модуль можно отследить на этапе анализа импортов, что даёт корректное значение `importer`.

Следующие хуки вызываются при закрытии сервера:

- [`buildEnd`](https://rolldown.rs/reference/Interface.Plugin#buildend)
- [`closeBundle`](https://rolldown.rs/reference/Interface.Plugin#closebundle)

Обратите внимание, что хук [`moduleParsed`](https://rolldown.rs/reference/Interface.Plugin#moduleparsed) **не** вызывается во время разработки, потому что Vite избегает полного разбора AST ради лучшей производительности.

[Хуки генерации вывода](https://rolldown.rs/apis/plugin-api#output-generation-hooks) (кроме `closeBundle`) **не** вызываются во время разработки.

## Специфичные для Vite хуки

Плагины Vite также могут предоставлять хуки, служащие специфичным для Vite целям. Эти хуки игнорируются Rollup.

### `config`

- **Тип:** `(config: UserConfig, env: { mode: 'build' | 'serve', command: string, isSsrBuild?: boolean, isPreview?: boolean }) => UserConfig | null | void`
- **Вид:** `async`, `sequential`

Изменяет конфигурацию Vite до её разрешения. Хук получает исходную пользовательскую конфигурацию (опции CLI, объединённые с файлом конфигурации) и текущее окружение конфигурации, которое предоставляет используемые `mode` и `command`. Он может вернуть частичный объект конфигурации, который будет глубоко объединён (deeply merged) с существующей конфигурацией, либо напрямую изменить конфигурацию (если стандартное объединение не позволяет достичь желаемого результата).

**Пример:**

```js
// вернуть частичную конфигурацию (рекомендуется)
const partialConfigPlugin = () => ({
  name: 'return-partial',
  config: () => ({
    resolve: {
      alias: {
        foo: 'bar',
      },
    },
  }),
})

// изменять конфигурацию напрямую (используйте только когда объединение не работает)
const mutateConfigPlugin = () => ({
  name: 'mutate-config',
  config(config, { command }) {
    if (command === 'build') {
      config.root = 'foo'
    }
  },
})
```

> **Внимание: Примечание**
> Пользовательские плагины разрешаются до запуска этого хука, поэтому внедрение других плагинов внутри хука `config` не даст никакого эффекта.

### `configResolved`

- **Тип:** `(config: ResolvedConfig) => void | Promise<void>`
- **Вид:** `async`, `parallel`

Вызывается после разрешения конфигурации Vite. Используйте этот хук, чтобы прочитать и сохранить окончательную разрешённую конфигурацию. Он также полезен, когда плагину нужно вести себя по-разному в зависимости от выполняемой команды.

**Пример:**

```js
const examplePlugin = () => {
  let config

  return {
    name: 'read-config',

    configResolved(resolvedConfig) {
      // сохранить разрешённую конфигурацию
      config = resolvedConfig
    },

    // использовать сохранённую конфигурацию в других хуках
    transform(code, id) {
      if (config.command === 'serve') {
        // dev: плагин вызван dev-сервером
      } else {
        // build: плагин вызван Rollup
      }
    },
  }
}
```

Обратите внимание, что значение `command` равно `serve` в режиме разработки (в CLI команды `vite`, `vite dev` и `vite serve` являются псевдонимами).

### `configureServer`

- **Тип:** `(server: ViteDevServer) => (() => void) | void | Promise<(() => void) | void>`
- **Вид:** `async`, `sequential`
- **См. также:** [ViteDevServer](https://vite.dev/guide/api-javascript#vitedevserver)

Хук для настройки dev-сервера. Наиболее распространённый случай использования — добавление собственных промежуточных обработчиков (middleware) во внутреннее приложение [connect](https://github.com/senchalabs/connect):

```js
const myPlugin = () => ({
  name: 'configure-server',
  configureServer(server) {
    server.middlewares.use((req, res, next) => {
      // собственная обработка запроса...
    })
  },
})
```

**Внедрение промежуточного обработчика после внутренних**

Хук `configureServer` вызывается до установки внутренних промежуточных обработчиков, поэтому по умолчанию пользовательские обработчики выполняются раньше внутренних. Если вы хотите внедрить обработчик **после** внутренних, вы можете вернуть функцию из `configureServer`, которая будет вызвана после установки внутренних обработчиков:

```js
const myPlugin = () => ({
  name: 'configure-server',
  configureServer(server) {
    // вернуть post-хук, который вызывается после установки
    // внутренних промежуточных обработчиков
    return () => {
      server.middlewares.use((req, res, next) => {
        // собственная обработка запроса...
      })
    }
  },
})
```

**Хранение доступа к серверу**

В некоторых случаях другим хукам плагина может потребоваться доступ к экземпляру dev-сервера (например, для доступа к WebSocket-серверу, наблюдателю за файловой системой или графу модулей). Этот хук также можно использовать для сохранения экземпляра сервера, чтобы обращаться к нему в других хуках:

```js
const myPlugin = () => {
  let server
  return {
    name: 'configure-server',
    configureServer(_server) {
      server = _server
    },
    transform(code, id) {
      if (server) {
        // использовать server...
      }
    },
  }
}
```

Обратите внимание, что `configureServer` не вызывается при выполнении продакшн-сборки, поэтому другие ваши хуки должны учитывать его возможное отсутствие.

### `configurePreviewServer`

- **Тип:** `(server: PreviewServer) => (() => void) | void | Promise<(() => void) | void>`
- **Вид:** `async`, `sequential`
- **См. также:** [PreviewServer](https://vite.dev/guide/api-javascript#previewserver)

То же самое, что [`configureServer`](https://vite.dev/guide/api-plugin.html#configureserver), но для сервера предпросмотра (preview). Как и `configureServer`, хук `configurePreviewServer` вызывается до установки других промежуточных обработчиков. Если вы хотите внедрить обработчик **после** остальных, вы можете вернуть функцию из `configurePreviewServer`, которая будет вызвана после установки внутренних обработчиков:

```js
const myPlugin = () => ({
  name: 'configure-preview-server',
  configurePreviewServer(server) {
    // вернуть post-хук, который вызывается после установки
    // остальных промежуточных обработчиков
    return () => {
      server.middlewares.use((req, res, next) => {
        // собственная обработка запроса...
      })
    }
  },
})
```

### `transformIndexHtml`

- **Тип:** `IndexHtmlTransformHook | { order?: 'pre' | 'post', handler: IndexHtmlTransformHook }`
- **Вид:** `async`, `sequential`

Специальный хук для преобразования HTML-файлов точек входа, таких как `index.html`. Хук получает текущую HTML-строку и контекст преобразования. Контекст предоставляет экземпляр [`ViteDevServer`](https://vite.dev/guide/api-javascript#vitedevserver) во время разработки и выходной бандл Rollup во время сборки.

Хук может быть асинхронным и может вернуть одно из следующего:
- Преобразованную HTML-строку
- Массив объектов-описателей тегов (`{ tag, attrs, children }`) для внедрения в существующий HTML. Каждый тег также может указывать, куда его следует внедрить (по умолчанию — в начало `<head>`)
- Объект, содержащий и то, и другое, в виде `{ html, tags }`

По умолчанию `order` равен `undefined`, и при этом хук применяется после того, как HTML уже преобразован. Чтобы внедрить скрипт, который должен пройти через конвейер плагинов Vite, `order: 'pre'` применит хук до обработки HTML. `order: 'post'` применяет хук после применения всех хуков с неопределённым (undefined) `order`.

**Базовый пример:**

```js
const htmlPlugin = () => {
  return {
    name: 'html-transform',
    transformIndexHtml(html) {
      return html.replace(
        /<title>(.*?)<\/title>/,
        `<title>Title replaced!</title>`,
      )
    },
  }
}
```

**Полная сигнатура хука:**

```ts
type IndexHtmlTransformHook = (
  html: string,
  ctx: {
    path: string
    filename: string
    server?: ViteDevServer
    bundle?: import('rolldown').OutputBundle
    chunk?: import('rolldown').OutputChunk
    originalUrl?: string
  },
) =>
  | IndexHtmlTransformResult
  | void
  | Promise<IndexHtmlTransformResult | void>

type IndexHtmlTransformResult =
  | string
  | HtmlTagDescriptor[]
  | {
      html: string
      tags: HtmlTagDescriptor[]
    }

interface HtmlTagDescriptor {
  tag: string
  /**
   * значения атрибутов при необходимости экранируются автоматически
   */
  attrs?: Record<string, string | boolean>
  children?: string | HtmlTagDescriptor[]
  /**
   * по умолчанию: 'head-prepend'
   */
  injectTo?: 'head' | 'body' | 'head-prepend' | 'body-prepend'
}
```

> **Внимание: Примечание**
> Этот хук не будет вызван, если вы используете фреймворк с собственной обработкой файлов точек входа (например, [SvelteKit](https://github.com/sveltejs/kit/discussions/8269#discussioncomment-4509145)).

### `handleHotUpdate`

- **Тип:** `(ctx: HmrContext) => Array<ModuleNode> | void | Promise<Array<ModuleNode> | void>`
- **Вид:** `async`, `sequential`
- **См. также:** [HMR API](https://vite.dev/guide/api-hmr)

Выполняет собственную обработку HMR-обновлений. Хук получает объект контекста со следующей сигнатурой:

```ts
interface HmrContext {
  file: string
  timestamp: number
  modules: Array<ModuleNode>
  read: () => string | Promise<string>
  server: ViteDevServer
}
```

- `modules` — это массив модулей, на которые влияет изменённый файл. Это массив, потому что один файл может отображаться на несколько обслуживаемых модулей (например, Vue SFC).

- `read` — это асинхронная функция чтения, которая возвращает содержимое файла. Она предоставляется потому, что на некоторых системах колбэк изменения файла может сработать слишком быстро, до того как редактор завершит обновление файла, и прямой вызов `fs.readFile` вернёт пустое содержимое. Переданная функция чтения нормализует это поведение.

Хук может:
- Отфильтровать и сузить список затронутых модулей, чтобы сделать HMR более точным.

- Вернуть пустой массив и выполнить полную перезагрузку:

  ```js
  handleHotUpdate({ server, modules, timestamp }) {
    // Инвалидировать модули вручную
    const invalidatedModules = new Set()
    for (const mod of modules) {
      server.moduleGraph.invalidateModule(
        mod,
        invalidatedModules,
        timestamp,
        true
      )
    }
    server.ws.send({ type: 'full-reload' })
    return []
  }
  ```

- Вернуть пустой массив и выполнить полностью собственную обработку HMR, отправляя клиенту пользовательские события:

  ```js
  handleHotUpdate({ server }) {
    server.ws.send({
      type: 'custom',
      event: 'special-update',
      data: {}
    })
    return []
  }
  ```

  Код на стороне клиента должен зарегистрировать соответствующий обработчик с помощью [HMR API](https://vite.dev/guide/api-hmr) (он может быть внедрён хуком `transform` того же плагина):

  ```js
  if (import.meta.hot) {
    import.meta.hot.on('special-update', (data) => {
      // выполнить собственное обновление
    })
  }
  ```

## Метаданные контекста плагина

Для хуков плагинов, у которых есть доступ к контексту плагина, Vite предоставляет дополнительные свойства в `this.meta`:

- `this.meta.viteVersion`: строка текущей версии Vite (например, `"8.0.0"`).

> **Совет: Обнаружение Vite на базе Rolldown**
>
> [`this.meta.rolldownVersion`](https://rolldown.rs/reference/Interface.PluginContextMeta#rolldownversion) доступно только для Vite на базе Rolldown (то есть Vite 8+). Вы можете использовать его, чтобы определить, работает ли текущий экземпляр Vite на базе Rolldown:
>
> ```ts
> function versionCheckPlugin(): Plugin {
>   return {
>     name: 'version-check',
>     buildStart() {
>       if (this.meta.rolldownVersion) {
>         // делать что-то только при работе на Vite на базе Rolldown
>       } else {
>         // делать что-то другое при работе на Vite на базе Rollup
>       }
>     },
>   }
> }
> ```

## Метаданные выходного бандла

Во время сборки Vite дополняет объекты выходных данных сборки Rolldown специфичным для Vite полем `viteMetadata`.

Оно доступно через:

- `RenderedChunk` (например, в `renderChunk` и `augmentChunkHash`)
- `OutputChunk` и `OutputAsset` (например, в `generateBundle` и `writeBundle`)

`viteMetadata` предоставляет:

- `viteMetadata.importedCss: Set<string>`
- `viteMetadata.importedAssets: Set<string>`

Это полезно при написании плагинов, которым нужно исследовать сгенерированный CSS и статические ресурсы, не полагаясь на [`build.manifest`](https://vite.dev/config/build-options#build-manifest).

Пример:

**`vite.config.ts`**

```ts
function outputMetadataPlugin(): Plugin {
  return {
    name: 'output-metadata-plugin',
    enforce: 'post',
    generateBundle(_, bundle) {
      for (const output of Object.values(bundle)) {
        const css = output.viteMetadata?.importedCss
        const assets = output.viteMetadata?.importedAssets
        if (!css?.size && !assets?.size) continue

        console.log(output.fileName, {
          css: css ? [...css] : [],
          assets: assets ? [...assets] : [],
        })
      }
    },
  }
}
```

## Порядок плагинов

Плагин Vite может дополнительно указать свойство `enforce` (по аналогии с загрузчиками webpack), чтобы скорректировать порядок своего применения. Значением `enforce` может быть либо `"pre"`, либо `"post"`. Разрешённые плагины будут располагаться в следующем порядке:

- Алиасы (Alias)
- Пользовательские плагины с `enforce: 'pre'`
- Основные плагины Vite
- Пользовательские плагины без значения enforce
- Плагины сборки Vite
- Пользовательские плагины с `enforce: 'post'`
- Плагины Vite, выполняемые после сборки (минификация, манифест, отчёты)

Обратите внимание, что это не то же самое, что порядок хуков, — они по-прежнему отдельно подчиняются своему [атрибуту `order`](https://rolldown.rs/reference/TypeAlias.ObjectHook#order), как обычно для хуков Rolldown.

## Условное применение

По умолчанию плагины вызываются как для serve, так и для build. В случаях, когда плагин должен применяться условно — только во время serve или build, — используйте свойство `apply`, чтобы вызывать его только во время `'build'` или `'serve'`:

```js
function myPlugin() {
  return {
    name: 'build-only',
    apply: 'build', // или 'serve'
  }
}
```

Для более точного контроля можно также использовать функцию:

```js
apply(config, { command }) {
  // применять только при сборке, но не для SSR
  return command === 'build' && !config.build.ssr
}
```

## Совместимость с плагинами Rolldown

Значительная часть плагинов Rolldown / Rollup будет работать напрямую как плагин Vite (например, `@rollup/plugin-alias` или `@rollup/plugin-json`), но не все, поскольку некоторые хуки плагинов не имеют смысла в контексте несобранного dev-сервера.

В целом, пока плагин Rolldown / Rollup удовлетворяет следующим критериям, он должен работать как плагин Vite:

- Он не использует хук [`moduleParsed`](https://rolldown.rs/reference/Interface.Plugin#moduleparsed).
- Он не полагается на специфичные для Rolldown опции, такие как [`transform.inject`](https://rolldown.rs/reference/InputOptions.transform#inject).
- У него нет сильной связанности между хуками фазы сборки бандла и хуками фазы вывода.

Если плагин Rolldown / Rollup имеет смысл только для фазы сборки, то его можно указать в `build.rolldownOptions.plugins`. Он будет работать так же, как плагин Vite с `enforce: 'post'` и `apply: 'build'`.

Вы также можете дополнить существующий плагин Rolldown / Rollup свойствами, специфичными только для Vite:

**`vite.config.js`**

```js
import example from 'rolldown-plugin-example'
import { defineConfig } from 'vite'

export default defineConfig({
  plugins: [
    {
      ...example(),
      enforce: 'post',
      apply: 'build',
    },
  ],
})
```

## Нормализация путей

Vite нормализует пути при разрешении id, используя разделители POSIX ( / ) и сохраняя при этом том (volume) в Windows. С другой стороны, Rollup по умолчанию оставляет разрешённые пути нетронутыми, поэтому разрешённые id имеют разделители win32 ( \\ ) в Windows. Однако плагины Rollup внутренне используют [служебную функцию `normalizePath`](https://github.com/rollup/plugins/tree/master/packages/pluginutils#normalizepath) из `@rollup/pluginutils`, которая преобразует разделители в POSIX перед выполнением сравнений. Это означает, что когда эти плагины используются в Vite, конфигурационные шаблоны `include` и `exclude` и другие подобные сравнения путей с разрешёнными id работают корректно.

Поэтому для плагинов Vite при сравнении путей с разрешёнными id важно сначала нормализовать пути, чтобы они использовали разделители POSIX. Эквивалентная служебная функция `normalizePath` экспортируется из модуля `vite`.

```js
import { normalizePath } from 'vite'

normalizePath('foo\\bar') // 'foo/bar'
normalizePath('foo/bar') // 'foo/bar'
```

## Фильтрация, шаблоны include/exclude

Vite предоставляет [функцию `createFilter` из `@rollup/pluginutils`](https://github.com/rollup/plugins/tree/master/packages/pluginutils#createfilter), чтобы поощрять специфичные для Vite плагины и интеграции использовать стандартный шаблон фильтрации include/exclude, который также используется в самом ядре Vite.

### Фильтры хуков

Rolldown представил [возможность фильтрации хуков](https://rolldown.rs/apis/plugin-api/hook-filters), чтобы уменьшить накладные расходы на взаимодействие между средами выполнения Rust и JavaScript. Эта возможность позволяет плагинам задавать шаблоны, определяющие, когда следует вызывать хуки, повышая производительность за счёт избегания ненужных вызовов хуков.

Это также поддерживается в Rollup 4.38.0+ и Vite 6.3.0+. Чтобы сделать ваш плагин обратно совместимым со старыми версиями, обязательно выполняйте фильтрацию также и внутри обработчиков хуков.

```js
export default function myPlugin() {
  const jsFileRegex = /\.js$/

  return {
    name: 'my-plugin',
    // Пример: вызывать transform только для .js-файлов
    transform: {
      filter: {
        id: jsFileRegex,
      },
      handler(code, id) {
        // Дополнительная проверка для обратной совместимости
        if (!jsFileRegex.test(id)) return null

        return {
          code: transformCode(code),
          map: null,
        }
      },
    },
  }
}
```

> **Совет**
> [`@rolldown/pluginutils`](https://www.npmjs.com/package/@rolldown/pluginutils) экспортирует некоторые утилиты для фильтров хуков, такие как `exactRegex` и `prefixRegex`. Для удобства они также реэкспортируются из `rolldown/filter`.

## Информация об import map для чанков

> **Информация: Экспериментально**
> Эта возможность является экспериментальной и может измениться в будущем.

Когда включена опция [`build.chunkImportMap`](https://vite.dev/config/build-options#build-chunkimportmap), инструкции импорта в сгенерированных чанках будут использовать уникальный ID для каждого чанка вместо пути к файлу.

Чтобы получить сопоставление ID чанка с путём к файлу, вы можете обратиться к import map, добавленной в бандл, в хуке `generateBundle` или хуке `writeBundle`. Import map имеет имя, заданное в [`build.rolldownOptions.experimental.chunkImportMap.fileName`](https://rolldown.rs/reference/InputOptions.experimental#chunkimportmap) (по умолчанию `importmap.json`).

```ts
function accessImportMap() {
  let config: ResolvedConfig
  return {
    name: 'access-import-map',
    configResolved(resolvedConfig) {
      config = resolvedConfig
    },
    generateBundle(options, bundle) {
      const chunkImportMap =
        config.build.rolldownOptions.experimental?.chunkImportMap
      if (chunkImportMap) {
        const importMapFilename =
          typeof chunkImportMap === 'object' && chunkImportMap.fileName
            ? chunkImportMap.fileName
            : 'importmap.json'
        const importMap = bundle[importMapFilename]! as OutputAsset
        const mapping = JSON.parse(importMap.source).imports
        console.log(mapping)
        // { "./entry.hash1.js": "./entry.hash2.js" }
      }
    },
  }
}
```

## Взаимодействие клиента и сервера

Начиная с Vite 2.9, мы предоставляем ряд утилит для плагинов, помогающих организовать взаимодействие с клиентами.

### От сервера к клиенту

На стороне плагина мы можем использовать `server.ws.send`, чтобы транслировать события клиенту:

**`vite.config.js`**

```js
export default defineConfig({
  plugins: [
    {
      // ...
      configureServer(server) {
        server.ws.on('connection', () => {
          server.ws.send('my:greetings', { msg: 'hello' })
        })
      },
    },
  ],
})
```

> **Совет: Примечание**
> Мы рекомендуем **всегда добавлять префикс** к именам ваших событий, чтобы избежать коллизий с другими плагинами.

На стороне клиента используйте [`hot.on`](https://vite.dev/guide/api-hmr.html#hot-on-event-cb), чтобы слушать события:

```ts
// на стороне клиента
if (import.meta.hot) {
  import.meta.hot.on('my:greetings', (data) => {
    console.log(data.msg) // hello
  })
}
```

### От клиента к серверу

Чтобы отправлять события от клиента к серверу, мы можем использовать [`hot.send`](https://vite.dev/guide/api-hmr.html#hot-send-event-data):

```ts
// на стороне клиента
if (import.meta.hot) {
  import.meta.hot.send('my:from-client', { msg: 'Hey!' })
}
```

Затем используйте `server.ws.on` и слушайте события на стороне сервера:

**`vite.config.js`**

```js
export default defineConfig({
  plugins: [
    {
      // ...
      configureServer(server) {
        server.ws.on('my:from-client', (data, client) => {
          console.log('Message from client:', data.msg) // Hey!
          // ответить только этому клиенту (если нужно)
          client.send('my:ack', { msg: 'Hi! I got your message!' })
        })
      },
    },
  ],
})
```

### TypeScript для пользовательских событий

Внутренне Vite выводит тип полезной нагрузки (payload) из интерфейса `CustomEventMap`; можно типизировать пользовательские события, расширив этот интерфейс:

> **Совет: Примечание**
> Обязательно указывайте расширение `.d.ts` при задании файлов деклараций TypeScript. Иначе TypeScript может не понять, какой файл пытается расширить модуль.

**`events.d.ts`**

```ts
import 'vite/types/customEvent.d.ts'

declare module 'vite/types/customEvent.d.ts' {
  interface CustomEventMap {
    'custom:foo': { msg: string }
    // 'event-key': payload
  }
}
```

Это расширение интерфейса используется `InferCustomEventPayload<T>` для вывода типа полезной нагрузки для события `T`. Дополнительную информацию о том, как используется этот интерфейс, смотрите в [документации по HMR API](https://vite.dev/guide/api-hmr#hmr-api).

```ts
type CustomFooPayload = InferCustomEventPayload<'custom:foo'>
import.meta.hot?.on('custom:foo', (payload) => {
  // Тип payload будет { msg: string }
})
import.meta.hot?.on('unknown:event', (payload) => {
  // Тип payload будет any
})
```
