# HMR API

> **Совет**
> Это клиентский HMR API. О том, как обрабатывать HMR-обновление в плагинах, см. [handleHotUpdate](https://vite.dev/guide/api-plugin#handlehotupdate).
>
> Ручной HMR API в первую очередь предназначен для авторов фреймворков и инструментария. Как конечному пользователю, вам, скорее всего, HMR уже настроен во фреймворк-специфичных стартовых шаблонах.

Vite предоставляет свой ручной HMR API через специальный объект `import.meta.hot`:

```ts
interface ImportMeta {
  readonly hot?: ViteHotContext
}

interface ViteHotContext {
  readonly data: any

  accept(): void
  accept(cb: (mod: ModuleNamespace | undefined) => void): void
  accept(dep: string, cb: (mod: ModuleNamespace | undefined) => void): void
  accept(
    deps: readonly string[],
    cb: (mods: Array<ModuleNamespace | undefined>) => void,
  ): void

  dispose(cb: (data: any) => void): void
  prune(cb: (data: any) => void): void
  invalidate(message?: string): void

  on<T extends CustomEventName>(
    event: T,
    cb: (payload: InferCustomEventPayload<T>) => void,
  ): void
  off<T extends CustomEventName>(
    event: T,
    cb: (payload: InferCustomEventPayload<T>) => void,
  ): void
  send<T extends CustomEventName>(
    event: T,
    data?: InferCustomEventPayload<T>,
  ): void
}
```

## Обязательная условная защита

Прежде всего убедитесь, что любое использование HMR API обёрнуто в условный блок, чтобы этот код можно было удалить (tree-shaking) в продакшене:

```js
if (import.meta.hot) {
  // код HMR
}
```

## IntelliSense для TypeScript

Vite предоставляет определения типов для `import.meta.hot` в [`vite/client.d.ts`](https://github.com/vitejs/vite/blob/main/packages/vite/client.d.ts). Вы можете добавить «vite/client» в `tsconfig.json`, чтобы TypeScript подхватил определения типов:

**tsconfig.json**

```json
{
  "compilerOptions": {
    "types": ["vite/client"]
  }
}
```

## `hot.accept(cb)`

Чтобы модуль принимал обновления самого себя (self-accept), используйте `import.meta.hot.accept` с колбэком, который получает обновлённый модуль:

```js
export const count = 1

if (import.meta.hot) {
  import.meta.hot.accept((newModule) => {
    if (newModule) {
      // newModule равен undefined, если произошла ошибка SyntaxError
      console.log('updated: count is now ', newModule.count)
    }
  })
}
```

Модуль, который «принимает» горячие обновления, считается **границей HMR** (HMR boundary).

```dot

  rankdir=RL

  root -> parent [label="импортирует" style=dashed]

```

HMR в Vite на самом деле не подменяет изначально импортированный модуль: если модуль-граница HMR реэкспортирует импорты из зависимости, то именно он отвечает за обновление этих реэкспортов (и такие экспорты должны использовать `let`). Кроме того, импортёры выше по цепочке от модуля-границы не будут уведомлены об изменении. Такая упрощённая реализация HMR достаточна для большинства сценариев разработки и при этом позволяет нам избежать затратной работы по генерации прокси-модулей.

Vite требует, чтобы вызов этой функции присутствовал в исходном коде именно в виде `import.meta.hot.accept(` (с учётом пробелов), — только тогда модуль сможет принимать обновления. Это требование статического анализа, который Vite выполняет для включения поддержки HMR у модуля.

## `hot.accept(deps, cb)`

Модуль также может принимать обновления от своих прямых зависимостей, не перезагружая себя:

```js
import { foo } from './foo.js'

foo()

if (import.meta.hot) {
  import.meta.hot.accept('./foo.js', (newFoo) => {
    // колбэк получает обновлённый модуль './foo.js'
    newFoo?.foo()
  })

  // Можно также принимать массив модулей-зависимостей:
  import.meta.hot.accept(
    ['./foo.js', './bar.js'],
    ([newFooModule, newBarModule]) => {
      // Колбэк получает массив, в котором не равен null только обновлённый
      // модуль. Если обновление прошло неуспешно (например, синтаксическая
      // ошибка), массив пуст
    },
  )
}
```

## `hot.dispose(cb)`

Самопринимающий (self-accepting) модуль или модуль, рассчитывающий на то, что его примут другие, может использовать `hot.dispose` для очистки любых устойчивых побочных эффектов, созданных его обновлённой копией:

```js
function setupSideEffect() {}

setupSideEffect()

if (import.meta.hot) {
  import.meta.hot.dispose((data) => {
    // очистка побочного эффекта
  })
}
```

## `hot.prune(cb)`

Регистрирует колбэк, который будет вызван, когда модуль перестанет импортироваться на странице. В отличие от `hot.dispose`, это можно использовать, если исходный код сам очищает побочные эффекты при обновлениях, а очистка нужна только при удалении модуля со страницы. Сейчас Vite использует это для импортов `.css`.

```js
function setupOrReuseSideEffect() {}

setupOrReuseSideEffect()

if (import.meta.hot) {
  import.meta.hot.prune((data) => {
    // очистка побочного эффекта
  })
}
```

## `hot.data`

Объект `import.meta.hot.data` сохраняется между разными экземплярами одного и того же обновляемого модуля. Его можно использовать для передачи информации от предыдущей версии модуля к следующей.

Обратите внимание, что переприсваивание самого `data` не поддерживается. Вместо этого следует изменять свойства объекта `data`, чтобы информация, добавленная другими обработчиками, сохранялась.

```js
// ok
import.meta.hot.data.someValue = 'hello'

// не поддерживается
import.meta.hot.data = { someValue: 'hello' }
```

## `hot.decline()`

В настоящее время это пустая операция (noop), оставленная для обратной совместимости. Это может измениться в будущем, если для неё найдётся новое применение. Чтобы указать, что модуль не поддерживает горячее обновление, используйте `hot.invalidate()`.

## `hot.invalidate(message?: string)`

Самопринимающий модуль во время выполнения может обнаружить, что не может обработать HMR-обновление, и тогда обновление нужно принудительно распространить на импортёров. Вызвав `import.meta.hot.invalidate()`, вы заставите HMR-сервер инвалидировать импортёров вызывающего модуля, как если бы вызывающий модуль не был самопринимающим. При этом сообщение будет записано и в консоль браузера, и в терминал. Вы можете передать сообщение, чтобы дать контекст о причине инвалидации.

Обратите внимание, что вы всегда должны вызывать `import.meta.hot.accept`, даже если планируете сразу же после этого вызвать `invalidate`, иначе HMR-клиент не будет отслеживать будущие изменения самопринимающего модуля. Чтобы ясно выразить свои намерения, рекомендуем вызывать `invalidate` внутри колбэка `accept`, например так:

```js
import.meta.hot.accept((module) => {
  // Вы можете использовать новый экземпляр модуля, чтобы решить, нужна ли инвалидация.
  if (cannotHandleUpdate(module)) {
    import.meta.hot.invalidate()
  }
})
```

## `hot.on(event, cb)`

Прослушивает HMR-событие.

Следующие HMR-события Vite отправляет автоматически:

- `'vite:beforeUpdate'` — когда обновление вот-вот будет применено (например, модуль будет заменён)
- `'vite:afterUpdate'` — когда обновление только что было применено (например, модуль был заменён)
- `'vite:beforeFullReload'` — когда вот-вот произойдёт полная перезагрузка
- `'vite:beforePrune'` — когда модули, которые больше не нужны, вот-вот будут удалены (pruned)
- `'vite:invalidate'` — когда модуль инвалидируется через `import.meta.hot.invalidate()`
- `'vite:error'` — когда возникает ошибка (например, синтаксическая)
- `'vite:ws:disconnect'` — когда соединение WebSocket потеряно
- `'vite:ws:connect'` — когда соединение WebSocket (повторно) установлено

Пользовательские HMR-события также могут отправляться из плагинов. Подробнее см. [handleHotUpdate](https://vite.dev/guide/api-plugin#handlehotupdate).

## `hot.off(event, cb)`

Удаляет колбэк из числа слушателей события.

## `hot.send(event, data)`

Отправляет пользовательские события обратно на dev-сервер Vite.

Если вызвать до установления соединения, данные будут буферизованы и отправлены, как только соединение будет установлено.

Подробнее см. [Взаимодействие клиента и сервера](https://vite.dev/guide/api-plugin.html#client-server-communication), включая раздел [Типизация пользовательских событий](https://vite.dev/guide/api-plugin.html#typescript-for-custom-events).
