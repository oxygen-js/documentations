# Справочник по конфигурации

При запуске `vite` из командной строки Vite автоматически попытается найти файл конфигурации с именем `vite.config.js` в [корне проекта](https://vite.dev/guide/#index-html-and-project-root) (поддерживаются и другие расширения JS и TS).

Самый базовый файл конфигурации выглядит так:

```js
export default {
  // опции конфигурации
}
```

Обратите внимание, что Vite поддерживает синтаксис ES-модулей в файле конфигурации, даже если проект не использует нативные Node ESM, например `"type": "module"` в `package.json`. В этом случае файл конфигурации автоматически предварительно обрабатывается перед загрузкой.

Вы также можете явно указать используемый файл конфигурации с помощью CLI-опции `--config` (путь разрешается относительно `cwd`):

```bash
vite --config my-config.js
```

Посмотрите интерактивный урок на Scrimba

> **ЗАГРУЗКА КОНФИГУРАЦИИ**
> По умолчанию Vite использует [Rolldown](https://rolldown.rs/) для того, чтобы собрать конфигурацию в бандл во временный файл и загрузить его. Это может вызывать проблемы при импорте TypeScript-файлов в монорепозитории. Если вы столкнётесь с какими-либо проблемами при таком подходе, можно указать `--configLoader runner`, чтобы вместо этого использовать [module runner](https://vite.dev/guide/api-environment-runtimes.html#modulerunner), который не будет создавать временную конфигурацию и будет преобразовывать любые файлы на лету. Учтите, что module runner не поддерживает CJS в файлах конфигурации, но внешние CJS-пакеты должны работать как обычно.
>
> Как вариант, если вы используете окружение с поддержкой TypeScript (например, `node --experimental-strip-types`), либо пишете только на чистом JavaScript, можно указать `--configLoader native`, чтобы использовать нативную среду выполнения окружения для загрузки файла конфигурации. Учтите, что обновления модулей, импортируемых файлом конфигурации, не отслеживаются, а значит, не приведут к автоматическому перезапуску сервера Vite.

## Автодополнение конфигурации

Поскольку Vite поставляется с типами TypeScript, вы можете задействовать автодополнение вашей IDE с помощью подсказок типов jsdoc:

```js
/** @type {import('vite').UserConfig} */
export default {
  // ...
}
```

Как вариант, можно использовать помощник `defineConfig`, который обеспечивает автодополнение без необходимости в аннотациях jsdoc:

```js
import { defineConfig } from 'vite'

export default defineConfig({
  // ...
})
```

Vite также поддерживает файлы конфигурации на TypeScript. Вы можете использовать `vite.config.ts` с помощником `defineConfig`, показанным выше, либо с оператором `satisfies`:

```ts
import type { UserConfig } from 'vite'

export default {
  // ...
} satisfies UserConfig
```

## Условная конфигурация

Если конфигурации необходимо условно определять опции в зависимости от команды (`serve` или `build`), от используемого [режима (mode)](https://vite.dev/guide/env-and-mode#modes), от того, является ли это SSR-сборкой (`isSsrBuild`), или от того, идёт ли предпросмотр сборки (`isPreview`), она может экспортировать функцию:

```js
export default defineConfig(({ command, mode, isSsrBuild, isPreview }) => {
  if (command === 'serve') {
    return {
      // конфигурация, специфичная для разработки
    }
  } else {
    // command === 'build'
    return {
      // конфигурация, специфичная для сборки
    }
  }
})
```

Важно отметить, что в API Vite значение `command` равно `serve` во время разработки (в CLI [`vite`](https://vite.dev/guide/cli#vite), `vite dev` и `vite serve` являются псевдонимами) и `build` при сборке для продакшена ([`vite build`](https://vite.dev/guide/cli#vite-build)).

`isSsrBuild` и `isPreview` — это дополнительные необязательные флаги для различения разновидностей команд `build` и `serve` соответственно. Некоторые инструменты, загружающие конфигурацию Vite, могут не поддерживать эти флаги и вместо значения будут передавать `undefined`. Поэтому рекомендуется использовать явное сравнение с `true` и `false`.

## Асинхронная конфигурация

Если конфигурации необходимо вызывать асинхронные функции, она может экспортировать асинхронную функцию. Эту асинхронную функцию также можно передать в `defineConfig` для улучшенной поддержки автодополнения:

```js
export default defineConfig(async ({ command, mode }) => {
  const data = await asyncFunction()
  return {
    // конфигурация vite
  }
})
```

## Использование переменных окружения в конфигурации

Переменные окружения, доступные во время вычисления самой конфигурации, — это только те, что уже существуют в текущем окружении процесса (`process.env`). Vite намеренно откладывает загрузку любых файлов `.env*` до тех пор, пока пользовательская конфигурация не будет разрешена, потому что набор загружаемых файлов зависит от опций конфигурации, таких как [`root`](https://vite.dev/guide/#index-html-and-project-root) и [`envDir`](https://vite.dev/config/shared-options.md#envdir), а также от итогового `mode`.

Это означает, что переменные, определённые в `.env`, `.env.local`, `.env.[mode]` или `.env.[mode].local`, **не** внедряются автоматически в `process.env` во время выполнения вашего `vite.config.*`. Они _действительно_ загружаются автоматически позже и предоставляются коду приложения через `import.meta.env` (с фильтром по префиксу `VITE_` по умолчанию) ровно так, как описано в разделе [Переменные окружения и режимы](https://vite.dev/guide/env-and-mode.html). Поэтому, если вам нужно лишь передать значения из файлов `.env*` в приложение, ничего вызывать в конфигурации не требуется.

Если же значения из файлов `.env*` должны влиять на саму конфигурацию (например, чтобы задать `server.port`, условно включить плагины или вычислить замены `define`), вы можете загрузить их вручную с помощью экспортируемого помощника [`loadEnv`](https://vite.dev/guide/api-javascript.html#loadenv).

```js
import { defineConfig, loadEnv } from 'vite'

export default defineConfig(({ mode }) => {
  // Загружаем env-файл на основе `mode` в текущем рабочем каталоге.
  // Установите третий параметр в '', чтобы загрузить все переменные
  // окружения независимо от префикса `VITE_`.
  const env = loadEnv(mode, process.cwd(), '')
  return {
    define: {
      // Задаём явную константу уровня приложения, выведенную из env-переменной.
      __APP_ENV__: JSON.stringify(env.APP_ENV),
    },
    // Пример: используем env-переменную, чтобы условно задать порт dev-сервера.
    server: {
      port: env.APP_PORT ? Number(env.APP_PORT) : 5173,
    },
  }
})
```

## Отладка файла конфигурации в VS Code

При поведении по умолчанию `--configLoader bundle` Vite записывает сгенерированный временный файл конфигурации в папку `node_modules/.vite-temp`, и при установке точки останова для отладки в файле конфигурации Vite возникнет ошибка «файл не найден». Чтобы решить эту проблему, добавьте следующую конфигурацию в `.vscode/settings.json`:

```json
{
  "debug.javascript.terminalOptions": {
    "resolveSourceMapLocations": [
      "${workspaceFolder}/**",
      "!**/node_modules/**",
      "**/node_modules/.vite-temp/**"
    ]
  }
}
```
