# Производительность

Хотя Vite быстр по умолчанию, по мере роста требований к проекту могут возникать проблемы с производительностью. Это руководство поможет вам выявить и устранить распространённые проблемы с производительностью, такие как:

- Медленный запуск сервера
- Медленная загрузка страниц
- Медленная сборка

## Проверьте настройки браузера

Некоторые расширения браузера могут вмешиваться в запросы и замедлять запуск и перезагрузку крупных приложений, особенно при использовании инструментов разработчика браузера. В таких случаях мы рекомендуем создать отдельный профиль для разработки без расширений или переключиться в режим инкогнито во время работы с dev-сервером Vite. Режим инкогнито также должен быть быстрее, чем обычный профиль без расширений.

Dev-сервер Vite выполняет жёсткое кэширование предварительно собранных зависимостей и реализует быстрые ответы 304 для исходного кода. Отключение кэша при открытых инструментах разработчика браузера может сильно повлиять на время запуска и полной перезагрузки страницы. Убедитесь, что опция «Disable Cache» не включена, пока вы работаете с сервером Vite.

## Проверьте настроенные плагины Vite

Внутренние и официальные плагины Vite оптимизированы так, чтобы выполнять минимально возможный объём работы, обеспечивая при этом совместимость с широкой экосистемой. Например, преобразования кода используют регулярные выражения в режиме разработки, но выполняют полный разбор при сборке для гарантии корректности.

Однако производительность плагинов сообщества находится вне контроля Vite, что может повлиять на опыт разработки. Вот несколько моментов, на которые стоит обратить внимание при использовании дополнительных плагинов Vite:

1. Крупные зависимости, которые используются только в определённых случаях, следует импортировать динамически, чтобы сократить время запуска Node.js. Примеры рефакторинга: [vite-plugin-react#212](https://github.com/vitejs/vite-plugin-react/pull/212) и [vite-plugin-pwa#224](https://github.com/vite-pwa/vite-plugin-pwa/pull/244).

2. Хуки `buildStart`, `config` и `configResolved` не должны выполнять длительные и объёмные операции. Эти хуки ожидаются (await) во время запуска dev-сервера, что задерживает момент, когда вы сможете открыть сайт в браузере.

3. Хуки `resolveId`, `load` и `transform` могут приводить к тому, что некоторые файлы загружаются медленнее других. Хотя иногда этого не избежать, всё же стоит поискать возможные места для оптимизации. Например, проверять, содержит ли `code` определённое ключевое слово, или соответствует ли `id` определённому расширению, прежде чем выполнять полное преобразование.

   Чем дольше преобразуется файл, тем значительнее будет каскад запросов (request waterfall) при загрузке сайта в браузере.

   Вы можете проверить, сколько времени занимает преобразование файла, с помощью `vite --debug plugin-transform` или [vite-plugin-inspect](https://github.com/antfu/vite-plugin-inspect). Обратите внимание, что асинхронные операции склонны давать неточные измерения времени, поэтому относитесь к числам как к грубой оценке, но они всё же должны выявить наиболее затратные операции.

> **Совет: Профилирование**
>
> Вы можете запустить `vite --profile`, открыть сайт и нажать `p + enter` в терминале, чтобы записать `.cpuprofile`. Затем такой инструмент, как [speedscope](https://www.speedscope.app), можно использовать для анализа профиля и выявления узких мест. Вы также можете [поделиться профилями](https://chat.vite.dev) с командой Vite, чтобы помочь нам выявить проблемы с производительностью.

## Сократите количество операций разрешения путей

Разрешение путей импорта может быть затратной операцией, если её худший случай встречается часто. Например, Vite поддерживает «угадывание» путей импорта с помощью опции [`resolve.extensions`](https://vite.dev/config/shared-options#resolve-extensions), которая по умолчанию равна `['.mjs', '.js', '.mts', '.ts', '.jsx', '.tsx', '.json']`.

Когда вы пытаетесь импортировать `./Component.jsx` через `import './Component'`, Vite выполнит следующие шаги для разрешения пути:

1. Проверить, существует ли `./Component` — нет.
2. Проверить, существует ли `./Component.mjs` — нет.
3. Проверить, существует ли `./Component.js` — нет.
4. Проверить, существует ли `./Component.mts` — нет.
5. Проверить, существует ли `./Component.ts` — нет.
6. Проверить, существует ли `./Component.jsx` — да!

Как видно, для разрешения одного пути импорта требуется в общей сложности 6 проверок файловой системы. Чем больше у вас неявных импортов, тем больше времени в сумме уходит на разрешение путей.

Поэтому обычно лучше явно указывать пути импорта, например `import './Component.jsx'`. Вы также можете сузить список для `resolve.extensions`, чтобы сократить общее количество проверок файловой системы, но при этом нужно убедиться, что он работает и для файлов в `node_modules`.

Если вы автор плагина, обязательно вызывайте [`this.resolve`](https://rollupjs.org/plugin-development/#this-resolve) только при необходимости, чтобы сократить количество проверок, описанных выше.

> **Совет: TypeScript**
>
> Если вы используете TypeScript, включите `"moduleResolution": "bundler"` и `"allowImportingTsExtensions": true` в `compilerOptions` вашего `tsconfig.json`, чтобы использовать расширения `.ts` и `.tsx` напрямую в коде.

## Избегайте barrel-файлов

Barrel-файлы — это файлы, которые повторно экспортируют API других файлов из того же каталога. Например:

**`src/utils/index.js`**

```js
export * from './color.js'
export * from './dom.js'
export * from './slash.js'
```

Когда вы импортируете лишь отдельный API, например `import { slash } from './utils'`, все файлы из этого barrel-файла нужно загрузить и преобразовать, поскольку они могут содержать API `slash`, а также могут содержать побочные эффекты, выполняющиеся при инициализации. Это означает, что при первоначальной загрузке страницы вы загружаете больше файлов, чем нужно, что приводит к более медленной загрузке страницы.

По возможности следует избегать barrel-файлов и импортировать отдельные API напрямую, например `import { slash } from './utils/slash.js'`. Больше информации можно найти в [issue #8237](https://github.com/vitejs/vite/issues/8237).

## Разогревайте часто используемые файлы

Dev-сервер Vite преобразует файлы только по запросу браузера, что позволяет ему быстро запускаться и применять преобразования только к используемым файлам. Он также может предварительно преобразовывать файлы, если предполагает, что определённые файлы вскоре будут запрошены. Однако каскад запросов всё равно может возникнуть, если некоторые файлы преобразуются дольше других. Например:

Дан граф импортов, где левый файл импортирует правый:

```
main.js -> BigComponent.vue -> big-utils.js -> large-data.json
```

Отношение импорта можно узнать только после того, как файл преобразован. Если `BigComponent.vue` преобразуется какое-то время, `big-utils.js` придётся дожидаться своей очереди, и так далее. Это вызывает внутренний каскад даже при встроенном предварительном преобразовании.

Vite позволяет разогревать файлы, которые, как вы знаете, используются часто, например `big-utils.js`, с помощью опции [`server.warmup`](https://vite.dev/config/server-options#server-warmup). Таким образом, `big-utils.js` будет готов и закэширован, чтобы его можно было немедленно отдать при запросе.

Найти часто используемые файлы можно, запустив `vite --debug transform` и изучив логи:

```bash
vite:transform 28.72ms /@vite/client +1ms
vite:transform 62.95ms /src/components/BigComponent.vue +1ms
vite:transform 102.54ms /src/utils/big-utils.js +1ms
```

**`vite.config.js`**

```js
export default defineConfig({
  server: {
    warmup: {
      clientFiles: [
        './src/components/BigComponent.vue',
        './src/utils/big-utils.js',
      ],
    },
  },
})
```

Обратите внимание, что разогревать следует только часто используемые файлы, чтобы не перегружать dev-сервер Vite при запуске. Дополнительную информацию смотрите в описании опции [`server.warmup`](https://vite.dev/config/server-options#server-warmup).

Использование [`--open` или `server.open`](https://vite.dev/config/server-options#server-open) также даёт прирост производительности, поскольку Vite автоматически разогреет точку входа вашего приложения или указанный для открытия URL.

## Используйте более лёгкий или нативный инструментарий

Сохранение высокой скорости Vite при растущей кодовой базе сводится к сокращению объёма работы для исходных файлов (JS/TS/CSS).

Примеры того, как делать меньше работы:

- Используйте CSS вместо Sass/Less/Stylus, когда это возможно (вложенность может обрабатываться PostCSS / Lightning CSS)
- Не преобразуйте SVG в компоненты UI-фреймворков (React, Vue и т. д.). Вместо этого импортируйте их как строки или URL.

Примеры использования нативного инструментария:

Хотя ядро Vite основано на нативном инструментарии, некоторые возможности по умолчанию всё же используют ненативный инструментарий для обеспечения лучшей совместимости и набора функций. Но для более крупных приложений это может стоить затрат.

- Попробуйте экспериментальную поддержку [LightningCSS](https://github.com/vitejs/vite/discussions/13835)
