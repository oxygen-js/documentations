# Возможности

На самом базовом уровне разработка с использованием Vite не так уж отличается от использования статического файлового сервера. Однако Vite предоставляет множество улучшений поверх нативных ESM-импортов для поддержки различных возможностей, которые обычно встречаются в конфигурациях на основе бандлеров.

## Разрешение и предварительная сборка npm-зависимостей

Нативные ES-импорты не поддерживают импорты «голых» модулей (bare module imports) вроде следующего:

```js
import { someMethod } from 'my-dep'
```

Приведённый выше импорт вызовет ошибку в браузере. Vite обнаруживает такие импорты голых модулей во всех обслуживаемых исходных файлах и выполняет следующее:

1. [Предварительно собирает](https://vite.dev/guide/dep-pre-bundling) их, чтобы повысить скорость загрузки страницы и преобразовать модули CommonJS / UMD в ESM. Шаг предварительной сборки выполняется с помощью [Rolldown](https://rolldown.rs/) и делает время холодного старта Vite значительно быстрее, чем у любого бандлера на основе JavaScript.

2. Переписывает импорты в корректные URL вроде `/node_modules/.vite/deps/my-dep.js?v=f3sf2ebd`, чтобы браузер мог правильно их импортировать.

**Зависимости жёстко кэшируются**

Vite кэширует запросы к зависимостям через HTTP-заголовки, поэтому, если вы хотите локально отредактировать/отладить зависимость, следуйте шагам [отсюда](https://vite.dev/guide/dep-pre-bundling#browser-cache).

## Горячая замена модулей

Vite предоставляет [HMR API](https://vite.dev/guide/api-hmr) поверх нативного ESM. Фреймворки с возможностями HMR могут использовать этот API для мгновенных, точечных обновлений без перезагрузки страницы и без сброса состояния приложения. Vite предоставляет встроенные HMR-интеграции для [однофайловых компонентов Vue (Single File Components)](https://github.com/vitejs/vite-plugin-vue/tree/main/packages/plugin-vue) и [React Fast Refresh](https://github.com/vitejs/vite-plugin-react/tree/main/packages/plugin-react). Также есть официальные интеграции для Preact через [@prefresh/vite](https://github.com/JoviDeCroock/prefresh/tree/main/packages/vite).

Обратите внимание, что настраивать это вручную не нужно — когда вы [создаёте приложение через `create-vite`](https://vite.dev/guide/), выбранные шаблоны уже будут иметь всё это предварительно настроенным.

## TypeScript

Vite поддерживает импорт `.ts` файлов «из коробки».

### Только транспиляция

Обратите внимание, что Vite выполняет только транспиляцию `.ts` файлов и **НЕ** выполняет проверку типов. Предполагается, что проверка типов выполняется вашей IDE и процессом сборки.

Причина, по которой Vite не выполняет проверку типов в рамках процесса преобразования, в том, что эти две задачи работают принципиально по-разному. Транспиляция может работать на уровне отдельных файлов и идеально согласуется с моделью компиляции по требованию (on-demand) в Vite. Для сравнения, проверка типов требует знания всего графа модулей. Втискивание проверки типов в конвейер преобразования Vite неизбежно поставит под угрозу преимущества Vite в скорости.

Задача Vite — привести ваши исходные модули к форме, которая может выполняться в браузере как можно быстрее. Для этого мы рекомендуем отделять проверки статического анализа от конвейера преобразования Vite. Этот принцип применим и к другим проверкам статического анализа, таким как ESLint.

- Для продакшн-сборок вы можете запускать `tsc --noEmit` в дополнение к команде сборки Vite.

- Во время разработки, если вам нужно нечто большее, чем подсказки IDE, мы рекомендуем запускать `tsc --noEmit --watch` в отдельном процессе или использовать [vite-plugin-checker](https://github.com/fi3ework/vite-plugin-checker), если вы предпочитаете, чтобы ошибки типов сообщались напрямую в браузере.

Vite использует [Oxc Transformer](https://oxc.rs/docs/guide/usage/transformer.html) для транспиляции TypeScript в JavaScript, что быстрее, чем «ванильный» `tsc`, и HMR-обновления могут отражаться в браузере менее чем за 50 мс.

Используйте синтаксис [Type-Only Imports and Export](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-3-8.html#type-only-imports-and-export), чтобы избежать потенциальных проблем, например некорректного включения импортов только типов в бандл:

```ts
import type { T } from 'only/types'
export type { T }
```

### Опции компилятора TypeScript

Vite учитывает некоторые опции в `tsconfig.json` и устанавливает соответствующие опции Oxc Transformer. Для каждого файла Vite использует ближайший родительский `tsconfig.json`, соответствующий файлу, либо конфигурацию, на которую ссылается его поле [`references`](https://www.typescriptlang.org/tsconfig/#references) и которая соответствует файлу. Vite считает конфигурацию соответствующей файлу, когда файл удовлетворяет полям [`files`](https://www.typescriptlang.org/tsconfig/#files), [`include`](https://www.typescriptlang.org/tsconfig/#include) и [`exclude`](https://www.typescriptlang.org/tsconfig/#exclude) этой конфигурации.

Когда опции заданы и в конфигурации Vite, и в `tsconfig.json`, значение из конфигурации Vite имеет приоритет.

Некоторые поля конфигурации в разделе `compilerOptions` в `tsconfig.json` требуют особого внимания.

#### `isolatedModules`

- [Документация TypeScript](https://www.typescriptlang.org/tsconfig#isolatedModules)

Должно быть установлено в `true`.

Это связано с тем, что Oxc transformer выполняет только транспиляцию без информации о типах, он не поддерживает некоторые возможности, такие как const enum и неявные импорты только типов.

Вы должны установить `"isolatedModules": true` в вашем `tsconfig.json` в разделе `compilerOptions`, чтобы TS предупреждал вас о возможностях, которые не работают с изолированной транспиляцией.

Если зависимость плохо работает с `"isolatedModules": true`, вы можете использовать `"skipLibCheck": true`, чтобы временно подавить ошибки, пока они не будут исправлены на стороне зависимости.

#### `useDefineForClassFields`

- [Документация TypeScript](https://www.typescriptlang.org/tsconfig#useDefineForClassFields)

Значение по умолчанию будет `true`, если целевой уровень TypeScript — `ES2022` или новее, включая `ESNext`. Это согласуется с [поведением TypeScript 4.3.2+](https://github.com/microsoft/TypeScript/pull/42663).
Для других целевых уровней TypeScript по умолчанию будет `false`.

`true` — это стандартное поведение среды выполнения ECMAScript.

Если вы используете библиотеку, которая сильно полагается на поля классов (class fields), будьте внимательны к тому, как библиотека предполагает их использование.
Хотя большинство библиотек ожидают `"useDefineForClassFields": true`, вы можете явно установить `useDefineForClassFields` в `false`, если ваша библиотека это не поддерживает.

#### `target`

- [Документация TypeScript](https://www.typescriptlang.org/tsconfig#target)

Vite игнорирует значение `target` в `tsconfig.json`, следуя тому же поведению, что и [esbuild](https://esbuild.github.io/).

Чтобы указать целевой уровень в режиме разработки, можно использовать опцию [`oxc.target`](https://vite.dev/config/shared-options.html#oxc), которая по умолчанию имеет значение `esnext` для минимальной транспиляции. В сборках опция [`build.target`](https://vite.dev/config/build-options.html#build-target) имеет более высокий приоритет над `oxc.target` и также может быть установлена при необходимости.

#### `emitDecoratorMetadata`

- [Документация TypeScript](https://www.typescriptlang.org/tsconfig#emitDecoratorMetadata)

Эта опция поддерживается лишь частично. Полная поддержка требует вывода типов компилятором TypeScript, что не поддерживается. Подробности см. в [документации Oxc Transformer](https://oxc.rs/docs/guide/usage/transformer/typescript.html#decorators).

#### `paths`

- [Документация TypeScript](https://www.typescriptlang.org/tsconfig/#paths)

Можно указать [`resolve.tsconfigPaths: true`](https://vite.dev/config/shared-options.md#resolve-tsconfigpaths), чтобы указать Vite использовать опцию `paths` в `tsconfig.json` для разрешения импортов.

Обратите внимание, что эта возможность имеет издержки по производительности и [команда TypeScript не рекомендует использовать эту опцию для изменения поведения внешних инструментов](https://www.typescriptlang.org/tsconfig/#paths:~:text=Note%20that%20this%20feature%20does%20not%20change%20how%20import%20paths%20are%20emitted%20by%20tsc%2C%20so%20paths%20should%20only%20be%20used%20to%20inform%20TypeScript%20that%20another%20tool%20has%20this%20mapping%20and%20will%20use%20it%20at%20runtime%20or%20when%20bundling.).

#### Другие опции компилятора, влияющие на результат сборки

- [`extends`](https://www.typescriptlang.org/tsconfig#extends)
- [`importsNotUsedAsValues`](https://www.typescriptlang.org/tsconfig#importsNotUsedAsValues)
- [`preserveValueImports`](https://www.typescriptlang.org/tsconfig#preserveValueImports)
- [`verbatimModuleSyntax`](https://www.typescriptlang.org/tsconfig#verbatimModuleSyntax)
- [`jsx`](https://www.typescriptlang.org/tsconfig#jsx)
- [`jsxFactory`](https://www.typescriptlang.org/tsconfig#jsxFactory)
- [`jsxFragmentFactory`](https://www.typescriptlang.org/tsconfig#jsxFragmentFactory)
- [`jsxImportSource`](https://www.typescriptlang.org/tsconfig#jsxImportSource)
- [`experimentalDecorators`](https://www.typescriptlang.org/tsconfig#experimentalDecorators)

> **Совет: `skipLibCheck`**
>
> Стартовые шаблоны Vite по умолчанию содержат `"skipLibCheck": true`, чтобы избежать проверки типов зависимостей, поскольку они могут поддерживать только определённые версии и конфигурации TypeScript. Подробнее можно узнать в [vuejs/vue-cli#5688](https://github.com/vuejs/vue-cli/pull/5688).

### Клиентские типы

Типы Vite по умолчанию предназначены для его Node.js API. Чтобы создать заглушки (shim) для окружения клиентского кода в приложении Vite, вы можете добавить `vite/client` в `compilerOptions.types` внутри `tsconfig.json`:

**tsconfig.json**

```json
{
  "compilerOptions": {
    "types": ["vite/client", "some-other-global-lib"]
  }
}
```

Обратите внимание, что если указан [`compilerOptions.types`](https://www.typescriptlang.org/tsconfig#types), то только эти пакеты будут включены в глобальную область видимости (вместо всех видимых пакетов «@types»). Это рекомендуется начиная с TS 5.9.

<details><summary>Использование тройной косой директивы (triple-slash)</summary>

В качестве альтернативы вы можете добавить файл декларации `d.ts`:

**vite-env.d.ts**

```typescript
/// <reference types="vite/client" />
```

</details>

`vite/client` предоставляет следующие заглушки типов:

- Импорты ресурсов (например, импорт файла `.svg`)
- Типы для внедряемых Vite [констант](https://vite.dev/guide/env-and-mode#env-variables) в `import.meta.env`
- Типы для [HMR API](https://vite.dev/guide/api-hmr) в `import.meta.hot`

> **Совет**
>
> Чтобы переопределить типизацию по умолчанию, добавьте файл определения типов, содержащий ваши типизации. Затем добавьте ссылку на типы перед `vite/client`.
>
> Например, чтобы сделать импорт по умолчанию для `*.svg` React-компонентом:
>
> - `vite-env-override.d.ts` (файл, содержащий ваши типизации):
>
>   ```ts
>   declare module '*.svg' {
>     const content: React.FC<React.SVGProps<SVGElement>>
>     export default content
>   }
>   ```
>
> - Если вы используете `compilerOptions.types`, убедитесь, что файл включён в `tsconfig.json`:
>
>   ```json
>   {
>     "include": ["src", "./vite-env-override.d.ts"]
>   }
>   ```
>
> - Если вы используете тройные косые директивы, обновите файл, содержащий ссылку на `vite/client` (обычно `vite-env.d.ts`):
>
>   ```ts
>   /// <reference types="./vite-env-override.d.ts" />
>   /// <reference types="vite/client" />
>   ```

## HTML

HTML-файлы находятся [в самом центре](https://vite.dev/guide/#index-html-and-project-root) проекта Vite, служа точками входа для вашего приложения и упрощая создание одностраничных и [многостраничных приложений](https://vite.dev/guide/build.html#multi-page-app).

К любым HTML-файлам в корне вашего проекта можно обращаться напрямую по соответствующему пути в каталоге:

- `<root>/index.html` -> `http://localhost:5173/`
- `<root>/about.html` -> `http://localhost:5173/about.html`
- `<root>/blog/index.html` -> `http://localhost:5173/blog/index.html`

Ресурсы, на которые ссылаются HTML-элементы, такие как `<script type="module" src>` и `<link href>`, обрабатываются и включаются в бандл как часть приложения. Полный список поддерживаемых элементов приведён ниже:

- `<audio src>`
- `<embed src>`
- `<img src>` и `<img srcset>`
- `<image href>` и `<image xlink:href>`
- `<input src>`
- `<link href>` и `<link imagesrcset>`
- `<object data>`
- `<script type="module" src>`
- `<source src>` и `<source srcset>`
- `<track src>`
- `<use href>` и `<use xlink:href>`
- `<video src>` и `<video poster>`
- `<meta content>`
  - Только если атрибут `name` соответствует `msapplication-tileimage`, `msapplication-square70x70logo`, `msapplication-square150x150logo`, `msapplication-wide310x150logo`, `msapplication-square310x310logo`, `msapplication-config` или `twitter:image`
  - Или только если атрибут `property` соответствует `og:image`, `og:image:url`, `og:image:secure_url`, `og:audio`, `og:audio:secure_url`, `og:video` или `og:video:secure_url`

```html
<!doctype html>
<html>
  <head>
    <link rel="icon" href="/favicon.ico" />
    <link rel="stylesheet" href="/src/styles.css" />
  </head>
  <body>
    <img src="/src/images/logo.svg" alt="logo" />
    <script type="module" src="/src/main.js"></script>
  </body>
</html>
```

Чтобы отказаться от обработки HTML для определённых элементов, вы можете добавить атрибут `vite-ignore` к элементу, что может быть полезно при ссылках на внешние ресурсы или CDN.

## Фреймворки

Все современные фреймворки поддерживают интеграции с Vite. Большинство плагинов фреймворков поддерживаются командами соответствующих фреймворков, за исключением официальных плагинов Vite для Vue и React, которые поддерживаются в организации vite:

- Поддержка Vue через [@vitejs/plugin-vue](https://github.com/vitejs/vite-plugin-vue/tree/main/packages/plugin-vue)
- Поддержка Vue JSX через [@vitejs/plugin-vue-jsx](https://github.com/vitejs/vite-plugin-vue/tree/main/packages/plugin-vue-jsx)
- Поддержка React через [@vitejs/plugin-react](https://github.com/vitejs/vite-plugin-react/tree/main/packages/plugin-react)
- Поддержка React с использованием SWC через [@vitejs/plugin-react-swc](https://github.com/vitejs/vite-plugin-react/tree/main/packages/plugin-react-swc)
- Поддержка [React Server Components (RSC)](https://react.dev/reference/rsc/server-components) через [@vitejs/plugin-rsc](https://github.com/vitejs/vite-plugin-react/tree/main/packages/plugin-rsc)

Для получения дополнительной информации ознакомьтесь с [руководством по плагинам](https://vite.dev/plugins/).

## JSX

Файлы `.jsx` и `.tsx` также поддерживаются «из коробки». Транспиляция JSX также обрабатывается через [Oxc Transformer](https://oxc.rs/docs/guide/usage/transformer.html).

Выбранный вами фреймворк уже настроит JSX «из коробки» (например, пользователям Vue следует использовать официальный плагин [@vitejs/plugin-vue-jsx](https://github.com/vitejs/vite-plugin-vue/tree/main/packages/plugin-vue-jsx), который предоставляет специфичные для Vue 3 возможности, включая HMR, глобальное разрешение компонентов, директивы и слоты).

Если вы используете JSX с собственным фреймворком, можно настроить пользовательские `jsxFactory` и `jsxFragment` с помощью [опции `oxc`](https://vite.dev/config/shared-options.md#oxc). Например, плагин Preact использовал бы:

**vite.config.js**

```js
import { defineConfig } from 'vite'

export default defineConfig({
  oxc: {
    jsx: {
      importSource: 'preact',
    },
  },
})
```

Подробнее в [документации Oxc Transformer](https://oxc.rs/docs/guide/usage/transformer/jsx.html).

Вы можете внедрять JSX-хелперы с помощью `jsxInject` (это опция, доступная только в Vite), чтобы избежать ручных импортов:

**vite.config.js**

```js
import { defineConfig } from 'vite'

export default defineConfig({
  oxc: {
    jsxInject: `import React from 'react'`,
  },
})
```

## CSS

Импорт `.css` файлов внедрит их содержимое на страницу через тег `<style>` с поддержкой HMR.

### Инлайнинг и перебазирование `@import`

Vite предварительно настроен на поддержку инлайнинга CSS `@import` через `postcss-import`. Алиасы Vite также учитываются для CSS `@import`. Кроме того, все ссылки CSS `url()`, даже если импортируемые файлы находятся в разных каталогах, всегда автоматически перебазируются для обеспечения корректности.

Алиасы `@import` и перебазирование URL также поддерживаются для файлов Sass и Less (см. [Препроцессоры CSS](#css-pre-processors)).

### PostCSS

Если проект содержит корректную конфигурацию PostCSS (в любом формате, поддерживаемом [postcss-load-config](https://github.com/postcss/postcss-load-config), например `postcss.config.js`), она будет автоматически применена ко всему импортируемому CSS.

Обратите внимание, что минификация CSS запускается после PostCSS и использует опцию [`build.cssTarget`](https://vite.dev/config/build-options.md#build-csstarget).

### CSS-модули

Любой CSS-файл с окончанием `.module.css` считается [файлом CSS-модуля](https://github.com/css-modules/css-modules). Импорт такого файла вернёт соответствующий объект модуля:

**example.module.css**

```css
.red {
  color: red;
}
```

```js
import classes from './example.module.css'
document.getElementById('foo').className = classes.red
```

Поведение CSS-модулей можно настроить с помощью [опции `css.modules`](https://vite.dev/config/shared-options.md#css-modules).

Если `css.modules.localsConvention` настроен на включение camelCase-локалей (например, `localsConvention: 'camelCaseOnly'`), вы также можете использовать именованные импорты:

```js
// .apply-color -> applyColor
import { applyColor } from './example.module.css'
document.getElementById('foo').className = applyColor
```

### Препроцессоры CSS

Поскольку Vite ориентируется только на современные браузеры, рекомендуется использовать нативные CSS-переменные с PostCSS-плагинами, реализующими черновики CSSWG (например, [postcss-nesting](https://github.com/csstools/postcss-plugins/tree/main/plugins/postcss-nesting)), и писать простой CSS, соответствующий будущим стандартам.

Тем не менее Vite предоставляет встроенную поддержку файлов `.scss`, `.sass`, `.less`, `.styl` и `.stylus`. Устанавливать специфичные для Vite плагины для них не нужно, но сам соответствующий препроцессор должен быть установлен:

```bash
# .scss и .sass
npm add -D sass-embedded # или sass

# .less
npm add -D less

# .styl и .stylus
npm add -D stylus
```

Если вы используете однофайловые компоненты Vue, это также автоматически включает `<style lang="sass">` и подобные.

Vite улучшает разрешение `@import` для Sass и Less, так что алиасы Vite также учитываются. Кроме того, относительные ссылки `url()` внутри импортируемых файлов Sass/Less, находящихся в каталогах, отличных от корневого файла, также автоматически перебазируются для обеспечения корректности. Перебазирование ссылок `url()`, начинающихся с переменной или интерполяции, не поддерживается из-за ограничений API.

Алиасы `@import` и перебазирование URL не поддерживаются для Stylus из-за ограничений его API.

Вы также можете использовать CSS-модули в сочетании с препроцессорами, добавив `.module` к расширению файла, например `style.module.scss`.

### Отключение внедрения CSS на страницу

Автоматическое внедрение содержимого CSS можно отключить с помощью query-параметра `?inline`. В этом случае обработанная строка CSS, как обычно, возвращается как экспорт по умолчанию модуля, но стили не внедряются на страницу.

```js
import './foo.css' // будет внедрён на страницу
import otherStyles from './bar.css?inline' // не будет внедрён
```

> **Примечание**
>
> Импорты по умолчанию и именованные импорты из CSS-файлов (например, `import style from './foo.css'`) удалены начиная с Vite 5. Используйте вместо этого query `?inline`.

### Lightning CSS

Vite по умолчанию использует [Lightning CSS](https://lightningcss.dev/) для минификации CSS в продакшн-сборках. Однако для другой обработки CSS по-прежнему используется PostCSS.

Есть экспериментальная поддержка использования Lightning CSS для полной обработки CSS. Вы можете включить её, добавив [`css.transformer: 'lightningcss'`](https://vite.dev/config/shared-options.md#css-transformer).

Чтобы настроить его, вы можете передать опции Lightning CSS в опцию конфигурации [`css.lightningcss`](https://vite.dev/config/shared-options.md#css-lightningcss). Для настройки CSS-модулей следует использовать [`css.lightningcss.cssModules`](https://lightningcss.dev/css-modules.html) вместо [`css.modules`](https://vite.dev/config/shared-options.md#css-modules) (которая настраивает способ обработки CSS-модулей в PostCSS).

```js
// Явно загрузить ресурс как URL (автоматически встраивается в зависимости от размера файла)

```

```js
// Загрузить ресурсы как строки
```

```js
// Загрузить веб-воркеры
```

```js
// Веб-воркеры, встроенные как base64-строки на этапе сборки
```

## JSON

JSON-файлы можно импортировать напрямую — именованные импорты также поддерживаются:

```js
// импортировать весь объект
import json from './example.json'
// импортировать поле верхнего уровня как именованный экспорт — помогает с tree-shaking!
import { field } from './example.json'
```

## Импорт через glob (Glob Import)

Vite поддерживает импорт нескольких модулей из файловой системы с помощью специальной функции `import.meta.glob`:

```js
const modules = import.meta.glob('./dir/*.js')
```

Приведённое выше будет преобразовано в следующее:

```js
// код, сгенерированный vite
const modules = {
  './dir/bar.js': () => import('./dir/bar.js'),
  './dir/foo.js': () => import('./dir/foo.js'),
}
```

Затем вы можете перебрать ключи объекта `modules`, чтобы получить доступ к соответствующим модулям:

```js
for (const path in modules) {
  modules[path]().then((mod) => {
    console.log(path, mod)
  })
}
```

Совпавшие файлы по умолчанию загружаются лениво через динамический импорт и будут разделены на отдельные чанки во время сборки. Если вы предпочитаете импортировать все модули напрямую (например, полагаясь на то, что побочные эффекты в этих модулях будут применены в первую очередь), вы можете передать `{ eager: true }` в качестве второго аргумента:

```js
const modules = import.meta.glob('./dir/*.js', { eager: true })
```

Приведённое выше будет преобразовано в следующее:

```js
// код, сгенерированный vite
import * as __vite_glob_0_0 from './dir/bar.js'
import * as __vite_glob_0_1 from './dir/foo.js'
const modules = {
  './dir/bar.js': __vite_glob_0_0,
  './dir/foo.js': __vite_glob_0_1,
}
```

### Несколько шаблонов (Multiple Patterns)

Первый аргумент может быть массивом glob-шаблонов, например

```js
const modules = import.meta.glob(['./dir/*.js', './another/*.js'])
```

### Отрицательные шаблоны (Negative Patterns)

Отрицательные glob-шаблоны также поддерживаются (с префиксом `!`). Чтобы исключить некоторые файлы из результата, вы можете добавить исключающие glob-шаблоны в первый аргумент:

```js
const modules = import.meta.glob(['./dir/*.js', '!**/bar.js'])
```

```js
// код, сгенерированный vite
const modules = {
  './dir/foo.js': () => import('./dir/foo.js'),
}
```

#### Именованные импорты (Named Imports)

Можно импортировать только части модулей с помощью опции `import`.

```ts
const modules = import.meta.glob('./dir/*.js', { import: 'setup' })
```

```ts
// код, сгенерированный vite
const modules = {
  './dir/bar.js': () => import('./dir/bar.js').then((m) => m.setup),
  './dir/foo.js': () => import('./dir/foo.js').then((m) => m.setup),
}
```

В сочетании с `eager` для этих модулей можно даже включить tree-shaking.

```ts
const modules = import.meta.glob('./dir/*.js', {
  import: 'setup',
  eager: true,
})
```

```ts
// код, сгенерированный vite:
import { setup as __vite_glob_0_0 } from './dir/bar.js'
import { setup as __vite_glob_0_1 } from './dir/foo.js'
const modules = {
  './dir/bar.js': __vite_glob_0_0,
  './dir/foo.js': __vite_glob_0_1,
}
```

Установите `import` в `default`, чтобы импортировать экспорт по умолчанию.

```ts
const modules = import.meta.glob('./dir/*.js', {
  import: 'default',
  eager: true,
})
```

```ts
// код, сгенерированный vite:
import { default as __vite_glob_0_0 } from './dir/bar.js'
import { default as __vite_glob_0_1 } from './dir/foo.js'
const modules = {
  './dir/bar.js': __vite_glob_0_0,
  './dir/foo.js': __vite_glob_0_1,
}
```

#### Пользовательские query (Custom Queries)

Вы также можете использовать опцию `query`, чтобы предоставить query для импортов, например, чтобы импортировать ресурсы [как строку](https://vite.dev/guide/assets.html#importing-asset-as-string) или [как URL](https://vite.dev/guide/assets.html#importing-asset-as-url):

```ts
const moduleStrings = import.meta.glob('./dir/*.svg', {
  query: '?raw',
  import: 'default',
})
const moduleUrls = import.meta.glob('./dir/*.svg', {
  query: '?url',
  import: 'default',
})
```

```ts
// код, сгенерированный vite:
const moduleStrings = {
  './dir/bar.svg': () => import('./dir/bar.svg?raw').then((m) => m['default']),
  './dir/foo.svg': () => import('./dir/foo.svg?raw').then((m) => m['default']),
}
const moduleUrls = {
  './dir/bar.svg': () => import('./dir/bar.svg?url').then((m) => m['default']),
  './dir/foo.svg': () => import('./dir/foo.svg?url').then((m) => m['default']),
}
```

Вы также можете предоставить пользовательские query для использования другими плагинами:

```ts
const modules = import.meta.glob('./dir/*.js', {
  query: { foo: 'bar', bar: true },
})
```

#### Базовый путь (Base Path)

Вы также можете использовать опцию `base`, чтобы задать базовый путь для импортов:

```ts
const modulesWithBase = import.meta.glob('./**/*.js', {
  base: './base',
})
```

```ts
// код, сгенерированный vite:
const modulesWithBase = {
  './dir/foo.js': () => import('./base/dir/foo.js'),
  './dir/bar.js': () => import('./base/dir/bar.js'),
}
```

Опция base может быть только путём к каталогу относительно файла-импортёра или абсолютным относительно корня проекта. Алиасы и виртуальные модули не поддерживаются.

Только те glob-шаблоны, которые являются относительными путями, интерпретируются как относительные к разрешённому базовому пути.

Все результирующие ключи модулей изменяются так, чтобы быть относительными к базовому пути, если он указан.

#### Сопоставление с учётом регистра (Case Sensitive Matching)

По умолчанию сопоставление glob-шаблонов чувствительно к регистру. Вы можете использовать опцию `caseSensitive`, чтобы изменить это поведение:

```ts
const modules = import.meta.glob('./dir/module*.js', {
  caseSensitive: false,
})
```

При `caseSensitive: false` glob будет сопоставлять файлы независимо от регистра (например, `Module.js`, `module.js`, `MODULE.js` — все будут сопоставлены шаблоном `module*.js`).

### Оговорки по импорту через glob (Glob Import Caveats)

Обратите внимание, что:

- Это возможность, специфичная для Vite, а не веб- или ES-стандарт.
- Glob-шаблоны обрабатываются как спецификаторы импорта: они должны быть либо относительными (начинаться с `./`), либо абсолютными (начинаться с `/`, разрешаются относительно корня проекта), либо путём-алиасом (см. [опцию `resolve.alias`](https://vite.dev/config/shared-options.md#resolve-alias)).
- Сопоставление glob выполняется с помощью [`tinyglobby`](https://github.com/SuperchupuDev/tinyglobby) — ознакомьтесь с его документацией по [поддерживаемым glob-шаблонам](https://superchupu.dev/tinyglobby/comparison).
- Также следует учитывать, что все аргументы в `import.meta.glob` должны быть **переданы как литералы**. Вы НЕ можете использовать в них переменные или выражения.

## Динамический импорт (Dynamic Import)

Подобно [импорту через glob](#glob-import), Vite также поддерживает динамический импорт с переменными.

```ts
const module = await import(`./dir/${file}.js`)
```

Обратите внимание, что переменные представляют имена файлов только на один уровень в глубину. Если `file` равно `'foo/bar'`, импорт завершится ошибкой. Для более сложного использования вы можете применить возможность [импорта через glob](#glob-import).

Также обратите внимание, что динамический импорт должен соответствовать следующим правилам, чтобы быть включённым в бандл:

- Импорты должны начинаться с `./` или `../`: ``import(`./dir/${foo}.js`)`` корректен, но ``import(`${foo}.js`)`` — нет.
- Импорты должны заканчиваться расширением файла: ``import(`./dir/${foo}.js`)`` корректен, но ``import(`./dir/${foo}`)`` — нет.
- Импорты в собственный каталог должны указывать шаблон имени файла: ``import(`./prefix-${foo}.js`)`` корректен, но ``import(`./${foo}.js`)`` — нет.

Эти правила введены для предотвращения случайного импорта файлов, которые не предназначены для включения в бандл. Например, без этих правил `import(foo)` включил бы в бандл всё в файловой системе.

## WebAssembly

Vite поддерживает импорт предварительно скомпилированных `.wasm` файлов двумя способами: напрямую как [ES-модуль](#esm-integration), когда вам нужны только экспорты модуля, или с помощью [`?init`](#manual-initialization), когда вам нужен явный контроль над инстанцированием.

### Интеграция с ESM (ESM Integration)

`.wasm` файл можно импортировать напрямую. Vite читает импорты и экспорты модуля из бинарного файла, инстанцирует его и повторно предоставляет его экспорты как именованные экспорты ES-модуля:

```js
import { add } from './add.wasm'

console.log(add(1, 2)) // 3
```

Если модуль WebAssembly объявляет собственные импорты, Vite разрешает их из JavaScript-модулей. Имя модуля каждого импорта обрабатывается как спецификатор импорта (разрешается относительно `.wasm` файла), и запрашиваемые члены автоматически подключаются в инстанс.

Это следует [предложению WebAssembly/ES Module Integration](https://github.com/WebAssembly/esm-integration). Поскольку модуль WebAssembly инстанцируется асинхронно, напрямую импортированный `.wasm` файл ведёт себя как асинхронный модуль и требует поддержки await верхнего уровня.

> **Совет: Поддержка TypeScript**
>
> Поскольку типы `.wasm` файлов неизвестны, TypeScript будет сообщать об ошибках вроде `Module '"*.wasm"' has no exported member 'add'`. Чтобы исправить это, включите [`allowArbitraryExtensions`](https://www.typescriptlang.org/tsconfig/#allowArbitraryExtensions) в вашем `tsconfig.json` и создайте файл декларации рядом с вашим `.wasm` файлом. С включённым `allowArbitraryExtensions` TypeScript будет искать файл декларации с именем `{filename}.d.wasm.ts` при разрешении импорта `.wasm`. Например, для `add.wasm` создайте `add.d.wasm.ts`:
>
> **add.d.wasm.ts**
>
> ```ts
> export function add(a: number, b: number): number
> ```

### Ручная инициализация (Manual Initialization)

Когда вам нужен контроль над тем, когда и как инстанцируется модуль, импортируйте его с помощью `?init`. Экспортом по умолчанию будет функция инициализации, которая возвращает Promise объекта [`WebAssembly.Instance`](https://developer.mozilla.org/en-US/docs/WebAssembly/JavaScript_interface/Instance):

```js
import init from './example.wasm?init'

init().then((instance) => {
  instance.exports.test()
})
```

Функция init также может принимать importObject, который передаётся в [`WebAssembly.instantiate`](https://developer.mozilla.org/en-US/docs/WebAssembly/JavaScript_interface/instantiate) в качестве второго аргумента:

```js
init({
  imports: {
    someFunc: () => {
      /* ... */
    },
  },
}).then(() => {
  /* ... */
})
```

В продакшн-сборке `.wasm` файлы меньше `assetInlineLimit` будут встроены как base64-строки. В противном случае они будут обрабатываться как [статический ресурс](https://vite.dev/guide/assets) и загружаться по требованию.

> **Внимание: для SSR-сборки поддерживаются только среды выполнения, совместимые с Node.js**
>
> Из-за отсутствия универсального способа загрузки файла внутренняя реализация как прямых импортов `.wasm`, так и `.wasm?init` полагается на модуль `node:fs`. Это означает, что эти возможности будут работать только в средах выполнения, совместимых с Node.js, для SSR-сборок.

### Доступ к модулю WebAssembly

Если вам нужен доступ к объекту `Module`, например, чтобы инстанцировать его несколько раз, используйте [явный импорт по URL](https://vite.dev/guide/assets#explicit-url-imports) для разрешения ресурса, а затем выполните инстанцирование:

```js
import wasmUrl from 'foo.wasm?url'

const main = async () => {
  const responsePromise = fetch(wasmUrl)
  const { module, instance } =
    await WebAssembly.instantiateStreaming(responsePromise)
  /* ... */
}

main()
```

## Веб-воркеры (Web Workers)

### Импорт с помощью конструкторов (Import with Constructors)

Скрипт веб-воркера можно импортировать с помощью [`new Worker()`](https://developer.mozilla.org/en-US/docs/Web/API/Worker/Worker) и [`new SharedWorker()`](https://developer.mozilla.org/en-US/docs/Web/API/SharedWorker/SharedWorker). По сравнению с суффиксами воркеров этот синтаксис ближе к стандартам и является **рекомендуемым** способом создания воркеров.

```ts
const worker = new Worker(new URL('./worker.js', import.meta.url))
```

Конструктор воркера также принимает опции, которые можно использовать для создания «модульных» воркеров:

```ts
const worker = new Worker(new URL('./worker.js', import.meta.url), {
  type: 'module',
})
```

Обнаружение воркера будет работать только в том случае, если конструктор `new URL()` используется непосредственно внутри объявления `new Worker()`. В противном случае он обрабатывается как [URL статического ресурса](https://vite.dev/guide/assets#new-url-url-import-meta-url). Кроме того, все параметры опций должны быть статическими значениями (то есть строковыми литералами).

### Импорт с суффиксами query (Import with Query Suffixes)

Скрипт веб-воркера можно импортировать напрямую, добавив `?worker` или `?sharedworker` к запросу импорта. Экспортом по умолчанию будет пользовательский конструктор воркера:

```js
import MyWorker from './worker?worker'

const worker = new MyWorker()
```

Скрипт воркера также может использовать ESM-операторы `import` вместо `importScripts()`. **Примечание**: во время разработки это полагается на [нативную поддержку браузера](https://caniuse.com/?search=module%20worker), но для продакшн-сборки это компилируется прочь (устраняется).

По умолчанию скрипт воркера будет выпущен как отдельный чанк в продакшн-сборке. Если вы хотите встроить воркер как base64-строки, добавьте query `inline`:

```js
import MyWorker from './worker?worker&inline'
```

Если вы хотите получить воркер как URL, добавьте query `url`:

```js
import MyWorker from './worker?worker&url'
```

Подробности о настройке бандлинга всех воркеров см. в разделе [Опции воркеров (Worker Options)](https://vite.dev/config/worker-options.md).

## Content Security Policy (CSP)

Для развёртывания CSP необходимо задать определённые директивы или конфигурации из-за внутреннего устройства Vite.

### [`'nonce-{RANDOM}'`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy/Sources#nonce-base64-value)

Когда установлена опция [`html.cspNonce`](https://vite.dev/config/shared-options#html-cspnonce), Vite добавляет атрибут nonce с указанным значением ко всем тегам `<script>` и `<style>`, а также к тегам `<link>` для таблиц стилей и предзагрузки модулей. Кроме того, когда эта опция установлена, Vite внедряет meta-тег (`<meta property="csp-nonce" nonce="PLACEHOLDER" />`).

Значение nonce из meta-тега с `property="csp-nonce"` будет использоваться Vite всякий раз, когда это необходимо, как во время разработки, так и после сборки.

> **Внимание**
>
> Убедитесь, что вы заменяете плейсхолдер уникальным значением для каждого запроса. Это важно для предотвращения обхода политики ресурса, что в противном случае можно легко сделать.

### [`data:`](<https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy/Sources#scheme-source:~:text=schemes%20(not%20recommended).-,data%3A,-Allows%20data%3A>)

По умолчанию во время сборки Vite встраивает небольшие ресурсы как data URI. Необходимо разрешить `data:` для соответствующих директив (например, [`img-src`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy/img-src), [`font-src`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy/font-src)) либо отключить это, установив [`build.assetsInlineLimit: 0`](https://vite.dev/config/build-options#build-assetsinlinelimit).

> **Внимание**
>
> Не разрешайте `data:` для [`script-src`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy/script-src). Это позволит внедрять произвольные скрипты.

## Лицензия (License)

Vite может сгенерировать файл со всеми лицензиями зависимостей, используемых в сборке, с помощью опции [`build.license`](https://vite.dev/config/build-options.md#build-license). Его можно разместить для отображения и указания зависимостей, используемых приложением.

**vite.config.js**

```js
import { defineConfig } from 'vite'

export default defineConfig({
  build: {
    license: true,
  },
})
```

Это сгенерирует файл `.vite/license.md` с выводом, который может выглядеть так:

```md
# Licenses

The app bundles dependencies which contain the following licenses:

## dep-1 - 1.2.3 (CC0-1.0)

CC0 1.0 Universal

...

## dep-2 - 4.5.6 (MIT)

MIT License

...
```

Чтобы обслуживать файл по другому пути, можно передать, например, `{ fileName: 'license.md' }`, чтобы он обслуживался по адресу `https://example.com/license.md`. Дополнительную информацию см. в документации [`build.license`](https://vite.dev/config/build-options.md#build-license).

## Оптимизации сборки

> Перечисленные ниже возможности применяются автоматически (за исключением экспериментальной возможности import map для чанков) в рамках процесса сборки, и нет необходимости в явной настройке, если только вы не хотите их отключить.

### Разделение CSS-кода (CSS Code Splitting)

Vite автоматически извлекает CSS, используемый модулями в асинхронном чанке, и генерирует для него отдельный файл. CSS-файл автоматически загружается через тег `<link>` при загрузке связанного асинхронного чанка, и гарантируется, что асинхронный чанк будет выполнен только после загрузки CSS, чтобы избежать [FOUC](https://en.wikipedia.org/wiki/Flash_of_unstyled_content#:~:text=A%20flash%20of%20unstyled%20content,before%20all%20information%20is%20retrieved.).

Если вы предпочитаете, чтобы весь CSS был извлечён в один файл, вы можете отключить разделение CSS-кода, установив [`build.cssCodeSplit`](https://vite.dev/config/build-options.md#build-csscodesplit) в `false`.

### Генерация директив предзагрузки (Preload Directives Generation)

Vite автоматически генерирует директивы `<link rel="modulepreload">` для чанков точек входа и их прямых импортов в собранном HTML.

### Оптимизация загрузки асинхронных чанков

В реальных приложениях Rollup часто генерирует «общие» чанки — код, который совместно используется двумя или более другими чанками. В сочетании с динамическими импортами довольно часто встречается следующий сценарий:

В неоптимизированных сценариях, когда импортируется асинхронный чанк `A`, браузеру придётся запросить и разобрать `A`, прежде чем он сможет понять, что ему также нужен общий чанк `C`. Это приводит к дополнительному сетевому обмену (round trip):

```
Entry ---> A ---> C
```

Vite автоматически переписывает вызовы динамического импорта с разделением кода, добавляя шаг предзагрузки, так что, когда запрашивается `A`, `C` загружается **параллельно**:

```
Entry ---> (A + C)
```

У `C` могут быть дальнейшие импорты, что приведёт к ещё большему числу обменов в неоптимизированном сценарии. Оптимизация Vite отслеживает все прямые импорты, чтобы полностью устранить обмены независимо от глубины импорта.

### Оптимизация import map для чанков (Chunk Import Map Optimization)

Чтобы повысить частоту попаданий в кэш для чанков, Vite может создать import map для чанков. Это предотвращает проблему каскадной инвалидации кэша, которая является проблемой ES-модулей.

Например, рассмотрим следующий сценарий:

```
Entry --> A ---> C
```

Если `C` обновляется, единственный чанк, который по своей сути нужно инвалидировать, — это `C`. Однако если `A` ссылается на `C` через обычный URL в статическом импорте (то есть хеш `C` включён в URL), содержимое `A` изменяется, поэтому `A` также нужно инвалидировать. То же самое относится к `Entry`.

Используя возможность import maps, этой проблемы можно избежать. Когда эта оптимизация включена, Vite создаёт import map, которая сопоставляет ID каждого чанка с его URL, и использует ID чанка в операторах импорта вместо URL. Таким образом, когда чанк обновляется, инвалидировать нужно только обновлённый чанк, а чанки, которые на него ссылаются, инвалидированы не будут.

Обратите внимание, что эта оптимизация в настоящее время не применяется к CSS и ресурсам. Если вы обновите ресурс, чанки, которые на него ссылаются, будут инвалидированы. Тем не менее инвалидация не будет каскадной, и чанк, импортирующий инвалидированный чанк, не будет инвалидирован.

Чтобы включить эту возможность, установите [`build.chunkImportMap`](https://vite.dev/config/build-options.md#build-chunkimportmap) в `true`.
