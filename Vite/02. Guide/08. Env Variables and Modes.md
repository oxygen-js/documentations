# Переменные окружения и режимы

Vite предоставляет некоторые константы в специальном объекте `import.meta.env`. Эти константы определяются как глобальные переменные во время разработки и статически заменяются во время сборки, чтобы tree-shaking работал эффективно.

<details>
<summary>Пример</summary>

```js
if (import.meta.env.DEV) {
  // код, который окажется здесь, будет удалён tree-shaking'ом в продакшн-сборках
  console.log('Dev mode')
}
```

</details>

[Посмотрите интерактивный урок на Scrimba](https://scrimba.com/intro-to-vite-c03p6pbbdq/~05an?via=vite)

## Встроенные константы

Некоторые встроенные константы доступны во всех случаях:

- **`import.meta.env.MODE`**: {string} [режим](#modes), в котором работает приложение.

- **`import.meta.env.BASE_URL`**: {string} базовый URL, с которого раздаётся приложение. Определяется [опцией конфигурации `base`](https://vite.dev/config/shared-options#base).

- **`import.meta.env.PROD`**: {boolean} работает ли приложение в продакшене (запуск dev-сервера с `NODE_ENV='production'` или запуск приложения, собранного с `NODE_ENV='production'`).

- **`import.meta.env.DEV`**: {boolean} работает ли приложение в режиме разработки (всегда противоположно `import.meta.env.PROD`).

- **`import.meta.env.SSR`**: {boolean} работает ли приложение на [сервере](https://vite.dev/guide/ssr#conditional-logic).

## Переменные окружения

Vite автоматически предоставляет переменные окружения в объекте `import.meta.env` в виде строк.

Переменные с префиксом `VITE_` будут доступны в клиентском исходном коде после сборки Vite в бандл. Чтобы случайно не раскрыть переменные окружения клиенту, избегайте этого префикса. В качестве примера рассмотрите следующее:

**.env**

```
VITE_SOME_KEY=123
DB_PASSWORD=foobar
```

Разобранное значение `VITE_SOME_KEY` — `"123"` — будет доступно на клиенте, а значение `DB_PASSWORD` — нет. Проверить это можно, добавив в код следующее:

```js
console.log(import.meta.env.VITE_SOME_KEY) // "123"
console.log(import.meta.env.DB_PASSWORD) // undefined
```

Если вы хотите настроить префикс переменных окружения, обратитесь к опции [envPrefix](https://vite.dev/config/shared-options#envprefix).

> **Совет: разбор переменных окружения**
>
> Как показано выше, `VITE_SOME_KEY` — число, но при разборе возвращается как строка. То же самое произойдёт и с булевыми переменными окружения. Обязательно преобразуйте значение к нужному типу при использовании его в коде.

> **Внимание: защита секретов**
>
> Переменные `VITE_*` _не должны_ содержать конфиденциальную информацию, такую как ключи API. Значения этих переменных встраиваются в ваш исходный код на этапе сборки. Для продакшн-развёртываний рассмотрите использование бэкенд-сервера или serverless/edge-функций, чтобы должным образом защитить секреты.

### Файлы `.env`

Vite использует [dotenv](https://github.com/motdotla/dotenv) для загрузки дополнительных переменных окружения из следующих файлов в вашем [каталоге окружения](https://vite.dev/config/shared-options#envdir):

```
.env                # загружается во всех случаях
.env.local          # загружается во всех случаях, игнорируется git
.env.[mode]         # загружается только в указанном режиме
.env.[mode].local   # загружается только в указанном режиме, игнорируется git
```

> **Совет: приоритеты загрузки переменных окружения**
>
> Файл окружения для конкретного режима (например, `.env.production`) имеет более высокий приоритет, чем общий (например, `.env`).
>
> Vite всегда загружает `.env` и `.env.local` в дополнение к файлу `.env.[mode]` для конкретного режима. Переменные, объявленные в файлах для конкретного режима, имеют приоритет над переменными из общих файлов, но переменные, определённые только в `.env` или `.env.local`, всё равно будут доступны в окружении.
>
> Кроме того, переменные окружения, которые уже существуют на момент запуска Vite, имеют наивысший приоритет и не будут перезаписаны файлами `.env`. Например, при запуске `VITE_SOME_KEY=123 vite build`.
>
> Файлы `.env` загружаются при запуске Vite. Перезапустите сервер после внесения изменений.

> **Внимание: пользователи Bun**
>
> При использовании [Bun](https://bun.sh) учтите, что Bun автоматически загружает файлы `.env` перед запуском вашего скрипта. Это встроенное поведение загружает переменные окружения напрямую в `process.env` и может конфликтовать с функцией Vite, поскольку Vite учитывает существующие значения `process.env`. Обходные пути смотрите в [oven-sh/bun#5515](https://github.com/oven-sh/bun/issues/5515).

Также Vite использует [dotenv-expand](https://github.com/motdotla/dotenv-expand) для подстановки переменных, записанных в файлах окружения, из коробки. Чтобы узнать больше о синтаксисе, ознакомьтесь с [их документацией](https://github.com/motdotla/dotenv-expand#what-rules-does-the-expansion-engine-follow).

Обратите внимание: если вы хотите использовать `$` внутри значения переменной окружения, его нужно экранировать с помощью `\`.

**.env**

```
KEY=123
NEW_KEY1=test$foo   # test
NEW_KEY2=test\$foo  # test$foo
NEW_KEY3=test$KEY   # test123
```

<details>
<summary>Подстановка переменных в обратном порядке</summary>

Vite поддерживает подстановку переменных в обратном порядке.
Например, приведённый ниже `.env` будет вычислен как `VITE_FOO=foobar`, `VITE_BAR=bar`.

**.env**

```
VITE_FOO=foo${VITE_BAR}
VITE_BAR=bar
```

Это не работает в shell-скриптах и других инструментах вроде `docker compose`.
Тем не менее Vite поддерживает такое поведение, поскольку оно давно поддерживается в `dotenv-expand`, и другие инструменты экосистемы JavaScript используют более старые версии, которые его поддерживают.

Чтобы избежать проблем совместимости, рекомендуется не полагаться на такое поведение. В будущем Vite может начать выводить предупреждения по этому поводу.

</details>

> **Внимание: игнорирование локальных файлов `.env`**
>
> Файлы `.env.*.local` предназначены только для локального использования и могут содержать конфиденциальные переменные. Добавьте `*.local` в ваш `.gitignore`, чтобы они не попадали в git.

## IntelliSense для TypeScript

По умолчанию Vite предоставляет определения типов для `import.meta.env` в [`vite/client.d.ts`](https://github.com/vitejs/vite/blob/main/packages/vite/client.d.ts). Хотя вы можете определять дополнительные пользовательские переменные окружения в файлах `.env.[mode]`, вам может понадобиться поддержка TypeScript IntelliSense для пользовательских переменных окружения с префиксом `VITE_`.

Для этого создайте файл `vite-env.d.ts` в каталоге `src` и дополните `ImportMetaEnv` следующим образом:

**vite-env.d.ts**

```typescript
interface ViteTypeOptions {
  // Добавив эту строку, вы можете сделать тип ImportMetaEnv строгим,
  // чтобы запретить неизвестные ключи.
  // strictImportMetaEnv: unknown
}

interface ImportMetaEnv {
  readonly VITE_APP_TITLE: string
  // другие переменные окружения...
}

interface ImportMeta {
  readonly env: ImportMetaEnv
}
```

Если ваш код полагается на типы из браузерных окружений, таких как [DOM](https://github.com/microsoft/TypeScript/blob/main/src/lib/dom.generated.d.ts) и [WebWorker](https://github.com/microsoft/TypeScript/blob/main/src/lib/webworker.generated.d.ts), вы можете обновить поле [lib](https://www.typescriptlang.org/tsconfig#lib) в `tsconfig.json`.

**tsconfig.json**

```json
{
  "lib": ["WebWorker"]
}
```

> **Внимание: импорты нарушают расширение типов**
>
> Если расширение `ImportMetaEnv` не работает, убедитесь, что в `vite-env.d.ts` нет ни одного оператора `import`. Подробнее смотрите в [документации TypeScript](https://www.typescriptlang.org/docs/handbook/2/modules.html#how-javascript-modules-are-defined).

## Замена констант в HTML

Vite также поддерживает замену констант в HTML-файлах. Любые свойства из `import.meta.env` можно использовать в HTML-файлах с помощью специального синтаксиса `%CONST_NAME%`:

```html
<h1>Vite is running in %MODE%</h1>
<p>Using data from %VITE_API_URL%</p>
```

Если переменная окружения отсутствует в `import.meta.env`, например `%NON_EXISTENT%`, она будет проигнорирована и не заменена — в отличие от `import.meta.env.NON_EXISTENT` в JS, где она заменяется на `undefined`.

Поскольку Vite используется многими фреймворками, он намеренно не навязывает мнения о сложных заменах вроде условных конструкций. Vite можно расширить с помощью [существующего стороннего плагина](https://github.com/vitejs/awesome-vite#transformers) или собственного плагина, реализующего [хук `transformIndexHtml`](https://vite.dev/guide/api-plugin#transformindexhtml).

## Режимы

По умолчанию dev-сервер (команда `dev`) работает в режиме `development`, а команда `build` — в режиме `production`.

Это значит, что при запуске `vite build` будут загружены переменные окружения из `.env.production`, если такой файл есть:

**.env.production**

```
VITE_APP_TITLE=My App
```

В приложении вы можете вывести заголовок с помощью `import.meta.env.VITE_APP_TITLE`.

В некоторых случаях вам может понадобиться запустить `vite build` с другим режимом, чтобы вывести другой заголовок. Вы можете переопределить режим по умолчанию для команды, передав флаг опции `--mode`. Например, если вы хотите собрать приложение для режима staging:

```bash
vite build --mode staging
```

И создать файл `.env.staging`:

**.env.staging**

```
VITE_APP_TITLE=My App (staging)
```

Поскольку `vite build` по умолчанию выполняет продакшн-сборку, вы можете изменить это и выполнить сборку для разработки, используя другой режим и конфигурацию файла `.env`:

**.env.testing**

```
NODE_ENV=development
```

### NODE_ENV и режимы

Важно отметить, что `NODE_ENV` (`process.env.NODE_ENV`) и режимы — это два разных понятия. Вот как разные команды влияют на `NODE_ENV` и режим:

| Команда                                              | NODE_ENV        | Режим           |
| ---------------------------------------------------- | --------------- | --------------- |
| `vite build`                                         | `"production"`  | `"production"`  |
| `vite build --mode development`                      | `"production"`  | `"development"` |
| `NODE_ENV=development vite build`                    | `"development"` | `"production"`  |
| `NODE_ENV=development vite build --mode development` | `"development"` | `"development"` |

Разные значения `NODE_ENV` и режима также отражаются на соответствующих свойствах `import.meta.env`:

| Команда                | `import.meta.env.PROD` | `import.meta.env.DEV` |
| ---------------------- | ---------------------- | --------------------- |
| `NODE_ENV=production`  | `true`                 | `false`               |
| `NODE_ENV=development` | `false`                | `true`                |
| `NODE_ENV=other`       | `false`                | `true`                |

| Команда              | `import.meta.env.MODE` |
| -------------------- | ---------------------- |
| `--mode production`  | `"production"`         |
| `--mode development` | `"development"`        |
| `--mode staging`     | `"staging"`            |

> **Совет: `NODE_ENV` в файлах `.env`**
>
> `NODE_ENV=...` можно задать как в команде, так и в вашем файле `.env`. Если `NODE_ENV` указан в файле `.env.[mode]`, то режим можно использовать для управления его значением. Тем не менее `NODE_ENV` и режимы остаются двумя разными понятиями.
>
> Основное преимущество указания `NODE_ENV=...` в команде в том, что это позволяет Vite определить значение на раннем этапе. Также это позволяет читать `process.env.NODE_ENV` в вашей конфигурации Vite, поскольку Vite может загрузить файлы окружения только после вычисления конфигурации.
