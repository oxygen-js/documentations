# Сборка для продакшна

Когда придёт время развернуть приложение в продакшне, просто выполните команду `vite build`. По умолчанию она использует `<root>/index.html` в качестве точки входа для сборки и создаёт бандл приложения, пригодный для раздачи через сервис статического хостинга. Загляните в раздел [Развёртывание статического сайта](https://vite.dev/guide/static-deploy) за руководствами по популярным сервисам.

Посмотрите интерактивный урок на Scrimba

## Совместимость с браузерами

По умолчанию продакшн-бандл рассчитан на современный браузер, входящий в число широко доступных целей [Baseline](https://web-platform-dx.github.io/web-features/). Диапазон поддержки браузеров по умолчанию таков:

- Chrome >=111
- Edge >=111
- Firefox >=114
- Safari >=16.4

Вы можете задать пользовательские цели через [опцию конфигурации `build.target`](https://vite.dev/config/build-options#build-target), при этом минимальная возможная цель — `es2015`. Если задать более низкую цель, Vite всё равно будет требовать эти минимальные диапазоны поддержки браузеров, поскольку он опирается на [нативный динамический импорт ESM](https://caniuse.com/es6-module-dynamic-import) и [`import.meta`](https://caniuse.com/mdn-javascript_operators_import_meta):

- Chrome >=64
- Firefox >=67
- Safari >=11.1
- Edge >=79

Обратите внимание, что по умолчанию Vite обрабатывает только преобразования синтаксиса и **не подключает полифилы**. Вы можете воспользоваться сервисом https://cdnjs.cloudflare.com/polyfill/, который автоматически генерирует бандлы полифилов на основе строки UserAgent браузера пользователя.

Поддержку устаревших браузеров можно обеспечить с помощью [@vitejs/plugin-legacy](https://github.com/vitejs/vite/tree/main/packages/plugin-legacy), который автоматически генерирует legacy-чанки и соответствующие полифилы возможностей языка ES. Legacy-чанки загружаются условно — только в браузерах без нативной поддержки ESM.

## Публичный базовый путь

- Связанная тема: [Работа со статическими ресурсами](https://vite.dev/guide/assets)

Если вы развёртываете проект по вложенному публичному пути, просто укажите [опцию конфигурации `base`](https://vite.dev/config/shared-options#base) — и все пути к ресурсам будут переписаны соответствующим образом. Эту опцию также можно задать флагом командной строки, например `vite build --base=/my/public/path/`.

URL-адреса ресурсов, импортированных из JS, ссылки CSS `url()` и ссылки на ресурсы в ваших `.html`-файлах — все они автоматически корректируются с учётом этой опции во время сборки.

Исключение — когда вам нужно динамически конкатенировать URL-адреса на лету. В этом случае можно использовать глобально внедряемую переменную `import.meta.env.BASE_URL`, которая будет содержать публичный базовый путь. Обратите внимание, что эта переменная статически заменяется во время сборки, поэтому она должна встречаться строго в неизменном виде (то есть `import.meta.env['BASE_URL']` работать не будет).

Для продвинутого управления базовым путём смотрите [Продвинутые опции базового пути](#advanced-base-options).

### Относительный базовый путь

Если вы не знаете базовый путь заранее, можно задать относительный базовый путь через `"base": "./"` или `"base": ""`. Тогда все сгенерированные URL-адреса будут относительными по отношению к каждому файлу.

> **Внимание: Поддержка старых браузеров при использовании относительного базового пути**
>
> Для относительного базового пути требуется поддержка `import.meta`. Если вам нужно поддерживать [браузеры, не поддерживающие `import.meta`](https://caniuse.com/mdn-javascript_operators_import_meta), вы можете воспользоваться [плагином `legacy`](https://github.com/vitejs/vite/tree/main/packages/plugin-legacy).

## Настройка сборки

Сборку можно настроить с помощью различных [опций конфигурации сборки](https://vite.dev/config/build-options). В частности, вы можете напрямую менять базовые [опции Rolldown](https://rolldown.rs/reference/) через `build.rolldownOptions`:

**`vite.config.js`**

```js
export default defineConfig({
  build: {
    rolldownOptions: {
      // https://rolldown.rs/reference/
    },
  },
})
```

Например, вы можете указать несколько выходных вариантов Rolldown с плагинами, применяемыми только во время сборки.

## Стратегия разбиения на чанки

Вы можете настроить способ разбиения на чанки с помощью [`build.rolldownOptions.output.codeSplitting`](https://rolldown.rs/reference/OutputOptions.codeSplitting) (см. [документацию Rolldown](https://rolldown.rs/in-depth/manual-code-splitting)). Если вы используете фреймворк, обратитесь к его документации по настройке способа разбиения на чанки.

## Обработка ошибок загрузки

Vite генерирует событие `vite:preloadError`, когда ему не удаётся загрузить динамические импорты. `event.payload` содержит исходную ошибку импорта. Если вызвать `event.preventDefault()`, ошибка не будет выброшена.

```js
window.addEventListener('vite:preloadError', (event) => {
  window.location.reload() // например, перезагрузить страницу
})
```

При новом развёртывании сервис хостинга может удалить ресурсы предыдущих развёртываний. В результате пользователь, посетивший ваш сайт до нового развёртывания, может столкнуться с ошибкой импорта. Эта ошибка возникает потому, что ресурсы, работающие на устройстве такого пользователя, устарели, и он пытается импортировать соответствующий старый чанк, который уже удалён. Это событие полезно для обработки подобной ситуации. В таком случае обязательно установите `Cache-Control: no-cache` для HTML-файла, иначе старые ресурсы будут по-прежнему упоминаться.

## Пересборка при изменении файлов

Вы можете включить наблюдатель rollup с помощью `vite build --watch`. Либо можно напрямую менять базовые [`WatcherOptions`](https://rolldown.rs/reference/InputOptions.watch) через `build.watch`:

**`vite.config.js`**

```js
export default defineConfig({
  build: {
    watch: {
      // https://rolldown.rs/reference/InputOptions.watch
    },
  },
})
```

При включённом флаге `--watch` изменения в собираемых файлах будут запускать пересборку. Обратите внимание, что изменения в конфигурации и её зависимостях требуют перезапуска команды сборки.

## Многостраничное приложение

Предположим, что у вас следующая структура исходного кода:

```
├── package.json
├── vite.config.js
├── index.html
├── main.js
└── nested
    ├── index.html
    └── nested.js
```

Во время разработки просто перейдите или сделайте ссылку на `/nested/` — это работает как ожидается, точно так же, как для обычного статического файлового сервера.

Во время сборки всё, что вам нужно сделать, — указать несколько `.html`-файлов в качестве точек входа:

**`vite.config.js`**

```js
import { dirname, resolve } from 'node:path'
import { defineConfig } from 'vite'

export default defineConfig({
  build: {
    rolldownOptions: {
      input: {
        main: resolve(import.meta.dirname, 'index.html'),
        nested: resolve(import.meta.dirname, 'nested/index.html'),
      },
    },
  },
})
```

Если вы указываете другой корень (root), помните, что `import.meta.dirname` при разрешении путей входа по-прежнему будет указывать на папку с вашим файлом `vite.config.js`. Поэтому вам нужно будет добавить ваш `root` в аргументы для `resolve`.

Обратите внимание, что для HTML-файлов Vite игнорирует имя, заданное для точки входа в объекте `rolldownOptions.input`, и вместо этого учитывает разрешённый id файла при генерации HTML-ресурса в папке dist. Это обеспечивает структуру, согласованную с тем, как работает dev-сервер.

## Режим библиотеки

Когда вы разрабатываете ориентированную на браузер библиотеку, вы, скорее всего, проводите большую часть времени на тестовой/демонстрационной странице, которая импортирует вашу настоящую библиотеку. С Vite вы можете использовать для этой цели ваш `index.html`, чтобы получить плавный процесс разработки.

Когда придёт время собрать библиотеку для распространения, используйте [опцию конфигурации `build.lib`](https://vite.dev/config/build-options#build-lib). Обязательно также вынесите во внешние зависимости (externalize) все зависимости, которые вы не хотите включать в бандл вашей библиотеки, например `vue` или `react`:

**vite.config.js (одна точка входа)**

```js
import { dirname, resolve } from 'node:path'
import { defineConfig } from 'vite'

export default defineConfig({
  build: {
    lib: {
      entry: resolve(import.meta.dirname, 'lib/main.js'),
      name: 'MyLib',
      // нужные расширения будут добавлены автоматически
      fileName: 'my-lib',
    },
    rolldownOptions: {
      // обязательно вынесите во внешние зависимости те, что не должны
      // попасть в бандл вашей библиотеки
      external: ['vue'],
      output: {
        // Предоставьте глобальные переменные для использования в UMD-сборке
        // для внешних зависимостей
        globals: {
          vue: 'Vue',
        },
      },
    },
  },
})
```

**vite.config.js (несколько точек входа)**

```js
import { dirname, resolve } from 'node:path'
import { defineConfig } from 'vite'

export default defineConfig({
  build: {
    lib: {
      entry: {
        'my-lib': resolve(import.meta.dirname, 'lib/main.js'),
        secondary: resolve(import.meta.dirname, 'lib/secondary.js'),
      },
      name: 'MyLib',
    },
    rolldownOptions: {
      // обязательно вынесите во внешние зависимости те, что не должны
      // попасть в бандл вашей библиотеки
      external: ['vue'],
      output: {
        // Предоставьте глобальные переменные для использования в UMD-сборке
        // для внешних зависимостей
        globals: {
          vue: 'Vue',
        },
      },
    },
  },
})
```

Файл точки входа должен содержать экспорты, которые смогут импортировать пользователи вашего пакета:

**`lib/main.js`**

```js
import Foo from './Foo.vue'
import Bar from './Bar.vue'
export { Foo, Bar }
```

Запуск `vite build` с такой конфигурацией использует пресет Rollup, ориентированный на поставку библиотек, и создаёт два формата бандла:

- `es` и `umd` (для одной точки входа)
- `es` и `cjs` (для нескольких точек входа)

Форматы можно настроить с помощью опции [`build.lib.formats`](https://vite.dev/config/build-options#build-lib).

```
$ vite build
building for production...
dist/my-lib.js      0.08 kB / gzip: 0.07 kB
dist/my-lib.umd.cjs 0.30 kB / gzip: 0.16 kB
```

Рекомендуемый `package.json` для вашей библиотеки:

**package.json (одна точка входа)**

```json
{
  "name": "my-lib",
  "type": "module",
  "files": ["dist"],
  "main": "./dist/my-lib.umd.cjs",
  "module": "./dist/my-lib.js",
  "exports": {
    ".": {
      "import": "./dist/my-lib.js",
      "require": "./dist/my-lib.umd.cjs"
    }
  }
}
```

**package.json (несколько точек входа)**

```json
{
  "name": "my-lib",
  "type": "module",
  "files": ["dist"],
  "main": "./dist/my-lib.cjs",
  "module": "./dist/my-lib.js",
  "exports": {
    ".": {
      "import": "./dist/my-lib.js",
      "require": "./dist/my-lib.cjs"
    },
    "./secondary": {
      "import": "./dist/secondary.js",
      "require": "./dist/secondary.cjs"
    }
  }
}
```

### Поддержка CSS

Если ваша библиотека импортирует какой-либо CSS, он будет собран в единый CSS-файл рядом с собранными JS-файлами, например `dist/my-lib.css`. Имя по умолчанию соответствует `build.lib.fileName`, но его также можно изменить с помощью [`build.lib.cssFileName`](https://vite.dev/config/build-options#build-lib).

Вы можете экспортировать CSS-файл в вашем `package.json`, чтобы пользователи могли его импортировать:

```json
{
  "name": "my-lib",
  "type": "module",
  "files": ["dist"],
  "main": "./dist/my-lib.umd.cjs",
  "module": "./dist/my-lib.js",
  "exports": {
    ".": {
      "import": "./dist/my-lib.js",
      "require": "./dist/my-lib.umd.cjs"
    },
    "./style.css": "./dist/my-lib.css"
  }
}
```

> **Совет: Расширения файлов**
>
> Если `package.json` не содержит `"type": "module"`, Vite сгенерирует другие расширения файлов для совместимости с Node.js. `.js` станет `.mjs`, а `.cjs` станет `.js`.

> **Совет: Переменные окружения**
>
> В режиме библиотеки все обращения к [`import.meta.env.*`](https://vite.dev/guide/env-and-mode) статически заменяются при сборке для продакшна. Однако обращения к `process.env.*` — нет, чтобы потребители вашей библиотеки могли изменять их динамически. Если это нежелательно, вы можете, например, использовать `define: { 'process.env.NODE_ENV': '"production"' }`, чтобы заменить их статически, либо использовать [`esm-env`](https://github.com/benmccann/esm-env) для лучшей совместимости с бандлерами и средами выполнения.

> **Внимание: Продвинутое использование**
>
> Режим библиотеки включает простую и «предвзятую» (opinionated) конфигурацию для ориентированных на браузер библиотек и библиотек JS-фреймворков. Если вы собираете небраузерные библиотеки или вам требуются продвинутые процессы сборки, вы можете напрямую использовать [tsdown](https://tsdown.dev/) или [Rolldown](https://rolldown.rs/).

## Продвинутые опции базового пути

> **Внимание**
>
> Эта возможность экспериментальная. [Оставить отзыв](https://github.com/vitejs/vite/discussions/13834).

В продвинутых сценариях развёрнутые ресурсы и публичные файлы могут располагаться по разным путям — например, чтобы использовать разные стратегии кэширования.
Пользователь может выбрать развёртывание по трём разным путям:

- Сгенерированные входные HTML-файлы (которые могут обрабатываться во время SSR)
- Сгенерированные хешированные ресурсы (JS, CSS и другие типы файлов, например изображения)
- Скопированные [публичные файлы](https://vite.dev/guide/assets#the-public-directory)

Единого статического [базового пути](#public-base-path) в таких сценариях недостаточно. Vite предоставляет экспериментальную поддержку продвинутых опций базового пути во время сборки — через `experimental.renderBuiltUrl`.

```ts
experimental: {
  renderBuiltUrl(filename, { hostType }) {
    if (hostType === 'js') {
      return { runtime: `window.__toCdnUrl(${JSON.stringify(filename)})` }
    } else {
      return { relative: true }
    }
  },
},
```

Если хешированные ресурсы и публичные файлы развёртываются не вместе, опции для каждой группы можно задать независимо, используя `type` ресурса, передаваемый во втором параметре `context` функции.

```ts
experimental: {
  renderBuiltUrl(filename, { hostId, hostType, type }) {
    if (type === 'public') {
      return 'https://www.domain.com/' + filename
    } else if (path.extname(hostId) === '.js') {
      return {
        runtime: `window.__assetsPath(${JSON.stringify(filename)})`
      }
    } else {
      return 'https://cdn.domain.com/assets/' + filename
    }
  },
},
```

Обратите внимание, что передаваемый `filename` — это декодированный URL, и если функция возвращает строку URL, она также должна быть декодированной. Vite автоматически обработает кодирование при рендеринге URL-адресов. Если возвращается объект с `runtime`, кодирование нужно обрабатывать самостоятельно там, где это необходимо, поскольку код runtime будет отрендерен как есть.
