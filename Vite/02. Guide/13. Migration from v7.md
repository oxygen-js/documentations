# Миграция с версии 7

Если вы переходите с `rolldown-vite` — технического превью-выпуска Vite с интегрированным Rolldown для версий 6 и 7 — применимы только те разделы, в заголовке которых стоит метка NRV.

## Изменение целевого браузера по умолчанию (NRV)

Значение по умолчанию для `build.target` и значение `'baseline-widely-available'` обновлены до более новых версий браузеров:

- Chrome 107 → 111
- Edge 107 → 111
- Firefox 104 → 114
- Safari 16.0 → 16.4

Эти версии браузеров соответствуют наборам возможностей [Baseline Widely Available](https://web-platform-dx.github.io/web-features/) по состоянию на 2026-01-01. Иными словами, все они были выпущены примерно два с половиной года назад.

## Rolldown

Vite 8 использует инструменты на основе [Rolldown](https://rolldown.rs/) и [Oxc](https://oxc.rs/) вместо [esbuild](https://esbuild.github.io/) и [Rollup](https://rollupjs.org/).

### Постепенная миграция

Пакет `rolldown-vite` реализует Vite 7 на Rolldown без остальных изменений Vite 8. Его можно использовать как промежуточный шаг для перехода на Vite 8. Чтобы переключиться с Vite 7 на `rolldown-vite`, см. [руководство по интеграции Rolldown](https://v7.vite.dev/guide/rolldown) в документации Vite 7.

Пользователям, переходящим с `rolldown-vite` на Vite 8, достаточно отменить изменения зависимостей в `package.json` и обновиться до Vite 8:

```json
{
  "devDependencies": {
    "vite": "npm:rolldown-vite@7.2.2"
    "vite": "^8.0.0"
  }
}
```

### Оптимизатор зависимостей теперь использует Rolldown

Для оптимизации зависимостей теперь используется Rolldown вместо esbuild. Vite по-прежнему поддерживает [`optimizeDeps.esbuildOptions`](https://vite.dev/config/dep-optimization-options#optimizedeps-esbuildoptions) для обратной совместимости, автоматически преобразуя его в [`optimizeDeps.rolldownOptions`](https://vite.dev/config/dep-optimization-options#optimizedeps-rolldownoptions). Параметр `optimizeDeps.esbuildOptions` теперь объявлен устаревшим и будет удалён в будущем, поэтому мы рекомендуем перейти на `optimizeDeps.rolldownOptions`.

Следующие параметры преобразуются автоматически:

- [`esbuildOptions.minify`](https://esbuild.github.io/api/#minify) -> [`rolldownOptions.output.minify`](https://rolldown.rs/reference/OutputOptions.minify)
- [`esbuildOptions.treeShaking`](https://esbuild.github.io/api/#tree-shaking) -> [`rolldownOptions.treeshake`](https://rolldown.rs/reference/InputOptions.treeshake)
- [`esbuildOptions.define`](https://esbuild.github.io/api/#define) -> [`rolldownOptions.transform.define`](https://rolldown.rs/reference/InputOptions.transform#define)
- [`esbuildOptions.loader`](https://esbuild.github.io/api/#loader) -> [`rolldownOptions.moduleTypes`](https://rolldown.rs/reference/InputOptions.moduleTypes)
- [`esbuildOptions.preserveSymlinks`](https://esbuild.github.io/api/#preserve-symlinks) -> [`!rolldownOptions.resolve.symlinks`](https://rolldown.rs/reference/InputOptions.resolve#symlinks)
- [`esbuildOptions.resolveExtensions`](https://esbuild.github.io/api/#resolve-extensions) -> [`rolldownOptions.resolve.extensions`](https://rolldown.rs/reference/InputOptions.resolve#extensions)
- [`esbuildOptions.mainFields`](https://esbuild.github.io/api/#main-fields) -> [`rolldownOptions.resolve.mainFields`](https://rolldown.rs/reference/InputOptions.resolve#mainfields)
- [`esbuildOptions.conditions`](https://esbuild.github.io/api/#conditions) -> [`rolldownOptions.resolve.conditionNames`](https://rolldown.rs/reference/InputOptions.resolve#conditionnames)
- [`esbuildOptions.keepNames`](https://esbuild.github.io/api/#keep-names) -> [`rolldownOptions.output.keepNames`](https://rolldown.rs/reference/OutputOptions.keepNames)
- [`esbuildOptions.platform`](https://esbuild.github.io/api/#platform) -> [`rolldownOptions.platform`](https://rolldown.rs/reference/InputOptions.platform)
- [`esbuildOptions.plugins`](https://esbuild.github.io/plugins/) -> [`rolldownOptions.plugins`](https://rolldown.rs/reference/InputOptions.plugins) (частичная поддержка)

Параметры, установленные слоем совместимости, можно получить в хуке `configResolved`:

```js
const plugin = {
  name: 'log-config',
  configResolved(config) {
    console.log('options', config.optimizeDeps.rolldownOptions)
  },
},
```

### Преобразования JavaScript через Oxc

Для преобразования JavaScript теперь используется Oxc вместо esbuild. Vite по-прежнему поддерживает параметр [`esbuild`](https://vite.dev/config/shared-options#esbuild) для обратной совместимости, автоматически преобразуя его в [`oxc`](https://vite.dev/config/shared-options#oxc). Параметр `esbuild` теперь объявлен устаревшим и будет удалён в будущем, поэтому мы рекомендуем перейти на `oxc`.

Следующие параметры преобразуются автоматически:

- `esbuild.jsxInject` -> `oxc.jsxInject`
- `esbuild.include` -> `oxc.include`
- `esbuild.exclude` -> `oxc.exclude`
- [`esbuild.jsx`](https://esbuild.github.io/api/#jsx) -> [`oxc.jsx`](https://oxc.rs/docs/guide/usage/transformer/jsx)
  - `esbuild.jsx: 'preserve'` -> `oxc.jsx: 'preserve'`
  - `esbuild.jsx: 'automatic'` -> `oxc.jsx: { runtime: 'automatic' }`
    - [`esbuild.jsxImportSource`](https://esbuild.github.io/api/#jsx-import-source) -> `oxc.jsx.importSource`
  - `esbuild.jsx: 'transform'` -> `oxc.jsx: { runtime: 'classic' }`
    - [`esbuild.jsxFactory`](https://esbuild.github.io/api/#jsx-factory) -> `oxc.jsx.pragma`
    - [`esbuild.jsxFragment`](https://esbuild.github.io/api/#jsx-fragment) -> `oxc.jsx.pragmaFrag`
  - [`esbuild.jsxDev`](https://esbuild.github.io/api/#jsx-dev) -> `oxc.jsx.development`
  - [`esbuild.jsxSideEffects`](https://esbuild.github.io/api/#jsx-side-effects) -> `oxc.jsx.pure`
- [`esbuild.define`](https://esbuild.github.io/api/#define) -> [`oxc.define`](https://oxc.rs/docs/guide/usage/transformer/global-variable-replacement#define)
- [`esbuild.banner`](https://esbuild.github.io/api/#banner) -> пользовательский плагин, использующий хук transform
- [`esbuild.footer`](https://esbuild.github.io/api/#footer) -> пользовательский плагин, использующий хук transform

Параметр [`esbuild.supported`](https://esbuild.github.io/api/#supported) не поддерживается Oxc. Если он вам нужен, см. [oxc-project/oxc#15373](https://github.com/oxc-project/oxc/issues/15373).

Параметры, установленные слоем совместимости, можно получить в хуке `configResolved`:

```js
const plugin = {
  name: 'log-config',
  configResolved(config) {
    console.log('options', config.oxc)
  },
},
```

В настоящее время преобразователь Oxc не поддерживает понижение (lowering) нативных декораторов, так как мы ждём продвижения спецификации, см. ([oxc-project/oxc#9170](https://github.com/oxc-project/oxc/issues/9170)).

#### Запасные варианты через esbuild

`esbuild` больше не используется Vite напрямую и теперь является необязательной зависимостью. Если вы используете плагин, который применяет функцию `transformWithEsbuild`, вам нужно установить `esbuild` как `devDependency`. Функция `transformWithEsbuild` объявлена устаревшей и будет удалена в будущем. Мы рекомендуем перейти на новую функцию `transformWithOxc`.

### Минификация JavaScript через Oxc

Для минификации JavaScript теперь используется минификатор Oxc вместо esbuild. Вы можете использовать устаревший параметр [`build.minify: 'esbuild'`](https://vite.dev/config/build-options#build-minify), чтобы вернуться к esbuild. Этот параметр конфигурации будет удалён в будущем, и вам потребуется установить `esbuild` как `devDependency`, так как Vite больше не зависит от esbuild напрямую.

Если вы использовали параметры `esbuild.minify*` для управления поведением минификации, теперь можно использовать `build.rolldownOptions.output.minify`. Если вы использовали параметр `esbuild.drop`, теперь можно использовать [параметры `build.rolldownOptions.output.minify.compress.drop*`](https://oxc.rs/docs/guide/usage/minifier/dead-code-elimination).

Мангление свойств (property mangling) и связанные с ним параметры ([`mangleProps`, `reserveProps`, `mangleQuoted`, `mangleCache`](https://esbuild.github.io/api/#mangle-props)) не поддерживаются Oxc. Если они вам нужны, см. [oxc-project/oxc#15375](https://github.com/oxc-project/oxc/issues/15375).

esbuild и минификатор Oxc делают немного разные предположения об исходном коде. Если вы подозреваете, что минификатор ломает ваш код, вы можете сравнить эти предположения здесь:

- [Предположения минификатора esbuild](https://esbuild.github.io/api/#minify-considerations)
- [Предположения минификатора Oxc](https://github.com/oxc-project/oxc/blob/main/crates/oxc_minifier/docs/ASSUMPTIONS.md)

Пожалуйста, сообщайте о любых обнаруженных вами проблемах, связанных с минификацией в ваших JavaScript-приложениях.

### Минификация CSS через Lightning CSS

По умолчанию для минификации CSS теперь используется [Lightning CSS](https://lightningcss.dev/). Вы можете использовать параметр [`build.cssMinify: 'esbuild'`](https://vite.dev/config/build-options#build-cssminify), чтобы вернуться к esbuild. Обратите внимание, что вам потребуется установить `esbuild` как `devDependency`.

Lightning CSS лучше поддерживает понижение синтаксиса, и размер вашего CSS-бандла может немного увеличиться.

### Согласованная совместимость с CommonJS

Импорт `default` из модуля CommonJS (CJS) теперь обрабатывается согласованным образом.

Если выполняется одно из следующих условий, импорт `default` — это значение `module.exports` импортируемого CJS-модуля. В противном случае импорт `default` — это значение `module.exports.default` импортируемого CJS-модуля:

- Импортирующий файл имеет расширение `.mjs` или `.mts`.
- В ближайшем `package.json` импортирующего файла поле `type` установлено в `module`.
- Значение `module.exports.__esModule` импортируемого CJS-модуля не равно true.

<details><summary>Предыдущее поведение</summary>

В режиме разработки, если выполняется одно из следующих условий, импорт `default` — это значение `module.exports` импортируемого CJS-модуля. В противном случае импорт `default` — это значение `module.exports.default` импортируемого CJS-модуля:

- _Импортирующий файл включён в оптимизацию зависимостей_ и имеет расширение `.mjs` или `.mts`.
- _Импортирующий файл включён в оптимизацию зависимостей_, и в ближайшем `package.json` импортирующего файла поле `type` установлено в `module`.
- Значение `module.exports.__esModule` импортируемого CJS-модуля не равно true.

При сборке условия были такими:

- Значение `module.exports.__esModule` импортируемого CJS-модуля не равно true.
- _Свойство `default` у `module.exports` отсутствует_.

(при условии, что [`build.commonjsOptions.defaultIsModuleExports`](https://github.com/rollup/plugins/tree/master/packages/commonjs#defaultismoduleexports) не изменён со значения по умолчанию `'auto'`)

</details>

Подробнее об этой проблеме см. в документации Rolldown: [Неоднозначный импорт `default` из CJS-модулей — Bundling CJS | Rolldown](https://rolldown.rs/in-depth/bundling-cjs#ambiguous-default-import-from-cjs-modules).

Это изменение может сломать некоторый существующий код, импортирующий CJS-модули. Вы можете использовать устаревший параметр `legacy.inconsistentCjsInterop: true`, чтобы временно восстановить прежнее поведение. Если вы обнаружите пакет, затронутый этим изменением, сообщите об этом автору пакета или отправьте ему pull request. Обязательно приложите ссылку на документацию Rolldown выше, чтобы автор мог понять контекст.

### Удалено разрешение модулей на основе определения формата

Когда в `package.json` присутствуют оба поля `browser` и `module`, Vite раньше разрешал поле на основе содержимого файла и выбирал ESM-файл для браузеров. Это было введено, потому что некоторые пакеты использовали поле `module` для указания на ESM-файлы для Node.js, а некоторые другие пакеты использовали поле `browser` для указания на UMD-файлы для браузеров. Поскольку современное поле `exports` решило эту проблему и теперь принято многими пакетами, Vite больше не использует эту эвристику и всегда учитывает порядок в параметре [`resolve.mainFields`](https://vite.dev/config/shared-options#resolve-mainfields). Если вы полагались на это поведение, вы можете использовать параметр [`resolve.alias`](https://vite.dev/config/shared-options#resolve-alias), чтобы сопоставить поле с нужным файлом, или применить патч с помощью вашего менеджера пакетов (например, `patch-package`, `pnpm patch`).

### Вызовы require для внешних (externalized) модулей

Вызовы `require` для внешних модулей теперь сохраняются как вызовы `require` и не преобразуются в инструкции `import`. Это сделано для сохранения семантики вызовов `require`. Если вы хотите преобразовать их в инструкции `import`, вы можете использовать [встроенный плагин Rolldown `esmExternalRequirePlugin`](https://rolldown.rs/builtin-plugins/esm-external-require), который реэкспортируется из `vite`.

```js
import { defineConfig, esmExternalRequirePlugin } from 'vite'

export default defineConfig({
  // ...
  plugins: [
    esmExternalRequirePlugin({
      external: ['react', 'vue', /^node:/],
    }),
  ],
})
```

Подробнее см. в документации Rolldown: [Внешние модули `require` — Bundling CJS | Rolldown](https://rolldown.rs/in-depth/bundling-cjs#require-external-modules).

### `import.meta.url` в UMD / IIFE

`import.meta.url` больше не полифилится в форматах вывода UMD / IIFE. По умолчанию он будет заменён на `undefined`. Если вы предпочитаете прежнее поведение, вы можете использовать параметр [`define`](https://vite.dev/config/shared-options#define) вместе с параметром [`build.rolldownOptions.output.intro`](https://rolldown.rs/reference/OutputOptions.intro). Подробнее см. в документации Rolldown: [Известные свойства `import.meta` — Non ESM Output Formats | Rolldown](https://rolldown.rs/in-depth/non-esm-output-formats#well-known-import-meta-properties).

### Удалён параметр `build.rollupOptions.watch.chokidar`

Параметр `build.rollupOptions.watch.chokidar` был удалён. Пожалуйста, перейдите на параметр [`build.rolldownOptions.watch.watcher`](https://rolldown.rs/reference/InputOptions.watch#watcher).

### Удалена объектная форма `build.rollupOptions.output.manualChunks` и объявлена устаревшей её функциональная форма

Объектная форма параметра `output.manualChunks` больше не поддерживается. Функциональная форма `output.manualChunks` объявлена устаревшей. У Rolldown есть более гибкий параметр [`codeSplitting`](https://rolldown.rs/reference/OutputOptions.codeSplitting). Подробнее о `codeSplitting` см. в документации Rolldown: [Ручное разделение кода — Rolldown](https://rolldown.rs/in-depth/manual-code-splitting).

### `build()` выбрасывает `BundleError`

_Это изменение затрагивает только пользователей JS API._

`build()` теперь выбрасывает [`BundleError`](https://rolldown.rs/reference/TypeAlias.BundleError) вместо необработанной ошибки, выброшенной в плагине. `BundleError` типизирован как `Error & { errors?: RolldownError[] }` и оборачивает отдельные ошибки в массив `errors`. Если вам нужны отдельные ошибки, обращайтесь к `.errors`:

```js
try {
  await build()
} catch (e) {
  if (e.errors) {
    for (const error of e.errors) {
      console.log(error.code) // код ошибки
    }
  }
}
```

### Поддержка типов модулей и автоопределение

_Это изменение затрагивает только авторов плагинов._

У Rolldown есть экспериментальная поддержка [типов модулей](https://rolldown.rs/guide/notable-features#module-types), аналогичная [параметру `loader` в esbuild](https://esbuild.github.io/api/#loader). Из-за этого Rolldown автоматически задаёт тип модуля на основе расширения разрешённого идентификатора. Если вы преобразуете содержимое из других типов модулей в JavaScript в хуках `load` или `transform`, вам может понадобиться добавить `moduleType: 'js'` к возвращаемому значению:

```js
const plugin = {
  name: 'txt-loader',
  load(id) {
    if (id.endsWith('.txt')) {
      const content = fs.readFile(id, 'utf-8')
      return {
        code: `export default ${JSON.stringify(content)}`,
        moduleType: 'js',
      }
    }
  },
}
```

### Прочие связанные устаревания

Следующие параметры объявлены устаревшими и будут удалены в будущем:

- `build.rollupOptions`: переименован в `build.rolldownOptions`
- `worker.rollupOptions`: переименован в `worker.rolldownOptions`
- `build.commonjsOptions`: теперь не выполняет никаких действий (no-op)
- `build.dynamicImportVarsOptions.warnOnError`: теперь не выполняет никаких действий (no-op)
- `resolve.alias[].customResolver`: вместо этого используйте пользовательский плагин с хуком `resolveId` и `enforce: 'pre'`

## Удалённые устаревшие возможности (NRV)

- Передача URL в `import.meta.hot.accept` больше не поддерживается. Пожалуйста, передавайте id вместо этого. ([#21382](https://github.com/vitejs/vite/pull/21382))

## Дополнительно

Ожидается, что эти ломающие изменения затронут лишь меньшинство сценариев использования:

- [Extglobs](https://github.com/micromatch/picomatch/blob/master/README.md#extglobs) пока не поддерживаются ([rolldown-vite#365](https://github.com/vitejs/rolldown-vite/issues/365))
- Устаревшие пространства имён TypeScript (legacy namespace) поддерживаются лишь частично. Подробнее см. [соответствующую документацию преобразователя Oxc](https://oxc.rs/docs/guide/usage/transformer/typescript.html#partial-namespace-support).
- `define` не разделяет ссылку для объектов: когда вы передаёте объект в качестве значения в `define`, каждая переменная получит отдельную копию объекта. Подробнее см. [соответствующую документацию преобразователя Oxc](https://oxc.rs/docs/guide/usage/transformer/global-variable-replacement#define).
- Изменения объекта `bundle` (`bundle` — это объект, передаваемый в хуки `generateBundle` / `writeBundle` и возвращаемый функцией `build`):
  - Присваивание `bundle[foo]` не поддерживается. Rollup также не рекомендует этого делать. Пожалуйста, используйте `this.emitFile()` вместо этого.
  - ссылка не разделяется между хуками ([rolldown-vite#410](https://github.com/vitejs/rolldown-vite/issues/410))
  - `structuredClone(bundle)` завершается ошибкой `DataCloneError: #<Object> could not be cloned`. Это больше не поддерживается. Пожалуйста, клонируйте его через `structuredClone({ ...bundle })`. ([rolldown-vite#128](https://github.com/vitejs/rolldown-vite/issues/128))
- Все параллельные хуки в Rollup работают как последовательные. Подробнее см. [документацию Rolldown](https://rolldown.rs/apis/plugin-api#sequential-hook-execution).
- `"use strict";` иногда не вставляется. Подробнее см. [документацию Rolldown](https://rolldown.rs/in-depth/directives).
- Преобразование в ES5 и ниже с помощью plugin-legacy не поддерживается ([rolldown-vite#452](https://github.com/vitejs/rolldown-vite/issues/452))
- Передача одного и того же браузера с несколькими его версиями в параметр `build.target` теперь вызывает ошибку: esbuild выбирал последнюю из версий, что, вероятно, не соответствовало вашим намерениям.
- Отсутствие поддержки в Rolldown: следующие возможности не поддерживаются Rolldown и больше не поддерживаются Vite.
  - `build.rollupOptions.output.format: 'system'` ([rolldown#2387](https://github.com/rolldown/rolldown/issues/2387))
  - `build.rollupOptions.output.format: 'amd'` ([rolldown#2387](https://github.com/rolldown/rolldown/issues/2528))
  - хук `shouldTransformCachedModule` ([rolldown#4389](https://github.com/rolldown/rolldown/issues/4389))
  - хук `resolveImportMeta` ([rolldown#1010](https://github.com/rolldown/rolldown/issues/1010))
  - хук `renderDynamicImport` ([rolldown#4532](https://github.com/rolldown/rolldown/issues/4532))
  - хук `resolveFileUrl`
- Функции `parseAst` / `parseAstAsync` теперь объявлены устаревшими в пользу функций `parseSync` / `parse`, которые имеют больше возможностей.
- комментарии удаляются перед хуком `renderChunk`, а не после хука `renderChunk`
- комментарии, отличные от перечисленных [здесь](https://rolldown.rs/reference/OutputOptions.comments), перемещаются, тогда как Rollup удаляет комментарии только если удаляется соседний код
