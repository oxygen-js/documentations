# Серверный рендеринг (SSR)

> **Примечание**
> SSR в данном случае относится к фронтенд-фреймворкам (например, React, Preact, Vue и Svelte), которые поддерживают запуск одного и того же приложения в Node.js, предварительный рендеринг его в HTML и последующую гидратацию на клиенте. Если вы ищете интеграцию с традиционными серверными фреймворками, вместо этого ознакомьтесь с [руководством по интеграции с бэкендом](https://vite.dev/guide/backend-integration).
>
> Дальнейшее руководство также предполагает наличие опыта работы с SSR в выбранном вами фреймворке и будет сосредоточено только на деталях интеграции, специфичных для Vite.

> **Внимание: Низкоуровневый API**
> Это низкоуровневый API, предназначенный для авторов библиотек и фреймворков. Если ваша цель — создать приложение, то сначала обязательно ознакомьтесь с более высокоуровневыми SSR-плагинами и инструментами в [разделе Awesome Vite SSR](https://github.com/vitejs/awesome-vite#ssr). При этом многие приложения успешно строятся напрямую поверх нативного низкоуровневого API Vite.
>
> В настоящее время Vite работает над улучшенным SSR API на основе [API окружений](https://github.com/vitejs/vite/discussions/16358). Подробности смотрите по ссылке.

## Структура исходников

Типичное SSR-приложение будет иметь следующую структуру исходных файлов:

```
- index.html
- server.js # основной сервер приложения
- src/
  - main.js          # экспортирует независимый от окружения (универсальный) код приложения
  - entry-client.js  # монтирует приложение в DOM-элемент
  - entry-server.js  # рендерит приложение с помощью SSR API фреймворка
```

Файл `index.html` должен ссылаться на `entry-client.js` и содержать плейсхолдер, в который будет внедряться отрендеренная на сервере разметка:

```html
<div id="app"><!--ssr-outlet--></div>
<script type="module" src="/src/entry-client.js"></script>
```

Вместо `<!--ssr-outlet-->` вы можете использовать любой предпочитаемый плейсхолдер, если его можно точно заменить.

## Условная логика

Если вам нужно выполнить условную логику в зависимости от того, SSR это или клиент, можно использовать

```js
if (import.meta.env.SSR) {
  // ... логика только для сервера
}
```

Это статически заменяется во время сборки, что позволяет выполнять tree-shaking для неиспользуемых веток.

## Настройка dev-сервера

При создании SSR-приложения вы, скорее всего, захотите иметь полный контроль над своим основным сервером и отделить Vite от продакшн-окружения. Поэтому рекомендуется использовать Vite в режиме middleware. Вот пример с [express](https://expressjs.com/):

```js
import fs from 'node:fs'
import path from 'node:path'
import express from 'express'
import { createServer as createViteServer } from 'vite'

async function createServer() {
  const app = express()

  // Создаём Vite-сервер в режиме middleware и настраиваем тип приложения
  // как 'custom', отключая собственную логику отдачи HTML в Vite, чтобы
  // родительский сервер мог взять управление на себя
  const vite = await createViteServer({
    server: { middlewareMode: true },
    appType: 'custom'
  })

  // Используем connect-экземпляр Vite как middleware. Если вы используете
  // собственный роутер express (express.Router()), следует использовать router.use
  // При перезапуске сервера (например, после того как пользователь изменит
  // vite.config.js) `vite.middlewares` по-прежнему будет той же самой
  // ссылкой (с новым внутренним стеком middleware, внедрённых Vite и плагинами).
  // Следующий код корректен даже после перезапусков.
  app.use(vite.middlewares)

  app.use('*all', async (req, res) => {
    // отдаём index.html — этим займёмся дальше
  })

  app.listen(5173)
}

createServer()
```

Здесь `vite` — это экземпляр [ViteDevServer](https://vite.dev/guide/api-javascript#vitedevserver). `vite.middlewares` — это экземпляр [Connect](https://github.com/senchalabs/connect), который можно использовать в качестве middleware в любом connect-совместимом фреймворке Node.js.

Следующий шаг — реализация обработчика `*` для отдачи отрендеренного на сервере HTML:

```js
app.use('*all', async (req, res, next) => {
  const url = req.originalUrl

  try {
    // 1. Читаем index.html
    let template = fs.readFileSync(
      path.resolve(import.meta.dirname, 'index.html'),
      'utf-8',
    )

    // 2. Применяем HTML-преобразования Vite. Это внедряет клиент Vite HMR,
    //    а также применяет HTML-преобразования из плагинов Vite, например,
    //    глобальные преамбулы из @vitejs/plugin-react
    template = await vite.transformIndexHtml(url, template)

    // 3. Загружаем серверную точку входа. ssrLoadModule автоматически преобразует
    //    ESM-исходный код так, чтобы его можно было использовать в Node.js!
    //    Бандлинг не требуется, при этом обеспечивается эффективная
    //    инвалидация, аналогичная HMR.
    const { render } = await vite.ssrLoadModule('/src/entry-server.js')

    // 4. Рендерим HTML приложения. Это предполагает, что экспортируемая
    //    из entry-server.js функция `render` вызывает соответствующие
    //    SSR API фреймворка, например ReactDOMServer.renderToString()
    const appHtml = await render(url)

    // 5. Внедряем отрендеренный приложением HTML в шаблон.
    const html = template.replace(`<!--ssr-outlet-->`, () => appHtml)

    // 6. Отправляем отрендеренный HTML обратно.
    res.status(200).set({ 'Content-Type': 'text/html' }).end(html)
  } catch (e) {
    // Если перехвачена ошибка, позволяем Vite исправить трассировку стека,
    // чтобы она сопоставлялась с вашим реальным исходным кодом.
    vite.ssrFixStacktrace(e)
    next(e)
  }
})
```

Скрипт `dev` в `package.json` также следует изменить, чтобы использовать вместо него серверный скрипт:

```diff
  "scripts": {
-   "dev": "vite"
+   "dev": "node server"
  }
```

## Сборка для продакшена

Чтобы подготовить SSR-проект к продакшену, нам нужно:

1. Собрать клиентскую сборку как обычно;
2. Собрать SSR-сборку, которую можно напрямую загрузить через `import()`, чтобы нам не приходилось проходить через `ssrLoadModule` в Vite;

Наши скрипты в `package.json` будут выглядеть так:

```json
{
  "scripts": {
    "dev": "node server",
    "build:client": "vite build --outDir dist/client",
    "build:server": "vite build --outDir dist/server --ssr src/entry-server.js"
  }
}
```

Обратите внимание на флаг `--ssr`, который указывает, что это SSR-сборка. Он также должен указывать SSR-точку входа.

Затем в `server.js` нам нужно добавить некоторую логику, специфичную для продакшена, проверяя `process.env.NODE_ENV`:

- Вместо чтения корневого `index.html` используйте `dist/client/index.html` в качестве шаблона, поскольку он содержит правильные ссылки на ресурсы клиентской сборки.

- Вместо `await vite.ssrLoadModule('/src/entry-server.js')` используйте `import('./dist/server/entry-server.js')` (этот файл является результатом SSR-сборки).

- Перенесите создание и всё использование dev-сервера `vite` в условные ветки только для разработки, затем добавьте middleware для отдачи статических файлов из `dist/client`.

За рабочей настройкой обращайтесь к [примерам проектов](#example-projects).

## Генерация директив предзагрузки

`vite build` поддерживает флаг `--ssrManifest`, который сгенерирует `.vite/ssr-manifest.json` в выходной директории сборки:

```diff
- "build:client": "vite build --outDir dist/client",
+ "build:client": "vite build --outDir dist/client --ssrManifest",
```

Приведённый выше скрипт теперь будет генерировать `dist/client/.vite/ssr-manifest.json` для клиентской сборки (да, SSR-манифест генерируется из клиентской сборки, потому что мы хотим сопоставить идентификаторы модулей с клиентскими файлами). Манифест содержит сопоставления идентификаторов модулей с их связанными чанками и файлами ресурсов.

Чтобы задействовать манифест, фреймворкам нужно предоставить способ собирать идентификаторы модулей тех компонентов, которые использовались во время серверного вызова рендеринга.

`@vitejs/plugin-vue` поддерживает это «из коробки» и автоматически регистрирует идентификаторы модулей использованных компонентов в связанном SSR-контексте Vue:

```js
const ctx = {}
const html = await vueServerRenderer.renderToString(app, ctx)
// ctx.modules теперь является Set идентификаторов модулей, которые использовались во время рендеринга
```

В продакшн-ветке `server.js` нам нужно прочитать манифест и передать его в функцию `render`, экспортируемую из `src/entry-server.js`. Это предоставит нам достаточно информации для рендеринга директив предзагрузки для файлов, используемых асинхронными маршрутами! Полный пример смотрите в [исходнике демо](https://github.com/vitejs/vite-plugin-vue/blob/main/playground/ssr-vue/src/entry-server.js). Вы также можете использовать эту информацию для [103 Early Hints](https://developer.mozilla.org/en-US/docs/Web/HTTP/Status/103).

## Предварительный рендеринг / SSG

Если маршруты и данные, необходимые для определённых маршрутов, известны заранее, мы можем предварительно отрендерить эти маршруты в статический HTML, используя ту же логику, что и продакшн-SSR. Это также можно рассматривать как форму генерации статических сайтов (SSG). Рабочий пример смотрите в [демо-скрипте предварительного рендеринга](https://github.com/vitejs/vite-plugin-vue/blob/main/playground/ssr-vue/prerender.js).

## Внешние зависимости SSR (SSR Externals)

По умолчанию при запуске SSR зависимости «выносятся вовне» (externalized) из системы модулей SSR-преобразования Vite. Это ускоряет как разработку, так и сборку.

Если зависимость нужно преобразовать конвейером Vite — например, потому что в ней используются нетранспилированные возможности Vite, — её можно добавить в [`ssr.noExternal`](https://vite.dev/config/ssr-options#ssr-noexternal).

Слинкованные зависимости по умолчанию не выносятся вовне, чтобы можно было воспользоваться HMR в Vite. Если это нежелательно — например, чтобы протестировать зависимости так, как если бы они не были слинкованы, — вы можете добавить её в [`ssr.external`](https://vite.dev/config/ssr-options#ssr-external).

> **Внимание: Работа с алиасами**
> Если вы настроили алиасы, перенаправляющие один пакет на другой, то для корректной работы с вынесенными вовне SSR-зависимостями вам, возможно, стоит настроить алиасы на реальные пакеты из `node_modules`. И [Yarn](https://classic.yarnpkg.com/en/docs/cli/add/#toc-yarn-add-alias), и [pnpm](https://pnpm.io/aliases/) поддерживают алиасинг через префикс `npm:`.

## Логика плагинов, специфичная для SSR

Некоторые фреймворки, такие как Vue или Svelte, компилируют компоненты в различные форматы в зависимости от того, клиент это или SSR. Для поддержки условных преобразований Vite передаёт дополнительное свойство `ssr` в объекте `options` следующих хуков плагинов:

- `resolveId`
- `load`
- `transform`

**Пример:**

```js
export function mySSRPlugin() {
  return {
    name: 'my-ssr',
    transform(code, id, options) {
      if (options?.ssr) {
        // выполнить преобразование, специфичное для ssr...
      }
    },
  }
}
```

Объект options в `load` и `transform` необязателен, Rollup в настоящее время не использует этот объект, но в будущем может расширить эти хуки дополнительными метаданными.

> **Примечание**
> До Vite 2.7 это сообщалось хукам плагинов через позиционный параметр `ssr` вместо использования объекта `options`. Все основные фреймворки и плагины обновлены, но вы можете встретить устаревшие статьи, использующие предыдущий API.

## Целевая платформа SSR (SSR Target)

По умолчанию целевой платформой для SSR-сборки является окружение node, но вы также можете запускать сервер в Web Worker. Разрешение точек входа пакетов различается для каждой платформы. Вы можете настроить целевую платформу на Web Worker, установив `ssr.target` в значение `'webworker'`.

## Бандл SSR (SSR Bundle)

В некоторых случаях, например для сред выполнения `webworker`, вам может понадобиться собрать вашу SSR-сборку в единый файл JavaScript. Вы можете включить это поведение, установив `ssr.noExternal` в `true`. Это сделает две вещи:

- Будет считать все зависимости как `noExternal`
- Будет выбрасывать ошибку, если импортируется любой встроенный модуль Node.js

## Условия разрешения SSR (SSR Resolve Conditions)

По умолчанию разрешение точек входа пакетов будет использовать условия, заданные в [`resolve.conditions`](https://vite.dev/config/shared-options#resolve-conditions), для SSR-сборки. Вы можете использовать [`ssr.resolve.conditions`](https://vite.dev/config/ssr-options#ssr-resolve-conditions) и [`ssr.resolve.externalConditions`](https://vite.dev/config/ssr-options#ssr-resolve-externalconditions), чтобы настроить это поведение.

## Vite CLI

CLI-команды `$ vite dev` и `$ vite preview` также можно использовать для SSR-приложений. Вы можете добавить свои SSR-middleware в сервер разработки с помощью [`configureServer`](https://vite.dev/guide/api-plugin#configureserver) и в сервер предпросмотра с помощью [`configurePreviewServer`](https://vite.dev/guide/api-plugin#configurepreviewserver).

> **Примечание**
> Используйте post-хук, чтобы ваш SSR-middleware запускался _после_ middleware Vite.
