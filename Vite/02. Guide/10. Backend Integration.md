# Интеграция с бэкендом

> **Примечание**
>
> Если вы хотите раздавать HTML с помощью традиционного бэкенда (например, Rails, Laravel), но использовать Vite для раздачи ресурсов, посмотрите готовые интеграции, перечисленные в [Awesome Vite](https://github.com/vitejs/awesome-vite#integrations-with-backends).
>
> Если вам нужна кастомная интеграция, вы можете следовать шагам из этого руководства, чтобы настроить её вручную.

1. В конфигурации Vite задайте точку входа и включите манифест сборки:

   **vite.config.js**

   ```js
   import { defineConfig } from 'vite'
   export default defineConfig({
     server: {
       cors: {
         // источник (origin), к которому вы будете обращаться через браузер
         origin: 'http://my-backend.example.com',
       },
     },
     build: {
       // генерирует .vite/manifest.json в outDir
       manifest: true,
       rolldownOptions: {
         // переопределяет стандартную точку входа .html
         input: '/path/to/main.js',
       },
     },
   })
   ```

   Если вы не отключили [полифилл предзагрузки модулей (module preload polyfill)](https://vite.dev/config/build-options#build-polyfillmodulepreload), вам также нужно импортировать этот полифилл в точке входа:

   ```js
   // добавьте в начало точки входа вашего приложения
   import 'vite/modulepreload-polyfill'
   ```

2. Для разработки внедрите следующее в HTML-шаблон вашего сервера (замените `http://localhost:5173` на локальный URL, на котором запущен Vite):

   ```html
   <!-- в режиме разработки -->
   <script type="module" src="http://localhost:5173/@vite/client"></script>
   <script type="module" src="http://localhost:5173/main.js"></script>
   ```

   Чтобы корректно раздавать ресурсы, у вас есть два варианта:
   - Убедитесь, что сервер настроен на проксирование запросов статических ресурсов на сервер Vite
   - Задайте [`server.origin`](https://vite.dev/config/server-options#server-origin), чтобы сгенерированные URL ресурсов разрешались с использованием URL бэкенд-сервера, а не относительного пути

   Это необходимо, чтобы такие ресурсы, как изображения, загружались корректно.

   Обратите внимание: если вы используете React с `@vitejs/plugin-react`, вам также потребуется добавить это перед приведёнными выше скриптами, поскольку плагин не может изменить раздаваемый вами HTML (замените `http://localhost:5173` на локальный URL, на котором запущен Vite):

   ```html
   <script type="module">
     import RefreshRuntime from 'http://localhost:5173/@react-refresh'
     RefreshRuntime.injectIntoGlobalHook(window)
     window.$RefreshReg$ = () => {}
     window.$RefreshSig$ = () => (type) => type
     window.__vite_plugin_react_preamble_installed__ = true
   </script>
   ```

3. Для продакшена после выполнения `vite build` рядом с другими файлами ресурсов будет сгенерирован файл `.vite/manifest.json`. Пример файла манифеста выглядит так:

   **.vite/manifest.json**

   ```json
   {
     "_shared-B7PI925R.js": {
       "file": "assets/shared-B7PI925R.js",
       "name": "shared",
       "css": ["assets/shared-ChJ_j-JJ.css"]
     },
     "_shared-ChJ_j-JJ.css": {
       "file": "assets/shared-ChJ_j-JJ.css",
       "src": "_shared-ChJ_j-JJ.css"
     },
     "logo.svg": {
       "file": "assets/logo-BuPIv-2h.svg",
       "src": "logo.svg"
     },
     "baz.js": {
       "file": "assets/baz-B2H3sXNv.js",
       "name": "baz",
       "src": "baz.js",
       "isDynamicEntry": true
     },
     "views/bar.js": {
       "file": "assets/bar-gkvgaI9m.js",
       "name": "bar",
       "src": "views/bar.js",
       "isEntry": true,
       "imports": ["_shared-B7PI925R.js"],
       "dynamicImports": ["baz.js"]
     },
     "views/foo.js": {
       "file": "assets/foo-BRBmoGS9.js",
       "name": "foo",
       "src": "views/foo.js",
       "isEntry": true,
       "imports": ["_shared-B7PI925R.js"],
       "css": ["assets/foo-5UjPuW-k.css"]
     }
   }
   ```

   Манифест сопоставляет исходные файлы с результатами их сборки и зависимостями:

   ```dot
   digraph manifest {
     rankdir=TB
     node [shape=box style="rounded,filled" fontname="Arial" fontsize=10 margin="0.2,0.1" fontcolor="${#3c3c43|#ffffff}" color="${#c2c2c4|#3c3f44}"]
     edge [color="${#67676c|#98989f}" fontname="Arial" fontsize=9 fontcolor="${#67676c|#98989f}"]
     bgcolor="transparent"

     foo [label="views/foo.js\n(entry)" fillcolor="${#e9eaff|#222541}"]
     bar [label="views/bar.js\n(entry)" fillcolor="${#e9eaff|#222541}"]
     shared [label="_shared-B7PI925R.js\n(common chunk)" fillcolor="${#f2ecfc|#2c273e}"]
     baz [label="baz.js\n(dynamic import)" fillcolor="${#fcf4dc|#38301a}"]
     foocss [label="foo.css" shape=ellipse fillcolor="${#fde4e8|#3a1d27}"]
     sharedcss [label="shared.css" shape=ellipse fillcolor="${#fde4e8|#3a1d27}"]
     logo [label="logo.svg\n(asset)" shape=ellipse fillcolor="${#def5ed|#15312d}"]

     foo -> shared [label="imports"]
     bar -> shared [label="imports"]
     bar -> baz [label="dynamicImports" style=dashed]
     foo -> foocss [label="css"]
     shared -> sharedcss [label="css"]
   }
   ```

   Манифест имеет структуру `Record<name, chunk>`, где каждый чанк соответствует интерфейсу `ManifestChunk`:

   ```ts
   interface ManifestChunk {
     /**
      * Имя входного файла этого чанка / ресурса, если известно
      */
     src?: string
     /**
      * Имя выходного файла этого чанка / ресурса
      */
     file: string
     /**
      * Список CSS-файлов, импортируемых этим чанком
      */
     css?: string[]
     /**
      * Список файлов ресурсов, импортируемых этим чанком, за исключением CSS-файлов
      */
     assets?: string[]
     /**
      * Является ли этот чанк или ресурс точкой входа
      */
     isEntry?: boolean
     /**
      * Имя этого чанка / ресурса, если известно
      */
     name?: string
     /**
      * Является ли этот чанк динамической точкой входа
      *
      * Это поле присутствует только в JS-чанках.
      */
     isDynamicEntry?: boolean
     /**
      * Список статически импортируемых этим чанком чанков
      *
      * Значения — это ключи манифеста. Это поле присутствует только в JS-чанках.
      */
     imports?: string[]
     /**
      * Список динамически импортируемых этим чанком чанков
      *
      * Значения — это ключи манифеста. Это поле присутствует только в JS-чанках.
      */
     dynamicImports?: string[]
   }
   ```

   Каждая запись в манифесте представляет одно из следующего:
   - **Чанки точек входа (Entry chunks)**: генерируются из файлов, указанных в [`build.rolldownOptions.input`](https://rolldown.rs/reference/InputOptions.input#input). У таких чанков `isEntry: true`, а их ключ — относительный путь src от корня проекта.
   - **Динамические чанки точек входа (Dynamic entry chunks)**: генерируются из динамических импортов. У таких чанков `isDynamicEntry: true`, а их ключ — относительный путь src от корня проекта.
   - **Чанки, не являющиеся точками входа (Non-entry chunks)**: их ключ — базовое имя сгенерированного файла с префиксом `_`.
   - **Чанки ресурсов (Asset chunks)**: генерируются из импортированных ресурсов, таких как изображения, шрифты. Их ключ — относительный путь src от корня проекта.
   - **CSS-файлы**: когда [`build.cssCodeSplit`](https://vite.dev/config/build-options#build-csscodesplit) равен `false`, генерируется единый CSS-файл с ключом `style.css`. Когда `build.cssCodeSplit` не равен `false`, ключ генерируется аналогично JS-чанкам (т. е. чанки точек входа не будут иметь префикс `_`, а чанки, не являющиеся точками входа, будут иметь префикс `_`).

   JS-чанки (чанки, отличные от ресурсов или CSS) содержат информацию о своих статических и динамических импортах (и то, и другое — это ключи, сопоставляемые с соответствующим чанком в манифесте). Чанки также перечисляют соответствующие им CSS-файлы и файлы ресурсов, если таковые имеются.

4. Вы можете использовать этот файл для формирования ссылок или директив предзагрузки с хешированными именами файлов.

   Ниже приведён пример HTML-шаблона для формирования корректных ссылок. Синтаксис здесь приведён только для пояснения — замените его на язык шаблонов вашего сервера. Функция `importedChunks` служит для иллюстрации и не предоставляется Vite.

   ```html
   <!-- в продакшене -->

   <!-- для каждого cssFile из manifest[name].css -->
   <link rel="stylesheet" href="/{{ cssFile }}" />

   <!-- для каждого chunk из importedChunks(manifest, name) -->
   <!-- для каждого cssFile из chunk.css -->
   <link rel="stylesheet" href="/{{ cssFile }}" />

   <script type="module" src="/{{ manifest[name].file }}"></script>

   <!-- для каждого chunk из importedChunks(manifest, name) -->
   <link rel="modulepreload" href="/{{ chunk.file }}" />
   ```

   В частности, бэкенд, генерирующий HTML, должен включать следующие теги при наличии файла манифеста и точки входа. Обратите внимание: соблюдение этого порядка рекомендуется для оптимальной производительности:
   1. Тег `<link rel="stylesheet">` для каждого файла из списка `css` чанка точки входа (если он есть)
   2. Рекурсивно обойдите все чанки из списка `imports` точки входа и включите тег `<link rel="stylesheet">` для каждого CSS-файла из списка `css` каждого импортированного чанка (если он есть).
   3. Тег для ключа `file` чанка точки входа. Это может быть `<script type="module">` для JavaScript или `<link rel="stylesheet">` для CSS.
   4. Опционально — тег `<link rel="modulepreload">` для `file` каждого импортированного JavaScript-чанка, снова рекурсивно обходя импорты, начиная с чанка точки входа.

   Следуя приведённому выше примеру манифеста, для точки входа `views/foo.js` в продакшене должны быть включены следующие теги:

   ```html
   <link rel="stylesheet" href="assets/foo-5UjPuW-k.css" />
   <link rel="stylesheet" href="assets/shared-ChJ_j-JJ.css" />
   <script type="module" src="assets/foo-BRBmoGS9.js"></script>
   <!-- опционально -->
   <link rel="modulepreload" href="assets/shared-B7PI925R.js" />
   ```

   А для точки входа `views/bar.js` должны быть включены следующие:

   ```html
   <link rel="stylesheet" href="assets/shared-ChJ_j-JJ.css" />
   <script type="module" src="assets/bar-gkvgaI9m.js"></script>
   <!-- опционально -->
   <link rel="modulepreload" href="assets/shared-B7PI925R.js" />
   ```

   <details><summary>Псевдореализация <code>importedChunks</code></summary>

   Пример псевдореализации `importedChunks` на TypeScript (её нужно будет адаптировать под ваш язык программирования и язык шаблонов):

   ```ts
   import type { Manifest, ManifestChunk } from 'vite'

   export default function importedChunks(
     manifest: Manifest,
     name: string,
   ): ManifestChunk[] {
     const seen = new Set<string>()

     function getImportedChunks(chunk: ManifestChunk): ManifestChunk[] {
       const chunks: ManifestChunk[] = []
       for (const file of chunk.imports ?? []) {
         const importee = manifest[file]
         if (seen.has(file)) {
           continue
         }
         seen.add(file)

         chunks.push(...getImportedChunks(importee))
         chunks.push(importee)
       }

       return chunks
     }

     return getImportedChunks(manifest[name])
   }
   ```

   </details>

   > **Поддержка import maps для чанков (экспериментально)**
   >
   > Если вы используете экспериментальную опцию [`build.chunkImportMap`](https://vite.dev/config/build-options#build-chunkimportmap), вам также нужно внедрить import map в HTML.
   >
   > Import map выводится в файл `importmap.json` в выходном каталоге. Обязательно внедрите тег `<script type="importmap">` перед любыми тегами `<script type="module">` или `<link rel="modulepreload">`.
