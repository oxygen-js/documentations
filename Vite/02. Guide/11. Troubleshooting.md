# Устранение неполадок

Дополнительную информацию смотрите также в [руководстве по устранению неполадок Rollup](https://rollupjs.org/troubleshooting/).

Если приведённые здесь советы не помогают, попробуйте задать вопрос в [GitHub Discussions](https://github.com/vitejs/vite/discussions) или в канале `#help` в [Vite Land Discord](https://chat.vite.dev).

## CLI

### `Error: Cannot find module 'C:\foo\bar&baz\vite\bin\vite.js'`

Путь к папке вашего проекта может содержать `&`, который не работает с `npm` в Windows ([npm/cmd-shim#45](https://github.com/npm/cmd-shim/issues/45)).

Вам потребуется сделать одно из двух:

- Перейти на другой менеджер пакетов (например, `pnpm`, `yarn`)
- Убрать `&` из пути к вашему проекту

## Конфигурация

### Этот пакет доступен только в формате ESM

При импорте пакета, доступного только в формате ESM, через `require` возникает следующая ошибка.

> Failed to resolve "foo". This package is ESM only but it was tried to load by `require`.

> Error [ERR_REQUIRE_ESM]: require() of ES Module /path/to/dependency.js from /path/to/vite.config.js not supported.
> Instead change the require of index.js in /path/to/vite.config.js to a dynamic import() which is available in all CommonJS modules.

В Node.js <=22 ESM-файлы по умолчанию нельзя загрузить через [`require`](https://nodejs.org/docs/latest-v22.x/api/esm.html#require).

Хотя это может работать при использовании [`--experimental-require-module`](https://nodejs.org/docs/latest-v22.x/api/modules.html#loading-ecmascript-modules-using-require), в Node.js >22 или в других средах выполнения, мы всё же рекомендуем преобразовать вашу конфигурацию в ESM одним из способов:

- добавить `"type": "module"` в ближайший `package.json`
- переименовать `vite.config.js`/`vite.config.ts` в `vite.config.mjs`/`vite.config.mts`

## Dev-сервер

### Запросы «зависают» навсегда

Если вы используете Linux, причиной проблемы могут быть ограничения на количество файловых дескрипторов и ограничения inotify. Поскольку Vite не собирает большинство файлов в бандл, браузеры могут запрашивать множество файлов, что требует множества файловых дескрипторов и приводит к превышению лимита.

Чтобы решить эту проблему:

- Увеличьте лимит файловых дескрипторов с помощью `ulimit`

  ```shell
  # Проверить текущий лимит
  $ ulimit -Sn
  # Изменить лимит (временно)
  $ ulimit -Sn 10000 # Возможно, потребуется изменить и жёсткий лимит
  # Перезапустите браузер
  ```

- Увеличьте следующие лимиты, связанные с inotify, с помощью `sysctl`

  ```shell
  # Проверить текущие лимиты
  $ sysctl fs.inotify
  # Изменить лимиты (временно)
  $ sudo sysctl fs.inotify.max_queued_events=16384
  $ sudo sysctl fs.inotify.max_user_instances=8192
  $ sudo sysctl fs.inotify.max_user_watches=524288
  ```

Если описанные выше шаги не помогают, попробуйте добавить `DefaultLimitNOFILE=65536` в качестве раскомментированного параметра в следующие файлы:

- /etc/systemd/system.conf
- /etc/systemd/user.conf

В Ubuntu Linux вам может потребоваться добавить строку `* - nofile 65536` в файл `/etc/security/limits.conf` вместо обновления конфигурационных файлов systemd.

Обратите внимание, что эти настройки сохраняются, но **требуется перезагрузка**.

Также возможен вариант, когда сервер запущен внутри devcontainer в VS Code — в этом случае запрос может выглядеть зависшим. Чтобы устранить эту проблему, смотрите
[Dev Containers / проброс портов в VS Code](#dev-containers-vs-code-port-forwarding).

### Vite падает с ошибкой ENOSPC

Если вы видите в Linux ошибку вроде этой:

> Error: ENOSPC: System limit for number of file watchers reached

Это происходит, когда в каталоге вашего проекта слишком много файлов (например, много изображений или ресурсов) и вы превышаете системный лимит наблюдателей за файлами (file watchers). В Linux по умолчанию установлен лимит примерно в 8192–10 000 наблюдателей.

Чтобы решить эту проблему, вы можете:

- Увеличить системный лимит наблюдателей за файлами:

  ```shell
  # Проверить текущий лимит
  $ cat /proc/sys/fs/inotify/max_user_watches
  # Увеличить лимит (временно)
  $ sudo sysctl fs.inotify.max_user_watches=524288
  # Сделать постоянным — добавить в /etc/sysctl.conf (или отредактировать, если файл уже существует)
  $ echo "fs.inotify.max_user_watches=524288" | sudo tee -a /etc/sysctl.conf
  $ sudo sysctl -p
  ```

- Исключить каталоги с большим количеством файлов из наблюдения с помощью [`server.watch.ignored`](https://vite.dev/config/server-options#server-watch)
- Использовать опрос (polling) вместо событий файловой системы с помощью [`server.watch.usePolling`](https://vite.dev/config/server-options#server-watch). Обратите внимание, что опрос потребляет больше ресурсов процессора

### Сетевые запросы перестают загружаться

При использовании самоподписанного SSL-сертификата Chrome игнорирует все директивы кэширования и перезагружает содержимое. Vite полагается на эти директивы кэширования.

Чтобы решить проблему, используйте доверенный SSL-сертификат.

Смотрите: [issue в Chrome](https://bugs.chromium.org/p/chromium/issues/detail?id=110649#c8)

#### macOS

Вы можете установить доверенный сертификат через CLI с помощью этой команды:

```
security add-trusted-cert -d -r trustRoot -k ~/Library/Keychains/login.keychain-db your-cert.cer
```

Либо импортировав его в приложение Keychain Access и установив уровень доверия для вашего сертификата в значение «Always Trust».

### 431 Request Header Fields Too Large

Когда сервер / WebSocket-сервер получает большой HTTP-заголовок, запрос будет отброшен и появится следующее предупреждение.

> Server responded with status code 431. See https://vite.dev/guide/troubleshooting.html#_431-request-header-fields-too-large.

Это происходит потому, что Node.js ограничивает размер заголовков запроса для защиты от [CVE-2018-12121](https://www.cve.org/CVERecord?id=CVE-2018-12121).

Чтобы этого избежать, попробуйте уменьшить размер заголовков запроса. Например, если cookie слишком длинный, удалите его. Либо вы можете использовать [`--max-http-header-size`](https://nodejs.org/api/cli.html#--max-http-header-sizesize), чтобы изменить максимальный размер заголовка.

### Dev Containers / проброс портов в VS Code

Если вы используете Dev Container или функцию проброса портов в VS Code, вам может потребоваться установить опцию [`server.host`](https://vite.dev/config/server-options#server-host) в значение `127.0.0.1` в конфигурации, чтобы всё заработало.

Это связано с тем, что [функция проброса портов в VS Code не поддерживает IPv6](https://github.com/microsoft/vscode-remote-release/issues/7029).

Подробнее смотрите в [#16522](https://github.com/vitejs/vite/issues/16522).

## HMR

### Vite обнаруживает изменение файла, но HMR не работает

Возможно, вы импортируете файл с другим регистром букв. Например, существует `src/foo.js`, а `src/bar.js` содержит:

```js
import './Foo.js' // должно быть './foo.js'
```

Связанный issue: [#964](https://github.com/vitejs/vite/issues/964)

### Vite не обнаруживает изменение файла

Если вы запускаете Vite в WSL2, при определённых условиях Vite не может отслеживать изменения файлов. Смотрите [опцию `server.watch`](https://vite.dev/config/server-options#server-watch).

### Вместо HMR происходит полная перезагрузка

Если HMR не обрабатывается ни Vite, ни плагином, происходит полная перезагрузка, так как это единственный способ обновить состояние.

Если HMR обрабатывается, но находится внутри циклической зависимости, полная перезагрузка также произойдёт, чтобы восстановить порядок выполнения. Чтобы решить эту проблему, попробуйте разорвать цикл. Вы можете запустить `vite --debug hmr`, чтобы вывести в лог путь циклической зависимости, если её вызвало изменение файла.

## Сборка

### Собранный файл не работает из-за ошибки CORS

Если выходной HTML-файл был открыт по протоколу `file`, скрипты не запустятся, и появится следующая ошибка.

> Access to script at 'file:///foo/bar.js' from origin 'null' has been blocked by CORS policy: Cross origin requests are only supported for protocol schemes: http, data, isolated-app, chrome-extension, chrome, https, chrome-untrusted.

> Cross-Origin Request Blocked: The Same Origin Policy disallows reading the remote resource at file:///foo/bar.js. (Reason: CORS request not http).

Подробнее о том, почему это происходит, смотрите [Reason: CORS request not HTTP - HTTP | MDN](https://developer.mozilla.org/en-US/docs/Web/HTTP/CORS/Errors/CORSRequestNotHttp).

Вам потребуется открыть файл по протоколу `http`. Проще всего этого добиться, запустив `npx vite preview`.

### Ошибка «No such file or directory» из-за чувствительности к регистру

Если вы сталкиваетесь с ошибками вроде `ENOENT: no such file or directory` или `Module not found`, это часто происходит, когда ваш проект разрабатывался на файловой системе, нечувствительной к регистру (Windows / macOS), но собирался на чувствительной к регистру (Linux). Убедитесь, что импорты используют правильный регистр.

### Ошибка `Failed to fetch dynamically imported module`

> TypeError: Failed to fetch dynamically imported module

Эта ошибка возникает в нескольких случаях:

- Рассинхронизация версий (version skew)
- Плохие условия сети
- Расширения браузера блокируют запросы

#### Рассинхронизация версий

Когда вы развёртываете новую версию приложения, HTML-файл и JS-файлы по-прежнему ссылаются на старые имена чанков, которые были удалены в новом развёртывании. Это происходит, когда:

1. У пользователей в браузере закэширована старая версия вашего приложения
2. Вы развёртываете новую версию с другими именами чанков (из-за изменений в коде)
3. Закэшированный HTML пытается загрузить чанки, которых больше не существует

Если вы используете фреймворк, сначала обратитесь к его документации, поскольку в нём может быть встроенное решение этой проблемы.

Чтобы решить эту проблему, вы можете:

- **Временно сохранять старые чанки**: рассмотрите возможность сохранения чанков предыдущего развёртывания на некоторое время, чтобы пользователи с кэшем могли плавно перейти на новую версию.
- **Использовать service worker**: реализуйте service worker, который будет заранее загружать (prefetch) все ресурсы и кэшировать их.
- **Заранее загружать динамические чанки (prefetch)**: обратите внимание, что это не поможет, если ваш HTML-файл закэширован браузером из-за заголовков `Cache-Control`.
- **Реализовать корректный запасной вариант (graceful fallback)**: реализуйте обработку ошибок для динамических импортов, чтобы перезагружать страницу, когда чанки отсутствуют. Подробнее смотрите [Обработка ошибок загрузки](https://vite.dev/guide/build#load-error-handling).

#### Плохие условия сети

Эта ошибка может возникать в нестабильных сетевых условиях. Например, когда запрос завершается неудачей из-за сетевых ошибок или недоступности сервера.

Обратите внимание, что вы не можете повторить динамический импорт из-за ограничений браузера ([whatwg/html#6768](https://github.com/whatwg/html/issues/6768)).

#### Расширения браузера блокируют запросы

Эта ошибка также может возникать, если расширения браузера (например, блокировщики рекламы) блокируют этот запрос.

Возможно, удастся обойти проблему, выбрав другое имя чанка с помощью [`build.rolldownOptions.output.chunkFileNames`](https://vite.dev/config/build-options#build-rolldownoptions), поскольку такие расширения часто блокируют запросы на основе имён файлов (например, имён, содержащих `ad`, `track`).

## Оптимизированные зависимости

### Устаревшие предварительно собранные зависимости при связывании с локальным пакетом

Ключ хэша, используемый для инвалидации оптимизированных зависимостей, зависит от содержимого lock-файла пакетов, применённых к зависимостям патчей и опций в файле конфигурации Vite, которые влияют на сборку node modules в бандл. Это означает, что Vite обнаружит, когда зависимость переопределена с помощью такой возможности, как [npm overrides](https://docs.npmjs.com/cli/v9/configuring-npm/package-json#overrides), и пересоберёт ваши зависимости при следующем запуске сервера. Vite не будет инвалидировать зависимости, когда вы используете такую возможность, как [npm link](https://docs.npmjs.com/cli/v9/commands/npm-link). Если вы связываете (link) или отвязываете (unlink) зависимость, вам потребуется принудительно запустить повторную оптимизацию при следующем запуске сервера с помощью `vite --force`. Мы рекомендуем использовать overrides, которые теперь поддерживаются каждым менеджером пакетов (смотрите также [pnpm overrides](https://pnpm.io/9.x/package_json#pnpmoverrides) и [yarn resolutions](https://yarnpkg.com/configuration/manifest/#resolutions)).

## Узкие места производительности

Если вы сталкиваетесь с какими-либо узкими местами производительности приложения, приводящими к медленной загрузке, вы можете запустить встроенный инспектор Node.js вместе с dev-сервером Vite или при сборке приложения, чтобы создать профиль загрузки процессора (CPU profile):

**dev server**

```bash
vite --profile --open
```

**build**

```bash
vite build --profile
```

> **Совет: Dev-сервер Vite**
> После того как ваше приложение откроется в браузере, просто дождитесь окончания его загрузки, затем вернитесь в терминал и нажмите клавишу `p` (это остановит инспектор Node.js), после чего нажмите клавишу `q`, чтобы остановить dev-сервер.

Инспектор Node.js сгенерирует файл `vite-profile-0.cpuprofile` в корневой папке; перейдите на https://www.speedscope.app/ и загрузите CPU profile с помощью кнопки `BROWSE`, чтобы изучить результат.

Вы можете установить [vite-plugin-inspect](https://github.com/antfu/vite-plugin-inspect), который позволяет исследовать промежуточное состояние плагинов Vite и также может помочь определить, какие плагины или промежуточные обработчики являются узким местом в ваших приложениях. Плагин можно использовать как в режиме разработки, так и в режиме сборки. Подробнее смотрите в файле readme.

## Прочее

### Модуль вынесен во внешние зависимости для совместимости с браузером

Когда вы используете модуль Node.js в браузере, Vite выведет следующее предупреждение.

> Module "fs" has been externalized for browser compatibility. Cannot access "fs.readFile" in client code.

Это происходит потому, что Vite не выполняет автоматический полифиллинг модулей Node.js.

Мы рекомендуем избегать модулей Node.js в коде для браузера, чтобы уменьшить размер бандла, хотя вы можете добавить полифиллы вручную. Если модуль импортируется из сторонней библиотеки (которая предназначена для использования в браузере), рекомендуется сообщить о проблеме разработчикам соответствующей библиотеки.

### Возникает Syntax Error / Type Error

Vite не может обрабатывать и не поддерживает код, который работает только в нестрогом режиме (sloppy mode). Это связано с тем, что Vite использует ESM, а внутри ESM всегда действует [строгий режим (strict mode)](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Strict_mode).

Например, вы можете увидеть такие ошибки.

> [ERROR] With statements cannot be used with the "esm" output format due to strict mode

> TypeError: Cannot create property 'foo' on boolean 'false'

Если такой код используется внутри зависимостей, вы можете воспользоваться [`patch-package`](https://github.com/ds300/patch-package) (или [`yarn patch`](https://yarnpkg.com/cli/patch), или [`pnpm patch`](https://pnpm.io/cli/patch)) в качестве обходного пути.

### Расширения браузера

Некоторые расширения браузера (например, блокировщики рекламы) могут препятствовать отправке клиентом Vite запросов к dev-серверу Vite. В этом случае вы можете увидеть белый экран без каких-либо ошибок в логе. Вы также можете увидеть следующую ошибку:

> TypeError: Failed to fetch dynamically imported module

Если у вас возникла эта проблема, попробуйте отключить расширения.

### Ссылки между дисками (cross drive links) в Windows

Если в вашем проекте в Windows есть ссылки между дисками (cross drive links), Vite может не работать.

Примеры ссылок между дисками:

- виртуальный диск, связанный с папкой с помощью команды `subst`
- symlink/junction на другой диск, созданный командой `mklink` (например, глобальный кэш Yarn)

Связанный issue: [#10802](https://github.com/vitejs/vite/issues/10802)

### Импорт по умолчанию неожиданно возвращает объект

Импорт по умолчанию возвращает объект `module.exports` для CJS-модулей, тогда как вы, возможно, ожидаете, что он вернёт значение `module.exports.default`.

Это может вызывать ошибки вроде:

> Element type is invalid: expected a string (for built-in components) or a class/function (for composite components) but got: object.

> foo is not a function

Подробнее об этой проблеме смотрите в документации Rolldown: [Ambiguous `default` import from CJS modules - Bundling CJS | Rolldown](https://rolldown.rs/in-depth/bundling-cjs#ambiguous-default-import-from-cjs-modules).
