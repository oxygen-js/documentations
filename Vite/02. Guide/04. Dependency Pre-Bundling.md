# Предварительная сборка зависимостей

Когда вы запускаете `vite` в первый раз, Vite выполняет предварительную сборку зависимостей вашего проекта, прежде чем загрузить сайт локально. По умолчанию это происходит автоматически и прозрачно.

## Зачем это нужно

Здесь Vite выполняет то, что мы называем «предварительной сборкой зависимостей». Этот процесс служит двум целям:

1. **Совместимость с CommonJS и UMD:** во время разработки Vite отдаёт весь код как нативные ES-модули. Поэтому Vite должен сначала преобразовать зависимости, поставляемые в формате CommonJS или UMD, в ESM.

   При преобразовании зависимостей CommonJS Vite выполняет умный анализ импортов, благодаря чему именованные импорты из модулей CommonJS работают ожидаемым образом, даже если экспорты присваиваются динамически (например, как в React):

   ```js
   // работает как ожидается
   import React, { useState } from 'react'
   ```

2. **Производительность:** Vite преобразует зависимости в формате ESM, состоящие из множества внутренних модулей, в единый модуль, чтобы улучшить производительность последующих загрузок страницы.

   Некоторые пакеты поставляют свои ES-модульные сборки в виде множества отдельных файлов, импортирующих друг друга. Например, [`lodash-es` содержит более 600 внутренних модулей](https://unpkg.com/browse/lodash-es/)! Когда мы делаем `import { debounce } from 'lodash-es'`, браузер одновременно отправляет более 600 HTTP-запросов! Хотя сервер без проблем их обрабатывает, большое количество запросов создаёт сетевую перегрузку на стороне браузера, из-за чего страница загружается заметно медленнее.

   Собрав `lodash-es` в один модуль на этапе предварительной сборки, мы теперь обходимся всего одним HTTP-запросом!

> **Примечание**
>
> Предварительная сборка зависимостей применяется только в режиме разработки.

## Автоматическое обнаружение зависимостей

Если существующий кэш не найден, Vite обходит ваш исходный код и автоматически обнаруживает импорты зависимостей (то есть «голые импорты», которые предполагается разрешать из `node_modules`) и использует эти найденные импорты как точки входа для предварительной сборки. Предварительная сборка выполняется с помощью [Rolldown](https://rolldown.rs/), поэтому обычно она очень быстрая.

Если уже после запуска сервера встречается импорт новой зависимости, которой ещё нет в кэше, Vite повторно запустит процесс сборки зависимостей и при необходимости перезагрузит страницу.

## Монорепозитории и связанные зависимости

В конфигурации с монорепозиторием зависимость может быть связанным (linked) пакетом из того же репозитория. Vite автоматически определяет зависимости, которые не разрешаются из `node_modules`, и рассматривает связанную зависимость как исходный код. Он не будет пытаться собрать связанную зависимость в бандл, а вместо этого проанализирует список её зависимостей.

Однако для этого связанная зависимость должна экспортироваться как ESM. Если это не так, вы можете добавить зависимость в [`optimizeDeps.include`](https://vite.dev/config/dep-optimization-options#optimizedeps-include) в вашей конфигурации.

**vite.config.js**

```js
import { defineConfig } from 'vite'
export default defineConfig({
  optimizeDeps: {
    include: ['linked-dep'],
  },
})
```

При внесении изменений в связанную зависимость перезапустите dev-сервер с опцией командной строки `--force`, чтобы изменения вступили в силу.

## Настройка поведения

Эвристики обнаружения зависимостей по умолчанию не всегда могут вас устраивать. В случаях, когда вы хотите явно включить или исключить зависимости из списка, используйте [параметры конфигурации `optimizeDeps`](https://vite.dev/config/dep-optimization-options).

Типичный сценарий использования `optimizeDeps.include` или `optimizeDeps.exclude` — когда у вас есть импорт, который не обнаруживается непосредственно в исходном коде. Например, возможно, импорт создаётся в результате преобразования плагином. Это означает, что Vite не сможет обнаружить импорт при первоначальном сканировании — он сможет обнаружить его только после того, как файл будет запрошен браузером и преобразован. Из-за этого сервер сразу же после запуска выполнит повторную сборку.

Для решения этой проблемы можно использовать как `include`, так и `exclude`. Если зависимость большая (со множеством внутренних модулей) или в формате CommonJS, вам следует включить её; если зависимость небольшая и уже является корректным ESM, вы можете исключить её и позволить браузеру загружать её напрямую.

Вы также можете дополнительно настроить Rolldown с помощью [параметра `optimizeDeps.rolldownOptions`](https://vite.dev/config/dep-optimization-options#optimizedeps-rolldownoptions). Например, добавить плагин Rolldown для обработки особых файлов в зависимостях или изменить [`target` сборки](https://rolldown.rs/reference/InputOptions.transform#target).

## Кэширование

### Кэш файловой системы

Vite кэширует предварительно собранные зависимости в `node_modules/.vite`. Он определяет необходимость повторного запуска этапа предварительной сборки на основе нескольких источников:

- Содержимое lock-файла менеджера пакетов, например `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml` или `bun.lock`.
- Время изменения папки с патчами.
- Соответствующие поля в вашем `vite.config.js`, если они есть.
- Значение `NODE_ENV`.

Этап предварительной сборки потребуется запустить повторно только тогда, когда изменится что-либо из перечисленного выше.

Если по какой-либо причине вы хотите принудительно заставить Vite пересобрать зависимости, вы можете либо запустить dev-сервер с опцией командной строки `--force`, либо вручную удалить директорию кэша `node_modules/.vite`.

### Кэш браузера

Разрешённые запросы зависимостей жёстко кэшируются с помощью HTTP-заголовков `max-age=31536000,immutable`, чтобы улучшить производительность перезагрузки страницы во время разработки. После кэширования эти запросы больше никогда не будут обращаться к dev-серверу. Они автоматически инвалидируются с помощью добавляемого параметра версии в запросе, если установлена другая версия (что отражается в lock-файле вашего менеджера пакетов). Если вы хотите отладить свои зависимости, внося локальные правки, вы можете:

1. Временно отключить кэш через вкладку Network в инструментах разработчика вашего браузера.
2. Перезапустить dev-сервер Vite с флагом `--force`, чтобы пересобрать зависимости.
3. Перезагрузить страницу.
