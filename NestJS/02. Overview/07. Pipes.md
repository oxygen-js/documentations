### Pipes

Pipe — это класс, помеченный декоратором `@Injectable()` и реализующий интерфейс `PipeTransform`.

<figure>
  <img class="illustrative-image" src="https://docs.nestjs.com/assets/Pipe_1.png" />
</figure>

У pipes есть два типичных сценария применения:

- **преобразование**: привести входные данные к нужному виду (например, из строки в целое число)
- **валидация**: проверить входные данные и, если они корректны, просто пропустить их без изменений; иначе выбросить исключение

В обоих случаях pipes работают с `аргументами`, которые обрабатывает <a href="02.%20Controllers.md#параметры-маршрута">обработчик маршрута контроллера</a>. Nest вставляет pipe непосредственно перед вызовом метода, и pipe получает аргументы, предназначенные для этого метода, и работает с ними. Все преобразования и проверки происходят именно в этот момент, после чего обработчик маршрута вызывается с (возможно) преобразованными аргументами.

В Nest есть набор встроенных pipes, которые можно использовать «из коробки». Вы также можете создавать собственные pipes. В этой главе мы познакомимся со встроенными pipes и покажем, как привязывать их к обработчикам маршрутов. Затем разберём несколько пользовательских pipes, чтобы показать, как написать такой класс с нуля.

> **Подсказка** Pipes выполняются внутри зоны исключений. Это значит, что когда pipe выбрасывает исключение, оно обрабатывается слоем исключений (глобальным фильтром исключений и любыми [фильтрами исключений](<06. Exception filters.md>), применёнными к текущему контексту). Из сказанного должно быть понятно: если в pipe выброшено исключение, ни один метод контроллера после этого не выполняется. Это даёт вам проверенный приём для валидации данных, поступающих в приложение из внешних источников, на границе системы.

#### Встроенные pipes

Nest поставляется с несколькими pipes, доступными «из коробки»:

- `ValidationPipe`
- `ParseIntPipe`
- `ParseFloatPipe`
- `ParseBoolPipe`
- `ParseArrayPipe`
- `ParseUUIDPipe`
- `ParseEnumPipe`
- `DefaultValuePipe`
- `ParseFilePipe`
- `ParseDatePipe`

Все они экспортируются из пакета `@nestjs/common`.

Быстро посмотрим на использование `ParseIntPipe`. Это пример сценария **преобразования**, в котором pipe гарантирует, что параметр метода-обработчика будет преобразован в целое число JavaScript (или будет выброшено исключение, если преобразование не удалось). Далее в этой главе мы покажем простую собственную реализацию `ParseIntPipe`. Описанные ниже приёмы применимы и к остальным встроенным преобразующим pipes (`ParseBoolPipe`, `ParseFloatPipe`, `ParseEnumPipe`, `ParseArrayPipe`, `ParseDatePipe` и `ParseUUIDPipe`, которые в этой главе мы будем называть `Parse*`-pipes).

#### Привязка pipes

Чтобы воспользоваться pipe, нужно привязать экземпляр его класса к подходящему контексту. В нашем примере с `ParseIntPipe` мы хотим связать pipe с конкретным методом-обработчиком маршрута и убедиться, что он выполняется до вызова метода. Делается это с помощью следующей конструкции, которую мы будем называть привязкой pipe на уровне параметра метода:

```typescript
@Get(':id')
async findOne(@Param('id', ParseIntPipe) id: number) {
  return this.catsService.findOne(id);
}
```

Это гарантирует одно из двух: либо параметр, который мы получаем в методе `findOne()`, является числом (как и ожидается при вызове `this.catsService.findOne()`), либо до вызова обработчика маршрута будет выброшено исключение.

Например, предположим, что маршрут вызывается так:

```bash
GET localhost:3000/abc
```

Nest выбросит примерно такое исключение:

```json
{
  "statusCode": 400,
  "message": "Validation failed (numeric string is expected)",
  "error": "Bad Request"
}
```

Исключение не даст телу метода `findOne()` выполниться.

В примере выше мы передаём класс (`ParseIntPipe`), а не экземпляр, оставляя создание объекта фреймворку и тем самым включая внедрение зависимостей. Как и в случае с pipes и guards, вместо этого можно передать готовый экземпляр. Передача экземпляра на месте полезна, если нужно настроить поведение встроенного pipe, передав ему параметры:

```typescript
@Get(':id')
async findOne(
  @Param('id', new ParseIntPipe({ errorHttpStatusCode: HttpStatus.NOT_ACCEPTABLE }))
  id: number,
) {
  return this.catsService.findOne(id);
}
```

Привязка остальных преобразующих pipes (всех **Parse\***-pipes) выполняется аналогично. Все они работают в контексте проверки параметров маршрута, параметров строки запроса и значений тела запроса.

Например, с параметром строки запроса:

```typescript
@Get()
async findOne(@Query('id', ParseIntPipe) id: number) {
  return this.catsService.findOne(id);
}
```

А вот пример использования `ParseUUIDPipe` для разбора строкового параметра и проверки того, что это UUID.

```typescript
@Get(':uuid')
async findOne(@Param('uuid', new ParseUUIDPipe()) uuid: string) {
  return this.catsService.findOne(uuid);
}
```

> **Подсказка** При использовании `ParseUUIDPipe()` разбирается UUID версии 3, 4 или 5; если вам нужна только конкретная версия UUID, вы можете передать её в параметрах pipe.

Выше мы рассмотрели примеры привязки различных встроенных pipes семейства `Parse*`. Привязка валидирующих pipes устроена немного иначе; об этом мы поговорим в следующем разделе.

> **Подсказка** Также см. [Приёмы валидации](<../04. Techniques/04. Validation.md>) — там много подробных примеров валидирующих pipes.

#### Пользовательские pipes

Как уже упоминалось, вы можете писать собственные pipes. Хотя Nest предоставляет надёжные встроенные `ParseIntPipe` и `ValidationPipe`, давайте напишем простые собственные версии каждого из них с нуля, чтобы понять, как устроены пользовательские pipes.

Начнём с простого `ValidationPipe`. Сперва он будет просто принимать входное значение и сразу возвращать его же, работая как тождественная функция.

**`validation.pipe.ts`**

```typescript
import { PipeTransform, Injectable, ArgumentMetadata } from '@nestjs/common';

@Injectable()
export class ValidationPipe implements PipeTransform {
  transform(value: any, metadata: ArgumentMetadata) {
    return value;
  }
}
```

> **Подсказка** `PipeTransform<T, R>` — это обобщённый интерфейс, который должен реализовывать любой pipe. В нём `T` обозначает тип входного значения `value`, а `R` — тип возвращаемого значения метода `transform()`.

Каждый pipe обязан реализовать метод `transform()`, чтобы выполнить контракт интерфейса `PipeTransform`. У этого метода два параметра:

- `value`
- `metadata`

Параметр `value` — это обрабатываемый в данный момент аргумент метода (до того, как его получит метод-обработчик маршрута), а `metadata` — метаданные этого аргумента. Объект метаданных содержит следующие свойства:

```typescript
export interface ArgumentMetadata {
  type: 'body' | 'query' | 'param' | 'custom';
  metatype?: Type<unknown>;
  data?: string;
}
```

Эти свойства описывают обрабатываемый аргумент.

<table>
  <tr>
    <td>
      <code>type</code>
    </td>
    <td>Указывает, является ли аргумент телом запроса
      <code>@Body()</code>, параметром строки запроса
      <code>@Query()</code>, параметром маршрута
      <code>@Param()</code> или пользовательским параметром (подробнее
      <a href="10.%20Custom%20decorators.md">здесь</a>).</td>
  </tr>
  <tr>
    <td>
      <code>metatype</code>
    </td>
    <td>
      Предоставляет метатип аргумента, например
      <code>String</code>. Примечание: значение равно
      <code>undefined</code>, если вы либо не указали тип в сигнатуре метода-обработчика маршрута, либо пишете на чистом JavaScript.
    </td>
  </tr>
  <tr>
    <td>
      <code>data</code>
    </td>
    <td>Строка, переданная в декоратор, например
      <code>@Body('string')</code>. Равна
      <code>undefined</code>, если скобки декоратора оставлены пустыми.</td>
  </tr>
</table>

> **Предупреждение** Интерфейсы TypeScript исчезают при транспиляции. Поэтому, если тип параметра метода объявлен как интерфейс, а не как класс, значением `metatype` будет `Object`.

#### Валидация на основе схемы

Сделаем наш валидирующий pipe чуть более полезным. Присмотримся к методу `create()` класса `CatsController`, где нам, скорее всего, хотелось бы убедиться в корректности объекта тела POST-запроса, прежде чем вызывать метод сервиса.

```typescript
@Post()
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
```

Сосредоточимся на параметре тела `createCatDto`. Его тип — `CreateCatDto`:

**`create-cat.dto.ts`**

```typescript
export class CreateCatDto {
  name: string;
  age: number;
  breed: string;
}
```

Мы хотим гарантировать, что любой входящий запрос к методу create содержит корректное тело. Значит, нужно проверить три поля объекта `createCatDto`. Можно было бы сделать это прямо в методе-обработчике маршрута, но это не лучший вариант: так нарушается **принцип единственной ответственности** (SRP).

Другой подход — создать **класс-валидатор** и делегировать задачу ему. Недостаток в том, что придётся не забывать вызывать этот валидатор в начале каждого метода.

А что если написать валидирующее middleware? Это могло бы сработать, но, к сожалению, невозможно создать **универсальное middleware**, пригодное для всех контекстов во всём приложении. Причина в том, что middleware ничего не знает о **контексте выполнения**, включая обработчик, который будет вызван, и его параметры.

Разумеется, именно для этого сценария и созданы pipes. Так что давайте доведём наш валидирующий pipe до ума.

#### Валидация объекта по схеме

Есть несколько подходов к аккуратной валидации объектов в духе [DRY](https://en.wikipedia.org/wiki/Don%27t_repeat_yourself). Один из распространённых — валидация **на основе схем**. Попробуем этот подход.

Библиотека [Zod](https://zod.dev/) позволяет создавать схемы очень просто, с читаемым API. Построим валидирующий pipe, использующий схемы Zod.

Начните с установки нужного пакета:

```bash
$ npm install --save zod
```

В примере кода ниже мы создаём простой класс, который принимает схему в качестве аргумента `constructor`. Затем мы применяем метод `schema.parse()`, проверяющий входящий аргумент по переданной схеме.

Как отмечалось ранее, **валидирующий pipe** либо возвращает значение без изменений, либо выбрасывает исключение.

В следующем разделе вы увидите, как передать нужную схему конкретному методу контроллера с помощью декоратора `@UsePipes()`. Благодаря этому наш валидирующий pipe становится переиспользуемым в разных контекстах — чего мы и добивались.

```typescript
import { PipeTransform, ArgumentMetadata, BadRequestException } from '@nestjs/common';
import { ZodSchema  } from 'zod';

export class ZodValidationPipe implements PipeTransform {
  constructor(private schema: ZodSchema) {}

  transform(value: unknown, metadata: ArgumentMetadata) {
    try {
      const parsedValue = this.schema.parse(value);
      return parsedValue;
    } catch (error) {
      throw new BadRequestException('Validation failed');
    }
  }
}
```

#### Привязка валидирующих pipes

Ранее мы видели, как привязывать преобразующие pipes (вроде `ParseIntPipe` и остальных `Parse*`-pipes).

Привязка валидирующих pipes тоже очень проста.

В данном случае мы хотим привязать pipe на уровне вызова метода. В нашем примере, чтобы использовать `ZodValidationPipe`, нужно сделать следующее:

1. Создать экземпляр `ZodValidationPipe`
2. Передать схему Zod, относящуюся к нужному контексту, в конструктор класса pipe
3. Привязать pipe к методу

Пример схемы Zod:

```typescript
import { z } from 'zod';

export const createCatSchema = z
  .object({
    name: z.string(),
    age: z.number(),
    breed: z.string(),
  })
  .required();

export type CreateCatDto = z.infer<typeof createCatSchema>;
```

Делаем это с помощью декоратора `@UsePipes()`, как показано ниже:

**`cats.controller.ts`**

```typescript
@Post()
@UsePipes(new ZodValidationPipe(createCatSchema))
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
```

> **Подсказка** Декоратор `@UsePipes()` импортируется из пакета `@nestjs/common`.

`ZodValidationPipe` можно применять к отдельным параметрам и вместе со встроенными pipes. Например, когда мы хотим проверить параметр пути `id` с помощью `ParseIntPipe` отдельно от тела запроса:

```typescript
@Put('/:id')
async update(
  @Param('id', ParseIntPipe) id: number,
  @Body(new ZodValidationPipe(createCatSchema)) body: CreateCatDto
): void {
  this.catsService.update(id, body);
}
```

> **Предупреждение** Библиотека `zod` требует, чтобы в вашем файле `tsconfig.json` была включена настройка `strictNullChecks`.

#### Class validator

> **Предупреждение** Приёмы из этого раздела требуют TypeScript и недоступны, если ваше приложение написано на чистом JavaScript.

Рассмотрим альтернативную реализацию нашего подхода к валидации.

Nest хорошо работает с библиотекой [class-validator](https://github.com/typestack/class-validator). Эта мощная библиотека позволяет использовать валидацию на основе декораторов. Такая валидация чрезвычайно удобна, особенно в сочетании с возможностями **pipes** в Nest, поскольку у нас есть доступ к `metatype` обрабатываемого свойства. Прежде чем начать, установим нужные пакеты:

```bash
$ npm i --save class-validator class-transformer
```

После установки мы можем добавить несколько декораторов к классу `CreateCatDto`. Здесь видно существенное преимущество этого подхода: класс `CreateCatDto` остаётся единственным источником истины для объекта тела POST-запроса (вместо того чтобы создавать отдельный класс для валидации).

**`create-cat.dto.ts`**

```typescript
import { IsString, IsInt } from 'class-validator';

export class CreateCatDto {
  @IsString()
  name: string;

  @IsInt()
  age: number;

  @IsString()
  breed: string;
}
```

> **Подсказка** Подробнее о декораторах class-validator читайте [здесь](https://github.com/typestack/class-validator#usage).

Теперь можно создать класс `ValidationPipe`, который использует эти аннотации.

**`validation.pipe.ts`**

```typescript
import { PipeTransform, Injectable, ArgumentMetadata, BadRequestException } from '@nestjs/common';
import { validate } from 'class-validator';
import { plainToInstance } from 'class-transformer';

@Injectable()
export class ValidationPipe implements PipeTransform<any> {
  async transform(value: any, { metatype }: ArgumentMetadata) {
    if (!metatype || !this.toValidate(metatype)) {
      return value;
    }
    const object = plainToInstance(metatype, value);
    const errors = await validate(object);
    if (errors.length > 0) {
      throw new BadRequestException('Validation failed');
    }
    return value;
  }

  private toValidate(metatype: Function): boolean {
    const types: Function[] = [String, Boolean, Number, Array, Object];
    return !types.includes(metatype);
  }
}
```

> **Подсказка** Напоминаем, что вам не нужно писать универсальный валидирующий pipe самостоятельно, поскольку `ValidationPipe` предоставляется Nest «из коробки». Встроенный `ValidationPipe` предлагает больше возможностей, чем пример, который мы построили в этой главе, — он намеренно оставлен простым, чтобы показать механику пользовательского pipe. Полное описание с множеством примеров вы найдёте [здесь](<../04. Techniques/04. Validation.md>).

> **Важно** Выше мы использовали библиотеку [class-transformer](https://github.com/typestack/class-transformer), созданную тем же автором, что и **class-validator**, поэтому они прекрасно работают вместе.

Разберём этот код. Во-первых, обратите внимание, что метод `transform()` помечен как `async`. Это возможно, потому что Nest поддерживает как синхронные, так и **асинхронные** pipes. Мы делаем метод `async`, поскольку некоторые проверки class-validator [могут быть асинхронными](https://github.com/typestack/class-validator#custom-validation-classes) (использовать промисы).

Далее заметьте, что мы применяем деструктуризацию, чтобы извлечь поле metatype (только этот член из `ArgumentMetadata`) в наш параметр `metatype`. Это просто сокращённая запись вместо того, чтобы получать целый `ArgumentMetadata` и отдельной инструкцией присваивать переменную metatype.

Затем обратите внимание на вспомогательную функцию `toValidate()`. Она отвечает за пропуск шага валидации, когда обрабатываемый аргумент имеет нативный тип JavaScript (к таким нельзя прикрепить декораторы валидации, поэтому прогонять их через валидацию незачем).

Далее мы используем функцию `plainToInstance()` из class-transformer, чтобы превратить обычный объект-аргумент JavaScript в типизированный объект и тем самым получить возможность применить валидацию. Это необходимо, потому что входящий объект тела POST-запроса после десериализации из сетевого запроса **не содержит никакой информации о типах** (так устроена нижележащая платформа, например Express). Class-validator должен использовать декораторы валидации, которые мы ранее объявили в нашем DTO, поэтому нам и нужно это преобразование, чтобы работать с входящим телом как с должным образом декорированным объектом, а не просто с обычным объектом.

Наконец, как уже отмечалось, поскольку это **валидирующий pipe**, он либо возвращает значение без изменений, либо выбрасывает исключение.

Последний шаг — привязать `ValidationPipe`. Pipes могут действовать в области параметра, метода, контроллера или глобально. Ранее, на примере валидирующего pipe на основе Zod, мы видели привязку на уровне метода.
В примере ниже мы привяжем экземпляр pipe к декоратору `@Body()` обработчика маршрута, чтобы наш pipe вызывался для проверки тела POST-запроса.

**`cats.controller.ts`**

```typescript
@Post()
async create(
  @Body(new ValidationPipe()) createCatDto: CreateCatDto,
) {
  this.catsService.create(createCatDto);
}
```

Pipes уровня параметра полезны, когда логика валидации касается только одного конкретного параметра.

#### Глобальные pipes

Поскольку `ValidationPipe` создавался максимально универсальным, раскрыть его потенциал полностью можно, зарегистрировав его как **глобальный** pipe, чтобы он применялся к каждому обработчику маршрута во всём приложении.

**`main.ts`**

```typescript
async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  app.useGlobalPipes(new ValidationPipe());
  await app.listen(process.env.PORT ?? 3000);
}
bootstrap();
```

> **Важно** В случае <a href="../14.%20FAQ/06.%20Hybrid%20application.md">гибридных приложений</a> метод `useGlobalPipes()` не настраивает pipes для шлюзов и микросервисов. Для «обычных» (негибридных) микросервисных приложений `useGlobalPipes()` действительно регистрирует pipes глобально.

Глобальные pipes применяются во всём приложении — к каждому контроллеру и каждому обработчику маршрута.

Учтите, что с точки зрения внедрения зависимостей глобальные pipes, зарегистрированные вне какого-либо модуля (с помощью `useGlobalPipes()`, как в примере выше), не могут внедрять зависимости, поскольку привязка выполнена вне контекста модуля. Чтобы решить эту проблему, вы можете зарегистрировать глобальный pipe **прямо из любого модуля** с помощью такой конструкции:

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { APP_PIPE } from '@nestjs/core';

@Module({
  providers: [
    {
      provide: APP_PIPE,
      useClass: ValidationPipe,
    },
  ],
})
export class AppModule {}
```

> **Подсказка** Если вы применяете этот подход для внедрения зависимостей в pipe, учтите: независимо от того, в каком модуле использована эта конструкция, pipe в действительности является глобальным. Где это лучше делать? Выберите модуль, в котором определён сам pipe (`ValidationPipe` в примере выше). Кроме того, `useClass` — не единственный способ регистрации пользовательского провайдера. Подробнее [здесь](<../03. Fundamentals/01. Custom providers.md>).

#### Встроенный ValidationPipe

Напоминаем, что вам не нужно писать универсальный валидирующий pipe самостоятельно, поскольку `ValidationPipe` предоставляется Nest «из коробки». Встроенный `ValidationPipe` предлагает больше возможностей, чем пример, который мы построили в этой главе, — он намеренно оставлен простым, чтобы показать механику пользовательского pipe. Полное описание с множеством примеров вы найдёте [здесь](<../04. Techniques/04. Validation.md>).

#### Сценарий преобразования

Валидация — не единственный сценарий применения пользовательских pipes. В начале главы мы упоминали, что pipe может также **преобразовывать** входные данные к нужному формату. Это возможно, потому что значение, возвращаемое функцией `transform`, полностью заменяет прежнее значение аргумента.

Когда это полезно? Иногда данные, переданные клиентом, нужно как-то изменить — например, преобразовать строку в целое число — прежде чем метод-обработчик маршрута сможет корректно с ними работать. Кроме того, некоторые обязательные поля могут отсутствовать, и мы хотели бы подставить значения по умолчанию. **Преобразующие pipes** справляются с этими задачами, вставляя обрабатывающую функцию между запросом клиента и обработчиком запроса.

Вот простой `ParseIntPipe`, отвечающий за разбор строки в целое число. (Как отмечено выше, в Nest есть встроенный, более продвинутый `ParseIntPipe`; здесь он приведён как простой пример пользовательского преобразующего pipe.)

**`parse-int.pipe.ts`**

```typescript
import { PipeTransform, Injectable, ArgumentMetadata, BadRequestException } from '@nestjs/common';

@Injectable()
export class ParseIntPipe implements PipeTransform<string, number> {
  transform(value: string, metadata: ArgumentMetadata): number {
    const val = parseInt(value, 10);
    if (isNaN(val)) {
      throw new BadRequestException('Validation failed');
    }
    return val;
  }
}
```

Затем мы можем привязать этот pipe к выбранному параметру, как показано ниже:

```typescript
@Get(':id')
async findOne(@Param('id', new ParseIntPipe()) id) {
  return this.catsService.findOne(id);
}
```

Ещё один полезный случай преобразования — выбрать **существующую сущность пользователя** из базы данных по идентификатору, переданному в запросе:

```typescript
@Get(':id')
findOne(@Param('id', UserByIdPipe) userEntity: UserEntity) {
  return userEntity;
}
```

Реализацию этого pipe мы оставляем читателю, но заметим, что, как и все остальные преобразующие pipes, он получает входное значение (`id`) и возвращает выходное (объект `UserEntity`). Это делает код более декларативным и соответствующим принципу [DRY](https://en.wikipedia.org/wiki/Don%27t_repeat_yourself), вынося шаблонный код из обработчика в общий pipe.

#### Значения по умолчанию

`Parse*`-pipes ожидают, что значение параметра определено. При получении `null` или `undefined` они выбрасывают исключение. Чтобы эндпоинт мог обрабатывать отсутствующие параметры строки запроса, нужно подставить значение по умолчанию до того, как `Parse*`-pipes начнут работать с этими значениями. Для этого и служит `DefaultValuePipe`. Достаточно создать экземпляр `DefaultValuePipe` в декораторе `@Query()` перед соответствующим `Parse*`-pipe, как показано ниже:

```typescript
@Get()
async findAll(
  @Query('activeOnly', new DefaultValuePipe(false), ParseBoolPipe) activeOnly: boolean,
  @Query('page', new DefaultValuePipe(0), ParseIntPipe) page: number,
) {
  return this.catsService.findAll({ activeOnly, page });
}
```
