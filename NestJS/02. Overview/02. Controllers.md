### Контроллеры

Контроллеры отвечают за обработку входящих **запросов** и отправку **ответов** клиенту.

<figure><img class="illustrative-image" src="https://docs.nestjs.com/assets/Controllers_1.png" /></figure>

Задача контроллера — обрабатывать определённые запросы приложения. Механизм **маршрутизации** определяет, какой контроллер будет обрабатывать каждый запрос. Часто у контроллера несколько маршрутов, и каждый маршрут может выполнять своё действие.

Для создания простого контроллера мы используем классы и **декораторы**. Декораторы связывают классы с необходимыми метаданными, что позволяет Nest построить карту маршрутизации, связывающую запросы с соответствующими контроллерами.

> **Подсказка** Чтобы быстро создать CRUD-контроллер со встроенной [валидацией](<../04. Techniques/04. Validation.md>), вы можете воспользоваться [генератором CRUD](<../13. Recipes/02. CRUD generator.md#генератор-crud-только-typescript>) из CLI: `nest g resource [name]`.

#### Маршрутизация

В следующем примере мы используем декоратор `@Controller()`, который **обязателен** для определения простого контроллера. Мы укажем необязательный префикс пути маршрута — `cats`. Использование префикса пути в декораторе `@Controller()` помогает сгруппировать связанные маршруты и уменьшить количество повторяющегося кода. Например, если мы хотим сгруппировать маршруты, управляющие сущностью «кот», под путём `/cats`, мы можем указать префикс `cats` в декораторе `@Controller()`. Тогда не придётся повторять эту часть пути в каждом маршруте файла.

**`cats.controller.ts`**

```typescript
import { Controller, Get } from '@nestjs/common';

@Controller('cats')
export class CatsController {
  @Get()
  findAll(): string {
    return 'This action returns all cats';
  }
}
```

> **Подсказка** Чтобы создать контроллер через CLI, просто выполните команду `$ nest g controller [name]`.

Декоратор HTTP-метода `@Get()`, размещённый перед методом `findAll()`, указывает Nest создать обработчик для конкретного эндпоинта HTTP-запросов. Этот эндпоинт определяется HTTP-методом запроса (в данном случае GET) и путём маршрута. Так что же такое путь маршрута? Путь маршрута для обработчика складывается из (необязательного) префикса, объявленного для контроллера, и пути, указанного в декораторе метода. Поскольку мы задали префикс (`cats`) для каждого маршрута и не добавили никакого конкретного пути в декоратор метода, Nest будет сопоставлять с этим обработчиком запросы `GET /cats`.

Как уже сказано, путь маршрута включает и необязательный префикс пути контроллера, **и** строку пути, указанную в декораторе метода. Например, если префикс контроллера — `cats`, а декоратор метода — `@Get('breed')`, итоговый маршрут будет `GET /cats/breed`.

В примере выше, когда к этому эндпоинту приходит GET-запрос, Nest направляет его в определённый пользователем метод `findAll()`. Обратите внимание: имя метода здесь выбрано совершенно произвольно. Объявить метод, к которому привязывается маршрут, необходимо, но само его имя для Nest никакого особого значения не имеет.

Этот метод вернёт код состояния 200 вместе с соответствующим ответом — в данном случае просто строкой. Почему так происходит? Чтобы объяснить, сначала нужно ввести понятие: Nest предлагает два **разных** способа работы с ответами:

<table>
  <tr>
    <td>Стандартный (рекомендуется)</td>
    <td>
      При использовании этого встроенного способа, если обработчик запроса возвращает объект или массив JavaScript, он <strong>автоматически</strong>
      сериализуется в JSON. Если же возвращается примитивный тип JavaScript (например, <code>string</code>, <code>number</code>, <code>boolean</code>), Nest отправит только значение, не пытаясь его сериализовать. Это делает работу с ответами простой: достаточно вернуть значение, а обо всём остальном позаботится Nest.
      <br />
      <br /> Кроме того, <strong>код состояния</strong> ответа по умолчанию всегда 200, за исключением POST-запросов,
      для которых используется 201. Это поведение легко изменить, добавив декоратор <code>@HttpCode(...)</code>
      на уровне обработчика (см. <a href='02.%20Controllers.md#код-состояния'>Коды состояния</a>).
    </td>
  </tr>
  <tr>
    <td>Специфичный для библиотеки</td>
    <td>
      Мы можем использовать специфичный для библиотеки (например, Express) <a href="https://expressjs.com/en/api.html#res" rel="nofollow" target="_blank">объект ответа</a>, который внедряется с помощью декоратора <code>@Res()</code> в сигнатуре метода-обработчика (например, <code>findAll(@Res() response)</code>).  При таком подходе вам доступны нативные методы работы с ответом, предоставляемые этим объектом.  Например, в Express можно формировать ответы кодом вида <code>response.status(200).send()</code>.
    </td>
  </tr>
</table>

> **Предупреждение** Nest определяет, когда обработчик использует `@Res()` или `@Next()` — это означает, что вы выбрали способ, специфичный для библиотеки. Если использовать оба подхода одновременно, стандартный подход для этого конкретного маршрута **автоматически отключается** и больше не работает как ожидается. Чтобы использовать оба подхода одновременно (например, внедрить объект ответа только для установки cookie или заголовков, оставив всё остальное фреймворку), необходимо задать опцию `passthrough` равной `true` в декораторе `@Res({ passthrough: true })`.

#### Объект запроса

Обработчикам часто нужен доступ к деталям **запроса** клиента. Nest предоставляет доступ к [объекту запроса](https://expressjs.com/en/api.html#req) нижележащей платформы (по умолчанию Express). Получить объект запроса можно, указав Nest внедрить его с помощью декоратора `@Req()` в сигнатуре обработчика.

**`cats.controller.ts`**

```typescript
import { Controller, Get, Req } from '@nestjs/common';
import type { Request } from 'express';

@Controller('cats')
export class CatsController {
  @Get()
  findAll(@Req() request: Request): string {
    return 'This action returns all cats';
  }
}
```

> **Подсказка** Чтобы воспользоваться типами `express` (как в примере с параметром `request: Request` выше), не забудьте установить пакет `@types/express`.

Объект запроса представляет HTTP-запрос и содержит свойства для строки запроса, параметров, HTTP-заголовков и тела (подробнее — [здесь](https://expressjs.com/en/api.html#req)). В большинстве случаев обращаться к этим свойствам вручную не нужно. Вместо этого можно использовать специальные декораторы, такие как `@Body()` или `@Query()`, доступные «из коробки». Ниже приведён список предоставляемых декораторов и соответствующих им объектов платформы.

<sup>\* </sup>Для совместимости типов между разными нижележащими HTTP-платформами (например, Express и Fastify) Nest предоставляет декораторы `@Res()` и `@Response()`. `@Res()` — это просто псевдоним для `@Response()`. Оба напрямую открывают интерфейс нативного объекта `response` соответствующей платформы. При их использовании стоит также импортировать типы нижележащей библиотеки (например, `@types/express`), чтобы получить максимум пользы. Учтите: когда вы внедряете `@Res()` или `@Response()` в метод-обработчик, вы переводите Nest в **режим, специфичный для библиотеки** для этого обработчика, и сами становитесь ответственны за формирование ответа. В этом случае вы обязаны так или иначе отправить ответ, вызвав метод объекта `response` (например, `res.json(...)` или `res.send(...)`), иначе HTTP-сервер зависнет.

> **Подсказка** О том, как создавать собственные декораторы, читайте в [этой](<10. Custom decorators.md>) главе.

#### Ресурсы

Ранее мы определили эндпоинт для получения ресурса «коты» (маршрут **GET**). Обычно нам также нужен эндпоинт для создания новых записей. Создадим обработчик **POST**:

**`cats.controller.ts`**

```typescript
import { Controller, Get, Post } from '@nestjs/common';

@Controller('cats')
export class CatsController {
  @Post()
  create(): string {
    return 'This action adds a new cat';
  }

  @Get()
  findAll(): string {
    return 'This action returns all cats';
  }
}
```

Вот так просто. Nest предоставляет декораторы для всех стандартных HTTP-методов: `@Get()`, `@Post()`, `@Put()`, `@Delete()`, `@Patch()`, `@Options()` и `@Head()`. Кроме того, `@All()` определяет эндпоинт, обрабатывающий их все.

#### Подстановочные знаки в маршрутах

NestJS поддерживает и маршруты на основе шаблонов. Например, звёздочку (`*`) можно использовать как подстановочный знак для совпадения с любой комбинацией символов в конце пути маршрута. В следующем примере метод `findAll()` будет выполняться для любого маршрута, начинающегося с `abcd/`, независимо от того, сколько символов идёт дальше.

```typescript
@Get('abcd/*')
findAll() {
  return 'This route uses a wildcard';
}
```

Путь маршрута `'abcd/*'` будет совпадать с `abcd/`, `abcd/123`, `abcd/abc` и так далее. Дефис (`-`) и точка (`.`) в строковых путях трактуются буквально.

Этот подход работает и в Express, и в Fastify. Однако в последнем выпуске Express (v5) система маршрутизации стала строже. В чистом Express для работы маршрута необходимо использовать именованный подстановочный знак — например, `abcd/*splat`, где `splat` — это просто имя параметра подстановки, не имеющее особого значения. Вы можете назвать его как угодно. При этом Nest предоставляет слой совместимости для Express, так что вы по-прежнему можете использовать звёздочку (`*`) в качестве подстановочного знака.

Что касается звёздочек **в середине маршрута**, Express требует именованных подстановочных знаков (например, `ab{*splat}cd`), а Fastify не поддерживает их вовсе.

#### Код состояния

Как уже отмечалось, **код состояния** ответа по умолчанию всегда **200**, за исключением POST-запросов, где по умолчанию используется **201**. Это поведение легко изменить с помощью декоратора `@HttpCode(...)` на уровне обработчика.

```typescript
@Post()
@HttpCode(204)
create() {
  return 'This action adds a new cat';
}
```

> **Подсказка** Импортируйте `HttpCode` из пакета `@nestjs/common`.

Часто код состояния не статичен, а зависит от разных факторов. В этом случае вы можете использовать специфичный для библиотеки объект **ответа** (внедряемый через `@Res()`) либо, в случае ошибки, выбросить исключение.

#### Заголовки ответа

Чтобы задать свой заголовок ответа, можно использовать либо декоратор `@Header()`, либо специфичный для библиотеки объект ответа (и вызвать `res.header()` напрямую).

```typescript
@Post()
@Header('Cache-Control', 'no-store')
create() {
  return 'This action adds a new cat';
}
```

> **Подсказка** Импортируйте `Header` из пакета `@nestjs/common`.

#### Перенаправление

Чтобы перенаправить ответ на определённый URL, можно использовать либо декоратор `@Redirect()`, либо специфичный для библиотеки объект ответа (и вызвать `res.redirect()` напрямую).

`@Redirect()` принимает два аргумента — `url` и `statusCode`, оба необязательные. Если `statusCode` не указан, по умолчанию используется `302` (`Found`).

```typescript
@Get()
@Redirect('https://nestjs.com', 301)
```

> **Подсказка** Иногда бывает нужно определить HTTP-код состояния или URL перенаправления динамически. Для этого верните объект, соответствующий интерфейсу `HttpRedirectResponse` (из `@nestjs/common`).

Возвращённые значения переопределяют любые аргументы, переданные декоратору `@Redirect()`. Например:

```typescript
@Get('docs')
@Redirect('https://docs.nestjs.com', 302)
getDocs(@Query('version') version) {
  if (version && version === '5') {
    return { url: 'https://docs.nestjs.com/v5/' };
  }
}
```

#### Параметры маршрута

Маршруты со статическими путями не подойдут, когда нужно принимать **динамические данные** в составе запроса (например, `GET /cats/1` для получения кота с id `1`). Чтобы определить маршруты с параметрами, добавьте в путь маршрута **токены** параметров, которые захватят динамические значения из URL. Такой подход показан на примере токена параметра маршрута в декораторе `@Get()` ниже. Обратиться к этим параметрам маршрута можно с помощью декоратора `@Param()`, который добавляется в сигнатуру метода.

> **Подсказка** Маршруты с параметрами следует объявлять после всех статических путей. Это не даст параметризованным путям перехватывать трафик, предназначенный статическим путям.

```typescript
@Get(':id')
findOne(@Param() params: any): string {
  console.log(params.id);
  return `This action returns a #${params.id} cat`;
}
```

Декоратор `@Param()` применяется к параметру метода (в примере выше — `params`) и делает параметры **маршрута** доступными как свойства этого параметра внутри метода. Как видно из кода, к параметру `id` можно обратиться через `params.id`. Кроме того, вы можете передать декоратору конкретный токен параметра и обращаться к параметру маршрута по имени прямо в теле метода.

> **Подсказка** Импортируйте `Param` из пакета `@nestjs/common`.

```typescript
@Get(':id')
findOne(@Param('id') id: string): string {
  return `This action returns a #${id} cat`;
}
```

#### Маршрутизация по поддоменам

Декоратор `@Controller` принимает опцию `host`, которая требует, чтобы HTTP-хост входящих запросов совпадал с определённым значением.

```typescript
@Controller({ host: 'admin.example.com' })
export class AdminController {
  @Get()
  index(): string {
    return 'Admin page';
  }
}
```

> **Предупреждение** Поскольку **Fastify** не поддерживает вложенные маршрутизаторы, при использовании маршрутизации по поддоменам рекомендуется использовать адаптер Express, применяемый по умолчанию.

Как и в случае с `path` маршрута, опция `host` может использовать токены для захвата динамического значения в соответствующей позиции имени хоста. Такое использование показано на примере токена параметра хоста в декораторе `@Controller()` ниже. Объявленные таким образом параметры хоста доступны через декоратор `@HostParam()`, который добавляется в сигнатуру метода.

```typescript
@Controller({ host: ':account.example.com' })
export class AccountController {
  @Get()
  getInfo(@HostParam('account') account: string) {
    return account;
  }
}
```

#### Разделение состояния

Разработчиков, пришедших из других языков программирования, может удивить, что в Nest почти всё разделяется между входящими запросами. Это касается таких ресурсов, как пул подключений к базе данных, singleton-сервисы с глобальным состоянием и многое другое. Важно понимать, что Node.js не использует многопоточную модель без состояния, где каждый запрос обрабатывается отдельным потоком. Поэтому использование singleton-экземпляров в Nest для наших приложений полностью **безопасно**.

Тем не менее есть отдельные пограничные случаи, когда контроллерам может потребоваться время жизни, привязанное к запросу. Примеры — кэширование на уровне запроса в GraphQL-приложениях, трассировка запросов или реализация мультиарендности. Подробнее об управлении областями внедрения читайте [здесь](<../03. Fundamentals/04. Injection scopes.md>).

#### Асинхронность

Мы любим современный JavaScript, особенно его акцент на **асинхронной** работе с данными. Именно поэтому Nest полностью поддерживает функции `async`. Каждая функция `async` должна возвращать `Promise`, что позволяет вернуть отложенное значение, которое Nest разрешит автоматически. Пример:

**`cats.controller.ts`**

```typescript
@Get()
async findAll(): Promise<any[]> {
  return [];
}
```

Этот код полностью корректен. Но Nest идёт дальше и позволяет обработчикам маршрутов возвращать также [потоки-observable](https://rxjs-dev.firebaseapp.com/guide/observable) из RxJS. Nest сам оформит подписку и разрешит последнее переданное значение, когда поток завершится.

**`cats.controller.ts`**

```typescript
@Get()
findAll(): Observable<any[]> {
  return of([]);
}
```

Оба подхода корректны, и вы можете выбрать тот, который лучше подходит для ваших задач.

#### Полезная нагрузка запроса

В предыдущем примере обработчик POST-маршрута не принимал никаких параметров от клиента. Исправим это, добавив декоратор `@Body()`.

Прежде чем продолжить (если вы используете TypeScript), нужно определить схему **DTO** (Data Transfer Object — объект передачи данных). DTO — это объект, описывающий, как данные должны передаваться по сети. Схему DTO можно определить с помощью интерфейсов **TypeScript** или обычных классов. Однако мы рекомендуем использовать здесь **классы**. Почему? Классы входят в стандарт JavaScript ES6, поэтому они сохраняются как реальные сущности в скомпилированном JavaScript. Интерфейсы же TypeScript удаляются при транспиляции, а значит, Nest не может обратиться к ним во время выполнения. Это важно, поскольку такие возможности, как **pipe**, полагаются на доступ к метатипу переменных во время выполнения, а это возможно только с классами.

Создадим класс `CreateCatDto`:

**`create-cat.dto.ts`**

```typescript
export class CreateCatDto {
  name: string;
  age: number;
  breed: string;
}
```

В нём всего три простых свойства. После этого мы можем использовать созданный DTO внутри `CatsController`:

**`cats.controller.ts`**

```typescript
@Post()
async create(@Body() createCatDto: CreateCatDto) {
  return 'This action adds a new cat';
}
```

> **Подсказка** Наш `ValidationPipe` умеет отфильтровывать свойства, которые не должны попадать в метод-обработчик. В этом случае мы можем составить белый список допустимых свойств, и любое свойство, не входящее в него, автоматически удаляется из результирующего объекта. В примере с `CreateCatDto` белый список — это свойства `name`, `age` и `breed`. Подробнее — [здесь](<../04. Techniques/04. Validation.md#удаление-лишних-свойств>).

#### Параметры строки запроса

Для работы с параметрами строки запроса в маршрутах используйте декоратор `@Query()`, который извлекает их из входящих запросов. Посмотрим, как это работает на практике.

Рассмотрим маршрут, где мы хотим отфильтровать список котов по таким параметрам, как `age` и `breed`. Сначала определим параметры запроса в `CatsController`:

**`cats.controller.ts`**

```typescript
@Get()
async findAll(@Query('age') age: number, @Query('breed') breed: string) {
  return `This action returns all cats filtered by age: ${age} and breed: ${breed}`;
}
```

В этом примере декоратор `@Query()` используется для извлечения значений `age` и `breed` из строки запроса. Например, запрос:

```plaintext
GET /cats?age=2&breed=Persian
```

даст `age`, равное `2`, и `breed`, равное `Persian`.

Если вашему приложению нужно обрабатывать более сложные параметры запроса, например вложенные объекты или массивы:

```plaintext
?filter[where][name]=John&filter[where][age]=30
?item[]=1&item[]=2
```

вам понадобится настроить HTTP-адаптер (Express или Fastify) на использование подходящего парсера строки запроса. В Express можно использовать парсер `extended`, который поддерживает сложные объекты запроса:

```typescript
const app = await NestFactory.create<NestExpressApplication>(AppModule);
app.set('query parser', 'extended');
```

В Fastify для этого есть опция `querystringParser`:

```typescript
const app = await NestFactory.create<NestFastifyApplication>(
  AppModule,
  new FastifyAdapter({
    querystringParser: (str) => qs.parse(str),
  }),
);
```

> **Подсказка** `qs` — парсер строки запроса с поддержкой вложенности и массивов. Установить его можно командой `npm install qs`.

#### Полный пример ресурса

Ниже приведён пример, демонстрирующий использование нескольких доступных декораторов для создания простого контроллера. Этот контроллер предоставляет несколько методов для доступа к внутренним данным и их изменения.

**`cats.controller.ts`**

```typescript
import { Controller, Get, Query, Post, Body, Put, Param, Delete } from '@nestjs/common';
import { CreateCatDto, UpdateCatDto, ListAllEntities } from './dto';

@Controller('cats')
export class CatsController {
  @Post()
  create(@Body() createCatDto: CreateCatDto) {
    return 'This action adds a new cat';
  }

  @Get()
  findAll(@Query() query: ListAllEntities) {
    return `This action returns all cats (limit: ${query.limit} items)`;
  }

  @Get(':id')
  findOne(@Param('id') id: string) {
    return `This action returns a #${id} cat`;
  }

  @Put(':id')
  update(@Param('id') id: string, @Body() updateCatDto: UpdateCatDto) {
    return `This action updates a #${id} cat`;
  }

  @Delete(':id')
  remove(@Param('id') id: string) {
    return `This action removes a #${id} cat`;
  }
}
```

> **Подсказка** Nest CLI предлагает генератор (схематик), который автоматически создаёт **весь шаблонный код**, избавляя вас от ручной работы и улучшая опыт разработки. Подробнее об этой возможности — [здесь](<../13. Recipes/02. CRUD generator.md>).

#### Запускаем всё это

Даже когда `CatsController` полностью определён, Nest ещё ничего о нём не знает и не создаст экземпляр класса автоматически.

Контроллеры всегда должны входить в состав модуля — поэтому мы указываем массив `controllers` внутри декоратора `@Module()`. Поскольку никаких других модулей, кроме корневого `AppModule`, мы пока не определяли, зарегистрируем `CatsController` в нём:

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { CatsController } from './cats/cats.controller';

@Module({
  controllers: [CatsController],
})
export class AppModule {}
```

Мы прикрепили метаданные к классу модуля с помощью декоратора `@Module()`, и теперь Nest без труда определит, какие контроллеры нужно подключить.

#### Подход, специфичный для библиотеки

До сих пор мы рассматривали стандартный для Nest способ работы с ответами. Другой подход — использовать специфичный для библиотеки [объект ответа](https://expressjs.com/en/api.html#res). Чтобы внедрить конкретный объект ответа, применяется декоратор `@Res()`. Чтобы подчеркнуть различия, перепишем `CatsController` так:

```typescript
import { Controller, Get, Post, Res, HttpStatus } from '@nestjs/common';
import { Response } from 'express';

@Controller('cats')
export class CatsController {
  @Post()
  create(@Res() res: Response) {
    res.status(HttpStatus.CREATED).send();
  }

  @Get()
  findAll(@Res() res: Response) {
     res.status(HttpStatus.OK).json([]);
  }
}
```

Такой подход работает и даёт больше гибкости за счёт полного контроля над объектом ответа (например, управление заголовками и доступ к специфичным для библиотеки возможностям), но применять его следует с осторожностью. В целом этот способ менее нагляден и имеет ряд недостатков. Главный из них — код становится зависимым от платформы, поскольку у разных нижележащих библиотек могут быть разные API объекта ответа. Кроме того, усложняется тестирование: вам, среди прочего, придётся создавать mock объекта ответа.

Более того, при таком подходе теряется совместимость с возможностями Nest, полагающимися на стандартную обработку ответов, — такими как interceptor'ы и декораторы `@HttpCode()` / `@Header()`. Чтобы этого избежать, можно включить опцию `passthrough`:

```typescript
@Get()
findAll(@Res({ passthrough: true }) res: Response) {
  res.status(HttpStatus.OK);
  return [];
}
```

При таком подходе вы можете взаимодействовать с нативным объектом ответа (например, устанавливать cookie или заголовки в зависимости от условий), оставляя всё остальное фреймворку.
