### Модули

Модуль — это класс, помеченный декоратором `@Module()`. Этот декоратор предоставляет метаданные, которые **Nest** использует, чтобы эффективно организовать структуру приложения и управлять ею.

<figure><img class="illustrative-image" src="https://docs.nestjs.com/assets/Modules_1.png" /></figure>

В каждом приложении Nest есть как минимум один модуль — **корневой модуль**, который служит отправной точкой для построения **графа приложения**. Этот граф — внутренняя структура, с помощью которой Nest разрешает связи и зависимости между модулями и провайдерами. Небольшое приложение может состоять из одного лишь корневого модуля, но так бывает редко. Модули **настоятельно рекомендуются** как действенный способ организации компонентов. В большинстве приложений у вас будет несколько модулей, каждый из которых инкапсулирует тесно связанный набор **возможностей**.

Декоратор `@Module()` принимает один объект, свойства которого описывают модуль:

|               |                                                                                                                                                                                                          |
| ------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `providers`   | провайдеры, которые будут созданы инжектором Nest и которые могут быть доступны как минимум в пределах этого модуля                                                                                      |
| `controllers` | набор контроллеров, определённых в этом модуле, экземпляры которых должны быть созданы                                                                                                                    |
| `imports`     | список импортируемых модулей, экспортирующих провайдеры, которые нужны в этом модуле                                                                                                                      |
| `exports`     | подмножество `providers`, которые предоставляются этим модулем и должны быть доступны в других модулях, импортирующих данный. Можно указать как сам провайдер, так и только его токен (значение `provide`) |

По умолчанию модуль **инкапсулирует** провайдеры: внедрять можно только те провайдеры, которые либо входят в текущий модуль, либо явно экспортированы из других импортированных модулей. Экспортируемые модулем провайдеры по сути и составляют его публичный интерфейс, или API.

#### Модули функциональности

В нашем примере `CatsController` и `CatsService` тесно связаны между собой и обслуживают одну предметную область приложения. Логично объединить их в модуль функциональности. Такой модуль группирует код, относящийся к конкретной возможности, что помогает поддерживать чёткие границы и лучшую организацию. Это особенно важно по мере роста приложения или команды и согласуется с принципами [SOLID](https://en.wikipedia.org/wiki/SOLID).

Далее создадим `CatsModule`, чтобы показать, как сгруппировать контроллер и сервис.

**`cats/cats.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

@Module({
  controllers: [CatsController],
  providers: [CatsService],
})
export class CatsModule {}
```

> **Подсказка** Чтобы создать модуль через CLI, просто выполните команду `$ nest g module cats`.

Выше мы определили `CatsModule` в файле `cats.module.ts` и перенесли всё, что относится к этому модулю, в каталог `cats`. Осталось импортировать этот модуль в корневой модуль (`AppModule`, определённый в файле `app.module.ts`).

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { CatsModule } from './cats/cats.module';

@Module({
  imports: [CatsModule],
})
export class AppModule {}
```

Вот как теперь выглядит структура каталогов:

```text
src
  cats
    dto
      create-cat.dto.ts
    interfaces
      cat.interface.ts
    cats.controller.ts
    cats.module.ts
    cats.service.ts
  app.module.ts
  main.ts
```

#### Общие модули

В Nest модули по умолчанию являются **синглтонами**, поэтому один и тот же экземпляр любого провайдера можно без труда использовать сразу в нескольких модулях.

<figure><img class="illustrative-image" src="https://docs.nestjs.com/assets/Shared_Module_1.png" /></figure>

Каждый модуль автоматически является **общим модулем**. Созданный однажды, он может быть повторно использован любым модулем. Допустим, мы хотим использовать один и тот же экземпляр `CatsService` в нескольких других модулях. Для этого сначала нужно **экспортировать** провайдер `CatsService`, добавив его в массив `exports` модуля:

**`cats.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

@Module({
  controllers: [CatsController],
  providers: [CatsService],
  exports: [CatsService]
})
export class CatsModule {}
```

Теперь любой модуль, импортирующий `CatsModule`, получает доступ к `CatsService` и будет использовать тот же экземпляр, что и все остальные модули, которые его импортируют.

Если бы мы регистрировали `CatsService` напрямую в каждом модуле, которому он нужен, это, конечно, работало бы, но каждый модуль получил бы собственный отдельный экземпляр `CatsService`. Это привело бы к росту потребления памяти, поскольку создавалось бы несколько экземпляров одного и того же сервиса, а также могло бы вызвать неожиданное поведение — например, рассогласование состояния, если сервис хранит какое-либо внутреннее состояние.

Инкапсулируя `CatsService` внутри модуля вроде `CatsModule` и экспортируя его, мы гарантируем, что один и тот же экземпляр `CatsService` переиспользуется во всех модулях, импортирующих `CatsModule`. Это не только снижает расход памяти, но и делает поведение более предсказуемым: все модули работают с одним экземпляром, что упрощает управление общим состоянием и ресурсами. В этом одно из ключевых преимуществ модульности и внедрения зависимостей в фреймворках вроде NestJS — сервисы эффективно используются во всём приложении.

#### Реэкспорт модулей

Как мы видели выше, модули могут экспортировать свои внутренние провайдеры. Кроме того, они могут реэкспортировать импортированные ими модули. В примере ниже `CommonModule` одновременно импортируется в `CoreModule` **и** экспортируется из него, благодаря чему становится доступен другим модулям, которые импортируют `CoreModule`.

```typescript
@Module({
  imports: [CommonModule],
  exports: [CommonModule],
})
export class CoreModule {}
```

#### Внедрение зависимостей

Класс модуля также может **внедрять** провайдеры (например, для целей конфигурации):

**`cats.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

@Module({
  controllers: [CatsController],
  providers: [CatsService],
})
export class CatsModule {
  constructor(private catsService: CatsService) {}
}
```

Однако сами классы модулей не могут быть внедрены как провайдеры из-за [циклической зависимости](<../03. Fundamentals/05. Circular dependency.md>) .

#### Глобальные модули

Если один и тот же набор модулей приходится импортировать повсюду, это быстро надоедает. В отличие от Nest, в [Angular](https://angular.dev) `providers` регистрируются в глобальной области видимости: будучи объявленными однажды, они доступны везде. Nest же инкапсулирует провайдеры в области модуля. Использовать провайдеры модуля в другом месте, не импортировав сам модуль, не получится.

Если вы хотите предоставить набор провайдеров, которые должны быть доступны везде «из коробки» (например, вспомогательные функции, подключения к базе данных и т. п.), сделайте модуль **глобальным** с помощью декоратора `@Global()`.

```typescript
import { Module, Global } from '@nestjs/common';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

@Global()
@Module({
  controllers: [CatsController],
  providers: [CatsService],
  exports: [CatsService],
})
export class CatsModule {}
```

Декоратор `@Global()` делает область видимости модуля глобальной. Глобальные модули следует регистрировать **только один раз** — как правило, в корневом или основном модуле. В примере выше провайдер `CatsService` будет доступен повсеместно, и модулям, которым нужно внедрить этот сервис, не придётся добавлять `CatsModule` в свой массив `imports`.

> **Подсказка** Делать всё глобальным не рекомендуется как приём проектирования. Глобальные модули помогают сократить шаблонный код, но обычно лучше использовать массив `imports`, чтобы предоставлять API модуля другим модулям управляемо и явно. Такой подход даёт более понятную структуру и лучшую поддерживаемость: другим модулям становятся доступны только нужные части модуля, а лишние связи между несвязанными частями приложения не возникают.

#### Динамические модули

Динамические модули в Nest позволяют создавать модули, настраиваемые во время выполнения. Это особенно полезно, когда нужны гибкие, настраиваемые модули, провайдеры которых создаются в зависимости от определённых параметров или конфигурации. Ниже — краткий обзор того, как работают **динамические модули**.

```typescript
import { Module, DynamicModule } from '@nestjs/common';
import { createDatabaseProviders } from './database.providers';
import { Connection } from './connection.provider';

@Module({
  providers: [Connection],
  exports: [Connection],
})
export class DatabaseModule {
  static forRoot(entities = [], options?): DynamicModule {
    const providers = createDatabaseProviders(options, entities);
    return {
      module: DatabaseModule,
      providers: providers,
      exports: providers,
    };
  }
}
```

> **Подсказка** Метод `forRoot()` может возвращать динамический модуль как синхронно, так и асинхронно (то есть через `Promise`).

Этот модуль по умолчанию определяет провайдер `Connection` (в метаданных декоратора `@Module()`), но дополнительно — в зависимости от объектов `entities` и `options`, переданных в метод `forRoot()`, — предоставляет набор провайдеров, например репозиториев. Обратите внимание, что свойства, возвращаемые динамическим модулем, **дополняют** (а не переопределяют) базовые метаданные модуля, заданные в декораторе `@Module()`. Именно поэтому из модуля экспортируются и статически объявленный провайдер `Connection`, **и** динамически созданные провайдеры репозиториев.

Если нужно зарегистрировать динамический модуль в глобальной области видимости, задайте свойству `global` значение `true`.

```typescript
{
  global: true,
  module: DatabaseModule,
  providers: providers,
  exports: providers,
}
```

> **Предупреждение** Как уже говорилось выше, делать всё глобальным — **не лучшее решение с точки зрения проектирования**.

`DatabaseModule` можно импортировать и настроить следующим образом:

```typescript
import { Module } from '@nestjs/common';
import { DatabaseModule } from './database/database.module';
import { User } from './users/entities/user.entity';

@Module({
  imports: [DatabaseModule.forRoot([User])],
})
export class AppModule {}
```

Если же вы хотите реэкспортировать динамический модуль, вызов метода `forRoot()` в массиве `exports` можно опустить:

```typescript
import { Module } from '@nestjs/common';
import { DatabaseModule } from './database/database.module';
import { User } from './users/entities/user.entity';

@Module({
  imports: [DatabaseModule.forRoot([User])],
  exports: [DatabaseModule],
})
export class AppModule {}
```
