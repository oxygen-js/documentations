### Interceptors

Interceptor (перехватчик) — это класс, помеченный декоратором `@Injectable()` и реализующий интерфейс `NestInterceptor`.

<figure><img class="illustrative-image" src="https://docs.nestjs.com/assets/Interceptors_1.png" /></figure>

Interceptors обладают набором полезных возможностей, навеянных техникой [аспектно-ориентированного программирования](https://en.wikipedia.org/wiki/Aspect-oriented_programming) (AOP). Они позволяют:

- добавлять дополнительную логику до и после выполнения метода
- преобразовывать результат, возвращаемый функцией
- преобразовывать исключение, выброшенное функцией
- расширять базовое поведение функции
- полностью подменять функцию при определённых условиях (например, ради кэширования)

#### Основы

Каждый interceptor реализует метод `intercept()`, принимающий два аргумента. Первый — экземпляр `ExecutionContext` (ровно тот же объект, что и для [guards](<08. Guards.md>)). `ExecutionContext` наследуется от `ArgumentsHost`. С `ArgumentsHost` мы уже встречались в главе о фильтрах исключений. Там мы выяснили, что это обёртка над аргументами, переданными исходному обработчику, и что она содержит разные массивы аргументов в зависимости от типа приложения. Подробнее об этом можно прочитать в главе о [фильтрах исключений](<06. Exception filters.md#arguments-host>).

#### Контекст выполнения

Расширяя `ArgumentsHost`, `ExecutionContext` добавляет ещё несколько вспомогательных методов, дающих дополнительные сведения о текущем процессе выполнения. Эти сведения помогают строить более универсальные interceptors, работающие с широким набором контроллеров, методов и контекстов выполнения. Подробнее об `ExecutionContext` читайте [здесь](<../03. Fundamentals/08. Execution context.md>).

#### Call handler

Второй аргумент — `CallHandler`. Интерфейс `CallHandler` реализует метод `handle()`, которым вы можете в нужный момент вызвать метод обработчика маршрута из своего interceptor. Если вы не вызовете `handle()` в своей реализации метода `intercept()`, метод обработчика маршрута вообще не будет выполнен.

Такой подход означает, что метод `intercept()` фактически **оборачивает** поток запроса/ответа. Благодаря этому вы можете реализовать собственную логику **и до, и после** выполнения конечного обработчика маршрута. Очевидно, что код в методе `intercept()` можно выполнить **до** вызова `handle()`, но как повлиять на то, что происходит после? Поскольку метод `handle()` возвращает `Observable`, мы можем использовать мощные операторы [RxJS](https://github.com/ReactiveX/rxjs) для дальнейшей работы с ответом. В терминологии аспектно-ориентированного программирования вызов обработчика маршрута (то есть вызов `handle()`) называется [Pointcut](https://en.wikipedia.org/wiki/Pointcut) — точкой, в которую вставляется наша дополнительная логика.

Рассмотрим, например, входящий запрос `POST /cats`. Он адресован обработчику `create()`, объявленному внутри `CatsController`. Если где-то по пути будет вызван interceptor, не вызывающий метод `handle()`, метод `create()` выполнен не будет. Как только `handle()` вызван (и его `Observable` возвращён), обработчик `create()` будет запущен. А когда через `Observable` придёт поток ответа, над ним можно выполнить дополнительные операции и вернуть вызывающей стороне итоговый результат.

#### Аспектный перехват

Первый сценарий, который мы рассмотрим, — использование interceptor для логирования действий пользователя (например, сохранение вызовов пользователя, асинхронная отправка событий или вычисление временной метки). Ниже приведён простой `LoggingInterceptor`:

**`logging.interceptor.ts`**

```typescript
import { Injectable, NestInterceptor, ExecutionContext, CallHandler } from '@nestjs/common';
import { Observable } from 'rxjs';
import { tap } from 'rxjs/operators';

@Injectable()
export class LoggingInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
    console.log('Before...');

    const now = Date.now();
    return next
      .handle()
      .pipe(
        tap(() => console.log(`After... ${Date.now() - now}ms`)),
      );
  }
}
```

> **Подсказка** `NestInterceptor<T, R>` — обобщённый интерфейс, где `T` обозначает тип `Observable<T>` (поддерживающего поток ответа), а `R` — тип значения, обёрнутого в `Observable<R>`.

> **Важно** Interceptors, как и контроллеры, провайдеры, guards и прочее, могут **внедрять зависимости** через свой `constructor`.

Так как `handle()` возвращает `Observable` из RxJS, в нашем распоряжении широкий выбор операторов для работы с потоком. В примере выше мы использовали оператор `tap()`: он вызывает нашу анонимную функцию логирования при штатном или аварийном завершении потока observable, но в остальном не вмешивается в цикл ответа.

#### Привязка interceptors

Чтобы подключить interceptor, мы используем декоратор `@UseInterceptors()`, импортируемый из пакета `@nestjs/common`. Как и [pipes](<07. Pipes.md>) с [guards](<08. Guards.md>), interceptors могут действовать в области контроллера, метода или глобально.

**`cats.controller.ts`**

```typescript
@UseInterceptors(LoggingInterceptor)
export class CatsController {}
```

> **Подсказка** Декоратор `@UseInterceptors()` импортируется из пакета `@nestjs/common`.

При такой конструкции каждый обработчик маршрута, объявленный в `CatsController`, будет использовать `LoggingInterceptor`. Когда кто-нибудь обратится к эндпоинту `GET /cats`, в стандартном выводе вы увидите следующее:

```typescript
Before...
After... 1ms
```

Обратите внимание, что мы передали класс `LoggingInterceptor` (а не экземпляр), оставив создание экземпляра фреймворку и тем самым включив внедрение зависимостей. Как и в случае с pipes, guards и фильтрами исключений, можно передать и готовый экземпляр:

**`cats.controller.ts`**

```typescript
@UseInterceptors(new LoggingInterceptor())
export class CatsController {}
```

Как уже говорилось, приведённая конструкция привязывает interceptor к каждому обработчику, объявленному в этом контроллере. Если нужно ограничить область действия interceptor одним методом, достаточно применить декоратор **на уровне метода**.

Чтобы подключить глобальный interceptor, используйте метод `useGlobalInterceptors()` экземпляра приложения Nest:

```typescript
const app = await NestFactory.create(AppModule);
app.useGlobalInterceptors(new LoggingInterceptor());
```

Глобальные interceptors действуют во всём приложении — для каждого контроллера и каждого обработчика маршрута. Что касается внедрения зависимостей, глобальные interceptors, зарегистрированные вне какого-либо модуля (через `useGlobalInterceptors()`, как в примере выше), не могут внедрять зависимости, поскольку регистрация происходит вне контекста модуля. Чтобы решить эту проблему, можно подключить interceptor **непосредственно из любого модуля** с помощью такой конструкции:

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { APP_INTERCEPTOR } from '@nestjs/core';

@Module({
  providers: [
    {
      provide: APP_INTERCEPTOR,
      useClass: LoggingInterceptor,
    },
  ],
})
export class AppModule {}
```

> **Подсказка** Используя этот подход для внедрения зависимостей в interceptor, учтите: независимо от того,
> в каком модуле применена такая конструкция, interceptor всё равно остаётся глобальным. Где же её размещать? Выбирайте модуль,
> в котором определён сам interceptor (`LoggingInterceptor` в примере выше). Кроме того, `useClass` — не единственный способ регистрации пользовательского провайдера. Подробнее читайте [здесь](<../03. Fundamentals/01. Custom providers.md>).

#### Преобразование ответа

Мы уже знаем, что `handle()` возвращает `Observable`. Поток содержит значение, **возвращённое** обработчиком маршрута, поэтому его легко изменить с помощью оператора `map()` из RxJS.

> **Предупреждение** Преобразование ответа не работает при использовании подхода, специфичного для библиотеки (обращаться напрямую к объекту `@Res()` в этом случае запрещено).

Создадим `TransformInterceptor`, который будет изменять каждый ответ простейшим образом, чтобы продемонстрировать сам процесс. Он воспользуется оператором `map()` из RxJS, чтобы поместить объект ответа в свойство `data` нового объекта и вернуть этот новый объект клиенту.

**`transform.interceptor.ts`**

```typescript
import { Injectable, NestInterceptor, ExecutionContext, CallHandler } from '@nestjs/common';
import { Observable } from 'rxjs';
import { map } from 'rxjs/operators';

export interface Response<T> {
  data: T;
}

@Injectable()
export class TransformInterceptor<T> implements NestInterceptor<T, Response<T>> {
  intercept(context: ExecutionContext, next: CallHandler): Observable<Response<T>> {
    return next.handle().pipe(map(data => ({ data })));
  }
}
```

> **Подсказка** Interceptors в Nest работают как с синхронными, так и с асинхронными методами `intercept()`. При необходимости достаточно сделать метод `async`.

С приведённой конструкцией обращение к эндпоинту `GET /cats` даст такой ответ (при условии, что обработчик маршрута возвращает пустой массив `[]`):

```json
{
  "data": []
}
```

Interceptors очень полезны для создания переиспользуемых решений задач, возникающих по всему приложению.
Представьте, например, что нам нужно превращать каждое значение `null` в пустую строку `''`. Это делается одной строкой кода, а привязав interceptor глобально, мы автоматически задействуем его в каждом зарегистрированном обработчике.

```typescript
import { Injectable, NestInterceptor, ExecutionContext, CallHandler } from '@nestjs/common';
import { Observable } from 'rxjs';
import { map } from 'rxjs/operators';

@Injectable()
export class ExcludeNullInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
    return next
      .handle()
      .pipe(map(value => value === null ? '' : value ));
  }
}
```

#### Преобразование исключений

Ещё один интересный сценарий — воспользоваться оператором `catchError()` из RxJS, чтобы подменять выбрасываемые исключения:

**`errors.interceptor.ts`**

```typescript
import {
  Injectable,
  NestInterceptor,
  ExecutionContext,
  BadGatewayException,
  CallHandler,
} from '@nestjs/common';
import { Observable, throwError } from 'rxjs';
import { catchError } from 'rxjs/operators';

@Injectable()
export class ErrorsInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
    return next
      .handle()
      .pipe(
        catchError(err => throwError(() => new BadGatewayException())),
      );
  }
}
```

#### Подмена потока

Есть несколько причин, по которым иногда хочется полностью предотвратить вызов обработчика и вернуть вместо этого другое значение. Очевидный пример — реализация кэша для сокращения времени ответа. Рассмотрим простой **cache interceptor**, возвращающий ответ из кэша. В реальном примере пришлось бы учесть и другие факторы — TTL, инвалидацию кэша, его размер и т. д., — но это выходит за рамки нашего обсуждения. Здесь мы приведём базовый пример, демонстрирующий основную идею.

**`cache.interceptor.ts`**

```typescript
import { Injectable, NestInterceptor, ExecutionContext, CallHandler } from '@nestjs/common';
import { Observable, of } from 'rxjs';

@Injectable()
export class CacheInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
    const isCached = true;
    if (isCached) {
      return of([]);
    }
    return next.handle();
  }
}
```

В нашем `CacheInterceptor` переменная `isCached` и ответ `[]` жёстко зашиты в код. Главное здесь то, что мы возвращаем новый поток, созданный оператором `of()` из RxJS, а значит, обработчик маршрута **не будет вызван** вовсе. Когда кто-нибудь обратится к эндпоинту, использующему `CacheInterceptor`, ответ (жёстко заданный пустой массив) вернётся немедленно. Чтобы получить универсальное решение, воспользуйтесь `Reflector` и создайте пользовательский декоратор. `Reflector` подробно описан в главе о [guards](<08. Guards.md>).

#### Другие операторы

Возможность манипулировать потоком с помощью операторов RxJS открывает множество возможностей. Рассмотрим ещё один распространённый сценарий. Допустим, вы хотите обрабатывать **таймауты** запросов к маршрутам. Если по истечении некоторого времени эндпоинт ничего не вернул, вы хотите завершить обработку ответом с ошибкой. Добиться этого позволяет следующая конструкция:

**`timeout.interceptor.ts`**

```typescript
import { Injectable, NestInterceptor, ExecutionContext, CallHandler, RequestTimeoutException } from '@nestjs/common';
import { Observable, throwError, TimeoutError } from 'rxjs';
import { catchError, timeout } from 'rxjs/operators';

@Injectable()
export class TimeoutInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
    return next.handle().pipe(
      timeout(5000),
      catchError(err => {
        if (err instanceof TimeoutError) {
          return throwError(() => new RequestTimeoutException());
        }
        return throwError(() => err);
      }),
    );
  };
};
```

Через 5 секунд обработка запроса будет отменена. Перед выбрасыванием `RequestTimeoutException` можно добавить собственную логику (например, освобождение ресурсов).
