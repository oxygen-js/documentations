### Guards

Guard — это класс, помеченный декоратором `@Injectable()` и реализующий интерфейс `CanActivate`.

<figure><img class="illustrative-image" src="https://docs.nestjs.com/assets/Guards_1.png" /></figure>

У guard-ов **единственная зона ответственности**. Они определяют, будет ли конкретный запрос обработан обработчиком маршрута, в зависимости от определённых условий (прав доступа, ролей, ACL и т. п.), известных во время выполнения. Обычно это называют **авторизацией**. В традиционных приложениях на Express авторизацию (и её «родственницу» — **аутентификацию**, с которой она обычно работает в паре) как правило реализовывали через [middleware](<05. Middleware.md>). Middleware — вполне подходящий выбор для аутентификации, поскольку такие вещи, как проверка токена и добавление свойств в объект `request`, слабо связаны с контекстом конкретного маршрута (и его метаданными).

Но middleware по своей природе «слеп». Он не знает, какой обработчик будет выполнен после вызова функции `next()`. **Guard-ы**, напротив, имеют доступ к экземпляру `ExecutionContext` и потому точно знают, что будет выполнено дальше. Как и фильтры исключений, pipe-ы и interceptor-ы, они спроектированы так, чтобы вы могли вклинить логику обработки ровно в нужную точку цикла запрос/ответ и сделать это декларативно. Это помогает сохранять код DRY и декларативным.

> **Подсказка** Guard-ы выполняются **после** всех middleware, но **до** любого interceptor-а или pipe-а.

#### Guard авторизации

Как уже сказано, **авторизация** — прекрасный сценарий применения guard-ов, поскольку определённые маршруты должны быть доступны только тогда, когда у вызывающей стороны (обычно у конкретного аутентифицированного пользователя) достаточно прав. `AuthGuard`, который мы сейчас напишем, предполагает, что пользователь аутентифицирован (и, следовательно, к заголовкам запроса приложен токен). Он извлечёт и проверит токен, а затем по полученной информации решит, можно ли пропустить запрос дальше.

**`auth.guard.ts`**

```typescript
import { Injectable, CanActivate, ExecutionContext } from '@nestjs/common';
import { Observable } from 'rxjs';

@Injectable()
export class AuthGuard implements CanActivate {
  canActivate(
    context: ExecutionContext,
  ): boolean | Promise<boolean> | Observable<boolean> {
    const request = context.switchToHttp().getRequest();
    return validateRequest(request);
  }
}
```

> **Подсказка** Если вам нужен реальный пример реализации механизма аутентификации в приложении, загляните в [эту главу](<../05. Security/01. Authentication.md>). Аналогично, более сложный пример авторизации есть на [этой странице](<../05. Security/02. Authorization.md>).

Логика внутри функции `validateRequest()` может быть сколь угодно простой или сложной. Главное в этом примере — показать, как guard-ы вписываются в цикл запрос/ответ.

Каждый guard обязан реализовать функцию `canActivate()`. Эта функция должна возвращать булево значение, указывающее, разрешён ли текущий запрос. Возвращать результат она может как синхронно, так и асинхронно (через `Promise` или `Observable`). Nest использует возвращённое значение, чтобы решить, что делать дальше:

- если возвращено `true`, запрос будет обработан;
- если возвращено `false`, Nest отклонит запрос.

#### Контекст выполнения

Функция `canActivate()` принимает единственный аргумент — экземпляр `ExecutionContext`. `ExecutionContext` наследуется от `ArgumentsHost`. С `ArgumentsHost` мы уже сталкивались в главе про фильтры исключений. В примере выше мы просто используем те же вспомогательные методы `ArgumentsHost`, что и раньше, чтобы получить ссылку на объект `Request`. Подробнее об этом — в разделе **Arguments host** главы про [фильтры исключений](<06. Exception filters.md#arguments-host>).

Расширяя `ArgumentsHost`, `ExecutionContext` добавляет несколько новых вспомогательных методов, дающих дополнительные сведения о текущем процессе выполнения. Эти сведения помогают строить более универсальные guard-ы, работающие с широким набором контроллеров, методов и контекстов выполнения. Подробнее об `ExecutionContext` читайте [здесь](<../03. Fundamentals/08. Execution context.md>).

#### Аутентификация на основе ролей

Построим более функциональный guard, разрешающий доступ только пользователям с определённой ролью. Начнём с базового шаблона guard-а и будем развивать его в следующих разделах. Пока он пропускает все запросы:

**`roles.guard.ts`**

```typescript
import { Injectable, CanActivate, ExecutionContext } from '@nestjs/common';
import { Observable } from 'rxjs';

@Injectable()
export class RolesGuard implements CanActivate {
  canActivate(
    context: ExecutionContext,
  ): boolean | Promise<boolean> | Observable<boolean> {
    return true;
  }
}
```

#### Привязка guard-ов

Как pipe-ы и фильтры исключений, guard-ы могут быть **на уровне контроллера**, на уровне метода или глобальными. Ниже мы настраиваем guard уровня контроллера с помощью декоратора `@UseGuards()`. Этот декоратор принимает один аргумент или список аргументов через запятую. Так вы можете одним объявлением применить нужный набор guard-ов.

```typescript
@Controller('cats')
@UseGuards(RolesGuard)
export class CatsController {}
```

> **Подсказка** Декоратор `@UseGuards()` импортируется из пакета `@nestjs/common`.

Выше мы передали класс `RolesGuard` (а не экземпляр), оставив создание экземпляра фреймворку и включив тем самым внедрение зависимостей. Как и в случае с pipe-ами и фильтрами исключений, можно передать и готовый экземпляр:

```typescript
@Controller('cats')
@UseGuards(new RolesGuard())
export class CatsController {}
```

Приведённая конструкция привязывает guard ко всем обработчикам, объявленным в этом контроллере. Если нужно применить guard только к одному методу, поставьте декоратор `@UseGuards()` **на уровне метода**.

Чтобы настроить глобальный guard, используйте метод `useGlobalGuards()` экземпляра приложения Nest:

```typescript
const app = await NestFactory.create(AppModule);
app.useGlobalGuards(new RolesGuard());
```

> **Важно** В случае гибридных приложений метод `useGlobalGuards()` по умолчанию не настраивает guard-ы для шлюзов и микросервисов (о том, как изменить это поведение, см. [Гибридное приложение](<../14. FAQ/06. Hybrid application.md>)). Для «обычных» (негибридных) микросервисных приложений `useGlobalGuards()` действительно подключает guard-ы глобально.

Глобальные guard-ы применяются во всём приложении — к каждому контроллеру и каждому обработчику маршрута. Что касается внедрения зависимостей, глобальные guard-ы, зарегистрированные вне какого-либо модуля (через `useGlobalGuards()`, как в примере выше), не могут внедрять зависимости, поскольку регистрация происходит вне контекста модуля. Чтобы решить эту проблему, можно настроить guard прямо из любого модуля такой конструкцией:

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { APP_GUARD } from '@nestjs/core';

@Module({
  providers: [
    {
      provide: APP_GUARD,
      useClass: RolesGuard,
    },
  ],
})
export class AppModule {}
```

> **Подсказка** При таком подходе к внедрению зависимостей для guard-а учтите, что независимо от того,
> в каком модуле применена эта конструкция, guard всё равно будет глобальным. Где это делать? Выбирайте модуль,
> в котором определён сам guard (`RolesGuard` в примере выше). Кроме того, `useClass` — не единственный способ
> регистрации пользовательского провайдера. Подробнее [здесь](<../03. Fundamentals/01. Custom providers.md>).

#### Задание ролей для каждого обработчика

Наш `RolesGuard` работает, но пока не слишком сообразителен. Мы ещё не используем важнейшую возможность guard-ов — [контекст выполнения](<../03. Fundamentals/08. Execution context.md>). Он пока ничего не знает о ролях и о том, какие роли разрешены для каждого обработчика. Например, у `CatsController` могут быть разные схемы прав для разных маршрутов. Одни доступны только администратору, другие открыты для всех. Как гибко и переиспользуемо сопоставить роли маршрутам?

Здесь в игру вступают **пользовательские метаданные** (подробнее [здесь](<../03. Fundamentals/08. Execution context.md#рефлексия-и-метаданные>)). Nest позволяет прикреплять к обработчикам маршрутов пользовательские **метаданные** — либо через декораторы, созданные статическим методом `Reflector.createDecorator`, либо через встроенный декоратор `@SetMetadata()`.

Например, создадим декоратор `@Roles()` методом `Reflector.createDecorator`, который прикрепит метаданные к обработчику. `Reflector` предоставляется фреймворком «из коробки» и экспортируется из пакета `@nestjs/core`.

**`roles.decorator.ts`**

```ts
import { Reflector } from '@nestjs/core';

export const Roles = Reflector.createDecorator<string[]>();
```

Декоратор `Roles` здесь — это функция, принимающая единственный аргумент типа `string[]`.

Теперь, чтобы использовать этот декоратор, просто пометим им обработчик:

**`cats.controller.ts`**

```typescript
@Post()
@Roles(['admin'])
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
```

Здесь мы прикрепили метаданные декоратора `Roles` к методу `create()`, указав, что доступ к этому маршруту должен быть только у пользователей с ролью `admin`.

Вместо метода `Reflector.createDecorator` можно было бы использовать встроенный декоратор `@SetMetadata()`. Подробнее об этом [здесь](<../03. Fundamentals/08. Execution context.md#низкоуровневый-подход>).

#### Собираем всё вместе

Вернёмся к нашему `RolesGuard` и свяжем всё воедино. Сейчас он всегда возвращает `true`, пропуская любой запрос. Мы хотим сделать возвращаемое значение зависимым от сравнения **ролей текущего пользователя** с ролями, которые фактически требуются для обрабатываемого маршрута. Чтобы получить доступ к ролям маршрута (пользовательским метаданным), снова воспользуемся вспомогательным классом `Reflector`:

**`roles.guard.ts`**

```typescript
import { Injectable, CanActivate, ExecutionContext } from '@nestjs/common';
import { Reflector } from '@nestjs/core';
import { Roles } from './roles.decorator';

@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    const roles = this.reflector.get(Roles, context.getHandler());
    if (!roles) {
      return true;
    }
    const request = context.switchToHttp().getRequest();
    const user = request.user;
    return matchRoles(roles, user.roles);
  }
}
```

> **Подсказка** В мире node.js принято прикреплять авторизованного пользователя к объекту `request`. Поэтому в примере выше мы предполагаем, что `request.user` содержит экземпляр пользователя и разрешённые роли. В вашем приложении такую связку вы, скорее всего, сделаете в собственном **guard-е аутентификации** (или в middleware). Подробнее об этом — в [этой главе](<../05. Security/01. Authentication.md>).

> **Предупреждение** Логика внутри функции `matchRoles()` может быть сколь угодно простой или сложной. Главное в этом примере — показать, как guard-ы вписываются в цикл запрос/ответ.

Подробнее о контекстно-зависимом использовании `Reflector` см. раздел <a href="../03.%20Fundamentals/08.%20Execution%20context.md#рефлексия-и-метаданные">Reflection and metadata</a> главы **Контекст выполнения**.

Когда пользователь с недостаточными привилегиями обращается к эндпоинту, Nest автоматически возвращает такой ответ:

```typescript
{
  "statusCode": 403,
  "message": "Forbidden resource",
  "error": "Forbidden"
}
```

Учтите, что «под капотом», когда guard возвращает `false`, фреймворк выбрасывает `ForbiddenException`. Если вы хотите вернуть другой ответ об ошибке, выбросьте собственное исключение. Например:

```typescript
throw new UnauthorizedException();
```

Любое исключение, выброшенное guard-ом, будет обработано [слоем исключений](<06. Exception filters.md>) (глобальным фильтром исключений и всеми фильтрами исключений, применёнными к текущему контексту).

> **Подсказка** Если вам нужен реальный пример реализации авторизации, загляните в [эту главу](<../05. Security/02. Authorization.md>).
