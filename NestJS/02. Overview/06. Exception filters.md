### Фильтры исключений

В Nest есть встроенный **слой исключений**, который отвечает за обработку всех необработанных исключений в приложении. Если исключение не обработано кодом вашего приложения, его перехватывает этот слой и автоматически отправляет подходящий понятный пользователю ответ.

<figure>
  <img class="illustrative-image" src="https://docs.nestjs.com/assets/Filter_1.png" />
</figure>

«Из коробки» этим занимается встроенный **глобальный фильтр исключений**, который обрабатывает исключения типа `HttpException` (и его подклассов). Если исключение **не распознано** (это не `HttpException` и не класс, унаследованный от `HttpException`), встроенный фильтр исключений формирует такой JSON-ответ по умолчанию:

```json
{
  "statusCode": 500,
  "message": "Internal server error"
}
```

> **Подсказка** Глобальный фильтр исключений частично поддерживает библиотеку `http-errors`. По сути, любое выброшенное исключение со свойствами `statusCode` и `message` будет корректно обработано и отправлено в ответе (вместо стандартного `InternalServerErrorException` для нераспознанных исключений).

#### Выбрасывание стандартных исключений

Nest предоставляет встроенный класс `HttpException`, доступный из пакета `@nestjs/common`. Для типичных приложений на базе HTTP REST/GraphQL API хорошей практикой считается отправлять стандартные объекты HTTP-ответа при возникновении определённых ошибок.

Например, в `CatsController` есть метод `findAll()` (обработчик маршрута `GET`). Предположим, что этот обработчик по какой-то причине выбрасывает исключение. Для демонстрации зададим его жёстко:

**`cats.controller.ts`**

```typescript
@Get()
async findAll() {
  throw new HttpException('Forbidden', HttpStatus.FORBIDDEN);
}
```

> **Подсказка** Здесь мы использовали `HttpStatus`. Это вспомогательное перечисление, импортируемое из пакета `@nestjs/common`.

Когда клиент обратится к этому эндпоинту, ответ будет выглядеть так:

```json
{
  "statusCode": 403,
  "message": "Forbidden"
}
```

Конструктор `HttpException` принимает два обязательных аргумента, определяющих ответ:

- Аргумент `response` задаёт тело JSON-ответа. Это может быть `string`
  или `object`, как описано ниже.
- Аргумент `status` задаёт [HTTP-код состояния](https://developer.mozilla.org/en-US/docs/Web/HTTP/Status).

По умолчанию тело JSON-ответа содержит два свойства:

- `statusCode`: по умолчанию — HTTP-код состояния, переданный в аргументе `status`
- `message`: краткое описание HTTP-ошибки на основе `status`

Чтобы переопределить только часть с сообщением в теле JSON-ответа, передайте строку в аргументе `response`. Чтобы переопределить всё тело JSON-ответа, передайте в аргументе `response` объект. Nest сериализует его и вернёт как тело JSON-ответа.

Второй аргумент конструктора — `status` — должен быть корректным HTTP-кодом состояния. Хорошая практика — использовать перечисление `HttpStatus`, импортированное из `@nestjs/common`.

Есть и **третий** (необязательный) аргумент конструктора — `options`, — с помощью которого можно указать [причину](https://nodejs.org/en/blog/release/v16.9.0/#error-cause) ошибки. Этот объект `cause` не сериализуется в объект ответа, но может быть полезен при логировании: он даёт ценную информацию о внутренней ошибке, из-за которой было выброшено `HttpException`.

Вот пример с переопределением всего тела ответа и указанием причины ошибки:

**`cats.controller.ts`**

```typescript
@Get()
async findAll() {
  try {
    await this.service.findAll()
  } catch (error) {
    throw new HttpException({
      status: HttpStatus.FORBIDDEN,
      error: 'This is a custom message',
    }, HttpStatus.FORBIDDEN, {
      cause: error
    });
  }
}
```

При таком коде ответ будет выглядеть так:

```json
{
  "status": 403,
  "error": "This is a custom message"
}
```

#### Логирование исключений

По умолчанию фильтр исключений не логирует встроенные исключения вроде `HttpException` (и любые исключения, унаследованные от него). Когда такие исключения выбрасываются, они не появляются в консоли, поскольку считаются частью нормального хода работы приложения. То же поведение действует и для других встроенных исключений, таких как `WsException` и `RpcException`.

Все эти исключения наследуются от базового класса `IntrinsicException`, экспортируемого из пакета `@nestjs/common`. Этот класс помогает отличить исключения, являющиеся частью нормальной работы приложения, от всех остальных.

Если вы хотите логировать такие исключения, создайте собственный фильтр исключений. Как это сделать, объясняется в следующем разделе.

#### Пользовательские исключения

Во многих случаях писать собственные исключения не потребуется: можно обойтись встроенным HTTP-исключением Nest, как описано в следующем разделе. Если же собственные исключения всё-таки нужны, хорошей практикой будет выстроить свою **иерархию исключений**, в которой они наследуются от базового класса `HttpException`. При таком подходе Nest распознает ваши исключения и автоматически позаботится об ответах с ошибками. Реализуем такое пользовательское исключение:

**`forbidden.exception.ts`**

```typescript
export class ForbiddenException extends HttpException {
  constructor() {
    super('Forbidden', HttpStatus.FORBIDDEN);
  }
}
```

Поскольку `ForbiddenException` расширяет базовый `HttpException`, он будет без проблем работать со встроенным обработчиком исключений, а значит, его можно использовать внутри метода `findAll()`.

**`cats.controller.ts`**

```typescript
@Get()
async findAll() {
  throw new ForbiddenException();
}
```

#### Встроенные HTTP-исключения

Nest предоставляет набор стандартных исключений, наследующих базовый `HttpException`. Они доступны из пакета `@nestjs/common` и покрывают многие из самых распространённых HTTP-исключений:

- `BadRequestException`
- `UnauthorizedException`
- `NotFoundException`
- `ForbiddenException`
- `NotAcceptableException`
- `RequestTimeoutException`
- `ConflictException`
- `GoneException`
- `HttpVersionNotSupportedException`
- `PayloadTooLargeException`
- `UnsupportedMediaTypeException`
- `UnprocessableEntityException`
- `InternalServerErrorException`
- `NotImplementedException`
- `ImATeapotException`
- `MethodNotAllowedException`
- `BadGatewayException`
- `ServiceUnavailableException`
- `GatewayTimeoutException`
- `PreconditionFailedException`

Все встроенные исключения тоже могут принимать причину ошибки `cause` и её описание через параметр `options`:

```typescript
throw new BadRequestException('Something bad happened', {
  cause: new Error(),
  description: 'Some error description',
});
```

При таком коде ответ будет выглядеть так:

```json
{
  "message": "Something bad happened",
  "error": "Some error description",
  "statusCode": 400
}
```

#### Фильтры исключений

Базовый (встроенный) фильтр исключений автоматически справляется со многими ситуациями, но иногда нужен **полный контроль** над слоем исключений. Например, вы можете захотеть добавить логирование или использовать другую JSON-схему в зависимости от каких-то динамических факторов. Именно для этого и предназначены **фильтры исключений**. Они позволяют управлять и ходом выполнения, и содержимым ответа, отправляемого клиенту.

Создадим фильтр исключений, который перехватывает исключения, являющиеся экземплярами класса `HttpException`, и реализует для них собственную логику ответа. Для этого нам понадобится доступ к объектам `Request` и `Response` нижележащей платформы. К объекту `Request` мы обратимся, чтобы получить исходный `url` и добавить его в логируемую информацию. Объект `Response` мы используем, чтобы напрямую управлять отправляемым ответом через метод `response.json()`.

**`http-exception.filter.ts`**

```typescript
import { ExceptionFilter, Catch, ArgumentsHost, HttpException } from '@nestjs/common';
import { Request, Response } from 'express';

@Catch(HttpException)
export class HttpExceptionFilter implements ExceptionFilter {
  catch(exception: HttpException, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    const request = ctx.getRequest<Request>();
    const status = exception.getStatus();

    response
      .status(status)
      .json({
        statusCode: status,
        timestamp: new Date().toISOString(),
        path: request.url,
      });
  }
}
```

> **Подсказка** Все фильтры исключений должны реализовывать обобщённый интерфейс `ExceptionFilter<T>`. Он требует определить метод `catch(exception: T, host: ArgumentsHost)` с указанной сигнатурой. `T` обозначает тип исключения.

> **Предупреждение** Если вы используете `@nestjs/platform-fastify`, вместо `response.json()` можно применять `response.send()`. Не забудьте импортировать правильные типы из `fastify`.

Декоратор `@Catch(HttpException)` привязывает к фильтру исключений необходимые метаданные, сообщая Nest, что этот конкретный фильтр ищет исключения типа `HttpException` и никакие другие. Декоратор `@Catch()` может принимать как один параметр, так и список через запятую. Это позволяет настроить фильтр сразу на несколько типов исключений.

#### Arguments host

Рассмотрим параметры метода `catch()`. Параметр `exception` — это обрабатываемый в данный момент объект исключения. Параметр `host` — это объект `ArgumentsHost`. `ArgumentsHost` — мощный служебный объект, который мы подробнее разберём в [главе о контексте выполнения](<../03. Fundamentals/08. Execution context.md>)\*. В этом примере кода мы используем его, чтобы получить ссылки на объекты `Request` и `Response`, передаваемые в исходный обработчик запроса (в контроллере, где возникло исключение). Здесь мы применили несколько вспомогательных методов `ArgumentsHost`, чтобы получить нужные объекты `Request` и `Response`. Подробнее об `ArgumentsHost` читайте [здесь](<../03. Fundamentals/08. Execution context.md>).

\*Такой уровень абстракции нужен потому, что `ArgumentsHost` работает во всех контекстах (например, в контексте HTTP-сервера, с которым мы работаем сейчас, а также в микросервисах и WebSocket). В главе о контексте выполнения мы увидим, как с помощью `ArgumentsHost` и его вспомогательных функций получить доступ к соответствующим <a href="../03.%20Fundamentals/08.%20Execution%20context.md#host-methods">нижележащим аргументам</a> для **любого** контекста выполнения. Это позволит писать универсальные фильтры исключений, работающие во всех контекстах.

#### Привязка фильтров

Привяжем наш новый `HttpExceptionFilter` к методу `create()` контроллера `CatsController`.

**`cats.controller.ts`**

```typescript
@Post()
@UseFilters(new HttpExceptionFilter())
async create(@Body() createCatDto: CreateCatDto) {
  throw new ForbiddenException();
}
```

> **Подсказка** Декоратор `@UseFilters()` импортируется из пакета `@nestjs/common`.

Здесь мы использовали декоратор `@UseFilters()`. Как и `@Catch()`, он может принимать один экземпляр фильтра или список экземпляров через запятую. В этом примере мы создали экземпляр `HttpExceptionFilter` прямо на месте. Как вариант, можно передать класс (вместо экземпляра), оставив создание экземпляра фреймворку и включив тем самым **внедрение зависимостей**.

**`cats.controller.ts`**

```typescript
@Post()
@UseFilters(HttpExceptionFilter)
async create(@Body() createCatDto: CreateCatDto) {
  throw new ForbiddenException();
}
```

> **Подсказка** По возможности применяйте фильтры через классы, а не экземпляры. Это снижает **расход памяти**, так как Nest может свободно переиспользовать экземпляры одного и того же класса во всём модуле.

В примере выше `HttpExceptionFilter` применяется только к одному обработчику маршрута `create()`, то есть имеет область видимости метода. Фильтры исключений можно применять на разных уровнях: на уровне метода контроллера/резолвера/шлюза, на уровне контроллера или глобально.
Например, чтобы применить фильтр на уровне контроллера, сделайте следующее:

**`cats.controller.ts`**

```typescript
@Controller()
@UseFilters(new HttpExceptionFilter())
export class CatsController {}
```

Такая конструкция подключает `HttpExceptionFilter` ко всем обработчикам маршрутов, определённым внутри `CatsController`.

Чтобы создать глобальный фильтр, сделайте следующее:

**`main.ts`**

```typescript
async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  app.useGlobalFilters(new HttpExceptionFilter());
  await app.listen(process.env.PORT ?? 3000);
}
bootstrap();
```

> **Предупреждение** Метод `useGlobalFilters()` не подключает фильтры к шлюзам и гибридным приложениям.

Глобальные фильтры действуют во всём приложении — для каждого контроллера и каждого обработчика маршрута. Что касается внедрения зависимостей: глобальные фильтры, зарегистрированные вне какого-либо модуля (через `useGlobalFilters()`, как в примере выше), не могут внедрять зависимости, поскольку регистрация происходит вне контекста модуля. Чтобы обойти это ограничение, глобальный фильтр можно зарегистрировать **прямо из любого модуля** такой конструкцией:

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { APP_FILTER } from '@nestjs/core';

@Module({
  providers: [
    {
      provide: APP_FILTER,
      useClass: HttpExceptionFilter,
    },
  ],
})
export class AppModule {}
```

> **Подсказка** Используя этот подход для внедрения зависимостей в фильтр, учтите: независимо от того, в каком модуле применена такая конструкция, фильтр остаётся глобальным. Где это лучше делать? Выберите модуль, в котором определён сам фильтр (`HttpExceptionFilter` в примере выше). Кроме того, `useClass` — не единственный способ регистрации пользовательского провайдера. Подробнее [здесь](<../03. Fundamentals/01. Custom providers.md>).

Таким способом можно добавить сколько угодно фильтров — просто перечислите каждый в массиве providers.

#### Перехват всего

Чтобы перехватывать **любое** необработанное исключение (независимо от его типа), оставьте список параметров декоратора `@Catch()` пустым, то есть `@Catch()`.

В примере ниже код не зависит от платформы, поскольку для отправки ответа используется [HTTP-адаптер](<../14. FAQ/02. HTTP adapter.md>), а специфичные для платформы объекты (`Request` и `Response`) напрямую не задействованы:

```typescript
import {
  ExceptionFilter,
  Catch,
  ArgumentsHost,
  HttpException,
  HttpStatus,
} from '@nestjs/common';
import { HttpAdapterHost } from '@nestjs/core';

@Catch()
export class CatchEverythingFilter implements ExceptionFilter {
  constructor(private readonly httpAdapterHost: HttpAdapterHost) {}

  catch(exception: unknown, host: ArgumentsHost): void {
    // В некоторых случаях `httpAdapter` может быть недоступен
    // в конструкторе, поэтому получаем его здесь.
    const { httpAdapter } = this.httpAdapterHost;

    const ctx = host.switchToHttp();

    const httpStatus =
      exception instanceof HttpException
        ? exception.getStatus()
        : HttpStatus.INTERNAL_SERVER_ERROR;

    const responseBody = {
      statusCode: httpStatus,
      timestamp: new Date().toISOString(),
      path: httpAdapter.getRequestUrl(ctx.getRequest()),
    };

    httpAdapter.reply(ctx.getResponse(), responseBody, httpStatus);
  }
}
```

> **Предупреждение** Если вы сочетаете фильтр, перехватывающий всё, с фильтром, привязанным к конкретному типу, фильтр «перехватить всё» следует объявлять первым, чтобы специализированный фильтр мог корректно обработать свой тип.

#### Наследование

Как правило, вы будете создавать полностью собственные фильтры исключений, заточенные под требования вашего приложения. Однако бывают случаи, когда хочется просто расширить встроенный **глобальный фильтр исключений** и переопределить поведение в зависимости от определённых условий.

Чтобы делегировать обработку исключения базовому фильтру, нужно унаследоваться от `BaseExceptionFilter` и вызвать унаследованный метод `catch()`.

**`all-exceptions.filter.ts`**

```typescript
import { Catch, ArgumentsHost } from '@nestjs/common';
import { BaseExceptionFilter } from '@nestjs/core';

@Catch()
export class AllExceptionsFilter extends BaseExceptionFilter {
  catch(exception: unknown, host: ArgumentsHost) {
    super.catch(exception, host);
  }
}
```

> **Предупреждение** Фильтры уровня метода и уровня контроллера, наследующие `BaseExceptionFilter`, не следует создавать через `new`. Вместо этого позвольте фреймворку создать их автоматически.

Глобальные фильтры **могут** расширять базовый фильтр. Сделать это можно двумя способами.

Первый способ — внедрить ссылку на `HttpAdapter` при создании пользовательского глобального фильтра:

```typescript
async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  const { httpAdapter } = app.get(HttpAdapterHost);
  app.useGlobalFilters(new AllExceptionsFilter(httpAdapter));

  await app.listen(process.env.PORT ?? 3000);
}
bootstrap();
```

Второй способ — использовать токен `APP_FILTER` <a href="06.%20Exception%20filters.md#привязка-фильтров">как показано здесь</a>.
