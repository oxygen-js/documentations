### Провайдеры

Провайдеры — одно из ключевых понятий Nest. Многие базовые классы Nest — сервисы, репозитории, фабрики, вспомогательные классы — можно рассматривать как провайдеры. Главная идея провайдера в том, что его можно **внедрить** как зависимость, позволяя объектам выстраивать между собой различные связи. Ответственность за «связывание» этих объектов в основном берёт на себя среда выполнения Nest.

<figure><img class="illustrative-image" src="https://docs.nestjs.com/assets/Components_1.png" /></figure>

В предыдущей главе мы создали простой `CatsController`. Контроллеры должны обрабатывать HTTP-запросы и делегировать более сложные задачи **провайдерам**. Провайдеры — это обычные классы JavaScript, объявленные в свойстве `providers` модуля NestJS. Подробнее об этом — в главе «Модули».

> **Подсказка** Поскольку Nest позволяет проектировать и организовывать зависимости в объектно-ориентированном стиле, мы настоятельно рекомендуем придерживаться [принципов SOLID](https://en.wikipedia.org/wiki/SOLID).

#### Сервисы

Начнём с создания простого `CatsService`. Этот сервис будет отвечать за хранение и извлечение данных и будет использоваться в `CatsController`. Из-за своей роли в управлении логикой приложения он идеально подходит на роль провайдера.

**`cats.service.ts`**

```typescript
import { Injectable } from '@nestjs/common';
import { Cat } from './interfaces/cat.interface';

@Injectable()
export class CatsService {
  private readonly cats: Cat[] = [];

  create(cat: Cat) {
    this.cats.push(cat);
  }

  findAll(): Cat[] {
    return this.cats;
  }
}
```

> **Подсказка** Чтобы создать сервис с помощью CLI, просто выполните команду `$ nest g service cats`.

Наш `CatsService` — обычный класс с одним свойством и двумя методами. Ключевое дополнение здесь — декоратор `@Injectable()`. Он добавляет к классу метаданные, сообщая, что `CatsService` может управляться [IoC](https://en.wikipedia.org/wiki/Inversion_of_control)-контейнером Nest.

Кроме того, в этом примере используется интерфейс `Cat`, который, скорее всего, выглядит примерно так:

**`interfaces/cat.interface.ts`**

```typescript
export interface Cat {
  name: string;
  age: number;
  breed: string;
}
```

Теперь, когда у нас есть класс сервиса для получения кошек, воспользуемся им внутри `CatsController`:

**`cats.controller.ts`**

```typescript
import { Controller, Get, Post, Body } from '@nestjs/common';
import { CreateCatDto } from './dto/create-cat.dto';
import { CatsService } from './cats.service';
import { Cat } from './interfaces/cat.interface';

@Controller('cats')
export class CatsController {
  constructor(private catsService: CatsService) {}

  @Post()
  async create(@Body() createCatDto: CreateCatDto) {
    this.catsService.create(createCatDto);
  }

  @Get()
  async findAll(): Promise<Cat[]> {
    return this.catsService.findAll();
  }
}
```

`CatsService` **внедряется** через конструктор класса. Обратите внимание на ключевое слово `private`. Это сокращение позволяет одной строкой и объявить, и инициализировать поле `catsService`, упрощая код.

#### Внедрение зависимостей

Nest построен вокруг мощного паттерна проектирования, известного как **внедрение зависимостей**. Мы настоятельно рекомендуем прочитать отличную статью об этой концепции в официальной [документации Angular](https://angular.dev/guide/di).

В Nest благодаря возможностям TypeScript управление зависимостями устроено просто: они разрешаются по своему типу. В примере ниже Nest разрешит `catsService`, создав и вернув экземпляр `CatsService` (или, в случае синглтона, вернув уже существующий экземпляр, если он был запрошен где-то ещё). Эта зависимость затем внедряется в конструктор вашего контроллера (либо присваивается указанному свойству):

```typescript
constructor(private catsService: CatsService) {}
```

#### Области видимости

Время жизни («область видимости») провайдера обычно совпадает с жизненным циклом приложения. При начальной загрузке приложения должна быть разрешена каждая зависимость, а значит, создан экземпляр каждого провайдера. Аналогично при остановке приложения все провайдеры уничтожаются. Впрочем, провайдер можно сделать и **привязанным к запросу** — тогда его время жизни определяется конкретным запросом, а не жизненным циклом приложения. Подробнее об этих приёмах — в главе [Области внедрения](<../03. Fundamentals/04. Injection scopes.md>).

#### Пользовательские провайдеры

В Nest есть встроенный контейнер инверсии управления («IoC»), который управляет связями между провайдерами. Эта возможность лежит в основе внедрения зависимостей, но на деле она гораздо мощнее, чем показано до сих пор. Определить провайдер можно несколькими способами: через обычные значения, классы, а также синхронные или асинхронные фабрики. Больше примеров объявления провайдеров — в главе [Внедрение зависимостей](<../03. Fundamentals/01. Custom providers.md>).

#### Необязательные провайдеры

Иногда встречаются зависимости, которые не обязательно должны быть разрешены. Например, ваш класс может зависеть от **объекта конфигурации**, но если его не передали, следует использовать значения по умолчанию. В таких случаях зависимость считается необязательной, и отсутствие провайдера конфигурации не должно приводить к ошибке.

Чтобы пометить провайдер как необязательный, используйте декоратор `@Optional()` в сигнатуре конструктора.

```typescript
import { Injectable, Optional, Inject } from '@nestjs/common';

@Injectable()
export class HttpService<T> {
  constructor(@Optional() @Inject('HTTP_OPTIONS') private httpClient: T) {}
}
```

В примере выше мы используем пользовательский провайдер, поэтому и указываем пользовательский **токен** `HTTP_OPTIONS`. В предыдущих примерах демонстрировалось внедрение через конструктор, когда зависимость обозначается классом в конструкторе. Подробнее о пользовательских провайдерах и о том, как работают связанные с ними токены, — в главе [Пользовательские провайдеры](<../03. Fundamentals/01. Custom providers.md>).

#### Внедрение через свойства

Приём, который мы использовали до сих пор, называется внедрением через конструктор: провайдеры передаются в метод-конструктор. В отдельных случаях бывает полезно **внедрение через свойства**. Например, если ваш класс верхнего уровня зависит от одного или нескольких провайдеров, прокидывание их всех вверх по цепочке через `super()` в подклассах становится обременительным. Чтобы этого избежать, можно применить декоратор `@Inject()` непосредственно к свойству.

```typescript
import { Injectable, Inject } from '@nestjs/common';

@Injectable()
export class HttpService<T> {
  @Inject('HTTP_OPTIONS')
  private readonly httpClient: T;
}
```

> **Предупреждение** Если ваш класс не наследуется от другого класса, обычно лучше использовать внедрение **через конструктор**. Конструктор явно перечисляет необходимые зависимости, что делает их лучше видимыми, а код — понятнее, чем в случае свойств класса, помеченных `@Inject`.

#### Регистрация провайдера

Теперь, когда у нас есть провайдер (`CatsService`) и его потребитель (`CatsController`), нужно зарегистрировать сервис в Nest, чтобы фреймворк смог выполнить внедрение. Для этого отредактируйте файл модуля (`app.module.ts`) и добавьте сервис в массив `providers` декоратора `@Module()`.

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { CatsController } from './cats/cats.controller';
import { CatsService } from './cats/cats.service';

@Module({
  controllers: [CatsController],
  providers: [CatsService],
})
export class AppModule {}
```

Теперь Nest сможет разрешить зависимости класса `CatsController`.

На этом этапе структура каталогов должна выглядеть так:

```text
src
  cats
    dto
      create-cat.dto.ts
    interfaces
      cat.interface.ts
    cats.controller.ts
    cats.service.ts
  app.module.ts
  main.ts
```
