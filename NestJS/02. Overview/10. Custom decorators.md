### Пользовательские декораторы маршрутов

Nest построен вокруг возможности языка под названием **декораторы**. Декораторы — хорошо известное понятие во многих распространённых языках программирования, но в мире JavaScript они всё ещё относительно новы. Чтобы лучше понять, как они работают, рекомендуем прочитать [эту статью](https://medium.com/google-developers/exploring-es7-decorators-76ecb65fb841). Вот простое определение:

<blockquote class="external">
  Декоратор ES2016 — это выражение, которое возвращает функцию и может принимать в качестве аргументов цель, имя и дескриптор свойства.
  Вы применяете его, ставя перед декоратором символ <code>@</code> и размещая его в самом начале того,
  что вы декорируете. Декораторы можно определять для класса, метода или свойства.
</blockquote>

Помимо этого вы можете создавать собственные **пользовательские декораторы**. Чем это полезно?

В мире node.js распространена практика прикреплять свойства к объекту **запроса**. Затем вы вручную извлекаете их в каждом обработчике маршрута примерно таким кодом:

```typescript
const user = req.user;
```

Чтобы сделать код более читаемым и понятным, вы можете создать декоратор `@User()` и переиспользовать его во всех своих контроллерах.

**`user.decorator.ts`**

```typescript
import { createParamDecorator, ExecutionContext } from '@nestjs/common';

export const User = createParamDecorator(
  (data: unknown, ctx: ExecutionContext) => {
    const request = ctx.switchToHttp().getRequest();
    return request.user;
  },
);
```

После этого его можно просто применять там, где он вам нужен.

```typescript
@Get()
async findOne(@User() user: UserEntity) {
  console.log(user);
}
```

#### Передача данных

Когда поведение декоратора зависит от каких-то условий, вы можете использовать параметр `data`, чтобы передать аргумент в фабричную функцию декоратора. Один из сценариев — пользовательский декоратор, извлекающий свойства из объекта запроса по ключу. Допустим, например, что наш <a href="../13.%20Recipes/04.%20Passport%20%28auth%29.md#реализация-стратегий-passport">слой аутентификации</a> проверяет запросы и прикрепляет к объекту запроса сущность пользователя. Сущность пользователя для аутентифицированного запроса может выглядеть так:

```json
{
  "id": 101,
  "firstName": "Alan",
  "lastName": "Turing",
  "email": "alan@email.com",
  "roles": ["admin"]
}
```

Определим декоратор, который принимает имя свойства в качестве ключа и возвращает соответствующее значение, если оно существует (либо undefined, если его нет или если объект `user` не был создан).

**`user.decorator.ts`**

```typescript
import { createParamDecorator, ExecutionContext } from '@nestjs/common';

export const User = createParamDecorator(
  (data: string, ctx: ExecutionContext) => {
    const request = ctx.switchToHttp().getRequest();
    const user = request.user;

    return data ? user?.[data] : user;
  },
);
```

Вот как теперь можно обратиться к конкретному свойству через декоратор `@User()` в контроллере:

```typescript
@Get()
async findOne(@User('firstName') firstName: string) {
  console.log(`Hello ${firstName}`);
}
```

Один и тот же декоратор можно использовать с разными ключами для доступа к разным свойствам. Если объект `user` вложенный или сложный, это делает реализацию обработчиков запросов проще и читабельнее.

> **Подсказка** Пользователям TypeScript стоит учитывать, что `createParamDecorator<T>()` — обобщённая функция. Это значит, что вы можете явно обеспечить типобезопасность, например `createParamDecorator<string>((data, ctx) => ...)`. Либо укажите тип параметра в фабричной функции, например `createParamDecorator((data: string, ctx) => ...)`. Если не сделать ни того, ни другого, типом `data` будет `any`.

#### Работа с pipes

Nest обращается с пользовательскими декораторами параметров так же, как со встроенными (`@Body()`, `@Param()` и `@Query()`). Это значит, что pipes выполняются и для параметров с пользовательскими аннотациями (в наших примерах — для аргумента `user`). Более того, pipe можно применить непосредственно к пользовательскому декоратору:

```typescript
@Get()
async findOne(
  @User(new ValidationPipe({ validateCustomDecorators: true }))
  user: UserEntity,
) {
  console.log(user);
}
```

> **Подсказка** Обратите внимание, что опция `validateCustomDecorators` должна быть установлена в true. По умолчанию `ValidationPipe` не валидирует аргументы, помеченные пользовательскими декораторами.

#### Композиция декораторов

Nest предоставляет вспомогательный метод для композиции нескольких декораторов. Предположим, например, что вы хотите объединить все декораторы, связанные с аутентификацией, в один. Это делается следующим образом:

**`auth.decorator.ts`**

```typescript
import { applyDecorators } from '@nestjs/common';

export function Auth(...roles: Role[]) {
  return applyDecorators(
    SetMetadata('roles', roles),
    UseGuards(AuthGuard, RolesGuard),
    ApiBearerAuth(),
    ApiUnauthorizedResponse({ description: 'Unauthorized' }),
  );
}
```

Затем этот пользовательский декоратор `@Auth()` можно использовать так:

```typescript
@Get('users')
@Auth('admin')
findAllUsers() {}
```

В результате все четыре декоратора применяются одним объявлением.

> **Предупреждение** Декоратор `@ApiHideProperty()` из пакета `@nestjs/swagger` не поддерживает композицию и не будет корректно работать с функцией `applyDecorators`.
