### Middleware

Middleware — это функция, которая вызывается **перед** обработчиком маршрута. Функции middleware имеют доступ к объектам [request](https://expressjs.com/en/4x/api.html#req) и [response](https://expressjs.com/en/4x/api.html#res), а также к функции `next()` в цикле «запрос — ответ» приложения. **Следующую** функцию middleware принято обозначать переменной с именем `next`.

<figure><img class="illustrative-image" src="https://docs.nestjs.com/assets/Middlewares_1.png" /></figure>

По умолчанию middleware в Nest эквивалентны middleware в [express](https://expressjs.com/en/guide/using-middleware.html). Следующее описание из официальной документации express перечисляет возможности middleware:

<blockquote class="external">
  Функции middleware могут выполнять следующие задачи:
  <ul>
    <li>выполнять любой код;</li>
    <li>вносить изменения в объекты запроса и ответа;</li>
    <li>завершать цикл «запрос — ответ»;</li>
    <li>вызывать следующую функцию middleware в стеке;</li>
    <li>если текущая функция middleware не завершает цикл «запрос — ответ», она обязана вызвать <code>next()</code>, чтобы
      передать управление следующей функции middleware. Иначе запрос останется висеть без ответа.</li>
  </ul>
</blockquote>

Собственный middleware в Nest реализуется либо функцией, либо классом с декоратором `@Injectable()`. Класс должен реализовывать интерфейс `NestMiddleware`, а к функции никаких особых требований нет. Начнём с реализации простого middleware в виде класса.

> **Предупреждение** `Express` и `fastify` обрабатывают middleware по-разному и предоставляют разные сигнатуры методов, подробнее [здесь](<../04. Techniques/19. Performance (Fastify).md#middleware>).

**`logger.middleware.ts`**

```typescript
import { Injectable, NestMiddleware } from '@nestjs/common';
import { Request, Response, NextFunction } from 'express';

@Injectable()
export class LoggerMiddleware implements NestMiddleware {
  use(req: Request, res: Response, next: NextFunction) {
    console.log('Request...');
    next();
  }
}
```

#### Внедрение зависимостей

Middleware в Nest полностью поддерживают внедрение зависимостей. Как и провайдеры с контроллерами, они могут **внедрять зависимости**, доступные в пределах того же модуля. Как обычно, это делается через `constructor`.

#### Применение middleware

В декораторе `@Module()` для middleware места нет. Вместо этого они настраиваются методом `configure()` класса модуля. Модули, включающие middleware, должны реализовывать интерфейс `NestModule`. Настроим `LoggerMiddleware` на уровне `AppModule`.

**`app.module.ts`**

```typescript
import { Module, NestModule, MiddlewareConsumer } from '@nestjs/common';
import { LoggerMiddleware } from './common/middleware/logger.middleware';
import { CatsModule } from './cats/cats.module';

@Module({
  imports: [CatsModule],
})
export class AppModule implements NestModule {
  configure(consumer: MiddlewareConsumer) {
    consumer
      .apply(LoggerMiddleware)
      .forRoutes('cats');
  }
}
```

В примере выше мы настроили `LoggerMiddleware` для обработчиков маршрута `/cats`, ранее определённых внутри `CatsController`. Мы также можем дополнительно ограничить middleware конкретным методом запроса, передав при настройке в метод `forRoutes()` объект с `path` маршрута и `method` запроса. Обратите внимание, что в примере ниже мы импортируем перечисление `RequestMethod`, чтобы указать нужный тип метода запроса.

**`app.module.ts`**

```typescript
import { Module, NestModule, RequestMethod, MiddlewareConsumer } from '@nestjs/common';
import { LoggerMiddleware } from './common/middleware/logger.middleware';
import { CatsModule } from './cats/cats.module';

@Module({
  imports: [CatsModule],
})
export class AppModule implements NestModule {
  configure(consumer: MiddlewareConsumer) {
    consumer
      .apply(LoggerMiddleware)
      .forRoutes({ path: 'cats', method: RequestMethod.GET });
  }
}
```

> **Подсказка** Метод `configure()` можно сделать асинхронным с помощью `async/await` (например, вы можете дождаться (`await`) завершения асинхронной операции внутри тела метода `configure()`).

> **Предупреждение** При использовании адаптера `express` приложение NestJS по умолчанию регистрирует `json` и `urlencoded` из пакета `body-parser`. Это значит, что если вы хотите настроить этот middleware через `MiddlewareConsumer`, вам нужно отключить глобальный middleware, установив флаг `bodyParser` в `false` при создании приложения через `NestFactory.create()`.

#### Шаблоны в маршрутах

Middleware в NestJS также поддерживают маршруты на основе шаблонов. Например, именованный wildcard (`*splat`) можно использовать для сопоставления с любой комбинацией символов в маршруте. В следующем примере middleware будет выполняться для любого маршрута, начинающегося с `abcd/`, независимо от того, сколько символов идёт дальше.

```typescript
forRoutes({
  path: 'abcd/*splat',
  method: RequestMethod.ALL,
});
```

> **Подсказка** `splat` — это просто имя параметра-wildcard, никакого особого смысла у него нет. Вы можете назвать его как угодно, например `*wildcard`.

Путь маршрута `'abcd/*'` совпадёт с `abcd/1`, `abcd/123`, `abcd/abc` и так далее. Дефис ( `-`) и точка (`.`) в строковых путях интерпретируются буквально. Однако `abcd/` без дополнительных символов маршруту не соответствует. Чтобы это работало, нужно заключить wildcard в фигурные скобки, сделав его необязательным:

```typescript
forRoutes({
  path: 'abcd/{*splat}',
  method: RequestMethod.ALL,
});
```

#### Middleware consumer

`MiddlewareConsumer` — вспомогательный класс. Он предоставляет несколько встроенных методов для управления middleware. Все они легко объединяются в **цепочку** в [текучем стиле](https://en.wikipedia.org/wiki/Fluent_interface). Метод `forRoutes()` может принимать одну строку, несколько строк, объект `RouteInfo`, класс контроллера и даже несколько классов контроллеров. В большинстве случаев вы, скорее всего, будете просто передавать список **контроллеров** через запятую. Ниже приведён пример с одним контроллером:

**`app.module.ts`**

```typescript
import { Module, NestModule, MiddlewareConsumer } from '@nestjs/common';
import { LoggerMiddleware } from './common/middleware/logger.middleware';
import { CatsModule } from './cats/cats.module';
import { CatsController } from './cats/cats.controller';

@Module({
  imports: [CatsModule],
})
export class AppModule implements NestModule {
  configure(consumer: MiddlewareConsumer) {
    consumer
      .apply(LoggerMiddleware)
      .forRoutes(CatsController);
  }
}
```

> **Подсказка** Метод `apply()` может принимать как один middleware, так и несколько аргументов, чтобы задать <a href="05.%20Middleware.md#несколько-middleware">несколько middleware</a>.

#### Исключение маршрутов

Иногда нам нужно **исключить** определённые маршруты из-под действия middleware. Это легко сделать методом `exclude()`. Метод `exclude()` принимает одну строку, несколько строк или объект `RouteInfo`, определяющий исключаемые маршруты.

Вот пример его использования:

```typescript
consumer
  .apply(LoggerMiddleware)
  .exclude(
    { path: 'cats', method: RequestMethod.GET },
    { path: 'cats', method: RequestMethod.POST },
    'cats/{*splat}',
  )
  .forRoutes(CatsController);
```

> **Подсказка** Метод `exclude()` поддерживает параметры-wildcard с помощью пакета [path-to-regexp](https://github.com/pillarjs/path-to-regexp#parameters).

В примере выше `LoggerMiddleware` будет привязан ко всем маршрутам, определённым внутри `CatsController`, **кроме** трёх, переданных в метод `exclude()`.

Такой подход даёт гибкость в применении или исключении middleware для конкретных маршрутов или шаблонов маршрутов.

#### Функциональные middleware

Класс `LoggerMiddleware`, который мы использовали, довольно прост. У него нет полей, дополнительных методов и зависимостей. Почему бы не определить его обычной функцией вместо класса? На самом деле так можно. Такой вид middleware называется **функциональным middleware**. Преобразуем middleware-логгер из класса в функцию, чтобы показать разницу:

**`logger.middleware.ts`**

```typescript
import { Request, Response, NextFunction } from 'express';

export function logger(req: Request, res: Response, next: NextFunction) {
  console.log(`Request...`);
  next();
};
```

И используем его внутри `AppModule`:

**`app.module.ts`**

```typescript
consumer
  .apply(logger)
  .forRoutes(CatsController);
```

> **Подсказка** Если вашему middleware не нужны зависимости, стоит рассмотреть более простой вариант — **функциональный middleware**.

#### Несколько middleware

Как упоминалось выше, чтобы привязать несколько middleware, выполняемых последовательно, достаточно передать список через запятую в метод `apply()`:

```typescript
consumer.apply(cors(), helmet(), logger).forRoutes(CatsController);
```

#### Глобальный middleware

Если мы хотим привязать middleware сразу ко всем зарегистрированным маршрутам, можно воспользоваться методом `use()`, который предоставляет экземпляр `INestApplication`:

**`main.ts`**

```typescript
const app = await NestFactory.create(AppModule);
app.use(logger);
await app.listen(process.env.PORT ?? 3000);
```

> **Подсказка** Обратиться к DI-контейнеру из глобального middleware невозможно. При использовании `app.use()` вместо этого можно применить [функциональный middleware](<05. Middleware.md#функциональные-middleware>). Либо использовать middleware-класс и подключить его через `.forRoutes('*')` внутри `AppModule` (или любого другого модуля).
