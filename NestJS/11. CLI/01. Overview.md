### Обзор

[Nest CLI](https://github.com/nestjs/nest-cli) — это инструмент командной строки, который помогает создавать, разрабатывать и поддерживать приложения на Nest. Он полезен во многих отношениях: генерирует каркас проекта, запускает его в режиме разработки, а также собирает и упаковывает приложение для продакшена. CLI воплощает лучшие архитектурные практики, помогая строить хорошо структурированные приложения.

#### Установка

**Примечание**: в этом руководстве описывается установка пакетов, включая Nest CLI, с помощью [npm](https://docs.npmjs.com/downloading-and-installing-node-js-and-npm). При желании вы можете использовать другие пакетные менеджеры. В npm есть несколько способов управлять тем, как командная строка вашей ОС находит исполняемый файл `nest`. Здесь мы описываем глобальную установку бинарника `nest` с ключом `-g`. Так удобнее, и именно этот подход подразумевается по всей документации. Учтите, что при глобальной установке **любого** пакета `npm` ответственность за использование правильной версии ложится на пользователя. Кроме того, это означает, что при наличии нескольких проектов все они будут работать с **одной и той же** версией CLI. Разумная альтернатива — программа [npx](https://github.com/npm/cli/blob/latest/docs/lib/content/commands/npx.md), входящая в состав `npm` (или аналогичные возможности других пакетных менеджеров): она гарантирует запуск **управляемой версии** Nest CLI. Рекомендуем ознакомиться с [документацией npx](https://github.com/npm/cli/blob/latest/docs/lib/content/commands/npx.md) и/или обратиться за подробностями к вашим DevOps-специалистам.

Установите CLI глобально командой `npm install -g` (подробности о глобальной установке — в **Примечании** выше).

```bash
$ npm install -g @nestjs/cli
```

> **Подсказка** Как вариант, вы можете использовать команду `npx @nestjs/cli@latest`, не устанавливая CLI глобально.

#### Базовый рабочий процесс

После установки команды CLI вызываются прямо из командной строки ОС через исполняемый файл `nest`. Список доступных команд `nest` можно посмотреть так:

```bash
$ nest --help
```

Справку по отдельной команде можно получить следующей конструкцией. Подставьте вместо `generate` в примере ниже любую команду — `new`, `add` и т. д., — чтобы получить подробную справку по ней:

```bash
$ nest generate --help
```

Чтобы создать, собрать и запустить новый простой проект Nest в режиме разработки, перейдите в папку, которая должна стать родительской для вашего нового проекта, и выполните следующие команды:

```bash
$ nest new my-nest-project
$ cd my-nest-project
$ npm run start:dev
```

Откройте в браузере [http://localhost:3000](http://localhost:3000), чтобы увидеть работающее приложение. При изменении любого исходного файла приложение автоматически пересобирается и перезагружается.

> **Подсказка** Для ускорения сборки рекомендуем использовать [сборщик SWC](<../13. Recipes/03. SWC (fast compiler).md>) (он в 10 раз производительнее стандартного компилятора TypeScript).

#### Структура проекта

При запуске `nest new` Nest генерирует шаблонную структуру приложения: создаёт новую папку и наполняет её начальным набором файлов. Вы можете и дальше работать в этой структуре по умолчанию, добавляя новые компоненты так, как описано в этой документации. Структуру проекта, создаваемую командой `nest new`, мы называем **стандартным режимом**. Nest также поддерживает альтернативную структуру для управления несколькими проектами и библиотеками — **режим монорепозитория**.

За исключением нескольких особенностей процесса **сборки** (по сути, режим монорепозитория упрощает сложности сборки, которые иногда возникают в проектах со структурой монорепозитория) и встроенной поддержки [библиотек](<03. Libraries.md>), все остальные возможности Nest и вся эта документация одинаково применимы и к стандартному режиму, и к режиму монорепозитория. Более того, вы в любой момент можете легко перейти со стандартного режима на монорепозиторий, так что это решение можно спокойно отложить, пока вы только знакомитесь с Nest.

Управлять несколькими проектами можно в любом из режимов. Ниже — краткая сводка отличий:

| Возможность                                                | Стандартный режим                                                  | Режим монорепозитория                                      |
| ---------------------------------------------------------- | ------------------------------------------------------------------ | ---------------------------------------------------------- |
| Несколько проектов                                         | Раздельная структура в файловой системе                            | Единая структура в файловой системе                        |
| `node_modules` и `package.json`                            | Отдельные экземпляры                                               | Общие для всего монорепозитория                            |
| Компилятор по умолчанию                                    | `tsc`                                                              | webpack                                                    |
| Настройки компилятора                                      | Задаются отдельно                                                  | Значения монорепозитория, переопределяемые для проекта     |
| Файлы конфигурации `eslint.config.mjs`, `.prettierrc` и др.| Задаются отдельно                                                  | Общие для всего монорепозитория                            |
| Команды `nest build` и `nest start`                        | По умолчанию нацелены на единственный проект в контексте           | По умолчанию нацелены на **проект по умолчанию** в монорепозитории |
| Библиотеки                                                 | Управляются вручную, обычно через npm-пакеты                       | Встроенная поддержка, включая управление путями и сборку   |

Чтобы решить, какой режим подходит вам больше, прочитайте разделы [Рабочие пространства](<02. Workspaces.md>) и [Библиотеки](<03. Libraries.md>).

#### Синтаксис команд CLI

Все команды `nest` имеют один и тот же формат:

```bash
nest commandOrAlias requiredArg [optionalArg] [options]
```

Например:

```bash
$ nest new my-nest-project --dry-run
```

Здесь `new` — это _commandOrAlias_. У команды `new` есть псевдоним `n`. `my-nest-project` — это _requiredArg_. Если _requiredArg_ не указан в командной строке, `nest` запросит его. Кроме того, у `--dry-run` есть короткая форма `-d`. С учётом этого следующая команда эквивалентна приведённой выше:

```bash
$ nest n my-nest-project -d
```

У большинства команд и у некоторых опций есть псевдонимы. Запустите `nest new --help`, чтобы посмотреть эти опции и псевдонимы и убедиться, что вы правильно поняли описанные конструкции.

#### Обзор команд

Выполните `nest <command> --help` для любой из перечисленных ниже команд, чтобы увидеть её собственные опции.

Подробное описание каждой команды приведено в разделе [использование](<04. Usage.md>).

| Команда    | Псевдоним | Описание                                                                                     |
| ---------- | ----- | ---------------------------------------------------------------------------------------------- |
| `new`      | `n`   | Генерирует каркас нового приложения в _стандартном режиме_ со всеми файлами, нужными для запуска. |
| `generate` | `g`   | Создаёт и/или изменяет файлы на основе схематика.                                              |
| `build`    |       | Компилирует приложение или рабочее пространство в выходную папку.                              |
| `start`    |       | Компилирует и запускает приложение (или проект по умолчанию в рабочем пространстве).           |
| `add`      |       | Подключает библиотеку, упакованную как **nest library**, выполняя её схематик установки.       |
| `info`     | `i`   | Показывает информацию об установленных пакетах nest и другие полезные сведения о системе.      |

#### Требования

Nest CLI требует бинарник Node.js, собранный с [поддержкой интернационализации](https://nodejs.org/api/intl.html) (ICU), — например, официальные сборки со [страницы загрузки Node.js](https://nodejs.org/en/download). Если вы столкнулись с ошибками, связанными с ICU, проверьте, что ваш бинарник удовлетворяет этому требованию.

```bash
node -p process.versions.icu
```

Если команда выводит `undefined`, значит, ваш бинарник Node.js собран без поддержки интернационализации.
