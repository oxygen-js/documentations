### Nest CLI и скрипты

В этом разделе подробнее рассказывается о том, как команда `nest` взаимодействует с компиляторами и скриптами. Это поможет DevOps-специалистам управлять средой разработки.

Приложение на Nest — это **обычное** TypeScript-приложение, которое перед запуском нужно скомпилировать в JavaScript. Шаг компиляции можно выполнить разными способами, и разработчики или команды вольны выбрать тот, который им удобнее. С учётом этого Nest предоставляет «из коробки» набор инструментов, преследующих следующие цели:

- Дать стандартный процесс сборки и запуска, доступный из командной строки, который «просто работает» с разумными значениями по умолчанию.
- Обеспечить **открытость** процесса сборки и запуска, чтобы разработчики могли напрямую обращаться к нижележащим инструментам и настраивать их родными средствами и опциями.
- Оставаться полностью стандартным фреймворком TypeScript/Node.js, чтобы всем конвейером компиляции, развёртывания и запуска можно было управлять любыми внешними инструментами по выбору команды.

Эта цель достигается сочетанием команды `nest`, локально установленного компилятора TypeScript и скриптов в `package.json`. Ниже описано, как эти технологии работают вместе. Это должно помочь вам понять, что происходит на каждом шаге сборки и запуска и как при необходимости изменить это поведение.

#### Бинарный файл nest

Команда `nest` — это бинарный файл уровня операционной системы (то есть запускается из командной строки ОС). Фактически эта команда охватывает 3 отдельные области, описанные ниже. Мы рекомендуем запускать подкоманды сборки (`nest build`) и запуска (`nest start`) через скрипты `package.json`, которые автоматически создаются при генерации каркаса проекта (см. [typescript starter](https://github.com/nestjs/typescript-starter), если вы хотите начать с клонирования репозитория, а не с запуска `nest new`).

#### Сборка

`nest build` — это обёртка над стандартным компилятором `tsc` или компилятором `swc` (для [стандартных проектов](<01. Overview.md#структура-проекта>)) либо над сборщиком webpack с использованием `ts-loader` (для [монорепозиториев](<01. Overview.md#структура-проекта>)). Она не добавляет никаких дополнительных возможностей или шагов компиляции, кроме встроенной обработки `tsconfig-paths`. Она существует потому, что большинству разработчиков, особенно начинающих знакомство с Nest, не требуется менять параметры компилятора (например, файл `tsconfig.json`), что порой бывает непросто.

Подробности смотрите в документации по [nest build](<04. Usage.md#nest-build>).

#### Запуск

`nest start` просто убеждается, что проект собран (то же самое, что `nest build`), а затем удобным и переносимым образом вызывает команду `node` для запуска скомпилированного приложения. Как и в случае со сборкой, вы можете настроить этот процесс как угодно — либо с помощью команды `nest start` и её опций, либо полностью заменив её. Весь процесс представляет собой стандартный конвейер сборки и запуска TypeScript-приложения, и вы вправе управлять им соответствующим образом.

Подробности смотрите в документации по [nest start](<04. Usage.md#nest-start>).

#### Генерация

Команды `nest generate`, как следует из названия, генерируют новые проекты Nest или компоненты внутри них.

#### Скрипты пакета

Запуск команд `nest` на уровне командной строки ОС требует, чтобы бинарный файл `nest` был установлен глобально. Это стандартная возможность npm, находящаяся вне прямого контроля Nest. Одно из следствий этого в том, что глобально установленный бинарный файл `nest` **не** управляется как зависимость проекта в `package.json`. Например, два разных разработчика могут запускать две разные версии бинарного файла `nest`. Стандартное решение — использовать скрипты пакета, чтобы работать с инструментами сборки и запуска как с зависимостями разработки.

Когда вы выполняете `nest new` или клонируете [typescript starter](https://github.com/nestjs/typescript-starter), Nest заполняет секцию скриптов в `package.json` нового проекта командами вроде `build` и `start`. Кроме того, он устанавливает нижележащие инструменты компиляции (такие как `typescript`) как **dev-зависимости**.

Скрипты сборки и запуска вызываются командами вида:

```bash
$ npm run build
```

и

```bash
$ npm run start
```

Эти команды используют возможности npm по запуску скриптов, чтобы выполнить `nest build` или `nest start` с помощью **локально установленного** бинарного файла `nest`. Применяя встроенные скрипты пакета, вы получаете полный контроль над зависимостями команд Nest CLI\*. Это означает, что при таком **рекомендуемом** способе использования все участники вашей организации гарантированно работают с одной и той же версией команд.

\*Это относится к командам `build` и `start`. Команды `nest new` и `nest generate` не входят в конвейер сборки и запуска, поэтому работают в ином контексте и не имеют встроенных скриптов в `package.json`.

Большинству разработчиков и команд рекомендуется использовать скрипты пакета для сборки и запуска проектов Nest. Вы можете полностью настроить поведение этих скриптов через их опции (`--path`, `--webpack`, `--webpackPath`) и/или через файлы настроек компилятора `tsc` либо webpack (например, `tsconfig.json`). Вы также вправе организовать полностью собственный процесс сборки для компиляции TypeScript (или даже выполнять TypeScript напрямую через `ts-node`).

#### Обратная совместимость

Поскольку приложения Nest — это чистые TypeScript-приложения, прежние версии скриптов сборки и запуска продолжат работать. Обновлять их не обязательно. Вы можете перейти на новые команды `nest build` и `nest start`, когда будете готовы, или продолжать использовать прежние либо собственные скрипты.

#### Миграция

Хотя вносить какие-либо изменения не требуется, вы можете захотеть перейти на новые команды CLI вместо таких инструментов, как `tsc-watch` или `ts-node`. В этом случае просто установите последнюю версию `@nestjs/cli` — и глобально, и локально:

```bash
$ npm install -g @nestjs/cli
$ cd  /some/project/root/folder
$ npm install -D @nestjs/cli
```

Затем вы можете заменить скрипты в секции `scripts` файла `package.json` на следующие:

```typescript
"build": "nest build",
"start": "nest start",
"start:dev": "nest start --watch",
"start:debug": "nest start --debug --watch",
```
