### Библиотеки

Многим приложениям приходится решать одни и те же типовые задачи или переиспользовать модульный компонент в разных контекстах. В Nest для этого есть несколько подходов, и каждый работает на своём уровне, помогая достичь разных архитектурных и организационных целей.

[Модули](<../02. Overview/04. Modules.md>) Nest удобны тем, что предоставляют контекст выполнения, позволяющий разделять компоненты внутри одного приложения. Модули также можно упаковать в [npm](https://npmjs.com) и получить переиспользуемую библиотеку, устанавливаемую в разные проекты. Это эффективный способ распространения настраиваемых переиспользуемых библиотек, которыми могут пользоваться разные, слабо связанные или вовсе не связанные между собой организации (например, распространение и установка сторонних библиотек).

Для совместного использования кода внутри тесно связанных групп (например, в границах компании или проекта) полезен более лёгкий подход к разделению компонентов. Для этого появились монорепозитории, а внутри монорепозитория **библиотека** даёт простой и лёгкий способ переиспользовать код. В монорепозитории Nest библиотеки позволяют легко собирать приложения из общих компонентов. Более того, это поощряет декомпозицию монолитных приложений и процессов разработки в сторону создания и композиции модульных компонентов.

#### Библиотеки Nest

Библиотека Nest — это проект Nest, который отличается от приложения тем, что не может запускаться самостоятельно. Чтобы её код выполнился, библиотеку нужно импортировать в содержащее её приложение. Встроенная поддержка библиотек, описанная в этом разделе, доступна только для **монорепозиториев** (проекты в стандартном режиме могут добиться похожего результата с помощью npm-пакетов).

Например, организация может разработать `AuthModule`, который реализует аутентификацию в соответствии с корпоративными политиками, действующими для всех внутренних приложений. Вместо того чтобы создавать такой модуль отдельно для каждого приложения или физически упаковывать код в npm и требовать установки в каждом проекте, монорепозиторий позволяет объявить этот модуль библиотекой. При такой организации все потребители библиотечного модуля видят актуальную версию `AuthModule` сразу после коммита. Это даёт заметные преимущества при координации разработки и сборки компонентов и упрощает сквозное тестирование.

#### Создание библиотек

Любая функциональность, пригодная для переиспользования, — кандидат на выделение в библиотеку. Решение о том, что должно стать библиотекой, а что остаться частью приложения, относится к архитектурному проектированию. Создание библиотеки — это не просто копирование кода из существующего приложения в новую библиотеку. При упаковке в библиотеку код должен быть отвязан от приложения. Это может потребовать **больше** времени на старте и вынудить принять решения по проектированию, с которыми вы не столкнулись бы при более тесно связанном коде. Но эти дополнительные усилия окупаются, когда библиотека ускоряет сборку сразу нескольких приложений.

Чтобы начать создание библиотеки, выполните следующую команду:

```bash
$ nest g library my-library
```

При запуске команды схематик `library` спросит у вас префикс (он же алиас) для библиотеки:

```bash
What prefix would you like to use for the library (default: @app)?
```

Так в вашем рабочем пространстве создаётся новый проект с именем `my-library`.
Проект типа «библиотека», как и проект типа «приложение», генерируется схематиком в папку с соответствующим именем. Библиотеки размещаются в папке `libs` в корне монорепозитория. Nest создаёт папку `libs` при создании первой библиотеки.

Файлы, генерируемые для библиотеки, немного отличаются от файлов приложения. Вот содержимое папки `libs` после выполнения команды выше:

```text
libs
  my-library
    src
      index.ts
      my-library.module.ts
      my-library.service.ts
    tsconfig.lib.json
```

В файле `nest-cli.json` появится новая запись о библиотеке под ключом `"projects"`:

```javascript
...
{
    "my-library": {
      "type": "library",
      "root": "libs/my-library",
      "entryFile": "index",
      "sourceRoot": "libs/my-library/src",
      "compilerOptions": {
        "tsConfigPath": "libs/my-library/tsconfig.lib.json"
      }
}
...
```

Метаданные библиотек и приложений в `nest-cli.json` различаются двумя моментами:

- свойство `"type"` имеет значение `"library"` вместо `"application"`
- свойство `"entryFile"` имеет значение `"index"` вместо `"main"`

Эти отличия указывают процессу сборки, что библиотеку нужно обрабатывать особым образом. Например, библиотека экспортирует свои функции через файл `index.js`.

Как и у проектов-приложений, у каждой библиотеки есть собственный файл `tsconfig.lib.json`, расширяющий корневой (общий для монорепозитория) `tsconfig.json`. При необходимости вы можете изменить его, чтобы задать специфичные для библиотеки опции компилятора.

Собрать библиотеку можно командой CLI:

```bash
$ nest build my-library
```

#### Использование библиотек

Когда конфигурационные файлы сгенерированы автоматически, пользоваться библиотеками просто. Как импортировать `MyLibraryService` из библиотеки `my-library` в приложение `my-project`?

Прежде всего отметим, что библиотечные модули используются точно так же, как любые другие модули Nest. Монорепозиторий берёт на себя управление путями, благодаря чему импорт библиотек и сборка происходят прозрачно. Чтобы воспользоваться `MyLibraryService`, нужно импортировать объявляющий его модуль. Изменим `my-project/src/app.module.ts` так, чтобы импортировать `MyLibraryModule`.

```typescript
import { Module } from '@nestjs/common';
import { AppController } from './app.controller';
import { AppService } from './app.service';
import { MyLibraryModule } from '@app/my-library';

@Module({
  imports: [MyLibraryModule],
  controllers: [AppController],
  providers: [AppService],
})
export class AppModule {}
```

Обратите внимание, что выше в строке `import` ES-модуля использован псевдоним пути `@app` — тот самый `prefix`, который мы указали в команде `nest g library`. «Под капотом» Nest реализует это через сопоставление путей в tsconfig. При добавлении библиотеки Nest обновляет ключ `"paths"` в глобальном (общем для монорепозитория) файле `tsconfig.json` следующим образом:

```javascript
"paths": {
    "@app/my-library": [
        "libs/my-library/src"
    ],
    "@app/my-library/*": [
        "libs/my-library/src/*"
    ]
}
```

Итак, если коротко, сочетание монорепозитория и библиотек делает подключение библиотечных модулей к приложениям простым и интуитивно понятным.

Этот же механизм позволяет собирать и разворачивать приложения, составленные из библиотек. После импорта `MyLibraryModule` команда `nest build` автоматически разрешает все модули и собирает приложение вместе со всеми библиотечными зависимостями для развёртывания. По умолчанию компилятор монорепозитория — **webpack**, поэтому результатом сборки будет один файл, объединяющий все транспилированные JavaScript-файлы. Вы также можете переключиться на `tsc`, как описано <a href="02.%20Workspaces.md#глобальные-опции-компилятора">здесь</a>.
