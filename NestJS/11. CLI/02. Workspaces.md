### Рабочие пространства

В Nest есть два режима организации кода:

- **стандартный режим**: подходит для разработки отдельных приложений со своими зависимостями и настройками, которым не нужно оптимизировать совместное использование модулей или сложные сборки. Это режим по умолчанию.
- **режим монорепозитория**: в этом режиме артефакты кода рассматриваются как части лёгкого **монорепозитория**; он может быть удобнее для команд разработчиков и/или окружений с несколькими проектами. Он автоматизирует часть процесса сборки, упрощая создание и композицию модульных компонентов, поощряет повторное использование кода, облегчает интеграционное тестирование, упрощает совместное использование общих для всех проектов артефактов — например, правил `eslint` и других конфигурационных политик, — и проще в использовании, чем альтернативы вроде подмодулей Git. В режиме монорепозитория применяется понятие **рабочего пространства**, описанного в файле `nest-cli.json`, которое координирует связи между компонентами монорепозитория.

Важно отметить, что практически все возможности Nest не зависят от выбранного режима организации кода. **Единственное**, на что влияет этот выбор, — как устроена композиция проектов и как формируются артефакты сборки. Вся остальная функциональность — от CLI до основных и дополнительных модулей — работает одинаково в обоих режимах.

Кроме того, вы в любой момент можете легко переключиться со **стандартного режима** на **режим монорепозитория**, так что это решение можно отложить до момента, когда преимущества того или иного подхода станут очевиднее.

#### Стандартный режим

Когда вы выполняете `nest new`, с помощью встроенной схемы (schematic) создаётся новый **проект**. Nest делает следующее:

1. Создаёт новую папку, соответствующую аргументу `name`, переданному в `nest new`.
2. Наполняет её файлами по умолчанию, соответствующими минимальному базовому приложению Nest. Ознакомиться с этими файлами можно в репозитории [typescript-starter](https://github.com/nestjs/typescript-starter).
3. Добавляет дополнительные файлы, такие как `nest-cli.json`, `package.json` и `tsconfig.json`, которые настраивают и включают различные инструменты для компиляции, тестирования и запуска приложения.

Дальше вы можете изменять стартовые файлы, добавлять новые компоненты и зависимости (например, через `npm install`) и в целом разрабатывать приложение так, как описано в остальной части документации.

#### Режим монорепозитория

Чтобы включить режим монорепозитория, начните со структуры _стандартного режима_ и добавьте **проекты**. Проект может быть либо полноценным **приложением** (добавляется в рабочее пространство командой `nest generate app`), либо **библиотекой** (добавляется командой `nest generate library`). Подробности об этих типах компонентов проекта мы обсудим ниже. Сейчас важно понимать одно: именно **добавление проекта** к существующей структуре стандартного режима **превращает её** в структуру монорепозитория. Рассмотрим пример.

Если выполнить:

```bash
$ nest new my-project
```

мы получим структуру _стандартного режима_ со следующим деревом папок:

```text
node_modules
src
  app.controller.ts
  app.module.ts
  app.service.ts
  main.ts
nest-cli.json
package.json
tsconfig.json
eslint.config.mjs
```

Преобразовать её в структуру монорепозитория можно так:

```bash
$ cd my-project
$ nest generate app my-app
```

В этот момент `nest` преобразует существующую структуру в структуру **режима монорепозитория**. Это приводит к нескольким важным изменениям. Дерево папок теперь выглядит так:

```text
apps
  my-app
    src
      app.controller.ts
      app.module.ts
      app.service.ts
      main.ts
    tsconfig.app.json
  my-project
    src
      app.controller.ts
      app.module.ts
      app.service.ts
      main.ts
    tsconfig.app.json
nest-cli.json
package.json
tsconfig.json
eslint.config.mjs
```

Схема `generate app` реорганизовала код: перенесла каждый проект-**приложение** в папку `apps` и добавила специфичный для проекта файл `tsconfig.app.json` в корневую папку каждого проекта. Исходное приложение `my-project` стало **проектом по умолчанию** для монорепозитория и теперь находится наравне с только что добавленным `my-app` в папке `apps`. Проекты по умолчанию мы рассмотрим ниже.

> **Предупреждение** Преобразование структуры стандартного режима в монорепозиторий работает только для проектов, следующих канонической структуре проекта Nest. В частности, при преобразовании схема пытается переместить папки `src` и `test` в папку проекта внутри корневой папки `apps`. Если проект не следует этой структуре, преобразование завершится ошибкой или даст непредсказуемый результат.

#### Проекты рабочего пространства

В монорепозитории для управления входящими в него сущностями используется понятие рабочего пространства. Рабочие пространства состоят из **проектов**. Проект может быть:

- **приложением**: полноценным приложением Nest с файлом `main.ts` для начальной загрузки приложения. Если не считать особенностей компиляции и сборки, проект-приложение внутри рабочего пространства функционально идентичен приложению в структуре _стандартного режима_.
- **библиотекой**: библиотека — это способ упаковать набор возможностей общего назначения (модули, провайдеры, контроллеры и т. д.), которые можно использовать в других проектах. Библиотека не может выполняться самостоятельно и не имеет файла `main.ts`. Подробнее о библиотеках читайте [здесь](<03. Libraries.md>).

В каждом рабочем пространстве есть **проект по умолчанию** (он должен быть проектом-приложением). Он задаётся свойством верхнего уровня `"root"` в файле `nest-cli.json`, которое указывает на корень проекта по умолчанию (подробнее см. раздел [Свойства CLI](<02. Workspaces.md#свойства-cli>) ниже). Обычно это приложение **стандартного режима**, с которого вы начали и которое затем преобразовали в монорепозиторий командой `nest generate app`. Если вы следуете этим шагам, свойство заполняется автоматически.

Проекты по умолчанию используются командами `nest` — такими как `nest build` и `nest start` — когда имя проекта не указано.

Например, в приведённой выше структуре монорепозитория команда

```bash
$ nest start
```

запустит приложение `my-project`. Чтобы запустить `my-app`, нужно выполнить:

```bash
$ nest start my-app
```

#### Приложения

Проекты-приложения, которые мы неформально называем просто «приложениями», — это полноценные приложения Nest, которые можно запускать и развёртывать. Проект-приложение создаётся командой `nest generate app`.

Эта команда автоматически генерирует каркас проекта, включая стандартные папки `src` и `test` из [typescript starter](https://github.com/nestjs/typescript-starter). В отличие от стандартного режима, проект-приложение в монорепозитории не содержит ни файла зависимостей (`package.json`), ни других конфигурационных артефактов вроде `.prettierrc` и `eslint.config.mjs`. Вместо них используются общие для монорепозитория зависимости и конфигурационные файлы.

Однако схема всё же создаёт специфичный для проекта файл `tsconfig.app.json` в корневой папке проекта. Этот файл конфигурации автоматически задаёт подходящие параметры сборки, включая корректную настройку папки для вывода компиляции. Файл расширяет `tsconfig.json` верхнего уровня (уровня монорепозитория), так что вы можете управлять глобальными настройками для всего монорепозитория и при необходимости переопределять их на уровне проекта.

#### Библиотеки

Как уже упоминалось, проекты-библиотеки, или просто «библиотеки», — это пакеты компонентов Nest, которые нужно включить в состав приложений, чтобы они работали. Проект-библиотека создаётся командой `nest generate library`. Решение о том, что именно должно входить в библиотеку, — вопрос архитектурного проектирования. Подробно библиотеки рассматриваются в главе [Библиотеки](<03. Libraries.md>).

#### Свойства CLI

Метаданные, необходимые для организации, сборки и развёртывания проектов как со стандартной структурой, так и со структурой монорепозитория, Nest хранит в файле `nest-cli.json`. Nest автоматически дополняет и обновляет этот файл по мере добавления проектов, поэтому обычно о нём не нужно думать и его не нужно редактировать. Тем не менее некоторые настройки может понадобиться изменить вручную, поэтому полезно иметь общее представление об этом файле.

После выполнения описанных выше шагов по созданию монорепозитория наш файл `nest-cli.json` выглядит так:

```javascript
{
  "collection": "@nestjs/schematics",
  "sourceRoot": "apps/my-project/src",
  "monorepo": true,
  "root": "apps/my-project",
  "compilerOptions": {
    "webpack": true,
    "tsConfigPath": "apps/my-project/tsconfig.app.json"
  },
  "projects": {
    "my-project": {
      "type": "application",
      "root": "apps/my-project",
      "entryFile": "main",
      "sourceRoot": "apps/my-project/src",
      "compilerOptions": {
        "tsConfigPath": "apps/my-project/tsconfig.app.json"
      }
    },
    "my-app": {
      "type": "application",
      "root": "apps/my-app",
      "entryFile": "main",
      "sourceRoot": "apps/my-app/src",
      "compilerOptions": {
        "tsConfigPath": "apps/my-app/tsconfig.app.json"
      }
    }
  }
}
```

Файл разделён на секции:

- глобальная секция со свойствами верхнего уровня, управляющими стандартными настройками и настройками уровня монорепозитория;
- свойство верхнего уровня (`"projects"`) с метаданными о каждом проекте. Эта секция присутствует только в структурах режима монорепозитория.

Свойства верхнего уровня следующие:

- `"collection"`: указывает на набор схем, используемых для генерации компонентов; обычно это значение изменять не следует;
- `"sourceRoot"`: указывает на корень исходного кода единственного проекта в структурах стандартного режима или _проекта по умолчанию_ в структурах режима монорепозитория;
- `"compilerOptions"`: набор пар, где ключи задают опции компилятора, а значения — их настройки; подробности см. ниже;
- `"generateOptions"`: набор пар, где ключи задают глобальные опции генерации, а значения — их настройки; подробности см. ниже;
- `"monorepo"`: (только для монорепозитория) для структуры режима монорепозитория это значение всегда `true`;
- `"root"`: (только для монорепозитория) указывает на корень _проекта по умолчанию_.

#### Глобальные опции компилятора

Эти свойства задают используемый компилятор, а также различные опции, влияющие на **любой** шаг компиляции — в рамках `nest build` или `nest start` — и независимо от компилятора, будь то `tsc` или webpack.

| Имя свойства        | Тип значения        | Описание                                                                                                                                                                                                                                                                  |
| ------------------- | ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `webpack`           | boolean             | Если `true`, используется [компилятор webpack](https://webpack.js.org/). Если `false` или свойство отсутствует, используется `tsc`. В режиме монорепозитория по умолчанию `true` (используется webpack), в стандартном режиме — `false` (используется `tsc`). Подробности ниже. (устарело: используйте вместо этого `builder`) |
| `tsConfigPath`      | string              | (**только для монорепозитория**) Указывает на файл с настройками `tsconfig.json`, которые будут использоваться при вызове `nest build` или `nest start` без опции `project` (например, когда собирается или запускается проект по умолчанию).                              |
| `webpackConfigPath` | string              | Указывает на файл с опциями webpack. Если не задан, Nest ищет файл `webpack.config.js`. Подробности ниже.                                                                                                                                                                 |
| `deleteOutDir`      | boolean             | Если `true`, при каждом вызове компилятора сначала удаляется папка вывода компиляции (заданная в `tsconfig.json`; по умолчанию `./dist`).                                                                                                                                 |
| `assets`            | array               | Включает автоматическое копирование не-TypeScript ресурсов при каждом запуске шага компиляции (копирование ресурсов **не** происходит при инкрементальной компиляции в режиме `--watch`). Подробности ниже.                                                                |
| `watchAssets`       | boolean             | Если `true`, запуск в режиме отслеживания **всех** не-TypeScript ресурсов. (Для более тонкой настройки отслеживаемых ресурсов см. раздел [Ресурсы](<02. Workspaces.md#ресурсы>) ниже).                                                                                            |
| `manualRestart`     | boolean             | Если `true`, включает сокращённую команду `rs` для ручного перезапуска сервера. Значение по умолчанию — `false`.                                                                                                                                                          |
| `builder`           | string/object       | Указывает CLI, какой `builder` использовать для компиляции проекта (`tsc`, `swc` или `webpack`). Чтобы настроить поведение сборщика, можно передать объект с двумя атрибутами: `type` (`tsc`, `swc` или `webpack`) и `options`.                                            |
| `typeCheck`         | boolean             | Если `true`, включает проверку типов для проектов на SWC (когда `builder` равен `swc`). Значение по умолчанию — `false`.                                                                                                                                                  |

#### Глобальные опции генерации

Эти свойства задают опции генерации по умолчанию, используемые командой `nest generate`.

| Имя свойства  | Тип значения        | Описание                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ------------- | ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `spec`        | boolean _или_ object | Если значение булево, `true` включает генерацию `spec`-файлов по умолчанию, а `false` — отключает. Флаг, переданный в командной строке CLI, переопределяет эту настройку, как и специфичная для проекта настройка `generateOptions` (подробнее ниже). Если значение — объект, каждый ключ соответствует имени схемы, а булево значение определяет, включена или отключена генерация spec-файлов по умолчанию для этой конкретной схемы. |
| `flat`        | boolean             | Если true, все команды генерации будут создавать плоскую структуру.                                                                                                                                                                                                                                                                                                                                                                            |

В следующем примере с помощью булева значения указывается, что генерация spec-файлов должна быть отключена по умолчанию для всех проектов:

```javascript
{
  "generateOptions": {
    "spec": false
  },
  ...
}
```

В следующем примере с помощью булева значения указывается, что плоская генерация файлов должна применяться по умолчанию ко всем проектам:

```javascript
{
  "generateOptions": {
    "flat": true
  },
  ...
}
```

В следующем примере генерация `spec`-файлов отключается только для схем `service` (например, `nest generate service...`):

```javascript
{
  "generateOptions": {
    "spec": {
      "service": false
    }
  },
  ...
}
```

> **Предупреждение** При задании `spec` в виде объекта ключ схемы генерации в настоящее время не поддерживает автоматическую обработку псевдонимов. Это значит, что если указать, например, `service: false` и попытаться сгенерировать сервис через псевдоним `s`, spec-файл всё равно будет создан. Чтобы и обычное имя схемы, и псевдоним работали как задумано, указывайте и обычное имя команды, и псевдоним, как показано ниже.
>
> ```javascript
> {
>   "generateOptions": {
>     "spec": {
>       "service": false,
>       "s": false
>     }
>   },
>   ...
> }
> ```

#### Опции генерации на уровне проекта

Помимо глобальных опций генерации, можно задать опции генерации на уровне отдельного проекта. Такие опции имеют в точности тот же формат, что и глобальные, но указываются непосредственно для каждого проекта.

Опции генерации на уровне проекта переопределяют глобальные опции генерации.

```javascript
{
  "projects": {
    "cats-project": {
      "generateOptions": {
        "spec": {
          "service": false
        }
      },
      ...
    }
  },
  ...
}
```

> **Предупреждение** Порядок приоритета опций генерации следующий. Опции, указанные в командной строке CLI, имеют приоритет над опциями уровня проекта. Опции уровня проекта переопределяют глобальные опции.

#### Выбор компилятора

Причина различия компиляторов по умолчанию в том, что для крупных проектов (более типичных для монорепозитория) webpack может дать существенные преимущества по времени сборки и позволяет получить единый файл, объединяющий все компоненты проекта. Если вы хотите генерировать отдельные файлы, задайте `"webpack"` значение `false` — тогда процесс сборки будет использовать `tsc` (или `swc`).

#### Опции webpack

Файл опций webpack может содержать стандартные [опции конфигурации webpack](https://webpack.js.org/configuration/). Например, чтобы указать webpack включать в бандл `node_modules` (которые по умолчанию исключаются), добавьте в `webpack.config.js` следующее:

```javascript
module.exports = {
  externals: [],
};
```

Поскольку конфигурационный файл webpack — это файл JavaScript, вы даже можете экспортировать функцию, которая принимает опции по умолчанию и возвращает изменённый объект:

```javascript
module.exports = function (options) {
  return {
    ...options,
    externals: [],
  };
};
```

#### Ресурсы

Компиляция TypeScript автоматически копирует вывод компилятора (файлы `.js` и `.d.ts`) в указанную папку вывода. Бывает удобно копировать и не-TypeScript файлы — например, файлы `.graphql`, `images`, файлы `.html` и другие ресурсы. Это позволяет рассматривать `nest build` (и любой начальный шаг компиляции) как лёгкий шаг **сборки для разработки**, в рамках которого вы можете править не-TypeScript файлы и итеративно компилировать и тестировать приложение.
Ресурсы должны находиться в папке `src`, иначе они не будут скопированы.

Значение ключа `assets` должно быть массивом элементов, задающих копируемые файлы. Элементами могут быть простые строки со спецификациями файлов в стиле `glob`, например:

```typescript
"assets": ["**/*.graphql"],
"watchAssets": true,
```

Для более тонкой настройки элементы могут быть объектами со следующими ключами:

- `"include"`: спецификации файлов в стиле `glob` для копируемых ресурсов;
- `"exclude"`: спецификации файлов в стиле `glob` для ресурсов, которые нужно **исключить** из списка `include`;
- `"outDir"`: строка с путём (относительно корневой папки), куда следует копировать ресурсы. По умолчанию — та же папка вывода, что настроена для вывода компилятора;
- `"watchAssets"`: булево значение; если `true`, запуск в режиме отслеживания указанных ресурсов.

Например:

```typescript
"assets": [
  { "include": "**/*.graphql", "exclude": "**/omitted.graphql", "watchAssets": true },
]
```

> **Предупреждение** Установка `watchAssets` в свойстве верхнего уровня `compilerOptions` переопределяет любые настройки `watchAssets` внутри свойства `assets`.

#### Свойства проектов

Этот элемент присутствует только в структурах режима монорепозитория. Обычно эти свойства редактировать не следует, так как Nest использует их для поиска проектов и их конфигурационных опций внутри монорепозитория.
