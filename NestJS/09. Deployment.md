### Развёртывание

Когда вы готовы развернуть своё приложение NestJS в продакшене, есть несколько ключевых шагов, которые помогут ему работать максимально эффективно. В этом руководстве мы разберём основные советы и лучшие практики, помогающие успешно развернуть приложение NestJS.

#### Предварительные требования

Перед развёртыванием приложения NestJS убедитесь, что у вас есть:

- Работающее приложение NestJS, готовое к развёртыванию.
- Доступ к платформе развёртывания или серверу, где вы сможете разместить приложение.
- Все необходимые переменные окружения, настроенные для вашего приложения.
- Все требуемые сервисы, например база данных, настроенные и готовые к работе.
- Как минимум LTS-версия Node.js, установленная на платформе развёртывания.

> **Подсказка** Если вы ищете облачную платформу для развёртывания приложения NestJS, обратите внимание на [Mau](https://mau.nestjs.com/ 'Deploy Nest') — нашу официальную платформу для развёртывания приложений NestJS на AWS. С Mau развернуть приложение NestJS так же просто, как нажать пару кнопок и выполнить одну команду:
>
> ```bash
> $ npm install -g @nestjs/mau
> $ mau deploy
> ```
>
> Как только развёртывание завершится, ваше приложение NestJS за считаные секунды будет запущено на AWS!

#### Сборка приложения

Чтобы собрать приложение NestJS, нужно скомпилировать код TypeScript в JavaScript. В результате появляется каталог `dist` со скомпилированными файлами. Собрать приложение можно следующей командой:

```bash
$ npm run build
```

Эта команда обычно «под капотом» запускает `nest build` — по сути обёртку над компилятором TypeScript с некоторыми дополнительными возможностями (копирование ассетов и т. п.). Если у вас есть собственный скрипт сборки, вы можете запустить его напрямую. Кроме того, для монорепозиториев NestJS CLI не забудьте передать имя собираемого проекта аргументом (`npm run build my-app`).

После успешной компиляции в корне проекта должен появиться каталог `dist` со скомпилированными файлами и точкой входа `main.js`. Если в корневом каталоге проекта есть какие-либо `.ts`-файлы (и ваш `tsconfig.json` настроен на их компиляцию), они тоже будут скопированы в каталог `dist`, немного изменив структуру каталогов (вместо `dist/main.js` вы получите `dist/src/main.js` — учитывайте это при настройке сервера).

#### Продакшен-окружение

Продакшен-окружение — это среда, в которой ваше приложение будет доступно внешним пользователям. Это может быть облачная платформа вроде [AWS](https://aws.amazon.com/) (с EC2, ECS и т. д.), [Azure](https://azure.microsoft.com/) или [Google Cloud](https://cloud.google.com/), либо выделенный сервер под вашим управлением, например [Hetzner](https://www.hetzner.com/).

Чтобы упростить развёртывание и избежать ручной настройки, вы можете воспользоваться сервисом вроде [Mau](https://mau.nestjs.com/ 'Deploy Nest') — нашей официальной платформы для развёртывания приложений NestJS на AWS. Подробности — в [этом разделе](todo).

Вот некоторые плюсы использования **облачной платформы** или сервиса вроде [Mau](https://mau.nestjs.com/ 'Deploy Nest'):

- Масштабируемость: легко масштабировать приложение по мере роста числа пользователей.
- Безопасность: встроенные средства защиты и сертификаты соответствия.
- Мониторинг: наблюдение за производительностью и состоянием приложения в реальном времени.
- Надёжность: гарантии высокой доступности приложения.

С другой стороны, облачные платформы обычно дороже самостоятельного хостинга, и у вас может быть меньше контроля над нижележащей инфраструктурой. Обычный VPS может оказаться хорошим выбором, если вы ищете более экономичное решение и обладаете техническими навыками для самостоятельного управления сервером, но помните, что обслуживание сервера, безопасность и резервное копирование придётся брать на себя.

#### NODE_ENV=production

Хотя технически между разработкой и продакшеном в Node.js и NestJS нет разницы, при запуске приложения в продакшен-окружении рекомендуется устанавливать переменную окружения `NODE_ENV` в значение `production`: некоторые библиотеки экосистемы ведут себя в зависимости от неё по-разному (например, включают или отключают отладочный вывод).

Установить переменную окружения `NODE_ENV` при запуске приложения можно так:

```bash
$ NODE_ENV=production node dist/main.js
```

Либо просто задайте её в панели управления вашего облачного провайдера или Mau.

#### Запуск приложения

Чтобы запустить приложение NestJS в продакшене, достаточно выполнить команду:

```bash
$ node dist/main.js # Скорректируйте путь в соответствии с расположением вашей точки входа
```

Эта команда запускает приложение, которое будет слушать указанный порт (обычно по умолчанию `3000`). Убедитесь, что он совпадает с портом, настроенным в вашем приложении.

Как вариант, можно использовать команду `nest start`. Она является обёрткой над `node dist/main.js`, но с одним важным отличием: перед запуском приложения она автоматически выполняет `nest build`, поэтому вручную запускать `npm run build` не требуется.

#### Проверки работоспособности

Проверки работоспособности (health checks) необходимы для отслеживания состояния вашего приложения NestJS в продакшене. Настроив эндпоинт проверки работоспособности, вы сможете регулярно убеждаться, что приложение работает как ожидается, и реагировать на проблемы до того, как они станут критическими.

В NestJS проверки работоспособности легко реализуются с помощью пакета **@nestjs/terminus**, который предоставляет мощный инструментарий для добавления проверок, включая подключения к базам данных, внешние сервисы и собственные проверки.

Как реализовать проверки работоспособности в приложении NestJS и обеспечить его постоянный мониторинг и отзывчивость, описано в [этом руководстве](<13. Recipes/11. Health checks.md>).

#### Логирование

Логирование необходимо любому приложению, готовому к продакшену. Оно помогает отслеживать ошибки, наблюдать за поведением и разбираться с проблемами. В NestJS вы можете использовать встроенный логгер или подключить внешние библиотеки, если вам нужны более продвинутые возможности.

Лучшие практики логирования:

- Логируйте ошибки, а не исключения: сосредоточьтесь на подробных сообщениях об ошибках — это ускорит отладку и решение проблем.
- Избегайте чувствительных данных: никогда не логируйте пароли, токены и другую чувствительную информацию.
- Используйте корреляционные идентификаторы: в распределённых системах добавляйте в логи уникальные идентификаторы (например, correlation ID), чтобы прослеживать запросы между сервисами.
- Используйте уровни логирования: разделяйте логи по важности (например, `info`, `warn`, `error`) и отключайте отладочные и подробные логи в продакшене, чтобы уменьшить шум.

> **Подсказка** Если вы используете [AWS](https://aws.amazon.com/) (через [Mau](https://mau.nestjs.com/ 'Deploy Nest') или напрямую), рассмотрите логирование в формате JSON — так логи проще разбирать и анализировать.

Для распределённых приложений крайне полезны централизованные сервисы логирования вроде ElasticSearch, Loggly или Datadog. Такие инструменты предлагают мощные возможности агрегации, поиска и визуализации логов, упрощая наблюдение за производительностью и поведением приложения.

#### Вертикальное и горизонтальное масштабирование

Эффективное масштабирование приложения NestJS критично для обработки возросшего трафика и поддержания оптимальной производительности. Существуют две основные стратегии: **вертикальное масштабирование** и **горизонтальное масштабирование**. Понимание этих подходов поможет спроектировать приложение так, чтобы оно эффективно справлялось с нагрузкой.

**Вертикальное масштабирование**, часто называемое «scaling up», подразумевает увеличение ресурсов одного сервера для повышения его производительности. Это может означать добавление CPU, оперативной памяти или дискового пространства к имеющейся машине. Ключевые моменты:

- Простота: вертикальное масштабирование обычно проще в реализации, поскольку нужно лишь обновить существующий сервер, а не управлять несколькими экземплярами.
- Ограничения: возможности масштабирования одной машины физически ограничены. Достигнув предела, вам придётся рассмотреть другие варианты.
- Экономичность: для приложений с умеренным трафиком вертикальное масштабирование может быть выгодным, так как не требует дополнительной инфраструктуры.

Пример: если ваше приложение NestJS размещено на виртуальной машине и вы замечаете, что в часы пик оно работает медленно, вы можете перевести ВМ на более крупный тип инстанса с большим объёмом ресурсов. Для этого достаточно зайти в панель управления вашего текущего провайдера и выбрать более мощный тип инстанса.

**Горизонтальное масштабирование**, или «scaling out», подразумевает добавление серверов или экземпляров для распределения нагрузки. Эта стратегия широко применяется в облачных средах и незаменима для приложений с высоким трафиком. Её преимущества и особенности:

- Рост ёмкости: добавляя новые экземпляры приложения, вы можете обслуживать больше одновременных пользователей без потери производительности.
- Резервирование: горизонтальное масштабирование даёт избыточность — отказ одного сервера не выводит из строя всё приложение. Трафик перераспределяется между оставшимися серверами.
- Балансировка нагрузки: чтобы эффективно управлять несколькими экземплярами, используйте балансировщики нагрузки (например, Nginx или AWS Elastic Load Balancing) для равномерного распределения входящего трафика между серверами.

Пример: при высоком трафике вы можете развернуть несколько экземпляров приложения NestJS в облачной среде и направлять запросы через балансировщик нагрузки, чтобы ни один экземпляр не стал узким местом.

Этот процесс упрощают технологии контейнеризации вроде [Docker](https://www.docker.com/) и платформы оркестрации контейнеров, такие как [Kubernetes](https://kubernetes.io/). Кроме того, вы можете задействовать облачные балансировщики нагрузки, например [AWS Elastic Load Balancing](https://aws.amazon.com/elasticloadbalancing/) или [Azure Load Balancer](https://azure.microsoft.com/en-us/services/load-balancer/), для распределения трафика между экземплярами приложения.

> **Подсказка** [Mau](https://mau.nestjs.com/ 'Deploy Nest') имеет встроенную поддержку горизонтального масштабирования на AWS, позволяя легко разворачивать несколько экземпляров приложения NestJS и управлять ими буквально в пару кликов.

#### Несколько дополнительных советов

При развёртывании приложения NestJS стоит помнить ещё о нескольких вещах:

- **Безопасность**: убедитесь, что ваше приложение защищено от распространённых угроз, таких как SQL-инъекции, XSS и т. д. Подробности — в категории «Безопасность».
- **Мониторинг**: используйте инструменты мониторинга вроде [Prometheus](https://prometheus.io/) или [New Relic](https://newrelic.com/) для отслеживания производительности и состояния приложения. Если вы используете облачного провайдера или Mau, у них могут быть встроенные сервисы мониторинга (например, [AWS CloudWatch](https://aws.amazon.com/cloudwatch/)).
- **Не зашивайте переменные окружения в код**: избегайте хардкода чувствительных данных — API-ключей, паролей, токенов. Используйте переменные окружения или менеджер секретов для безопасного хранения и получения этих значений.
- **Резервные копии**: регулярно создавайте резервные копии данных, чтобы не потерять их в случае инцидента.
- **Автоматизируйте развёртывание**: используйте CI/CD-конвейеры, чтобы автоматизировать процесс развёртывания и обеспечить единообразие окружений.
- **Ограничение частоты запросов**: внедрите ограничение частоты запросов, чтобы предотвратить злоупотребления и защитить приложение от DDoS-атак. Подробности — в [главе об ограничении частоты запросов](<05. Security/07. Rate limiting.md>); для продвинутой защиты можно использовать сервис вроде [AWS WAF](https://aws.amazon.com/waf/).

#### Упаковка приложения в Docker

[Docker](https://www.docker.com/) — это платформа, использующая контейнеризацию, чтобы разработчики могли упаковывать приложения вместе с их зависимостями в стандартизированную единицу — контейнер. Контейнеры легковесны, переносимы и изолированы, что делает их идеальными для развёртывания приложений в самых разных средах — от локальной разработки до продакшена.

Преимущества упаковки приложения NestJS в Docker:

- Единообразие: Docker гарантирует, что приложение работает одинаково на любой машине, устраняя проблему «у меня на компьютере работает».
- Изоляция: каждый контейнер выполняется в собственной изолированной среде, что предотвращает конфликты зависимостей.
- Масштабируемость: с Docker легко масштабировать приложение, запуская несколько контейнеров на разных машинах или облачных инстансах.
- Переносимость: контейнеры легко перемещать между средами, что упрощает развёртывание приложения на разных платформах.

Чтобы установить Docker, следуйте инструкциям на [официальном сайте](https://www.docker.com/get-started). После установки вы можете создать в своём проекте NestJS файл `Dockerfile`, описывающий шаги сборки образа контейнера.

`Dockerfile` — это текстовый файл с инструкциями, по которым Docker собирает образ контейнера.

Вот пример Dockerfile для приложения NestJS:

```bash
# Используем официальный образ Node.js как базовый
FROM node:20

# Задаём рабочий каталог внутри контейнера
WORKDIR /usr/src/app

# Копируем package.json и package-lock.json в рабочий каталог
COPY package*.json ./

# Устанавливаем зависимости приложения
RUN npm install

# Копируем остальные файлы приложения
COPY . .

# Собираем приложение NestJS
RUN npm run build

# Открываем порт приложения
EXPOSE 3000

# Команда запуска приложения
CMD ["node", "dist/main"]
```

> **Подсказка** Не забудьте заменить `node:20` на подходящую версию Node.js, которую вы используете в проекте. Доступные образы Node.js для Docker можно найти в [официальном репозитории Docker Hub](https://hub.docker.com/_/node).

Это базовый Dockerfile: он настраивает окружение Node.js, устанавливает зависимости, собирает приложение NestJS и запускает его. Вы можете доработать этот файл под требования своего проекта (использовать другие базовые образы, оптимизировать процесс сборки, устанавливать только продакшен-зависимости и т. д.).

Создадим также файл `.dockerignore`, чтобы указать, какие файлы и каталоги Docker должен игнорировать при сборке образа. Создайте `.dockerignore` в корне проекта:

```bash
node_modules
dist
*.log
*.md
.git
```

Этот файл гарантирует, что в образ контейнера не попадут лишние файлы, и образ останется лёгким. Теперь, когда Dockerfile готов, вы можете собрать образ Docker. Откройте терминал, перейдите в каталог проекта и выполните следующую команду:

```bash
docker build -t my-nestjs-app .
```

В этой команде:

- `-t my-nestjs-app`: помечает образ именем `my-nestjs-app`.
- `.`: указывает текущий каталог как контекст сборки.

После сборки образа его можно запустить как контейнер. Выполните следующую команду:

```bash
docker run -p 3000:3000 my-nestjs-app
```

В этой команде:

- `-p 3000:3000`: пробрасывает порт 3000 хост-машины на порт 3000 внутри контейнера.
- `my-nestjs-app`: указывает, какой образ запускать.

Теперь ваше приложение NestJS должно работать внутри контейнера Docker.

Если вы хотите развернуть образ Docker у облачного провайдера или поделиться им с другими, его нужно отправить в реестр Docker (например, [Docker Hub](https://hub.docker.com/), [AWS ECR](https://aws.amazon.com/ecr/) или [Google Container Registry](https://cloud.google.com/container-registry)).

Определившись с реестром, вы можете отправить образ, выполнив следующие шаги:

```bash
docker login # Вход в реестр Docker
docker tag my-nestjs-app your-dockerhub-username/my-nestjs-app # Пометка образа тегом
docker push your-dockerhub-username/my-nestjs-app # Отправка образа
```

Замените `your-dockerhub-username` на своё имя пользователя Docker Hub или на соответствующий URL реестра. После отправки образа вы можете скачать его на любой машине и запустить как контейнер.

Облачные провайдеры вроде AWS, Azure и Google Cloud предлагают управляемые сервисы для контейнеров, упрощающие развёртывание контейнеров и управление ими в больших масштабах. Такие сервисы предоставляют автомасштабирование, балансировку нагрузки и мониторинг, облегчая эксплуатацию приложения NestJS в продакшене.

Mau создан простым и интуитивно понятным, чтобы вы могли сосредоточиться на создании приложений и не думать о нижележащей инфраструктуре. «Под капотом» мы используем **Amazon Web Services**, чтобы дать вам мощную и надёжную платформу, скрывая при этом всю сложность AWS. Всю тяжёлую работу мы берём на себя, а вы можете сосредоточиться на разработке приложений и развитии бизнеса.

[Mau](https://mau.nestjs.com/ 'Deploy Nest') отлично подходит стартапам, малому и среднему бизнесу, крупным компаниям и разработчикам, которые хотят быстро всё запустить, не тратя много времени на изучение инфраструктуры и управление ею. Пользоваться им невероятно просто, а инфраструктуру можно поднять за считаные минуты. Кроме того, за кулисами используется AWS, так что вы получаете все преимущества AWS без хлопот, связанных с его сложностью.

<figure><img src="https://docs.nestjs.com/assets/mau-metrics.png" /></figure>

С [Mau](https://mau.nestjs.com/ 'Deploy Nest') вы можете:

- Развёртывать приложения NestJS в несколько кликов (API, микросервисы и т. д.).
- Разворачивать **базы данных**, такие как:
  - PostgreSQL
  - MySQL
  - MongoDB (DocumentDB)
  - Redis
  - и другие
- Настраивать брокеры сообщений:
  - RabbitMQ
  - Kafka
  - NATS
- Разворачивать задачи по расписанию (**CRON-задания**) и фоновые воркеры.
- Разворачивать lambda-функции и serverless-приложения.
- Настраивать **CI/CD-конвейеры** для автоматического развёртывания.
- И многое другое!

Чтобы развернуть приложение NestJS с помощью Mau, просто выполните следующую команду:

```bash
$ npm install -g @nestjs/mau
$ mau deploy
```

Регистрируйтесь и [разворачивайте с Mau](https://mau.nestjs.com/ 'Deploy Nest') — ваши приложения NestJS заработают на AWS за считаные минуты!
