### Пользовательские провайдеры

В предыдущих главах мы затрагивали разные аспекты **внедрения зависимостей (DI)** и его применения в Nest. Один из примеров — внедрение зависимостей [через конструктор](<../02. Overview/03. Providers.md#внедрение-зависимостей>), с помощью которого экземпляры (чаще всего провайдеры-сервисы) попадают в классы. Вряд ли вас удивит, что внедрение зависимостей заложено в ядро Nest на самом фундаментальном уровне. Пока мы рассмотрели лишь один основной приём. По мере усложнения приложения вам может понадобиться весь арсенал системы DI, поэтому разберём её подробнее.

#### Основы DI

Внедрение зависимостей — это приём [инверсии управления (IoC)](https://en.wikipedia.org/wiki/Inversion_of_control), при котором создание зависимостей вы поручаете IoC-контейнеру (в нашем случае — среде выполнения NestJS), вместо того чтобы делать это императивно в собственном коде. Разберём, что происходит в этом примере из [главы о провайдерах](<../02. Overview/03. Providers.md>).

Сначала мы объявляем провайдер. Декоратор `@Injectable()` помечает класс `CatsService` как провайдер.

**`cats.service.ts`**

```typescript
import { Injectable } from '@nestjs/common';
import { Cat } from './interfaces/cat.interface';

@Injectable()
export class CatsService {
  private readonly cats: Cat[] = [];

  findAll(): Cat[] {
    return this.cats;
  }
}
```

Затем мы просим Nest внедрить этот провайдер в класс нашего контроллера:

**`cats.controller.ts`**

```typescript
import { Controller, Get } from '@nestjs/common';
import { CatsService } from './cats.service';
import { Cat } from './interfaces/cat.interface';

@Controller('cats')
export class CatsController {
  constructor(private catsService: CatsService) {}

  @Get()
  async findAll(): Promise<Cat[]> {
    return this.catsService.findAll();
  }
}
```

Наконец, мы регистрируем провайдер в IoC-контейнере Nest:

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { CatsController } from './cats/cats.controller';
import { CatsService } from './cats/cats.service';

@Module({
  controllers: [CatsController],
  providers: [CatsService],
})
export class AppModule {}
```

Что именно происходит «под капотом», чтобы это заработало? В процессе есть три ключевых шага:

1. В `cats.service.ts` декоратор `@Injectable()` объявляет класс `CatsService` классом, которым может управлять IoC-контейнер Nest.
2. В `cats.controller.ts` класс `CatsController` объявляет зависимость от токена `CatsService` через внедрение в конструктор:

```typescript
constructor(private catsService: CatsService)
```

3. В `app.module.ts` мы связываем токен `CatsService` с классом `CatsService` из файла `cats.service.ts`. Как именно происходит это связывание (его также называют _регистрацией_), мы <a href="01.%20Custom%20providers.md#стандартные-провайдеры">разберём ниже</a>.

Когда IoC-контейнер Nest создаёт экземпляр `CatsController`, он сначала ищет его зависимости\*. Обнаружив зависимость `CatsService`, он выполняет поиск по токену `CatsService`, который возвращает класс `CatsService` — согласно шагу регистрации (пункт 3 выше). Если предполагается область `SINGLETON` (поведение по умолчанию), Nest либо создаст экземпляр `CatsService`, закэширует его и вернёт, либо вернёт уже имеющийся в кэше экземпляр.

\*Это объяснение несколько упрощено, чтобы передать суть. Один важный момент мы обошли стороной: анализ кода на предмет зависимостей — процесс весьма сложный, и происходит он во время начальной загрузки приложения. Ключевая его особенность в том, что анализ зависимостей (или «построение графа зависимостей») **транзитивен**. В примере выше, если бы у самого `CatsService` были зависимости, они тоже были бы разрешены. Граф зависимостей гарантирует, что зависимости разрешаются в правильном порядке — по сути, «снизу вверх». Благодаря этому механизму разработчику не приходится самому управлять такими сложными графами.

#### Стандартные провайдеры

Присмотримся к декоратору `@Module()`. В `app.module` мы объявляем:

```typescript
@Module({
  controllers: [CatsController],
  providers: [CatsService],
})
```

Свойство `providers` принимает массив провайдеров. До сих пор мы передавали их простым списком имён классов. На самом деле запись `providers: [CatsService]` — это сокращение более полного синтаксиса:

```typescript
providers: [
  {
    provide: CatsService,
    useClass: CatsService,
  },
];
```

Теперь, увидев эту явную конструкцию, мы можем понять процесс регистрации. Здесь мы прямо связываем токен `CatsService` с классом `CatsService`. Сокращённая запись — не более чем удобство для самого частого случая, когда токен используется, чтобы запросить экземпляр одноимённого класса.

#### Пользовательские провайдеры

Что делать, когда ваши задачи выходят за рамки того, что дают _стандартные провайдеры_? Вот несколько примеров:

- Вы хотите создать собственный экземпляр вместо того, чтобы Nest создавал экземпляр класса (или возвращал закэшированный).
- Вы хотите переиспользовать существующий класс во второй зависимости.
- Вы хотите подменить класс mock-версией для тестирования.

Для таких случаев Nest позволяет определять пользовательские провайдеры. Есть несколько способов их описать — рассмотрим их по очереди.

> **Подсказка** Если у вас возникают трудности с разрешением зависимостей, задайте переменную окружения `NEST_DEBUG` — при запуске вы получите дополнительные логи разрешения зависимостей.

#### Провайдеры-значения: `useValue`

Синтаксис `useValue` удобен, когда нужно внедрить константное значение, поместить внешнюю библиотеку в контейнер Nest или заменить реальную реализацию mock-объектом. Допустим, вы хотите заставить Nest использовать mock-версию `CatsService` для тестирования.

```typescript
import { CatsService } from './cats.service';

const mockCatsService = {
  /* реализация-заглушка
  ...
  */
};

@Module({
  imports: [CatsModule],
  providers: [
    {
      provide: CatsService,
      useValue: mockCatsService,
    },
  ],
})
export class AppModule {}
```

В этом примере токен `CatsService` будет разрешаться в mock-объект `mockCatsService`. `useValue` требует значения — здесь это объектный литерал с тем же интерфейсом, что и у заменяемого класса `CatsService`. Благодаря [структурной типизации](https://www.typescriptlang.org/docs/handbook/type-compatibility.html) в TypeScript вы можете использовать любой объект с совместимым интерфейсом, включая объектный литерал или экземпляр класса, созданный через `new`.

#### Токены провайдеров, не основанные на классах

До сих пор в качестве токенов провайдеров мы использовали имена классов (значение свойства `provide` у провайдера в массиве `providers`). Это соответствует стандартному подходу [внедрения через конструктор](<../02. Overview/03. Providers.md#внедрение-зависимостей>), где токеном тоже выступает имя класса. (Если понятие токена всё ещё не до конца ясно, вернитесь к разделу <a href="01.%20Custom%20providers.md#основы-di">Основы DI</a>.) Иногда хочется большей гибкости — использовать в качестве DI-токена строки или символы. Например:

```typescript
import { connection } from './connection';

@Module({
  providers: [
    {
      provide: 'CONNECTION',
      useValue: connection,
    },
  ],
})
export class AppModule {}
```

В этом примере мы связываем строковый токен (`'CONNECTION'`) с уже существующим объектом `connection`, импортированным из внешнего файла.

> **Важно** Помимо строк, в качестве значений токенов можно использовать [символы](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Symbol) JavaScript или [перечисления](https://www.typescriptlang.org/docs/handbook/enums.html) TypeScript.

Ранее мы видели, как внедрить провайдер стандартным способом — [через конструктор](<../02. Overview/03. Providers.md#внедрение-зависимостей>). Этот подход **требует**, чтобы зависимость была объявлена с именем класса. Пользовательский провайдер `'CONNECTION'` использует строковый токен. Посмотрим, как внедрить такой провайдер. Для этого применяется декоратор `@Inject()`. Он принимает единственный аргумент — токен.

```typescript
@Injectable()
export class CatsRepository {
  constructor(@Inject('CONNECTION') connection: Connection) {}
}
```

> **Подсказка** Декоратор `@Inject()` импортируется из пакета `@nestjs/common`.

В примерах выше мы для наглядности используем строку `'CONNECTION'` напрямую, но ради чистоты кода лучше выносить токены в отдельный файл, например `constants.ts`. Относитесь к ним так же, как к символам или перечислениям, которые объявляются в собственном файле и импортируются там, где нужны.

#### Интерфейсы и абстрактные классы

Типы и интерфейсы TypeScript стираются при компиляции, поэтому Nest не может обратиться к ним во время выполнения. Это значит, что интерфейс способен описать форму зависимости, но сам по себе не может служить DI-токеном.

Поскольку Nest разрешает провайдеры по токенам времени выполнения, при регистрации провайдера для интерфейса используйте строковый токен или `Symbol`:

```typescript
export interface LoggerService {
  log(message: string): void;
}

export const LOGGER_SERVICE = Symbol('LOGGER_SERVICE');

@Injectable()
export class PinoLoggerService implements LoggerService {
  log(message: string) {
    // детали реализации
  }
}

@Module({
  providers: [
    {
      provide: LOGGER_SERVICE,
      useClass: PinoLoggerService,
    },
  ],
})
export class AppModule {}
```

Чтобы внедрить такой провайдер, передайте этот токен декоратору `@Inject()`:

```typescript
@Injectable()
export class CatsService {
  constructor(
    @Inject(LOGGER_SERVICE)
    private readonly logger: LoggerService,
  ) {}
}
```

Абстрактные классы, в отличие от интерфейсов, существуют во время выполнения. Абстрактный класс можно использовать одновременно и как контракт TypeScript, и как DI-токен:

```typescript
export abstract class LoggerService {
  abstract log(message: string): void;
}

@Injectable()
export class PinoLoggerService implements LoggerService {
  log(message: string) {
    // детали реализации
  }
}

@Module({
  providers: [
    {
      provide: LoggerService,
      useClass: PinoLoggerService,
    },
  ],
})
export class AppModule {}
```

Когда токеном служит абстрактный класс, при внедрении через конструктор можно напрямую указать его тип — декоратор `@Inject()` не нужен:

```typescript
@Injectable()
export class CatsService {
  constructor(private readonly logger: LoggerService) {}
}
```

Используйте строковые токены или `Symbol`, когда DI-токен времени выполнения должен быть отвязан от конкретного класса. Символьные токены особенно полезны в библиотеках и крупных приложениях: каждый символ имеет уникальную идентичность во время выполнения, что помогает избежать случайных коллизий, возможных, когда никак не связанные провайдеры используют один и тот же строковый токен. Работая с символьным токеном, экспортируйте его из общего файла и переиспользуйте один и тот же экземпляр символа везде, где провайдер регистрируется и внедряется. Абстрактные классы подойдут, если один и тот же артефакт должен быть и контрактом, и токеном времени выполнения, а вам удобнее более простое внедрение через конструктор. Обычный интерфейс по-прежнему хорош, когда тип нужен только для проверок на этапе компиляции и DI-токен не требуется.

#### Провайдеры-классы: `useClass`

Синтаксис `useClass` позволяет динамически определять класс, в который будет разрешаться токен. Допустим, у нас есть абстрактный (или предлагаемый по умолчанию) класс `ConfigService`. В зависимости от текущего окружения мы хотим, чтобы Nest предоставлял разные реализации сервиса конфигурации. Такую стратегию реализует следующий код.

```typescript
const configServiceProvider = {
  provide: ConfigService,
  useClass:
    process.env.NODE_ENV === 'development'
      ? DevelopmentConfigService
      : ProductionConfigService,
};

@Module({
  providers: [configServiceProvider],
})
export class AppModule {}
```

Обратим внимание на пару деталей в этом примере. Заметьте, что сначала мы определяем `configServiceProvider` объектным литералом, а затем передаём его в свойство `providers` декоратора модуля. Это лишь вопрос организации кода: функционально всё равнозначно примерам, которые мы использовали в этой главе ранее.

Кроме того, токеном здесь служит имя класса `ConfigService`. В любой класс, зависящий от `ConfigService`, Nest внедрит экземпляр указанного класса (`DevelopmentConfigService` или `ProductionConfigService`), переопределив реализацию по умолчанию, которая могла быть объявлена где-то ещё (например, `ConfigService` с декоратором `@Injectable()`).

#### Провайдеры-фабрики: `useFactory`

Синтаксис `useFactory` позволяет создавать провайдеры **динамически**. Фактическим провайдером станет значение, возвращённое фабричной функцией. Сама фабрика может быть сколь угодно простой или сложной. Простая фабрика может не зависеть ни от каких других провайдеров. Более сложная может сама внедрять другие провайдеры, необходимые ей для вычисления результата. Для второго случая в синтаксисе провайдера-фабрики есть пара связанных механизмов:

1. Фабричная функция может принимать (необязательные) аргументы.
2. (Необязательное) свойство `inject` принимает массив провайдеров, которые Nest разрешит и передаст фабричной функции в качестве аргументов при создании экземпляра. Эти провайдеры также можно пометить как необязательные. Два списка должны соответствовать друг другу: Nest передаёт экземпляры из списка `inject` в фабричную функцию в том же порядке. Пример ниже это демонстрирует.

```typescript
const connectionProvider = {
  provide: 'CONNECTION',
  useFactory: (optionsProvider: MyOptionsProvider, optionalProvider?: string) => {
    const options = optionsProvider.get();
    return new DatabaseConnection(options);
  },
  inject: [MyOptionsProvider, { token: 'SomeOptionalProvider', optional: true }],
  //       \______________/             \__________________/
  //        Этот провайдер               Провайдер с этим токеном
  //        обязателен.                  может разрешиться в `undefined`.
};

@Module({
  providers: [
    connectionProvider,
    MyOptionsProvider, // провайдер на основе класса
    // { provide: 'SomeOptionalProvider', useValue: 'anything' },
  ],
})
export class AppModule {}
```

#### Провайдеры-псевдонимы: `useExisting`

Синтаксис `useExisting` позволяет создавать псевдонимы для существующих провайдеров. Так появляется два способа обратиться к одному и тому же провайдеру. В примере ниже (строковый) токен `'AliasedLoggerService'` — псевдоним для (классового) токена `LoggerService`. Допустим, у нас есть две разные зависимости: одна на `'AliasedLoggerService'`, другая на `LoggerService`. Если обе объявлены с областью `SINGLETON`, они разрешатся в один и тот же экземпляр.

```typescript
@Injectable()
class LoggerService {
  /* детали реализации */
}

const loggerAliasProvider = {
  provide: 'AliasedLoggerService',
  useExisting: LoggerService,
};

@Module({
  providers: [LoggerService, loggerAliasProvider],
})
export class AppModule {}
```

#### Провайдеры, не являющиеся сервисами

Хотя провайдеры чаще всего поставляют сервисы, этим их применение не ограничивается. Провайдер может предоставить **любое** значение. Например, он может выдавать массив объектов конфигурации в зависимости от текущего окружения:

```typescript
const configFactory = {
  provide: 'CONFIG',
  useFactory: () => {
    return process.env.NODE_ENV === 'development' ? devConfig : prodConfig;
  },
};

@Module({
  providers: [configFactory],
})
export class AppModule {}
```

#### Экспорт пользовательского провайдера

Как и любой провайдер, пользовательский провайдер доступен только в объявившем его модуле. Чтобы он стал виден другим модулям, его нужно экспортировать. Для этого можно указать либо его токен, либо объект провайдера целиком.

Следующий пример показывает экспорт по токену:

```typescript
const connectionFactory = {
  provide: 'CONNECTION',
  useFactory: (optionsProvider: OptionsProvider) => {
    const options = optionsProvider.get();
    return new DatabaseConnection(options);
  },
  inject: [OptionsProvider],
};

@Module({
  providers: [connectionFactory],
  exports: ['CONNECTION'],
})
export class AppModule {}
```

Либо экспортируйте объект провайдера целиком:

```typescript
const connectionFactory = {
  provide: 'CONNECTION',
  useFactory: (optionsProvider: OptionsProvider) => {
    const options = optionsProvider.get();
    return new DatabaseConnection(options);
  },
  inject: [OptionsProvider],
};

@Module({
  providers: [connectionFactory],
  exports: [connectionFactory],
})
export class AppModule {}
```
