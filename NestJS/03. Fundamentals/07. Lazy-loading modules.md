### Ленивая загрузка модулей

По умолчанию модули загружаются «жадно»: как только загружается приложение, загружаются и все модули, независимо от того, нужны они прямо сейчас или нет. Для большинства приложений это нормально, но может стать узким местом для приложений и воркеров, работающих в **serverless-среде**, где задержка запуска («холодный старт») критична.

Ленивая загрузка помогает сократить время начальной загрузки, подгружая только те модули, которые нужны конкретному вызову serverless-функции. Кроме того, вы можете асинхронно догружать остальные модули после того, как функция «прогрелась», чтобы ещё сильнее ускорить последующие вызовы (отложенная регистрация модулей).

> **Подсказка** Если вы знакомы с фреймворком **[Angular](https://angular.dev/)**, вам, возможно, уже встречался термин «[ленивая загрузка модулей](https://angular.dev/guide/ngmodules/lazy-loading#lazy-loading-basics)». Имейте в виду, что в Nest этот приём **работает иначе**, так что воспринимайте его как совершенно отдельную возможность со схожим названием.

> **Предупреждение** Обратите внимание, что [методы-хуки жизненного цикла](<09. Lifecycle events.md>) не вызываются в лениво загружаемых модулях и сервисах.

#### Начало работы

Для загрузки модулей по требованию Nest предоставляет класс `LazyModuleLoader`, который можно внедрить в класс обычным способом:

**`cats.service.ts`**

```typescript
@Injectable()
export class CatsService {
  constructor(private lazyModuleLoader: LazyModuleLoader) {}
}
```

> **Подсказка** Класс `LazyModuleLoader` импортируется из пакета `@nestjs/core`.

Как вариант, ссылку на провайдер `LazyModuleLoader` можно получить прямо в файле начальной загрузки приложения (`main.ts`):

```typescript
// "app" — экземпляр Nest-приложения
const lazyModuleLoader = app.get(LazyModuleLoader);
```

После этого любой модуль можно загрузить такой конструкцией:

```typescript
const { LazyModule } = await import('./lazy.module');
const moduleRef = await this.lazyModuleLoader.load(() => LazyModule);
```

> **Подсказка** Лениво загружаемые модули **кэшируются** при первом вызове метода `LazyModuleLoader#load`. Это значит, что каждая последующая попытка загрузить `LazyModule` будет **очень быстрой** и вернёт закэшированный экземпляр вместо повторной загрузки модуля.
>
> ```bash
> Load "LazyModule" attempt: 1
> time: 2.379ms
> Load "LazyModule" attempt: 2
> time: 0.294ms
> Load "LazyModule" attempt: 3
> time: 0.303ms
> ```
>
> Кроме того, лениво загружаемые модули используют тот же граф модулей, что и модули, загруженные «жадно» при начальной загрузке приложения, а также любые другие ленивые модули, зарегистрированные позже.

Здесь `lazy.module.ts` — это файл TypeScript, экспортирующий **обычный модуль Nest** (никаких дополнительных изменений не требуется).

Метод `LazyModuleLoader#load` возвращает [ссылку на модуль](<06. Module reference.md>) (`LazyModule`), которая позволяет пройтись по внутреннему списку провайдеров и получить ссылку на любой из них по его токену внедрения.

Например, допустим, у нас есть `LazyModule` со следующим определением:

```typescript
@Module({
  providers: [LazyService],
  exports: [LazyService],
})
export class LazyModule {}
```

> **Подсказка** Лениво загружаемые модули нельзя зарегистрировать как **глобальные модули** — в этом просто нет смысла (ведь они регистрируются лениво, по требованию, когда все статически зарегистрированные модули уже созданы). Точно так же не будут корректно работать и зарегистрированные **глобальные усилители** (guard-ы, interceptor-ы и т. п.).

Теперь мы можем получить ссылку на провайдер `LazyService`:

```typescript
const { LazyModule } = await import('./lazy.module');
const moduleRef = await this.lazyModuleLoader.load(() => LazyModule);

const { LazyService } = await import('./lazy.service');
const lazyService = moduleRef.get(LazyService);
```

> **Предупреждение** Если вы используете **Webpack**, не забудьте обновить файл `tsconfig.json`: задайте `compilerOptions.module` значение `"esnext"` и добавьте свойство `compilerOptions.moduleResolution` со значением `"node"`:
>
> ```json
> {
>   "compilerOptions": {
>     "module": "esnext",
>     "moduleResolution": "node",
>     ...
>   }
> }
> ```
>
> С этими настройками вы сможете воспользоваться возможностью [разделения кода](https://webpack.js.org/guides/code-splitting/).

#### Ленивая загрузка контроллеров, шлюзов и резолверов

Поскольку контроллеры (или резолверы в GraphQL-приложениях) в Nest представляют наборы маршрутов, путей и топиков (или запросов и мутаций), **лениво загружать их** с помощью класса `LazyModuleLoader` нельзя.

> **Предупреждение** Контроллеры, [резолверы](<../06. GraphQL/02. Resolvers.md>) и [шлюзы](<../07. WebSockets/01. Gateways.md>), зарегистрированные внутри лениво загружаемых модулей, будут вести себя не так, как ожидается. Точно так же нельзя по требованию регистрировать middleware-функции (через реализацию интерфейса `MiddlewareConsumer`).

Допустим, вы создаёте REST API (HTTP-приложение) на драйвере Fastify «под капотом» (с пакетом `@nestjs/platform-fastify`). Fastify не позволяет регистрировать маршруты после того, как приложение готово и успешно принимает сообщения. Значит, даже если бы мы проанализировали сопоставления маршрутов, зарегистрированные в контроллерах модуля, все лениво загруженные маршруты остались бы недоступны, поскольку зарегистрировать их во время выполнения невозможно.

Аналогично, некоторые транспортные стратегии из пакета `@nestjs/microservices` (включая Kafka, gRPC и RabbitMQ) требуют подписки на конкретные топики или каналы до установления соединения. Как только приложение начинает принимать сообщения, фреймворк уже не сможет подписаться на новые топики.

Наконец, пакет `@nestjs/graphql` с включённым подходом code first автоматически генерирует схему GraphQL «на лету» на основе метаданных. Значит, все классы должны быть загружены заранее — иначе построить корректную схему не получится.

#### Типичные сценарии использования

Чаще всего лениво загружаемые модули встречаются там, где ваш воркер, задача по расписанию, lambda- или serverless-функция либо webhook должны запускать разные сервисы (разную логику) в зависимости от входных аргументов (пути маршрута, даты, параметров запроса и т. п.). С другой стороны, для монолитных приложений, где время запуска не так важно, ленивая загрузка модулей может не иметь особого смысла.
