### События жизненного цикла

Приложение Nest, как и каждый его элемент, имеет жизненный цикл, которым управляет Nest. Nest предоставляет **хуки жизненного цикла**, которые дают доступ к ключевым событиям жизненного цикла и возможность выполнить действие (запустить зарегистрированный код в ваших модулях, провайдерах или контроллерах) в момент их наступления.

#### Последовательность жизненного цикла

На схеме ниже показана последовательность ключевых событий жизненного цикла приложения — от начальной загрузки до завершения процесса Node. Весь жизненный цикл можно разделить на три фазы: **инициализация**, **работа** и **завершение**. Опираясь на этот жизненный цикл, вы можете спланировать правильную инициализацию модулей и сервисов, управлять активными соединениями и корректно завершать работу приложения при получении сигнала на остановку.

#### События жизненного цикла

События жизненного цикла происходят во время начальной загрузки и завершения работы приложения. Nest вызывает зарегистрированные методы-хуки жизненного цикла у модулей, провайдеров и контроллеров при каждом из следующих событий (**хуки завершения работы** предварительно нужно включить, как описано [ниже](<09. Lifecycle events.md#завершение-работы-приложения>)). Как показано на схеме выше, Nest также вызывает соответствующие низкоуровневые методы, чтобы начать и прекратить приём соединений.

В таблице ниже `onModuleInit` и `onApplicationBootstrap` срабатывают только в том случае, если вы явно вызываете `app.init()` или `app.listen()`.

В таблице ниже `onModuleDestroy`, `beforeApplicationShutdown` и `onApplicationShutdown` срабатывают только в том случае, если вы явно вызываете `app.close()` либо если процесс получает специальный системный сигнал (например, SIGTERM) и вы корректно вызвали `enableShutdownHooks` при начальной загрузке приложения (см. ниже раздел **Завершение работы приложения**).

| Метод-хук жизненного цикла      | Событие жизненного цикла, вызывающее этот метод                                                                                                                                                                   |
| ------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `onModuleInit()`                | Вызывается после того, как разрешены зависимости модуля-владельца.                                                                                                                                                |
| `onApplicationBootstrap()`      | Вызывается после инициализации всех модулей, но до начала приёма соединений.                                                                                                                                      |
| `onModuleDestroy()`\*           | Вызывается после получения сигнала на завершение (например, `SIGTERM`).                                                                                                                                           |
| `beforeApplicationShutdown()`\* | Вызывается после завершения всех обработчиков `onModuleDestroy()` (Promise разрешены или отклонены);<br />по его завершении (Promise разрешены или отклонены) все существующие соединения будут закрыты (вызывается `app.close()`). |
| `onApplicationShutdown()`\*     | Вызывается после закрытия соединений (`app.close()` разрешается).                                                                                                                                                 |

\* Для этих событий, если вы не вызываете `app.close()` явно, нужно включить их явным образом, чтобы они работали с системными сигналами вроде `SIGTERM`. См. раздел [Завершение работы приложения](<09. Lifecycle events.md#завершение-работы-приложения>) ниже.

> **Предупреждение** Перечисленные выше хуки жизненного цикла не срабатывают для классов с **областью запроса** (request-scoped). Такие классы не привязаны к жизненному циклу приложения, и время их существования непредсказуемо. Они создаются исключительно для каждого запроса и автоматически удаляются сборщиком мусора после отправки ответа.

> **Подсказка** Порядок выполнения `onModuleInit()` и `onApplicationBootstrap()` напрямую зависит от порядка импорта модулей, с ожиданием предыдущего хука.

#### Использование

Каждый хук жизненного цикла представлен интерфейсом. Технически интерфейсы необязательны, поскольку после компиляции TypeScript их не существует. Тем не менее использовать их — хорошая практика: вы получаете строгую типизацию и подсказки редактора. Чтобы зарегистрировать хук жизненного цикла, реализуйте соответствующий интерфейс. Например, чтобы зарегистрировать метод, вызываемый при инициализации модуля в конкретном классе (контроллере, провайдере или модуле), реализуйте интерфейс `OnModuleInit`, добавив метод `onModuleInit()`, как показано ниже:

```typescript
import { Injectable, OnModuleInit } from '@nestjs/common';

@Injectable()
export class UsersService implements OnModuleInit {
  onModuleInit() {
    console.log(`The module has been initialized.`);
  }
}
```

#### Асинхронная инициализация

Хуки `OnModuleInit` и `OnApplicationBootstrap` позволяют отложить процесс инициализации приложения (вернуть `Promise` либо пометить метод как `async` и внутри него дождаться (`await`) завершения асинхронного вызова).

```typescript
async onModuleInit(): Promise<void> {
  await this.fetch();
}
```

#### Завершение работы приложения

Хуки `onModuleDestroy()`, `beforeApplicationShutdown()` и `onApplicationShutdown()` вызываются в фазе завершения (в ответ на явный вызов `app.close()` или при получении системных сигналов вроде SIGTERM, если это включено). Эта возможность часто используется вместе с [Kubernetes](https://kubernetes.io/) для управления жизненным циклом контейнеров, в [Heroku](https://www.heroku.com/) для dyno и в аналогичных сервисах.

Слушатели хуков завершения работы потребляют системные ресурсы, поэтому по умолчанию они отключены. Чтобы использовать эти хуки, вы **должны включить слушателей**, вызвав `enableShutdownHooks()`:

```typescript
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);

  // Начинает прослушивание хуков завершения работы
  app.enableShutdownHooks();

  await app.listen(process.env.PORT ?? 3000);
}
bootstrap();
```

> **Предупреждение** Из-за особенностей платформы поддержка хуков завершения работы приложения в NestJS на Windows ограничена. Можно рассчитывать, что заработает `SIGINT`, а также `SIGBREAK` и, отчасти, `SIGHUP` — [подробнее](https://nodejs.org/api/process.html#process_signal_events). Однако `SIGTERM` на Windows не заработает никогда, потому что завершение процесса в диспетчере задач безусловно, «то есть у приложения нет способа обнаружить или предотвратить его». Вот [соответствующая документация](https://docs.libuv.org/en/v1.x/signal.html) libuv, где можно узнать больше о том, как на Windows обрабатываются `SIGINT`, `SIGBREAK` и другие сигналы. См. также документацию Node.js о [событиях сигналов процесса](https://nodejs.org/api/process.html#process_signal_events)

> **Информация** `enableShutdownHooks` расходует память, запуская слушателей. Если вы запускаете несколько приложений Nest в одном процессе Node (например, при параллельном прогоне тестов в Jest), Node может пожаловаться на избыточное число слушателей. Именно поэтому `enableShutdownHooks` не включён по умолчанию. Учитывайте это, когда запускаете несколько экземпляров в одном процессе Node.

Получив сигнал на завершение, приложение вызовет все зарегистрированные методы `onModuleDestroy()`, `beforeApplicationShutdown()`, а затем `onApplicationShutdown()` (в описанной выше последовательности), передав соответствующий сигнал первым параметром. Если зарегистрированная функция ожидает асинхронный вызов (возвращает promise), Nest не продолжит последовательность, пока этот promise не будет разрешён или отклонён.

```typescript
@Injectable()
class UsersService implements OnApplicationShutdown {
  onApplicationShutdown(signal: string) {
    console.log(signal); // например, "SIGINT"
  }
}
```

> **Информация** Вызов `app.close()` не завершает процесс Node, а лишь запускает хуки `onModuleDestroy()` и `onApplicationShutdown()`. Поэтому, если есть интервалы, длительные фоновые задачи и т. п., процесс не завершится автоматически.
