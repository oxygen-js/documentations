### Тестирование

Автоматизированное тестирование считается неотъемлемой частью любой серьёзной разработки. Автоматизация позволяет быстро и просто повторять отдельные тесты или наборы тестов в ходе разработки. Это помогает добиться того, чтобы релизы соответствовали целям по качеству и производительности. Автоматизация увеличивает покрытие и ускоряет обратную связь для разработчиков. Она повышает продуктивность каждого разработчика и гарантирует запуск тестов в ключевых точках жизненного цикла разработки — при коммите в систему контроля версий, при интеграции функциональности и при выпуске версии.

Такие тесты бывают самых разных видов: модульные, сквозные (e2e), интеграционные и так далее. Польза от них несомненна, но их настройка бывает утомительной. Nest стремится продвигать лучшие практики разработки, включая эффективное тестирование, поэтому включает следующие возможности, помогающие разработчикам и командам создавать и автоматизировать тесты. Nest:

- автоматически генерирует каркас модульных тестов для компонентов и e2e-тестов для приложений
- предоставляет готовый инструментарий (например, тест-раннер, который создаёт изолированный загрузчик модуля/приложения)
- «из коробки» интегрируется с [Jest](https://github.com/facebook/jest) и [Supertest](https://github.com/visionmedia/supertest), оставаясь при этом независимым от конкретных инструментов тестирования
- делает систему внедрения зависимостей Nest доступной в тестовой среде, чтобы упростить создание моков компонентов

Как уже сказано, вы можете использовать любой **фреймворк тестирования**, поскольку Nest не навязывает конкретный инструментарий. Достаточно заменить нужные элементы (например, тест-раннер), и вы по-прежнему сможете пользоваться готовыми средствами тестирования Nest.

#### Установка

Для начала установите необходимый пакет:

```bash
$ npm i --save-dev @nestjs/testing
```

#### Модульное тестирование

В примере ниже мы тестируем два класса: `CatsController` и `CatsService`. Как уже упоминалось, в качестве фреймворка тестирования по умолчанию используется [Jest](https://github.com/facebook/jest). Он выступает тест-раннером, а также предоставляет функции проверок и утилиты для тестовых двойников, помогающие делать моки, шпионы и т. д. В следующем простом тесте мы вручную создаём экземпляры этих классов и убеждаемся, что контроллер и сервис соблюдают свой API-контракт.

**`cats.controller.spec.ts`**

```typescript
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

describe('CatsController', () => {
  let catsController: CatsController;
  let catsService: CatsService;

  beforeEach(() => {
    catsService = new CatsService();
    catsController = new CatsController(catsService);
  });

  describe('findAll', () => {
    it('should return an array of cats', async () => {
      const result = ['test'];
      jest.spyOn(catsService, 'findAll').mockImplementation(() => result);

      expect(await catsController.findAll()).toBe(result);
    });
  });
});
```

> **Подсказка** Держите файлы тестов рядом с классами, которые они тестируют. Файлы тестов должны иметь суффикс `.spec` или `.test`.

Пример выше тривиален, поэтому мы, по сути, не тестируем ничего специфичного для Nest. Более того, мы даже не используем внедрение зависимостей (обратите внимание, что мы передаём экземпляр `CatsService` в `catsController`). Такую форму тестирования, когда экземпляры тестируемых классов создаются вручную, часто называют **изолированным тестированием**, поскольку оно не зависит от фреймворка. Перейдём к более продвинутым возможностям, которые помогут тестировать приложения, активнее использующие возможности Nest.

#### Утилиты тестирования

Пакет `@nestjs/testing` предоставляет набор утилит, делающих процесс тестирования более надёжным. Перепишем предыдущий пример с помощью встроенного класса `Test`:

**`cats.controller.spec.ts`**

```typescript
import { Test } from '@nestjs/testing';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

describe('CatsController', () => {
  let catsController: CatsController;
  let catsService: CatsService;

  beforeEach(async () => {
    const moduleRef = await Test.createTestingModule({
        controllers: [CatsController],
        providers: [CatsService],
      }).compile();

    catsService = moduleRef.get(CatsService);
    catsController = moduleRef.get(CatsController);
  });

  describe('findAll', () => {
    it('should return an array of cats', async () => {
      const result = ['test'];
      jest.spyOn(catsService, 'findAll').mockImplementation(() => result);

      expect(await catsController.findAll()).toBe(result);
    });
  });
});
```

Класс `Test` полезен тем, что предоставляет контекст выполнения приложения, который по сути имитирует полноценную среду выполнения Nest, но при этом даёт вам точки подключения для удобного управления экземплярами классов, включая моки и переопределения. У класса `Test` есть метод `createTestingModule()`, принимающий объект метаданных модуля (тот же объект, который вы передаёте в декоратор `@Module()`). Этот метод возвращает экземпляр `TestingModule`, у которого, в свою очередь, есть несколько методов. Для модульных тестов важен метод `compile()`. Он выполняет начальную загрузку модуля вместе с его зависимостями (аналогично тому, как приложение загружается в обычном файле `main.ts` через `NestFactory.create()`) и возвращает модуль, готовый к тестированию.

> **Подсказка** Метод `compile()` **асинхронный**, поэтому его нужно дожидаться через await. После компиляции модуля вы можете получить любой объявленный в нём **статический** экземпляр (контроллеры и провайдеры) методом `get()`.

`TestingModule` наследуется от класса [module reference](<06. Module reference.md>), а значит, обладает его способностью динамически разрешать провайдеры с областью видимости (transient или request-scoped). Для этого используется метод `resolve()` (метод `get()` умеет получать только статические экземпляры).

```typescript
const moduleRef = await Test.createTestingModule({
  controllers: [CatsController],
  providers: [CatsService],
}).compile();

catsService = await moduleRef.resolve(CatsService);
```

> **Предупреждение** Метод `resolve()` возвращает уникальный экземпляр провайдера из собственного **поддерева DI-контейнера**. У каждого поддерева свой идентификатор контекста. Поэтому, если вы вызовете этот метод несколько раз и сравните ссылки на экземпляры, они окажутся разными.

> **Подсказка** Подробнее о возможностях module reference читайте [здесь](<06. Module reference.md>).

Вместо рабочей версии любого провайдера вы можете подставить [пользовательский провайдер](<01. Custom providers.md>) для целей тестирования. Например, можно замокать сервис базы данных вместо подключения к настоящей базе. Переопределения мы разберём в следующем разделе, но они доступны и в модульных тестах.

#### Автоматическое создание моков

Nest также позволяет задать фабрику моков, применяемую ко всем отсутствующим зависимостям. Это удобно, когда в классе много зависимостей и мокать каждую заняло бы много времени и настроек. Чтобы воспользоваться этой возможностью, к `createTestingModule()` нужно добавить в цепочку метод `useMocker()`, передав ему фабрику моков зависимостей. Эта фабрика может принимать необязательный токен — токен экземпляра, любой допустимый для провайдера Nest токен — и возвращает реализацию мока. Ниже пример создания универсального мокера с помощью [`jest-mock`](https://www.npmjs.com/package/jest-mock) и конкретного мока для `CatsService` через `jest.fn()`.

```typescript
// ...
import { ModuleMocker, MockMetadata } from 'jest-mock';

const moduleMocker = new ModuleMocker(global);

describe('CatsController', () => {
  let controller: CatsController;

  beforeEach(async () => {
    const moduleRef = await Test.createTestingModule({
      controllers: [CatsController],
    })
      .useMocker((token) => {
        const results = ['test1', 'test2'];
        if (token === CatsService) {
          return { findAll: jest.fn().mockResolvedValue(results) };
        }
        if (typeof token === 'function') {
          const mockMetadata = moduleMocker.getMetadata(
            token,
          ) as MockMetadata<any, any>;
          const Mock = moduleMocker.generateFromMetadata(
            mockMetadata,
          ) as ObjectConstructor;
          return new Mock();
        }
      })
      .compile();

    controller = moduleRef.get(CatsController);
  });
});
```

Эти моки можно получать из тестового контейнера так же, как обычные пользовательские провайдеры: `moduleRef.get(CatsService)`.

> **Подсказка** Универсальную фабрику моков, например `createMock` из [`@golevelup/ts-jest`](https://github.com/golevelup/nestjs/tree/master/packages/testing), также можно передать напрямую.

> **Подсказка** Провайдеры `REQUEST` и `INQUIRER` нельзя замокать автоматически, потому что они уже предопределены в контексте. Однако их можно _переопределить_ синтаксисом пользовательского провайдера или методом `.overrideProvider`.

#### Сквозное тестирование

В отличие от модульного тестирования, сосредоточенного на отдельных модулях и классах, сквозное (e2e) тестирование охватывает взаимодействие классов и модулей на более высоком уровне — ближе к тому, как с рабочей системой взаимодействуют конечные пользователи. По мере роста приложения вручную проверять сквозное поведение каждого эндпоинта API становится сложно. Автоматизированные сквозные тесты помогают убедиться, что поведение системы в целом корректно и соответствует требованиям проекта. Для e2e-тестов используется конфигурация, похожая на ту, что мы разобрали в разделе о **модульном тестировании**. Кроме того, Nest упрощает использование библиотеки [Supertest](https://github.com/visionmedia/supertest) для имитации HTTP-запросов.

**`cats.e2e-spec.ts`**

```typescript
import * as request from 'supertest';
import { Test } from '@nestjs/testing';
import { CatsModule } from '../../src/cats/cats.module';
import { CatsService } from '../../src/cats/cats.service';
import { INestApplication } from '@nestjs/common';

describe('Cats', () => {
  let app: INestApplication;
  let catsService = { findAll: () => ['test'] };

  beforeAll(async () => {
    const moduleRef = await Test.createTestingModule({
      imports: [CatsModule],
    })
      .overrideProvider(CatsService)
      .useValue(catsService)
      .compile();

    app = moduleRef.createNestApplication();
    await app.init();
  });

  it(`/GET cats`, () => {
    return request(app.getHttpServer())
      .get('/cats')
      .expect(200)
      .expect({
        data: catsService.findAll(),
      });
  });

  afterAll(async () => {
    await app.close();
  });
});
```

> **Подсказка** Если вы используете [Fastify](<../04. Techniques/19. Performance (Fastify).md>) в качестве HTTP-адаптера, потребуется немного другая конфигурация; у него есть встроенные средства тестирования:
>
> ```ts
> let app: NestFastifyApplication;
>
> beforeAll(async () => {
>   app = moduleRef.createNestApplication<NestFastifyApplication>(
>     new FastifyAdapter(),
>   );
>
>   await app.init();
>   await app.getHttpAdapter().getInstance().ready();
> });
>
> it(`/GET cats`, () => {
>   return app
>     .inject({
>       method: 'GET',
>       url: '/cats',
>     })
>     .then((result) => {
>       expect(result.statusCode).toEqual(200);
>       expect(result.payload).toEqual(/* expectedPayload */);
>     });
> });
>
> afterAll(async () => {
>   await app.close();
> });
> ```

В этом примере мы опираемся на некоторые из описанных ранее понятий. Помимо использованного ранее метода `compile()`, здесь применяется метод `createNestApplication()`, создающий полноценную среду выполнения Nest.

Стоит учитывать одну особенность: когда приложение компилируется методом `compile()`, `HttpAdapterHost#httpAdapter` в этот момент будет undefined. Так происходит потому, что на этапе компиляции HTTP-адаптер и сервер ещё не созданы. Если вашему тесту нужен `httpAdapter`, создавайте экземпляр приложения методом `createNestApplication()` либо переработайте проект так, чтобы избавиться от этой зависимости на этапе инициализации графа зависимостей.

Итак, разберём пример по частям:

Мы сохраняем ссылку на запущенное приложение в переменной `app`, чтобы использовать её для имитации HTTP-запросов.

HTTP-тесты имитируются функцией `request()` из Supertest. Мы хотим, чтобы эти HTTP-запросы попадали в наше запущенное приложение Nest, поэтому передаём в `request()` ссылку на HTTP-слушатель, лежащий в основе Nest (его, в свою очередь, может предоставлять платформа Express). Отсюда и конструкция `request(app.getHttpServer())`. Вызов `request()` возвращает обёрнутый HTTP-сервер, уже подключённый к приложению Nest, с методами для имитации настоящего HTTP-запроса. Например, `request(...).get('/cats')` инициирует запрос к приложению Nest, идентичный **реальному** HTTP-запросу вида `get '/cats'`, пришедшему по сети.

В этом примере мы также предоставляем альтернативную (тестовый двойник) реализацию `CatsService`, которая просто возвращает жёстко заданное значение, поддающееся проверке. Для подстановки такой альтернативной реализации используется `overrideProvider()`. Аналогично Nest предоставляет методы для переопределения модулей, guard'ов, interceptor'ов, фильтров и pipe'ов: `overrideModule()`, `overrideGuard()`, `overrideInterceptor()`, `overrideFilter()` и `overridePipe()` соответственно.

Каждый из методов переопределения (кроме `overrideModule()`) возвращает объект с тремя методами, повторяющими описанные для [пользовательских провайдеров](<01. Custom providers.md>):

- `useClass`: вы указываете класс, экземпляр которого будет создан и подставлен вместо объекта (провайдера, guard'а и т. д.).
- `useValue`: вы указываете экземпляр, который заменит объект.
- `useFactory`: вы указываете функцию, возвращающую экземпляр, который заменит объект.

В свою очередь, `overrideModule()` возвращает объект с методом `useModule()`, которым вы задаёте модуль, заменяющий исходный:

```typescript
const moduleRef = await Test.createTestingModule({
  imports: [AppModule],
})
  .overrideModule(CatsModule)
  .useModule(AlternateCatsModule)
  .compile();
```

Каждый из методов переопределения, в свою очередь, возвращает экземпляр `TestingModule`, поэтому их можно объединять в цепочки в [текучем стиле](https://en.wikipedia.org/wiki/Fluent_interface). В конце такой цепочки следует вызвать `compile()`, чтобы Nest создал и инициализировал модуль.

Иногда также бывает нужно подставить собственный логгер, например при запуске тестов на CI-сервере. Воспользуйтесь методом `setLogger()` и передайте объект, реализующий интерфейс `LoggerService`, чтобы указать `TestModuleBuilder`, как вести логирование во время тестов (по умолчанию в консоль выводятся только логи уровня «error»).

#### Переопределение глобально зарегистрированных enhancer'ов

Если у вас есть глобально зарегистрированный guard (или pipe, interceptor, фильтр), для его переопределения потребуется несколько дополнительных шагов. Напомним, как выглядит исходная регистрация:

```typescript
providers: [
  {
    provide: APP_GUARD,
    useClass: JwtAuthGuard,
  },
],
```

Здесь guard регистрируется как «multi»-провайдер через токен `APP_*`. Чтобы получить возможность заменить `JwtAuthGuard`, регистрация должна ссылаться на существующий провайдер:

```typescript
providers: [
  {
    provide: APP_GUARD,
    useExisting: JwtAuthGuard,
    // ^^^^^^^^ обратите внимание: 'useExisting' вместо 'useClass'
  },
  JwtAuthGuard,
],
```

> **Подсказка** Замените `useClass` на `useExisting`, чтобы сослаться на зарегистрированный провайдер, а не заставлять Nest создавать экземпляр за токеном.

Теперь `JwtAuthGuard` виден Nest как обычный провайдер, который можно переопределить при создании `TestingModule`:

```typescript
const moduleRef = await Test.createTestingModule({
  imports: [AppModule],
})
  .overrideProvider(JwtAuthGuard)
  .useClass(MockAuthGuard)
  .compile();
```

Теперь во всех ваших тестах для каждого запроса будет использоваться `MockAuthGuard`.

#### Тестирование экземпляров с областью видимости запроса

Провайдеры [с областью видимости запроса](<04. Injection scopes.md>) создаются заново для каждого входящего **запроса**. После завершения обработки запроса экземпляр удаляется сборщиком мусора. Это создаёт проблему: мы не можем получить доступ к поддереву внедрения зависимостей, созданному специально для тестируемого запроса.

Из разделов выше мы знаем, что метод `resolve()` позволяет получить динамически созданный экземпляр класса. Кроме того, как описано <a href="06.%20Module%20reference.md#разрешение-scoped-провайдеров">здесь</a>, мы знаем, что можно передать уникальный идентификатор контекста, чтобы управлять жизненным циклом поддерева DI-контейнера. Как воспользоваться этим при тестировании?

Стратегия такова: заранее сгенерировать идентификатор контекста и заставить Nest использовать именно его при создании поддерева для всех входящих запросов. Так мы сможем получать экземпляры, созданные для тестируемого запроса.

Для этого примените `jest.spyOn()` к `ContextIdFactory`:

```typescript
const contextId = ContextIdFactory.create();
jest
  .spyOn(ContextIdFactory, 'getByRequest')
  .mockImplementation(() => contextId);
```

Теперь с помощью `contextId` мы можем обращаться к единственному сгенерированному поддереву DI-контейнера для любого последующего запроса.

```typescript
catsService = await moduleRef.resolve(CatsService, contextId);
```
