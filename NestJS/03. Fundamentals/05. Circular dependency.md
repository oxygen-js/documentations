### Циклическая зависимость

Циклическая зависимость возникает, когда два класса зависят друг от друга. Например, классу A нужен класс B, а классу B — класс A. В Nest циклические зависимости могут возникать как между модулями, так и между провайдерами.

Циклических зависимостей по возможности следует избегать, но это не всегда осуществимо. Для таких случаев Nest предлагает два способа их разрешения между провайдерами. В этой главе описан первый приём — **опережающие ссылки** (forward referencing), и второй — использование класса **ModuleRef** для получения экземпляра провайдера из DI-контейнера.

Также описано разрешение циклических зависимостей между модулями.

> **Предупреждение** Циклическая зависимость может возникнуть и при использовании «barrel-файлов» / файлов index.ts для группировки импортов. Для классов модулей и провайдеров barrel-файлы использовать не стоит. Например, их не следует применять при импорте файлов из того же каталога, где лежит сам barrel-файл: `cats/cats.controller` не должен импортировать `cats`, чтобы получить файл `cats/cats.service`. Подробнее см. [этот issue на GitHub](https://github.com/nestjs/nest/issues/1181#issuecomment-430197191).

#### Опережающая ссылка

**Опережающая ссылка** позволяет Nest ссылаться на ещё не определённые классы с помощью вспомогательной функции `forwardRef()`. Например, если `CatsService` и `CommonService` зависят друг от друга, обе стороны этой связи могут использовать `@Inject()` вместе с `forwardRef()`, чтобы разрешить циклическую зависимость. Иначе Nest не сможет создать их экземпляры, поскольку не все необходимые метаданные будут доступны. Пример:

**`cats.service.ts`**

```typescript
@Injectable()
export class CatsService {
  constructor(
    @Inject(forwardRef(() => CommonService))
    private commonService: CommonService,
  ) {}
}
```

> **Подсказка** Функция `forwardRef()` импортируется из пакета `@nestjs/common`.

Это закрывает одну сторону связи. Теперь сделаем то же самое для `CommonService`:

**`common.service.ts`**

```typescript
@Injectable()
export class CommonService {
  constructor(
    @Inject(forwardRef(() => CatsService))
    private catsService: CatsService,
  ) {}
}
```

> **Предупреждение** Порядок создания экземпляров не определён. Убедитесь, что ваш код не зависит от того, какой конструктор будет вызван первым. Циклические зависимости, завязанные на провайдеры с областью `Scope.REQUEST`, могут приводить к неопределённым зависимостям. Подробнее [здесь](https://github.com/nestjs/nest/issues/5778)

#### Альтернатива: класс ModuleRef

Вместо `forwardRef()` можно переработать код и использовать класс `ModuleRef`, чтобы получать провайдер на одной из сторон (в остальном) циклической связи. Подробнее о вспомогательном классе `ModuleRef` — [здесь](<06. Module reference.md>).

#### Опережающая ссылка между модулями

Чтобы разрешить циклические зависимости между модулями, используйте ту же вспомогательную функцию `forwardRef()` с обеих сторон связи модулей. Например:

**`common.module.ts`**

```typescript
@Module({
  imports: [forwardRef(() => CatsModule)],
})
export class CommonModule {}
```

Это закрывает одну сторону связи. Теперь сделаем то же самое для `CatsModule`:

**`cats.module.ts`**

```typescript
@Module({
  imports: [forwardRef(() => CommonModule)],
})
export class CatsModule {}
```
