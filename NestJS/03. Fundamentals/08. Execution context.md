### Контекст выполнения

Nest предоставляет несколько вспомогательных классов, которые упрощают написание приложений, работающих сразу в нескольких контекстах приложения (например, в контексте HTTP-сервера Nest, микросервисов и WebSocket-приложений). Эти утилиты дают информацию о текущем контексте выполнения, на основе которой можно строить универсальные [guard](<../02. Overview/08. Guards.md>), [фильтры](<../02. Overview/06. Exception filters.md>) и [interceptor](<../02. Overview/09. Interceptors.md>), работающие в широком наборе контроллеров, методов и контекстов выполнения.

В этой главе мы рассмотрим два таких класса: `ArgumentsHost` и `ExecutionContext`.

#### Класс ArgumentsHost

Класс `ArgumentsHost` предоставляет методы для получения аргументов, передаваемых обработчику. Он позволяет выбрать подходящий контекст (например, HTTP, RPC (микросервис) или WebSockets), из которого нужно извлечь аргументы. Фреймворк передаёт экземпляр `ArgumentsHost` — обычно он фигурирует как параметр `host` — в тех местах, где он может вам понадобиться. Например, метод `catch()` [фильтра исключений](<../02. Overview/06. Exception filters.md#arguments-host>) вызывается с экземпляром `ArgumentsHost`.

`ArgumentsHost` — это просто абстракция над аргументами обработчика. Например, для приложений с HTTP-сервером (когда используется `@nestjs/platform-express`) объект `host` инкапсулирует массив Express `[request, response, next]`, где `request` — объект запроса, `response` — объект ответа, а `next` — функция, управляющая циклом «запрос-ответ» приложения. Для [GraphQL](<../06. GraphQL/01. Quick start.md>)-приложений, напротив, объект `host` содержит массив `[root, args, context, info]`.

#### Текущий контекст приложения

При построении универсальных [guard](<../02. Overview/08. Guards.md>), [фильтров](<../02. Overview/06. Exception filters.md>) и [interceptor](<../02. Overview/09. Interceptors.md>), рассчитанных на работу в нескольких контекстах приложения, нам нужен способ определить, в приложении какого типа сейчас выполняется наш метод. Для этого служит метод `getType()` класса `ArgumentsHost`:

```typescript
if (host.getType() === 'http') {
  // сделать что-то, что важно только в контексте обычных HTTP-запросов (REST)
} else if (host.getType() === 'rpc') {
  // сделать что-то, что важно только в контексте запросов к микросервисам
} else if (host.getType<GqlContextType>() === 'graphql') {
  // сделать что-то, что важно только в контексте GraphQL-запросов
}
```

> **Подсказка** `GqlContextType` импортируется из пакета `@nestjs/graphql`.

Зная тип приложения, мы можем писать более универсальные компоненты, как показано ниже.

#### Аргументы обработчика

Чтобы получить массив аргументов, передаваемых обработчику, можно воспользоваться методом `getArgs()` объекта host.

```typescript
const [req, res, next] = host.getArgs();
```

Отдельный аргумент можно достать по индексу с помощью метода `getArgByIndex()`:

```typescript
const request = host.getArgByIndex(0);
const response = host.getArgByIndex(1);
```

В этих примерах мы получили объекты запроса и ответа по индексу, что обычно не рекомендуется, так как привязывает приложение к конкретному контексту выполнения. Вместо этого вы можете сделать код надёжнее и пригоднее для повторного использования, применив один из вспомогательных методов объекта `host` для переключения на подходящий контекст приложения. Методы переключения контекста показаны ниже.

```typescript
/**
 * Переключить контекст на RPC.
 */
switchToRpc(): RpcArgumentsHost;
/**
 * Переключить контекст на HTTP.
 */
switchToHttp(): HttpArgumentsHost;
/**
 * Переключить контекст на WebSockets.
 */
switchToWs(): WsArgumentsHost;
```

Перепишем предыдущий пример с использованием метода `switchToHttp()`. Вызов `host.switchToHttp()` возвращает объект `HttpArgumentsHost`, соответствующий контексту HTTP-приложения. У объекта `HttpArgumentsHost` есть два полезных метода, с помощью которых можно извлечь нужные объекты. Здесь мы также применяем утверждения типов Express, чтобы получить объекты с нативными типами Express:

```typescript
const ctx = host.switchToHttp();
const request = ctx.getRequest<Request>();
const response = ctx.getResponse<Response>();
```

Аналогично у `WsArgumentsHost` и `RpcArgumentsHost` есть методы, возвращающие соответствующие объекты в контекстах микросервисов и WebSockets. Вот методы `WsArgumentsHost`:

```typescript
export interface WsArgumentsHost {
  /**
   * Возвращает объект данных.
   */
  getData<T>(): T;
  /**
   * Возвращает объект клиента.
   */
  getClient<T>(): T;
}
```

А вот методы `RpcArgumentsHost`:

```typescript
export interface RpcArgumentsHost {
  /**
   * Возвращает объект данных.
   */
  getData<T>(): T;

  /**
   * Возвращает объект контекста.
   */
  getContext<T>(): T;
}
```

#### Класс ExecutionContext

`ExecutionContext` расширяет `ArgumentsHost`, предоставляя дополнительные сведения о текущем процессе выполнения. Как и в случае с `ArgumentsHost`, Nest передаёт экземпляр `ExecutionContext` там, где он может вам понадобиться, — например, в методе `canActivate()` [guard](<../02. Overview/08. Guards.md#контекст-выполнения>) и в методе `intercept()` [interceptor](<../02. Overview/09. Interceptors.md#контекст-выполнения>). Он предоставляет следующие методы:

```typescript
export interface ExecutionContext extends ArgumentsHost {
  /**
   * Возвращает тип класса контроллера, которому принадлежит текущий обработчик.
   */
  getClass<T>(): Type<T>;
  /**
   * Возвращает ссылку на обработчик (метод), который будет вызван следующим
   * в конвейере обработки запроса.
   */
  getHandler(): Function;
}
```

Метод `getHandler()` возвращает ссылку на обработчик, который вот-вот будет вызван. Метод `getClass()` возвращает тип класса `Controller`, которому принадлежит этот обработчик. Например, если в HTTP-контексте текущий обрабатываемый запрос — это `POST`-запрос, привязанный к методу `create()` контроллера `CatsController`, то `getHandler()` вернёт ссылку на метод `create()`, а `getClass()` — **класс** `CatsController` (а не его экземпляр).

```typescript
const methodKey = ctx.getHandler().name; // "create"
const className = ctx.getClass().name; // "CatsController"
```

Возможность обращаться к ссылкам и на текущий класс, и на метод-обработчик даёт большую гибкость. Главное — она позволяет из guard или interceptor получать доступ к метаданным, заданным либо декораторами, созданными через `Reflector#createDecorator`, либо встроенным декоратором `@SetMetadata()`. Этот сценарий разбирается ниже.

#### Рефлексия и метаданные

Nest позволяет прикреплять к обработчикам маршрутов **пользовательские метаданные** — через декораторы, созданные методом `Reflector#createDecorator`, и через встроенный декоратор `@SetMetadata()`. В этом разделе сравним оба подхода и посмотрим, как читать метаданные внутри guard или interceptor.

Чтобы создавать строго типизированные декораторы с помощью `Reflector#createDecorator`, нужно указать аргумент типа. Например, создадим декоратор `Roles`, принимающий на вход массив строк.

**`roles.decorator.ts`**

```ts
import { Reflector } from '@nestjs/core';

export const Roles = Reflector.createDecorator<string[]>();
```

Декоратор `Roles` здесь — это функция, принимающая единственный аргумент типа `string[]`.

Теперь, чтобы воспользоваться этим декоратором, достаточно пометить им обработчик:

**`cats.controller.ts`**

```typescript
@Post()
@Roles(['admin'])
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
```

Здесь мы прикрепили метаданные декоратора `Roles` к методу `create()`, указав, что доступ к этому маршруту должен быть только у пользователей с ролью `admin`.

Чтобы получить роль (роли) маршрута, то есть пользовательские метаданные, мы снова воспользуемся вспомогательным классом `Reflector`. `Reflector` внедряется в класс обычным способом:

**`roles.guard.ts`**

```typescript
@Injectable()
export class RolesGuard {
  constructor(private reflector: Reflector) {}
}
```

> **Подсказка** Класс `Reflector` импортируется из пакета `@nestjs/core`.

Теперь, чтобы прочитать метаданные обработчика, используйте метод `get()`:

```typescript
const roles = this.reflector.get(Roles, context.getHandler());
```

Метод `Reflector#get` позволяет легко получить метаданные, принимая два аргумента: ссылку на декоратор и **контекст** (цель декоратора), из которого нужно их извлечь. В этом примере указанный **декоратор** — `Roles` (см. файл `roles.decorator.ts` выше). Контекст задаётся вызовом `context.getHandler()`, благодаря чему извлекаются метаданные обрабатываемого в данный момент обработчика маршрута. Напомним, что `getHandler()` даёт нам **ссылку** на функцию-обработчик маршрута.

Кроме того, контроллер можно организовать иначе — задать метаданные на уровне контроллера, и тогда они применятся ко всем маршрутам этого класса.

**`cats.controller.ts`**

```typescript
@Roles(['admin'])
@Controller('cats')
export class CatsController {}
```

В этом случае, чтобы извлечь метаданные контроллера, вторым аргументом вместо `context.getHandler()` передаётся `context.getClass()` (чтобы контекстом для извлечения метаданных стал класс контроллера):

**`roles.guard.ts`**

```typescript
const roles = this.reflector.get(Roles, context.getClass());
```

Поскольку метаданные можно задавать на нескольких уровнях, вам может потребоваться извлечь и объединить метаданные из нескольких контекстов. Класс `Reflector` предоставляет для этого два вспомогательных метода. Оба извлекают метаданные **сразу** и контроллера, и метода, но комбинируют их по-разному.

Рассмотрим сценарий, в котором вы задали метаданные `Roles` на обоих уровнях.

**`cats.controller.ts`**

```typescript
@Roles(['user'])
@Controller('cats')
export class CatsController {
  @Post()
  @Roles(['admin'])
  async create(@Body() createCatDto: CreateCatDto) {
    this.catsService.create(createCatDto);
  }
}
```

Если вы хотите задать `'user'` в качестве роли по умолчанию и выборочно переопределять её для отдельных методов, вам, скорее всего, подойдёт метод `getAllAndOverride()`.

```typescript
const roles = this.reflector.getAllAndOverride(Roles, [context.getHandler(), context.getClass()]);
```

Guard с таким кодом, выполняясь в контексте метода `create()` с приведёнными выше метаданными, получит в `roles` значение `['admin']`.

Чтобы получить метаданные обоих уровней и объединить их (этот метод объединяет как массивы, так и объекты), используйте метод `getAllAndMerge()`:

```typescript
const roles = this.reflector.getAllAndMerge(Roles, [context.getHandler(), context.getClass()]);
```

В результате `roles` будет содержать `['user', 'admin']`.

В обоих методах объединения первым аргументом передаётся ключ метаданных, а вторым — массив целевых контекстов метаданных (то есть вызовов методов `getHandler()` и/или `getClass()`).

#### Низкоуровневый подход

Как упоминалось ранее, вместо `Reflector#createDecorator` для прикрепления метаданных к обработчику можно использовать встроенный декоратор `@SetMetadata()`.

**`cats.controller.ts`**

```typescript
@Post()
@SetMetadata('roles', ['admin'])
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
```

> **Подсказка** Декоратор `@SetMetadata()` импортируется из пакета `@nestjs/common`.

Приведённой выше конструкцией мы прикрепили к методу `create()` метаданные `roles` (`roles` — ключ метаданных, `['admin']` — связанное с ним значение). Хотя это работает, использовать `@SetMetadata()` напрямую в маршрутах — не лучшая практика. Вместо этого вы можете создать собственные декораторы, как показано ниже:

**`roles.decorator.ts`**

```typescript
import { SetMetadata } from '@nestjs/common';

export const Roles = (...roles: string[]) => SetMetadata('roles', roles);
```

Такой подход намного чище и читаемее и отчасти напоминает подход с `Reflector#createDecorator`. Разница в том, что с `@SetMetadata` у вас больше контроля над ключом и значением метаданных, а также можно создавать декораторы, принимающие более одного аргумента.

Теперь, когда у нас есть собственный декоратор `@Roles()`, мы можем применить его к методу `create()`.

**`cats.controller.ts`**

```typescript
@Post()
@Roles('admin')
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
```

Чтобы получить роль (роли) маршрута, то есть пользовательские метаданные, мы снова воспользуемся вспомогательным классом `Reflector`:

**`roles.guard.ts`**

```typescript
@Injectable()
export class RolesGuard {
  constructor(private reflector: Reflector) {}
}
```

> **Подсказка** Класс `Reflector` импортируется из пакета `@nestjs/core`.

Теперь, чтобы прочитать метаданные обработчика, используйте метод `get()`.

```typescript
const roles = this.reflector.get<string[]>('roles', context.getHandler());
```

Здесь вместо ссылки на декоратор мы передаём первым аргументом **ключ** метаданных (в нашем случае — `'roles'`). Всё остальное остаётся таким же, как в примере с `Reflector#createDecorator`.
