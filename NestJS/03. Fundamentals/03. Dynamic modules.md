### Динамические модули

[Глава о модулях](<../02. Overview/04. Modules.md>) описывает основы модулей Nest и содержит краткое введение в [динамические модули](<../02. Overview/04. Modules.md#динамические-модули>). Эта глава раскрывает тему динамических модулей подробнее. По её завершении вы будете хорошо понимать, что это такое, а также как и когда их применять.

#### Введение

Большинство примеров кода в разделе **Обзор** используют обычные, или статические, модули. Модули объединяют в группы такие компоненты, как [провайдеры](<../02. Overview/03. Providers.md>) и [контроллеры](<../02. Overview/02. Controllers.md>), которые вместе образуют модульную часть приложения. Они предоставляют этим компонентам контекст выполнения, или область видимости. Например, провайдеры, объявленные в модуле, видны остальным участникам этого модуля без необходимости их экспортировать. Если провайдер должен быть виден за пределами модуля, его сначала экспортируют из модуля-владельца, а затем импортируют в модуль-потребитель.

Разберём знакомый пример.

Сначала объявим `UsersModule`, который предоставляет и экспортирует `UsersService`. `UsersModule` — это **модуль-владелец** для `UsersService`.

```typescript
import { Module } from '@nestjs/common';
import { UsersService } from './users.service';

@Module({
  providers: [UsersService],
  exports: [UsersService],
})
export class UsersModule {}
```

Далее объявим `AuthModule`, который импортирует `UsersModule`, благодаря чему экспортированные из `UsersModule` провайдеры становятся доступны внутри `AuthModule`:

```typescript
import { Module } from '@nestjs/common';
import { AuthService } from './auth.service';
import { UsersModule } from '../users/users.module';

@Module({
  imports: [UsersModule],
  providers: [AuthService],
  exports: [AuthService],
})
export class AuthModule {}
```

Эти конструкции позволяют внедрить `UsersService`, например, в `AuthService`, который находится в `AuthModule`:

```typescript
import { Injectable } from '@nestjs/common';
import { UsersService } from '../users/users.service';

@Injectable()
export class AuthService {
  constructor(private usersService: UsersService) {}
  /*
    Реализация, использующая this.usersService
  */
}
```

Будем называть это **статической** привязкой модулей. Вся информация, необходимая Nest для связывания модулей, уже объявлена в модуле-владельце и модуле-потребителе. Разберём, что происходит в этом процессе. Nest делает `UsersService` доступным внутри `AuthModule` так:

1. Создаёт экземпляр `UsersModule`, включая транзитивный импорт других модулей, которые использует сам `UsersModule`, и транзитивное разрешение всех зависимостей (см. [Пользовательские провайдеры](<01. Custom providers.md>)).
2. Создаёт экземпляр `AuthModule` и делает экспортированные провайдеры `UsersModule` доступными компонентам в `AuthModule` (так, как если бы они были объявлены в самом `AuthModule`).
3. Внедряет экземпляр `UsersService` в `AuthService`.

#### Сценарий использования динамического модуля

При статической привязке модулей у модуля-потребителя нет возможности **повлиять** на то, как настроены провайдеры модуля-владельца. Почему это важно? Представьте модуль общего назначения, который должен вести себя по-разному в разных сценариях. Это похоже на концепцию «плагина» во многих системах, где универсальный механизм требует некоторой настройки, прежде чем потребитель сможет им воспользоваться.

Хороший пример в Nest — **модуль конфигурации**. Многим приложениям удобно вынести детали конфигурации в отдельный модуль конфигурации. Это позволяет легко менять настройки приложения в разных развёртываниях: например, база данных для разработки у разработчиков, staging-база для тестового окружения и т. д. Когда управление параметрами конфигурации делегировано модулю конфигурации, исходный код приложения остаётся независимым от этих параметров.

Сложность в том, что сам модуль конфигурации, будучи универсальным (подобно «плагину»), должен настраиваться модулем-потребителем. Здесь на сцену выходят _динамические модули_. С их помощью мы можем сделать модуль конфигурации **динамическим**, чтобы модуль-потребитель через API управлял тем, как модуль конфигурации настраивается в момент импорта.

Иными словами, динамические модули предоставляют API для импорта одного модуля в другой с настройкой свойств и поведения импортируемого модуля — в отличие от статических привязок, которые мы рассматривали до сих пор.

#### Пример модуля конфигурации

В этом разделе мы будем опираться на базовую версию примера из [главы о конфигурации](<../04. Techniques/01. Configuration.md#service>). Готовая версия по итогам этой главы доступна как рабочий [пример здесь](https://github.com/nestjs/nest/tree/master/sample/25-dynamic-modules).

Наша задача — сделать так, чтобы `ConfigModule` принимал объект `options` для настройки. Вот возможность, которую мы хотим поддержать. В базовом примере расположение файла `.env` жёстко задано корневой папкой проекта. Допустим, мы хотим сделать его настраиваемым, чтобы файлы `.env` можно было хранить в любой выбранной вами папке. Например, представьте, что вы хотите хранить разные файлы `.env` в папке `config` под корнем проекта (то есть в папке, соседней с `src`). И вы хотите выбирать разные папки при использовании `ConfigModule` в разных проектах.

Динамические модули дают возможность передавать параметры в импортируемый модуль, чтобы изменить его поведение. Посмотрим, как это работает. Удобно начать с конечной цели — как это должно выглядеть с точки зрения модуля-потребителя, — и двигаться в обратную сторону. Сначала быстро вспомним пример _статического_ импорта `ConfigModule` (то есть подхода, не позволяющего влиять на поведение импортируемого модуля). Обратите особое внимание на массив `imports` в декораторе `@Module()`:

```typescript
import { Module } from '@nestjs/common';
import { AppController } from './app.controller';
import { AppService } from './app.service';
import { ConfigModule } from './config/config.module';

@Module({
  imports: [ConfigModule],
  controllers: [AppController],
  providers: [AppService],
})
export class AppModule {}
```

Теперь посмотрим, как мог бы выглядеть импорт _динамического модуля_, в котором мы передаём объект конфигурации. Сравните разницу в массиве `imports` между этими двумя примерами:

```typescript
import { Module } from '@nestjs/common';
import { AppController } from './app.controller';
import { AppService } from './app.service';
import { ConfigModule } from './config/config.module';

@Module({
  imports: [ConfigModule.register({ folder: './config' })],
  controllers: [AppController],
  providers: [AppService],
})
export class AppModule {}
```

Разберём, что происходит в динамическом примере выше. Из каких частей он состоит?

1. `ConfigModule` — обычный класс, поэтому можно заключить, что у него есть **статический метод** `register()`. Мы знаем, что он статический, потому что вызываем его на классе `ConfigModule`, а не на его **экземпляре**. Примечание: этот метод, который мы скоро напишем, может называться как угодно, но по соглашению его следует называть `forRoot()` или `register()`.
2. Метод `register()` пишем мы сами, поэтому он может принимать любые аргументы. В нашем случае это будет простой объект `options` с подходящими свойствами — типичный вариант.
3. Можно заключить, что метод `register()` должен возвращать нечто вроде `module`, поскольку его возвращаемое значение попадает в знакомый список `imports`, который, как мы видели, содержит список модулей.

На самом деле метод `register()` вернёт `DynamicModule`. Динамический модуль — это не более чем модуль, созданный во время выполнения, с точно такими же свойствами, как у статического модуля, плюс одно дополнительное свойство `module`. Быстро вспомним объявление статического модуля, обращая внимание на параметры модуля, переданные в декоратор:

```typescript
@Module({
  imports: [DogsModule],
  controllers: [CatsController],
  providers: [CatsService],
  exports: [CatsService]
})
```

Динамические модули должны возвращать объект с точно таким же интерфейсом плюс дополнительное свойство `module`. Свойство `module` служит именем модуля и должно совпадать с именем класса модуля, как показано в примере ниже.

> **Подсказка** У динамического модуля все свойства объекта параметров модуля необязательны, **кроме** `module`.

Что насчёт статического метода `register()`? Теперь видно, что его задача — вернуть объект с интерфейсом `DynamicModule`. Вызывая его, мы фактически добавляем модуль в список `imports` — так же, как в статическом случае мы указывали бы там имя класса модуля. Иначе говоря, API динамических модулей просто возвращает модуль, но вместо фиксации свойств в декораторе `@Module` мы задаём их программно.

Осталось прояснить ещё пару деталей, чтобы картина стала полной:

1. Теперь можно утверждать, что свойство `imports` декоратора `@Module()` принимает не только имя класса модуля (например, `imports: [UsersModule]`), но и функцию, **возвращающую** динамический модуль (например, `imports: [ConfigModule.register(...)]`).
2. Динамический модуль сам может импортировать другие модули. В этом примере мы так делать не будем, но если динамический модуль зависит от провайдеров из других модулей, вы импортируете их через необязательное свойство `imports`. Это полностью аналогично тому, как вы объявляете метаданные статического модуля с помощью декоратора `@Module()`.

Вооружившись этим пониманием, посмотрим, как должно выглядеть объявление нашего динамического `ConfigModule`. Попробуем.

```typescript
import { DynamicModule, Module } from '@nestjs/common';
import { ConfigService } from './config.service';

@Module({})
export class ConfigModule {
  static register(): DynamicModule {
    return {
      module: ConfigModule,
      providers: [ConfigService],
      exports: [ConfigService],
    };
  }
}
```

Теперь должно быть понятно, как всё связано. Вызов `ConfigModule.register(...)` возвращает объект `DynamicModule` со свойствами, по сути теми же, что до сих пор мы передавали как метаданные через декоратор `@Module()`.

> **Подсказка** Импортируйте `DynamicModule` из `@nestjs/common`.

Однако наш динамический модуль пока не очень интересен: мы так и не добавили возможность **настраивать** его, о которой говорили. Займёмся этим.

#### Настройка модуля

Очевидное решение для настройки поведения `ConfigModule` — передавать ему объект `options` в статическом методе `register()`, как мы и предполагали выше. Взглянем ещё раз на свойство `imports` модуля-потребителя:

```typescript
import { Module } from '@nestjs/common';
import { AppController } from './app.controller';
import { AppService } from './app.service';
import { ConfigModule } from './config/config.module';

@Module({
  imports: [ConfigModule.register({ folder: './config' })],
  controllers: [AppController],
  providers: [AppService],
})
export class AppModule {}
```

Это аккуратно решает задачу передачи объекта `options` в динамический модуль. Но как затем использовать этот объект внутри `ConfigModule`? Подумаем минуту. Мы знаем, что `ConfigModule` по сути служит носителем, который предоставляет и экспортирует внедряемый сервис — `ConfigService` — для использования другими провайдерами. И именно `ConfigService` нужно прочитать объект `options`, чтобы настроить своё поведение. Допустим на время, что мы каким-то образом умеем передавать `options` из метода `register()` в `ConfigService`. С этим допущением внесём в сервис несколько изменений, чтобы его поведение зависело от свойств объекта `options`. (**Примечание**: пока мы _ещё не_ определили, как именно передавать этот объект, поэтому просто зададим `options` жёстко. Мы исправим это через минуту.)

```typescript
import { Injectable } from '@nestjs/common';
import * as fs from 'node:fs';
import * as path from 'node:path';
import * as dotenv from 'dotenv';
import { EnvConfig } from './interfaces';

@Injectable()
export class ConfigService {
  private readonly envConfig: EnvConfig;

  constructor() {
    const options = { folder: './config' };

    const filePath = `${process.env.NODE_ENV || 'development'}.env`;
    const envFile = path.resolve(__dirname, '../../', options.folder, filePath);
    this.envConfig = dotenv.parse(fs.readFileSync(envFile));
  }

  get(key: string): string {
    return this.envConfig[key];
  }
}
```

Теперь наш `ConfigService` умеет находить файл `.env` в папке, указанной в `options`.

Остаётся как-то внедрить объект `options` из шага `register()` в `ConfigService`. И, конечно, мы сделаем это через _внедрение зависимостей_. Это ключевой момент, поэтому убедитесь, что вы его поняли. Наш `ConfigModule` предоставляет `ConfigService`. `ConfigService`, в свою очередь, зависит от объекта `options`, который доступен только во время выполнения. Значит, во время выполнения нам нужно сначала привязать объект `options` к IoC-контейнеру Nest, а затем позволить Nest внедрить его в `ConfigService`. Вспомните из главы **Пользовательские провайдеры**, что провайдеры могут [содержать любое значение](<01. Custom providers.md#провайдеры-не-являющиеся-сервисами>), а не только сервисы, — так что использовать внедрение зависимостей для простого объекта `options` вполне уместно.

Сначала разберёмся с привязкой объекта параметров к IoC-контейнеру. Делаем это в статическом методе `register()`. Помните, что мы динамически конструируем модуль, и одно из свойств модуля — список его провайдеров. Значит, нужно объявить наш объект параметров как провайдер. Это сделает его внедряемым в `ConfigService`, чем мы воспользуемся на следующем шаге. В коде ниже обратите внимание на массив `providers`:

```typescript
import { DynamicModule, Module } from '@nestjs/common';
import { ConfigService } from './config.service';

@Module({})
export class ConfigModule {
  static register(options: Record<string, any>): DynamicModule {
    return {
      module: ConfigModule,
      providers: [
        {
          provide: 'CONFIG_OPTIONS',
          useValue: options,
        },
        ConfigService,
      ],
      exports: [ConfigService],
    };
  }
}
```

Теперь можно завершить процесс, внедрив провайдер `'CONFIG_OPTIONS'` в `ConfigService`. Напомним, что при объявлении провайдера с неклассовым токеном нужно использовать декоратор `@Inject()`, [как описано здесь](<01. Custom providers.md#токены-провайдеров-не-основанные-на-классах>).

```typescript
import * as fs from 'node:fs';
import * as path from 'node:path';
import * as dotenv from 'dotenv';
import { Injectable, Inject } from '@nestjs/common';
import { EnvConfig } from './interfaces';

@Injectable()
export class ConfigService {
  private readonly envConfig: EnvConfig;

  constructor(@Inject('CONFIG_OPTIONS') private options: Record<string, any>) {
    const filePath = `${process.env.NODE_ENV || 'development'}.env`;
    const envFile = path.resolve(__dirname, '../../', options.folder, filePath);
    this.envConfig = dotenv.parse(fs.readFileSync(envFile));
  }

  get(key: string): string {
    return this.envConfig[key];
  }
}
```

И последнее замечание: для простоты выше мы использовали строковый токен внедрения (`'CONFIG_OPTIONS'`), но лучшая практика — объявить его как константу (или `Symbol`) в отдельном файле и импортировать этот файл. Например:

```typescript
export const CONFIG_OPTIONS = 'CONFIG_OPTIONS';
```

#### Соглашения сообщества

Вы могли видеть методы вроде `forRoot`, `register` и `forFeature` в некоторых пакетах `@nestjs/` и задаваться вопросом, чем они различаются. Строгого правила нет, но пакеты `@nestjs/` стараются следовать таким соглашениям.

Создавая модуль с методом:

- `register` — вы рассчитываете настроить динамический модуль конкретной конфигурацией, используемой только вызывающим модулем. Например, у Nest есть `@nestjs/axios`: `HttpModule.register({ baseUrl: 'someUrl' })`. Если в другом модуле вы напишете `HttpModule.register({ baseUrl: 'somewhere else' })`, конфигурация будет другой. Так можно делать для любого числа модулей.

- `forRoot` — вы рассчитываете настроить динамический модуль один раз и переиспользовать эту конфигурацию в нескольких местах (возможно, даже не замечая этого, поскольку она скрыта за абстракцией). Именно поэтому у вас один `GraphQLModule.forRoot()`, один `TypeOrmModule.forRoot()` и т. д.

- `forFeature` — вы рассчитываете использовать конфигурацию, заданную в `forRoot` динамического модуля, но с изменениями под нужды вызывающего модуля (например, к какому репозиторию этот модуль должен иметь доступ или какой контекст должен использовать логгер).

У всех этих методов обычно есть и `async`-аналоги — `registerAsync`, `forRootAsync` и `forFeatureAsync`, — которые означают то же самое, но используют механизм внедрения зависимостей Nest и для конфигурации.

#### Построитель настраиваемых модулей

Поскольку вручную создавать гибко настраиваемые динамические модули с `async`-методами (`registerAsync`, `forRootAsync` и т. д.) довольно сложно, особенно новичкам, Nest предоставляет класс `ConfigurableModuleBuilder`, который упрощает этот процесс и позволяет построить «чертёж» модуля буквально в несколько строк кода.

Например, возьмём пример, который мы использовали выше (`ConfigModule`), и переведём его на `ConfigurableModuleBuilder`. Прежде чем начать, создадим отдельный интерфейс, описывающий, какие параметры принимает наш `ConfigModule`.

```typescript
export interface ConfigModuleOptions {
  folder: string;
}
```

После этого создайте новый отдельный файл (рядом с существующим `config.module.ts`) и назовите его `config.module-definition.ts`. В этом файле воспользуемся `ConfigurableModuleBuilder`, чтобы построить определение `ConfigModule`.

**`config.module-definition.ts`**

```typescript
import { ConfigurableModuleBuilder } from '@nestjs/common';
import { ConfigModuleOptions } from './interfaces/config-module-options.interface';

export const { ConfigurableModuleClass, MODULE_OPTIONS_TOKEN } =
  new ConfigurableModuleBuilder<ConfigModuleOptions>().build();
```

Теперь откроем файл `config.module.ts` и изменим его реализацию так, чтобы задействовать автоматически сгенерированный `ConfigurableModuleClass`:

```typescript
import { Module } from '@nestjs/common';
import { ConfigService } from './config.service';
import { ConfigurableModuleClass } from './config.module-definition';

@Module({
  providers: [ConfigService],
  exports: [ConfigService],
})
export class ConfigModule extends ConfigurableModuleClass {}
```

Наследование от `ConfigurableModuleClass` означает, что теперь `ConfigModule` предоставляет не только метод `register` (как раньше, при собственной реализации), но и метод `registerAsync`, который позволяет потребителям настраивать модуль асинхронно — например, с помощью асинхронных фабрик:

```typescript
@Module({
  imports: [
    ConfigModule.register({ folder: './config' }),
    // или как вариант:
    // ConfigModule.registerAsync({
    //   useFactory: () => {
    //     return {
    //       folder: './config',
    //     }
    //   },
    //   inject: [...любые дополнительные зависимости...]
    // }),
  ],
})
export class AppModule {}
```

Метод `registerAsync` принимает в качестве аргумента следующий объект:

```typescript
{
  /**
   * Токен внедрения, разрешающийся в класс, экземпляр которого будет создан как провайдер.
   * Класс должен реализовывать соответствующий интерфейс.
   */
  useClass?: Type<
    ConfigurableModuleOptionsFactory<ModuleOptions, FactoryClassMethodKey>
  >;
  /**
   * Функция, возвращающая параметры (или Promise, разрешающийся в параметры)
   * для настройки модуля.
   */
  useFactory?: (...args: any[]) => Promise<ModuleOptions> | ModuleOptions;
  /**
   * Зависимости, которые может внедрить фабрика.
   */
  inject?: FactoryProvider['inject'];
  /**
   * Токен внедрения, разрешающийся в существующий провайдер. Провайдер должен
   * реализовывать соответствующий интерфейс.
   */
  useExisting?: Type<
    ConfigurableModuleOptionsFactory<ModuleOptions, FactoryClassMethodKey>
  >;
}
```

Разберём эти свойства по порядку:

- `useFactory` — функция, возвращающая объект конфигурации. Она может быть как синхронной, так и асинхронной. Чтобы внедрить зависимости в фабричную функцию, используйте свойство `inject`. Именно этот вариант мы применили в примере выше.
- `inject` — массив зависимостей, которые будут внедрены в фабричную функцию. Порядок зависимостей должен совпадать с порядком параметров фабричной функции.
- `useClass` — класс, экземпляр которого будет создан как провайдер. Класс должен реализовывать соответствующий интерфейс. Обычно это класс с методом `create()`, возвращающим объект конфигурации. Подробнее об этом — в разделе [Собственное имя метода класса](<03. Dynamic modules.md#собственное-имя-метода-класса>) ниже.
- `useExisting` — вариант `useClass`, позволяющий использовать существующий провайдер вместо того, чтобы Nest создавал новый экземпляр класса. Это удобно, когда вы хотите использовать провайдер, уже зарегистрированный в модуле. Учтите, что класс должен реализовывать тот же интерфейс, что и в случае `useClass` (то есть предоставлять метод `create()`, если только вы не переопределили имя метода по умолчанию — см. раздел [Собственное имя метода класса](<03. Dynamic modules.md#собственное-имя-метода-класса>) ниже).

Всегда выбирайте только один из перечисленных вариантов (`useFactory`, `useClass` или `useExisting`) — они взаимоисключающие.

Наконец, обновим класс `ConfigService`, чтобы он внедрял сгенерированный провайдер параметров модуля вместо `'CONFIG_OPTIONS'`, который мы использовали до сих пор.

```typescript
@Injectable()
export class ConfigService {
  constructor(@Inject(MODULE_OPTIONS_TOKEN) private options: ConfigModuleOptions) { ... }
}
```

#### Собственное имя метода класса

По умолчанию `ConfigurableModuleClass` предоставляет методы `register` и парный к нему `registerAsync`. Чтобы использовать другое имя метода, воспользуйтесь методом `ConfigurableModuleBuilder#setClassMethodName`:

**`config.module-definition.ts`**

```typescript
export const { ConfigurableModuleClass, MODULE_OPTIONS_TOKEN } =
  new ConfigurableModuleBuilder<ConfigModuleOptions>().setClassMethodName('forRoot').build();
```

Такая конструкция укажет `ConfigurableModuleBuilder` сгенерировать класс, который вместо них предоставляет `forRoot` и `forRootAsync`. Пример:

```typescript
@Module({
  imports: [
    ConfigModule.forRoot({ folder: './config' }), // <-- обратите внимание: используется "forRoot" вместо "register"
    // или как вариант:
    // ConfigModule.forRootAsync({
    //   useFactory: () => {
    //     return {
    //       folder: './config',
    //     }
    //   },
    //   inject: [...любые дополнительные зависимости...]
    // }),
  ],
})
export class AppModule {}
```

#### Собственный класс-фабрика параметров

Поскольку метод `registerAsync` (или `forRootAsync`, или любое другое имя — в зависимости от конфигурации) позволяет потребителю передать описание провайдера, разрешающегося в конфигурацию модуля, потребитель библиотеки может передать класс, который будет использоваться для построения объекта конфигурации.

```typescript
@Module({
  imports: [
    ConfigModule.registerAsync({
      useClass: ConfigModuleOptionsFactory,
    }),
  ],
})
export class AppModule {}
```

По умолчанию такой класс должен предоставлять метод `create()`, возвращающий объект конфигурации модуля. Однако если в вашей библиотеке принято другое соглашение об именовании, вы можете изменить это поведение и указать `ConfigurableModuleBuilder` ожидать другой метод, например `createConfigOptions`, с помощью метода `ConfigurableModuleBuilder#setFactoryMethodName`:

**`config.module-definition.ts`**

```typescript
export const { ConfigurableModuleClass, MODULE_OPTIONS_TOKEN } =
  new ConfigurableModuleBuilder<ConfigModuleOptions>().setFactoryMethodName('createConfigOptions').build();
```

Теперь класс `ConfigModuleOptionsFactory` должен предоставлять метод `createConfigOptions` (вместо `create`):

```typescript
@Module({
  imports: [
    ConfigModule.registerAsync({
      useClass: ConfigModuleOptionsFactory, // <-- этот класс должен предоставлять метод "createConfigOptions"
    }),
  ],
})
export class AppModule {}
```

#### Дополнительные параметры

Бывают пограничные случаи, когда модулю нужны дополнительные параметры, определяющие его поведение (хороший пример такого параметра — флаг `isGlobal`, или просто `global`), которые при этом не должны попадать в провайдер `MODULE_OPTIONS_TOKEN` (поскольку они не важны сервисам и провайдерам, зарегистрированным внутри модуля: например, `ConfigService` незачем знать, зарегистрирован ли его модуль-владелец как глобальный).

В таких случаях можно использовать метод `ConfigurableModuleBuilder#setExtras`. Посмотрите пример:

```typescript
export const { ConfigurableModuleClass, MODULE_OPTIONS_TOKEN } =
  new ConfigurableModuleBuilder<ConfigModuleOptions>()
    .setExtras(
      {
        isGlobal: true,
      },
      (definition, extras) => ({
        ...definition,
        global: extras.isGlobal,
      }),
    )
    .build();
```

В примере выше первый аргумент метода `setExtras` — объект со значениями по умолчанию для «дополнительных» свойств. Второй аргумент — функция, принимающая автоматически сгенерированное определение модуля (с `provider`, `exports` и т. д.) и объект `extras`, представляющий дополнительные свойства (заданные потребителем либо взятые по умолчанию). Возвращаемое значение этой функции — изменённое определение модуля. В данном случае мы берём свойство `extras.isGlobal` и присваиваем его свойству `global` определения модуля (которое, в свою очередь, определяет, является ли модуль глобальным; подробнее [здесь](<../02. Overview/04. Modules.md#динамические-модули>)).

Теперь при использовании этого модуля можно передать дополнительный флаг `isGlobal`:

```typescript
@Module({
  imports: [
    ConfigModule.register({
      isGlobal: true,
      folder: './config',
    }),
  ],
})
export class AppModule {}
```

Однако, поскольку `isGlobal` объявлен как «дополнительное» свойство, он не будет доступен в провайдере `MODULE_OPTIONS_TOKEN`:

```typescript
@Injectable()
export class ConfigService {
  constructor(
    @Inject(MODULE_OPTIONS_TOKEN) private options: ConfigModuleOptions,
  ) {
    // в объекте "options" не будет свойства "isGlobal"
    // ...
  }
}
```

#### Расширение автоматически сгенерированных методов

Автоматически сгенерированные статические методы (`register`, `registerAsync` и т. д.) при необходимости можно расширить:

```typescript
import { Module } from '@nestjs/common';
import { ConfigService } from './config.service';
import {
  ConfigurableModuleClass,
  ASYNC_OPTIONS_TYPE,
  OPTIONS_TYPE,
} from './config.module-definition';

@Module({
  providers: [ConfigService],
  exports: [ConfigService],
})
export class ConfigModule extends ConfigurableModuleClass {
  static register(options: typeof OPTIONS_TYPE): DynamicModule {
    return {
      // здесь ваша собственная логика
      ...super.register(options),
    };
  }

  static registerAsync(options: typeof ASYNC_OPTIONS_TYPE): DynamicModule {
    return {
      // здесь ваша собственная логика
      ...super.registerAsync(options),
    };
  }
}
```

Обратите внимание на использование типов `OPTIONS_TYPE` и `ASYNC_OPTIONS_TYPE`, которые должны экспортироваться из файла определения модуля:

```typescript
export const {
  ConfigurableModuleClass,
  MODULE_OPTIONS_TOKEN,
  OPTIONS_TYPE,
  ASYNC_OPTIONS_TYPE,
} = new ConfigurableModuleBuilder<ConfigModuleOptions>().build();
```
