### Области внедрения

Тем, кто пришёл из других языков программирования, может показаться неожиданным, что в Nest почти всё разделяется между входящими запросами. У нас есть пул соединений с базой данных, сервисы-синглтоны с глобальным состоянием и так далее. Помните, что Node.js не следует многопоточной модели запрос/ответ без состояния, в которой каждый запрос обрабатывается отдельным потоком. Поэтому использование экземпляров-синглтонов для наших приложений полностью **безопасно**.

Тем не менее бывают пограничные случаи, когда время жизни на уровне запроса — именно то, что нужно: например, кэширование в пределах запроса в GraphQL-приложениях, трассировка запросов и мультиарендность. Области внедрения дают механизм, позволяющий получить нужное поведение времени жизни провайдера.

#### Область провайдера

Провайдер может иметь любую из следующих областей:

<table>
  <tr>
    <td><code>DEFAULT</code></td>
    <td>Единственный экземпляр провайдера разделяется всем приложением. Время жизни экземпляра напрямую привязано к жизненному циклу приложения. После начальной загрузки приложения все провайдеры-синглтоны уже созданы. Область синглтона используется по умолчанию.</td>
  </tr>
  <tr>
    <td><code>REQUEST</code></td>
    <td>Новый экземпляр провайдера создаётся отдельно для каждого входящего <strong>запроса</strong>. После завершения обработки запроса экземпляр собирается сборщиком мусора.</td>
  </tr>
  <tr>
    <td><code>TRANSIENT</code></td>
    <td>Transient-провайдеры не разделяются между потребителями. Каждый потребитель, внедряющий такой провайдер, получает новый, персональный экземпляр.</td>
  </tr>
</table>

> **Подсказка** Для большинства сценариев **рекомендуется** область синглтона. Разделение провайдеров между потребителями и между запросами означает, что экземпляр можно закэшировать, а его инициализация произойдёт только один раз — при старте приложения.

#### Использование

Область внедрения задаётся свойством `scope` в объекте опций декоратора `@Injectable()`:

```typescript
import { Injectable, Scope } from '@nestjs/common';

@Injectable({ scope: Scope.REQUEST })
export class CatsService {}
```

Аналогично для [пользовательских провайдеров](<01. Custom providers.md>) свойство `scope` указывается в развёрнутой форме регистрации провайдера:

```typescript
{
  provide: 'CACHE_MANAGER',
  useClass: CacheManager,
  scope: Scope.TRANSIENT,
}
```

> **Подсказка** Перечисление `Scope` импортируется из `@nestjs/common`

Область синглтона используется по умолчанию и объявлять её не нужно. Если вы всё же хотите явно объявить провайдер синглтоном, укажите в свойстве `scope` значение `Scope.DEFAULT`.

> **Важно** WebSocket-шлюзы не должны использовать провайдеры с областью запроса, поскольку обязаны вести себя как синглтоны. Каждый шлюз инкапсулирует реальный сокет и не может быть создан в нескольких экземплярах. Это ограничение распространяется и на некоторые другие провайдеры, например [_стратегии Passport_](<../13. Recipes/04. Passport (auth).md#стратегии-с-областью-запроса>) или _cron-контроллеры_.

#### Область контроллера

Контроллеры тоже могут иметь область, которая распространяется на все объявленные в них обработчики методов запроса. Как и в случае провайдеров, область контроллера определяет его время жизни. Для контроллера с областью запроса новый экземпляр создаётся на каждый входящий запрос и собирается сборщиком мусора после завершения обработки запроса.

Область контроллера объявляется свойством `scope` объекта `ControllerOptions`:

```typescript
@Controller({
  path: 'cats',
  scope: Scope.REQUEST,
})
export class CatsController {}
```

#### Иерархия областей

Область `REQUEST` распространяется вверх по цепочке внедрения. Контроллер, зависящий от провайдера с областью запроса, сам становится провайдером с областью запроса.

Представьте такой граф зависимостей: `CatsController <- CatsService <- CatsRepository`. Если `CatsService` имеет область запроса (а остальные — синглтоны по умолчанию), `CatsController` тоже станет запросным, поскольку зависит от внедрённого сервиса. А `CatsRepository`, который ни от чего не зависит, останется синглтоном.

Зависимости с областью transient этому правилу не подчиняются. Если синглтон `DogsService` внедряет transient-провайдер `LoggerService`, он получит его новый экземпляр. Однако сам `DogsService` останется синглтоном, поэтому его внедрение где угодно _не_ приведёт к созданию нового экземпляра `DogsService`. Если такое поведение всё же нужно, `DogsService` необходимо явно пометить как `TRANSIENT`.

#### Провайдер запроса

В приложении на базе HTTP-сервера (например, с `@nestjs/platform-express` или `@nestjs/platform-fastify`) при работе с провайдерами области запроса может понадобиться доступ к ссылке на исходный объект запроса. Это делается внедрением объекта `REQUEST`.

Провайдер `REQUEST` по своей природе имеет область запроса, поэтому указывать область `REQUEST` явно не требуется. Более того, даже если вы это сделаете, указание будет проигнорировано. Любой провайдер, зависящий от провайдера с областью запроса, автоматически получает область запроса, и изменить это поведение нельзя.

```typescript
import { Injectable, Scope, Inject } from '@nestjs/common';
import { REQUEST } from '@nestjs/core';
import { Request } from 'express';

@Injectable({ scope: Scope.REQUEST })
export class CatsService {
  constructor(@Inject(REQUEST) private request: Request) {}
}
```

Из-за различий в нижележащих платформах и протоколах в микросервисных и GraphQL-приложениях доступ к входящему запросу устроен немного иначе. В [GraphQL](<../06. GraphQL/01. Quick start.md>)-приложениях вместо `REQUEST` внедряется `CONTEXT`:

```typescript
import { Injectable, Scope, Inject } from '@nestjs/common';
import { CONTEXT } from '@nestjs/graphql';

@Injectable({ scope: Scope.REQUEST })
export class CatsService {
  constructor(@Inject(CONTEXT) private context) {}
}
```

После этого вы настраиваете значение `context` (в `GraphQLModule`) так, чтобы оно содержало `request` в качестве свойства.

#### Провайдер INQUIRER

Если нужно узнать класс, в котором был сконструирован провайдер, — например, в провайдерах логирования или метрик, — можно внедрить токен `INQUIRER`.

```typescript
import { Inject, Injectable, Scope } from '@nestjs/common';
import { INQUIRER } from '@nestjs/core';

@Injectable({ scope: Scope.TRANSIENT })
export class HelloService {
  constructor(@Inject(INQUIRER) private parentClass: object) {}

  sayHello(message: string) {
    console.log(`${this.parentClass?.constructor?.name}: ${message}`);
  }
}
```

А затем использовать его так:

```typescript
import { Injectable } from '@nestjs/common';
import { HelloService } from './hello.service';

@Injectable()
export class AppService {
  constructor(private helloService: HelloService) {}

  getRoot(): string {
    this.helloService.sayHello('My name is getRoot');

    return 'Hello world!';
  }
}
```

В примере выше при вызове `AppService#getRoot` в консоль будет выведено `"AppService: My name is getRoot"`.

#### Производительность

Использование провайдеров с областью запроса влияет на производительность приложения. Хотя Nest старается кэшировать как можно больше метаданных, ему всё равно приходится создавать экземпляр вашего класса на каждый запрос. Из-за этого среднее время ответа и общие результаты нагрузочных тестов ухудшатся. Если провайдеру не обязательно иметь область запроса, настоятельно рекомендуется использовать область синглтона по умолчанию.

> **Подсказка** Хотя всё это звучит довольно пугающе, грамотно спроектированное приложение, использующее провайдеры с областью запроса, не должно терять по задержке больше ~5%.

#### Durable-провайдеры

Провайдеры с областью запроса, как сказано выше, могут увеличивать задержку: наличие хотя бы одного такого провайдера (внедрённого в экземпляр контроллера или глубже — в один из его провайдеров) делает запросным и сам контроллер. А значит, его придётся пересоздавать на каждый отдельный запрос (и затем собирать сборщиком мусора). То есть, скажем, при 30 тысячах параллельных запросов возникнет 30 тысяч эфемерных экземпляров контроллера (и его провайдеров с областью запроса).

Наличие общего провайдера, от которого зависит большинство остальных (представьте соединение с базой данных или сервис логирования), автоматически переводит все эти провайдеры в область запроса. Это может стать проблемой в **мультиарендных приложениях**, особенно там, где есть центральный провайдер «источника данных» с областью запроса, который берёт заголовки или токен из объекта запроса и по их значениям получает соответствующее соединение с базой или схему (для конкретного арендатора).

Например, предположим, что вашим приложением попеременно пользуются 10 разных клиентов. У каждого клиента **свой выделенный источник данных**, и нужно гарантировать, что клиент A никогда не доберётся до базы клиента B. Одно из решений — объявить провайдер «источника данных» с областью запроса, который по объекту запроса определяет «текущего клиента» и получает соответствующую ему базу. С таким подходом превратить приложение в мультиарендное можно за несколько минут. Но у него есть серьёзный недостаток: поскольку от провайдера «источника данных» почти наверняка зависит значительная часть компонентов приложения, все они неявно станут запросными, и вы, без сомнения, увидите влияние на производительность.

А если бы существовало решение получше? Раз клиентов всего 10, почему бы не иметь 10 отдельных [поддеревьев DI](<06. Module reference.md#разрешение-scoped-провайдеров>) — по одному на клиента (вместо пересоздания дерева на каждый запрос)? Если ваши провайдеры не зависят ни от одного свойства, действительно уникального для каждого следующего запроса (например, UUID запроса), а есть лишь некоторые атрибуты, по которым запросы можно агрегировать (классифицировать), то _пересоздавать поддерево DI_ на каждый входящий запрос незачем.

Вот тут и пригождаются **durable-провайдеры**.

Прежде чем помечать провайдеры как durable, нужно зарегистрировать **стратегию**, которая сообщит Nest, что считать «общими атрибутами запроса», и опишет логику группировки запросов — сопоставления их с соответствующими поддеревьями DI.

```typescript
import {
  HostComponentInfo,
  ContextId,
  ContextIdFactory,
  ContextIdStrategy,
} from '@nestjs/core';
import { Request } from 'express';

const tenants = new Map<string, ContextId>();

export class AggregateByTenantContextIdStrategy implements ContextIdStrategy {
  attach(contextId: ContextId, request: Request) {
    const tenantId = request.headers['x-tenant-id'] as string;
    let tenantSubTreeId: ContextId;

    if (tenants.has(tenantId)) {
      tenantSubTreeId = tenants.get(tenantId);
    } else {
      tenantSubTreeId = ContextIdFactory.create();
      tenants.set(tenantId, tenantSubTreeId);
    }

    // Если дерево не durable, возвращаем исходный объект "contextId"
    return (info: HostComponentInfo) =>
      info.isTreeDurable ? tenantSubTreeId : contextId;
  }
}
```

> **Подсказка** Как и область запроса, durability распространяется вверх по цепочке внедрения. Это значит, что если A зависит от B, помеченного как `durable`, то A неявно тоже становится durable (если только для провайдера A свойство `durable` не выставлено явно в `false`).

> **Предупреждение** Учтите, что эта стратегия не подходит для приложений, работающих с большим числом арендаторов.

Значение, возвращаемое методом `attach`, сообщает Nest, какой идентификатор контекста использовать для данного хоста. В нашем случае мы указали, что вместо исходного, автоматически сгенерированного объекта `contextId` должен использоваться `tenantSubTreeId`, когда хост-компонент (например, контроллер с областью запроса) помечен как durable (о том, как помечать провайдеры durable, — ниже). Кроме того, в примере выше **полезная нагрузка не регистрируется** (полезная нагрузка = провайдер `REQUEST`/`CONTEXT`, представляющий «корень» — родителя поддерева).

Если вы хотите зарегистрировать полезную нагрузку для durable-дерева, используйте вместо этого такую конструкцию:

```typescript
// Возвращаемое значение метода `AggregateByTenantContextIdStrategy#attach`:
return {
  resolve: (info: HostComponentInfo) =>
    info.isTreeDurable ? tenantSubTreeId : contextId,
  payload: { tenantId },
};
```

Теперь при каждом внедрении провайдера `REQUEST` (или `CONTEXT` для GraphQL-приложений) через `@Inject(REQUEST)`/`@Inject(CONTEXT)` будет внедряться объект `payload` (состоящий в данном случае из единственного свойства — `tenantId`).

Итак, когда стратегия готова, её можно зарегистрировать где-нибудь в коде (она всё равно применяется глобально) — например, в файле `main.ts`:

```typescript
ContextIdFactory.apply(new AggregateByTenantContextIdStrategy());
```

> **Подсказка** Класс `ContextIdFactory` импортируется из пакета `@nestjs/core`.

Пока регистрация происходит до того, как в приложение придёт первый запрос, всё будет работать как задумано.

Наконец, чтобы превратить обычный провайдер в durable, достаточно выставить флаг `durable` в `true` и изменить его область на `Scope.REQUEST` (это не нужно, если область REQUEST уже присутствует в цепочке внедрения):

```typescript
import { Injectable, Scope } from '@nestjs/common';

@Injectable({ scope: Scope.REQUEST, durable: true })
export class CatsService {}
```

Аналогично для [пользовательских провайдеров](<01. Custom providers.md>) свойство `durable` указывается в развёрнутой форме регистрации провайдера:

```typescript
{
  provide: 'foobar',
  useFactory: () => { ... },
  scope: Scope.REQUEST,
  durable: true,
}
```
