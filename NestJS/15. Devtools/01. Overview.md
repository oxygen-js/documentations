### Обзор

> **Подсказка** Эта глава посвящена интеграции Nest Devtools с фреймворком Nest. Если вам нужно само приложение Devtools, зайдите на сайт [Devtools](https://devtools.nestjs.com).

Чтобы начать отладку локального приложения, откройте файл `main.ts` и задайте атрибуту `snapshot` значение `true` в объекте опций приложения:

```typescript
async function bootstrap() {
  const app = await NestFactory.create(AppModule, {
    snapshot: true,
  });
  await app.listen(process.env.PORT ?? 3000);
}
```

Так фреймворк получит указание собирать метаданные, необходимые Nest Devtools для визуализации графа вашего приложения.

Далее установим требуемую зависимость:

```bash
$ npm i @nestjs/devtools-integration
```

> **Предупреждение** Если в приложении используется пакет `@nestjs/graphql`, обязательно установите его последнюю версию (`npm i @nestjs/graphql@11`).

Теперь, когда зависимость на месте, откроем файл `app.module.ts` и импортируем только что установленный `DevtoolsModule`:

```typescript
@Module({
  imports: [
    DevtoolsModule.register({
      http: process.env.NODE_ENV !== 'production',
    }),
  ],
  controllers: [AppController],
  providers: [AppService],
})
export class AppModule {}
```

> **Предупреждение** Переменная окружения `NODE_ENV` проверяется здесь потому, что этот модуль ни в коем случае нельзя использовать в production!

После импорта `DevtoolsModule` и запуска приложения (`npm run start:dev`) вы сможете перейти по адресу [Devtools](https://devtools.nestjs.com) и увидеть построенный граф.

<figure><img src="https://docs.nestjs.com/assets/devtools/modules-graph.png" /></figure>

> **Подсказка** Как видно на скриншоте выше, каждый модуль связан с `InternalCoreModule`. `InternalCoreModule` — это глобальный модуль, который всегда импортируется в корневой модуль. Поскольку он зарегистрирован как глобальный узел, Nest автоматически создаёт рёбра между всеми модулями и узлом `InternalCoreModule`. Если вы хотите скрыть глобальные модули на графе, воспользуйтесь флажком «**Hide global modules**» (в боковой панели).

Итак, `DevtoolsModule` заставляет ваше приложение поднять дополнительный HTTP-сервер (на порту 8000), который приложение Devtools использует для интроспекции.

Чтобы лишний раз убедиться, что всё работает как задумано, переключите отображение графа в режим «Classes». Вы должны увидеть такой экран:

<figure><img src="https://docs.nestjs.com/assets/devtools/classes-graph.png" /></figure>

Чтобы сосредоточиться на конкретном узле, щёлкните по прямоугольнику — на графе появится всплывающее окно с кнопкой **«Focus»**. Найти нужный узел можно и через строку поиска в боковой панели.

> **Подсказка** При нажатии кнопки **Inspect** приложение перенаправит вас на страницу `/debug` с выбранным узлом.

<figure><img src="https://docs.nestjs.com/assets/devtools/node-popup.png" /></figure>

> **Подсказка** Чтобы экспортировать граф в виде изображения, нажмите кнопку **Export as PNG** в правом углу графа.

С помощью элементов управления в боковой панели (слева) вы можете регулировать глубину связей и, например, визуализировать конкретное поддерево приложения:

<figure><img src="https://docs.nestjs.com/assets/devtools/subtree-view.png" /></figure>

Это особенно полезно, когда в команду приходят **новые разработчики** и вам нужно показать им устройство приложения. Ещё этой возможностью удобно пользоваться, чтобы визуализировать отдельный модуль (например, `TasksModule`) со всеми его зависимостями — например, при разбиении крупного приложения на модули поменьше (скажем, на отдельные микросервисы).

#### Разбор ошибки «Cannot resolve dependency»

> **Примечание** Эта возможность поддерживается начиная с `@nestjs/core` версии `v9.3.10`.

Пожалуй, самое частое сообщение об ошибке, которое вам доводилось видеть, — о том, что Nest не может разрешить зависимости провайдера. С Nest Devtools вы без труда определите причину проблемы и поймёте, как её устранить.

Сначала откройте файл `main.ts` и измените вызов `bootstrap()`:

```typescript
bootstrap().catch((err) => {
  fs.writeFileSync('graph.json', PartialGraphHost.toString() ?? '');
  process.exit(1);
});
```

Также обязательно задайте `abortOnError` значение `false`:

```typescript
const app = await NestFactory.create(AppModule, {
  snapshot: true,
  abortOnError: false, // <--- ВОТ ЭТО
});
```

Теперь каждый раз, когда приложение не сможет запуститься из-за ошибки **«Cannot resolve dependency»**, в корневом каталоге будет появляться файл `graph.json` (он представляет частичный граф). Этот файл можно перетащить в Devtools (не забудьте переключить текущий режим с «Interactive» на «Preview»):

<figure><img src="https://docs.nestjs.com/assets/devtools/drag-and-drop.png" /></figure>

После успешной загрузки вы увидите такой граф и диалоговое окно:

<figure><img src="https://docs.nestjs.com/assets/devtools/partial-graph-modules-view.png" /></figure>

Как видите, подсвеченный `TasksModule` — это именно то, что стоит изучить. Кроме того, в диалоговом окне уже приведены рекомендации по устранению проблемы.

Если переключиться на представление «Classes», картина будет такой:

<figure><img src="https://docs.nestjs.com/assets/devtools/partial-graph-classes-view.png" /></figure>

Этот граф показывает, что `DiagnosticsService`, который мы хотим внедрить в `TasksService`, не найден в контексте модуля `TasksModule`, и для исправления, скорее всего, достаточно импортировать `DiagnosticsModule` в `TasksModule`!

#### Обозреватель маршрутов

Перейдя на страницу **Routes explorer**, вы увидите все зарегистрированные точки входа:

<figure><img src="https://docs.nestjs.com/assets/devtools/routes.png" /></figure>

> **Подсказка** На этой странице отображаются не только HTTP-маршруты, но и все остальные точки входа (например, WebSockets, gRPC, резолверы GraphQL и т. д.).

Точки входа сгруппированы по контроллерам, которым они принадлежат. Найти конкретную точку входа можно через строку поиска.

Если щёлкнуть по конкретной точке входа, отобразится **граф потока выполнения**. Он показывает порядок выполнения для этой точки входа (например, guard-ы, interceptor-ы, pipe-ы и т. д., привязанные к данному маршруту). Это особенно полезно, когда нужно понять, как выглядит цикл «запрос — ответ» для конкретного маршрута, или разобраться, почему тот или иной guard/interceptor/pipe не выполняется.

#### Песочница

Чтобы выполнять JavaScript-код на лету и взаимодействовать с приложением в реальном времени, перейдите на страницу **Sandbox**:

<figure><img src="https://docs.nestjs.com/assets/devtools/sandbox.png" /></figure>

Песочницу можно использовать для тестирования и отладки эндпоинтов API **в реальном времени**: разработчики быстро находят и исправляют проблемы, не прибегая, например, к HTTP-клиенту. Также можно обойти слой аутентификации, так что лишний шаг с входом в систему или отдельная тестовая учётная запись больше не нужны. В событийно-ориентированных приложениях события можно инициировать прямо из песочницы и наблюдать за реакцией приложения.

Всё, что попадает в лог, направляется в консоль песочницы, поэтому происходящее хорошо видно.

Просто выполняйте код **на лету** и мгновенно получайте результат — без пересборки приложения и перезапуска сервера.

#### Просмотр статических файлов

Чтобы сохранить сериализованный граф в файл, используйте следующий код:

```typescript
await app.listen(process.env.PORT ?? 3000); // ИЛИ await app.init()
fs.writeFileSync('./graph.json', app.get(SerializedGraph).toString());
```

> **Подсказка** `SerializedGraph` экспортируется из пакета `@nestjs/core`.

Затем этот файл можно перетащить или загрузить:

<figure><img src="https://docs.nestjs.com/assets/devtools/drag-and-drop.png" /></figure>

Это удобно, когда нужно поделиться графом с кем-то ещё (например, с коллегой) или проанализировать его офлайн.
