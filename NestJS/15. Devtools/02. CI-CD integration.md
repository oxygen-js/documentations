#### Публикация графов

Сначала настроим файл начальной загрузки приложения (`main.ts`) на использование класса `GraphPublisher` (экспортируется из `@nestjs/devtools-integration` — подробнее см. предыдущую главу):

```typescript
async function bootstrap() {
  const shouldPublishGraph = process.env.PUBLISH_GRAPH === "true";

  const app = await NestFactory.create(AppModule, {
    snapshot: true,
    preview: shouldPublishGraph,
  });

  if (shouldPublishGraph) {
    await app.init();

    const publishOptions = { ... } // ПРИМЕЧАНИЕ: содержимое этого объекта опций зависит от используемого CI/CD-провайдера
    const graphPublisher = new GraphPublisher(app);
    await graphPublisher.publish(publishOptions);

    await app.close();
  } else {
    await app.listen(process.env.PORT ?? 3000);
  }
}
```

Как видно, здесь мы используем `GraphPublisher`, чтобы опубликовать сериализованный граф в централизованном реестре. `PUBLISH_GRAPH` — это пользовательская переменная окружения, позволяющая управлять тем, публиковать граф (сценарий CI/CD) или нет (обычный запуск приложения). Кроме того, мы устанавливаем атрибут `preview` в `true`. С этим флагом приложение запускается в режиме предпросмотра: конструкторы (и хуки жизненного цикла) всех контроллеров, enhancer-ов и провайдеров приложения выполняться не будут. Обратите внимание: это **не обязательно**, но упрощает жизнь, поскольку в таком случае нам не придётся подключаться к базе данных и тому подобному при запуске приложения в конвейере CI/CD.

Содержимое объекта `publishOptions` зависит от используемого CI/CD-провайдера. Инструкции для самых популярных провайдеров приведены ниже, в следующих разделах.

#### Отчёты

Devtools генерируют отчёт для каждой сборки, **ЕСЛИ** в централизованном реестре уже сохранён соответствующий снимок. Например, если вы создаёте PR в ветку `master`, для которой граф уже был опубликован, приложение сможет обнаружить различия и сгенерировать отчёт. В противном случае отчёт создан не будет.

Чтобы посмотреть отчёты, перейдите на соответствующую страницу проекта (см. организации).

<figure><img src="https://docs.nestjs.com/assets/devtools/report.png" /></figure>

Это особенно полезно для выявления изменений, которые могли остаться незамеченными при код-ревью. Допустим, кто-то изменил область видимости **глубоко вложенного провайдера**. Такое изменение может быть неочевидно для ревьюера, но с Devtools мы легко его заметим и убедимся, что оно было сделано намеренно. Или, если мы убрали guard с конкретного эндпоинта, он будет помечен в отчёте как затронутый. А если для этого маршрута нет интеграционных или e2e-тестов, мы можем и не заметить, что он больше не защищён, — а когда заметим, будет уже поздно.

Аналогично: если мы работаем с **большой кодовой базой** и делаем модуль глобальным, мы увидим, сколько рёбер добавилось в граф, а это в большинстве случаев признак того, что что-то идёт не так.

#### Предпросмотр сборки

Для каждого опубликованного графа можно вернуться назад во времени и посмотреть, как он выглядел раньше, нажав кнопку **Preview**. Более того, если отчёт был сгенерирован, различия будут подсвечены прямо на графе:

- зелёные узлы — добавленные элементы
- светло-белые узлы — изменённые элементы
- красные узлы — удалённые элементы

См. скриншот ниже:

<figure><img src="https://docs.nestjs.com/assets/devtools/nodes-selection.png" /></figure>

Возможность вернуться назад во времени позволяет исследовать и диагностировать проблему, сравнивая текущий граф с предыдущим. В зависимости от вашей настройки каждому pull request (или даже каждому коммиту) будет соответствовать снимок в реестре, так что вы легко сможете посмотреть, что изменилось. Считайте Devtools своего рода Git, который понимает, как Nest строит граф вашего приложения, и умеет его **визуализировать**.

#### Интеграции: GitHub Actions

Начнём с создания нового GitHub workflow в каталоге `.github/workflows` нашего проекта — назовём его, например, `publish-graph.yml`. Внутри этого файла используем следующее определение:

```yaml
name: Devtools

on:
  push:
    branches:
      - master
  pull_request:
    branches:
      - '*'

jobs:
  publish:
    if: github.actor!= 'dependabot[bot]'
    name: Publish graph
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '16'
          cache: 'npm'
      - name: Install dependencies
        run: npm ci
      - name: Setup Environment (PR)
        if: ${{ github.event_name == 'pull_request' }}
        shell: bash
        run: |
          echo "COMMIT_SHA=${{ github.event.pull_request.head.sha }}" >>\${GITHUB_ENV}
      - name: Setup Environment (Push)
        if: ${{ github.event_name == 'push' }}
        shell: bash
        run: |
          echo "COMMIT_SHA=\${GITHUB_SHA}" >> \${GITHUB_ENV}
      - name: Publish
        run: PUBLISH_GRAPH=true npm run start
        env:
          DEVTOOLS_API_KEY: CHANGE_THIS_TO_YOUR_API_KEY
          REPOSITORY_NAME: ${{ github.event.repository.name }}
          BRANCH_NAME: ${{ github.head_ref || github.ref_name }}
          TARGET_SHA: ${{ github.event.pull_request.base.sha }}
```

В идеале переменную окружения `DEVTOOLS_API_KEY` следует получать из GitHub Secrets, подробнее об этом [здесь](https://docs.github.com/en/actions/security-guides/encrypted-secrets#creating-encrypted-secrets-for-a-repository) .

Этот workflow будет запускаться для каждого pull request, нацеленного на ветку `master`, либо при прямом коммите в `master`. Свободно подстраивайте конфигурацию под нужды вашего проекта. Главное — предоставить необходимые переменные окружения, чтобы класс `GraphPublisher` мог работать.

Однако есть одна переменная, которую нужно обновить, прежде чем этот workflow заработает, — `DEVTOOLS_API_KEY`. Сгенерировать выделенный API-ключ для вашего проекта можно на этой [странице](https://devtools.nestjs.com/settings/manage-api-keys).

Наконец, вернёмся к файлу `main.ts` и заполним объект `publishOptions`, который ранее оставили пустым.

```typescript
const publishOptions = {
  apiKey: process.env.DEVTOOLS_API_KEY,
  repository: process.env.REPOSITORY_NAME,
  owner: process.env.GITHUB_REPOSITORY_OWNER,
  sha: process.env.COMMIT_SHA,
  target: process.env.TARGET_SHA,
  trigger: process.env.GITHUB_BASE_REF ? 'pull' : 'push',
  branch: process.env.BRANCH_NAME,
};
```

Для максимально удобной работы обязательно подключите к проекту **приложение GitHub**, нажав кнопку «Integrate GitHub app» (см. скриншот ниже). Обратите внимание: это не обязательно.

#### Интеграции: Gitlab Pipelines

Начнём с создания нового конфигурационного файла Gitlab CI в корневом каталоге проекта — назовём его, например, `.gitlab-ci.yml`. Внутри этого файла используем следующее определение:

```typescript
const publishOptions = {
  apiKey: process.env.DEVTOOLS_API_KEY,
  repository: process.env.REPOSITORY_NAME,
  owner: process.env.GITHUB_REPOSITORY_OWNER,
  sha: process.env.COMMIT_SHA,
  target: process.env.TARGET_SHA,
  trigger: process.env.GITHUB_BASE_REF ? 'pull' : 'push',
  branch: process.env.BRANCH_NAME,
};
```

> **Подсказка** В идеале переменную окружения `DEVTOOLS_API_KEY` следует получать из секретов.

Этот workflow будет запускаться для каждого pull request, нацеленного на ветку `master`, либо при прямом коммите в `master`. Свободно подстраивайте конфигурацию под нужды вашего проекта. Главное — предоставить необходимые переменные окружения, чтобы класс `GraphPublisher` мог работать.

Однако есть одна переменная (в этом определении workflow), которую нужно обновить, прежде чем он заработает, — `DEVTOOLS_API_KEY`. Сгенерировать выделенный API-ключ для вашего проекта можно на этой **странице** .

Наконец, вернёмся к файлу `main.ts` и заполним объект `publishOptions`, который ранее оставили пустым.

```yaml
image: node:16

stages:
  - build

cache:
  key:
    files:
      - package-lock.json
  paths:
    - node_modules/

workflow:
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
      when: always
    - if: $CI_COMMIT_BRANCH == "master" && $CI_PIPELINE_SOURCE == "push"
      when: always
    - when: never

install_dependencies:
  stage: build
  script:
    - npm ci

publish_graph:
  stage: build
  needs:
    - install_dependencies
  script: npm run start
  variables:
    PUBLISH_GRAPH: 'true'
    DEVTOOLS_API_KEY: 'CHANGE_THIS_TO_YOUR_API_KEY'
```

#### Другие инструменты CI/CD

Интеграцию Nest Devtools с CI/CD можно использовать с любым инструментом CI/CD на ваш выбор (например, [Bitbucket Pipelines](https://bitbucket.org/product/features/pipelines) , [CircleCI](https://circleci.com/) и т. д.), так что не ограничивайтесь описанными здесь провайдерами.

Взгляните на следующую конфигурацию объекта `publishOptions`, чтобы понять, какая информация нужна для публикации графа для конкретного коммита, сборки или PR.

```typescript
const publishOptions = {
  apiKey: process.env.DEVTOOLS_API_KEY,
  repository: process.env.CI_PROJECT_NAME,
  owner: process.env.CI_PROJECT_ROOT_NAMESPACE,
  sha: process.env.CI_COMMIT_SHA,
  target: process.env.CI_MERGE_REQUEST_DIFF_BASE_SHA,
  trigger: process.env.CI_MERGE_REQUEST_DIFF_BASE_SHA ? 'pull' : 'push',
  branch: process.env.CI_COMMIT_BRANCH ?? process.env.CI_MERGE_REQUEST_SOURCE_BRANCH_NAME,
};
```

Большая часть этой информации предоставляется через встроенные переменные окружения CI/CD (см. [список встроенных переменных окружения CircleCI](https://circleci.com/docs/variables/#built-in-environment-variables) и [переменные Bitbucket](https://support.atlassian.com/bitbucket-cloud/docs/variables-and-secrets/) ).

Что касается конфигурации конвейера для публикации графов, мы рекомендуем использовать следующие триггеры:

- событие `push` — только если текущая ветка представляет среду развёртывания, например `master`, `main`, `staging`, `production` и т. д.
- событие `pull request` — всегда либо когда **целевая ветка** представляет среду развёртывания (см. выше)
