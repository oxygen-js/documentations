### CQRS

Ход работы простого [CRUD](https://en.wikipedia.org/wiki/Create,_read,_update_and_delete)-приложения (Create, Read, Update and Delete) можно описать так:

1. Слой контроллеров обрабатывает HTTP-запросы и делегирует задачи слою сервисов.
2. В слое сервисов сосредоточена большая часть бизнес-логики.
3. Сервисы используют репозитории / DAO для изменения и сохранения сущностей.
4. Сущности выступают контейнерами для значений, с сеттерами и геттерами.

Такой подход обычно достаточен для небольших и средних приложений, но для крупных и более сложных он может оказаться не лучшим выбором. В таких случаях модель **CQRS** (Command and Query Responsibility Segregation — разделение ответственности команд и запросов) может быть более подходящей и масштабируемой (в зависимости от требований приложения). Преимущества этой модели:

- **Разделение ответственности**. Модель разносит операции чтения и записи по отдельным моделям.
- **Масштабируемость**. Операции чтения и записи можно масштабировать независимо друг от друга.
- **Гибкость**. Модель позволяет использовать разные хранилища данных для чтения и для записи.
- **Производительность**. Модель позволяет использовать хранилища данных, оптимизированные под чтение и под запись по отдельности.

Для поддержки этой модели Nest предоставляет легковесный [модуль CQRS](https://github.com/nestjs/cqrs). В этой главе описано, как им пользоваться.

#### Установка

Сначала установите нужный пакет:

```bash
$ npm install --save @nestjs/cqrs
```

После установки перейдите в корневой модуль приложения (обычно `AppModule`) и импортируйте `CqrsModule.forRoot()`:

```typescript
import { Module } from '@nestjs/common';
import { CqrsModule } from '@nestjs/cqrs';

@Module({
  imports: [CqrsModule.forRoot()],
})
export class AppModule {}
```

Этот модуль принимает необязательный объект конфигурации. Доступны следующие параметры:

| Свойство                      | Описание                                                                                                                     | По умолчанию                      |
| ----------------------------- | ---------------------------------------------------------------------------------------------------------------------------- | --------------------------------- |
| `commandPublisher`            | Издатель, отвечающий за отправку команд в систему.                                                                           | `DefaultCommandPubSub`            |
| `eventPublisher`              | Издатель, публикующий события, чтобы их можно было разослать или обработать.                                                  | `DefaultPubSub`                   |
| `queryPublisher`              | Издатель, публикующий запросы, которые могут запускать операции получения данных.                                            | `DefaultQueryPubSub`              |
| `unhandledExceptionPublisher` | Издатель, отвечающий за обработку необработанных исключений и обеспечивающий их отслеживание и передачу.                     | `DefaultUnhandledExceptionPubSub` |
| `eventIdProvider`             | Сервис, выдающий уникальные идентификаторы событий: он их генерирует или извлекает из экземпляров событий.                    | `DefaultEventIdProvider`          |
| `rethrowUnhandled`            | Определяет, нужно ли повторно выбрасывать необработанные исключения после их обработки; полезно при отладке и учёте ошибок.  | `false`                           |

#### Команды

Команды используются для изменения состояния приложения. Они должны быть ориентированы на задачу, а не на данные. Когда команда отправлена, её обрабатывает соответствующий **обработчик команды** (Command Handler). Обработчик отвечает за обновление состояния приложения.

**`heroes-game.service.ts`**

```typescript
@Injectable()
export class HeroesGameService {
  constructor(private commandBus: CommandBus) {}

  async killDragon(heroId: string, killDragonDto: KillDragonDto) {
    return this.commandBus.execute(
      new KillDragonCommand(heroId, killDragonDto.dragonId)
    );
  }
}
```

В приведённом фрагменте мы создаём экземпляр класса `KillDragonCommand` и передаём его в метод `execute()` объекта `CommandBus`. Вот сам класс команды:

**`kill-dragon.command.ts`**

```typescript
export class KillDragonCommand extends Command<{
  actionId: string // Этот тип описывает результат выполнения команды
}> {
  constructor(
    public readonly heroId: string,
    public readonly dragonId: string,
  ) {
    super();
  }
}
```

Как видите, класс `KillDragonCommand` наследуется от класса `Command`. Класс `Command` — это простой вспомогательный класс, экспортируемый пакетом `@nestjs/cqrs`, который позволяет задать возвращаемый тип команды. В данном случае возвращаемый тип — объект со свойством `actionId`. Теперь при отправке команды `KillDragonCommand` возвращаемый тип метода `CommandBus#execute()` будет выведен как `Promise<{ actionId: string }>`. Это удобно, когда нужно вернуть какие-то данные из обработчика команды.

> **Подсказка** Наследование от класса `Command` необязательно. Оно нужно только в том случае, если вы хотите задать возвращаемый тип команды.

`CommandBus` представляет собой **поток** команд. Он отвечает за передачу команд соответствующим обработчикам. Метод `execute()` возвращает promise, который разрешается значением, возвращённым обработчиком.

Создадим обработчик для команды `KillDragonCommand`.

**`kill-dragon.handler.ts`**

```typescript
@CommandHandler(KillDragonCommand)
export class KillDragonHandler implements ICommandHandler<KillDragonCommand> {
  constructor(private repository: HeroesRepository) {}

  async execute(command: KillDragonCommand) {
    const { heroId, dragonId } = command;
    const hero = this.repository.findOneById(+heroId);

    hero.killEnemy(dragonId);
    await this.repository.persist(hero);

    // "ICommandHandler<KillDragonCommand>" обязывает вернуть значение, соответствующее возвращаемому типу команды
    return {
      actionId: crypto.randomUUID(), // Это значение будет возвращено вызывающей стороне
    }
  }
}
```

Этот обработчик получает сущность `Hero` из репозитория, вызывает метод `killEnemy()` и затем сохраняет изменения. Класс `KillDragonHandler` реализует интерфейс `ICommandHandler`, который требует реализации метода `execute()`. Метод `execute()` получает объект команды в качестве аргумента.

Обратите внимание, что `ICommandHandler<KillDragonCommand>` обязывает вернуть значение, соответствующее возвращаемому типу команды. В данном случае это объект со свойством `actionId`. Это касается только команд, унаследованных от класса `Command`. В остальных случаях вы можете возвращать что угодно.

Наконец, не забудьте зарегистрировать `KillDragonHandler` как провайдер в модуле:

```typescript
providers: [KillDragonHandler];
```

#### Запросы

Запросы используются для получения данных из состояния приложения. Они должны быть ориентированы на данные, а не на задачу. Когда запрос отправлен, его обрабатывает соответствующий **обработчик запроса** (Query Handler). Обработчик отвечает за получение данных.

`QueryBus` устроен по тому же принципу, что и `CommandBus`. Обработчики запросов должны реализовывать интерфейс `IQueryHandler` и быть помечены декоратором `@QueryHandler()`. Пример:

```typescript
export class GetHeroQuery extends Query<Hero> {
  constructor(public readonly heroId: string) {}
}
```

Подобно классу `Command`, класс `Query` — это простой вспомогательный класс, экспортируемый пакетом `@nestjs/cqrs`, который позволяет задать возвращаемый тип запроса. В данном случае возвращаемый тип — объект `Hero`. Теперь при отправке запроса `GetHeroQuery` возвращаемый тип метода `QueryBus#execute()` будет выведен как `Promise<Hero>`.

Чтобы получить героя, нужно создать обработчик запроса:

**`get-hero.handler.ts`**

```typescript
@QueryHandler(GetHeroQuery)
export class GetHeroHandler implements IQueryHandler<GetHeroQuery> {
  constructor(private repository: HeroesRepository) {}

  async execute(query: GetHeroQuery) {
    return this.repository.findOneById(query.heroId);
  }
}
```

Класс `GetHeroHandler` реализует интерфейс `IQueryHandler`, который требует реализации метода `execute()`. Метод `execute()` получает объект запроса в качестве аргумента и должен вернуть данные, соответствующие возвращаемому типу запроса (в данном случае — объект `Hero`).

Наконец, не забудьте зарегистрировать `GetHeroHandler` как провайдер в модуле:

```typescript
providers: [GetHeroHandler];
```

Теперь, чтобы отправить запрос, воспользуйтесь `QueryBus`:

```typescript
const hero = await this.queryBus.execute(new GetHeroQuery(heroId)); // тип "hero" будет автоматически выведен как "Hero"
```

#### События

События используются, чтобы уведомлять другие части приложения об изменениях его состояния. Они отправляются **моделями** или напрямую через `EventBus`. Когда событие отправлено, его обрабатывают соответствующие **обработчики событий** (Event Handlers). Обработчики могут, например, обновлять модель чтения.

Для примера создадим класс события:

**`hero-killed-dragon.event.ts`**

```typescript
export class HeroKilledDragonEvent {
  constructor(
    public readonly heroId: string,
    public readonly dragonId: string,
  ) {}
}
```

События можно отправлять напрямую методом `EventBus.publish()`, но их также можно отправлять из модели. Обновим модель `Hero` так, чтобы она отправляла событие `HeroKilledDragonEvent` при вызове метода `killEnemy()`.

**`hero.model.ts`**

```typescript
export class Hero extends AggregateRoot {
  constructor(private id: string) {
    super();
  }

  killEnemy(enemyId: string) {
    // Бизнес-логика
    this.apply(new HeroKilledDragonEvent(this.id, enemyId));
  }
}
```

Метод `apply()` используется для отправки событий. Он принимает объект события в качестве аргумента. Однако, поскольку наша модель ничего не знает об `EventBus`, её нужно связать с ним. Сделать это можно с помощью класса `EventPublisher`.

**`kill-dragon.handler.ts`**

```typescript
@CommandHandler(KillDragonCommand)
export class KillDragonHandler implements ICommandHandler<KillDragonCommand> {
  constructor(
    private repository: HeroesRepository,
    private publisher: EventPublisher,
  ) {}

  async execute(command: KillDragonCommand) {
    const { heroId, dragonId } = command;
    const hero = this.publisher.mergeObjectContext(
      await this.repository.findOneById(+heroId),
    );
    hero.killEnemy(dragonId);
    hero.commit();
  }
}
```

Метод `EventPublisher#mergeObjectContext` встраивает издателя событий в переданный объект. Этот объект должен реализовывать интерфейс `IAggregateRoot` (или наследоваться от класса `AggregateRoot`). После встраивания объект сможет публиковать события в поток событий.

Обратите внимание, что в этом примере мы также вызываем у модели метод `commit()`. Он используется для отправки всех накопленных событий. Чтобы события отправлялись автоматически, можно задать свойству `autoCommit` значение `true`:

```typescript
export class Hero extends AggregateRoot {
  constructor(private id: string) {
    super();
    this.autoCommit = true;
  }
}
```

Если издателя событий нужно встроить не в существующий объект, а в класс, можно воспользоваться методом `EventPublisher#mergeClassContext`:

```typescript
const HeroModel = this.publisher.mergeClassContext(Hero);
const hero = new HeroModel('id'); // <-- HeroModel — это класс
```

Теперь каждый экземпляр класса `HeroModel` сможет публиковать события без использования метода `mergeObjectContext()`.

#### Гибкие корни агрегатов

Класс `AggregateRoot` — это конкретная реализация, от которой можно наследоваться, чтобы добавить моделям предметной области событийные возможности. Однако при таком подходе сущности домена обязаны напрямую наследоваться от `AggregateRoot`, что может стать ограничением, если в вашем приложении уже сложилась своя иерархия наследования сущностей (например, базовый класс `Entity` или предметно-ориентированные базовые классы вроде `Monster`, `Vehicle` и т. п.).

Ради большей гибкости пакет `@nestjs/cqrs` предлагает три разных подхода к реализации корней агрегатов.

**Подход 1: традиционный (наследование класса)**

Это стандартный подход, показанный в предыдущих примерах. Он отлично работает в простых сценариях и в проектах, создаваемых с нуля.

```typescript
export class Hero extends AggregateRoot {
  constructor(private id: string) {
    super();
  }

  killEnemy(enemyId: string) {
    this.apply(new HeroKilledDragonEvent(this.id, enemyId));
  }
}
```

**Подход 2: миксин (для существующих иерархий)**

Если у вас уже есть базовый класс и наследоваться от `AggregateRoot` напрямую нельзя, можно воспользоваться функцией-миксином `WithAggregateRoot<EventBase, TBase>()`. Она позволяет добавить поведение корня агрегата к любому существующему базовому классу.

**`dragon.model.ts`**

```typescript
abstract class Monster {
  constructor(protected readonly id: string) {}
  abstract roar(): void;
}

export class Dragon extends WithAggregateRoot(Monster) {
  roar(): void {
    console.log('Roarrrr!');
  }

  die(): void {
    this.roar();
    this.apply(new DragonDiedEvent(this.id)); // Теперь доступно благодаря миксину!
  }
}
```

**Подход 3: собственная реализация**

Для максимального контроля — или если вы хотите сохранить слой предметной области полностью независимым от фреймворка — можно реализовать интерфейс `IAggregateRoot` напрямую. `EventPublisher` принимает любой объект, реализующий этот интерфейс.

```typescript
export class CustomEntity implements IAggregateRoot {
  private events: IEvent[] = [];

  getUncommittedEvents() {
    return this.events;
  }

  publish(event: IEvent) {
    // собственная логика
  }

  commit() {
    // собственная логика
  }

  uncommit() {
    // собственная логика
  }

  apply(event: IEvent) {
    this.events.push(event);
  }

  loadFromHistory(history: IEvent[]) {
    // собственная логика
  }
}
```

Все три подхода без проблем работают с `EventPublisher`, который принимает любой объект, реализующий интерфейс `IAggregateRoot`.

#### Ручная публикация событий

```typescript
this.eventBus.publish(new HeroKilledDragonEvent());
```

> **Подсказка** `EventBus` — внедряемый класс.

У каждого события может быть несколько **обработчиков событий**.

**`hero-killed-dragon.handler.ts`**

```typescript
@EventsHandler(HeroKilledDragonEvent)
export class HeroKilledDragonHandler implements IEventHandler<HeroKilledDragonEvent> {
  constructor(private repository: HeroesRepository) {}

  handle(event: HeroKilledDragonEvent) {
    // Бизнес-логика
  }
}
```

> **Подсказка** Учтите, что, начав использовать обработчики событий, вы выходите за пределы привычного контекста HTTP-запроса.
>
> - Ошибки в `CommandHandlers` по-прежнему могут быть перехвачены встроенными [фильтрами исключений](<../02. Overview/06. Exception filters.md>).
> - Ошибки в `EventHandlers` фильтрами исключений не перехватываются: обрабатывать их придётся вручную. Либо простым `try/catch`, либо через [саги](<12. CQRS.md#саги>), запуская компенсирующее событие, либо любым другим удобным вам способом.
> - HTTP-ответы из `CommandHandlers` по-прежнему можно отправить клиенту.
> - HTTP-ответы из `EventHandlers` — нельзя. Если нужно передать информацию клиенту, воспользуйтесь [WebSocket](<../07. WebSockets/01. Gateways.md>), [SSE](<../04. Techniques/20. Server-Sent Events.md>) или любым другим подходящим решением.

Как и в случае с командами и запросами, не забудьте зарегистрировать `HeroKilledDragonHandler` как провайдер в модуле:

```typescript
providers: [HeroKilledDragonHandler];
```

#### Саги

Сага — это длительный процесс, который слушает события и может запускать новые команды. Обычно она применяется для управления сложными рабочими процессами в приложении. Например, когда пользователь регистрируется, сага может слушать событие `UserRegisteredEvent` и отправлять пользователю приветственное письмо.

Саги — чрезвычайно мощная возможность. Одна сага может слушать от 1 до \* событий. С помощью библиотеки [RxJS](https://github.com/ReactiveX/rxjs) можно фильтровать, преобразовывать, разветвлять и объединять потоки событий, выстраивая сложные сценарии. Каждая сага возвращает Observable, порождающий экземпляр команды. Эта команда затем отправляется **асинхронно** через `CommandBus`.

Создадим сагу, которая слушает событие `HeroKilledDragonEvent` и отправляет команду `DropAncientItemCommand`.

**`heroes-game.saga.ts`**

```typescript
@Injectable()
export class HeroesGameSagas {
  @Saga()
  dragonKilled = (events$: Observable<any>): Observable<ICommand> => {
    return events$.pipe(
      ofType(HeroKilledDragonEvent),
      map((event) => new DropAncientItemCommand(event.heroId, fakeItemID)),
    );
  }
}
```

> **Подсказка** Оператор `ofType` и декоратор `@Saga()` экспортируются пакетом `@nestjs/cqrs`.

Декоратор `@Saga()` помечает метод как сагу. Аргумент `events$` — это Observable-поток всех событий. Оператор `ofType` фильтрует поток по указанному типу события. Оператор `map` преобразует событие в новый экземпляр команды.

В этом примере мы преобразуем `HeroKilledDragonEvent` в команду `DropAncientItemCommand`. Команда `DropAncientItemCommand` затем автоматически отправляется через `CommandBus`.

Как и в случае обработчиков запросов, команд и событий, не забудьте зарегистрировать `HeroesGameSagas` как провайдер в модуле:

```typescript
providers: [HeroesGameSagas];
```

#### Необработанные исключения

Обработчики событий выполняются асинхронно, поэтому они всегда должны корректно обрабатывать исключения, чтобы приложение не оказалось в несогласованном состоянии. Если исключение не обработано, `EventBus` создаст объект `UnhandledExceptionInfo` и поместит его в поток `UnhandledExceptionBus`. Этот поток представляет собой `Observable`, который можно использовать для обработки необработанных исключений.

```typescript
private destroy$ = new Subject<void>();

constructor(private unhandledExceptionsBus: UnhandledExceptionBus) {
  this.unhandledExceptionsBus
    .pipe(takeUntil(this.destroy$))
    .subscribe((exceptionInfo) => {
      // Здесь обрабатываем исключение
      // например, отправляем его во внешний сервис, завершаем процесс или публикуем новое событие
    });
}

onModuleDestroy() {
  this.destroy$.next();
  this.destroy$.complete();
}
```

Чтобы отфильтровать исключения, можно воспользоваться оператором `ofType`:

```typescript
this.unhandledExceptionsBus
  .pipe(
    takeUntil(this.destroy$),
    UnhandledExceptionBus.ofType(TransactionNotAllowedException),
  )
  .subscribe((exceptionInfo) => {
    // Здесь обрабатываем исключение
  });
```

Здесь `TransactionNotAllowedException` — то исключение, которое мы хотим отобрать.

Объект `UnhandledExceptionInfo` содержит следующие свойства:

```typescript
export interface UnhandledExceptionInfo<
  Cause = IEvent | ICommand,
  Exception = any,
> {
  /**
   * Выброшенное исключение.
   */
  exception: Exception;
  /**
   * Причина исключения (ссылка на событие или команду).
   */
  cause: Cause;
}
```

#### Подписка на все события

`CommandBus`, `QueryBus` и `EventBus` — все они являются **Observable**. Это значит, что можно подписаться на весь поток и, например, обработать все события: вывести их в консоль или сохранить в хранилище событий.

```typescript
private destroy$ = new Subject<void>();

constructor(private eventBus: EventBus) {
  this.eventBus
    .pipe(takeUntil(this.destroy$))
    .subscribe((event) => {
      // Сохраняем события в базу данных
    });
}

onModuleDestroy() {
  this.destroy$.next();
  this.destroy$.complete();
}
```

#### Область запроса

Тем, кто пришёл из других языков программирования, может показаться неожиданным, что в Nest большинство объектов разделяется между входящими запросами. Это касается пула соединений с базой данных, синглтон-сервисов с глобальным состоянием и многого другого. Помните, что Node.js не следует многопоточной модели «запрос/ответ» без состояния, где каждый запрос обрабатывается отдельным потоком. Поэтому использование экземпляров-синглтонов в наших приложениях **безопасно**.

Тем не менее есть пограничные случаи, когда время жизни обработчика, привязанное к запросу, может оказаться желательным. Например, кэширование в рамках запроса в GraphQL-приложениях, трассировка запросов или мультиарендность. Подробнее об управлении областями можно прочитать [здесь](<../03. Fundamentals/04. Injection scopes.md>).

Использование провайдеров с областью запроса вместе с CQRS может быть непростым, поскольку `CommandBus`, `QueryBus` и `EventBus` — синглтоны. К счастью, пакет `@nestjs/cqrs` упрощает задачу, автоматически создавая новый экземпляр обработчика с областью запроса для каждой обрабатываемой команды, запроса или события.

Чтобы сделать обработчик привязанным к запросу, можно:

1. Использовать зависимость от провайдера с областью запроса.
2. Явно задать область `REQUEST` в декораторе `@CommandHandler`, `@QueryHandler` или `@EventsHandler`, как показано ниже:

```typescript
@CommandHandler(KillDragonCommand, {
  scope: Scope.REQUEST,
})
export class KillDragonHandler {
  // Реализация
}
```

Чтобы внедрить полезную нагрузку запроса в любой провайдер с областью запроса, используйте декоратор `@Inject(REQUEST)`. Однако природа этой полезной нагрузки в CQRS зависит от контекста: это может быть HTTP-запрос, задача по расписанию или любая другая операция, инициирующая команду.

Полезная нагрузка должна быть экземпляром класса, унаследованного от `AsyncContext` (предоставляется пакетом `@nestjs/cqrs`), который выступает контекстом запроса и хранит данные, доступные на протяжении всего жизненного цикла запроса.

```typescript
import { AsyncContext } from '@nestjs/cqrs';

export class MyRequest extends AsyncContext {
  constructor(public readonly user: User) {
    super();
  }
}
```

При выполнении команды передайте собственный контекст запроса вторым аргументом метода `CommandBus#execute`:

```typescript
const myRequest = new MyRequest(user);
await this.commandBus.execute(
  new KillDragonCommand(heroId, killDragonDto.dragonId),
  myRequest,
);
```

Благодаря этому экземпляр `MyRequest` становится доступен соответствующему обработчику как провайдер `REQUEST`:

```typescript
@CommandHandler(KillDragonCommand, {
  scope: Scope.REQUEST,
})
export class KillDragonHandler {
  constructor(
    @Inject(REQUEST) private request: MyRequest, // Внедряем контекст запроса
  ) {}

  // Реализация обработчика
}
```

Тот же подход применим и к запросам:

```typescript
const myRequest = new MyRequest(user);
const hero = await this.queryBus.execute(new GetHeroQuery(heroId), myRequest);
```

И в обработчике запроса:

```typescript
@QueryHandler(GetHeroQuery, {
  scope: Scope.REQUEST,
})
export class GetHeroHandler {
  constructor(
    @Inject(REQUEST) private request: MyRequest, // Внедряем контекст запроса
  ) {}

  // Реализация обработчика
}
```

Для событий провайдер запроса тоже можно передать в `EventBus#publish`, но это встречается реже. Вместо этого используйте `EventPublisher`, чтобы встроить провайдер запроса в модель:

```typescript
const hero = this.publisher.mergeObjectContext(
  await this.repository.findOneById(+heroId),
  this.request, // Внедряем контекст запроса здесь
);
```

Обработчики событий с областью запроса, подписанные на эти события, получат доступ к провайдеру запроса.

Саги всегда являются синглтонами, поскольку управляют длительными процессами. Тем не менее провайдер запроса можно извлечь из объектов событий:

```typescript
@Saga()
dragonKilled = (events$: Observable<any>): Observable<ICommand> => {
  return events$.pipe(
    ofType(HeroKilledDragonEvent),
    map((event) => {
      const request = AsyncContext.of(event); // Получаем контекст запроса
      const command = new DropAncientItemCommand(event.heroId, fakeItemID);

      AsyncContext.merge(request, command); // Встраиваем контекст запроса в команду
      return command;
    }),
  );
}
```

Как вариант, используйте метод `request.attachTo(command)`, чтобы привязать контекст запроса к команде.
