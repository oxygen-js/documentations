### Async Local Storage

`AsyncLocalStorage` — это [API Node.js](https://nodejs.org/api/async_context.html#class-asynclocalstorage) (построенный на API `async_hooks`), который даёт альтернативный способ распространять локальное состояние по приложению без явной передачи его параметром функции. Он похож на thread-local storage в других языках.

Основная идея Async Local Storage в том, что мы можем _обернуть_ некоторый вызов функции вызовом `AsyncLocalStorage#run`. Весь код, вызываемый внутри обёрнутого вызова, получает доступ к одному и тому же `store`, уникальному для каждой цепочки вызовов.

В контексте NestJS это значит: если мы найдём в жизненном цикле запроса точку, где можно обернуть весь остальной код запроса, то сможем читать и изменять состояние, видимое только этому запросу. Это может служить альтернативой провайдерам с областью REQUEST и обходить некоторые их ограничения.

Кроме того, ALS можно использовать для распространения контекста лишь по части системы (например, объекта _транзакции_) без явной передачи его между сервисами, что повышает изоляцию и инкапсуляцию.

#### Собственная реализация

Сам NestJS не предоставляет встроенной абстракции над `AsyncLocalStorage`, поэтому разберём, как реализовать её самостоятельно для простейшего HTTP-случая — так лучше понять всю концепцию:

> **Информация** О готовом [специальном пакете](<18. Async local storage.md#nestjs-cls>) читайте ниже.

1. Сначала создайте новый экземпляр `AsyncLocalStorage` в каком-нибудь общем файле. Раз уж мы используем NestJS, оформим его как модуль с пользовательским провайдером.

**`als.module.ts`**

```ts
@Module({
  providers: [
    {
      provide: AsyncLocalStorage,
      useValue: new AsyncLocalStorage(),
    },
  ],
  exports: [AsyncLocalStorage],
})
export class AlsModule {}
```
> **Подсказка** `AsyncLocalStorage` импортируется из `async_hooks`.

2. Нас интересует только HTTP, поэтому воспользуемся middleware, чтобы обернуть функцию `next` вызовом `AsyncLocalStorage#run`. Поскольку middleware — первое, куда попадает запрос, `store` станет доступен во всех enhancer-ах и во всей остальной системе.

**`app.module.ts`**

```ts
@Module({
  imports: [AlsModule],
  providers: [CatsService],
  controllers: [CatsController],
})
export class AppModule implements NestModule {
  constructor(
    // внедряем AsyncLocalStorage в конструктор модуля,
    private readonly als: AsyncLocalStorage
  ) {}

  configure(consumer: MiddlewareConsumer) {
    // привязываем middleware,
    consumer
      .apply((req, res, next) => {
        // заполняем store значениями по умолчанию
        // на основе запроса,
        const store = {
          userId: req.headers['x-user-id'],
        };
        // и передаём функцию "next" как коллбэк
        // в метод "als.run" вместе со store.
        this.als.run(store, () => next());
      })
      .forRoutes('*path');
  }
}
```

3. Теперь в любой точке жизненного цикла запроса мы можем обратиться к экземпляру локального хранилища.

**`cats.service.ts`**

```ts
@Injectable()
export class CatsService {
  constructor(
    // Мы можем внедрить предоставленный экземпляр ALS.
    private readonly als: AsyncLocalStorage,
    private readonly catsRepository: CatsRepository,
  ) {}

  getCatForUser() {
    // Метод "getStore" всегда возвращает экземпляр store,
    // связанный с данным запросом.
    const userId = this.als.getStore()["userId"] as number;
    return this.catsRepository.getForUser(userId);
  }
}
```

4. Вот и всё. Теперь у нас есть способ делиться состоянием, связанным с запросом, без внедрения целого объекта `REQUEST`.

> **Предупреждение** Учтите: приём полезен во многих сценариях, но по своей природе он затемняет поток выполнения (создаёт неявный контекст), поэтому пользуйтесь им обдуманно и особенно избегайте создания контекстных «[божественных объектов](https://en.wikipedia.org/wiki/God_object)».

### NestJS CLS

Пакет [nestjs-cls](https://github.com/Papooch/nestjs-cls) даёт ряд улучшений в удобстве разработки по сравнению с «голым» `AsyncLocalStorage` (`CLS` — сокращение от _continuation-local storage_). Он прячет реализацию за `ClsModule`, который предлагает разные способы инициализации `store` для разных транспортов (не только HTTP), а также поддержку строгой типизации.

Обращаться к store можно через внедряемый `ClsService` либо полностью убрать его из бизнес-логики, используя [Proxy Providers](https://www.npmjs.com/package/nestjs-cls#proxy-providers).

> **Информация** `nestjs-cls` — сторонний пакет, он не сопровождается основной командой NestJS. О любых найденных проблемах с библиотекой сообщайте в [соответствующем репозитории](https://github.com/Papooch/nestjs-cls/issues).

#### Установка

Помимо peer-зависимости от библиотек `@nestjs`, пакет использует только встроенный API Node.js. Устанавливается как любой другой пакет.

```bash
npm i nestjs-cls
```

#### Использование

Функциональность, аналогичную описанной [выше](<18. Async local storage.md#собственная-реализация>), можно реализовать с помощью `nestjs-cls` так:

1. Импортируйте `ClsModule` в корневой модуль.

**`app.module.ts`**

```ts
@Module({
  imports: [
    // Регистрируем ClsModule,
    ClsModule.forRoot({
      middleware: {
        // автоматически подключаем
        // ClsMiddleware для всех маршрутов
        mount: true,
        // и используем метод setup, чтобы
        // задать значения store по умолчанию.
        setup: (cls, req) => {
          cls.set('userId', req.headers['x-user-id']);
        },
      },
    }),
  ],
  providers: [CatsService],
  controllers: [CatsController],
})
export class AppModule {}
```

2. После чего можно использовать `ClsService` для доступа к значениям store.

**`cats.service.ts`**

```ts
@Injectable()
export class CatsService {
  constructor(
    // Мы можем внедрить предоставленный экземпляр ClsService,
    private readonly cls: ClsService,
    private readonly catsRepository: CatsRepository,
  ) {}

  getCatForUser() {
    // и использовать метод "get" для получения любого сохранённого значения.
    const userId = this.cls.get('userId');
    return this.catsRepository.getForUser(userId);
  }
}
```

3. Чтобы получить строгую типизацию значений store, которыми управляет `ClsService` (а заодно и автодополнение строковых ключей), при внедрении можно использовать необязательный параметр типа `ClsService<MyClsStore>`.

```ts
export interface MyClsStore extends ClsStore {
  userId: number;
}
```

> **Подсказка** Также можно позволить пакету автоматически генерировать идентификатор запроса и позднее получать его через `cls.getId()` либо получать весь объект Request через `cls.get(CLS_REQ)`.
#### Тестирование

Поскольку `ClsService` — обычный внедряемый провайдер, в модульных тестах его можно полностью заменить заглушкой.

Однако в некоторых интеграционных тестах может понадобиться настоящая реализация `ClsService`. В таком случае нужно обернуть контекстозависимый фрагмент кода вызовом `ClsService#run` или `ClsService#runWith`.

```ts
describe('CatsService', () => {
  let service: CatsService
  let cls: ClsService
  const mockCatsRepository = createMock<CatsRepository>()

  beforeEach(async () => {
    const module = await Test.createTestingModule({
      // Настраиваем тестовый модуль в основном как обычно.
      providers: [
        CatsService,
        {
          provide: CatsRepository
          useValue: mockCatsRepository
        }
      ],
      imports: [
        // Импортируем статическую версию ClsModule, которая предоставляет
        // только ClsService, но никак не настраивает store.
        ClsModule
      ],
    }).compile()

    service = module.get(CatsService)

    // Также получаем ClsService для дальнейшего использования.
    cls = module.get(ClsService)
  })

  describe('getCatForUser', () => {
    it('retrieves cat based on user id', async () => {
      const expectedUserId = 42
      mocksCatsRepository.getForUser.mockImplementationOnce(
        (id) => ({ userId: id })
      )

      // Оборачиваем тестовый вызов в метод `runWith`,
      // в который можно передать вручную составленные значения store.
      const cat = await cls.runWith(
        { userId: expectedUserId },
        () => service.getCatForUser()
      )

      expect(cat.userId).toEqual(expectedUserId)
    })
  })
})
```
