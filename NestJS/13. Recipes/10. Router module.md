### Модуль маршрутизации

> **Подсказка** Эта глава относится только к приложениям на основе HTTP.

В HTTP-приложении (например, в REST API) путь маршрута для обработчика определяется склейкой (необязательного) префикса, объявленного для контроллера (внутри декоратора `@Controller`),
и пути, указанного в декораторе метода (например, `@Get('users')`). Подробнее об этом — в [этом разделе](<../02. Overview/02. Controllers.md#маршрутизация>). Кроме того,
вы можете задать [глобальный префикс](<../14. FAQ/04. Global path prefix.md>) для всех маршрутов, зарегистрированных в приложении, либо включить [версионирование](<../04. Techniques/07. Versioning.md>).

Бывают и пограничные случаи, когда удобно задать префикс на уровне модуля (а значит, для всех контроллеров, зарегистрированных внутри этого модуля).
Например, представьте REST-приложение, которое предоставляет несколько разных эндпоинтов, используемых определённой частью приложения под названием «Dashboard».
В таком случае вместо повторения префикса `/dashboard` в каждом контроллере можно воспользоваться вспомогательным модулем `RouterModule`:

```typescript
@Module({
  imports: [
    DashboardModule,
    RouterModule.register([
      {
        path: 'dashboard',
        module: DashboardModule,
      },
    ]),
  ],
})
export class AppModule {}
```

> **Подсказка** Класс `RouterModule` экспортируется из пакета `@nestjs/core`.

Помимо этого, вы можете описывать иерархические структуры. Это значит, что у каждого модуля могут быть дочерние модули (`children`).
Дочерние модули наследуют префикс родителя. В следующем примере мы зарегистрируем `AdminModule` как родительский модуль для `DashboardModule` и `MetricsModule`.

```typescript
@Module({
  imports: [
    AdminModule,
    DashboardModule,
    MetricsModule,
    RouterModule.register([
      {
        path: 'admin',
        module: AdminModule,
        children: [
          {
            path: 'dashboard',
            module: DashboardModule,
          },
          {
            path: 'metrics',
            module: MetricsModule,
          },
        ],
      },
    ])
  ],
});
```

> **Подсказка** Пользоваться этой возможностью следует очень осторожно: злоупотребление ею со временем сделает код трудным в сопровождении.

В примере выше любой контроллер, зарегистрированный внутри `DashboardModule`, получит дополнительный префикс `/admin/dashboard` (поскольку модуль склеивает пути сверху вниз — рекурсивно — от родителя к дочерним элементам).
Аналогично каждый контроллер, определённый внутри `MetricsModule`, получит дополнительный префикс уровня модуля `/admin/metrics`.
