### Suites

[Suites](https://suites.dev) — [открытый](https://github.com/suites-dev/suites) фреймворк модульного тестирования для TypeScript-фреймворков с внедрением зависимостей. Он используется как **альтернатива** ручному созданию моков, многословной настройке тестов с множеством конфигураций моков и работе с нетипизированными тестовыми двойниками (моками, заглушками и т. п.).

Suites считывает метаданные сервисов nestjs во время выполнения и автоматически генерирует полностью типизированные моки для всех зависимостей.
Это избавляет от шаблонного кода настройки моков и обеспечивает типобезопасность тестов. Suites можно применять вместе с `Test.createTestingModule()`, но лучше всего он проявляет себя в сфокусированном модульном тестировании.
Используйте `Test.createTestingModule()`, когда нужно проверить связывание модулей, декораторы, guards и interceptors.
Используйте Suites для быстрых модульных тестов с автоматической генерацией моков.

Подробнее о тестировании на уровне модулей рассказано в главе [основы тестирования](<../03. Fundamentals/12. Testing.md>).

> **Примечание** `Suites` — сторонний пакет, он не поддерживается основной командой NestJS. Сообщайте о проблемах в [соответствующий репозиторий](https://github.com/suites-dev/suites).

#### Начало работы

Это руководство показывает, как тестировать сервисы NestJS с помощью Suites. Рассматриваются как изолированное тестирование (все зависимости замоканы), так и «социальное» (sociable) тестирование с частью реальных реализаций.

#### Установка Suites

Убедитесь, что runtime-зависимости NestJS установлены:

```bash
$ npm install @nestjs/common @nestjs/core reflect-metadata
```

Установите ядро Suites, адаптер для NestJS и адаптер тестовых двойников:

```bash
$ npm install --save-dev @suites/unit @suites/di.nestjs @suites/doubles.jest
```

Адаптер тестовых двойников (`@suites/doubles.jest`) предоставляет обёртки над возможностями мокирования Jest. Он экспортирует функции `mock()` и `stub()`, создающие типобезопасные тестовые двойники.

Убедитесь, что доступны Jest и TypeScript:

```bash
$ npm install --save-dev ts-jest @types/jest jest typescript
```

```bash

```

> **Подсказка** Убедитесь, что в `compilerOptions` вашего tsconfig указано `"emitDecoratorMetadata": true` (стандарт для NestJS).

#### Пример сервиса

В этом руководстве используется простой `UserService` с двумя зависимостями:

**`user.repository.ts`**

```typescript
import { Injectable } from '@nestjs/common';

@Injectable()
export class UserRepository {
  async findById(id: string): Promise<User | null> {
    // Запрос к базе данных
  }

  async save(user: User): Promise<User> {
    // Сохранение в базу данных
  }
}
```
**`user.service.ts`**

```typescript
import { Injectable, NotFoundException } from '@nestjs/common';
import { Logger } from '@nestjs/common';

@Injectable()
export class UserService {
  constructor(
    private repository: UserRepository,
    private logger: Logger,
  ) {}

  async findById(id: string): Promise<User> {
    const user = await this.repository.findById(id);
    if (!user) {
      throw new NotFoundException(`User ${id} not found`);
    }
    this.logger.log(`Found user ${id}`);
    return user;
  }

  async create(email: string, name: string): Promise<User> {
    const user = { id: generateId(), email, name };
    await this.repository.save(user);
    this.logger.log(`Created user ${user.id}`);
    return user;
  }
}
```

#### Написание модульного теста

Используйте `TestBed.solitary()`, чтобы создать изолированные тесты, в которых замоканы все зависимости:

**`user.service.spec.ts`**

```typescript
import { TestBed, type Mocked } from '@suites/unit';
import { UserService } from './user.service';
import { UserRepository } from './user.repository';
import { Logger } from '@nestjs/common';

describe('User Service Unit Spec', () => {
  let userService: UserService;
  let repository: Mocked<UserRepository>;
  let logger: Mocked<Logger>;

  beforeAll(async () => {
    const { unit, unitRef } = await TestBed.solitary(UserService).compile();

    userService = unit;
    repository = unitRef.get(UserRepository);
    logger = unitRef.get(Logger);
  });

  it('should find user by id', async () => {
    const user = { id: '1', email: 'test@example.com', name: 'Test' };
    repository.findById.mockResolvedValue(user);

    const result = await userService.findById('1');

    expect(result).toEqual(user);
    expect(logger.log).toHaveBeenCalled();
  });
});
```

`TestBed.solitary()` анализирует конструктор и создаёт типизированные моки для всех зависимостей.
Тип `Mocked<T>` обеспечивает поддержку IntelliSense при настройке моков.

#### Предварительная настройка моков

Настроить поведение мока до компиляции можно с помощью `.mock().impl()`:

**`user.service.spec.ts`**

```typescript
import { TestBed } from '@suites/unit';
import { UserService } from './user.service';
import { UserRepository } from './user.repository';

describe('User Service Unit Spec - pre-configured', () => {
  let unit: UserService;
  let repository: Mocked<UserRepository>;
  
  beforeAll(async () => {
    const { unit: underTest, unitRef } = await TestBed.solitary(UserService)
      .mock(UserRepository)
      .impl(stubFn => ({
        findById: stubFn().mockResolvedValue({ id: '1', email: 'test@example.com', name: 'Test' })
      }))
      .compile();
    
    repository = unitRef.get(UserRepository);
    unit = underTest;
  })
  
  it('should find user with pre-configured mock', async () => {
    const result = await unit.findById('1');
    
    expect(repository.findById).toHaveBeenCalled();
    expect(result.email).toBe('test@example.com');
  });
});
```

Параметр `stubFn` соответствует установленному адаптеру тестовых двойников (`jest.fn()` для Jest, `vi.fn()` для Vitest, `sinon.stub()` для Sinon).

#### Тестирование с реальными зависимостями

Используйте `TestBed.sociable()` вместе с `.expose()`, чтобы задействовать реальные реализации отдельных зависимостей:

**`user.service.spec.ts`**

```typescript
import { TestBed, Mocked } from '@suites/unit';
import { UserService } from './user.service';
import { UserRepository } from './user.repository';
import { Logger } from '@nestjs/common';

describe('UserService - with real logger', () => {
  let userService: UserService;
  let repository: Mocked<UserRepository>;

  beforeAll(async () => {
    const { unit, unitRef } = await TestBed.sociable(UserService)
      .expose(Logger)
      .compile();

    userService = unit;
    repository = unitRef.get(UserRepository);
  });

  it('should log when finding user', async () => {
    const user = { id: '1', email: 'test@example.com' };
    repository.findById.mockResolvedValue(user);

    await userService.findById('1');

    // Logger выполняется по-настоящему, мок не нужен
  });
});
```

`.expose(Logger)` создаёт экземпляр `Logger` с его реальной реализацией, оставляя остальные зависимости замоканными.

#### Зависимости на основе токенов

Suites умеет работать с пользовательскими токенами внедрения (строками или символами):

**`config.service.ts`**

```typescript
import { Injectable, Inject } from '@nestjs/common';

export const CONFIG_OPTIONS = 'CONFIG_OPTIONS';

@Injectable()
export class ConfigService {
  constructor(
    @Inject(CONFIG_OPTIONS) private options: { apiKey: string },
  ) {}

  getApiKey(): string {
    return this.options.apiKey;
  }
}
```

Обращаться к зависимостям на основе токенов можно через `unitRef.get()`:

**`config.service.spec.ts`**

```typescript
import { TestBed } from '@suites/unit';
import { ConfigService, CONFIG_OPTIONS, ConfigOptions } from './config.service';

describe('Config Service Unit Spec', () => {
  let configService: ConfigService;
  let options: ConfigOptions;

  beforeAll(async () => {
    const { unit, unitRef } = await TestBed.solitary(ConfigService).compile();
    configService = unit;

    options = unitRef.get<ConfigOptions>(CONFIG_OPTIONS);
  });

  it('should return api key', () => { ... });
});
```

#### Прямое использование mock() и stub()

Тем, кто предпочитает прямой контроль без `TestBed`, пакет-адаптер тестовых двойников предоставляет функции `mock()` и `stub()`:

**`user.service.spec.ts`**

```typescript
import { mock } from '@suites/unit';
import { UserRepository } from './user.repository';

describe('User Service Unit Spec', () => {
  it('should work with direct mocks', async () => {
    const repository = mock<UserRepository>();
    const logger = mock<Logger>();

    const service = new UserService(repository, logger);

    // ...
  });
});
```

`mock()` создаёт типизированный объект-мок, а `stub()` оборачивает нижележащую библиотеку мокирования (в этом примере — Jest), предоставляя методы вроде `mockResolvedValue()`
Эти функции поставляются установленным адаптером тестовых двойников (`@suites/doubles.jest`), который адаптирует нативные возможности мокирования тестового фреймворка.

> **Подсказка** Функция `mock()` — альтернатива `createMock` из `@golevelup/ts-jest`. Обе создают типизированные объекты-моки. Подробнее о `createMock` см. в главе [основы тестирования](<../03. Fundamentals/12. Testing.md#автоматическое-создание-моков>).

#### Итоги

**Используйте `Test.createTestingModule()` для:**
- Проверки конфигурации модулей и связывания провайдеров
- Тестирования декораторов, guards, interceptors и pipes
- Проверки внедрения зависимостей между модулями
- Тестирования полного контекста приложения вместе с middleware

**Используйте Suites для:**
- Быстрых модульных тестов, сосредоточенных на бизнес-логике
- Автоматической генерации моков для множества зависимостей
- Типобезопасных тестовых двойников с поддержкой IntelliSense

Разделяйте тесты по назначению: Suites — для модульных тестов, проверяющих поведение отдельного сервиса, а `Test.createTestingModule()` — для интеграционных тестов, проверяющих конфигурацию модулей.
