### Prisma

[Prisma](https://www.prisma.io) — это [open-source](https://github.com/prisma/prisma) ORM для Node.js и TypeScript. Она используется как **альтернатива** написанию чистого SQL или применению других инструментов доступа к базе данных: построителей SQL-запросов (вроде [knex.js](https://knexjs.org/)) и ORM (вроде [TypeORM](https://typeorm.io/) и [Sequelize](https://sequelize.org/)). На данный момент Prisma поддерживает PostgreSQL, MySQL, SQL Server, SQLite, MongoDB и CockroachDB ([в статусе Preview](https://www.prisma.io/docs/orm/reference/supported-databases)).

Prisma можно использовать и с чистым JavaScript, но она ориентирована на TypeScript и обеспечивает уровень типобезопасности, выходящий за рамки гарантий других ORM в экосистеме TypeScript. Подробное сравнение гарантий типобезопасности Prisma и TypeORM доступно [здесь](https://www.prisma.io/docs/orm/more/comparisons/prisma-and-typeorm#type-safety).

> **Примечание** Если вы хотите быстро составить представление о том, как работает Prisma, пройдите [Быстрый старт](https://www.prisma.io/docs/getting-started/prisma-orm/quickstart/prisma-postgres) или прочитайте [Введение](https://www.prisma.io/docs/orm/overview/introduction/what-is-prisma) в [документации](https://www.prisma.io/docs). Кроме того, в репозитории [`prisma-examples`](https://github.com/prisma/prisma-examples/) есть готовые к запуску примеры для [REST](https://github.com/prisma/prisma-examples/tree/b53fad046a6d55f0090ddce9fd17ec3f9b95cab3/orm/nest) и [GraphQL](https://github.com/prisma/prisma-examples/tree/b53fad046a6d55f0090ddce9fd17ec3f9b95cab3/orm/nest-graphql).

#### Начало работы

В этом рецепте вы с нуля познакомитесь с совместной работой NestJS и Prisma. Вы построите пример приложения на NestJS с REST API, способным читать и записывать данные в базу.

Для целей этого руководства мы возьмём базу [SQLite](https://sqlite.org/), чтобы не тратить силы на поднятие сервера БД. Обратите внимание: руководство подойдёт и тем, кто использует PostgreSQL или MySQL — в нужных местах будут даны дополнительные указания для этих баз.

> **Примечание** Если у вас уже есть проект и вы рассматриваете переход на Prisma, воспользуйтесь руководством по [добавлению Prisma в существующий проект](https://www.prisma.io/docs/getting-started/setup-prisma/add-to-existing-project-typescript-postgres). Если вы переезжаете с TypeORM, прочитайте руководство [Миграция с TypeORM на Prisma](https://www.prisma.io/docs/guides/migrate-from-typeorm).

#### Создание проекта NestJS

Для начала установите NestJS CLI и создайте каркас приложения следующими командами:

```bash
$ npm install -g @nestjs/cli
$ nest new hello-prisma
```

Подробнее о файлах проекта, созданных этой командой, — на странице [Первые шаги](<../02. Overview/01. First steps.md>). Учтите также, что теперь можно запустить приложение командой `npm start`. REST API, работающий по адресу `http://localhost:3000/`, пока обслуживает единственный маршрут, реализованный в `src/app.controller.ts`. По ходу руководства вы добавите новые маршруты для сохранения и получения данных о _пользователях_ и _постах_.

#### Настройка Prisma

Начните с установки Prisma CLI как зависимости для разработки:

```bash
$ cd hello-prisma
$ npm install prisma --save-dev
```

В следующих шагах мы будем пользоваться [Prisma CLI](https://www.prisma.io/docs/orm/tools/prisma-cli). Рекомендуемая практика — вызывать CLI локально, добавляя префикс `npx`:

```bash
$ npx prisma
```

Если вы используете Yarn, установить Prisma CLI можно так:

```bash
$ yarn add prisma --dev
```

После установки вызывайте её с префиксом `yarn`:

```bash
$ yarn prisma
```

Теперь создайте начальную конфигурацию Prisma командой `init` из Prisma CLI:

```bash
$ npx prisma init
```

Эта команда создаёт новый каталог `prisma` со следующим содержимым:

- `schema.prisma`: описывает подключение к базе данных и содержит схему БД
- `prisma.config.ts`: файл конфигурации для ваших проектов
- `.env`: файл [dotenv](https://github.com/motdotla/dotenv), обычно используемый для хранения учётных данных базы в виде набора переменных окружения

#### Указание пути вывода генератора

Задайте `path` вывода для сгенерированного клиента Prisma — либо передав `--output ../src/generated/prisma` при выполнении prisma init, либо прямо в схеме Prisma:

```groovy
generator client {
  provider        = "prisma-client"
  output          = "../src/generated/prisma"
}
```

#### Настройка формата модулей

Задайте в генераторе `moduleFormat` значение `cjs`:

```groovy
generator client {
  provider        = "prisma-client"
  output          = "../src/generated/prisma"
  moduleFormat    = "cjs"
}
```

> **Примечание** Настройка `moduleFormat` необходима, поскольку Prisma v7 по умолчанию поставляется как ES-модуль, что не работает с CommonJS-конфигурацией NestJS. Значение `cjs` заставляет Prisma генерировать CommonJS-модуль вместо ESM.

#### Настройка подключения к базе данных

Подключение к базе настраивается в блоке `datasource` файла `schema.prisma`. По умолчанию там указан `postgresql`, но раз в этом руководстве используется SQLite, нужно изменить поле `provider` блока `datasource` на `sqlite`:

```groovy
datasource db {
  provider = "sqlite"
}

generator client {
  provider      = "prisma-client"
  output        = "../src/generated/prisma"
  moduleFormat  = "cjs"
}
```

Теперь откройте `.env` и приведите переменную окружения `DATABASE_URL` к такому виду:

```bash
DATABASE_URL="file:./dev.db"
```

Убедитесь, что у вас настроен [ConfigModule](<../04. Techniques/01. Configuration.md>), иначе переменная `DATABASE_URL` не будет подхвачена из `.env`.

Базы SQLite — это обычные файлы, для работы с ними сервер не нужен. Поэтому вместо URL подключения с _хостом_ и _портом_ достаточно указать локальный файл, в данном случае `dev.db`. Он будет создан на следующем шаге.

<details><summary>Раскройте, если вы используете PostgreSQL, MySQL, MsSQL или Azure SQL</summary>

В случае PostgreSQL и MySQL URL подключения должен указывать на _сервер базы данных_. Подробнее о требуемом формате URL подключения — [здесь](https://www.prisma.io/docs/orm/reference/connection-urls).

**PostgreSQL**

Если вы используете PostgreSQL, файлы `schema.prisma` и `.env` нужно изменить так:

**`schema.prisma`**

```groovy
datasource db {
  provider = "postgresql"
}

generator client {
  provider = "prisma-client"
  output          = "../src/generated/prisma"
  moduleFormat  = "cjs"
}
```

**`.env`**

```bash
DATABASE_URL="postgresql://USER:PASSWORD@HOST:PORT/DATABASE?schema=SCHEMA"
```

Замените написанные заглавными буквами подстановки своими учётными данными базы. Если вы не уверены, что указать вместо `SCHEMA`, — скорее всего, подойдёт значение по умолчанию `public`:

```bash
DATABASE_URL="postgresql://USER:PASSWORD@HOST:PORT/DATABASE?schema=public"
```

Если хотите разобраться, как поднять базу PostgreSQL, воспользуйтесь руководством по [настройке бесплатной базы PostgreSQL на Heroku](https://dev.to/prisma/how-to-setup-a-free-postgresql-database-on-heroku-1dc1).

**MySQL**

Если вы используете MySQL, файлы `schema.prisma` и `.env` нужно изменить так:

**`schema.prisma`**

```groovy
datasource db {
  provider = "mysql"
}

generator client {
  provider = "prisma-client"
  output          = "../src/generated/prisma"
  moduleFormat  = "cjs"
}
```

**`.env`**

```bash
DATABASE_URL="mysql://USER:PASSWORD@HOST:PORT/DATABASE"
```

Замените написанные заглавными буквами подстановки своими учётными данными базы.

**Microsoft SQL Server / Azure SQL Server**

Если вы используете Microsoft SQL Server или Azure SQL Server, файлы `schema.prisma` и `.env` нужно изменить так:

**`schema.prisma`**

```groovy
datasource db {
  provider = "sqlserver"
}

generator client {
  provider = "prisma-client"
  output          = "../src/generated/prisma"
  moduleFormat  = "cjs"
}
```

**`.env`**

Замените написанные заглавными буквами подстановки своими учётными данными базы. Если вы не уверены, что указать вместо `encrypt`, — скорее всего, подойдёт значение по умолчанию `true`:

```bash
DATABASE_URL="sqlserver://HOST:PORT;database=DATABASE;user=USER;password=PASSWORD;encrypt=true"
```

</details>

#### Создание двух таблиц с помощью Prisma Migrate

В этом разделе вы создадите в базе две новые таблицы с помощью [Prisma Migrate](https://www.prisma.io/docs/orm/prisma-migrate/getting-started). Prisma Migrate генерирует SQL-файлы миграций по декларативному описанию модели данных в схеме Prisma. Эти файлы миграций полностью настраиваемы, так что вы можете задействовать любые дополнительные возможности используемой базы или добавить свои команды — например, для заполнения данными.

Добавьте в файл `schema.prisma` следующие две модели:

```groovy
model User {
  id    Int     @default(autoincrement()) @id
  email String  @unique
  name  String?
  posts Post[]
}

model Post {
  id        Int      @default(autoincrement()) @id
  title     String
  content   String?
  published Boolean? @default(false)
  author    User?    @relation(fields: [authorId], references: [id])
  authorId  Int?
}
```

Когда модели Prisma готовы, можно сгенерировать SQL-файлы миграций и применить их к базе. Выполните в терминале следующие команды:

```bash
$ npx prisma migrate dev --name init
```

Команда `prisma migrate dev` генерирует SQL-файлы и сразу же применяет их к базе. В нашем случае в уже существующем каталоге `prisma` были созданы такие файлы миграции:

```bash
$ tree prisma
prisma
├── dev.db
├── migrations
│   └── 20201207100915_init
│       └── migration.sql
└── schema.prisma
```

<details><summary>Раскройте, чтобы посмотреть сгенерированные SQL-инструкции</summary>

В вашей базе SQLite были созданы следующие таблицы:

```sql

    "id" INTEGER NOT NULL PRIMARY KEY AUTOINCREMENT,
    "email" TEXT NOT NULL,
    "name" TEXT
);

-- CreateTable

    "id" INTEGER NOT NULL PRIMARY KEY AUTOINCREMENT,
    "title" TEXT NOT NULL,
    "content" TEXT,
    "published" BOOLEAN DEFAULT false,
    "authorId" INTEGER,

    FOREIGN KEY ("authorId") REFERENCES "User"("id") ON DELETE SET NULL ON UPDATE CASCADE

-- CreateIndex
```

#### Установка и генерация Prisma Client

Prisma Client — типобезопасный клиент базы данных, который _генерируется_ по описанию ваших моделей Prisma. Благодаря такому подходу Prisma Client предоставляет [CRUD](https://www.prisma.io/docs/orm/prisma-client/queries/crud)-операции, _подогнанные_ именно под ваши модели.

Чтобы установить Prisma Client в проект, выполните в терминале команду:

```bash
$ npm install @prisma/client
```

После установки запустите команду generate, чтобы сгенерировать типы и клиент для вашего проекта. При любых изменениях схемы команду `generate` нужно запускать повторно, чтобы типы оставались в актуальном состоянии.

```bash
$ npx prisma generate
```

Помимо Prisma Client вам понадобится адаптер драйвера для используемого типа базы. Для SQLite можно установить драйвер `@prisma/adapter-better-sqlite3`.

```bash
npm install @prisma/adapter-better-sqlite3
```

<details> <summary>Раскройте, если вы используете PostgreSQL, MySQL, MsSQL или AzureSQL</summary>

- Для PostgreSQL

```bash
npm install @prisma/adapter-pg
```

- Для MySQL, MsSQL, AzureSQL:

```bash
npm install @prisma/adapter-mariadb
```

</details>

#### Использование Prisma Client в сервисах NestJS

Теперь вы можете отправлять запросы к базе через Prisma Client. Если хотите подробнее изучить построение запросов с Prisma Client, загляните в [документацию по API](https://www.prisma.io/docs/orm/reference/prisma-client-reference).

При настройке приложения NestJS API запросов Prisma Client стоит спрятать за сервисом. Для начала создайте новый `PrismaService`, который возьмёт на себя создание экземпляра `PrismaClient` и подключение к базе.

Внутри каталога `src` создайте файл `prisma.service.ts` и добавьте в него следующий код:

```typescript
import { Injectable } from '@nestjs/common';
import { PrismaClient } from './generated/prisma/client';
import { PrismaBetterSqlite3 } from '@prisma/adapter-better-sqlite3';

@Injectable()
export class PrismaService extends PrismaClient {
  constructor() {
    const adapter = new PrismaBetterSqlite3({ url: process.env.DATABASE_URL });
    super({ adapter });
  }
}
```

Далее можно написать сервисы для обращений к базе для моделей `User` и `Post` из вашей схемы Prisma.

Всё так же внутри каталога `src` создайте файл `user.service.ts` и добавьте в него следующий код:

```typescript
import { Injectable } from '@nestjs/common';
import { PrismaService } from './prisma.service';
import { User, Prisma } from 'generated/prisma';

@Injectable()
export class UsersService {
  constructor(private prisma: PrismaService) {}

  async user(
    userWhereUniqueInput: Prisma.UserWhereUniqueInput,
  ): Promise<User | null> {
    return this.prisma.user.findUnique({
      where: userWhereUniqueInput,
    });
  }

  async users(params: {
    skip?: number;
    take?: number;
    cursor?: Prisma.UserWhereUniqueInput;
    where?: Prisma.UserWhereInput;
    orderBy?: Prisma.UserOrderByWithRelationInput;
  }): Promise<User[]> {
    const { skip, take, cursor, where, orderBy } = params;
    return this.prisma.user.findMany({
      skip,
      take,
      cursor,
      where,
      orderBy,
    });
  }

  async createUser(data: Prisma.UserCreateInput): Promise<User> {
    return this.prisma.user.create({
      data,
    });
  }

  async updateUser(params: {
    where: Prisma.UserWhereUniqueInput;
    data: Prisma.UserUpdateInput;
  }): Promise<User> {
    const { where, data } = params;
    return this.prisma.user.update({
      data,
      where,
    });
  }

  async deleteUser(where: Prisma.UserWhereUniqueInput): Promise<User> {
    return this.prisma.user.delete({
      where,
    });
  }
}
```

Обратите внимание, как здесь используются сгенерированные Prisma Client типы, чтобы методы вашего сервиса были корректно типизированы. Так вы избавляетесь от шаблонного кода: не нужно описывать типы моделей и создавать дополнительные файлы интерфейсов или DTO.

Теперь сделайте то же самое для модели `Post`.

Всё так же внутри каталога `src` создайте файл `post.service.ts` и добавьте в него следующий код:

```typescript
import { Injectable } from '@nestjs/common';
import { PrismaService } from './prisma.service';
import { Post, Prisma } from 'generated/prisma';

@Injectable()
export class PostsService {
  constructor(private prisma: PrismaService) {}

  async post(
    postWhereUniqueInput: Prisma.PostWhereUniqueInput,
  ): Promise<Post | null> {
    return this.prisma.post.findUnique({
      where: postWhereUniqueInput,
    });
  }

  async posts(params: {
    skip?: number;
    take?: number;
    cursor?: Prisma.PostWhereUniqueInput;
    where?: Prisma.PostWhereInput;
    orderBy?: Prisma.PostOrderByWithRelationInput;
  }): Promise<Post[]> {
    const { skip, take, cursor, where, orderBy } = params;
    return this.prisma.post.findMany({
      skip,
      take,
      cursor,
      where,
      orderBy,
    });
  }

  async createPost(data: Prisma.PostCreateInput): Promise<Post> {
    return this.prisma.post.create({
      data,
    });
  }

  async updatePost(params: {
    where: Prisma.PostWhereUniqueInput;
    data: Prisma.PostUpdateInput;
  }): Promise<Post> {
    const { data, where } = params;
    return this.prisma.post.update({
      data,
      where,
    });
  }

  async deletePost(where: Prisma.PostWhereUniqueInput): Promise<Post> {
    return this.prisma.post.delete({
      where,
    });
  }
}
```

Пока что `UsersService` и `PostsService` просто оборачивают CRUD-запросы, доступные в Prisma Client. В реальном приложении сервис также был бы местом для бизнес-логики. Например, в `UsersService` мог бы появиться метод `updatePassword`, отвечающий за смену пароля пользователя.

Не забудьте зарегистрировать новые сервисы в модуле приложения.

##### Реализация маршрутов REST API в главном контроллере приложения

Наконец, воспользуйтесь созданными в предыдущих разделах сервисами, чтобы реализовать различные маршруты приложения. Для целей этого руководства все маршруты мы поместим в уже существующий класс `AppController`.

Замените содержимое файла `app.controller.ts` следующим кодом:

```typescript
import {
  Controller,
  Get,
  Param,
  Post,
  Body,
  Put,
  Delete,
} from '@nestjs/common';
import { UsersService } from './user.service';
import { PostsService } from './post.service';
import { User as UserModel, Post as PostModel } from 'generated/prisma';

@Controller()
export class AppController {
  constructor(
    private readonly userService: UsersService,
    private readonly postService: PostsService,
  ) {}

  @Get('post/:id')
  async getPostById(@Param('id') id: string): Promise<PostModel> {
    return this.postService.post({ id: Number(id) });
  }

  @Get('feed')
  async getPublishedPosts(): Promise<PostModel[]> {
    return this.postService.posts({
      where: { published: true },
    });
  }

  @Get('filtered-posts/:searchString')
  async getFilteredPosts(
    @Param('searchString') searchString: string,
  ): Promise<PostModel[]> {
    return this.postService.posts({
      where: {
        OR: [
          {
            title: { contains: searchString },
          },
          {
            content: { contains: searchString },
          },
        ],
      },
    });
  }

  @Post('post')
  async createDraft(
    @Body() postData: { title: string; content?: string; authorEmail: string },
  ): Promise<PostModel> {
    const { title, content, authorEmail } = postData;
    return this.postService.createPost({
      title,
      content,
      author: {
        connect: { email: authorEmail },
      },
    });
  }

  @Post('user')
  async signupUser(
    @Body() userData: { name?: string; email: string },
  ): Promise<UserModel> {
    return this.userService.createUser(userData);
  }

  @Put('publish/:id')
  async publishPost(@Param('id') id: string): Promise<PostModel> {
    return this.postService.updatePost({
      where: { id: Number(id) },
      data: { published: true },
    });
  }

  @Delete('post/:id')
  async deletePost(@Param('id') id: string): Promise<PostModel> {
    return this.postService.deletePost({ id: Number(id) });
  }
}
```

Этот контроллер реализует следующие маршруты:

###### `GET`

- `/post/:id`: получить один пост по его `id`
- `/feed`: получить все _опубликованные_ посты
- `/filter-posts/:searchString`: отфильтровать посты по `title` или `content`

###### `POST`

- `/post`: создать новый пост
  - Тело запроса:
    - `title: String` (обязательно): заголовок поста
    - `content: String` (необязательно): содержимое поста
    - `authorEmail: String` (обязательно): email пользователя, создающего пост
- `/user`: создать нового пользователя
  - Тело запроса:
    - `email: String` (обязательно): адрес электронной почты пользователя
    - `name: String` (необязательно): имя пользователя

###### `PUT`

- `/publish/:id`: опубликовать пост по его `id`

###### `DELETE`

- `/post/:id`: удалить пост по его `id`

#### Итоги

В этом рецепте вы узнали, как использовать Prisma вместе с NestJS для реализации REST API. Контроллер, реализующий маршруты API, обращается к `PrismaService`, который, в свою очередь, через Prisma Client отправляет запросы к базе данных, чтобы обеспечить входящие запросы нужными данными.

Если вы хотите глубже изучить работу NestJS с Prisma, обязательно посмотрите следующие материалы:

- [NestJS и Prisma](https://www.prisma.io/nestjs)
- [Готовые к запуску примеры проектов для REST и GraphQL](https://github.com/prisma/prisma-examples/)
- [Стартовый набор, готовый к продакшену](https://github.com/notiz-dev/nestjs-prisma-starter#instructions)
- [Видео: доступ к базам данных из NestJS с помощью Prisma (5 мин)](https://www.youtube.com/watch?v=UlVJ340UEuk&ab_channel=Prisma) от [Marc Stammerjohann](https://github.com/marcjulian)
