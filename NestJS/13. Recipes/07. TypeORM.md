### SQL (TypeORM)

##### Эта глава относится только к TypeScript

> **Предупреждение** В этой статье вы узнаете, как с нуля создать `DatabaseModule` на основе пакета **TypeORM**, используя механизм пользовательских провайдеров. Как следствие, такое решение содержит немало накладных расходов, которых можно избежать, воспользовавшись готовым и доступным «из коробки» пакетом `@nestjs/typeorm`. Подробнее об этом — [здесь](<../04. Techniques/02. Database.md>).

[TypeORM](https://github.com/typeorm/typeorm) — безусловно самый зрелый объектно-реляционный маппер (ORM), доступный в мире node.js. Поскольку он написан на TypeScript, он очень хорошо работает с фреймворком Nest.

#### Начало работы

Чтобы начать знакомство с этой библиотекой, нужно установить все необходимые зависимости:

```bash
$ npm install --save typeorm mysql2
```

Первым делом нужно установить соединение с базой данных с помощью `new DataSource().initialize()` — класса, импортированного из пакета `typeorm`. Функция `initialize()` возвращает `Promise`, поэтому нам нужно создать [асинхронный провайдер](<../03. Fundamentals/02. Asynchronous providers.md>).

**`database.providers.ts`**

```typescript
import { DataSource } from 'typeorm';

export const databaseProviders = [
  {
    provide: 'DATA_SOURCE',
    useFactory: async () => {
      const dataSource = new DataSource({
        type: 'mysql',
        host: 'localhost',
        port: 3306,
        username: 'root',
        password: 'root',
        database: 'test',
        entities: [
            __dirname + '/../**/*.entity{.ts,.js}',
        ],
        synchronize: true,
      });

      return dataSource.initialize();
    },
  },
];
```

> **Предупреждение** Не используйте `synchronize: true` в production — иначе вы можете потерять рабочие данные.

> **Подсказка** Следуя лучшим практикам, мы объявили пользовательский провайдер в отдельном файле с суффиксом `*.providers.ts`.

Далее нужно экспортировать эти провайдеры, чтобы сделать их **доступными** для остального приложения.

**`database.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { databaseProviders } from './database.providers';

@Module({
  providers: [...databaseProviders],
  exports: [...databaseProviders],
})
export class DatabaseModule {}
```

Теперь мы можем внедрить объект `DATA_SOURCE` с помощью декоратора `@Inject()`. Каждый класс, зависящий от асинхронного провайдера `DATA_SOURCE`, будет ждать, пока разрешится `Promise`.

#### Паттерн «репозиторий»

[TypeORM](https://github.com/typeorm/typeorm) поддерживает паттерн проектирования «репозиторий», поэтому у каждой сущности есть собственный Repository. Эти репозитории можно получить из соединения с базой данных.

Но сначала нам нужна хотя бы одна сущность. Мы переиспользуем сущность `Photo` из официальной документации.

**`photo.entity.ts`**

```typescript
import { Entity, Column, PrimaryGeneratedColumn } from 'typeorm';

@Entity()
export class Photo {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ length: 500 })
  name: string;

  @Column('text')
  description: string;

  @Column()
  filename: string;

  @Column('int')
  views: number;

  @Column()
  isPublished: boolean;
}
```

Сущность `Photo` находится в каталоге `photo`. Этот каталог представляет `PhotoModule`. Теперь создадим провайдер **Repository**:

**`photo.providers.ts`**

```typescript
import { DataSource } from 'typeorm';
import { Photo } from './photo.entity';

export const photoProviders = [
  {
    provide: 'PHOTO_REPOSITORY',
    useFactory: (dataSource: DataSource) => dataSource.getRepository(Photo),
    inject: ['DATA_SOURCE'],
  },
];
```

> **Предупреждение** В реальных приложениях следует избегать **магических строк**. И `PHOTO_REPOSITORY`, и `DATA_SOURCE` стоит хранить в отдельном файле `constants.ts`.

Теперь мы можем внедрить `Repository<Photo>` в `PhotoService` с помощью декоратора `@Inject()`:

**`photo.service.ts`**

```typescript
import { Injectable, Inject } from '@nestjs/common';
import { Repository } from 'typeorm';
import { Photo } from './photo.entity';

@Injectable()
export class PhotoService {
  constructor(
    @Inject('PHOTO_REPOSITORY')
    private photoRepository: Repository<Photo>,
  ) {}

  async findAll(): Promise<Photo[]> {
    return this.photoRepository.find();
  }
}
```

Соединение с базой данных **асинхронно**, но Nest делает этот процесс полностью незаметным для конечного пользователя. `PhotoRepository` ждёт соединения с БД, а `PhotoService` откладывается до тех пор, пока репозиторий не будет готов к использованию. Приложение целиком стартует, когда создан экземпляр каждого класса.

Вот итоговый `PhotoModule`:

**`photo.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { DatabaseModule } from '../database/database.module';
import { photoProviders } from './photo.providers';
import { PhotoService } from './photo.service';

@Module({
  imports: [DatabaseModule],
  providers: [
    ...photoProviders,
    PhotoService,
  ],
})
export class PhotoModule {}
```

> **Подсказка** Не забудьте импортировать `PhotoModule` в корневой `AppModule`.
