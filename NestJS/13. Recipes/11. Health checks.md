### Проверки работоспособности (Terminus)

Интеграция с Terminus даёт вам проверки работоспособности типа **readiness/liveness**. Такие проверки крайне важны при сложной
организации бэкенда. Если коротко, проверка работоспособности в веб-разработке обычно представляет собой специальный адрес, например `https://my-website.com/health/readiness`.
Сервис или компонент вашей инфраструктуры (например, [Kubernetes](https://kubernetes.io/)) постоянно обращается к этому адресу. В зависимости от HTTP-кода состояния, возвращаемого на `GET`-запрос по этому адресу, сервис предпримет те или иные действия, если получит ответ «неработоспособен».
Поскольку определение «работоспособности» и «неработоспособности» зависит от типа предоставляемого вами сервиса, интеграция с **Terminus** предлагает
набор **индикаторов работоспособности** (health indicators).

Например, если ваш веб-сервер хранит данные в MongoDB, жизненно важно знать, работает ли MongoDB.
В этом случае можно воспользоваться `MongooseHealthIndicator`. При правильной настройке — об этом далее — адрес проверки работоспособности будет возвращать
HTTP-код состояния «работоспособен» или «неработоспособен» в зависимости от того, запущена ли MongoDB.

#### Начало работы

Чтобы начать работу с `@nestjs/terminus`, нужно установить требуемую зависимость.

```bash
$ npm install --save @nestjs/terminus
```

#### Настройка проверки работоспособности

Проверка работоспособности представляет собой сводку по **индикаторам работоспособности**. Индикатор проверяет, находится ли сервис в работоспособном или неработоспособном состоянии. Проверка считается успешной, если все назначенные индикаторы работают штатно. Поскольку многим приложениям нужны схожие индикаторы, [`@nestjs/terminus`](https://github.com/nestjs/terminus) предоставляет набор готовых индикаторов, например:

- `HttpHealthIndicator`
- `TypeOrmHealthIndicator`
- `MongooseHealthIndicator`
- `SequelizeHealthIndicator`
- `MikroOrmHealthIndicator`
- `PrismaHealthIndicator`
- `MicroserviceHealthIndicator`
- `GRPCHealthIndicator`
- `MemoryHealthIndicator`
- `DiskHealthIndicator`

Чтобы приступить к первой проверке работоспособности, создадим `HealthModule` и импортируем в него `TerminusModule`, добавив его в массив imports.

> **Подсказка** Чтобы создать модуль с помощью [Nest CLI](<../11. CLI/01. Overview.md>), просто выполните команду `$ nest g module health`.

**`health.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { TerminusModule } from '@nestjs/terminus';

@Module({
  imports: [TerminusModule]
})
export class HealthModule {}
```

Наши проверки работоспособности могут выполняться через [контроллер](<../02. Overview/02. Controllers.md>), который легко создать с помощью [Nest CLI](<../11. CLI/01. Overview.md>).

```bash
$ nest g controller health
```

> **Информация** Настоятельно рекомендуется включить в приложении хуки завершения работы (shutdown hooks). Интеграция с Terminus использует это событие жизненного цикла, если оно включено. Подробнее о хуках завершения работы читайте [здесь](<../03. Fundamentals/09. Lifecycle events.md#завершение-работы-приложения>).

#### Проверка работоспособности по HTTP

После того как мы установили `@nestjs/terminus`, импортировали `TerminusModule` и создали новый контроллер, можно приступать к созданию проверки работоспособности.

Для `HTTPHealthIndicator` требуется пакет `@nestjs/axios`, поэтому убедитесь, что он установлен:

```bash
$ npm i --save @nestjs/axios axios
```

Теперь можно настроить `HealthController`:

**`health.controller.ts`**

```typescript
import { Controller, Get } from '@nestjs/common';
import { HealthCheckService, HttpHealthIndicator, HealthCheck } from '@nestjs/terminus';

@Controller('health')
export class HealthController {
  constructor(
    private health: HealthCheckService,
    private http: HttpHealthIndicator,
  ) {}

  @Get()
  @HealthCheck()
  check() {
    return this.health.check([
      () => this.http.pingCheck('nestjs-docs', 'https://docs.nestjs.com'),
    ]);
  }
}
```

**`health.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { TerminusModule } from '@nestjs/terminus';
import { HttpModule } from '@nestjs/axios';
import { HealthController } from './health.controller';

@Module({
  imports: [TerminusModule, HttpModule],
  controllers: [HealthController],
})
export class HealthModule {}
```

Теперь наша проверка работоспособности будет отправлять _GET_-запрос по адресу `https://docs.nestjs.com`. Если
мы получим оттуда успешный ответ, маршрут `http://localhost:3000/health` вернёт
следующий объект с кодом состояния 200.

```json
{
  "status": "ok",
  "info": {
    "nestjs-docs": {
      "status": "up"
    }
  },
  "error": {},
  "details": {
    "nestjs-docs": {
      "status": "up"
    }
  }
}
```

Интерфейс этого объекта ответа доступен в пакете `@nestjs/terminus` под именем `HealthCheckResult`.

|           |                                                                                                                                                                                             |                                      |
|-----------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|--------------------------------------|
| `status`  | Если хотя бы один индикатор работоспособности дал сбой, статус будет `'error'`. Если приложение NestJS завершает работу, но всё ещё принимает HTTP-запросы, проверка получит статус `'shutting_down'`. | `'error' \| 'ok' \| 'shutting_down'` |
| `info`    | Объект с информацией о каждом индикаторе со статусом `'up'` — иными словами, «работоспособен».                                                                              | `object`                             |
| `error`   | Объект с информацией о каждом индикаторе со статусом `'down'` — иными словами, «неработоспособен».                                                                          | `object`                             |
| `details` | Объект со всей информацией по каждому индикатору работоспособности                                                                                                                  | `object`                             |

##### Проверка конкретных кодов HTTP-ответа

В некоторых случаях требуется проверить конкретные условия и валидировать ответ. Например, предположим, что
`https://my-external-service.com` возвращает код ответа `204`. С помощью `HttpHealthIndicator.responseCheck` вы можете
проверять именно этот код ответа, а все остальные считать признаком неработоспособности.

В приведённом ниже примере любой код ответа, отличный от `204`, будет означать неработоспособность. Третий параметр
требует передать функцию (синхронную или асинхронную), возвращающую булево значение: считается ли ответ
работоспособным (`true`) или неработоспособным (`false`).

**`health.controller.ts`**

```typescript
// Внутри класса `HealthController`

@Get()
@HealthCheck()
check() {
  return this.health.check([
    () =>
      this.http.responseCheck(
        'my-external-service',
        'https://my-external-service.com',
        (res) => res.status === 204,
      ),
  ]);
}
```

#### Индикатор работоспособности TypeOrm

Terminus позволяет добавить в проверку работоспособности проверки базы данных. Чтобы начать работу с этим индикатором,
изучите [главу о базах данных](<../04. Techniques/02. Database.md>) и убедитесь, что подключение к базе данных в вашем приложении установлено.

> **Подсказка** Под капотом `TypeOrmHealthIndicator` просто выполняет SQL-команду `SELECT 1`, которую часто используют для проверки того, что база данных всё ещё жива. Если вы используете базу данных Oracle, применяется `SELECT 1 FROM DUAL`.

**`health.controller.ts`**

```typescript
@Controller('health')
export class HealthController {
  constructor(
    private health: HealthCheckService,
    private db: TypeOrmHealthIndicator,
  ) {}

  @Get()
  @HealthCheck()
  check() {
    return this.health.check([
      () => this.db.pingCheck('database'),
    ]);
  }
}
```

Если база данных доступна, при `GET`-запросе к `http://localhost:3000/health` вы увидите такой JSON-результат:

```json
{
  "status": "ok",
  "info": {
    "database": {
      "status": "up"
    }
  },
  "error": {},
  "details": {
    "database": {
      "status": "up"
    }
  }
}
```

Если ваше приложение использует [несколько баз данных](<../04. Techniques/02. Database.md#несколько-баз-данных>), необходимо внедрить каждое
подключение в `HealthController`. После этого достаточно передать ссылку на подключение в `TypeOrmHealthIndicator`.

**`health.controller.ts`**

```typescript
@Controller('health')
export class HealthController {
  constructor(
    private health: HealthCheckService,
    private db: TypeOrmHealthIndicator,
    @InjectConnection('albumsConnection')
    private albumsConnection: Connection,
    @InjectConnection()
    private defaultConnection: Connection,
  ) {}

  @Get()
  @HealthCheck()
  check() {
    return this.health.check([
      () => this.db.pingCheck('albums-database', { connection: this.albumsConnection }),
      () => this.db.pingCheck('database', { connection: this.defaultConnection }),
    ]);
  }
}
```

#### Индикатор работоспособности диска

С помощью `DiskHealthIndicator` можно проверять, сколько дискового пространства занято. Для начала внедрите `DiskHealthIndicator`
в свой `HealthController`. В примере ниже проверяется использование хранилища по пути `/` (в Windows можно использовать `C:\\`).
Если занято более 50 % от общего объёма, проверка работоспособности вернёт признак неработоспособности.

**`health.controller.ts`**

```typescript
@Controller('health')
export class HealthController {
  constructor(
    private readonly health: HealthCheckService,
    private readonly disk: DiskHealthIndicator,
  ) {}

  @Get()
  @HealthCheck()
  check() {
    return this.health.check([
      () => this.disk.checkStorage('storage', { path: '/', thresholdPercent: 0.5 }),
    ]);
  }
}
```

Функция `DiskHealthIndicator.checkStorage` также позволяет задать фиксированный объём пространства.
Пример ниже даст признак неработоспособности, если по пути `/my-app/` будет превышено 250 ГБ.

**`health.controller.ts`**

```typescript
// Внутри класса `HealthController`

@Get()
@HealthCheck()
check() {
  return this.health.check([
    () => this.disk.checkStorage('storage', {  path: '/', threshold: 250 * 1024 * 1024 * 1024, })
  ]);
}
```

#### Индикатор работоспособности памяти

Чтобы убедиться, что процесс не превышает определённый лимит памяти, можно использовать `MemoryHealthIndicator`.
Пример ниже проверяет heap вашего процесса.

> **Подсказка** Heap — это область памяти, в которой располагается динамически выделяемая память (то есть память, выделенная через malloc). Выделенная из heap память остаётся занятой, пока не произойдёт одно из следующего:
> - память освобождена (_free_);
> - программа завершилась.

**`health.controller.ts`**

```typescript
@Controller('health')
export class HealthController {
  constructor(
    private health: HealthCheckService,
    private memory: MemoryHealthIndicator,
  ) {}

  @Get()
  @HealthCheck()
  check() {
    return this.health.check([
      () => this.memory.checkHeap('memory_heap', 150 * 1024 * 1024),
    ]);
  }
}
```

Также можно проверить показатель RSS процесса с помощью `MemoryHealthIndicator.checkRSS`. Этот пример
вернёт код неработоспособности, если процесс занял более 150 МБ.

> **Подсказка** RSS (Resident Set Size) показывает, сколько памяти выделено процессу и находится в оперативной памяти.
> В него не входит память, выгруженная в swap. В него входит память разделяемых библиотек — при условии, что страницы
> этих библиотек действительно находятся в памяти. В него входит вся память стека и heap.

**`health.controller.ts`**

```typescript
// Внутри класса `HealthController`

@Get()
@HealthCheck()
check() {
  return this.health.check([
    () => this.memory.checkRSS('memory_rss', 150 * 1024 * 1024),
  ]);
}
```

#### Пользовательский индикатор работоспособности

Иногда готовые индикаторы из `@nestjs/terminus` не покрывают всех ваших потребностей. В этом случае вы можете создать собственный индикатор под свои задачи.

Начнём с создания сервиса, который и будет нашим пользовательским индикатором. Чтобы в общих чертах понять, как устроен индикатор, создадим для примера `DogHealthIndicator`. Этот сервис должен иметь состояние `'up'`, если каждый объект `Dog` имеет тип `'goodboy'`. Если условие не выполняется, он должен выбросить ошибку.

**`dog.health.ts`**

```typescript
import { Injectable } from '@nestjs/common';
import { HealthIndicatorService } from '@nestjs/terminus';

export interface Dog {
  name: string;
  type: string;
}

@Injectable()
export class DogHealthIndicator {
  constructor(
    private readonly healthIndicatorService: HealthIndicatorService
  ) {}

  private dogs: Dog[] = [
    { name: 'Fido', type: 'goodboy' },
    { name: 'Rex', type: 'badboy' },
  ];

  async isHealthy(key: string){
    const indicator = this.healthIndicatorService.check(key);
    const badboys = this.dogs.filter(dog => dog.type === 'badboy');
    const isHealthy = badboys.length === 0;

    if (!isHealthy) {
      return indicator.down({ badboys: badboys.length });
    }

    return indicator.up();
  }
}
```

Далее нужно зарегистрировать индикатор работоспособности как провайдер.

**`health.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { TerminusModule } from '@nestjs/terminus';
import { DogHealthIndicator } from './dog.health';

@Module({
  controllers: [HealthController],
  imports: [TerminusModule],
  providers: [DogHealthIndicator]
})
export class HealthModule { }
```

> **Подсказка** В реальном приложении `DogHealthIndicator` следует объявлять в отдельном модуле, например `DogModule`, который затем импортируется в `HealthModule`.

Последний необходимый шаг — добавить ставший доступным индикатор в нужный эндпоинт проверки работоспособности. Для этого вернёмся к `HealthController` и добавим его в функцию `check`.

**`health.controller.ts`**

```typescript
import { HealthCheckService, HealthCheck } from '@nestjs/terminus';
import { Injectable, Dependencies, Get } from '@nestjs/common';
import { DogHealthIndicator } from './dog.health';

@Injectable()
export class HealthController {
  constructor(
    private health: HealthCheckService,
    private dogHealthIndicator: DogHealthIndicator
  ) {}

  @Get()
  @HealthCheck()
  healthCheck() {
    return this.health.check([
      () => this.dogHealthIndicator.isHealthy('dog'),
    ])
  }
}
```

#### Логирование

Terminus логирует только сообщения об ошибках — например, когда проверка работоспособности не прошла. Метод `TerminusModule.forRoot()` даёт больше контроля над тем, как логируются ошибки,
и позволяет полностью взять логирование на себя.

В этом разделе мы разберём, как создать пользовательский логгер `TerminusLogger`. Он расширяет встроенный логгер,
поэтому вы можете переопределить только те части, которые вам нужны.

> **Информация** Если вы хотите больше узнать о пользовательских логгерах в NestJS, [читайте здесь](<../04. Techniques/10. Logging.md#внедрение-собственного-логгера>).

**`terminus-logger.service.ts`**

```typescript
import { Injectable, Scope, ConsoleLogger } from '@nestjs/common';

@Injectable({ scope: Scope.TRANSIENT })
export class TerminusLogger extends ConsoleLogger {
  error(message: any, stack?: string, context?: string): void;
  error(message: any, ...optionalParams: any[]): void;
  error(
    message: unknown,
    stack?: unknown,
    context?: unknown,
    ...rest: unknown[]
  ): void {
    // Здесь переопределите, как должны логироваться сообщения об ошибках
  }
}
```

После создания пользовательского логгера достаточно передать его в `TerminusModule.forRoot()` следующим образом.

**`health.module.ts`**

```typescript
@Module({
imports: [
  TerminusModule.forRoot({
    logger: TerminusLogger,
  }),
],
})
export class HealthModule {}
```

Чтобы полностью подавить любые сообщения от Terminus, включая сообщения об ошибках, настройте его так.

**`health.module.ts`**

```typescript
@Module({
imports: [
  TerminusModule.forRoot({
    logger: false,
  }),
],
})
export class HealthModule {}
```

Terminus позволяет настроить, как ошибки проверок работоспособности отображаются в логах.

| Стиль лога ошибок          | Описание                                                                                                                        | Пример                                                              |
|:------------------|:-----------------------------------------------------------------------------------------------------------------------------------|:---------------------------------------------------------------------|
| `json`  (по умолчанию) | При ошибке выводит сводку результата проверки работоспособности в виде JSON-объекта                                                     | <figure><img src="https://docs.nestjs.com/assets/Terminus_Error_Log_Json.png" /></figure>   |
| `pretty`          | При ошибке выводит сводку результата проверки в отформатированных рамках, выделяя успешные и ошибочные результаты | <figure><img src="https://docs.nestjs.com/assets/Terminus_Error_Log_Pretty.png" /></figure> |

Изменить стиль лога можно с помощью параметра конфигурации `errorLogStyle`, как в примере ниже.

**`health.module.ts`**

```typescript
@Module({
  imports: [
    TerminusModule.forRoot({
      errorLogStyle: 'pretty',
    }),
  ]
})
export class HealthModule {}
```

#### Тайм-аут плавного завершения работы

Если вашему приложению нужно отложить процесс завершения работы, Terminus может взять это на себя.
Такая настройка особенно полезна при работе с оркестратором вроде Kubernetes.
Задав задержку чуть больше интервала readiness-проверки, вы можете добиться нулевого простоя при остановке контейнеров.

**`health.module.ts`**

```typescript
@Module({
  imports: [
    TerminusModule.forRoot({
      gracefulShutdownTimeoutMs: 1000,
    }),
  ]
})
export class HealthModule {}
```
