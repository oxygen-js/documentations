### Nest Commander

В дополнение к документации об [автономных приложениях](<../10. Standalone apps.md>) существует пакет [nest-commander](https://jmcdo29.github.io/nest-commander) для написания консольных приложений в структуре, привычной по обычным приложениям Nest.

> **Информация** `nest-commander` — сторонний пакет, который не сопровождается основной командой NestJS. Сообщайте обо всех найденных в библиотеке проблемах в [соответствующем репозитории](https://github.com/jmcdo29/nest-commander/issues/new/choose)

#### Установка

Как и любой другой пакет, перед использованием его нужно установить.

```bash
$ npm i nest-commander
```

#### Файл команды

`nest-commander` позволяет легко писать новые консольные приложения с помощью [декораторов](https://www.typescriptlang.org/docs/handbook/decorators.html): декоратор `@Command()` применяется к классам, а декоратор `@Option()` — к методам этого класса. Каждый файл команды должен реализовывать абстрактный класс `CommandRunner` и быть помечен декоратором `@Command()`.

Nest рассматривает каждую команду как `@Injectable()`, поэтому привычное внедрение зависимостей работает так, как вы и ожидаете. Обратить внимание стоит только на абстрактный класс `CommandRunner`, который должна реализовывать каждая команда. Этот абстрактный класс гарантирует, что у всех команд есть метод `run`, возвращающий `Promise<void>` и принимающий параметры `string[], Record<string, any>`. Именно из метода `run` вы запускаете всю свою логику: он получает все параметры, не совпавшие с флагами опций, и передаёт их в виде массива — на случай, если вы действительно рассчитываете работать с несколькими параметрами. Что до опций (`Record<string, any>`), имена их свойств совпадают со значением `name`, переданным в декораторы `@Option()`, а значения — с тем, что вернул обработчик опции. Если вам нужна более строгая типизация, ничто не мешает описать для опций собственный интерфейс.

#### Запуск команды

Подобно тому как в приложении NestJS мы используем `NestFactory` для создания сервера и запускаем его через `listen`, пакет `nest-commander` предоставляет простой API для запуска. Импортируйте `CommandFactory`, вызовите `static`-метод `run` и передайте в него корневой модуль вашего приложения. Выглядеть это будет примерно так:

```ts
import { CommandFactory } from 'nest-commander';
import { AppModule } from './app.module';

async function bootstrap() {
  await CommandFactory.run(AppModule);
}

bootstrap();
```

По умолчанию при использовании `CommandFactory` логгер Nest отключён. Тем не менее его можно передать вторым аргументом функции `run`. Вы можете передать либо собственный логгер NestJS, либо массив уровней логирования, которые хотите оставить — здесь бывает полезно указать хотя бы `['error']`, если вам нужны только сообщения об ошибках Nest.

```ts
import { CommandFactory } from 'nest-commander';
import { AppModule } from './app.module';
import { LogService } from './log.service';

async function bootstrap() {
  await CommandFactory.run(AppModule, new LogService());

  // или, если нужны только предупреждения и ошибки Nest
  await CommandFactory.run(AppModule, ['warn', 'error']);
}

bootstrap();
```

Вот и всё. «Под капотом» `CommandFactory` сам позаботится о вызове `NestFactory` и о вызове `app.close()` там, где это нужно, так что об утечках памяти можно не беспокоиться. Если требуется обработка ошибок, всегда можно обернуть вызов `run` в `try/catch` либо добавить `.catch()` к вызову `bootstrap()`.

#### Тестирование

Какой смысл писать превосходный консольный скрипт, если его нельзя так же легко протестировать? К счастью, в `nest-commander` есть утилиты, которые прекрасно вписываются в экосистему NestJS и покажутся знакомыми любому, кто работал с Nest. Вместо `CommandFactory` для сборки команды в тестовом режиме используйте `CommandTestFactory` и передавайте в него метаданные — очень похоже на то, как работает `Test.createTestingModule` из `@nestjs/testing`. По сути, «под капотом» используется именно этот пакет. Вам также по-прежнему доступно построение цепочки с методами `overrideProvider` перед вызовом `compile()`, так что подменять части внедрения зависимостей можно прямо в тесте.

#### Собираем всё вместе

Приведённый ниже класс соответствует CLI-команде, которая принимает подкоманду `basic` либо вызывается напрямую, поддерживает флаги `-n`, `-s` и `-b` (вместе с их длинными вариантами) и использует собственные парсеры для каждой опции. Флаг `--help` также поддерживается, как это принято в commander.

```ts
import { Command, CommandRunner, Option } from 'nest-commander';
import { LogService } from './log.service';

interface BasicCommandOptions {
  string?: string;
  boolean?: boolean;
  number?: number;
}

@Command({ name: 'basic', description: 'A parameter parse' })
export class BasicCommand extends CommandRunner {
  constructor(private readonly logService: LogService) {
    super()
  }

  async run(
    passedParam: string[],
    options?: BasicCommandOptions,
  ): Promise<void> {
    if (options?.boolean !== undefined && options?.boolean !== null) {
      this.runWithBoolean(passedParam, options.boolean);
    } else if (options?.number) {
      this.runWithNumber(passedParam, options.number);
    } else if (options?.string) {
      this.runWithString(passedParam, options.string);
    } else {
      this.runWithNone(passedParam);
    }
  }

  @Option({
    flags: '-n, --number [number]',
    description: 'A basic number parser',
  })
  parseNumber(val: string): number {
    return Number(val);
  }

  @Option({
    flags: '-s, --string [string]',
    description: 'A string return',
  })
  parseString(val: string): string {
    return val;
  }

  @Option({
    flags: '-b, --boolean [boolean]',
    description: 'A boolean parser',
  })
  parseBoolean(val: string): boolean {
    return JSON.parse(val);
  }

  runWithString(param: string[], option: string): void {
    this.logService.log({ param, string: option });
  }

  runWithNumber(param: string[], option: number): void {
    this.logService.log({ param, number: option });
  }

  runWithBoolean(param: string[], option: boolean): void {
    this.logService.log({ param, boolean: option });
  }

  runWithNone(param: string[]): void {
    this.logService.log({ param });
  }
}
```

Убедитесь, что класс команды добавлен в модуль

```ts
@Module({
  providers: [LogService, BasicCommand],
})
export class AppModule {}
```

А теперь, чтобы запустить CLI, в файле main.ts достаточно написать следующее

```ts
async function bootstrap() {
  await CommandFactory.run(AppModule);
}

bootstrap();
```

Вот так просто у вас появилось консольное приложение.
