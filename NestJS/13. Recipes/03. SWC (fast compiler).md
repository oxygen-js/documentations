### SWC

[SWC](https://swc.rs/) (Speedy Web Compiler) — расширяемая платформа на Rust, которую можно использовать и для компиляции, и для сборки.
Связка SWC с Nest CLI — отличный и простой способ существенно ускорить разработку.

> **Подсказка** SWC примерно **в 20 раз быстрее** стандартного компилятора TypeScript.

#### Установка

Для начала установите несколько пакетов:

```bash
$ npm i --save-dev @swc/cli @swc/core
```

#### Начало работы

После завершения установки вы можете использовать сборщик `swc` вместе с Nest CLI:

```bash
$ nest start -b swc
# OR nest start --builder swc
```

> **Подсказка** Если ваш репозиторий — монорепозиторий, загляните в [этот раздел](<03. SWC (fast compiler).md#монорепозиторий>).

Вместо передачи флага `-b` можно просто задать свойству `compilerOptions.builder` значение `"swc"` в файле `nest-cli.json`:

```json
{
  "compilerOptions": {
    "builder": "swc"
  }
}
```

Чтобы настроить поведение сборщика, передайте объект с двумя атрибутами — `type` (`"swc"`) и `options`:

```json
{
  "compilerOptions": {
    "builder": {
      "type": "swc",
      "options": {
        "swcrcPath": "infrastructure/.swcrc",
      }
    }
  }
}
```

Например, чтобы swc компилировал файлы `.jsx` и `.tsx`, сделайте так:

```json
{
  "compilerOptions": {
    "builder": {
      "type": "swc",
      "options": { "extensions": [".ts", ".tsx", ".js", ".jsx"] }
    },
  }
}
```

Чтобы запустить приложение в режиме отслеживания изменений, выполните команду:

```bash
$ nest start -b swc -w
# OR nest start --builder swc --watch
```

#### Проверка типов

SWC сам по себе не выполняет проверку типов (в отличие от стандартного компилятора TypeScript), поэтому для её включения нужен флаг `--type-check`:

```bash
$ nest start -b swc --type-check
```

Эта команда укажет Nest CLI запускать `tsc` в режиме `noEmit` параллельно с SWC, что обеспечит асинхронную проверку типов. И здесь вместо передачи флага `--type-check` можно просто задать свойству `compilerOptions.typeCheck` значение `true` в файле `nest-cli.json`:

```json
{
  "compilerOptions": {
    "builder": "swc",
    "typeCheck": true
  }
}
```

#### Плагины CLI (SWC)

Флаг `--type-check` автоматически запустит **плагины NestJS CLI** и создаст файл сериализованных метаданных, который затем сможет загрузить приложение во время выполнения.

#### Конфигурация SWC

Сборщик SWC уже преднастроен под требования NestJS-приложений. Тем не менее вы можете изменить конфигурацию, создав файл `.swcrc` в корневом каталоге и настроив параметры по своему усмотрению.

```json
{
  "$schema": "https://swc.rs/schema.json",
  "sourceMaps": true,
  "jsc": {
    "parser": {
      "syntax": "typescript",
      "decorators": true,
      "dynamicImport": true
    },
    "baseUrl": "./"
  },
  "minify": false
}
```

#### Монорепозиторий

Если ваш репозиторий — монорепозиторий, то вместо сборщика `swc` придётся настроить `webpack` на использование `swc-loader`.

Сначала установим нужный пакет:

```bash
$ npm i --save-dev swc-loader
```

После установки создайте в корневом каталоге приложения файл `webpack.config.js` со следующим содержимым:

```js
const swcDefaultConfig = require('@nestjs/cli/lib/compiler/defaults/swc-defaults').swcDefaultsFactory().swcOptions;

module.exports = {
  module: {
    rules: [
      {
        test: /\.ts$/,
        exclude: /node_modules/,
        use: {
          loader: 'swc-loader',
          options: swcDefaultConfig,
        },
      },
    ],
  },
};
```

#### Монорепозиторий и плагины CLI

Если вы используете плагины CLI, `swc-loader` не загрузит их автоматически. Вместо этого придётся создать отдельный файл, который загрузит их вручную. Для этого
объявите файл `generate-metadata.ts` рядом с файлом `main.ts` со следующим содержимым:

```ts
import { PluginMetadataGenerator } from '@nestjs/cli/lib/compiler/plugins/plugin-metadata-generator';
import { ReadonlyVisitor } from '@nestjs/swagger/dist/plugin';

const generator = new PluginMetadataGenerator();
generator.generate({
  visitors: [new ReadonlyVisitor({ introspectComments: true, pathToSource: __dirname })],
  outputDir: __dirname,
  watch: true,
  tsconfigPath: 'apps/<name>/tsconfig.app.json',
});
```

> **Подсказка** В этом примере мы использовали плагин `@nestjs/swagger`, но вы можете взять любой плагин на свой выбор.

Метод `generate()` принимает следующие опции:

|                    |                                                                                                |
| ------------------ | ---------------------------------------------------------------------------------------------- |
| `watch`            | Отслеживать ли изменения в проекте.                                                            |
| `tsconfigPath`     | Путь к файлу `tsconfig.json`. Указывается относительно текущего рабочего каталога (`process.cwd()`). |
| `outputDir`        | Путь к каталогу, в который будет сохранён файл метаданных.                                     |
| `visitors`         | Массив visitor'ов, которые будут использованы для генерации метаданных.                        |
| `filename`         | Имя файла метаданных. По умолчанию `metadata.ts`.                                              |
| `printDiagnostics` | Выводить ли диагностику в консоль. По умолчанию `true`.                                        |

Наконец, скрипт `generate-metadata` можно запустить в отдельном окне терминала следующей командой:

```bash
$ npx ts-node src/generate-metadata.ts
# OR npx ts-node apps/{YOUR_APP}/src/generate-metadata.ts
```

#### Типичные подводные камни

Если в приложении используется TypeORM/MikroORM или любая другая ORM, вы можете столкнуться с проблемами циклических импортов. SWC плохо справляется с **циклическими импортами**, поэтому применяйте следующий обходной путь:

```typescript
@Entity()
export class User {
  @OneToOne(() => Profile, (profile) => profile.user)
  profile: Relation<Profile>; // <--- обратите внимание на тип "Relation<>" вместо просто "Profile"
}
```

> **Подсказка** Тип `Relation` экспортируется из пакета `typeorm`.

Это не даёт типу свойства попасть в метаданные свойства в транспилированном коде и тем самым предотвращает проблемы с циклическими зависимостями.

Если ваша ORM не предлагает похожего обходного пути, вы можете определить тип-обёртку самостоятельно:

```typescript
/**
 * Тип-обёртка, позволяющий обойти проблему циклических зависимостей
 * ESM-модулей, вызванную сохранением типа свойства в метаданных рефлексии.
 */
export type WrapperType<T> = T; // WrapperType === Relation
```

Для всех [циклических внедрений зависимостей](<../03. Fundamentals/05. Circular dependency.md>) в вашем проекте также потребуется использовать описанный выше пользовательский тип-обёртку:

```typescript
@Injectable()
export class UsersService {
  constructor(
    @Inject(forwardRef(() => ProfileService))
    private readonly profileService: WrapperType<ProfileService>,
  ) {};
}
```

### Jest + SWC

Чтобы использовать SWC вместе с Jest, установите следующие пакеты:

```bash
$ npm i --save-dev jest @swc/core @swc/jest
```

После установки обновите файл `package.json`/`jest.config.js` (в зависимости от вашей конфигурации), добавив следующее содержимое:

```json
{
  "jest": {
    "transform": {
      "^.+\\.(t|j)s?$": ["@swc/jest"]
    }
  }
}
```

Кроме того, в файл `.swcrc` понадобится добавить следующие свойства `transform`: `legacyDecorator`, `decoratorMetadata`:

```json
{
  "$schema": "https://swc.rs/schema.json",
  "sourceMaps": true,
  "jsc": {
    "parser": {
      "syntax": "typescript",
      "decorators": true,
      "dynamicImport": true
    },
    "transform": {
      "legacyDecorator": true,
      "decoratorMetadata": true
    },
    "baseUrl": "./"
  },
  "minify": false
}
```

Если в проекте используются плагины NestJS CLI, вам придётся запускать `PluginMetadataGenerator` вручную. Подробнее об этом — в [этом разделе](<03. SWC (fast compiler).md#монорепозиторий-и-плагины-cli>).

### Vitest

[Vitest](https://vitest.dev/) — быстрый и лёгкий тест-раннер, спроектированный для работы с Vite. Он предлагает современное, быстрое и удобное решение для тестирования, которое можно интегрировать с NestJS-проектами.

#### Установка

Для начала установите нужные пакеты:

```bash
$ npm i --save-dev vitest unplugin-swc @swc/core @vitest/coverage-v8
```

#### Конфигурация

Создайте в корневом каталоге приложения файл `vitest.config.ts` со следующим содержимым:

```ts
import swc from 'unplugin-swc';
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    globals: true,
    root: './',
  },
  plugins: [
    // Это необходимо для сборки тестовых файлов с помощью SWC
    swc.vite({
      // Явно задаём тип модуля, чтобы это значение не унаследовалось из файла конфигурации `.swcrc`
      module: { type: 'es6' },
    }),
  ],
  resolve: {
    alias: {
      // Гарантирует, что Vitest корректно разрешает алиасы путей TypeScript
      'src': resolve(__dirname, './src'),
    },
  },
});
```

Этот файл конфигурации настраивает окружение Vitest, корневой каталог и плагин SWC. Также стоит создать отдельный файл конфигурации
для e2e-тестов с дополнительным полем `include`, задающим регулярное выражение для путей к тестам:

```ts
import swc from 'unplugin-swc';
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    include: ['**/*.e2e-spec.ts'],
    globals: true,
    root: './',
  },
  plugins: [swc.vite()],
});
```

Кроме того, вы можете задать опции `alias`, чтобы поддержать пути TypeScript в тестах:

```ts
import swc from 'unplugin-swc';
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    include: ['**/*.e2e-spec.ts'],
    globals: true,
    alias: {
      '@src': './src',
      '@test': './test',
    },
    root: './',
  },
  resolve: {
    alias: {
      '@src': './src',
      '@test': './test',
    },
  },
  plugins: [swc.vite()],
});
```

### Алиасы путей

В отличие от Jest, Vitest не разрешает алиасы путей TypeScript вроде `src/` автоматически. Это может приводить к ошибкам разрешения зависимостей во время тестирования. Чтобы решить эту проблему, добавьте в файл `vitest.config.ts` следующую конфигурацию `resolve.alias`:

```ts
import { resolve } from 'path';

export default defineConfig({
  resolve: {
    alias: {
      'src': resolve(__dirname, './src'),
    },
  },
});
```
Это гарантирует, что Vitest корректно разрешит импорты модулей, и предотвратит ошибки, связанные с отсутствующими зависимостями.

#### Обновление импортов в E2E-тестах

Замените в E2E-тестах импорты вида `import * as request from 'supertest'` на `import request from 'supertest'`. Это необходимо, потому что Vitest в связке с Vite ожидает импорт по умолчанию для supertest. Импорт через пространство имён в этой конфигурации может вызвать проблемы.

Наконец, обновите тестовые скрипты в файле package.json следующим образом:

```json
{
  "scripts": {
    "test": "vitest run",
    "test:watch": "vitest",
    "test:cov": "vitest run --coverage",
    "test:debug": "vitest --inspect-brk --inspect --logHeapUsage --threads=false",
    "test:e2e": "vitest run --config ./vitest.config.e2e.ts"
  }
}
```

Эти скрипты настраивают Vitest на запуск тестов, отслеживание изменений, формирование отчётов о покрытии кода и отладку. Скрипт test:e2e предназначен именно для запуска E2E-тестов с отдельным файлом конфигурации.

С такой настройкой вы сможете пользоваться преимуществами Vitest в вашем NestJS-проекте, включая более быстрое выполнение тестов и более современный опыт тестирования.

> **Подсказка** Рабочий пример можно посмотреть в этом [репозитории](https://github.com/TrilonIO/nest-vitest)
