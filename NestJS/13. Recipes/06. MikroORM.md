### MikroORM

Этот рецепт поможет начать работу с MikroORM в Nest. MikroORM — это TypeScript ORM для Node.js, построенная на паттернах Data Mapper, Unit of Work и Identity Map. Это отличная альтернатива TypeORM, и переход с TypeORM должен пройти довольно легко. Полную документацию по MikroORM можно найти [здесь](https://mikro-orm.io/docs).

> **Информация** `@mikro-orm/nestjs` — сторонний пакет, который не поддерживается основной командой NestJS. Обо всех найденных в библиотеке проблемах сообщайте в [соответствующий репозиторий](https://github.com/mikro-orm/nestjs).

#### Установка

Проще всего интегрировать MikroORM в Nest через [модуль `@mikro-orm/nestjs`](https://github.com/mikro-orm/nestjs).
Просто установите его рядом с Nest, MikroORM и нужным драйвером:

```bash
$ npm i @mikro-orm/core @mikro-orm/nestjs @mikro-orm/sqlite
```

MikroORM также поддерживает `postgres`, `sqlite` и `mongo`. Полный список драйверов см. в [официальной документации](https://mikro-orm.io/docs/usage-with-sql/).

После установки можно импортировать `MikroOrmModule` в корневой `AppModule`.

```typescript
import { SqliteDriver } from '@mikro-orm/sqlite';

@Module({
  imports: [
    MikroOrmModule.forRoot({
      entities: ['./dist/entities'],
      entitiesTs: ['./src/entities'],
      dbName: 'my-db-name.sqlite3',
      driver: SqliteDriver,
    }),
  ],
  controllers: [AppController],
  providers: [AppService],
})
export class AppModule {}
```

Метод `forRoot()` принимает тот же объект конфигурации, что и `init()` из пакета MikroORM. Полное описание конфигурации приведено на [этой странице](https://mikro-orm.io/docs/configuration).

Как вариант, можно [настроить CLI](https://mikro-orm.io/docs/installation#setting-up-the-commandline-tool), создав файл конфигурации `mikro-orm.config.ts`, и затем вызывать `forRoot()` без аргументов.

```typescript
@Module({
  imports: [
    MikroOrmModule.forRoot(),
  ],
  ...
})
export class AppModule {}
```

Но это не сработает со сборщиками, применяющими tree shaking: в таком случае лучше передавать конфигурацию явно:

```typescript
import config from './mikro-orm.config'; // ваша конфигурация ORM

@Module({
  imports: [
    MikroOrmModule.forRoot(config),
  ],
  ...
})
export class AppModule {}
```

После этого `EntityManager` можно будет внедрять в любом месте проекта (без импорта каких-либо модулей где-либо ещё).

```ts
// Импортируйте всё из пакета вашего драйвера или из `@mikro-orm/knex`
import { EntityManager, MikroORM } from '@mikro-orm/sqlite';

@Injectable()
export class MyService {
  constructor(
    private readonly orm: MikroORM,
    private readonly em: EntityManager,
  ) {}
}
```

> **Информация** Обратите внимание, что `EntityManager` импортируется из пакета `@mikro-orm/driver`, где driver — это `mysql`, `sqlite`, `postgres` или тот драйвер, который вы используете. Если в зависимостях установлен `@mikro-orm/knex`, `EntityManager` можно импортировать и оттуда.

#### Репозитории

MikroORM поддерживает паттерн проектирования «репозиторий». Для каждой сущности можно создать репозиторий. Полную документацию по репозиториям читайте [здесь](https://mikro-orm.io/docs/repositories). Чтобы указать, какие репозитории следует зарегистрировать в текущей области видимости, используйте метод `forFeature()`. Например, вот так:

> **Информация** Базовые сущности регистрировать через `forFeature()` **не нужно** — репозиториев для них нет.
> С другой стороны, базовые сущности должны входить в список в `forRoot()` (или в конфигурации ORM в целом).

```typescript
// photo.module.ts
@Module({
  imports: [MikroOrmModule.forFeature([Photo])],
  providers: [PhotoService],
  controllers: [PhotoController],
})
export class PhotoModule {}
```

и импортировать его в корневой `AppModule`:

```typescript
// app.module.ts
@Module({
  imports: [MikroOrmModule.forRoot(...), PhotoModule],
})
export class AppModule {}
```

Так мы сможем внедрить `PhotoRepository` в `PhotoService` с помощью декоратора `@InjectRepository()`:

```typescript
@Injectable()
export class PhotoService {
  constructor(
    @InjectRepository(Photo)
    private readonly photoRepository: EntityRepository<Photo>,
  ) {}
}
```

#### Использование пользовательских репозиториев

При использовании пользовательских репозиториев декоратор `@InjectRepository()` больше не нужен, так как Nest DI разрешает зависимости по ссылкам на классы.

```ts
// `**./author.entity.ts**`
@Entity({ repository: () => AuthorRepository })
export class Author {
  // чтобы работал вывод типов в `em.getRepository()`
  [EntityRepositoryType]?: AuthorRepository;
}

// `**./author.repository.ts**`
export class AuthorRepository extends EntityRepository<Author> {
  // ваши собственные методы...
}
```

Поскольку имя пользовательского репозитория совпадает с тем, что вернул бы `getRepositoryToken()`, декоратор `@InjectRepository()` больше не требуется:

```ts
@Injectable()
export class MyService {
  constructor(private readonly repo: AuthorRepository) {}
}
```

#### Автоматическая загрузка сущностей

Добавлять сущности в массив entities параметров подключения вручную довольно утомительно. Кроме того, ссылки на сущности из корневого модуля нарушают границы предметных областей приложения и приводят к утечке деталей реализации в другие его части. Чтобы решить эту проблему, можно использовать статические glob-пути.

Однако учтите, что glob-пути не поддерживаются webpack, поэтому при сборке приложения в монорепозитории воспользоваться ими не получится. Для таких случаев предусмотрено альтернативное решение. Чтобы сущности загружались автоматически, задайте свойству `autoLoadEntities` объекта конфигурации (передаваемого в метод `forRoot()`) значение `true`, как показано ниже:

```ts
@Module({
  imports: [
    MikroOrmModule.forRoot({
      ...
      autoLoadEntities: true,
    }),
  ],
})
export class AppModule {}
```

С этой опцией каждая сущность, зарегистрированная через метод `forFeature()`, будет автоматически добавлена в массив entities объекта конфигурации.

> **Информация** Учтите, что сущности, не зарегистрированные через метод `forFeature()`, а лишь упомянутые
> в другой сущности (через связь), настройка `autoLoadEntities` не подхватит.

> **Информация** Кроме того, `autoLoadEntities` никак не влияет на CLI MikroORM — для него по-прежнему
> нужна конфигурация с полным списком сущностей. Зато там можно использовать glob-шаблоны, так как
> CLI не проходит через webpack.

#### Сериализация

> **Примечание** MikroORM оборачивает каждую связь сущности в объект `Reference<T>` или `Collection<T>`, чтобы обеспечить лучшую типобезопасность. Из-за этого [встроенный сериализатор Nest](<../04. Techniques/06. Serialization.md>) не увидит обёрнутые связи. Иными словами, если вы возвращаете сущности MikroORM из обработчиков HTTP или WebSocket, ни одна из их связей сериализована НЕ будет.

К счастью, у MikroORM есть [API сериализации](https://mikro-orm.io/docs/serializing), которым можно пользоваться вместо `ClassSerializerInterceptor`.

```typescript
@Entity()
export class Book {
  @Property({ hidden: true }) // Эквивалент `@Exclude` из class-transformer
  hiddenField = Date.now();

  @Property({ persist: false }) // Аналог `@Expose()` из class-transformer. Будет существовать только в памяти и попадёт в сериализацию.
  count?: number;

  @ManyToOne({
    serializer: (value) => value.name,
    serializedName: 'authorName',
  }) // Эквивалент `@Transform()` из class-transformer
  author: Author;
}
```

#### Обработчики очередей с областью видимости запроса

Как сказано в [документации](https://mikro-orm.io/docs/identity-map), для каждого запроса нужно чистое состояние. Это происходит автоматически благодаря помощнику `RequestContext`, зарегистрированному через middleware.

Но middleware выполняются только для обычных обработчиков HTTP-запросов — а что, если метод с областью видимости запроса нужен вне их? Пример — обработчики очередей или задачи по расписанию.

Здесь пригодится декоратор `@CreateRequestContext()`. Он требует, чтобы вы сначала внедрили экземпляр `MikroORM` в текущий контекст: именно он будет использован для создания контекста. «Под капотом» декоратор регистрирует для вашего метода новый контекст запроса и выполняет метод внутри этого контекста.

```ts
@Injectable()
export class MyService {
  constructor(private readonly orm: MikroORM) {}

  @CreateRequestContext()
  async doSomething() {
    // это выполнится в отдельном контексте
  }
}
```

> **Примечание** Как следует из названия, этот декоратор всегда создаёт новый контекст — в отличие от альтернативного `@EnsureRequestContext`, который создаёт контекст, только если код ещё не выполняется внутри другого.

#### Тестирование

Пакет `@mikro-orm/nestjs` предоставляет функцию `getRepositoryToken()`, которая возвращает готовый токен для заданной сущности, что позволяет подменять репозиторий заглушкой.

```typescript
@Module({
  providers: [
    PhotoService,
    {
      // или, если у вас пользовательский репозиторий: `provide: PhotoRepository`
      provide: getRepositoryToken(Photo),
      useValue: mockedRepository,
    },
  ],
})
export class PhotoModule {}
```
