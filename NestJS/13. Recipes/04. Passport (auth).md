### Passport (аутентификация)

[Passport](https://github.com/jaredhanson/passport) — самая популярная библиотека аутентификации для node.js, хорошо известная сообществу и успешно применяемая во многих промышленных приложениях. Интегрировать её с приложением на **Nest** несложно с помощью модуля `@nestjs/passport`. На верхнем уровне Passport выполняет последовательность шагов, чтобы:

- Аутентифицировать пользователя, проверив его «учётные данные» (например, имя пользователя и пароль, JSON Web Token ([JWT](https://jwt.io/)) или identity-токен от провайдера идентификации)
- Управлять состоянием аутентификации (выдавая переносимый токен, такой как JWT, или создавая [сессию Express](https://github.com/expressjs/session))
- Прикреплять информацию об аутентифицированном пользователе к объекту `Request` для дальнейшего использования в обработчиках маршрутов

У Passport богатая экосистема [стратегий](http://www.passportjs.org/), реализующих различные механизмы аутентификации. Идея проста, но набор доступных стратегий велик и очень разнообразен. Passport абстрагирует эти разнородные шаги в единый стандартный шаблон, а модуль `@nestjs/passport` оборачивает и приводит этот шаблон к привычным конструкциям Nest.

В этой главе мы реализуем полноценное сквозное решение для аутентификации в RESTful API-сервере с помощью этих мощных и гибких модулей. Описанные здесь идеи можно применить к любой стратегии Passport, чтобы настроить свою схему аутентификации. Следуя шагам этой главы, вы построите весь пример целиком.

#### Требования к аутентификации

Опишем требования подробнее. В нашем сценарии клиенты сначала аутентифицируются по имени пользователя и паролю. После успешной аутентификации сервер выдаёт JWT, который можно отправлять как [bearer-токен в заголовке authorization](https://tools.ietf.org/html/rfc6750) в последующих запросах для подтверждения аутентификации. Мы также создадим защищённый маршрут, доступный только тем запросам, которые содержат действительный JWT.

Начнём с первого требования — аутентификации пользователя. Затем расширим его выдачей JWT. Наконец, создадим защищённый маршрут, проверяющий наличие действительного JWT в запросе.

Сначала нужно установить необходимые пакеты. Passport предоставляет стратегию [passport-local](https://github.com/jaredhanson/passport-local), реализующую механизм аутентификации по имени пользователя и паролю, — как раз то, что нужно для этой части сценария.

```bash
$ npm install --save @nestjs/passport passport passport-local
$ npm install --save-dev @types/passport-local
```

> **Важно** Для **любой** выбранной стратегии Passport вам всегда понадобятся пакеты `@nestjs/passport` и `passport`. Затем нужно установить пакет конкретной стратегии (например, `passport-jwt` или `passport-local`), реализующий нужный механизм аутентификации. Кроме того, вы можете установить определения типов для стратегии Passport — как показано выше с `@types/passport-local`, — что упрощает написание кода на TypeScript.

#### Реализация стратегий Passport

Теперь мы готовы реализовать аутентификацию. Начнём с обзора процесса, применимого к **любой** стратегии Passport. Полезно воспринимать Passport как маленький самостоятельный фреймворк. Изящество этого фреймворка в том, что он сводит процесс аутентификации к нескольким базовым шагам, которые вы настраиваете под реализуемую стратегию. Он похож на фреймворк потому, что настраивается через параметры (обычные JSON-объекты) и собственный код в виде функций обратного вызова, которые Passport вызывает в нужный момент. Модуль `@nestjs/passport` оборачивает этот фреймворк в пакет в стиле Nest, упрощая интеграцию в приложение Nest. Ниже мы будем использовать `@nestjs/passport`, но сначала рассмотрим, как работает **чистый Passport**.

В чистом Passport стратегия настраивается двумя вещами:

1. Набором опций, специфичных для этой стратегии. Например, в JWT-стратегии вы можете указать секрет для подписи токенов.
2. «Проверяющим обратным вызовом» (verify callback) — здесь вы сообщаете Passport, как взаимодействовать с вашим хранилищем пользователей (где вы управляете учётными записями). Здесь вы проверяете, существует ли пользователь (и/или создаёте нового) и действительны ли его учётные данные. Библиотека Passport ожидает, что этот обратный вызов вернёт полного пользователя в случае успешной проверки либо null в случае неудачи (неудачей считается либо отсутствие пользователя, либо, в случае passport-local, несовпадение пароля).

В `@nestjs/passport` стратегия Passport настраивается наследованием от класса `PassportStrategy`. Опции стратегии (пункт 1 выше) вы передаёте, вызывая метод `super()` в своём подклассе — при желании с объектом опций. Проверяющий обратный вызов (пункт 2 выше) вы реализуете в виде метода `validate()` в этом же подклассе.

Начнём с генерации `AuthModule` и внутри него — `AuthService`:

```bash
$ nest g module auth
$ nest g service auth
```

По ходу реализации `AuthService` окажется удобным инкапсулировать операции с пользователями в `UsersService`, так что сгенерируем этот модуль и сервис прямо сейчас:

```bash
$ nest g module users
$ nest g service users
```

Замените содержимое сгенерированных файлов, как показано ниже. В нашем демонстрационном приложении `UsersService` просто хранит захардкоженный список пользователей в памяти и предоставляет метод поиска по имени пользователя. В реальном приложении именно здесь вы построили бы модель пользователя и слой хранения на основе выбранной библиотеки (например, TypeORM, Sequelize, Mongoose и т. д.).

**`users/users.service.ts`**

```typescript
import { Injectable } from '@nestjs/common';

// Здесь должен быть настоящий класс/интерфейс, представляющий сущность пользователя
export type User = any;

@Injectable()
export class UsersService {
  private readonly users = [
    {
      userId: 1,
      username: 'john',
      password: 'changeme',
    },
    {
      userId: 2,
      username: 'maria',
      password: 'guess',
    },
  ];

  async findOne(username: string): Promise<User | undefined> {
    return this.users.find(user => user.username === username);
  }
}
```

В `UsersModule` требуется единственное изменение — добавить `UsersService` в массив `exports` декоратора `@Module`, чтобы сервис был виден за пределами модуля (вскоре мы используем его в `AuthService`).

**`users/users.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { UsersService } from './users.service';

@Module({
  providers: [UsersService],
  exports: [UsersService],
})
export class UsersModule {}
```

Задача нашего `AuthService` — получить пользователя и проверить пароль. Для этого создадим метод `validateUser()`. В коде ниже мы используем удобный spread-оператор ES6, чтобы убрать свойство с паролем из объекта пользователя перед возвратом. Вскоре мы будем вызывать метод `validateUser()` из локальной стратегии Passport.

**`auth/auth.service.ts`**

```typescript
import { Injectable } from '@nestjs/common';
import { UsersService } from '../users/users.service';

@Injectable()
export class AuthService {
  constructor(private usersService: UsersService) {}

  async validateUser(username: string, pass: string): Promise<any> {
    const user = await this.usersService.findOne(username);
    if (user && user.password === pass) {
      const { password, ...result } = user;
      return result;
    }
    return null;
  }
}
```

> **Предупреждение** Разумеется, в настоящем приложении пароль не хранят в открытом виде. Вместо этого используют библиотеку вроде [bcrypt](https://github.com/kelektiv/node.bcrypt.js#readme) с односторонним алгоритмом хэширования с солью. При таком подходе вы храните только хэши паролей и сравниваете сохранённый пароль с хэшем **входящего** пароля, никогда не сохраняя и не раскрывая пароли пользователей в открытом виде. Чтобы не усложнять демонстрационное приложение, мы нарушаем это безусловное правило и используем открытый текст. **Не делайте так в реальном приложении!**

Теперь обновим `AuthModule`, чтобы он импортировал `UsersModule`.

**`auth/auth.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { AuthService } from './auth.service';
import { UsersModule } from '../users/users.module';

@Module({
  imports: [UsersModule],
  providers: [AuthService],
})
export class AuthModule {}
```

#### Реализация Passport local

Теперь можно реализовать **локальную стратегию аутентификации** Passport. Создайте файл `local.strategy.ts` в папке `auth` и добавьте следующий код:

**`auth/local.strategy.ts`**

```typescript
import { Strategy } from 'passport-local';
import { PassportStrategy } from '@nestjs/passport';
import { Injectable, UnauthorizedException } from '@nestjs/common';
import { AuthService } from './auth.service';

@Injectable()
export class LocalStrategy extends PassportStrategy(Strategy) {
  constructor(private authService: AuthService) {
    super();
  }

  async validate(username: string, password: string): Promise<any> {
    const user = await this.authService.validateUser(username, password);
    if (!user) {
      throw new UnauthorizedException();
    }
    return user;
  }
}
```

Мы следовали описанному ранее рецепту, общему для всех стратегий Passport. В нашем случае с passport-local опций настройки нет, поэтому конструктор просто вызывает `super()` без объекта опций.

> **Подсказка** В вызов `super()` можно передать объект опций, чтобы изменить поведение стратегии Passport. В этом примере стратегия passport-local по умолчанию ожидает в теле запроса свойства `username` и `password`. Передайте объект опций, чтобы указать другие имена свойств, например: `super({ usernameField: 'email' })`. Подробнее — в [документации Passport](http://www.passportjs.org/docs/configure/).

Мы также реализовали метод `validate()`. Для каждой стратегии Passport вызывает проверяющую функцию (в `@nestjs/passport` она реализуется методом `validate()`) с подходящим для этой стратегии набором параметров. Для локальной стратегии Passport ожидает метод `validate()` со следующей сигнатурой: `validate(username: string, password:string): any`.

Основная работа по проверке выполняется в `AuthService` (с помощью `UsersService`), поэтому этот метод довольно прост. Метод `validate()` для **любой** стратегии Passport строится по схожему принципу и различается лишь тем, как представлены учётные данные. Если пользователь найден и данные верны, он возвращается, чтобы Passport мог завершить свою работу (например, создать свойство `user` в объекте `Request`) и конвейер обработки запроса продолжился. Если пользователь не найден, мы выбрасываем исключение и передаём его <a href="../02.%20Overview/06.%20Exception%20filters.md">слою исключений</a>.

Как правило, единственное существенное различие метода `validate()` между стратегиями — **как именно** вы определяете, существует ли пользователь и действителен ли он. Например, в JWT-стратегии в зависимости от требований мы можем проверять, соответствует ли `userId` из декодированного токена записи в базе пользователей или списку отозванных токенов. Благодаря этому подход с наследованием и реализацией специфичной для стратегии проверки остаётся единообразным, изящным и расширяемым.

Осталось настроить `AuthModule` на использование только что описанных возможностей Passport. Приведите `auth.module.ts` к следующему виду:

**`auth/auth.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { AuthService } from './auth.service';
import { UsersModule } from '../users/users.module';
import { PassportModule } from '@nestjs/passport';
import { LocalStrategy } from './local.strategy';

@Module({
  imports: [UsersModule, PassportModule],
  providers: [AuthService, LocalStrategy],
})
export class AuthModule {}
```

#### Встроенные guard'ы Passport

В главе <a href="../02.%20Overview/08.%20Guards.md">Guards</a> описана основная задача guard'ов: определять, будет ли запрос обработан обработчиком маршрута. Это по-прежнему верно, и вскоре мы воспользуемся этой стандартной возможностью. Однако в контексте модуля `@nestjs/passport` появляется небольшой нюанс, который поначалу может сбивать с толку, поэтому обсудим его сразу. С точки зрения аутентификации ваше приложение может находиться в двух состояниях:

1. пользователь/клиент **не** вошёл в систему (не аутентифицирован)
2. пользователь/клиент **вошёл** в систему (аутентифицирован)

В первом случае (пользователь не вошёл) нам нужно выполнять две разные задачи:

- Ограничивать маршруты, доступные неаутентифицированному пользователю (то есть запрещать доступ к закрытым маршрутам). Для этого мы используем guard'ы в их привычной роли, повесив guard на защищённые маршруты. Как вы, вероятно, догадываетесь, в этом guard'е мы будем проверять наличие действительного JWT, поэтому займёмся им позже — когда научимся выдавать JWT.

- Инициировать сам **шаг аутентификации**, когда ранее неаутентифицированный пользователь пытается войти. Именно на этом шаге мы **выдаём** JWT действительному пользователю. Немного подумав, мы понимаем, что для начала аутентификации нужно отправить имя пользователя и пароль методом `POST`, поэтому создадим для этого маршрут `POST /auth/login`. Возникает вопрос: как именно вызвать стратегию passport-local в этом маршруте?

Ответ прост: с помощью другого, немного иного типа guard'а. Модуль `@nestjs/passport` предоставляет встроенный guard, который делает это за нас. Этот guard вызывает стратегию Passport и запускает описанные выше шаги (получение учётных данных, выполнение проверяющей функции, создание свойства `user` и т. д.).

Второй перечисленный выше случай (вошедший пользователь) опирается на обычный тип guard'а, который мы уже обсуждали, — он открывает вошедшим пользователям доступ к защищённым маршрутам.

#### Маршрут входа

Стратегия готова, и теперь можно реализовать простейший маршрут `/auth/login` и применить встроенный guard, запускающий поток passport-local.

Откройте файл `app.controller.ts` и замените его содержимое следующим:

**`app.controller.ts`**

```typescript
import { Controller, Request, Post, UseGuards } from '@nestjs/common';
import { AuthGuard } from '@nestjs/passport';

@Controller()
export class AppController {
  @UseGuards(AuthGuard('local'))
  @Post('auth/login')
  async login(@Request() req) {
    return req.user;
  }
}
```

С помощью `@UseGuards(AuthGuard('local'))` мы используем `AuthGuard`, который `@nestjs/passport` **автоматически подготовил** для нас, когда мы унаследовались от стратегии passport-local. Разберём это подробнее. Наша локальная стратегия Passport имеет имя по умолчанию `'local'`. Мы указываем это имя в декораторе `@UseGuards()`, чтобы связать его с кодом из пакета `passport-local`. Это нужно, чтобы однозначно определить, какую стратегию вызывать, если в приложении их несколько (каждая может предоставлять свой `AuthGuard`). Пока у нас всего одна стратегия, но вскоре появится вторая, поэтому такое различение необходимо.

Чтобы протестировать маршрут, пусть `/auth/login` пока просто возвращает пользователя. Это заодно демонстрирует ещё одну возможность Passport: он автоматически создаёт объект `user` на основе значения, возвращённого из метода `validate()`, и присваивает его объекту `Request` как `req.user`. Позже мы заменим это кодом, создающим и возвращающим JWT.

Поскольку это API-маршруты, протестируем их широко доступной утилитой [cURL](https://curl.haxx.se/). Вы можете использовать любой из объектов `user`, захардкоженных в `UsersService`.

```bash
$ # POST to /auth/login
$ curl -X POST http://localhost:3000/auth/login -d '{"username": "john", "password": "changeme"}' -H "Content-Type: application/json"
$ # result -> {"userId":1,"username":"john"}
```

Хотя это работает, передача имени стратегии напрямую в `AuthGuard()` вносит в код магические строки. Вместо этого мы рекомендуем создать собственный класс, как показано ниже:

**`auth/local-auth.guard.ts`**

```typescript
import { Injectable } from '@nestjs/common';
import { AuthGuard } from '@nestjs/passport';

@Injectable()
export class LocalAuthGuard extends AuthGuard('local') {}
```

Теперь можно обновить обработчик маршрута `/auth/login` и использовать `LocalAuthGuard`:

```typescript
@UseGuards(LocalAuthGuard)
@Post('auth/login')
async login(@Request() req) {
  return req.user;
}
```

#### Маршрут выхода

Для выхода можно создать дополнительный маршрут, вызывающий `req.logout()` для очистки сессии пользователя. Это типичный подход при аутентификации на основе сессий, но он неприменим к JWT.

```typescript
@UseGuards(LocalAuthGuard)
@Post('auth/logout')
async logout(@Request() req) {
  return req.logout();
}
```

#### Работа с JWT

Мы готовы перейти к JWT-части нашей системы аутентификации. Уточним требования:

- Позволить пользователям аутентифицироваться по имени пользователя и паролю, получая JWT для последующих обращений к защищённым эндпоинтам API. Мы уже близки к выполнению этого требования: осталось написать код, выдающий JWT.
- Создать маршруты API, защищённые проверкой наличия действительного JWT в виде bearer-токена

Для поддержки JWT понадобится установить ещё несколько пакетов:

```bash
$ npm install --save @nestjs/jwt passport-jwt
$ npm install --save-dev @types/passport-jwt
```

Пакет `@nestjs/jwt` (подробнее [здесь](https://github.com/nestjs/jwt)) — вспомогательный пакет для работы с JWT. Пакет `passport-jwt` — это пакет Passport, реализующий JWT-стратегию, а `@types/passport-jwt` предоставляет определения типов TypeScript.

Рассмотрим подробнее, как обрабатывается запрос `POST /auth/login`. Мы пометили маршрут встроенным `AuthGuard`, предоставленным стратегией passport-local. Это означает, что:

1. Обработчик маршрута **будет вызван только в том случае, если пользователь прошёл проверку**
2. Параметр `req` будет содержать свойство `user` (заполненное Passport в ходе аутентификации passport-local)

С учётом этого мы наконец можем сгенерировать настоящий JWT и вернуть его из этого маршрута. Чтобы сохранить аккуратную модульность сервисов, генерацию JWT вынесем в `authService`. Откройте файл `auth.service.ts` в папке `auth`, добавьте метод `login()` и импортируйте `JwtService`, как показано ниже:

**`auth/auth.service.ts`**

```typescript
import { Injectable } from '@nestjs/common';
import { UsersService } from '../users/users.service';
import { JwtService } from '@nestjs/jwt';

@Injectable()
export class AuthService {
  constructor(
    private usersService: UsersService,
    private jwtService: JwtService
  ) {}

  async validateUser(username: string, pass: string): Promise<any> {
    const user = await this.usersService.findOne(username);
    if (user && user.password === pass) {
      const { password, ...result } = user;
      return result;
    }
    return null;
  }

  async login(user: any) {
    const payload = { username: user.username, sub: user.userId };
    return {
      access_token: this.jwtService.sign(payload),
    };
  }
}
```

Мы используем библиотеку `@nestjs/jwt`, которая предоставляет функцию `sign()` для генерации JWT из подмножества свойств объекта `user`; результат возвращается как простой объект с единственным свойством `access_token`. Обратите внимание: для хранения значения `userId` мы выбрали имя свойства `sub` — в соответствии со стандартами JWT. Не забудьте внедрить провайдер JwtService в `AuthService`.

Теперь нужно обновить `AuthModule`, чтобы импортировать новые зависимости и настроить `JwtModule`.

#### Реализация Passport JWT

Теперь можно перейти к последнему требованию: защите эндпоинтов проверкой наличия действительного JWT в запросе. Passport поможет и здесь. Он предоставляет стратегию [passport-jwt](https://github.com/mikenicholson/passport-jwt) для защиты RESTful-эндпоинтов с помощью JSON Web Token. Начните с создания файла `jwt.strategy.ts` в папке `auth` и добавьте следующий код:

**`auth/jwt.strategy.ts`**

```typescript
import { ExtractJwt, Strategy } from 'passport-jwt';
import { PassportStrategy } from '@nestjs/passport';
import { Injectable } from '@nestjs/common';
import { jwtConstants } from './constants';

@Injectable()
export class JwtStrategy extends PassportStrategy(Strategy) {
  constructor() {
    super({
      jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
      ignoreExpiration: false,
      secretOrKey: jwtConstants.secret,
    });
  }

  async validate(payload: any) {
    return { userId: payload.sub, username: payload.username };
  }
}
```

В `JwtStrategy` мы следовали тому же рецепту, что и для всех стратегий Passport. Эта стратегия требует некоторой инициализации, поэтому мы передаём объект опций в вызов `super()`. Подробнее о доступных опциях — [здесь](https://github.com/mikenicholson/passport-jwt#configure-strategy). В нашем случае это:

- `jwtFromRequest`: задаёт способ извлечения JWT из `Request`. Мы используем стандартный подход — bearer-токен в заголовке Authorization наших API-запросов. Другие варианты описаны [здесь](https://github.com/mikenicholson/passport-jwt#extracting-the-jwt-from-the-request).
- `ignoreExpiration`: явности ради мы выбираем значение по умолчанию `false`, которое возлагает проверку срока действия JWT на модуль Passport. Это означает, что если в маршрут придёт просроченный JWT, запрос будет отклонён с ответом `401 Unauthorized`. Passport удобно делает это автоматически.
- `secretOrKey`: мы пользуемся простым вариантом — симметричным секретом для подписи токена. Другие варианты, например открытый ключ в формате PEM, могут быть уместнее в промышленных приложениях (подробнее [здесь](https://github.com/mikenicholson/passport-jwt#configure-strategy)). В любом случае, как уже предупреждалось, **не публикуйте этот секрет**.

Метод `validate()` заслуживает отдельного разговора. В jwt-стратегии Passport сначала проверяет подпись JWT и декодирует JSON. Затем он вызывает наш метод `validate()`, передавая декодированный JSON единственным параметром. Благодаря принципу работы подписи JWT **мы гарантированно получаем действительный токен**, который ранее подписали и выдали действительному пользователю.

В итоге наш обратный вызов `validate()` тривиален: мы просто возвращаем объект со свойствами `userId` и `username`. Напомним ещё раз, что Passport построит объект `user` на основе возвращённого значения метода `validate()` и прикрепит его как свойство объекта `Request`.

Кроме того, вы можете вернуть массив: первое значение будет использовано для создания объекта `user`, а второе — для создания объекта `authInfo`.

Стоит также отметить, что такой подход оставляет нам место (своего рода «хуки») для встраивания в процесс другой бизнес-логики. Например, в методе `validate()` можно обратиться к базе данных, чтобы получить больше сведений о пользователе, и получить в `Request` более насыщенный объект `user`. Здесь же мы можем выполнить дополнительную проверку токена — например, поискать `userId` в списке отозванных токенов, что позволит реализовать отзыв токенов. Модель, реализованная в нашем примере, — это быстрая модель «stateless JWT», в которой каждый вызов API немедленно авторизуется по наличию действительного JWT, а небольшой объём информации о вызывающем (его `userId` и `username`) доступен в конвейере обработки запроса.

Добавьте новую `JwtStrategy` в провайдеры `AuthModule`:

**`auth/auth.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { AuthService } from './auth.service';
import { LocalStrategy } from './local.strategy';
import { JwtStrategy } from './jwt.strategy';
import { UsersModule } from '../users/users.module';
import { PassportModule } from '@nestjs/passport';
import { JwtModule } from '@nestjs/jwt';
import { jwtConstants } from './constants';

@Module({
  imports: [
    UsersModule,
    PassportModule,
    JwtModule.register({
      secret: jwtConstants.secret,
      signOptions: { expiresIn: '60s' },
    }),
  ],
  providers: [AuthService, LocalStrategy, JwtStrategy],
  exports: [AuthService],
})
export class AuthModule {}
```

Импортируя тот же секрет, которым мы подписывали JWT, мы гарантируем, что этап **проверки** в Passport и этап **подписи** в AuthService используют общий секрет.

Наконец, определим класс `JwtAuthGuard`, наследующий встроенный `AuthGuard`:

**`auth/jwt-auth.guard.ts`**

```typescript
import { Injectable } from '@nestjs/common';
import { AuthGuard } from '@nestjs/passport';

@Injectable()
export class JwtAuthGuard extends AuthGuard('jwt') {}
```

#### Реализация защищённого маршрута и guard'ов JWT-стратегии

Теперь можно реализовать защищённый маршрут и связанный с ним guard.

Откройте файл `app.controller.ts` и обновите его, как показано ниже:

**`app.controller.ts`**

```typescript
import { Controller, Get, Request, Post, UseGuards } from '@nestjs/common';
import { JwtAuthGuard } from './auth/jwt-auth.guard';
import { LocalAuthGuard } from './auth/local-auth.guard';
import { AuthService } from './auth/auth.service';

@Controller()
export class AppController {
  constructor(private authService: AuthService) {}

  @UseGuards(LocalAuthGuard)
  @Post('auth/login')
  async login(@Request() req) {
    return this.authService.login(req.user);
  }

  @UseGuards(JwtAuthGuard)
  @Get('profile')
  getProfile(@Request() req) {
    return req.user;
  }
}
```

И вновь мы применяем `AuthGuard`, который модуль `@nestjs/passport` автоматически подготовил для нас при настройке модуля passport-jwt. На этот guard мы ссылаемся по имени по умолчанию — `jwt`. При обращении к маршруту `GET /profile` guard автоматически вызовет нашу настроенную стратегию passport-jwt, проверит JWT и присвоит свойство `user` объекту `Request`.

Убедитесь, что приложение запущено, и протестируйте маршруты с помощью `cURL`.

```bash
$ # GET /profile
$ curl http://localhost:3000/profile
$ # result -> {"statusCode":401,"message":"Unauthorized"}

$ # POST /auth/login
$ curl -X POST http://localhost:3000/auth/login -d '{"username": "john", "password": "changeme"}' -H "Content-Type: application/json"
$ # result -> {"access_token":"eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2Vybm... }

$ # GET /profile using access_token returned from previous step as bearer code
$ curl http://localhost:3000/profile -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2Vybm..."
$ # result -> {"userId":1,"username":"john"}
```

Обратите внимание, что в `AuthModule` мы задали срок действия JWT `60 seconds`. Это, вероятно, слишком короткий срок, а подробности истечения и обновления токенов выходят за рамки этой статьи. Тем не менее мы выбрали такое значение, чтобы продемонстрировать важное свойство JWT и стратегии passport-jwt. Если после аутентификации подождать 60 секунд и лишь затем выполнить запрос `GET /profile`, вы получите ответ `401 Unauthorized`. Причина в том, что Passport автоматически проверяет срок действия JWT, избавляя вас от необходимости делать это в приложении.

На этом реализация аутентификации через JWT завершена. Клиенты на JavaScript (например, Angular/React/Vue) и другие JavaScript-приложения теперь могут аутентифицироваться и безопасно взаимодействовать с нашим API-сервером.

#### Расширение guard'ов

В большинстве случаев штатного класса `AuthGuard` достаточно. Однако бывают случаи, когда хочется просто расширить стандартную обработку ошибок или логику аутентификации. Для этого можно унаследовать встроенный класс и переопределить его методы в подклассе.

```typescript
import {
  ExecutionContext,
  Injectable,
  UnauthorizedException,
} from '@nestjs/common';
import { AuthGuard } from '@nestjs/passport';

@Injectable()
export class JwtAuthGuard extends AuthGuard('jwt') {
  canActivate(context: ExecutionContext) {
    // Добавьте здесь свою логику аутентификации,
    // например, вызовите super.logIn(request), чтобы создать сессию.
    return super.canActivate(context);
  }

  handleRequest(err, user, info) {
    // Вы можете выбросить исключение на основе аргументов "info" или "err"
    if (err || !user) {
      throw err || new UnauthorizedException();
    }
    return user;
  }
}
```

Помимо расширения стандартной обработки ошибок и логики аутентификации, мы можем пропустить аутентификацию через цепочку стратегий. Первая же стратегия, завершившаяся успехом, перенаправлением или ошибкой, остановит цепочку. При неудачах аутентификация последовательно проходит по всем стратегиям и в итоге завершается неудачей, если не сработала ни одна из них.

```typescript
export class JwtAuthGuard extends AuthGuard(['strategy_jwt_1', 'strategy_jwt_2', '...']) { ... }
```

#### Глобальное включение аутентификации

Если подавляющее большинство ваших эндпоинтов должно быть защищено по умолчанию, вы можете зарегистрировать guard аутентификации как [глобальный guard](<../02. Overview/08. Guards.md#привязка-guard-ов>) и вместо использования декоратора `@UseGuards()` над каждым контроллером просто помечать, какие маршруты являются публичными.

Сначала зарегистрируйте `JwtAuthGuard` как глобальный guard с помощью следующей конструкции (в любом модуле):

```typescript
providers: [
  {
    provide: APP_GUARD,
    useClass: JwtAuthGuard,
  },
],
```

После этого Nest автоматически привяжет `JwtAuthGuard` ко всем эндпоинтам.

Теперь нужно предусмотреть механизм объявления маршрутов публичными. Для этого создадим пользовательский декоратор с помощью фабричной функции `SetMetadata`.

```typescript
import { SetMetadata } from '@nestjs/common';

export const IS_PUBLIC_KEY = 'isPublic';
export const Public = () => SetMetadata(IS_PUBLIC_KEY, true);
```

В файле выше мы экспортировали две константы. Одна — ключ метаданных с именем `IS_PUBLIC_KEY`, вторая — сам новый декоратор, который мы назовём `Public` (вы можете назвать его `SkipAuth` или `AllowAnon` — как удобнее в вашем проекте).

Теперь, когда у нас есть собственный декоратор `@Public()`, мы можем применять его к любому методу:

```typescript
@Public()
@Get()
findAll() {
  return [];
}
```

Наконец, нужно, чтобы `JwtAuthGuard` возвращал `true`, когда найдены метаданные `"isPublic"`. Для этого воспользуемся классом `Reflector` (подробнее [здесь](<../02. Overview/08. Guards.md#собираем-всё-вместе>)).

```typescript
@Injectable()
export class JwtAuthGuard extends AuthGuard('jwt') {
  constructor(private reflector: Reflector) {
    super();
  }

  canActivate(context: ExecutionContext) {
    const isPublic = this.reflector.getAllAndOverride<boolean>(IS_PUBLIC_KEY, [
      context.getHandler(),
      context.getClass(),
    ]);
    if (isPublic) {
      return true;
    }
    return super.canActivate(context);
  }
}
```

#### Стратегии с областью запроса

API Passport основан на регистрации стратегий в глобальном экземпляре библиотеки. Поэтому стратегии не рассчитаны на опции, зависящие от запроса, и на динамическое создание экземпляра под каждый запрос (подробнее о провайдерах с [областью запроса](<../03. Fundamentals/04. Injection scopes.md>)). Если вы объявите стратегию с областью запроса, Nest никогда не создаст её экземпляр, поскольку она не привязана к конкретному маршруту. Физически невозможно определить, какие «request-scoped» стратегии нужно выполнять для каждого запроса.

Тем не менее есть способы динамически разрешать провайдеры с областью запроса внутри стратегии. Для этого мы воспользуемся возможностью [ссылки на модуль](<../03. Fundamentals/06. Module reference.md>).

Сначала откройте файл `local.strategy.ts` и внедрите `ModuleRef` обычным способом:

```typescript
constructor(private moduleRef: ModuleRef) {
  super({
    passReqToCallback: true,
  });
}
```

> **Подсказка** Класс `ModuleRef` импортируется из пакета `@nestjs/core`.

Обязательно установите свойство конфигурации `passReqToCallback` в `true`, как показано выше.

На следующем шаге экземпляр запроса будет использован для получения текущего идентификатора контекста вместо создания нового (подробнее о контексте запроса — [здесь](<../03. Fundamentals/06. Module reference.md#получение-текущего-поддерева>)).

Теперь внутри метода `validate()` класса `LocalStrategy` используйте метод `getByRequest()` класса `ContextIdFactory`, чтобы создать идентификатор контекста на основе объекта запроса, и передайте его в вызов `resolve()`:

```typescript
async validate(
  request: Request,
  username: string,
  password: string,
) {
  const contextId = ContextIdFactory.getByRequest(request);
  // "AuthService" — провайдер с областью запроса
  const authService = await this.moduleRef.resolve(AuthService, contextId);
  ...
}
```

В примере выше метод `resolve()` асинхронно вернёт экземпляр провайдера `AuthService` с областью запроса (мы предполагаем, что `AuthService` помечен как провайдер с областью запроса).

#### Настройка Passport

Любые стандартные параметры настройки Passport передаются тем же способом — через метод `register()`. Доступные параметры зависят от реализуемой стратегии. Например:

```typescript
PassportModule.register({ session: true });
```

Кроме того, стратегиям можно передавать объект опций в их конструкторах. Для локальной стратегии можно передать, например:

```typescript
constructor(private authService: AuthService) {
  super({
    usernameField: 'email',
    passwordField: 'password',
  });
}
```

Имена свойств смотрите на официальном [сайте Passport](http://www.passportjs.org/docs/oauth/).

#### Именованные стратегии

Реализуя стратегию, вы можете задать ей имя, передав второй аргумент в функцию `PassportStrategy`. Если этого не сделать, у каждой стратегии будет имя по умолчанию (например, 'jwt' для jwt-стратегии):

```typescript
export class JwtStrategy extends PassportStrategy(Strategy, 'myjwt')
```

После этого вы ссылаетесь на неё в декораторе, например `@UseGuards(AuthGuard('myjwt'))`.

#### GraphQL

Чтобы использовать AuthGuard с [GraphQL](<../06. GraphQL/01. Quick start.md>), унаследуйте встроенный класс `AuthGuard` и переопределите метод `getRequest()`.

```typescript
@Injectable()
export class GqlAuthGuard extends AuthGuard('jwt') {
  getRequest(context: ExecutionContext) {
    const ctx = GqlExecutionContext.create(context);
    return ctx.getContext().req;
  }
}
```

Чтобы получить текущего аутентифицированного пользователя в резолвере graphql, можно определить декоратор `@CurrentUser()`:

```typescript
import { createParamDecorator, ExecutionContext } from '@nestjs/common';
import { GqlExecutionContext } from '@nestjs/graphql';

export const CurrentUser = createParamDecorator(
  (data: unknown, context: ExecutionContext) => {
    const ctx = GqlExecutionContext.create(context);
    return ctx.getContext().req.user;
  },
);
```

Чтобы использовать приведённый декоратор в резолвере, обязательно укажите его как параметр запроса или мутации:

```typescript
@Query(() => User)
@UseGuards(GqlAuthGuard)
whoAmI(@CurrentUser() user: User) {
  return this.usersService.findById(user.id);
}
```

Для стратегии passport-local также потребуется добавить аргументы контекста GraphQL в тело запроса, чтобы Passport мог получить к ним доступ при проверке. Иначе вы получите ошибку Unauthorized.

```typescript
@Injectable()
export class GqlLocalAuthGuard extends AuthGuard('local') {
  getRequest(context: ExecutionContext) {
    const gqlExecutionContext = GqlExecutionContext.create(context);
    const gqlContext = gqlExecutionContext.getContext();
    const gqlArgs = gqlExecutionContext.getArgs();

    gqlContext.req.body = { ...gqlContext.req.body, ...gqlArgs };
    return gqlContext.req;
  }
}
```
