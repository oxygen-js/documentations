### Руководство по миграции

Эта статья — подробное руководство по переходу с NestJS версии 10 на версию 11. О новых возможностях v11 читайте в [этой статье](https://trilon.io/blog/announcing-nestjs-11-whats-new). Обновление содержит несколько небольших ломающих изменений, но большинства пользователей они вряд ли коснутся. Полный список изменений доступен [здесь](https://github.com/nestjs/nest/releases/tag/v11.0.0).

#### Обновление пакетов

Пакеты можно обновить и вручную, но для более гладкого процесса мы рекомендуем [npm-check-updates (ncu)](https://npmjs.com/package/npm-check-updates).

#### Express v5

После нескольких лет разработки Express v5 официально вышел в 2024 году и стал стабильной версией в 2025-м. В NestJS 11 Express v5 — версия по умолчанию, интегрированная во фреймворк. Для большинства пользователей обновление пройдёт незаметно, однако важно помнить, что Express v5 приносит ряд ломающих изменений. Подробные указания смотрите в [руководстве по миграции на Express v5](https://expressjs.com/en/guide/migrating-5.html).

Одно из самых заметных изменений в Express v5 — переработанный алгоритм сопоставления маршрутов. В сопоставление строк пути с входящими запросами внесены следующие изменения:

- Символ подстановки `*` теперь должен иметь имя — так же, как параметры: вместо `/*` используйте `/*splat` или `/{*splat}`. `splat` — просто имя параметра-подстановки, никакого особого смысла у него нет. Назвать его можно как угодно, например `*wildcard`
- Символ необязательности `?` больше не поддерживается, вместо него используйте фигурные скобки: `/:file{.:ext}`.
- Символы регулярных выражений не поддерживаются.
- Некоторые символы зарезервированы, чтобы избежать путаницы при обновлении: `(()[]?+!)`; для их экранирования используйте `\`.
- Имена параметров теперь допускают любые корректные идентификаторы JavaScript либо запись в кавычках вида `:"this"`.

Таким образом, маршруты, работавшие в Express v4, могут перестать работать в Express v5. Например:

```typescript
@Get('users/*')
findAll() {
  // В NestJS 11 это автоматически преобразуется в корректный маршрут Express v5.
  // Хотя такой вариант ещё может работать, использовать этот синтаксис подстановки
  // в Express v5 больше не рекомендуется.
  return 'This route should not work in Express v5';
}
```

Чтобы исправить это, обновите маршрут, применив именованную подстановку:

```typescript
@Get('users/*splat')
findAll() {
  return 'This route will work in Express v5';
}
```

> **Предупреждение** Обратите внимание: `*splat` — именованная подстановка, которая сопоставляется с любым путём, кроме корневого. Если нужно сопоставлять и корневой путь (`/users`), используйте `/users/{*splat}`, заключив подстановку в фигурные скобки (необязательная группа). Учтите, что `splat` — просто имя параметра-подстановки без особого смысла. Назвать его можно как угодно, например `*wildcard`.

Аналогично, если у вас есть middleware, работающий на всех маршрутах, путь тоже может потребоваться переписать с именованной подстановкой:

```typescript
// В NestJS 11 это автоматически преобразуется в корректный маршрут Express v5.
// Хотя такой вариант ещё может работать, использовать этот синтаксис подстановки
// в Express v5 больше не рекомендуется.
forRoutes('*'); // <-- В Express v5 это работать не должно
```

Вместо этого укажите путь с именованной подстановкой:

```typescript
forRoutes('{*splat}'); // <-- Это будет работать в Express v5
```

Заметьте, что `{*splat}` — именованная подстановка, сопоставляющаяся с любым путём, включая корневой. Внешние фигурные скобки делают путь необязательным.

#### Разбор параметров строки запроса

> **Примечание** Это изменение касается только Express v5.

В Express v5 параметры строки запроса больше не разбираются библиотекой `qs` по умолчанию. Вместо неё используется парсер `simple`, который не поддерживает вложенные объекты и массивы.

В результате строки запроса вида:

```plaintext
?filter[where][name]=John&filter[where][age]=30
?item[]=1&item[]=2
```

больше не будут разбираться так, как вы ожидаете. Чтобы вернуть прежнее поведение, настройте Express на парсер `extended` (использовавшийся по умолчанию в Express v4), задав параметру `query parser` значение `extended`:

```typescript
import { NestFactory } from '@nestjs/core';
import { NestExpressApplication } from '@nestjs/platform-express';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create<NestExpressApplication>(AppModule); // <-- Обязательно используйте <NestExpressApplication>
  app.set('query parser', 'extended'); // <-- Добавьте эту строку
  await app.listen(3000);
}
bootstrap();
```

#### Fastify v5

`@nestjs/platform-fastify` v11 наконец поддерживает Fastify v5. Для большинства пользователей обновление пройдёт незаметно, однако Fastify v5 содержит несколько ломающих изменений, которые вряд ли затронут большинство пользователей NestJS. Подробности смотрите в [руководстве по миграции на Fastify v5](https://fastify.dev/docs/v5.1.x/Guides/Migration-Guide-V5/).

> **Подсказка** Сопоставление путей в Fastify v5 не изменилось (за исключением middleware, см. раздел ниже), поэтому синтаксис подстановки можно использовать как и раньше. Поведение осталось прежним, и маршруты с подстановками (например, `*`) продолжат работать как ожидается.

#### CORS в Fastify

По умолчанию разрешены только [методы из белого списка CORS](https://fetch.spec.whatwg.org/#methods). Если нужно разрешить дополнительные методы (такие как `PUT`, `PATCH` или `DELETE`), их следует явно указать в параметре `methods`.

```typescript
const methods = ['GET', 'POST', 'PUT', 'PATCH', 'DELETE']; // ИЛИ строка через запятую: 'GET,POST,PUT,PATCH,DELETE'

const app = await NestFactory.create<NestFastifyApplication>(
  AppModule,
  new FastifyAdapter(),
  { cors: { methods } },
);

// ИЛИ, как вариант, можно использовать метод `enableCors`
const app = await NestFactory.create<NestFastifyApplication>(
  AppModule,
  new FastifyAdapter(),
);
app.enableCors({ methods });
```

#### Регистрация middleware в Fastify

Для сопоставления **путей middleware** в `@nestjs/platform-fastify` NestJS 11 теперь использует последнюю версию пакета [path-to-regexp](https://www.npmjs.com/package/path-to-regexp). Поэтому синтаксис `(.*)` для сопоставления со всеми путями больше не поддерживается — вместо него следует использовать именованные подстановки.

Например, если у вас есть middleware, применяемый ко всем маршрутам:

```typescript
// В NestJS 11 это автоматически преобразуется в корректный маршрут, даже если вы ничего не поменяете.
.forRoutes('(.*)');
```

Его нужно переписать с именованной подстановкой:

```typescript
.forRoutes('*splat');
```

Здесь `splat` — произвольное имя параметра-подстановки. Назвать его можно как угодно.

#### Алгоритм разрешения модулей

Начиная с NestJS 11 алгоритм разрешения модулей улучшен: для большинства приложений он повышает производительность и снижает потребление памяти. Это изменение не требует ручного вмешательства, но в некоторых пограничных случаях поведение может отличаться от предыдущих версий.

В NestJS v10 и более ранних версиях динамическим модулям присваивался уникальный непрозрачный ключ, сгенерированный из динамических метаданных модуля. По этому ключу модуль опознавался в реестре модулей. Например, если вы подключали `TypeOrmModule.forFeature([User])` в нескольких модулях, NestJS убирал дубликаты и рассматривал их как единый узел модуля в реестре. Этот процесс называется дедупликацией узлов.

С выходом NestJS v11 мы больше не генерируем предсказуемые хэши для динамических модулей. Вместо этого эквивалентность модулей определяется по ссылкам на объекты. Чтобы использовать один и тот же динамический модуль в нескольких модулях, просто присвойте его переменной и импортируйте её везде, где нужно. Новый подход даёт больше гибкости и обеспечивает более эффективную работу с динамическими модулями.

Новый алгоритм может повлиять на ваши интеграционные тесты, если вы активно используете динамические модули: без описанной выше ручной дедупликации в `TestingModule` может оказаться несколько экземпляров одной зависимости. Из-за этого подменять метод становится сложнее — нужно попасть в нужный экземпляр. Возможные варианты:

- Выполнить дедупликацию динамического модуля, который вы хотите подменить
- Использовать `module.select(ParentModule).get(Target)`, чтобы найти нужный экземпляр
- Подменить все экземпляры с помощью `module.get(Target, { each: true })`
- Либо вернуть тест на старый алгоритм: `Test.createTestingModule({}, { moduleIdGeneratorAlgorithm: 'deep-hash' })`

#### Вывод типов в Reflector

NestJS 11 приносит несколько улучшений в класс `Reflector`, расширяя его функциональность и вывод типов для значений метаданных. Эти изменения делают работу с метаданными более понятной и надёжной.

1. `getAllAndMerge` теперь возвращает объект, а не массив с единственным элементом, когда запись метаданных всего одна, а `value` имеет тип `object`. Это повышает единообразие при работе с метаданными на основе объектов.
2. Возвращаемый тип `getAllAndOverride` изменён с `T` на `T | undefined`. Так лучше отражается возможность того, что метаданные не найдены, и обеспечивается корректная обработка случаев с undefined.
3. Преобразованный аргумент типа у `ReflectableDecorator` теперь корректно выводится во всех методах.

Эти улучшения повышают удобство разработки, обеспечивая лучшую типобезопасность и обработку метаданных в NestJS 11.

#### Порядок выполнения хуков жизненного цикла

Хуки завершения теперь выполняются в порядке, обратном их аналогам при инициализации. То есть хуки `OnModuleDestroy`, `BeforeApplicationShutdown` и `OnApplicationShutdown` выполняются в обратном порядке.

Представьте следующую ситуацию:

```plaintext
// Where A, B, and C are modules and "->" represents the module dependency.
A -> B -> C
```

В этом случае хуки `OnModuleInit` выполняются в таком порядке:

```plaintext
C -> B -> A
```

А хуки `OnModuleDestroy` — в обратном:

```plaintext
A -> B -> C
```

> **Подсказка** Глобальные модули считаются зависимостью всех остальных модулей. Это означает, что глобальные модули инициализируются первыми, а уничтожаются последними.

#### Порядок регистрации middleware

В NestJS v11 поведение при регистрации middleware изменилось. Раньше порядок регистрации определялся топологической сортировкой графа зависимостей модулей: очередь задавалась расстоянием от корневого модуля независимо от того, зарегистрирован ли middleware в глобальном модуле или в обычном. Глобальные модули в этом отношении рассматривались как обычные, что приводило к непоследовательному поведению, особенно на фоне других возможностей фреймворка.

Начиная с v11 middleware, зарегистрированный в глобальных модулях, **выполняется первым**, независимо от его положения в графе зависимостей модулей. Это гарантирует, что глобальный middleware всегда отрабатывает раньше любого middleware из импортированных модулей, обеспечивая единообразный и предсказуемый порядок.

#### Модуль кэширования

`CacheModule` (из пакета `@nestjs/cache-manager`) обновлён под последнюю версию пакета `cache-manager`. Это обновление приносит несколько ломающих изменений, в том числе переход на [Keyv](https://keyv.org/) — единый интерфейс к хранилищам «ключ-значение» поверх разных бэкендов через адаптеры хранилищ.

Ключевое отличие новой версии от предыдущей — настройка внешних хранилищ. Раньше, чтобы зарегистрировать хранилище Redis, вы, скорее всего, писали примерно так:

```ts
// Старая версия — больше не поддерживается
CacheModule.registerAsync({
  useFactory: async () => {
    const store = await redisStore({
      socket: {
        host: 'localhost',
        port: 6379,
      },
    });

    return {
      store,
    };
  },
}),
```

В новой версии для настройки хранилища следует использовать адаптер `Keyv`:

```ts
// Новая версия — поддерживается
CacheModule.registerAsync({
  useFactory: async () => {
    return {
      stores: [
        new KeyvRedis('redis://localhost:6379'),
      ],
    };
  },
}),
```

Здесь `KeyvRedis` импортируется из пакета `@keyv/redis`. Подробнее — в [документации по кэшированию](<04. Techniques/05. Caching.md>).

> **Предупреждение** В этом обновлении данные, которыми управляет библиотека Keyv, хранятся в кэше как объект с полями `value` и `expires`, например: `{"value": "yourData", "expires": 1678901234567}`. Keyv автоматически извлекает поле `value` при обращении через своё API, но это изменение важно учитывать, если вы работаете с данными кэша напрямую (например, в обход API cache-manager) или вам нужна совместимость с данными, записанными предыдущей версией `@nestjs/cache-manager`.

#### Модуль конфигурации

Если вы используете `ConfigModule` из пакета `@nestjs/config`, обратите внимание на несколько ломающих изменений в `@nestjs/config@4.0.0`. Главное из них — изменившийся порядок чтения конфигурационных переменных методом `ConfigService#get`. Новый порядок таков:

- Внутренняя конфигурация (пространства имён конфигурации и пользовательские файлы конфигурации)
- Проверенные переменные окружения (если валидация включена и задана схема)
- Объект `process.env`

Раньше сначала читались проверенные переменные окружения и объект `process.env`, из-за чего внутренняя конфигурация не могла их переопределить. После обновления внутренняя конфигурация всегда имеет приоритет над переменными окружения.

Кроме того, параметр `ignoreEnvVars`, ранее позволявший отключить валидацию объекта `process.env`, объявлен устаревшим. Вместо него используйте параметр `validatePredefined` (значение `false` отключает валидацию предопределённых переменных окружения). Предопределённые переменные окружения — это переменные `process.env`, установленные до импорта модуля. Например, если вы запускаете приложение командой `PORT=3000 node main.js`, переменная `PORT` считается предопределённой. А вот переменные, загруженные `ConfigModule` из файла `.env`, к предопределённым не относятся.

Также появился новый параметр `skipProcessEnv`. Он позволяет полностью запретить методу `ConfigService#get` обращаться к объекту `process.env` — это удобно, когда нужно ограничить прямое чтение переменных окружения сервисом.

#### Модуль Terminus

Если вы используете `TerminusModule` и написали собственный индикатор работоспособности, обратите внимание: в версии 11 появился новый API. Новый `HealthIndicatorService` призван сделать пользовательские индикаторы работоспособности читаемее и удобнее для тестирования.

До версии 11 индикатор работоспособности мог выглядеть так:

```typescript
@Injectable()
export class DogHealthIndicator extends HealthIndicator {
  constructor(private readonly httpService: HttpService) {
    super();
  }

  async isHealthy(key: string) {
    try {
      const badboys = await this.getBadboys();
      const isHealthy = badboys.length === 0;

      const result = this.getStatus(key, isHealthy, {
        badboys: badboys.length,
      });

      if (!isHealthy) {
        throw new HealthCheckError('Dog check failed', result);
      }

      return result;
    } catch (error) {
      const result = this.getStatus(key, isHealthy);
      throw new HealthCheckError('Dog check failed', result);
    }
  }

  private getBadboys() {
    return firstValueFrom(
      this.httpService.get<Dog[]>('https://example.com/dog').pipe(
        map((response) => response.data),
        map((dogs) => dogs.filter((dog) => dog.state === DogState.BAD_BOY)),
      ),
    );
  }
}
```

Начиная с версии 11 рекомендуется использовать новый API `HealthIndicatorService`, упрощающий реализацию. Вот как теперь можно написать тот же индикатор работоспособности:

```typescript
@Injectable()
export class DogHealthIndicator {
  constructor(
    private readonly httpService: HttpService,
    //  Внедряем `HealthIndicatorService`, предоставляемый `TerminusModule`
    private readonly healthIndicatorService: HealthIndicatorService,
  ) {}

  async isHealthy(key: string) {
    // Запускаем проверку индикатора работоспособности для заданного ключа
    const indicator = this.healthIndicatorService.check(key);

    try {
      const badboys = await this.getBadboys();
      const isHealthy = badboys.length === 0;

      if (!isHealthy) {
        // Помечаем индикатор как "down" и добавляем в ответ дополнительные сведения
        return indicator.down({ badboys: badboys.length });
      }

      // Помечаем индикатор работоспособности как up
      return indicator.up();
    } catch (error) {
      return indicator.down('Unable to retrieve dogs');
    }
  }

  private getBadboys() {
    // ...
  }
}
```

Ключевые изменения:

- `HealthIndicatorService` заменяет устаревшие классы `HealthIndicator` и `HealthCheckError`, предоставляя более чистый API для проверок работоспособности.
- Метод `check` позволяет удобно отслеживать состояние (`up` или `down`) и добавлять дополнительные метаданные в ответы проверок работоспособности.

> **Информация** Обратите внимание: классы `HealthIndicator` и `HealthCheckError` помечены как устаревшие и будут удалены в следующем мажорном релизе.

#### Node.js v16 и v18 больше не поддерживаются

Начиная с NestJS 11 поддержка Node.js v16 прекращена, поскольку 11 сентября 2023 года эта версия достигла конца жизненного цикла (EOL). Аналогично, поддержка безопасности Node.js v18 заканчивается 30 апреля 2025 года, поэтому мы отказались и от неё.

Теперь NestJS 11 требует **Node.js v20 или выше**.

Для наилучшего результата мы настоятельно рекомендуем использовать последнюю LTS-версию Node.js.

#### Mau — официальная платформа развёртывания

Если вы пропустили анонс: в 2024 году мы запустили нашу официальную платформу развёртывания [Mau](https://www.mau.nestjs.com/).
Mau — полностью управляемая платформа, упрощающая развёртывание приложений NestJS. С Mau вы можете развернуть приложение в облаке (**AWS**, Amazon Web Services) одной командой, управлять переменными окружения и следить за производительностью приложения в реальном времени.

Mau делает подготовку и обслуживание инфраструктуры делом нескольких нажатий. Платформа задумана простой и интуитивной, чтобы вы могли сосредоточиться на создании приложений, а не на инфраструктуре под ними. «Под капотом» мы используем Amazon Web Services, давая вам мощную и надёжную платформу и скрывая всю сложность AWS. Всю тяжёлую работу мы берём на себя, а вы занимаетесь своими приложениями и развитием бизнеса.

```bash
$ npm install -g @nestjs/mau
$ mau deploy
```

Подробнее о Mau читайте [в этой главе](<09. Deployment.md#простое-развёртывание-с-mau>).
