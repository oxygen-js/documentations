### Валидация

Проверять корректность любых данных, поступающих в веб-приложение, — хорошая практика. Для автоматической валидации входящих запросов Nest предлагает несколько pipe, доступных сразу «из коробки»:

- `ValidationPipe`
- `ParseIntPipe`
- `ParseBoolPipe`
- `ParseArrayPipe`
- `ParseUUIDPipe`

`ValidationPipe` использует мощный пакет [class-validator](https://github.com/typestack/class-validator) и его декларативные декораторы валидации. `ValidationPipe` даёт удобный способ применять правила валидации ко всем входящим клиентским данным, при этом сами правила задаются простыми аннотациями в объявлениях локальных классов/DTO внутри каждого модуля.

#### Обзор

В главе [Pipes](<../02. Overview/07. Pipes.md>) мы разобрали, как создавать простые pipe и привязывать их к контроллерам, методам или ко всему приложению, чтобы показать, как всё это работает. Обязательно перечитайте ту главу, чтобы лучше понять материал этой. Здесь мы сосредоточимся на различных **реальных** сценариях использования `ValidationPipe` и покажем некоторые его продвинутые возможности настройки.

#### Использование встроенного ValidationPipe

Для начала установим необходимую зависимость.

```bash
$ npm i --save class-validator class-transformer
```

> **Подсказка** `ValidationPipe` экспортируется из пакета `@nestjs/common`.

Поскольку этот pipe использует библиотеки [`class-validator`](https://github.com/typestack/class-validator) и [`class-transformer`](https://github.com/typestack/class-transformer), доступно множество опций. Настраиваются они через объект конфигурации, передаваемый в pipe. Вот встроенные опции:

```typescript
export interface ValidationPipeOptions extends ValidatorOptions {
  transform?: boolean;
  disableErrorMessages?: boolean;
  exceptionFactory?: (errors: ValidationError[]) => any;
  errorFormat?: 'list' | 'grouped';
}
```

#### Автоматическая валидация

Начнём с привязки `ValidationPipe` на уровне приложения, чтобы все эндпоинты были защищены от получения некорректных данных.

```typescript
async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  app.useGlobalPipes(new ValidationPipe());
  await app.listen(process.env.PORT ?? 3000);
}
bootstrap();
```

Чтобы проверить наш pipe, создадим простой эндпоинт.

```typescript
@Post()
create(@Body() createUserDto: CreateUserDto) {
  return 'This action adds a new user';
}
```

> **Подсказка** Поскольку TypeScript не сохраняет метаданные об **обобщениях и интерфейсах**, при их использовании в DTO `ValidationPipe` может оказаться не в состоянии корректно проверить входящие данные. Поэтому в DTO лучше использовать конкретные классы.

> **Подсказка** При импорте DTO нельзя использовать импорт только типа, так как он будет стёрт во время выполнения, — то есть пишите `import { CreateUserDto }`, а не `import type { CreateUserDto }`.

Теперь добавим несколько правил валидации в `CreateUserDto`. Делается это с помощью декораторов из пакета `class-validator`, подробно описанных [здесь](https://github.com/typestack/class-validator#validation-decorators). Таким образом, любой маршрут, использующий `CreateUserDto`, будет автоматически применять эти правила валидации.

```typescript
import { IsEmail, IsNotEmpty } from 'class-validator';

export class CreateUserDto {
  @IsEmail()
  email: string;

  @IsNotEmpty()
  password: string;
}
```

С такими правилами, если в наш эндпоинт придёт запрос с некорректным свойством `email` в теле, приложение автоматически ответит кодом `400 Bad Request` и следующим телом ответа:

```json
{
  "statusCode": 400,
  "error": "Bad Request",
  "message": ["email must be an email"]
}
```

Помимо тела запроса, `ValidationPipe` можно применять и к другим свойствам объекта запроса. Представим, что мы хотим принимать `:id` в пути эндпоинта. Чтобы для этого параметра принимались только числа, можно использовать такую конструкцию:

```typescript
@Get(':id')
findOne(@Param() params: FindOneParams) {
  return 'This action returns a user';
}
```

`FindOneParams`, как и DTO, — это просто класс, который задаёт правила валидации с помощью `class-validator`. Выглядеть он будет так:

```typescript
import { IsNumberString } from 'class-validator';

export class FindOneParams {
  @IsNumberString()
  id: string;
}
```

#### Отключение подробных ошибок

Сообщения об ошибках помогают понять, что было не так в запросе. Однако в некоторых production-окружениях подробные ошибки предпочитают отключать. Для этого передайте в `ValidationPipe` объект опций:

```typescript
app.useGlobalPipes(
  new ValidationPipe({
    disableErrorMessages: true,
  }),
);
```

В результате подробные сообщения об ошибках не будут выводиться в теле ответа.

#### Удаление лишних свойств

Наш `ValidationPipe` также умеет отфильтровывать свойства, которые не должны попадать в обработчик метода. В этом случае мы задаём **белый список** допустимых свойств, а всё, что в него не входит, автоматически удаляется из итогового объекта. Например, если обработчик ожидает свойства `email` и `password`, а запрос содержит ещё и `age`, это свойство может быть автоматически удалено из итогового DTO. Чтобы включить такое поведение, задайте `whitelist` значение `true`.

```typescript
app.useGlobalPipes(
  new ValidationPipe({
    whitelist: true,
  }),
);
```

Со значением true из объекта автоматически удаляются свойства вне белого списка (то есть те, у которых нет ни одного декоратора в классе валидации).

Как вариант, можно прерывать обработку запроса при наличии свойств вне белого списка и возвращать пользователю ответ с ошибкой. Для этого задайте опции `forbidNonWhitelisted` значение `true` в сочетании с `whitelist`, равным `true`.

#### Преобразование объектов с полезной нагрузкой

Полезная нагрузка, приходящая по сети, — это обычные объекты JavaScript. `ValidationPipe` может автоматически преобразовывать её в объекты, типизированные согласно их классам DTO. Чтобы включить автоматическое преобразование, задайте `transform` значение `true`. Сделать это можно на уровне метода:

**`cats.controller.ts`**

```typescript
@Post()
@UsePipes(new ValidationPipe({ transform: true }))
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
```

Чтобы включить это поведение глобально, задайте опцию для глобального pipe:

```typescript
app.useGlobalPipes(
  new ValidationPipe({
    transform: true,
  }),
);
```

При включённом автоматическом преобразовании `ValidationPipe` будет также приводить примитивные типы. В следующем примере метод `findOne()` принимает один аргумент, представляющий извлечённый параметр пути `id`:

```typescript
@Get(':id')
findOne(@Param('id') id: number) {
  console.log(typeof id === 'number'); // true
  return 'This action returns a user';
}
```

По умолчанию каждый параметр пути и каждый query-параметр приходит по сети как `string`. В примере выше мы указали для `id` тип `number` (в сигнатуре метода). Поэтому `ValidationPipe` попытается автоматически преобразовать строковый идентификатор в число.

#### Явное преобразование

В предыдущем разделе мы показали, как `ValidationPipe` может неявно преобразовывать query-параметры и параметры пути на основе ожидаемого типа. Однако для этого требуется включённое автоматическое преобразование.

Как вариант (при отключённом автоматическом преобразовании), можно явно приводить значения с помощью `ParseIntPipe` или `ParseBoolPipe` (обратите внимание, что `ParseStringPipe` не нужен, поскольку, как упоминалось выше, каждый параметр пути и query-параметр по умолчанию приходит по сети как `string`).

```typescript
@Get(':id')
findOne(
  @Param('id', ParseIntPipe) id: number,
  @Query('sort', ParseBoolPipe) sort: boolean,
) {
  console.log(typeof id === 'number'); // true
  console.log(typeof sort === 'boolean'); // true
  return 'This action returns a user';
}
```

> **Подсказка** `ParseIntPipe` и `ParseBoolPipe` экспортируются из пакета `@nestjs/common`.

#### Mapped types

По мере разработки таких возможностей, как **CRUD** (Create/Read/Update/Delete), часто бывает удобно строить варианты базового типа сущности. Nest предоставляет несколько служебных функций, которые выполняют преобразования типов и упрощают эту задачу.

> **Предупреждение** Если ваше приложение использует пакет `@nestjs/swagger`, обратитесь к [этой главе](<../12. OpenAPI/05. Mapped Types.md>) за подробностями о Mapped Types. Аналогично, если вы используете пакет `@nestjs/graphql`, см. [эту главу](<../06. GraphQL/10. Mapped types.md>). Оба пакета сильно опираются на типы, поэтому требуют импорта из соответствующего места. Если же вы использовали `@nestjs/mapped-types` (вместо подходящего пакета — `@nestjs/swagger` или `@nestjs/graphql` в зависимости от типа приложения), вы можете столкнуться с различными недокументированными побочными эффектами.

При построении типов для валидации входных данных (их же называют DTO) часто удобно иметь варианты **create** и **update** одного и того же типа. Например, вариант **create** может требовать все поля, тогда как в варианте **update** все поля могут быть необязательными.

Чтобы упростить эту задачу и свести к минимуму шаблонный код, Nest предоставляет служебную функцию `PartialType()`.

Функция `PartialType()` возвращает тип (класс), в котором все свойства исходного типа сделаны необязательными. Допустим, у нас есть такой тип **create**:

```typescript
export class CreateCatDto {
  name: string;
  age: number;
  breed: string;
}
```

По умолчанию все эти поля обязательны. Чтобы создать тип с теми же полями, но необязательными, используйте `PartialType()`, передав ссылку на класс (`CreateCatDto`) в качестве аргумента:

```typescript
export class UpdateCatDto extends PartialType(CreateCatDto) {}
```

> **Подсказка** Функция `PartialType()` импортируется из пакета `@nestjs/mapped-types`.

Функция `PickType()` строит новый тип (класс), выбирая набор свойств из исходного типа. Допустим, мы начинаем с такого типа:

```typescript
export class CreateCatDto {
  name: string;
  age: number;
  breed: string;
}
```

Выбрать набор свойств из этого класса можно с помощью служебной функции `PickType()`:

```typescript
export class UpdateCatAgeDto extends PickType(CreateCatDto, ['age'] as const) {}
```

> **Подсказка** Функция `PickType()` импортируется из пакета `@nestjs/mapped-types`.

Функция `OmitType()` строит тип, беря все свойства исходного типа и удаляя определённый набор ключей. Допустим, мы начинаем с такого типа:

```typescript
export class CreateCatDto {
  name: string;
  age: number;
  breed: string;
}
```

Мы можем получить производный тип со всеми свойствами, **кроме** `name`, как показано ниже. В этой конструкции второй аргумент `OmitType` — массив имён свойств.

```typescript
export class UpdateCatDto extends OmitType(CreateCatDto, ['name'] as const) {}
```

> **Подсказка** Функция `OmitType()` импортируется из пакета `@nestjs/mapped-types`.

Функция `IntersectionType()` объединяет два типа в один новый тип (класс). Допустим, мы начинаем с двух типов:

```typescript
export class CreateCatDto {
  name: string;
  breed: string;
}

export class AdditionalCatInfo {
  color: string;
}
```

Мы можем получить новый тип, объединяющий все свойства обоих типов.

```typescript
export class UpdateCatDto extends IntersectionType(
  CreateCatDto,
  AdditionalCatInfo,
) {}
```

> **Подсказка** Функция `IntersectionType()` импортируется из пакета `@nestjs/mapped-types`.

Служебные функции преобразования типов можно комбинировать. Например, следующий код даст тип (класс) со всеми свойствами типа `CreateCatDto`, кроме `name`, причём все эти свойства станут необязательными:

```typescript
export class UpdateCatDto extends PartialType(
  OmitType(CreateCatDto, ['name'] as const),
) {}
```

#### Разбор и валидация массивов

TypeScript не сохраняет метаданные об обобщениях и интерфейсах, поэтому при их использовании в DTO `ValidationPipe` может оказаться не в состоянии корректно проверить входящие данные. Например, в следующем коде `createUserDtos` не будет проверен как надо:

```typescript
@Post()
createBulk(@Body() createUserDtos: CreateUserDto[]) {
  return 'This action adds new users';
}
```

Чтобы проверить массив, создайте отдельный класс со свойством-обёрткой над массивом или используйте `ParseArrayPipe`.

```typescript
@Post()
createBulk(
  @Body(new ParseArrayPipe({ items: CreateUserDto }))
  createUserDtos: CreateUserDto[],
) {
  return 'This action adds new users';
}
```

Кроме того, `ParseArrayPipe` может пригодиться при разборе query-параметров. Рассмотрим метод `findByIds()`, который возвращает пользователей по идентификаторам, переданным в query-параметрах.

```typescript
@Get()
findByIds(
  @Query('ids', new ParseArrayPipe({ items: Number, separator: ',' }))
  ids: number[],
) {
  return 'This action returns users by ids';
}
```

Эта конструкция проверяет входящие query-параметры HTTP-запроса `GET` вида:

```bash
GET /?ids=1,2,3
```

#### WebSocket и микросервисы

Хотя в этой главе примеры приведены для приложений в стиле HTTP (например, Express или Fastify), `ValidationPipe` точно так же работает для WebSocket и микросервисов, независимо от используемого транспорта.
