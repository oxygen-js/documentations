### Сериализация

Сериализация — это процесс, который выполняется перед тем, как объекты возвращаются в сетевом ответе. Это подходящее место для того, чтобы задать правила преобразования и очистки данных, отправляемых клиенту. Например, конфиденциальные данные вроде паролей всегда должны исключаться из ответа. Либо некоторые свойства могут требовать дополнительного преобразования — скажем, отправки лишь части свойств сущности. Выполнять такие преобразования вручную утомительно, легко допустить ошибку и остаётся неуверенность в том, что учтены все случаи.

#### Обзор

Nest предоставляет встроенную возможность, которая помогает выполнять эти операции просто и понятно. Interceptor `ClassSerializerInterceptor` использует мощный пакет [class-transformer](https://github.com/typestack/class-transformer), давая декларативный и расширяемый способ преобразования объектов. Основное, что он делает, — берёт значение, возвращённое обработчиком метода, и применяет к нему функцию `instanceToPlain()` из [class-transformer](https://github.com/typestack/class-transformer). При этом применяются правила, заданные декораторами `class-transformer` на классе сущности или DTO, как описано ниже.

> **Подсказка** Сериализация не применяется к ответам типа [StreamableFile](<15. Streaming files.md#класс-streamablefile>).

#### Исключение свойств

Предположим, мы хотим автоматически исключать свойство `password` из сущности пользователя. Аннотируем сущность следующим образом:

```typescript
import { Exclude } from 'class-transformer';

export class UserEntity {
  id: number;
  firstName: string;
  lastName: string;

  @Exclude()
  password: string;

  constructor(partial: Partial<UserEntity>) {
    Object.assign(this, partial);
  }
}
```

Теперь рассмотрим контроллер с обработчиком метода, который возвращает экземпляр этого класса.

```typescript
@UseInterceptors(ClassSerializerInterceptor)
@Get()
findOne(): UserEntity {
  return new UserEntity({
    id: 1,
    firstName: 'John',
    lastName: 'Doe',
    password: 'password',
  });
}
```

> **Предупреждение** Обратите внимание: необходимо вернуть именно экземпляр класса. Если вернуть обычный объект JavaScript, например `{ user: new UserEntity() }`, объект не будет сериализован должным образом.

> **Подсказка** `ClassSerializerInterceptor` импортируется из `@nestjs/common`.

При запросе к этому эндпоинту клиент получит следующий ответ:

```json
{
  "id": 1,
  "firstName": "John",
  "lastName": "Doe"
}
```

Обратите внимание, что interceptor можно применить на уровне всего приложения (как описано [здесь](<../02. Overview/09. Interceptors.md#привязка-interceptors>)). Сочетание interceptor'а и объявления класса сущности гарантирует, что **любой** метод, возвращающий `UserEntity`, обязательно удалит свойство `password`. Это даёт вам централизованное соблюдение данного бизнес-правила.

#### Раскрытие свойств

Вы можете использовать декоратор `@Expose()`, чтобы задать псевдонимы для свойств или выполнить функцию для вычисления значения свойства (по аналогии с функциями-**геттерами**), как показано ниже.

```typescript
@Expose()
get fullName(): string {
  return `${this.firstName} ${this.lastName}`;
}
```

#### Преобразование

Дополнительное преобразование данных выполняется с помощью декоратора `@Transform()`. Например, следующая конструкция возвращает свойство name объекта `RoleEntity` вместо всего объекта целиком.

```typescript
@Transform(({ value }) => value.name)
role: RoleEntity;
```

#### Передача параметров

Вам может понадобиться изменить поведение функций преобразования, принятое по умолчанию. Чтобы переопределить настройки по умолчанию, передайте их в объекте `options` через декоратор `@SerializeOptions()`.

```typescript
@SerializeOptions({
  excludePrefixes: ['_'],
})
@Get()
findOne(): UserEntity {
  return new UserEntity();
}
```

> **Подсказка** Декоратор `@SerializeOptions()` импортируется из `@nestjs/common`.

Параметры, переданные через `@SerializeOptions()`, передаются вторым аргументом в лежащую в основе функцию `instanceToPlain()`. В этом примере мы автоматически исключаем все свойства, начинающиеся с префикса `_`.

#### Преобразование обычных объектов

Вы можете принудительно включить преобразования на уровне контроллера с помощью декоратора `@SerializeOptions`. Это гарантирует, что все ответы будут преобразованы в экземпляры указанного класса с применением всех декораторов class-validator или class-transformer, даже если возвращаются обычные объекты. Такой подход делает код чище: не нужно каждый раз создавать экземпляр класса или вызывать `plainToInstance`.

В примере ниже, несмотря на то что в обеих ветках условия возвращаются обычные объекты JavaScript, они будут автоматически преобразованы в экземпляры `UserEntity` с применением соответствующих декораторов:

```typescript
@UseInterceptors(ClassSerializerInterceptor)
@SerializeOptions({ type: UserEntity })
@Get()
findOne(@Query() { id }: { id: number }): UserEntity {
  if (id === 1) {
    return {
      id: 1,
      firstName: 'John',
      lastName: 'Doe',
      password: 'password',
    };
  }

  return {
    id: 2,
    firstName: 'Kamil',
    lastName: 'Mysliwiec',
    password: 'password2',
  };
}
```

> **Подсказка** Указав ожидаемый тип возвращаемого значения контроллера, вы задействуете проверку типов TypeScript и убедитесь, что возвращаемый обычный объект соответствует форме DTO или сущности. Функция `plainToInstance` такой подсказки типов не даёт, что может привести к ошибкам, если обычный объект не совпадает с ожидаемой структурой DTO или сущности.

#### WebSockets и микросервисы

Хотя в этой главе примеры приводятся для приложений в стиле HTTP (например, Express или Fastify), `ClassSerializerInterceptor` работает точно так же для WebSockets и микросервисов, независимо от используемого способа транспорта.
