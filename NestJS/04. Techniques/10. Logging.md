### Logger

В Nest есть встроенный текстовый логгер, который используется при начальной загрузке приложения и в ряде других случаев — например, для вывода перехваченных исключений (то есть системное логирование). Эта функциональность предоставляется классом `Logger` из пакета `@nestjs/common`. Вы можете полностью управлять поведением системы логирования, в том числе:

- полностью отключить логирование
- задать уровень детализации логов (например, показывать ошибки, предупреждения, отладочную информацию и т. д.)
- настроить форматирование сообщений (raw, json, с цветом и т. д.)
- переопределить отметку времени в логгере по умолчанию (например, использовать формат даты по стандарту ISO8601)
- полностью заменить логгер по умолчанию
- настроить логгер по умолчанию, расширив его
- задействовать внедрение зависимостей, чтобы упростить сборку и тестирование приложения

Встроенный логгер (или собственную реализацию) можно использовать и для логирования событий и сообщений уровня самого приложения.

Если вашему приложению нужна интеграция с внешними системами логирования, автоматическая запись логов в файлы или пересылка логов в централизованный сервис, вы можете реализовать полностью собственное решение на базе какой-либо библиотеки логирования для Node.js. Один из популярных вариантов — [Pino](https://github.com/pinojs/pino), известный высокой производительностью и гибкостью.

#### Базовая настройка

Чтобы отключить логирование, задайте свойству `logger` значение `false` в (необязательном) объекте параметров приложения Nest, который передаётся вторым аргументом в метод `NestFactory.create()`.

```typescript
const app = await NestFactory.create(AppModule, {
  logger: false,
});
await app.listen(process.env.PORT ?? 3000);
```

Чтобы включить конкретные уровни логирования, задайте свойству `logger` массив строк с нужными уровнями:

```typescript
const app = await NestFactory.create(AppModule, {
  logger: ['error', 'warn'],
});
await app.listen(process.env.PORT ?? 3000);
```

Значениями в массиве могут быть любые сочетания `'log'`, `'fatal'`, `'error'`, `'warn'`, `'debug'` и `'verbose'`.

> **Подсказка** Уровни логирования в Nest каскадные (наследуемые). Это значит, что при указании конкретного уровня (например, `'log'`) автоматически включаются все уровни более высокой важности (например, `'warn'`, `'error'` и `'fatal'`).

Чтобы отключить цветной вывод, передайте в свойство `logger` объект `ConsoleLogger` со свойством `colors`, равным `false`.

```typescript
const app = await NestFactory.create(AppModule, {
  logger: new ConsoleLogger({
    colors: false,
  }),
});
```

Чтобы задать префикс для каждого сообщения лога, передайте объект `ConsoleLogger` с заданным атрибутом `prefix`:

```typescript
const app = await NestFactory.create(AppModule, {
  logger: new ConsoleLogger({
    prefix: 'MyApp', // По умолчанию — "Nest"
  }),
});
```

#### Логирование в JSON

Логирование в JSON необходимо для современной наблюдаемости приложений и интеграции с системами управления логами. Чтобы включить JSON-логирование в приложении NestJS, настройте объект `ConsoleLogger`, задав его свойству `json` значение `true`. Затем передайте эту конфигурацию логгера в свойство `logger` при создании экземпляра приложения.

```typescript
const app = await NestFactory.create(AppModule, {
  logger: new ConsoleLogger({
    json: true,
  }),
});
```

При такой конфигурации логи выводятся в структурированном формате JSON, что упрощает интеграцию с внешними системами — агрегаторами логов и облачными платформами. Например, платформы вроде **AWS ECS** (Elastic Container Service) поддерживают JSON-логи изначально, что открывает продвинутые возможности:

- **Фильтрация логов**: легко сузить выборку логов по полям вроде уровня, отметки времени или пользовательских метаданных.
- **Поиск и анализ**: использовать инструменты запросов для анализа и отслеживания трендов в поведении приложения.

Кроме того, если вы используете [NestJS Mau](https://mau.nestjs.com), JSON-логирование упрощает просмотр логов в хорошо организованном структурированном виде, что особенно полезно при отладке и мониторинге производительности.

> **Примечание** Когда `json` равен `true`, `ConsoleLogger` автоматически отключает раскраску текста, задавая свойству `colors` значение `false`. Это гарантирует, что вывод останется корректным JSON без артефактов форматирования. Однако в целях разработки это поведение можно переопределить, явно задав `colors` равным `true`. Тогда JSON-логи будут цветными, что делает записи более читаемыми при локальной отладке.

При включённом JSON-логировании вывод выглядит так (в одну строку):

```json
{
  "level": "log",
  "pid": 19096,
  "timestamp": 1607370779834,
  "message": "Starting Nest application...",
  "context": "NestFactory"
}
```

Различные варианты можно посмотреть в этом [Pull Request](https://github.com/nestjs/nest/pull/14121).

#### Использование логгера для логирования приложения

Мы можем скомбинировать несколько описанных выше приёмов, чтобы обеспечить единообразное поведение и форматирование как системного логирования Nest, так и логирования событий и сообщений нашего приложения.

Хорошая практика — создавать экземпляр класса `Logger` из `@nestjs/common` в каждом сервисе. Имя сервиса можно передать аргументом `context` в конструктор `Logger`:

```typescript
import { Logger, Injectable } from '@nestjs/common';

@Injectable()
class MyService {
  private readonly logger = new Logger(MyService.name);

  doSomething() {
    this.logger.log('Doing something...');
  }
}
```

В реализации логгера по умолчанию `context` выводится в квадратных скобках — как `NestFactory` в примере ниже:

```bash
[Nest] 19096   - 12/08/2019, 7:12:59 AM   [NestFactory] Starting Nest application...
```

Если мы передаём собственный логгер через `app.useLogger()`, Nest действительно будет использовать его внутри себя. Это значит, что наш код не зависит от конкретной реализации, а логгер по умолчанию легко заменяется на собственный вызовом `app.useLogger()`.

Таким образом, если следовать шагам из предыдущего раздела и вызвать `app.useLogger(app.get(MyLogger))`, то последующие вызовы `this.logger.log()` из `MyService` приведут к вызовам метода `log` у экземпляра `MyLogger`.

Для большинства случаев этого достаточно. Но если нужна более глубокая настройка (например, добавление и вызов собственных методов), переходите к следующему разделу.

#### Логи с отметками времени

Чтобы включить вывод отметки времени для каждого сообщения, при создании экземпляра логгера можно использовать необязательный параметр `timestamp: true`.

```typescript
import { Logger, Injectable } from '@nestjs/common';

@Injectable()
class MyService {
  private readonly logger = new Logger(MyService.name, { timestamp: true });

  doSomething() {
    this.logger.log('Doing something with timestamp here ->');
  }
}
```

Это даст вывод следующего вида:

```bash
[Nest] 19096   - 04/19/2024, 7:12:59 AM   [MyService] Doing something with timestamp here +5ms
```

Обратите внимание на `+5ms` в конце строки. Для каждой записи лога вычисляется разница во времени с предыдущим сообщением и выводится в конце строки.

#### Собственная реализация

Вы можете передать Nest собственную реализацию логгера для системного логирования, задав свойству `logger` объект, удовлетворяющий интерфейсу `LoggerService`. Например, можно указать Nest использовать встроенный глобальный объект JavaScript `console` (который реализует интерфейс `LoggerService`):

```typescript
const app = await NestFactory.create(AppModule, {
  logger: console,
});
await app.listen(process.env.PORT ?? 3000);
```

Реализовать собственный логгер несложно. Достаточно реализовать все методы интерфейса `LoggerService`, как показано ниже.

```typescript
import { LoggerService, Injectable } from '@nestjs/common';

@Injectable()
export class MyLogger implements LoggerService {
  /**
   * Записать лог уровня 'log'.
   */
  log(message: any, ...optionalParams: any[]) {}

  /**
   * Записать лог уровня 'fatal'.
   */
  fatal(message: any, ...optionalParams: any[]) {}

  /**
   * Записать лог уровня 'error'.
   */
  error(message: any, ...optionalParams: any[]) {}

  /**
   * Записать лог уровня 'warn'.
   */
  warn(message: any, ...optionalParams: any[]) {}

  /**
   * Записать лог уровня 'debug'.
   */
  debug?(message: any, ...optionalParams: any[]) {}

  /**
   * Записать лог уровня 'verbose'.
   */
  verbose?(message: any, ...optionalParams: any[]) {}
}
```

Затем экземпляр `MyLogger` можно передать через свойство `logger` объекта параметров приложения Nest.

```typescript
const app = await NestFactory.create(AppModule, {
  logger: new MyLogger(),
});
await app.listen(process.env.PORT ?? 3000);
```

Этот приём прост, но он не задействует внедрение зависимостей для класса `MyLogger`. Это создаёт определённые сложности, особенно при тестировании, и ограничивает переиспользуемость `MyLogger`. Более удачное решение описано в разделе <a href="10.%20Logging.md#внедрение-зависимостей">Внедрение зависимостей</a> ниже.

#### Расширение встроенного логгера

Вместо того чтобы писать логгер с нуля, вы, возможно, сможете решить свою задачу, расширив встроенный класс `ConsoleLogger` и переопределив отдельные аспекты поведения реализации по умолчанию.

```typescript
import { ConsoleLogger } from '@nestjs/common';

export class MyLogger extends ConsoleLogger {
  error(message: any, stack?: string, context?: string) {
    // здесь добавьте свою логику
    super.error(...arguments);
  }
}
```

Такой расширенный логгер можно применять в модулях функциональности, как описано в разделе <a href="10.%20Logging.md#использование-логгера-для-логирования-приложения">Использование логгера для логирования приложения</a> ниже.

Указать Nest использовать ваш расширенный логгер для системного логирования можно, передав его экземпляр через свойство `logger` объекта параметров приложения (как показано в разделе <a href="10.%20Logging.md#собственная-реализация">Собственная реализация</a> выше), либо приёмом из раздела <a href="10.%20Logging.md#внедрение-зависимостей">Внедрение зависимостей</a> ниже. В этом случае не забудьте вызвать `super`, как в примере кода выше, чтобы делегировать конкретный вызов метода логирования родительскому (встроенному) классу — тогда Nest сможет опираться на ожидаемые им встроенные возможности.

#### Внедрение зависимостей

Для более продвинутых сценариев логирования стоит воспользоваться внедрением зависимостей. Например, вы можете захотеть внедрить `ConfigService` в свой логгер для его настройки, а затем внедрить собственный логгер в другие контроллеры и/или провайдеры. Чтобы включить внедрение зависимостей для собственного логгера, создайте класс, реализующий `LoggerService`, и зарегистрируйте его как провайдер в каком-нибудь модуле. Например, вы можете:

1. Определить класс `MyLogger`, который либо расширяет встроенный `ConsoleLogger`, либо полностью его заменяет, как показано в предыдущих разделах. Обязательно реализуйте интерфейс `LoggerService`.
2. Создать `LoggerModule`, как показано ниже, и предоставить `MyLogger` из этого модуля.

```typescript
import { Module } from '@nestjs/common';
import { MyLogger } from './my-logger.service';

@Module({
  providers: [MyLogger],
  exports: [MyLogger],
})
export class LoggerModule {}
```

При такой конструкции ваш собственный логгер становится доступен любому другому модулю. Поскольку класс `MyLogger` входит в состав модуля, он может использовать внедрение зависимостей (например, внедрить `ConfigService`). Нужен ещё один приём, чтобы Nest использовал этот логгер и для системного логирования (например, при начальной загрузке и обработке ошибок).

Так как создание экземпляра приложения (`NestFactory.create()`) происходит вне контекста какого-либо модуля, оно не участвует в обычной фазе внедрения зависимостей при инициализации. Поэтому нужно убедиться, что хотя бы один модуль приложения импортирует `LoggerModule` — это заставит Nest создать одиночный экземпляр (синглтон) класса `MyLogger`.

Затем мы можем указать Nest использовать тот же самый экземпляр `MyLogger` такой конструкцией:

```typescript
const app = await NestFactory.create(AppModule, {
  bufferLogs: true,
});
app.useLogger(app.get(MyLogger));
await app.listen(process.env.PORT ?? 3000);
```

> **Примечание** В примере выше мы задали `bufferLogs` равным `true`, чтобы все логи буферизовались до тех пор, пока не будет подключён собственный логгер (в данном случае `MyLogger`) и процесс инициализации приложения не завершится успешно либо с ошибкой. Если инициализация завершится ошибкой, Nest откатится к исходному `ConsoleLogger` для вывода сообщений об ошибках. Кроме того, вы можете задать `autoFlushLogs` равным `false` (по умолчанию `true`), чтобы сбрасывать логи вручную (методом `Logger.flush()`).

Здесь мы используем метод `get()` экземпляра `NestApplication`, чтобы получить синглтон-экземпляр объекта `MyLogger`. По сути, это способ «внедрить» экземпляр логгера для использования самим Nest. Вызов `app.get()` возвращает синглтон-экземпляр `MyLogger` и полагается на то, что этот экземпляр был предварительно внедрён в другом модуле, как описано выше.

Провайдер `MyLogger` можно также внедрять в классы модулей функциональности — тем самым обеспечивается единообразное логирование как системных событий Nest, так и событий приложения. Подробнее см. <a href="10.%20Logging.md#использование-логгера-для-логирования-приложения">Использование логгера для логирования приложения</a> и <a href="10.%20Logging.md#внедрение-собственного-логгера">Внедрение собственного логгера</a> ниже.

#### Внедрение собственного логгера

Для начала расширьте встроенный логгер примерно так, как показано ниже. Мы передаём параметр `scope` в качестве метаданных конфигурации для класса `ConsoleLogger`, указывая [transient](<../03. Fundamentals/04. Injection scopes.md>)-область, чтобы каждый модуль функциональности получал собственный экземпляр логгера. Это важно, потому что каждый модуль задаёт своему логгеру уникальный контекст через `setContext()` — с синглтоном этот вызов перезаписывал бы контекст во всём приложении. В этом примере мы не переопределяем отдельные методы `ConsoleLogger` (такие как `log()`, `warn()` и т. д.), хотя вы можете это сделать.

```typescript
import { Injectable, Scope, ConsoleLogger } from '@nestjs/common';

@Injectable({ scope: Scope.TRANSIENT })
export class MyLogger extends ConsoleLogger {
  customLog() {
    this.log('Please feed the cat!');
  }
}
```

Далее создайте `LoggerModule` такой конструкцией:

```typescript
import { Module } from '@nestjs/common';
import { MyLogger } from './my-logger.service';

@Module({
  providers: [MyLogger],
  exports: [MyLogger],
})
export class LoggerModule {}
```

Затем импортируйте `LoggerModule` в свой модуль функциональности. Поскольку мы расширили стандартный `Logger`, нам доступен удобный метод `setContext`. Так что можно сразу пользоваться собственным логгером с учётом контекста:

```typescript
import { Injectable } from '@nestjs/common';
import { MyLogger } from './my-logger.service';

@Injectable()
export class CatsService {
  private readonly cats: Cat[] = [];

  constructor(private myLogger: MyLogger) {
    // Благодаря transient-области у CatsService есть собственный уникальный экземпляр MyLogger,
    // поэтому установка контекста здесь не затронет другие экземпляры в других сервисах
    this.myLogger.setContext('CatsService');
  }

  findAll(): Cat[] {
    // Можно вызывать все стандартные методы
    this.myLogger.warn('About to return cats!');
    // И свои собственные методы
    this.myLogger.customLog();
    return this.cats;
  }
}
```

Наконец, укажите Nest использовать экземпляр вашего логгера в файле `main.ts`, как показано ниже. Разумеется, в этом примере мы не меняли поведение логгера (не переопределяли методы `Logger` вроде `log()`, `warn()` и т. д.), поэтому такой шаг фактически не нужен. Но он **был бы** нужен, если бы вы добавили в эти методы свою логику и хотели, чтобы Nest использовал ту же реализацию.

```typescript
const app = await NestFactory.create(AppModule, {
  bufferLogs: true,
});
app.useLogger(new MyLogger());
await app.listen(process.env.PORT ?? 3000);
```

> **Подсказка** Вместо `bufferLogs: true` можно временно отключить логгер инструкцией `logger: false`. Учтите, что если передать `logger: false` в `NestFactory.create`, то до вызова `useLogger` ничего логироваться не будет, и вы можете пропустить важные ошибки инициализации. Если вас устраивает, что часть первых сообщений будет выведена логгером по умолчанию, можно просто не указывать параметр `logger: false`.

#### Использование внешнего логгера

У промышленных приложений часто есть особые требования к логированию: продвинутая фильтрация, форматирование, централизованный сбор логов. Встроенный логгер Nest предназначен для наблюдения за поведением самой системы Nest и может пригодиться для базового форматированного текстового логирования в модулях функциональности во время разработки, но в продакшене обычно применяют специализированные модули логирования вроде [Winston](https://github.com/winstonjs/winston). Как и в любом стандартном приложении Node.js, вы можете в полной мере пользоваться такими модулями в Nest.
