### Очереди

Очереди — это мощный паттерн проектирования, который помогает справляться с типичными задачами масштабирования и производительности приложения. Вот несколько примеров проблем, которые очереди помогают решить:

- Сглаживание пиков нагрузки. Например, если пользователи могут запускать ресурсоёмкие задачи в произвольные моменты времени, вы можете добавлять эти задачи в очередь вместо того, чтобы выполнять их синхронно. Затем воркеры будут забирать задачи из очереди контролируемым образом. По мере роста приложения вы легко добавите новых потребителей очереди, чтобы масштабировать обработку задач на бэкенде.
- Разбиение монолитных задач, которые иначе блокировали бы цикл событий Node.js. Например, если пользовательский запрос требует ресурсоёмких вычислений вроде перекодирования аудио, вы можете делегировать эту задачу другим процессам, освободив процессы, обращённые к пользователю, и сохранив их отзывчивость.
- Надёжный канал связи между разными сервисами. Например, вы можете ставить задачи (jobs) в очередь в одном процессе или сервисе, а обрабатывать их в другом. Вы можете получать уведомления (подписавшись на события состояния) о завершении, ошибке или других изменениях состояния в жизненном цикле задачи из любого процесса или сервиса. Если производители или потребители очереди выходят из строя, их состояние сохраняется, и обработка задач может автоматически возобновиться после перезапуска узлов.

Nest предоставляет пакет `@nestjs/bullmq` для интеграции с BullMQ и пакет `@nestjs/bull` для интеграции с Bull. Оба пакета — это абстракции-обёртки над соответствующими библиотеками, которые были разработаны одной и той же командой. Bull сейчас находится в режиме поддержки: команда сосредоточена на исправлении ошибок, тогда как BullMQ активно развивается и предлагает современную реализацию на TypeScript и другой набор возможностей. Если Bull отвечает вашим требованиям, он остаётся надёжным и проверенным в бою выбором. Пакеты Nest позволяют удобно интегрировать в приложение Nest очереди как BullMQ, так и Bull.

И BullMQ, и Bull используют [Redis](https://redis.io/) для сохранения данных задач, поэтому Redis должен быть установлен в вашей системе. Поскольку они работают поверх Redis, архитектура очередей может быть полностью распределённой и не зависеть от платформы. Например, часть <a href="09.%20Queues.md#производители">производителей</a>, <a href="09.%20Queues.md#потребители">потребителей</a> и <a href="09.%20Queues.md#слушатели-событий">слушателей</a> очереди может работать в Nest на одном (или нескольких) узлах, а другие производители, потребители и слушатели — на других платформах Node.js на других узлах сети.

Эта глава посвящена пакетам `@nestjs/bullmq` и `@nestjs/bull`. Мы также рекомендуем прочитать документацию [BullMQ](https://docs.bullmq.io/) и [Bull](https://github.com/OptimalBits/bull/blob/master/REFERENCE.md), чтобы лучше понять контекст и детали конкретных реализаций.

#### Установка BullMQ

Чтобы начать работу с BullMQ, сначала установим необходимые зависимости.

```bash
$ npm install --save @nestjs/bullmq bullmq
```

После завершения установки мы можем импортировать `BullModule` в корневой `AppModule`.

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { BullModule } from '@nestjs/bullmq';

@Module({
  imports: [
    BullModule.forRoot({
      connection: {
        host: 'localhost',
        port: 6379,
      },
    }),
  ],
})
export class AppModule {}
```

Метод `forRoot()` используется для регистрации объекта конфигурации пакета `bullmq`, который будет применяться ко всем очередям, зарегистрированным в приложении (если не указано иное). Для справки — вот несколько свойств объекта конфигурации:

- `connection: ConnectionOptions` — параметры настройки соединения с Redis. Подробнее см. [Connections](https://docs.bullmq.io/guide/connections). Необязательный.
- `prefix: string` — префикс для всех ключей очереди. Необязательный.
- `defaultJobOptions: JobOpts` — параметры, задающие настройки по умолчанию для новых задач. Подробнее см. [JobOpts](https://github.com/OptimalBits/bull/blob/master/REFERENCE.md#queueadd). Необязательный.
- `settings: AdvancedSettings` — расширенные настройки конфигурации очереди. Обычно их не следует менять. Подробнее см. [AdvancedSettings](https://github.com/OptimalBits/bull/blob/master/REFERENCE.md#queue). Необязательный.
- `extraOptions` — дополнительные параметры инициализации модуля. См. [Ручная регистрация](<09. Queues.md#ручная-регистрация>)

Все параметры необязательны и дают детальный контроль над поведением очереди. Они передаются напрямую в конструктор `Queue` из BullMQ. Подробнее об этих и других параметрах читайте [здесь](https://api.docs.bullmq.io/interfaces/v4.QueueOptions.html).

Чтобы зарегистрировать очередь, импортируйте динамический модуль `BullModule.registerQueue()`:

```typescript
BullModule.registerQueue({
  name: 'audio',
});
```

> **Подсказка** Чтобы создать несколько очередей, передайте методу `registerQueue()` несколько объектов конфигурации через запятую.

Метод `registerQueue()` используется для создания и/или регистрации очередей. Очереди являются общими для модулей и процессов, которые подключаются к одной и той же базе Redis с одними и теми же учётными данными. Каждая очередь уникальна по своему свойству name. Имя очереди используется одновременно как токен внедрения (для внедрения очереди в контроллеры/провайдеры) и как аргумент декораторов, связывающих классы потребителей и слушателей с очередями.

Вы также можете переопределить некоторые из заранее заданных параметров для конкретной очереди:

```typescript
BullModule.registerQueue({
  name: 'audio',
  connection: {
    port: 6380,
  },
});
```

BullMQ также поддерживает отношения «родитель — потомок» между задачами. Эта возможность позволяет создавать потоки (flows), в которых задачи являются узлами деревьев произвольной глубины. Подробнее о них читайте [здесь](https://docs.bullmq.io/guide/flows).

Чтобы добавить поток, сделайте следующее:

```typescript
BullModule.registerFlowProducer({
  name: 'flowProducerName',
});
```

Поскольку задачи сохраняются в Redis, при каждом создании очереди с определённым именем (например, при запуске или перезапуске приложения) она пытается обработать старые задачи, оставшиеся от предыдущей незавершённой сессии.

У каждой очереди может быть один или несколько производителей, потребителей и слушателей. Потребители забирают задачи из очереди в определённом порядке: FIFO (по умолчанию), LIFO или согласно приоритетам. Управление порядком обработки очереди обсуждается <a href="09.%20Queues.md#потребители">здесь</a>.

#### Именованные конфигурации

Если ваши очереди подключаются к нескольким разным экземплярам Redis, вы можете воспользоваться приёмом под названием **именованные конфигурации**. Эта возможность позволяет зарегистрировать несколько конфигураций под заданными ключами, на которые затем можно ссылаться в параметрах очереди.

Например, если у вас есть дополнительный экземпляр Redis (помимо основного), используемый несколькими очередями в приложении, вы можете зарегистрировать его конфигурацию так:

```typescript
BullModule.forRoot('alternative-config', {
  connection: {
    port: 6381,
  },
});
```

В примере выше `'alternative-config'` — это просто ключ конфигурации (им может быть любая произвольная строка).

Теперь вы можете сослаться на эту конфигурацию в объекте параметров `registerQueue()`:

```typescript
BullModule.registerQueue({
  configKey: 'alternative-config',
  name: 'video',
});
```

#### Производители

Производители задач добавляют задачи в очереди. Обычно производители — это сервисы приложения ([провайдеры](<../02. Overview/03. Providers.md>) Nest). Чтобы добавлять задачи в очередь, сначала внедрите очередь в сервис:

```typescript
import { Injectable } from '@nestjs/common';
import { Queue } from 'bullmq';
import { InjectQueue } from '@nestjs/bullmq';

@Injectable()
export class AudioService {
  constructor(@InjectQueue('audio') private audioQueue: Queue) {}
}
```

> **Подсказка** Декоратор `@InjectQueue()` определяет очередь по её имени, указанному при вызове метода `registerQueue()` (например, `'audio'`).

Теперь добавьте задачу, вызвав метод очереди `add()` и передав определённый вами объект задачи. Задачи представляются сериализуемыми объектами JavaScript (именно в таком виде они хранятся в базе Redis). Форма передаваемой задачи произвольна: используйте её, чтобы выразить семантику вашего объекта задачи. Также нужно указать имя. Это позволяет создавать специализированных <a href="09.%20Queues.md#потребители">потребителей</a>, которые будут обрабатывать только задачи с заданным именем.

```typescript
const job = await this.audioQueue.add('transcode', {
  foo: 'bar',
});
```

#### Параметры задачи

С задачами могут быть связаны дополнительные параметры. Передайте объект параметров после аргумента `job` в методе `Queue.add()`. Вот некоторые из свойств параметров задачи:

- `priority`: `number` — необязательное значение приоритета. Диапазон — от 1 (наивысший приоритет) до MAX_INT (наинизший приоритет). Учтите, что использование приоритетов слегка сказывается на производительности, поэтому применяйте их осторожно.
- `delay`: `number` — время (в миллисекундах) ожидания до того, как задача может быть обработана. Учтите, что для точных задержек часы сервера и клиентов должны быть синхронизированы.
- `attempts`: `number` — общее количество попыток выполнить задачу до её успешного завершения.
- `repeat`: `RepeatOpts` — повторять задачу согласно cron-спецификации. См. [RepeatOpts](https://github.com/OptimalBits/bull/blob/master/REFERENCE.md#queueadd).
- `backoff`: `number | BackoffOpts` — настройка задержки перед автоматическими повторами, если задача завершилась с ошибкой. См. [BackoffOpts](https://github.com/OptimalBits/bull/blob/master/REFERENCE.md#queueadd).
- `lifo`: `boolean` — если true, добавляет задачу в правый конец очереди вместо левого (по умолчанию false).
- `jobId`: `number` | `string` — переопределение идентификатора задачи. По умолчанию идентификатор задачи — уникальное
  целое число, но этой настройкой его можно переопределить. Если вы используете этот параметр, уникальность `jobId` — ваша ответственность. При попытке добавить задачу с уже существующим id она добавлена не будет.
- `removeOnComplete`: `boolean | number` — если true, удаляет задачу после её успешного завершения. Число задаёт количество сохраняемых задач. По умолчанию задача остаётся в наборе завершённых.
- `removeOnFail`: `boolean | number` — если true, удаляет задачу, когда она завершается с ошибкой после всех попыток. Число задаёт количество сохраняемых задач. По умолчанию задача остаётся в наборе неудачных.
- `stackTraceLimit`: `number` — ограничивает количество строк стека вызовов, которые будут записаны в трассировку.

Ниже — несколько примеров настройки задач с помощью параметров.

Чтобы отложить старт задачи, используйте свойство конфигурации `delay`.

```typescript
const job = await this.audioQueue.add(
  'transcode',
  {
    foo: 'bar',
  },
  { delay: 3000 }, // задержка 3 секунды
);
```

Чтобы добавить задачу в правый конец очереди (обрабатывать задачу по принципу **LIFO** (Last In First Out)), задайте свойству `lifo` объекта конфигурации значение `true`.

```typescript
const job = await this.audioQueue.add(
  'transcode',
  {
    foo: 'bar',
  },
  { lifo: true },
);
```

Чтобы задать приоритет задачи, используйте свойство `priority`.

```typescript
const job = await this.audioQueue.add(
  'transcode',
  {
    foo: 'bar',
  },
  { priority: 2 },
);
```

Полный список параметров смотрите в документации API [здесь](https://api.docs.bullmq.io/types/v4.JobsOptions.html) и [здесь](https://api.docs.bullmq.io/interfaces/v4.BaseJobOptions.html).

#### Потребители

Потребитель — это **класс**, определяющий методы, которые либо обрабатывают добавленные в очередь задачи, либо слушают события очереди, либо и то и другое. Объявите класс потребителя с помощью декоратора `@Processor()`:

```typescript
import { Processor } from '@nestjs/bullmq';

@Processor('audio')
export class AudioConsumer {}
```

> **Подсказка** Потребители должны быть зарегистрированы как `providers`, чтобы пакет `@nestjs/bullmq` смог их обнаружить.

Строковый аргумент декоратора (например, `'audio'`) — это имя очереди, с которой будут связаны методы класса.

```typescript
import { Processor, WorkerHost } from '@nestjs/bullmq';
import { Job } from 'bullmq';

@Processor('audio')
export class AudioConsumer extends WorkerHost {
  async process(job: Job<any, any, string>): Promise<any> {
    let progress = 0;
    for (let i = 0; i < 100; i++) {
      await doSomething(job.data);
      progress += 1;
      await job.updateProgress(progress);
    }
    return {};
  }
}
```

Метод process вызывается всякий раз, когда воркер простаивает, а в очереди есть задачи для обработки. Этот метод-обработчик получает объект `job` в качестве единственного аргумента. Значение, возвращённое обработчиком, сохраняется в объекте задачи и может быть получено позднее — например, в слушателе события завершения.

У объектов `Job` есть несколько методов, позволяющих взаимодействовать с их состоянием. Например, в коде выше используется метод `updateProgress()` для обновления прогресса задачи. Полное описание API объекта `Job` смотрите [здесь](https://api.docs.bullmq.io/classes/v4.Job.html).

В более старой версии, Bull, вы могли указать, что метод-обработчик задач будет обрабатывать **только** задачи определённого типа (задачи с конкретным `name`), передав это `name` в декоратор `@Process()`, как показано ниже.

> **Предупреждение** В BullMQ это не работает, читайте дальше.

```typescript
@Process('transcode')
async transcode(job: Job<unknown>) { ... }
```

Такое поведение не поддерживается в BullMQ из-за путаницы, которую оно порождало. Вместо этого вам нужны конструкции switch, чтобы вызывать разные сервисы или логику для каждого имени задачи:

```typescript
import { Processor, WorkerHost } from '@nestjs/bullmq';
import { Job } from 'bullmq';

@Processor('audio')
export class AudioConsumer extends WorkerHost {
  async process(job: Job<any, any, string>): Promise<any> {
    switch (job.name) {
      case 'transcode': {
        let progress = 0;
        for (i = 0; i < 100; i++) {
          await doSomething(job.data);
          progress += 1;
          await job.progress(progress);
        }
        return {};
      }
      case 'concatenate': {
        await doSomeLogic2();
        break;
      }
    }
  }
}
```

Это описано в разделе [named processor](https://docs.bullmq.io/patterns/named-processor) документации BullMQ.

#### Потребители с областью видимости запроса

Когда потребитель помечен как имеющий область видимости запроса (подробнее об областях внедрения — [здесь](<../03. Fundamentals/04. Injection scopes.md#область-провайдера>)), для каждой задачи будет создаваться отдельный экземпляр класса. После завершения задачи экземпляр будет удалён сборщиком мусора.

```typescript
@Processor({
  name: 'audio',
  scope: Scope.REQUEST,
})
```

Поскольку классы потребителей с областью видимости запроса создаются динамически и привязаны к одной задаче, вы можете внедрить `JOB_REF` через конструктор стандартным способом.

```typescript
constructor(@Inject(JOB_REF) jobRef: Job) {
  console.log(jobRef);
}
```

> **Подсказка** Токен `JOB_REF` импортируется из пакета `@nestjs/bullmq`.

#### Слушатели событий

BullMQ порождает набор полезных событий при изменении состояния очереди и/или задачи. На эти события можно подписаться на уровне воркера с помощью декоратора `@OnWorkerEvent(event)` либо на уровне очереди — с помощью отдельного класса-слушателя и декоратора `@OnQueueEvent(event)`.

События воркера должны объявляться внутри класса-<a href="09.%20Queues.md#потребители">потребителя</a> (то есть внутри класса, помеченного декоратором `@Processor()`). Чтобы слушать событие, используйте декоратор `@OnWorkerEvent(event)` с нужным вам событием. Например, чтобы слушать событие, генерируемое при переходе задачи в активное состояние в очереди `audio`, используйте такую конструкцию:

```typescript
import { Processor, Process, OnWorkerEvent } from '@nestjs/bullmq';
import { Job } from 'bullmq';

@Processor('audio')
export class AudioConsumer {
  @OnWorkerEvent('active')
  onActive(job: Job) {
    console.log(
      `Processing job ${job.id} of type ${job.name} with data ${job.data}...`,
    );
  }

  // ...
}
```

Полный список событий и их аргументов в виде свойств WorkerListener можно посмотреть [здесь](https://api.docs.bullmq.io/interfaces/v4.WorkerListener.html).

Слушатели QueueEvent должны использовать декоратор `@QueueEventsListener(queue)` и наследовать класс `QueueEventsHost`, предоставляемый `@nestjs/bullmq`. Чтобы слушать событие, используйте декоратор `@OnQueueEvent(event)` с нужным вам событием. Например, чтобы слушать событие, генерируемое при переходе задачи в активное состояние в очереди `audio`, используйте такую конструкцию:

```typescript
import {
  QueueEventsHost,
  QueueEventsListener,
  OnQueueEvent,
} from '@nestjs/bullmq';

@QueueEventsListener('audio')
export class AudioEventsListener extends QueueEventsHost {
  @OnQueueEvent('active')
  onActive(job: { jobId: string; prev?: string }) {
    console.log(`Processing job ${job.jobId}...`);
  }

  // ...
}
```

> **Подсказка** Слушатели QueueEvent должны быть зарегистрированы как `providers`, чтобы пакет `@nestjs/bullmq` смог их обнаружить.

Полный список событий и их аргументов в виде свойств QueueEventsListener можно посмотреть [здесь](https://api.docs.bullmq.io/interfaces/v4.QueueEventsListener.html).

#### Управление очередями

У очередей есть API, позволяющий выполнять управляющие операции: приостановку и возобновление, получение количества задач в различных состояниях и многое другое. Полное описание API очереди доступно [здесь](https://api.docs.bullmq.io/classes/v4.Queue.html). Вызывайте любой из этих методов непосредственно на объекте `Queue`, как показано ниже в примерах с приостановкой и возобновлением.

Приостановите очередь вызовом метода `pause()`. Приостановленная очередь не будет обрабатывать новые задачи до возобновления, но уже обрабатываемые задачи будут выполняться до их завершения.

```typescript
await audioQueue.pause();
```

Чтобы возобновить приостановленную очередь, используйте метод `resume()`:

```typescript
await audioQueue.resume();
```

#### Отдельные процессы

Обработчики задач могут также выполняться в отдельном (форкнутом) процессе ([источник](https://docs.bullmq.io/guide/workers/sandboxed-processors)). У этого есть несколько преимуществ:

- Процесс изолирован в песочнице, поэтому его падение не затрагивает воркер.
- Вы можете выполнять блокирующий код, не влияя на очередь (задачи не будут зависать).
- Гораздо лучшая утилизация многоядерных процессоров.
- Меньше соединений с redis.

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { BullModule } from '@nestjs/bullmq';
import { join } from 'node:path';

@Module({
  imports: [
    BullModule.registerQueue({
      name: 'audio',
      processors: [join(__dirname, 'processor.js')],
    }),
  ],
})
export class AppModule {}
```

> **Предупреждение** Обратите внимание: поскольку ваша функция выполняется в форкнутом процессе, внедрение зависимостей (и IoC-контейнер) будут недоступны. Это значит, что функция-процессор должна содержать (или создавать) все экземпляры внешних зависимостей, которые ей нужны.

#### Асинхронная конфигурация

Возможно, вы захотите передавать параметры `bullmq` асинхронно, а не статически. В этом случае используйте метод `forRootAsync()`, который предлагает несколько способов работы с асинхронной конфигурацией. Аналогично, если вы хотите передавать параметры очереди асинхронно, используйте метод `registerQueueAsync()`.

Один из подходов — использовать фабричную функцию:

```typescript
BullModule.forRootAsync({
  useFactory: () => ({
    connection: {
      host: 'localhost',
      port: 6379,
    },
  }),
});
```

Наша фабрика ведёт себя как любой другой [асинхронный провайдер](<../03. Fundamentals/02. Asynchronous providers.md>) (например, она может быть `async` и способна внедрять зависимости через `inject`).

```typescript
BullModule.forRootAsync({
  imports: [ConfigModule],
  useFactory: async (configService: ConfigService) => ({
    connection: {
      host: configService.get('QUEUE_HOST'),
      port: configService.get('QUEUE_PORT'),
    },
  }),
  inject: [ConfigService],
});
```

Как вариант, можно использовать синтаксис `useClass`:

```typescript
BullModule.forRootAsync({
  useClass: BullConfigService,
});
```

Конструкция выше создаст экземпляр `BullConfigService` внутри `BullModule` и использует его для получения объекта параметров через вызов `createSharedConfiguration()`. Учтите, что для этого `BullConfigService` должен реализовывать интерфейс `SharedBullConfigurationFactory`, как показано ниже:

```typescript
@Injectable()
class BullConfigService implements SharedBullConfigurationFactory {
  createSharedConfiguration(): BullModuleOptions {
    return {
      connection: {
        host: 'localhost',
        port: 6379,
      },
    };
  }
}
```

Чтобы предотвратить создание `BullConfigService` внутри `BullModule` и использовать провайдер, импортированный из другого модуля, вы можете применить синтаксис `useExisting`.

```typescript
BullModule.forRootAsync({
  imports: [ConfigModule],
  useExisting: ConfigService,
});
```

Эта конструкция работает так же, как `useClass`, с одним важным отличием: `BullModule` просмотрит импортированные модули, чтобы переиспользовать существующий `ConfigService`, вместо создания нового.

Аналогично, если вы хотите передавать параметры очереди асинхронно, используйте метод `registerQueueAsync()` — только не забудьте указать атрибут `name` вне фабричной функции.

```typescript
BullModule.registerQueueAsync({
  name: 'audio',
  useFactory: () => ({
    redis: {
      host: 'localhost',
      port: 6379,
    },
  }),
});
```

#### Ручная регистрация

По умолчанию `BullModule` автоматически регистрирует компоненты BullMQ (очереди, процессоры и сервисы-слушатели событий) в функции жизненного цикла `onModuleInit`. Однако в некоторых случаях такое поведение может оказаться нежелательным. Чтобы отключить автоматическую регистрацию, включите `manualRegistration` в `BullModule` следующим образом:

```typescript
BullModule.forRoot({
  extraOptions: {
    manualRegistration: true,
  },
});
```

Чтобы зарегистрировать эти компоненты вручную, внедрите `BullRegistrar` и вызовите функцию `register` — лучше всего внутри `OnModuleInit` или `OnApplicationBootstrap`.

```typescript
import { Injectable, OnModuleInit } from '@nestjs/common';
import { BullRegistrar } from '@nestjs/bullmq';

@Injectable()
export class AudioService implements OnModuleInit {
  constructor(private bullRegistrar: BullRegistrar) {}

  onModuleInit() {
    if (yourConditionHere) {
      this.bullRegistrar.register();
    }
  }
}
```

Пока вы не вызовете функцию `BullRegistrar#register`, ни один компонент BullMQ работать не будет — то есть ни одна задача не будет обработана.

#### Установка Bull

> **Примечание** Если вы решили использовать BullMQ, пропустите этот раздел и следующие главы.

Чтобы начать работу с Bull, сначала установим необходимые зависимости.

```bash
$ npm install --save @nestjs/bull bull
```

После завершения установки мы можем импортировать `BullModule` в корневой `AppModule`.

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { BullModule } from '@nestjs/bull';

@Module({
  imports: [
    BullModule.forRoot({
      redis: {
        host: 'localhost',
        port: 6379,
      },
    }),
  ],
})
export class AppModule {}
```

Метод `forRoot()` используется для регистрации объекта конфигурации пакета `bull`, который будет применяться ко всем очередям, зарегистрированным в приложении (если не указано иное). Объект конфигурации состоит из следующих свойств:

- `limiter: RateLimiter` — параметры, управляющие скоростью обработки задач очереди. Подробнее см. [RateLimiter](https://github.com/OptimalBits/bull/blob/master/REFERENCE.md#queue). Необязательный.
- `redis: RedisOpts` — параметры настройки соединения с Redis. Подробнее см. [RedisOpts](https://github.com/OptimalBits/bull/blob/master/REFERENCE.md#queue). Необязательный.
- `prefix: string` — префикс для всех ключей очереди. Необязательный.
- `defaultJobOptions: JobOpts` — параметры, задающие настройки по умолчанию для новых задач. Подробнее см. [JobOpts](https://github.com/OptimalBits/bull/blob/master/REFERENCE.md#queueadd). Необязательный. **Примечание: они не действуют, если вы планируете задачи через FlowProducer. Объяснение — в [bullmq#1034](https://github.com/taskforcesh/bullmq/issues/1034).**
- `settings: AdvancedSettings` — расширенные настройки конфигурации очереди. Обычно их не следует менять. Подробнее см. [AdvancedSettings](https://github.com/OptimalBits/bull/blob/master/REFERENCE.md#queue). Необязательный.

Все параметры необязательны и дают детальный контроль над поведением очереди. Они передаются напрямую в конструктор `Queue` из Bull. Подробнее об этих параметрах читайте [здесь](https://github.com/OptimalBits/bull/blob/master/REFERENCE.md#queue).

Чтобы зарегистрировать очередь, импортируйте динамический модуль `BullModule.registerQueue()`:

```typescript
BullModule.registerQueue({
  name: 'audio',
});
```

> **Подсказка** Чтобы создать несколько очередей, передайте методу `registerQueue()` несколько объектов конфигурации через запятую.

Метод `registerQueue()` используется для создания и/или регистрации очередей. Очереди являются общими для модулей и процессов, которые подключаются к одной и той же базе Redis с одними и теми же учётными данными. Каждая очередь уникальна по своему свойству name. Имя очереди используется одновременно как токен внедрения (для внедрения очереди в контроллеры/провайдеры) и как аргумент декораторов, связывающих классы потребителей и слушателей с очередями.

Вы также можете переопределить некоторые из заранее заданных параметров для конкретной очереди:

```typescript
BullModule.registerQueue({
  name: 'audio',
  redis: {
    port: 6380,
  },
});
```

Поскольку задачи сохраняются в Redis, при каждом создании очереди с определённым именем (например, при запуске или перезапуске приложения) она пытается обработать старые задачи, оставшиеся от предыдущей незавершённой сессии.

У каждой очереди может быть один или несколько производителей, потребителей и слушателей. Потребители забирают задачи из очереди в определённом порядке: FIFO (по умолчанию), LIFO или согласно приоритетам. Управление порядком обработки очереди обсуждается <a href="09.%20Queues.md#потребители">здесь</a>.

#### Именованные конфигурации

Если ваши очереди подключаются к нескольким экземплярам Redis, вы можете воспользоваться приёмом под названием **именованные конфигурации**. Эта возможность позволяет зарегистрировать несколько конфигураций под заданными ключами, на которые затем можно ссылаться в параметрах очереди.

Например, если у вас есть дополнительный экземпляр Redis (помимо основного), используемый несколькими очередями в приложении, вы можете зарегистрировать его конфигурацию так:

```typescript
BullModule.forRoot('alternative-config', {
  redis: {
    port: 6381,
  },
});
```

В примере выше `'alternative-config'` — это просто ключ конфигурации (им может быть любая произвольная строка).

Теперь вы можете сослаться на эту конфигурацию в объекте параметров `registerQueue()`:

```typescript
BullModule.registerQueue({
  configKey: 'alternative-config',
  name: 'video',
});
```

#### Производители

Производители задач добавляют задачи в очереди. Обычно производители — это сервисы приложения ([провайдеры](<../02. Overview/03. Providers.md>) Nest). Чтобы добавлять задачи в очередь, сначала внедрите очередь в сервис:

```typescript
import { Injectable } from '@nestjs/common';
import { Queue } from 'bull';
import { InjectQueue } from '@nestjs/bull';

@Injectable()
export class AudioService {
  constructor(@InjectQueue('audio') private audioQueue: Queue) {}
}
```

> **Подсказка** Декоратор `@InjectQueue()` определяет очередь по её имени, указанному при вызове метода `registerQueue()` (например, `'audio'`).

Теперь добавьте задачу, вызвав метод очереди `add()` и передав определённый вами объект задачи. Задачи представляются сериализуемыми объектами JavaScript (именно в таком виде они хранятся в базе Redis). Форма передаваемой задачи произвольна: используйте её, чтобы выразить семантику вашего объекта задачи.

```typescript
const job = await this.audioQueue.add({
  foo: 'bar',
});
```

#### Именованные задачи

У задач могут быть уникальные имена. Это позволяет создавать специализированных <a href="09.%20Queues.md#потребители">потребителей</a>, которые будут обрабатывать только задачи с заданным именем.

```typescript
const job = await this.audioQueue.add('transcode', {
  foo: 'bar',
});
```

> **Предупреждение** При использовании именованных задач вы должны создать процессоры для каждого уникального имени, добавляемого в очередь, иначе очередь будет сообщать об отсутствии процессора для данной задачи. Подробнее об обработке именованных задач — <a href="09.%20Queues.md#потребители">здесь</a>.

#### Параметры задачи

С задачами могут быть связаны дополнительные параметры. Передайте объект параметров после аргумента `job` в методе `Queue.add()`. Свойства параметров задачи таковы:

- `priority`: `number` — необязательное значение приоритета. Диапазон — от 1 (наивысший приоритет) до MAX_INT (наинизший приоритет). Учтите, что использование приоритетов слегка сказывается на производительности, поэтому применяйте их осторожно.
- `delay`: `number` — время (в миллисекундах) ожидания до того, как задача может быть обработана. Учтите, что для точных задержек часы сервера и клиентов должны быть синхронизированы.
- `attempts`: `number` — общее количество попыток выполнить задачу до её успешного завершения.
- `repeat`: `RepeatOpts` — повторять задачу согласно cron-спецификации. См. [RepeatOpts](https://github.com/OptimalBits/bull/blob/master/REFERENCE.md#queueadd).
- `backoff`: `number | BackoffOpts` — настройка задержки перед автоматическими повторами, если задача завершилась с ошибкой. См. [BackoffOpts](https://github.com/OptimalBits/bull/blob/master/REFERENCE.md#queueadd).
- `lifo`: `boolean` — если true, добавляет задачу в правый конец очереди вместо левого (по умолчанию false).
- `timeout`: `number` — число миллисекунд, по истечении которых задача должна завершиться ошибкой по тайм-ауту.
- `jobId`: `number` | `string` — переопределение идентификатора задачи. По умолчанию идентификатор задачи — уникальное
  целое число, но этой настройкой его можно переопределить. Если вы используете этот параметр, уникальность `jobId` — ваша ответственность. При попытке добавить задачу с уже существующим id она добавлена не будет.
- `removeOnComplete`: `boolean | number` — если true, удаляет задачу после её успешного завершения. Число задаёт количество сохраняемых задач. По умолчанию задача остаётся в наборе завершённых.
- `removeOnFail`: `boolean | number` — если true, удаляет задачу, когда она завершается с ошибкой после всех попыток. Число задаёт количество сохраняемых задач. По умолчанию задача остаётся в наборе неудачных.
- `stackTraceLimit`: `number` — ограничивает количество строк стека вызовов, которые будут записаны в трассировку.

Ниже — несколько примеров настройки задач с помощью параметров.

Чтобы отложить старт задачи, используйте свойство конфигурации `delay`.

```typescript
const job = await this.audioQueue.add(
  {
    foo: 'bar',
  },
  { delay: 3000 }, // задержка 3 секунды
);
```

Чтобы добавить задачу в правый конец очереди (обрабатывать задачу по принципу **LIFO** (Last In First Out)), задайте свойству `lifo` объекта конфигурации значение `true`.

```typescript
const job = await this.audioQueue.add(
  {
    foo: 'bar',
  },
  { lifo: true },
);
```

Чтобы задать приоритет задачи, используйте свойство `priority`.

```typescript
const job = await this.audioQueue.add(
  {
    foo: 'bar',
  },
  { priority: 2 },
);
```

#### Потребители

Потребитель — это **класс**, определяющий методы, которые либо обрабатывают добавленные в очередь задачи, либо слушают события очереди, либо и то и другое. Объявите класс потребителя с помощью декоратора `@Processor()`:

```typescript
import { Processor } from '@nestjs/bull';

@Processor('audio')
export class AudioConsumer {}
```

> **Подсказка** Потребители должны быть зарегистрированы как `providers`, чтобы пакет `@nestjs/bull` смог их обнаружить.

Строковый аргумент декоратора (например, `'audio'`) — это имя очереди, с которой будут связаны методы класса.

Внутри класса потребителя объявляйте обработчики задач, помечая методы-обработчики декоратором `@Process()`.

```typescript
import { Processor, Process } from '@nestjs/bull';
import { Job } from 'bull';

@Processor('audio')
export class AudioConsumer {
  @Process()
  async transcode(job: Job<unknown>) {
    let progress = 0;
    for (let i = 0; i < 100; i++) {
      await doSomething(job.data);
      progress += 1;
      await job.progress(progress);
    }
    return {};
  }
}
```

Помеченный декоратором метод (например, `transcode()`) вызывается всякий раз, когда воркер простаивает, а в очереди есть задачи для обработки. Этот метод-обработчик получает объект `job` в качестве единственного аргумента. Значение, возвращённое обработчиком, сохраняется в объекте задачи и может быть получено позднее — например, в слушателе события завершения.

У объектов `Job` есть несколько методов, позволяющих взаимодействовать с их состоянием. Например, в коде выше используется метод `progress()` для обновления прогресса задачи. Полное описание API объекта `Job` смотрите [здесь](https://github.com/OptimalBits/bull/blob/master/REFERENCE.md#job).

Вы можете указать, что метод-обработчик задач будет обрабатывать **только** задачи определённого типа (задачи с конкретным `name`), передав это `name` в декоратор `@Process()`, как показано ниже. В одном классе потребителя может быть несколько обработчиков `@Process()`, соответствующих каждому типу задач (`name`). Когда вы используете именованные задачи, убедитесь, что для каждого имени есть свой обработчик.

```typescript
@Process('transcode')
async transcode(job: Job<unknown>) { ... }
```

> **Предупреждение** Если для одной очереди определено несколько потребителей, параметр `concurrency` в `@Process({ concurrency: 1 })` не подействует. Минимальное значение `concurrency` будет равно количеству определённых потребителей. Это верно и в том случае, когда обработчики `@Process()` используют разные `name` для обработки именованных задач.

#### Потребители с областью видимости запроса

Когда потребитель помечен как имеющий область видимости запроса (подробнее об областях внедрения — [здесь](<../03. Fundamentals/04. Injection scopes.md#область-провайдера>)), для каждой задачи будет создаваться отдельный экземпляр класса. После завершения задачи экземпляр будет удалён сборщиком мусора.

```typescript
@Processor({
  name: 'audio',
  scope: Scope.REQUEST,
})
```

Поскольку классы потребителей с областью видимости запроса создаются динамически и привязаны к одной задаче, вы можете внедрить `JOB_REF` через конструктор стандартным способом.

```typescript
constructor(@Inject(JOB_REF) jobRef: Job) {
  console.log(jobRef);
}
```

> **Подсказка** Токен `JOB_REF` импортируется из пакета `@nestjs/bull`.

#### Слушатели событий

Bull порождает набор полезных событий при изменении состояния очереди и/или задачи. Nest предоставляет набор декораторов, позволяющих подписаться на основной набор стандартных событий. Они экспортируются из пакета `@nestjs/bull`.

Слушатели событий должны объявляться внутри класса-<a href="09.%20Queues.md#потребители">потребителя</a> (то есть внутри класса, помеченного декоратором `@Processor()`). Чтобы слушать событие, используйте один из декораторов из таблицы ниже для объявления обработчика этого события. Например, чтобы слушать событие, генерируемое при переходе задачи в активное состояние в очереди `audio`, используйте такую конструкцию:

```typescript
import { Processor, Process, OnQueueActive } from '@nestjs/bull';
import { Job } from 'bull';

@Processor('audio')
export class AudioConsumer {

  @OnQueueActive()
  onActive(job: Job) {
    console.log(
      `Processing job ${job.id} of type ${job.name} with data ${job.data}...`,
    );
  }
  ...
```

Поскольку Bull работает в распределённой (многоузловой) среде, в нём вводится понятие локальности событий. Оно отражает тот факт, что события могут порождаться либо целиком внутри одного процесса, либо в общих очередях из разных процессов. **Локальное** событие — это событие, возникающее при действии или смене состояния очереди в локальном процессе. Иначе говоря, когда ваши производители и потребители событий локальны для одного процесса, все события в очередях являются локальными.

Когда очередь используется совместно несколькими процессами, появляется возможность **глобальных** событий. Чтобы слушатель в одном процессе получал уведомление о событии, вызванном другим процессом, он должен зарегистрироваться на глобальное событие.

Обработчики событий вызываются всякий раз, когда порождается соответствующее событие. Обработчик вызывается с сигнатурой, показанной в таблице ниже, что даёт доступ к относящейся к событию информации. Одно важное различие между сигнатурами локальных и глобальных обработчиков событий разобрано ниже.

<table>
  <tr>
    <th>Локальные слушатели событий</th>
    <th>Глобальные слушатели событий</th>
    <th>Сигнатура метода-обработчика / когда срабатывает</th>
  </tr>
  <tr>
    <td><code>@OnQueueError()</code></td><td><code>@OnGlobalQueueError()</code></td><td><code>handler(error: Error)</code> — произошла ошибка. <code>error</code> содержит вызвавшую её ошибку.</td>
  </tr>
  <tr>
    <td><code>@OnQueueWaiting()</code></td><td><code>@OnGlobalQueueWaiting()</code></td><td><code>handler(jobId: number | string)</code> — задача ожидает обработки, как только освободится воркер. <code>jobId</code> содержит id задачи, перешедшей в это состояние.</td>
  </tr>
  <tr>
    <td><code>@OnQueueActive()</code></td><td><code>@OnGlobalQueueActive()</code></td><td><code>handler(job: Job)</code> — задача <code>job</code>началась. </td>
  </tr>
  <tr>
    <td><code>@OnQueueStalled()</code></td><td><code>@OnGlobalQueueStalled()</code></td><td><code>handler(job: Job)</code> — задача <code>job</code> помечена как зависшая. Это полезно для отладки воркеров, которые падают или приостанавливают цикл событий.</td>
  </tr>
  <tr>
    <td><code>@OnQueueProgress()</code></td><td><code>@OnGlobalQueueProgress()</code></td><td><code>handler(job: Job, progress: number)</code> — прогресс задачи <code>job</code> обновлён до значения <code>progress</code>.</td>
  </tr>
  <tr>
    <td><code>@OnQueueCompleted()</code></td><td><code>@OnGlobalQueueCompleted()</code></td><td><code>handler(job: Job, result: any)</code> — задача <code>job</code> успешно завершена с результатом <code>result</code>.</td>
  </tr>
  <tr>
    <td><code>@OnQueueFailed()</code></td><td><code>@OnGlobalQueueFailed()</code></td><td><code>handler(job: Job, err: Error)</code> — задача <code>job</code> завершилась неудачей по причине <code>err</code>.</td>
  </tr>
  <tr>
    <td><code>@OnQueuePaused()</code></td><td><code>@OnGlobalQueuePaused()</code></td><td><code>handler()</code> — очередь приостановлена.</td>
  </tr>
  <tr>
    <td><code>@OnQueueResumed()</code></td><td><code>@OnGlobalQueueResumed()</code></td><td><code>handler(job: Job)</code> — очередь возобновлена.</td>
  </tr>
  <tr>
    <td><code>@OnQueueCleaned()</code></td><td><code>@OnGlobalQueueCleaned()</code></td><td><code>handler(jobs: Job[], type: string)</code> — старые задачи очищены из очереди. <code>jobs</code> — массив очищенных задач, <code>type</code> — тип очищенных задач.</td>
  </tr>
  <tr>
    <td><code>@OnQueueDrained()</code></td><td><code>@OnGlobalQueueDrained()</code></td><td><code>handler()</code> — порождается всякий раз, когда очередь обработала все ожидающие задачи (даже если остались ещё не обработанные отложенные задачи).</td>
  </tr>
  <tr>
    <td><code>@OnQueueRemoved()</code></td><td><code>@OnGlobalQueueRemoved()</code></td><td><code>handler(job: Job)</code> — задача <code>job</code> успешно удалена.</td>
  </tr>

При прослушивании глобальных событий сигнатуры методов могут слегка отличаться от их локальных аналогов. В частности, любая сигнатура, которая в локальной версии получает объекты `job`, в глобальной версии получает `jobId` (`number`). Чтобы в таком случае получить ссылку на сам объект `job`, используйте метод `Queue#getJob`. Этот вызов нужно ожидать через await, поэтому обработчик должен быть объявлен как `async`. Например:

```typescript
@OnGlobalQueueCompleted()
async onGlobalCompleted(jobId: number, result: any) {
  const job = await this.immediateQueue.getJob(jobId);
  console.log('(Global) on completed: job ', job.id, ' -> result: ', result);
}
```

> **Подсказка** Чтобы обратиться к объекту `Queue` (для вызова `getJob()`), его, разумеется, нужно внедрить. Кроме того, очередь должна быть зарегистрирована в том модуле, в котором вы её внедряете.

Помимо специализированных декораторов-слушателей событий, вы можете использовать обобщённый декоратор `@OnQueueEvent()` в сочетании с перечислениями `BullQueueEvents` или `BullQueueGlobalEvents`. Подробнее о событиях читайте [здесь](https://github.com/OptimalBits/bull/blob/master/REFERENCE.md#events).

#### Управление очередями

У очередей есть API, позволяющий выполнять управляющие операции: приостановку и возобновление, получение количества задач в различных состояниях и многое другое. Полное описание API очереди доступно [здесь](https://github.com/OptimalBits/bull/blob/master/REFERENCE.md#queue). Вызывайте любой из этих методов непосредственно на объекте `Queue`, как показано ниже в примерах с приостановкой и возобновлением.

Приостановите очередь вызовом метода `pause()`. Приостановленная очередь не будет обрабатывать новые задачи до возобновления, но уже обрабатываемые задачи будут выполняться до их завершения.

```typescript
await audioQueue.pause();
```

Чтобы возобновить приостановленную очередь, используйте метод `resume()`:

```typescript
await audioQueue.resume();
```

#### Отдельные процессы

Обработчики задач могут также выполняться в отдельном (форкнутом) процессе ([источник](https://github.com/OptimalBits/bull#separate-processes)). У этого есть несколько преимуществ:

- Процесс изолирован в песочнице, поэтому его падение не затрагивает воркер.
- Вы можете выполнять блокирующий код, не влияя на очередь (задачи не будут зависать).
- Гораздо лучшая утилизация многоядерных процессоров.
- Меньше соединений с redis.

**`app.module.ts`**

```ts
import { Module } from '@nestjs/common';
import { BullModule } from '@nestjs/bull';
import { join } from 'path';

@Module({
  imports: [
    BullModule.registerQueue({
      name: 'audio',
      processors: [join(__dirname, 'processor.js')],
    }),
  ],
})
export class AppModule {}
```

Обратите внимание: поскольку ваша функция выполняется в форкнутом процессе, внедрение зависимостей (и IoC-контейнер) будут недоступны. Это значит, что функция-процессор должна содержать (или создавать) все экземпляры внешних зависимостей, которые ей нужны.

**`processor.ts`**

```ts
import { Job, DoneCallback } from 'bull';

export default function (job: Job, cb: DoneCallback) {
  console.log(`[${process.pid}] ${JSON.stringify(job.data)}`);
  cb(null, 'It works');
}
```

#### Асинхронная конфигурация

Возможно, вы захотите передавать параметры `bull` асинхронно, а не статически. В этом случае используйте метод `forRootAsync()`, который предлагает несколько способов работы с асинхронной конфигурацией.

Один из подходов — использовать фабричную функцию:

```typescript
BullModule.forRootAsync({
  useFactory: () => ({
    redis: {
      host: 'localhost',
      port: 6379,
    },
  }),
});
```

Наша фабрика ведёт себя как любой другой [асинхронный провайдер](<../03. Fundamentals/02. Asynchronous providers.md>) (например, она может быть `async` и способна внедрять зависимости через `inject`).

```typescript
BullModule.forRootAsync({
  imports: [ConfigModule],
  useFactory: async (configService: ConfigService) => ({
    redis: {
      host: configService.get('QUEUE_HOST'),
      port: configService.get('QUEUE_PORT'),
    },
  }),
  inject: [ConfigService],
});
```

Как вариант, можно использовать синтаксис `useClass`:

```typescript
BullModule.forRootAsync({
  useClass: BullConfigService,
});
```

Конструкция выше создаст экземпляр `BullConfigService` внутри `BullModule` и использует его для получения объекта параметров через вызов `createSharedConfiguration()`. Учтите, что для этого `BullConfigService` должен реализовывать интерфейс `SharedBullConfigurationFactory`, как показано ниже:

```typescript
@Injectable()
class BullConfigService implements SharedBullConfigurationFactory {
  createSharedConfiguration(): BullModuleOptions {
    return {
      redis: {
        host: 'localhost',
        port: 6379,
      },
    };
  }
}
```

Чтобы предотвратить создание `BullConfigService` внутри `BullModule` и использовать провайдер, импортированный из другого модуля, вы можете применить синтаксис `useExisting`.

```typescript
BullModule.forRootAsync({
  imports: [ConfigModule],
  useExisting: ConfigService,
});
```

Эта конструкция работает так же, как `useClass`, с одним важным отличием: `BullModule` просмотрит импортированные модули, чтобы переиспользовать существующий `ConfigService`, вместо создания нового.

Аналогично, если вы хотите передавать параметры очереди асинхронно, используйте метод `registerQueueAsync()` — только не забудьте указать атрибут `name` вне фабричной функции.

```typescript
BullModule.registerQueueAsync({
  name: 'audio',
  useFactory: () => ({
    redis: {
      host: 'localhost',
      port: 6379,
    },
  }),
});
```
