### База данных

Nest не привязан к конкретной базе данных и позволяет легко интегрироваться с любой SQL- или NoSQL-базой. В вашем распоряжении множество вариантов — выбирайте по своим предпочтениям. На самом общем уровне подключение Nest к базе данных сводится к загрузке подходящего драйвера Node.js для этой базы, точно так же, как вы поступили бы в [Express](https://expressjs.com/en/guide/database-integration.html) или Fastify.

Вы также можете напрямую использовать любую универсальную **библиотеку** интеграции с базами данных для Node.js или ORM — например, [MikroORM](https://mikro-orm.io/) (см. [рецепт MikroORM](<../13. Recipes/06. MikroORM.md>)), [Sequelize](https://sequelize.org/) (см. [интеграцию с Sequelize](<02. Database.md#интеграция-с-sequelize>)), [Knex.js](https://knexjs.org/) (см. [руководство по Knex.js](https://dev.to/nestjs/build-a-nestjs-module-for-knex-js-or-other-resource-based-libraries-in-5-minutes-12an)), [TypeORM](https://github.com/typeorm/typeorm) и [Prisma](https://www.github.com/prisma/prisma) (см. [рецепт Prisma](<../13. Recipes/14. Prisma.md>)), чтобы работать на более высоком уровне абстракции.

Для удобства Nest «из коробки» предоставляет тесную интеграцию с TypeORM и Sequelize через пакеты `@nestjs/typeorm` и `@nestjs/sequelize` соответственно — о них пойдёт речь в этой главе, — а также с Mongoose через `@nestjs/mongoose`, которому посвящена [эта глава](<03. Mongo.md>). Эти интеграции добавляют дополнительные возможности, специфичные для NestJS: внедрение моделей и репозиториев, тестируемость и асинхронную конфигурацию, что делает работу с выбранной базой данных ещё проще.

### Интеграция с TypeORM

Для интеграции с SQL- и NoSQL-базами данных Nest предоставляет пакет `@nestjs/typeorm`. [TypeORM](https://github.com/typeorm/typeorm) — наиболее зрелый Object Relational Mapper (ORM) из доступных для TypeScript. Поскольку он написан на TypeScript, он хорошо сочетается с фреймворком Nest.

Чтобы начать работу, сначала установим необходимые зависимости. В этой главе мы будем использовать популярную реляционную СУБД [MySQL](https://www.mysql.com/), однако TypeORM поддерживает множество реляционных баз данных: PostgreSQL, Oracle, Microsoft SQL Server, SQLite и даже NoSQL-базы вроде MongoDB. Описанная в этой главе процедура будет одинаковой для любой базы данных, поддерживаемой TypeORM. Вам потребуется лишь установить соответствующие клиентские библиотеки для выбранной базы.

```bash
$ npm install --save @nestjs/typeorm typeorm mysql2
```

После завершения установки мы можем импортировать `TypeOrmModule` в корневой `AppModule`.

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';

@Module({
  imports: [
    TypeOrmModule.forRoot({
      type: 'mysql',
      host: 'localhost',
      port: 3306,
      username: 'root',
      password: 'root',
      database: 'test',
      entities: [],
      synchronize: true,
    }),
  ],
})
export class AppModule {}
```

> **Предупреждение** Настройку `synchronize: true` не следует использовать в production — иначе вы можете потерять production-данные.

Метод `forRoot()` поддерживает все свойства конфигурации, доступные конструктору `DataSource` из пакета [TypeORM](https://typeorm.io/data-source-options#common-data-source-options). Кроме того, есть несколько дополнительных свойств, описанных ниже.

<table>
  <tr>
    <td><code>retryAttempts</code></td>
    <td>Количество попыток подключения к базе данных (по умолчанию: <code>10</code>)</td>
  </tr>
  <tr>
    <td><code>retryDelay</code></td>
    <td>Задержка между повторными попытками подключения (мс) (по умолчанию: <code>3000</code>)</td>
  </tr>
  <tr>
    <td><code>autoLoadEntities</code></td>
    <td>Если <code>true</code>, сущности загружаются автоматически (по умолчанию: <code>false</code>)</td>
  </tr>
</table>

> **Подсказка** Подробнее об опциях источника данных читайте [здесь](https://typeorm.io/data-source-options).

После этого объекты TypeORM `DataSource` и `EntityManager` станут доступны для внедрения во всём проекте (без необходимости импортировать какие-либо модули), например:

**`app.module.ts`**

```typescript
import { DataSource } from 'typeorm';

@Module({
  imports: [TypeOrmModule.forRoot(), UsersModule],
})
export class AppModule {
  constructor(private dataSource: DataSource) {}
}
```

#### Паттерн Repository

[TypeORM](https://github.com/typeorm/typeorm) поддерживает **паттерн проектирования repository**, поэтому у каждой сущности есть собственный репозиторий. Эти репозитории можно получить из источника данных.

Чтобы продолжить пример, нам нужна хотя бы одна сущность. Опишем сущность `User`.

**`user.entity.ts`**

```typescript
import { Entity, Column, PrimaryGeneratedColumn } from 'typeorm';

@Entity()
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  firstName: string;

  @Column()
  lastName: string;

  @Column({ default: true })
  isActive: boolean;
}
```

> **Подсказка** Подробнее о сущностях читайте в [документации TypeORM](https://typeorm.io/docs/entity/entities/).

Файл сущности `User` лежит в каталоге `users`. Этот каталог содержит все файлы, относящиеся к `UsersModule`. Вы вправе сами решать, где хранить файлы моделей, однако мы рекомендуем создавать их рядом с их **доменом**, в соответствующем каталоге модуля.

Чтобы начать использовать сущность `User`, нужно сообщить о ней TypeORM, добавив её в массив `entities` в опциях метода `forRoot()` модуля (если только вы не используете статический glob-путь):

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { User } from './users/user.entity';

@Module({
  imports: [
    TypeOrmModule.forRoot({
      type: 'mysql',
      host: 'localhost',
      port: 3306,
      username: 'root',
      password: 'root',
      database: 'test',
      entities: [User],
      synchronize: true,
    }),
  ],
})
export class AppModule {}
```

Теперь посмотрим на `UsersModule`:

**`users.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { UsersService } from './users.service';
import { UsersController } from './users.controller';
import { User } from './user.entity';

@Module({
  imports: [TypeOrmModule.forFeature([User])],
  providers: [UsersService],
  controllers: [UsersController],
})
export class UsersModule {}
```

Этот модуль использует метод `forFeature()`, чтобы определить, какие репозитории зарегистрированы в текущей области видимости. После этого мы можем внедрить `UsersRepository` в `UsersService` с помощью декоратора `@InjectRepository()`:

**`users.service.ts`**

```typescript
import { Injectable } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository } from 'typeorm';
import { User } from './user.entity';

@Injectable()
export class UsersService {
  constructor(
    @InjectRepository(User)
    private usersRepository: Repository<User>,
  ) {}

  findAll(): Promise<User[]> {
    return this.usersRepository.find();
  }

  findOne(id: number): Promise<User | null> {
    return this.usersRepository.findOneBy({ id });
  }

  async remove(id: number): Promise<void> {
    await this.usersRepository.delete(id);
  }
}
```

> **Важно** Не забудьте импортировать `UsersModule` в корневой `AppModule`.

Если вы хотите использовать репозиторий за пределами модуля, импортирующего `TypeOrmModule.forFeature`, вам нужно повторно экспортировать сгенерированные им провайдеры.
Сделать это можно, экспортировав модуль целиком, вот так:

**`users.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { User } from './user.entity';

@Module({
  imports: [TypeOrmModule.forFeature([User])],
  exports: [TypeOrmModule]
})
export class UsersModule {}
```

Теперь, если мы импортируем `UsersModule` в `UserHttpModule`, мы сможем использовать `@InjectRepository(User)` в провайдерах последнего.

**`users-http.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { UsersModule } from './users.module';
import { UsersService } from './users.service';
import { UsersController } from './users.controller';

@Module({
  imports: [UsersModule],
  providers: [UsersService],
  controllers: [UsersController]
})
export class UserHttpModule {}
```

#### Связи

Связи — это ассоциации между двумя и более таблицами. Они строятся на общих полях таблиц, обычно с участием первичных и внешних ключей.

Существует три типа связей:

<table>
  <tr>
    <td><code>Один к одному</code></td>
    <td>Каждой строке основной таблицы соответствует ровно одна связанная строка во внешней таблице. Для определения такой связи используйте декоратор <code>@OneToOne()</code>.</td>
  </tr>
  <tr>
    <td><code>Один ко многим / Многие к одному</code></td>
    <td>Каждой строке основной таблицы соответствует одна или несколько связанных строк во внешней таблице. Для определения такой связи используйте декораторы <code>@OneToMany()</code> и <code>@ManyToOne()</code>.</td>
  </tr>
  <tr>
    <td><code>Многие ко многим</code></td>
    <td>Каждой строке основной таблицы соответствует множество связанных строк во внешней таблице, и каждой записи внешней таблицы — множество связанных строк в основной. Для определения такой связи используйте декоратор <code>@ManyToMany()</code>.</td>
  </tr>
</table>

Чтобы определить связи в сущностях, используйте соответствующие **декораторы**. Например, чтобы указать, что у каждого `User` может быть несколько фотографий, используйте декоратор `@OneToMany()`.

**`user.entity.ts`**

```typescript
import { Entity, Column, PrimaryGeneratedColumn, OneToMany } from 'typeorm';
import { Photo } from '../photos/photo.entity';

@Entity()
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  firstName: string;

  @Column()
  lastName: string;

  @Column({ default: true })
  isActive: boolean;

  @OneToMany(type => Photo, photo => photo.user)
  photos: Photo[];
}
```

> **Подсказка** Чтобы узнать больше о связях в TypeORM, загляните в [документацию TypeORM](https://typeorm.io/docs/relations/relations).

#### Автоматическая загрузка сущностей

Вручную добавлять сущности в массив `entities` опций источника данных бывает утомительно. Кроме того, обращение к сущностям из корневого модуля нарушает границы доменов приложения и приводит к утечке деталей реализации в другие его части. Для решения этой проблемы предусмотрен альтернативный подход. Чтобы сущности загружались автоматически, задайте свойству `autoLoadEntities` объекта конфигурации (передаваемого в метод `forRoot()`) значение `true`, как показано ниже:

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';

@Module({
  imports: [
    TypeOrmModule.forRoot({
      ...
      autoLoadEntities: true,
    }),
  ],
})
export class AppModule {}
```

С этой опцией каждая сущность, зарегистрированная через метод `forFeature()`, будет автоматически добавлена в массив `entities` объекта конфигурации.

> **Предупреждение** Обратите внимание: сущности, не зарегистрированные через метод `forFeature()`, а лишь упомянутые в другой сущности (через связь), настройкой `autoLoadEntities` включены не будут.

#### Вынесение определения сущности

Вы можете описать сущность и её колонки прямо в модели с помощью декораторов. Но некоторые предпочитают описывать сущности и их колонки в отдельных файлах, используя [«схемы сущностей»](https://typeorm.io/docs/entity/separating-entity-definition).

```typescript
import { EntitySchema } from 'typeorm';
import { User } from './user.entity';

export const UserSchema = new EntitySchema<User>({
  name: 'User',
  target: User,
  columns: {
    id: {
      type: Number,
      primary: true,
      generated: true,
    },
    firstName: {
      type: String,
    },
    lastName: {
      type: String,
    },
    isActive: {
      type: Boolean,
      default: true,
    },
  },
  relations: {
    photos: {
      type: 'one-to-many',
      target: 'Photo', // имя PhotoSchema
    },
  },
});
```

> **Предупреждение** Если вы задаёте опцию `target`, значение опции `name` должно совпадать с именем целевого класса.
> Если `target` не указан, вы можете использовать любое имя.

Nest позволяет использовать экземпляр `EntitySchema` везде, где ожидается `Entity`, например:

```typescript
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { UserSchema } from './user.schema';
import { UsersController } from './users.controller';
import { UsersService } from './users.service';

@Module({
  imports: [TypeOrmModule.forFeature([UserSchema])],
  providers: [UsersService],
  controllers: [UsersController],
})
export class UsersModule {}
```

#### Транзакции в TypeORM

Транзакция базы данных обозначает единицу работы, выполняемую системой управления базами данных над базой и обрабатываемую согласованно и надёжно, независимо от других транзакций. Как правило, транзакция представляет любое изменение в базе данных ([подробнее](https://en.wikipedia.org/wiki/Database_transaction)).

Существует множество стратегий работы с [транзакциями TypeORM](https://typeorm.io/docs/advanced-topics/transactions/). Мы рекомендуем использовать класс `QueryRunner`, поскольку он даёт полный контроль над транзакцией.

Сначала нужно обычным образом внедрить объект `DataSource` в класс:

```typescript
@Injectable()
export class UsersService {
  constructor(private dataSource: DataSource) {}
}
```

> **Подсказка** Класс `DataSource` импортируется из пакета `typeorm`.

Теперь мы можем использовать этот объект для создания транзакции.

```typescript
async createMany(users: User[]) {
  const queryRunner = this.dataSource.createQueryRunner();

  await queryRunner.connect();
  await queryRunner.startTransaction();
  try {
    await queryRunner.manager.save(users[0]);
    await queryRunner.manager.save(users[1]);

    await queryRunner.commitTransaction();
  } catch (err) {
    // раз возникли ошибки, откатим внесённые изменения
    await queryRunner.rollbackTransaction();
  } finally {
    // созданный вручную queryRunner необходимо освободить
    await queryRunner.release();
  }
}
```

> **Подсказка** Обратите внимание: `dataSource` используется только для создания `QueryRunner`. Однако для тестирования такого класса пришлось бы мокать весь объект `DataSource` (у которого немало методов). Поэтому мы рекомендуем использовать вспомогательный класс-фабрику (например, `QueryRunnerFactory`) и описать интерфейс с ограниченным набором методов, необходимых для работы с транзакциями. Такой приём делает создание моков совсем простым.

Как вариант, можно применить подход с колбэком — метод `transaction` объекта `DataSource` ([подробнее](https://typeorm.io/docs/advanced-topics/transactions/#creating-and-using-transactions)).

```typescript
async createMany(users: User[]) {
  await this.dataSource.transaction(async manager => {
    await manager.save(users[0]);
    await manager.save(users[1]);
  });
}
```

#### Подписчики

С помощью [подписчиков](https://typeorm.io/docs/advanced-topics/listeners-and-subscribers#what-is-a-subscriber) TypeORM вы можете отслеживать определённые события сущностей.

```typescript
import {
  DataSource,
  EntitySubscriberInterface,
  EventSubscriber,
  InsertEvent,
} from 'typeorm';
import { User } from './user.entity';

@EventSubscriber()
export class UserSubscriber implements EntitySubscriberInterface<User> {
  constructor(dataSource: DataSource) {
    dataSource.subscribers.push(this);
  }

  listenTo() {
    return User;
  }

  beforeInsert(event: InsertEvent<User>) {
    console.log(`BEFORE USER INSERTED: `, event.entity);
  }
}
```

> **Предупреждение** Подписчики на события не могут иметь [область видимости запроса](<../03. Fundamentals/04. Injection scopes.md>).

Теперь добавьте класс `UserSubscriber` в массив `providers`:

```typescript
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { User } from './user.entity';
import { UsersController } from './users.controller';
import { UsersService } from './users.service';
import { UserSubscriber } from './user.subscriber';

@Module({
  imports: [TypeOrmModule.forFeature([User])],
  providers: [UsersService, UserSubscriber],
  controllers: [UsersController],
})
export class UsersModule {}
```

#### Миграции

[Миграции](https://typeorm.io/docs/advanced-topics/migrations/) позволяют пошагово обновлять схему базы данных, поддерживая её в соответствии с моделью данных приложения и сохраняя при этом уже имеющиеся данные. Для создания, выполнения и отката миграций TypeORM предоставляет отдельный [CLI](https://typeorm.io/docs/advanced-topics/migrations/#creating-a-new-migration).

Классы миграций существуют отдельно от исходного кода приложения Nest. Их жизненным циклом управляет CLI TypeORM. Поэтому в миграциях нельзя воспользоваться внедрением зависимостей и другими специфичными для Nest возможностями. Подробнее о миграциях — в руководстве из [документации TypeORM](https://typeorm.io/docs/advanced-topics/migrations/).

#### Несколько баз данных

Некоторым проектам требуется несколько подключений к базам данных. Этого тоже можно добиться с помощью данного модуля. Чтобы работать с несколькими подключениями, сначала создайте их. В этом случае именование источников данных становится **обязательным**.

Предположим, у вас есть сущность `Album`, хранящаяся в отдельной базе данных.

```typescript
const defaultOptions = {
  type: 'postgres',
  port: 5432,
  username: 'user',
  password: 'password',
  database: 'db',
  synchronize: true,
};

@Module({
  imports: [
    TypeOrmModule.forRoot({
      ...defaultOptions,
      host: 'user_db_host',
      entities: [User],
    }),
    TypeOrmModule.forRoot({
      ...defaultOptions,
      name: 'albumsConnection',
      host: 'album_db_host',
      entities: [Album],
    }),
  ],
})
export class AppModule {}
```

> **Важно** Если вы не зададите `name` для источника данных, ему будет присвоено имя `default`. Учтите, что не должно быть нескольких подключений без имени или с одинаковым именем — иначе они будут перезаписывать друг друга.

> **Важно** Если вы используете `TypeOrmModule.forRootAsync`, имя источника данных нужно задать **также** и вне `useFactory`. Например:
>
> ```typescript
> TypeOrmModule.forRootAsync({
>   name: 'albumsConnection',
>   useFactory: ...,
>   inject: ...,
> }),
> ```
>
> Подробности см. в [этом issue](https://github.com/nestjs/typeorm/issues/86).

Теперь сущности `User` и `Album` зарегистрированы каждая со своим источником данных. При такой схеме нужно указывать методу `TypeOrmModule.forFeature()` и декоратору `@InjectRepository()`, какой источник данных использовать. Если имя источника не передано, используется источник `default`.

```typescript
@Module({
  imports: [
    TypeOrmModule.forFeature([User]),
    TypeOrmModule.forFeature([Album], 'albumsConnection'),
  ],
})
export class AppModule {}
```

Вы также можете внедрить `DataSource` или `EntityManager` для конкретного источника данных:

```typescript
@Injectable()
export class AlbumsService {
  constructor(
    @InjectDataSource('albumsConnection')
    private dataSource: DataSource,
    @InjectEntityManager('albumsConnection')
    private entityManager: EntityManager,
  ) {}
}
```

Кроме того, любой `DataSource` можно внедрить в провайдеры:

```typescript
@Module({
  providers: [
    {
      provide: AlbumsService,
      useFactory: (albumsConnection: DataSource) => {
        return new AlbumsService(albumsConnection);
      },
      inject: [getDataSourceToken('albumsConnection')],
    },
  ],
})
export class AlbumsModule {}
```

#### Тестирование

При модульном тестировании приложения обычно хочется обойтись без подключения к базе данных, чтобы наборы тестов оставались независимыми, а выполнялись как можно быстрее. Но наши классы могут зависеть от репозиториев, получаемых из экземпляра источника данных (подключения). Как быть? Решение — создать mock-репозитории. Для этого мы настраиваем [пользовательские провайдеры](<../03. Fundamentals/01. Custom providers.md>). Каждый зарегистрированный репозиторий автоматически представлен токеном `<EntityName>Repository`, где `EntityName` — имя класса вашей сущности.

Пакет `@nestjs/typeorm` предоставляет функцию `getRepositoryToken()`, которая возвращает готовый токен для заданной сущности.

```typescript
@Module({
  providers: [
    UsersService,
    {
      provide: getRepositoryToken(User),
      useValue: mockRepository,
    },
  ],
})
export class UsersModule {}
```

Теперь в качестве `UsersRepository` будет использоваться подменный `mockRepository`. Всякий раз, когда какой-либо класс запросит `UsersRepository` через декоратор `@InjectRepository()`, Nest подставит зарегистрированный объект `mockRepository`.

#### Асинхронная конфигурация

Возможно, вы захотите передавать опции модуля репозитория асинхронно, а не статически. В этом случае используйте метод `forRootAsync()`, который предлагает несколько способов работы с асинхронной конфигурацией.

Один из подходов — использовать фабричную функцию:

```typescript
TypeOrmModule.forRootAsync({
  useFactory: () => ({
    type: 'mysql',
    host: 'localhost',
    port: 3306,
    username: 'root',
    password: 'root',
    database: 'test',
    entities: [],
    synchronize: true,
  }),
});
```

Наша фабрика ведёт себя как любой другой [асинхронный провайдер](<../03. Fundamentals/02. Asynchronous providers.md>) (например, она может быть `async` и способна внедрять зависимости через `inject`).

```typescript
TypeOrmModule.forRootAsync({
  imports: [ConfigModule],
  useFactory: (configService: ConfigService) => ({
    type: 'mysql',
    host: configService.get('HOST'),
    port: +configService.get('PORT'),
    username: configService.get('USERNAME'),
    password: configService.get('PASSWORD'),
    database: configService.get('DATABASE'),
    entities: [],
    synchronize: true,
  }),
  inject: [ConfigService],
});
```

Как вариант, можно воспользоваться синтаксисом `useClass`:

```typescript
TypeOrmModule.forRootAsync({
  useClass: TypeOrmConfigService,
});
```

Приведённая конструкция создаст экземпляр `TypeOrmConfigService` внутри `TypeOrmModule` и будет использовать его для получения объекта опций через вызов `createTypeOrmOptions()`. Учтите, что при этом `TypeOrmConfigService` должен реализовывать интерфейс `TypeOrmOptionsFactory`, как показано ниже:

```typescript
@Injectable()
export class TypeOrmConfigService implements TypeOrmOptionsFactory {
  createTypeOrmOptions(): TypeOrmModuleOptions {
    return {
      type: 'mysql',
      host: 'localhost',
      port: 3306,
      username: 'root',
      password: 'root',
      database: 'test',
      entities: [],
      synchronize: true,
    };
  }
}
```

Чтобы не создавать `TypeOrmConfigService` внутри `TypeOrmModule`, а использовать провайдер, импортированный из другого модуля, можно применить синтаксис `useExisting`.

```typescript
TypeOrmModule.forRootAsync({
  imports: [ConfigModule],
  useExisting: ConfigService,
});
```

Эта конструкция работает так же, как `useClass`, с одним принципиальным отличием: `TypeOrmModule` просмотрит импортированные модули, чтобы переиспользовать существующий `ConfigService`, вместо создания нового экземпляра.

> **Подсказка** Убедитесь, что свойство `name` определено на том же уровне, что и свойства `useFactory`, `useClass` или `useValue`. Это позволит Nest корректно зарегистрировать источник данных под соответствующим токеном внедрения.

#### Пользовательская фабрика DataSource

Вместе с асинхронной конфигурацией через `useFactory`, `useClass` или `useExisting` вы можете дополнительно указать функцию `dataSourceFactory`, которая позволит предоставить собственный источник данных TypeORM вместо того, чтобы его создавал `TypeOrmModule`.

`dataSourceFactory` получает `DataSourceOptions` TypeORM, сформированные при асинхронной конфигурации через `useFactory`, `useClass` или `useExisting`, и возвращает `Promise`, разрешающийся объектом `DataSource` TypeORM.

```typescript
TypeOrmModule.forRootAsync({
  imports: [ConfigModule],
  inject: [ConfigService],
  // Используйте useFactory, useClass или useExisting
  // для настройки DataSourceOptions.
  useFactory: (configService: ConfigService) => ({
    type: 'mysql',
    host: configService.get('HOST'),
    port: +configService.get('PORT'),
    username: configService.get('USERNAME'),
    password: configService.get('PASSWORD'),
    database: configService.get('DATABASE'),
    entities: [],
    synchronize: true,
  }),
  // dataSource получает сформированные DataSourceOptions
  // и возвращает Promise<DataSource>.
  dataSourceFactory: async (options) => {
    const dataSource = await new DataSource(options).initialize();
    return dataSource;
  },
});
```

> **Подсказка** Класс `DataSource` импортируется из пакета `typeorm`.

Рабочий пример доступен [здесь](https://github.com/nestjs/nest/tree/master/sample/05-sql-typeorm).

### Интеграция с Sequelize

Альтернатива TypeORM — ORM [Sequelize](https://sequelize.org/) вместе с пакетом `@nestjs/sequelize`. Дополнительно мы задействуем пакет [sequelize-typescript](https://github.com/RobinBuschmann/sequelize-typescript), который предоставляет набор дополнительных декораторов для декларативного описания сущностей.

Чтобы начать работу, сначала установим необходимые зависимости. В этой главе мы будем использовать популярную реляционную СУБД [MySQL](https://www.mysql.com/), однако Sequelize поддерживает множество реляционных баз данных: PostgreSQL, MySQL, Microsoft SQL Server, SQLite и MariaDB. Описанная в этой главе процедура будет одинаковой для любой базы данных, поддерживаемой Sequelize. Вам потребуется лишь установить соответствующие клиентские библиотеки для выбранной базы.

```bash
$ npm install --save @nestjs/sequelize sequelize sequelize-typescript mysql2
$ npm install --save-dev @types/sequelize
```

После завершения установки мы можем импортировать `SequelizeModule` в корневой `AppModule`.

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { SequelizeModule } from '@nestjs/sequelize';

@Module({
  imports: [
    SequelizeModule.forRoot({
      dialect: 'mysql',
      host: 'localhost',
      port: 3306,
      username: 'root',
      password: 'root',
      database: 'test',
      models: [],
    }),
  ],
})
export class AppModule {}
```

Метод `forRoot()` поддерживает все свойства конфигурации, доступные конструктору Sequelize ([подробнее](https://sequelize.org/docs/v6/getting-started/#connecting-to-a-database)). Кроме того, есть несколько дополнительных свойств, описанных ниже.

<table>
  <tr>
    <td><code>retryAttempts</code></td>
    <td>Количество попыток подключения к базе данных (по умолчанию: <code>10</code>)</td>
  </tr>
  <tr>
    <td><code>retryDelay</code></td>
    <td>Задержка между повторными попытками подключения (мс) (по умолчанию: <code>3000</code>)</td>
  </tr>
  <tr>
    <td><code>autoLoadModels</code></td>
    <td>Если <code>true</code>, модели загружаются автоматически (по умолчанию: <code>false</code>)</td>
  </tr>
  <tr>
    <td><code>keepConnectionAlive</code></td>
    <td>Если <code>true</code>, подключение не будет закрыто при завершении работы приложения (по умолчанию: <code>false</code>)</td>
  </tr>
  <tr>
    <td><code>synchronize</code></td>
    <td>Если <code>true</code>, автоматически загруженные модели будут синхронизированы (по умолчанию: <code>true</code>)</td>
  </tr>
</table>

После этого объект `Sequelize` станет доступен для внедрения во всём проекте (без необходимости импортировать какие-либо модули), например:

**`app.service.ts`**

```typescript
import { Injectable } from '@nestjs/common';
import { Sequelize } from 'sequelize-typescript';

@Injectable()
export class AppService {
  constructor(private sequelize: Sequelize) {}
}
```

#### Модели

Sequelize реализует паттерн Active Record. При таком подходе для взаимодействия с базой данных вы используете классы моделей напрямую. Чтобы продолжить пример, нам нужна хотя бы одна модель. Опишем модель `User`.

**`user.model.ts`**

```typescript
import { Column, Model, Table } from 'sequelize-typescript';

@Table
export class User extends Model {
  @Column
  firstName: string;

  @Column
  lastName: string;

  @Column({ defaultValue: true })
  isActive: boolean;
}
```

> **Подсказка** Подробнее о доступных декораторах читайте [здесь](https://github.com/RobinBuschmann/sequelize-typescript#column).

Файл модели `User` лежит в каталоге `users`. Этот каталог содержит все файлы, относящиеся к `UsersModule`. Вы вправе сами решать, где хранить файлы моделей, однако мы рекомендуем создавать их рядом с их **доменом**, в соответствующем каталоге модуля.

Чтобы начать использовать модель `User`, нужно сообщить о ней Sequelize, добавив её в массив `models` в опциях метода `forRoot()` модуля:

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { SequelizeModule } from '@nestjs/sequelize';
import { User } from './users/user.model';

@Module({
  imports: [
    SequelizeModule.forRoot({
      dialect: 'mysql',
      host: 'localhost',
      port: 3306,
      username: 'root',
      password: 'root',
      database: 'test',
      models: [User],
    }),
  ],
})
export class AppModule {}
```

Теперь посмотрим на `UsersModule`:

**`users.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { SequelizeModule } from '@nestjs/sequelize';
import { User } from './user.model';
import { UsersController } from './users.controller';
import { UsersService } from './users.service';

@Module({
  imports: [SequelizeModule.forFeature([User])],
  providers: [UsersService],
  controllers: [UsersController],
})
export class UsersModule {}
```

Этот модуль использует метод `forFeature()`, чтобы определить, какие модели зарегистрированы в текущей области видимости. После этого мы можем внедрить `UserModel` в `UsersService` с помощью декоратора `@InjectModel()`:

**`users.service.ts`**

```typescript
import { Injectable } from '@nestjs/common';
import { InjectModel } from '@nestjs/sequelize';
import { User } from './user.model';

@Injectable()
export class UsersService {
  constructor(
    @InjectModel(User)
    private userModel: typeof User,
  ) {}

  async findAll(): Promise<User[]> {
    return this.userModel.findAll();
  }

  findOne(id: string): Promise<User> {
    return this.userModel.findOne({
      where: {
        id,
      },
    });
  }

  async remove(id: string): Promise<void> {
    const user = await this.findOne(id);
    await user.destroy();
  }
}
```

> **Важно** Не забудьте импортировать `UsersModule` в корневой `AppModule`.

Если вы хотите использовать модель за пределами модуля, импортирующего `SequelizeModule.forFeature`, вам нужно повторно экспортировать сгенерированные им провайдеры.
Сделать это можно, экспортировав модуль целиком, вот так:

**`users.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { SequelizeModule } from '@nestjs/sequelize';
import { User } from './user.entity';

@Module({
  imports: [SequelizeModule.forFeature([User])],
  exports: [SequelizeModule]
})
export class UsersModule {}
```

Теперь, если мы импортируем `UsersModule` в `UserHttpModule`, мы сможем использовать `@InjectModel(User)` в провайдерах последнего.

**`users-http.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { UsersModule } from './users.module';
import { UsersService } from './users.service';
import { UsersController } from './users.controller';

@Module({
  imports: [UsersModule],
  providers: [UsersService],
  controllers: [UsersController]
})
export class UserHttpModule {}
```

#### Связи

Связи — это ассоциации между двумя и более таблицами. Они строятся на общих полях таблиц, обычно с участием первичных и внешних ключей.

Существует три типа связей:

<table>
  <tr>
    <td><code>Один к одному</code></td>
    <td>Каждой строке основной таблицы соответствует ровно одна связанная строка во внешней таблице</td>
  </tr>
  <tr>
    <td><code>Один ко многим / Многие к одному</code></td>
    <td>Каждой строке основной таблицы соответствует одна или несколько связанных строк во внешней таблице</td>
  </tr>
  <tr>
    <td><code>Многие ко многим</code></td>
    <td>Каждой строке основной таблицы соответствует множество связанных строк во внешней таблице, и каждой записи внешней таблицы — множество связанных строк в основной</td>
  </tr>
</table>

Чтобы определить связи в моделях, используйте соответствующие **декораторы**. Например, чтобы указать, что у каждого `User` может быть несколько фотографий, используйте декоратор `@HasMany()`.

**`user.model.ts`**

```typescript
import { Column, Model, Table, HasMany } from 'sequelize-typescript';
import { Photo } from '../photos/photo.model';

@Table
export class User extends Model {
  @Column
  firstName: string;

  @Column
  lastName: string;

  @Column({ defaultValue: true })
  isActive: boolean;

  @HasMany(() => Photo)
  photos: Photo[];
}
```

> **Подсказка** Чтобы узнать больше об ассоциациях в Sequelize, прочитайте [эту](https://github.com/RobinBuschmann/sequelize-typescript#model-association) главу.

#### Автоматическая загрузка моделей

Вручную добавлять модели в массив `models` опций подключения бывает утомительно. Кроме того, обращение к моделям из корневого модуля нарушает границы доменов приложения и приводит к утечке деталей реализации в другие его части. Чтобы решить эту проблему, включите автоматическую загрузку моделей, задав свойствам `autoLoadModels` и `synchronize` объекта конфигурации (передаваемого в метод `forRoot()`) значение `true`, как показано ниже:

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { SequelizeModule } from '@nestjs/sequelize';

@Module({
  imports: [
    SequelizeModule.forRoot({
      ...
      autoLoadModels: true,
      synchronize: true,
    }),
  ],
})
export class AppModule {}
```

С этой опцией каждая модель, зарегистрированная через метод `forFeature()`, будет автоматически добавлена в массив `models` объекта конфигурации.

> **Предупреждение** Обратите внимание: модели, не зарегистрированные через метод `forFeature()`, а лишь упомянутые в другой модели (через ассоциацию), включены не будут.

#### Транзакции в Sequelize

Транзакция базы данных обозначает единицу работы, выполняемую системой управления базами данных над базой и обрабатываемую согласованно и надёжно, независимо от других транзакций. Как правило, транзакция представляет любое изменение в базе данных ([подробнее](https://en.wikipedia.org/wiki/Database_transaction)).

Существует множество стратегий работы с [транзакциями Sequelize](https://sequelize.org/docs/v6/other-topics/transactions/). Ниже приведён пример реализации управляемой транзакции (с автоматическим колбэком).

Сначала нужно обычным образом внедрить объект `Sequelize` в класс:

```typescript
@Injectable()
export class UsersService {
  constructor(private sequelize: Sequelize) {}
}
```

> **Подсказка** Класс `Sequelize` импортируется из пакета `sequelize-typescript`.

Теперь мы можем использовать этот объект для создания транзакции.

```typescript
async createMany() {
  try {
    await this.sequelize.transaction(async t => {
      const transactionHost = { transaction: t };

      await this.userModel.create(
          { firstName: 'Abraham', lastName: 'Lincoln' },
          transactionHost,
      );
      await this.userModel.create(
          { firstName: 'John', lastName: 'Boothe' },
          transactionHost,
      );
    });
  } catch (err) {
    // Транзакция была откачена
    // err — это то, чем был отклонён промис, возвращённый в колбэк транзакции
  }
}
```

> **Подсказка** Обратите внимание: экземпляр `Sequelize` используется только для запуска транзакции. Однако для тестирования такого класса пришлось бы мокать весь объект `Sequelize` (у которого немало методов). Поэтому мы рекомендуем использовать вспомогательный класс-фабрику (например, `TransactionRunner`) и описать интерфейс с ограниченным набором методов, необходимых для работы с транзакциями. Такой приём делает создание моков совсем простым.

#### Миграции

[Миграции](https://sequelize.org/docs/v6/other-topics/migrations/) позволяют пошагово обновлять схему базы данных, поддерживая её в соответствии с моделью данных приложения и сохраняя при этом уже имеющиеся данные. Для создания, выполнения и отката миграций Sequelize предоставляет отдельный [CLI](https://sequelize.org/docs/v6/other-topics/migrations/#installing-the-cli).

Классы миграций существуют отдельно от исходного кода приложения Nest. Их жизненным циклом управляет CLI Sequelize. Поэтому в миграциях нельзя воспользоваться внедрением зависимостей и другими специфичными для Nest возможностями. Подробнее о миграциях — в руководстве из [документации Sequelize](https://sequelize.org/docs/v6/other-topics/migrations/#installing-the-cli).

#### Несколько баз данных

Некоторым проектам требуется несколько подключений к базам данных. Этого тоже можно добиться с помощью данного модуля. Чтобы работать с несколькими подключениями, сначала создайте их. В этом случае именование подключений становится **обязательным**.

Предположим, у вас есть сущность `Album`, хранящаяся в отдельной базе данных.

```typescript
const defaultOptions = {
  dialect: 'postgres',
  port: 5432,
  username: 'user',
  password: 'password',
  database: 'db',
  synchronize: true,
};

@Module({
  imports: [
    SequelizeModule.forRoot({
      ...defaultOptions,
      host: 'user_db_host',
      models: [User],
    }),
    SequelizeModule.forRoot({
      ...defaultOptions,
      name: 'albumsConnection',
      host: 'album_db_host',
      models: [Album],
    }),
  ],
})
export class AppModule {}
```

> **Важно** Если вы не зададите `name` для подключения, ему будет присвоено имя `default`. Учтите, что не должно быть нескольких подключений без имени или с одинаковым именем — иначе они будут перезаписывать друг друга.

Теперь модели `User` и `Album` зарегистрированы каждая со своим подключением. При такой схеме нужно указывать методу `SequelizeModule.forFeature()` и декоратору `@InjectModel()`, какое подключение использовать. Если имя подключения не передано, используется подключение `default`.

```typescript
@Module({
  imports: [
    SequelizeModule.forFeature([User]),
    SequelizeModule.forFeature([Album], 'albumsConnection'),
  ],
})
export class AppModule {}
```

Вы также можете внедрить экземпляр `Sequelize` для конкретного подключения:

```typescript
@Injectable()
export class AlbumsService {
  constructor(
    @InjectConnection('albumsConnection')
    private sequelize: Sequelize,
  ) {}
}
```

Кроме того, любой экземпляр `Sequelize` можно внедрить в провайдеры:

```typescript
@Module({
  providers: [
    {
      provide: AlbumsService,
      useFactory: (albumsSequelize: Sequelize) => {
        return new AlbumsService(albumsSequelize);
      },
      inject: [getDataSourceToken('albumsConnection')],
    },
  ],
})
export class AlbumsModule {}
```

#### Тестирование

При модульном тестировании приложения обычно хочется обойтись без подключения к базе данных, чтобы наборы тестов оставались независимыми, а выполнялись как можно быстрее. Но наши классы могут зависеть от моделей, получаемых из экземпляра подключения. Как быть? Решение — создать mock-модели. Для этого мы настраиваем [пользовательские провайдеры](<../03. Fundamentals/01. Custom providers.md>). Каждая зарегистрированная модель автоматически представлена токеном `<ModelName>Model`, где `ModelName` — имя класса вашей модели.

Пакет `@nestjs/sequelize` предоставляет функцию `getModelToken()`, которая возвращает готовый токен для заданной модели.

```typescript
@Module({
  providers: [
    UsersService,
    {
      provide: getModelToken(User),
      useValue: mockModel,
    },
  ],
})
export class UsersModule {}
```

Теперь в качестве `UserModel` будет использоваться подменная `mockModel`. Всякий раз, когда какой-либо класс запросит `UserModel` через декоратор `@InjectModel()`, Nest подставит зарегистрированный объект `mockModel`.

#### Асинхронная конфигурация

Возможно, вы захотите передавать опции `SequelizeModule` асинхронно, а не статически. В этом случае используйте метод `forRootAsync()`, который предлагает несколько способов работы с асинхронной конфигурацией.

Один из подходов — использовать фабричную функцию:

```typescript
SequelizeModule.forRootAsync({
  useFactory: () => ({
    dialect: 'mysql',
    host: 'localhost',
    port: 3306,
    username: 'root',
    password: 'root',
    database: 'test',
    models: [],
  }),
});
```

Наша фабрика ведёт себя как любой другой [асинхронный провайдер](<../03. Fundamentals/02. Asynchronous providers.md>) (например, она может быть `async` и способна внедрять зависимости через `inject`).

```typescript
SequelizeModule.forRootAsync({
  imports: [ConfigModule],
  useFactory: (configService: ConfigService) => ({
    dialect: 'mysql',
    host: configService.get('HOST'),
    port: +configService.get('PORT'),
    username: configService.get('USERNAME'),
    password: configService.get('PASSWORD'),
    database: configService.get('DATABASE'),
    models: [],
  }),
  inject: [ConfigService],
});
```

Как вариант, можно воспользоваться синтаксисом `useClass`:

```typescript
SequelizeModule.forRootAsync({
  useClass: SequelizeConfigService,
});
```

Приведённая конструкция создаст экземпляр `SequelizeConfigService` внутри `SequelizeModule` и будет использовать его для получения объекта опций через вызов `createSequelizeOptions()`. Учтите, что при этом `SequelizeConfigService` должен реализовывать интерфейс `SequelizeOptionsFactory`, как показано ниже:

```typescript
@Injectable()
class SequelizeConfigService implements SequelizeOptionsFactory {
  createSequelizeOptions(): SequelizeModuleOptions {
    return {
      dialect: 'mysql',
      host: 'localhost',
      port: 3306,
      username: 'root',
      password: 'root',
      database: 'test',
      models: [],
    };
  }
}
```

Чтобы не создавать `SequelizeConfigService` внутри `SequelizeModule`, а использовать провайдер, импортированный из другого модуля, можно применить синтаксис `useExisting`.

```typescript
SequelizeModule.forRootAsync({
  imports: [ConfigModule],
  useExisting: ConfigService,
});
```

Эта конструкция работает так же, как `useClass`, с одним принципиальным отличием: `SequelizeModule` просмотрит импортированные модули, чтобы переиспользовать существующий `ConfigService`, вместо создания нового экземпляра.
