### Загрузка файлов

Для обработки загрузки файлов Nest предоставляет встроенный модуль на основе middleware-пакета [multer](https://github.com/expressjs/multer) для Express. Multer обрабатывает данные, отправленные в формате `multipart/form-data`, который в основном используется для загрузки файлов через HTTP-запрос `POST`. Модуль полностью настраиваемый, и вы можете подстроить его поведение под требования своего приложения.

> **Предупреждение** Multer не может обрабатывать данные, которые не соответствуют поддерживаемому формату multipart (`multipart/form-data`). Кроме того, учтите, что этот пакет несовместим с `FastifyAdapter`.

Для лучшей типобезопасности установим пакет с типами Multer:

```shell
$ npm i -D @types/multer
```

После установки этого пакета мы можем использовать тип `Express.Multer.File` (импортировать его можно так: `import { Express } from 'express'`).

#### Простой пример

Чтобы загрузить один файл, достаточно привязать interceptor `FileInterceptor()` к обработчику маршрута и извлечь `file` из `request` с помощью декоратора `@UploadedFile()`.

```typescript
@Post('upload')
@UseInterceptors(FileInterceptor('file'))
uploadFile(@UploadedFile() file: Express.Multer.File) {
  console.log(file);
}
```

> **Подсказка** Декоратор `FileInterceptor()` экспортируется из пакета `@nestjs/platform-express`. Декоратор `@UploadedFile()` экспортируется из `@nestjs/common`.

Декоратор `FileInterceptor()` принимает два аргумента:

- `fieldName`: строка с именем поля HTML-формы, содержащего файл
- `options`: необязательный объект типа `MulterOptions`. Это тот же объект, который принимает конструктор multer (подробнее [здесь](https://github.com/expressjs/multer#multeropts)).

> **Предупреждение** `FileInterceptor()` может быть несовместим со сторонними облачными провайдерами вроде Google Firebase и других.

#### Валидация файлов

Нередко бывает полезно проверить метаданные входящего файла — например, размер или MIME-тип. Для этого вы можете создать собственный [Pipe](<../02. Overview/07. Pipes.md>) и привязать его к параметру, помеченному декоратором `UploadedFile`. Пример ниже показывает, как можно реализовать простой pipe для проверки размера файла:

```typescript
import { PipeTransform, Injectable, ArgumentMetadata } from '@nestjs/common';

@Injectable()
export class FileSizeValidationPipe implements PipeTransform {
  transform(value: any, metadata: ArgumentMetadata) {
    // "value" — это объект, содержащий атрибуты и метаданные файла
    const oneKb = 1000;
    return value.size < oneKb;
  }
}
```

Использовать его совместно с `FileInterceptor` можно так:

```typescript
@Post('file')
@UseInterceptors(FileInterceptor('file'))
uploadFileAndValidate(@UploadedFile(
  new FileSizeValidationPipe(),
  // здесь можно добавить другие pipes
) file: Express.Multer.File, ) {
  return file;
}
```

Nest предоставляет встроенный pipe для типичных сценариев, который упрощает и стандартизирует добавление новых. Он называется `ParseFilePipe`, и использовать его можно так:

```typescript
@Post('file')
uploadFileAndPassValidation(
  @Body() body: SampleDto,
  @UploadedFile(
    new ParseFilePipe({
      validators: [
        // ... Set of file validator instances here
      ]
    })
  )
  file: Express.Multer.File,
) {
  return {
    body,
    file: file.buffer.toString(),
  };
}
```

Как видите, нужно указать массив валидаторов файлов, которые выполнит `ParseFilePipe`. Интерфейс валидатора мы обсудим ниже, но стоит упомянуть, что у этого pipe есть ещё два **необязательных** параметра:

<table>
  <tr>
    <td><code>errorHttpStatusCode</code></td>
    <td>HTTP-код состояния, возвращаемый в случае, если <b>любой</b> из валидаторов не прошёл. По умолчанию <code>400</code> (BAD REQUEST)</td>
  </tr>
  <tr>
    <td><code>exceptionFactory</code></td>
    <td>Фабрика, которая получает сообщение об ошибке и возвращает ошибку.</td>
  </tr>
</table>

Теперь вернёмся к интерфейсу `FileValidator`. Чтобы использовать валидаторы с этим pipe, нужно либо взять встроенные реализации, либо написать собственный `FileValidator`. Смотрите пример ниже:

```typescript
export abstract class FileValidator<TValidationOptions = Record<string, any>> {
  constructor(protected readonly validationOptions: TValidationOptions) {}

  /**
   * Указывает, следует ли считать этот файл корректным, согласно параметрам, переданным в конструктор.
   * @param file файл из объекта запроса
   */
  abstract isValid(file?: any): boolean | Promise<boolean>;

  /**
   * Формирует сообщение об ошибке на случай неудачной проверки.
   * @param file файл из объекта запроса
   */
  abstract buildErrorMessage(file: any): string;
}
```

> **Подсказка** Интерфейс `FileValidator` поддерживает асинхронную валидацию через функцию `isValid`. Чтобы воспользоваться типобезопасностью, вы можете также типизировать параметр `file` как `Express.Multer.File`, если используете express (драйвер по умолчанию).

`FileValidator` — это обычный класс, который имеет доступ к объекту файла и проверяет его согласно параметрам, переданным клиентом. В Nest есть две встроенные реализации `FileValidator`, которые вы можете использовать в своём проекте:

- `MaxFileSizeValidator` — проверяет, что размер файла меньше указанного значения (в `bytes`)
- `FileTypeValidator` — проверяет, что MIME-тип файла соответствует заданной строке или регулярному выражению. По умолчанию определяет MIME-тип по содержимому файла — его [magic number](https://www.ibm.com/support/pages/what-magic-number)

Чтобы понять, как их использовать вместе с упомянутым `ParseFilePipe`, возьмём изменённый фрагмент последнего примера:

```typescript
@UploadedFile(
  new ParseFilePipe({
    validators: [
      new MaxFileSizeValidator({ maxSize: 1000 }),
      new FileTypeValidator({ fileType: 'image/jpeg' }),
    ],
  }),
)
file: Express.Multer.File,
```

> **Подсказка** Если валидаторов становится слишком много или их параметры загромождают файл, вы можете вынести этот массив в отдельный файл и импортировать сюда как именованную константу, например `fileValidators`.

Наконец, вы можете воспользоваться специальным классом `ParseFilePipeBuilder`, который позволяет собирать и конструировать валидаторы. Применяя его так, как показано ниже, вы избавитесь от ручного создания каждого валидатора и сможете передавать их параметры напрямую:

```typescript
@UploadedFile(
  new ParseFilePipeBuilder()
    .addFileTypeValidator({
      fileType: 'jpeg',
    })
    .addMaxSizeValidator({
      maxSize: 1000
    })
    .build({
      errorHttpStatusCode: HttpStatus.UNPROCESSABLE_ENTITY
    }),
)
file: Express.Multer.File,
```

> **Подсказка** По умолчанию наличие файла обязательно, но вы можете сделать его необязательным, добавив параметр `fileIsRequired: false` в параметры функции `build` (на том же уровне, что и `errorHttpStatusCode`).

#### Массив файлов

Чтобы загрузить массив файлов (с одним и тем же именем поля), используйте декоратор `FilesInterceptor()` (обратите внимание на множественное число **Files** в названии декоратора). Этот декоратор принимает три аргумента:

- `fieldName`: как описано выше
- `maxCount`: необязательное число, задающее максимальное количество принимаемых файлов
- `options`: необязательный объект `MulterOptions`, как описано выше

При использовании `FilesInterceptor()` извлекайте файлы из `request` с помощью декоратора `@UploadedFiles()`.

```typescript
@Post('upload')
@UseInterceptors(FilesInterceptor('files'))
uploadFile(@UploadedFiles() files: Array<Express.Multer.File>) {
  console.log(files);
}
```

> **Подсказка** Декоратор `FilesInterceptor()` экспортируется из пакета `@nestjs/platform-express`. Декоратор `@UploadedFiles()` экспортируется из `@nestjs/common`.

#### Несколько файлов

Чтобы загрузить несколько файлов (каждый со своим именем поля), используйте декоратор `FileFieldsInterceptor()`. Этот декоратор принимает два аргумента:

- `uploadedFields`: массив объектов, в каждом из которых задаётся обязательное свойство `name` со строковым значением — именем поля, как описано выше, — и необязательное свойство `maxCount`, также описанное выше
- `options`: необязательный объект `MulterOptions`, как описано выше

При использовании `FileFieldsInterceptor()` извлекайте файлы из `request` с помощью декоратора `@UploadedFiles()`.

```typescript
@Post('upload')
@UseInterceptors(FileFieldsInterceptor([
  { name: 'avatar', maxCount: 1 },
  { name: 'background', maxCount: 1 },
]))
uploadFile(@UploadedFiles() files: { avatar?: Express.Multer.File[], background?: Express.Multer.File[] }) {
  console.log(files);
}
```

#### Любые файлы

Чтобы загрузить все поля с произвольными именами, используйте декоратор `AnyFilesInterceptor()`. Он может принимать необязательный объект `options`, описанный выше.

При использовании `AnyFilesInterceptor()` извлекайте файлы из `request` с помощью декоратора `@UploadedFiles()`.

```typescript
@Post('upload')
@UseInterceptors(AnyFilesInterceptor())
uploadFile(@UploadedFiles() files: Array<Express.Multer.File>) {
  console.log(files);
}
```

#### Без файлов

Чтобы принимать `multipart/form-data`, но запретить загрузку любых файлов, используйте `NoFilesInterceptor`. Он помещает multipart-данные в тело запроса в виде атрибутов. Любой файл, отправленный с запросом, приведёт к `BadRequestException`.

```typescript
@Post('upload')
@UseInterceptors(NoFilesInterceptor())
handleMultiPartData(@Body() body) {
  console.log(body)
}
```

#### Параметры по умолчанию

Вы можете задавать параметры multer в файловых interceptors, как описано выше. Чтобы задать параметры по умолчанию, вызовите статический метод `register()` при импорте `MulterModule`, передав в него поддерживаемые параметры. Вы можете использовать все параметры, перечисленные [здесь](https://github.com/expressjs/multer#multeropts).

```typescript
MulterModule.register({
  dest: './upload',
});
```

> **Подсказка** Класс `MulterModule` экспортируется из пакета `@nestjs/platform-express`.

#### Асинхронная настройка

Когда параметры `MulterModule` нужно задать асинхронно, а не статически, используйте метод `registerAsync()`. Как и для большинства динамических модулей, Nest предлагает несколько приёмов асинхронной настройки.

Один из приёмов — использовать фабричную функцию:

```typescript
MulterModule.registerAsync({
  useFactory: () => ({
    dest: './upload',
  }),
});
```

Как и другие [фабричные провайдеры](<../03. Fundamentals/01. Custom providers.md#провайдеры-фабрики-usefactory>), наша фабричная функция может быть `async` и внедрять зависимости через `inject`.

```typescript
MulterModule.registerAsync({
  imports: [ConfigModule],
  useFactory: async (configService: ConfigService) => ({
    dest: configService.get<string>('MULTER_DEST'),
  }),
  inject: [ConfigService],
});
```

Кроме того, вы можете настроить `MulterModule` с помощью класса, а не фабрики, как показано ниже:

```typescript
MulterModule.registerAsync({
  useClass: MulterConfigService,
});
```

Приведённая конструкция создаёт экземпляр `MulterConfigService` внутри `MulterModule` и использует его для получения нужного объекта параметров. Обратите внимание, что в этом примере `MulterConfigService` должен реализовывать интерфейс `MulterOptionsFactory`, как показано ниже. `MulterModule` вызовет метод `createMulterOptions()` у созданного экземпляра переданного класса.

```typescript
@Injectable()
class MulterConfigService implements MulterOptionsFactory {
  createMulterOptions(): MulterModuleOptions {
    return {
      dest: './upload',
    };
  }
}
```

Если вы хотите переиспользовать существующий провайдер параметров вместо создания его отдельной копии внутри `MulterModule`, используйте синтаксис `useExisting`.

```typescript
MulterModule.registerAsync({
  imports: [ConfigModule],
  useExisting: ConfigService,
});
```

Вы также можете передать методу `registerAsync()` так называемые `extraProviders`. Эти провайдеры будут объединены с провайдерами модуля.

```typescript
MulterModule.registerAsync({
  imports: [ConfigModule],
  useClass: ConfigService,
  extraProviders: [MyAdditionalProvider],
});
```

Это удобно, когда нужно предоставить дополнительные зависимости фабричной функции или конструктору класса.
