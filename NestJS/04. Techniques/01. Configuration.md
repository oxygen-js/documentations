### Конфигурация

Приложения часто работают в разных **средах**. В зависимости от среды должны применяться разные настройки конфигурации. Например, локальная среда обычно использует конкретные учётные данные базы данных, действительные только для локального экземпляра БД. В production-среде будет использоваться отдельный набор учётных данных. Поскольку конфигурационные переменные меняются, лучшая практика — [хранить конфигурационные переменные](https://12factor.net/config) в окружении.

Заданные извне переменные окружения доступны внутри Node.js через глобальный объект `process.env`. Можно было бы решить проблему нескольких сред, задавая переменные окружения отдельно в каждой из них. Но такой подход быстро становится неудобным, особенно в средах разработки и тестирования, где эти значения нужно легко подменять и изменять.

В приложениях на Node.js для описания каждой среды принято использовать файлы `.env` с парами «ключ — значение», где каждый ключ соответствует определённому значению. Тогда запуск приложения в разных средах сводится к подстановке нужного файла `.env`.

Хороший способ применить эту технику в Nest — создать `ConfigModule`, который предоставляет `ConfigService`, загружающий подходящий файл `.env`. Вы можете написать такой модуль самостоятельно, но для удобства Nest предлагает пакет `@nestjs/config` «из коробки». Ему и посвящена эта глава.

#### Установка

Для начала установим необходимую зависимость.

```bash
$ npm i --save @nestjs/config
```

> **Подсказка** Пакет `@nestjs/config` внутри использует [dotenv](https://github.com/motdotla/dotenv).

> **Примечание** `@nestjs/config` требует TypeScript 4.1 или новее.

#### Начало работы

После установки можно импортировать `ConfigModule`. Как правило, его импортируют в корневой `AppModule` и управляют его поведением через статический метод `.forRoot()`. На этом шаге пары «ключ — значение» переменных окружения разбираются и разрешаются. Далее мы рассмотрим несколько способов обращения к классу `ConfigService` модуля `ConfigModule` из других функциональных модулей.

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { ConfigModule } from '@nestjs/config';

@Module({
  imports: [ConfigModule.forRoot()],
})
export class AppModule {}
```

Приведённый код загрузит и разберёт файл `.env` из расположения по умолчанию (корневой каталог проекта), объединит пары «ключ — значение» из `.env` с переменными окружения, присвоенными `process.env`, и сохранит результат в приватной структуре, доступной через `ConfigService`. Метод `forRoot()` регистрирует провайдер `ConfigService`, который предоставляет метод `get()` для чтения этих разобранных и объединённых конфигурационных переменных. Поскольку `@nestjs/config` опирается на [dotenv](https://github.com/motdotla/dotenv), для разрешения конфликтов имён переменных окружения используются правила этого пакета. Если ключ присутствует и в среде выполнения как переменная окружения (например, задан через экспорт в shell вида `export DATABASE_USER=test`), и в файле `.env`, приоритет имеет переменная окружения среды выполнения.

Пример файла `.env` выглядит примерно так:

```json
DATABASE_USER=test
DATABASE_PASSWORD=test
```

Если некоторые переменные окружения должны быть доступны ещё до загрузки `ConfigModule` и начальной загрузки приложения Nest (например, чтобы передать конфигурацию микросервиса в метод `NestFactory#createMicroservice`), воспользуйтесь опцией `--env-file` в Nest CLI. Она позволяет указать путь к файлу `.env`, который нужно загрузить до старта приложения. Поддержка флага `--env-file` появилась в Node v20, подробности см. в [документации](https://nodejs.org/dist/v20.18.1/docs/api/cli.html#--env-fileconfig).

```bash
$ nest start --env-file .env
```

#### Пользовательский путь к env-файлу

По умолчанию пакет ищет файл `.env` в корневом каталоге приложения. Чтобы указать другой путь к файлу `.env`, задайте свойство `envFilePath` в (необязательном) объекте опций, передаваемом в `forRoot()`:

```typescript
ConfigModule.forRoot({
  envFilePath: '.development.env',
});
```

Можно указать и несколько путей к файлам `.env`:

```typescript
ConfigModule.forRoot({
  envFilePath: ['.env.development.local', '.env.development'],
});
```

Если переменная найдена в нескольких файлах, приоритет имеет первый из них.

#### Отключение загрузки переменных окружения

Если вы не хотите загружать файл `.env`, а хотите просто обращаться к переменным окружения среды выполнения (как при экспорте в shell вида `export DATABASE_USER=test`), установите свойство `ignoreEnvFile` объекта опций в `true`:

```typescript
ConfigModule.forRoot({
  ignoreEnvFile: true,
});
```

#### Глобальное использование модуля

Чтобы использовать `ConfigModule` в других модулях, его нужно импортировать (как и любой другой модуль Nest). Как вариант, объявите его [глобальным модулем](<../02. Overview/04. Modules.md#глобальные-модули>), установив свойство `isGlobal` объекта опций в `true`, как показано ниже. В этом случае импортировать `ConfigModule` в других модулях не потребуется — достаточно загрузить его в корневом модуле (например, в `AppModule`).

```typescript
ConfigModule.forRoot({
  isGlobal: true,
});
```

#### Пользовательские файлы конфигурации

В более сложных проектах можно использовать пользовательские файлы конфигурации, возвращающие вложенные объекты конфигурации. Это позволяет группировать связанные настройки по назначению (например, настройки базы данных) и хранить их в отдельных файлах, управляя ими независимо друг от друга.

Пользовательский файл конфигурации экспортирует фабричную функцию, возвращающую объект конфигурации. Этот объект может быть произвольно вложенным простым объектом JavaScript. Объект `process.env` будет содержать полностью разрешённые пары «ключ — значение» переменных окружения (с разрешением и объединением файла `.env` и заданных извне переменных, как описано <a href="01.%20Configuration.md#начало-работы">выше</a>). Поскольку возвращаемый объект конфигурации формируете вы, в него можно добавить любую логику приведения значений к нужному типу, задания значений по умолчанию и т. д. Например:

**`config/configuration.ts`**

```typescript
export default () => ({
  port: parseInt(process.env.PORT, 10) || 3000,
  database: {
    host: process.env.DATABASE_HOST,
    port: parseInt(process.env.DATABASE_PORT, 10) || 5432
  }
});
```

Этот файл загружается через свойство `load` объекта опций, передаваемого в метод `ConfigModule.forRoot()`:

```typescript
import configuration from './config/configuration';

@Module({
  imports: [
    ConfigModule.forRoot({
      load: [configuration],
    }),
  ],
})
export class AppModule {}
```

> **Важно** Значение свойства `load` — массив, что позволяет загружать несколько файлов конфигурации (например, `load: [databaseConfig, authConfig]`)

С помощью пользовательских файлов конфигурации можно работать и с файлами других форматов, например YAML. Вот пример конфигурации в формате YAML:

```yaml
http:
  host: 'localhost'
  port: 8080

db:
  postgres:
    url: 'localhost'
    port: 5432
    database: 'yaml-db'

  sqlite:
    database: 'sqlite.db'
```

Для чтения и разбора YAML-файлов можно воспользоваться пакетом `js-yaml`.

```bash
$ npm i js-yaml
$ npm i -D @types/js-yaml
```

После установки пакета используем функцию `yaml#load`, чтобы загрузить созданный выше YAML-файл.

**`config/configuration.ts`**

```typescript
import { readFileSync } from 'node:fs';
import { join } from 'node:path';
import * as yaml from 'js-yaml';

const YAML_CONFIG_FILENAME = 'config.yaml';

export default () => {
  return yaml.load(
    readFileSync(join(__dirname, YAML_CONFIG_FILENAME), 'utf8'),
  ) as Record<string, any>;
};
```

> **Примечание** Nest CLI не переносит ваши «ассеты» (файлы, не относящиеся к TS) в папку `dist` автоматически во время сборки. Чтобы ваши YAML-файлы копировались, укажите это в объекте `compilerOptions#assets` в файле `nest-cli.json`. Например, если папка `config` находится на одном уровне с папкой `src`, добавьте `compilerOptions#assets` со значением `"assets": [{"include": "../config/*.yaml", "outDir": "./dist/config"}]`. Подробнее [здесь](<../11. CLI/02. Workspaces.md#ресурсы>).

Небольшое замечание: файлы конфигурации не валидируются автоматически, даже если вы используете опцию `validationSchema` в `ConfigModule` NestJS. Если вам нужна валидация или какие-либо преобразования, выполняйте их внутри фабричной функции, где вы полностью контролируете объект конфигурации. Это позволяет реализовать любую нужную вам логику проверки.

Например, если вы хотите убедиться, что порт находится в определённом диапазоне, добавьте в фабричную функцию шаг валидации:

**`config/configuration.ts`**

```typescript
export default () => {
  const config = yaml.load(
    readFileSync(join(__dirname, YAML_CONFIG_FILENAME), 'utf8'),
  ) as Record<string, any>;

  if (config.http.port < 1024 || config.http.port > 49151) {
    throw new Error('HTTP port must be between 1024 and 49151');
  }

  return config;
};
```

Теперь, если порт окажется вне указанного диапазона, приложение выбросит ошибку при старте.

#### Использование `ConfigService`

Чтобы получать значения конфигурации из `ConfigService`, сначала нужно внедрить `ConfigService`. Как и с любым провайдером, необходимо импортировать содержащий его модуль — `ConfigModule` — в тот модуль, где он будет использоваться (если только вы не установили свойство `isGlobal` в объекте опций метода `ConfigModule.forRoot()` в `true`). Импортируйте его в функциональный модуль, как показано ниже.

**`feature.module.ts`**

```typescript
@Module({
  imports: [ConfigModule],
  // ...
})
```

Затем его можно внедрить обычным способом — через конструктор:

```typescript
constructor(private configService: ConfigService) {}
```

> **Подсказка** `ConfigService` импортируется из пакета `@nestjs/config`.

И использовать в своём классе:

```typescript
// получить переменную окружения
const dbUser = this.configService.get<string>('DATABASE_USER');

// получить значение из пользовательской конфигурации
const dbHost = this.configService.get<string>('database.host');
```

Как показано выше, метод `configService.get()` возвращает простую переменную окружения по её имени. Можно указать подсказку типа для TypeScript, передав тип (например, `get<string>(...)`). Метод `get()` умеет также обходить вложенный объект пользовательской конфигурации (созданный через <a href="01.%20Configuration.md#пользовательские-файлы-конфигурации">пользовательский файл конфигурации</a>), как во втором примере выше.

Можно получить и весь вложенный объект пользовательской конфигурации целиком, указав интерфейс в качестве подсказки типа:

```typescript
interface DatabaseConfig {
  host: string;
  port: number;
}

const dbConfig = this.configService.get<DatabaseConfig>('database');

// теперь можно использовать `dbConfig.port` и `dbConfig.host`
const port = dbConfig.port;
```

Метод `get()` принимает необязательный второй аргумент — значение по умолчанию, которое будет возвращено, если ключ отсутствует:

```typescript
// использовать "localhost", если "database.host" не задан
const dbHost = this.configService.get<string>('database.host', 'localhost');
```

У `ConfigService` есть два необязательных обобщённых параметра (типовых аргумента). Первый помогает предотвратить обращение к несуществующему свойству конфигурации. Используйте его так:

```typescript
interface EnvironmentVariables {
  PORT: number;
  TIMEOUT: string;
}

// где-то в коде
constructor(private configService: ConfigService<EnvironmentVariables>) {
  const port = this.configService.get('PORT', { infer: true });

  // Ошибка TypeScript: так нельзя, поскольку свойство URL не объявлено в EnvironmentVariables
  const url = this.configService.get('URL', { infer: true });
}
```

Если свойство `infer` установлено в `true`, метод `ConfigService#get` автоматически выведет тип свойства на основе интерфейса — так, например, `typeof port === "number"` (если вы не используете флаг TypeScript `strictNullChecks`), поскольку в интерфейсе `EnvironmentVariables` у `PORT` тип `number`.

Кроме того, благодаря `infer` можно вывести тип свойства вложенного объекта пользовательской конфигурации даже при использовании точечной нотации:

```typescript
constructor(private configService: ConfigService<{ database: { host: string } }>) {
  const dbHost = this.configService.get('database.host', { infer: true })!;
  // typeof dbHost === "string"                                          |
  //                                              +--> оператор утверждения non-null
}
```

Второй обобщённый параметр опирается на первый и работает как утверждение типа, убирающее все типы `undefined`, которые методы `ConfigService` могут возвращать при включённом `strictNullChecks`. Например:

```typescript
// ...
constructor(private configService: ConfigService<{ PORT: number }, true>) {
  //                                                               ^^^^
  const port = this.configService.get('PORT', { infer: true });
  //    ^^^ Тип port будет 'number', так что утверждения типа TS больше не нужны
}
```

> **Подсказка** Чтобы метод `ConfigService#get` брал значения исключительно из пользовательских файлов конфигурации и игнорировал переменные `process.env`, установите опцию `skipProcessEnv` в `true` в объекте опций метода `forRoot()` модуля `ConfigModule`.

#### Пространства имён конфигурации

`ConfigModule` позволяет определять и загружать несколько пользовательских файлов конфигурации, как показано в разделе <a href="01.%20Configuration.md#пользовательские-файлы-конфигурации">Пользовательские файлы конфигурации</a> выше. Сложными иерархиями объектов конфигурации можно управлять с помощью вложенных объектов, как описано в том разделе. В качестве альтернативы можно вернуть объект конфигурации с пространством имён, используя функцию `registerAs()`:

**`config/database.config.ts`**

```typescript
export default registerAs('database', () => ({
  host: process.env.DATABASE_HOST,
  port: process.env.DATABASE_PORT || 5432
}));
```

Как и в случае пользовательских файлов конфигурации, внутри фабричной функции `registerAs()` объект `process.env` будет содержать полностью разрешённые пары «ключ — значение» переменных окружения (с разрешением и объединением файла `.env` и заданных извне переменных, как описано <a href="01.%20Configuration.md#начало-работы">выше</a>).

> **Подсказка** Функция `registerAs` экспортируется из пакета `@nestjs/config`.

Загрузите конфигурацию с пространством имён через свойство `load` объекта опций метода `forRoot()` — так же, как обычный пользовательский файл конфигурации:

```typescript
import databaseConfig from './config/database.config';

@Module({
  imports: [
    ConfigModule.forRoot({
      load: [databaseConfig],
    }),
  ],
})
export class AppModule {}
```

Теперь, чтобы получить значение `host` из пространства имён `database`, используйте точечную нотацию. В качестве префикса к имени свойства укажите `'database'` — имя пространства имён (переданное первым аргументом в функцию `registerAs()`):

```typescript
const dbHost = this.configService.get<string>('database.host');
```

Разумная альтернатива — внедрить пространство имён `database` напрямую. Это даёт преимущества строгой типизации:

```typescript
constructor(
  @Inject(databaseConfig.KEY)
  private dbConfig: ConfigType<typeof databaseConfig>,
) {}
```

> **Подсказка** `ConfigType` экспортируется из пакета `@nestjs/config`.

#### Конфигурации с пространствами имён в модулях

Чтобы использовать конфигурацию с пространством имён как объект конфигурации для другого модуля приложения, воспользуйтесь методом `.asProvider()` объекта конфигурации. Этот метод превращает вашу конфигурацию с пространством имён в провайдер, который затем можно передать в `forRootAsync()` (или любой аналогичный метод) нужного модуля.

Пример:

```typescript
import databaseConfig from './config/database.config';

@Module({
  imports: [
    TypeOrmModule.forRootAsync(databaseConfig.asProvider()),
  ],
})
```

Чтобы понять, как работает метод `.asProvider()`, рассмотрим возвращаемое им значение:

```typescript
// Возвращаемое значение метода .asProvider()
{
  imports: [ConfigModule.forFeature(databaseConfig)],
  useFactory: (configuration: ConfigType<typeof databaseConfig>) => configuration,
  inject: [databaseConfig.KEY]
}
```

Такая структура позволяет без лишних усилий встраивать конфигурации с пространствами имён в ваши модули, сохраняя приложение организованным и модульным и не заставляя писать шаблонный повторяющийся код.

#### Кэширование переменных окружения

Поскольку обращение к `process.env` может быть медленным, вы можете установить свойство `cache` объекта опций, передаваемого в `ConfigModule.forRoot()`, чтобы повысить производительность метода `ConfigService#get` при работе с переменными из `process.env`.

```typescript
ConfigModule.forRoot({
  cache: true,
});
```

#### Частичная регистрация

До сих пор мы обрабатывали файлы конфигурации в корневом модуле (например, `AppModule`) методом `forRoot()`. Возможно, структура вашего проекта сложнее и файлы конфигурации, относящиеся к отдельным возможностям, расположены в разных каталогах. Чтобы не загружать все эти файлы в корневом модуле, пакет `@nestjs/config` предлагает механизм **частичной регистрации**, который подключает только файлы конфигурации, связанные с конкретным функциональным модулем. Для частичной регистрации используйте статический метод `forFeature()` внутри функционального модуля:

```typescript
import databaseConfig from './config/database.config';

@Module({
  imports: [ConfigModule.forFeature(databaseConfig)],
})
export class DatabaseModule {}
```

> **Предупреждение** В некоторых случаях к свойствам, загруженным через частичную регистрацию, следует обращаться в хуке `onModuleInit()`, а не в конструкторе. Дело в том, что метод `forFeature()` выполняется во время инициализации модуля, а порядок инициализации модулей не определён. Если вы обратитесь к загруженным таким образом значениям из другого модуля в конструкторе, модуль, от которого зависит конфигурация, может быть ещё не инициализирован. Метод `onModuleInit()` выполняется только после инициализации всех модулей, от которых он зависит, поэтому этот приём безопасен.

#### Валидация по схеме

Стандартная практика — выбрасывать исключение при старте приложения, если обязательные переменные окружения не заданы или не соответствуют определённым правилам валидации. Пакет `@nestjs/config` предлагает два способа сделать это:

- Встроенный валидатор [Joi](https://github.com/sideway/joi). С Joi вы описываете схему объекта и проверяете по ней объекты JavaScript.
- Пользовательская функция `validate()`, принимающая переменные окружения на вход.

Чтобы использовать Joi, установим пакет Joi:

```bash
$ npm install --save joi
```

Теперь можно описать схему валидации Joi и передать её через свойство `validationSchema` объекта опций метода `forRoot()`:

**`app.module.ts`**

```typescript
import * as Joi from 'joi';

@Module({
  imports: [
    ConfigModule.forRoot({
      validationSchema: Joi.object({
        NODE_ENV: Joi.string()
          .valid('development', 'production', 'test', 'provision')
          .default('development'),
        PORT: Joi.number().port().default(3000),
      }),
    }),
  ],
})
export class AppModule {}
```

По умолчанию все ключи схемы считаются необязательными. Здесь мы задали значения по умолчанию для `NODE_ENV` и `PORT`, которые будут использоваться, если эти переменные не заданы в окружении (в файле `.env` или в окружении процесса). В качестве альтернативы можно применить метод валидации `required()`, чтобы значение обязательно было определено в окружении (в файле `.env` или в окружении процесса). В этом случае, если переменная не задана, шаг валидации выбросит исключение. Подробнее о построении схем валидации см. в [методах валидации Joi](https://joi.dev/api/?v=17.3.0#example).

По умолчанию неизвестные переменные окружения (те, чьи ключи отсутствуют в схеме) разрешены и не вызывают исключения валидации. Также по умолчанию сообщается обо всех ошибках валидации. Это поведение можно изменить, передав объект опций через ключ `validationOptions` в объекте опций `forRoot()`. Этот объект может содержать любые стандартные свойства опций валидации, предоставляемые [опциями валидации Joi](https://joi.dev/api/?v=17.3.0#anyvalidatevalue-options). Например, чтобы изменить обе описанные выше настройки на противоположные, передайте опции так:

**`app.module.ts`**

```typescript
import * as Joi from 'joi';

@Module({
  imports: [
    ConfigModule.forRoot({
      validationSchema: Joi.object({
        NODE_ENV: Joi.string()
          .valid('development', 'production', 'test', 'provision')
          .default('development'),
        PORT: Joi.number().port().default(3000),
      }),
      validationOptions: {
        allowUnknown: false,
        abortEarly: true,
      },
    }),
  ],
})
export class AppModule {}
```

Пакет `@nestjs/config` использует следующие настройки по умолчанию:

- `allowUnknown`: определяет, разрешены ли неизвестные ключи среди переменных окружения. По умолчанию `true`
- `abortEarly`: если true, валидация останавливается на первой ошибке; если false, возвращаются все ошибки. По умолчанию `false`.

Учтите: если вы решили передать объект `validationOptions`, все настройки, которые вы не указали явно, примут стандартные значения по умолчанию `Joi` (а не значения по умолчанию `@nestjs/config`). Например, если оставить `allowUnknowns` неуказанным в своём объекте `validationOptions`, он получит значение по умолчанию `Joi` — `false`. Поэтому, вероятно, безопаснее всего указывать в своём объекте **обе** эти настройки.

> **Подсказка** Чтобы отключить валидацию предопределённых переменных окружения, установите атрибут `validatePredefined` в `false` в объекте опций метода `forRoot()`. Предопределённые переменные окружения — это переменные процесса (`process.env`), заданные до импорта модуля. Например, если вы запускаете приложение командой `PORT=3000 node main.js`, то `PORT` является предопределённой переменной окружения.

#### Пользовательская функция валидации

В качестве альтернативы можно задать **синхронную** функцию `validate`, которая принимает объект с переменными окружения (из env-файла и из процесса) и возвращает объект с проверенными переменными окружения, чтобы при необходимости вы могли их преобразовать или изменить. Если функция выбросит ошибку, приложение не запустится.

В этом примере мы воспользуемся пакетами `class-transformer` и `class-validator`. Сначала нужно определить:

- класс с ограничениями валидации,
- функцию валидации, использующую функции `plainToInstance` и `validateSync`.

**`env.validation.ts`**

```typescript
import { plainToInstance } from 'class-transformer';
import { IsEnum, IsNumber, Max, Min, validateSync } from 'class-validator';

enum Environment {
  Development = "development",
  Production = "production",
  Test = "test",
  Provision = "provision",
}

class EnvironmentVariables {
  @IsEnum(Environment)
  NODE_ENV: Environment;

  @IsNumber()
  @Min(0)
  @Max(65535)
  PORT: number;
}

export function validate(config: Record<string, unknown>) {
  const validatedConfig = plainToInstance(
    EnvironmentVariables,
    config,
    { enableImplicitConversion: true },
  );
  const errors = validateSync(validatedConfig, { skipMissingProperties: false });

  if (errors.length > 0) {
    throw new Error(errors.toString());
  }
  return validatedConfig;
}
```

Теперь используйте функцию `validate` как опцию конфигурации `ConfigModule`:

**`app.module.ts`**

```typescript
import { validate } from './env.validation';

@Module({
  imports: [
    ConfigModule.forRoot({
      validate,
    }),
  ],
})
export class AppModule {}
```

#### Пользовательские функции-геттеры

`ConfigService` определяет обобщённый метод `get()` для получения значения конфигурации по ключу. Мы можем также добавить функции-`getter`, чтобы код выглядел естественнее:

```typescript
@Injectable()
export class ApiConfigService {
  constructor(private configService: ConfigService) {}

  get isAuthEnabled(): boolean {
    return this.configService.get('AUTH_ENABLED') === 'true';
  }
}
```

Теперь функцию-геттер можно использовать так:

**`app.service.ts`**

```typescript
@Injectable()
export class AppService {
  constructor(apiConfigService: ApiConfigService) {
    if (apiConfigService.isAuthEnabled) {
      // Аутентификация включена
    }
  }
}
```

#### Хук загрузки переменных окружения

Если конфигурация модуля зависит от переменных окружения, а эти переменные загружаются из файла `.env`, используйте хук `ConfigModule.envVariablesLoaded`, чтобы убедиться, что файл был загружен до обращения к объекту `process.env`. Пример:

```typescript
export async function getStorageModule() {
  await ConfigModule.envVariablesLoaded;
  return process.env.STORAGE === 'S3' ? S3StorageModule : DefaultStorageModule;
}
```

Такая конструкция гарантирует, что после разрешения промиса `ConfigModule.envVariablesLoaded` все конфигурационные переменные загружены.

#### Условная конфигурация модулей

Иногда требуется загружать модуль условно, задавая условие через переменную окружения. К счастью, `@nestjs/config` предоставляет `ConditionalModule`, который позволяет это сделать.

```typescript
@Module({
  imports: [
    ConfigModule.forRoot(),
    ConditionalModule.registerWhen(FooModule, 'USE_FOO'),
  ],
})
export class AppModule {}
```

Приведённый модуль загрузит `FooModule` только в том случае, если в файле `.env` переменная окружения `USE_FOO` не имеет значения `false`. Вы также можете передать собственное условие — функцию, принимающую ссылку на `process.env` и возвращающую булево значение для `ConditionalModule`:

```typescript
@Module({
  imports: [
    ConfigModule.forRoot(),
    ConditionalModule.registerWhen(
      FooBarModule,
      (env: NodeJS.ProcessEnv) => !!env['foo'] && !!env['bar'],
    ),
  ],
})
export class AppModule {}
```

Важно убедиться, что при использовании `ConditionalModule` в приложении также загружен `ConfigModule`, чтобы хук `ConfigModule.envVariablesLoaded` можно было корректно использовать. Если хук не переключится в true в течение 5 секунд (или заданного пользователем таймаута в миллисекундах, передаваемого третьим параметром опций метода `registerWhen`), `ConditionalModule` выбросит ошибку, и Nest прервёт запуск приложения.

#### Раскрываемые переменные

Пакет `@nestjs/config` поддерживает раскрытие (expansion) переменных окружения. С помощью этой техники можно создавать вложенные переменные окружения, когда одна переменная упоминается в определении другой. Например:

```json
APP_URL=mywebsite.com
SUPPORT_EMAIL=support@${APP_URL}
```

При такой конструкции переменная `SUPPORT_EMAIL` разрешится в `'support@mywebsite.com'`. Обратите внимание на синтаксис `${...}`, который запускает разрешение значения переменной `APP_URL` внутри определения `SUPPORT_EMAIL`.

> **Подсказка** Для этой возможности пакет `@nestjs/config` внутри использует [dotenv-expand](https://github.com/motdotla/dotenv-expand).

Включите раскрытие переменных окружения через свойство `expandVariables` в объекте опций, передаваемом в метод `forRoot()` модуля `ConfigModule`:

**`app.module.ts`**

```typescript
@Module({
  imports: [
    ConfigModule.forRoot({
      // ...
      expandVariables: true,
    }),
  ],
})
export class AppModule {}
```

#### Использование в `main.ts`

Хотя наша конфигурация хранится в сервисе, её можно использовать и в файле `main.ts`. Так вы сможете хранить в ней такие значения, как порт приложения или хост для CORS.

Чтобы получить к ней доступ, используйте метод `app.get()`, передав ссылку на сервис:

```typescript
const configService = app.get(ConfigService);
```

Дальше её можно использовать как обычно, вызывая метод `get` с ключом конфигурации:

```typescript
const port = configService.get('PORT');
```
