### Версионирование

> **Подсказка** Эта глава относится только к приложениям на основе HTTP.

Версионирование позволяет иметь **разные версии** контроллеров или отдельных маршрутов в рамках одного приложения. Приложения меняются очень часто, и нередко приходится вносить обратно несовместимые изменения, продолжая при этом поддерживать предыдущую версию приложения.

#### Версионирование по URI

При версионировании по URI используется версия, переданная в URI запроса, например `https://example.com/v1/route` и `https://example.com/v2/route`.

> **Важно** При версионировании по URI версия автоматически добавляется в URI после <a href="../14.%20FAQ/04.%20Global%20path%20prefix.md">глобального префикса пути</a> (если он задан) и перед путями контроллеров и маршрутов.

Чтобы включить версионирование по URI в вашем приложении, сделайте следующее:

**`main.ts`**

```typescript
const app = await NestFactory.create(AppModule);
// или "app.enableVersioning()"
app.enableVersioning({
  type: VersioningType.URI,
});
await app.listen(process.env.PORT ?? 3000);
```

> **Важно** По умолчанию к версии в URI автоматически добавляется префикс `v`, однако значение префикса можно настроить, задав ключ `prefix` равным нужному префиксу или `false`, если префикс не нужен.

> **Подсказка** Перечисление `VersioningType` можно использовать для свойства `type`; оно импортируется из пакета `@nestjs/common`.

#### Версионирование по заголовку

При версионировании по заголовку версия задаётся пользовательским заголовком запроса: значение этого заголовка и будет версией, используемой для запроса.

Пример HTTP-запросов для версионирования по заголовку:

Чтобы включить **версионирование по заголовку** в вашем приложении, сделайте следующее:

**`main.ts`**

```typescript
const app = await NestFactory.create(AppModule);
app.enableVersioning({
  type: VersioningType.HEADER,
  header: 'Custom-Header',
});
await app.listen(process.env.PORT ?? 3000);
```

Свойство `header` должно содержать имя заголовка, в котором будет передаваться версия запроса.

> **Подсказка** Перечисление `VersioningType` можно использовать для свойства `type`; оно импортируется из пакета `@nestjs/common`.

#### Версионирование по типу содержимого

При версионировании по типу содержимого версия задаётся заголовком запроса `Accept`.

Внутри заголовка `Accept` версия отделяется от типа содержимого точкой с запятой — `;`. Далее должна следовать пара «ключ-значение», представляющая версию для этого запроса, например `Accept: application/json;v=2`. Ключ при определении версии трактуется скорее как префикс, поэтому настраивается он вместе с разделителем.

Чтобы включить **версионирование по типу содержимого** в вашем приложении, сделайте следующее:

**`main.ts`**

```typescript
const app = await NestFactory.create(AppModule);
app.enableVersioning({
  type: VersioningType.MEDIA_TYPE,
  key: 'v=',
});
await app.listen(process.env.PORT ?? 3000);
```

Свойство `key` должно содержать ключ и разделитель той пары «ключ-значение», в которой передаётся версия. Для примера `Accept: application/json;v=2` свойству `key` следует задать значение `v=`.

> **Подсказка** Перечисление `VersioningType` можно использовать для свойства `type`; оно импортируется из пакета `@nestjs/common`.

#### Пользовательское версионирование

При пользовательском версионировании для указания версии (или версий) используется любой аспект запроса. Входящий запрос анализируется
функцией `extractor`, которая возвращает строку или массив строк.

Если клиент указывает несколько версий, функция-экстрактор может вернуть массив строк, отсортированный
от наибольшей/самой новой версии к наименьшей/самой старой. Версии сопоставляются с маршрутами по порядку — от наибольшей к
наименьшей.

Если `extractor` возвращает пустую строку или пустой массив, ни один маршрут не сопоставляется и возвращается 404.

Например, если входящий запрос сообщает, что поддерживает версии `1`, `2` и `3`, `extractor` **ОБЯЗАН** вернуть `[3, 2, 1]`. Это гарантирует, что первой будет выбрана максимально возможная версия маршрута.

Если извлечены версии `[3, 2, 1]`, но маршруты существуют только для версий `2` и `1`, будет выбран маршрут, соответствующий версии `2`
(версия `3` автоматически игнорируется).

> **Важно** Выбор наибольшей подходящей версии на основе массива, возвращённого функцией `extractor`, > **надёжно не работает** с адаптером Express из-за особенностей его архитектуры. Одиночная версия (строка или
> массив из 1 элемента) в Express работает без проблем. Fastify корректно поддерживает и выбор наибольшей подходящей версии,
> и выбор одиночной версии.

Чтобы включить **пользовательское версионирование** в вашем приложении, создайте функцию `extractor` и передайте её приложению
следующим образом:

**`main.ts`**

```typescript
// Пример экстрактора, который извлекает список версий из пользовательского заголовка и превращает его в отсортированный массив.
// В этом примере используется Fastify, но запросы Express можно обрабатывать аналогичным образом.
const extractor = (request: FastifyRequest): string | string[] =>
  [request.headers['custom-versioning-field'] ?? '']
     .flatMap(v => v.split(','))
     .filter(v => !!v)
     .sort()
     .reverse()

const app = await NestFactory.create(AppModule);
app.enableVersioning({
  type: VersioningType.CUSTOM,
  extractor,
});
await app.listen(process.env.PORT ?? 3000);
```

#### Использование

Версионирование позволяет задавать версии контроллеров и отдельных маршрутов, а также даёт способ исключить отдельные ресурсы из версионирования. Порядок использования одинаков независимо от того, какой тип версионирования применяется в вашем приложении.

> **Важно** Если версионирование включено для приложения, но контроллер или маршрут не указывает версию, любые запросы к такому контроллеру или маршруту получат статус ответа `404`. Аналогично, если поступает запрос с версией, для которой нет соответствующего контроллера или маршрута, он тоже получит статус ответа `404`.

#### Версии контроллеров

Версию можно задать контроллеру — тогда она распространяется на все его маршруты.

Чтобы добавить версию контроллеру, сделайте следующее:

**`cats.controller.ts`**

```typescript
@Controller({
  version: '1',
})
export class CatsControllerV1 {
  @Get('cats')
  findAll(): string {
    return 'This action returns all cats for version 1';
  }
}
```

#### Версии маршрутов

Версию можно задать отдельному маршруту. Такая версия переопределяет любую другую версию, влияющую на этот маршрут, в том числе версию контроллера.

Чтобы добавить версию отдельному маршруту, сделайте следующее:

**`cats.controller.ts`**

```typescript
import { Controller, Get, Version } from '@nestjs/common';

@Controller()
export class CatsController {
  @Version('1')
  @Get('cats')
  findAllV1(): string {
    return 'This action returns all cats for version 1';
  }

  @Version('2')
  @Get('cats')
  findAllV2(): string {
    return 'This action returns all cats for version 2';
  }
}
```

#### Несколько версий

Контроллеру или маршруту можно задать несколько версий. Для этого укажите в качестве версии массив.

Чтобы добавить несколько версий, сделайте следующее:

**`cats.controller.ts`**

```typescript
@Controller({
  version: ['1', '2'],
})
export class CatsController {
  @Get('cats')
  findAll(): string {
    return 'This action returns all cats for version 1 or 2';
  }
}
```

#### Версия «Neutral»

Некоторым контроллерам или маршрутам версия безразлична: они работают одинаково независимо от неё. Для таких случаев версию можно задать символом `VERSION_NEUTRAL`.

Входящий запрос будет сопоставлен с контроллером или маршрутом `VERSION_NEUTRAL` независимо от версии, указанной в запросе, а также если версия в запросе вообще отсутствует.

> **Важно** При версионировании по URI у ресурса `VERSION_NEUTRAL` версия в URI отсутствует.

Чтобы добавить контроллер или маршрут, нейтральный к версии, сделайте следующее:

**`cats.controller.ts`**

```typescript
import { Controller, Get, VERSION_NEUTRAL } from '@nestjs/common';

@Controller({
  version: VERSION_NEUTRAL,
})
export class CatsController {
  @Get('cats')
  findAll(): string {
    return 'This action returns all cats regardless of version';
  }
}
```

#### Глобальная версия по умолчанию

Если вы не хотите указывать версию для каждого контроллера или отдельного маршрута либо хотите задать конкретную версию как версию по умолчанию для всех контроллеров и маршрутов, где версия не указана, вы можете задать `defaultVersion` следующим образом:

**`main.ts`**

```typescript
app.enableVersioning({
  // ...
  defaultVersion: '1'
  // или
  defaultVersion: ['1', '2']
  // или
  defaultVersion: VERSION_NEUTRAL
});
```

#### Версионирование middleware

[Middleware](<../02. Overview/05. Middleware.md>) также может использовать метаданные версионирования, чтобы настраиваться на версию конкретного маршрута. Для этого передайте номер версии одним из параметров метода `MiddlewareConsumer.forRoutes()`:

**`app.module.ts`**

```typescript
import { Module, NestModule, MiddlewareConsumer } from '@nestjs/common';
import { LoggerMiddleware } from './common/middleware/logger.middleware';
import { CatsModule } from './cats/cats.module';
import { CatsController } from './cats/cats.controller';

@Module({
  imports: [CatsModule],
})
export class AppModule implements NestModule {
  configure(consumer: MiddlewareConsumer) {
    consumer
      .apply(LoggerMiddleware)
      .forRoutes({ path: 'cats', method: RequestMethod.GET, version: '2' });
  }
}
```

С приведённым выше кодом `LoggerMiddleware` будет применяться только к версии '2' эндпоинта `/cats`.

> **Важно** Middleware работает с любым типом версионирования, описанным в этом разделе: `URI`, `Header`, `Media Type` или `Custom`.
