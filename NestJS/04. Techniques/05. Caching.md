### Кэширование

Кэширование — мощный и простой **приём** повышения производительности приложения. Выступая в роли временного слоя хранения, оно ускоряет доступ к часто используемым данным и избавляет от необходимости раз за разом получать или вычислять одну и ту же информацию. В результате время отклика сокращается, а общая эффективность растёт.

#### Установка

Чтобы начать работу с кэшированием в Nest, установите пакет `@nestjs/cache-manager` вместе с пакетом `cache-manager`.

```bash
$ npm install @nestjs/cache-manager cache-manager
```

По умолчанию всё хранится в памяти; поскольку `cache-manager` использует «под капотом» [Keyv](https://keyv.org/docs/), вы легко можете перейти на более продвинутое хранилище, например Redis, установив соответствующий пакет. Подробнее об этом — далее.

#### Кэш в памяти

Чтобы включить кэширование в приложении, импортируйте `CacheModule` и настройте его методом `register()`:

```typescript
import { Module } from '@nestjs/common';
import { CacheModule } from '@nestjs/cache-manager';
import { AppController } from './app.controller';

@Module({
  imports: [CacheModule.register()],
  controllers: [AppController],
})
export class AppModule {}
```

Такая настройка инициализирует кэширование в памяти с параметрами по умолчанию, и вы сразу можете начать кэшировать данные.

#### Работа с хранилищем кэша

Чтобы взаимодействовать с экземпляром cache manager, внедрите его в свой класс по токену `CACHE_MANAGER`:

```typescript
constructor(@Inject(CACHE_MANAGER) private cacheManager: Cache) {}
```

> **Подсказка** Класс `Cache` и токен `CACHE_MANAGER` импортируются из пакета `@nestjs/cache-manager`.

Метод `get` экземпляра `Cache` (из пакета `cache-manager`) используется для получения элементов из кэша. Если элемента в кэше нет, вернётся `undefined` (в `cache-manager` версии 6 и ранее возвращался `null`). При миграции считайте оба значения ложными.

```typescript
const value = await this.cacheManager.get('key');
```

Чтобы добавить элемент в кэш, используйте метод `set`:

```typescript
await this.cacheManager.set('key', 'value');
```

> **Примечание** Хранилище кэша в памяти может хранить значения только тех типов, которые поддерживаются [алгоритмом структурированного клонирования](https://developer.mozilla.org/en-US/docs/Web/API/Web_Workers_API/Structured_clone_algorithm#javascript_types).

Вы можете вручную задать TTL (время жизни в миллисекундах) для конкретного ключа:

```typescript
await this.cacheManager.set('key', 'value', 1000);
```

Здесь `1000` — это TTL в миллисекундах; в данном случае элемент кэша устареет через одну секунду.

Чтобы отключить устаревание кэша, установите свойство конфигурации `ttl` в `0`:

```typescript
await this.cacheManager.set('key', 'value', 0);
```

Чтобы удалить элемент из кэша, используйте метод `del`:

```typescript
await this.cacheManager.del('key');
```

Чтобы очистить кэш целиком, используйте метод `clear`:

```typescript
await this.cacheManager.clear();
```

#### Автоматическое кэширование ответов

> **Предупреждение** В приложениях на [GraphQL](<../06. GraphQL/01. Quick start.md>) interceptor'ы выполняются отдельно для каждого резолвера поля. Поэтому `CacheModule` (который использует interceptor'ы для кэширования ответов) работать корректно не будет.

Чтобы включить автоматическое кэширование ответов, просто подключите `CacheInterceptor` там, где нужно кэшировать данные.

```typescript
@Controller()
@UseInterceptors(CacheInterceptor)
export class AppController {
  @Get()
  findAll(): string[] {
    return [];
  }
}
```

> **Предупреждение** Кэшируются только эндпоинты `GET`. Кроме того, маршруты HTTP-сервера, которые внедряют нативный объект ответа (`@Res()`), не могут использовать Cache Interceptor. Подробнее см.
> <a href="../02.%20Overview/09.%20Interceptors.md#преобразование-ответа">response mapping</a>.

Чтобы сократить объём шаблонного кода, вы можете привязать `CacheInterceptor` глобально ко всем эндпоинтам:

```typescript
import { Module } from '@nestjs/common';
import { CacheModule, CacheInterceptor } from '@nestjs/cache-manager';
import { AppController } from './app.controller';
import { APP_INTERCEPTOR } from '@nestjs/core';

@Module({
  imports: [CacheModule.register()],
  controllers: [AppController],
  providers: [
    {
      provide: APP_INTERCEPTOR,
      useClass: CacheInterceptor,
    },
  ],
})
export class AppModule {}
```

#### Время жизни (TTL)

Значение `ttl` по умолчанию — `0`, то есть кэш никогда не устаревает. Чтобы задать собственное значение [TTL](https://en.wikipedia.org/wiki/Time_to_live), передайте параметр `ttl` в метод `register()`, как показано ниже:

```typescript
CacheModule.register({
  ttl: 5000, // миллисекунды
});
```

#### Использование модуля глобально

Если вы хотите использовать `CacheModule` в других модулях, его нужно импортировать (как и любой модуль Nest). Либо объявите его [глобальным модулем](<../02. Overview/04. Modules.md#глобальные-модули>), установив свойство `isGlobal` объекта параметров в `true`, как показано ниже. В этом случае импортировать `CacheModule` в других модулях не потребуется — достаточно загрузить его в корневом модуле (например, в `AppModule`).

```typescript
CacheModule.register({
  isGlobal: true,
});
```

#### Переопределение глобального кэша

При включённом глобальном кэше записи сохраняются под `CacheKey`, который генерируется автоматически на основе пути маршрута. Некоторые настройки кэша (`@CacheKey()` и `@CacheTTL()`) можно переопределить для отдельных методов, задавая свои стратегии кэширования для конкретных методов контроллера. Это особенно актуально при использовании [разных хранилищ кэша.](<05. Caching.md#different-stores>)

Декоратор `@CacheTTL()` можно применить на уровне контроллера, чтобы задать TTL кэширования для всего контроллера. Если TTL задан и на уровне контроллера, и на уровне метода, приоритет имеет значение, указанное на уровне метода.

```typescript
@Controller()
@CacheTTL(50)
export class AppController {
  @CacheKey('custom_key')
  @CacheTTL(20)
  findAll(): string[] {
    return [];
  }
}
```

> **Подсказка** Декораторы `@CacheKey()` и `@CacheTTL()` импортируются из пакета `@nestjs/cache-manager`.

Декоратор `@CacheKey()` можно использовать как вместе с соответствующим `@CacheTTL()`, так и без него, и наоборот. Можно переопределить только `@CacheKey()` или только `@CacheTTL()`. Настройки, не переопределённые декоратором, возьмут значения по умолчанию, зарегистрированные глобально (см. [Настройка кэширования](<05. Caching.md#customize-caching>)).

#### WebSockets и микросервисы

Вы также можете применять `CacheInterceptor` к подписчикам WebSocket, а также к шаблонам микросервисов (независимо от используемого способа транспорта).

```typescript
@CacheKey('events')
@UseInterceptors(CacheInterceptor)
@SubscribeMessage('events')
handleEvent(client: Client, data: string[]): Observable<string[]> {
  return [];
}
```

Однако в этом случае требуется дополнительный декоратор `@CacheKey()`, задающий ключ, по которому кэшированные данные будут сохраняться и извлекаться. Также обратите внимание, что **кэшировать всё подряд не следует**. Действия, выполняющие бизнес-операции, а не просто запрашивающие данные, кэшировать нельзя.

Кроме того, вы можете задать время устаревания кэша (TTL) декоратором `@CacheTTL()` — оно переопределит глобальное значение TTL по умолчанию.

```typescript
@CacheTTL(10)
@UseInterceptors(CacheInterceptor)
@SubscribeMessage('events')
handleEvent(client: Client, data: string[]): Observable<string[]> {
  return [];
}
```

> **Подсказка** Декоратор `@CacheTTL()` можно использовать как вместе с соответствующим декоратором `@CacheKey()`, так и без него.

#### Настройка отслеживания

По умолчанию Nest использует URL запроса (в HTTP-приложении) или ключ кэша (в приложениях на websockets и микросервисах, заданный декоратором `@CacheKey()`), чтобы связать записи кэша с вашими эндпоинтами. Тем не менее иногда требуется отслеживание по другим признакам — например, по HTTP-заголовкам (скажем, по `Authorization`, чтобы корректно различать эндпоинты `profile`).

Чтобы этого добиться, создайте подкласс `CacheInterceptor` и переопределите метод `trackBy()`.

```typescript
@Injectable()
class HttpCacheInterceptor extends CacheInterceptor {
  trackBy(context: ExecutionContext): string | undefined {
    return 'key';
  }
}
```

#### Использование альтернативных хранилищ кэша

Переключиться на другое хранилище кэша просто. Сначала установите нужный пакет. Например, для использования Redis установите пакет `@keyv/redis`:

```bash
$ npm install @keyv/redis
```

После этого вы можете зарегистрировать `CacheModule` с несколькими хранилищами, как показано ниже:

```typescript
import { Module } from '@nestjs/common';
import { CacheModule } from '@nestjs/cache-manager';
import { AppController } from './app.controller';
import KeyvRedis from '@keyv/redis';
import { Keyv } from 'keyv';
import { KeyvCacheableMemory } from 'cacheable';

@Module({
  imports: [
    CacheModule.registerAsync({
      useFactory: async () => {
        return {
          stores: [
            new Keyv({
              store: new KeyvCacheableMemory({ ttl: 60000, lruSize: 5000 }),
            }),
            new KeyvRedis('redis://localhost:6379'),
          ],
        };
      },
    }),
  ],
  controllers: [AppController],
})
export class AppModule {}
```

В этом примере мы зарегистрировали два хранилища: `CacheableMemory` и `KeyvRedis`. `CacheableMemory` — простое хранилище в памяти, создаваемое через адаптер `KeyvCacheableMemory`, а `KeyvRedis` — хранилище Redis. Массив `stores` задаёт хранилища, которые вы хотите использовать. Первое хранилище в массиве считается основным, остальные — запасными.

Подробнее о доступных хранилищах читайте в [документации Keyv](https://keyv.org/docs/).

#### Асинхронная конфигурация

Вам может понадобиться передавать параметры модуля асинхронно, а не статически на этапе компиляции. В этом случае используйте метод `registerAsync()`, который предлагает несколько способов асинхронной настройки.

Один из подходов — фабричная функция:

```typescript
CacheModule.registerAsync({
  useFactory: () => ({
    ttl: 5,
  }),
});
```

Наша фабрика ведёт себя так же, как и все прочие асинхронные фабрики модулей (она может быть `async` и способна внедрять зависимости через `inject`).

```typescript
CacheModule.registerAsync({
  imports: [ConfigModule],
  useFactory: async (configService: ConfigService) => ({
    ttl: configService.get('CACHE_TTL'),
  }),
  inject: [ConfigService],
});
```

Как вариант, вы можете использовать способ `useClass`:

```typescript
CacheModule.registerAsync({
  useClass: CacheConfigService,
});
```

Приведённая конструкция создаст экземпляр `CacheConfigService` внутри `CacheModule` и использует его для получения объекта параметров. Чтобы предоставлять параметры конфигурации, `CacheConfigService` должен реализовывать интерфейс `CacheOptionsFactory`:

```typescript
@Injectable()
class CacheConfigService implements CacheOptionsFactory {
  createCacheOptions(): CacheModuleOptions {
    return {
      ttl: 5,
    };
  }
}
```

Если вы хотите использовать существующий провайдер конфигурации, импортированный из другого модуля, воспользуйтесь синтаксисом `useExisting`:

```typescript
CacheModule.registerAsync({
  imports: [ConfigModule],
  useExisting: ConfigService,
});
```

Это работает так же, как `useClass`, с одним важным отличием: `CacheModule` просмотрит импортированные модули и переиспользует уже созданный `ConfigService` вместо того, чтобы создавать собственный.

> **Подсказка** У `CacheModule#register`, `CacheModule#registerAsync` и `CacheOptionsFactory` есть необязательный обобщённый параметр (аргумент типа), позволяющий сузить параметры конфигурации до конкретного хранилища и сделать их типобезопасными.

Также в метод `registerAsync()` можно передать так называемые `extraProviders`. Эти провайдеры будут объединены с провайдерами модуля.

```typescript
CacheModule.registerAsync({
  imports: [ConfigModule],
  useClass: ConfigService,
  extraProviders: [MyAdditionalProvider],
});
```

Это удобно, когда нужно предоставить дополнительные зависимости фабричной функции или конструктору класса.
