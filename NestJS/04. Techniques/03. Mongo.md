### Mongo

Nest поддерживает два способа интеграции с базой данных [MongoDB](https://www.mongodb.com/). Вы можете либо использовать встроенный модуль [TypeORM](https://github.com/typeorm/typeorm), описанный [здесь](<02. Database.md>), в котором есть коннектор для MongoDB, либо [Mongoose](https://mongoosejs.com) — самый популярный инструмент объектного моделирования для MongoDB. В этой главе мы разберём второй вариант, с использованием специального пакета `@nestjs/mongoose`.

Начните с установки [необходимых зависимостей](https://github.com/Automattic/mongoose):

```bash
$ npm i @nestjs/mongoose mongoose
```

После завершения установки можно импортировать `MongooseModule` в корневой `AppModule`.

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { MongooseModule } from '@nestjs/mongoose';

@Module({
  imports: [MongooseModule.forRoot('mongodb://localhost/nest')],
})
export class AppModule {}
```

Метод `forRoot()` принимает тот же объект конфигурации, что и `mongoose.connect()` из пакета Mongoose, — он описан [здесь](https://mongoosejs.com/docs/connections.html).

#### Внедрение модели

В Mongoose всё строится вокруг [схемы](http://mongoosejs.com/docs/guide.html). Каждая схема сопоставляется с коллекцией MongoDB и задаёт форму документов внутри этой коллекции. На основе схем определяются [модели](https://mongoosejs.com/docs/models.html). Модели отвечают за создание и чтение документов из нижележащей базы данных MongoDB.

Схемы можно создавать декораторами NestJS или вручную средствами самого Mongoose. Использование декораторов заметно сокращает объём шаблонного кода и улучшает общую читаемость.

Определим `CatSchema`:

**`schemas/cat.schema.ts`**

```typescript
import { Prop, Schema, SchemaFactory } from '@nestjs/mongoose';
import { HydratedDocument } from 'mongoose';

export type CatDocument = HydratedDocument<Cat>;

@Schema()
export class Cat {
  @Prop()
  name: string;

  @Prop()
  age: number;

  @Prop()
  breed: string;
}

export const CatSchema = SchemaFactory.createForClass(Cat);
```

> **Подсказка** Обратите внимание: вы также можете сгенерировать «сырое» определение схемы с помощью класса `DefinitionsFactory` (из `nestjs/mongoose`). Это позволяет вручную изменить определение схемы, сгенерированное на основе предоставленных вами метаданных. Приём полезен в отдельных пограничных случаях, когда выразить всё декораторами затруднительно.

Декоратор `@Schema()` помечает класс как определение схемы. Он сопоставляет наш класс `Cat` с одноимённой коллекцией MongoDB, но с добавлением «s» на конце — так что итоговое имя коллекции в mongo будет `cats`. Декоратор принимает единственный необязательный аргумент — объект опций схемы. Считайте его тем объектом, который вы обычно передаёте вторым аргументом в конструктор класса `mongoose.Schema` (например, `new mongoose.Schema(_, options)`). Подробнее о доступных опциях схемы читайте в [этой](https://mongoosejs.com/docs/guide.html#options) главе.

Декоратор `@Prop()` определяет свойство документа. Например, в приведённом выше определении схемы мы определили три свойства: `name`, `age` и `breed`. [Типы схемы](https://mongoosejs.com/docs/schematypes.html) для этих свойств выводятся автоматически благодаря возможностям метаданных TypeScript (и рефлексии). Однако в более сложных сценариях, где типы не могут быть выведены неявно (например, массивы или вложенные структуры объектов), типы нужно указывать явно:

```typescript
@Prop([String])
tags: string[];
```

Кроме того, декоратор `@Prop()` принимает аргумент — объект опций ([подробнее](https://mongoosejs.com/docs/schematypes.html#schematype-options) о доступных опциях). С его помощью вы можете указать, обязательно ли свойство, задать значение по умолчанию или пометить его как неизменяемое. Например:

```typescript
@Prop({ required: true })
name: string;
```

Если вы хотите указать связь с другой моделью, чтобы позже выполнять populate, декоратор `@Prop()` тоже подойдёт. Например, если у `Cat` есть `Owner`, который хранится в другой коллекции `owners`, у свойства должны быть указаны тип и ref. Например:

```typescript
import * as mongoose from 'mongoose';
import { Owner } from '../owners/schemas/owner.schema';

// внутри определения класса
@Prop({ type: mongoose.Schema.Types.ObjectId, ref: 'Owner' })
owner: Owner;
```

Если владельцев несколько, конфигурация свойства должна выглядеть так:

```typescript
@Prop({ type: [{ type: mongoose.Schema.Types.ObjectId, ref: 'Owner' }] })
owners: Owner[];
```

Если вы не собираетесь всегда выполнять populate ссылки на другую коллекцию, рассмотрите вариант с типом `mongoose.Types.ObjectId`:

```typescript
@Prop({ type: { type: mongoose.Schema.Types.ObjectId, ref: 'Owner' } })
// Это гарантирует, что поле не спутают с уже загруженной (populated) ссылкой
owner: mongoose.Types.ObjectId;
```

Затем, когда потребуется выборочно выполнить populate, вы можете использовать функцию репозитория, указывающую корректный тип:

```typescript
import { Owner } from './schemas/owner.schema';

// например, внутри сервиса или репозитория
async findAllPopulated() {
  return this.catModel.find().populate<{ owner: Owner }>("owner");
}
```

> **Подсказка** Если связанного документа для populate нет, тип может быть `Owner | null` — в зависимости от вашей [конфигурации Mongoose](https://mongoosejs.com/docs/populate.html#doc-not-found). Либо будет выброшена ошибка, и тогда типом будет `Owner`.

Наконец, в декоратор можно передать и **сырое** определение схемы. Это удобно, когда, например, свойство представляет вложенный объект, не описанный отдельным классом. Для этого используйте функцию `raw()` из пакета `@nestjs/mongoose`:

```typescript
@Prop(raw({
  firstName: { type: String },
  lastName: { type: String }
}))
details: Record<string, any>;
```

Если же вы предпочитаете **не использовать декораторы**, схему можно определить вручную. Например:

```typescript
export const CatSchema = new mongoose.Schema({
  name: String,
  age: Number,
  breed: String,
});
```

Файл `cat.schema` лежит в папке внутри каталога `cats`, где мы также определяем `CatsModule`. Хранить файлы схем можно где угодно, но мы рекомендуем размещать их рядом с относящимися к ним объектами **предметной области**, в каталоге соответствующего модуля.

Посмотрим на `CatsModule`:

**`cats.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { MongooseModule } from '@nestjs/mongoose';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';
import { Cat, CatSchema } from './schemas/cat.schema';

@Module({
  imports: [MongooseModule.forFeature([{ name: Cat.name, schema: CatSchema }])],
  controllers: [CatsController],
  providers: [CatsService],
})
export class CatsModule {}
```

`MongooseModule` предоставляет метод `forFeature()` для настройки модуля, в том числе для указания, какие модели должны быть зарегистрированы в текущей области. Если вы хотите использовать эти модели и в другом модуле, добавьте MongooseModule в секцию `exports` модуля `CatsModule`, а сам `CatsModule` импортируйте в другой модуль.

После регистрации схемы вы можете внедрить модель `Cat` в `CatsService` с помощью декоратора `@InjectModel()`:

**`cats.service.ts`**

```typescript
import { Model } from 'mongoose';
import { Injectable } from '@nestjs/common';
import { InjectModel } from '@nestjs/mongoose';
import { Cat } from './schemas/cat.schema';
import { CreateCatDto } from './dto/create-cat.dto';

@Injectable()
export class CatsService {
  constructor(@InjectModel(Cat.name) private catModel: Model<Cat>) {}

  async create(createCatDto: CreateCatDto): Promise<Cat> {
    const createdCat = new this.catModel(createCatDto);
    return createdCat.save();
  }

  async findAll(): Promise<Cat[]> {
    return this.catModel.find().exec();
  }
}
```

#### Соединение

Иногда требуется доступ к нативному объекту [Mongoose Connection](https://mongoosejs.com/docs/api.html#Connection). Например, чтобы выполнить нативные вызовы API на объекте соединения. Внедрить Mongoose Connection можно с помощью декоратора `@InjectConnection()`:

```typescript
import { Injectable } from '@nestjs/common';
import { InjectConnection } from '@nestjs/mongoose';
import { Connection } from 'mongoose';

@Injectable()
export class CatsService {
  constructor(@InjectConnection() private connection: Connection) {}
}
```

#### Сессии

Чтобы начать сессию в Mongoose, рекомендуется внедрять соединение с базой данных через `@InjectConnection`, а не вызывать `mongoose.startSession()` напрямую. Такой подход обеспечивает лучшую интеграцию с системой внедрения зависимостей NestJS и корректное управление соединениями.

Вот пример запуска сессии:

```typescript
import { InjectConnection } from '@nestjs/mongoose';
import { Connection } from 'mongoose';

@Injectable()
export class CatsService {
  constructor(@InjectConnection() private readonly connection: Connection) {}

  async startTransaction() {
    const session = await this.connection.startSession();
    session.startTransaction();
    // Здесь логика вашей транзакции
  }
}
```

В этом примере `@InjectConnection()` используется для внедрения соединения Mongoose в сервис. Получив соединение, вы можете вызвать `connection.startSession()`, чтобы начать новую сессию. Она позволяет управлять транзакциями базы данных, обеспечивая атомарность операций в нескольких запросах. Начав сессию, не забудьте зафиксировать или отменить транзакцию в соответствии с вашей логикой.

#### Несколько баз данных

Некоторым проектам нужно несколько подключений к базам данных. Этот модуль позволяет решить и такую задачу. Чтобы работать с несколькими соединениями, сначала создайте их. В этом случае именование соединений становится **обязательным**.

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { MongooseModule } from '@nestjs/mongoose';

@Module({
  imports: [
    MongooseModule.forRoot('mongodb://localhost/test', {
      connectionName: 'cats',
    }),
    MongooseModule.forRoot('mongodb://localhost/users', {
      connectionName: 'users',
    }),
  ],
})
export class AppModule {}
```

> **Важно** Учтите, что у вас не должно быть нескольких соединений без имени или с одинаковыми именами, иначе они будут перезаписывать друг друга.

При такой настройке нужно сообщить функции `MongooseModule.forFeature()`, какое соединение следует использовать.

```typescript
@Module({
  imports: [
    MongooseModule.forFeature([{ name: Cat.name, schema: CatSchema }], 'cats'),
  ],
})
export class CatsModule {}
```

Вы также можете внедрить `Connection` для конкретного соединения:

```typescript
import { Injectable } from '@nestjs/common';
import { InjectConnection } from '@nestjs/mongoose';
import { Connection } from 'mongoose';

@Injectable()
export class CatsService {
  constructor(@InjectConnection('cats') private connection: Connection) {}
}
```

Чтобы внедрить конкретное `Connection` в пользовательский провайдер (например, в factory-провайдер), используйте функцию `getConnectionToken()`, передав ей имя соединения в качестве аргумента.

```typescript
{
  provide: CatsService,
  useFactory: (catsConnection: Connection) => {
    return new CatsService(catsConnection);
  },
  inject: [getConnectionToken('cats')],
}
```

Если вам нужно просто внедрить модель из именованной базы данных, передайте имя соединения вторым параметром декоратора `@InjectModel()`.

**`cats.service.ts`**

```typescript
@Injectable()
export class CatsService {
  constructor(@InjectModel(Cat.name, 'cats') private catModel: Model<Cat>) {}
}
```

#### Хуки (middleware)

Middleware (их также называют pre- и post-хуками) — это функции, которым передаётся управление во время выполнения асинхронных функций. Middleware задаётся на уровне схемы и удобно для написания плагинов ([источник](https://mongoosejs.com/docs/middleware.html)). Вызов `pre()` или `post()` после компиляции модели в Mongoose не работает. Чтобы зарегистрировать хук **до** регистрации модели, используйте метод `forFeatureAsync()` модуля `MongooseModule` вместе с factory-провайдером (то есть `useFactory`). При таком подходе вы получаете доступ к объекту схемы и можете зарегистрировать на ней хук методом `pre()` или `post()`. Пример:

```typescript
@Module({
  imports: [
    MongooseModule.forFeatureAsync([
      {
        name: Cat.name,
        useFactory: () => {
          const schema = CatSchema;
          schema.pre('save', function () {
            console.log('Hello from pre save');
          });
          return schema;
        },
      },
    ]),
  ],
})
export class AppModule {}
```

Как и другие [factory-провайдеры](<../03. Fundamentals/01. Custom providers.md#провайдеры-фабрики-usefactory>), наша фабричная функция может быть `async` и внедрять зависимости через `inject`.

```typescript
@Module({
  imports: [
    MongooseModule.forFeatureAsync([
      {
        name: Cat.name,
        imports: [ConfigModule],
        useFactory: (configService: ConfigService) => {
          const schema = CatSchema;
          schema.pre('save', function() {
            console.log(
              `${configService.get('APP_NAME')}: Hello from pre save`,
            ),
          });
          return schema;
        },
        inject: [ConfigService],
      },
    ]),
  ],
})
export class AppModule {}
```

#### Плагины

Чтобы зарегистрировать [плагин](https://mongoosejs.com/docs/plugins.html) для конкретной схемы, используйте метод `forFeatureAsync()`.

```typescript
@Module({
  imports: [
    MongooseModule.forFeatureAsync([
      {
        name: Cat.name,
        useFactory: () => {
          const schema = CatSchema;
          schema.plugin(require('mongoose-autopopulate'));
          return schema;
        },
      },
    ]),
  ],
})
export class AppModule {}
```

Чтобы зарегистрировать плагин сразу для всех схем, вызовите метод `.plugin()` объекта `Connection`. Обратиться к соединению нужно до создания моделей; для этого используйте `connectionFactory`:

**`app.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { MongooseModule } from '@nestjs/mongoose';

@Module({
  imports: [
    MongooseModule.forRoot('mongodb://localhost/test', {
      connectionFactory: (connection) => {
        connection.plugin(require('mongoose-autopopulate'));
        return connection;
      }
    }),
  ],
})
export class AppModule {}
```

#### Дискриминаторы

[Дискриминаторы](https://mongoosejs.com/docs/discriminators.html) — механизм наследования схем. Они позволяют иметь несколько моделей с пересекающимися схемами поверх одной и той же нижележащей коллекции MongoDB.

Допустим, вы хотите отслеживать разные типы событий в одной коллекции. У каждого события будет отметка времени.

**`event.schema.ts`**

```typescript
@Schema({ discriminatorKey: 'kind' })
export class Event {
  @Prop({
    type: String,
    required: true,
    enum: [ClickedLinkEvent.name, SignUpEvent.name],
  })
  kind: string;

  @Prop({ type: Date, required: true })
  time: Date;
}

export const EventSchema = SchemaFactory.createForClass(Event);
```

> **Подсказка** Mongoose различает разные модели-дискриминаторы по «ключу дискриминатора», которым по умолчанию является `__t`. Mongoose добавляет в ваши схемы строковый путь `__t` и использует его, чтобы отслеживать, экземпляром какого дискриминатора является документ.
> Вы также можете задать путь для различения с помощью опции `discriminatorKey`.

Экземпляры `SignUpEvent` и `ClickedLinkEvent` будут храниться в той же коллекции, что и обобщённые события.

Теперь определим класс `ClickedLinkEvent`:

**`click-link-event.schema.ts`**

```typescript
@Schema()
export class ClickedLinkEvent {
  kind: string;
  time: Date;

  @Prop({ type: String, required: true })
  url: string;
}

export const ClickedLinkEventSchema = SchemaFactory.createForClass(ClickedLinkEvent);
```

И класс `SignUpEvent`:

**`sign-up-event.schema.ts`**

```typescript
@Schema()
export class SignUpEvent {
  kind: string;
  time: Date;

  @Prop({ type: String, required: true })
  user: string;
}

export const SignUpEventSchema = SchemaFactory.createForClass(SignUpEvent);
```

Теперь используйте опцию `discriminators`, чтобы зарегистрировать дискриминатор для конкретной схемы. Она работает и с `MongooseModule.forFeature`, и с `MongooseModule.forFeatureAsync`:

**`event.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { MongooseModule } from '@nestjs/mongoose';

@Module({
  imports: [
    MongooseModule.forFeature([
      {
        name: Event.name,
        schema: EventSchema,
        discriminators: [
          { name: ClickedLinkEvent.name, schema: ClickedLinkEventSchema },
          { name: SignUpEvent.name, schema: SignUpEventSchema },
        ],
      },
    ]),
  ]
})
export class EventsModule {}
```

#### Тестирование

При модульном тестировании приложения обычно хочется обойтись без подключения к базе данных, чтобы наборы тестов было проще настраивать и они быстрее выполнялись. Но наши классы могут зависеть от моделей, получаемых из экземпляра соединения. Как разрешить такие классы? Решение — создать mock-модели.

Чтобы упростить это, пакет `@nestjs/mongoose` предоставляет функцию `getModelToken()`, которая возвращает готовый [токен внедрения](<../03. Fundamentals/01. Custom providers.md#основы-di>) на основе имени токена. С помощью этого токена вы легко подставите mock-реализацию любым из стандартных приёмов [пользовательских провайдеров](<../03. Fundamentals/01. Custom providers.md>), включая `useClass`, `useValue` и `useFactory`. Например:

```typescript
@Module({
  providers: [
    CatsService,
    {
      provide: getModelToken(Cat.name),
      useValue: catModel,
    },
  ],
})
export class CatsModule {}
```

В этом примере жёстко заданный `catModel` (экземпляр объекта) будет предоставлен всякий раз, когда любой потребитель внедряет `Model<Cat>` с помощью декоратора `@InjectModel()`.

#### Асинхронная конфигурация

Когда параметры модуля нужно передавать асинхронно, а не статически, используйте метод `forRootAsync()`. Как и для большинства динамических модулей, Nest предлагает несколько приёмов работы с асинхронной конфигурацией.

Один из приёмов — использовать фабричную функцию:

```typescript
MongooseModule.forRootAsync({
  useFactory: () => ({
    uri: 'mongodb://localhost/nest',
  }),
});
```

Как и другие [factory-провайдеры](<../03. Fundamentals/01. Custom providers.md#провайдеры-фабрики-usefactory>), наша фабричная функция может быть `async` и внедрять зависимости через `inject`.

```typescript
MongooseModule.forRootAsync({
  imports: [ConfigModule],
  useFactory: async (configService: ConfigService) => ({
    uri: configService.get<string>('MONGODB_URI'),
  }),
  inject: [ConfigService],
});
```

Кроме того, `MongooseModule` можно настроить с помощью класса вместо фабрики:

```typescript
MongooseModule.forRootAsync({
  useClass: MongooseConfigService,
});
```

Приведённая конструкция создаёт экземпляр `MongooseConfigService` внутри `MongooseModule` и использует его для формирования нужного объекта опций. Обратите внимание: в этом примере `MongooseConfigService` должен реализовывать интерфейс `MongooseOptionsFactory`, как показано ниже. `MongooseModule` вызовет метод `createMongooseOptions()` на созданном экземпляре переданного класса.

```typescript
@Injectable()
export class MongooseConfigService implements MongooseOptionsFactory {
  createMongooseOptions(): MongooseModuleOptions {
    return {
      uri: 'mongodb://localhost/nest',
    };
  }
}
```

Если вы хотите переиспользовать существующий провайдер опций, а не создавать его приватную копию внутри `MongooseModule`, используйте синтаксис `useExisting`.

```typescript
MongooseModule.forRootAsync({
  imports: [ConfigModule],
  useExisting: ConfigService,
});
```

#### События соединения

Вы можете слушать [события соединения](https://mongoosejs.com/docs/connections.html#connection-events) Mongoose с помощью опции конфигурации `onConnectionCreate`. Это позволяет выполнять собственную логику при каждом установлении соединения. Например, вы можете зарегистрировать обработчики событий `connected`, `open`, `disconnected`, `reconnected` и `disconnecting`, как показано ниже:

```typescript
MongooseModule.forRoot('mongodb://localhost/test', {
  onConnectionCreate: (connection: Connection) => {
    connection.on('connected', () => console.log('connected'));
    connection.on('open', () => console.log('open'));
    connection.on('disconnected', () => console.log('disconnected'));
    connection.on('reconnected', () => console.log('reconnected'));
    connection.on('disconnecting', () => console.log('disconnecting'));

    return connection;
  },
}),
```

В этом фрагменте кода мы устанавливаем соединение с базой данных MongoDB по адресу `mongodb://localhost/test`. Опция `onConnectionCreate` позволяет настроить конкретные обработчики событий для наблюдения за состоянием соединения:

- `connected`: срабатывает, когда соединение успешно установлено.
- `open`: срабатывает, когда соединение полностью открыто и готово к операциям.
- `disconnected`: вызывается при потере соединения.
- `reconnected`: вызывается, когда соединение восстановлено после разрыва.
- `disconnecting`: происходит, когда соединение находится в процессе закрытия.

Свойство `onConnectionCreate` можно также включить в асинхронную конфигурацию, создаваемую через `MongooseModule.forRootAsync()`:

```typescript
MongooseModule.forRootAsync({
  useFactory: () => ({
    uri: 'mongodb://localhost/test',
    onConnectionCreate: (connection: Connection) => {
      // Здесь регистрируем обработчики событий
      return connection;
    },
  }),
}),
```

Это даёт гибкий способ управлять событиями соединения и эффективно обрабатывать изменения его состояния.

#### Вложенные документы

Чтобы вложить поддокументы в родительский документ, вы можете определить схемы следующим образом:

**`name.schema.ts`**

```typescript
@Schema()
export class Name {
  @Prop()
  firstName: string;

  @Prop()
  lastName: string;
}

export const NameSchema = SchemaFactory.createForClass(Name);
```

А затем сослаться на поддокумент в родительской схеме:

**`person.schema.ts`**

```typescript
@Schema()
export class Person {
  @Prop(NameSchema)
  name: Name;
}

export const PersonSchema = SchemaFactory.createForClass(Person);

export type PersonDocumentOverride = {
  name: Types.Subdocument<Types.ObjectId> & Name;
};

export type PersonDocument = HydratedDocument<Person, PersonDocumentOverride>;
```

Если нужно включить несколько поддокументов, используйте их массив. Важно соответствующим образом переопределить тип свойства:

**`name.schema.ts`**

```typescript
@Schema()
export class Person {
  @Prop([NameSchema])
  name: Name[];
}

export const PersonSchema = SchemaFactory.createForClass(Person);

export type PersonDocumentOverride = {
  name: Types.DocumentArray<Name>;
};

export type PersonDocument = HydratedDocument<Person, PersonDocumentOverride>;
```

#### Виртуальные свойства

В Mongoose **виртуальное свойство** (virtual) — это свойство, которое существует у документа, но не сохраняется в MongoDB. Оно не хранится в базе, а вычисляется динамически при каждом обращении. Виртуальные свойства обычно используют для производных или вычисляемых значений — например, для объединения полей (скажем, свойство `fullName` из `firstName` и `lastName`) или для свойств, опирающихся на уже имеющиеся данные документа.

```ts
class Person {
  @Prop()
  firstName: string;

  @Prop()
  lastName: string;

  @Virtual({
    get: function (this: Person) {
      return `${this.firstName} ${this.lastName}`;
    },
  })
  fullName: string;
}
```

> **Подсказка** Декоратор `@Virtual()` импортируется из пакета `@nestjs/mongoose`.

В этом примере виртуальное свойство `fullName` вычисляется из `firstName` и `lastName`. Хотя при обращении оно ведёт себя как обычное свойство, в документ MongoDB оно никогда не сохраняется.:
