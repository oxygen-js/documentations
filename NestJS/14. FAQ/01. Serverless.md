### Serverless

Serverless-вычисления — это модель облачных вычислений, при которой облачный провайдер выделяет машинные ресурсы по требованию, беря обслуживание серверов на себя. Когда приложение не используется, вычислительные ресурсы ему не выделяются. Оплата зависит от фактического объёма ресурсов, потреблённых приложением ([источник](https://en.wikipedia.org/wiki/Serverless_computing)).

При **serverless-архитектуре** вы сосредотачиваетесь исключительно на отдельных функциях в коде приложения. Такие сервисы, как AWS Lambda, Google Cloud Functions и Microsoft Azure Functions, берут на себя всё физическое оборудование, операционную систему виртуальной машины и управление веб-сервером.

> **Подсказка** В этой главе не разбираются плюсы и минусы serverless-функций и не рассматриваются особенности конкретных облачных провайдеров.

#### Холодный старт

Холодный старт — это первый за некоторое время запуск вашего кода. В зависимости от используемого облачного провайдера он может включать несколько разных операций: от загрузки кода и начальной загрузки среды выполнения до собственно запуска вашего кода.
Этот процесс добавляет **существенную задержку**, величина которой зависит от ряда факторов: языка, количества пакетов, необходимых приложению, и т. д.

Холодный старт важен, и хотя часть факторов нам неподконтрольна, со своей стороны мы всё же можем многое сделать, чтобы сократить его.

Nest можно воспринимать как полноценный фреймворк, рассчитанный на сложные корпоративные приложения, но он также **подходит и для гораздо более «простых» приложений** (или скриптов). Например, с помощью [автономных приложений](<../10. Standalone apps.md>) вы можете использовать систему внедрения зависимостей Nest в простых воркерах, CRON-задачах, CLI-утилитах или serverless-функциях.

#### Бенчмарки

Чтобы лучше понять, во сколько обходится использование Nest или других известных библиотек (например, `express`) в контексте serverless-функций, сравним, сколько времени среде выполнения Node требуется на запуск следующих скриптов:

```typescript
// #1 Express
import * as express from 'express';

async function bootstrap() {
  const app = express();
  app.get('/', (req, res) => res.send('Hello world!'));
  await new Promise<void>((resolve) => app.listen(3000, resolve));
}
bootstrap();

// #2 Nest (с @nestjs/platform-express)
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule, { logger: ['error'] });
  await app.listen(process.env.PORT ?? 3000);
}
bootstrap();

// #3 Nest как автономное приложение (без HTTP-сервера)
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';
import { AppService } from './app.service';

async function bootstrap() {
  const app = await NestFactory.createApplicationContext(AppModule, {
    logger: ['error'],
  });
  console.log(app.get(AppService).getHello());
}
bootstrap();

// #4 Чистый скрипт на Node.js
async function bootstrap() {
  console.log('Hello world!');
}
bootstrap();
```

Для всех этих скриптов использовался компилятор `tsc` (TypeScript), поэтому код не собирался в бандл (`webpack` не применялся).

|                                      |                   |
| ------------------------------------ | ----------------- |
| Express                              | 0.0079s (7.9ms)   |
| Nest с `@nestjs/platform-express`    | 0.1974s (197.4ms) |
| Nest (автономное приложение)         | 0.1117s (111.7ms) |
| Чистый скрипт на Node.js             | 0.0071s (7.1ms)   |

> **Примечание** Машина: MacBook Pro Mid 2014, 2.5 GHz Quad-Core Intel Core i7, 16 GB 1600 MHz DDR3, SSD.

Теперь повторим все замеры, но на этот раз соберём приложение в один исполняемый JavaScript-файл с помощью `webpack` (если у вас установлен [Nest CLI](<../11. CLI/01. Overview.md>), достаточно выполнить `nest build --webpack`).
Однако вместо конфигурации `webpack` по умолчанию, поставляемой с Nest CLI, мы соберём в бандл и все зависимости (`node_modules`), вот так:

```javascript
module.exports = (options, webpack) => {
  const lazyImports = [
    '@nestjs/microservices/microservices-module',
    '@nestjs/websockets/socket-module',
  ];

  return {
    ...options,
    externals: [],
    plugins: [
      ...options.plugins,
      new webpack.IgnorePlugin({
        checkResource(resource) {
          if (lazyImports.includes(resource)) {
            try {
              require.resolve(resource);
            } catch (err) {
              return true;
            }
          }
          return false;
        },
      }),
    ],
  };
};
```

> **Подсказка** Чтобы Nest CLI использовал эту конфигурацию, создайте файл `webpack.config.js` в корневом каталоге проекта.

С такой конфигурацией получились следующие результаты:

|                                      |                  |
| ------------------------------------ | ---------------- |
| Express                              | 0.0068s (6.8ms)  |
| Nest с `@nestjs/platform-express`    | 0.0815s (81.5ms) |
| Nest (автономное приложение)         | 0.0319s (31.9ms) |
| Чистый скрипт на Node.js             | 0.0066s (6.6ms)  |

> **Примечание** Машина: MacBook Pro Mid 2014, 2.5 GHz Quad-Core Intel Core i7, 16 GB 1600 MHz DDR3, SSD.

> **Подсказка** Результат можно улучшить ещё сильнее, применив дополнительные приёмы минификации и оптимизации кода (плагины `webpack` и т. п.).

Как видите, способ компиляции (и то, собираете ли вы код в бандл) критически важен и заметно влияет на общее время запуска. С `webpack` время начальной загрузки автономного приложения Nest (стартовый проект с одним модулем, контроллером и сервисом) можно снизить в среднем до ~32 мс, а для обычного HTTP-приложения NestJS на базе express — до ~81,5 мс.

Для более сложных приложений Nest, например с 10 ресурсами (сгенерированными схематиком `$ nest g resource` = 10 модулей, 10 контроллеров, 10 сервисов, 20 классов DTO, 50 HTTP-эндпоинтов + `AppModule`), общее время запуска на MacBook Pro Mid 2014, 2.5 GHz Quad-Core Intel Core i7, 16 GB 1600 MHz DDR3, SSD составляет примерно 0.1298s (129.8ms). Запускать монолитное приложение как serverless-функцию в любом случае обычно не слишком осмысленно, так что воспринимайте этот бенчмарк скорее как пример того, насколько может расти время начальной загрузки по мере роста приложения.

#### Оптимизации времени выполнения

До сих пор мы говорили об оптимизациях на этапе компиляции. Они никак не связаны с тем, как вы объявляете провайдеры и загружаете модули Nest, а это играет существенную роль по мере роста приложения.

Например, представьте, что подключение к базе данных объявлено как [асинхронный провайдер](<../03. Fundamentals/02. Asynchronous providers.md>). Асинхронные провайдеры задуманы так, чтобы откладывать запуск приложения до завершения одной или нескольких асинхронных задач.
Это значит, что если вашей serverless-функции в среднем нужно 2 с на подключение к базе данных (при начальной загрузке), вашему эндпоинту потребуется как минимум две дополнительные секунды (потому что придётся дождаться установки соединения), прежде чем он отправит ответ (при холодном старте, когда приложение ещё не было запущено).

Как видите, в **serverless-окружении**, где время начальной загрузки имеет значение, структурировать провайдеры приходится несколько иначе.
Ещё один хороший пример — использование Redis для кэширования, но только в определённых сценариях. Возможно, в этом случае не стоит объявлять подключение к Redis асинхронным провайдером: оно замедлит начальную загрузку, даже если для конкретного вызова функции не требуется.

Кроме того, иногда можно лениво загружать целые модули с помощью класса `LazyModuleLoader`, как описано в [этой главе](<../03. Fundamentals/07. Lazy-loading modules.md>). Кэширование и здесь отличный пример.
Представьте, что в приложении есть, скажем, `CacheModule`, который внутри подключается к Redis и экспортирует `CacheService` для работы с хранилищем Redis. Если он нужен не при всех возможных вызовах функции,
его можно загружать лениво, по требованию. Так вы получите более быстрый старт (при холодном старте) для всех вызовов, которым кэширование не требуется.

```typescript
if (request.method === RequestMethod[RequestMethod.GET]) {
  const { CacheModule } = await import('./cache.module');
  const moduleRef = await this.lazyModuleLoader.load(() => CacheModule);

  const { CacheService } = await import('./cache.service');
  const cacheService = moduleRef.get(CacheService);

  return cacheService.get(ENDPOINT_KEY);
}
```

Ещё один хороший пример — вебхук или воркер, который в зависимости от определённых условий (например, входных аргументов) выполняет разные операции.
В таком случае в обработчике маршрута можно задать условие, лениво загружающее подходящий модуль для конкретного вызова функции, а все остальные модули загружать так же лениво.

```typescript
if (workerType === WorkerType.A) {
  const { WorkerAModule } = await import('./worker-a.module');
  const moduleRef = await this.lazyModuleLoader.load(() => WorkerAModule);
  // ...
} else if (workerType === WorkerType.B) {
  const { WorkerBModule } = await import('./worker-b.module');
  const moduleRef = await this.lazyModuleLoader.load(() => WorkerBModule);
  // ...
}
```

#### Пример интеграции

То, как должен выглядеть входной файл вашего приложения (обычно `main.ts`), **зависит от нескольких факторов**, поэтому **единого шаблона**, подходящего для любого сценария, не существует.
Например, файл инициализации, необходимый для запуска serverless-функции, различается у разных облачных провайдеров (AWS, Azure, GCP и т. д.).
Кроме того, код приложения будет отличаться в зависимости от того, хотите ли вы запустить обычное HTTP-приложение с множеством маршрутов/эндпоинтов или предоставить единственный маршрут (либо выполнить конкретный фрагмент кода) — например, при подходе «эндпоинт на функцию» можно использовать `NestFactory.createApplicationContext` вместо запуска HTTP-сервера, настройки middleware и т. д.

Исключительно в иллюстративных целях мы интегрируем Nest (с `@nestjs/platform-express`, то есть с запуском полноценного HTTP-роутера) с фреймворком [Serverless](https://www.serverless.com/) (в данном случае с прицелом на AWS Lambda). Как уже говорилось, ваш код будет отличаться в зависимости от выбранного облачного провайдера и множества других факторов.

Сначала установим необходимые пакеты:

```bash
$ npm i @codegenie/serverless-express aws-lambda
$ npm i -D @types/aws-lambda serverless-offline
```

> **Подсказка** Чтобы ускорить циклы разработки, мы устанавливаем плагин `serverless-offline`, эмулирующий AWS λ и API Gateway.

Когда установка завершится, создадим файл `serverless.yml` для настройки фреймворка Serverless:

```yaml
service: serverless-example

plugins:
  - serverless-offline

provider:
  name: aws
  runtime: nodejs14.x

functions:
  main:
    handler: dist/main.handler
    events:
      - http:
          method: ANY
          path: /
      - http:
          method: ANY
          path: '{proxy+}'
```

> **Подсказка** Подробнее о фреймворке Serverless читайте в [официальной документации](https://www.serverless.com/framework/docs/).

Теперь можно перейти к файлу `main.ts` и дополнить код начальной загрузки необходимым шаблонным кодом:

```typescript
import { NestFactory } from '@nestjs/core';
import serverlessExpress from '@codegenie/serverless-express';
import { Callback, Context, Handler } from 'aws-lambda';
import { AppModule } from './app.module';

let server: Handler;

async function bootstrap(): Promise<Handler> {
  const app = await NestFactory.create(AppModule);
  await app.init();

  const expressApp = app.getHttpAdapter().getInstance();
  return serverlessExpress({ app: expressApp });
}

export const handler: Handler = async (
  event: any,
  context: Context,
  callback: Callback,
) => {
  server = server ?? (await bootstrap());
  return server(event, context, callback);
};
```

> **Подсказка** Для создания нескольких serverless-функций и разделения общих модулей между ними рекомендуем использовать [режим монорепозитория в CLI](<../11. CLI/02. Workspaces.md#режим-монорепозитория>).

> **Предупреждение** Если вы используете пакет `@nestjs/swagger`, потребуется несколько дополнительных шагов, чтобы он корректно работал в контексте serverless-функции. Подробности в [этом обсуждении](https://github.com/nestjs/swagger/issues/199).

Далее откройте файл `tsconfig.json` и включите опцию `esModuleInterop`, чтобы пакет `@codegenie/serverless-express` загружался корректно.

```json
{
  "compilerOptions": {
    ...
    "esModuleInterop": true
  }
}
```

Теперь можно собрать приложение (командой `nest build` или `tsc`) и запустить лямбда-функцию локально через CLI `serverless`:

```bash
$ npm run build
$ npx serverless offline
```

Когда приложение запустится, откройте браузер и перейдите по адресу `http://localhost:3000/dev/[ANY_ROUTE]` (где `[ANY_ROUTE]` — любой зарегистрированный в приложении эндпоинт).

В разделах выше мы показали, что использование `webpack` и сборка приложения в бандл могут заметно повлиять на общее время начальной загрузки.
Однако, чтобы это заработало в нашем примере, в файл `webpack.config.js` нужно добавить несколько дополнительных настроек. В частности,
чтобы наша функция `handler` была подхвачена, необходимо изменить свойство `output.libraryTarget` на `commonjs2`.

```javascript
return {
  ...options,
  externals: [],
  output: {
    ...options.output,
    libraryTarget: 'commonjs2',
  },
  // ... остальная часть конфигурации
};
```

После этого вы можете компилировать код функции командой `$ nest build --webpack` (а затем проверять его через `$ npx serverless offline`).

Также рекомендуется (но **не обязательно**, так как это замедлит сборку) установить пакет `terser-webpack-plugin` и переопределить его конфигурацию, чтобы имена классов сохранялись при минификации production-сборки. Иначе возможна некорректная работа `class-validator` в вашем приложении.

```javascript
const TerserPlugin = require('terser-webpack-plugin');

return {
  ...options,
  externals: [],
  optimization: {
    minimizer: [
      new TerserPlugin({
        terserOptions: {
          keep_classnames: true,
        },
      }),
    ],
  },
  output: {
    ...options.output,
    libraryTarget: 'commonjs2',
  },
  // ... остальная часть конфигурации
};
```

#### Использование автономного приложения

В качестве альтернативы, если вы хотите сохранить функцию максимально лёгкой и вам не нужны возможности, связанные с HTTP (маршрутизация, а также guard'ы, interceptor'ы, pipe'ы и т. д.),
можно воспользоваться `NestFactory.createApplicationContext` (как упоминалось ранее) вместо запуска полноценного HTTP-сервера (и `express` «под капотом»):

**`main.ts`**

```typescript
import { HttpStatus } from '@nestjs/common';
import { NestFactory } from '@nestjs/core';
import { Callback, Context, Handler } from 'aws-lambda';
import { AppModule } from './app.module';
import { AppService } from './app.service';

export const handler: Handler = async (
  event: any,
  context: Context,
  callback: Callback,
) => {
  const appContext = await NestFactory.createApplicationContext(AppModule);
  const appService = appContext.get(AppService);

  return {
    body: appService.getHello(),
    statusCode: HttpStatus.OK,
  };
};
```

> **Подсказка** Учтите, что `NestFactory.createApplicationContext` не оборачивает методы контроллеров enhancer'ами (guard'ами, interceptor'ами и т. д.). Для этого нужно использовать метод `NestFactory.create`.

Объект `event` можно также передать, скажем, в провайдер `EventsService`, который обработает его и вернёт соответствующее значение (в зависимости от входных данных и вашей бизнес-логики).

```typescript
export const handler: Handler = async (
  event: any,
  context: Context,
  callback: Callback,
) => {
  const appContext = await NestFactory.createApplicationContext(AppModule);
  const eventsService = appContext.get(EventsService);
  return eventsService.process(event);
};
```
