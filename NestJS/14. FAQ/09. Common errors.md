### Типичные ошибки

В процессе разработки на NestJS вы можете столкнуться с различными ошибками, пока осваиваете фреймворк.

#### Ошибка «Cannot resolve dependency»

> **Подсказка** Обратите внимание на [NestJS Devtools](<../15. Devtools/01. Overview.md#разбор-ошибки-cannot-resolve-dependency>) — они помогут без труда разобраться с ошибкой «Cannot resolve dependency».

Пожалуй, самое частое сообщение об ошибке связано с тем, что Nest не может разрешить зависимости провайдера. Обычно оно выглядит примерно так:

```bash
Nest can't resolve dependencies of the <provider> (?). Please make sure that the argument <unknown_token> at index [<index>] is available in the <module> context.

Potential solutions:
- Is <module> a valid NestJS module?
- If <unknown_token> is a provider, is it part of the current <module>?
- If <unknown_token> is exported from a separate @Module, is that module imported within <module>?
  @Module({
    imports: [ /* the Module containing <unknown_token> */ ]
  })
```

Чаще всего причина этой ошибки — отсутствие `<provider>` в массиве `providers` модуля. Убедитесь, что провайдер действительно находится в массиве `providers` и что вы следуете [стандартным практикам работы с провайдерами NestJS](<../03. Fundamentals/01. Custom providers.md#основы-di>).

Есть несколько распространённых подводных камней. Один из них — помещение провайдера в массив `imports`. В этом случае в сообщении об ошибке имя провайдера окажется на месте `<module>`.

Если вы столкнулись с этой ошибкой при разработке, посмотрите на модуль, упомянутый в сообщении, и на его `providers`. Для каждого провайдера из массива `providers` убедитесь, что у модуля есть доступ ко всем его зависимостям. Часто `providers` дублируются в «модуле функциональности» и в «корневом модуле», из-за чего Nest пытается создать экземпляр провайдера дважды. Скорее всего, вместо дублирования следует добавить модуль, содержащий `<provider>`, в массив `imports` «корневого модуля».

Если `<unknown_token>` в примере выше — это `dependency`, возможно, у вас циклический импорт файлов. Это не то же самое, что [циклическая зависимость](<09. Common errors.md#ошибка-circular-dependency>), описанная ниже: здесь провайдеры не зависят друг от друга в конструкторах, а просто два файла импортируют друг друга. Типичный случай — файл модуля объявляет токен и импортирует провайдер, а провайдер импортирует константу-токен из файла модуля. Если вы используете barrel-файлы, убедитесь, что импорты через них тоже не создают таких циклических импортов.

Если `<unknown_token>` в примере выше — это `Object`, значит, вы внедряете зависимость по типу/интерфейсу без подходящего токена провайдера. Чтобы это исправить, убедитесь, что:

1. вы импортируете ссылку на класс либо используете пользовательский токен с декоратором `@Inject()`. Прочитайте [страницу о пользовательских провайдерах](<../03. Fundamentals/01. Custom providers.md>);
2. для провайдеров на основе классов вы импортируете конкретные классы, а не только тип через синтаксис [`import type ...`](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-3-8.html#type-only-imports-and-export).

Также убедитесь, что вы не внедрили провайдер сам в себя, — в NestJS самовнедрение не допускается. В этом случае `<unknown_token>`, скорее всего, будет совпадать с `<provider>`.

Если вы работаете в **монорепозитории**, вы можете столкнуться с той же ошибкой, но в роли `<unknown_token>` будет выступать провайдер ядра `ModuleRef`:

```bash
Nest can't resolve dependencies of the <provider> (?).
Please make sure that the argument ModuleRef at index [<index>] is available in the <module> context.
...
```

Обычно это происходит, когда проект загружает два Node-модуля пакета `@nestjs/core`, например так:

```text
.
├── package.json
├── apps
│   └── api
│       └── node_modules
│           └── @nestjs/bull
│               └── node_modules
│                   └── @nestjs/core
└── node_modules
    ├── (other packages)
    └── @nestjs/core
```

Решения:

- Для рабочих пространств **Yarn** используйте [возможность nohoist](https://classic.yarnpkg.com/blog/2018/02/15/nohoist), чтобы предотвратить подъём пакета `@nestjs/core`.
- Для рабочих пространств **pnpm** укажите `@nestjs/core` в peerDependencies другого модуля и добавьте `"dependenciesMeta": {"other-module-name": {"injected": true }}` в package.json приложения, где этот модуль импортируется. См.: [dependenciesmetainjected](https://pnpm.io/package_json#dependenciesmetainjected)

#### Ошибка «Circular dependency»

Иногда бывает трудно избежать [циклических зависимостей](<../03. Fundamentals/05. Circular dependency.md>) в приложении. Чтобы помочь Nest их разрешить, придётся предпринять некоторые шаги. Ошибки, возникающие из-за циклических зависимостей, выглядят так:

```bash
Nest cannot create the <module> instance.
The module at index [<index>] of the <module> "imports" array is undefined.

Potential causes:
- A circular dependency between modules. Use forwardRef() to avoid it. Read more: https://docs.nestjs.com/fundamentals/circular-dependency
- The module at index [<index>] is of type "undefined". Check your import statements and the type of the module.

Scope [<module_import_chain>]
# example chain AppModule -> FooModule
```

Циклические зависимости могут возникать как из-за взаимной зависимости провайдеров, так и из-за того, что файлы TypeScript зависят друг от друга ради констант — например, когда константы экспортируются из файла модуля и импортируются в файле сервиса. В последнем случае рекомендуется вынести константы в отдельный файл. В первом случае следуйте руководству по циклическим зависимостям и убедитесь, что и модули, **и** провайдеры помечены `forwardRef`.

#### Отладка ошибок зависимостей

Помимо ручной проверки корректности зависимостей, начиная с Nest 8.1.0 вы можете задать переменной окружения `NEST_DEBUG` строку, приводящуюся к истинному значению, и получить дополнительную отладочную информацию в логах, пока Nest разрешает все зависимости приложения.

На изображении выше жёлтым выделен класс-хост внедряемой зависимости, синим — имя внедряемой зависимости или её токен внедрения, а фиолетовым — модуль, в котором зависимость ищется. С их помощью обычно удаётся проследить процесс разрешения зависимостей и понять, что происходит и почему возникают проблемы с внедрением зависимостей.

#### Бесконечный цикл «File change detected»

С этой проблемой могут столкнуться пользователи Windows с TypeScript версии 4.9 и выше.
Она возникает при попытке запустить приложение в режиме наблюдения, например через `npm run start:dev`, и проявляется как бесконечный цикл сообщений в логе:

```bash
XX:XX:XX AM - File change detected. Starting incremental compilation...
XX:XX:XX AM - Found 0 errors. Watching for file changes.
```

Когда вы запускаете приложение в режиме наблюдения с помощью NestJS CLI, под капотом вызывается `tsc --watch`, а начиная с версии 4.9 TypeScript использует [новую стратегию](https://devblogs.microsoft.com/typescript/announcing-typescript-4-9/#file-watching-now-uses-file-system-events) обнаружения изменений в файлах, которая, вероятно, и является причиной проблемы.
Чтобы её устранить, добавьте в файл tsconfig.json настройку после опции `"compilerOptions"`:

```bash
"watchOptions": {
    "watchFile": "fixedPollingInterval"
  }
```

Это указывает TypeScript использовать для отслеживания изменений в файлах опрос (polling) вместо событий файловой системы (нового способа по умолчанию), который на некоторых машинах может вызывать проблемы.
Подробнее об опции `"watchFile"` можно прочитать в [документации TypeScript](https://www.typescriptlang.org/tsconfig#watch-watchDirectory).
