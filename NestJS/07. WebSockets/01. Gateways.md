### Шлюзы

Большинство концепций, рассмотренных в других разделах этой документации, — внедрение зависимостей, декораторы, фильтры исключений, pipes, guards и interceptors — в равной степени применимы и к шлюзам. Где возможно, Nest абстрагирует детали реализации, чтобы одни и те же компоненты работали на HTTP-платформах, в WebSockets и в микросервисах. Этот раздел посвящён тем аспектам Nest, которые специфичны для WebSockets.

В Nest шлюз — это просто класс, помеченный декоратором `@WebSocketGateway()`. Технически шлюзы не зависят от платформы, что делает их совместимыми с любой библиотекой WebSockets, для которой создан адаптер. «Из коробки» поддерживаются две WS-платформы: [socket.io](https://github.com/socketio/socket.io) и [ws](https://github.com/websockets/ws). Вы можете выбрать ту, что лучше подходит под ваши задачи. Кроме того, можно написать собственный адаптер, следуя этому [руководству](<06. Adapters.md>).

<figure><img class="illustrative-image" src="https://docs.nestjs.com/assets/Gateways_1.png" /></figure>

> **Подсказка** Шлюзы можно рассматривать как [провайдеры](<../02. Overview/03. Providers.md>): это значит, что они могут внедрять зависимости через конструктор класса. Также сами шлюзы могут быть внедрены в другие классы (провайдеры и контроллеры).

#### Установка

Чтобы начать создавать приложения на базе WebSockets, сначала установите нужный пакет:

```bash
$ npm i --save @nestjs/websockets @nestjs/platform-socket.io
```

#### Обзор

Как правило, каждый шлюз слушает тот же порт, что и **HTTP-сервер**, — если только ваше приложение не является веб-приложением или вы не поменяли порт вручную. Это поведение по умолчанию можно изменить, передав аргумент в декоратор `@WebSocketGateway(80)`, где `80` — выбранный номер порта. Также можно задать используемое шлюзом [пространство имён](https://socket.io/docs/v4/namespaces/) следующей конструкцией:

```typescript
@WebSocketGateway(80, { namespace: 'events' })
```

> **Предупреждение** Экземпляры шлюзов не создаются, пока на них не сослались в массиве providers существующего модуля.

Любую поддерживаемую [опцию](https://socket.io/docs/v4/server-options/) можно передать в конструктор сокета вторым аргументом декоратора `@WebSocketGateway()`, как показано ниже:

```typescript
@WebSocketGateway(81, { transports: ['websocket'] })
```

Теперь шлюз слушает соединения, но мы ещё не подписались ни на какие входящие сообщения. Создадим обработчик, который подпишется на сообщения `events` и ответит пользователю ровно теми же данными.

**`events.gateway.ts`**

```typescript
@SubscribeMessage('events')
handleEvent(@MessageBody() data: string): string {
  return data;
}
```

> **Подсказка** Декораторы `@SubscribeMessage()` и `@MessageBody()` импортируются из пакета `@nestjs/websockets`.

Созданный шлюз можно зарегистрировать в нашем модуле.

**`events.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { EventsGateway } from './events.gateway';

@Module({
  providers: [EventsGateway]
})
export class EventsModule {}
```

Также в декоратор можно передать ключ свойства, чтобы извлечь его из тела входящего сообщения:

**`events.gateway.ts`**

```typescript
@SubscribeMessage('events')
handleEvent(@MessageBody('id') id: number): number {
  // id === messageBody.id
  return id;
}
```

Если вы предпочитаете обходиться без декораторов, следующий код функционально эквивалентен:

**`events.gateway.ts`**

```typescript
@SubscribeMessage('events')
handleEvent(client: Socket, data: string): string {
  return data;
}
```

В примере выше функция `handleEvent()` принимает два аргумента. Первый — специфичный для платформы [экземпляр сокета](https://socket.io/docs/v4/server-api/#socket), второй — данные, полученные от клиента. Такой подход, однако, не рекомендуется, поскольку требует создавать заглушку для экземпляра `socket` в каждом юнит-тесте.

При получении сообщения `events` обработчик отправляет подтверждение с теми же данными, что пришли по сети. Кроме того, сообщения можно отправлять способом, специфичным для библиотеки, — например, методом `client.emit()`. Чтобы получить доступ к экземпляру подключённого сокета, используйте декоратор `@ConnectedSocket()`.

**`events.gateway.ts`**

```typescript
@SubscribeMessage('events')
handleEvent(
  @MessageBody() data: string,
  @ConnectedSocket() client: Socket,
): string {
  return data;
}
```

> **Подсказка** Декоратор `@ConnectedSocket()` импортируется из пакета `@nestjs/websockets`.

Однако в этом случае вы не сможете задействовать interceptors. Если отвечать пользователю не нужно, просто опустите оператор `return` (или явно верните «ложное» значение, например `undefined`).

Теперь, когда клиент отправляет сообщение так:

```typescript
socket.emit('events', { name: 'Nest' });
```

будет выполнен метод `handleEvent()`. Чтобы слушать сообщения, отправленные из этого обработчика, клиент должен подключить соответствующий обработчик подтверждения:

```typescript
socket.emit('events', { name: 'Nest' }, (data) => console.log(data));
```

Возврат значения из обработчика сообщения неявно отправляет подтверждение, но в более сложных сценариях часто требуется прямой контроль над колбэком подтверждения.

Декоратор параметра `@Ack()` позволяет внедрить функцию-колбэк `ack` прямо в обработчик сообщения.
Без этого декоратора колбэк передаётся третьим аргументом метода.

**`events.gateway.ts`**

```typescript
@SubscribeMessage('events')
handleEvent(
  @MessageBody() data: string,
  @Ack() ack: (response: { status: string; data: string }) => void,
) {
  ack({ status: 'received', data });
}
```

#### Несколько ответов

Подтверждение отправляется только один раз. Более того, оно не поддерживается нативной реализацией WebSockets. Чтобы обойти это ограничение, можно вернуть объект из двух свойств: `event` — имя отправляемого события и `data` — данные, которые нужно передать клиенту.

**`events.gateway.ts`**

```typescript
@SubscribeMessage('events')
handleEvent(@MessageBody() data: unknown): WsResponse<unknown> {
  const event = 'events';
  return { event, data };
}
```

> **Подсказка** Интерфейс `WsResponse` импортируется из пакета `@nestjs/websockets`.

> **Предупреждение** Если поле `data` полагается на `ClassSerializerInterceptor`, вам следует возвращать экземпляр класса, реализующего `WsResponse`, поскольку этот interceptor игнорирует ответы в виде обычных объектов JavaScript.

Чтобы получать входящие ответы, клиент должен добавить ещё один обработчик события.

```typescript
socket.on('events', (data) => console.log(data));
```

#### Асинхронные ответы

Обработчики сообщений могут отвечать как синхронно, так и **асинхронно**. Соответственно, методы с `async` поддерживаются. Обработчик сообщения также может вернуть `Observable` — в этом случае значения результата будут отправляться до тех пор, пока поток не завершится.

**`events.gateway.ts`**

```typescript
@SubscribeMessage('events')
onEvent(@MessageBody() data: unknown): Observable<WsResponse<number>> {
  const event = 'events';
  const response = [1, 2, 3];

  return from(response).pipe(
    map(data => ({ event, data })),
  );
}
```

В примере выше обработчик сообщения ответит **3 раза** — по одному разу на каждый элемент массива.

#### Хуки жизненного цикла

Доступны 3 полезных хука жизненного цикла. У каждого из них есть соответствующий интерфейс; они описаны в таблице ниже:

<table>
  <tr>
    <td>
      <code>OnGatewayInit</code>
    </td>
    <td>
      Обязывает реализовать метод <code>afterInit()</code>. Принимает аргументом специфичный для библиотеки экземпляр сервера
      (и при необходимости распространяет остальные аргументы).
    </td>
  </tr>
  <tr>
    <td>
      <code>OnGatewayConnection</code>
    </td>
    <td>
      Обязывает реализовать метод <code>handleConnection()</code>. Принимает аргументом специфичный для библиотеки экземпляр
      клиентского сокета.
    </td>
  </tr>
  <tr>
    <td>
      <code>OnGatewayDisconnect</code>
    </td>
    <td>
      Обязывает реализовать метод <code>handleDisconnect()</code>. Принимает аргументом специфичный для библиотеки экземпляр
      клиентского сокета.
    </td>
  </tr>
</table>

> **Подсказка** Каждый интерфейс жизненного цикла доступен из пакета `@nestjs/websockets`.

#### Сервер и пространство имён

Иногда может понадобиться прямой доступ к нативному, **специфичному для платформы** экземпляру сервера. Ссылка на этот объект передаётся аргументом в метод `afterInit()` (интерфейс `OnGatewayInit`). Другой вариант — использовать декоратор `@WebSocketServer()`.

```typescript
@WebSocketServer()
server: Server;
```

Также можно получить соответствующее пространство имён через атрибут `namespace`:

```typescript
@WebSocketGateway({ namespace: 'my-namespace' })
export class EventsGateway {
  @WebSocketServer()
  namespace: Namespace;
}
```

Декоратор `@WebSocketServer()` внедряет экземпляр сервера, обращаясь к метаданным, сохранённым декоратором `@WebSocketGateway()`. Если вы передали в `@WebSocketGateway()` опцию namespace, декоратор `@WebSocketServer()` вернёт экземпляр `Namespace` вместо экземпляра `Server`.

> **Важно** Декоратор `@WebSocketServer()` импортируется из пакета `@nestjs/websockets`.

Nest автоматически присвоит этому свойству экземпляр сервера, как только тот будет готов к использованию.
