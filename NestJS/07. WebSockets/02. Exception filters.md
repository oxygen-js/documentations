### Фильтры исключений

Единственное отличие HTTP-слоя [фильтров исключений](<../02. Overview/06. Exception filters.md>) от соответствующего слоя веб-сокетов состоит в том, что вместо `HttpException` следует выбрасывать `WsException`.

```typescript
throw new WsException('Invalid credentials.');
```

> **Подсказка** Класс `WsException` импортируется из пакета `@nestjs/websockets`.

В примере выше Nest обработает выброшенное исключение и отправит сообщение `exception` со следующей структурой:

```typescript
{
  status: 'error',
  message: 'Invalid credentials.'
}
```

#### Фильтры

Фильтры исключений для веб-сокетов работают точно так же, как HTTP-фильтры исключений. В примере ниже используется вручную созданный фильтр уровня метода. Как и в HTTP-приложениях, вы можете применять фильтры уровня шлюза (то есть добавить декоратор `@UseFilters()` к классу шлюза).

```typescript
@UseFilters(new WsExceptionFilter())
@SubscribeMessage('events')
onEvent(client, data: any): WsResponse<any> {
  const event = 'events';
  return { event, data };
}
```

#### Наследование

Обычно вы создаёте полностью настроенные под требования приложения фильтры исключений. Однако бывают случаи, когда достаточно просто расширить **базовый фильтр исключений** и переопределить поведение в зависимости от тех или иных условий.

Чтобы делегировать обработку исключения базовому фильтру, нужно унаследоваться от `BaseWsExceptionFilter` и вызвать унаследованный метод `catch()`.

```typescript
import { Catch, ArgumentsHost } from '@nestjs/common';
import { BaseWsExceptionFilter } from '@nestjs/websockets';

@Catch()
export class AllExceptionsFilter extends BaseWsExceptionFilter {
  catch(exception: unknown, host: ArgumentsHost) {
    super.catch(exception, host);
  }
}
```

Приведённая реализация — лишь каркас, демонстрирующий подход. В вашей реализации расширенного фильтра исключений будет собственная **бизнес-логика** (например, обработка различных условий).
