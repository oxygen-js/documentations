### Скалярные типы

У объектного типа GraphQL есть имя и поля, но в какой-то момент эти поля должны разрешиться в конкретные данные. Здесь и вступают в игру скалярные типы: они представляют собой листья запроса (подробнее [здесь](https://graphql.org/learn/schema/#scalar-types)). GraphQL включает следующие типы по умолчанию: `Int`, `Float`, `String`, `Boolean` и `ID`. Помимо этих встроенных типов вам может понадобиться поддержка собственных атомарных типов данных (например, `Date`).

#### Подход code first

Подход code first поставляется с пятью скалярными типами, три из которых — просто псевдонимы существующих типов GraphQL.

- `ID` (псевдоним `GraphQLID`) — уникальный идентификатор, часто используемый для повторного получения объекта или как ключ кэша
- `Int` (псевдоним `GraphQLInt`) — 32-битное целое со знаком
- `Float` (псевдоним `GraphQLFloat`) — число с плавающей точкой двойной точности со знаком
- `GraphQLISODateTime` — строка с датой и временем в UTC (используется по умолчанию для представления типа `Date`)
- `GraphQLTimestamp` — целое число со знаком, представляющее дату и время как количество миллисекунд от начала эпохи UNIX

Для представления типа `Date` по умолчанию используется `GraphQLISODateTime` (например, `2019-12-03T09:54:33Z`). Чтобы вместо него использовать `GraphQLTimestamp`, задайте свойству `dateScalarMode` объекта `buildSchemaOptions` значение `'timestamp'`:

```typescript
GraphQLModule.forRoot({
  buildSchemaOptions: {
    dateScalarMode: 'timestamp',
  }
}),
```

Аналогично, для представления типа `number` по умолчанию используется `GraphQLFloat`. Чтобы вместо него использовать `GraphQLInt`, задайте свойству `numberScalarMode` объекта `buildSchemaOptions` значение `'integer'`:

```typescript
GraphQLModule.forRoot({
  buildSchemaOptions: {
    numberScalarMode: 'integer',
  }
}),
```

Кроме того, вы можете создавать собственные скалярные типы.

#### Переопределение стандартного скалярного типа

Чтобы создать собственную реализацию скалярного типа `Date`, достаточно объявить новый класс.

```typescript
import { Scalar, CustomScalar } from '@nestjs/graphql';
import { Kind, ValueNode } from 'graphql';

@Scalar('Date', () => Date)
export class DateScalar implements CustomScalar<number, Date> {
  description = 'Date custom scalar type';

  parseValue(value: number): Date {
    return new Date(value); // значение от клиента
  }

  serialize(value: Date): number {
    return value.getTime(); // значение, отправляемое клиенту
  }

  parseLiteral(ast: ValueNode): Date {
    if (ast.kind === Kind.INT) {
      return new Date(ast.value);
    }
    return null;
  }
}
```

После этого зарегистрируйте `DateScalar` как провайдер.

```typescript
@Module({
  providers: [DateScalar],
})
export class CommonModule {}
```

Теперь в наших классах можно использовать тип `Date`.

```typescript
@Field()
creationDate: Date;
```

#### Импорт стороннего скалярного типа

Чтобы воспользоваться сторонним скалярным типом, импортируйте его и зарегистрируйте как резолвер. Для примера возьмём пакет `graphql-type-json`. Этот npm-пакет определяет скалярный тип GraphQL `JSON`.

Начните с установки пакета:

```bash
$ npm i --save graphql-type-json
```

После установки передадим собственный резолвер в метод `forRoot()`:

```typescript
import GraphQLJSON from 'graphql-type-json';

@Module({
  imports: [
    GraphQLModule.forRoot({
      resolvers: { JSON: GraphQLJSON },
    }),
  ],
})
export class AppModule {}
```

Теперь в наших классах можно использовать тип `JSON`.

```typescript
@Field(() => GraphQLJSON)
info: JSON;
```

Набор полезных скалярных типов вы найдёте в пакете [graphql-scalars](https://www.npmjs.com/package/graphql-scalars).

#### Создание собственного скалярного типа

Чтобы определить собственный скалярный тип, создайте новый экземпляр `GraphQLScalarType`. Мы создадим скалярный тип `UUID`.

```typescript
const regex = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i;

function validate(uuid: unknown): string | never {
  if (typeof uuid !== 'string' || !regex.test(uuid)) {
    throw new Error('invalid uuid');
  }
  return uuid;
}

export const CustomUuidScalar = new GraphQLScalarType({
  name: 'UUID',
  description: 'A simple UUID parser',
  serialize: (value) => validate(value),
  parseValue: (value) => validate(value),
  parseLiteral: (ast) => validate(ast.value),
});
```

Передаём собственный резолвер в метод `forRoot()`:

```typescript
@Module({
  imports: [
    GraphQLModule.forRoot({
      resolvers: { UUID: CustomUuidScalar },
    }),
  ],
})
export class AppModule {}
```

Теперь в наших классах можно использовать тип `UUID`.

```typescript
@Field(() => CustomUuidScalar)
uuid: string;
```

#### Подход schema first

Чтобы определить собственный скалярный тип (подробнее о скалярных типах [здесь](https://www.apollographql.com/docs/graphql-tools/scalars.html)), создайте определение типа и отдельный резолвер. Здесь, как и в официальной документации, для примера возьмём пакет `graphql-type-json`. Этот npm-пакет определяет скалярный тип GraphQL `JSON`.

Начните с установки пакета:

```bash
$ npm i --save graphql-type-json
```

После установки передадим собственный резолвер в метод `forRoot()`:

```typescript
import GraphQLJSON from 'graphql-type-json';

@Module({
  imports: [
    GraphQLModule.forRoot({
      typePaths: ['./**/*.graphql'],
      resolvers: { JSON: GraphQLJSON },
    }),
  ],
})
export class AppModule {}
```

Теперь скалярный тип `JSON` можно использовать в наших определениях типов:

```graphql
scalar JSON

type Foo {
  field: JSON
}
```

Ещё один способ определить скалярный тип — создать простой класс. Допустим, мы хотим дополнить нашу схему типом `Date`.

```typescript
import { Scalar, CustomScalar } from '@nestjs/graphql';
import { Kind, ValueNode } from 'graphql';

@Scalar('Date')

    return value.getTime(); // значение, отправляемое клиенту
  }

  parseLiteral(ast: ValueNode): Date {
    if (ast.kind === Kind.INT) {
      return new Date(ast.value);
    }
    return null;
  }
}
```

После этого зарегистрируйте `DateScalar` как провайдер.

```typescript
@Module({
  providers: [DateScalar],
})
export class CommonModule {}
```

Теперь скалярный тип `Date` можно использовать в определениях типов.

```graphql
scalar Date
```

По умолчанию для всех скалярных типов генерируется TypeScript-определение `any`, что не назовёшь типобезопасным.
Однако вы можете настроить, как Nest генерирует типы для ваших скалярных типов, указав это при генерации:

```typescript
import { GraphQLDefinitionsFactory } from '@nestjs/graphql';
import { join } from 'path';

const definitionsFactory = new GraphQLDefinitionsFactory();

definitionsFactory.generate({
  typePaths: ['./src/**/*.graphql'],
  path: join(process.cwd(), 'src/graphql.ts'),
  outputAs: 'class',
  defaultScalarType: 'unknown',
  customScalarTypeMapping: {
    DateTime: 'Date',
    BigNumber: '_BigNumber',
  },
  additionalHeader: "import _BigNumber from 'bignumber.js'",
});
```

> **Подсказка** Как вариант, можно указать ссылку на тип, например: `DateTime: Date`. В этом случае `GraphQLDefinitionsFactory` извлечёт свойство name указанного типа (`Date.name`) для генерации TS-определений. Учтите: для типов, не являющихся встроенными (пользовательских), потребуется добавить оператор импорта.

Теперь, имея следующие пользовательские скалярные типы GraphQL:

```graphql
scalar DateTime
scalar BigNumber
scalar Payload
```

мы получим в `src/graphql.ts` такие сгенерированные TypeScript-определения:

```typescript
import _BigNumber from 'bignumber.js';

export type DateTime = Date;
export type BigNumber = _BigNumber;
export type Payload = unknown;
```

Здесь мы использовали свойство `customScalarTypeMapping`, чтобы задать соответствие типов, которые хотим объявить для наших скалярных типов. Мы
также указали свойство `additionalHeader`, чтобы добавить импорты, необходимые для этих определений типов. Наконец, мы задали
`defaultScalarType` равным `'unknown'`, чтобы все скалярные типы, не перечисленные в `customScalarTypeMapping`, получали псевдоним
`unknown` вместо `any` ([TypeScript рекомендует](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-3-0.html#new-unknown-top-type) использовать его начиная с версии 3.0 ради большей типобезопасности).

> **Подсказка** Обратите внимание, что мы импортировали `_BigNumber` из `bignumber.js`; это сделано, чтобы избежать [циклических ссылок между типами](https://github.com/Microsoft/TypeScript/issues/12525#issuecomment-263166239).
