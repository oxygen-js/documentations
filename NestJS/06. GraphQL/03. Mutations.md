### Мутации

Большинство обсуждений GraphQL сосредоточено на получении данных, но любой полноценной платформе данных нужен и способ изменять данные на сервере. В REST любой запрос технически может привести к побочным эффектам на сервере, однако хорошая практика предписывает не изменять данные в GET-запросах. В GraphQL ситуация похожа: технически любой query можно реализовать так, чтобы он записывал данные. Тем не менее, как и в REST, рекомендуется соблюдать соглашение, по которому все операции записи выполняются явно через мутацию (подробнее [здесь](https://graphql.org/learn/queries/#mutations)).

В официальной документации [Apollo](https://www.apollographql.com/docs/graphql-tools/generate-schema.html) в качестве примера используется мутация `upvotePost()`. Она реализует метод, увеличивающий значение свойства `votes` у поста. Чтобы создать аналогичную мутацию в Nest, воспользуемся декоратором `@Mutation()`.

#### Code first

Добавим ещё один метод в `AuthorResolver`, использованный в предыдущем разделе (см. [резолверы](<02. Resolvers.md>)).

```typescript
@Mutation(() => Post)
async upvotePost(@Args({ name: 'postId', type: () => Int }) postId: number) {
  return this.postsService.upvoteById({ id: postId });
}
```

> **Подсказка** Все декораторы (например, `@Resolver`, `@ResolveField`, `@Args` и т. д.) экспортируются из пакета `@nestjs/graphql`.

В результате будет сгенерирована следующая часть GraphQL-схемы в SDL:

```graphql
type Mutation {
  upvotePost(postId: Int!): Post
}
```

Метод `upvotePost()` принимает аргумент `postId` (`Int`) и возвращает обновлённую сущность `Post`. По причинам, описанным в разделе [резолверы](<02. Resolvers.md>), ожидаемый тип приходится указывать явно.

Если мутация должна принимать объект в качестве аргумента, можно создать **input-тип**. Input-тип — это особая разновидность объектного типа, который можно передавать в качестве аргумента (подробнее [здесь](https://graphql.org/learn/schema/#input-types)). Для объявления input-типа используйте декоратор `@InputType()`.

```typescript
import { InputType, Field } from '@nestjs/graphql';

@InputType()
export class UpvotePostInput {
  @Field()
  postId: number;
}
```

> **Подсказка** Декоратор `@InputType()` принимает объект настроек в качестве аргумента, поэтому вы можете, например, задать описание input-типа. Учтите, что из-за ограничений системы рефлексии метаданных TypeScript вам придётся либо указывать тип вручную с помощью декоратора `@Field`, либо использовать [плагин CLI](<14. CLI Plugin.md>).

Затем этот тип можно использовать в классе резолвера:

```typescript
@Mutation(() => Post)
async upvotePost(
  @Args('upvotePostData') upvotePostData: UpvotePostInput,
) {}
```

#### Schema first

Расширим наш `AuthorResolver`, использованный в предыдущем разделе (см. [резолверы](<02. Resolvers.md>)).

```typescript
@Mutation()
async upvotePost(@Args('postId') postId: number) {
  return this.postsService.upvoteById({ id: postId });
}
```

Обратите внимание: выше мы исходили из того, что бизнес-логика вынесена в `PostsService` (получение поста и увеличение его свойства `votes`). Логика внутри класса `PostsService` может быть сколь угодно простой или сложной. Основная цель этого примера — показать, как резолверы могут взаимодействовать с другими провайдерами.

Последний шаг — добавить нашу мутацию в существующее определение типов.

```graphql
type Author {
  id: Int!
  firstName: String
  lastName: String
  posts: [Post]
}

type Post {
  id: Int!
  title: String
  votes: Int
}

type Query {
  author(id: Int!): Author
}

type Mutation {
  upvotePost(postId: Int!): Post
}
```

Теперь мутацию `upvotePost(postId: Int!): Post` можно вызывать как часть GraphQL API вашего приложения.
