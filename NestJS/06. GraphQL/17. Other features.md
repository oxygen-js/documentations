### Прочие возможности

В мире GraphQL много споров о том, как решать такие задачи, как **аутентификация** или **побочные эффекты** операций. Обрабатывать их внутри бизнес-логики? Использовать функцию высшего порядка, дополняющую запросы и мутации логикой авторизации? Или применять [директивы схемы](https://www.apollographql.com/docs/apollo-server/schema/directives/)? Единого универсального ответа на эти вопросы нет.

Nest помогает решать такие задачи своими кроссплатформенными возможностями — [guards](<../02. Overview/08. Guards.md>) и [interceptors](<../02. Overview/09. Interceptors.md>). Философия здесь в том, чтобы сократить дублирование и дать инструменты для создания хорошо структурированных, читаемых и единообразных приложений.

#### Обзор

Стандартные [guards](<../02. Overview/08. Guards.md>), [interceptors](<../02. Overview/09. Interceptors.md>), [фильтры](<../02. Overview/06. Exception filters.md>) и [pipes](<../02. Overview/07. Pipes.md>) применяются в GraphQL точно так же, как в любом RESTful-приложении. Кроме того, вы легко можете создавать собственные декораторы, воспользовавшись возможностью [пользовательских декораторов](<../02. Overview/10. Custom decorators.md>). Рассмотрим пример обработчика GraphQL-запроса.

```typescript
@Query('author')
@UseGuards(AuthGuard)
async getAuthor(@Args('id', ParseIntPipe) id: number) {
  return this.authorsService.findOneById(id);
}
```

Как видите, GraphQL работает с guards и pipes так же, как обработчики HTTP REST. Благодаря этому логику аутентификации можно вынести в guard и даже переиспользовать один и тот же класс guard сразу в REST- и GraphQL-интерфейсах API. Аналогично interceptors одинаково работают в обоих типах приложений:

```typescript
@Mutation()
@UseInterceptors(EventsInterceptor)
async upvotePost(@Args('postId') postId: number) {
  return this.postsService.upvoteById({ id: postId });
}
```

#### Контекст выполнения

Поскольку во входящем запросе GraphQL приходят данные другого вида, [контекст выполнения](<../03. Fundamentals/08. Execution context.md>), который получают guards и interceptors, в GraphQL несколько отличается от REST. У резолверов GraphQL свой набор аргументов: `root`, `args`, `context` и `info`. Поэтому guards и interceptors должны преобразовать обобщённый `ExecutionContext` в `GqlExecutionContext`. Делается это просто:

```typescript
import { CanActivate, ExecutionContext, Injectable } from '@nestjs/common';
import { GqlExecutionContext } from '@nestjs/graphql';

@Injectable()
export class AuthGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const ctx = GqlExecutionContext.create(context);
    return true;
  }
}
```

Объект контекста GraphQL, возвращаемый `GqlExecutionContext.create()`, предоставляет метод **get** для каждого аргумента резолвера GraphQL (`getArgs()`, `getContext()` и т. д.). После преобразования мы можем легко получить любой GraphQL-аргумент текущего запроса.

#### Фильтры исключений

Стандартные [фильтры исключений](<../02. Overview/06. Exception filters.md>) Nest совместимы и с GraphQL-приложениями. Как и в случае с `ExecutionContext`, GraphQL-приложения должны преобразовать объект `ArgumentsHost` в объект `GqlArgumentsHost`.

```typescript
@Catch(HttpException)
export class HttpExceptionFilter implements GqlExceptionFilter {
  catch(exception: HttpException, host: ArgumentsHost) {
    const gqlHost = GqlArgumentsHost.create(host);
    return exception;
  }
}
```

> **Подсказка** И `GqlExceptionFilter`, и `GqlArgumentsHost` импортируются из пакета `@nestjs/graphql`.

Обратите внимание: в отличие от REST, здесь для формирования ответа не используется нативный объект `response`.

#### Пользовательские декораторы

Как уже упоминалось, возможность [пользовательских декораторов](<../02. Overview/10. Custom decorators.md>) работает с резолверами GraphQL как ожидается.

```typescript
export const User = createParamDecorator(
  (data: unknown, ctx: ExecutionContext) =>
    GqlExecutionContext.create(ctx).getContext().user,
);
```

Пользовательский декоратор `@User()` применяется так:

```typescript
@Mutation()
async upvotePost(
  @User() user: UserEntity,
  @Args('postId') postId: number,
) {}
```

> **Подсказка** В примере выше мы исходили из того, что объект `user` помещён в контекст вашего GraphQL-приложения.

#### Запуск enhancers на уровне резолверов полей

В контексте GraphQL Nest не выполняет **enhancers** (общее название для interceptors, guards и фильтров) на уровне полей — [см. этот issue](https://github.com/nestjs/graphql/issues/320#issuecomment-511193229): они запускаются только для методов верхнего уровня `@Query()`/`@Mutation()`. Чтобы Nest выполнял interceptors, guards или фильтры для методов, помеченных `@ResolveField()`, задайте параметр `fieldResolverEnhancers` в `GqlModuleOptions`. Передайте ему список из `'interceptors'`, `'guards'` и/или `'filters'` — что нужно:

```typescript
GraphQLModule.forRoot({
  fieldResolverEnhancers: ['interceptors']
}),
```

> **Предупреждение** Включение enhancers для резолверов полей может привести к проблемам с производительностью, если вы возвращаете много записей и резолвер поля выполняется тысячи раз. Поэтому при включении `fieldResolverEnhancers` мы советуем пропускать выполнение тех enhancers, которые для ваших резолверов полей строго не необходимы. Сделать это можно с помощью следующей вспомогательной функции:

```typescript
export function isResolvingGraphQLField(context: ExecutionContext): boolean {
  if (context.getType<GqlContextType>() === 'graphql') {
    const gqlContext = GqlExecutionContext.create(context);
    const info = gqlContext.getInfo();
    const parentType = info.parentType.name;
    return parentType !== 'Query' && parentType !== 'Mutation';
  }
  return false;
}
```

#### Создание собственного драйвера

Nest «из коробки» предоставляет два официальных драйвера — `@nestjs/apollo` и `@nestjs/mercurius`, — а также API, позволяющий разработчикам создавать новые **собственные драйверы**. С помощью такого драйвера можно интегрировать любую GraphQL-библиотеку или расширить существующую интеграцию, добавив к ней дополнительные возможности.

Например, чтобы интегрировать пакет `express-graphql`, можно создать такой класс драйвера:

```typescript
import { AbstractGraphQLDriver, GqlModuleOptions } from '@nestjs/graphql';
import { graphqlHTTP } from 'express-graphql';

class ExpressGraphQLDriver extends AbstractGraphQLDriver {
  async start(options: GqlModuleOptions<any>): Promise<void> {
    options = await this.graphQlFactory.mergeWithSchema(options);

    const { httpAdapter } = this.httpAdapterHost;
    httpAdapter.use(
      '/graphql',
      graphqlHTTP({
        schema: options.schema,
        graphiql: true,
      }),
    );
  }

  async stop() {}
}
```

А затем использовать его так:

```typescript
GraphQLModule.forRoot({
  driver: ExpressGraphQLDriver,
});
```
