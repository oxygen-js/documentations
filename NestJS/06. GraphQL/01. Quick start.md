## Используем силу TypeScript и GraphQL

[GraphQL](https://graphql.org/) — мощный язык запросов для API и среда выполнения, которая обслуживает эти запросы на ваших существующих данных. Это изящный подход, решающий многие проблемы, типичные для REST API. Для общего понимания советуем прочитать это [сравнение](https://www.apollographql.com/blog/graphql-vs-rest) GraphQL и REST. GraphQL в связке с [TypeScript](https://www.typescriptlang.org/) помогает повысить типобезопасность ваших GraphQL-запросов, обеспечивая сквозную типизацию.

В этой главе мы предполагаем базовое знакомство с GraphQL и сосредоточимся на работе со встроенным модулем `@nestjs/graphql`. `GraphQLModule` можно настроить на использование сервера [Apollo](https://www.apollographql.com/) (через драйвер `@nestjs/apollo`) или [Mercurius](https://github.com/mercurius-js/mercurius) (через `@nestjs/mercurius`). Мы предоставляем официальные интеграции с этими проверенными GraphQL-пакетами, чтобы дать простой способ использовать GraphQL вместе с Nest (другие интеграции смотрите [здесь](<01. Quick start.md#сторонние-интеграции>)).

Вы также можете написать собственный драйвер (подробнее об этом [здесь](<17. Other features.md#создание-собственного-драйвера>)).

#### Установка

Начните с установки необходимых пакетов:

```bash
# For Express and Apollo (default)
$ npm i @nestjs/graphql @nestjs/apollo @apollo/server @as-integrations/express5 graphql

# For Fastify and Apollo
# npm i @nestjs/graphql @nestjs/apollo @apollo/server @as-integrations/fastify graphql

# For Fastify and Mercurius
# npm i @nestjs/graphql @nestjs/mercurius graphql mercurius
```

> **Предупреждение** Пакеты `@nestjs/graphql@>=9` и `@nestjs/apollo^10` совместимы с **Apollo v3** (подробности — в [руководстве по миграции](https://www.apollographql.com/docs/apollo-server/migration/) на Apollo Server 3), тогда как `@nestjs/graphql@^8` поддерживает только **Apollo v2** (например, пакет `apollo-server-express@2.x.x`).

#### Обзор

Nest предлагает два способа построения GraphQL-приложений: **code first** и **schema first**. Выбирайте тот, что вам удобнее. Большинство глав этого раздела о GraphQL разделены на две основные части: одну следует читать, если вы выбрали **code first**, другую — если **schema first**.

При подходе **code first** вы используете декораторы и классы TypeScript, чтобы сгенерировать соответствующую GraphQL-схему. Этот подход удобен, если вы предпочитаете работать исключительно с TypeScript и не переключаться между синтаксисами разных языков.

При подходе **schema first** источником истины служат файлы GraphQL SDL (Schema Definition Language). SDL — не зависящий от языка способ обмениваться файлами схемы между разными платформами. Nest автоматически генерирует определения TypeScript (классы или интерфейсы) на основе GraphQL-схем, избавляя вас от написания лишнего шаблонного кода.

#### Первые шаги с GraphQL и TypeScript

> **Подсказка** В следующих главах мы будем работать с пакетом `@nestjs/apollo`. Если вместо него вы хотите использовать `mercurius`, перейдите к [этому разделу](<01. Quick start.md#интеграция-с-mercurius>).

После установки пакетов можно импортировать `GraphQLModule` и настроить его статическим методом `forRoot()`.

```typescript
import { Module } from '@nestjs/common';
import { GraphQLModule } from '@nestjs/graphql';
import { ApolloDriver, ApolloDriverConfig } from '@nestjs/apollo';

@Module({
  imports: [
    GraphQLModule.forRoot<ApolloDriverConfig>({
      driver: ApolloDriver,
    }),
  ],
})
export class AppModule {}
```

> **Подсказка** Для интеграции с `mercurius` следует использовать `MercuriusDriver` и `MercuriusDriverConfig`. Оба экспортируются из пакета `@nestjs/mercurius`.

Метод `forRoot()` принимает объект параметров в качестве аргумента. Эти параметры передаются нижележащему экземпляру драйвера (подробнее о доступных настройках: [Apollo](https://www.apollographql.com/docs/apollo-server/api/apollo-server) и [Mercurius](https://github.com/mercurius-js/mercurius/blob/master/docs/api/options.md#plugin-options)). Например, если вы хотите отключить `playground` и выключить режим `debug` (для Apollo), передайте такие параметры:

```typescript
import { Module } from '@nestjs/common';
import { GraphQLModule } from '@nestjs/graphql';
import { ApolloDriver, ApolloDriverConfig } from '@nestjs/apollo';

@Module({
  imports: [
    GraphQLModule.forRoot<ApolloDriverConfig>({
      driver: ApolloDriver,
      playground: false,
    }),
  ],
})
export class AppModule {}
```

В этом случае параметры будут переданы в конструктор `ApolloServer`.

#### GraphQL playground

Playground — это графическая интерактивная браузерная IDE для GraphQL, по умолчанию доступная по тому же URL, что и сам GraphQL-сервер. Чтобы её открыть, нужен настроенный и запущенный базовый GraphQL-сервер. Чтобы увидеть её прямо сейчас, установите и соберите [рабочий пример отсюда](https://github.com/nestjs/nest/tree/master/sample/23-graphql-code-first). #### Code first

При подходе **code first** вы используете декораторы и классы TypeScript, чтобы сгенерировать соответствующую GraphQL-схему.

Чтобы применить подход code first, начните с добавления свойства `autoSchemaFile` в объект параметров:

```typescript
GraphQLModule.forRoot<ApolloDriverConfig>({
  driver: ApolloDriver,
  autoSchemaFile: join(process.cwd(), 'src/schema.gql'),
}),
```

Значение свойства `autoSchemaFile` — путь, по которому будет создана автоматически сгенерированная схема. Кроме того, схему можно генерировать на лету в памяти. Чтобы включить это, задайте `autoSchemaFile` значение `true`:

```typescript
GraphQLModule.forRoot<ApolloDriverConfig>({
  driver: ApolloDriver,
  autoSchemaFile: true,
}),
```

По умолчанию типы в сгенерированной схеме идут в том порядке, в каком они определены в подключённых модулях. Чтобы отсортировать схему лексикографически, задайте свойству `sortSchema` значение `true`:

```typescript
GraphQLModule.forRoot<ApolloDriverConfig>({
  driver: ApolloDriver,
  autoSchemaFile: join(process.cwd(), 'src/schema.gql'),
  sortSchema: true,
}),
```

#### Schema first

Чтобы применить подход schema first, начните с добавления свойства `typePaths` в объект параметров. Свойство `typePaths` указывает, где `GraphQLModule` должен искать файлы определения схемы GraphQL SDL, которые вы будете писать. Эти файлы будут объединены в памяти; это позволяет разбить схему на несколько файлов и разместить их рядом с соответствующими резолверами.

```typescript
GraphQLModule.forRoot<ApolloDriverConfig>({
  driver: ApolloDriver,
  typePaths: ['./**/*.graphql'],
}),
```

Обычно вам также понадобятся определения TypeScript (классы и интерфейсы), соответствующие типам GraphQL SDL. Создавать их вручную — работа избыточная и утомительная. К тому же мы остаёмся без единого источника истины: каждое изменение в SDL заставляет править и определения TypeScript. Чтобы решить эту проблему, пакет `@nestjs/graphql` умеет **автоматически генерировать** определения TypeScript из абстрактного синтаксического дерева ([AST](https://en.wikipedia.org/wiki/Abstract_syntax_tree)). Чтобы включить эту возможность, добавьте при настройке `GraphQLModule` свойство `definitions`.

```typescript
GraphQLModule.forRoot<ApolloDriverConfig>({
  driver: ApolloDriver,
  typePaths: ['./**/*.graphql'],
  definitions: {
    path: join(process.cwd(), 'src/graphql.ts'),
  },
}),
```

Свойство `path` объекта `definitions` указывает, куда сохранять сгенерированный код на TypeScript. По умолчанию все сгенерированные типы TypeScript создаются как интерфейсы. Чтобы вместо этого генерировать классы, задайте свойству `outputAs` значение `'class'`.

```typescript
GraphQLModule.forRoot<ApolloDriverConfig>({
  driver: ApolloDriver,
  typePaths: ['./**/*.graphql'],
  definitions: {
    path: join(process.cwd(), 'src/graphql.ts'),
    outputAs: 'class',
  },
}),
```

Описанный подход генерирует определения TypeScript динамически при каждом запуске приложения. Иногда удобнее написать небольшой скрипт, генерирующий их по требованию. Допустим, мы создаём такой скрипт `generate-typings.ts`:

```typescript
import { GraphQLDefinitionsFactory } from '@nestjs/graphql';
import { join } from 'node:path';

const definitionsFactory = new GraphQLDefinitionsFactory();
definitionsFactory.generate({
  typePaths: ['./src/**/*.graphql'],
  path: join(process.cwd(), 'src/graphql.ts'),
  outputAs: 'class',
});
```

Теперь его можно запускать по необходимости:

```bash
$ ts-node generate-typings
```

> **Подсказка** Скрипт можно предварительно скомпилировать (например, с помощью `tsc`) и запускать через `node`.

Чтобы включить для скрипта режим наблюдения (автоматическая генерация типов при изменении любого файла `.graphql`), передайте методу `generate()` параметр `watch`.

```typescript
definitionsFactory.generate({
  typePaths: ['./src/**/*.graphql'],
  path: join(process.cwd(), 'src/graphql.ts'),
  outputAs: 'class',
  watch: true,
});
```

Чтобы автоматически генерировать дополнительное поле `__typename` для каждого объектного типа, включите параметр `emitTypenameField`:

```typescript
definitionsFactory.generate({
  // ...
  emitTypenameField: true,
});
```

Чтобы генерировать резолверы (запросы, мутации, подписки) как обычные поля без аргументов, включите параметр `skipResolverArgs`:

```typescript
definitionsFactory.generate({
  // ...
  skipResolverArgs: true,
});
```

Чтобы генерировать перечисления как объединения типов TypeScript вместо обычных enum, задайте параметру `enumsAsTypes` значение `true`:

```typescript
definitionsFactory.generate({
  // ...
  enumsAsTypes: true,
});
```

#### Apollo Sandbox

Чтобы для локальной разработки использовать в качестве GraphQL IDE [Apollo Sandbox](https://www.apollographql.com/blog/announcement/platform/apollo-sandbox-an-open-graphql-ide-for-local-development/) вместо `graphql-playground`, примените такую конфигурацию:

```typescript
import { ApolloDriver, ApolloDriverConfig } from '@nestjs/apollo';
import { Module } from '@nestjs/common';
import { GraphQLModule } from '@nestjs/graphql';
import { ApolloServerPluginLandingPageLocalDefault } from '@apollo/server/plugin/landingPage/default';

@Module({
  imports: [
    GraphQLModule.forRoot<ApolloDriverConfig>({
      driver: ApolloDriver,
      playground: false,
      plugins: [ApolloServerPluginLandingPageLocalDefault()],
    }),
  ],
})
export class AppModule {}
```

#### Доступ к сгенерированной схеме

В некоторых случаях (например, в сквозных тестах) может понадобиться ссылка на объект сгенерированной схемы. В сквозных тестах вы затем сможете выполнять запросы через объект `graphql`, не поднимая никаких HTTP-слушателей.

Получить сгенерированную схему (как при подходе code first, так и при schema first) можно с помощью класса `GraphQLSchemaHost`:

```typescript
const { schema } = app.get(GraphQLSchemaHost);
```

> **Подсказка** Вызывать геттер `GraphQLSchemaHost#schema` нужно после инициализации приложения (после того, как хук `onModuleInit` был вызван методом `app.listen()` или `app.init()`).

#### Асинхронная конфигурация

Когда параметры модуля нужно передавать асинхронно, а не статически, используйте метод `forRootAsync()`. Как и для большинства динамических модулей, Nest предлагает несколько приёмов работы с асинхронной конфигурацией.

Один из них — использовать фабричную функцию:

```typescript
GraphQLModule.forRootAsync<ApolloDriverConfig>({
  driver: ApolloDriver,
  useFactory: () => ({
    typePaths: ['./**/*.graphql'],
  }),
}),
```

Как и другие фабричные провайдеры, наша фабричная функция может быть <a href="../03.%20Fundamentals/01.%20Custom%20providers.md#провайдеры-фабрики-usefactory">асинхронной</a> и получать зависимости через `inject`.

```typescript
GraphQLModule.forRootAsync<ApolloDriverConfig>({
  driver: ApolloDriver,
  imports: [ConfigModule],
  useFactory: async (configService: ConfigService) => ({
    typePaths: configService.get<string>('GRAPHQL_TYPE_PATHS'),
  }),
  inject: [ConfigService],
}),
```

Кроме того, `GraphQLModule` можно настроить с помощью класса вместо фабрики, как показано ниже:

```typescript
GraphQLModule.forRootAsync<ApolloDriverConfig>({
  driver: ApolloDriver,
  useClass: GqlConfigService,
}),
```

Такая конструкция создаёт экземпляр `GqlConfigService` внутри `GraphQLModule` и использует его для создания объекта параметров. Обратите внимание, что в этом примере `GqlConfigService` должен реализовывать интерфейс `GqlOptionsFactory`, как показано ниже. `GraphQLModule` вызовет метод `createGqlOptions()` у экземпляра переданного класса.

```typescript
@Injectable()
class GqlConfigService implements GqlOptionsFactory {
  createGqlOptions(): ApolloDriverConfig {
    return {
      typePaths: ['./**/*.graphql'],
    };
  }
}
```

Если вы хотите переиспользовать существующий провайдер параметров, а не создавать его отдельную копию внутри `GraphQLModule`, используйте синтаксис `useExisting`.

```typescript
GraphQLModule.forRootAsync<ApolloDriverConfig>({
  imports: [ConfigModule],
  useExisting: ConfigService,
}),
```

#### Интеграция с Mercurius

Вместо Apollo пользователи Fastify (подробнее [здесь](<../04. Techniques/19. Performance (Fastify).md>)) могут взять драйвер `@nestjs/mercurius`.

```typescript
import { Module } from '@nestjs/common';
import { GraphQLModule } from '@nestjs/graphql';
import { MercuriusDriver, MercuriusDriverConfig } from '@nestjs/mercurius';

@Module({
  imports: [
    GraphQLModule.forRoot<MercuriusDriverConfig>({
      driver: MercuriusDriver,
      graphiql: true,
    }),
  ],
})
export class AppModule {}
```

> **Подсказка** После запуска приложения откройте браузер и перейдите на `http://localhost:3000/graphiql`. Вы должны увидеть [GraphQL IDE](https://github.com/graphql/graphiql).

Метод `forRoot()` принимает объект параметров в качестве аргумента. Эти параметры передаются нижележащему экземпляру драйвера. Подробнее о доступных настройках — [здесь](https://github.com/mercurius-js/mercurius/blob/master/docs/api/options.md#plugin-options).

#### Несколько эндпоинтов

Ещё одна полезная возможность модуля `@nestjs/graphql` — обслуживать сразу несколько эндпоинтов. Это позволяет решать, какие модули включать в какой эндпоинт. По умолчанию `GraphQL` ищет резолверы по всему приложению. Чтобы ограничить сканирование лишь частью модулей, используйте свойство `include`.

```typescript
GraphQLModule.forRoot({
  include: [CatsModule],
}),
```

> **Предупреждение** Если вы используете `@apollo/server` вместе с пакетом `@as-integrations/fastify` и несколькими GraphQL-эндпоинтами в одном приложении, обязательно включите настройку `disableHealthCheck` в конфигурации `GraphQLModule`.
