### Плагин CLI

> **Предупреждение** Эта глава относится только к подходу code first.

Система метаданных и рефлексии TypeScript имеет ряд ограничений: например, с её помощью невозможно определить, из каких свойств состоит класс, или понять, является ли конкретное свойство обязательным. Однако часть этих ограничений можно обойти на этапе компиляции. Nest предоставляет плагин, который расширяет процесс компиляции TypeScript и сокращает объём необходимого шаблонного кода.

> **Подсказка** Плагин подключается **по желанию**. Если вам так удобнее, вы можете объявлять все декораторы вручную или только те, которые вам действительно нужны.

#### Обзор

Плагин GraphQL автоматически:

- размечает декоратором `@Field` все свойства классов input object, object type и args, если не использован `@HideField`
- устанавливает свойство `nullable` в зависимости от знака вопроса (например, `name?: string` даст `nullable: true`)
- устанавливает свойство `type` в зависимости от типа (поддерживаются в том числе массивы)
- генерирует описания свойств на основе комментариев (если `introspectComments` установлен в `true`)

Обратите внимание: чтобы плагин анализировал ваши файлы, их имена **должны** иметь один из следующих суффиксов: `['.input.ts', '.args.ts', '.entity.ts', '.model.ts']` (например, `author.entity.ts`). Если вы используете другой суффикс, поведение плагина можно настроить через опцию `typeFileNameSuffix` (см. ниже).

С тем, что мы разобрали до сих пор, приходится дублировать много кода, чтобы пакет понял, как объявить ваш тип в GraphQL. Например, простой класс `Author` можно определить так:

**`authors/models/author.model.ts`**

```typescript
@ObjectType()
export class Author {
  @Field(type => ID)
  id: number;

  @Field({ nullable: true })
  firstName?: string;

  @Field({ nullable: true })
  lastName?: string;

  @Field(type => [Post])
  posts: Post[];
}
```

Для проектов среднего размера это не такая уж проблема, но при большом количестве классов код становится многословным и трудным в сопровождении.

Если включить плагин GraphQL, приведённое выше определение класса можно записать просто так:

**`authors/models/author.model.ts`**

```typescript
@ObjectType()
export class Author {
  @Field(type => ID)
  id: number;
  firstName?: string;
  lastName?: string;
  posts: Post[];
}
```

Плагин добавляет нужные декораторы на лету, опираясь на **абстрактное синтаксическое дерево** (AST). Благодаря этому вам не придётся возиться с декораторами `@Field`, разбросанными по всему коду.

> **Подсказка** Плагин автоматически сгенерирует все недостающие свойства GraphQL, но если вам нужно их переопределить, просто задайте их явно через `@Field()`.

#### Интроспекция комментариев

Если включена интроспекция комментариев, плагин CLI будет генерировать описания полей на основе комментариев.

Например, рассмотрим свойство `roles`:

```typescript
/**
 * Список ролей пользователя
 */
@Field(() => [String], {
  description: `A list of user's roles`
})
roles: string[];
```

Здесь приходится дублировать значение описания. С включённой опцией `introspectComments` плагин CLI может извлечь эти комментарии и автоматически подставить описания свойств. Теперь поле выше объявляется просто так:

```typescript
/**
 * Список ролей пользователя
 */
roles: string[];
```

#### Использование плагина CLI

Чтобы включить плагин, откройте `nest-cli.json` (если вы используете [Nest CLI](<../11. CLI/01. Overview.md>)) и добавьте следующую конфигурацию `plugins`:

```javascript
{
  "collection": "@nestjs/schematics",
  "sourceRoot": "src",
  "compilerOptions": {
    "plugins": ["@nestjs/graphql"]
  }
}
```

Свойство `options` позволяет настроить поведение плагина.

```javascript
{
  "collection": "@nestjs/schematics",
  "sourceRoot": "src",
  "compilerOptions": {
    "plugins": [
      {
        "name": "@nestjs/graphql",
        "options": {
          "typeFileNameSuffix": [".input.ts", ".args.ts"],
          "introspectComments": true
        }
      }
    ]
  }
}
```

Свойство `options` должно соответствовать следующему интерфейсу:

```typescript
export interface PluginOptions {
  typeFileNameSuffix?: string[];
  introspectComments?: boolean;
}
```

Если вы не используете CLI, а вместо этого поддерживаете собственную конфигурацию `webpack`, этот плагин можно применить совместно с `ts-loader`:

```javascript
getCustomTransformers: (program: any) => ({
  before: [require('@nestjs/graphql/plugin').before({}, program)]
}),
```

#### Сборщик SWC

В стандартных конфигурациях (не монорепозиторий) для использования плагинов CLI со сборщиком SWC нужно включить проверку типов, как описано [здесь](<../13. Recipes/03. SWC (fast compiler).md#проверка-типов>).

```bash
$ nest start -b swc --type-check
```

Для монорепозиториев следуйте инструкциям [здесь](<../13. Recipes/03. SWC (fast compiler).md#монорепозиторий-и-плагины-cli>).

```bash
$ npx ts-node src/generate-metadata.ts
# ИЛИ npx ts-node apps/{YOUR_APP}/src/generate-metadata.ts
```

Теперь файл с сериализованными метаданными нужно загрузить в методе `GraphQLModule`, как показано ниже:

```typescript
import metadata from './metadata'; // <-- файл, автоматически сгенерированный "PluginMetadataGenerator"

GraphQLModule.forRoot<...>({
  ..., // другие опции
  metadata,
}),
```

#### Интеграция с `ts-jest` (e2e-тесты)

При запуске e2e-тестов с включённым плагином вы можете столкнуться с проблемами компиляции схемы. Например, одна из самых частых ошибок:

```json
Object type <name> must define one or more fields.
```

Так происходит потому, что конфигурация `jest` нигде не импортирует плагин `@nestjs/graphql/plugin`.

Чтобы это исправить, создайте в каталоге с e2e-тестами такой файл:

```javascript
const transformer = require('@nestjs/graphql/plugin');

module.exports.name = 'nestjs-graphql-transformer';
// номер версии следует менять при каждом изменении конфигурации ниже — иначе jest не заметит изменений
module.exports.version = 1;

module.exports.factory = (cs) => {
  return transformer.before(
    {
      // опции @nestjs/graphql/plugin (могут быть пустыми)
    },
    cs.program, // "cs.tsCompiler.program" для старых версий Jest (<= v27)
  );
};
```

После этого подключите AST-трансформер в конфигурационном файле `jest`. По умолчанию (в стартовом приложении) файл конфигурации e2e-тестов находится в папке `test` и называется `jest-e2e.json`.

```json
{
  ... // другая конфигурация
  "globals": {
    "ts-jest": {
      "astTransformers": {
        "before": ["<path to the file created above>"]
      }
    }
  }
}
```

Если вы используете `jest@^29`, воспользуйтесь фрагментом ниже, так как предыдущий подход объявлен устаревшим.

```json
{
  ... // другая конфигурация
  "transform": {
    "^.+\\.(t|j)s$": [
      "ts-jest",
      {
        "astTransformers": {
          "before": ["<path to the file created above>"]
        }
      }
    ]
  }
}
```
