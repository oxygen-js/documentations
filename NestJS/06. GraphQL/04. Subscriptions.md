### Подписки

Помимо получения данных через запросы (queries) и их изменения через мутации (mutations), спецификация GraphQL поддерживает третий тип операций — `subscription`. Подписки GraphQL — это способ отправлять данные с сервера тем клиентам, которые решили получать сообщения от сервера в реальном времени. Подписки похожи на запросы тем, что тоже задают набор полей для доставки клиенту, но вместо немедленного возврата единственного ответа открывается канал, и результат отправляется клиенту каждый раз, когда на сервере происходит определённое событие.

Типичный сценарий использования подписок — уведомление клиентской стороны о конкретных событиях: например, о создании нового объекта, об изменении полей и так далее (подробнее [здесь](https://www.apollographql.com/docs/react/data/subscriptions)).

#### Включение подписок с драйвером Apollo

Чтобы включить подписки, выставьте свойство `installSubscriptionHandlers` в `true`.

```typescript
GraphQLModule.forRoot<ApolloDriverConfig>({
  driver: ApolloDriver,
  installSubscriptionHandlers: true,
}),
```

> **Предупреждение** Опция конфигурации `installSubscriptionHandlers` удалена из последней версии Apollo server и вскоре будет объявлена устаревшей и в этом пакете. По умолчанию `installSubscriptionHandlers` использует `subscriptions-transport-ws` ([подробнее](https://github.com/apollographql/subscriptions-transport-ws)), но мы настоятельно рекомендуем вместо неё библиотеку `graphql-ws` ([подробнее](https://github.com/enisdenjo/graphql-ws)).

Чтобы перейти на пакет `graphql-ws`, используйте следующую конфигурацию:

```typescript
GraphQLModule.forRoot<ApolloDriverConfig>({
  driver: ApolloDriver,
  subscriptions: {
    'graphql-ws': true
  },
}),
```

> **Подсказка** Вы можете использовать оба пакета (`subscriptions-transport-ws` и `graphql-ws`) одновременно — например, ради обратной совместимости.

#### Code first

Чтобы создать подписку в подходе code first, мы используем декоратор `@Subscription()` (экспортируется из пакета `@nestjs/graphql`) и класс `PubSub` из пакета `graphql-subscriptions`, который предоставляет простой **API публикации/подписки**.

Следующий обработчик подписки берёт на себя **подписку** на событие, вызывая `PubSub#asyncIterableIterator`. Этот метод принимает единственный аргумент — `triggerName`, соответствующий имени топика события.

```typescript
const pubSub = new PubSub();

@Resolver(() => Author)
export class AuthorResolver {
  // ...
  @Subscription(() => Comment)
  commentAdded() {
    return pubSub.asyncIterableIterator('commentAdded');
  }
}
```

> **Подсказка** Все декораторы экспортируются из пакета `@nestjs/graphql`, а класс `PubSub` — из пакета `graphql-subscriptions`.

> **Примечание** `PubSub` — это класс с простым API `publish` и `subscribe`. Подробнее о нём — [здесь](https://www.apollographql.com/docs/graphql-subscriptions/setup.html). Учтите, что документация Apollo предупреждает: реализация по умолчанию не подходит для production (подробнее [здесь](https://github.com/apollographql/graphql-subscriptions#getting-started-with-your-first-subscription)). В production-приложениях следует использовать реализацию `PubSub` поверх внешнего хранилища (подробнее [здесь](https://github.com/apollographql/graphql-subscriptions#pubsub-implementations)).

В результате в SDL-описании схемы GraphQL будет сгенерирована такая часть:

```graphql
type Subscription {
  commentAdded(): Comment!
}
```

Обратите внимание, что подписки по определению возвращают объект с единственным свойством верхнего уровня, ключ которого — имя подписки. Это имя либо наследуется от имени метода-обработчика подписки (то есть `commentAdded` выше), либо задаётся явно через опцию с ключом `name`, передаваемую вторым аргументом декоратора `@Subscription()`, как показано ниже.

```typescript
@Subscription(() => Comment, {
  name: 'commentAdded',
})
subscribeToCommentAdded() {
  return pubSub.asyncIterableIterator('commentAdded');
}
```

Эта конструкция даёт тот же SDL, что и предыдущий пример, но позволяет отвязать имя метода от имени подписки.

#### Публикация

Теперь, чтобы опубликовать событие, воспользуемся методом `PubSub#publish`. Его часто вызывают внутри мутации, чтобы инициировать обновление на стороне клиента при изменении части графа объектов. Например:

**`posts/posts.resolver.ts`**

```typescript
@Mutation(() => Comment)
async addComment(
  @Args('postId', { type: () => Int }) postId: number,
  @Args('comment', { type: () => Comment }) comment: CommentInput,
) {
  const newComment = this.commentsService.addComment({ id: postId, comment });
  pubSub.publish('commentAdded', { commentAdded: newComment });
  return newComment;
}
```

Метод `PubSub#publish` принимает первым параметром `triggerName` (снова — считайте это именем топика события), а вторым — полезную нагрузку события. Как уже говорилось, подписка по определению возвращает значение, и у этого значения есть форма. Посмотрите ещё раз на сгенерированный SDL для нашей подписки `commentAdded`:

```graphql
type Subscription {
  commentAdded(): Comment!
}
```

Отсюда видно, что подписка должна вернуть объект со свойством верхнего уровня `commentAdded`, значением которого является объект `Comment`. Важно, что форма полезной нагрузки события, отправляемой методом `PubSub#publish`, должна соответствовать форме значения, которое ожидается от подписки. Так, в примере выше выражение `pubSub.publish('commentAdded', { commentAdded: newComment })` публикует событие `commentAdded` с полезной нагрузкой правильной формы. Если формы не совпадут, подписка завершится ошибкой на этапе валидации GraphQL.

#### Фильтрация подписок

Чтобы отсеивать определённые события, задайте в свойстве `filter` функцию-фильтр. Она работает похоже на функцию, передаваемую в `filter` массива. Функция принимает два аргумента: `payload` с полезной нагрузкой события (в том виде, в каком её отправил издатель) и `variables` с любыми аргументами, переданными в запросе на подписку. Возвращает булево значение, определяющее, нужно ли публиковать это событие слушающим клиентам.

```typescript
@Subscription(() => Comment, {
  filter: (payload, variables) =>
    payload.commentAdded.title === variables.title,
})
commentAdded(@Args('title') title: string) {
  return pubSub.asyncIterableIterator('commentAdded');
}
```

#### Изменение полезной нагрузки подписки

Чтобы преобразовать публикуемую полезную нагрузку события, задайте в свойстве `resolve` функцию. Она получает полезную нагрузку события (в том виде, в каком её отправил издатель) и возвращает нужное значение.

```typescript
@Subscription(() => Comment, {
  resolve: value => value,
})
commentAdded() {
  return pubSub.asyncIterableIterator('commentAdded');
}
```

> **Примечание** Если вы используете опцию `resolve`, возвращать нужно развёрнутую полезную нагрузку (например, в нашем случае — сразу объект `newComment`, а не объект `{ commentAdded: newComment }`).

Если вам нужен доступ к внедрённым провайдерам (например, чтобы проверить данные через внешний сервис), используйте следующую конструкцию.

```typescript
@Subscription(() => Comment, {
  resolve(this: AuthorResolver, value) {
    // "this" ссылается на экземпляр "AuthorResolver"
    return value;
  }
})
commentAdded() {
  return pubSub.asyncIterableIterator('commentAdded');
}
```

Та же конструкция работает и с фильтрами:

```typescript
@Subscription(() => Comment, {
  filter(this: AuthorResolver, payload, variables) {
    // "this" ссылается на экземпляр "AuthorResolver"
    return payload.commentAdded.title === variables.title;
  }
})
commentAdded() {
  return pubSub.asyncIterableIterator('commentAdded');
}
```

#### Аутентификация поверх WebSocket

Проверить, аутентифицирован ли пользователь, можно внутри колбэка `onConnect`, который задаётся в опциях `subscriptions`.

Первым аргументом `onConnect` получит `connectionParams`, переданные в `SubscriptionClient` ([подробнее](https://www.apollographql.com/docs/react/data/subscriptions/#5-authenticate-over-websocket-optional)).

```typescript
GraphQLModule.forRoot<ApolloDriverConfig>({
  driver: ApolloDriver,
  subscriptions: {
    'subscriptions-transport-ws': {
      onConnect: (connectionParams) => {
        const authToken = connectionParams.authToken;
        if (!isValid(authToken)) {
          throw new Error('Token is not valid');
        }
        // извлекаем информацию о пользователе из токена
        const user = parseToken(authToken);
        // возвращаем данные пользователя, чтобы позже добавить их в контекст
        return { user };
      },
    }
  },
  context: ({ connection }) => {
    // connection.context будет равен тому, что вернул колбэк "onConnect"
  },
}),
```

В этом примере `authToken` отправляется клиентом лишь однажды — при первом установлении соединения.
Все подписки, созданные в рамках этого соединения, будут иметь тот же `authToken`, а значит и ту же информацию о пользователе.

> **Примечание** В `subscriptions-transport-ws` есть баг, из-за которого соединения могут пропускать этап `onConnect` ([подробнее](https://github.com/apollographql/subscriptions-transport-ws/issues/349)). Не считайте, что `onConnect` обязательно был вызван, когда пользователь начинает подписку, и всегда проверяйте, что `context` заполнен.

Если вы используете пакет `graphql-ws`, сигнатура колбэка `onConnect` будет немного другой:

```typescript
GraphQLModule.forRoot<ApolloDriverConfig>({
  driver: ApolloDriver,
  subscriptions: {
    'graphql-ws': {
      onConnect: (context: Context<any>) => {
        const { connectionParams, extra } = context;
        // проверка пользователя остаётся такой же, как в примере выше
        // при использовании graphql-ws дополнительное значение контекста
        // следует сохранять в поле extra
        extra.user = { user: {} };
      },
    },
  },
  context: ({ extra }) => {
    // теперь ваше дополнительное значение контекста доступно через поле extra
  },
});
```

#### Публикация

Теперь, чтобы опубликовать событие, воспользуемся методом `PubSub#publish`. Его часто вызывают внутри мутации, чтобы инициировать обновление на стороне клиента при изменении части графа объектов. Например:

**`posts/posts.resolver.ts`**

```typescript
@Mutation(() => Comment)

  @Args('postId', { type: () => Int }) postId: number,

  const newComment = this.commentsService.addComment({ id: postId, comment });
  await pubSub.publish({
    topic: 'commentAdded',
    payload: {
      commentAdded: newComment

}

  posts: [Post]
}

type Post {
  id: Int!
  title: String
  votes: Int
}

type Query {
  author(id: Int!): Author
}

type Comment {
  id: String
  content: String
}

type Subscription {
  commentAdded(title: String!): Comment
}
```

Так мы создали единственную подписку `commentAdded(title: String!): Comment`.

#### PubSub

В примерах выше использовался emitter `PubSub` по умолчанию ([mqemitter](https://github.com/mcollina/mqemitter))
Предпочтительный подход (для production) — использовать `mqemitter-redis`. Как вариант, можно предоставить собственную реализацию `PubSub` (подробнее [здесь](https://github.com/mercurius-js/mercurius/blob/master/docs/subscriptions.md))

```typescript
GraphQLModule.forRoot<MercuriusDriverConfig>({
  driver: MercuriusDriver,
  subscription: {
    emitter: require('mqemitter-redis')({
      port: 6579,
      host: '127.0.0.1',
    }),
  },
});
```

#### Аутентификация поверх WebSocket

Проверить, аутентифицирован ли пользователь, можно внутри колбэка `verifyClient`, который задаётся в опциях `subscription`.

Первым аргументом `verifyClient` получит объект `info`, из которого можно извлечь заголовки запроса.

```typescript
GraphQLModule.forRoot<MercuriusDriverConfig>({
  driver: MercuriusDriver,
  subscription: {
    verifyClient: (info, next) => {
      const authorization = info.req.headers?.authorization as string;
      if (!authorization?.startsWith('Bearer ')) {
        return next(false);
      }
      next(true);
    },
  }
}),
```
