### Федерация

Федерация позволяет разбить монолитный сервер GraphQL на независимые микросервисы. Она состоит из двух компонентов: шлюза и одного или нескольких федеративных микросервисов. Каждый микросервис содержит часть схемы, а шлюз объединяет эти схемы в одну, с которой работает клиент.

Как сказано в [документации Apollo](https://blog.apollographql.com/apollo-federation-f260cf525d21), федерация построена на следующих ключевых принципах:

- Построение графа должно быть **декларативным.** При федерации вы описываете граф декларативно внутри самой схемы, а не пишете императивный код сшивания схем.
- Код должен разделяться по **зонам ответственности**, а не по типам. Часто ни одна команда не контролирует все аспекты важного типа вроде User или Product, поэтому определение таких типов должно быть распределено между командами и кодовыми базами, а не централизовано.
- Граф должен быть простым в использовании для клиентов. Вместе федеративные сервисы образуют целостный, ориентированный на продукт граф, точно отражающий то, как его используют на клиенте.
- Это по-прежнему **GraphQL**, использующий только соответствующие спецификации возможности языка. Реализовать федерацию можно на любом языке, не только на JavaScript.

> **Предупреждение** В настоящее время федерация не поддерживает подписки.

В следующих разделах мы соберём демонстрационное приложение из шлюза и двух федеративных эндпоинтов: сервиса Users и сервиса Posts.

#### Федерация с Apollo

Начните с установки необходимых зависимостей:

```bash
$ npm install --save @apollo/subgraph
```

#### Schema first

«Сервис User» предоставляет простую схему. Обратите внимание на директиву `@key`: она сообщает планировщику запросов Apollo, что конкретный экземпляр `User` можно получить, указав его `id`. Также обратите внимание, что мы расширяем (`extend`) тип `Query`.

```graphql
type User @key(fields: "id") {
  id: ID!
  name: String!
}

extend type Query {
  getUser(id: ID!): User
}
```

Резолвер предоставляет один дополнительный метод — `resolveReference()`. Он вызывается шлюзом Apollo всякий раз, когда связанному ресурсу требуется экземпляр User. Пример этого мы увидим далее, в сервисе Posts. Учтите, что метод должен быть помечен декоратором `@ResolveReference()`.

```typescript
import { Args, Query, Resolver, ResolveReference } from '@nestjs/graphql';
import { UsersService } from './users.service';

@Resolver('User')
export class UsersResolver {
  constructor(private usersService: UsersService) {}

  @Query()
  getUser(@Args('id') id: string) {
    return this.usersService.findById(id);
  }

  @ResolveReference()
  resolveReference(reference: { __typename: string; id: string }) {
    return this.usersService.findById(reference.id);
  }
}
```

Наконец, свяжем всё воедино, зарегистрировав `GraphQLModule` и передав в объект конфигурации драйвер `ApolloFederationDriver`:

```typescript
import {
  ApolloFederationDriver,
  ApolloFederationDriverConfig,
} from '@nestjs/apollo';
import { Module } from '@nestjs/common';
import { GraphQLModule } from '@nestjs/graphql';
import { UsersResolver } from './users.resolver';

@Module({
  imports: [
    GraphQLModule.forRoot<ApolloFederationDriverConfig>({
      driver: ApolloFederationDriver,
      typePaths: ['**/*.graphql'],
    }),
  ],
  providers: [UsersResolver],
})
export class AppModule {}
```

#### Пример федерации: Posts

Сервис Post должен отдавать агрегированные посты через запрос `getPosts`, а также расширять наш тип `User` полем `user.posts`.

#### Schema first

«Сервис Posts» ссылается на тип `User` в своей схеме, помечая его ключевым словом `extend`. Кроме того, он объявляет у типа `User` дополнительное свойство (`posts`). Обратите внимание на директиву `@key`, которая используется для сопоставления экземпляров User, и на директиву `@external`, указывающую, что поле `id` управляется в другом месте.

```graphql
type Post @key(fields: "id") {
  id: ID!
  title: String!
  body: String!
  user: User
}

extend type User @key(fields: "id") {
  id: ID! @external
  posts: [Post]
}

extend type Query {
  getPosts: [Post]
}
```

В примере ниже `PostsResolver` предоставляет метод `getUser()`, который возвращает ссылку, содержащую `__typename` и дополнительные свойства, необходимые вашему приложению для разрешения ссылки, — в данном случае `id`. `__typename` используется шлюзом GraphQL, чтобы определить микросервис, отвечающий за тип User, и получить соответствующий экземпляр. При выполнении метода `resolveReference()` будет вызван описанный выше «сервис Users».

```typescript
import { Query, Resolver, Parent, ResolveField } from '@nestjs/graphql';
import { PostsService } from './posts.service';
import { Post } from './posts.interfaces';

@Resolver('Post')
export class PostsResolver {
  constructor(private postsService: PostsService) {}

  @Query('getPosts')
  getPosts() {
    return this.postsService.findAll();
  }

  @ResolveField('user')
  getUser(@Parent() post: Post) {
    return { __typename: 'User', id: post.userId };
  }
}
```

Наконец, нужно зарегистрировать `GraphQLModule` — так же, как мы делали это в разделе про «сервис Users».

```typescript
import {
  ApolloFederationDriver,
  ApolloFederationDriverConfig,
} from '@nestjs/apollo';
import { Module } from '@nestjs/common';
import { GraphQLModule } from '@nestjs/graphql';
import { PostsResolver } from './posts.resolver';

@Module({
  imports: [
    GraphQLModule.forRoot<ApolloFederationDriverConfig>({
      driver: ApolloFederationDriver,
      typePaths: ['**/*.graphql'],
    }),
  ],
  providers: [PostsResolver],
})
export class AppModule {}
```

#### Code first

Сначала нам нужно объявить класс, представляющий сущность `User`. Хотя сама сущность живёт в другом сервисе, здесь мы будем её использовать (расширяя её определение). Обратите внимание на директивы `@extends` и `@external`.

```ts
import { Directive, ObjectType, Field, ID } from '@nestjs/graphql';
import { Post } from './post.entity';

@ObjectType()
@Directive('@extends')
@Directive('@key(fields: "id")')
export class User {
  @Field(() => ID)
  @Directive('@external')
  id: number;

  @Field(() => [Post])
  posts?: Post[];
}
```

Теперь создадим соответствующий резолвер для нашего расширения сущности `User`:

```ts
import { Parent, ResolveField, Resolver } from '@nestjs/graphql';
import { PostsService } from './posts.service';
import { Post } from './post.entity';
import { User } from './user.entity';

@Resolver(() => User)
export class UsersResolver {
  constructor(private readonly postsService: PostsService) {}

  @ResolveField(() => [Post])
  public posts(@Parent() user: User): Post[] {
    return this.postsService.forAuthor(user.id);
  }
}
```

Также нужно определить класс сущности `Post`:

```ts
import { Directive, Field, ID, Int, ObjectType } from '@nestjs/graphql';
import { User } from './user.entity';

@ObjectType()
@Directive('@key(fields: "id")')
export class Post {
  @Field(() => ID)
  id: number;

  @Field()
  title: string;

  @Field(() => Int)
  authorId: number;

  @Field(() => User)
  user?: User;
}
```

И его резолвер:

```ts
import { Query, Args, ResolveField, Resolver, Parent } from '@nestjs/graphql';
import { PostsService } from './posts.service';
import { Post } from './post.entity';
import { User } from './user.entity';

@Resolver(() => Post)
export class PostsResolver {
  constructor(private readonly postsService: PostsService) {}

  @Query(() => Post)
  findPost(@Args('id') id: number): Post {
    return this.postsService.findOne(id);
  }

  @Query(() => [Post])
  getPosts(): Post[] {
    return this.postsService.all();
  }

  @ResolveField(() => User)
  user(@Parent() post: Post): any {
    return { __typename: 'User', id: post.authorId };
  }
}
```

И наконец соберём всё в модуле. Обратите внимание на параметры сборки схемы, где мы указываем, что `User` — это «осиротевший» (внешний) тип.

```ts
import {
  ApolloFederationDriver,
  ApolloFederationDriverConfig,
} from '@nestjs/apollo';
import { Module } from '@nestjs/common';
import { User } from './user.entity';
import { PostsResolver } from './posts.resolvers';
import { UsersResolver } from './users.resolvers';
import { PostsService } from './posts.service'; // В примере не приводится

@Module({
  imports: [
    GraphQLModule.forRoot<ApolloFederationDriverConfig>({
      driver: ApolloFederationDriver,
      autoSchemaFile: true,
      buildSchemaOptions: {
        orphanedTypes: [User],
      },
    }),
  ],
  providers: [PostsResolver, UsersResolver, PostsService],
})
export class AppModule {}
```

#### Пример федерации: шлюз

Начните с установки необходимой зависимости:

```bash
$ npm install --save @apollo/gateway
```

Шлюзу нужно передать список эндпоинтов, а соответствующие схемы он обнаружит сам. Поэтому реализация сервиса-шлюза будет одинаковой и для подхода code first, и для schema first.

```typescript
import { IntrospectAndCompose } from '@apollo/gateway';
import { ApolloGatewayDriver, ApolloGatewayDriverConfig } from '@nestjs/apollo';
import { Module } from '@nestjs/common';
import { GraphQLModule } from '@nestjs/graphql';

@Module({
  imports: [
    GraphQLModule.forRoot<ApolloGatewayDriverConfig>({
      driver: ApolloGatewayDriver,
      server: {
        // ... параметры сервера Apollo
        cors: true,
      },
      gateway: {
        supergraphSdl: new IntrospectAndCompose({
          subgraphs: [
            { name: 'users', url: 'http://user-service/graphql' },
            { name: 'posts', url: 'http://post-service/graphql' },
          ],
        }),
      },
    }),
  ],
})
export class AppModule {}
```

И наконец соберём всё в модуле. Обратите внимание на параметры сборки схемы, где мы указываем, что `User` — это «осиротевший» (внешний) тип.

```ts
import {
  MercuriusFederationDriver,
  MercuriusFederationDriverConfig,
} from '@nestjs/mercurius';
import { Module } from '@nestjs/common';
import { User } from './user.entity';
import { PostsResolver } from './posts.resolvers';
import { UsersResolver } from './users.resolvers';
import { PostsService } from './posts.service'; // В примере не приводится

@Module({
  imports: [
    GraphQLModule.forRoot<MercuriusFederationDriverConfig>({
      driver: MercuriusFederationDriver,
      autoSchemaFile: true,
      federationMetadata: true,
      buildSchemaOptions: {
        orphanedTypes: [User],
      },
    }),
  ],
  providers: [PostsResolver, UsersResolver, PostsService],
})
export class AppModule {}
```

#### Пример федерации: шлюз

Шлюзу нужно передать список эндпоинтов, а соответствующие схемы он обнаружит сам. Поэтому реализация сервиса-шлюза будет одинаковой и для подхода code first, и для schema first.

```typescript
import {
  MercuriusGatewayDriver,
  MercuriusGatewayDriverConfig,
} from '@nestjs/mercurius';
import { Module } from '@nestjs/common';
import { GraphQLModule } from '@nestjs/graphql';

@Module({
  imports: [
    GraphQLModule.forRoot<MercuriusGatewayDriverConfig>({
      driver: MercuriusGatewayDriver,
      gateway: {
        services: [
          { name: 'users', url: 'http://user-service/graphql' },
          { name: 'posts', url: 'http://post-service/graphql' },
        ],
      },
    }),
  ],
})
export class AppModule {}
```

### Federation 2

Как сказано в [документации Apollo](https://www.apollographql.com/docs/federation/federation-2/new-in-federation-2), Federation 2 улучшает опыт разработчика по сравнению с исходной Apollo Federation (в этом документе она называется Federation 1) и обратно совместима с большинством исходных суперграфов.

> **Предупреждение** Mercurius поддерживает Federation 2 не полностью. Список библиотек с поддержкой Federation 2 приведён [здесь](https://www.apollographql.com/docs/federation/supported-subgraphs#javascript--typescript).

В следующих разделах мы обновим предыдущий пример до Federation 2.

#### Пример федерации: Users

Одно из изменений в Federation 2 состоит в том, что у сущностей больше нет исходного подграфа, поэтому расширять `Query` больше не требуется. Подробнее см. [раздел о сущностях](https://www.apollographql.com/docs/federation/federation-2/new-in-federation-2#entities) в документации Apollo Federation 2.

#### Schema first

Достаточно убрать из схемы ключевое слово `extend`.

```graphql
type User @key(fields: "id") {
  id: ID!
  name: String!
}

type Query {
  getUser(id: ID!): User
}
```

#### Code first

Чтобы использовать Federation 2, нужно указать версию федерации в опции `autoSchemaFile`.

```ts
import {
  ApolloFederationDriver,
  ApolloFederationDriverConfig,
} from '@nestjs/apollo';
import { Module } from '@nestjs/common';
import { UsersResolver } from './users.resolver';
import { UsersService } from './users.service'; // В этом примере не приводится

@Module({
  imports: [
    GraphQLModule.forRoot<ApolloFederationDriverConfig>({
      driver: ApolloFederationDriver,
      autoSchemaFile: {
        federation: 2,
      },
    }),
  ],
  providers: [UsersResolver, UsersService],
})
export class AppModule {}
```

#### Пример федерации: Posts

По той же причине, что и выше, расширять `User` и `Query` больше не требуется.

#### Schema first

Достаточно убрать из схемы директивы `extend` и `external`

```graphql
type Post @key(fields: "id") {
  id: ID!
  title: String!
  body: String!
  user: User
}

type User @key(fields: "id") {
  id: ID!
  posts: [Post]
}

type Query {
  getPosts: [Post]
}
```

#### Code first

Поскольку сущность `User` мы больше не расширяем, можно просто убрать у `User` директивы `extends` и `external`.

```ts
import { Directive, ObjectType, Field, ID } from '@nestjs/graphql';
import { Post } from './post.entity';

@ObjectType()
@Directive('@key(fields: "id")')
export class User {
  @Field(() => ID)
  id: number;

  @Field(() => [Post])
  posts?: Post[];
}
```

Кроме того, как и в сервисе User, нужно указать в `GraphQLModule` использование Federation 2.

```ts
import {
  ApolloFederationDriver,
  ApolloFederationDriverConfig,
} from '@nestjs/apollo';
import { Module } from '@nestjs/common';
import { User } from './user.entity';
import { PostsResolver } from './posts.resolvers';
import { UsersResolver } from './users.resolvers';
import { PostsService } from './posts.service'; // В примере не приводится

@Module({
  imports: [
    GraphQLModule.forRoot<ApolloFederationDriverConfig>({
      driver: ApolloFederationDriver,
      autoSchemaFile: {
        federation: 2,
      },
      buildSchemaOptions: {
        orphanedTypes: [User],
      },
    }),
  ],
  providers: [PostsResolver, UsersResolver, PostsService],
})
export class AppModule {}
```
