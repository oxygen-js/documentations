### Резолверы

Резолверы описывают, как превратить операцию [GraphQL](https://graphql.org/) (query, mutation или subscription) в данные. Они возвращают данные той же формы, которую мы задали в схеме, — либо синхронно, либо в виде промиса, разрешающегося в результат этой формы. Обычно **карту резолверов** составляют вручную. Пакет `@nestjs/graphql`, напротив, генерирует карту резолверов автоматически, используя метаданные, предоставленные декораторами, которыми вы аннотируете классы. Чтобы продемонстрировать, как с помощью возможностей пакета создать GraphQL API, мы построим простой API авторов.

#### Code first

При подходе code first мы не идём привычным путём создания схемы GraphQL через написание GraphQL SDL вручную. Вместо этого мы используем декораторы TypeScript, чтобы сгенерировать SDL из определений классов TypeScript. Пакет `@nestjs/graphql` читает метаданные, заданные через декораторы, и автоматически генерирует за вас схему.

#### Объектные типы

Большая часть определений в схеме GraphQL — это **объектные типы**. Каждый объектный тип, который вы определяете, должен представлять доменный объект, с которым может понадобиться работать клиенту приложения. Например, нашему примеру API нужно уметь получать список авторов и их постов, поэтому мы должны определить типы `Author` и `Post`, поддерживающие эту функциональность.

Если бы мы использовали подход schema first, такую схему мы описали бы на SDL так:

```graphql
type Author {
  id: Int!
  firstName: String
  lastName: String
  posts: [Post!]!
}
```

В нашем же случае, при подходе code first, мы определяем схемы с помощью классов TypeScript, аннотируя поля этих классов декораторами TypeScript. Эквивалент приведённого выше SDL в подходе code first выглядит так:

**`authors/models/author.model.ts`**

```typescript
import { Field, Int, ObjectType } from '@nestjs/graphql';
import { Post } from './post';

@ObjectType()
export class Author {
  @Field(type => Int)
  id: number;

  @Field({ nullable: true })
  firstName?: string;

  @Field({ nullable: true })
  lastName?: string;

  @Field(type => [Post])
  posts: Post[];
}
```

> **Подсказка** Система рефлексии метаданных TypeScript имеет ряд ограничений, из-за которых невозможно, например, определить, из каких свойств состоит класс, или понять, является ли конкретное свойство необязательным или обязательным. Из-за этих ограничений нам приходится либо явно использовать декоратор `@Field()` в классах определения схемы, чтобы предоставить метаданные о GraphQL-типе каждого поля и его необязательности, либо применять [плагин CLI](<14. CLI Plugin.md>), который сделает это за нас.

Объектный тип `Author`, как и любой класс, состоит из набора полей, каждое из которых объявляет тип. Тип поля соответствует [типу GraphQL](https://graphql.org/learn/schema/). GraphQL-типом поля может быть либо другой объектный тип, либо скалярный тип. Скалярный тип GraphQL — это примитив (например, `ID`, `String`, `Boolean` или `Int`), разрешающийся в единственное значение.

> **Подсказка** Помимо встроенных скалярных типов GraphQL, вы можете определять собственные скалярные типы (читайте [подробнее](<05. Scalars.md>)).

Приведённое выше определение объектного типа `Author` заставит Nest **сгенерировать** тот SDL, который мы показали выше:

```graphql
type Author {
  id: Int!
  firstName: String
  lastName: String
  posts: [Post!]!
}
```

Декоратор `@Field()` принимает необязательную функцию типа (например, `type => Int`) и, опционально, объект настроек.

Функция типа обязательна, когда возможна неоднозначность между системой типов TypeScript и системой типов GraphQL. В частности: она **не** требуется для типов `string` и `boolean`; она **требуется** для `number` (который должен быть сопоставлен либо с `Int`, либо с `Float` в GraphQL). Функция типа должна просто возвращать нужный тип GraphQL (как показано в различных примерах этих глав).

Объект настроек может содержать любые из следующих пар ключ/значение:

- `nullable`: указывает, может ли поле быть nullable (в `@nestjs/graphql` каждое поле по умолчанию non-nullable); `boolean`
- `description`: задаёт описание поля; `string`
- `deprecationReason`: помечает поле как устаревшее; `string`

Например:

```typescript
@Field({ description: `Book title`, deprecationReason: 'Not useful in v2 schema' })
title: string;
```

> **Подсказка** Вы также можете добавить описание или пометить устаревшим весь объектный тип: `@ObjectType({ description: 'Author model' })`.

Когда поле является массивом, тип массива нужно указать вручную в функции типа декоратора `Field()`:

```typescript
@Field(type => [Post])
posts: Post[];
```

> **Подсказка** С помощью квадратных скобок (`[ ]`) можно указать глубину массива. Например, запись `[[Int]]` описывает матрицу целых чисел.

Чтобы объявить nullable элементы массива (а не сам массив), задайте свойству `nullable` значение `'items'`:

```typescript
@Field(type => [Post], { nullable: 'items' })
posts: Post[];
```

> **Подсказка** Если nullable должны быть и массив, и его элементы, задайте `nullable` значение `'itemsAndList'`.

Теперь, когда объектный тип `Author` создан, определим объектный тип `Post`.

**`posts/models/post.model.ts`**

```typescript
import { Field, Int, ObjectType } from '@nestjs/graphql';

@ObjectType()
export class Post {
  @Field(type => Int)
  id: number;

  @Field()
  title: string;

  @Field(type => Int, { nullable: true })
  votes?: number;
}
```

Объектный тип `Post` приведёт к генерации следующей части схемы GraphQL в формате SDL:

```graphql
type Post {
  id: Int!
  title: String!
  votes: Int
}
```

#### Резолвер в подходе code first

На данный момент мы определили объекты (определения типов), которые могут существовать в нашем графе данных, но у клиентов пока нет способа взаимодействовать с этими объектами. Чтобы это исправить, нужно создать класс резолвера. В методе code first класс резолвера одновременно определяет функции-резолверы **и** генерирует **тип Query**. Это станет понятно на примере ниже:

**`authors/authors.resolver.ts`**

```typescript
@Resolver(() => Author)
export class AuthorsResolver {
  constructor(
    private authorsService: AuthorsService,
    private postsService: PostsService,
  ) {}

  @Query(() => Author)
  async author(@Args('id', { type: () => Int }) id: number) {
    return this.authorsService.findOneById(id);
  }

  @ResolveField()
  async posts(@Parent() author: Author) {
    const { id } = author;
    return this.postsService.findAll({ authorId: id });
  }
}
```

> **Подсказка** Все декораторы (например, `@Resolver`, `@ResolveField`, `@Args` и т. д.) экспортируются из пакета `@nestjs/graphql`.

Вы можете определить несколько классов резолверов. Nest объединит их во время выполнения. Подробнее об организации кода см. раздел [Модуль](<02. Resolvers.md#модуль>) ниже.

> **Примечание** Логика внутри классов `AuthorsService` и `PostsService` может быть настолько простой или сложной, насколько это нужно. Главная цель примера — показать, как строятся резолверы и как они могут взаимодействовать с другими провайдерами.

В примере выше мы создали `AuthorsResolver`, который определяет одну функцию-резолвер запроса и одну функцию-резолвер поля. Чтобы создать резолвер, мы создаём класс, методы которого являются функциями-резолверами, и аннотируем класс декоратором `@Resolver()`.

В этом примере мы определили обработчик запроса, получающий объект автора по `id`, переданному в запросе. Чтобы указать, что метод является обработчиком запроса, используется декоратор `@Query()`.

Аргумент, передаваемый декоратору `@Resolver()`, необязателен, но становится важен, когда граф перестаёт быть тривиальным. Он служит для передачи родительского объекта, который используют функции-резолверы полей при обходе графа объектов вниз.

В нашем примере, поскольку класс содержит функцию-**резолвер поля** (для свойства `posts` объектного типа `Author`), мы **обязаны** передать декоратору `@Resolver()` значение, указывающее, какой класс является родительским типом (то есть имя соответствующего класса `ObjectType`) для всех резолверов полей, определённых внутри этого класса. Как ясно из примера, при написании функции-резолвера поля необходимо обращаться к родительскому объекту (объекту, членом которого является разрешаемое поле). В этом примере мы заполняем массив постов автора резолвером поля, который вызывает сервис, принимающий `id` автора в качестве аргумента. Отсюда и необходимость указать родительский объект в декораторе `@Resolver()`. Обратите внимание на соответствующее использование декоратора параметра метода `@Parent()`, который затем извлекает ссылку на этот родительский объект в резолвере поля.

Мы можем определить несколько функций-резолверов `@Query()` (как внутри этого класса, так и в любом другом классе резолвера), и они будут объединены в единое определение **типа Query** в сгенерированном SDL вместе с соответствующими записями в карте резолверов. Это позволяет определять запросы рядом с моделями и сервисами, которые они используют, и держать их аккуратно организованными по модулям.

> **Подсказка** Nest CLI предоставляет генератор (schematic), который автоматически создаёт **весь шаблонный код**, помогая избежать всей этой ручной работы и значительно упрощая жизнь разработчика. Подробнее об этой возможности читайте [здесь](<../13. Recipes/02. CRUD generator.md>).

#### Имена типов запросов

В примерах выше декоратор `@Query()` генерирует имя типа запроса в схеме GraphQL на основе имени метода. Например, рассмотрим следующую конструкцию из примера выше:

```typescript
@Query(() => Author)
async author(@Args('id', { type: () => Int }) id: number) {
  return this.authorsService.findOneById(id);
}
```

Она порождает следующую запись для запроса author в нашей схеме (тип запроса получает то же имя, что и метод):

```graphql
type Query {
  author(id: Int!): Author
}
```

> **Подсказка** Подробнее о запросах GraphQL читайте [здесь](https://graphql.org/learn/queries/).

По соглашению мы предпочитаем разделять эти имена: например, для метода-обработчика запроса нам удобнее имя вида `getAuthor()`, но имя типа запроса при этом должно остаться `author`. То же касается и резолверов полей. Сделать это легко, передав имена сопоставления в качестве аргументов декораторов `@Query()` и `@ResolveField()`:

**`authors/authors.resolver.ts`**

```typescript
@Resolver(() => Author)
export class AuthorsResolver {
  constructor(
    private authorsService: AuthorsService,
    private postsService: PostsService,
  ) {}

  @Query(() => Author, { name: 'author' })
  async getAuthor(@Args('id', { type: () => Int }) id: number) {
    return this.authorsService.findOneById(id);
  }

  @ResolveField('posts', () => [Post])
  async getPosts(@Parent() author: Author) {
    const { id } = author;
    return this.postsService.findAll({ authorId: id });
  }
}
```

Приведённый выше метод-обработчик `getAuthor` приведёт к генерации следующей части схемы GraphQL в формате SDL:

```graphql
type Query {
  author(id: Int!): Author
}
```

#### Настройки декоратора Query

Объект настроек декоратора `@Query()` (в который мы выше передавали `{name: 'author'}`) принимает ряд пар ключ/значение:

- `name`: имя запроса; `string`
- `description`: описание, которое будет использовано для генерации документации схемы GraphQL (например, в GraphQL playground); `string`
- `deprecationReason`: задаёт метаданные, помечающие запрос как устаревший (например, в GraphQL playground); `string`
- `nullable`: может ли запрос вернуть null в качестве ответа; `boolean`, либо `'items'`, либо `'itemsAndList'` (о значениях `'items'` и `'itemsAndList'` см. выше)

#### Настройки декоратора Args

Декоратор `@Args()` используется для извлечения аргументов из запроса, чтобы применить их в методе-обработчике. Это работает очень похоже на [извлечение параметров маршрута в REST](<../02. Overview/02. Controllers.md#параметры-маршрута>).

Обычно ваш декоратор `@Args()` будет простым и не потребует объектного аргумента, как в методе `getAuthor()` выше. Например, если тип идентификатора — строка, достаточно следующей конструкции, которая просто выбирает поле с указанным именем из входящего запроса GraphQL для использования в качестве аргумента метода.

```typescript
@Args('id') id: string
```

В случае `getAuthor()` используется тип `number`, и это создаёт сложность. Тип TypeScript `number` не даёт достаточно информации об ожидаемом представлении в GraphQL (например, `Int` или `Float`). Поэтому нам приходится **явно** передавать ссылку на тип. Мы делаем это, передавая второй аргумент декоратору `Args()` — объект настроек аргумента:

```typescript
@Query(() => Author, { name: 'author' })
async getAuthor(@Args('id', { type: () => Int }) id: number) {
  return this.authorsService.findOneById(id);
}
```

Объект настроек позволяет указать следующие необязательные пары ключ/значение:

- `type`: функция, возвращающая тип GraphQL
- `defaultValue`: значение по умолчанию; `any`
- `description`: метаданные описания; `string`
- `deprecationReason`: помечает поле устаревшим и предоставляет метаданные с объяснением причины; `string`
- `nullable`: может ли поле быть nullable

Методы-обработчики запросов могут принимать несколько аргументов. Представим, что мы хотим получить автора по его `firstName` и `lastName`. В этом случае мы можем вызвать `@Args` дважды:

```typescript
getAuthor(
  @Args('firstName', { nullable: true }) firstName?: string,
  @Args('lastName', { defaultValue: '' }) lastName?: string,
) {}
```

> **Подсказка** В случае `firstName`, который является nullable-полем GraphQL, добавлять к типу этого поля значения `null` или `undefined` не обязательно. Просто имейте в виду, что в резолверах вам понадобится проверка на эти возможные «пустые» типы, поскольку nullable-поле GraphQL позволит им дойти до вашего резолвера.

#### Выделенный класс аргументов

При использовании встроенных вызовов `@Args()` код вроде приведённого выше разрастается. Вместо этого можно создать выделенный класс аргументов `GetAuthorArgs` и обращаться к нему в методе-обработчике так:

```typescript
@Args() args: GetAuthorArgs
```

Создайте класс `GetAuthorArgs` с помощью `@ArgsType()`:

**`authors/dto/get-author.args.ts`**

```typescript
import { MinLength } from 'class-validator';
import { Field, ArgsType } from '@nestjs/graphql';

@ArgsType()
class GetAuthorArgs {
  @Field({ nullable: true })
  firstName?: string;

  @Field({ defaultValue: '' })
  @MinLength(3)
  lastName: string;
}
```

> **Подсказка** И снова, из-за ограничений системы рефлексии метаданных TypeScript, необходимо либо вручную указывать тип и необязательность через декоратор `@Field`, либо использовать [плагин CLI](<14. CLI Plugin.md>). Кроме того, в случае `firstName`, который является nullable-полем GraphQL, добавлять к типу этого поля значения `null` или `undefined` не обязательно. Просто имейте в виду, что в резолверах вам понадобится проверка на эти возможные «пустые» типы, поскольку nullable-поле GraphQL позволит им дойти до вашего резолвера. 

В результате будет сгенерирована следующая часть схемы GraphQL в формате SDL:

```graphql
type Query {
  author(firstName: String, lastName: String = ''): Author
}
```

> **Подсказка** Обратите внимание, что классы аргументов вроде `GetAuthorArgs` прекрасно сочетаются с `ValidationPipe` (читайте [подробнее](<../04. Techniques/04. Validation.md>)).

#### Наследование классов

Вы можете использовать стандартное наследование классов TypeScript, чтобы создавать базовые классы с общими служебными возможностями (поля и их свойства, валидации и т. д.), которые затем расширяются. Например, у вас может быть набор аргументов пагинации, всегда включающий стандартные поля `offset` и `limit`, но также и другие поля-индексы, специфичные для конкретного типа. Иерархию классов можно построить так.

Базовый класс `@ArgsType()`:

```typescript
@ArgsType()
class PaginationArgs {
  @Field(() => Int)
  offset: number = 0;

  @Field(() => Int)
  limit: number = 10;
}
```

Специфичный для типа подкласс базового класса `@ArgsType()`:

```typescript
@ArgsType()
class GetAuthorArgs extends PaginationArgs {
  @Field({ nullable: true })
  firstName?: string;

  @Field({ defaultValue: '' })
  @MinLength(3)
  lastName: string;
}
```

Такой же подход применим и к объектам `@ObjectType()`. Определите общие свойства в базовом классе:

```typescript
@ObjectType()
class Character {
  @Field(() => Int)
  id: number;

  @Field()
  name: string;
}
```

И добавьте специфичные для типа свойства в подклассах:

```typescript
@ObjectType()
class Warrior extends Character {
  @Field()
  level: number;
}
```

Наследование можно использовать и с резолвером. Типобезопасность обеспечивается сочетанием наследования и обобщённых типов TypeScript. Например, чтобы создать базовый класс с обобщённым запросом `findAll`, используйте такую конструкцию:

```typescript
function BaseResolver<T extends Type<unknown>>(classRef: T): any {
  @Resolver({ isAbstract: true })
  abstract class BaseResolverHost {
    @Query(() => [classRef], { name: `findAll${classRef.name}` })
    async findAll(): Promise<T[]> {
      return [];
    }
  }
  return BaseResolverHost;
}
```

Обратите внимание на следующее:

- Явный тип возвращаемого значения (`any` выше) обязателен; иначе TypeScript будет ругаться на использование приватного определения класса. Рекомендуется: определить интерфейс вместо использования `any`.
- `Type` импортируется из пакета `@nestjs/common`
- Свойство `isAbstract: true` указывает, что SDL (выражения языка определения схемы) для этого класса генерировать не нужно. Заметьте, это свойство можно задавать и для других типов, чтобы подавить генерацию SDL.

Вот как можно сгенерировать конкретный подкласс `BaseResolver`:

```typescript
@Resolver(() => Recipe)
export class RecipesResolver extends BaseResolver(Recipe) {
  constructor(private recipesService: RecipesService) {
    super();
  }
}
```

Эта конструкция сгенерирует следующий SDL:

```graphql
type Query {
  findAllRecipe: [Recipe!]!
}
```

#### Обобщённые типы

Один из вариантов использования обобщённых типов мы уже видели выше. Эта мощная возможность TypeScript позволяет создавать полезные абстракции. Например, вот пример реализации курсорной пагинации по мотивам [этой документации](https://graphql.org/learn/pagination/#pagination-and-edges):

```typescript
import { Field, ObjectType, Int } from '@nestjs/graphql';
import { Type } from '@nestjs/common';

interface IEdgeType<T> {
  cursor: string;
  node: T;
}

export interface IPaginatedType<T> {
  edges: IEdgeType<T>[];
  nodes: T[];
  totalCount: number;
  hasNextPage: boolean;
}

export function Paginated<T>(classRef: Type<T>): Type<IPaginatedType<T>> {
  @ObjectType(`${classRef.name}Edge`)
  abstract class EdgeType {
    @Field(() => String)
    cursor: string;

    @Field(() => classRef)
    node: T;
  }

  @ObjectType({ isAbstract: true })
  abstract class PaginatedType implements IPaginatedType<T> {
    @Field(() => [EdgeType], { nullable: true })
    edges: EdgeType[];

    @Field(() => [classRef], { nullable: true })
    nodes: T[];

    @Field(() => Int)
    totalCount: number;

    @Field()
    hasNextPage: boolean;
  }
  return PaginatedType as Type<IPaginatedType<T>>;
}
```

Определив такой базовый класс, мы теперь легко можем создавать специализированные типы, наследующие это поведение. Например:

```typescript
@ObjectType()
class PaginatedAuthor extends Paginated(Author) {}
```

#### Schema first

Как упоминалось в [предыдущей](<01. Quick start.md>) главе, при подходе schema first мы начинаем с ручного определения типов схемы на SDL (читайте [подробнее](https://graphql.org/learn/schema/#type-language)). Рассмотрим следующие определения типов SDL.

> **Подсказка** Для удобства в этой главе мы собрали весь SDL в одном месте (например, в одном файле `.graphql`, как показано ниже). На практике вам может оказаться удобнее организовать код модульно. Например, бывает полезно создавать отдельные файлы SDL с определениями типов для каждой доменной сущности вместе со связанными сервисами, кодом резолверов и классом определения модуля Nest в отдельном каталоге для этой сущности. Nest объединит все отдельные определения типов схемы во время выполнения.

```graphql
type Author {
  id: Int!
  firstName: String
  lastName: String
  posts: [Post]
}

type Post {
  id: Int!
  title: String!
  votes: Int
}

type Query {
  author(id: Int!): Author
}
```

#### Резолвер в подходе schema first

Приведённая выше схема предоставляет единственный запрос — `author(id: Int!): Author`.

> **Подсказка** Подробнее о запросах GraphQL читайте [здесь](https://graphql.org/learn/queries/).

Теперь создадим класс `AuthorsResolver`, разрешающий запросы авторов:

**`authors/authors.resolver.ts`**

```typescript
@Resolver('Author')
export class AuthorsResolver {
  constructor(
    private authorsService: AuthorsService,
    private postsService: PostsService,
  ) {}

  @Query()
  async author(@Args('id') id: number) {
    return this.authorsService.findOneById(id);
  }

  @ResolveField()
  async posts(@Parent() author) {
    const { id } = author;
    return this.postsService.findAll({ authorId: id });
  }
}
```

> **Подсказка** Все декораторы (например, `@Resolver`, `@ResolveField`, `@Args` и т. д.) экспортируются из пакета `@nestjs/graphql`.

> **Примечание** Логика внутри классов `AuthorsService` и `PostsService` может быть настолько простой или сложной, насколько это нужно. Главная цель примера — показать, как строятся резолверы и как они могут взаимодействовать с другими провайдерами.

Декоратор `@Resolver()` обязателен. Он принимает необязательный строковый аргумент с именем класса. Это имя класса требуется всякий раз, когда класс содержит декораторы `@ResolveField()`, — чтобы сообщить Nest, что декорированный метод связан с родительским типом (в нашем примере — типом `Author`). Как вариант, вместо указания `@Resolver()` в начале класса это можно делать для каждого метода:

```typescript
@Resolver('Author')
@ResolveField()
async posts(@Parent() author) {
  const { id } = author;
  return this.postsService.findAll({ authorId: id });
}
```

В этом случае (декоратор `@Resolver()` на уровне метода), если внутри класса несколько декораторов `@ResolveField()`, `@Resolver()` придётся добавить ко всем из них. Лучшей практикой это не считается (поскольку создаёт лишние накладные расходы).

> **Подсказка** Любой аргумент с именем класса, переданный в `@Resolver()`, **не** влияет на запросы (декоратор `@Query()`) и мутации (декоратор `@Mutation()`).

> **Предупреждение** Использование декоратора `@Resolver` на уровне метода не поддерживается в подходе **code first**.

В примерах выше декораторы `@Query()` и `@ResolveField()` связываются с типами схемы GraphQL на основе имени метода. Например, рассмотрим следующую конструкцию из примера выше:

```typescript
@Query()
async author(@Args('id') id: number) {
  return this.authorsService.findOneById(id);
}
```

Она порождает следующую запись для запроса author в нашей схеме (тип запроса получает то же имя, что и метод):

```graphql
type Query {
  author(id: Int!): Author
}
```

По соглашению мы предпочли бы разделить их, используя для методов резолвера имена вроде `getAuthor()` или `getPosts()`. Сделать это легко, передав имя сопоставления в качестве аргумента декоратора:

**`authors/authors.resolver.ts`**

```typescript
@Resolver('Author')
export class AuthorsResolver {
  constructor(
    private authorsService: AuthorsService,
    private postsService: PostsService,
  ) {}

  @Query('author')
  async getAuthor(@Args('id') id: number) {
    return this.authorsService.findOneById(id);
  }

  @ResolveField('posts')
  async getPosts(@Parent() author) {
    const { id } = author;
    return this.postsService.findAll({ authorId: id });
  }
}
```

> **Подсказка** Nest CLI предоставляет генератор (schematic), который автоматически создаёт **весь шаблонный код**, помогая избежать всей этой ручной работы и значительно упрощая жизнь разработчика. Подробнее об этой возможности читайте [здесь](<../13. Recipes/02. CRUD generator.md>).

#### Генерация типов

Предположим, что мы используем подход schema first и включили генерацию типов (с `outputAs: 'class'`, как показано в [предыдущей](<01. Quick start.md>) главе). Тогда после запуска приложения будет сгенерирован следующий файл (в том месте, которое вы указали в методе `GraphQLModule.forRoot()`). Например, в `src/graphql.ts`:

**`graphql.ts`**

```typescript
export class Author {
  id: number;
  firstName?: string;
  lastName?: string;
  posts?: Post[];
}
export class Post {
  id: number;
  title: string;
  votes?: number;
}

export abstract class IQuery {
  abstract author(id: number): Author | Promise<Author>;
}
```

Генерируя классы (вместо интерфейсов, как в технике по умолчанию), вы можете использовать декларативные **декораторы** валидации в сочетании с подходом schema first, что является чрезвычайно полезным приёмом (читайте [подробнее](<../04. Techniques/04. Validation.md>)). Например, вы можете добавить декораторы `class-validator` к сгенерированному классу `CreatePostInput`, как показано ниже, чтобы задать минимальную и максимальную длину строки для поля `title`:

```typescript
import { MinLength, MaxLength } from 'class-validator';

export class CreatePostInput {
  @MinLength(3)
  @MaxLength(50)
  title: string;
}
```

> **Важно** Чтобы включить автоматическую валидацию входных данных (и параметров), используйте `ValidationPipe`. Подробнее о валидации читайте [здесь](<../04. Techniques/04. Validation.md>), а конкретно о pipe'ах — [здесь](<../02. Overview/07. Pipes.md>).

Однако если вы добавите декораторы прямо в автоматически генерируемый файл, они будут **перезаписаны** при каждой генерации файла. Вместо этого создайте отдельный файл и просто расширьте сгенерированный класс.

```typescript
import { MinLength, MaxLength } from 'class-validator';
import { Post } from '../../graphql.ts';

export class CreatePostInput extends Post {
  @MinLength(3)
  @MaxLength(50)
  title: string;
}
```

#### Декораторы аргументов GraphQL

К стандартным аргументам резолвера GraphQL можно обращаться с помощью специальных декораторов. Ниже приведено сопоставление декораторов Nest и параметров Apollo, которые они представляют.

<table>
  <tbody>
    <tr>
      <td><code>@Root()</code> и <code>@Parent()</code></td>
      <td><code>root</code>/<code>parent</code></td>
    </tr>
    <tr>
      <td><code>@Context(param?: string)</code></td>
      <td><code>context</code> / <code>context[param]</code></td>
    </tr>
    <tr>
      <td><code>@Info(param?: string)</code></td>
      <td><code>info</code> / <code>info[param]</code></td>
    </tr>
    <tr>
      <td><code>@Args(param?: string)</code></td>
      <td><code>args</code> / <code>args[param]</code></td>
    </tr>
  </tbody>
</table>

Эти аргументы означают следующее:

- `root`: объект, содержащий результат, возвращённый резолвером родительского поля, либо — в случае поля верхнего уровня `Query` — значение `rootValue`, переданное из конфигурации сервера.
- `context`: объект, общий для всех резолверов в рамках конкретного запроса; обычно используется для хранения состояния, относящегося к запросу.
- `info`: объект с информацией о состоянии выполнения запроса.
- `args`: объект с аргументами, переданными полю в запросе.

#### Модуль

После выполнения описанных выше шагов мы декларативно указали всю информацию, необходимую `GraphQLModule` для генерации карты резолверов. `GraphQLModule` использует рефлексию, чтобы проанализировать метаданные, предоставленные через декораторы, и автоматически преобразует классы в корректную карту резолверов.

Единственное, о чём ещё нужно позаботиться, — это **предоставить** классы резолверов (`AuthorsResolver`), то есть перечислить их как `provider` в каком-либо модуле, и импортировать этот модуль (`AuthorsModule`) где-нибудь, чтобы Nest мог его использовать.

Например, мы можем сделать это в `AuthorsModule`, который также может предоставлять другие сервисы, необходимые в этом контексте. Не забудьте импортировать `AuthorsModule` где-нибудь (например, в корневом модуле или в каком-то другом модуле, импортируемом корневым).

**`authors/authors.module.ts`**

```typescript
@Module({
  imports: [PostsModule],
  providers: [AuthorsService, AuthorsResolver],
})
export class AuthorsModule {}
```

> **Подсказка** Полезно организовывать код по так называемой **доменной модели** (примерно так же, как вы организовывали бы точки входа в REST API). При этом подходе держите модели (классы `ObjectType`), резолверы и сервисы вместе внутри модуля Nest, представляющего доменную модель. Храните все эти компоненты в одной папке на модуль. Если вы поступите так и будете использовать [Nest CLI](<../11. CLI/01. Overview.md>) для генерации каждого элемента, Nest автоматически свяжет все эти части между собой (разместит файлы в подходящих папках, добавит записи в массивы `provider` и `imports` и т. д.).
