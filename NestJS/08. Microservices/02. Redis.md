### Redis

Транспорт [Redis](https://redis.io/) реализует модель обмена сообщениями publish/subscribe и опирается на возможность [Pub/Sub](https://redis.io/topics/pubsub) в Redis. Публикуемые сообщения распределяются по каналам, при этом отправитель не знает, какие подписчики (если они вообще есть) в итоге получат сообщение. Каждый микросервис может подписаться на любое количество каналов. Кроме того, можно одновременно подписаться сразу на несколько каналов. Обмен сообщениями через каналы работает по принципу **fire-and-forget** («отправил и забыл»): если сообщение опубликовано, а заинтересованных подписчиков нет, оно удаляется и восстановить его невозможно. Таким образом, нет гарантии, что сообщение или событие будет обработано хотя бы одним сервисом. Одно и то же сообщение может быть получено несколькими подписчиками.

#### Установка

Чтобы начать создавать микросервисы на базе Redis, сначала установите необходимый пакет:

```bash
$ npm i --save ioredis
```

#### Обзор

Чтобы использовать транспорт Redis, передайте в метод `createMicroservice()` следующий объект опций:

**`main.ts`**

```typescript
const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  transport: Transport.REDIS,
  options: {
    host: 'localhost',
    port: 6379,
  },
});
```

> **Подсказка** Перечисление `Transport` импортируется из пакета `@nestjs/microservices`.

#### Опции

Свойство `options` зависит от выбранного транспорта. Транспорт <strong>Redis</strong> предоставляет описанные ниже свойства.

<table>
  <tr>
    <td><code>host</code></td>
    <td>URL подключения</td>
  </tr>
  <tr>
    <td><code>port</code></td>
    <td>Порт подключения</td>
  </tr>
  <tr>
    <td><code>retryAttempts</code></td>
    <td>Количество повторных попыток отправки сообщения (по умолчанию: <code>0</code>)</td>
  </tr>
  <tr>
    <td><code>retryDelay</code></td>
    <td>Задержка между повторными попытками отправки сообщения (мс) (по умолчанию: <code>0</code>)</td>
  </tr>
   <tr>
    <td><code>wildcards</code></td>
    <td>Включает подписки с подстановочными знаками в Redis, указывая транспорту использовать «под капотом» <code>psubscribe</code>/<code>pmessage</code>. (по умолчанию: <code>false</code>)</td>
  </tr>
</table>

Все свойства, поддерживаемые официальным клиентом [ioredis](https://redis.github.io/ioredis/index.html#RedisOptions), также поддерживаются этим транспортом.

#### Клиент

Как и для других транспортов микросервисов, у вас есть <a href="01.%20Overview.md#клиент-класс-производитель">несколько вариантов</a> создания экземпляра `ClientProxy` для Redis.

Один из способов — использовать `ClientsModule`. Чтобы создать экземпляр клиента через `ClientsModule`, импортируйте его и вызовите метод `register()`, передав объект опций с теми же свойствами, что показаны выше для метода `createMicroservice()`, а также свойство `name`, которое будет использоваться как токен внедрения. Подробнее о `ClientsModule` читайте <a href="01.%20Overview.md#клиент-класс-производитель">здесь</a>.

```typescript
@Module({
  imports: [
    ClientsModule.register([
      {
        name: 'MATH_SERVICE',
        transport: Transport.REDIS,
        options: {
          host: 'localhost',
          port: 6379,
        }
      },
    ]),
  ]
  ...
})
```

Также можно использовать другие способы создания клиента (`ClientProxyFactory` или `@Client()`). Прочитать о них можно <a href="01.%20Overview.md#клиент-класс-производитель">здесь</a>.

#### Контекст

В более сложных сценариях может потребоваться доступ к дополнительной информации о входящем запросе. При использовании транспорта Redis вы можете обратиться к объекту `RedisContext`.

```typescript
@MessagePattern('notifications')
getNotifications(@Payload() data: number[], @Ctx() context: RedisContext) {
  console.log(`Channel: ${context.getChannel()}`);
}
```

> **Подсказка** `@Payload()`, `@Ctx()` и `RedisContext` импортируются из пакета `@nestjs/microservices`.

#### Подстановочные знаки

Чтобы включить поддержку подстановочных знаков, задайте опции `wildcards` значение `true`. Это укажет транспорту использовать «под капотом» `psubscribe` и `pmessage`.

```typescript
const app = await NestFactory.createMicroservice(AppModule, {
  transport: Transport.REDIS,
  options: {
    // Другие опции
    wildcards: true,
  },
});
```

Не забудьте передать опцию `wildcards` и при создании экземпляра клиента.

С включённой опцией вы можете использовать подстановочные знаки в шаблонах сообщений и событий. Например, чтобы подписаться на все каналы, начинающиеся с `notifications`, используйте следующий шаблон:

```typescript
@EventPattern('notifications.*')
```

#### Обновления статуса экземпляра

Чтобы получать обновления о состоянии соединения и экземпляра нижележащего драйвера в реальном времени, вы можете подписаться на поток `status`. Этот поток выдаёт обновления статуса, специфичные для выбранного драйвера. Для драйвера Redis поток `status` выдаёт события `connected`, `disconnected` и `reconnecting`.

```typescript
this.client.status.subscribe((status: RedisStatus) => {
  console.log(status);
});
```

> **Подсказка** Тип `RedisStatus` импортируется из пакета `@nestjs/microservices`.

Аналогично можно подписаться на поток `status` сервера, чтобы получать уведомления о его состоянии.

```typescript
const server = app.connectMicroservice<MicroserviceOptions>(...);
server.status.subscribe((status: RedisStatus) => {
  console.log(status);
});
```

#### Прослушивание событий Redis

В некоторых случаях может потребоваться прослушивать внутренние события, выдаваемые микросервисом. Например, вы можете слушать событие `error`, чтобы выполнять дополнительные действия при возникновении ошибки. Для этого используйте метод `on()`, как показано ниже:

```typescript
this.client.on('error', (err) => {
  console.error(err);
});
```

Точно так же можно слушать внутренние события сервера:

```typescript
server.on<RedisEvents>('error', (err) => {
  console.error(err);
});
```

> **Подсказка** Тип `RedisEvents` импортируется из пакета `@nestjs/microservices`.

#### Доступ к нижележащему драйверу

В более сложных сценариях может потребоваться доступ к экземпляру нижележащего драйвера. Это бывает полезно, например, для ручного закрытия соединения или использования специфичных для драйвера методов. Однако имейте в виду, что в большинстве случаев обращаться к драйверу напрямую **не требуется**.

Для этого используйте метод `unwrap()`, который возвращает экземпляр нижележащего драйвера. Параметр обобщённого типа должен задавать ожидаемый тип экземпляра драйвера.

```typescript
const [pub, sub] =
  this.client.unwrap<[import('ioredis').Redis, import('ioredis').Redis]>();
```

Аналогично можно обратиться к экземпляру нижележащего драйвера сервера:

```typescript
const [pub, sub] =
  server.unwrap<[import('ioredis').Redis, import('ioredis').Redis]>();
```

Обратите внимание: в отличие от других транспортов, транспорт Redis возвращает кортеж из двух экземпляров `ioredis`: первый используется для публикации сообщений, а второй — для подписки на них.
