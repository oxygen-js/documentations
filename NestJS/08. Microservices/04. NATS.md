### NATS

[NATS](https://nats.io) — простая, безопасная и высокопроизводительная система обмена сообщениями с открытым исходным кодом, предназначенная для облачных приложений, обмена сообщениями в IoT и микросервисных архитектур. Сервер NATS написан на языке Go, но клиентские библиотеки для взаимодействия с ним доступны для десятков основных языков программирования. NATS поддерживает доставку как **At Most Once**, так и **At Least Once**. Он может работать где угодно — от крупных серверов и облачных инстансов до пограничных шлюзов и даже устройств интернета вещей.

#### Установка

Чтобы начать создавать микросервисы на базе NATS, сначала установите необходимый пакет:

```bash
$ npm i --save nats
```

#### Обзор

Чтобы использовать транспорт NATS, передайте в метод `createMicroservice()` следующий объект опций:

**`main.ts`**

```typescript
const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  transport: Transport.NATS,
  options: {
    servers: ['nats://localhost:4222'],
  },
});
```

> **Подсказка** Перечисление `Transport` импортируется из пакета `@nestjs/microservices`.

#### Опции

Объект `options` зависит от выбранного транспорта. Транспорт <strong>NATS</strong> предоставляет свойства, описанные [здесь](https://github.com/nats-io/node-nats#connection-options), а также следующие свойства:

<table>
  <tr>
    <td><code>queue</code></td>
    <td>Очередь, на которую должен подписаться ваш сервер (оставьте <code>undefined</code>, чтобы не использовать эту настройку). Подробнее о группах очередей NATS читайте <a href="04.%20NATS.md#группы-очередей">ниже</a>.
    </td> 
  </tr>
  <tr>
    <td><code>gracefulShutdown</code></td>
    <td>Включает корректное завершение работы. Когда опция включена, сервер сначала отписывается от всех каналов и только затем закрывает соединение. По умолчанию <code>false</code>.
  </tr>
  <tr>
    <td><code>gracePeriod</code></td>
    <td>Время в миллисекундах, которое сервер ожидает после отписки от всех каналов. По умолчанию <code>10000</code> мс.
  </tr>
</table>

#### Клиент

Как и для других транспортов микросервисов, у вас есть <a href="01.%20Overview.md#клиент-класс-производитель">несколько вариантов</a> создания экземпляра `ClientProxy` для NATS.

Один из способов — использовать `ClientsModule`. Чтобы создать экземпляр клиента через `ClientsModule`, импортируйте его и вызовите метод `register()`, передав объект опций с теми же свойствами, что показаны выше для метода `createMicroservice()`, а также свойство `name`, которое будет использоваться как токен внедрения. Подробнее о `ClientsModule` читайте <a href="01.%20Overview.md#клиент-класс-производитель">здесь</a>.

```typescript
@Module({
  imports: [
    ClientsModule.register([
      {
        name: 'MATH_SERVICE',
        transport: Transport.NATS,
        options: {
          servers: ['nats://localhost:4222'],
        }
      },
    ]),
  ]
  ...
})
```

Также можно использовать другие способы создания клиента (`ClientProxyFactory` или `@Client()`). Прочитать о них можно <a href="01.%20Overview.md#клиент-класс-производитель">здесь</a>.

#### Запрос-ответ

Для стиля обмена сообщениями **запрос-ответ** ([подробнее](<01. Overview.md#запрос-ответ>)) транспорт NATS не использует встроенный механизм [Request-Reply](https://docs.nats.io/nats-concepts/reqreply). Вместо этого «запрос» публикуется в заданный subject методом `publish()` с уникальным именем subject для ответа, а обработчики слушают этот subject и отправляют ответы в subject для ответа. Subject'ы для ответа динамически направляются обратно запрашивающей стороне независимо от местоположения обеих сторон.

#### На основе событий

Для стиля обмена сообщениями **на основе событий** ([подробнее](<01. Overview.md#на-основе-событий>)) транспорт NATS использует встроенный механизм [Publish-Subscribe](https://docs.nats.io/nats-concepts/pubsub). Издатель отправляет сообщение в subject, и любой активный подписчик, слушающий этот subject, получает сообщение. Подписчики также могут подписываться на subject'ы с подстановочными знаками, которые работают примерно как регулярные выражения. Такую схему «один ко многим» иногда называют fan-out.

#### Группы очередей

NATS предоставляет встроенный механизм балансировки нагрузки — [распределённые очереди](https://docs.nats.io/nats-concepts/queue). Чтобы создать подписку на очередь, используйте свойство `queue`, как показано ниже:

**`main.ts`**

```typescript
const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  transport: Transport.NATS,
  options: {
    servers: ['nats://localhost:4222'],
    queue: 'cats_queue',
  },
});
```

#### Контекст

В более сложных сценариях может потребоваться доступ к дополнительной информации о входящем запросе. При использовании транспорта NATS вы можете обратиться к объекту `NatsContext`.

```typescript
@MessagePattern('notifications')
getNotifications(@Payload() data: number[], @Ctx() context: NatsContext) {
  console.log(`Subject: ${context.getSubject()}`);
}
```

> **Подсказка** `@Payload()`, `@Ctx()` и `NatsContext` импортируются из пакета `@nestjs/microservices`.

#### Подстановочные знаки

Подписка может быть как на явно заданный subject, так и содержать подстановочные знаки.

```typescript
@MessagePattern('time.us.*')
getDate(@Payload() data: number[], @Ctx() context: NatsContext) {
  console.log(`Subject: ${context.getSubject()}`); // например, "time.us.east"
  return new Date().toLocaleTimeString(...);
}
```

#### Построители записей

Для настройки опций сообщения можно использовать класс `NatsRecordBuilder` (отметим, что это применимо и к потокам на основе событий). Например, чтобы добавить заголовок `x-version`, используйте метод `setHeaders`:

```typescript
import * as nats from 'nats';

// где-то в вашем коде
const headers = nats.headers();
headers.set('x-version', '1.0.0');

const record = new NatsRecordBuilder(':cat:').setHeaders(headers).build();
this.client.send('replace-emoji', record).subscribe(...);
```

> **Подсказка** Класс `NatsRecordBuilder` экспортируется из пакета `@nestjs/microservices`.

Эти заголовки можно прочитать и на стороне сервера, обратившись к `NatsContext`:

```typescript
@MessagePattern('replace-emoji')
replaceEmoji(@Payload() data: string, @Ctx() context: NatsContext): string {
  const headers = context.getHeaders();
  return headers['x-version'] === '1.0.0' ? '🐱' : '🐈';
}
```

Иногда требуется задать заголовки сразу для нескольких запросов — их можно передать как опции в `ClientProxyFactory`:

```typescript
import { Module } from '@nestjs/common';
import { ClientProxyFactory, Transport } from '@nestjs/microservices';

@Module({
  providers: [
    {
      provide: 'API_v1',
      useFactory: () =>
        ClientProxyFactory.create({
          transport: Transport.NATS,
          options: {
            servers: ['nats://localhost:4222'],
            headers: { 'x-version': '1.0.0' },
          },
        }),
    },
  ],
})
export class ApiModule {}
```

#### Обновления статуса экземпляра

Чтобы получать обновления о состоянии соединения и экземпляра нижележащего драйвера в реальном времени, вы можете подписаться на поток `status`. Этот поток выдаёт обновления статуса, специфичные для выбранного драйвера. Для драйвера NATS поток `status` выдаёт события `connected`, `disconnected` и `reconnecting`.

```typescript
this.client.status.subscribe((status: NatsStatus) => {
  console.log(status);
});
```

> **Подсказка** Тип `NatsStatus` импортируется из пакета `@nestjs/microservices`.

Аналогично можно подписаться на поток `status` сервера, чтобы получать уведомления о его состоянии.

```typescript
const server = app.connectMicroservice<MicroserviceOptions>(...);
server.status.subscribe((status: NatsStatus) => {
  console.log(status);
});
```

#### Прослушивание событий Nats

В некоторых случаях может потребоваться прослушивать внутренние события, выдаваемые микросервисом. Например, вы можете слушать событие `error`, чтобы выполнять дополнительные действия при возникновении ошибки. Для этого используйте метод `on()`, как показано ниже:

```typescript
this.client.on('error', (err) => {
  console.error(err);
});
```

Точно так же можно слушать внутренние события сервера:

```typescript
server.on<NatsEvents>('error', (err) => {
  console.error(err);
});
```

> **Подсказка** Тип `NatsEvents` импортируется из пакета `@nestjs/microservices`.

#### Доступ к нижележащему драйверу

В более сложных сценариях может потребоваться доступ к экземпляру нижележащего драйвера. Это бывает полезно, например, для ручного закрытия соединения или использования специфичных для драйвера методов. Однако имейте в виду, что в большинстве случаев обращаться к драйверу напрямую **не требуется**.

Для этого используйте метод `unwrap()`, который возвращает экземпляр нижележащего драйвера. Параметр обобщённого типа должен задавать ожидаемый тип экземпляра драйвера.

```typescript
const natsConnection = this.client.unwrap<import('nats').NatsConnection>();
```

Аналогично можно обратиться к экземпляру нижележащего драйвера сервера:

```typescript
const natsConnection = server.unwrap<import('nats').NatsConnection>();
```
