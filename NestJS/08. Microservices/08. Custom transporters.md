### Пользовательские транспорты

Nest предоставляет разнообразные **транспорты** «из коробки», а также API, позволяющий разработчикам создавать новые пользовательские транспортные стратегии.
Транспорты дают возможность соединять компоненты по сети с помощью подключаемого коммуникационного слоя и очень простого протокола сообщений уровня приложения (полностью читайте в [статье](https://dev.to/nestjs/integrate-nestjs-with-external-services-using-microservice-transporters-part-1-p3)).

> **Подсказка** Создание микросервиса на Nest вовсе не обязательно означает, что вы должны использовать пакет `@nestjs/microservices`. Например, если вы хотите взаимодействовать с внешними сервисами (скажем, с другими микросервисами, написанными на других языках), вам могут не понадобиться все возможности библиотеки `@nestjs/microservice`.
> На деле, если вам не нужны декораторы (`@EventPattern` или `@MessagePattern`), позволяющие декларативно объявлять подписчиков, для большинства сценариев будет достаточно запустить [автономное приложение](<../10. Standalone apps.md>) и вручную поддерживать соединение и подписку на каналы — это к тому же даст больше гибкости.

С помощью пользовательского транспорта вы можете интегрировать любую систему обмена сообщениями или протокол (включая Google Cloud Pub/Sub, Amazon Kinesis и другие) либо расширить существующий, добавив к нему дополнительные возможности (например, [QoS](https://github.com/mqttjs/MQTT.js/blob/master/README.md#qos) для MQTT).

> **Подсказка** Чтобы лучше понять, как работают микросервисы Nest и как можно расширить возможности существующих транспортов, рекомендуем прочитать серии статей [NestJS Microservices in Action](https://dev.to/johnbiundo/series/4724) и [Advanced NestJS Microservices](https://dev.to/nestjs/part-1-introduction-and-setup-1a2l).

#### Создание стратегии

Сначала определим класс, представляющий наш пользовательский транспорт.

```typescript
import { CustomTransportStrategy, Server } from '@nestjs/microservices';

class GoogleCloudPubSubServer
  extends Server
  implements CustomTransportStrategy
{
  /**
   * Срабатывает при вызове "app.listen()".
   */
  listen(callback: () => void) {
    callback();
  }

  /**
   * Срабатывает при завершении работы приложения.
   */
  close() {}

  /**
   * Этот метод можно проигнорировать, если вы не хотите, чтобы пользователи
   * транспорта могли регистрировать слушателей событий. Большинству пользовательских
   * реализаций он не нужен.
   */
  on(event: string, callback: Function) {
    throw new Error('Method not implemented.');
  }

  /**
   * Этот метод можно проигнорировать, если вы не хотите, чтобы пользователи транспорта
   * могли получить нижележащий нативный сервер. Большинству пользовательских
   * реализаций он не нужен.
   */
  unwrap<T = never>(): T {
    throw new Error('Method not implemented.');
  }
}
```

> **Предупреждение** Обратите внимание: в этой главе мы не будем реализовывать полнофункциональный сервер Google Cloud Pub/Sub, поскольку это потребовало бы погружения в технические детали конкретного транспорта.

В примере выше мы объявили класс `GoogleCloudPubSubServer` и предоставили методы `listen()` и `close()`, обязательные согласно интерфейсу `CustomTransportStrategy`.
Кроме того, наш класс наследуется от класса `Server`, импортируемого из пакета `@nestjs/microservices`, который даёт несколько полезных методов — например, методы, используемые средой выполнения Nest для регистрации обработчиков сообщений. В качестве альтернативы, если вы хотите расширить возможности существующей транспортной стратегии, можно унаследоваться от соответствующего класса сервера, например `ServerRedis`.
По соглашению мы добавили к имени класса суффикс `"Server"`, так как он будет отвечать за подписку на сообщения и события (и при необходимости за ответы на них).

Теперь мы можем использовать нашу собственную стратегию вместо встроенного транспорта:

```typescript
const app = await NestFactory.createMicroservice<MicroserviceOptions>(
  AppModule,
  {
    strategy: new GoogleCloudPubSubServer(),
  },
);
```

По сути, вместо обычного объекта опций транспорта со свойствами `transport` и `options` мы передаём единственное свойство `strategy`, значением которого является экземпляр нашего класса-транспорта.

Возвращаясь к классу `GoogleCloudPubSubServer`: в реальном приложении в методе `listen()` мы бы устанавливали соединение с брокером сообщений или внешним сервисом и регистрировали подписчиков либо слушали определённые каналы (а затем снимали подписки и закрывали соединение в методе очистки `close()`),
но поскольку это требует хорошего понимания того, как микросервисы Nest взаимодействуют друг с другом, рекомендуем прочитать эту [серию статей](https://dev.to/nestjs/part-1-introduction-and-setup-1a2l).
В этой же главе мы сосредоточимся на возможностях класса `Server` и на том, как их применять для создания пользовательских стратегий.

Допустим, например, что где-то в нашем приложении объявлен следующий обработчик сообщений:

```typescript
@MessagePattern('echo')
echo(@Payload() data: object) {
  return data;
}
```

Этот обработчик сообщений будет автоматически зарегистрирован средой выполнения Nest. С помощью класса `Server` вы можете увидеть, какие шаблоны сообщений зарегистрированы, а также получить доступ к назначенным им методам и выполнить их.
Чтобы проверить это, добавим простой `console.log` внутри метода `listen()` перед вызовом функции `callback`:

```typescript
listen(callback: () => void) {
  console.log(this.messageHandlers);
  callback();
}
```

После перезапуска приложения вы увидите в терминале следующую запись:

```typescript
Map { 'echo' => [AsyncFunction] { isEventHandler: false } }
```

> **Подсказка** Если бы мы использовали декоратор `@EventPattern`, вывод был бы таким же, но со свойством `isEventHandler`, равным `true`.

Как видите, свойство `messageHandlers` — это коллекция `Map` со всеми обработчиками сообщений (и событий), в которой ключами служат шаблоны.
Теперь по ключу (например, `"echo"`) можно получить ссылку на обработчик сообщений:

```typescript
async listen(callback: () => void) {
  const echoHandler = this.messageHandlers.get('echo');
  console.log(await echoHandler('Hello world!'));
  callback();
}
```

Как только мы выполним `echoHandler`, передав произвольную строку в качестве аргумента (здесь — `"Hello world!"`), мы должны увидеть её в консоли:

```json
Hello world!
```

Это означает, что наш метод-обработчик был выполнен корректно.

При использовании `CustomTransportStrategy` вместе с [interceptor'ами](<../02. Overview/09. Interceptors.md>) обработчики оборачиваются в потоки RxJS. Это значит, что для выполнения логики, лежащей в основе потока, на них нужно подписаться (например, чтобы перейти к логике контроллера после выполнения interceptor'а).

Пример этого приведён ниже:

```typescript
async listen(callback: () => void) {
  const echoHandler = this.messageHandlers.get('echo');
  const streamOrResult = await echoHandler('Hello World');
  if (isObservable(streamOrResult)) {
    streamOrResult.subscribe();
  }
  callback();
}
```

#### Клиентский прокси

Как мы упоминали в первом разделе, использовать пакет `@nestjs/microservices` для создания микросервисов не обязательно, но если вы всё же решите это сделать и вам нужно интегрировать пользовательскую стратегию, потребуется предоставить ещё и класс «клиента».

> **Подсказка** Повторим: реализация полнофункционального класса клиента, совместимого со всеми возможностями `@nestjs/microservices` (например, с потоковой передачей), требует хорошего понимания используемых фреймворком приёмов взаимодействия. Подробнее — в этой [статье](https://dev.to/nestjs/part-4-basic-client-component-16f9).

Чтобы взаимодействовать с внешним сервисом и отправлять или публиковать сообщения (или события), можно либо воспользоваться SDK-пакетом конкретной библиотеки, либо реализовать собственный класс клиента, наследующий `ClientProxy`:

```typescript
import { ClientProxy, ReadPacket, WritePacket } from '@nestjs/microservices';

class GoogleCloudPubSubClient extends ClientProxy {
  async connect(): Promise<any> {}
  async close() {}
  async dispatchEvent(packet: ReadPacket<any>): Promise<any> {}
  publish(
    packet: ReadPacket<any>,
    callback: (packet: WritePacket<any>) => void,
  ): Function {}
  unwrap<T = never>(): T {
    throw new Error('Method not implemented.');
  }
}
```

> **Предупреждение** Обратите внимание: в этой главе мы не будем реализовывать полнофункциональный клиент Google Cloud Pub/Sub, поскольку это потребовало бы погружения в технические детали конкретного транспорта.

Как видите, класс `ClientProxy` требует от нас реализовать несколько методов для установки и закрытия соединения, а также для публикации сообщений (`publish`) и событий (`dispatchEvent`).
Заметьте: если вам не нужна поддержка стиля взаимодействия «запрос — ответ», метод `publish()` можно оставить пустым. Точно так же, если не нужна поддержка взаимодействия на основе событий, пропустите метод `dispatchEvent()`.

Чтобы понаблюдать, что и когда выполняется в этих методах, добавим несколько вызовов `console.log`:

```typescript
class GoogleCloudPubSubClient extends ClientProxy {
  async connect(): Promise<any> {
    console.log('connect');
  }

  async close() {
    console.log('close');
  }

  async dispatchEvent(packet: ReadPacket<any>): Promise<any> {
    return console.log('event to dispatch: ', packet);
  }

  publish(
    packet: ReadPacket<any>,
    callback: (packet: WritePacket<any>) => void,
  ): Function {
    console.log('message:', packet);

    // В реальном приложении функция "callback" должна вызываться
    // с полезной нагрузкой, присланной обратно отвечающей стороной. Здесь же мы просто имитируем
    // (с задержкой в 5 секунд), что ответ пришёл, передавая те же "data", что и отправили изначально.
    //
    // Логическое поле "isDisposed" в WritePacket сообщает ответу, что дальнейших данных
    // не ожидается. Если оно не передано или равно false, данные просто будут отправлены в Observable.
    setTimeout(() => callback({ 
      response: packet.data,
      isDisposed: true,
    }), 5000);

    return () => console.log('teardown');
  }

  unwrap<T = never>(): T {
    throw new Error('Method not implemented.');
  }
}
```

Теперь создадим экземпляр класса `GoogleCloudPubSubClient` и вызовем метод `send()` (который вы могли видеть в предыдущих главах), подписавшись на возвращённый поток observable.

```typescript
const googlePubSubClient = new GoogleCloudPubSubClient();
googlePubSubClient
  .send('pattern', 'Hello world!')
  .subscribe((response) => console.log(response));
```

Теперь в терминале должен появиться следующий вывод:

```typescript
connect
message: { pattern: 'pattern', data: 'Hello world!' }
Hello world! // <-- after 5 seconds
```

Чтобы проверить, корректно ли выполняется наш метод очистки (возвращаемый методом `publish()`), применим к потоку оператор timeout со значением 2 секунды — так, чтобы он сработал раньше, чем `setTimeout` вызовет функцию `callback`.

```typescript
const googlePubSubClient = new GoogleCloudPubSubClient();
googlePubSubClient
  .send('pattern', 'Hello world!')
  .pipe(timeout(2000))
  .subscribe(
    (response) => console.log(response),
    (error) => console.error(error.message),
  );
```

> **Подсказка** Оператор `timeout` импортируется из пакета `rxjs/operators`.

С применённым оператором `timeout` вывод в терминале должен выглядеть так:

```typescript
connect
message: { pattern: 'pattern', data: 'Hello world!' }
teardown // <-- teardown
Timeout has occurred
```

Чтобы отправить событие (вместо сообщения), используйте метод `emit()`:

```typescript
googlePubSubClient.emit('event', 'Hello world!');
```

И вот что вы должны увидеть в консоли:

```typescript
connect
event to dispatch:  { pattern: 'event', data: 'Hello world!' }
```

#### Сериализация сообщений

Если вам нужно добавить собственную логику вокруг сериализации ответов на стороне клиента, можно воспользоваться пользовательским классом, наследующим класс `ClientProxy` или один из его потомков. Для изменения успешных запросов можно переопределить метод `serializeResponse`, а для изменения любых ошибок, проходящих через этот клиент, — метод `serializeError`. Чтобы задействовать такой класс, передайте сам класс в метод `ClientsModule.register()` через свойство `customClass`. Ниже приведён пример пользовательского `ClientProxy`, который сериализует каждую ошибку в `RpcException`.

**`error-handling.proxy.ts`**

```typescript
import { ClientTCP, RpcException } from '@nestjs/microservices';

class ErrorHandlingProxy extends ClientTCP {
  serializeError(err: Error) {
    return new RpcException(err);
  }
}
```

а затем используйте его в `ClientsModule` следующим образом:

**`app.module.ts`**

```typescript
@Module({
  imports: [
    ClientsModule.register([{
      name: 'CustomProxy',
      customClass: ErrorHandlingProxy,
    }]),
  ]
})
export class AppModule
```

> **Подсказка** В `customClass` передаётся сам класс, а не его экземпляр. Nest создаст экземпляр «под капотом» за вас и передаст новому `ClientProxy` все опции, заданные в свойстве `options`.
