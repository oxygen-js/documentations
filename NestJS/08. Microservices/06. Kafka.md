### Kafka

[Kafka](https://kafka.apache.org/) — распределённая платформа потоковой передачи с открытым исходным кодом, обладающая тремя ключевыми возможностями:

- Публикация потоков записей и подписка на них — подобно очереди сообщений или корпоративной системе обмена сообщениями.
- Отказоустойчивое и долговременное хранение потоков записей.
- Обработка потоков записей по мере их появления.

Проект Kafka нацелен на то, чтобы дать единую платформу с высокой пропускной способностью и малой задержкой для обработки потоков данных в реальном времени. Он очень хорошо интегрируется с Apache Storm и Spark для анализа потоковых данных в реальном времени.

#### Установка

Чтобы начать создавать микросервисы на основе Kafka, сначала установите необходимый пакет:

```bash
$ npm i --save kafkajs
```

#### Обзор

Как и в других реализациях транспортного слоя микросервисов Nest, механизм транспорта Kafka выбирается через свойство `transport` объекта опций, передаваемого в метод `createMicroservice()`, вместе с необязательным свойством `options`, как показано ниже:

**`main.ts`**

```typescript
const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  transport: Transport.KAFKA,
  options: {
    client: {
      brokers: ['localhost:9092'],
    }
  }
});
```

> **Подсказка** Перечисление `Transport` импортируется из пакета `@nestjs/microservices`.

#### Клиент

По сравнению с другими транспортами микросервисов у Kafka есть небольшое отличие. Вместо класса `ClientProxy` здесь используется класс `ClientKafkaProxy`.

Как и в случае других транспортов микросервисов, для создания экземпляра `ClientKafkaProxy` у вас есть <a href="01.%20Overview.md#клиент-класс-производитель">несколько вариантов</a>.

Один из способов создать экземпляр — воспользоваться `ClientsModule`. Чтобы создать экземпляр клиента с помощью `ClientsModule`, импортируйте его и вызовите метод `register()`, передав объект опций с теми же свойствами, что показаны выше для метода `createMicroservice()`, а также свойство `name`, которое будет использоваться как токен внедрения. Подробнее о `ClientsModule` читайте <a href="01.%20Overview.md#клиент-класс-производитель">здесь</a>.

```typescript
@Module({
  imports: [
    ClientsModule.register([
      {
        name: 'HERO_SERVICE',
        transport: Transport.KAFKA,
        options: {
          client: {
            clientId: 'hero',
            brokers: ['localhost:9092'],
          },
          consumer: {
            groupId: 'hero-consumer'
          }
        }
      },
    ]),
  ]
  ...
})
```

Можно использовать и другие способы создания клиента (`ClientProxyFactory` или `@Client()`). Прочитать о них можно <a href="01.%20Overview.md#клиент-класс-производитель">здесь</a>.

Декоратор `@Client()` используется так:

```typescript
@Client({
  transport: Transport.KAFKA,
  options: {
    client: {
      clientId: 'hero',
      brokers: ['localhost:9092'],
    },
    consumer: {
      groupId: 'hero-consumer'
    }
  }
})
client: ClientKafkaProxy;
```

#### Шаблон сообщений

Шаблон сообщений микросервиса Kafka использует два топика — для канала запроса и канала ответа. Метод `ClientKafkaProxy.send()` отправляет сообщения с [обратным адресом](https://www.enterpriseintegrationpatterns.com/patterns/messaging/ReturnAddress.html), связывая с сообщением-запросом [идентификатор корреляции](https://www.enterpriseintegrationpatterns.com/patterns/messaging/CorrelationIdentifier.html), топик ответа и партицию ответа. Поэтому экземпляр `ClientKafkaProxy` перед отправкой сообщения должен быть подписан на топик ответа и получить как минимум одну партицию.

Следовательно, на каждое работающее приложение Nest должна приходиться как минимум одна партиция топика ответа. Например, если у вас запущено 4 приложения Nest, а в топике ответа всего 3 партиции, то одно из приложений Nest выдаст ошибку при попытке отправить сообщение.

Когда запускаются новые экземпляры `ClientKafkaProxy`, они присоединяются к группе потребителей и подписываются на свои топики. Этот процесс вызывает перебалансировку партиций топиков, назначенных потребителям группы.

Обычно партиции топиков распределяются круговым (round robin) партиционером, который назначает партиции набору потребителей, отсортированных по именам, задаваемым случайным образом при запуске приложения. Однако когда к группе присоединяется новый потребитель, он может оказаться в любом месте этого набора. Возникает ситуация, при которой уже существующие потребители получают другие партиции, если оказываются в списке после нового потребителя. В результате потребители, которым назначены другие партиции, теряют сообщения-ответы на запросы, отправленные до перебалансировки.

Чтобы потребители `ClientKafkaProxy` не теряли сообщения-ответы, применяется специальный встроенный партиционер Nest. Он назначает партиции набору потребителей, отсортированному по высокоточным меткам времени (`process.hrtime()`), которые задаются при запуске приложения.

#### Подписка на ответы на сообщения

> **Примечание** Этот раздел актуален, только если вы используете стиль сообщений [«запрос — ответ»](<01. Overview.md#запрос-ответ>) (с декоратором `@MessagePattern` и методом `ClientKafkaProxy.send`). Для взаимодействия [на основе событий](<01. Overview.md#на-основе-событий>) (декоратор `@EventPattern` и метод `ClientKafkaProxy.emit`) подписка на топик ответа не требуется.

Класс `ClientKafkaProxy` предоставляет метод `subscribeToResponseOf()`. Он принимает имя топика запроса в качестве аргумента и добавляет производное от него имя топика ответа в коллекцию топиков ответов. Этот метод обязателен при реализации шаблона сообщений.

**`heroes.controller.ts`**

```typescript
onModuleInit() {
  this.client.subscribeToResponseOf('hero.kill.dragon');
}
```

Если экземпляр `ClientKafkaProxy` создаётся асинхронно, метод `subscribeToResponseOf()` необходимо вызвать до вызова метода `connect()`.

**`heroes.controller.ts`**

```typescript
async onModuleInit() {
  this.client.subscribeToResponseOf('hero.kill.dragon');
  await this.client.connect();
}
```

#### Входящие сообщения

Nest получает входящие сообщения Kafka в виде объекта со свойствами `key`, `value` и `headers`, значения которых имеют тип `Buffer`. Затем Nest разбирает эти значения, преобразуя буферы в строки. Если строка «похожа на объект», Nest пытается разобрать её как `JSON`. После этого `value` передаётся соответствующему обработчику.

#### Исходящие сообщения

Исходящие сообщения Kafka Nest отправляет после процесса сериализации — при публикации событий или отправке сообщений. Это происходит с аргументами, передаваемыми в методы `emit()` и `send()` класса `ClientKafkaProxy`, либо со значениями, возвращаемыми из метода с декоратором `@MessagePattern`. При такой сериализации объекты, не являющиеся строками или буферами, приводятся к строке с помощью `JSON.stringify()` или метода прототипа `toString()`.

**`heroes.controller.ts`**

```typescript
@Controller()
export class HeroesController {
  @MessagePattern('hero.kill.dragon')
  killDragon(@Payload() message: KillDragonMessage): any {
    const dragonId = message.dragonId;
    const items = [
      { id: 1, name: 'Mythical Sword' },
      { id: 2, name: 'Key to Dungeon' },
    ];
    return items;
  }
}
```

> **Подсказка** `@Payload()` импортируется из пакета `@nestjs/microservices`.

Исходящим сообщениям можно также задавать ключ, передавая объект со свойствами `key` и `value`. Ключи важны для выполнения [требования сопартиционирования](https://docs.confluent.io/current/ksql/docs/developer-guide/partition-data.html#co-partitioning-requirements).

**`heroes.controller.ts`**

```typescript
@Controller()
export class HeroesController {
  @MessagePattern('hero.kill.dragon')
  killDragon(@Payload() message: KillDragonMessage): any {
    const realm = 'Nest';
    const heroId = message.heroId;
    const dragonId = message.dragonId;

    const items = [
      { id: 1, name: 'Mythical Sword' },
      { id: 2, name: 'Key to Dungeon' },
    ];

    return {
      headers: {
        realm
      },
      key: heroId,
      value: items
    }
  }
}
```

Кроме того, сообщения, передаваемые в таком формате, могут содержать пользовательские заголовки, задаваемые в хеш-свойстве `headers`. Значения этого хеша должны быть либо типа `string`, либо типа `Buffer`.

**`heroes.controller.ts`**

```typescript
@Controller()
export class HeroesController {
  @MessagePattern('hero.kill.dragon')
  killDragon(@Payload() message: KillDragonMessage): any {
    const realm = 'Nest';
    const heroId = message.heroId;
    const dragonId = message.dragonId;

    const items = [
      { id: 1, name: 'Mythical Sword' },
      { id: 2, name: 'Key to Dungeon' },
    ];

    return {
      headers: {
        kafka_nestRealm: realm
      },
      key: heroId,
      value: items
    }
  }
}
```

#### На основе событий

Хотя метод «запрос — ответ» идеален для обмена сообщениями между сервисами, он менее уместен, когда ваш стиль обмена основан на событиях (что, в свою очередь, идеально для Kafka), — когда вы просто хотите публиковать события, **не дожидаясь ответа**. В этом случае вам ни к чему накладные расходы на поддержание двух топиков, требуемые схемой «запрос — ответ».

Подробнее об этом — в двух разделах: [Обзор: на основе событий](<01. Overview.md#на-основе-событий>) и [Обзор: публикация событий](<01. Overview.md#публикация-событий>).

#### Контекст

В более сложных сценариях вам может понадобиться доступ к дополнительной информации о входящем запросе. При использовании транспорта Kafka можно обратиться к объекту `KafkaContext`.

```typescript
@MessagePattern('hero.kill.dragon')
killDragon(@Payload() message: KillDragonMessage, @Ctx() context: KafkaContext) {
  console.log(`Topic: ${context.getTopic()}`);
}
```

> **Подсказка** `@Payload()`, `@Ctx()` и `KafkaContext` импортируются из пакета `@nestjs/microservices`.

Чтобы получить доступ к исходному объекту `IncomingMessage` из Kafka, воспользуйтесь методом `getMessage()` объекта `KafkaContext`:

```typescript
@MessagePattern('hero.kill.dragon')
killDragon(@Payload() message: KillDragonMessage, @Ctx() context: KafkaContext) {
  const originalMessage = context.getMessage();
  const partition = context.getPartition();
  const { headers, timestamp } = originalMessage;
}
```

При этом `IncomingMessage` соответствует следующему интерфейсу:

```typescript
interface IncomingMessage {
  topic: string;
  partition: number;
  timestamp: string;
  size: number;
  attributes: number;
  offset: string;
  key: any;
  value: any;
  headers: Record<string, any>;
}
```

Если обработка каждого полученного сообщения в вашем обработчике занимает много времени, стоит воспользоваться callback-функцией `heartbeat`. Чтобы получить функцию `heartbeat`, используйте метод `getHeartbeat()` объекта `KafkaContext`:

```typescript
@MessagePattern('hero.kill.dragon')
async killDragon(@Payload() message: KillDragonMessage, @Ctx() context: KafkaContext) {
  const heartbeat = context.getHeartbeat();

  // Выполняем медленную обработку
  await doWorkPart1();

  // Отправляем heartbeat, чтобы не превысить sessionTimeout
  await heartbeat();

  // Снова выполняем медленную обработку
  await doWorkPart2();
}
```

#### Соглашения об именовании

Компоненты микросервиса Kafka добавляют описание своей роли к опциям `client.clientId` и `consumer.groupId`, чтобы избежать коллизий между клиентскими и серверными компонентами микросервисов Nest. По умолчанию компоненты `ClientKafkaProxy` добавляют к обеим этим опциям `-client`, а компоненты `ServerKafka` — `-server`. Обратите внимание, как именно преобразуются приведённые ниже значения (показано в комментариях).

**`main.ts`**

```typescript
const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  transport: Transport.KAFKA,
  options: {
    client: {
      clientId: 'hero', // hero-server
      brokers: ['localhost:9092'],
    },
    consumer: {
      groupId: 'hero-consumer' // hero-consumer-server
    },
  }
});
```

И для клиента:

**`heroes.controller.ts`**

```typescript
@Client({
  transport: Transport.KAFKA,
  options: {
    client: {
      clientId: 'hero', // hero-client
      brokers: ['localhost:9092'],
    },
    consumer: {
      groupId: 'hero-consumer' // hero-consumer-client
    }
  }
})
client: ClientKafkaProxy;
```

> **Подсказка** Соглашения об именовании клиента и потребителя Kafka можно настроить, унаследовав `ClientKafkaProxy` и `KafkaServer` в собственном пользовательском провайдере и переопределив конструктор.

Поскольку шаблон сообщений микросервиса Kafka использует два топика — для канала запроса и канала ответа, — шаблон ответа должен быть производным от топика запроса. По умолчанию имя топика ответа складывается из имени топика запроса с добавлением `.reply`.

**`heroes.controller.ts`**

```typescript
onModuleInit() {
  this.client.subscribeToResponseOf('hero.get'); // hero.get.reply
}
```

> **Подсказка** Соглашения об именовании топиков ответов Kafka можно настроить, унаследовав `ClientKafkaProxy` в собственном пользовательском провайдере и переопределив метод `getResponsePatternName`.

#### Повторяемые исключения

Как и в других транспортах, все необработанные исключения автоматически оборачиваются в `RpcException` и преобразуются в «дружелюбный к пользователю» формат. Однако бывают пограничные случаи, когда вы хотите обойти этот механизм и позволить драйверу `kafkajs` обработать исключения самостоятельно. Выброс исключения при обработке сообщения указывает `kafkajs` **повторить** его (доставить заново), а значит, даже если обработчик сообщения (или события) был вызван, смещение (offset) не будет зафиксировано в Kafka.

> **Предупреждение** Для обработчиков событий (взаимодействие на основе событий) все необработанные исключения по умолчанию считаются **повторяемыми исключениями**.

Для этого можно воспользоваться специальным классом `KafkaRetriableException`:

```typescript
throw new KafkaRetriableException('...');
```

> **Подсказка** Класс `KafkaRetriableException` экспортируется из пакета `@nestjs/microservices`.

### Пользовательская обработка исключений

Помимо стандартных механизмов обработки ошибок, для событий Kafka можно создать пользовательский фильтр исключений, управляющий логикой повторов. Например, приведённый ниже код показывает, как пропустить проблемное событие после настраиваемого числа повторных попыток:

```typescript
import { Catch, ArgumentsHost, Logger } from '@nestjs/common';
import { BaseExceptionFilter } from '@nestjs/core';
import { KafkaContext } from '@nestjs/microservices';
import { Producer } from 'kafkajs';

@Catch()
export class KafkaMaxRetryExceptionFilter extends BaseExceptionFilter {
  private readonly logger = new Logger(KafkaMaxRetryExceptionFilter.name);

  constructor(
    private readonly producer: Producer,
    private readonly maxRetries: number,
    // Необязательная пользовательская функция, выполняемая при превышении числа повторов
    private readonly skipHandler?: (message: any) => Promise<void>,
  ) {
    super();
  }

  async catch(exception: unknown, host: ArgumentsHost) {
    const kafkaContext = host.switchToRpc().getContext<KafkaContext>();
    const message = kafkaContext.getMessage();
    const currentRetryCount = this.getRetryCountFromContext(kafkaContext);

    if (currentRetryCount >= this.maxRetries) {
      this.logger.warn(
        `Max retries (${
          this.maxRetries
        }) exceeded for message: ${JSON.stringify(message)}`,
      );

      if (this.skipHandler) {
        try {
          await this.skipHandler(message);
        } catch (err) {
          this.logger.error('Error in skipHandler:', err);
        }
      }

      try {
        await this.commitOffset(kafkaContext);
      } catch (commitError) {
        this.logger.error('Failed to commit offset:', commitError);
      }
      return; // Прекращаем распространение исключения
    }

    // Повторно публикуем сообщение в тот же топик с увеличенным счётчиком повторов
    try {
      await this.republishWithRetry(kafkaContext, currentRetryCount + 1);
      await this.commitOffset(kafkaContext);
    } catch (republishError) {
      this.logger.error('Failed to republish message for retry:', republishError);
      // Откатываемся к стандартной обработке исключений
      super.catch(exception, host);
    }
  }

  private getRetryCountFromContext(context: KafkaContext): number {
    const headers = context.getMessage().headers || {};
    const retryHeader = headers['retry-count'];
    if (!retryHeader) {
      return 0;
    }
    // Значения заголовков — буферы, поэтому сначала преобразуем в строку
    const value = Buffer.isBuffer(retryHeader)
      ? retryHeader.toString()
      : String(retryHeader);
    return parseInt(value, 10) || 0;
  }

  private async republishWithRetry(
    context: KafkaContext,
    retryCount: number,
  ): Promise<void> {
    const topic = context.getTopic();
    const message = context.getMessage();

    await this.producer.send({
      topic,
      messages: [
        {
          key: message.key,
          value: message.value,
          headers: {
            ...message.headers,
            'retry-count': retryCount.toString(),
          },
        },
      ],
    });
  }

  private async commitOffset(context: KafkaContext): Promise<void> {
    const consumer = context.getConsumer();
    if (!consumer) {
      throw new Error('Consumer instance is not available from KafkaContext.');
    }

    const topic = context.getTopic();
    const partition = context.getPartition();
    const message = context.getMessage();
    const offset = message.offset;

    if (!topic || partition === undefined || offset === undefined) {
      throw new Error(
        'Incomplete Kafka message context for committing offset.',
      );
    }

    await consumer.commitOffsets([
      {
        topic,
        partition,
        // При фиксации смещения фиксируем следующий номер (то есть текущее смещение + 1)
        offset: (Number(offset) + 1).toString(),
      },
    ]);
  }
}
```

Этот фильтр даёт возможность повторять обработку события Kafka заданное число раз. При возникновении исключения он повторно публикует сообщение в тот же топик с увеличенным заголовком `retry-count`, после чего фиксирует текущее смещение. По достижении максимального числа повторов он вызывает пользовательский `skipHandler` (если тот задан) и фиксирует смещение, фактически пропуская проблемное событие. Это позволяет обрабатывать последующие события без остановок.

Подключить этот фильтр можно, зарегистрировав его глобально или на уровне контроллера. Обратите внимание, что вам потребуется предоставить экземпляр производителя Kafka:

**`kafka-retry.filter.ts`**

```typescript
import { Inject, Injectable } from '@nestjs/common';
import { Producer } from 'kafkajs';

@Injectable()
export class AppKafkaRetryFilter extends KafkaMaxRetryExceptionFilter {
  constructor(@Inject('KAFKA_PRODUCER') producer: Producer) {
    super(producer, 5); // maxRetries = 5
  }
}
```

**`my-event.handler.ts`**

```typescript
@Controller()
@UseFilters(AppKafkaRetryFilter)
export class MyEventHandler {
  @EventPattern('your-topic')
  async handleEvent(@Payload() data: any, @Ctx() context: KafkaContext) {
    // Ваша логика обработки события...
  }
}
```

Не забудьте предоставить производителя Kafka в вашем модуле:

**`app.module.ts`**

```typescript
import { Kafka } from 'kafkajs';

@Module({
  providers: [
    AppKafkaRetryFilter,
    {
      provide: 'KAFKA_PRODUCER',
      useFactory: async () => {
        const kafka = new Kafka({ brokers: ['localhost:9092'] });
        const producer = kafka.producer();
        await producer.connect();
        return producer;
      },
    },
  ],
})
export class AppModule {}
```

#### Фиксация смещений

Фиксация смещений (offsets) крайне важна при работе с Kafka. По умолчанию сообщения фиксируются автоматически через определённое время. Подробнее — в [документации KafkaJS](https://kafka.js.org/docs/consuming#autocommit). `KafkaContext` даёт способ обратиться к активному потребителю для ручной фиксации смещений. Это потребитель KafkaJS, работающий так же, как в [нативной реализации KafkaJS](https://kafka.js.org/docs/consuming#manual-committing).

```typescript
@EventPattern('user.created')
async handleUserCreated(@Payload() data: IncomingMessage, @Ctx() context: KafkaContext) {
  // бизнес-логика

  const { offset } = context.getMessage();
  const partition = context.getPartition();
  const topic = context.getTopic();
  const consumer = context.getConsumer();
  await consumer.commitOffsets([{ topic, partition, offset }])
}
```

Чтобы отключить автоматическую фиксацию сообщений, задайте `autoCommit: false` в конфигурации `run`:

**`main.ts`**

```typescript
const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  transport: Transport.KAFKA,
  options: {
    client: {
      brokers: ['localhost:9092'],
    },
    run: {
      autoCommit: false
    }
  }
});
```

#### Обновления состояния экземпляра

Чтобы получать в реальном времени сведения о соединении и состоянии нижележащего экземпляра драйвера, можно подписаться на поток `status`. Этот поток предоставляет обновления состояния, специфичные для выбранного драйвера. Для драйвера Kafka поток `status` выдаёт события `connected`, `disconnected`, `rebalancing`, `crashed` и `stopped`.

```typescript
this.client.status.subscribe((status: KafkaStatus) => {
  console.log(status);
});
```

> **Подсказка** Тип `KafkaStatus` импортируется из пакета `@nestjs/microservices`.

Аналогично можно подписаться на поток `status` сервера, чтобы получать уведомления о его состоянии.

```typescript
const server = app.connectMicroservice<MicroserviceOptions>(...);
server.status.subscribe((status: KafkaStatus) => {
  console.log(status);
});
```

#### Нижележащие производитель и потребитель

В более сложных сценариях вам может понадобиться доступ к нижележащим экземплярам производителя и потребителя. Это бывает полезно, например, для ручного закрытия соединения или использования специфичных для драйвера методов. Однако имейте в виду, что в большинстве случаев обращаться к драйверу напрямую **не требуется**.

Для этого можно воспользоваться геттерами `producer` и `consumer`, которые предоставляет экземпляр `ClientKafkaProxy`.

```typescript
const producer = this.client.producer;
const consumer = this.client.consumer;
```
