### Обзор

Помимо традиционной (иногда её называют монолитной) архитектуры приложений, Nest изначально поддерживает микросервисный стиль разработки. Большинство концепций, описанных в других разделах этой документации, — внедрение зависимостей, декораторы, фильтры исключений, pipes, guards и interceptor'ы — в равной степени применимы и к микросервисам. Везде, где это возможно, Nest скрывает детали реализации, чтобы одни и те же компоненты работали и на платформах на основе HTTP, и в WebSockets, и в микросервисах. Этот раздел посвящён тем аспектам Nest, которые специфичны для микросервисов.

В Nest микросервис — это, по сути, приложение, которое использует иной **транспортный** слой, нежели HTTP.

<figure><img class="illustrative-image" src="https://docs.nestjs.com/assets/Microservices_1.png" /></figure>

Nest поддерживает несколько встроенных реализаций транспортного слоя, называемых **транспортами**; они отвечают за передачу сообщений между разными экземплярами микросервисов. Большинство транспортов изначально поддерживают оба стиля обмена сообщениями: **запрос — ответ** и **на основе событий**. Nest скрывает детали реализации каждого транспорта за единым интерфейсом для обоих стилей. Благодаря этому легко переключиться с одного транспортного слоя на другой — например, чтобы задействовать особенности надёжности или производительности конкретного транспорта — без изменений в коде приложения.

#### Установка

Чтобы приступить к разработке микросервисов, сначала установите нужный пакет:

```bash
$ npm i --save @nestjs/microservices
```

#### Начало работы

Чтобы создать экземпляр микросервиса, используйте метод `createMicroservice()` класса `NestFactory`:

**`main.ts`**

```typescript
import { NestFactory } from '@nestjs/core';
import { Transport, MicroserviceOptions } from '@nestjs/microservices';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.createMicroservice<MicroserviceOptions>(
    AppModule,
    {
      transport: Transport.TCP,
    },
  );
  await app.listen();
}
bootstrap();
```

> **Подсказка** По умолчанию микросервисы используют транспортный слой **TCP**.

#### Шаблоны сообщений и событий

Микросервисы распознают и сообщения, и события по **шаблонам**. Шаблон — это обычное значение, например литеральный объект или строка. Шаблоны автоматически сериализуются и отправляются по сети вместе с данными сообщения. Таким образом отправители и потребители сообщений согласуют, какие запросы обрабатываются какими обработчиками.

#### Запрос — ответ

Стиль «запрос — ответ» полезен, когда нужно **обмениваться** сообщениями между различными внешними сервисами. Эта парадигма гарантирует, что сервис действительно получил сообщение (и вам не придётся вручную реализовывать протокол подтверждений). Однако подход «запрос — ответ» подходит не всегда. Например, потоковые транспорты вроде [Kafka](https://docs.confluent.io/3.0.0/streams/) или [NATS streaming](https://github.com/nats-io/node-nats-streaming), использующие журнальное хранение, оптимизированы под другой круг задач, ближе к парадигме обмена событиями (подробнее см. [обмен сообщениями на основе событий](<01. Overview.md#на-основе-событий>)).

Чтобы включить тип сообщений «запрос — ответ», Nest создаёт два логических канала: один для передачи данных, другой для ожидания входящих ответов. Для некоторых нижележащих транспортов, например [NATS](https://nats.io/), поддержка двух каналов доступна «из коробки». Для остальных Nest компенсирует это, создавая отдельные каналы вручную. Работает это хорошо, но может добавить накладных расходов. Поэтому, если стиль «запрос — ответ» вам не нужен, стоит рассмотреть подход на основе событий.

Чтобы создать обработчик сообщений в парадигме «запрос — ответ», используйте декоратор `@MessagePattern()`, импортируемый из пакета `@nestjs/microservices`. Этот декоратор следует применять только внутри классов [контроллеров](<../02. Overview/02. Controllers.md>), поскольку именно они служат точками входа в приложение. В провайдерах он не даст эффекта — среда выполнения Nest его проигнорирует.

**`math.controller.ts`**

```typescript
import { Controller } from '@nestjs/common';
import { MessagePattern } from '@nestjs/microservices';

@Controller()
export class MathController {
  @MessagePattern({ cmd: 'sum' })
  accumulate(data: number[]): number {
    return (data || []).reduce((a, b) => a + b);
  }
}
```

В коде выше **обработчик сообщений** `accumulate()` слушает сообщения, соответствующие шаблону `{ cmd: 'sum' }`. Обработчик принимает единственный аргумент — `data`, переданные клиентом. В данном случае это массив чисел, которые нужно просуммировать.

#### Асинхронные ответы

Обработчики сообщений могут отвечать как синхронно, так и **асинхронно**, то есть методы `async` поддерживаются.

```typescript
@MessagePattern({ cmd: 'sum' })
async accumulate(data: number[]): Promise<number> {
  return (data || []).reduce((a, b) => a + b);
}
```

Обработчик сообщений также может вернуть `Observable` — в этом случае значения результата будут отправляться до завершения потока.

```typescript
@MessagePattern({ cmd: 'sum' })
accumulate(data: number[]): Observable<number> {
  return from([1, 2, 3]);
}
```

В примере выше обработчик сообщений ответит **трижды** — по одному разу на каждый элемент массива.

#### На основе событий

Метод «запрос — ответ» прекрасно подходит для обмена сообщениями между сервисами, но хуже годится для обмена на основе событий — когда вы просто хотите публиковать **события**, не дожидаясь ответа. В таких случаях накладные расходы на поддержание двух каналов для «запроса — ответа» излишни.

Например, если вам нужно уведомить другой сервис о том, что в этой части системы наступило определённое условие, стиль сообщений на основе событий подходит идеально.

Чтобы создать обработчик событий, используйте декоратор `@EventPattern()`, импортируемый из пакета `@nestjs/microservices`.

```typescript
@EventPattern('user_created')
async handleUserCreated(data: Record<string, unknown>) {
  // бизнес-логика
}
```

> **Подсказка** Для **одного** шаблона события можно зарегистрировать несколько обработчиков — все они будут автоматически запущены параллельно.

**Обработчик событий** `handleUserCreated()` слушает событие `'user_created'`. Обработчик принимает единственный аргумент — `data`, переданные клиентом (в данном случае это полезная нагрузка события, отправленная по сети).

#### Дополнительные сведения о запросе

В более сложных сценариях вам может понадобиться доступ к дополнительным сведениям о входящем запросе. Например, при использовании NATS с подписками по шаблону может потребоваться получить исходный subject, в который отправитель послал сообщение. Аналогично в Kafka может понадобиться доступ к заголовкам сообщения. Для этого можно воспользоваться встроенными декораторами, как показано ниже:

```typescript
@MessagePattern('time.us.*')
getDate(@Payload() data: number[], @Ctx() context: NatsContext) {
  console.log(`Subject: ${context.getSubject()}`); // например, "time.us.east"
  return new Date().toLocaleTimeString(...);
}
```

> **Подсказка** `@Payload()`, `@Ctx()` и `NatsContext` импортируются из `@nestjs/microservices`.

> **Подсказка** В декоратор `@Payload()` можно передать имя свойства, чтобы извлечь конкретное свойство из объекта входящей полезной нагрузки, например `@Payload('id')`.

#### Клиент (класс-производитель)

Клиентское приложение Nest может обмениваться сообщениями с микросервисом Nest или публиковать в него события с помощью класса `ClientProxy`. Этот класс предоставляет несколько методов, в частности `send()` (для обмена в стиле «запрос — ответ») и `emit()` (для обмена на основе событий), которые позволяют взаимодействовать с удалённым микросервисом. Получить экземпляр этого класса можно следующими способами.

Один из подходов — импортировать `ClientsModule`, который предоставляет статический метод `register()`. Этот метод принимает массив объектов, описывающих транспорты микросервисов. Каждый объект обязан содержать свойство `name`, а также может содержать необязательные свойства `transport` (по умолчанию `Transport.TCP`) и `options`.

Свойство `name` выступает **токеном внедрения**, по которому вы можете внедрить экземпляр `ClientProxy` там, где он нужен. Значением этого свойства может быть произвольная строка или символ JavaScript, как описано [здесь](<../03. Fundamentals/01. Custom providers.md#токены-провайдеров-не-основанные-на-классах>).

Свойство `options` — это объект с теми же свойствами, что мы видели ранее в методе `createMicroservice()`.

```typescript
@Module({
  imports: [
    ClientsModule.register([
      { name: 'MATH_SERVICE', transport: Transport.TCP },
    ]),
  ],
})
```

Как вариант, вы можете использовать метод `registerAsync()`, если во время настройки нужно получить конфигурацию или выполнить другие асинхронные операции.

```typescript
@Module({
  imports: [
    ClientsModule.registerAsync([
      {
        imports: [ConfigModule],
        name: 'MATH_SERVICE',
        useFactory: async (configService: ConfigService) => ({
          transport: Transport.TCP,
          options: {
            url: configService.get('URL'),
          },
        }),
        inject: [ConfigService],
      },
    ]),
  ],
})
```

После импорта модуля вы можете внедрить экземпляр `ClientProxy`, настроенный с указанными параметрами для транспорта `'MATH_SERVICE'`, с помощью декоратора `@Inject()`.

```typescript
constructor(
  @Inject('MATH_SERVICE') private client: ClientProxy,
) {}
```

> **Подсказка** Классы `ClientsModule` и `ClientProxy` импортируются из пакета `@nestjs/microservices`.

Иногда конфигурацию транспорта нужно получать из другого сервиса (например, из `ConfigService`), а не жёстко прописывать в клиентском приложении. Для этого можно зарегистрировать [пользовательский провайдер](<../03. Fundamentals/01. Custom providers.md>) с помощью класса `ClientProxyFactory`. У этого класса есть статический метод `create()`, который принимает объект параметров транспорта и возвращает настроенный экземпляр `ClientProxy`.

```typescript
@Module({
  providers: [
    {
      provide: 'MATH_SERVICE',
      useFactory: (configService: ConfigService) => {
        const mathSvcOptions = configService.getMathSvcOptions();
        return ClientProxyFactory.create(mathSvcOptions);
      },
      inject: [ConfigService],
    }
  ]
  ...
})
```

> **Подсказка** `ClientProxyFactory` импортируется из пакета `@nestjs/microservices`.

Ещё один вариант — декоратор свойства `@Client()`.

```typescript
@Client({ transport: Transport.TCP })
client: ClientProxy;
```

> **Подсказка** Декоратор `@Client()` импортируется из пакета `@nestjs/microservices`.

Использовать декоратор `@Client()` предпочтительным способом не назовёшь: такой код труднее тестировать и сложнее переиспользовать экземпляр клиента.

`ClientProxy` работает **лениво**. Он не устанавливает соединение сразу — соединение создаётся перед первым вызовом микросервиса и затем переиспользуется при всех последующих вызовах. Однако если вы хотите отложить начальную загрузку приложения до установления соединения, вы можете инициировать его вручную методом `connect()` объекта `ClientProxy` внутри хука жизненного цикла `OnApplicationBootstrap`.

```typescript
async onApplicationBootstrap() {
  await this.client.connect();
}
```

Если соединение установить не удастся, метод `connect()` завершится отказом с соответствующим объектом ошибки.

#### Отправка сообщений

`ClientProxy` предоставляет метод `send()`. Этот метод предназначен для вызова микросервиса и возвращает `Observable` с ответом. Таким образом, мы легко можем подписаться на отправляемые значения.

```typescript
accumulate(): Observable<number> {
  const pattern = { cmd: 'sum' };
  const payload = [1, 2, 3];
  return this.client.send<number>(pattern, payload);
}
```

Метод `send()` принимает два аргумента: `pattern` и `payload`. `pattern` должен совпадать с шаблоном, заданным в декораторе `@MessagePattern()`. `payload` — это сообщение, которое мы хотим передать удалённому микросервису. Метод возвращает **холодный `Observable`**, а значит, вам нужно явно на него подписаться, прежде чем сообщение будет отправлено.

#### Публикация событий

Чтобы отправить событие, используйте метод `emit()` объекта `ClientProxy`. Этот метод публикует событие в брокере сообщений.

```typescript
async publish() {
  this.client.emit<number>('user_created', new UserCreatedEvent());
}
```

Метод `emit()` принимает два аргумента: `pattern` и `payload`. `pattern` должен совпадать с шаблоном, заданным в декораторе `@EventPattern()`, а `payload` — это данные события, которые вы хотите передать удалённому микросервису. Метод возвращает **горячий `Observable`** (в отличие от холодного `Observable`, который возвращает `send()`); это значит, что прокси немедленно попытается доставить событие независимо от того, подписались вы на observable явно или нет.

#### Область запроса

Тем, кто пришёл из других языков программирования, может показаться неожиданным, что в Nest большинство сущностей разделяются между входящими запросами. Это касается и пула соединений с базой данных, и сервисов-синглтонов с глобальным состоянием, и многого другого. Помните, что Node.js не следует многопоточной модели «запрос/ответ» без состояния, где каждый запрос обрабатывается отдельным потоком. Поэтому использование экземпляров-синглтонов в наших приложениях **безопасно**.

Тем не менее бывают частные случаи, когда время жизни обработчика на уровне запроса всё же желательно. Например, кэширование для каждого запроса в приложениях GraphQL, отслеживание запросов или мультиарендность. Подробнее об управлении областями видимости читайте [здесь](<../03. Fundamentals/04. Injection scopes.md>).

Обработчики и провайдеры с областью запроса могут внедрять `RequestContext` с помощью декоратора `@Inject()` в сочетании с токеном `CONTEXT`:

```typescript
import { Injectable, Scope, Inject } from '@nestjs/common';
import { CONTEXT, RequestContext } from '@nestjs/microservices';

@Injectable({ scope: Scope.REQUEST })
export class CatsService {
  constructor(@Inject(CONTEXT) private ctx: RequestContext) {}
}
```

Это даёт доступ к объекту `RequestContext`, у которого есть два свойства:

```typescript
export interface RequestContext<T = any> {
  pattern: string | Record<string, any>;
  data: T;
}
```

Свойство `data` — это полезная нагрузка сообщения, отправленная его производителем. Свойство `pattern` — шаблон, по которому определяется подходящий обработчик для входящего сообщения.

#### Обновления состояния экземпляра

Чтобы получать обновления о соединении и состоянии нижележащего экземпляра драйвера в реальном времени, вы можете подписаться на поток `status`. Этот поток передаёт обновления состояния, специфичные для выбранного драйвера. Например, если вы используете транспорт TCP (используемый по умолчанию), поток `status` отправляет события `connected` и `disconnected`.

```typescript
this.client.status.subscribe((status: TcpStatus) => {
  console.log(status);
});
```

> **Подсказка** Тип `TcpStatus` импортируется из пакета `@nestjs/microservices`.

Аналогично вы можете подписаться на поток `status` сервера, чтобы получать уведомления о его состоянии.

```typescript
const server = app.connectMicroservice<MicroserviceOptions>(...);
server.status.subscribe((status: TcpStatus) => {
  console.log(status);
});
```

#### Прослушивание внутренних событий

В некоторых случаях бывает нужно слушать внутренние события, отправляемые микросервисом. Например, вы можете слушать событие `error`, чтобы выполнять дополнительные действия при возникновении ошибки. Для этого используйте метод `on()`, как показано ниже:

```typescript
this.client.on('error', (err) => {
  console.error(err);
});
```

Точно так же можно слушать внутренние события сервера:

```typescript
server.on<TcpEvents>('error', (err) => {
  console.error(err);
});
```

> **Подсказка** Тип `TcpEvents` импортируется из пакета `@nestjs/microservices`.

#### Доступ к нижележащему драйверу

В более сложных сценариях вам может понадобиться доступ к экземпляру нижележащего драйвера. Это бывает полезно, например, чтобы вручную закрыть соединение или воспользоваться методами, специфичными для драйвера. Однако имейте в виду, что в большинстве случаев обращаться к драйверу напрямую **не требуется**.

Для этого используйте метод `unwrap()`, который возвращает экземпляр нижележащего драйвера. В параметре обобщённого типа следует указать ожидаемый тип экземпляра драйвера.

```typescript
const netServer = this.client.unwrap<Server>();
```

Здесь `Server` — тип, импортированный из модуля `net`.

Аналогично можно получить доступ к нижележащему экземпляру драйвера сервера:

```typescript
const netServer = server.unwrap<Server>();
```

#### Обработка таймаутов

В распределённых системах микросервисы иногда бывают недоступны или не работают. Чтобы не ждать бесконечно долго, используйте таймауты. Таймаут — крайне полезный шаблон при взаимодействии с другими сервисами. Чтобы применить таймауты к вызовам микросервисов, воспользуйтесь оператором `timeout` из [RxJS](https://rxjs.dev). Если микросервис не ответит за указанное время, будет выброшено исключение, которое вы сможете перехватить и обработать нужным образом.

Для этого вам понадобится пакет [`rxjs`](https://github.com/ReactiveX/rxjs). Просто используйте оператор `timeout` внутри pipe:

```typescript
this.client
  .send<TResult, TInput>(pattern, data)
  .pipe(timeout(5000));
```

> **Подсказка** Оператор `timeout` импортируется из пакета `rxjs/operators`.

Если через 5 секунд микросервис не ответит, будет выброшена ошибка.

#### Поддержка TLS

При взаимодействии за пределами частной сети важно шифровать трафик для обеспечения безопасности. В NestJS этого можно добиться с помощью TLS поверх TCP, используя встроенный модуль Node [TLS](https://nodejs.org/api/tls.html). Nest предоставляет встроенную поддержку TLS в своём транспорте TCP, что позволяет шифровать обмен данными между микросервисами или клиентами.

Чтобы включить TLS для TCP-сервера, вам понадобятся закрытый ключ и сертификат в формате PEM. Они добавляются в параметры сервера через `tlsOptions` с указанием файлов ключа и сертификата, как показано ниже:

```typescript
import * as fs from 'fs';
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';
import { MicroserviceOptions, Transport } from '@nestjs/microservices';

async function bootstrap() {
  const key = fs.readFileSync('<pathToKeyFile>', 'utf8').toString();
  const cert = fs.readFileSync('<pathToCertFile>', 'utf8').toString();

  const app = await NestFactory.createMicroservice<MicroserviceOptions>(
    AppModule,
    {
      transport: Transport.TCP,
      options: {
        tlsOptions: {
          key,
          cert,
        },
      },
    },
  );

  await app.listen();
}
bootstrap();
```

Чтобы клиент мог безопасно взаимодействовать по TLS, мы также определяем объект `tlsOptions`, но на этот раз с сертификатом CA. Это сертификат удостоверяющего центра, подписавшего сертификат сервера. Он гарантирует, что клиент доверяет сертификату сервера и может установить защищённое соединение.

```typescript
import { Module } from '@nestjs/common';
import { ClientsModule, Transport } from '@nestjs/microservices';

@Module({
  imports: [
    ClientsModule.register([
      {
        name: 'MATH_SERVICE',
        transport: Transport.TCP,
        options: {
          tlsOptions: {
            ca: [fs.readFileSync('<pathToCaFile>', 'utf-8').toString()],
          },
        },
      },
    ]),
  ],
})
export class AppModule {}
```

Если в вашей конфигурации несколько доверенных удостоверяющих центров, можно передать массив CA.

Когда всё настроено, вы можете внедрять `ClientProxy` как обычно, с помощью декоратора `@Inject()`, и использовать клиент в своих сервисах. Это обеспечит шифрование обмена данными между вашими микросервисами NestJS, а детали шифрования возьмёт на себя модуль `TLS` из Node.

Подробнее см. [документацию по TLS](https://nodejs.org/api/tls.html) в Node.

#### Динамическая конфигурация

Когда микросервис нужно настроить с помощью `ConfigService` (из пакета `@nestjs/config`), но контекст внедрения доступен только после создания экземпляра микросервиса, решением служит `AsyncMicroserviceOptions`. Такой подход позволяет настраивать микросервис динамически и обеспечивает гладкую интеграцию с `ConfigService`.

```typescript
import { ConfigService } from '@nestjs/config';
import { AsyncMicroserviceOptions, Transport } from '@nestjs/microservices';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.createMicroservice<AsyncMicroserviceOptions>(
    AppModule,
    {
      useFactory: (configService: ConfigService) => ({
        transport: Transport.TCP,
        options: {
          host: configService.get<string>('HOST'),
          port: configService.get<number>('PORT'),
        },
      }),
      inject: [ConfigService],
    },
  );

  await app.listen();
}
bootstrap();
```
