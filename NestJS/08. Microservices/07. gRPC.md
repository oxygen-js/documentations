### gRPC

[gRPC](https://github.com/grpc/grpc-node) — современный высокопроизводительный RPC-фреймворк с открытым исходным кодом, способный работать в любом окружении. Он эффективно связывает сервисы внутри дата-центров и между ними, поддерживая подключаемые механизмы балансировки нагрузки, трассировки, проверки работоспособности и аутентификации.

Как и многие RPC-системы, gRPC основан на идее описания сервиса через функции (методы), которые можно вызывать удалённо. Для каждого метода вы задаёте типы параметров и возвращаемого значения. Сервисы, параметры и возвращаемые типы описываются в файлах `.proto` с помощью открытого языконезависимого механизма Google <a href="https://protobuf.dev">protocol buffers</a>.

С транспортом gRPC Nest использует файлы `.proto` для динамической привязки клиентов и серверов, что упрощает реализацию удалённых вызовов процедур с автоматической сериализацией и десериализацией структурированных данных.

#### Установка

Чтобы начать создавать микросервисы на базе gRPC, сначала установите необходимые пакеты:

```bash
$ npm i --save @grpc/grpc-js @grpc/proto-loader
```

#### Обзор

Как и в других реализациях транспортного слоя микросервисов Nest, механизм транспорта gRPC выбирается через свойство `transport` объекта опций, передаваемого в метод `createMicroservice()`. В примере ниже мы настроим сервис героев. Свойство `options` содержит метаданные этого сервиса; его свойства описаны <a href="07.%20gRPC.md#опции">ниже</a>.

**`main.ts`**

```typescript
const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  transport: Transport.GRPC,
  options: {
    package: 'hero',
    protoPath: join(__dirname, 'hero/hero.proto'),
  },
});
```

> **Подсказка** Функция `join()` импортируется из пакета `path`; enum `Transport` — из пакета `@nestjs/microservices`.

В файл `nest-cli.json` мы добавляем свойство `assets`, позволяющее распространять файлы, не относящиеся к TypeScript, и `watchAssets` — чтобы включить отслеживание всех таких ресурсов. В нашем случае мы хотим, чтобы файлы `.proto` автоматически копировались в папку `dist`.

```json
{
  "compilerOptions": {
    "assets": ["**/*.proto"],
    "watchAssets": true
  }
}
```

#### Пример gRPC-сервиса

Определим наш пример gRPC-сервиса под названием `HeroesService`. В приведённом выше объекте `options` свойство `protoPath` задаёт путь к файлу определений `.proto` — `hero.proto`. Файл `hero.proto` описан с помощью <a href="https://developers.google.com/protocol-buffers">protocol buffers</a>. Вот как он выглядит:

```typescript
// hero/hero.proto
syntax = "proto3";

package hero;

service HeroesService {
  rpc FindOne (HeroById) returns (Hero) {}
}

message HeroById {
  int32 id = 1;
}

message Hero {
  int32 id = 1;
  string name = 2;
}
```

Наш `HeroesService` предоставляет метод `FindOne()`. Этот метод ожидает входной аргумент типа `HeroById` и возвращает сообщение `Hero` (в protocol buffers элементы `message` описывают и типы параметров, и типы возвращаемых значений).

Далее нужно реализовать сервис. Чтобы определить обработчик, соответствующий этому описанию, мы используем в контроллере декоратор `@GrpcMethod()`, как показано ниже. Этот декоратор предоставляет метаданные, необходимые для объявления метода как метода gRPC-сервиса.

> **Подсказка** Декоратор `@MessagePattern()` (<a href="01.%20Overview.md#запрос-ответ">подробнее</a>), представленный в предыдущих главах о микросервисах, с микросервисами на базе gRPC не используется. Для них его роль фактически выполняет декоратор `@GrpcMethod()`.

**`heroes.controller.ts`**

```typescript
@Controller()
export class HeroesController {
  @GrpcMethod('HeroesService', 'FindOne')
  findOne(data: HeroById, metadata: Metadata, call: ServerUnaryCall<any, any>): Hero {
    const items = [
      { id: 1, name: 'John' },
      { id: 2, name: 'Doe' },
    ];
    return items.find(({ id }) => id === data.id);
  }
}
```

> **Подсказка** Декоратор `@GrpcMethod()` импортируется из пакета `@nestjs/microservices`, а `Metadata` и `ServerUnaryCall` — из пакета `grpc`.

Показанный выше декоратор принимает два аргумента. Первый — имя сервиса (например, `'HeroesService'`), соответствующее определению сервиса `HeroesService` в `hero.proto`. Второй (строка `'FindOne'`) соответствует rpc-методу `FindOne()`, определённому внутри `HeroesService` в файле `hero.proto`.

Метод-обработчик `findOne()` принимает три аргумента: `data`, переданные вызывающей стороной, `metadata`, где хранятся метаданные gRPC-запроса, и `call` — для доступа к свойствам объекта `GrpcCall`, таким как `sendMetadata`, для отправки метаданных клиенту.

Оба аргумента декоратора `@GrpcMethod()` необязательны. Если вызвать его без второго аргумента (например, `'FindOne'`), Nest автоматически сопоставит rpc-метод из файла `.proto` с обработчиком, преобразовав имя обработчика в upper camel case (например, обработчик `findOne` будет сопоставлен с определением rpc-вызова `FindOne`). Это показано ниже.

**`heroes.controller.ts`**

```typescript
@Controller()
export class HeroesController {
  @GrpcMethod('HeroesService')
  findOne(data: HeroById, metadata: Metadata, call: ServerUnaryCall<any, any>): Hero {
    const items = [
      { id: 1, name: 'John' },
      { id: 2, name: 'Doe' },
    ];
    return items.find(({ id }) => id === data.id);
  }
}
```

Вы также можете опустить первый аргумент `@GrpcMethod()`. В этом случае Nest автоматически сопоставит обработчик с определением сервиса из файла proto-определений на основе имени **класса**, в котором объявлен обработчик. Например, в следующем коде класс `HeroesService` связывает свои методы-обработчики с определением сервиса `HeroesService` в файле `hero.proto` по совпадению имени `'HeroesService'`.

**`heroes.controller.ts`**

```typescript
@Controller()
export class HeroesService {
  @GrpcMethod()
  findOne(data: HeroById, metadata: Metadata, call: ServerUnaryCall<any, any>): Hero {
    const items = [
      { id: 1, name: 'John' },
      { id: 2, name: 'Doe' },
    ];
    return items.find(({ id }) => id === data.id);
  }
}
```

#### Клиент

Приложения Nest могут выступать в роли gRPC-клиентов, обращаясь к сервисам, описанным в файлах `.proto`. Доступ к удалённым сервисам осуществляется через объект `ClientGrpc`. Получить объект `ClientGrpc` можно несколькими способами.

Предпочтительный подход — импортировать `ClientsModule`. Метод `register()` привязывает пакет сервисов, описанных в файле `.proto`, к токену внедрения и настраивает сервис. Свойство `name` — это токен внедрения. Для gRPC-сервисов используйте `transport: Transport.GRPC`. Свойство `options` — объект с теми же свойствами, которые описаны <a href="07.%20gRPC.md#опции">выше</a>.

```typescript
imports: [
  ClientsModule.register([
    {
      name: 'HERO_PACKAGE',
      transport: Transport.GRPC,
      options: {
        package: 'hero',
        protoPath: join(__dirname, 'hero/hero.proto'),
      },
    },
  ]),
];
```

> **Подсказка** Метод `register()` принимает массив объектов. Чтобы зарегистрировать несколько пакетов, передайте список объектов регистрации через запятую.

После регистрации мы можем внедрить настроенный объект `ClientGrpc` через `@Inject()`. Затем с помощью метода `getService()` объекта `ClientGrpc` получаем экземпляр сервиса, как показано ниже.

```typescript
@Injectable()
export class AppService implements OnModuleInit {
  private heroesService: HeroesService;

  constructor(@Inject('HERO_PACKAGE') private client: ClientGrpc) {}

  onModuleInit() {
    this.heroesService = this.client.getService<HeroesService>('HeroesService');
  }

  getHero(): Observable<string> {
    return this.heroesService.findOne({ id: 1 });
  }
}
```

> **Предупреждение** gRPC-клиент не будет отправлять поля, в именах которых есть подчёркивание `_`, если в конфигурации proto-загрузчика не задана опция `keepCase` со значением `true` (`options.loader.keepcase` в конфигурации транспорта микросервиса).

Обратите внимание на небольшое отличие от подхода, применяемого в других транспортах микросервисов. Вместо класса `ClientProxy` мы используем класс `ClientGrpc`, предоставляющий метод `getService()`. Обобщённый метод `getService()` принимает имя сервиса в качестве аргумента и возвращает его экземпляр (если он доступен).

Кроме того, для создания объекта `ClientGrpc` можно использовать декоратор `@Client()`:

```typescript
@Injectable()
export class AppService implements OnModuleInit {
  @Client({
    transport: Transport.GRPC,
    options: {
      package: 'hero',
      protoPath: join(__dirname, 'hero/hero.proto'),
    },
  })
  client: ClientGrpc;

  private heroesService: HeroesService;

  onModuleInit() {
    this.heroesService = this.client.getService<HeroesService>('HeroesService');
  }

  getHero(): Observable<string> {
    return this.heroesService.findOne({ id: 1 });
  }
}
```

Наконец, для более сложных сценариев мы можем внедрить динамически настроенный клиент с помощью класса `ClientProxyFactory`, как описано <a href="01.%20Overview.md#клиент-класс-производитель">здесь</a>.

В любом случае мы получаем ссылку на объект-прокси `HeroesService`, предоставляющий тот же набор методов, что определён в файле `.proto`. Теперь, когда мы обращаемся к этому объекту-прокси (то есть к `heroesService`), система gRPC автоматически сериализует запросы, пересылает их в удалённую систему, возвращает ответ и десериализует его. Поскольку gRPC скрывает от нас эти детали сетевого взаимодействия, `heroesService` выглядит и ведёт себя как локальный провайдер.

Учтите, что все методы сервиса именуются в **lower camel case** (чтобы следовать естественному соглашению языка). Так, например, хотя определение `HeroesService` в нашем файле `.proto` содержит функцию `FindOne()`, экземпляр `heroesService` будет предоставлять метод `findOne()`.

```typescript
interface HeroesService {
  findOne(data: { id: number }): Observable<any>;
}
```

Обработчик сообщений также может возвращать `Observable` — в этом случае значения результата будут отдаваться до тех пор, пока поток не завершится.

**`heroes.controller.ts`**

```typescript
@Get()
call(): Observable<any> {
  return this.heroesService.findOne({ id: 1 });
}
```

Чтобы отправить метаданные gRPC (вместе с запросом), вы можете передать второй аргумент:

```typescript
call(): Observable<any> {
  const metadata = new Metadata();
  metadata.add('Set-Cookie', 'yummy_cookie=choco');

  return this.heroesService.findOne({ id: 1 }, metadata);
}
```

> **Подсказка** Класс `Metadata` импортируется из пакета `grpc`.

Обратите внимание, что для этого потребуется обновить интерфейс `HeroesService`, который мы определили несколькими шагами ранее.

#### Рефлексия gRPC

[Спецификация gRPC Server Reflection](https://grpc.io/docs/guides/reflection/#overview) — это стандарт, позволяющий gRPC-клиентам запрашивать сведения об API, который предоставляет сервер, — примерно так же, как OpenAPI-документ описывает REST API. Это заметно упрощает работу с инструментами отладки вроде grpc-ui или postman.

Чтобы добавить на сервер поддержку рефлексии gRPC, сначала установите пакет с необходимой реализацией:

```bash
$ npm i --save @grpc/reflection
```

Затем её можно подключить к gRPC-серверу через хук `onLoadPackageDefinition` в опциях сервера:

**`main.ts`**

```typescript
import { ReflectionService } from '@grpc/reflection';

const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  options: {
    onLoadPackageDefinition: (pkg, server) => {
      new ReflectionService(pkg).addToServer(server);
    },
  },
});
```

Теперь ваш сервер будет отвечать на сообщения с запросом сведений об API согласно спецификации рефлексии.

#### Потоковая передача в gRPC

Сам по себе gRPC поддерживает долгоживущие соединения, традиционно называемые `streams` (потоками). Потоки полезны в таких сценариях, как чаты, наблюдение за событиями или передача данных по частям. Подробности смотрите в официальной документации [здесь](https://grpc.io/docs/guides/concepts/).

Nest поддерживает обработчики gRPC-потоков двумя способами:

- Обработчик на основе RxJS `Subject` + `Observable`: удобен, когда ответы формируются прямо внутри метода контроллера или передаются потребителю `Subject`/`Observable`.
- Обработчик чистого потока gRPC-вызова: удобен, когда поток передаётся некоторому исполнителю, который берёт на себя дальнейшую диспетчеризацию для стандартного обработчика потока Node `Duplex`.

#### Пример потоковой передачи

Определим новый пример gRPC-сервиса под названием `HelloService`. Файл `hello.proto` описан с помощью <a href="https://developers.google.com/protocol-buffers">protocol buffers</a>. Вот как он выглядит:

```typescript
// hello/hello.proto
syntax = "proto3";

package hello;

service HelloService {
  rpc BidiHello(stream HelloRequest) returns (stream HelloResponse);
  rpc LotsOfGreetings(stream HelloRequest) returns (HelloResponse);
}

message HelloRequest {
  string greeting = 1;
}

message HelloResponse {
  string reply = 1;
}
```

> **Подсказка** Метод `LotsOfGreetings` можно просто реализовать с помощью декоратора `@GrpcMethod` (как в примерах выше), поскольку возвращаемый поток может отдавать несколько значений.

На основе этого файла `.proto` определим интерфейс `HelloService`:

```typescript
interface HelloService {
  bidiHello(upstream: Observable<HelloRequest>): Observable<HelloResponse>;
  lotsOfGreetings(
    upstream: Observable<HelloRequest>,
  ): Observable<HelloResponse>;
}

interface HelloRequest {
  greeting: string;
}

interface HelloResponse {
  reply: string;
}
```

> **Подсказка** Интерфейс proto может быть сгенерирован автоматически пакетом [ts-proto](https://github.com/stephenh/ts-proto), подробнее [здесь](https://github.com/stephenh/ts-proto/blob/main/NESTJS.markdown).

#### Стратегия Subject

Декоратор `@GrpcStreamMethod()` передаёт параметр функции в виде RxJS `Observable`. Благодаря этому мы можем принимать и обрабатывать несколько сообщений.

```typescript
@GrpcStreamMethod()
bidiHello(messages: Observable<any>, metadata: Metadata, call: ServerDuplexStream<any, any>): Observable<any> {
  const subject = new Subject();

  const onNext = message => {
    console.log(message);
    subject.next({
      reply: 'Hello, world!'
    });
  };
  const onComplete = () => subject.complete();
  messages.subscribe({
    next: onNext,
    complete: onComplete,
  });

  return subject.asObservable();
}
```

> **Предупреждение** Чтобы поддерживать полнодуплексное взаимодействие с декоратором `@GrpcStreamMethod()`, метод контроллера должен возвращать RxJS `Observable`.

> **Подсказка** Классы и интерфейсы `Metadata` и `ServerUnaryCall` импортируются из пакета `grpc`.

Согласно определению сервиса (в файле `.proto`), метод `BidiHello` должен передавать запросы сервису потоком. Чтобы отправлять в поток несколько асинхронных сообщений со стороны клиента, мы задействуем класс `ReplaySubject` из RxJS.

```typescript
const helloService = this.client.getService<HelloService>('HelloService');
const helloRequest$ = new ReplaySubject<HelloRequest>();

helloRequest$.next({ greeting: 'Hello (1)!' });
helloRequest$.next({ greeting: 'Hello (2)!' });
helloRequest$.complete();

return helloService.bidiHello(helloRequest$);
```

В примере выше мы записали в поток два сообщения (вызовы `next()`) и уведомили сервис о том, что отправка данных завершена (вызов `complete()`).

#### Обработчик потока вызова

Когда возвращаемое значение метода определено как `stream`, декоратор `@GrpcStreamCall()` передаёт параметр функции в виде `grpc.ServerDuplexStream`, который поддерживает стандартные методы вроде `.on('data', callback)`, `.write(message)` или `.cancel()`. Полная документация по доступным методам находится [здесь](https://grpc.github.io/grpc/node/grpc-ClientDuplexStream.html).

Если же возвращаемое значение метода не является `stream`, декоратор `@GrpcStreamCall()` передаёт два параметра функции: соответственно `grpc.ServerReadableStream` (подробнее [здесь](https://grpc.github.io/grpc/node/grpc-ServerReadableStream.html)) и `callback`.

Начнём с реализации `BidiHello`, который должен поддерживать полнодуплексное взаимодействие.

```typescript
@GrpcStreamCall()
bidiHello(requestStream: any) {
  requestStream.on('data', message => {
    console.log(message);
    requestStream.write({
      reply: 'Hello, world!'
    });
  });
}
```

> **Подсказка** Этот декоратор не требует возвращать какое-либо определённое значение. Предполагается, что поток будет обрабатываться так же, как любой другой стандартный тип потока.

В примере выше мы использовали метод `write()` для записи объектов в поток ответа. Колбэк, переданный вторым параметром в метод `.on()`, будет вызываться каждый раз, когда наш сервис получает новую порцию данных.

Реализуем метод `LotsOfGreetings`.

```typescript
@GrpcStreamCall()
lotsOfGreetings(requestStream: any, callback: (err: unknown, value: HelloResponse) => void) {
  requestStream.on('data', message => {
    console.log(message);
  });
  requestStream.on('end', () => callback(null, { reply: 'Hello, world!' }));
}
```

Здесь мы использовали функцию `callback`, чтобы отправить ответ после того, как обработка `requestStream` завершена.

#### Проверки работоспособности

При запуске gRPC-приложения в оркестраторе вроде Kubernetes вам может понадобиться знать, запущено ли оно и находится ли в работоспособном состоянии. [Спецификация gRPC Health Check](https://grpc.io/docs/guides/health-checking/) — стандарт, позволяющий gRPC-клиентам сообщать о своём состоянии, чтобы оркестратор мог действовать соответствующим образом.

Чтобы добавить поддержку проверок работоспособности gRPC, сначала установите пакет [grpc-node](https://github.com/grpc/grpc-node/tree/master/packages/grpc-health-check):

```bash
$ npm i --save grpc-health-check
```

Затем его можно подключить к gRPC-сервису через хук `onLoadPackageDefinition` в опциях gRPC-сервера, как показано ниже. Обратите внимание, что `protoPath` должен включать и пакет проверки работоспособности, и пакет hero.

**`main.ts`**

```typescript
import { HealthImplementation, protoPath as healthCheckProtoPath } from 'grpc-health-check';

const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  options: {
    protoPath: [
      healthCheckProtoPath,
      protoPath: join(__dirname, 'hero/hero.proto'),
    ],
    onLoadPackageDefinition: (pkg, server) => {
      const healthImpl = new HealthImplementation({
        '': 'UNKNOWN',
      });

      healthImpl.addToServer(server);
      healthImpl.setStatus('', 'SERVING');
    },
  },
});
```

> **Подсказка** [gRPC health probe](https://github.com/grpc-ecosystem/grpc-health-probe) — полезная CLI-утилита для тестирования проверок работоспособности gRPC в контейнеризованном окружении.

#### Метаданные gRPC

Метаданные — это информация о конкретном RPC-вызове в виде списка пар «ключ-значение», где ключи являются строками, а значения обычно тоже строки, но могут быть и двоичными данными. Для самого gRPC метаданные непрозрачны — они позволяют клиенту передавать серверу связанную с вызовом информацию и наоборот. Метаданные могут включать токены аутентификации, идентификаторы запросов и метки для мониторинга, а также сведения о данных, например количество записей в наборе.

Чтобы прочитать метаданные в обработчике `@GrpcMethod()`, используйте второй аргумент (metadata) типа `Metadata` (импортируется из пакета `grpc`).

Чтобы отправить метаданные обратно из обработчика, используйте метод `ServerUnaryCall#sendMetadata()` (третий аргумент обработчика).

**`heroes.controller.ts`**

```typescript
@Controller()
export class HeroesService {
  @GrpcMethod()
  findOne(data: HeroById, metadata: Metadata, call: ServerUnaryCall<any, any>): Hero {
    const serverMetadata = new Metadata();
    const items = [
      { id: 1, name: 'John' },
      { id: 2, name: 'Doe' },
    ];

    serverMetadata.add('Set-Cookie', 'yummy_cookie=choco');
    call.sendMetadata(serverMetadata);

    return items.find(({ id }) => id === data.id);
  }
}
```

Точно так же, чтобы прочитать метаданные в обработчиках, помеченных декоратором `@GrpcStreamMethod()` ([стратегия subject](<07. gRPC.md#стратегия-subject>)), используйте второй аргумент (metadata) типа `Metadata` (импортируется из пакета `grpc`).

Чтобы отправить метаданные обратно из обработчика, используйте метод `ServerDuplexStream#sendMetadata()` (третий аргумент обработчика).

Чтобы прочитать метаданные внутри [обработчиков потока вызова](<07. gRPC.md#обработчик-потока-вызова>) (обработчиков, помеченных декоратором `@GrpcStreamCall()`), подпишитесь на событие `metadata` у ссылки `requestStream`:

```typescript
requestStream.on('metadata', (metadata: Metadata) => {
  const meta = metadata.get('X-Meta');
});
```
