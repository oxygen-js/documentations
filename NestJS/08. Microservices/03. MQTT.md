### MQTT

[MQTT](https://mqtt.org/) (Message Queuing Telemetry Transport) — это открытый легковесный протокол обмена сообщениями, оптимизированный под низкие задержки. Он предоставляет масштабируемый и экономичный способ соединять устройства по модели **публикации/подписки**. Система обмена сообщениями на основе MQTT состоит из публикующего сервера, брокера и одного или нескольких клиентов. Протокол рассчитан на устройства с ограниченными ресурсами и на сети с узкой полосой пропускания, высокими задержками или низкой надёжностью.

#### Установка

Чтобы приступить к созданию микросервисов на основе MQTT, сначала установите нужный пакет:

```bash
$ npm i --save mqtt
```

#### Обзор

Чтобы использовать транспорт MQTT, передайте методу `createMicroservice()` следующий объект параметров:

**`main.ts`**

```typescript
const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  transport: Transport.MQTT,
  options: {
    url: 'mqtt://localhost:1883',
  },
});
```

> **Подсказка** Перечисление `Transport` импортируется из пакета `@nestjs/microservices`.

#### Параметры

Объект `options` специфичен для выбранного транспорта. Транспорт <strong>MQTT</strong> предоставляет свойства, описанные [здесь](https://github.com/mqttjs/MQTT.js/#mqttclientstreambuilder-options).

#### Клиент

Как и в случае других микросервисных транспортов, у вас есть <a href="01.%20Overview.md#клиент-класс-производитель">несколько способов</a> создать экземпляр `ClientProxy` для MQTT.

Один из способов — использовать `ClientsModule`. Чтобы создать экземпляр клиента с помощью `ClientsModule`, импортируйте его и вызовите метод `register()`, передав объект параметров с теми же свойствами, что были показаны выше для метода `createMicroservice()`, а также свойство `name`, которое будет использоваться как токен внедрения. Подробнее о `ClientsModule` читайте <a href="01.%20Overview.md#клиент-класс-производитель">здесь</a>.

```typescript
@Module({
  imports: [
    ClientsModule.register([
      {
        name: 'MATH_SERVICE',
        transport: Transport.MQTT,
        options: {
          url: 'mqtt://localhost:1883',
        }
      },
    ]),
  ]
  ...
})
```

Можно использовать и другие способы создания клиента (`ClientProxyFactory` или `@Client()`). О них можно прочитать <a href="01.%20Overview.md#клиент-класс-производитель">здесь</a>.

#### Контекст

В более сложных сценариях вам может понадобиться доступ к дополнительной информации о входящем запросе. При использовании транспорта MQTT вы можете обратиться к объекту `MqttContext`.

```typescript
@MessagePattern('notifications')
getNotifications(@Payload() data: number[], @Ctx() context: MqttContext) {
  console.log(`Topic: ${context.getTopic()}`);
}
```

> **Подсказка** `@Payload()`, `@Ctx()` и `MqttContext` импортируются из пакета `@nestjs/microservices`.

Чтобы получить доступ к исходному [пакету](https://github.com/mqttjs/mqtt-packet) mqtt, используйте метод `getPacket()` объекта `MqttContext`:

```typescript
@MessagePattern('notifications')
getNotifications(@Payload() data: number[], @Ctx() context: MqttContext) {
  console.log(context.getPacket());
}
```

#### Подстановочные знаки

Подписка может быть на конкретный топик, а может содержать подстановочные знаки. Доступны два таких знака: `+` и `#`. `+` — одноуровневый подстановочный знак, а `#` — многоуровневый, охватывающий сразу несколько уровней топика.

```typescript
@MessagePattern('sensors/+/temperature/+')
getTemperature(@Ctx() context: MqttContext) {
  console.log(`Topic: ${context.getTopic()}`);
}
```

#### Качество обслуживания (QoS)

Любая подписка, созданная декораторами `@MessagePattern` или `@EventPattern`, оформляется с QoS 0. Если требуется более высокий уровень QoS, его можно задать глобально в блоке `subscribeOptions` при установке соединения:

**`main.ts`**

```typescript
const app = await NestFactory.createMicroservice<MicroserviceOptions>(AppModule, {
  transport: Transport.MQTT,
  options: {
    url: 'mqtt://localhost:1883',
    subscribeOptions: {
      qos: 2
    },
  },
});
```

#### QoS для отдельного шаблона

Вы можете переопределить QoS подписки MQTT для конкретного шаблона, указав `qos` в поле `extras` декоратора шаблона. Если значение не задано, по умолчанию используется глобальное `subscribeOptions.qos`.

```typescript
@EventPattern('critical-events', { extras: { qos: 2 } })
handleCriticalEvent(@Payload() data: any) {
  // Эта подписка использует QoS 2
}

@EventPattern('metrics', { extras: { qos: 0 } })
handleMetrics(@Payload() data: any) {
  // Эта подписка использует QoS 0
}
```

> **Подсказка** Настройка QoS для отдельного шаблона не влияет на существующее поведение. Если `extras.qos` не указан, подписка использует глобальное значение `subscribeOptions.qos`.

#### Построители записей

Чтобы настроить параметры сообщения (изменить уровень QoS, установить флаги Retain или DUP либо добавить дополнительные свойства к полезной нагрузке), вы можете воспользоваться классом `MqttRecordBuilder`. Например, чтобы задать `QoS` равным `2`, используйте метод `setQoS`:

```typescript
const userProperties = { 'x-version': '1.0.0' };
const record = new MqttRecordBuilder(':cat:')
  .setProperties({ userProperties })
  .setQoS(1)
  .build();
client.send('replace-emoji', record).subscribe(...);
```

> **Подсказка** Класс `MqttRecordBuilder` экспортируется из пакета `@nestjs/microservices`.

Эти же параметры можно прочитать и на стороне сервера, обратившись к `MqttContext`.

```typescript
@MessagePattern('replace-emoji')
replaceEmoji(@Payload() data: string, @Ctx() context: MqttContext): string {
  const { properties: { userProperties } } = context.getPacket();
  return userProperties['x-version'] === '1.0.0' ? '🐱' : '🐈';
}
```

Иногда бывает нужно задать пользовательские свойства сразу для нескольких запросов — эти параметры можно передать в `ClientProxyFactory`.

```typescript
import { Module } from '@nestjs/common';
import { ClientProxyFactory, Transport } from '@nestjs/microservices';

@Module({
  providers: [
    {
      provide: 'API_v1',
      useFactory: () =>
        ClientProxyFactory.create({
          transport: Transport.MQTT,
          options: {
            url: 'mqtt://localhost:1833',
            userProperties: { 'x-version': '1.0.0' },
          },
        }),
    },
  ],
})
export class ApiModule {}
```

#### Обновления состояния экземпляра

Чтобы получать в реальном времени сведения о соединении и состоянии нижележащего экземпляра драйвера, вы можете подписаться на поток `status`. Этот поток выдаёт обновления состояния, специфичные для выбранного драйвера. Для драйвера MQTT поток `status` порождает события `connected`, `disconnected`, `reconnecting` и `closed`.

```typescript
this.client.status.subscribe((status: MqttStatus) => {
  console.log(status);
});
```

> **Подсказка** Тип `MqttStatus` импортируется из пакета `@nestjs/microservices`.

Аналогично вы можете подписаться на поток `status` сервера, чтобы получать уведомления о его состоянии.

```typescript
const server = app.connectMicroservice<MicroserviceOptions>(...);
server.status.subscribe((status: MqttStatus) => {
  console.log(status);
});
```

#### Прослушивание событий MQTT

Иногда бывает нужно отслеживать внутренние события, порождаемые микросервисом. Например, вы можете слушать событие `error`, чтобы запускать дополнительные действия при возникновении ошибки. Для этого используйте метод `on()`, как показано ниже:

```typescript
this.client.on('error', (err) => {
  console.error(err);
});
```

Точно так же можно слушать внутренние события сервера:

```typescript
server.on<MqttEvents>('error', (err) => {
  console.error(err);
});
```

> **Подсказка** Тип `MqttEvents` импортируется из пакета `@nestjs/microservices`.

#### Доступ к нижележащему драйверу

В более сложных сценариях вам может понадобиться доступ к экземпляру нижележащего драйвера. Это бывает полезно, например, чтобы вручную закрыть соединение или воспользоваться специфичными для драйвера методами. Учтите, однако, что в большинстве случаев обращаться к драйверу напрямую **не требуется**.

Для этого можно использовать метод `unwrap()`, возвращающий экземпляр нижележащего драйвера. В параметре обобщённого типа следует указать ожидаемый тип экземпляра драйвера.

```typescript
const mqttClient = this.client.unwrap<import('mqtt').MqttClient>();
```

Точно так же можно получить доступ к экземпляру нижележащего драйвера на стороне сервера:

```typescript
const mqttClient = server.unwrap<import('mqtt').MqttClient>();
```
