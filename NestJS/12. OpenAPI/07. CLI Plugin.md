### Плагин CLI

Система отражения метаданных в [TypeScript](https://www.typescriptlang.org/docs/handbook/decorators.html) имеет ряд ограничений: например, она не позволяет определить, из каких свойств состоит класс, или понять, является ли конкретное свойство необязательным. Однако часть этих ограничений можно обойти на этапе компиляции. Nest предоставляет плагин, который дополняет процесс компиляции TypeScript и сокращает объём шаблонного кода.

> **Подсказка** Плагин подключается **по желанию**. Если вам так удобнее, вы можете объявлять все декораторы вручную либо указывать только те, что вам нужны.

#### Обзор

Плагин Swagger автоматически:

- добавляет ко всем свойствам DTO аннотацию `@ApiProperty`, если только не используется `@ApiHideProperty`;
- задаёт свойство `required` в зависимости от знака вопроса (например, для `name?: string` будет установлено `required: false`);
- задаёт свойство `type` или `enum` в зависимости от типа (поддерживаются и массивы);
- задаёт свойство `default` на основе присвоенного значения по умолчанию;
- задаёт ряд правил валидации на основе декораторов `class-validator` (если `classValidatorShim` равно `true`);
- добавляет к каждому эндпоинту декоратор ответа с корректным статусом и `type` (модель ответа);
- генерирует описания свойств и эндпоинтов на основе комментариев (если `introspectComments` равно `true`);
- генерирует примеры значений для свойств на основе комментариев (если `introspectComments` равно `true`).

Обратите внимание: чтобы плагин анализировал ваши файлы, их имена **должны** иметь один из следующих суффиксов: `['.dto.ts', '.entity.ts']` (например, `create-user.dto.ts`).

Если вы используете другой суффикс, поведение плагина можно изменить с помощью опции `dtoFileNameSuffix` (см. ниже).

Раньше, чтобы обеспечить интерактивную работу со Swagger UI, приходилось дублировать много кода — иначе пакет не знал, как объявлять ваши модели и компоненты в спецификации. Например, простой класс `CreateUserDto` описывался так:

```typescript
export class CreateUserDto {
  @ApiProperty()
  email: string;

  @ApiProperty()
  password: string;

  @ApiProperty({ enum: RoleEnum, default: [], isArray: true })
  roles: RoleEnum[] = [];

  @ApiProperty({ required: false, default: true })
  isEnabled?: boolean = true;
}
```

Для проектов среднего размера это не большая проблема, но при большом количестве классов код становится многословным и трудным в поддержке.

Если [включить плагин Swagger](<07. CLI Plugin.md#использование-плагина-cli>), то приведённое выше определение класса можно записать просто так:

```typescript
export class CreateUserDto {
  email: string;
  password: string;
  roles: RoleEnum[] = [];
  isEnabled?: boolean = true;
}
```

> **Примечание** Плагин Swagger выводит аннотации @ApiProperty() из типов TypeScript и декораторов class-validator. Это помогает понятно описать ваш API в генерируемой документации Swagger UI. Однако валидацию во время выполнения по-прежнему обеспечивают декораторы class-validator. Поэтому такие валидаторы, как `IsEmail()`, `IsNumber()` и другие, нужно продолжать использовать.

Следовательно, если вы рассчитываете на автоматические аннотации при генерации документации, но при этом хотите сохранить валидацию во время выполнения, декораторы class-validator всё равно необходимы.

> **Подсказка** Если в DTO вы используете [утилиты mapped types](<05. Mapped Types.md>) (например, `PartialType`), импортируйте их из `@nestjs/swagger`, а не из `@nestjs/mapped-types`, — иначе плагин не подхватит схему.

Плагин добавляет нужные декораторы на лету, опираясь на **абстрактное синтаксическое дерево**. Благодаря этому вам не придётся возиться с декораторами `@ApiProperty`, разбросанными по всему коду.

> **Подсказка** Плагин автоматически создаёт все недостающие свойства swagger, но если нужно их переопределить, просто задайте их явно через `@ApiProperty()`.

#### Интроспекция комментариев

Когда включена интроспекция комментариев, плагин CLI генерирует описания и примеры значений для свойств на основе комментариев.

Например, возьмём свойство `roles`:

```typescript
/**
 * Список ролей пользователя
 * @example ['admin']
 */
@ApiProperty({
  description: `A list of user's roles`,
  example: ['admin'],
})
roles: RoleEnum[] = [];
```

Здесь и описание, и пример приходится дублировать. Если включить `introspectComments`, плагин CLI сможет извлечь эти комментарии и автоматически подставить описания (а также примеры, если они заданы) для свойств. Теперь приведённое выше свойство объявляется просто так:

```typescript
/**
 * Список ролей пользователя
 * @example ['admin']
 */
roles: RoleEnum[] = [];
```

Есть опции плагина `dtoKeyOfComment` и `controllerKeyOfComment`, которые задают, в какие свойства декораторов `ApiProperty` и `ApiOperation` соответственно плагин подставляет значения. Смотрите пример ниже:

```typescript
export class SomeController {
  /**
   * Создать некий ресурс
   */
  @Post()
  create() {}
}
```

Это эквивалентно следующей инструкции:

```typescript
@ApiOperation({ summary: "Create some resource" })
```

> **Подсказка** Для моделей действует та же логика, но применяется декоратор `ApiProperty`.

Для контроллеров можно указать не только краткое описание (summary), но и подробное описание (remarks), теги (например,` @deprecated`) и примеры ответов:

```ts
/**
 * Создать нового кота
 *
 * @remarks Эта операция позволяет создать нового кота.
 *
 * @deprecated
 * @throws {500} Что-то пошло не так.
 * @throws {400} Некорректный запрос.
 */
@Post()
async create(): Promise<Cat> {}
```

#### Использование плагина CLI

Чтобы включить плагин, откройте `nest-cli.json` (если вы используете [Nest CLI](<../11. CLI/01. Overview.md>)) и добавьте следующую конфигурацию `plugins`:

```javascript
{
  "collection": "@nestjs/schematics",
  "sourceRoot": "src",
  "compilerOptions": {
    "plugins": ["@nestjs/swagger"]
  }
}
```

Поведение плагина можно настроить с помощью свойства `options`.

```javascript
{
  "collection": "@nestjs/schematics",
  "sourceRoot": "src",
  "compilerOptions": {
    "plugins": [
      {
        "name": "@nestjs/swagger",
        "options": {
          "classValidatorShim": false,
          "introspectComments": true,
          "skipAutoHttpCode": true
        }
      }
    ]
  }
}
```

Свойство `options` должно соответствовать следующему интерфейсу:

```typescript
export interface PluginOptions {
  dtoFileNameSuffix?: string[];
  controllerFileNameSuffix?: string[];
  classValidatorShim?: boolean;
  dtoKeyOfComment?: string;
  controllerKeyOfComment?: string;
  introspectComments?: boolean;
  skipAutoHttpCode?: boolean;
  esmCompatible?: boolean;
}
```

При изменении опций плагина обязательно удаляйте папку `/dist` и пересобирайте приложение.
Если вы не используете CLI, а вместо этого у вас своя конфигурация `webpack`, плагин можно применить вместе с `ts-loader`:

```javascript
getCustomTransformers: (program: any) => ({
  before: [require('@nestjs/swagger/plugin').before({}, program)]
}),
```

#### Сборщик SWC

В стандартных конфигурациях (не монорепозиторий), чтобы использовать плагины CLI со сборщиком SWC, нужно включить проверку типов, как описано [здесь](<../13. Recipes/03. SWC (fast compiler).md#проверка-типов>).

```bash
$ nest start -b swc --type-check
```

Для монорепозиториев следуйте инструкциям [отсюда](<../13. Recipes/03. SWC (fast compiler).md#монорепозиторий-и-плагины-cli>).

```bash
$ npx ts-node src/generate-metadata.ts
# ИЛИ npx ts-node apps/{YOUR_APP}/src/generate-metadata.ts
```

Теперь файл с сериализованными метаданными нужно загрузить методом `SwaggerModule#loadPluginMetadata`, как показано ниже:

```typescript
import metadata from './metadata'; // <-- файл, автоматически сгенерированный "PluginMetadataGenerator"

await SwaggerModule.loadPluginMetadata(metadata); // <-- вот здесь
const document = SwaggerModule.createDocument(app, config);
```

#### Интеграция с `ts-jest` (e2e-тесты)

Для запуска e2e-тестов `ts-jest` компилирует исходные файлы на лету, в памяти. Это значит, что компилятор Nest CLI не используется, плагины не применяются и преобразования AST не выполняются.

Чтобы включить плагин, создайте в каталоге с e2e-тестами такой файл:

```javascript
const transformer = require('@nestjs/swagger/plugin');

module.exports.name = 'nestjs-swagger-transformer';
// номер версии нужно менять при каждом изменении конфигурации ниже — иначе jest не заметит изменений
module.exports.version = 1;

module.exports.factory = (cs) => {
  return transformer.before(
    {
      // опции @nestjs/swagger/plugin (могут быть пустыми)
    },
    cs.program, // "cs.tsCompiler.program" для старых версий Jest (<= v27)
  );
};
```

После этого подключите AST-трансформер в конфигурационном файле `jest`. По умолчанию (в стартовом приложении) файл конфигурации e2e-тестов находится в папке `test` и называется `jest-e2e.json`.

Если вы используете `jest@<29`, воспользуйтесь фрагментом ниже.

```json
{
  ... // остальная конфигурация
  "globals": {
    "ts-jest": {
      "astTransformers": {
        "before": ["<path to the file created above>"]
      }
    }
  }
}
```

Если вы используете `jest@^29`, воспользуйтесь фрагментом ниже, поскольку предыдущий подход объявлен устаревшим.

```json
{
  ... // остальная конфигурация
  "transform": {
    "^.+\\.(t|j)s$": [
      "ts-jest",
      {
        "astTransformers": {
          "before": ["<path to the file created above>"]
        }
      }
    ]
  }
}
```

#### Устранение неполадок с `jest` (e2e-тесты)

Если `jest` словно не замечает изменений в вашей конфигурации, возможно, он уже **закэшировал** результат сборки. Чтобы применить новую конфигурацию, нужно очистить каталог кэша Jest.

Для очистки кэша выполните в папке вашего проекта NestJS следующую команду:

```bash
$ npx jest --clearCache
```

Если автоматическая очистка кэша не сработала, папку кэша можно удалить вручную такими командами:

```bash
# Найдите каталог кэша jest (обычно /tmp/jest_rs),
# выполнив следующую команду в корне вашего проекта NestJS
$ npx jest --showConfig | grep cache
# пример результата:
#   "cache": true,
#   "cacheDirectory": "/tmp/jest_rs"

# Удалите или очистите каталог кэша Jest
$ rm -rf  <cacheDirectory value>
# пример:
# rm -rf /tmp/jest_rs
```
