### Типы и параметры

`SwaggerModule` ищет в обработчиках маршрутов все декораторы `@Body()`, `@Query()` и `@Param()`, чтобы сформировать документ API. Кроме того, с помощью рефлексии он создаёт соответствующие определения моделей. Рассмотрим следующий код:

```typescript
@Post()
async create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
```

> **Подсказка** Чтобы задать определение тела запроса явно, используйте декоратор `@ApiBody()` (импортируется из пакета `@nestjs/swagger`).

На основе `CreateCatDto` в Swagger UI будет создано такое определение модели:

<figure><img src="https://docs.nestjs.com/assets/swagger-dto.png" /></figure>

Как видите, определение пустое, хотя у класса объявлено несколько свойств. Чтобы `SwaggerModule` увидел свойства класса, их нужно либо пометить декоратором `@ApiProperty()`, либо воспользоваться плагином CLI (подробнее — в разделе **Плагин**), который сделает это автоматически:

```typescript
import { ApiProperty } from '@nestjs/swagger';

export class CreateCatDto {
  @ApiProperty()
  name: string;

  @ApiProperty()
  age: number;

  @ApiProperty()
  breed: string;
}
```

> **Подсказка** Вместо того чтобы вручную помечать каждое свойство, попробуйте плагин Swagger (см. раздел [Плагин](<07. CLI Plugin.md>)) — он сделает это за вас автоматически.

Откроем браузер и проверим сгенерированную модель `CreateCatDto`:

<figure><img src="https://docs.nestjs.com/assets/swagger-dto2.png" /></figure>

Кроме того, декоратор `@ApiProperty()` позволяет задавать различные свойства [объекта Schema](https://swagger.io/specification/#schemaObject):

```typescript
@ApiProperty({
  description: 'The age of a cat',
  minimum: 1,
  default: 1,
})
age: number;
```

> **Подсказка** Вместо явной записи `@ApiProperty({ required: false })` можно использовать сокращённый декоратор `@ApiPropertyOptional()`.

Чтобы явно задать тип свойства, используйте ключ `type`:

```typescript
@ApiProperty({
  type: Number,
})
age: number;
```

#### Массивы

Если свойство является массивом, тип массива нужно указать вручную, как показано ниже:

```typescript
@ApiProperty({ type: [String] })
names: string[];
```

> **Подсказка** Попробуйте плагин Swagger (см. раздел [Плагин](<07. CLI Plugin.md>)) — он определяет массивы автоматически.

Либо укажите тип первым элементом массива (как показано выше), либо задайте свойству `isArray` значение `true`.

#### Циклические зависимости

Если между классами есть циклические зависимости, передавайте информацию о типе в `SwaggerModule` через отложенную функцию:

```typescript
@ApiProperty({ type: () => Node })
node: Node;
```

> **Подсказка** Попробуйте плагин Swagger (см. раздел [Плагин](<07. CLI Plugin.md>)) — он определяет циклические зависимости автоматически.

#### Обобщения и интерфейсы

Поскольку TypeScript не хранит метаданные об обобщениях и интерфейсах, при их использовании в DTO `SwaggerModule` может не суметь корректно сформировать определения моделей во время выполнения. Например, следующий код модуль Swagger разберёт неверно:

```typescript
createBulk(@Body() usersDto: CreateUserDto[])
```

Чтобы обойти это ограничение, задайте тип явно:

```typescript
@ApiBody({ type: [CreateUserDto] })
createBulk(@Body() usersDto: CreateUserDto[])
```

#### Перечисления

Чтобы распознать `enum`, нужно вручную задать свойство `enum` у `@ApiProperty`, передав массив значений.

```typescript
@ApiProperty({ enum: ['Admin', 'Moderator', 'User']})
role: UserRole;
```

Как вариант, объявите настоящее перечисление TypeScript:

```typescript
export enum UserRole {
  Admin = 'Admin',
  Moderator = 'Moderator',
  User = 'User',
}
```

После этого перечисление можно использовать напрямую с декоратором параметра `@Query()` в сочетании с декоратором `@ApiQuery()`.

```typescript
@ApiQuery({ name: 'role', enum: UserRole })
async filterByRole(@Query('role') role: UserRole = UserRole.User) {}
```

<figure><img src="https://docs.nestjs.com/assets/enum_query.gif" /></figure>

Если задать `isArray` значение **true**, `enum` можно будет выбирать в режиме **множественного выбора**:

#### Схема перечислений

По умолчанию свойство `enum` добавляет к `parameter` «сырое» определение [Enum](https://swagger.io/docs/specification/data-models/enums/).

```yaml
- breed:
    type: 'string'
    enum:
      - Persian
      - Tabby
      - Siamese
```

В большинстве случаев такая спецификация вполне подходит. Однако если вы используете инструмент, который принимает спецификацию **на вход** и генерирует **клиентский** код, вы можете столкнуться с дублированием `enums` в сгенерированном коде. Рассмотрим такой фрагмент:

```typescript
// сгенерированный клиентский код
export class CatDetail {
  breed: CatDetailEnum;
}

export class CatInformation {
  breed: CatInformationEnum;
}

export enum CatDetailEnum {
  Persian = 'Persian',
  Tabby = 'Tabby',
  Siamese = 'Siamese',
}

export enum CatInformationEnum {
  Persian = 'Persian',
  Tabby = 'Tabby',
  Siamese = 'Siamese',
}
```

> **Подсказка** Приведённый фрагмент сгенерирован инструментом [NSwag](https://github.com/RicoSuter/NSwag).

Как видите, у вас получились два совершенно одинаковых `enums`.
Чтобы решить эту проблему, передайте в декораторе `enumName` вместе со свойством `enum`.

```typescript
export class CatDetail {
  @ApiProperty({ enum: CatBreed, enumName: 'CatBreed' })
  breed: CatBreed;
}
```

Свойство `enumName` позволяет `@nestjs/swagger` вынести `CatBreed` в отдельную `schema`, благодаря чему перечисление `CatBreed` становится переиспользуемым. Спецификация будет выглядеть так:

```yaml
CatDetail:
  type: 'object'
  properties:
    ...
    - breed:
        schema:
          $ref: '#/components/schemas/CatBreed'
CatBreed:
  type: string
  enum:
    - Persian
    - Tabby
    - Siamese
```

> **Подсказка** Любой **декоратор**, принимающий свойство `enum`, принимает и `enumName`.

#### Примеры значений свойств

Задать один пример для свойства можно с помощью ключа `example`:

```typescript
@ApiProperty({
  example: 'persian',
})
breed: string;
```

Если нужно указать несколько примеров, используйте ключ `examples`, передав объект следующей структуры:

```typescript
@ApiProperty({
  examples: {
    Persian: { value: 'persian' },
    Tabby: { value: 'tabby' },
    Siamese: { value: 'siamese' },
    'Scottish Fold': { value: 'scottish_fold' },
  },
})
breed: string;
```

#### «Сырые» определения

В некоторых случаях — например, для глубоко вложенных массивов или матриц — тип приходится описывать вручную:

```typescript
@ApiProperty({
  type: 'array',
  items: {
    type: 'array',
    items: {
      type: 'number',
    },
  },
})
coords: number[][];
```

Точно так же можно задавать «сырые» схемы объектов:

```typescript
@ApiProperty({
  type: 'object',
  properties: {
    name: {
      type: 'string',
      example: 'Error'
    },
    status: {
      type: 'number',
      example: 400
    }
  },
  required: ['name', 'status']
})
rawDefinition: Record<string, any>;
```

Чтобы вручную описать входные/выходные данные в классах контроллеров, используйте свойство `schema`:

```typescript
@ApiBody({
  schema: {
    type: 'array',
    items: {
      type: 'array',
      items: {
        type: 'number',
      },
    },
  },
})
async create(@Body() coords: number[][]) {}
```

#### Дополнительные модели

Чтобы объявить дополнительные модели, на которые нет прямых ссылок в контроллерах, но которые должны быть учтены модулем Swagger, используйте декоратор `@ApiExtraModels()`:

```typescript
@ApiExtraModels(ExtraModel)
export class CreateCatDto {}
```

> **Подсказка** Для конкретного класса модели `@ApiExtraModels()` достаточно применить один раз.

Как вариант, можно передать в метод `SwaggerModule.createDocument()` объект параметров со свойством `extraModels`:

```typescript
const documentFactory = () =>
  SwaggerModule.createDocument(app, options, {
    extraModels: [ExtraModel],
  });
```

Чтобы получить ссылку (`$ref`) на вашу модель, используйте функцию `getSchemaPath(ExtraModel)`:

```typescript
'application/vnd.api+json': {
   schema: { $ref: getSchemaPath(ExtraModel) },
},
```

#### oneOf, anyOf, allOf

Для комбинирования схем используйте ключевые слова `oneOf`, `anyOf` или `allOf` ([подробнее](https://swagger.io/docs/specification/data-models/oneof-anyof-allof-not/)).

```typescript
@ApiProperty({
  oneOf: [
    { $ref: getSchemaPath(Cat) },
    { $ref: getSchemaPath(Dog) },
  ],
})
pet: Cat | Dog;
```

Если нужно описать полиморфный массив (то есть массив, элементы которого относятся к разным схемам), опишите тип вручную «сырым» определением (см. выше).

```typescript
type Pet = Cat | Dog;

@ApiProperty({
  type: 'array',
  items: {
    oneOf: [
      { $ref: getSchemaPath(Cat) },
      { $ref: getSchemaPath(Dog) },
    ],
  },
})
pets: Pet[];
```

> **Подсказка** Функция `getSchemaPath()` импортируется из `@nestjs/swagger`.

И `Cat`, и `Dog` должны быть объявлены как дополнительные модели с помощью декоратора `@ApiExtraModels()` (на уровне класса).

#### Имя и описание схемы

Как вы могли заметить, имя сгенерированной схемы образуется от имени исходного класса модели (например, модель `CreateCatDto` даёт схему `CreateCatDto`). Если вы хотите изменить имя схемы, используйте декоратор `@ApiSchema()`.

Вот пример:

```typescript
@ApiSchema({ name: 'CreateCatRequest' })
class CreateCatDto {}
```

Приведённая выше модель будет преобразована в схему `CreateCatRequest`.

По умолчанию описание к сгенерированной схеме не добавляется. Добавить его можно через атрибут `description`:

```typescript
@ApiSchema({ description: 'Description of the CreateCatDto schema' })
class CreateCatDto {}
```

Тогда описание попадёт в схему:

```yaml
schemas:
  CreateCatDto:
    type: object
    description: Description of the CreateCatDto schema
```
