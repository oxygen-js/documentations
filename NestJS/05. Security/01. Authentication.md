### Аутентификация

Аутентификация — **важнейшая** часть большинства приложений. Существует множество подходов и стратегий её реализации. Выбор подхода для конкретного проекта зависит от его требований. В этой главе рассматривается несколько подходов к аутентификации, которые можно адаптировать под самые разные требования.

Сформулируем требования. В нашем сценарии клиенты сначала аутентифицируются по имени пользователя и паролю. После успешной аутентификации сервер выдаёт JWT, который в последующих запросах передаётся как [bearer-токен](https://tools.ietf.org/html/rfc6750) в заголовке authorization и подтверждает факт аутентификации. Мы также создадим защищённый маршрут, доступный только тем запросам, которые содержат корректный JWT.

Начнём с первого требования — аутентификации пользователя. Затем расширим её выдачей JWT. Наконец, создадим защищённый маршрут, который проверяет наличие корректного JWT в запросе.

#### Создание модуля аутентификации

Начнём с генерации `AuthModule`, а в нём — `AuthService` и `AuthController`. В `AuthService` мы реализуем логику аутентификации, а в `AuthController` — предоставим соответствующие эндпоинты.

```bash
$ nest g module auth
$ nest g controller auth
$ nest g service auth
```

По ходу реализации `AuthService` окажется удобным инкапсулировать операции над пользователями в `UsersService`, поэтому сгенерируем этот модуль и сервис прямо сейчас:

```bash
$ nest g module users
$ nest g service users
```

Замените содержимое сгенерированных файлов по умолчанию на приведённое ниже. В нашем демонстрационном приложении `UsersService` просто хранит жёстко заданный список пользователей в памяти и предоставляет метод поиска по имени пользователя. В реальном приложении именно здесь вы бы построили модель пользователя и слой хранения, используя выбранную библиотеку (например, TypeORM, Sequelize, Mongoose и т. д.).

**`users/users.service.ts`**

```typescript
import { Injectable } from '@nestjs/common';

// Здесь должен быть настоящий класс/интерфейс, представляющий сущность пользователя
export type User = any;

@Injectable()
export class UsersService {
  private readonly users = [
    {
      userId: 1,
      username: 'john',
      password: 'changeme',
    },
    {
      userId: 2,
      username: 'maria',
      password: 'guess',
    },
  ];

  async findOne(username: string): Promise<User | undefined> {
    return this.users.find(user => user.username === username);
  }
}
```

В `UsersModule` требуется единственное изменение — добавить `UsersService` в массив exports декоратора `@Module`, чтобы сервис был виден за пределами этого модуля (вскоре мы используем его в `AuthService`).

**`users/users.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { UsersService } from './users.service';

@Module({
  providers: [UsersService],
  exports: [UsersService],
})
export class UsersModule {}
```

#### Реализация эндпоинта «Вход»

Задача `AuthService` — получить пользователя и проверить пароль. Для этого создадим метод `signIn()`. В коде ниже мы используем удобный spread-оператор ES6, чтобы убрать свойство password из объекта пользователя перед возвратом. Это распространённая практика при возврате объектов пользователей: раскрывать чувствительные поля вроде паролей и других ключей безопасности не следует.

**`auth/auth.service.ts`**

```typescript
import { Injectable, UnauthorizedException } from '@nestjs/common';
import { UsersService } from '../users/users.service';

@Injectable()
export class AuthService {
  constructor(private readonly usersService: UsersService) {}

  async signIn(username: string, pass: string): Promise<any> {
    const user = await this.usersService.findOne(username);
    if (user?.password !== pass) {
      throw new UnauthorizedException();
    }
    const { password, ...result } = user;
    // TODO: Сгенерировать JWT и вернуть его здесь
    // вместо объекта пользователя
    return result;
  }
}
```

> **Предупреждение** Разумеется, в реальном приложении пароль не хранят в открытом виде. Вместо этого используют библиотеку вроде [bcrypt](https://github.com/kelektiv/node.bcrypt.js#readme) с односторонним хэшированием и солью. При таком подходе хранятся только хэши паролей, а сохранённое значение сравнивается с хэшем **входящего** пароля — то есть пароли пользователей никогда не хранятся и не раскрываются в открытом виде. Чтобы не усложнять демонстрационное приложение, мы нарушаем это безусловное правило и используем открытый текст. **Не делайте так в настоящем приложении!**

Теперь обновим `AuthModule`, чтобы он импортировал `UsersModule`.

**`auth/auth.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { AuthService } from './auth.service';
import { AuthController } from './auth.controller';
import { UsersModule } from '../users/users.module';

@Module({
  imports: [UsersModule],
  providers: [AuthService],
  controllers: [AuthController],
})
export class AuthModule {}
```

Теперь откроем `AuthController` и добавим в него метод `signIn()`. Этот метод будет вызываться клиентом для аутентификации пользователя. Он принимает имя пользователя и пароль в теле запроса и возвращает JWT-токен, если пользователь аутентифицирован.

**`auth/auth.controller.ts`**

```typescript
import { Body, Controller, Post, HttpCode, HttpStatus } from '@nestjs/common';
import { AuthService } from './auth.service';

@Controller('auth')
export class AuthController {
  constructor(private readonly authService: AuthService) {}

  @HttpCode(HttpStatus.OK)
  @Post('login')
  signIn(@Body() signInDto: Record<string, any>) {
    return this.authService.signIn(signInDto.username, signInDto.password);
  }
}
```

> **Подсказка** В идеале вместо типа `Record<string, any>` следовало бы описать форму тела запроса с помощью класса DTO. Подробнее см. главу [валидация](<../04. Techniques/04. Validation.md>).

#### JWT-токен

Мы готовы перейти к JWT-части нашей системы аутентификации. Пересмотрим и уточним требования:

- Позволить пользователям аутентифицироваться по имени пользователя и паролю, возвращая JWT для последующих обращений к защищённым эндпоинтам API. Мы уже близки к выполнению этого требования — осталось написать код, выдающий JWT.
- Создать маршруты API, защищённые проверкой наличия корректного JWT в виде bearer-токена.

Для поддержки работы с JWT понадобится установить ещё один пакет:

```bash
$ npm install --save @nestjs/jwt
```

> **Подсказка** Пакет `@nestjs/jwt` (подробнее [здесь](https://github.com/nestjs/jwt)) — вспомогательный пакет для работы с JWT, включая генерацию и проверку JWT-токенов.

Чтобы сервисы оставались аккуратно разделёнными, генерацию JWT мы реализуем в `authService`. Откройте файл `auth.service.ts` в папке `auth`, внедрите `JwtService` и обновите метод `signIn` так, чтобы он генерировал JWT-токен:

**`auth/auth.service.ts`**

```typescript
import { Injectable, UnauthorizedException } from '@nestjs/common';
import { UsersService } from '../users/users.service';
import { JwtService } from '@nestjs/jwt';

@Injectable()
export class AuthService {
  constructor(
    private usersService: UsersService,
    private jwtService: JwtService
  ) {}

  async signIn(
    username: string,
    pass: string,
  ): Promise<{ access_token: string }> {
    const user = await this.usersService.findOne(username);
    if (user?.password !== pass) {
      throw new UnauthorizedException();
    }
    const payload = { sub: user.userId, username: user.username };
    return {
      // 💡 Здесь секретный ключ JWT, которым подписывается полезная нагрузка, —
      // это ключ, переданный в JwtModule
      access_token: await this.jwtService.signAsync(payload),
    };
  }
}
```

Мы используем библиотеку `@nestjs/jwt`, которая предоставляет функцию `signAsync()` для генерации JWT из подмножества свойств объекта `user`; результат возвращается как простой объект с единственным свойством `access_token`. Обратите внимание: для хранения значения `userId` выбрано имя свойства `sub` — в соответствии со стандартами JWT.

Теперь нужно обновить `AuthModule`, чтобы импортировать новые зависимости и настроить `JwtModule`.

Сначала создайте в папке `auth` файл `constants.ts` и добавьте следующий код:

**`auth/constants.ts`**

```typescript
export const jwtConstants = {
  secret: 'DO NOT USE THIS VALUE. INSTEAD, CREATE A COMPLEX SECRET AND KEEP IT SAFE OUTSIDE OF THE SOURCE CODE.',
};
```

Этот ключ мы используем совместно на этапах подписания и проверки JWT.

> **Предупреждение** **Не публикуйте этот ключ**. Здесь мы поступили так лишь для наглядности, но в промышленной системе **ключ необходимо защитить** подходящими средствами — хранилищем секретов, переменной окружения или сервисом конфигурации.

Теперь откройте `auth.module.ts` в папке `auth` и приведите его к такому виду:

**`auth/auth.module.ts`**

```typescript
import { Module } from '@nestjs/common';
import { AuthService } from './auth.service';
import { UsersModule } from '../users/users.module';
import { JwtModule } from '@nestjs/jwt';
import { AuthController } from './auth.controller';
import { jwtConstants } from './constants';

@Module({
  imports: [
    UsersModule,
    JwtModule.register({
      global: true,
      secret: jwtConstants.secret,
      signOptions: { expiresIn: '60s' },
    }),
  ],
  providers: [AuthService],
  controllers: [AuthController],
  exports: [AuthService],
})
export class AuthModule {}
```

> **Подсказка** Мы регистрируем `JwtModule` как глобальный, чтобы упростить себе жизнь. Это значит, что импортировать `JwtModule` где-либо ещё в приложении не потребуется.

Мы настраиваем `JwtModule` через `register()`, передавая объект конфигурации. Подробнее о `JwtModule` в Nest см. [здесь](https://github.com/nestjs/jwt/blob/master/README.md), а о доступных параметрах конфигурации — [здесь](https://github.com/auth0/node-jsonwebtoken#usage).

Давайте снова протестируем маршруты с помощью cURL. Проверить можно на любом из объектов `user`, жёстко заданных в `UsersService`.

```bash
$ # POST to /auth/login
$ curl -X POST http://localhost:3000/auth/login -d '{"username": "john", "password": "changeme"}' -H "Content-Type: application/json"
{"access_token":"eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."}
$ # Note: above JWT truncated
```

#### Реализация guard аутентификации

Теперь можно заняться последним требованием: защитить эндпоинты, потребовав наличия корректного JWT в запросе. Для этого создадим `AuthGuard`, которым будем защищать маршруты.

**`auth/auth.guard.ts`**

```typescript
import {
  CanActivate,
  ExecutionContext,
  Injectable,
  UnauthorizedException,
} from '@nestjs/common';
import { JwtService } from '@nestjs/jwt';
import { Request } from 'express';

@Injectable()
export class AuthGuard implements CanActivate {
  constructor(private readonly jwtService: JwtService) {}

  async canActivate(context: ExecutionContext): Promise<boolean> {
    const request = context.switchToHttp().getRequest();
    const token = this.extractTokenFromHeader(request);
    if (!token) {
      throw new UnauthorizedException();
    }
    try {
      // 💡 Здесь секретный ключ JWT, которым проверяется полезная нагрузка, —
      // это ключ, переданный в JwtModule
      const payload = await this.jwtService.verifyAsync(token);
      // 💡 Здесь мы присваиваем полезную нагрузку объекту запроса,
      // чтобы обращаться к ней в обработчиках маршрутов
      request['user'] = payload;
    } catch {
      throw new UnauthorizedException();
    }
    return true;
  }

  private extractTokenFromHeader(request: Request): string | undefined {
    const [type, token] = request.headers.authorization?.split(' ') ?? [];
    return type === 'Bearer' ? token : undefined;
  }
}
```

Теперь можно реализовать защищённый маршрут и зарегистрировать `AuthGuard` для его защиты.

Откройте файл `auth.controller.ts` и приведите его к такому виду:

**`auth.controller.ts`**

```typescript
import {
  Body,
  Controller,
  Get,
  HttpCode,
  HttpStatus,
  Post,
  Request,
  UseGuards
} from '@nestjs/common';
import { AuthGuard } from './auth.guard';
import { AuthService } from './auth.service';

@Controller('auth')
export class AuthController {
  constructor(private readonly authService: AuthService) {}

  @HttpCode(HttpStatus.OK)
  @Post('login')
  signIn(@Body() signInDto: Record<string, any>) {
    return this.authService.signIn(signInDto.username, signInDto.password);
  }

  @UseGuards(AuthGuard)
  @Get('profile')
  getProfile(@Request() req) {
    return req.user;
  }
}
```

Мы применяем только что созданный `AuthGuard` к маршруту `GET /profile`, тем самым защищая его.

Убедитесь, что приложение запущено, и протестируйте маршруты с помощью `cURL`.

```bash
$ # GET /profile
$ curl http://localhost:3000/auth/profile
{"statusCode":401,"message":"Unauthorized"}

$ # POST /auth/login
$ curl -X POST http://localhost:3000/auth/login -d '{"username": "john", "password": "changeme"}' -H "Content-Type: application/json"
{"access_token":"eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2Vybm..."}

$ # GET /profile using access_token returned from previous step as bearer code
$ curl http://localhost:3000/auth/profile -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2Vybm..."
{"sub":1,"username":"john","iat":...,"exp":...}
```

Обратите внимание: в `AuthModule` мы задали срок жизни JWT в `60 секунд`. Это слишком короткий срок, а тонкости истечения и обновления токенов выходят за рамки этой статьи. Тем не менее такое значение выбрано, чтобы продемонстрировать важное свойство JWT. Если после аутентификации подождать 60 секунд и затем выполнить запрос `GET /auth/profile`, вы получите ответ `401 Unauthorized`. Дело в том, что `@nestjs/jwt` автоматически проверяет срок действия JWT, избавляя вас от необходимости делать это в приложении.

На этом реализация аутентификации через JWT завершена. Теперь JavaScript-клиенты (например, на Angular/React/Vue) и другие JavaScript-приложения могут аутентифицироваться и безопасно взаимодействовать с нашим API-сервером.

#### Включение аутентификации глобально

Если подавляющее большинство ваших эндпоинтов должно быть защищено по умолчанию, вы можете зарегистрировать guard аутентификации как [глобальный guard](<../02. Overview/08. Guards.md#привязка-guard-ов>) и вместо использования декоратора `@UseGuards()` над каждым контроллером просто помечать те маршруты, которые должны быть публичными.

Сначала зарегистрируйте `AuthGuard` как глобальный guard с помощью следующей конструкции (в любом модуле, например в `AuthModule`):

```typescript
providers: [
  {
    provide: APP_GUARD,
    useClass: AuthGuard,
  },
],
```

После этого Nest автоматически привяжет `AuthGuard` ко всем эндпоинтам.

Теперь нужен механизм объявления маршрутов публичными. Для этого создадим пользовательский декоратор с помощью фабричной функции `SetMetadata`.

```typescript
import { SetMetadata } from '@nestjs/common';

export const IS_PUBLIC_KEY = 'isPublic';
export const Public = () => SetMetadata(IS_PUBLIC_KEY, true);
```

В файле выше мы экспортировали две константы. Первая — ключ метаданных `IS_PUBLIC_KEY`, вторая — сам новый декоратор, который мы назовём `Public` (можно назвать его `SkipAuth` или `AllowAnon` — как удобнее в вашем проекте).

Теперь, когда у нас есть пользовательский декоратор `@Public()`, им можно пометить любой метод:

```typescript
@Public()
@Get()
findAll() {
  return [];
}
```

Наконец, нужно, чтобы `AuthGuard` возвращал `true`, когда обнаружены метаданные `"isPublic"`. Для этого воспользуемся классом `Reflector` (подробнее [здесь](<../02. Overview/08. Guards.md#собираем-всё-вместе>)).

```typescript
@Injectable()
export class AuthGuard implements CanActivate {
  constructor(private readonly jwtService: JwtService, private reflector: Reflector) {}

  async canActivate(context: ExecutionContext): Promise<boolean> {
    const isPublic = this.reflector.getAllAndOverride<boolean>(IS_PUBLIC_KEY, [
      context.getHandler(),
      context.getClass(),
    ]);
    if (isPublic) {
      // 💡 Обратите внимание на это условие
      return true;
    }

    const request = context.switchToHttp().getRequest();
    const token = this.extractTokenFromHeader(request);
    if (!token) {
      throw new UnauthorizedException();
    }
    try {
      // 💡 Здесь секретный ключ JWT, которым проверяется полезная нагрузка, —
      // это ключ, переданный в JwtModule
      const payload = await this.jwtService.verifyAsync(token);
      // 💡 Здесь мы присваиваем полезную нагрузку объекту запроса,
      // чтобы обращаться к ней в обработчиках маршрутов
      request['user'] = payload;
    } catch {
      throw new UnauthorizedException();
    }
    return true;
  }

  private extractTokenFromHeader(request: Request): string | undefined {
    const [type, token] = request.headers.authorization?.split(' ') ?? [];
    return type === 'Bearer' ? token : undefined;
  }
}
```
