### Авторизация

**Авторизация** — это процесс, определяющий, что пользователю разрешено делать. Например, администратору позволено создавать, редактировать и удалять записи, а обычному пользователю — только читать их.

Авторизация ортогональна аутентификации и независима от неё. Тем не менее для авторизации необходим механизм аутентификации.

Существует множество подходов и стратегий авторизации. Выбор для конкретного проекта зависит от требований к приложению. В этой главе представлено несколько подходов к авторизации, которые можно адаптировать под самые разные требования.

#### Базовая реализация RBAC

Управление доступом на основе ролей (**RBAC**) — это независимый от конкретных политик механизм контроля доступа, построенный вокруг ролей и привилегий. В этом разделе мы покажем, как реализовать простейший механизм RBAC с помощью [guard-ов](<../02. Overview/08. Guards.md>) Nest.

Сначала создадим перечисление `Role`, описывающее роли в системе:

**`role.enum.ts`**

```typescript
export enum Role {
  User = 'user',
  Admin = 'admin',
}
```

> **Подсказка** В более сложных системах роли можно хранить в базе данных или получать от внешнего провайдера аутентификации.

Теперь создадим декоратор `@Roles()`. Он позволит указывать, какие роли требуются для доступа к тем или иным ресурсам.

**`roles.decorator.ts`**

```typescript
import { SetMetadata } from '@nestjs/common';
import { Role } from '../enums/role.enum';

export const ROLES_KEY = 'roles';
export const Roles = (...roles: Role[]) => SetMetadata(ROLES_KEY, roles);
```

Теперь, когда у нас есть собственный декоратор `@Roles()`, им можно пометить любой обработчик маршрута.

**`cats.controller.ts`**

```typescript
@Post()
@Roles(Role.Admin)
create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
```

Наконец, создадим класс `RolesGuard`, который сравнит роли, назначенные текущему пользователю, с ролями, необходимыми для обрабатываемого маршрута. Чтобы получить роли маршрута (пользовательские метаданные), мы воспользуемся вспомогательным классом `Reflector` — он доступен «из коробки» и экспортируется из пакета `@nestjs/core`.

**`roles.guard.ts`**

```typescript
import { Injectable, CanActivate, ExecutionContext } from '@nestjs/common';
import { Reflector } from '@nestjs/core';

@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    const requiredRoles = this.reflector.getAllAndOverride<Role[]>(ROLES_KEY, [
      context.getHandler(),
      context.getClass(),
    ]);
    if (!requiredRoles) {
      return true;
    }
    const { user } = context.switchToHttp().getRequest();
    return requiredRoles.some((role) => user.roles?.includes(role));
  }
}
```

> **Подсказка** Подробнее о том, как использовать `Reflector` с учётом контекста, читайте в разделе [Рефлексия и метаданные](<../03. Fundamentals/08. Execution context.md#рефлексия-и-метаданные>) главы о контексте выполнения.

> **Важно** Этот пример назван «**базовым**», поскольку мы проверяем наличие ролей только на уровне обработчика маршрута. В реальных приложениях бывают эндпоинты и обработчики, выполняющие несколько операций, каждая из которых требует своего набора разрешений. В таком случае придётся предусмотреть механизм проверки ролей где-то внутри бизнес-логики, а сопровождать это несколько сложнее: не будет единого места, связывающего разрешения с конкретными действиями.

В этом примере мы исходили из того, что `request.user` содержит объект пользователя и разрешённые роли (в свойстве `roles`). В вашем приложении такая связь, скорее всего, устанавливается в собственном **guard-е аутентификации** — подробности см. в главе об [аутентификации](<01. Authentication.md>).

Чтобы пример работал, класс `User` должен выглядеть так:

```typescript
class User {
  // ...other properties
  roles: Role[];
}
```

Наконец, не забудьте зарегистрировать `RolesGuard` — например, на уровне контроллера или глобально:

```typescript
providers: [
  {
    provide: APP_GUARD,
    useClass: RolesGuard,
  },
],
```

Когда пользователь с недостаточными привилегиями обращается к эндпоинту, Nest автоматически возвращает такой ответ:

```typescript
{
  "statusCode": 403,
  "message": "Forbidden resource",
  "error": "Forbidden"
}
```

> **Подсказка** Если вы хотите вернуть другой ответ с ошибкой, вместо возврата логического значения выбрасывайте собственное исключение.

#### Авторизация на основе claim-ов

При создании учётной записи ей могут быть назначены один или несколько claim-ов, выданных доверенной стороной. Claim — это пара «имя — значение», описывающая, что субъект может делать, а не то, чем он является.

Чтобы реализовать в Nest авторизацию на основе claim-ов, можно пройти те же шаги, что описаны выше в разделе о [RBAC](<02. Authorization.md#базовая-реализация-rbac>), с одним существенным отличием: вместо проверки конкретных ролей нужно сравнивать **разрешения**. Каждому пользователю назначается набор разрешений. Аналогично каждый ресурс или эндпоинт определяет, какие разрешения нужны для доступа к нему (например, через отдельный декоратор `@RequirePermissions()`).

**`cats.controller.ts`**

```typescript
@Post()
@RequirePermissions(Permission.CREATE_CAT)
create(@Body() createCatDto: CreateCatDto) {
  this.catsService.create(createCatDto);
}
```

> **Подсказка** В примере выше `Permission` (по аналогии с `Role` из раздела о RBAC) — это перечисление TypeScript, содержащее все доступные в системе разрешения.

#### Интеграция с CASL

[CASL](https://casl.js.org/) — изоморфная библиотека авторизации, ограничивающая доступ клиента к ресурсам. Она рассчитана на постепенное внедрение и легко масштабируется от простой авторизации на основе claim-ов до полноценной авторизации на основе субъектов и атрибутов.

Для начала установите пакет `@casl/ability`:

```bash
$ npm i @casl/ability
```

> **Подсказка** В этом примере мы выбрали CASL, но вы можете использовать любую другую библиотеку, например `accesscontrol` или `acl`, — в зависимости от предпочтений и потребностей проекта.

После установки, чтобы наглядно показать механику CASL, определим два класса-сущности: `User` и `Article`.

```typescript
class User {
  id: number;
  isAdmin: boolean;
}
```

Класс `User` состоит из двух свойств: `id` — уникальный идентификатор пользователя и `isAdmin` — признак наличия у пользователя прав администратора.

```typescript
class Article {
  id: number;
  isPublished: boolean;
  authorId: number;
}
```

У класса `Article` три свойства: `id`, `isPublished` и `authorId`. `id` — уникальный идентификатор статьи, `isPublished` указывает, опубликована ли статья, а `authorId` — идентификатор пользователя, написавшего её.

Теперь уточним требования для нашего примера:

- Администраторы могут управлять (создавать, читать, изменять, удалять) всеми сущностями.
- Пользователи имеют доступ ко всему только на чтение.
- Пользователи могут изменять свои статьи (`article.authorId === userId`).
- Уже опубликованные статьи удалять нельзя (`article.isPublished === true`).

С учётом этого начнём с создания перечисления `Action`, описывающего все возможные действия пользователей над сущностями:

```typescript
export enum Action {
  Manage = 'manage',
  Create = 'create',
  Read = 'read',
  Update = 'update',
  Delete = 'delete',
}
```

> **Важно** `manage` — специальное ключевое слово CASL, означающее «любое действие».

Чтобы инкапсулировать библиотеку CASL, сгенерируем `CaslModule` и `CaslAbilityFactory`.

```bash
$ nest g module casl
$ nest g class casl/casl-ability.factory
```

Теперь определим в `CaslAbilityFactory` метод `createForUser()`. Он будет создавать объект `Ability` для заданного пользователя:

```typescript
type Subjects = InferSubjects<typeof Article | typeof User> | 'all';

export type AppAbility = MongoAbility<[Action, Subjects]>;

@Injectable()
export class CaslAbilityFactory {
  createForUser(user: User) {
    const { can, cannot, build } = new AbilityBuilder(createMongoAbility);

    if (user.isAdmin) {
      can(Action.Manage, 'all'); // доступ ко всему на чтение и запись
    } else {
      can(Action.Read, 'all'); // доступ ко всему только на чтение
    }

    can(Action.Update, Article, { authorId: user.id });
    cannot(Action.Delete, Article, { isPublished: true });

    return build({
      // Подробности: https://casl.js.org/v6/en/guide/subject-type-detection#use-classes-as-subject-types
      detectSubjectType: (item) =>
        item.constructor as ExtractSubjectType<Subjects>,
    });
  }
}
```

> **Важно** `all` — специальное ключевое слово CASL, означающее «любой субъект».

> **Подсказка** Начиная с CASL v6 классом ability по умолчанию является `MongoAbility`: он заменил устаревший `Ability` и лучше поддерживает разрешения на основе условий в синтаксисе, похожем на MongoDB. Несмотря на название, он не привязан к MongoDB — он работает с любыми данными, просто сопоставляя объекты с условиями, записанными в Mongo-подобном синтаксисе.

> **Подсказка** Классы `MongoAbility`, `AbilityBuilder`, `AbilityClass` и `ExtractSubjectType` экспортируются из пакета `@casl/ability`.

> **Подсказка** Опция `detectSubjectType` подсказывает CASL, как определить тип субъекта по объекту. Подробнее — в [документации CASL](https://casl.js.org/v6/en/guide/subject-type-detection#use-classes-as-subject-types).

В примере выше мы создали экземпляр `MongoAbility` с помощью класса `AbilityBuilder`. Как вы, вероятно, догадались, `can` и `cannot` принимают одинаковые аргументы, но имеют противоположный смысл: `can` разрешает выполнять действие над указанным субъектом, а `cannot` запрещает. Обе функции принимают до четырёх аргументов. Подробнее о них — в официальной [документации CASL](https://casl.js.org/v6/en/guide/intro).

Наконец, не забудьте добавить `CaslAbilityFactory` в массивы `providers` и `exports` в определении модуля `CaslModule`:

```typescript
import { Module } from '@nestjs/common';
import { CaslAbilityFactory } from './casl-ability.factory';

@Module({
  providers: [CaslAbilityFactory],
  exports: [CaslAbilityFactory],
})
export class CaslModule {}
```

После этого `CaslAbilityFactory` можно внедрить в любой класс обычным способом — через конструктор, при условии что `CaslModule` импортирован в соответствующем контексте:

```typescript
constructor(private caslAbilityFactory: CaslAbilityFactory) {}
```

Затем используйте её в классе следующим образом.

```typescript
const ability = this.caslAbilityFactory.createForUser(user);
if (ability.can(Action.Read, 'all')) {
  // «user» имеет доступ ко всему на чтение
}
```

> **Подсказка** Подробнее о классе `MongoAbility` читайте в официальной [документации CASL](https://casl.js.org/v6/en/guide/intro).

Допустим, у нас есть пользователь, не являющийся администратором. Он должен иметь возможность читать статьи, но создавать новые или удалять существующие ему запрещено.

```typescript
const user = new User();
user.isAdmin = false;

const ability = this.caslAbilityFactory.createForUser(user);
ability.can(Action.Read, Article); // true
ability.can(Action.Delete, Article); // false
ability.can(Action.Create, Article); // false
```

> **Подсказка** Хотя методы `can` и `cannot` есть и у `MongoAbility`, и у `AbilityBuilder`, назначение у них разное, а аргументы немного отличаются.

Кроме того, согласно нашим требованиям, пользователь должен иметь возможность изменять свои статьи:

```typescript
const user = new User();
user.id = 1;

const article = new Article();
article.authorId = user.id;

const ability = this.caslAbilityFactory.createForUser(user);
ability.can(Action.Update, article); // true

article.authorId = 2;
ability.can(Action.Update, article); // false
```

Как видите, экземпляр `MongoAbility` позволяет проверять разрешения довольно наглядно. Точно так же `AbilityBuilder` даёт возможность описывать разрешения (и задавать различные условия). Больше примеров вы найдёте в официальной документации.

#### Продвинутый уровень: реализация `PoliciesGuard`

В этом разделе мы покажем, как построить несколько более сложный guard, проверяющий, соответствует ли пользователь определённым **политикам авторизации**, которые настраиваются на уровне метода (при желании его можно расширить и на политики уровня класса). В примере мы используем пакет CASL исключительно для наглядности — применять именно эту библиотеку не обязательно. Также нам пригодится провайдер `CaslAbilityFactory`, созданный в предыдущем разделе.

Сначала сформулируем требования. Цель — предоставить механизм, позволяющий задавать проверки политик для каждого обработчика маршрута. Мы поддержим и объекты, и функции (для более простых проверок и для тех, кому ближе функциональный стиль).

Начнём с определения интерфейсов для обработчиков политик:

```typescript
import { AppAbility } from '../casl/casl-ability.factory';

interface IPolicyHandler {
  handle(ability: AppAbility): boolean;
}

type PolicyHandlerCallback = (ability: AppAbility) => boolean;

export type PolicyHandler = IPolicyHandler | PolicyHandlerCallback;
```

Как сказано выше, мы предусмотрели два способа задать обработчик политики: объект (экземпляр класса, реализующего интерфейс `IPolicyHandler`) и функция (соответствующая типу `PolicyHandlerCallback`).

Теперь создадим декоратор `@CheckPolicies()`. Он позволит указывать, какие политики должны выполняться для доступа к конкретным ресурсам.

```typescript
export const CHECK_POLICIES_KEY = 'check_policy';
export const CheckPolicies = (...handlers: PolicyHandler[]) =>
  SetMetadata(CHECK_POLICIES_KEY, handlers);
```

Далее создадим `PoliciesGuard`, который извлечёт и выполнит все обработчики политик, привязанные к обработчику маршрута.

```typescript
@Injectable()
export class PoliciesGuard implements CanActivate {
  constructor(
    private reflector: Reflector,
    private caslAbilityFactory: CaslAbilityFactory,
  ) {}

  async canActivate(context: ExecutionContext): Promise<boolean> {
    const policyHandlers =
      this.reflector.get<PolicyHandler[]>(
        CHECK_POLICIES_KEY,
        context.getHandler(),
      ) || [];

    const { user } = context.switchToHttp().getRequest();
    const ability = this.caslAbilityFactory.createForUser(user);

    return policyHandlers.every((handler) =>
      this.execPolicyHandler(handler, ability),
    );
  }

  private execPolicyHandler(handler: PolicyHandler, ability: AppAbility) {
    if (typeof handler === 'function') {
      return handler(ability);
    }
    return handler.handle(ability);
  }
}
```

> **Подсказка** В этом примере мы исходили из того, что `request.user` содержит объект пользователя. В вашем приложении такая связь, скорее всего, устанавливается в собственном **guard-е аутентификации** — подробности см. в главе об [аутентификации](<01. Authentication.md>).

Разберём пример по шагам. `policyHandlers` — это массив обработчиков, назначенных методу через декоратор `@CheckPolicies()`. Затем мы вызываем метод `CaslAbilityFactory#create`, который создаёт объект `Ability`, позволяющий проверить, достаточно ли у пользователя прав для выполнения определённых действий. Этот объект передаётся обработчику политики — функции либо экземпляру класса, реализующего `IPolicyHandler` с методом `handle()`, возвращающим логическое значение. Наконец, с помощью `Array#every` мы убеждаемся, что каждый обработчик вернул `true`.

Наконец, чтобы проверить этот guard, привяжите его к любому обработчику маршрута и зарегистрируйте встроенный обработчик политики (функциональный подход):

```typescript
@Get()
@UseGuards(PoliciesGuard)
@CheckPolicies((ability: AppAbility) => ability.can(Action.Read, Article))
findAll() {
  return this.articlesService.findAll();
}
```

Как вариант, можно определить класс, реализующий интерфейс `IPolicyHandler`:

```typescript
export class ReadArticlePolicyHandler implements IPolicyHandler {
  handle(ability: AppAbility) {
    return ability.can(Action.Read, Article);
  }
}
```

И использовать его так:

```typescript
@Get()
@UseGuards(PoliciesGuard)
@CheckPolicies(new ReadArticlePolicyHandler())
findAll() {
  return this.articlesService.findAll();
}
```

> **Важно** Поскольку обработчик политики приходится создавать прямо на месте с помощью `new`, класс `ReadArticlePolicyHandler` не может использовать внедрение зависимостей. Это решается методом `ModuleRef#get` (подробнее [здесь](<../03. Fundamentals/06. Module reference.md>)). По сути, вместо регистрации функций и экземпляров через декоратор `@CheckPolicies()` нужно разрешить передачу `Type<IPolicyHandler>`. Тогда внутри guard-а вы сможете получить экземпляр по ссылке на тип: `moduleRef.get(YOUR_HANDLER_TYPE)` — или даже динамически создать его методом `ModuleRef#create`.
