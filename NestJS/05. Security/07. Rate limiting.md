### Ограничение частоты запросов

Распространённый приём защиты приложений от атак перебором — **ограничение частоты запросов** (rate limiting). Для начала вам понадобится установить пакет `@nestjs/throttler`.

```bash
$ npm i --save @nestjs/throttler
```

После установки `ThrottlerModule` настраивается так же, как любой другой пакет Nest — методами `forRoot` или `forRootAsync`.

**`app.module.ts`**

```typescript
@Module({
  imports: [
     ThrottlerModule.forRoot({
      throttlers: [
        {
          ttl: 60000,
          limit: 10,
        },
      ],
    }),
  ],
})
export class AppModule {}
```

Приведённый выше код задаёт глобальные параметры `ttl` (время жизни в миллисекундах) и `limit` (максимальное число запросов в пределах ttl) для тех маршрутов вашего приложения, которые защищены guard'ом.

После импорта модуля вы можете выбрать способ привязки `ThrottlerGuard`. Подойдёт любой из вариантов привязки, описанных в разделе [guards](<../02. Overview/08. Guards.md>). Например, чтобы привязать guard глобально, добавьте в любой модуль такой провайдер:

```typescript
{
  provide: APP_GUARD,
  useClass: ThrottlerGuard
}
```

#### Несколько наборов ограничений

Иногда требуется задать сразу несколько наборов ограничений — например, не более 3 вызовов в секунду, 20 вызовов за 10 секунд и 100 вызовов в минуту. Для этого опишите их в массиве с именованными опциями, на которые затем можно ссылаться в декораторах `@SkipThrottle()` и `@Throttle()`, чтобы снова изменить параметры.

**`app.module.ts`**

```typescript
@Module({
  imports: [
    ThrottlerModule.forRoot([
      {
        name: 'short',
        ttl: 1000,
        limit: 3,
      },
      {
        name: 'medium',
        ttl: 10000,
        limit: 20
      },
      {
        name: 'long',
        ttl: 60000,
        limit: 100
      }
    ]),
  ],
})
export class AppModule {}
```

#### Настройка

Может возникнуть ситуация, когда вы хотите привязать guard к контроллеру или глобально, но отключить ограничение частоты для одного или нескольких эндпоинтов. Для этого используйте декоратор `@SkipThrottle()`, отменяющий работу throttler'а для целого класса или отдельного маршрута. Декоратор `@SkipThrottle()` также может принимать объект со строковыми ключами и булевыми значениями — на случай, если нужно исключить _большинство_ маршрутов контроллера, но не все, а также настроить поведение отдельно для каждого набора ограничений, если их несколько. Если объект не передан, по умолчанию применяется `{ default: true }`

```typescript
@SkipThrottle()
@Controller('users')
export class UsersController {}
```

Декоратор `@SkipThrottle()` можно использовать как для пропуска маршрута или класса, так и для отмены пропуска для отдельного маршрута внутри пропускаемого класса.

```typescript
@SkipThrottle()
@Controller('users')
export class UsersController {
  // К этому маршруту ограничение частоты применяется.
  @SkipThrottle({ default: false })
  dontSkip() {
    return 'List users work with Rate limiting.';
  }
  // Этот маршрут пропустит ограничение частоты.
  doSkip() {
    return 'List users work without Rate limiting.';
  }
}
```

Существует также декоратор `@Throttle()`, позволяющий переопределить `limit` и `ttl`, заданные в глобальном модуле, и сделать ограничения строже или мягче. Его тоже можно применять как к классу, так и к методу. Начиная с версии 5 декоратор принимает объект, ключи которого — имена наборов ограничений, а значения — объекты с ключами limit и ttl и целочисленными значениями, аналогично опциям, передаваемым в корневой модуль. Если в исходных опциях имя не задано, используйте строку `default`. Настраивать нужно так:

```typescript
// Переопределяем стандартную конфигурацию ограничения частоты и его длительности.
@Throttle({ default: { limit: 3, ttl: 60000 } })
@Get()
findAll() {
  return "List users works with custom rate limiting.";
}
```

#### Прокси

Если ваше приложение работает за прокси-сервером, важно настроить HTTP-адаптер так, чтобы он доверял прокси. Чтобы включить настройку `trust proxy`, обратитесь к описанию соответствующих опций HTTP-адаптера для [Express](http://expressjs.com/en/guide/behind-proxies.html) и [Fastify](https://www.fastify.io/docs/latest/Reference/Server/#trustproxy).

Вот пример включения `trust proxy` для адаптера Express:

**`main.ts`**

```typescript
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';
import { NestExpressApplication } from '@nestjs/platform-express';

async function bootstrap() {
  const app = await NestFactory.create<NestExpressApplication>(AppModule);
  app.set('trust proxy', 'loopback'); // Доверять запросам с адреса обратной петли
  await app.listen(3000);
}

bootstrap();
```

Включение `trust proxy` позволяет получать исходный IP-адрес из заголовка `X-Forwarded-For`. Вы также можете изменить поведение приложения, переопределив метод `getTracker()` так, чтобы он извлекал IP-адрес из этого заголовка вместо `req.ip`. Следующий пример показывает, как это сделать для Express и Fastify:

**`throttler-behind-proxy.guard.ts`**

```typescript
import { ThrottlerGuard } from '@nestjs/throttler';
import { Injectable } from '@nestjs/common';

@Injectable()
export class ThrottlerBehindProxyGuard extends ThrottlerGuard {
  protected async getTracker(req: Record<string, any>): Promise<string> {
    return req.ips.length ? req.ips[0] : req.ip; // подстройте извлечение IP под свои задачи
  }
}
```

> **Подсказка** API объекта запроса `req` для express описан [здесь](https://expressjs.com/en/api.html#req.ips), а для fastify — [здесь](https://www.fastify.io/docs/latest/Reference/Request/).

#### Websockets

Этот модуль может работать с websockets, но требует расширения класса. Вы можете унаследоваться от `ThrottlerGuard` и переопределить метод `handleRequest` следующим образом:

```typescript
@Injectable()
export class WsThrottlerGuard extends ThrottlerGuard {
  async handleRequest(requestProps: ThrottlerRequest): Promise<boolean> {
    const {
      context,
      limit,
      ttl,
      throttler,
      blockDuration,
      getTracker,
      generateKey,
    } = requestProps;

    const client = context.switchToWs().getClient();
    const tracker = client._socket.remoteAddress;
    const key = generateKey(context, tracker, throttler.name);
    const { totalHits, timeToExpire, isBlocked, timeToBlockExpire } =
      await this.storageService.increment(
        key,
        ttl,
        limit,
        blockDuration,
        throttler.name,
      );

    const getThrottlerSuffix = (name: string) =>
      name === 'default' ? '' : `-${name}`;

    // Выбрасываем ошибку, когда пользователь достиг своего лимита.
    if (isBlocked) {
      await this.throwThrottlingException(context, {
        limit,
        ttl,
        key,
        tracker,
        totalHits,
        timeToExpire,
        isBlocked,
        timeToBlockExpire,
      });
    }

    return true;
  }
}
```

> **Подсказка** Если вы используете ws, `_socket` необходимо заменить на `conn`

При работе с WebSockets нужно помнить о нескольких моментах:

- Guard нельзя зарегистрировать через `APP_GUARD` или `app.useGlobalGuards()`
- При достижении лимита Nest сгенерирует событие `exception`, поэтому убедитесь, что для него есть слушатель

> **Подсказка** Если вы используете пакет `@nestjs/platform-ws`, можно применять `client._socket.remoteAddress`.

> **Подсказка** Когда вы настраиваете [несколько наборов ограничений](<07. Rate limiting.md#несколько-наборов-ограничений>), метод `handleRequest()` выполняется по одному разу для каждого набора. Используйте `throttler.name` из `ThrottlerRequest` при формировании ключа хранилища и при передаче `ThrottlerLimitDetail`, как показано выше, чтобы каждый именованный throttler отслеживал собственный лимит.

#### GraphQL

`ThrottlerGuard` можно использовать и для работы с GraphQL-запросами. Здесь guard также расширяется, но на этот раз переопределяется метод `getRequestResponse`

```typescript
@Injectable()
export class GqlThrottlerGuard extends ThrottlerGuard {
  getRequestResponse(context: ExecutionContext) {
    const gqlCtx = GqlExecutionContext.create(context);
    const ctx = gqlCtx.getContext();
    return { req: ctx.req, res: ctx.res };
  }
}
```

#### Асинхронная конфигурация

Возможно, вы захотите получать конфигурацию ограничения частоты асинхронно, а не синхронно. Для этого есть метод `forRootAsync()`, поддерживающий внедрение зависимостей и `async`-методы.

Один из подходов — использовать фабричную функцию:

```typescript
@Module({
  imports: [
    ThrottlerModule.forRootAsync({
      imports: [ConfigModule],
      inject: [ConfigService],
      useFactory: (config: ConfigService) => [
        {
          ttl: config.get('THROTTLE_TTL'),
          limit: config.get('THROTTLE_LIMIT'),
        },
      ],
    }),
  ],
})
export class AppModule {}
```

Также можно применить синтаксис `useClass`:

```typescript
@Module({
  imports: [
    ThrottlerModule.forRootAsync({
      imports: [ConfigModule],
      useClass: ThrottlerConfigService,
    }),
  ],
})
export class AppModule {}
```

Это возможно при условии, что `ThrottlerConfigService` реализует интерфейс `ThrottlerOptionsFactory`.

#### Хранилища

Встроенное хранилище — это кэш в памяти, который отслеживает выполненные запросы, пока не истечёт TTL, заданный глобальными опциями. Вы можете подставить собственное хранилище в опцию `storage` модуля `ThrottlerModule`, если его класс реализует интерфейс `ThrottlerStorage`.

Для распределённых серверов можно использовать хранилище от сообщества на базе [Redis](https://github.com/jmcdo29/nest-lab/tree/main/packages/throttler-storage-redis), чтобы иметь единый источник истины.

> **Примечание** `ThrottlerStorage` можно импортировать из `@nestjs/throttler`.

#### Помощники для работы со временем

Есть несколько вспомогательных методов, делающих указание интервалов более читаемым, если вам это удобнее прямого указания чисел. `@nestjs/throttler` экспортирует пять помощников: `seconds`, `minutes`, `hours`, `days` и `weeks`. Чтобы воспользоваться ими, просто вызовите `seconds(5)` или любой другой из них — и получите нужное количество миллисекунд.

#### Руководство по миграции

Большинству будет достаточно обернуть свои опции в массив.

Если вы используете пользовательское хранилище, оберните `ttl` и `limit` в массив и присвойте его свойству `throttlers` объекта опций.

Любой декоратор `@SkipThrottle()` может использоваться для обхода ограничения частоты на конкретных маршрутах или методах. Он принимает необязательный булев параметр, по умолчанию равный `true`. Это удобно, когда нужно отключить ограничение частоты для отдельных эндпоинтов.

Все декораторы `@Throttle()` теперь также должны принимать объект со строковыми ключами, соответствующими именам контекстов throttler'а (снова `'default'`, если имя не задано), и значениями-объектами с ключами `limit` и `ttl`.

> **Важно** Значение `ttl` теперь задаётся в **миллисекундах**. Если для читаемости вы хотите
> оставить ttl в секундах, используйте помощник `seconds` из этого пакета. Он просто
> умножает ttl на 1000, переводя его в миллисекунды.

Подробности см. в [Changelog](https://github.com/nestjs/throttler/blob/master/CHANGELOG.md#500)
