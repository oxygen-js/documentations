# Изучение каталога .claude

> Откуда Claude Code читает CLAUDE.md, settings.json, хуки, навыки, команды, субагентов, рабочие процессы, правила и автопамять. Изучите каталог .claude в вашем проекте и ~/.claude в домашнем каталоге.

Claude Code читает инструкции, настройки, навыки, субагентов и память из каталога вашего проекта и из `~/.claude` в вашем домашнем каталоге. Коммитьте файлы проекта в git, чтобы делиться ими с командой; файлы в `~/.claude` — это личная конфигурация, которая применяется ко всем вашим проектам.

В Windows `~/.claude` разрешается в `%USERPROFILE%\.claude`. Если вы задали [`CLAUDE_CONFIG_DIR`](https://code.claude.com/docs/en/env-vars), каждый путь `~/.claude` на этой странице находится внутри указанного каталога.

Большинство пользователей редактируют только `CLAUDE.md` и `settings.json`. Остальная часть каталога необязательна: добавляйте навыки, правила или субагентов по мере необходимости.

## Изучение каталога

Ниже описан каждый файл: что он делает, когда загружается и пример его содержимого. Файлы сгруппированы по двум областям — уровень проекта и глобальный уровень (`~/`).

**Метки файлов:** `committed` — файл коммитится в git; `gitignored` — добавляется в `.gitignore`; `local` — только локально, не коммитится; «Claude пишет» — файл создаёт и ведёт сам Claude.

### Уровень проекта: `your-project/`

#### `CLAUDE.md`

*Инструкции проекта, которые Claude читает в каждой сессии.* (метка: `committed`)

**Когда загружается.** Загружается в контекст в начале каждой сессии.

Специфичные для проекта инструкции, определяющие, как Claude работает в этом репозитории. Разместите здесь свои соглашения, часто используемые команды и архитектурный контекст, чтобы Claude работал с теми же предположениями, что и ваша команда.

**Советы:**

- Ориентируйтесь на объём менее 200 строк. Более длинные файлы всё равно загружаются целиком, но могут снижать точность следования инструкциям
- CLAUDE.md загружается в каждую сессию. Если что-то важно только для конкретных задач, перенесите это в [навык](https://code.claude.com/docs/en/skills) или в [правило](https://code.claude.com/docs/en/memory#organize-rules-with-claude/rules/) с ограничением по путям, чтобы оно загружалось только при необходимости
- Перечислите команды, которые вы запускаете чаще всего, — сборку, тесты, форматирование, — чтобы Claude знал их, не заставляя вас объяснять каждый раз
- Выполните `/memory`, чтобы открыть и отредактировать CLAUDE.md прямо из сессии
- Также работает в `.claude/CLAUDE.md`, если вы предпочитаете не засорять корень проекта

Этот пример — для проекта на TypeScript и React. В нём перечислены команды сборки и тестирования, соглашения фреймворка, которым должен следовать Claude, и специфичные для проекта правила, такие как стиль экспортов и раскладка файлов.

```md
# Project conventions

## Commands
- Build: `npm run build`
- Test: `npm test`
- Lint: `npm run lint`

## Stack
- TypeScript with strict mode
- React 19, functional components only

## Rules
- Named exports, never default exports
- Tests live next to source: `foo.ts` -> `foo.test.ts`
- All API routes return `{ data, error }` shape
```

#### `.mcp.json`

*MCP-серверы уровня проекта, общие для вашей команды.* (метка: `committed`)

**Когда загружается.** Серверы подключаются при начале сессии. Схемы инструментов по умолчанию отложены и загружаются по требованию через [поиск инструментов](https://code.claude.com/docs/en/mcp#scale-with-mcp-tool-search).

Настраивает серверы Model Context Protocol (MCP), которые дают Claude доступ к внешним инструментам: базам данных, API, браузерам и другим. Этот файл содержит серверы уровня проекта, которые использует вся ваша команда. Личные серверы, которые вы хотите оставить только себе, размещаются в `~/.claude.json`.

**Советы:**

- Используйте ссылки на переменные окружения для секретов: `${GITHUB_TOKEN}`
- Находится в корне проекта, а не внутри `.claude/`
- Для серверов, нужных только вам, выполните `claude mcp add --scope user`. Это запишет их в `~/.claude.json`, а не в `.mcp.json`

Этот пример настраивает MCP-сервер GitHub, чтобы Claude мог читать issue и открывать pull request'ы. Ссылка `${GITHUB_TOKEN}` читается из окружения вашей оболочки, когда Claude Code запускает сервер, поэтому токен никогда не попадает в файл.

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}
```

#### `.worktreeinclude`

*Игнорируемые git файлы для копирования в новые worktree.* (метка: `committed`)

**Когда загружается.** Читается, когда Claude создаёт git-worktree через `--worktree`, инструмент `EnterWorktree` или субагента с `isolation: worktree`.

Перечисляет игнорируемые git файлы, которые нужно скопировать из вашего основного репозитория в каждый новый worktree. Worktree — это свежие выгрузки, поэтому неотслеживаемые файлы вроде `.env` в них по умолчанию отсутствуют. Шаблоны здесь используют синтаксис `.gitignore`. Копируются только файлы, которые соответствуют шаблону и при этом игнорируются git, поэтому отслеживаемые файлы никогда не дублируются.

**Советы:**

- Находится в корне проекта, а не внутри `.claude/`
- Только для git: если вы настроите [хук WorktreeCreate](https://code.claude.com/docs/en/hooks#worktreecreate) для другой системы контроля версий, этот файл не читается. Копируйте файлы внутри скрипта своего хука
- Также применяется к параллельным сессиям в [десктопном приложении](https://code.claude.com/docs/en/desktop#work-in-parallel-with-sessions)

Этот пример копирует ваши локальные файлы окружения и конфигурацию с секретами в каждый worktree, создаваемый Claude. Комментарии начинаются с # , а пустые строки игнорируются — так же, как в .gitignore.

```text
# Local environment
.env
.env.local

# API credentials
config/secrets.json
```

#### `.claude/`

*Конфигурация, правила и расширения уровня проекта.*

Всё, что Claude Code читает специфично для этого проекта. Если вы используете git, коммитьте большинство файлов здесь, чтобы ваша команда пользовалась ими совместно; несколько файлов, например settings.local.json, автоматически попадают в .gitignore. Метка каждого файла показывает, какой именно.

#### `.claude/settings.json`

*Разрешения, хуки и конфигурация.* (метка: `committed`)

**Когда загружается.** Переопределяет глобальный `~/.claude/settings.json`. Локальные настройки, флаги CLI и управляемые настройки переопределяют этот файл.

Настройки, которые Claude Code применяет напрямую. Разрешения управляют тем, какие команды и инструменты может использовать Claude; хуки запускают ваши скрипты в определённые моменты сессии. В отличие от CLAUDE.md, который Claude читает как рекомендации, эти настройки принудительно применяются независимо от того, следует им Claude или нет.

**Общие ключи:**

- [permissions](https://code.claude.com/docs/en/permissions): разрешать, запрещать или запрашивать подтверждение перед тем, как Claude использует определённые инструменты или команды
- [hooks](https://code.claude.com/docs/en/hooks): запускать ваши собственные скрипты на событиях вроде «перед вызовом инструмента» или «после редактирования файла»
- [statusLine](https://code.claude.com/docs/en/statusline): настроить строку, отображаемую внизу, пока Claude работает
- [model](https://code.claude.com/docs/en/settings#available-settings): выбрать модель по умолчанию для этого проекта
- [env](https://code.claude.com/docs/en/settings#environment-variables): переменные окружения, задаваемые в каждой сессии
- [outputStyle](https://code.claude.com/docs/en/output-styles): выбрать пользовательский стиль системного промпта из output-styles/

**Советы:**

- Шаблоны разрешений для Bash поддерживают подстановочные знаки: `Bash(npm test *)` соответствует любой команде, начинающейся с `npm test`
- Настройки-массивы, такие как `permissions.allow`, объединяются по всем областям; скалярные настройки, такие как `model`, используют наиболее специфичное значение

Этот пример разрешает команды `npm test` и `npm run` без запроса подтверждения, блокирует `rm -rf` и запускает Prettier для файлов после того, как Claude их отредактирует или запишет.

```json
{
  "permissions": {
    "allow": [
      "Bash(npm test *)",
      "Bash(npm run *)"
    ],
    "deny": [
      "Bash(rm -rf *)"
    ]
  },
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
      }]
    }]
  }
}
```

#### `.claude/settings.local.json`

*Ваши личные переопределения настроек для этого проекта.* (метка: `gitignored`)

**Когда загружается.** Высший приоритет среди редактируемых пользователем файлов настроек; флаги CLI и управляемые настройки всё же имеют приоритет.

Личные настройки, которые имеют приоритет над настройками проекта по умолчанию. Тот же формат JSON, что и в settings.json, но не коммитится. Используйте это, когда вам нужны разрешения или значения по умолчанию, отличные от командной конфигурации.

**Советы:**

- Та же схема, что и у settings.json. Настройки-массивы, такие как `permissions.allow`, объединяются по областям; скалярные настройки, такие как `model`, используют локальное значение
- Claude Code добавляет этот файл в `~/.config/git/ignore` при первой записи. Если вы используете собственный `core.excludesFile`, добавьте шаблон и туда. Чтобы поделиться правилом игнорирования с командой, добавьте его также в `.gitignore` проекта

Этот пример добавляет разрешения для Docker поверх всего, что разрешает командный settings.json.

```json
{
  "permissions": {
    "allow": [
      "Bash(docker *)"
    ]
  }
}
```

[Полная документация →](https://code.claude.com/docs/en/settings)

#### `.claude/rules/`

*Тематические инструкции, опционально ограниченные путями к файлам.*

**Когда загружается.** Правила без `paths:` загружаются в начале сессии. Правила с `paths:` загружаются, когда соответствующий файл попадает в контекст.

Инструкции проекта, разбитые на тематические файлы, которые могут загружаться условно в зависимости от путей к файлам. Правило без frontmatter `paths:` загружается в начале сессии, как CLAUDE.md; правило с `paths:` загружается только тогда, когда Claude читает соответствующий файл.

Как и CLAUDE.md, правила — это рекомендации, которые Claude читает, а не конфигурация, которую Claude Code принудительно применяет. Для гарантированного поведения используйте [хуки](https://code.claude.com/docs/en/hooks) или [разрешения](https://code.claude.com/docs/en/permissions).

**Советы:**

- Используйте frontmatter `paths:` с glob-шаблонами, чтобы ограничить правила каталогами или типами файлов
- Подкаталоги работают: `.claude/rules/frontend/react.md` обнаруживается автоматически
- Когда CLAUDE.md приближается к 200 строкам, начинайте разбивать его на правила

[Полная документация →](https://code.claude.com/docs/en/memory#organize-rules-with-claude/rules/)

#### `.claude/rules/testing.md`

*Соглашения по тестам, ограниченные тестовыми файлами.* (метка: `committed`)

**Когда загружается.** Загружается, когда Claude читает файл, соответствующий glob-шаблонам `paths:` ниже.

Пример правила, которое загружается только тогда, когда Claude работает с тестовыми файлами. Glob-шаблоны `paths:` во frontmatter определяют, какие файлы его запускают; здесь — всё, что заканчивается на .test.ts или .test.tsx. Для других файлов это правило в контекст не загружается.

```md
---
paths:
  - "**/*.test.ts"
  - "**/*.test.tsx"
---

# Testing Rules

- Use descriptive test names: "should [expected] when [condition]"
- Mock external dependencies, not internal modules
- Clean up side effects in afterEach
```

#### `.claude/rules/api-design.md`

*Соглашения по API, ограниченные кодом бэкенда.* (метка: `committed`)

**Когда загружается.** Загружается, когда Claude читает файл, соответствующий glob-шаблону `paths:` ниже.

Второй пример, показывающий правило, ограниченное кодом бэкенда. Glob-шаблон `paths:` соответствует файлам в src/api/, поэтому эти соглашения загружаются только тогда, когда Claude редактирует маршруты API.

```md
---
paths:
  - "src/api/**/*.ts"
---

# API Design Rules

- All endpoints must validate input with Zod schemas
- Return shape: { data: T } | { error: string }
- Rate limit all public endpoints
```

#### `.claude/skills/`

*Переиспользуемые промпты, которые вы или Claude вызываете по имени.*

**Когда загружается.** Вызывается через `/skill-name` или когда Claude сопоставляет задачу с навыком.

Каждый навык — это папка с файлом SKILL.md плюс любые вспомогательные файлы, которые ему нужны. По умолчанию навык может вызывать как вы, так и Claude. Управляйте этим через frontmatter: `disable-model-invocation: true` для рабочих процессов только для пользователя, таких как `/deploy`, или `user-invocable: false`, чтобы скрыть навык из меню `/`, оставив Claude возможность его вызывать.

**Советы:**

- Навыки принимают аргументы: `/deploy staging` передаёт "staging" как `$ARGUMENTS`. Используйте `$0`, `$1` и так далее для позиционного доступа
- Frontmatter `description` определяет, когда Claude автоматически вызывает навык
- Размещайте справочные документы рядом с SKILL.md. Claude знает путь к каталогу навыка и может читать вспомогательные файлы, когда вы их упоминаете

[Полная документация →](https://code.claude.com/docs/en/skills)

#### `.claude/skills/security-review/`

*Навык, объединяющий SKILL.md со вспомогательными файлами.*

#### `.claude/skills/security-review/SKILL.md`

*Точка входа: триггер, возможность вызова, инструкции.* (метка: `committed`)

**Когда загружается.** Пользователь вводит `/security-review <target>`; Claude не может вызвать этот навык автоматически.

Этот навык использует `disable-model-invocation: true`, поэтому запустить его можете только вы; Claude никогда не вызывает его сам.

Строка `` !`...` `` выполняет команду оболочки и вставляет её вывод в промпт. `$ARGUMENTS` подставляет то, что вы ввели после имени навыка. Claude видит путь к каталогу навыка, поэтому упоминание вложенного файла вроде checklist.md позволяет Claude прочитать его.

```md
---
description: Reviews code changes for security vulnerabilities, authentication gaps, and injection risks
disable-model-invocation: true
argument-hint: <branch-or-path>
---

## Diff to review

!`git diff $ARGUMENTS`

Audit the changes above for:

1. Injection vulnerabilities (SQL, XSS, command)
2. Authentication and authorization gaps
3. Hardcoded secrets or credentials

Use checklist.md in this skill directory for the full review checklist.

Report findings with severity ratings and remediation steps.
```

#### `.claude/skills/security-review/checklist.md`

*Вспомогательный файл, поставляемый вместе с навыком.* (метка: `committed`)

**Когда загружается.** Claude читает его по требованию во время выполнения навыка.

Навыки могут включать любые вспомогательные файлы: справочные документы, шаблоны, скрипты. Путь к каталогу навыка добавляется в начало SKILL.md, поэтому Claude может читать вложенные файлы по имени. Для скриптов в командах внедрения bash используйте плейсхолдер `${CLAUDE_SKILL_DIR}`.

```md
# Security Review Checklist

## Input Validation
- [ ] All user input sanitized before DB queries
- [ ] File upload MIME types validated
- [ ] Path traversal prevented on file operations

## Authentication
- [ ] JWT tokens expire after 24 hours
- [ ] API keys stored in environment variables
- [ ] Passwords hashed with bcrypt or argon2
```

#### `.claude/commands/`

*Промпты в одном файле, вызываемые через `/name`.*

> **Примечание.** Команды и навыки теперь являются одним и тем же механизмом. Для новых рабочих процессов используйте [skills/](https://code.claude.com/docs/en/skills): тот же вызов `/name`, плюс вы можете включать вспомогательные файлы.

**Когда загружается.** Пользователь вводит `/command-name`.

Файл `commands/deploy.md` создаёт `/deploy` так же, как это делает навык `skills/deploy/SKILL.md`, и оба могут вызываться Claude автоматически. Навыки используют каталог с SKILL.md, что позволяет включать справочные документы, шаблоны или скрипты вместе с промптом.

**Советы:**

- Используйте `$ARGUMENTS` в файле, чтобы принимать параметры: `/fix-issue 123`
- Если навык и команда имеют одинаковое имя, приоритет у навыка
- Новые команды обычно лучше делать навыками; команды по-прежнему поддерживаются

[Полная документация →](https://code.claude.com/docs/en/skills)

#### `.claude/commands/fix-issue.md`

*Вызывается как `/fix-issue <number>`.* (метка: `committed`)

> **Примечание.** Команды и навыки теперь являются одним и тем же механизмом. Для новых рабочих процессов используйте [skills/](https://code.claude.com/docs/en/skills): тот же вызов `/name`, плюс вы можете включать вспомогательные файлы.

Пример команды для исправления issue на GitHub. Введите `/fix-issue 123`, и строка `` !`...` `` выполнит `gh issue view 123` в вашей оболочке, вставив вывод в промпт перед тем, как его увидит Claude.

`$ARGUMENTS` подставляет то, что вы ввели после имени команды. Для позиционного доступа используйте `$0`, `$1` и так далее.

```md
---
argument-hint: <issue-number>
---

!`gh issue view $ARGUMENTS`

Investigate and fix the issue above.

1. Trace the bug to its root cause
2. Implement the fix
3. Write or update tests
4. Summarize what you changed and why
```

#### `.claude/output-styles/`

*Стили вывода уровня проекта, если ваша команда ими делится.*

**Когда загружается.** Применяется в начале сессии, когда выбран через настройку outputStyle.

Стили вывода обычно личные, поэтому большинство из них находятся в `~/.claude/output-styles/`. Разместите стиль здесь, если ваша команда им делится, например режим ревью, которым пользуются все. Полное объяснение и пример см. в разделе `output-styles/` блока «Глобальный уровень» ниже.

[Полная документация →](https://code.claude.com/docs/en/sub-agents#enable-persistent-memory)

## Что не показано

Обозреватель охватывает файлы, которые вы создаёте и редактируете. Несколько связанных файлов находятся в других местах:

| Файл | Расположение | Назначение |
| --- | --- | --- |
| `managed-settings.json` | Системный уровень, зависит от ОС | Принудительные корпоративные настройки, которые вы не можете переопределить. См. [настройки, управляемые сервером](https://code.claude.com/docs/en/server-managed-settings). |
| `CLAUDE.local.md` | Корень проекта | Ваши личные предпочтения для этого проекта, загружаются вместе с CLAUDE.md. Создайте его вручную и добавьте в `.gitignore`. |
| Установленные плагины | `~/.claude/plugins` | Клонированные маркетплейсы, установленные версии плагинов и данные каждого плагина, управляемые командами `claude plugin`. Осиротевшие версии удаляются через 7 дней после обновления или удаления плагина. См. [кэширование плагинов](https://code.claude.com/docs/en/plugins-reference#plugin-caching-and-file-resolution). |

В `~/.claude` также хранятся данные, которые Claude Code записывает во время работы: транскрипты, история промптов, снимки файлов, кэши и логи. См. ниже раздел «Данные приложения».

## Справочник по файлам

В этой таблице перечислены все файлы, которые охватывает обозреватель. Файлы уровня проекта находятся в вашем репозитории в каталоге `.claude/` (или в корне для `CLAUDE.md`, `.mcp.json` и `.worktreeinclude`). Файлы глобального уровня находятся в `~/.claude/` и применяются ко всем проектам.

> **Примечание.** Несколько вещей могут переопределить то, что вы указали в этих файлах:
>
> * [Управляемые настройки](https://code.claude.com/docs/en/server-managed-settings), развёрнутые вашей организацией, имеют приоритет над всем остальным
> * Флаги CLI, такие как `--permission-mode` или `--settings`, переопределяют `settings.json` для этой сессии
> * Некоторые переменные окружения имеют приоритет над соответствующей настройкой, но это зависит от конкретной переменной: см. [справочник по переменным окружения](https://code.claude.com/docs/en/env-vars) по каждой
>
> Полный порядок см. в разделе [приоритет настроек](https://code.claude.com/docs/en/settings#settings-precedence).

## Данные приложения

Помимо конфигурации, которую создаёте вы, в `~/.claude` хранятся данные, которые Claude Code записывает во время сессий. Эти файлы — в открытом виде (plaintext). Всё, что проходит через инструмент, попадает в транскрипт на диске: содержимое файлов, вывод команд, вставленный текст.

### Очищается автоматически

Файлы по путям ниже удаляются при запуске, как только они старше [`cleanupPeriodDays`](https://code.claude.com/docs/en/settings#available-settings). По умолчанию — 30 дней.

| Путь внутри `~/.claude/` | Содержимое |
| --- | --- |
| `projects/<project>/<session>.jsonl` | Полный транскрипт разговора: каждое сообщение, вызов инструмента и результат инструмента |
| `projects/<project>/<session>/subagents/` | Транскрипты разговоров [субагентов](https://code.claude.com/docs/en/sub-agents), удаляются вместе с транскриптом родительской сессии, когда тот устаревает |
| `projects/<project>/<session>/tool-results/` | Большие выводы инструментов, вынесенные в отдельные файлы |
| `file-history/<session>/` | Снимки файлов до редактирования, которые изменил Claude; используются для [восстановления из контрольных точек](https://code.claude.com/docs/en/checkpointing) |
| `plans/` | Файлы планов, записанные в [режиме планирования](https://code.claude.com/docs/en/permission-modes#analyze-before-you-edit-with-plan-mode) |
| `debug/` | Отладочные логи по сессиям, записываются только при запуске с `--debug` или командой `/debug` |
| `paste-cache/`, `image-cache/` | Содержимое больших вставок и прикреплённых изображений |
| `session-env/` | Метаданные окружения по сессиям |
| `tasks/` | Списки задач по сессиям, записанные инструментами задач |
| `shell-snapshots/` | Захваченное окружение оболочки, используемое инструментом Bash. Удаляется при корректном выходе. Очистка убирает то, что осталось после сбоя. |
| `backups/` | Копии `~/.claude.json` с отметкой времени, сделанные перед миграциями конфигурации |
| `feedback-bundles/` | Отредактированные (с удалением конфиденциальных данных) архивы транскриптов, записываемые командой `/feedback` у сторонних провайдеров, для отправки вашей команде по работе с клиентами в Anthropic |
| `todos/`, `statsig/`, `logs/` | Устаревшие каталоги из старых версий. Больше не записываются. Очистка удаляет их содержимое, а затем и пустой каталог. |

### Хранится, пока вы не удалите

Следующие пути не охватываются автоматической очисткой и сохраняются бессрочно.

| Путь внутри `~/.claude/` | Содержимое |
| --- | --- |
| `history.jsonl` | Каждый введённый вами промпт с отметкой времени и путём проекта. Используется для повтора по стрелке вверх. |
| `stats-cache.json` | Агрегированные счётчики токенов и стоимости, показываемые командой `/usage` |
| `remote-settings.json` | Кэшированная копия [настроек, управляемых сервером](https://code.claude.com/docs/en/server-managed-settings) для вашей организации. Присутствует только если организация их настроила. Обновляется при каждом запуске. |

Другие небольшие файлы кэша и блокировок появляются в зависимости от используемых функций, и их можно безопасно удалять.

### Хранение в открытом виде

Транскрипты и история не шифруются на диске. Единственная защита — права доступа к файлам ОС. Если инструмент читает файл `.env` или команда выводит учётные данные, это значение записывается в `projects/<project>/<session>.jsonl`. Чтобы снизить риск раскрытия:

* Уменьшите `cleanupPeriodDays`, чтобы сократить срок хранения транскриптов
* Задайте переменную окружения [`CLAUDE_CODE_SKIP_PROMPT_HISTORY`](https://code.claude.com/docs/en/env-vars), чтобы не записывать транскрипты и историю промптов ни в каком режиме. В неинтерактивном режиме можно вместо этого передать `--no-session-persistence` вместе с `-p` или задать `persistSession: false` в Agent SDK.
* Используйте [правила разрешений](https://code.claude.com/docs/en/permissions), чтобы запретить чтение файлов с учётными данными

### Очистка локальных данных

Выполните `claude project purge`, чтобы удалить состояние, которое Claude Code хранит для одного проекта. Команда требует Claude Code v2.1.124 или новее. Она удаляет:

* Транскрипты и автопамять в `projects/`
* Записи `tasks/`, `debug/` и `file-history/` по сессиям
* Соответствующие строки промптов в `history.jsonl`
* Запись проекта в `~/.claude.json`

Команда печатает полный план удаления и запрашивает подтверждение, прежде чем что-либо удалять.

Предпросмотр плана без удаления чего-либо:

```bash
claude project purge ~/work/my-repo --dry-run
```

Удаление с единственным запросом подтверждения:

```bash
claude project purge ~/work/my-repo
```

Опустите путь, чтобы выбрать проект из интерактивного списка.

Пропустите запрос подтверждения для использования в скриптах:

```bash
claude project purge ~/work/my-repo --yes
```

Передайте `--all` вместо пути, чтобы очистить состояние сразу для всех проектов; при этом `history.jsonl` удаляется целиком, а не фильтруется. Передайте `-i`, чтобы пройти по плану удаления по одному пункту за раз.

Команда не трогает `shell-snapshots/` и `backups/`, поскольку они не привязаны к проекту, и предупреждает о них в выводе плана. Она завершается со статусом 1, если по указанному пути нет совпадающего состояния.

Вы также можете удалить любой из перечисленных выше путей данных приложения вручную. На новые сессии это не влияет. В таблице ниже показано, что вы теряете для прошлых сессий.

| Удалить | Вы теряете |
| --- | --- |
| `~/.claude/projects/` | Возобновление, продолжение и откат прошлых сессий |
| `~/.claude/history.jsonl` | Повтор промптов по стрелке вверх |
| `~/.claude/file-history/` | Восстановление из контрольных точек для прошлых сессий |
| `~/.claude/stats-cache.json` | Исторические итоги, показываемые командой `/usage` |
| `~/.claude/remote-settings.json` | Ничего. Заново загружается при следующем запуске. |
| `~/.claude/debug/`, `~/.claude/plans/`, `~/.claude/paste-cache/`, `~/.claude/image-cache/`, `~/.claude/session-env/`, `~/.claude/tasks/`, `~/.claude/shell-snapshots/`, `~/.claude/backups/` | Ничего заметного для пользователя |
| `~/.claude/todos/`, `~/.claude/statsig/`, `~/.claude/logs/` | Ничего. Устаревшие каталоги, не записываемые текущими версиями. |

Не удаляйте `~/.claude.json`, `~/.claude/settings.json` или `~/.claude/plugins/`: в них хранятся ваша аутентификация, предпочтения и установленные плагины.
