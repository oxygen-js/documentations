# Кэширование промптов

> Claude Code управляет кэшированием промптов автоматически. Узнайте, почему смена модели вызывает медленный ход без кэша, во что обходится `/compact`, почему правки CLAUDE.md не применяются посреди сессии и как проверить долю попаданий в кэш.

Кэширование промптов делает Claude Code быстрее и экономичнее. Без кэширования API обрабатывал бы всю вашу историю заново на каждом ходу. С кэшированием он повторно использует то, что уже обработал, и выполняет новую работу только для того, что изменилось.

Claude Code сам управляет кэшированием промптов, если только вы его не [отключите](#disable-prompt-caching). Тем не менее полезно понимать, как работает кэширование промптов, потому что некоторые действия инвалидируют кэш и делают следующий ответ более медленным и дорогим на время его перестроения. На этой странице рассказано, какие это действия, почему некоторые настройки применяются только после перезапуска и как проверить производительность кэша, когда потребление кажется высоким.

## Как устроен кэш

Каждый раз, когда вы отправляете сообщение в Claude Code, он создаёт новый запрос к API. Модель ничего не запоминает между запросами, поэтому Claude Code повторно отправляет весь контекст: системный промпт, контекст вашего проекта, все предыдущие сообщения и результаты работы инструментов, а также ваше новое сообщение. Новый контент добавляется в конец, а значит большая часть каждого запроса идентична предыдущему. Кэширование промптов — это то, как API избегает повторной обработки той части, которая не изменилась.

API кэширует, сопоставляя начало каждого запроса, называемое префиксом, с содержимым, которое он недавно обработал. На обычном ходу префикс — это весь предыдущий запрос, и новым является только последний обмен репликами. Совпадение точное, поэтому изменение где угодно в префиксе приводит к пересчёту всего, что идёт после него. Кэширования по файлам или по сегментам нет. См. [как работает кэширование промптов](https://platform.claude.com/docs/en/build-with-claude/prompt-caching#how-prompt-caching-works) в справочнике API, чтобы понять лежащий в основе механизм.

Чтобы извлечь максимум из сопоставления префиксов, Claude Code упорядочивает каждый запрос так, чтобы контент, который редко меняется между ходами, шёл первым:

| Слой              | Содержимое                                                       | Меняется, когда                                                              |
| ----------------- | --------------------------------------------------------------- | --------------------------------------------------------------------------- |
| Системный промпт  | Базовые инструкции, определения инструментов, стиль вывода       | Меняется набор загруженных определений инструментов или обновляется Claude Code |
| Контекст проекта  | CLAUDE.md, автопамять, правила без области действия               | Запускается сессия либо после `/clear` или `/compact`                        |
| Диалог            | Ваши сообщения, ответы Claude, результаты работы инструментов     | На каждом ходу                                                               |

Изменение в слое диалога оставляет системный промпт и контекст проекта закэшированными. Изменение системного промпта инвалидирует всё, потому что весь последующий контент теперь находится за другим префиксом. Третий столбец приводит распространённые триггеры, а не исчерпывающий список, и разделы ниже охватывают полный набор, включая контент вроде стиля вывода, который фиксируется в начале сессии.

Правило сопоставления префиксов объясняет большинство поведений на этой странице. Например, [режим планирования](https://code.claude.com/docs/en/permission-modes#analyze-before-you-edit-with-plan-mode) и [загрузка навыков](https://code.claude.com/docs/en/skills) добавляют свои инструкции как сообщения диалога, поэтому закэшированный префикс остаётся нетронутым.

Две настройки вообще не являются частью текста промпта, поэтому их нет в таблице слоёв, но обе входят в ключ кэша:

* **Модель**: у каждой модели свой кэш. Смена модели пересчитывает весь запрос, даже когда содержимое идентично. См. [Смена моделей](#switching-models) ниже.
* **Уровень усилий**: у каждого уровня усилий свой кэш для одной и той же модели. Изменение его посреди сессии пересчитывает весь запрос, и Claude Code просит вас подтвердить перед применением изменения. См. [Изменение уровня усилий](#changing-effort-level) ниже.

> **Совет.** Выбирайте модель и уровень усилий в начале сессии, а `/compact` приберегите для естественных пауз между задачами. Чем меньше изменений вы делаете посреди задачи, тем выше ваша доля попаданий в кэш.

### Где живёт кэш

Кэширование происходит на стороне сервера, в той инфраструктуре, которая обслуживает вашу модель. Где именно — зависит от того, как вы аутентифицируетесь:

* **API-ключ, подписка Claude или [Claude Platform на AWS](https://code.claude.com/docs/en/claude-platform-on-aws)**: кэш живёт в инфраструктуре Anthropic, доступ к которой осуществляется через [Claude API](https://platform.claude.com/docs)
* **Bedrock или Vertex AI**: кэш живёт в обслуживающей инфраструктуре вашего облачного провайдера
* **Foundry**: запросы направляются в инфраструктуру Anthropic
* **Пользовательский `ANTHROPIC_BASE_URL` или [LLM-шлюз](https://code.claude.com/docs/en/llm-gateway)**: кэш живёт там, куда пересылаются ваши запросы, и работает ли кэширование — зависит от шлюза

Что каждый провайдер хранит и обрабатывает, см. в разделе [использование данных](https://code.claude.com/docs/en/data-usage). Где бы ни жил кэш, записи истекают после периода бездействия, и раздел [Время жизни кэша](#cache-lifetime) ниже охватывает TTL и то, как его продлить.

## Действия, которые инвалидируют кэш

Эти действия приводят к тому, что следующий запрос промахивается мимо части кэша или всего кэша целиком. Вы видите одноразовый более медленный и дорогой ход, после которого новый префикс кэшируется. Большинства из них можно избежать посреди задачи, как только вы узнаёте, что у них есть цена. Смена модели может казаться бесплатной, пока вы не заметите следующий за ней медленный ход.

### Смена моделей

У каждой модели свой кэш. Переключение с помощью [`/model`](https://code.claude.com/docs/en/model-config#setting-your-model) означает, что следующий запрос читает всю историю диалога без попаданий в кэш, даже если содержимое идентично.

[Настройка модели `opusplan`](https://code.claude.com/docs/en/model-config#opusplan-model-setting) разрешается в Opus во время режима планирования и в Sonnet во время выполнения, поэтому каждое переключение режима планирования является сменой модели и начинает свежий кэш.

[Автоматический откат модели](https://code.claude.com/docs/en/model-config#automatic-model-fallback) на Fable 5 также является сменой модели. Когда классификатор безопасности помечает запрос, Claude Code перезапускает его на модели Opus по умолчанию, и сессия продолжается там.

### Изменение уровня усилий

Кэш ключуется по [уровню усилий](https://code.claude.com/docs/en/model-config#adjust-effort-level), а также по модели, поэтому переключение с помощью `/effort` означает, что следующий запрос читает всю историю диалога без попаданий в кэш. После того как диалог начался, Claude Code показывает диалог подтверждения перед применением изменения усилий, которое инвалидировало бы кэш. Изменение, которое разрешается в тот же уровень, что уже действует, — например, явная установка значения по умолчанию для модели, — пропускает диалог и сохраняет кэш.

### Включение быстрого режима

Включение [быстрого режима](https://code.claude.com/docs/en/fast-mode) добавляет заголовок запроса, который является частью ключа кэша, поэтому следующий запрос читает всю историю диалога без попаданий в кэш. Эти незакэшированные входные токены оплачиваются по [тарифам быстрого режима](https://code.claude.com/docs/en/fast-mode#understand-the-cost-tradeoff), и именно поэтому включение его в начале сессии обходится дешевле, чем включение глубоко внутри долгой сессии. Включение быстрого режима из модели, отличной от Opus, также [меняет вашу модель](#switching-models), что само по себе начинает свежий кэш.

Эта цена платится один раз за диалог. После первого хода в быстром режиме Claude Code продолжает отправлять заголовок и меняет только настройку скорости запроса, которая не является частью ключа кэша. Выключение быстрого режима, [автоматический откат к стандартной скорости](https://code.claude.com/docs/en/fast-mode#handle-rate-limits) после превышения лимита запросов и повторное включение его позже — всё это сохраняет кэш. `/clear` и `/compact` сбрасывают это, поскольку они всё равно перестраивают кэш в этих точках.

> **Примечание.** Сохранение заголовка между переключениями требует Claude Code v2.1.86 или новее. На более ранних версиях каждое переключение быстрого режима и откат по лимиту запросов инвалидирует кэш.

### Подключение или отключение MCP-сервера

Определения инструментов находятся в слое системного промпта, поэтому кэш инвалидируется, когда набор определений инструментов в запросе меняется между ходами. Переключение [инструмента-советника](https://code.claude.com/docs/en/advisor) является исключением: его определение находится после точки разрыва кэша, поэтому включение или отключение `/advisor` сохраняет закэшированный префикс нетронутым. Приводит ли к этому изменение [MCP-сервера](https://code.claude.com/docs/en/mcp), зависит от того, отложены ли его инструменты [поиском инструментов](https://code.claude.com/docs/en/mcp#scale-with-mcp-tool-search) или загружены в префикс:

* **Отложенные инструменты**, вариант по умолчанию на поддерживаемых моделях: подключение сервера, его отключение или изменение его списка инструментов только добавляет новый контент и не затрагивает ничего уже закэшированного.
* **Инструменты, загруженные в префикс**: любое их изменение инвалидирует кэш. Это происходит, когда [поиск инструментов недоступен или отключён](https://code.claude.com/docs/en/mcp#configure-tool-search), например на моделях Haiku, на Vertex AI или с пользовательским шлюзом `ANTHROPIC_BASE_URL`. Это также происходит для сервера или инструмента, помеченного [`alwaysLoad`](https://code.claude.com/docs/en/mcp#exempt-a-server-from-deferral), и для определений, оставленных заранее [загрузкой на основе порога](https://code.claude.com/docs/en/mcp#configure-tool-search).

Когда инструменты загружаются в префикс, самая частая причина инвалидации — подключение или отключение сервера посреди сессии, что может произойти без каких-либо действий с вашей стороны: процесс stdio-сервера завершается, HTTP-сессия истекает или сервер [переподключается автоматически после кратковременного сбоя](https://code.claude.com/docs/en/mcp#automatic-reconnection). Подключённый сервер также может отправить [динамическое обновление инструментов](https://code.claude.com/docs/en/mcp#dynamic-tool-updates), которое меняет его список инструментов.

Редактирование вашей MCP-конфигурации само по себе не меняет кэш. Новая конфигурация вступает в силу только после перезапуска, когда сервер подключается или отключается.

### Включение или отключение плагина

[Плагины](https://code.claude.com/docs/en/plugins) объединяют несколько типов компонентов, и цена изменения зависит от того, какие компоненты предоставляет плагин. Навыки (skills), команды, агенты, хуки, LSP-серверы, мониторы и темы никогда не инвалидируют кэш: всё, что они добавляют к запросу, добавляется после существующего диалога, поэтому следующий запрос оплачивает новый контент, но по-прежнему читает всё, что было до него, из кэша.

Исключение — плагин, который предоставляет [MCP-серверы](https://code.claude.com/docs/en/plugins-reference#mcp-servers). Включение или отключение такого следует тем же правилам, что и [подключение или отключение MCP-сервера](#connecting-or-disconnecting-an-mcp-server): кэш выживает, когда инструменты сервера отложены, и следующий запрос перечитывает весь диалог, когда они загружаются в префикс.

Изменения плагинов применяются, когда вы запускаете [`/reload-plugins`](https://code.claude.com/docs/en/discover-plugins#apply-plugin-changes-without-restarting) или начинаете новую сессию. Цена — будь то добавленные объявления или полное перечитывание — проявляется на первом ходу после перезагрузки, а не когда вы запускаете `/plugin install`, `/plugin enable` или `/plugin disable`. Начиная с v2.1.163, когда перезагрузка вызвала бы полное перечитывание, `/reload-plugins` показывает предупреждение и не применяет перезагрузку. Передайте `--force`, чтобы применить её всё равно.

Отключение плагина, который вы включили ранее в сессии, восстанавливает прежнюю форму запроса. Если этот префикс всё ещё находится в пределах своего [времени жизни кэша](#cache-lifetime), следующий запрос читает более старую запись кэша вместо перестроения.

### Запрет целого инструмента

Добавление голого имени инструмента вроде `Bash` или `WebFetch` как [правила запрета](https://code.claude.com/docs/en/permissions#manage-permissions) полностью удаляет этот инструмент из контекста Claude. Определения встроенных инструментов загружаются в слой системного промпта, поэтому добавление или удаление одного из таких правил посреди сессии инвалидирует кэш. Изменение вступает в силу на следующем ходу, независимо от того, добавляете ли вы его через `/permissions` или [редактируя файл настроек напрямую](https://code.claude.com/docs/en/settings#when-edits-take-effect).

Только правило запрета, которое совпадает в позиции имени инструмента, имеет этот эффект: голое имя инструмента, эквивалентная форма `Bash(*)` или [glob по имени инструмента](https://code.claude.com/docs/en/permissions#tool-name-wildcards) вроде `"*"`. Glob, который совпадает только с MCP-инструментами, например `"mcp__*"`, удаляет эти инструменты тем же способом, но оставляет кэш нетронутым, когда совпавшие инструменты [отложены](#connecting-or-disconnecting-an-mcp-server) (вариант по умолчанию), поскольку отложенные определения никогда не были в закэшированном префиксе. Правила запрета с областью действия вроде `Bash(rm *)`, а также все правила разрешения и запроса не меняют, какие инструменты видит Claude. Claude Code проверяет их, когда Claude пытается сделать вызов, оставляя префикс нетронутым.

### Сжатие диалога

[Сжатие](https://code.claude.com/docs/en/context-window#what-survives-compaction) заменяет историю ваших сообщений сводкой. По замыслу это инвалидирует слой диалога, поскольку у следующего запроса новая, более короткая история, которая не разделяет префикс со старой. Claude Code повторно использует слой системного промпта и перезагружает контекст проекта с диска, что даёт попадание в кэш только если CLAUDE.md и память не изменились с начала сессии.

Чтобы создать сводку, Claude Code отправляет разовый запрос с тем же системным промптом, инструментами и историей, что и ваш диалог, плюс инструкцию по суммированию, добавленную как финальное пользовательское сообщение. Поскольку он разделяет ваш префикс, этот запрос читает существующий кэш вместо повторной обработки всей истории. Большая часть времени сжатия уходит на генерацию сводки, а не на промах мимо кэша. Следующий ход перестраивает кэш диалога только для гораздо более короткой сводки, поэтому ход после сжатия не является медленной частью.

> **Совет.** Сжатие работает вам на пользу, когда контекст, который вы отбрасываете, — это контент, который вам больше не нужен. Чтобы выбрать, когда происходят его накладные расходы, запускайте `/compact` в естественной паузе в вашей работе, например между задачами, вместо того чтобы ждать, пока автосжатие сработает посреди задачи. Если вы пошли по пути, от которого хотите полностью отказаться, используйте [`/rewind`](#rewinding-the-conversation) к более раннему ходу. Перемотка отсекает историю обратно к префиксу, который уже закэширован, вместо построения нового, как это делает сжатие.

### Обновление Claude Code

Новая версия Claude Code обычно обновляет системный промпт или определения инструментов, поэтому первый запрос после обновления перестраивает кэш с самого начала. [Автообновление](https://code.claude.com/docs/en/setup#auto-updates) загружает новые версии в фоне, но применяет их при следующем запуске, никогда посреди сессии, поэтому вы видите это как незакэшированный первый ход после перезапуска, а не как неожиданность во время сессии. Установите `DISABLE_AUTOUPDATER=1`, чтобы контролировать, когда применяются обновления.

> **Примечание.** [Возобновление сессии](https://code.claude.com/docs/en/sessions#resume-a-session) после обновления обрабатывает всю историю диалога заново без попаданий в кэш, поскольку история теперь находится за другим системным промптом. Цена масштабируется с тем, насколько длинный возобновляемый диалог, поэтому первый ход обратно в долгую сессию может быть самым дорогим запросом, который вы отправляете.

### Редактирование файлов в вашем репозитории

Содержимое файлов попадает в контекст только когда Claude их читает, а операции чтения добавляются в диалог. Редактирование файла, который Claude ранее прочитал, не меняет задним числом более раннее чтение в истории. Вместо этого Claude Code добавляет `<system-reminder>`, отмечающий, что файл изменился, и Claude перечитывает его при необходимости.

### Редактирование CLAUDE.md посреди сессии

Ваши файлы CLAUDE.md в корне проекта и на уровне пользователя читаются один раз в начале сессии и хранятся в памяти. Их редактирование посреди сессии не инвалидирует кэш, но правка при этом и не применяется. Claude продолжает работать с версией, которая была загружена в начале сессии. Новый контент загружается при следующем `/clear`, `/compact` или перезапуске.

[Вложенные файлы CLAUDE.md в подкаталогах](https://code.claude.com/docs/en/memory) и [правила с frontmatter `paths:`](https://code.claude.com/docs/en/memory#path-specific-rules) загружаются позже, когда Claude впервые читает совпадающий файл. Редактирование такого до его загрузки действительно вступает в силу. После его загрузки контент становится частью истории диалога, поэтому правка посреди сессии не меняет его задним числом.

### Изменение стиля вывода

[Стиль вывода](https://code.claude.com/docs/en/output-styles) является частью системного промпта, который Claude Code читает один раз в начале сессии. Изменение его через `/config` или настройку `outputStyle` посреди сессии не инвалидирует кэш, но изменение при этом и не применяется. Claude продолжает использовать стиль, который был загружен в начале сессии. Новый стиль загружается при следующем `/clear` или перезапуске.

### Изменение режима разрешений

Переключение между [режимами разрешений](https://code.claude.com/docs/en/permission-modes), например с default на accept edits, не меняет системный промпт или определения инструментов, поэтому изменения режима безопасны для кэша. Исключение — режим планирования с настройкой модели [`opusplan`](https://code.claude.com/docs/en/model-config#opusplan-model-setting), которая переключает модель между Opus и Sonnet при входе в режим планирования или выходе из него. Это делает переключение режима [сменой модели](#switching-models).

### Вызов навыков и команд

[Навыки](https://code.claude.com/docs/en/skills) и [команды](https://code.claude.com/docs/en/commands) внедряют свои инструкции как пользовательские сообщения в точке вызова. Ничто более раннее в диалоге не меняется.

### Запуск `/recap`

[`/recap`](https://code.claude.com/docs/en/interactive-mode#session-recap) генерирует сводку для отображения в вашем терминале. В отличие от `/compact`, он добавляет сводку как вывод команды, а не заменяет историю ваших сообщений, поэтому закэшированный префикс остаётся нетронутым.

### Перемотка диалога

[`/rewind`](https://code.claude.com/docs/en/checkpointing) отсекает ваш диалог обратно к более раннему ходу. Оставшаяся история — тот же контент, из которого был построен кэш в тот момент, и слои системного промпта и контекста проекта не изменились, поэтому следующий запрос попадает в более раннюю запись кэша. Каждый ход с тех пор читался через этот префикс, что поддерживало запись «тёплой», даже если исходный ход был давнее, чем TTL.

Восстановление контрольных точек файлов вместе с диалогом не имеет отдельного эффекта на кэш. Содержимое файлов попадает в контекст только когда Claude их читает, так же как и при [редактировании файлов в вашем репозитории](#editing-files-in-your-repository).

## Время жизни кэша

Закэшированные префиксы истекают после периода бездействия. Каждый запрос, попадающий в кэш, сбрасывает таймер, поэтому кэш остаётся «тёплым», пока вы продолжаете работать. После достаточно долгого перерыва следующий запрос пересчитывает весь ввод и заново устанавливает кэш, поэтому первый ход после того, как вы отошли, может быть заметно медленнее.

Время жизни (TTL) определяет, какой длины перерыв кэш переживает. API предлагает два: пятиминутный TTL и [часовой TTL](https://platform.claude.com/docs/en/build-with-claude/prompt-caching#1-hour-cache-duration), который поддерживает кэш «тёплым» через более длительные перерывы, но [оплачивает записи в кэш по более высокому тарифу](https://platform.claude.com/docs/en/build-with-claude/prompt-caching#pricing). Claude Code выбирает TTL за вас на основе того, как вы аутентифицируетесь, и вы можете переопределить его переменными окружения.

### На подписке Claude

На подписке Claude, Claude Code запрашивает часовой TTL автоматически. Потребление включено в ваш план, а не оплачивается за токен, поэтому более длинный TTL не стоит вам ничего дополнительно и влияет только на то, как долго ваш кэш остаётся «тёплым».

Если вы превысили лимит потребления вашего плана и Claude Code использует [кредиты потребления](https://support.claude.com/en/articles/12429409-extra-usage-for-paid-claude-plans), это потребление оплачивается вами, поэтому Claude Code автоматически снижает TTL до пяти минут.

### На API-ключе или у стороннего провайдера

На API-ключе, Bedrock, Vertex, Foundry или Claude Platform на AWS вы платите по тарифам за токен, поэтому TTL по умолчанию остаётся на более дешёвых пяти минутах. Чтобы подключить [часовой TTL](https://platform.claude.com/docs/en/build-with-claude/prompt-caching#1-hour-cache-duration), установите `ENABLE_PROMPT_CACHING_1H=1`.

На Bedrock поддержка кэширования промптов, минимальная длина кэшируемого префикса и доступность часового TTL — всё это зависит от модели. Если счётчики токенов кэша остаются на нуле, проверьте [поддерживаемые модели, регионы и ограничения](https://docs.aws.amazon.com/bedrock/latest/userguide/prompt-caching.html#prompt-caching-models) в документации Bedrock.

### Переопределение TTL

Установите `FORCE_PROMPT_CACHING_5M=1`, чтобы принудительно задать пятиминутный TTL независимо от аутентификации. Это полезно, когда вы отлаживаете поведение кэша, сравниваете два TTL или переопределяете `ENABLE_PROMPT_CACHING_1H`, установленный в [управляемых настройках](https://code.claude.com/docs/en/settings#settings-files).

## Область действия кэша

В Claude Code кэш фактически ограничен одной машиной и каталогом. Системный промпт встраивает рабочий каталог, платформу, оболочку, версию ОС и пути автопамяти, поэтому две сессии в разных каталогах строят разные префиксы и промахиваются мимо кэша друг друга. Это включает рабочие деревья (worktrees) одного и того же репозитория, поскольку у каждого рабочего дерева свой рабочий каталог.

Сессии, которые вы запускаете параллельно в одном каталоге, строят совпадающие префиксы и читают кэш друг друга. Последовательные сессии разделяют префикс только когда снимок статуса git при запуске совпадает, поскольку системный промпт также захватывает ветку и недавние коммиты.

Лежащий в основе кэш API шире. Кэши изолированы между организациями, а у некоторых провайдеров — [между рабочими пространствами внутри организации](https://platform.claude.com/docs/en/build-with-claude/prompt-caching#cache-storage-and-sharing). В пределах этих границ любые два запроса с одинаковой моделью и префиксом читают один и тот же кэш. Для вызывающих Agent SDK, которые запускают флоты автоматизированных процессов, см. [улучшение кэширования промптов между пользователями и машинами](https://code.claude.com/docs/en/agent-sdk/modifying-system-prompts#improve-prompt-caching-across-users-and-machines), чтобы подавить машиноспецифичные разделы системного промпта и разделять кэш между машинами.

## Проверка производительности кэша

Производительность кэша проявляется как два счётчика токенов, которые API сообщает в каждом ответе. Самый прямой способ наблюдать за ними вживую — это [скрипт строки состояния](https://code.claude.com/docs/en/statusline), который читает объект `current_usage`:

| Поле                          | Значение                                                                                    |
| ----------------------------- | ------------------------------------------------------------------------------------------- |
| `cache_creation_input_tokens` | Токены, записанные в кэш на этом ходу, оплачиваются по тарифу записи в кэш                    |
| `cache_read_input_tokens`     | Токены, обслуженные из кэша на этом ходу, оплачиваются примерно по 10% от стандартного тарифа на ввод |

Высокое соотношение чтения к созданию означает, что кэширование работает хорошо. Если создание остаётся высоким ход за ходом, что-то меняется в вашем префиксе. Раздел [действия, которые инвалидируют кэш](#actions-that-invalidate-the-cache) перечисляет обычные причины.

Для видимости в масштабе организации экспортёр OpenTelemetry сообщает токены чтения и создания кэша по каждому пользователю и сессии. См. [Мониторинг потребления](https://code.claude.com/docs/en/monitoring-usage) для справочника по метрике и атрибутам событий.

## Субагенты и кэш

[Субагент](https://code.claude.com/docs/en/sub-agents) начинает свой собственный диалог со своим собственным системным промптом и набором инструментов, отдельным от родительского. Он строит свой собственный кэш, начиная без попаданий в кэш на первом вызове и разогреваясь на протяжении своих собственных ходов. Субагенты используют пятиминутный TTL даже на подписке, поскольку автоматический часовой TTL применяется к основному диалогу.

Кэш родителя не затрагивается. Со стороны родителя вызов субагента и его результат добавляются в диалог, оставляя префикс родителя нетронутым.

[Форк](https://code.claude.com/docs/en/sub-agents#fork-the-current-conversation), напротив, наследует системный промпт, инструменты и историю диалога родителя в точности, поэтому его первый запрос читает кэш родителя. Вызов суммирования при сжатии, описанный в разделе [Сжатие диалога](#compacting-the-conversation), использует тот же подход с разделением префикса.

## Отключение кэширования промптов

Отключение кэширования иногда полезно при отладке поведения кэширования с конкретной моделью или провайдером. Чтобы отключить его, установите одну из этих переменных окружения в `1`:

| Переменная                      | Эффект                     |
| ------------------------------- | -------------------------- |
| `DISABLE_PROMPT_CACHING`        | Отключить для всех моделей  |
| `DISABLE_PROMPT_CACHING_HAIKU`  | Отключить только для Haiku  |
| `DISABLE_PROMPT_CACHING_SONNET` | Отключить только для Sonnet |
| `DISABLE_PROMPT_CACHING_OPUS`   | Отключить только для Opus   |
| `DISABLE_PROMPT_CACHING_FABLE`  | Отключить только для Fable  |

Чтобы задать политику кэширования в масштабе организации, поместите любую из этих или [переменных TTL](#cache-lifetime) в блок `env` [управляемых настроек](https://code.claude.com/docs/en/settings#settings-files). Для обычного использования оставляйте кэширование включённым.
