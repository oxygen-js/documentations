# Хранение инструкций и воспоминаний

> Задавайте Claude постоянные инструкции с помощью файлов CLAUDE.md и позвольте Claude автоматически накапливать знания с помощью автопамяти.

Каждая сессия Claude Code начинается с чистого окна контекста. Знания между сессиями переносят два механизма:

* **Файлы CLAUDE.md**: инструкции, которые вы пишете, чтобы дать Claude постоянный контекст
* **Автопамять**: заметки, которые Claude пишет сам на основе ваших исправлений и предпочтений

На этой странице рассказывается, как:

* [Писать и организовывать файлы CLAUDE.md](#claude-md-files)
* [Ограничивать правила определёнными типами файлов](#organize-rules-with-claude/rules/) с помощью `.claude/rules/`
* [Настроить автопамять](#auto-memory), чтобы Claude делал заметки автоматически
* [Устранять неполадки](#troubleshoot-memory-issues), когда инструкции не выполняются

## CLAUDE.md против автопамяти

В Claude Code есть две взаимодополняющие системы памяти. Обе загружаются в начале каждого разговора. Claude воспринимает их как контекст, а не как принудительную конфигурацию. Чтобы заблокировать действие независимо от того, что решит Claude, используйте вместо этого [хук PreToolUse](https://code.claude.com/docs/en/hooks-guide). Чем более конкретны и лаконичны ваши инструкции, тем более последовательно Claude им следует.

|                      | Файлы CLAUDE.md                                   | Автопамять                                                      |
| :------------------- | :------------------------------------------------ | :--------------------------------------------------------------- |
| **Кто пишет**        | Вы                                                | Claude                                                          |
| **Что содержит**     | Инструкции и правила                              | Знания и закономерности                                          |
| **Область действия** | Проект, пользователь или организация             | Для каждого репозитория, общая для всех рабочих деревьев (worktrees) |
| **Куда загружается** | В каждую сессию                                   | В каждую сессию (первые 200 строк или 25 КБ)                     |
| **Для чего использовать** | Стандарты написания кода, рабочие процессы, архитектура проекта | Команды сборки, находки при отладке, предпочтения, которые обнаруживает Claude |

Используйте файлы CLAUDE.md, когда хотите направлять поведение Claude. Автопамять позволяет Claude учиться на ваших исправлениях без ручных усилий.

Субагенты также могут вести собственную автопамять. Подробности см. в [настройке субагентов](https://code.claude.com/docs/en/sub-agents#enable-persistent-memory).

## Файлы CLAUDE.md

Файлы CLAUDE.md — это markdown-файлы, которые дают Claude постоянные инструкции для проекта, вашего личного рабочего процесса или всей вашей организации. Вы пишете эти файлы обычным текстом; Claude читает их в начале каждой сессии.

### Когда добавлять в CLAUDE.md

Относитесь к CLAUDE.md как к месту, где вы записываете то, что иначе пришлось бы объяснять снова. Добавляйте туда записи, когда:

* Claude совершает одну и ту же ошибку во второй раз
* Проверка кода выявляет то, что Claude должен был знать об этой кодовой базе
* Вы вводите в чат то же исправление или уточнение, которое вводили в прошлой сессии
* Новому члену команды понадобился бы тот же контекст, чтобы продуктивно работать

Ограничивайтесь фактами, которые Claude должен держать в каждой сессии: командами сборки, соглашениями, структурой проекта, правилами «всегда делай X». Если запись представляет собой многошаговую процедуру или важна только для одной части кодовой базы, перенесите её в [навык (skill)](https://code.claude.com/docs/en/skills) или в [правило с привязкой к путям](#organize-rules-with-claude/rules/). В [обзоре расширений](https://code.claude.com/docs/en/features-overview#build-your-setup-over-time) рассказано, когда использовать каждый механизм.

### Выбор места для файлов CLAUDE.md

Файлы CLAUDE.md могут располагаться в нескольких местах, у каждого из которых своя область действия. В таблице ниже они перечислены в порядке загрузки, от самой широкой области действия к самой узкой, так что инструкция проекта появляется в контексте после инструкции пользователя.

| Область действия         | Расположение                                                                                                                                                            | Назначение                                                | Примеры использования                                               | Доступно кому                   |
| ------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------- | ------------------------------------------------------------------- | ------------------------------- |
| **Управляемая политика** | • macOS: `/Library/Application Support/ClaudeCode/CLAUDE.md`<br />• Linux и WSL: `/etc/claude-code/CLAUDE.md`<br />• Windows: `C:\Program Files\ClaudeCode\CLAUDE.md`   | Инструкции для всей организации, управляемые IT/DevOps     | Корпоративные стандарты написания кода, политики безопасности, требования соответствия | Всем пользователям в организации |
| **Инструкции пользователя** | `~/.claude/CLAUDE.md`                                                                                                                                               | Личные предпочтения для всех проектов                     | Предпочтения по оформлению кода, личные сокращения для инструментов | Только вам (во всех проектах)   |
| **Инструкции проекта**   | `./CLAUDE.md` или `./.claude/CLAUDE.md`                                                                                                                                | Общие для команды инструкции проекта                     | Архитектура проекта, стандарты написания кода, распространённые рабочие процессы | Членам команды через систему контроля версий |
| **Локальные инструкции** | `./CLAUDE.local.md`                                                                                                                                                    | Личные предпочтения для конкретного проекта; добавьте в `.gitignore` | URL-адреса вашей песочницы, предпочитаемые тестовые данные | Только вам (в текущем проекте)  |

Файлы CLAUDE.md и CLAUDE.local.md в иерархии каталогов выше рабочего каталога загружаются целиком при запуске. Файлы в подкаталогах загружаются по мере необходимости, когда Claude читает файлы в этих каталогах. Полный порядок разрешения см. в разделе [Как загружаются файлы CLAUDE.md](#how-claude-md-files-load).

Для крупных проектов вы можете разбить инструкции на тематические файлы с помощью [правил проекта](#organize-rules-with-claude/rules/). Правила позволяют ограничивать инструкции определёнными типами файлов или подкаталогами.

### Настройка CLAUDE.md проекта

CLAUDE.md проекта может храниться либо в `./CLAUDE.md`, либо в `./.claude/CLAUDE.md`. Создайте этот файл и добавьте инструкции, которые применимы ко всем, кто работает над проектом: команды сборки и тестирования, стандарты написания кода, архитектурные решения, соглашения об именовании и распространённые рабочие процессы. Эти инструкции передаются вашей команде через систему контроля версий, поэтому сосредоточьтесь на стандартах уровня проекта, а не на личных предпочтениях.

> **Совет.** Запустите `/init`, чтобы автоматически сгенерировать стартовый CLAUDE.md. Claude анализирует вашу кодовую базу и создаёт файл с командами сборки, инструкциями по тестированию и обнаруженными им соглашениями проекта. Если CLAUDE.md уже существует, `/init` предлагает улучшения, а не перезаписывает его. Дальше дорабатывайте файл инструкциями, которые Claude не смог бы обнаружить самостоятельно.
>
> Установите `CLAUDE_CODE_NEW_INIT=1`, чтобы включить интерактивный многоэтапный процесс. `/init` спрашивает, какие артефакты настроить: файлы CLAUDE.md, навыки и хуки. Затем он исследует вашу кодовую базу с помощью субагента, восполняет пробелы уточняющими вопросами и представляет предложение для проверки, прежде чем записывать какие-либо файлы.

### Как писать эффективные инструкции

Файлы CLAUDE.md загружаются в окно контекста в начале каждой сессии, потребляя токены наравне с вашим разговором. [Визуализация окна контекста](https://code.claude.com/docs/en/context-window) показывает, где загружается CLAUDE.md относительно остального стартового контекста. Поскольку это контекст, а не принудительная конфигурация, то, как вы пишете инструкции, влияет на то, насколько надёжно Claude им следует. Лучше всего работают конкретные, лаконичные и хорошо структурированные инструкции.

**Размер**: старайтесь укладываться менее чем в 200 строк на один файл CLAUDE.md. Более длинные файлы потребляют больше контекста и снижают следование инструкциям. Если ваши инструкции разрастаются, используйте [правила с привязкой к путям](#path-specific-rules), чтобы инструкции загружались только когда Claude работает с подходящими файлами. Вы также можете разбить содержимое на [импорты](#import-additional-files) для организации, хотя импортированные файлы всё равно загружаются и попадают в окно контекста при запуске.

**Структура**: используйте markdown-заголовки и маркированные списки, чтобы группировать связанные инструкции. Claude просматривает структуру так же, как читатели: организованные разделы воспринимать легче, чем плотные абзацы.

**Конкретность**: пишите инструкции, достаточно конкретные, чтобы их можно было проверить. Например:

* «Используйте отступ в 2 пробела» вместо «Форматируйте код правильно»
* «Запускайте `npm test` перед коммитом» вместо «Тестируйте свои изменения»
* «API-обработчики находятся в `src/api/handlers/`» вместо «Держите файлы в порядке»

**Согласованность**: если два правила противоречат друг другу, Claude может выбрать одно из них произвольно. Периодически просматривайте свои файлы CLAUDE.md, вложенные файлы CLAUDE.md в подкаталогах и [`.claude/rules/`](#organize-rules-with-claude/rules/), чтобы удалять устаревшие или противоречивые инструкции. В монорепозиториях используйте [`claudeMdExcludes`](#exclude-specific-claude-md-files), чтобы пропускать файлы CLAUDE.md других команд, не относящиеся к вашей работе.

### Импорт дополнительных файлов

Файлы CLAUDE.md могут импортировать дополнительные файлы с помощью синтаксиса `@path/to/import`. Импортированные файлы разворачиваются и загружаются в контекст при запуске вместе с CLAUDE.md, который на них ссылается.

Допускаются как относительные, так и абсолютные пути. Относительные пути разрешаются относительно файла, содержащего импорт, а не рабочего каталога. Импортированные файлы могут рекурсивно импортировать другие файлы с максимальной глубиной в четыре перехода.

При разборе импортов пропускаются markdown-фрагменты кода и огороженные код-блоки. Чтобы упомянуть путь в вашем CLAUDE.md, не импортируя его, заключите его в обратные кавычки: запись `` `@README` `` оставляет текст буквальным, тогда как `@README` вне обратных кавычек импортирует файл.

Чтобы подтянуть README, package.json и руководство по рабочему процессу, сошлитесь на них с помощью синтаксиса `@` в любом месте вашего CLAUDE.md:

```text
See @README for project overview and @package.json for available npm commands for this project.

# Additional Instructions
- git workflow @docs/git-instructions.md
```

Для приватных предпочтений конкретного проекта, которые не должны попадать в систему контроля версий, создайте `CLAUDE.local.md` в корне проекта. Он загружается вместе с `CLAUDE.md` и обрабатывается так же. Добавьте `CLAUDE.local.md` в ваш `.gitignore`, чтобы он не коммитился; запуск `/init` с выбором личного варианта сделает это за вас.

Если вы работаете с несколькими рабочими деревьями (worktrees) одного репозитория, то файл `CLAUDE.local.md`, добавленный в gitignore, существует только в том рабочем дереве, где вы его создали. Чтобы поделиться личными инструкциями между рабочими деревьями, импортируйте вместо этого файл из вашего домашнего каталога:

```text
# Individual Preferences
- @~/.claude/my-project-instructions.md
```

> **Предупреждение.** Когда Claude Code впервые встречает внешние импорты в проекте, он показывает диалог подтверждения со списком файлов. Если вы откажетесь, импорты останутся отключёнными, и диалог больше не появится.

Более структурированный подход к организации инструкций описан в разделе [`.claude/rules/`](#organize-rules-with-claude/rules/).

### AGENTS.md

Claude Code читает `CLAUDE.md`, а не `AGENTS.md`. Если ваш репозиторий уже использует `AGENTS.md` для других агентов написания кода, создайте `CLAUDE.md`, который импортирует его, чтобы оба инструмента читали одни и те же инструкции без их дублирования. Вы также можете добавить инструкции, специфичные для Claude, ниже импорта. Claude загружает импортированный файл в начале сессии, а затем добавляет остальное:

```markdown CLAUDE.md
@AGENTS.md

## Claude Code

Use plan mode for changes under `src/billing/`.
```

Символическая ссылка (symlink) тоже подойдёт, если вам не нужно добавлять контент, специфичный для Claude:

```bash
ln -s AGENTS.md CLAUDE.md
```

В Windows создание символической ссылки требует прав администратора или режима разработчика, поэтому используйте вместо этого импорт `@AGENTS.md`.

Запуск [`/init`](https://code.claude.com/docs/en/commands) в репозитории, где уже есть `AGENTS.md`, читает его и включает соответствующие части в сгенерированный `CLAUDE.md`. Он также читает конфигурации других инструментов, такие как `.cursorrules`, `.devin/rules/` и `.windsurfrules`.

### Как загружаются файлы CLAUDE.md

Claude Code читает файлы CLAUDE.md, поднимаясь вверх по дереву каталогов от вашего текущего рабочего каталога и проверяя каждый каталог по пути на наличие файлов `CLAUDE.md` и `CLAUDE.local.md`. Это означает, что если вы запускаете Claude Code в `foo/bar/`, он загружает инструкции из `foo/bar/CLAUDE.md`, `foo/CLAUDE.md` и любых файлов `CLAUDE.local.md` рядом с ними.

Все обнаруженные файлы объединяются в контекст, а не переопределяют друг друга. По всему дереву каталогов содержимое упорядочивается от корня файловой системы вниз к вашему рабочему каталогу. В примере с `foo/bar/` файл `foo/CLAUDE.md` появляется в контексте перед `foo/bar/CLAUDE.md`, так что инструкции, расположенные ближе к месту запуска Claude, читаются последними. Внутри каждого каталога `CLAUDE.local.md` добавляется после `CLAUDE.md`, так что ваши личные заметки — это последнее, что Claude читает на этом уровне.

Claude также обнаруживает файлы `CLAUDE.md` и `CLAUDE.local.md` в подкаталогах ниже вашего текущего рабочего каталога. Вместо загрузки при запуске они включаются, когда Claude читает файлы в этих подкаталогах.

Если вы работаете в большом монорепозитории, где подхватываются файлы CLAUDE.md других команд, используйте [`claudeMdExcludes`](#exclude-specific-claude-md-files), чтобы пропускать их. Полную структуру корневых и покаталожных файлов CLAUDE.md и правил см. в разделе [Монорепозитории и большие репозитории](https://code.claude.com/docs/en/large-codebases).

Блочные HTML-комментарии (`<!-- maintainer notes -->`) в файлах CLAUDE.md удаляются перед тем, как содержимое внедряется в контекст Claude. Используйте их, чтобы оставлять заметки для людей-сопровождающих, не тратя на них токены контекста. Комментарии внутри код-блоков сохраняются. Когда вы открываете файл CLAUDE.md напрямую с помощью инструмента Read, комментарии остаются видимыми.

#### Загрузка из дополнительных каталогов

Флаг `--add-dir` даёт Claude доступ к дополнительным каталогам за пределами вашего основного рабочего каталога. По умолчанию файлы CLAUDE.md из этих каталогов не загружаются.

Чтобы также загружать файлы памяти из дополнительных каталогов, установите переменную окружения `CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD`:

```bash
CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1 claude --add-dir ../shared-config
```

Это загружает `CLAUDE.md`, `.claude/CLAUDE.md`, `.claude/rules/*.md` и `CLAUDE.local.md` из дополнительного каталога. `CLAUDE.local.md` пропускается, если вы исключаете `local` из [`--setting-sources`](https://code.claude.com/docs/en/cli-reference).

### Организация правил с помощью `.claude/rules/`

Для более крупных проектов вы можете организовать инструкции в несколько файлов с помощью каталога `.claude/rules/`. Это делает инструкции модульными и более удобными в сопровождении для команд. Правила также можно [привязать к определённым путям файлов](#path-specific-rules), чтобы они загружались в контекст только когда Claude работает с подходящими файлами, снижая шум и экономя место в контексте.

> **Примечание.** Правила загружаются в контекст в каждой сессии или при открытии подходящих файлов. Для инструкций, специфичных для задачи, которым не нужно постоянно находиться в контексте, используйте вместо этого [навыки](https://code.claude.com/docs/en/skills), которые загружаются только когда вы их вызываете или когда Claude определяет, что они относятся к вашему промпту.

#### Настройка правил

Размещайте markdown-файлы в каталоге `.claude/rules/` вашего проекта. Каждый файл должен охватывать одну тему и иметь описательное имя, например `testing.md` или `api-design.md`. Все файлы `.md` обнаруживаются рекурсивно, поэтому вы можете организовать правила в подкаталоги, такие как `frontend/` или `backend/`:

```text
your-project/
├── .claude/
│   ├── CLAUDE.md           # Main project instructions
│   └── rules/
│       ├── code-style.md   # Code style guidelines
│       ├── testing.md      # Testing conventions
│       └── security.md     # Security requirements
```

Правила без [frontmatter `paths`](#path-specific-rules) загружаются при запуске с тем же приоритетом, что и `.claude/CLAUDE.md`.

#### Правила с привязкой к путям

Правила можно привязать к определённым файлам с помощью YAML-frontmatter с полем `paths`. Эти условные правила применяются только когда Claude работает с файлами, соответствующими указанным шаблонам.

```markdown
---
paths:
  - "src/api/**/*.ts"
---

# API Development Rules

- All API endpoints must include input validation
- Use the standard error response format
- Include OpenAPI documentation comments
```

Правила без поля `paths` загружаются безусловно и применяются ко всем файлам. Правила с привязкой к путям срабатывают, когда Claude читает файлы, соответствующие шаблону, а не при каждом использовании инструмента.

Используйте glob-шаблоны в поле `paths`, чтобы сопоставлять файлы по расширению, каталогу или любой их комбинации:

| Шаблон                 | Соответствует                            |
| ---------------------- | ---------------------------------------- |
| `**/*.ts`              | Всем файлам TypeScript в любом каталоге  |
| `src/**/*`             | Всем файлам в каталоге `src/`            |
| `*.md`                 | Markdown-файлам в корне проекта          |
| `src/components/*.tsx` | React-компонентам в определённом каталоге |

Вы можете указать несколько шаблонов и использовать раскрытие скобок, чтобы сопоставить несколько расширений в одном шаблоне:

```markdown
---
paths:
  - "src/**/*.{ts,tsx}"
  - "lib/**/*.ts"
  - "tests/**/*.test.ts"
---
```

#### Совместное использование правил в разных проектах с помощью символических ссылок

Каталог `.claude/rules/` поддерживает символические ссылки, поэтому вы можете вести общий набор правил и связывать их с несколькими проектами. Символические ссылки разрешаются и загружаются в обычном порядке, а циклические символические ссылки обнаруживаются и корректно обрабатываются.

В этом примере связываются как общий каталог, так и отдельный файл:

```bash
ln -s ~/shared-claude-rules .claude/rules/shared
ln -s ~/company-standards/security.md .claude/rules/security.md
```

#### Правила на уровне пользователя

Личные правила в `~/.claude/rules/` применяются к каждому проекту на вашей машине. Используйте их для предпочтений, не связанных с конкретным проектом:

```text
~/.claude/rules/
├── preferences.md    # Your personal coding preferences
└── workflows.md      # Your preferred workflows
```

Правила на уровне пользователя загружаются перед правилами проекта, что даёт правилам проекта более высокий приоритет.

### Управление CLAUDE.md для больших команд

Для организаций, развёртывающих Claude Code в разных командах, вы можете централизовать инструкции и управлять тем, какие файлы CLAUDE.md загружаются.

#### Развёртывание CLAUDE.md для всей организации

Организации могут развернуть централизованно управляемый CLAUDE.md, который применяется ко всем пользователям на машине. Этот файл нельзя исключить с помощью индивидуальных настроек.

1. **Создайте файл в расположении для управляемых политик**
   * macOS: `/Library/Application Support/ClaudeCode/CLAUDE.md`
   * Linux и WSL: `/etc/claude-code/CLAUDE.md`
   * Windows: `C:\Program Files\ClaudeCode\CLAUDE.md`
2. **Разверните его с помощью вашей системы управления конфигурацией**
   Используйте MDM, групповые политики (Group Policy), Ansible или аналогичные инструменты, чтобы распространить файл по машинам разработчиков. Другие варианты конфигурации для всей организации см. в разделе [управляемые настройки](https://code.claude.com/docs/en/permissions#managed-settings).

Ключ `claudeMd` позволяет поместить содержимое управляемого CLAUDE.md прямо внутрь `managed-settings.json` вместо развёртывания отдельного файла.

**Область действия**: каждая сессия Claude Code на машине, в каждом репозитории. Для указаний, специфичных для репозитория, вместо этого закоммитьте CLAUDE.md проекта.

**Приоритет**: такой же, как у управляемого файла CLAUDE.md. Загружается перед CLAUDE.md пользователя и проекта.

**Где учитывается**: только в управляемых настройках и настройках политик. Установка `claudeMd` в пользовательских, проектных или локальных настройках не даёт эффекта.

В примере ниже поведенческие инструкции добавляются прямо в файл управляемых настроек:

```json
{
  "claudeMd": "Always run `make lint` before committing.\nNever push directly to main."
}
```

Управляемый CLAUDE.md и [управляемые настройки](https://code.claude.com/docs/en/settings#settings-files) служат разным целям. Используйте настройки для технического принуждения, а CLAUDE.md — для указаний по поведению:

| Задача                                         | Где настраивать                                          |
| :--------------------------------------------- | :-------------------------------------------------------- |
| Блокировка определённых инструментов, команд или путей файлов | Управляемые настройки: `permissions.deny`                |
| Принудительная изоляция песочницы              | Управляемые настройки: `sandbox.enabled`                 |
| Переменные окружения и маршрутизация к API-провайдеру | Управляемые настройки: `env`                             |
| Метод аутентификации и привязка к организации  | Управляемые настройки: `forceLoginMethod`, `forceLoginOrgUUID` |
| Указания по стилю и качеству кода              | Управляемый CLAUDE.md                                    |
| Напоминания об обработке данных и соответствии требованиям | Управляемый CLAUDE.md                                    |
| Поведенческие инструкции для Claude            | Управляемый CLAUDE.md                                    |

Правила настроек принудительно применяются клиентом независимо от того, что решит сделать Claude. Инструкции CLAUDE.md формируют поведение Claude, но не являются жёстким слоем принуждения.

#### Исключение определённых файлов CLAUDE.md

В больших монорепозиториях родительские файлы CLAUDE.md могут содержать инструкции, не относящиеся к вашей работе. Настройка `claudeMdExcludes` позволяет пропускать определённые файлы по пути или glob-шаблону.

В этом примере исключаются CLAUDE.md верхнего уровня и каталог правил из родительской папки. Добавьте это в `.claude/settings.local.json`, чтобы исключение оставалось локальным для вашей машины:

```json
{
  "claudeMdExcludes": [
    "**/monorepo/CLAUDE.md",
    "/home/user/monorepo/other-team/.claude/rules/**"
  ]
}
```

Шаблоны сопоставляются с абсолютными путями файлов с помощью glob-синтаксиса. Вы можете настроить `claudeMdExcludes` на любом [уровне настроек](https://code.claude.com/docs/en/settings#settings-files): пользовательском, проектном, локальном или управляемой политики. Массивы объединяются между уровнями.

Файлы CLAUDE.md управляемой политики нельзя исключить. Это гарантирует, что инструкции для всей организации всегда применяются независимо от индивидуальных настроек.

## Автопамять

Автопамять позволяет Claude накапливать знания между сессиями без того, чтобы вы что-либо писали. Claude сохраняет заметки для себя по ходу работы: команды сборки, находки при отладке, заметки об архитектуре, предпочтения по стилю кода и привычки рабочего процесса. Claude сохраняет что-то не в каждой сессии. Он решает, что стоит запомнить, исходя из того, будет ли информация полезна в будущем разговоре.

> **Примечание.** Автопамять требует Claude Code версии v2.1.59 или новее. Проверьте вашу версию с помощью `claude --version`.

### Включение или отключение автопамяти

Автопамять включена по умолчанию. Чтобы переключить её, откройте `/memory` в сессии и используйте переключатель автопамяти или установите `autoMemoryEnabled` в настройках вашего проекта:

```json
{
  "autoMemoryEnabled": false
}
```

Чтобы отключить автопамять через переменную окружения, установите `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1`.

### Место хранения

Каждый проект получает собственный каталог памяти по пути `~/.claude/projects/<project>/memory/`. Путь `<project>` выводится из git-репозитория, поэтому все рабочие деревья (worktrees) и подкаталоги в пределах одного репозитория используют один общий каталог автопамяти. Вне git-репозитория вместо этого используется корень проекта.

Чтобы хранить автопамять в другом месте, установите `autoMemoryDirectory` в вашем `settings.json`. Он считывается из любой [области настроек](https://code.claude.com/docs/en/settings#settings-precedence): пользовательской, проектной, локальной, политики или `--settings`.

```json
{
  "autoMemoryDirectory": "~/my-custom-memory-dir"
}
```

Значение должно быть абсолютным путём или начинаться с `~/`. Когда оно задано в `.claude/settings.json` или `.claude/settings.local.json` проекта, значение учитывается только после того, как вы примете диалог доверия к рабочему пространству для этой папки — тот же барьер, что управляет хуками.

Каталог содержит точку входа `MEMORY.md` и необязательные тематические файлы:

```text
~/.claude/projects/<project>/memory/
├── MEMORY.md          # Concise index, loaded into every session
├── debugging.md       # Detailed notes on debugging patterns
├── api-conventions.md # API design decisions
└── ...                # Any other topic files Claude creates
```

`MEMORY.md` выступает индексом каталога памяти. Claude читает и записывает файлы в этом каталоге на протяжении вашей сессии, используя `MEMORY.md`, чтобы отслеживать, что и где хранится.

Автопамять локальна для машины. Все рабочие деревья (worktrees) и подкаталоги в пределах одного git-репозитория используют один общий каталог автопамяти. Файлы не разделяются между машинами или облачными средами.

### Как это работает

Первые 200 строк `MEMORY.md` или первые 25 КБ — в зависимости от того, что наступит раньше — загружаются в начале каждого разговора. Содержимое за пределами этого порога не загружается в начале сессии. Claude поддерживает `MEMORY.md` лаконичным, перенося подробные заметки в отдельные тематические файлы.

Это ограничение применяется только к `MEMORY.md`. Файлы CLAUDE.md загружаются целиком независимо от длины, хотя более короткие файлы обеспечивают лучшее следование инструкциям.

Тематические файлы, такие как `debugging.md` или `patterns.md`, не загружаются при запуске. Claude читает их по мере необходимости с помощью своих стандартных инструментов для работы с файлами, когда ему нужна информация.

Claude читает и записывает файлы памяти во время вашей сессии. Когда вы видите «Writing memory» или «Recalled memory» в интерфейсе Claude Code, Claude активно обновляет или читает данные из `~/.claude/projects/<project>/memory/`.

### Аудит и редактирование вашей памяти

Файлы автопамяти — это обычный markdown, который вы можете редактировать или удалять в любое время. Запустите [`/memory`](#view-and-edit-with-%2Fmemory), чтобы просматривать и открывать файлы памяти прямо из сессии.

## Просмотр и редактирование с помощью `/memory`

Команда `/memory` перечисляет все файлы CLAUDE.md, CLAUDE.local.md и файлы правил, загруженные в вашей текущей сессии, позволяет включать или отключать автопамять и предоставляет ссылку для открытия папки автопамяти. Выберите любой файл, чтобы открыть его в вашем редакторе.

Когда вы просите Claude что-то запомнить, например «always use pnpm, not npm» или «remember that the API tests require a local Redis instance», Claude сохраняет это в автопамять. Чтобы вместо этого добавить инструкции в CLAUDE.md, попросите Claude напрямую, например «add this to CLAUDE.md», или отредактируйте файл сами через `/memory`.

## Устранение неполадок с памятью

Это самые распространённые проблемы с CLAUDE.md и автопамятью, а также шаги для их диагностики.

### Claude не следует моему CLAUDE.md

Содержимое CLAUDE.md доставляется как сообщение пользователя после системного промпта, а не как часть самого системного промпта. Claude читает его и пытается ему следовать, но нет гарантии строгого соблюдения, особенно для расплывчатых или противоречивых инструкций.

Для диагностики:

* Запустите `/memory`, чтобы убедиться, что ваши файлы CLAUDE.md и CLAUDE.local.md загружаются. Если файла нет в списке, Claude его не видит.
* Проверьте, что нужный CLAUDE.md находится в расположении, которое загружается для вашей сессии (см. [Выбор места для файлов CLAUDE.md](#choose-where-to-put-claude-md-files)).
* Сделайте инструкции более конкретными. «Используйте отступ в 2 пробела» работает лучше, чем «форматируйте код красиво».
* Ищите противоречивые инструкции в разных файлах CLAUDE.md. Если два файла дают разные указания для одного и того же поведения, Claude может выбрать одно произвольно.

Если инструкция — это что-то, что должно выполняться в определённый момент, например перед каждым коммитом или после каждого редактирования файла, оформите её вместо этого как [хук](https://code.claude.com/docs/en/hooks-guide). Хуки выполняются как команды оболочки в фиксированные моменты жизненного цикла и применяются независимо от того, что решит сделать Claude.

Для инструкций, которые вы хотите иметь на уровне системного промпта, используйте [`--append-system-prompt`](https://code.claude.com/docs/en/cli-reference#system-prompt-flags). Его нужно передавать при каждом запуске, поэтому он лучше подходит для скриптов и автоматизации, чем для интерактивного использования.

> **Совет.** Используйте [хук `InstructionsLoaded`](https://code.claude.com/docs/en/hooks#instructionsloaded), чтобы логировать, какие именно файлы инструкций загружаются, когда они загружаются и почему. Это полезно для отладки правил с привязкой к путям или файлов с отложенной загрузкой в подкаталогах.

### Я не знаю, что сохранила автопамять

Запустите `/memory` и выберите папку автопамяти, чтобы просмотреть, что сохранил Claude. Всё это обычный markdown, который вы можете читать, редактировать или удалять.

### Мой CLAUDE.md слишком большой

Файлы длиннее 200 строк потребляют больше контекста и могут снижать следование инструкциям. Используйте [правила с привязкой к путям](#path-specific-rules), чтобы загружать инструкции только когда Claude работает с подходящими файлами, или урезайте содержимое, которое не нужно в каждой сессии. Разбиение на [импорты `@path`](#import-additional-files) помогает организации, но не уменьшает контекст, поскольку импортированные файлы загружаются при запуске.

### Инструкции как будто теряются после `/compact`

CLAUDE.md из корня проекта переживает уплотнение: после `/compact` Claude перечитывает его с диска и снова внедряет в сессию. Вложенные файлы CLAUDE.md в подкаталогах не внедряются повторно автоматически; они перезагружаются, когда Claude в следующий раз читает файл в этом подкаталоге.

Если инструкция исчезла после уплотнения, она либо была дана только в разговоре, либо находится во вложенном CLAUDE.md, который ещё не перезагрузился. Добавьте инструкции, данные только в разговоре, в CLAUDE.md, чтобы они сохранялись. Полную разбивку см. в разделе [Что переживает уплотнение](https://code.claude.com/docs/en/context-window#what-survives-compaction).

Указания по размеру, структуре и конкретности см. в разделе [Как писать эффективные инструкции](#write-effective-instructions).
