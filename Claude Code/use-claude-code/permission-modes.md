# Режимы разрешений

> Управляйте тем, спрашивает ли Claude разрешение перед редактированием файлов или запуском команд. Переключайте режимы клавишами Shift+Tab в CLI или используйте селектор режимов в VS Code, Desktop и claude.ai.

Когда Claude хочет отредактировать файл, выполнить команду оболочки или сделать сетевой запрос, он приостанавливается и просит вас одобрить действие. Режимы разрешений управляют тем, как часто происходит эта пауза. Выбранный режим определяет ход сессии: режим по умолчанию заставляет вас проверять каждое действие по мере его появления, тогда как более свободные режимы позволяют Claude работать более длинными непрерывными отрезками и отчитываться по завершении. Выбирайте больше контроля для чувствительной работы или меньше прерываний, когда вы доверяете направлению.

## Доступные режимы

Каждый режим предлагает свой компромисс между удобством и контролем. В таблице ниже показано, что Claude может делать без запроса разрешения в каждом режиме.

| Режим                                                               | Что выполняется без запроса                                                                        | Лучше всего для                              |
| :------------------------------------------------------------------ | :------------------------------------------------------------------------------------------------- | :------------------------------------------- |
| `default`                                                           | Только чтение                                                                                      | Начало работы, чувствительная работа         |
| [`acceptEdits`](#auto-approve-file-edits-with-acceptedits-mode)     | Чтение, редактирование файлов и распространённые команды файловой системы (`mkdir`, `touch`, `mv`, `cp` и т. д.) | Итеративная работа над кодом, который вы проверяете |
| [`plan`](#analyze-before-you-edit-with-plan-mode)                   | Только чтение                                                                                      | Изучение кодовой базы перед её изменением     |
| [`auto`](#eliminate-prompts-with-auto-mode)                         | Всё, с фоновыми проверками безопасности                                                            | Длительные задачи, снижение усталости от запросов |
| [`dontAsk`](#allow-only-pre-approved-tools-with-dontask-mode)       | Только предварительно одобренные инструменты                                                       | Закрытые CI и скрипты                         |
| [`bypassPermissions`](#skip-all-checks-with-bypasspermissions-mode) | Всё                                                                                                | Только изолированные контейнеры и виртуальные машины |

Во всех режимах, кроме `bypassPermissions`, запись в [защищённые пути](#protected-paths) никогда не одобряется автоматически, что защищает состояние репозитория и собственную конфигурацию Claude от случайного повреждения.

Режимы задают базовый уровень. Поверх них накладывайте [правила разрешений](https://code.claude.com/docs/en/permissions#manage-permissions), чтобы заранее одобрить или заблокировать конкретные инструменты. Правила запрета (deny) и явные правила запроса (ask) действуют в каждом режиме, включая `bypassPermissions`. Правила разрешения (allow) не действуют в этом режиме, потому что всё остальное уже одобрено.

## Переключение режимов разрешений

Вы можете переключать режимы в середине сессии, при запуске или в качестве постоянного значения по умолчанию. Режим задаётся через эти элементы управления, а не путём просьбы к Claude в чате. Выберите свой интерфейс ниже, чтобы узнать, как его изменить.

### CLI

**Во время сессии**: нажмите `Shift+Tab`, чтобы циклически переключаться `default` → `acceptEdits` → `plan`. Текущий режим отображается в строке состояния. Не каждый режим входит в цикл по умолчанию:

* `auto`: появляется, когда ваша учётная запись соответствует [требованиям режима auto](#eliminate-prompts-with-auto-mode); переключение на него меняет режимы без запроса подтверждения
* `bypassPermissions`: появляется после того, как вы запустили с `--permission-mode bypassPermissions`, `--dangerously-skip-permissions` или `--allow-dangerously-skip-permissions`; вариант `--allow-` добавляет режим в цикл, не активируя его
* `dontAsk`: никогда не появляется в цикле; задайте его с помощью `--permission-mode dontAsk`

Включённые необязательные режимы вставляются после `plan`, причём `bypassPermissions` идёт первым, а `auto` — последним. Если у вас включены оба, вы пройдёте через `bypassPermissions` на пути к `auto`.

**При запуске**: передайте режим как флаг.

```bash
claude --permission-mode plan
```

**В качестве значения по умолчанию**: задайте `defaultMode` в [настройках](https://code.claude.com/docs/en/settings#settings-files).

```json
{
  "permissions": {
    "defaultMode": "acceptEdits"
  }
}
```

Тот же флаг `--permission-mode` работает с `-p` для [неинтерактивных запусков](https://code.claude.com/docs/en/headless).

### VS Code

**Во время сессии**: щёлкните по индикатору режима внизу поля ввода промпта.

**В качестве значения по умолчанию**: задайте `claudeCode.initialPermissionMode` в настройках VS Code или используйте панель настроек расширения Claude Code.

Индикатор режима показывает следующие подписи, сопоставленные с режимом, который каждая из них применяет:

| Подпись в интерфейсе | Режим               |
| :------------------- | :------------------ |
| Ask before edits     | `default`           |
| Edit automatically   | `acceptEdits`       |
| Plan mode            | `plan`              |
| Auto mode            | `auto`              |
| Bypass permissions   | `bypassPermissions` |

Режим auto появляется в индикаторе режима, когда ваша учётная запись соответствует всем требованиям, перечисленным в [разделе о режиме auto](#eliminate-prompts-with-auto-mode). Настройка `claudeCode.initialPermissionMode` не принимает значение `auto`. Чтобы запускаться в режиме auto по умолчанию, вместо этого задайте `defaultMode` в ваших [пользовательских настройках](https://code.claude.com/docs/en/settings#settings-files). Claude Code игнорирует `defaultMode: "auto"` в проектных и локальных настройках.

Для режима Bypass permissions требуется переключатель **Allow dangerously skip permissions** в настройках расширения, прежде чем он появится в индикаторе режима.

Подробности, специфичные для расширения, см. в [руководстве по VS Code](https://code.claude.com/docs/en/vs-code).

### JetBrains

Плагин JetBrains запускает Claude Code в терминале IDE, поэтому переключение режимов работает так же, как в CLI: нажмите `Shift+Tab` для циклического переключения или передайте `--permission-mode` при запуске.

### Desktop

Используйте селектор режимов рядом с кнопкой отправки. Режимы Auto и Bypass permissions появляются только после того, как вы включите их в настройках Desktop. См. [руководство по Desktop](https://code.claude.com/docs/en/desktop#choose-a-permission-mode).

### Веб и мобильные устройства

Используйте раскрывающийся список режимов рядом с полем ввода промпта на [claude.ai/code](https://claude.ai/code) или в мобильном приложении. Запросы разрешений появляются в claude.ai для одобрения. Какие режимы отображаются, зависит от того, где выполняется сессия:

* **Облачные сессии** в [Claude Code в вебе](https://code.claude.com/docs/en/claude-code-on-the-web): Accept edits, Plan mode и Auto mode. Accept edits соответствует режиму `default`: облачная среда заранее одобряет редактирование файлов независимо от режима, поэтому в раскрывающемся списке отображается Accept edits вместо Ask permissions. `defaultMode: "acceptEdits"` из настроек по-прежнему учитывается. Режим Auto появляется только тогда, когда ваша организация разрешает его, а выбранная модель его поддерживает. Bypass permissions недоступен.
* **Сессии [Remote Control](https://code.claude.com/docs/en/remote-control)** на вашей локальной машине: Ask permissions, Auto accept edits и Plan mode. Режимы Auto и Bypass permissions недоступны.

Для Remote Control вы также можете задать начальный режим при запуске хоста:

```bash
claude remote-control --permission-mode acceptEdits
```

## Автоматическое одобрение редактирования файлов в режиме acceptEdits

Режим `acceptEdits` позволяет Claude создавать и редактировать файлы в вашем рабочем каталоге без запроса. Пока этот режим активен, в строке состояния отображается `⏵⏵ accept edits on`.

Помимо редактирования файлов, режим `acceptEdits` автоматически одобряет распространённые команды файловой системы Bash: `mkdir`, `touch`, `rm`, `rmdir`, `mv`, `cp` и `sed`. Эти команды также автоматически одобряются, когда им предшествуют безопасные переменные окружения, такие как `LANG=C` или `NO_COLOR=1`, либо обёртки процессов, такие как `timeout`, `nice` или `nohup`. Как и редактирование файлов, автоматическое одобрение применяется только к путям внутри вашего рабочего каталога или `additionalDirectories`. Пути за пределами этой области, запись в [защищённые пути](#protected-paths) и все прочие команды Bash по-прежнему требуют запроса.

Когда включён [инструмент PowerShell](https://code.claude.com/docs/en/tools-reference#powershell-tool), режим `acceptEdits` также автоматически одобряет `Set-Content`, `Add-Content`, `Clear-Content` и `Remove-Item` для путей в области действия, наряду с их распространёнными псевдонимами. Действуют те же правила области действия и защищённых путей.

Используйте `acceptEdits`, когда хотите проверять изменения в редакторе или через `git diff` постфактум, а не одобрять каждое редактирование по ходу дела. Нажмите `Shift+Tab` один раз из режима по умолчанию, чтобы войти в него, или запустите с ним сразу:

```bash
claude --permission-mode acceptEdits
```

## Анализ перед редактированием в режиме планирования

Режим планирования указывает Claude исследовать и предлагать изменения, не внося их. Claude читает файлы, выполняет команды оболочки для изучения и составляет план, но не редактирует ваш исходный код. Запросы разрешений по-прежнему применяются так же, как в режиме по умолчанию.

Войдите в режим планирования, нажав `Shift+Tab` или добавив к отдельному промпту префикс `/plan`. Вы также можете запуститься в режиме планирования из CLI:

```bash
claude --permission-mode plan
```

Нажмите `Shift+Tab` ещё раз, чтобы выйти из режима планирования, не одобряя план.

### Проверка и одобрение плана

Когда план готов, Claude представляет его и спрашивает, как действовать дальше. Из этого запроса вы можете:

* Одобрить и запуститься в режиме auto
* Одобрить и принимать редактирования
* Одобрить и проверять каждое редактирование вручную
* Продолжить планирование с обратной связью
* Доработать с помощью [Ultraplan](https://code.claude.com/docs/en/ultraplan) для проверки в браузере

Одобрение плана выводит из режима планирования и переключает сессию в тот режим разрешений, который описывает каждый вариант одобрения, так что Claude начинает редактирование. Чтобы снова спланировать, вернитесь в режим планирования с помощью `Shift+Tab` или добавьте к следующему промпту префикс `/plan`.

Нажмите `Ctrl+G`, чтобы открыть предложенный план в текстовом редакторе по умолчанию и отредактировать его напрямую, прежде чем Claude продолжит. Когда включён [`showClearContextOnPlanAccept`](https://code.claude.com/docs/en/settings#available-settings), каждый вариант одобрения также предлагает сначала очистить контекст планирования.

Принятие плана также автоматически именует сессию по содержимому плана, если только вы уже не задали имя с помощью `--name` или `/rename`.

### Установка режима планирования по умолчанию

Чтобы сделать режим планирования режимом по умолчанию для проекта, задайте `defaultMode` в `.claude/settings.json`:

```json
{
  "permissions": {
    "defaultMode": "plan"
  }
}
```

## Устранение запросов разрешений с помощью режима auto

> **Примечание.** Режим auto требует Claude Code v2.1.83 или новее.

Режим auto позволяет Claude выполнять действия без рутинных запросов разрешений. Отдельная модель-классификатор проверяет действия перед их выполнением, блокируя всё, что выходит за рамки вашего запроса, нацелено на нераспознанную инфраструктуру или выглядит как продиктованное враждебным содержимым, которое прочитал Claude. Явные [правила запроса (ask)](https://code.claude.com/docs/en/permissions#manage-permissions) по-прежнему принудительно вызывают запрос.

Режим auto также побуждает Claude продолжать работу, не останавливаясь для уточняющих вопросов, хотя Claude всё же задаёт их, когда ваш промпт или навык явно на это полагается. Для более выраженного автономного поведения с сохранением запросов разрешений вместо этого задайте [стиль вывода Proactive](https://code.claude.com/docs/en/output-styles).

> **Предупреждение.** Режим auto — это исследовательская предварительная версия (research preview). Он сокращает запросы разрешений, но не гарантирует безопасность. Используйте его для задач, в общем направлении которых вы уверены, а не как замену проверки для чувствительных операций.

Режим auto доступен только тогда, когда ваша учётная запись соответствует всем этим требованиям:

* **Тарифный план**: Все тарифные планы.
* **Владелец**: в тарифах Team и Enterprise Владелец должен включить его в [настройках администратора Claude Code](https://claude.ai/admin-settings/claude-code), прежде чем пользователи смогут его активировать. Администраторы также могут заблокировать его, задав `permissions.disableAutoMode` значение `"disable"` в [управляемых настройках](https://code.claude.com/docs/en/permissions#managed-settings).
* **Модель**: в Anthropic API — Claude Opus 4.6 или новее, либо Sonnet 4.6 или новее. В Amazon Bedrock, Google Cloud Vertex AI, Microsoft Foundry и сессиях с входом через [Claude apps gateway](https://code.claude.com/docs/en/claude-apps-gateway) — только Claude Sonnet 5, Opus 4.7 и Opus 4.8. Более старые модели, включая Sonnet 4.5, Opus 4.5, Haiku и модели claude-3, не поддерживаются ни на одном провайдере.
* **Провайдер**: доступен по умолчанию в Anthropic API. В Amazon Bedrock, Google Cloud Vertex AI, Microsoft Foundry и сессиях с входом через Claude apps gateway режим auto выключен, пока вы не [зададите `CLAUDE_CODE_ENABLE_AUTO_MODE`](#enable-auto-mode-on-bedrock-vertex-ai-or-foundry).

Если Claude Code сообщает, что режим auto недоступен, одно из этих требований не выполнено; это не временный сбой. Отдельное сообщение, называющее модель и говорящее, что режим auto «не может определить безопасность» действия, — это временный сбой классификатора; см. [справочник по ошибкам](https://code.claude.com/docs/en/errors#auto-mode-cannot-determine-the-safety-of-an-action).

Если вы задали `defaultMode: "auto"` в [настройках](https://code.claude.com/docs/en/settings#available-settings), а сессия запускается в режиме `default` без ошибки, эта настройка, вероятно, находится в `.claude/settings.json` или `.claude/settings.local.json`. Claude Code v2.1.142 и новее игнорируют `auto` из этих файлов, чтобы репозиторий не мог сам предоставить себе режим auto. Переместите её в `~/.claude/settings.json`.

### Включение режима auto в Bedrock, Vertex AI или Foundry

В [Amazon Bedrock](https://code.claude.com/docs/en/amazon-bedrock), [Google Cloud Vertex AI](https://code.claude.com/docs/en/google-vertex-ai), [Microsoft Foundry](https://code.claude.com/docs/en/microsoft-foundry) и сессиях с входом через [Claude apps gateway](https://code.claude.com/docs/en/claude-apps-gateway) режим auto не появляется в цикле `Shift+Tab`, пока `CLAUDE_CODE_ENABLE_AUTO_MODE` не установлен в `1`. Переменная работает в Claude Code v2.1.158 и новее. На этих провайдерах поддерживаются только Claude Sonnet 5, Opus 4.7 и Opus 4.8.

Чтобы включить его для одного разработчика, добавьте переменную в блок `env` в `~/.claude/settings.json`:

```json
{
  "env": {
    "CLAUDE_CODE_ENABLE_AUTO_MODE": "1"
  }
}
```

Чтобы включить его для вашей организации, добавьте тот же блок `env` в [управляемые настройки](https://code.claude.com/docs/en/settings#settings-files).

После установки переменной режим auto появляется в цикле `Shift+Tab` для каждой сессии. Чтобы сделать его начальным режимом по умолчанию, также задайте `"permissions": {"defaultMode": "auto"}` в пользовательских или управляемых настройках. На этих провайдерах Claude Code игнорирует `defaultMode: "auto"`, если также не задан `CLAUDE_CODE_ENABLE_AUTO_MODE`.

Чтобы запретить разработчикам включать режим auto, задайте `disableAutoMode` значение `"disable"` в управляемых настройках. Это переопределяет переменную включения.

Если вы подключаетесь через [шлюз LLM](https://code.claude.com/docs/en/llm-gateway), настроенный с `ANTHROPIC_BASE_URL`, режим auto может быть уже доступен без переменной включения, потому что шлюз маршрутизирует запросы через Anthropic API. Это не относится к сессии с входом через [Claude apps gateway](https://code.claude.com/docs/en/claude-apps-gateway), которая является отдельным классом провайдера и требует переменной включения. Настройка `disableAutoMode` применяется одинаково в обеих конфигурациях.

### Что классификатор блокирует по умолчанию

Классификатор доверяет вашему рабочему каталогу и настроенным удалённым репозиториям (remotes) вашего репозитория. Всё остальное считается внешним, пока вы не [настроите доверенную инфраструктуру](https://code.claude.com/docs/en/auto-mode-config).

**Заблокировано по умолчанию**:

* Загрузка и выполнение кода, например `curl | bash`
* Отправка чувствительных данных на внешние конечные точки
* Развёртывания и миграции в продакшене
* Массовое удаление в облачном хранилище
* Предоставление разрешений IAM или репозитория
* Изменение общей инфраструктуры
* Необратимое уничтожение файлов, существовавших до сессии
* Принудительная отправка (force push) или отправка напрямую в `main`
* `git reset --hard`, `git checkout -- .`, `git restore .`, `git clean -fd`, `git stash drop` или `git stash clear`, которые, по предположению классификатора, отбросили бы незакоммиченные изменения
* `git commit --amend`, когда коммит на HEAD не был создан в этой сессии
* `terraform destroy`, `pulumi destroy`, `cdk destroy` или `terragrunt destroy`, а также применение плана, который уничтожает ресурсы

Claude Code v2.1.195 и новее блокируют по умолчанию больше категорий. Некоторые из них зависят от записей [environment](https://code.claude.com/docs/en/auto-mode-config#define-trusted-infrastructure), таких как чувствительные удалённые цели и защищённые области IaC, которые вы можете сузить до конкретных имён.

* Запись в менеджер секретов или изменение записей DNS либо сертификатов TLS
* Слияние pull request, который не одобрил ни один человек, одобрение собственного pull request Claude или отключение проверок CI
* Публикация комментария, который сам по себе является командой для автоматизации, например `atlantis apply` или бота `/deploy` либо `/merge`
* Переключение, поэтапное развёртывание или удаление продакшен-флага функциональности (feature flag)
* Применение изменений инфраструктуры к защищённой области IaC либо вывод из эксплуатации (drain) и удаление узлов кластера
* Запись в общий вычислительный кластер, выходящая за пределы названного вами ресурса, например селектор меток или `--all`, который затрагивает задания других пользователей
* Создание ресурсов Kubernetes, которые работают на каждом узле или перехватывают трафик кластера, таких как DaemonSets и admission webhooks
* Интерактивные оболочки или перенаправления портов (port-forward) в чувствительную удалённую цель
* Открытие туннеля или обратной оболочки (reverse shell), делающих локальную службу доступной из публичного интернета
* Вывод действующих учётных данных или токена в транскрипт или файл
* Доступ к местоположению с PII или регулируемыми данными либо копирование данных из него
* Маршрутизация установки пакета в обход вашего внутреннего реестра пакетов к публичному реестру
* Выполнение команды с флагом, который отключает защитный механизм, например `--insecure`
* Действия браузера [Claude в Chrome](https://code.claude.com/docs/en/chrome), которые могли бы отправить содержимое страницы, файлы cookie или учётные данные за пределы источника (off-origin)

**Разрешено по умолчанию**:

* Локальные операции с файлами в вашем рабочем каталоге
* Установка зависимостей, объявленных в ваших lock-файлах или манифестах
* Чтение `.env` и отправка учётных данных в соответствующий им API
* HTTP-запросы только для чтения
* Отправка в ветку, с которой вы начали, или в ветку, созданную Claude

Claude Code v2.1.195 и новее также разрешают следующее по умолчанию:

* Удаление именно тех заданий, которые Claude создал ранее в той же сессии
* Чтение, проверка или написание связанного с безопасностью кода, конфигураций и моделей угроз в рамках вашей задачи
* Сообщения между агентами, работающими вместе в одной мультиагентной сессии
* Отправка данных на доверенные домены, в бакеты и службы, которые вы перечислили в [`environment`](https://code.claude.com/docs/en/auto-mode-config#define-trusted-infrastructure). Это касается только потока данных, а не разрушительных операций или операций с учётными данными на той же инфраструктуре
* Переход [Claude в Chrome](https://code.claude.com/docs/en/chrome) на доверенный внутренний домен, localhost или названный вами URL

Запросы сетевого доступа из песочницы маршрутизируются через классификатор, а не разрешаются по умолчанию. Выполните `claude auto-mode defaults`, чтобы увидеть полные списки правил. Если рутинные действия блокируются, администратор может добавить доверенные репозитории, бакеты и службы через настройку `autoMode.environment`: см. [Настройка режима auto](https://code.claude.com/docs/en/auto-mode-config).

### Границы, которые вы задаёте в разговоре

Классификатор трактует границы, которые вы задаёте в разговоре, как сигнал к блокировке. Если вы говорите Claude «не отправляй (don't push)» или «подожди, пока я проверю, прежде чем развёртывать», классификатор блокирует соответствующие действия, даже когда правила по умолчанию их разрешили бы. Граница остаётся в силе, пока вы не снимете её в последующем сообщении. Собственное суждение Claude о том, что условие выполнено, не снимает её.

Границы не хранятся как правила. Классификатор перечитывает их из транскрипта при каждой проверке, поэтому граница может быть потеряна, если [уплотнение контекста](https://code.claude.com/docs/en/costs#reduce-token-usage) удалит сообщение, которое её задало. Для твёрдой гарантии вместо этого добавьте [правило запрета (deny)](https://code.claude.com/docs/en/permissions#permission-rule-syntax).

### Когда режим auto откатывается

Каждое отклонённое действие показывает уведомление и появляется в `/permissions` на вкладке Recently denied, где вы можете нажать `r`, чтобы повторить его с ручным одобрением.

Если классификатор блокирует действие 3 раза подряд или 20 раз в общей сложности, режим auto приостанавливается, и Claude Code возобновляет запросы. Одобрение запрошенного действия возобновляет режим auto. Эти пороги не настраиваются. Любое разрешённое действие сбрасывает счётчик подряд идущих блокировок, тогда как общий счётчик сохраняется на протяжении сессии и сбрасывается только тогда, когда его собственный лимит вызывает откат.

В [неинтерактивном режиме](https://code.claude.com/docs/en/headless) с флагом `-p` повторные блокировки прерывают сессию, поскольку некому отправить запрос.

Повторные блокировки обычно означают, что классификатору не хватает контекста о вашей инфраструктуре. Используйте `/feedback`, чтобы сообщить о ложных срабатываниях, или попросите администратора [настроить доверенную инфраструктуру](https://code.claude.com/docs/en/auto-mode-config).

### Как классификатор оценивает действия

Каждое действие проходит через фиксированный порядок принятия решений. Побеждает первый подходящий шаг:

1. Действия, соответствующие вашим [правилам разрешения или запрета (allow или deny)](https://code.claude.com/docs/en/permissions#manage-permissions), разрешаются немедленно, за исключением записи в [защищённые пути](#protected-paths), которая маршрутизируется к классификатору, даже когда подходит правило разрешения
2. Действия только для чтения и редактирование файлов в вашем рабочем каталоге одобряются автоматически, за исключением записи в [защищённые пути](#protected-paths)
3. Всё остальное отправляется к классификатору
4. Если классификатор блокирует, Claude получает причину и пробует альтернативу

При входе в режим auto широкие правила разрешения, дающие возможность выполнять произвольный код, отбрасываются:

* Всеохватывающие `Bash(*)` или `PowerShell(*)`
* Интерпретаторы с подстановочными знаками, такие как `Bash(python*)`
* Команды запуска менеджеров пакетов
* Правила разрешения `Agent`

Узкие правила, такие как `Bash(npm test)`, сохраняются. Отброшенные правила восстанавливаются, когда вы выходите из режима auto.

Классификатор видит сообщения пользователя, вызовы инструментов и содержимое вашего CLAUDE.md. Результаты инструментов удаляются, поэтому враждебное содержимое в файле или веб-странице не может напрямую им манипулировать. Отдельный серверный зонд сканирует входящие результаты инструментов и помечает подозрительное содержимое, прежде чем Claude его прочитает. Подробнее о том, как эти уровни работают вместе, см. [анонс режима auto](https://claude.com/blog/auto-mode) и [подробный технический разбор](https://www.anthropic.com/engineering/claude-code-auto-mode).

### Как режим auto обрабатывает субагентов

Классификатор проверяет работу [субагентов](https://code.claude.com/docs/en/sub-agents) в трёх точках:

1. Перед запуском субагента оценивается описание делегированной задачи, так что опасно выглядящая задача блокируется на этапе порождения.
2. Пока субагент выполняется, каждое его действие проходит через классификатор с теми же правилами, что и у родительской сессии, а любой `permissionMode` во фронтматтере субагента игнорируется.
3. Когда субагент завершает работу, классификатор проверяет всю историю его действий; если эта завершающая проверка отмечает проблему, к результатам субагента добавляется предупреждение о безопасности.

Шаг 1 требует Claude Code v2.1.178 или новее. Более ранние версии применяли классификатор на шагах 2 и 3, но не оценивали описание задачи перед запуском субагента.

### Стоимость и задержка

Классификатор работает на настроенной на сервере модели, которая не зависит от вашего выбора `/model`, поэтому переключение моделей не меняет доступность классификатора. Вызовы классификатора учитываются в вашем расходе токенов. Каждая проверка отправляет часть транскрипта плюс ожидающее действие, добавляя один сетевой обмен перед выполнением. Чтения и редактирование в рабочем каталоге вне защищённых путей пропускают классификатор, поэтому накладные расходы возникают в основном из-за команд оболочки и сетевых операций.

## Разрешение только предварительно одобренных инструментов в режиме dontAsk

Режим `dontAsk` автоматически отклоняет каждый вызов инструмента, который иначе вызвал бы запрос. Выполняться могут только действия, соответствующие вашим правилам `permissions.allow`, и [команды Bash только для чтения](https://code.claude.com/docs/en/permissions#read-only-commands); явные [правила `ask`](https://code.claude.com/docs/en/permissions#manage-permissions) отклоняются, а не вызывают запрос. Это делает режим полностью неинтерактивным для конвейеров CI или ограниченных сред, где вы заранее определяете точно, что Claude может делать. Облачные сессии в [Claude Code в вебе](https://code.claude.com/docs/en/claude-code-on-the-web) игнорируют `defaultMode: "dontAsk"`; подробности см. в [bypassPermissions](#skip-all-checks-with-bypasspermissions-mode).

Задайте его при запуске с помощью флага:

```bash
claude --permission-mode dontAsk
```

## Пропуск всех проверок в режиме bypassPermissions

Режим `bypassPermissions` отключает запросы разрешений и проверки безопасности, так что вызовы инструментов выполняются немедленно. Начиная с v2.1.126 это включает запись в [защищённые пути](#protected-paths), для которой более ранние версии всё ещё запрашивали разрешение. Явные [правила запроса (ask)](https://code.claude.com/docs/en/permissions#manage-permissions) по-прежнему принудительно вызывают запрос в этом режиме, а удаления, нацеленные на корень файловой системы или домашний каталог, такие как `rm -rf /` и `rm -rf ~`, по-прежнему вызывают запрос как предохранитель против ошибки модели. Используйте этот режим только в изолированных средах, таких как контейнеры, виртуальные машины или dev-контейнеры без доступа в интернет, где Claude Code не может повредить вашу хост-систему.

Вы не можете войти в `bypassPermissions` из сессии, которая была запущена без одного из включающих флагов; перезапустите с ним, чтобы включить его:

```bash
claude --permission-mode bypassPermissions
```

Флаг `--dangerously-skip-permissions` эквивалентен.

В Linux и macOS Claude Code отказывается запускаться в этом режиме при работе от имени root или под `sudo`:

```text
--dangerously-skip-permissions cannot be used with root/sudo privileges for security reasons
```

Проверка автоматически пропускается внутри распознанной песочницы. Чтобы работать автономно в контейнере, используйте конфигурацию [dev-контейнера](https://code.claude.com/docs/en/devcontainer), которая запускает Claude Code от имени пользователя без прав root.

[Claude Code в вебе](https://code.claude.com/docs/en/claude-code-on-the-web) не учитывает `defaultMode: "bypassPermissions"` или `"dontAsk"` из ваших файлов настроек, поэтому зафиксированные в репозитории настройки не могут запустить облачную сессию в режиме обхода разрешений. Настройка молча игнорируется, и сессия вместо этого запускается в режиме, показанном в раскрывающемся списке режимов. О том, какие режимы предлагают облачные сессии, см. [Переключение режимов разрешений](#switch-permission-modes).

> **Предупреждение.** `bypassPermissions` не обеспечивает никакой защиты от инъекции промптов или непреднамеренных действий. Для фоновых проверок безопасности со значительно меньшим числом запросов разрешений вместо этого используйте [режим auto](#eliminate-prompts-with-auto-mode). Администраторы могут заблокировать этот режим, задав `permissions.disableBypassPermissionsMode` значение `"disable"` в [управляемых настройках](https://code.claude.com/docs/en/permissions#managed-settings).

## Защищённые пути

Запись в небольшой набор путей никогда не одобряется автоматически во всех режимах, кроме `bypassPermissions`. Это предотвращает случайное повреждение состояния репозитория и собственной конфигурации Claude.

| Режим                            | Запись в защищённые пути         |
| :------------------------------- | :------------------------------- |
| `default`, `acceptEdits`, `plan` | Запрос                           |
| `auto`                           | Маршрутизируется к классификатору |
| `dontAsk`                        | Отклоняется                      |
| `bypassPermissions`              | Разрешено                        |

Правила [`permissions.allow`](https://code.claude.com/docs/en/permissions#manage-permissions) в файлах настроек не одобряют заранее запись в защищённые пути. Проверка безопасности выполняется до того, как Claude Code оценивает правила разрешения из настроек, поэтому запись вида `Edit(.claude/**)` в `~/.claude/settings.json` или `.claude/settings.json` не меняет результат для каждого режима в таблице выше. В режимах, которые запрашивают, запрос на запись в `.claude/` предлагает вариант **Yes, and allow Claude to edit its own settings for this session**, который одобряет последующие записи в `.claude/` в этой сессии без повторного запроса.

Защищённые каталоги:

* `.git`
* `.config/git`
* `.vscode`
* `.idea`
* `.husky`
* `.cargo`
* `.devcontainer`
* `.yarn`
* `.mvn`
* `.claude`, за исключением `.claude/worktrees`, где Claude хранит собственные git-worktree

Защищённые файлы:

* `.gitconfig`, `.gitmodules`
* `.bashrc`, `.bash_profile`, `.bash_login`, `.bash_aliases`, `.bash_logout`, `.zshrc`, `.zprofile`, `.zshenv`, `.zlogin`, `.zlogout`, `.profile`, `.envrc`
* `.npmrc`, `.yarnrc`, `.yarnrc.yml`, `.pnp.cjs`, `.pnp.loader.mjs`, `.pnpmfile.cjs`, `bunfig.toml`, `.bunfig.toml`
* `.bazelrc`, `.bazelversion`, `.bazeliskrc`
* `.pre-commit-config.yaml`, `lefthook.yml`, `lefthook.yaml`, `.lefthook.yml`, `.lefthook.yaml`
* `gradle-wrapper.properties`, `maven-wrapper.properties`
* `.devcontainer.json`
* `.ripgreprc`, `pyrightconfig.json`
* `.mcp.json`, `.claude.json`
