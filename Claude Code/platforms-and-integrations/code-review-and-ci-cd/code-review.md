# Обзор кода

> Настройте автоматические обзоры PR, которые выявляют логические ошибки, уязвимости безопасности и регрессии с помощью многоагентного анализа всей вашей кодовой базы

> **Примечание.** Обзор кода находится в статусе research preview и доступен для подписок [Team и Enterprise](https://claude.ai/admin-settings/claude-code). Он недоступен для организаций с включённым [Zero Data Retention](https://code.claude.com/docs/en/zero-data-retention).

Обзор кода анализирует ваши pull request в GitHub и публикует найденные проблемы в виде встроенных комментариев к тем строкам кода, где были обнаружены проблемы. Группа специализированных агентов изучает изменения в коде в контексте всей вашей кодовой базы, отыскивая логические ошибки, уязвимости безопасности, сломанные граничные случаи и неочевидные регрессии.

Найденные проблемы помечаются по степени серьёзности и не одобряют и не блокируют ваш PR, поэтому существующие рабочие процессы обзора остаются нетронутыми. Вы можете настроить, что именно отмечает Claude, добавив в репозиторий файл `CLAUDE.md` или `REVIEW.md`.

Чтобы запускать Claude в вашей собственной CI-инфраструктуре вместо этого управляемого сервиса, см. [GitHub Actions](https://code.claude.com/docs/en/github-actions) или [GitLab CI/CD](https://code.claude.com/docs/en/gitlab-ci-cd). Для репозиториев на самостоятельно размещённом инстансе GitHub см. [GitHub Enterprise Server](https://code.claude.com/docs/en/github-enterprise-server).

## Как работают обзоры

После того как Владелец (Owner) [включит Обзор кода](#set-up-code-review) для вашей организации, обзоры запускаются при открытии PR, при каждом push или по ручному запросу — в зависимости от настроенного поведения репозитория. Комментарий `@claude review` [запускает обзоры на PR](#manually-trigger-reviews) в любом режиме.

Когда обзор запускается, несколько агентов параллельно анализируют дифф и окружающий код на инфраструктуре Anthropic. Каждый агент ищет свой класс проблем, затем этап верификации сверяет кандидатов с фактическим поведением кода, чтобы отфильтровать ложные срабатывания. Результаты дедуплицируются, ранжируются по степени серьёзности и публикуются в виде встроенных комментариев к конкретным строкам, где были найдены проблемы, с общей сводкой в теле обзора. Если проблем не найдено, Обзор кода обновляет check run в GitHub, показывая, что проблем не обнаружено. Claude также может опубликовать короткий подтверждающий комментарий к PR.

Стоимость обзоров масштабируется с размером и сложностью PR, а сами обзоры в среднем завершаются за 20 минут. Владельцы могут отслеживать активность обзоров и расходы через [панель аналитики](#view-usage).

### Уровни серьёзности

Каждая найденная проблема помечается уровнем серьёзности:

| Маркер | Серьёзность    | Значение                                                                       |
| :----- | :------------- | :----------------------------------------------------------------------------- |
| 🔴     | Important      | Баг, который следует исправить до слияния                                       |
| 🟡     | Nit            | Незначительная проблема, которую стоит исправить, но она не блокирует           |
| 🟣     | Pre-existing   | Баг, который уже существует в кодовой базе, но не был внесён этим PR             |

Найденные проблемы включают сворачиваемый раздел с расширенными рассуждениями, который можно развернуть, чтобы понять, почему Claude отметил проблему и как он её проверил.

### Оценка найденных проблем и ответы на них

Каждый комментарий обзора от Claude приходит с уже прикреплёнными 👍 и 👎, так что обе кнопки отображаются в интерфейсе GitHub для оценки в один клик. Нажмите 👍, если найденная проблема была полезной, или 👎, если она была ошибочной или лишней. Anthropic собирает количество реакций после слияния PR и использует их для настройки рецензента. Реакции не запускают повторный обзор и ничего не меняют в PR.

Ответ на встроенный комментарий не побуждает Claude отвечать или обновлять PR. Чтобы отреагировать на найденную проблему, исправьте код и сделайте push. Если PR подписан на обзоры по push, следующий запуск закроет тред, когда проблема будет исправлена. Чтобы запросить свежий обзор без push, оставьте комментарий `@claude review once` в виде [комментария верхнего уровня к PR](#manually-trigger-reviews).

### Вывод check run

Помимо встроенных комментариев обзора, каждый обзор заполняет check run **Claude Code Review**, который появляется рядом с вашими CI-проверками. Разверните его ссылку **Details**, чтобы увидеть сводку по каждой найденной проблеме в одном месте, отсортированную по серьёзности:

| Серьёзность  | Файл:Строка               | Проблема                                                                          |
| ------------ | ------------------------- | -------------------------------------------------------------------------------- |
| 🔴 Important | `src/auth/session.ts:142` | Обновление токена конкурирует с выходом из системы, оставляя устаревшие сессии активными |
| 🟡 Nit       | `src/auth/session.ts:88`  | `parseExpiry` молча возвращает 0 при некорректном вводе                           |

Каждая найденная проблема также появляется как аннотация во вкладке **Files changed**, отмеченная непосредственно на соответствующих строках диффа. Проблемы Important отображаются с красным маркером, nit — с жёлтым предупреждением, а уже существующие баги — с серым уведомлением. Аннотации и таблица серьёзности записываются в check run независимо от встроенных комментариев обзора, поэтому они остаются доступными, даже если GitHub отклонит встроенный комментарий к строке, которая сместилась.

Check run всегда завершается с нейтральным заключением, так что он никогда не блокирует слияние через правила защиты веток. Если вы хотите ограничивать слияния на основе найденных Обзором кода проблем, считывайте разбивку по серьёзности из вывода check run в вашей собственной CI. Последняя строка текста Details — это машиночитаемый комментарий, который ваш рабочий процесс может разобрать с помощью `gh` и jq:

```bash
gh api repos/OWNER/REPO/check-runs/CHECK_RUN_ID \
  --jq '.output.text | split("bughunter-severity: ")[1] | split(" -->")[0] | fromjson'
```

Это возвращает JSON-объект с количеством по каждому уровню серьёзности, например `{"normal": 2, "nit": 1, "pre_existing": 0}`. Ключ `normal` содержит количество найденных проблем Important; ненулевое значение означает, что Claude нашёл хотя бы один баг, который стоит исправить до слияния.

### Что проверяет Обзор кода

По умолчанию Обзор кода сосредоточен на корректности: багах, которые сломали бы продакшн, а не на предпочтениях форматирования или отсутствующем покрытии тестами. Вы можете расширить то, что он проверяет, [добавив файлы с указаниями](#customize-reviews) в ваш репозиторий.

## Настройка Обзора кода

Владелец включает Обзор кода один раз для организации и выбирает, какие репозитории включить.

1. **Откройте настройки администратора Claude Code**

   Перейдите на [claude.ai/admin-settings/claude-code](https://claude.ai/admin-settings/claude-code) и найдите раздел Code Review. Вам нужна роль Owner или Primary Owner в вашей организации Claude и разрешение на установку GitHub Apps в вашей организации GitHub.

2. **Начните настройку**

   Нажмите **Setup**. Это запускает процесс установки GitHub App.

3. **Установите Claude GitHub App**

   Следуйте подсказкам, чтобы установить Claude GitHub App в вашу организацию GitHub. Приложение запрашивает следующие разрешения на репозиторий:

   * **Contents**: чтение и запись
   * **Issues**: чтение и запись
   * **Pull requests**: чтение и запись

   Обзор кода использует доступ на чтение к contents и доступ на запись к pull requests. Более широкий набор разрешений также поддерживает [GitHub Actions](https://code.claude.com/docs/en/github-actions), если вы включите это позже.

4. **Выберите репозитории**

   Выберите, какие репозитории включить для Обзора кода. Если вы не видите репозиторий, убедитесь, что вы предоставили Claude GitHub App доступ к нему во время установки. Вы можете добавить больше репозиториев позже.

5. **Настройте триггеры обзора для каждого репозитория**

   После завершения настройки в разделе Code Review отображаются ваши репозитории в виде таблицы. Для каждого репозитория используйте выпадающий список **Review Behavior**, чтобы выбрать, когда запускаются обзоры:

   * **Once after PR creation**: обзор запускается один раз при открытии PR или пометке его готовым к обзору
   * **After every push**: обзор запускается при каждом push в ветку PR, выявляя новые проблемы по мере развития PR и автоматически закрывая треды, когда вы исправляете отмеченные проблемы
   * **Manual**: обзоры запускаются только тогда, когда кто-то [оставляет комментарий `@claude review` или `@claude review once` на PR](#manually-trigger-reviews); `@claude review` также подписывает PR на обзоры при последующих push

   Обзор при каждом push запускает наибольшее количество обзоров и стоит дороже всего. Ручной режим полезен для репозиториев с высоким трафиком, где вы хотите подключать к обзору конкретные PR, или чтобы начинать обзор ваших PR только когда они готовы.

Таблица репозиториев также показывает среднюю стоимость обзора для каждого репозитория на основе недавней активности. Используйте меню действий строки, чтобы включить или выключить Обзор кода для каждого репозитория или полностью удалить репозиторий.

Чтобы проверить настройку, откройте тестовый PR. Если вы выбрали автоматический триггер, в течение нескольких минут появится check run с названием **Claude Code Review**. Если вы выбрали Manual, оставьте комментарий `@claude review` на PR, чтобы запустить первый обзор. Если check run не появляется, убедитесь, что репозиторий указан в ваших настройках администратора и что Claude GitHub App имеет к нему доступ.

## Ручной запуск обзоров

Две команды в комментариях запускают обзор по требованию. Обе работают независимо от настроенного триггера репозитория, поэтому вы можете использовать их, чтобы подключать конкретные PR к обзору в режиме Manual или получать немедленный повторный обзор в других режимах.

| Команда               | Что она делает                                                                          |
| :-------------------- | :-------------------------------------------------------------------------------------- |
| `@claude review`      | Запускает обзор и подписывает PR на обзоры по push в дальнейшем                          |
| `@claude review once` | Запускает единичный обзор без подписки PR на будущие push                                |

Используйте `@claude review once`, когда вы хотите получить отзыв о текущем состоянии PR, но не хотите, чтобы каждый последующий push влёк за собой обзор. Это полезно для долгоживущих PR с частыми push или когда вам нужно разовое второе мнение без изменения поведения обзора для PR.

Чтобы любая из команд запустила обзор:

* Опубликуйте её как комментарий верхнего уровня к PR, а не как встроенный комментарий к строке диффа
* Поместите команду в начало комментария, с `once` на той же строке, если вы используете разовую форму
* У вас должен быть доступ уровня owner, member или collaborator к репозиторию
* PR должен быть открыт

В отличие от автоматических триггеров, ручные триггеры срабатывают на черновых (draft) PR, поскольку явный запрос сигнализирует, что вы хотите обзор сейчас, независимо от статуса черновика.

Если обзор уже выполняется на этом PR, запрос ставится в очередь до завершения выполняющегося обзора. Вы можете отслеживать ход выполнения через check run на PR.

## Настройка обзоров

Обзор кода читает два файла из вашего репозитория, чтобы определить, что именно отмечать. Они различаются тем, насколько сильно влияют на обзор:

* **`CLAUDE.md`**: общие инструкции проекта, которые Claude Code использует для всех задач, а не только для обзоров. Обзор кода читает его как контекст проекта и отмечает вновь внесённые нарушения как nit.
* **`REVIEW.md`**: инструкции только для обзора, внедряемые непосредственно в каждый агент конвейера обзора с наивысшим приоритетом. Используйте его, чтобы изменить, что отмечается, с какой серьёзностью и как сообщается о найденных проблемах.

### CLAUDE.md

Обзор кода читает файлы `CLAUDE.md` вашего репозитория и рассматривает вновь внесённые нарушения как найденные проблемы уровня [nit](#severity-levels). Это работает в обе стороны: если ваш PR изменяет код так, что утверждение в `CLAUDE.md` становится устаревшим, Claude также отметит, что документацию нужно обновить.

Claude читает файлы `CLAUDE.md` на каждом уровне иерархии ваших каталогов, поэтому правила в `CLAUDE.md` подкаталога применяются только к файлам по этому пути. См. [документацию по памяти](https://code.claude.com/docs/en/memory) для получения дополнительной информации о том, как работает `CLAUDE.md`.

Для указаний, специфичных для обзора, которые вы не хотите применять к обычным сессиям Claude Code, используйте вместо этого [`REVIEW.md`](#review-md).

### REVIEW.md

`REVIEW.md` — это файл в корне вашего репозитория, который переопределяет поведение Обзора кода в вашем репозитории. Его содержимое внедряется в системный промпт каждого агента конвейера обзора как блок инструкций наивысшего приоритета, имеющий приоритет над указаниями обзора по умолчанию.

Поскольку он вставляется дословно, `REVIEW.md` — это обычные инструкции: [синтаксис импорта `@`](https://code.claude.com/docs/en/memory#import-additional-files) не разворачивается, а упоминаемые файлы не считываются в промпт. Поместите правила, которые вы хотите принудительно применять, непосредственно в файл.

#### Что можно настроить

`REVIEW.md` — это свободный markdown, поэтому всё, что вы можете выразить как инструкцию обзора, попадает в область применения. Приведённые ниже шаблоны на практике оказывают наибольшее влияние.

**Серьёзность**: переопределите, что означает 🔴 Important для вашего репозитория. Стандартная калибровка ориентирована на продакшн-код; репозиторий документации, репозиторий конфигурации или прототип могут захотеть гораздо более узкое определение. Явно укажите, какие классы найденных проблем являются Important, а какие — не более чем Nit. Вы также можете повысить в другую сторону, например рассматривать любое нарушение `CLAUDE.md` как Important, а не как nit по умолчанию.

**Объём nit**: ограничьте, сколько комментариев 🟡 Nit публикует один обзор. Текст и файлы конфигурации можно шлифовать бесконечно. Ограничение вроде «сообщать не более пяти nit, остальные упоминать числом в сводке» сохраняет обзоры действенными.

**Правила пропуска**: перечислите пути, шаблоны веток и категории найденных проблем, где Claude не должен публиковать никаких проблем. Обычные кандидаты — сгенерированный код, lockfiles, вендорные зависимости и ветки, созданные машинами, а также всё, что ваша CI уже проверяет, например линтинг или проверку орфографии. Для путей, которые заслуживают некоторого обзора, но не полной тщательности, установите более высокую планку вместо полного пропуска: «в `scripts/` сообщать только если почти наверняка и серьёзно».

**Проверки, специфичные для репозитория**: добавьте правила, которые вы хотите отмечать на каждом PR, например «новые API-маршруты должны иметь интеграционный тест». Поскольку `REVIEW.md` внедряется с наивысшим приоритетом, они срабатывают надёжнее, чем те же правила в длинном `CLAUDE.md`.

**Планка верификации**: требуйте доказательств, прежде чем публиковать определённый класс найденных проблем. Например, «заявления о поведении требуют цитаты `file:line` в исходном коде, а не вывода из именования» сокращает ложные срабатывания, которые иначе стоили бы автору лишнего цикла.

**Сходимость повторных обзоров**: укажите Claude, как вести себя, когда PR уже был проверен. Правило вроде «после первого обзора подавлять новые nit и публиковать только проблемы Important» не даёт однострочному исправлению дойти до седьмого раунда из-за одного лишь стиля.

**Форма сводки**: попросите, чтобы тело обзора открывалось однострочным подсчётом вроде `2 factual, 4 style` и начиналось с «no factual issues», когда это так. Автор хочет знать форму работы прежде деталей.

#### Пример

Этот `REVIEW.md` перекалибровывает серьёзность для бэкенд-сервиса, ограничивает nit, пропускает сгенерированные файлы и добавляет проверки, специфичные для репозитория.

```markdown
# Review instructions

## What Important means here

Reserve Important for findings that would break behavior, leak data,
or block a rollback: incorrect logic, unscoped database queries, PII
in logs or error messages, and migrations that aren't backward
compatible. Style, naming, and refactoring suggestions are Nit at
most.

## Cap the nits

Report at most five Nits per review. If you found more, say "plus N
similar items" in the summary instead of posting them inline. If
everything you found is a Nit, lead the summary with "No blocking
issues."

## Do not report

- Anything CI already enforces: lint, formatting, type errors
- Generated files under `src/gen/` and any `*.lock` file
- Test-only code that intentionally violates production rules

## Always check

- New API routes have an integration test
- Log lines don't include email addresses, user IDs, or request bodies
- Database queries are scoped to the caller's tenant
```

#### Держите его сфокусированным

Длина имеет цену: длинный `REVIEW.md` размывает наиболее важные правила. Ограничьте его инструкциями, которые меняют поведение обзора, а общий контекст проекта оставьте в `CLAUDE.md`.

## Просмотр использования

Перейдите на [claude.ai/analytics/code-review](https://claude.ai/analytics/code-review), чтобы увидеть активность Обзора кода по всей вашей организации. Панель показывает:

| Раздел               | Что она показывает                                                                            |
| :------------------- | :------------------------------------------------------------------------------------------- |
| PRs reviewed         | Ежедневное количество проверенных pull request за выбранный временной диапазон                |
| Cost weekly          | Еженедельные расходы на Обзор кода                                                            |
| Feedback             | Количество комментариев обзора, которые были автоматически закрыты, потому что разработчик устранил проблему |
| Repository breakdown | Количество проверенных PR и закрытых комментариев по каждому репозиторию                      |

Таблица репозиториев в настройках администратора также показывает среднюю стоимость обзора для каждого репозитория. Показатели стоимости на панели являются оценками для мониторинга активности; для точных расходов, соответствующих счёту, обращайтесь к вашему счёту Anthropic.

## Цены

Обзор кода оплачивается на основе использования токенов. Каждый обзор стоит в среднем \$15-25, масштабируясь с размером PR, сложностью кодовой базы и тем, сколько проблем требует верификации. Использование Обзора кода оплачивается отдельно через [кредиты использования](https://support.claude.com/en/articles/12429409-extra-usage-for-paid-claude-plans) и не засчитывается в использование, включённое в ваш план.

Выбранный вами триггер обзора влияет на общую стоимость:

* **Once after PR creation**: запускается один раз на PR
* **After every push**: запускается при каждом push, умножая стоимость на количество push
* **Manual**: обзоры не запускаются, пока кто-то не оставит комментарий `@claude review` на PR

В любом режиме комментарий `@claude review` [подключает PR к обзорам по push](#manually-trigger-reviews), поэтому дополнительная стоимость накапливается за каждый push после этого комментария. Чтобы запустить единичный обзор без подписки на будущие push, оставьте вместо этого комментарий `@claude review once`.

Расходы появляются в вашем счёте Anthropic независимо от того, использует ли ваша организация Amazon Bedrock или Google Vertex AI для других функций Claude Code. Чтобы установить месячный лимит расходов для Обзора кода, перейдите на [claude.ai/admin-settings/usage](https://claude.ai/admin-settings/usage) и настройте лимит для сервиса Claude Code Review.

Отслеживайте расходы через график еженедельной стоимости в [аналитике](#view-usage) или столбец средней стоимости по репозиторию в настройках администратора.

## Устранение неполадок

Запуски обзоров выполняются по принципу «максимальных усилий» (best-effort). Неудачный запуск никогда не блокирует ваш PR, но и не повторяется сам по себе. В этом разделе рассказывается, как восстановиться после неудачного запуска и куда смотреть, когда check run сообщает о проблемах, которые вы не можете найти.

### Повторный запуск неудачного или прерванного по таймауту обзора

Когда инфраструктура обзора сталкивается с внутренней ошибкой или превышает лимит времени, check run завершается с заголовком **Code review encountered an error** или **Code review timed out**. Заключение по-прежнему нейтральное, поэтому ничто не блокирует ваше слияние, но найденные проблемы не публикуются.

Чтобы запустить обзор снова, оставьте комментарий `@claude review once` на PR. Это запускает свежий обзор без подписки PR на будущие push. Если PR уже подписан на обзоры по push, push нового коммита также запускает новый обзор.

Кнопка **Re-run** во вкладке Checks в GitHub не запускает Обзор кода повторно. Используйте вместо этого команду в комментарии или новый push.

### Обзор не запустился, и PR показывает сообщение о лимите расходов

Когда достигнут месячный лимит расходов вашей организации, Обзор кода публикует один комментарий на PR, объясняющий, что обзор был пропущен. Обзоры возобновляются автоматически в начале следующего платёжного периода или сразу же, когда администратор повышает лимит на [claude.ai/admin-settings/usage](https://claude.ai/admin-settings/usage).

### Найдите проблемы, которые не отображаются как встроенные комментарии

Если заголовок check run говорит, что проблемы были найдены, но вы не видите встроенных комментариев обзора на диффе, поищите в этих других местах, где отображаются найденные проблемы:

* **Check run Details**: нажмите **Details** рядом с проверкой Claude Code Review во вкладке Checks. Таблица серьёзности перечисляет каждую найденную проблему с её файлом, строкой и сводкой независимо от того, был ли принят встроенный комментарий.
* **Аннотации Files changed**: откройте вкладку **Files changed** на PR. Найденные проблемы отображаются как аннотации, прикреплённые непосредственно к строкам диффа, отдельно от комментариев обзора.
* **Тело обзора**: если вы сделали push на PR во время выполнения обзора, некоторые найденные проблемы могут ссылаться на строки, которых больше нет в текущем диффе. Они появляются под заголовком **Additional findings** в тексте тела обзора, а не как встроенные комментарии.

## Обзор диффа локально

[Команда `/code-review`](https://code.claude.com/docs/en/commands) проверяет дифф в вашем терминале без установки GitHub App. Выполните её в любой сессии Claude Code: она сообщает о багах корректности и о чистке в части переиспользования, упрощения и эффективности. По умолчанию локальный обзор охватывает коммиты вашей ветки, опережающие её upstream, плюс любые незакоммиченные изменения в рабочем дереве. Передайте `--comment`, чтобы опубликовать найденные проблемы как встроенные комментарии к PR, или `--fix`, чтобы применить найденные проблемы к вашему рабочему дереву после обзора.

Более низкие [уровни усилий](https://code.claude.com/docs/en/model-config#adjust-effort-level) возвращают меньше найденных проблем с более высокой уверенностью, тогда как `high` — `max` дают более широкое покрытие и могут включать неуверенные находки. Без аргумента усилия обзор использует текущий уровень усилий сессии. Чтобы проверить что-то отличное от диффа по умолчанию, передайте цель: путь к файлу, номер PR, имя ветки или диапазон ref, например `main...my-feature`. Форма диапазона ref проверяет закоммиченный дифф, который содержал бы pull request из `my-feature` в `main`, независимо от того, как настроен upstream ветки.

`/code-review ultra --fix` запускает более глубокий [ultrareview](https://code.claude.com/docs/en/ultrareview) в облаке, а затем применяет его находки к вашему рабочему дереву, когда они возвращаются в вашу сессию. Ultrareview использует собственную область: ваша текущая ветка против ветки репозитория по умолчанию, плюс любые незакоммиченные и застейдженные изменения в рабочем дереве.

Команда называлась `/simplify` до v2.1.147, когда она применяла исправления по умолчанию. Начиная с v2.1.154, `/simplify` запускает отдельный обзор только для чистки, который применяет исправления без поиска багов. Если вы использовали `/simplify` в скриптах для поиска багов, переключитесь на `/code-review --fix`, который остался неизменным.

## Связанные ресурсы

Обзор кода спроектирован для работы вместе с остальной частью Claude Code. Если вы хотите запускать обзоры локально перед открытием PR, нуждаетесь в самостоятельно размещённой настройке или хотите глубже разобраться в том, как `CLAUDE.md` формирует поведение Claude во всех инструментах, эти страницы — хорошие следующие остановки:

* [Команды](https://code.claude.com/docs/en/commands): запускайте `/code-review` в локальной сессии Claude Code, чтобы проверить дифф перед push
* [GitHub Actions](https://code.claude.com/docs/en/github-actions): запускайте Claude в ваших собственных рабочих процессах GitHub Actions для пользовательской автоматизации помимо обзора кода
* [GitLab CI/CD](https://code.claude.com/docs/en/gitlab-ci-cd): самостоятельно размещённая интеграция Claude для конвейеров GitLab
* [Память](https://code.claude.com/docs/en/memory): как работают файлы `CLAUDE.md` во всём Claude Code
* [Аналитика](https://code.claude.com/docs/en/analytics): отслеживайте использование Claude Code помимо обзора кода
