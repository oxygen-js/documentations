# Руководство по безопасности

> Установите плагин security-guidance, чтобы Claude проверял собственные изменения кода на уязвимости и исправлял их в той же сессии.

Плагин руководства по безопасности заставляет Claude проверять собственные изменения кода на распространённые уязвимости прямо в процессе работы и исправлять найденное в той же сессии. Плагин выявляет такие проблемы, как инъекции, небезопасная десериализация и небезопасные DOM API, ещё до того, как код попадёт в pull request, снижая нагрузку по проверке безопасности, которая иначе легла бы на людей-ревьюеров ниже по потоку.

После установки плагин работает автоматически. Ничего вызывать не нужно, и не нужно помнить отдельную команду.

Плагин — это внутрисессионный компаньон [Обзора кода](https://code.claude.com/docs/en/code-review), который запускается на pull request'ах. Этот плагин уменьшает то, что доходит до PR. Обзор кода ловит то, что дошло. О том, как плагин сочетается с проверкой по требованию и сканированием в CI, см. [Как это сочетается с другими инструментами безопасности](#how-this-fits-with-other-security-tools).

## Предварительные требования

* Claude Code CLI версии 2.1.144 или новее
* Python 3.8 или новее в вашем `PATH`. Плагин пробует `python3`, `python` и `py -3` именно в этом порядке
* Git-репозиторий для каталога, в котором вы работаете. Проверки в конце хода и при коммите сравниваются с состоянием git и молча пропускаются вне репозитория. Проверка отдельного редактирования по шаблонам работает где угодно

При первом запуске плагин создаёт виртуальное окружение в `~/.claude/security/` и устанавливает в него Claude Agent SDK, для чего нужны `pip` и доступ к сети. Если эта установка не удастся, проверка при коммите откатывается к одноразовой проверке вместо агентной. В Windows шаг создания виртуального окружения пропускается, поэтому агентная проверка при коммите запускается только если `claude-agent-sdk` уже импортируется, а иначе откатывается таким же образом.

## Установите плагин

В сессии Claude Code установите из [официального маркетплейса Anthropic](https://code.claude.com/docs/en/discover-plugins#official-anthropic-marketplace):

```text
/plugin install security-guidance@claude-plugins-official
```

Установка запрашивает область (scope). Выберите область пользователя, чтобы записать плагин в ваши пользовательские настройки, тогда он будет загружаться в каждой новой локальной сессии, которую вы запускаете на этой машине. Если Claude Code сообщает, что маркетплейс не найден, сначала выполните `/plugin marketplace add anthropics/claude-plugins-official`, затем повторите установку.

Затем активируйте его в текущей сессии командой `/reload-plugins`, которая применяет ожидающие изменения плагинов без перезапуска:

```text
/reload-plugins
```

### Включение в облачных сессиях и общих репозиториях

Плагины с областью пользователя не переносятся в [Claude Code в вебе](https://code.claude.com/docs/en/claude-code-on-the-web), поскольку эти сессии выполняются на инфраструктуре Anthropic, а не на вашей машине. Чтобы включить плагин там или включить его для всех, кто клонирует репозиторий, объявите его в закоммиченных настройках проекта:

```json .claude/settings.json
{
  "enabledPlugins": {
    "security-guidance@claude-plugins-official": true
  }
}
```

Администраторы могут включить плагин для всей организации, задав [`enabledPlugins`](https://code.claude.com/docs/en/settings#plugin-settings) в [управляемых настройках](https://code.claude.com/docs/en/admin-setup).

## Что проверяет плагин

Плагин проверяет работу Claude в трёх точках, каждая с разной глубиной:

* [При каждом редактировании файла](#on-each-file-edit): быстрое сопоставление по шаблонам для рискованных вызовов, без обращения к модели
* [В конце каждого хода](#at-the-end-of-each-turn): фоновый обзор моделью всего, что изменилось за этот ход
* [При каждом коммите или push, который делает Claude](#on-each-commit-or-push-claude-makes): более глубокий агентный обзор, читающий окружающий код

Вы можете расширить каждый слой, [добавив собственные правила](#add-your-own-rules). Встроенные проверки нельзя удалить по отдельности, но вы можете [отключить каждый слой](#disable-or-uninstall) независимо.

### При каждом редактировании файла

Когда Claude записывает в файл, плагин сканирует новое содержимое на наличие известных рискованных шаблонов. Это сопоставление по шаблонам без обращения к модели, поэтому оно не добавляет расходов на использование.

Примеры категорий шаблонов:

* Динамическое исполнение кода: `eval(`, `new Function`, `os.system`, `child_process.exec`
* Небезопасная десериализация: `pickle`
* DOM-инъекции: `dangerouslySetInnerHTML`, `.innerHTML =`, `document.write`
* Файлы рабочих процессов: изменения в `.github/workflows/`, которые могут предоставлять разрешения на уровне репозитория

Проверка выполняется после того, как редактирование применено, и добавляет предупреждение в контекст Claude для следующего шага. Каждое предупреждение срабатывает один раз на шаблон на файл за сессию, поэтому повторные совпадения в том же файле не засоряют беседу.

Вы можете [добавить собственные шаблоны](#add-custom-per-edit-patterns) в этот слой с помощью файла `security-patterns.yaml`.

### В конце каждого хода

Ход — это один раунд ответа Claude: вы отправляете сообщение, Claude работает и отвечает, и ход завершается. После каждого хода плагин вычисляет git diff всего, что изменилось в рабочем дереве за этот ход, включая изменения от инструментов редактирования Claude, команд Bash и субагентов, и отправляет его отдельному обзору Claude, сфокусированному на безопасности. Обзор выполняется в фоне, поэтому ответ Claude не задерживается. Если обзор находит проблемы, Claude заново получает промпт с результатами и решает их как последующее действие.

Это ловит проблемы, которые сопоставление строк не может, например:

* Обход авторизации
* Небезопасные прямые ссылки на объекты (insecure direct object references)
* Инъекции
* Подделка запросов на стороне сервера (server-side request forgery)
* Слабая криптография

Вы видите и находку, и её решение Claude прямо в своей сессии. Обзор охватывает до 30 изменённых файлов за ход и срабатывает не более трёх раз подряд, прежде чем передать управление вам.

### При каждом коммите или push, который делает Claude

Когда Claude выполняет `git commit` или `git push` через свой инструмент Bash, плагин запускает более глубокий агентный обзор изменения в фоне. Этот обзор читает окружающий код, включая вызывающие места, санитайзеры и связанные файлы, чтобы решить, реальна ли находка, прежде чем сообщать о ней. Дополнительный контекст удерживает число ложных срабатываний низким для шаблонов, которые выглядят опасно в отрыве, но безопасны в вашей кодовой базе.

Этот слой срабатывает только на коммитах и push'ах, которые Claude делает через свой инструмент Bash. Коммиты, которые вы выполняете из собственной оболочки, включая shell-escape `!` внутри сессии, не проверяются. Обзоры коммитов и push'ей ограничены 20 в скользящий час. Если находки обзора при коммите дублируют то, о чём уже сообщил обзор в конце хода, Claude не получает повторный промпт, поэтому чистый коммит не даёт видимого вывода от этого слоя.

### Независимость обзора и ограничения

Плагин не просит тот же экземпляр Claude, который написал код, оценивать самого себя. Проверка при редактировании — детерминированное сопоставление строк без участия модели. Обзоры в конце хода и при коммите выполняются как отдельный вызов Claude со свежим контекстом и промптом, ориентированным на безопасность: ревьюер начинает с diff'а, не имеет привязанности к исходному подходу и получает инструкцию только искать проблемы.

Ни один из слоёв не блокирует записи или коммиты. Находки доходят до пишущего Claude как инструкции, Claude решает их в беседе, а модель обзора может пропустить проблемы. Относитесь к плагину как к одному слою эшелонированной защиты, а не как к полноценному решению по безопасности. См. [Как это сочетается с другими инструментами безопасности](#how-this-fits-with-other-security-tools).

## Добавьте собственные правила

У плагина есть две точки расширения: файл руководства в формате Markdown для обзоров, опирающихся на модель, и файл шаблонов в YAML или JSON для сопоставления строк при редактировании. Оба дополняют встроенное поведение. Вы можете добавлять проверки, но не можете отключать встроенные из этих файлов.

### Добавьте руководство для обзоров, опирающихся на модель

Создайте `.claude/claude-security-guidance.md` в своём проекте и опишите свою модель угроз и чек-лист обзора обычным языком. Обзоры, опирающиеся на модель, загружают его как дополнительный контекст наряду со встроенным чек-листом уязвимостей.

Следующий пример предназначен для веб-сервиса с админ-маршрутами, ограниченными по роли, и политикой логирования данных клиентов:

```markdown .claude/claude-security-guidance.md
# Security guidance for this repo

- Do not log `customer_id` or `account_number` at INFO level or above.
- All routes under `/admin` must call `require_role("admin")` before any database read.
- Use `crypto.timingSafeEqual` for token comparison instead of `===`.
```

Эти правила — руководство для ревьюера, а не детерминированные ограничители. Плагин представляет нарушения как находки, которые Claude должен исправить, но он не блокирует записи и не гарантирует, что каждое нарушение будет поймано. Руководство только дополняет: правило, которое велит игнорировать класс уязвимости, не подавляет эти находки. Для жёсткого принуждения сочетайте плагин с [хуком, который блокирует редактирование](https://code.claude.com/docs/en/hooks-guide#block-edits-to-protected-files) или проверкой в CI.

### Добавьте собственные шаблоны для редактирования

Создайте `.claude/security-patterns.yaml`, чтобы добавить правила на основе regex или подстрок в [проверку по шаблонам при редактировании](#on-each-file-edit). Они выполняются как детерминированные сопоставления строк наряду со встроенными шаблонами:

```yaml .claude/security-patterns.yaml
patterns:
  - rule_name: internal_api_key
    substrings: ["sk_live_", "AKIA"]
    reminder: "Hardcoded API key prefix. Load credentials from the secret manager."
  - rule_name: tenant_unfiltered_query
    regex: "\\.objects\\.all\\(\\)"
    paths: ["**/src/tenants/**"]
    reminder: "Multi-tenant code must filter by org_id."
```

| Поле            | Тип    | Описание                                                                                                                                                        |
| :-------------- | :----- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `rule_name`     | строка | Идентификатор, показываемый в предупреждении                                                                                                                    |
| `reminder`      | строка | Текст предупреждения, добавляемый в контекст Claude, ограничен 1 КБ                                                                                             |
| `regex`         | строка | Python-regex, сопоставляемый с отредактированным содержимым                                                                                                     |
| `substrings`    | список | Литеральные подстроки; укажите это или `regex`                                                                                                                  |
| `paths`         | список | Необязательные glob-шаблоны; правило применяется только к совпадающим файлам. Globs сопоставляются с полным путём файла, поэтому предваряйте проектно-относительные шаблоны символами `**/` |
| `exclude_paths` | список | Необязательные glob-шаблоны для пропуска; то же сопоставление, что и у `paths`                                                                                   |

Плагин также читает `.claude/security-patterns.yml` и `.claude/security-patterns.json` с той же схемой. JSON работает на любой установке Python. Формы YAML требуют, чтобы PyYAML был импортируем, а плагин не устанавливает его за вас. Плагин загружает до 50 пользовательских правил и пропускает regex'ы, которые выглядят склонными к катастрофическому откату (catastrophic backtracking).

### Места поиска файлов правил

Плагин ищет `claude-security-guidance.md` и `security-patterns.yaml` в одних и тех же местах, независимо от того, как плагин был включён:

| Область         | Путь                                        | Примечания                                    |
| :-------------- | :------------------------------------------ | :-------------------------------------------- |
| Пользователь    | `~/.claude/claude-security-guidance.md`     | Применяется ко всем проектам на вашей машине  |
| Проект          | `.claude/claude-security-guidance.md`       | Закоммичен вместе с репозиторием              |
| Локальный проект| `.claude/claude-security-guidance.local.md` | В gitignore, для персональных переопределений |

Плагин загружает все существующие места и объединяет их, с суммарным ограничением 8 КБ для файла руководства. Администраторы могут распространять правила для всей организации, отправляя файл области пользователя в `~/.claude/` через управление устройствами. Те же пути применяются к `security-patterns.yaml`.

## Стоимость использования

[Проверка по шаблонам при редактировании](#on-each-file-edit) не обращается к модели и не добавляет расходов. Обзоры [в конце хода](#at-the-end-of-each-turn) и [при коммите](#on-each-commit-or-push-claude-makes) каждый расходуют дополнительное использование модели, которое засчитывается в ваше [использование](https://code.claude.com/docs/en/costs), как любой другой запрос к Claude. Обзор при коммите агентный и может занимать несколько ходов модели на коммит, с ограничением 20 обзоров в скользящий час. Ожидайте примерно один вызов обзора на ход, изменяющий файлы, и один более глубокий обзор на коммит, оба с учётом ограничений выше.

Оба обзора, опирающихся на модель, по умолчанию используют Claude Opus 4.7. Задайте `SECURITY_REVIEW_MODEL`, чтобы выбрать другую модель для обзора в конце хода, и `SG_AGENTIC_MODEL` для обзора при коммите.

Плагин доступен на всех планах.

## Отключение или удаление

Чтобы отключить отдельные слои, сохранив остальные, задайте соответствующую переменную окружения:

| Переменная                      | Эффект                                                                     |
| :------------------------------ | :------------------------------------------------------------------------- |
| `ENABLE_PATTERN_RULES=0`        | Отключить [проверку по шаблонам при редактировании](#on-each-file-edit)    |
| `ENABLE_STOP_REVIEW=0`          | Отключить [обзор diff в конце хода](#at-the-end-of-each-turn)              |
| `ENABLE_COMMIT_REVIEW=0`        | Отключить [обзор при коммите и push](#on-each-commit-or-push-claude-makes) |
| `ENABLE_CODE_SECURITY_REVIEW=0` | Отключить сразу все обзоры, опирающиеся на модель                          |
| `SECURITY_GUIDANCE_DISABLE=1`   | Полностью отключить плагин без удаления                                    |

Чтобы приостановить плагин в вашей области пользователя:

```text
/plugin disable security-guidance@claude-plugins-official
```

Чтобы удалить его из вашей области пользователя:

```text
/plugin uninstall security-guidance@claude-plugins-official
```

Если плагин был включён через `.claude/settings.json` проекта, отключение его через `/plugin` записывает переопределение в ваш `.claude/settings.local.json`, а не редактирует закоммиченный файл, поэтому плагин остаётся выключенным для вас, но не затрагивает коллег. Если он был включён через [управляемые настройки](https://code.claude.com/docs/en/admin-setup), отключить его может только администратор.

## Как плагин интегрируется с Claude Code

Плагин целиком построен на [хуках](https://code.claude.com/docs/en/hooks) — механизме запуска вашего собственного кода в определённых точках цикла Claude. Он регистрирует:

| Событие хука                                                     | Назначение                                                                     |
| :--------------------------------------------------------------- | :---------------------------------------------------------------------------- |
| `SessionStart`                                                   | Инициализация Python-окружения плагина                                         |
| `UserPromptSubmit`                                               | Захват базового состояния рабочего дерева, с которым сравнивает обзор в конце хода |
| `PostToolUse` на `Edit`, `Write` и `NotebookEdit`               | Сопоставление по шаблонам при редактировании                                   |
| `Stop`                                                           | Обзор diff в конце хода, выполняемый в фоне                                    |
| `PostToolUse` на `Bash`, отфильтрованный до `git commit` и `git push` | Обзор при коммите и push, выполняемый в фоне                              |

Если вы создаёте собственные хуки, [исходный код плагина](https://github.com/anthropics/claude-plugins-official/tree/main/plugins/security-guidance) — рабочий пример запуска отдельного вызова модели из хука и передачи результата обратно в сессию.

## Как это сочетается с другими инструментами безопасности

Плагин — один слой в подходе эшелонированной защиты. Он ловит проблемы раньше всех, пока код ещё в редакторе, но это не гарантия и не замена более поздним проверкам. Типичный набор:

| Этап            | Инструмент                                               | Что охватывает                                                                                   |
| :-------------- | :------------------------------------------------------- | :---------------------------------------------------------------------------------------------- |
| В сессии        | Плагин руководства по безопасности                       | Распространённые уязвимости в коде, который пишет Claude, исправляемые в той же сессии           |
| По требованию   | [`/security-review`](https://code.claude.com/docs/en/commands#all-commands) | Одноразовый проход по безопасности текущей ветки, запускаемый по вашему запросу |
| На pull request | [Обзор кода](https://code.claude.com/docs/en/code-review), планы Team и Enterprise | Многоагентный обзор корректности и безопасности с полным контекстом кодовой базы |
| В CI            | Ваши существующие статические анализаторы и сканеры зависимостей | Правила для конкретных языков, проверки цепочки поставок и обеспечение политик, которые плагин не пытается выполнять |

Каждый более поздний этап ловит то, что пропустили ранние. Ценность плагина — в уменьшении объёма, который до них доходит, а не в устранении необходимости в них.

## Устранение неполадок

Плагин пишет диагностику времени выполнения в `~/.claude/security/log.txt`. Проверьте там в первую очередь, если обзоры не появляются.

Распространённые причины, по которым слой обзора пропускается без сообщения в беседе:

* Каталог не является git-репозиторием: обзоры в конце хода и при коммите требуют состояния git и пропускаются вне репозитория
* В сессии нет аутентификации Anthropic: обзоры, опирающиеся на модель, пропускаются, и выполняется только проверка по шаблонам при редактировании
* Файл `security-patterns.yaml` присутствует, но PyYAML не импортируется: файл игнорируется. Используйте `security-patterns.json` вместо него
