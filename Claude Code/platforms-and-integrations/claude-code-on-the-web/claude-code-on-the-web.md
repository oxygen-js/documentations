# Claude Code в вебе

> Настройте облачные окружения, скрипты настройки, сетевой доступ и Docker в песочнице Anthropic. Перемещайте сессии между вебом и терминалом с помощью `--remote` и `--teleport`.

> **Примечание.** Claude Code в вебе находится в режиме исследовательского превью (research preview) для пользователей Pro, Max и Team, а также для корпоративных (Enterprise) пользователей с премиум-местами или местами Chat + Claude Code.

Claude Code в вебе выполняет задачи на облачной инфраструктуре под управлением Anthropic по адресу [claude.ai/code](https://claude.ai/code). Сессии сохраняются, даже если вы закроете браузер, и вы можете следить за ними из мобильного приложения Claude.

> **Совет.** Впервые используете Claude Code в вебе? Начните с раздела [Начало работы](https://code.claude.com/docs/en/web-quickstart), чтобы подключить свой аккаунт GitHub и отправить первую задачу.

## Варианты аутентификации в GitHub

Облачным сессиям нужен доступ к вашим репозиториям GitHub, чтобы клонировать код и отправлять ветки. Предоставить доступ можно двумя способами:

| Метод            | Как это работает                                                                                | Кому подходит                                                                       |
| :--------------- | :-------------------------------------------------------------------------------------------- | :--------------------------------------------------------------------------------- |
| **GitHub App**   | Авторизуйте приложение Claude GitHub App во время [онбординга в вебе](https://code.claude.com/docs/en/web-quickstart). | Онбординг в браузере; команды, которым нужно [Автоисправление](#auto-fix-pull-requests) |
| **`/web-setup`** | Выполните `/web-setup` в терминале, чтобы синхронизировать токен вашего локального `gh` CLI с аккаунтом Claude. | Индивидуальные разработчики, которые уже используют `gh`                            |

> **Примечание.** При любом из методов облачная сессия может получить доступ к любому репозиторию, который видит подключаемый аккаунт GitHub, а не только к репозиториям, на которые установлено приложение Claude GitHub App. Установка приложения включает вебхуки PR для [Автоисправления](#auto-fix-pull-requests); это не контроль доступа на уровне сессии. Чтобы ограничить, к каким репозиториям ваша команда может обращаться из облачных сессий, ограничьте доступ на самом GitHub, например ограничив членство в команде или репозитории для подключённых аккаунтов GitHub.

Работает любой из методов. [`/schedule`](https://code.claude.com/docs/en/routines) проверяет наличие любой из форм доступа и предлагает выполнить `/web-setup`, если не настроена ни одна. См. [Подключение из терминала](https://code.claude.com/docs/en/web-quickstart#connect-from-your-terminal) для пошагового руководства по `/web-setup`.

Приложение GitHub App обязательно для [Автоисправления](#auto-fix-pull-requests), которое использует приложение для получения вебхуков PR. Если вы подключились через `/web-setup` и позже захотите использовать Автоисправление, установите приложение на эти репозитории.

Владельцы Team и Enterprise могут отключить `/web-setup` с помощью переключателя Quick web setup на странице [claude.ai/admin-settings/claude-code](https://claude.ai/admin-settings/claude-code).

> **Примечание.** Организации с включённым [нулевым хранением данных (Zero Data Retention)](https://code.claude.com/docs/en/zero-data-retention) не могут использовать `/web-setup` или другие возможности облачных сессий.

## Облачное окружение

Каждая сессия выполняется в свежей виртуальной машине под управлением Anthropic с клонированным репозиторием. В этом разделе описано, что доступно при запуске сессии и как это настраивать.

### Что доступно в облачных сессиях

Облачные сессии стартуют из свежего клона вашего репозитория. Доступно всё, что закоммичено в репозиторий. Всё, что вы установили или настроили только на своей машине, в сессии недоступно. Политика вашей организации приходит отдельно через [серверно-управляемые настройки](https://code.claude.com/docs/en/server-managed-settings).

|                                                                            | Доступно в облачных сессиях | Почему                                                                                                                                                                                                                                                                                                                  |
| :------------------------------------------------------------------------- | :-------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `CLAUDE.md` вашего репозитория                                             | Да                          | Часть клона                                                                                                                                                                                                                                                                                                            |
| Хуки в `.claude/settings.json` вашего репозитория                          | Да                          | Часть клона                                                                                                                                                                                                                                                                                                            |
| MCP-серверы в `.mcp.json` вашего репозитория                               | Да                          | Часть клона                                                                                                                                                                                                                                                                                                            |
| `.claude/rules/` вашего репозитория                                        | Да                          | Часть клона                                                                                                                                                                                                                                                                                                            |
| `.claude/skills/`, `.claude/agents/`, `.claude/commands/` вашего репозитория | Да                        | Часть клона                                                                                                                                                                                                                                                                                                            |
| Плагины, объявленные в `.claude/settings.json`                             | Да                          | Устанавливаются при запуске сессии из [маркетплейса](https://code.claude.com/docs/en/plugin-marketplaces), который вы объявили. Требуется сетевой доступ до источника маркетплейса                                                                                                                                       |
| [Серверно-управляемые настройки](https://code.claude.com/docs/en/server-managed-settings) вашей организации | Да | Загружаются с серверов Anthropic при запуске сессии. См. [Покрытие поверхностей](https://code.claude.com/docs/en/model-config#surface-coverage) о том, как `availableModels` применяется в облачных сессиях. Настройки, развёрнутые на вашем устройстве через MDM или управляемые файлы настроек, не применяются, потому что сессия выполняется на виртуальной машине под управлением Anthropic |
| Ваш пользовательский `~/.claude/CLAUDE.md`                                 | Нет                         | Находится на вашей машине, а не в репозитории                                                                                                                                                                                                                                                                          |
| Ваши пользовательские `~/.claude/skills/`, `~/.claude/agents/`, `~/.claude/commands/` | Нет               | Находятся на вашей машине, а не в репозитории. Вместо этого закоммитьте их в каталог `.claude/` репозитория. Навыки (skills), которые вы включаете на claude.ai, загружаются в облачные сессии автоматически                                                                                                             |
| Плагины, включённые только в ваших пользовательских настройках             | Нет                         | Пользовательский `enabledPlugins` находится в `~/.claude/settings.json`. Вместо этого объявите их в `.claude/settings.json` репозитория                                                                                                                                                                                 |
| MCP-серверы, добавленные через `claude mcp add`                            | Нет                         | Они записываются в вашу локальную пользовательскую конфигурацию, а не в репозиторий. Вместо этого объявите сервер в [`.mcp.json`](https://code.claude.com/docs/en/mcp#project-scope)                                                                                                                                     |
| Статические API-токены и учётные данные                                    | Нет                         | Выделенного хранилища секретов пока нет. См. ниже                                                                                                                                                                                                                                                                       |
| Интерактивная аутентификация вроде AWS SSO                                 | Нет                         | Не поддерживается. SSO требует входа через браузер, который не может выполняться в облачной сессии                                                                                                                                                                                                                       |

Чтобы сделать вашу собственную конфигурацию доступной в облачных сессиях, закоммитьте её в репозиторий; политика организации приходит отдельно через [серверно-управляемые настройки](https://code.claude.com/docs/en/server-managed-settings).

Выделенное хранилище секретов пока недоступно. И переменные окружения, и скрипты настройки хранятся в конфигурации окружения, видимой любому, кто может редактировать это окружение. Если вам нужны секреты в облачной сессии, добавляйте их как переменные окружения, учитывая эту видимость.

### Работа с задачами (issues) и pull request'ами GitHub

В облачные сессии встроены инструменты GitHub, которые позволяют Claude читать задачи, перечислять pull request'ы, получать диффы и оставлять комментарии без какой-либо настройки. Эти инструменты аутентифицируются через [прокси GitHub](#github-proxy) с помощью того метода, который вы настроили в разделе [Варианты аутентификации в GitHub](#github-authentication-options), поэтому ваш токен никогда не попадает в контейнер.

`gh` CLI не предустановлен. Если вам нужна команда `gh`, которую не покрывают встроенные инструменты, например `gh release` или `gh workflow run`, установите и аутентифицируйте её самостоятельно:

1. **Установите gh в скрипте настройки**

   Добавьте `apt update && apt install -y gh` в ваш [скрипт настройки](#setup-scripts).

2. **Укажите токен**

   Добавьте переменную окружения `GH_TOKEN` в ваши [настройки окружения](#configure-your-environment) с персональным токеном доступа GitHub. `gh` читает `GH_TOKEN` автоматически, поэтому шаг `gh auth login` не нужен.

### Привязка вывода к сессии

У каждой облачной сессии есть URL транскрипта на claude.ai, и сессия может прочитать свой собственный идентификатор из переменной окружения `CLAUDE_CODE_REMOTE_SESSION_ID`. Используйте это, чтобы поместить отслеживаемую ссылку в тела PR, сообщения коммитов, посты в Slack или сгенерированные отчёты, чтобы ревьюер мог открыть запуск, который их создал.

Начиная с v2.1.179, коммиты, которые Claude создаёт в веб-сессии, включают git-трейлер `Claude-Session: <url>`, а тела PR содержат URL сессии на отдельной строке. Начиная с v2.1.182, установите [`attribution.sessionUrl`](https://code.claude.com/docs/en/settings#attribution-settings) в `false`, чтобы опустить трейлер и ссылку в теле PR.

Чтобы включить ссылку на сессию во что-то отличное от коммита или PR, например в сообщение Slack, которое публикует Claude, или в файл отчёта, который он пишет, попросите Claude выполнить следующую команду и использовать её вывод. Команда преобразует префикс `cse_` в значении переменной окружения в префикс `session_`, который ожидает URL транскрипта:

```bash
echo "https://claude.ai/code/${CLAUDE_CODE_REMOTE_SESSION_ID/#cse_/session_}"
```

### Запуск тестов, старт сервисов и добавление пакетов

Claude запускает тесты как часть работы над задачей. Попросите об этом в своём промпте, например «fix the failing tests in `tests/`» или «run pytest after each change». Раннеры тестов, такие как pytest, jest и cargo test, предустановлены и работают без дополнительной настройки.

PostgreSQL и Redis предустановлены, но по умолчанию не запущены. Попросите Claude запустить каждый из них во время сессии:

```bash
service postgresql start
```

```bash
service redis-server start
```

Docker доступен для запуска контейнеризированных сервисов. Попросите Claude выполнить `docker compose up`, чтобы запустить сервисы вашего проекта. Сетевой доступ для скачивания образов подчиняется [уровню доступа](#access-levels) вашего окружения, а [Доверенные значения по умолчанию](#default-allowed-domains) включают Docker Hub и другие распространённые реестры.

Если ваши образы большие или медленно скачиваются, добавьте `docker compose pull` или `docker compose build` в ваш [скрипт настройки](#setup-scripts). Скачанные образы сохраняются в [кэшированном окружении](#environment-caching), поэтому в каждой новой сессии они уже на диске. Кэш хранит только файлы, а не запущенные процессы, поэтому Claude всё равно запускает контейнеры в каждой сессии.

Чтобы добавить пакеты, которых нет в предустановленных, используйте [скрипт настройки](#setup-scripts). Вывод скрипта [кэшируется](#environment-caching), поэтому пакеты, которые вы там устанавливаете, доступны в начале каждой сессии без переустановки. Вы также можете попросить Claude установить пакеты в середине сессии, но такие установки не переносятся в другие сессии.

### Лимиты ресурсов

Облачные сессии выполняются с примерными потолками ресурсов, которые со временем могут меняться:

* 4 vCPU
* 16 ГБ ОЗУ
* 30 ГБ диска

Задачи, требующие значительно больше памяти, например крупные сборки или ресурсоёмкие тесты, могут завершиться сбоем или быть прерваны. Для нагрузок за пределами этих лимитов используйте [Remote Control](https://code.claude.com/docs/en/remote-control), чтобы запускать Claude Code на собственном оборудовании.

### Настройка окружения

Окружения управляют [сетевым доступом](#network-access), переменными окружения и [скриптом настройки](#setup-scripts), который выполняется перед запуском сессии. См. [Установленные инструменты](#installed-tools) о том, что доступно без какой-либо настройки. Управлять окружениями можно из веб-интерфейса или из терминала:

| Действие                             | Как                                                                                                                                                                                                                            |
| :----------------------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Добавить окружение                   | Выберите текущее окружение, чтобы открыть селектор, затем выберите **Add environment**. Диалог включает имя, уровень сетевого доступа, переменные окружения и скрипт настройки.                                                |
| Редактировать окружение              | Выберите значок облака с именем текущего окружения, чтобы открыть селектор, наведите курсор на окружение и нажмите значок настроек, появляющийся справа.                                                                        |
| Заархивировать окружение             | Откройте окружение для редактирования и выберите **Archive**. Заархивированные окружения скрыты из селектора, но существующие сессии продолжают работать.                                                                       |
| Задать значение по умолчанию для `--remote` | Выполните `/remote-env` в терминале. Если у вас одно окружение, эта команда показывает вашу текущую конфигурацию. `/remote-env` только выбирает значение по умолчанию; добавляйте, редактируйте и архивируйте окружения из веб-интерфейса. |

Переменные окружения используют формат `.env` с одной парой `KEY=value` на строку. Не заключайте значения в кавычки, поскольку кавычки сохраняются как часть значения. В этом примере определяются три переменные:

```text
NODE_ENV=development
LOG_LEVEL=debug
DATABASE_URL=postgres://localhost:5432/myapp
```

## Скрипты настройки

Скрипт настройки — это скрипт Bash, который выполняется при запуске новой облачной сессии, до старта Claude Code. Используйте скрипты настройки, чтобы устанавливать зависимости, настраивать инструменты или получать всё, что нужно сессии и что не предустановлено.

Скрипты выполняются от имени root на Ubuntu 24.04, поэтому `apt install` и большинство менеджеров пакетов для языков работают.

Чтобы добавить скрипт настройки, откройте диалог настроек окружения и введите свой скрипт в поле **Setup script**.

В этом примере устанавливается `gh` CLI, который не предустановлен:

```bash
#!/bin/bash
apt update && apt install -y gh
```

Если скрипт завершается с ненулевым кодом, сессия не запускается. Добавляйте `|| true` к некритичным командам, чтобы не блокировать сессию из-за случайного сбоя установки.

Поддерживайте общее время выполнения скрипта примерно в пределах пяти минут, чтобы [кэш окружения](#environment-caching) мог собраться. Запускайте независимые установки параллельно с помощью `&` и `wait`. Если одна загрузка не укладывается в пятиминутный лимит, перенесите её в [хук SessionStart](#setup-scripts-vs-sessionstart-hooks), который запускает её в фоне.

> **Примечание.** Скриптам настройки, устанавливающим пакеты, нужен сетевой доступ до реестров. Сетевой доступ **Trusted** по умолчанию разрешает соединения с [распространёнными реестрами пакетов](#default-allowed-domains), включая npm, PyPI, RubyGems и crates.io. Скрипты не смогут установить пакеты, если ваше окружение использует сетевой доступ **None**.

### Кэширование окружения

Скрипт настройки выполняется при первом запуске сессии в окружении. После его завершения Anthropic делает снимок файловой системы и повторно использует этот снимок как отправную точку для последующих сессий. Новые сессии стартуют с вашими зависимостями, инструментами и образами Docker, уже находящимися на диске, а шаг скрипта настройки пропускается. Это сохраняет быстрый запуск, даже когда скрипт устанавливает крупные наборы инструментов или скачивает образы контейнеров.

Кэш захватывает файлы, а не запущенные процессы. Всё, что скрипт настройки записывает на диск, переносится. Сервисы или контейнеры, которые он запускает, — нет, поэтому запускайте их в каждой сессии, попросив Claude или с помощью [хука SessionStart](#setup-scripts-vs-sessionstart-hooks).

Скрипт настройки выполняется снова для пересборки кэша, когда вы меняете скрипт настройки окружения или разрешённые сетевые хосты, а также когда кэш достигает истечения срока примерно через семь дней. Возобновление существующей сессии никогда не запускает скрипт настройки повторно.

Вам не нужно включать кэширование или управлять снимками самостоятельно.

### Скрипты настройки против хуков SessionStart

Используйте скрипт настройки, чтобы установить то, что нужно облаку, но что уже есть на вашем ноутбуке, например языковую среду выполнения или CLI-инструмент. Используйте [хук SessionStart](https://code.claude.com/docs/en/hooks#sessionstart) для настройки проекта, которая должна выполняться везде — и в облаке, и локально, — например `npm install`.

Оба выполняются в начале сессии, но относятся к разным местам:

|                  | Скрипты настройки                                                                            | Хуки SessionStart                                             |
| ---------------- | ------------------------------------------------------------------------------------------- | ------------------------------------------------------------ |
| Привязаны к      | Облачному окружению                                                                          | Вашему репозиторию                                            |
| Настраиваются в  | Интерфейсе облачного окружения                                                               | `.claude/settings.json` в вашем репозитории                  |
| Выполняются      | До запуска Claude Code, когда нет доступного [кэшированного окружения](#environment-caching) | После запуска Claude Code, в каждой сессии, включая возобновлённые |
| Область          | Только облачные окружения                                                                    | И локальные, и облачные                                       |

Хуки SessionStart также можно определить локально в вашем пользовательском `~/.claude/settings.json`, но пользовательские настройки не переносятся в облачные сессии. В облаке хуки берутся из репозитория и из [серверно-управляемых настроек](https://code.claude.com/docs/en/server-managed-settings) вашей организации.

### Установка зависимостей с помощью хука SessionStart

Чтобы устанавливать зависимости только в облачных сессиях, добавьте хук SessionStart в `.claude/settings.json` вашего репозитория:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup|resume",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/scripts/install_pkgs.sh"
          }
        ]
      }
    ]
  }
}
```

Создайте скрипт по пути `scripts/install_pkgs.sh` и сделайте его исполняемым с помощью `chmod +x`. Переменная окружения `CLAUDE_CODE_REMOTE` установлена в `true` в облачных сессиях, поэтому её можно использовать, чтобы пропустить локальное выполнение:

```bash
#!/bin/bash

if [ "$CLAUDE_CODE_REMOTE" != "true" ]; then
  exit 0
fi

npm install
pip install -r requirements.txt
exit 0
```

У хуков SessionStart есть некоторые ограничения в облачных сессиях:

* **Нет ограничения только облаком**: хуки выполняются и в локальных, и в облачных сессиях. Чтобы пропустить локальное выполнение, проверяйте переменную окружения `CLAUDE_CODE_REMOTE`, как показано выше.
* **Требуется сетевой доступ**: командам установки нужно достичь реестров пакетов. Если ваше окружение использует сетевой доступ **None**, эти хуки завершатся сбоем. [Список разрешённых по умолчанию](#default-allowed-domains) при уровне **Trusted** покрывает npm, PyPI, RubyGems и crates.io.
* **Совместимость с прокси**: весь исходящий трафик проходит через [прокси безопасности](#security-proxy). Некоторые менеджеры пакетов работают с этим прокси некорректно. Bun — известный пример.
* **Добавляет задержку запуска**: хуки выполняются при каждом запуске или возобновлении сессии, в отличие от скриптов настройки, которые выигрывают от [кэширования окружения](#environment-caching). Держите скрипты установки быстрыми, проверяя, присутствуют ли зависимости, прежде чем переустанавливать их.

Чтобы сохранить переменные окружения для последующих команд Bash, записывайте в файл по пути `$CLAUDE_ENV_FILE`. Подробности см. в [хуках SessionStart](https://code.claude.com/docs/en/hooks#sessionstart).

Замена базового образа собственным образом Docker пока не поддерживается. Используйте скрипт настройки, чтобы установить то, что вам нужно, поверх [предоставленного образа](#installed-tools), или запускайте свой образ как контейнер рядом с Claude с помощью `docker compose`.

## Сетевой доступ

Сетевой доступ управляет исходящими соединениями из облачного окружения. Каждое окружение задаёт один уровень доступа, и вы можете расширить его собственными разрешёнными доменами. По умолчанию используется **Trusted**, что разрешает реестры пакетов и другие [домены из списка разрешённых](#default-allowed-domains).

Чтобы изменить сетевой доступ окружения, [откройте его для редактирования](#configure-your-environment) и используйте селектор **Network access** в диалоге. Отдельной страницы Environments нет. Значок облака появляется везде, где вы запускаете облачную сессию или настраиваете [рутину (routine)](https://code.claude.com/docs/en/routines#environments-and-network-access).

> **Примечание.** Трафик коннекторов MCP направляется через серверы Anthropic, поэтому коннекторы, которые вы включаете в сессии или рутине, работают без добавления их хостов в **Allowed domains**. Коннекторы настраиваются для каждой сессии или каждой рутины; удаляйте любые ненужные, чтобы ограничить, к каким инструментам может обращаться Claude. Это опирается на тот же канал, направленный на Anthropic, который отмечен в разделе [Безопасность и изоляция](#security-and-isolation).

### Уровни доступа

Выберите уровень доступа при создании или редактировании окружения:

| Уровень     | Исходящие соединения                                                                          |
| :---------- | :------------------------------------------------------------------------------------------- |
| **None**    | Нет исходящего сетевого доступа                                                              |
| **Trusted** | Только [домены из списка разрешённых](#default-allowed-domains): реестры пакетов, GitHub, облачные SDK |
| **Full**    | Любой домен                                                                                  |
| **Custom**  | Ваш собственный список разрешённых, при желании включающий значения по умолчанию            |

Операции GitHub используют [отдельный прокси](#github-proxy), независимый от этой настройки.

### Разрешение конкретных доменов

Чтобы разрешить домены, которых нет в списке Trusted, выберите **Custom** в настройках сетевого доступа окружения. Появится поле **Allowed domains**. Вводите по одному домену на строку:

```text
api.example.com
*.internal.example.com
registry.example.com
```

Используйте `*.` для сопоставления поддоменов по шаблону. Отметьте **Also include default list of common package managers**, чтобы сохранить [Доверенные домены](#default-allowed-domains) наряду с вашими собственными записями, или оставьте его неотмеченным, чтобы разрешить только то, что вы перечислили.

### Прокси GitHub

Из соображений безопасности все операции GitHub проходят через выделенный прокси-сервис, который прозрачно обрабатывает все взаимодействия с git. Внутри песочницы git-клиент аутентифицируется с помощью специально созданного ограниченного по области действия учётного данного. Этот прокси:

* Безопасно управляет аутентификацией в GitHub: git-клиент использует ограниченное учётное данное внутри песочницы, которое прокси проверяет и преобразует в ваш настоящий токен аутентификации GitHub
* Ограничивает операции git push текущей рабочей веткой ради безопасности
* Обеспечивает клонирование, получение (fetch) и операции с PR, сохраняя границы безопасности

### Прокси безопасности

Окружения работают за сетевым прокси HTTP/HTTPS в целях безопасности и предотвращения злоупотреблений. Весь исходящий интернет-трафик проходит через этот прокси, который обеспечивает:

* Защиту от вредоносных запросов
* Ограничение частоты запросов и предотвращение злоупотреблений
* Фильтрацию контента для повышенной безопасности
* Аудиторский след запрошенных имён хостов на уровне DNS

## Перемещение задач между вебом и терминалом

Эти рабочие процессы требуют [Claude Code CLI](https://code.claude.com/docs/en/quickstart), вошедшего в тот же аккаунт claude.ai. Вы можете запускать новые облачные сессии из терминала или переносить облачные сессии в терминал, чтобы продолжить локально. Облачные сессии сохраняются, даже если вы закроете ноутбук, и вы можете следить за ними откуда угодно, включая мобильное приложение Claude.

> **Примечание.** Из CLI передача сессии односторонняя: вы можете перенести облачные сессии в терминал с помощью `--teleport`, но не можете отправить существующую терминальную сессию в веб. Флаг `--remote` создаёт новую облачную сессию для вашего текущего репозитория. [Приложение для десктопа](https://code.claude.com/docs/en/desktop#continue-in-another-surface) предоставляет меню «Continue in», которое может отправить локальную сессию в веб.

### Из терминала в веб

Запустите облачную сессию из командной строки с флагом `--remote`:

```bash
claude --remote "Fix the authentication bug in src/auth/login.ts"
```

Это создаёт новую облачную сессию на claude.ai. Сессия клонирует GitHub-удалёнку вашего текущего каталога на вашей текущей ветке, поэтому сначала отправьте изменения (push), если у вас есть локальные коммиты, так как виртуальная машина клонирует с GitHub, а не с вашей машины. `--remote` работает с одним репозиторием за раз. Задача выполняется в облаке, пока вы продолжаете работать локально.

Начиная с v2.1.195, CLI показывает живой чек-лист шагов настройки, таких как клонирование репозитория и запуск вашего [скрипта настройки](#setup-scripts), пока запускается облачный контейнер. Сообщения, которые вы вводите во время подготовки контейнера, ставятся в очередь и отправляются, как только сессия готова.

> **Примечание.** `--remote` создаёт облачные сессии. `--remote-control` не связан с этим: он открывает локальную сессию CLI для мониторинга из веба. См. [Remote Control](https://code.claude.com/docs/en/remote-control).

Используйте `/tasks` в Claude Code CLI, чтобы проверить прогресс, или откройте сессию на claude.ai либо в мобильном приложении Claude, чтобы взаимодействовать напрямую. Оттуда вы можете направлять Claude, давать обратную связь или отвечать на вопросы, как в любой другой беседе.

#### Советы для облачных задач

**Планируйте локально, выполняйте удалённо**: для сложных задач запустите Claude в режиме планирования, чтобы совместно проработать подход, а затем отправьте работу в облако:

```bash
claude --permission-mode plan
```

В режиме планирования Claude читает файлы, выполняет команды для исследования и предлагает план, не редактируя исходный код. Как только вы будете довольны, сохраните план в репозиторий, закоммитьте и отправьте (push), чтобы облачная виртуальная машина могла его склонировать. Затем запустите облачную сессию для автономного выполнения:

```bash
claude --remote "Execute the migration plan in docs/migration-plan.md"
```

Этот шаблон даёт вам контроль над стратегией, позволяя Claude автономно выполнять работу в облаке.

**Планируйте в облаке с помощью ultraplan**: чтобы составить и просмотреть сам план в веб-сессии, используйте [ultraplan](https://code.claude.com/docs/en/ultraplan). Claude генерирует план в Claude Code в вебе, пока вы продолжаете работать, затем вы комментируете разделы в браузере и выбираете, выполнить его удалённо или отправить план обратно в терминал.

**Запускайте задачи параллельно**: каждая команда `--remote` создаёт свою собственную облачную сессию, которая выполняется независимо. Вы можете запустить несколько задач, и все они будут выполняться одновременно в отдельных сессиях:

```bash
claude --remote "Fix the flaky test in auth.spec.ts"
claude --remote "Update the API documentation"
claude --remote "Refactor the logger to use structured output"
```

Отслеживайте все сессии с помощью `/tasks` в Claude Code CLI. Когда сессия завершается, вы можете создать PR из веб-интерфейса или [телепортировать](#from-web-to-terminal) сессию в терминал, чтобы продолжить работу.

#### Отправка локальных репозиториев без GitHub

Когда вы запускаете `claude --remote` из репозитория, не подключённого к GitHub, Claude Code упаковывает ваш локальный репозиторий и загружает его напрямую в облачную сессию. Пакет включает всю историю вашего репозитория по всем веткам, плюс любые незакоммиченные изменения в отслеживаемых файлах.

Этот запасной вариант активируется автоматически, когда доступ к GitHub недоступен. Чтобы принудительно включить его даже при подключённом GitHub, установите `CCR_FORCE_BUNDLE=1`:

```bash
CCR_FORCE_BUNDLE=1 claude --remote "Run the test suite and fix any failures"
```

Упакованные репозитории должны соответствовать этим ограничениям:

* Каталог должен быть git-репозиторием хотя бы с одним коммитом
* Упакованный репозиторий должен быть менее 100 МБ. Более крупные репозитории откатываются к упаковке только текущей ветки, затем к единственному сжатому снимку рабочего дерева и завершаются сбоем, только если снимок всё ещё слишком велик
* Неотслеживаемые файлы не включаются; выполните `git add` для файлов, которые вы хотите показать облачной сессии
* Сессии, созданные из пакета, не могут отправлять изменения (push) обратно в удалённый репозиторий, если у вас также не настроена [аутентификация в GitHub](#github-authentication-options)

### Из веба в терминал

Перенесите облачную сессию в терминал любым из этих способов:

* **С помощью `--teleport`**: из командной строки выполните `claude --teleport` для интерактивного выбора сессии или `claude --teleport <session-id>`, чтобы возобновить конкретную сессию напрямую. Если у вас есть незакоммиченные изменения, вам предложат сначала спрятать их в stash.
* **С помощью `/teleport`**: внутри существующей сессии CLI выполните `/teleport` или `/tp`, чтобы открыть тот же выбор сессии без перезапуска Claude Code.
* **Из `/tasks`**: выполните `/tasks`, чтобы увидеть ваши фоновые сессии, затем нажмите `t`, чтобы телепортироваться в одну из них.
* **Из веб-интерфейса**: выберите **Open in CLI**, чтобы скопировать команду, которую вы можете вставить в терминал.

Когда вы телепортируете сессию, Claude проверяет, что вы находитесь в правильном репозитории, получает и переключается на ветку из облачной сессии и загружает полную историю беседы в ваш терминал.

`--teleport` отличается от `--resume`. `--resume` заново открывает беседу из локальной истории этой машины и не перечисляет облачные сессии; `--teleport` переносит облачную сессию и её ветку.

#### Требования для телепорта

Телепорт проверяет эти требования перед возобновлением сессии. Если какое-либо требование не выполнено, вы увидите ошибку или вам предложат решить проблему.

| Требование            | Подробности                                                                                                                |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| Чистое состояние git  | В вашем рабочем каталоге не должно быть незакоммиченных изменений. Телепорт предлагает спрятать изменения в stash при необходимости. |
| Правильный репозиторий | Вы должны запускать `--teleport` из checkout того же репозитория, а не форка.                                             |
| Доступная ветка       | Ветка из облачной сессии должна быть отправлена (push) в удалённый репозиторий. Телепорт автоматически получает и переключается на неё. |
| Тот же аккаунт        | Вы должны быть аутентифицированы в том же аккаунте claude.ai, что использовался в облачной сессии.                        |

#### `--teleport` недоступен

Телепорт требует аутентификации по подписке claude.ai. Если вы аутентифицированы через API-ключ, Bedrock, Vertex AI или Microsoft Foundry, выполните `/login`, чтобы войти под своим аккаунтом claude.ai. Если вы уже вошли через claude.ai, но `--teleport` всё ещё недоступен, ваша организация могла отключить облачные сессии.

## Работа с сессиями

Сессии отображаются в боковой панели на claude.ai/code. Оттуда вы можете просматривать изменения, делиться с коллегами по команде, архивировать завершённую работу или удалять сессии навсегда.

### Управление контекстом

Облачные сессии поддерживают [встроенные команды](https://code.claude.com/docs/en/commands), которые выдают текстовый вывод. Команды, открывающие интерактивный выбор в терминале, такие как `/model` или `/config`, недоступны.

Специально для управления контекстом:

| Команда    | Работает в облачных сессиях | Примечания                                                                                                              |
| :--------- | :-------------------------- | :-------------------------------------------------------------------------------------------------------------------- |
| `/compact` | Да                          | Суммирует беседу, чтобы освободить контекст. Принимает необязательные инструкции фокуса, например `/compact keep the test output` |
| `/context` | Да                          | Показывает, что сейчас находится в окне контекста                                                                     |
| `/clear`   | Нет                         | Вместо этого начните новую сессию из боковой панели                                                                   |

Автоуплотнение (auto-compaction) запускается автоматически, когда окно контекста приближается к вместимости. Чтобы запустить его раньше, установите [`CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`](https://code.claude.com/docs/en/env-vars) в ваших [переменных окружения](#configure-your-environment). Например, `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=70` уплотняет при 70% вместимости, вместо того чтобы ждать, пока окно почти заполнится. Чтобы изменить эффективный размер окна для расчётов уплотнения, используйте [`CLAUDE_CODE_AUTO_COMPACT_WINDOW`](https://code.claude.com/docs/en/env-vars).

[Субагенты](https://code.claude.com/docs/en/sub-agents) работают так же, как локально. Claude может порождать их с помощью инструмента Task, чтобы вынести исследование или параллельную работу в отдельное окно контекста, сохраняя основную беседу более лёгкой. Субагенты, определённые в `.claude/agents/` вашего репозитория, подхватываются автоматически.

[Команды агентов (agent teams)](https://code.claude.com/docs/en/agent-teams) по умолчанию отключены, но их можно включить, добавив `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` в ваши [переменные окружения](#configure-your-environment).

### Просмотр изменений

Каждая сессия показывает индикатор диффа с добавленными и удалёнными строками, например `+42 -18`. Выберите его, чтобы открыть представление диффа, оставить встроенные комментарии на конкретных строках и отправить их Claude со следующим сообщением. См. [Просмотр и итерация](https://code.claude.com/docs/en/web-quickstart#review-and-iterate) для полного руководства, включая создание PR. Чтобы Claude автоматически отслеживал PR на предмет сбоев CI и комментариев к ревью, см. [Автоисправление pull request'ов](#auto-fix-pull-requests).

### Обмен сессиями

Чтобы поделиться сессией, переключите её видимость согласно типам аккаунтов ниже. После этого поделитесь ссылкой на сессию как есть. Получатели видят последнее состояние при открытии ссылки, но их представление не обновляется в реальном времени.

#### Обмен из аккаунта Enterprise или Team

Для аккаунтов Enterprise и Team есть два варианта видимости — **Private** и **Team**. Видимость Team делает сессию видимой другим участникам вашей организации claude.ai. Сессии [Claude в Slack](https://code.claude.com/docs/en/slack) автоматически публикуются с видимостью Team.

Проверка доступа к репозиторию включена по умолчанию, на основе аккаунта GitHub, подключённого к аккаунту получателя. Отображаемое имя вашего аккаунта видно всем получателям с доступом.

#### Обмен из аккаунта Max или Pro

Для аккаунтов Max и Pro есть два варианта видимости — **Private** и **Public**. Видимость Public делает сессию видимой любому пользователю, вошедшему в claude.ai.

Проверяйте сессию на предмет конфиденциального содержимого перед тем, как делиться. Сессии могут содержать код и учётные данные из приватных репозиториев GitHub. Проверка доступа к репозиторию не включена по умолчанию.

Чтобы требовать от получателей наличия доступа к репозиторию или скрыть ваше имя из общих сессий, перейдите в Settings > Claude Code > Sharing settings.

### Архивация сессий

Вы можете архивировать сессии, чтобы поддерживать порядок в списке сессий. Заархивированные сессии скрыты из списка сессий по умолчанию, но их можно просмотреть, отфильтровав по заархивированным сессиям.

Чтобы заархивировать сессию, наведите курсор на сессию в боковой панели и выберите значок архивации.

### Удаление сессий

Удаление сессии навсегда убирает сессию и её данные. Это действие нельзя отменить. Удалить сессию можно двумя способами:

* **Из боковой панели**: отфильтруйте по заархивированным сессиям, затем наведите курсор на сессию, которую хотите удалить, и выберите значок удаления
* **Из меню сессии**: откройте сессию, выберите выпадающий список рядом с заголовком сессии и выберите **Delete**

Перед удалением сессии вас попросят подтвердить действие.

## Автоисправление pull request'ов

Claude может отслеживать pull request и автоматически реагировать на сбои CI и комментарии к ревью. Claude подписывается на активность GitHub в PR, и когда проверка не проходит или ревьюер оставляет комментарий, Claude исследует и отправляет исправление, если оно очевидно.

> **Примечание.** Автоисправление требует, чтобы приложение Claude GitHub App было установлено на вашем репозитории. Если вы ещё этого не сделали, установите его со [страницы GitHub App](https://github.com/apps/claude) или по запросу во время [настройки](https://code.claude.com/docs/en/web-quickstart#connect-github-and-create-an-environment).

Есть несколько способов включить автоисправление в зависимости от того, откуда пришёл PR и какое устройство вы используете:

* **PR, созданные в Claude Code в вебе**: откройте строку состояния CI и выберите **Auto-fix**
* **Из терминала**: выполните [`/autofix-pr`](https://code.claude.com/docs/en/commands), находясь на ветке PR. Claude Code обнаруживает открытый PR с помощью `gh`, порождает веб-сессию и включает автоисправление за один шаг
* **Из мобильного приложения**: скажите Claude автоисправить PR, например «watch this PR and fix any CI failures or review comments»
* **Любой существующий PR**: вставьте URL PR в сессию и скажите Claude автоисправить его

Автоисправление — это переключатель на уровне PR. Чтобы прекратить мониторинг, откройте строку состояния CI в веб-сессии и снимите переключатель **Auto-fix** или скажите Claude прекратить отслеживать PR.

### Как Claude реагирует на активность PR

Когда автоисправление активно, Claude получает события GitHub для PR, включая новые комментарии к ревью и сбои проверок CI. Для каждого события Claude исследует и решает, как действовать:

* **Очевидные исправления**: если Claude уверен в исправлении и оно не конфликтует с более ранними инструкциями, Claude вносит изменение, отправляет его (push) и объясняет, что было сделано, в сессии
* **Неоднозначные запросы**: если комментарий ревьюера можно интерпретировать несколькими способами или он затрагивает что-то архитектурно значимое, Claude спрашивает вас перед тем, как действовать
* **Дублирующие события или события без действия**: если событие является дубликатом или не требует изменений, Claude отмечает это в сессии и двигается дальше

GitHub не выдаёт вебхук, когда базовая ветка продвигается вперёд и создаёт конфликт слияния, поэтому автоисправление не может само реагировать на конфликты. Чтобы разрешить конфликт, откройте сессию и попросите Claude выполнить rebase.

Claude может отвечать на потоки комментариев к ревью на GitHub в рамках их разрешения. Эти ответы публикуются с использованием вашего аккаунта GitHub, поэтому они появляются под вашим именем пользователя, но каждый ответ помечается как исходящий от Claude Code, чтобы ревьюеры знали, что он написан агентом, а не вами напрямую.

> **Предупреждение.** Если ваш репозиторий использует автоматизацию, запускаемую комментариями, такую как Atlantis, Terraform Cloud или пользовательские GitHub Actions, срабатывающие на события `issue_comment`, учитывайте, что Claude может отвечать от вашего имени, что может запускать эти рабочие процессы. Проверьте автоматизацию вашего репозитория перед включением автоисправления и рассмотрите возможность отключения автоисправления для репозиториев, где комментарий к PR может разворачивать инфраструктуру или выполнять привилегированные операции.

## Безопасность и изоляция

Каждая облачная сессия отделена от вашей машины и от других сессий несколькими слоями:

* **Изолированные виртуальные машины**: каждая сессия выполняется в изолированной виртуальной машине под управлением Anthropic
* **Контроль сетевого доступа**: сетевой доступ по умолчанию ограничен и может быть отключён. При работе с отключённым сетевым доступом Claude Code всё ещё может связываться с API Anthropic, что может позволить данным покинуть виртуальную машину.
* **Защита учётных данных**: конфиденциальные учётные данные, такие как учётные данные git или ключи подписи, никогда не находятся внутри песочницы с Claude Code. Аутентификация обрабатывается через безопасный прокси с использованием ограниченных по области действия учётных данных.
* **Безопасный анализ**: код анализируется и изменяется внутри изолированных виртуальных машин перед созданием PR

## Устранение неполадок

Для ошибок API времени выполнения, которые появляются в беседе, таких как `API Error: 500`, `529 Overloaded`, `429` или `Prompt is too long`, см. [Справочник по ошибкам](https://code.claude.com/docs/en/errors). Эти ошибки и их исправления общие с CLI и приложением для десктопа. Разделы ниже охватывают проблемы, специфичные для облачных сессий.

### Не удалось создать сессию

Если новая сессия не запускается с ошибкой `Session creation failed` или зависает на подготовке, Claude Code не смог выделить облачное окружение.

* Проверьте [status.claude.com](https://status.claude.com) на предмет инцидентов с облачными сессиями
* Повторите попытку через минуту, так как ёмкость выделяется по запросу
* Убедитесь, что ваш репозиторий доступен. Подключаемый аккаунт GitHub должен иметь доступ к репозиторию на GitHub — либо через авторизацию Claude GitHub App, либо через токен `gh`, синхронизированный с помощью `/web-setup`. Установка приложения на репозиторий не требуется. См. [Варианты аутентификации в GitHub](#github-authentication-options).

### Сессия Remote Control истекла или доступ запрещён

`--teleport` подключается через ту же инфраструктуру сессий Remote Control, что и облачные сессии, поэтому ошибки аутентификации и истечения срока сессии проявляются с формулировками Remote Control. Вы можете увидеть `Remote Control session expired` или `Access denied`. Токен подключения недолговечен и ограничен вашим аккаунтом.

* Выполните `/login` локально, чтобы обновить учётные данные, затем переподключитесь
* Убедитесь, что вы вошли в тот же аккаунт, которому принадлежит сессия
* Если вы видите `Remote Control may not be available for this organization`, владелец не включил облачные сессии для вашей организации

### Окружение истекло

Облачные сессии останавливаются после периода бездействия, и лежащее в основе окружение освобождается. Из локального терминала это проявляется как `Could not resume session ... its environment has expired. Creating a fresh session instead.` В вебе сессия помечается как истёкшая в списке сессий.

Заново откройте сессию с [claude.ai/code](https://claude.ai/code), чтобы подготовить свежее окружение с восстановленной историей беседы.

## Ограничения

Прежде чем полагаться на облачные сессии для рабочего процесса, учтите эти ограничения:

* **Лимиты запросов**: Claude Code в вебе разделяет лимиты запросов со всем остальным использованием Claude и Claude Code в вашем аккаунте. Запуск нескольких задач параллельно расходует лимиты пропорционально больше. Отдельной платы за вычисления облачной виртуальной машины нет.
* **Аутентификация репозитория**: вы можете перемещать сессии из веба в локальную среду только когда вы аутентифицированы в том же аккаунте
* **Ограничения платформы**: клонирование репозитория и создание pull request требуют GitHub. Самостоятельно размещённые экземпляры [GitHub Enterprise Server](https://code.claude.com/docs/en/github-enterprise-server) поддерживаются для планов Team и Enterprise. GitLab, Bitbucket и другие репозитории, не относящиеся к GitHub, можно отправить в облачные сессии как [локальный пакет](#send-local-repositories-without-github), но сессия не может отправить результаты обратно в удалённый репозиторий
* **Список разрешённых IP организации**: облачные сессии обращаются к API Anthropic из инфраструктуры под управлением Anthropic, а не из вашей сети. Если в вашей организации включён [список разрешённых IP](https://support.claude.com/en/articles/13200993-restrict-access-to-claude-with-ip-allowlisting), каждая облачная сессия завершается ошибкой аутентификации. То же относится к [Code Review](https://code.claude.com/docs/en/code-review) и [Рутинам (Routines)](https://code.claude.com/docs/en/routines). Обратитесь в [поддержку Anthropic](https://support.claude.com/), чтобы исключить сервисы, размещённые на стороне Anthropic, из списка разрешённых IP вашей организации.
