# Рутины (Routines)

> Переведите Claude Code в режим автопилота. Определяйте рутины, которые запускаются по расписанию, срабатывают по вызовам API или реагируют на события GitHub, работая на управляемой Anthropic облачной инфраструктуре.

> **Примечание.** Рутины находятся в стадии исследовательского превью (research preview). Поведение, лимиты и поверхность API могут измениться.

Рутина — это сохранённая конфигурация Claude Code: промпт, один или несколько репозиториев и набор [коннекторов](https://code.claude.com/docs/en/mcp), упакованные один раз и запускаемые автоматически. Рутины выполняются на управляемой Anthropic облачной инфраструктуре, поэтому продолжают работать, когда ваш ноутбук закрыт.

К каждой рутине можно прикрепить один или несколько триггеров:

* **По расписанию (Scheduled)**: запуск с повторяющейся периодичностью — ежечасно, еженощно или еженедельно, либо однократно в конкретный момент в будущем
* **API**: запуск по требованию путём отправки HTTP POST на индивидуальный для рутины эндпоинт с bearer-токеном
* **GitHub**: автоматический запуск в ответ на события репозитория, такие как pull request'ы или релизы

Одна рутина может сочетать несколько триггеров. Например, рутина проверки PR может запускаться еженощно, срабатывать из скрипта деплоя, а также реагировать на каждый новый PR.

Рутины доступны на планах Pro, Max, Team и Enterprise при включённом [Claude Code в вебе](https://code.claude.com/docs/en/claude-code-on-the-web). Создавайте и управляйте ими на [claude.ai/code/routines](https://claude.ai/code/routines) или из CLI командой `/schedule`.

Владельцы (Owners) команд Team и Enterprise могут отключить рутины для всех участников с помощью переключателя Routines на [claude.ai/admin-settings/claude-code](https://claude.ai/admin-settings/claude-code). При отключении существующие рутины перестают запускаться, а участники не могут создавать новые.

Эта страница охватывает создание рутины, настройку каждого типа триггера, управление запусками и то, как применяются лимиты использования.

## Примеры сценариев использования

Каждый пример сочетает тип триггера с той работой, для которой подходят рутины: автономной, повторяемой и привязанной к чёткому результату.

**Поддержание бэклога.** Триггер по расписанию запускается каждый будний вечер и работает с вашим трекером задач через коннектор. Рутина читает задачи, открытые с момента последнего запуска, назначает метки, назначает ответственных на основе области кода, на которую есть ссылки, и публикует сводку в Slack, чтобы команда начинала день с приведённой в порядок очередью.

**Сортировка алертов.** Ваш инструмент мониторинга вызывает API-эндпоинт рутины, когда превышен порог ошибок, передавая тело алерта как `text`. Рутина извлекает стек-трейс, соотносит его с недавними коммитами в репозитории и открывает черновой pull request с предлагаемым исправлением и ссылкой обратно на алерт. Дежурный проверяет PR вместо того, чтобы начинать с пустого терминала.

**Индивидуальная проверка кода.** Триггер GitHub запускается на `pull_request.opened`. Рутина применяет собственный чек-лист проверки вашей команды, оставляет встроенные комментарии по вопросам безопасности, производительности и стиля, а также добавляет итоговый комментарий, чтобы люди-ревьюеры могли сосредоточиться на дизайне вместо механических проверок.

**Проверка деплоя.** Ваш CD-пайплайн вызывает API-эндпоинт рутины после каждого продакшн-деплоя. Рутина прогоняет дымовые проверки (smoke checks) против новой сборки, сканирует логи ошибок на предмет регрессий и публикует вердикт «go» или «no-go» в канал релизов до закрытия окна деплоя.

**Расхождение документации.** Триггер по расписанию запускается еженедельно. Рутина сканирует смёрдженные PR с момента последнего запуска, помечает документацию, которая ссылается на изменённые API, и открывает PR с обновлениями против репозитория документации для проверки редактором.

**Портирование библиотеки.** Триггер GitHub запускается на `pull_request.closed`, отфильтрованный по смёрдженным PR в одном репозитории SDK. Рутина портирует изменение в параллельный SDK на другом языке и открывает соответствующий PR, поддерживая синхронность двух библиотек без того, чтобы человек заново реализовывал каждое изменение.

В разделах ниже пошагово рассматриваются создание рутины и настройка каждого из этих типов триггеров.

## Создание рутины

Создайте рутину в вебе на [claude.ai/code/routines](https://claude.ai/code/routines), в приложении Desktop или из CLI. Все три интерфейса пишут в один и тот же облачный аккаунт, поэтому рутина, созданная в одном из них, немедленно появляется в остальных. В приложении Desktop нажмите **Routines** в боковой панели, затем **New routine** и выберите **Remote**; выбор **Local** вместо этого создаёт [запланированную задачу Desktop](https://code.claude.com/docs/en/desktop-scheduled-tasks), которая выполняется на вашей машине, а не в облаке.

Форма создания настраивает промпт рутины, репозитории, окружение, коннекторы и триггеры.

Рутины выполняются автономно как полноценные облачные сессии Claude Code: нет выбора режима разрешений и нет запросов на подтверждение во время запуска. Сессия может выполнять команды оболочки, использовать [навыки (skills)](https://code.claude.com/docs/en/skills), закоммиченные в клонированный репозиторий, и вызывать любые коннекторы, которые вы включили. То, к чему может обратиться рутина, определяется выбранными вами репозиториями и их настройкой push в ветки, [сетевым доступом и переменными окружения](https://code.claude.com/docs/en/claude-code-on-the-web#the-cloud-environment) и коннекторами, которые вы включили. Ограничивайте каждый из этих аспектов тем, что рутине действительно нужно.

Рутины принадлежат вашему индивидуальному аккаунту claude.ai. Они не разделяются с коллегами по команде и учитываются в дневной квоте запусков вашего аккаунта. Всё, что рутина делает через вашу подключённую личность GitHub или коннекторы, отображается как совершённое вами: коммиты и pull request'ы несут ваш GitHub-пользователь, а сообщения Slack, тикеты Linear или другие действия коннекторов используют ваши связанные аккаунты для этих сервисов.

### Создание в вебе

1. **Откройте форму создания**

   Перейдите на [claude.ai/code/routines](https://claude.ai/code/routines) и нажмите **New routine**.

2. **Назовите рутину и напишите промпт**

   Дайте рутине описательное имя и напишите промпт, который Claude выполняет каждый раз. Промпт — самая важная часть: рутина выполняется автономно, поэтому промпт должен быть самодостаточным и явно описывать, что делать и как выглядит успех.

   Поле ввода промпта включает селектор модели. Claude использует выбранную модель при каждом запуске.

3. **Выберите репозитории**

   Добавьте один или несколько репозиториев GitHub, в которых будет работать Claude. Каждый репозиторий клонируется в начале запуска, начиная с ветки по умолчанию. Claude создаёт ветки с префиксом `claude/` для своих изменений.

4. **Выберите окружение**

   Выберите [облачное окружение](https://code.claude.com/docs/en/claude-code-on-the-web#the-cloud-environment) для рутины. Окружения контролируют, к чему имеет доступ облачная сессия:

   * **Сетевой доступ**: задайте уровень доступа к интернету, доступный во время каждого запуска
   * **Переменные окружения**: предоставьте API-ключи, токены или другие секреты, которые Claude может использовать
   * **Скрипт настройки**: устанавливает зависимости и инструменты, которые нужны рутине. Результат [кэшируется](https://code.claude.com/docs/en/claude-code-on-the-web#environment-caching), поэтому скрипт не запускается заново при каждой сессии

   Предоставляется окружение **Default** с сетевым доступом **Trusted**, который разрешает [набор по умолчанию](https://code.claude.com/docs/en/claude-code-on-the-web#default-allowed-domains) реестров пакетов, API облачных провайдеров, реестров контейнеров и распространённых доменов для разработки, но блокирует всё остальное. Если вашей рутине нужно обратиться к вашим собственным сервисам или домену вне этого списка, отредактируйте [сетевой доступ](https://code.claude.com/docs/en/claude-code-on-the-web#network-access) окружения перед запуском. Чтобы использовать отдельное окружение, сначала [создайте его](https://code.claude.com/docs/en/claude-code-on-the-web#configure-your-environment).

5. **Выберите триггер**

   В разделе **Select a trigger** выберите, как запускается рутина. Вы можете выбрать один тип триггера или скомбинировать несколько.

   #### Schedule

   Выберите предустановленную частоту для повторяющегося запуска или запланируйте однократный запуск в конкретный момент времени. См. [Добавление триггера по расписанию](#добавление-триггера-по-расписанию) для обработки часовых поясов, разброса (stagger), пользовательских cron-интервалов и однократных запусков.

   #### GitHub event

   Выберите репозиторий, событие, на которое нужно реагировать, и опциональные фильтры. См. [Добавление триггера GitHub](#добавление-триггера-github) для полного списка поддерживаемых событий и полей фильтрации.

   #### API

   Выберите здесь **API**, затем сохраните рутину. URL и токен генерируются после сохранения рутины, поскольку они зависят от ID рутины. См. [Добавление триггера API](#добавление-триггера-api), чтобы скопировать URL и сгенерировать токен.

6. **Проверьте коннекторы и разрешения**

   Вкладки **Connectors** и **Permissions** внизу формы контролируют, к чему может обратиться рутина.

   В разделе Connectors по умолчанию включены все ваши подключённые [MCP-коннекторы](https://code.claude.com/docs/en/mcp). Удалите те, которые рутине не нужны. Claude может использовать любой инструмент из включённого коннектора, включая запись, без запроса разрешения во время запуска.

   В разделе Permissions включите **Allow unrestricted branch pushes** для любого репозитория, где Claude должен иметь возможность делать push в существующие ветки, а не только в ветки с префиксом `claude/`.

7. **Создайте рутину**

   Нажмите **Create**. Рутина появляется в списке и запускается в следующий раз, когда сработает один из её триггеров. Чтобы запустить немедленно, нажмите **Run now** на странице деталей рутины.

   Каждый запуск создаёт новую сессию рядом с вашими другими сессиями, где вы можете увидеть, что сделал Claude, проверить изменения и создать pull request.

### Создание из CLI

Запустите `/schedule` в любой сессии, чтобы создать рутину по расписанию в разговорном режиме. Вы также можете передать описание напрямую — для повторяющейся рутины, например `/schedule daily PR review at 9am`, или однократной, например `/schedule clean up feature flag in one week`. Claude проходит через ту же информацию, которую собирает веб-форма, затем сохраняет рутину в вашем аккаунте.

`/schedule` в CLI создаёт только рутины по расписанию. Чтобы добавить триггер API или GitHub, отредактируйте рутину в вебе на [claude.ai/code/routines](https://claude.ai/code/routines).

CLI также поддерживает управление существующими рутинами. Запустите `/schedule list`, чтобы увидеть все рутины, `/schedule update`, чтобы изменить одну, или `/schedule run`, чтобы запустить её немедленно.

## Настройка триггеров

Рутина запускается, когда срабатывает один из её триггеров. Вы можете прикрепить любую комбинацию триггеров по расписанию, API и GitHub к одной и той же рутине, а также добавлять или удалять их в любой момент из раздела **Select a trigger** формы редактирования рутины.

### Добавление триггера по расписанию

Триггер по расписанию запускает рутину с повторяющейся периодичностью или однократно в конкретный момент в будущем. Выберите предустановленную частоту в разделе **Select a trigger**: hourly (ежечасно), daily (ежедневно), weekdays (по будням) или weekly (еженедельно). Время вводится в вашей локальной зоне и конвертируется автоматически, поэтому рутина запускается в это время по настенным часам независимо от того, где расположена облачная инфраструктура.

Запуски могут начинаться на несколько минут позже запланированного времени из-за разброса (stagger). Смещение постоянно для каждой рутины.

Для пользовательского интервала, такого как каждые два часа или первого числа каждого месяца, выберите ближайшую предустановку в форме, затем запустите `/schedule update` в CLI, чтобы задать конкретное cron-выражение. Минимальный интервал — один час; выражения, которые запускаются чаще, отклоняются.

#### Планирование однократного запуска

Однократное расписание запускает рутину один раз в конкретный момент времени. Используйте его, чтобы напомнить себе позже на неделе, открыть PR с очисткой после завершения раскатки или запустить последующую задачу, когда прибудет вышестоящее изменение. После срабатывания рутина автоматически отключается, и веб-интерфейс помечает её как **Ran** (Выполнена). Чтобы запустить её снова, отредактируйте рутину и задайте новое однократное время.

Создайте однократный запуск из CLI, описав время на естественном языке. Claude разрешает фразу относительно текущего времени и подтверждает абсолютную метку времени перед сохранением.

```text
/schedule tomorrow at 9am, summarize yesterday's merged PRs
```

```text
/schedule in 2 weeks, open a cleanup PR that removes the feature flag
```

К однократным меткам времени применяется та же конверсия из локального времени в UTC, что и для повторяющихся расписаний.

Однократные запуски не учитываются в дневном лимите запусков рутин. Они расходуют обычное использование подписки вашего плана, как любая другая сессия. Подробности см. в разделе [Использование и лимиты](#использование-и-лимиты).

### Добавление триггера API

Триггер API даёт рутине выделенный HTTP-эндпоинт. Отправка POST на эндпоинт с bearer-токеном рутины запускает новую сессию и возвращает URL сессии. Используйте это, чтобы связать Claude Code с системами оповещения, пайплайнами деплоя, внутренними инструментами или где угодно, где вы можете сделать аутентифицированный HTTP-запрос.

Триггеры API добавляются к существующей рутине из веба. CLI в настоящее время не может создавать или отзывать токены.

1. **Откройте рутину для редактирования**

   Перейдите на [claude.ai/code/routines](https://claude.ai/code/routines), нажмите на рутину, которую хотите запускать через API, затем нажмите значок карандаша, чтобы открыть **Edit routine**.

2. **Добавьте триггер API**

   Прокрутите до раздела **Select a trigger** ниже поля **Instructions**, нажмите **Add another trigger** и выберите **API**.

3. **Скопируйте URL и сгенерируйте токен**

   Модальное окно показывает URL для этой рутины вместе с примером команды curl. Скопируйте URL, затем нажмите **Generate token** и немедленно скопируйте токен. Токен показывается один раз и не может быть получен позже, поэтому храните его в безопасном месте, например в хранилище секретов вашего инструмента оповещения.

4. **Вызовите эндпоинт**

   Отправляйте токен в заголовке `Authorization: Bearer`, когда делаете POST на URL. Раздел [Запуск рутины](#запуск-рутины) ниже показывает полный пример.

У каждой рутины есть свой собственный токен, ограниченный запуском только этой рутины. Чтобы сменить его (ротация) или отозвать, вернитесь в то же модальное окно и нажмите **Regenerate** или **Revoke**.

#### Запуск рутины

Отправьте POST-запрос на эндпоинт `/fire` с bearer-токеном в заголовке `Authorization`. Тело запроса принимает опциональное поле `text` для контекста, специфичного для запуска, такого как тело алерта или упавший лог, которое передаётся рутине наряду с её сохранённым промптом. Значение — это свободный текст, который не парсится: если вы отправите JSON или другую структурированную полезную нагрузку, рутина получит её как литеральную строку.

Пример ниже запускает рутину из оболочки:

```bash
curl -X POST https://api.anthropic.com/v1/claude_code/routines/trig_01ABCDEFGHJKLMNOPQRSTUVW/fire \
  -H "Authorization: Bearer sk-ant-oat01-xxxxx" \
  -H "anthropic-beta: experimental-cc-routine-2026-04-01" \
  -H "anthropic-version: 2023-06-01" \
  -H "Content-Type: application/json" \
  -d '{"text": "Sentry alert SEN-4521 fired in prod. Stack trace attached."}'
```

Успешный запрос возвращает JSON-тело с ID и URL новой сессии:

```json
{
  "type": "routine_fire",
  "claude_code_session_id": "session_01HJKLMNOPQRSTUVWXYZ",
  "claude_code_session_url": "https://claude.ai/code/session_01HJKLMNOPQRSTUVWXYZ"
}
```

Откройте URL сессии в браузере, чтобы наблюдать за запуском в реальном времени, проверить изменения или продолжить разговор вручную.

> **Предупреждение.** Эндпоинт `/fire` поставляется под бета-заголовком `experimental-cc-routine-2026-04-01`. Форматы запросов и ответов, лимиты частоты и семантика токенов могут измениться, пока функция находится в исследовательском превью. Ломающие изменения поставляются за новыми датированными версиями бета-заголовка, а две самые последние предыдущие версии заголовка продолжают работать, чтобы у вызывающих сторон было время на миграцию.

#### Справочник API

Полный справочник по API, включая все ответы об ошибках, правила валидации и лимиты полей, см. в [Запуск рутины через API](https://platform.claude.com/docs/en/api/claude-code/routines-fire) в документации Claude Platform.

Эндпоинт `/fire` доступен только пользователям claude.ai и не является частью поверхности API Claude Platform.

### Добавление триггера GitHub

Триггер GitHub автоматически запускает новую сессию, когда на подключённом репозитории происходит подходящее событие. Каждое подходящее событие запускает свою собственную сессию.

> **Примечание.** Во время исследовательского превью события веб-хуков GitHub подчиняются почасовым лимитам на уровне рутины и на уровне аккаунта. События сверх лимита отбрасываются, пока окно не сбросится. Смотрите ваши текущие лимиты на [claude.ai/code/routines](https://claude.ai/code/routines).

Триггеры GitHub настраиваются только из веб-интерфейса.

1. **Откройте рутину для редактирования**

   Перейдите на [claude.ai/code/routines](https://claude.ai/code/routines), нажмите на рутину, затем нажмите значок карандаша, чтобы открыть **Edit routine**.

2. **Добавьте триггер события GitHub**

   Прокрутите до раздела **Select a trigger**, нажмите **Add another trigger** и выберите **GitHub event**.

3. **Установите приложение Claude GitHub App**

   Приложение Claude GitHub App должно быть установлено на репозиторий, на который вы хотите подписаться. Настройка триггера предложит установить его, если оно ещё не установлено.

   > **Примечание.** Запуск `/web-setup` в CLI предоставляет доступ к репозиторию для клонирования, но не устанавливает приложение Claude GitHub App и не включает доставку веб-хуков. Триггеры GitHub требуют установки приложения Claude GitHub App, которую предлагает настройка триггера.

4. **Настройте триггер**

   Выберите репозиторий, выберите событие из списка [поддерживаемых событий](#поддерживаемые-события) и опционально добавьте фильтры. Сохраните триггер.

#### Поддерживаемые события

Триггеры GitHub могут подписываться на любую из следующих категорий событий. Внутри каждой категории вы можете выбрать конкретное действие, например `pull_request.opened`, или реагировать на все действия в категории.

| Событие | Срабатывает, когда |
| :--- | :--- |
| Pull request | PR открыт, закрыт, назначен, помечен меткой, синхронизирован или иным образом обновлён |
| Release | Релиз создан, опубликован, отредактирован или удалён |

#### Фильтрация pull request'ов

Используйте фильтры, чтобы сузить, какие pull request'ы запускают новую сессию. Для срабатывания рутины должны совпасть все условия фильтра. Доступные поля фильтрации:

| Фильтр | Совпадает с |
| :--- | :--- |
| Author | Имя пользователя GitHub автора PR |
| Title | Текст заголовка PR |
| Body | Текст описания PR |
| Base branch | Ветка, на которую нацелен PR |
| Head branch | Ветка, из которой исходит PR |
| Labels | Метки, применённые к PR |
| Is draft | Находится ли PR в состоянии черновика |
| Is merged | Был ли PR смёрджен |

Каждый фильтр сочетает поле с оператором: equals (равно), contains (содержит), starts with (начинается с), is one of (одно из), is not one of (не одно из) или matches regex (совпадает с регулярным выражением).

Оператор `matches regex` проверяет всё значение поля, а не подстроку внутри него. Чтобы совпасть с любым заголовком, содержащим `hotfix`, напишите `.*hotfix.*`. Без окружающих `.*` фильтр совпадёт только с заголовком, который в точности равен `hotfix`, без чего-либо до или после. Для литерального сопоставления подстроки без синтаксиса регулярных выражений используйте вместо этого оператор `contains`.

Несколько примеров комбинаций фильтров:

* **Проверка модуля auth**: base branch `main`, head branch содержит `auth-provider`. Отправляет любой PR, затрагивающий аутентификацию, специализированному ревьюеру.
* **Только готовые к проверке**: is draft равно `false`. Пропускает черновики, чтобы рутина запускалась только когда PR готов к проверке.
* **Бэкпорт по метке**: labels включают `needs-backport`. Запускает рутину порта-в-другую-ветку только когда мейнтейнер помечает PR.

#### Как сессии сопоставляются с событиями

Каждое подходящее событие GitHub запускает новую сессию. Повторное использование сессий между событиями недоступно для рутин, запускаемых по триггеру GitHub, поэтому два обновления PR порождают две независимые сессии.

## Управление рутинами

Нажмите на рутину в списке, чтобы открыть её страницу деталей. Страница деталей показывает репозитории рутины, коннекторы, промпт, расписание, токены API, триггеры GitHub и список прошлых запусков.

### Просмотр запусков и взаимодействие с ними

Нажмите на любой запуск, чтобы открыть его как полноценную сессию. Оттуда вы можете увидеть, что сделал Claude, проверить изменения, создать pull request или продолжить разговор. Каждая сессия запуска работает как любая другая сессия: используйте выпадающее меню рядом с заголовком сессии, чтобы переименовать, заархивировать или удалить её.

> **Примечание.** Зелёный статус в списке запусков означает, что сессия запустилась и завершилась без инфраструктурной ошибки. Он не означает, что задача в вашем промпте выполнена успешно. Откройте запуск, чтобы прочитать транскрипт и подтвердить, что Claude действительно сделал. Заблокированные сетевые запросы, отсутствующие инструменты коннекторов и сбои на уровне задачи — всё это отображается там, а не в индикаторе статуса.

### Редактирование и управление рутинами

Со страницы деталей рутины вы можете:

* Нажать **Run now**, чтобы запустить немедленно, не дожидаясь следующего запланированного времени.
* Использовать переключатель в разделе **Repeats**, чтобы приостановить или возобновить расписание. Приостановленные рутины сохраняют свою конфигурацию, но не запускаются, пока вы их снова не включите.
* Нажать значок карандаша, чтобы открыть **Edit routine** и изменить имя, промпт, репозитории, окружение, коннекторы или любой из триггеров рутины. Раздел **Select a trigger** — это место, где вы добавляете или удаляете расписания, токены API и триггеры событий GitHub.
* Нажать значок удаления, чтобы удалить рутину. Прошлые сессии, созданные рутиной, остаются в вашем списке сессий.

### Репозитории и разрешения на ветки

Рутинам нужен доступ к GitHub для клонирования репозиториев. Когда вы создаёте рутину из CLI командой `/schedule`, Claude проверяет, подключён ли GitHub к вашему аккаунту, и предлагает запустить `/web-setup`, если нет. См. [Варианты аутентификации GitHub](https://code.claude.com/docs/en/claude-code-on-the-web#github-authentication-options) для двух способов предоставить доступ.

Каждый добавленный вами репозиторий клонируется при каждом запуске. Claude начинает с ветки по умолчанию репозитория, если только ваш промпт не указывает иное.

По умолчанию Claude может делать push только в ветки с префиксом `claude/`. Это предотвращает случайное изменение рутинами защищённых или долгоживущих веток. Чтобы снять это ограничение для конкретного репозитория, включите **Allow unrestricted branch pushes** для этого репозитория при создании или редактировании рутины.

### Коннекторы

Рутины могут использовать ваши подключённые MCP-коннекторы для чтения из внешних сервисов и записи в них во время каждого запуска. Например, рутина, сортирующая запросы в поддержку, может читать из канала Slack и создавать задачи в Linear.

Коннекторы — это [интеграции claude.ai](https://code.claude.com/docs/en/mcp#use-mcp-servers-from-claude-ai) в вашем аккаунте. MCP-серверы, добавленные вами локально в CLI командой `claude mcp add`, хранятся на вашей машине, а не в вашем аккаунте claude.ai, поэтому не появляются в списке коннекторов. Чтобы использовать один из таких серверов в рутине, добавьте его как коннектор на [claude.ai/customize/connectors](https://claude.ai/customize/connectors) или объявите его в закоммиченном [`.mcp.json`](https://code.claude.com/docs/en/mcp#project-scope), чтобы он был частью клонированного репозитория.

Когда вы создаёте рутину, все ваши текущие подключённые коннекторы включаются по умолчанию. Удалите те, которые не нужны, чтобы ограничить, к каким инструментам Claude имеет доступ во время запуска. Вы также можете добавить коннекторы прямо из формы рутины.

Чтобы управлять коннекторами или добавлять их вне формы рутины, посетите **Settings > Connectors** на claude.ai или используйте `/schedule update` в CLI.

### Окружения и сетевой доступ

Каждая рутина выполняется в [облачном окружении](https://code.claude.com/docs/en/claude-code-on-the-web#the-cloud-environment), которое контролирует сетевой доступ, переменные окружения и скрипты настройки. Рутина наследует сетевую политику окружения при каждом запуске.

Окружение **Default** использует сетевой доступ **Trusted**: доступен [список разрешённых по умолчанию](https://code.claude.com/docs/en/claude-code-on-the-web#default-allowed-domains) реестров пакетов, API облачных провайдеров, реестров контейнеров и распространённых доменов для разработки, но произвольные домены недоступны. Исходящие запросы к другим хостам завершаются с ошибкой `403` и `x-deny-reason: host_not_allowed`. Трафик MCP-коннекторов маршрутизируется через серверы Anthropic, поэтому коннекторы, которые вы добавляете к рутине, работают без добавления их хостов в **Allowed domains**. Удалите любые коннекторы, которые вам не нужны, в разделе [Коннекторы](#коннекторы).

Чтобы разрешить дополнительные домены:

1. **Откройте рутину для редактирования**

   На странице деталей рутины нажмите значок карандаша, чтобы открыть **Edit routine**.

2. **Откройте селектор окружения**

   Ниже поля **Instructions** выберите значок облака, показывающий имя вашего окружения, например **Default**.

3. **Откройте настройки окружения**

   Наведите курсор на окружение в списке и нажмите значок настроек, появляющийся справа.

4. **Измените уровень сетевого доступа**

   В диалоге **Update cloud environment** измените **Network access** на **Custom** и введите свои домены в **Allowed domains**. Отметьте **Also include default list of common package managers**, чтобы сохранить [список разрешённых по умолчанию](https://code.claude.com/docs/en/claude-code-on-the-web#default-allowed-domains) наряду с вашими пользовательскими доменами. Выберите **Full** вместо этого для неограниченного доступа.

5. **Сохраните**

   Нажмите **Save changes**. Новая политика применяется со следующего запуска.

См. [Сетевой доступ](https://code.claude.com/docs/en/claude-code-on-the-web#network-access) для подробностей об уровнях доступа и списке разрешённых по умолчанию.

## Использование и лимиты

Рутины расходуют использование подписки так же, как интерактивные сессии. В дополнение к стандартным лимитам подписки у рутин есть дневной лимит на то, сколько запусков может стартовать на аккаунт. Смотрите ваше текущее потребление и оставшиеся дневные запуски рутин на [claude.ai/code/routines](https://claude.ai/code/routines) или [claude.ai/settings/usage](https://claude.ai/settings/usage).

Когда рутина достигает дневного лимита или лимита использования вашей подписки, организации с включёнными кредитами использования (usage credits) могут продолжать запускать рутины с оплатой сверх лимита (metered overage). Без кредитов использования дополнительные запуски отклоняются, пока окно не сбросится. Включите кредиты использования в **Settings > Billing** на claude.ai.

Однократные запуски не учитываются в дневном лимите рутин. Они расходуют ваше обычное использование подписки, как любая другая сессия, но освобождены от дневной квоты запусков рутин на аккаунт.

## Устранение неполадок

### `/schedule` показывает «No commands match» или «Unknown command»

CLI скрывает `/schedule`, когда одно из его требований не выполнено, поэтому меню команд показывает `No commands match "/schedule"`, пока вы печатаете, а отправка команды возвращает `Unknown command: /schedule`. Причина обычно одна из следующих:

* Вы аутентифицированы с помощью ключа API Console или облачного провайдера, такого как Bedrock, Vertex или Foundry. `/schedule` требует входа по подписке claude.ai. Если в вашей оболочке задана `ANTHROPIC_API_KEY` или `ANTHROPIC_AUTH_TOKEN`, или в `settings.json` задан `apiKeyHelper`, сначала удалите это, поскольку они имеют приоритет над входом в claude.ai
* `DISABLE_TELEMETRY`, `DO_NOT_TRACK`, `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC` или `DISABLE_GROWTHBOOK` задана в вашем окружении оболочки или в блоке `env` [файла `settings.json`](https://code.claude.com/docs/en/settings#available-settings). Они отключают загрузку фича-флагов, от которой зависит `/schedule`
* Вы находитесь внутри сессии Claude Code в вебе. Управляйте рутинами из [веб-интерфейса](https://claude.ai/code/routines) вместо этого
* Ваш CLI старше v2.1.81. Запустите `claude update`

Вы всегда можете создавать рутины и управлять ими на [claude.ai/code/routines](https://claude.ai/code/routines) независимо от того, как настроен CLI.

### «Routines are disabled by your organization's policy»

Владелец (Owner) в вашей организации Team или Enterprise, вероятно, отключил переключатель **Routines** на [claude.ai/admin-settings/claude-code](https://claude.ai/admin-settings/claude-code). Это серверная настройка организации, поэтому её нельзя переопределить из вашей локальной конфигурации. Попросите Владельца включить рутины для вашей организации.
