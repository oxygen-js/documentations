С тех пор как появились сигналы (Signals), команда NgRx работала над хранилищем (store), которое опирается на этот новый реактивный кирпичик. Новый NgRx Signal Store вышел вместе с NgRx 17, целиком построен на сигналах и при этом отлично расширяется.

В этой статье мы разберём, как встроить его в своё приложение. Для этого мы рассмотрим 3+1 разных «вкусов» его использования.

📂 [Исходный код](https://github.com/manfredsteyer/standalone-example-cli)

## Устанавливаем пакет

Чтобы поставить Signal Store, достаточно добавить в приложение пакет `@ngrx/signals`:

```
npm i @ngrx/signals
```

## Вкус 1: Лёгкий вариант с signalState

🔀 _Ветка:_ `arc-signal-store`

Самый лёгкий способ управлять сигналами через Signal Store — это его функция `signalState` (не путайте её с функцией `signalStore`). Она создаёт простой контейнер, который управляет переданным состоянием (state) с помощью сигналов. Этот контейнер представлен типом `SignalState`:

```
@Injectable({ providedIn: 'root' })

import { signalState } from '@ngrx/signals';

[...]

export class FlightBookingFacade {

    [...]

    private state = signalState({
        from: 'Paris',
        to: 'London',
        preferences: {
          directConnection: false,
          maxPrice: 350,
        },        
        flights: [] as Flight[],
        basket: {} as Record<number, boolean>,
    });

    // получаем сигналы только для чтения
    flights = this.state.flights;
    from = this.state.from;
    to = this.state.to;
    basket = this.state.basket;

  [...]
}
```

Каждое свойство состояния верхнего уровня получает свой собственный сигнал. Эти свойства отдаются как сигналы только для чтения (read-only), что отделяет чтение от записи: те, кто пользуется сигналами, могут лишь читать их значения. А для обновления состояния сервис, который инкапсулирует это состояние, предоставляет методы (см. ниже). Так гарантируется, что состояние можно менять только строго определённым образом.

Кроме того, вложенные объекты — например, тот, что задан свойством `preferences` выше, — превращаются во вложенные сигналы. Поэтому можно получить как весь объект `preferences` целиком в виде сигнала, так и его отдельные свойства:

```
const ps = this.state.preferences();
const direct = this.state.preferences.directConnection();
```

Пока что для массивов это не реализовано: предполагается, что будущие сигнальные компоненты (Signal Components) в Angular решат эту задачу, создавая по сигналу на каждый перебираемый элемент.

### Выбираем и вычисляем сигналы

Поскольку Signal Store отдаёт состояние в виде сигналов, мы можем напрямую использовать функцию Angular `computed`:

```
selected = computed(() =>
  this.flights().filter((f) => this.basket()[f.id])
);
```

Здесь `computed` играет ту же роль, что и селекторы (Selectors) в классическом NgRx Store на базе Redux: он позволяет вычислять разные представления состояния под разные сценарии. Эти так называемые View Models пересчитываются только тогда, когда меняется хотя бы один из лежащих в их основе сигналов.

### Обновляем состояние

Для обновления `SignalState` в Signal Store есть функция `patchState`:

```
import { patchState } from '@ngrx/signals';

[...]

updateCriteria(from: string, to: string): void {
  patchState(this.state, { from, to })
}
```

Здесь мы передаём контейнер состояния и частичное состояние (partial state). Как вариант, можно передать функцию, которая принимает текущее состояние и преобразует его в новое:

```
updateBasket(id: number, selected: boolean): void {
  patchState(this.state, state => ({
    basket: {
      ...state.basket,
      [id]: selected,
    },
  }));
}
```

## Побочные эффекты

Помимо обновления состояния, методы могут запускать и побочные эффекты (side effects) — например, загрузку и сохранение объектов:

```
async load() {
  if (!this.from() || !this.to()) return;

  const flights = await this.flightService.findPromise(
    this.from(),
    this.to()
  );

  patchState(this.state, { flights });
}
```

### Отделяем намерение от исполнения

Иногда тот, кто вызывает `patchState`, знает лишь то, что какое-то состояние нужно обновить, но не знает, где именно оно находится. На такие случаи можно завести апдейтеры (Updaters). Апдейтер — это просто функция, которая принимает текущее состояние и возвращает его обновлённую версию:

```
type BasketSlice = { basket: Record<number, boolean> };
type BasketUpdateter = (state: BasketSlice) => BasketSlice;

export function updateBasket(flightId: number, selected: boolean): BasketUpdateter {
  return (state) => ({
    ...state,
    basket: {
      ...state.basket,
      [flightId]: selected,
    },
  });
}
```

Также можно вернуть и частичное состояние — оно будет «подмешано» (patched) к текущему состоянию:

```
type BasketSlice = { basket: Record<number, boolean> };
type BasketUpdateter = (state: BasketSlice) => BasketSlice;

export function updateBasket(flightId: number, selected: boolean): BasketUpdateter {
  return (state) => ({
    basket: {
      ...state.basket,
      [flightId]: selected,
    },
  });
}
```

Если проецировать текущее состояние не нужно, можно просто вернуть частичное состояние. В этом случае внутреннюю функцию можно опустить:

```
export function updateFlights(flights: Flight[]) {
  return { flights };
}
```

Апдейтеры можно определять на «суверенной территории» самого `Store` (то есть `signalState`). Для потребителя это просто чёрный ящик:

```
patchState(updateBasket(id, selected))
```

Передача апдейтера в `patchState` выражает намерение. Это похоже на диспатч экшена (Action) в классическом NgRx store. Но, в отличие от Redux, никакого механизма событий тут нет, и мы не можем запретить вызывающему коду напрямую передавать собственные апдейтеры. Именно поэтому я прячу `SignalStore` за фасадом.

## Вкус 2: Мощный вариант с signalStore

🔀 _Ветка:_ `arc-signal-store-2`

Как и `signalState`, функция `signalStore` создаёт контейнер, управляющий состоянием через сигналы. Но теперь этот контейнер — полноценное хранилище (Store), в котором есть не только сигналы состояния, но и вычисляемые сигналы (computed Signals), а также методы для обновления состояния и для запуска побочных эффектов. Поэтому писать фасад вручную, как выше, нужно гораздо реже.

Технически Store — это сервис Angular, собранный из нескольких готовых фич (features):

```
export const FlightBookingStore = signalStore(
  { providedIn: 'root' },
  withState({
    from: 'Paris',
    to: 'London',
    initialized: false,
    flights: [] as Flight[],
    basket: {} as Record<number, boolean>,
  }),

  // Подключаем дополнительные фичи
  withComputed([...]),
  withMethods([...]),
  withHooks([...]),
)
```

В этом случае сервис тоже регистрируется в корневой области видимости (root scope). Если убрать `{ providedIn: 'root' }`, сервис придётся регистрировать вручную — например, при бутстрапе приложения, в конфигурации роутера или на уровне компонента.

### Выбираем и вычисляем сигналы

Фича `withComputed` берёт store с его сигналами состояния и описывает объект из вычисляемых сигналов:

```
withComputed((store) => ({
  selected: computed(() => store.flights().filter((f) => store.basket()[f.id])),
  criteria: computed(() => ({ from: store.from(), to: store.to() })),
})),
```

Возвращённые вычисляемые сигналы становятся частью store. Более компактный вариант — сразу деструктурировать переданный store:

```
withComputed(({ flights, basket, from, to }) => ({
  selected: selectSignal(() => flights().filter((f) => basket()[f.id])),
  criteria: selectSignal(() => ({ from: from(), to: to() })),
})),
```

### Методы для обновления состояния и побочных эффектов

Аналогично `withComputed`, фича `withMethods` тоже берёт store и возвращает объект с методами:

```
withMethods((state) => {
  const { basket, flights, from, to, initialized } = state;
  const flightService = inject(FlightService);

  return {
    updateCriteria: (from: string, to: string) => {
      patchState(state, { from, to });
    },
    updateBasket: (flightId: number, selected: boolean) => {
      patchState(state, {
        basket: {
          ...basket(),
          [flightId]: selected,
        },
      });
    },
    delay: () => {
      const currentFlights = flights();
      const flight = currentFlights[0];

      const date = addMinutes(flight.date, 15);
      const updFlight = { ...flight, date };
      const updFlights = [updFlight, ...currentFlights.slice(1)];

      patchState(state, { flights: updFlights });
    },
    load: async () => {
      if (!from() || !to()) return;
      const flights = await flightService.findPromise(from(), to());
      patchState(state, { flights });
    }     
  };
}),
```

`withMethods` выполняется в контексте внедрения зависимостей (injection context), поэтому внутри можно использовать `inject`, чтобы получить нужные сервисы. После того как `withMethods` отработала, полученные методы добавляются в store.

### Используем Store

С точки зрения вызывающего кода store очень похож на показанный выше фасад. Мы можем внедрить его в компонент-потребитель:

```
@Component([...])
export class FlightSearchComponent {
  private store = inject(FlightBookingStore);

  from = this.store.from;
  to = this.store.to;
  basket = this.store.basket;
  flights = this.store.flights;
  selected = this.store.selected;

  async search() {
    this.store.load();
  }

  delay(): void {
    this.store.delay();
  }

  updateCriteria(from: string, to: string): void {
    this.store.updateCriteria(from, to);
  }

  updateBasket(id: number, selected: boolean): void {
    this.store.updateBasket(id, selected);
  }
}
```

### Хуки (Hooks)

Функция `withHooks` даёт ещё одну фичу — она позволяет настроить хуки жизненного цикла, которые срабатывают при инициализации и при уничтожении store:

```
withHooks({
  onInit({ load }) {
    load()
  },
  onDestroy({ flights }) {
    console.log('flights are destroyed now', flights());
  },
}),
```

Оба хука получают store. И снова с помощью деструктуризации можно сосредоточиться лишь на части членов store.

### rxMethod

🔀 _Ветка:_ `arc-signal-store-rx`

Сигналы просты в использовании, но они не полностью заменяют RxJS. Чтобы пользоваться RxJS и его мощными операторами, Signal Store предоставляет дополнительную точку входа `@ngrx/signals/rxjs-interop` с функцией `rxMethod<T>`. Она позволяет работать с Observable, который представляет побочные эффекты, автоматически запускающиеся при изменении определённых значений:

```
import { rxMethod } from '@ngrx/signals/rxjs-interop';

[...]

withMethods(({ $update, basket, flights, from, to, initialized }) => {
  const flightService = inject(FlightService);

  return {
    [...]
    connectCriteria: rxMethod<Criteria>((c$) => c$.pipe(
      filter(c => c.from.length >= 3 && c.to.length >= 3),
      debounceTime(300),
      switchMap((c) => flightService.find(c.from, c.to)),
      tap(flights => patchState(state, { flights }))
    ))
  }
});
```

Параметр типа `T` задаёт тип, с которым работает `rxMethod`. Хотя `rxMethod` получает `Obserable<T>`, вызывающий код может передать `Observable<T>`, `Signal<T>` или просто `T` напрямую. В двух последних случаях переданные значения преобразуются в Observable.

Определив `rxMethod`, где-нибудь ещё в приложении — например, в хуке или в обычном методе — можно вызвать этот эффект:

```
withHooks({
  onInit({ loadBy, criteria }) {
    connectCriteria(criteria());
  },
})
```

Здесь передаётся сигнал criteria — вычисляемый сигнал. Каждый раз, когда этот сигнал меняется, эффект внутри `connectCriteria` выполняется заново.

## Пользовательские фичи — дорога к новым вкусам

🔀 _Ветка:_ `arc-signal-store-custom`

Помимо настройки Store встроенными фичами, каждый может написать собственную фичу, чтобы автоматизировать повторяющиеся задачи. Песочница, которую сделал [Марко Станимирович](https://twitter.com/MarkoStDev) — контрибьютор NgRx, стоящий за Signal Store, — содержит несколько примеров таких фич.

Один из примеров в этом репозитории — [фича CallState](https://github.com/markostanimirovic/ngrx-signal-store-playground/blob/main/src/app/shared/call-state.feature.ts), задающая свойство состояния, которое сообщает о статусе текущего HTTP-запроса:

```
export type CallState = 'init' | 'loading' | 'loaded' | { error: string };
```

В этом разделе на этом примере я покажу, как создавать пользовательские фичи.

### Определяем пользовательские фичи

Фичу обычно создают вызовом `signalStoreFeature`. Эта функция надстраивает новую фичу поверх существующих:

```
// Взято из: https://github.com/markostanimirovic/ngrx-signal-store-playground/blob/main/src/app/shared/call-state.feature.ts

import { computed } from '@angular/core';
import {
  signalStoreFeature,
  withComputed,
  withState,
} from '@ngrx/signals';

export type CallState = 'init' | 'loading' | 'loaded' | { error: string };

export function withCallState() {
  return signalStoreFeature(
    withState<{ callState: CallState }>({ callState: 'init' }),
    withComputed(({ callState }) => ({
      loading: computed(() => callState() === 'loading'),
      loaded: computed(() => callState() === 'loaded'),
      error: computed(() => {
        const state = callState();
        return typeof state === 'object' ? state.error : null
      }),
    }))
  );
}
```

Для свойств состояния, которые добавляет фича, можно предоставить апдейтеры (`Updaters`):

```
export function setLoading(): { callState: CallState } {
  return { callState: 'loading' };
}

export function setLoaded(): { callState: CallState } {
  return { callState: 'loaded' };
}

export function setError(error: string): { callState: CallState } {
  return { callState: { error } };
}
```

Апдейтеры позволяют потребителю менять состояние фичи, не зная, как именно оно устроено.

### Используем пользовательские фичи

Чтобы использовать пользовательские фичи, просто вызовите соответствующую фабрику при настройке store:

```
export const FlightBookingStore = signalStore(
  { providedIn: 'root' },
  withState({ [...] }),

  // Добавляем фичу:
  withCallState(),
  [...]

  withMethods([...])
  [...]
);
```

Предоставленные свойства, методы и апдейтеры можно использовать в методах Store:

```
load: async () => {
  if (!from() || !to()) return;

  // Устанавливаем callState через апдейтер
  patchState(state, setLoading());

  const flights = await flightService.findPromise(from(), to());
  patchState(state, { flights });

  // Устанавливаем callState через апдейтер
  patchState(state, setLoaded());
},
```

Потребитель store тоже видит свойства, которые предоставляет фича:

```
private store = inject(FlightBookingStore);

flights = this.store.flightEntities;
loading = this.store.loading;
```

Поскольку каждая фича преобразует свойства и методы Store, важно вызывать их в правильном порядке. Если предположить, что методы, зарегистрированные через `withMethods`, используют `CallState`, то `withCallState` нужно вызвать раньше `withMethods`.

## Вкус 3: Встроенные фичи вроде управления сущностями

🔀 _Ветка:_ `arc-signal-store-entities`

NgRx Signal Store уже идёт с удобным расширением для управления сущностями (entities). Оно находится в дополнительной точке входа `@ngrx/signals/entities` и предоставляет структуры данных для сущностей, а также несколько апдейтеров — например, для вставки сущностей или для обновления одной сущности по id.

Чтобы настроить управление сущностями, просто вызовите функцию `withEntities`:

```
import { withEntities } from '@ngrx/signals/entities';

const BooksStore = signalStore(
  [...]

  // Описываем сущность
  withEntities({ entity: type<Flight>(), collection: 'flight' }),

  // withEntities создал для нас сигнал flightEntities:
  withComputed(({ flightEntities, basket, from, to }) => ({
    selected: computed(() => flightEntities().filter((f) => basket()[f.id])),
    criteria: computed(() => ({ from: from(), to: to() })),
  })),

  withMethods((state) => {
    const { basket, flightEntities, from, to, initialized } = state;
    const flightService = inject(FlightService);

    return {
      [...],

      load: async () => {
        if (!from() || !to()) return;
        patchState(state, setLoading());

        const flights = await flightService.findPromise(from(), to());

        // Обновляем сущности готовым апдейтером setAllEntities
        patchState(state, setAllEntities(flights, { collection: 'flight' }));
        patchState(state, setLoaded());
      },

      [...],
    };
  }),
);
```

Переданное имя коллекции предотвращает конфликты имён. В нашем случае коллекция называется `flight`, и поэтому фича создаёт несколько свойств, начинающихся с `flight`, например `flightEntities`.

Готовых апдейтеров довольно много:

* `addEntity`
* `addEntities`
* `removeEntity`
* `removeEntities`
* `removeAllEntities`
* `setEntity`
* `setEntities`
* `setAllEntities`
* `updateEntity`
* `updateEntities`
* `updateAllEntities`

Как и в `@ngrx/entities`, внутри сущности хранятся в нормализованном виде. То есть они лежат в словаре, где первичные ключи отображаются на объекты сущностей. Так их проще «склеивать» во View Models, нужные для конкретных сценариев.

Поскольку нашу коллекцию мы назвали `flight`, `withEntities` создаёт сигнал `flightEntityMap`, отображающий id рейсов на наши объекты рейсов. Также он создаёт сигнал `flightIds`, содержащий все id по порядку. Оба используются добавленным вычисляемым сигналом `flightEntities`, который мы применили выше. Он возвращает все рейсы в виде массива, соблюдая порядок id из `flightIds`. Поэтому, если нужно переставить рейсы местами, просто обновите свойство `flightIds` соответствующим образом.

Чтобы строить структуры вроде `flightEntityMap`, апдейтерам нужно знать, как называется id сущности. По умолчанию предполагается свойство `id`. Если id называется иначе, об этом можно сообщить апдейтеру через свойство `idKey`:

```
patchState(state, setAllEntities(flights, { collection: 'flight', idKey: 'flightId' }));
```

Переданное свойство должно быть `string` или `number`. Если оно другого типа или его вообще нет, вы получите ошибку компиляции.

## Заключение

Будущий NgRx Signal Store позволяет управлять состоянием с помощью сигналов. Самый лёгкий способ использовать эту библиотеку — просто взять контейнер `SignalState`. Эта структура данных предоставляет по сигналу на каждое свойство состояния. Эти сигналы доступны только для чтения. Для обновления состояния можно использовать функцию `patchState`. А чтобы обновления происходили только строго определённым образом, `signalstate` можно спрятать за фасадом.

`SignalStore` мощнее и позволяет регистрировать опциональные фичи. Они задают состояние, которым нужно управлять, а также методы, работающие с ним. `SignalStore` можно предоставить как сервис и внедрять его в потребителей.

Ещё `SignalStore` предоставляет механизм расширения для реализации пользовательских фич, упрощающих повторяющиеся задачи. «Из коробки» Signal Store идёт с весьма удобной фичей для управления сущностями.
