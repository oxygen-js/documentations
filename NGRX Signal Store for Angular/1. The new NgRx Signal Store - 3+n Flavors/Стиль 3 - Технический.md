С момента появления сигналов (Signals) команда NgRx работала над хранилищем (store), использующим этот новый реактивный примитив. Новый NgRx Signal Store был выпущен с NgRx 17, полностью основан на сигналах и обладает высокой расширяемостью.

В статье рассматривается, как интегрировать его в приложение. Для этого демонстрируются 3+1 различных вариантов его использования.

📂 [Исходный код](https://github.com/manfredsteyer/standalone-example-cli)

## Получение пакета

Для установки Signal Store достаточно добавить в приложение пакет `@ngrx/signals`:

```
npm i @ngrx/signals
```

## Вариант 1: Легковесный — на основе signalState

🔀 _Ветка:_ `arc-signal-store`

Очень легковесный способ управления сигналами через Signal Store — это функция `signalState` (не путать с функцией `signalStore`). Она создаёт простой контейнер для управления переданным состоянием (state) посредством сигналов. Этот контейнер представлен типом `SignalState`:

```
@Injectable({ providedIn: 'root' })

import { signalState } from '@ngrx/signals';

[...]

export class FlightBookingFacade {

    [...]

    private state = signalState({
        from: 'Paris',
        to: 'London',
        preferences: {
          directConnection: false,
          maxPrice: 350,
        },        
        flights: [] as Flight[],
        basket: {} as Record<number, boolean>,
    });

    // извлекаем сигналы только для чтения
    flights = this.state.flights;
    from = this.state.from;
    to = this.state.to;
    basket = this.state.basket;

  [...]
}
```

Каждое свойство состояния верхнего уровня получает собственный сигнал. Эти свойства предоставляются как сигналы только для чтения (read-only), что обеспечивает разделение чтения и записи: потребители, использующие сигналы, могут только читать их значения. Для обновления состояния сервис, инкапсулирующий состояние, предоставляет методы (см. ниже). Это гарантирует, что состояние может быть обновлено только строго определённым образом.

Кроме того, вложенные объекты — такие как предоставленный свойством `preferences` выше — преобразуются во вложенные сигналы (то есть `signalState` рекурсивно оборачивает поля объекта в сигналы). Следовательно, можно получить как весь объект `preferences` в виде сигнала, так и отдельные его свойства:

```
const ps = this.state.preferences();
const direct = this.state.preferences.directConnection();
```

В настоящее время для массивов это не реализовано, поскольку планируемые в Angular сигнальные компоненты (Signal Components) решат данный сценарий, создавая сигнал для каждого итерируемого элемента.

### Выборка и вычисление сигналов

Поскольку Signal Store предоставляет состояние в виде сигналов, можно напрямую использовать функцию Angular `computed`:

```
selected = computed(() =>
  this.flights().filter((f) => this.basket()[f.id])
);
```

Здесь `computed` выполняет ту же роль, что и селекторы (Selectors) в основанном на Redux NgRx Store: он позволяет вычислять различные представления состояния под разные сценарии использования. Эти так называемые View Models пересчитываются лениво — только при изменении хотя бы одного из сигналов-зависимостей (мемоизация).

### Обновление состояния

Для обновления `SignalState` Signal Store предоставляет функцию `patchState`:

```
import { patchState } from '@ngrx/signals';

[...]

updateCriteria(from: string, to: string): void {
  patchState(this.state, { from, to })
}
```

Здесь передаётся контейнер состояния и частичное состояние (partial state). В качестве альтернативы можно передать функцию, принимающую текущее состояние и преобразующую его в новое:

```
updateBasket(id: number, selected: boolean): void {
  patchState(this.state, state => ({
    basket: {
      ...state.basket,
      [id]: selected,
    },
  }));
}
```

## Побочные эффекты (Side Effects)

Помимо обновления состояния, методы могут запускать побочные эффекты (side effects) — например, загрузку и сохранение объектов:

```
async load() {
  if (!this.from() || !this.to()) return;

  const flights = await this.flightService.findPromise(
    this.from(),
    this.to()
  );

  patchState(this.state, { flights });
}
```

### Отделение намерения от исполнения

Иногда вызывающий код `patchState` знает лишь то, что некоторое состояние требуется обновить, но не знает, где оно расположено. Для таких случаев можно предоставить апдейтеры (Updaters). Апдейтер — это просто функция, принимающая текущее состояние и возвращающая его обновлённую версию:

```
type BasketSlice = { basket: Record<number, boolean> };
type BasketUpdateter = (state: BasketSlice) => BasketSlice;

export function updateBasket(flightId: number, selected: boolean): BasketUpdateter {
  return (state) => ({
    ...state,
    basket: {
      ...state.basket,
      [flightId]: selected,
    },
  });
}
```

Допустимо также вернуть и частичное состояние — оно будет применено (patched) поверх текущего состояния:

```
type BasketSlice = { basket: Record<number, boolean> };
type BasketUpdateter = (state: BasketSlice) => BasketSlice;

export function updateBasket(flightId: number, selected: boolean): BasketUpdateter {
  return (state) => ({
    basket: {
      ...state.basket,
      [flightId]: selected,
    },
  });
}
```

Если проекция текущего состояния не требуется, достаточно вернуть частичное состояние. В этом случае внутреннюю функцию можно опустить:

```
export function updateFlights(flights: Flight[]) {
  return { flights };
}
```

Апдейтеры можно определять на «суверенной территории» самого `Store` (то есть `signalState`). Для потребителя это просто чёрный ящик:

```
patchState(updateBasket(id, selected))
```

Передача апдейтера в `patchState` выражает намерение (intention). Это аналогично диспатчу (dispatch) экшена (Action) в классическом NgRx store. Однако, в отличие от Redux, здесь не задействован механизм событий (eventing), и нельзя запретить вызывающему коду напрямую передавать собственные апдейтеры. По последней причине я скрываю `SignalStore` за фасадом.

## Вариант 2: Полнофункциональный — на основе signalStore

🔀 _Ветка:_ `arc-signal-store-2`

Аналогично `signalState`, функция `signalStore` создаёт контейнер, управляющий состоянием через сигналы. Однако теперь этот контейнер — полноценное хранилище (Store), содержащее не только сигналы состояния, но и вычисляемые сигналы (computed Signals), а также методы для обновления состояния и для запуска побочных эффектов. Следовательно, потребность в ручном создании фасада (как выше) ниже.

Технически Store — это сервис Angular, скомпонованный из нескольких заранее существующих фич (features):

```
export const FlightBookingStore = signalStore(
  { providedIn: 'root' },
  withState({
    from: 'Paris',
    to: 'London',
    initialized: false,
    flights: [] as Flight[],
    basket: {} as Record<number, boolean>,
  }),

  // Активация дополнительных фич
  withComputed([...]),
  withMethods([...]),
  withHooks([...]),
)
```

В этом случае сервис также регистрируется в корневой области видимости (root scope). При отсутствии `{ providedIn: 'root' }` сервис необходимо регистрировать вручную — например, предоставляя его при бутстрапе приложения, в конфигурации роутера или на уровне компонента.

### Выборка и вычисление сигналов

Фича `withComputed` принимает store с его сигналами состояния и определяет объект с вычисляемыми сигналами:

```
withComputed((store) => ({
  selected: computed(() => store.flights().filter((f) => store.basket()[f.id])),
  criteria: computed(() => ({ from: store.from(), to: store.to() })),
})),
```

Возвращённые вычисляемые сигналы становятся частью store. Более компактный вариант — прямая деструктуризация переданного store:

```
withComputed(({ flights, basket, from, to }) => ({
  selected: selectSignal(() => flights().filter((f) => basket()[f.id])),
  criteria: selectSignal(() => ({ from: from(), to: to() })),
})),
```

### Методы для обновления состояния и побочных эффектов

Аналогично `withComputed`, фича `withMethods` также принимает store и возвращает объект с методами:

```
withMethods((state) => {
  const { basket, flights, from, to, initialized } = state;
  const flightService = inject(FlightService);

  return {
    updateCriteria: (from: string, to: string) => {
      patchState(state, { from, to });
    },
    updateBasket: (flightId: number, selected: boolean) => {
      patchState(state, {
        basket: {
          ...basket(),
          [flightId]: selected,
        },
      });
    },
    delay: () => {
      const currentFlights = flights();
      const flight = currentFlights[0];

      const date = addMinutes(flight.date, 15);
      const updFlight = { ...flight, date };
      const updFlights = [updFlight, ...currentFlights.slice(1)];

      patchState(state, { flights: updFlights });
    },
    load: async () => {
      if (!from() || !to()) return;
      const flights = await flightService.findPromise(from(), to());
      patchState(state, { flights });
    }     
  };
}),
```

`withMethods` выполняется в контексте внедрения зависимостей (injection context), поэтому внутри допустимо использовать `inject` для получения сервисов. После выполнения `withMethods` полученные методы добавляются в store.

### Потребление Store

С точки зрения вызывающего кода store во многом аналогичен показанному выше фасаду. Его можно внедрить в потребляющий компонент:

```
@Component([...])
export class FlightSearchComponent {
  private store = inject(FlightBookingStore);

  from = this.store.from;
  to = this.store.to;
  basket = this.store.basket;
  flights = this.store.flights;
  selected = this.store.selected;

  async search() {
    this.store.load();
  }

  delay(): void {
    this.store.delay();
  }

  updateCriteria(from: string, to: string): void {
    this.store.updateCriteria(from, to);
  }

  updateBasket(id: number, selected: boolean): void {
    this.store.updateBasket(id, selected);
  }
}
```

### Хуки (Hooks)

Функция `withHooks` предоставляет ещё одну фичу, позволяющую настроить хуки жизненного цикла, выполняемые при инициализации или уничтожении store:

```
withHooks({
  onInit({ load }) {
    load()
  },
  onDestroy({ flights }) {
    console.log('flights are destroyed now', flights());
  },
}),
```

Оба хука получают store. И снова, используя деструктуризацию, можно сосредоточиться на подмножестве членов store.

### rxMethod

🔀 _Ветка:_ `arc-signal-store-rx`

Хотя сигналы просты в использовании, они не являются полной заменой RxJS. Для задействования RxJS и его мощных операторов Signal Store предоставляет вторичную точку входа (secondary entry point) `@ngrx/signals/rxjs-interop`, содержащую функцию `rxMethod<T>`. Она позволяет работать с Observable, представляющим побочные эффекты, которые автоматически выполняются при изменении определённых значений:

```
import { rxMethod } from '@ngrx/signals/rxjs-interop';

[...]

withMethods(({ $update, basket, flights, from, to, initialized }) => {
  const flightService = inject(FlightService);

  return {
    [...]
    connectCriteria: rxMethod<Criteria>((c$) => c$.pipe(
      filter(c => c.from.length >= 3 && c.to.length >= 3),
      debounceTime(300),
      switchMap((c) => flightService.find(c.from, c.to)),
      tap(flights => patchState(state, { flights }))
    ))
  }
});
```

Параметр типа `T` определяет тип, с которым работает `rxMethod`. Хотя `rxMethod` получает `Obserable<T>`, вызывающий код может передать `Observable<T>`, `Signal<T>` или `T` напрямую. В двух последних случаях переданные значения преобразуются в Observable (для `Signal<T>` — через интероп сигнала в поток).

После определения `rxMethod` где-либо ещё в приложении — например, в хуке или обычном методе — этот эффект можно вызвать:

```
withHooks({
  onInit({ loadBy, criteria }) {
    connectCriteria(criteria());
  },
})
```

Здесь передаётся сигнал criteria — вычисляемый сигнал. Всякий раз при изменении этого сигнала эффект внутри `connectCriteria` выполняется повторно.

## Пользовательские фичи — путь к новым вариантам использования

🔀 _Ветка:_ `arc-signal-store-custom`

Помимо конфигурирования Store встроенными фичами, каждый может написать собственные фичи для автоматизации повторяющихся задач. Песочница, предоставленная [Марко Станимировичем](https://twitter.com/MarkoStDev) — контрибьютором NgRx, стоящим за Signal Store, — содержит несколько примеров таких фич.

Один из примеров в этом репозитории — [фича CallState](https://github.com/markostanimirovic/ngrx-signal-store-playground/blob/main/src/app/shared/call-state.feature.ts), определяющая свойство состояния, информирующее о состоянии текущего HTTP-вызова:

```
export type CallState = 'init' | 'loading' | 'loaded' | { error: string };
```

В этом разделе на данном примере поясняется, как предоставлять пользовательские фичи.

### Определение пользовательских фич

Фича обычно создаётся вызовом `signalStoreFeature`. Эта функция конструирует новую фичу поверх существующих.

```
// Взято из: https://github.com/markostanimirovic/ngrx-signal-store-playground/blob/main/src/app/shared/call-state.feature.ts

import { computed } from '@angular/core';
import {
  signalStoreFeature,
  withComputed,
  withState,
} from '@ngrx/signals';

export type CallState = 'init' | 'loading' | 'loaded' | { error: string };

export function withCallState() {
  return signalStoreFeature(
    withState<{ callState: CallState }>({ callState: 'init' }),
    withComputed(({ callState }) => ({
      loading: computed(() => callState() === 'loading'),
      loaded: computed(() => callState() === 'loaded'),
      error: computed(() => {
        const state = callState();
        return typeof state === 'object' ? state.error : null
      }),
    }))
  );
}
```

Для свойств состояния, добавляемых фичей, можно предоставить апдейтеры (`Updaters`):

```
export function setLoading(): { callState: CallState } {
  return { callState: 'loading' };
}

export function setLoaded(): { callState: CallState } {
  return { callState: 'loaded' };
}

export function setError(error: string): { callState: CallState } {
  return { callState: { error } };
}
```

Апдейтеры позволяют потребителю изменять состояние фичи, фактически не зная, как оно структурировано.

### Использование пользовательских фич

Для использования пользовательских фич достаточно вызвать предоставленную фабрику при настройке store:

```
export const FlightBookingStore = signalStore(
  { providedIn: 'root' },
  withState({ [...] }),

  // Добавляем фичу:
  withCallState(),
  [...]

  withMethods([...])
  [...]
);
```

Предоставленные свойства, методы и апдейтеры можно использовать в методах Store:

```
load: async () => {
  if (!from() || !to()) return;

  // Установка callState через апдейтер
  patchState(state, setLoading());

  const flights = await flightService.findPromise(from(), to());
  patchState(state, { flights });

  // Установка callState через апдейтер
  patchState(state, setLoaded());
},
```

Потребитель store также видит свойства, предоставленные фичей:

```
private store = inject(FlightBookingStore);

flights = this.store.flightEntities;
loading = this.store.loading;
```

Поскольку каждая фича трансформирует свойства и методы Store, важно вызывать их в правильном порядке. Если предположить, что методы, зарегистрированные через `withMethods`, используют `CallState`, то `withCallState` должна быть вызвана перед `withMethods`.

## Вариант 3: Встроенные фичи, например управление сущностями

🔀 _Ветка:_ `arc-signal-store-entities`

NgRx Signal Store уже поставляется с удобным расширением для управления сущностями (entities). Оно располагается во вторичной точке входа `@ngrx/signals/entities` и предоставляет структуры данных для сущностей, а также несколько апдейтеров — например, для вставки сущностей или для обновления одной сущности по id.

Для настройки управления сущностями достаточно вызвать функцию `withEntities`:

```
import { withEntities } from '@ngrx/signals/entities';

const BooksStore = signalStore(
  [...]

  // Определение сущности
  withEntities({ entity: type<Flight>(), collection: 'flight' }),

  // withEntities создал для нас сигнал flightEntities:
  withComputed(({ flightEntities, basket, from, to }) => ({
    selected: computed(() => flightEntities().filter((f) => basket()[f.id])),
    criteria: computed(() => ({ from: from(), to: to() })),
  })),

  withMethods((state) => {
    const { basket, flightEntities, from, to, initialized } = state;
    const flightService = inject(FlightService);

    return {
      [...],

      load: async () => {
        if (!from() || !to()) return;
        patchState(state, setLoading());

        const flights = await flightService.findPromise(from(), to());

        // Обновление сущностей готовым апдейтером setAllEntities
        patchState(state, setAllEntities(flights, { collection: 'flight' }));
        patchState(state, setLoaded());
      },

      [...],
    };
  }),
);
```

Переданное имя коллекции (collection) предотвращает конфликты имён. В нашем случае коллекция называется `flight`, поэтому фича создаёт несколько свойств, начинающихся с `flight`, например `flightEntities`.

Существует довольно большой набор готовых апдейтеров:

* `addEntity`
* `addEntities`
* `removeEntity`
* `removeEntities`
* `removeAllEntities`
* `setEntity`
* `setEntities`
* `setAllEntities`
* `updateEntity`
* `updateEntities`
* `updateAllEntities`

Подобно `@ngrx/entities`, внутренне сущности хранятся в нормализованном виде (normalized). Это означает, что они хранятся в словаре (dictionary), отображающем их первичные ключи на объекты сущностей. Это упрощает их объединение (join) в View Models, необходимые для конкретных сценариев использования.

Поскольку наша коллекция называется `flight`, `withEntities` создаёт сигнал `flightEntityMap`, отображающий id рейсов на объекты рейсов. Также он создаёт сигнал `flightIds`, содержащий все id в исходном порядке. Оба используются также добавленным вычисляемым сигналом `flightEntities`, применённым выше. Он возвращает все рейсы в виде массива, соблюдая порядок id внутри `flightIds`. Следовательно, чтобы переставить позиции рейсов, достаточно соответствующим образом обновить свойство `flightIds`.

Для построения структур, таких как `flightEntityMap`, апдейтерам необходимо знать, как называется id сущности. По умолчанию предполагается свойство `id`. Если id называется иначе, об этом можно сообщить апдейтеру с помощью свойства `idKey`:

```
patchState(state, setAllEntities(flights, { collection: 'flight', idKey: 'flightId' }));
```

Переданное свойство должно иметь тип `string` или `number`. Если оно имеет иной тип данных или вовсе отсутствует, возникнет ошибка компиляции (compilation error).

## Заключение

Будущий NgRx Signal Store позволяет управлять состоянием с помощью сигналов. Наиболее легковесный вариант использования этой библиотеки — просто взять контейнер `SignalState`. Эта структура данных предоставляет сигнал для каждого свойства состояния. Эти сигналы доступны только для чтения. Для обновления состояния можно использовать функцию `patchState`. Чтобы гарантировать, что обновления происходят только строго определённым образом, `signalstate` можно скрыть за фасадом.

`SignalStore` обладает большими возможностями и позволяет регистрировать опциональные фичи. Они определяют состояние, которым требуется управлять, а также методы, оперирующие им. `SignalStore` можно предоставить как сервис и внедрять в его потребителей.

`SignalStore` также предоставляет механизм расширения для реализации пользовательских фич, упрощающих повторяющиеся задачи. «Из коробки» Signal Store поставляется с весьма удобной фичей для управления сущностями.
