С момента появления сигналов (Signals) команда NgRx работала над хранилищем (store), которое использует этот новый реактивный строительный блок. Этот новый NgRx Signal Store был выпущен вместе с NgRx 17, полностью основан на сигналах и обладает высокой расширяемостью.

Эта статья показывает, как встроить его в ваше приложение. Для этого она демонстрирует 3+1 различных разновидностей его использования.

📂 [Исходный код](https://github.com/manfredsteyer/standalone-example-cli)

## Получение пакета

Чтобы установить Signal Store, вам достаточно добавить в ваше приложение пакет `@ngrx/signals`:

```
npm i @ngrx/signals
```

## Разновидность 1: Лёгкая с signalState

🔀 _Ветка:_ `arc-signal-store`

Очень лёгкий способ управлять сигналами с помощью Signal Store — это его функция `signalState` (не путать с функцией `signalStore`). Она создаёт простой контейнер для управления переданным состоянием (state) с помощью сигналов. Этот контейнер представлен типом `SignalState`:

```
@Injectable({ providedIn: 'root' })

import { signalState } from '@ngrx/signals';

[...]

export class FlightBookingFacade {

    [...]

    private state = signalState({
        from: 'Paris',
        to: 'London',
        preferences: {
          directConnection: false,
          maxPrice: 350,
        },        
        flights: [] as Flight[],
        basket: {} as Record<number, boolean>,
    });

    // получаем сигналы только для чтения
    flights = this.state.flights;
    from = this.state.from;
    to = this.state.to;
    basket = this.state.basket;

  [...]
}
```

Каждое свойство состояния верхнего уровня получает свой собственный сигнал. Эти свойства извлекаются как сигналы только для чтения (read-only), что обеспечивает разделение между чтением и записью: потребители, использующие сигналы, могут лишь читать их значения. Для обновления состояния сервис, инкапсулирующий состояние, предоставляет методы (см. ниже). Это гарантирует, что состояние может быть обновлено только строго определённым образом.

Кроме того, вложенные объекты, такие как тот, что предоставлен свойством `preferences` выше, приводят к вложенным сигналам. Следовательно, можно извлечь как весь объект `preferences` в виде сигнала, так и его свойства:

```
const ps = this.state.preferences();
const direct = this.state.preferences.directConnection();
```

В настоящее время это не реализовано для массивов, поскольку планируемые в Angular сигнальные компоненты (Signal Components) решат этот сценарий, создавая сигнал для каждого перебираемого элемента.

### Выбор и вычисление сигналов

Поскольку Signal Store предоставляет состояние в виде сигналов, мы можем напрямую использовать функцию Angular `computed`:

```
selected = computed(() =>
  this.flights().filter((f) => this.basket()[f.id])
);
```

Здесь `computed` служит той же цели, что и селекторы (Selectors) в основанном на Redux NgRx Store: он позволяет нам вычислять различные представления состояния для различных сценариев использования. Эти так называемые View Models пересчитываются только тогда, когда изменяется хотя бы один из лежащих в их основе сигналов.

### Обновление состояния

Для обновления `SignalState` Signal Store предоставляет нам функцию `patchState`:

```
import { patchState } from '@ngrx/signals';

[...]

updateCriteria(from: string, to: string): void {
  patchState(this.state, { from, to })
}
```

Здесь мы передаём контейнер состояния и частичное состояние (partial state). В качестве альтернативы можно передать функцию, принимающую текущее состояние и преобразующую его в новое состояние:

```
updateBasket(id: number, selected: boolean): void {
  patchState(this.state, state => ({
    basket: {
      ...state.basket,
      [id]: selected,
    },
  }));
}
```

## Побочные эффекты

Помимо обновления состояния, методы также могут запускать побочные эффекты (side effects), такие как загрузка и сохранение объектов:

```
async load() {
  if (!this.from() || !this.to()) return;

  const flights = await this.flightService.findPromise(
    this.from(),
    this.to()
  );

  patchState(this.state, { flights });
}
```

### Отделение намерения от исполнения

Иногда вызывающий код `patchState` знает только то, что некоторое состояние нужно обновить, не зная, где оно расположено. Для таких случаев можно предоставить апдейтеры (Updaters). Апдейтеры — это просто функции, принимающие текущее состояние и возвращающие его обновлённую версию:

```
type BasketSlice = { basket: Record<number, boolean> };
type BasketUpdateter = (state: BasketSlice) => BasketSlice;

export function updateBasket(flightId: number, selected: boolean): BasketUpdateter {
  return (state) => ({
    ...state,
    basket: {
      ...state.basket,
      [flightId]: selected,
    },
  });
}
```

Также допустимо просто вернуть частичное состояние. Оно будет применено (patched) поверх текущего состояния:

```
type BasketSlice = { basket: Record<number, boolean> };
type BasketUpdateter = (state: BasketSlice) => BasketSlice;

export function updateBasket(flightId: number, selected: boolean): BasketUpdateter {
  return (state) => ({
    basket: {
      ...state.basket,
      [flightId]: selected,
    },
  });
}
```

Если вам не нужно проецировать текущее состояние, то достаточно вернуть частичное состояние. В этом случае внутреннюю функцию можно пропустить:

```
export function updateFlights(flights: Flight[]) {
  return { flights };
}
```

Апдейтеры можно определять на «суверенной территории» `Store` (то есть `signalState`). Для потребителя это просто чёрный ящик:

```
patchState(updateBasket(id, selected))
```

Передача апдейтера в `patchState` выражает намерение. Это похоже на отправку (dispatch) экшена (Action) в классическом NgRx store. Однако, в отличие от Redux, здесь не задействован механизм событий, и мы не можем помешать вызывающему коду напрямую передавать свои собственные апдейтеры. По последней причине я скрываю `SignalStore` за фасадом.

## Разновидность 2: Мощная с signalStore

🔀 _Ветка:_ `arc-signal-store-2`

Подобно `signalState`, функция `signalStore` создаёт контейнер, управляющий состоянием с помощью сигналов. Однако теперь этот контейнер — полноценное хранилище (Store), которое содержит не только сигналы состояния, но и вычисляемые сигналы (computed Signals), а также методы для обновления состояния и для запуска побочных эффектов. Следовательно, потребность в создании фасада вручную, как показано выше, меньше.

Технически Store — это сервис Angular, составленный из нескольких заранее существующих функций (features):

```
export const FlightBookingStore = signalStore(
  { providedIn: 'root' },
  withState({
    from: 'Paris',
    to: 'London',
    initialized: false,
    flights: [] as Flight[],
    basket: {} as Record<number, boolean>,
  }),

  // Активация дополнительных функций
  withComputed([...]),
  withMethods([...]),
  withHooks([...]),
)
```

В этом случае сервис также регистрируется в корневой области видимости (root scope). При пропуске `{ providedIn: 'root' }` сервис нужно регистрировать вручную, например, предоставляя его при бутстрапе приложения, в конфигурации роутера или на уровне компонента.

### Выбор и вычисление сигналов

Функция `withComputed` принимает store с его сигналами состояния и определяет объект с вычисляемыми сигналами:

```
withComputed((store) => ({
  selected: computed(() => store.flights().filter((f) => store.basket()[f.id])),
  criteria: computed(() => ({ from: store.from(), to: store.to() })),
})),
```

Возвращённые вычисляемые сигналы становятся частью store. Более компактная версия может предполагать прямую деструктуризацию переданного store:

```
withComputed(({ flights, basket, from, to }) => ({
  selected: selectSignal(() => flights().filter((f) => basket()[f.id])),
  criteria: selectSignal(() => ({ from: from(), to: to() })),
})),
```

### Методы для обновления состояния и побочных эффектов

Подобно `withComputed`, `withMethods` также принимает store и возвращает объект с методами:

```
withMethods((state) => {
  const { basket, flights, from, to, initialized } = state;
  const flightService = inject(FlightService);

  return {
    updateCriteria: (from: string, to: string) => {
      patchState(state, { from, to });
    },
    updateBasket: (flightId: number, selected: boolean) => {
      patchState(state, {
        basket: {
          ...basket(),
          [flightId]: selected,
        },
      });
    },
    delay: () => {
      const currentFlights = flights();
      const flight = currentFlights[0];

      const date = addMinutes(flight.date, 15);
      const updFlight = { ...flight, date };
      const updFlights = [updFlight, ...currentFlights.slice(1)];

      patchState(state, { flights: updFlights });
    },
    load: async () => {
      if (!from() || !to()) return;
      const flights = await flightService.findPromise(from(), to());
      patchState(state, { flights });
    }     
  };
}),
```

`withMethods` выполняется в контексте внедрения (injection context) и, следовательно, может использовать `inject` для получения сервисов. После того как `withMethods` был выполнен, полученные методы добавляются в store.

### Потребление Store

С точки зрения вызывающего кода store очень похож на показанный выше фасад. Мы можем внедрить его в потребляющий компонент:

```
@Component([...])
export class FlightSearchComponent {
  private store = inject(FlightBookingStore);

  from = this.store.from;
  to = this.store.to;
  basket = this.store.basket;
  flights = this.store.flights;
  selected = this.store.selected;

  async search() {
    this.store.load();
  }

  delay(): void {
    this.store.delay();
  }

  updateCriteria(from: string, to: string): void {
    this.store.updateCriteria(from, to);
  }

  updateBasket(id: number, selected: boolean): void {
    this.store.updateBasket(id, selected);
  }
}
```

### Хуки (Hooks)

Функция `withHooks` предоставляет ещё одну функцию, позволяющую настроить хуки жизненного цикла, выполняющиеся при инициализации или уничтожении store:

```
withHooks({
  onInit({ load }) {
    load()
  },
  onDestroy({ flights }) {
    console.log('flights are destroyed now', flights());
  },
}),
```

Оба хука получают store. И снова, используя деструктуризацию, вы можете сосредоточиться на подмножестве членов store.

### rxMethod

🔀 _Ветка:_ `arc-signal-store-rx`

Хотя сигналы просты в использовании, они не являются полной заменой RxJS. Для задействования RxJS и его мощных операторов Signal Store предоставляет вторичную точку входа `@ngrx/signals/rxjs-interop`, содержащую функцию `rxMethod<T>`. Она позволяет работать с Observable, представляющим побочные эффекты, которые автоматически запускаются при изменении определённых значений:

```
import { rxMethod } from '@ngrx/signals/rxjs-interop';

[...]

withMethods(({ $update, basket, flights, from, to, initialized }) => {
  const flightService = inject(FlightService);

  return {
    [...]
    connectCriteria: rxMethod<Criteria>((c$) => c$.pipe(
      filter(c => c.from.length >= 3 && c.to.length >= 3),
      debounceTime(300),
      switchMap((c) => flightService.find(c.from, c.to)),
      tap(flights => patchState(state, { flights }))
    ))
  }
});
```

Параметр типа `T` определяет тип, с которым работает `rxMethod`. Хотя `rxMethod` получает `Obserable<T>`, вызывающий код также может передать `Observable<T>`, `Signal<T>` или `T` напрямую. В двух последних случаях переданные значения преобразуются в Observable.

После определения `rxMethod` где-нибудь ещё в приложении, например, в хуке или обычном методе, можно вызвать этот эффект:

```
withHooks({
  onInit({ loadBy, criteria }) {
    connectCriteria(criteria());
  },
})
```

Здесь передаётся сигнал criteria — вычисляемый сигнал. Каждый раз, когда этот сигнал изменяется, эффект внутри `connectCriteria` выполняется повторно.

## Пользовательские функции — путь к новым разновидностям

🔀 _Ветка:_ `arc-signal-store-custom`

Помимо конфигурирования Store встроенными функциями, каждый может написать свои собственные функции для автоматизации повторяющихся задач. Песочница, предоставленная [Марко Станимировичем](https://twitter.com/MarkoStDev) — контрибьютором NgRx, стоящим за Signal Store, — содержит несколько примеров таких функций.

Один из примеров, найденных в этом репозитории, — это [функция CallState](https://github.com/markostanimirovic/ngrx-signal-store-playground/blob/main/src/app/shared/call-state.feature.ts), определяющая свойство состояния, информирующее о состоянии текущего HTTP-вызова:

```
export type CallState = 'init' | 'loading' | 'loaded' | { error: string };
```

В этом разделе я использую данный пример, чтобы объяснить, как предоставлять пользовательские функции.

### Определение пользовательских функций

Функция обычно создаётся вызовом `signalStoreFeature`. Эта функция конструирует новую функцию поверх существующих.

```
// Взято из: https://github.com/markostanimirovic/ngrx-signal-store-playground/blob/main/src/app/shared/call-state.feature.ts

import { computed } from '@angular/core';
import {
  signalStoreFeature,
  withComputed,
  withState,
} from '@ngrx/signals';

export type CallState = 'init' | 'loading' | 'loaded' | { error: string };

export function withCallState() {
  return signalStoreFeature(
    withState<{ callState: CallState }>({ callState: 'init' }),
    withComputed(({ callState }) => ({
      loading: computed(() => callState() === 'loading'),
      loaded: computed(() => callState() === 'loaded'),
      error: computed(() => {
        const state = callState();
        return typeof state === 'object' ? state.error : null
      }),
    }))
  );
}
```

Для свойств состояния, добавленных функцией, можно предоставить апдейтеры (`Updaters`):

```
export function setLoading(): { callState: CallState } {
  return { callState: 'loading' };
}

export function setLoaded(): { callState: CallState } {
  return { callState: 'loaded' };
}

export function setError(error: string): { callState: CallState } {
  return { callState: { error } };
}
```

Апдейтеры позволяют потребителю изменять состояние функции, фактически не зная, как оно структурировано.

### Использование пользовательских функций

Для использования пользовательских функций просто вызовите предоставленную фабрику при настройке store:

```
export const FlightBookingStore = signalStore(
  { providedIn: 'root' },
  withState({ [...] }),

  // Добавляем функцию:
  withCallState(),
  [...]

  withMethods([...])
  [...]
);
```

Предоставленные свойства, методы и апдейтеры можно использовать в методах Store:

```
load: async () => {
  if (!from() || !to()) return;

  // Установка callState через апдейтер
  patchState(state, setLoading());

  const flights = await flightService.findPromise(from(), to());
  patchState(state, { flights });

  // Установка callState через апдейтер
  patchState(state, setLoaded());
},
```

Потребитель store также видит свойства, предоставленные функцией:

```
private store = inject(FlightBookingStore);

flights = this.store.flightEntities;
loading = this.store.loading;
```

Поскольку каждая функция преобразует свойства и методы Store, убедитесь, что вы вызываете их в правильном порядке. Если предположить, что методы, зарегистрированные с помощью `withMethods`, используют `CallState`, то `withCallState` должен быть вызван перед `withMethods`.

## Разновидность 3: Встроенные функции, такие как управление сущностями

🔀 _Ветка:_ `arc-signal-store-entities`

NgRx Signal Store уже поставляется с удобным расширением для управления сущностями (entities). Его можно найти во вторичной точке входа `@ngrx/signals/entities`; оно предоставляет структуры данных для сущностей, а также несколько апдейтеров, например, для вставки сущностей или для обновления одной сущности по id.

Чтобы настроить управление сущностями, просто вызовите функцию `withEntities`:

```
import { withEntities } from '@ngrx/signals/entities';

const BooksStore = signalStore(
  [...]

  // Определение сущности
  withEntities({ entity: type<Flight>(), collection: 'flight' }),

  // withEntities создал для нас сигнал flightEntities:
  withComputed(({ flightEntities, basket, from, to }) => ({
    selected: computed(() => flightEntities().filter((f) => basket()[f.id])),
    criteria: computed(() => ({ from: from(), to: to() })),
  })),

  withMethods((state) => {
    const { basket, flightEntities, from, to, initialized } = state;
    const flightService = inject(FlightService);

    return {
      [...],

      load: async () => {
        if (!from() || !to()) return;
        patchState(state, setLoading());

        const flights = await flightService.findPromise(from(), to());

        // Обновление сущностей готовым апдейтером setAllEntities
        patchState(state, setAllEntities(flights, { collection: 'flight' }));
        patchState(state, setLoaded());
      },

      [...],
    };
  }),
);
```

Переданное имя коллекции предотвращает конфликты имён. В нашем случае коллекция называется `flight`, и поэтому функция создаёт несколько свойств, начинающихся с `flight`, например, `flightEntities`.

Существует довольно большое количество готовых к использованию апдейтеров:

* `addEntity`
* `addEntities`
* `removeEntity`
* `removeEntities`
* `removeAllEntities`
* `setEntity`
* `setEntities`
* `setAllEntities`
* `updateEntity`
* `updateEntities`
* `updateAllEntities`

Подобно `@ngrx/entities`, внутренне сущности хранятся в нормализованном виде. Это означает, что они хранятся в словаре, отображающем их первичные ключи на объекты сущностей. Это упрощает их объединение в View Models, необходимые для конкретных сценариев использования.

Поскольку мы называем нашу коллекцию `flight`, `withEntities` создаёт сигнал `flightEntityMap`, отображающий id рейсов на наши объекты рейсов. Также он создаёт сигнал `flightIds`, содержащий все id по порядку. Оба используются также добавленным вычисляемым сигналом `flightEntities`, использованным выше. Он возвращает все рейсы в виде массива, соблюдая порядок id внутри `flightIds`. Следовательно, если вы хотите переставить позиции наших рейсов, просто обновите свойство `flightIds` соответствующим образом.

Для построения структур, таких как `flightEntityMap`, апдейтерам нужно знать, как называется id сущности. По умолчанию предполагается свойство `id`. Если id называется иначе, вы можете сообщить об этом апдейтеру с помощью свойства `idKey`:

```
patchState(state, setAllEntities(flights, { collection: 'flight', idKey: 'flightId' }));
```

Переданное свойство должно быть `string` или `number`. Если оно другого типа данных или если его вообще не существует, вы получите ошибку компиляции.

## Заключение

Будущий NgRx Signal Store позволяет управлять состоянием с помощью сигналов. Самый лёгкий вариант использования этой библиотеки — просто взять контейнер `SignalState`. Эта структура данных предоставляет сигнал для каждого свойства состояния. Эти сигналы доступны только для чтения. Для обновления состояния можно использовать функцию `patchState`. Чтобы убедиться, что обновления происходят только строго определённым образом, `signalstate` можно скрыть за фасадом.

`SignalStore` мощнее и позволяет регистрировать опциональные функции. Они определяют состояние, которым нужно управлять, а также методы, работающие с ним. `SignalStore` можно предоставить как сервис и внедрять в его потребителей.

`SignalStore` также предоставляет механизм расширения для реализации пользовательских функций, упрощающих повторяющиеся задачи. «Из коробки» Signal Store поставляется с весьма удобной функцией для управления сущностями.
