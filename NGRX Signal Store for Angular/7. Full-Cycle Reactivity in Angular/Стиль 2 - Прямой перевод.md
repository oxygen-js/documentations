## Загрузка данных с помощью Resource API

Сервис доступа к данным заботится о создании ресурсов и мутаций. Для загрузки он опирается на _httpResource_:

```typescript
@Injectable({
  providedIn: 'root',
})
export class FlightService {
  […]

  findResourceById(id: Signal<number>) {
    return httpResource<Flight>(
      () =>
        !id()
          ? undefined
          : {
              url: 'https://demo.angulararchitects.io/api/flight',
              params: {
                id: id(),
              },
            },  
      {
        defaultValue: initFlight,
      }
    );
  }
}
```

Всякий раз, когда id в переданном сигнале меняется, ресурс пересчитывает объект запроса, а затем использует его для загрузки запрошенного рейса. Однако есть одно исключение: если id является _falsy_ (например, _0_), объект запроса получает значение _undefined_. По соглашению ресурс в этом случае не загружает никаких данных. Эта процедура откладывает загрузку до тех пор, пока действительно не появится «настоящий» id.

Чтобы предотвратить возврат ресурсом результата _undefined_ до того, как вернётся первый вызов, в примере определяется _defaultValue_. _initFlight_ служит Null-объектом.

## Сохранение данных с помощью Mutation API

Mutation API, предоставляемый NgRx Toolkit, является аналогом Angular Resource API для изменения данных. Помимо _rxMutation_ для произвольных изменений, которые возвращают свои результаты как Observable, _httpMutation_ подходит для изменений, которые могут быть напрямую представлены HTTP-сообщениями:

```typescript
import { httpMutation } from '@angular-architects/ngrx-toolkit';

[…]

@Injectable({
  providedIn: 'root',
})
export class FlightService {
  […]

  createSaveMutation(options: Partial<HttpMutationOptions<Flight, Flight>>) {
    return httpMutation({
      ...options,
      request: (flight) => ({
        url: 'https://demo.angulararchitects.io/api/flight',
        method: 'POST',
        body: flight,
      }),
      operator: concatOp
    });
  }

  […]
}
```

И здесь сервис доступа к данным служит фабрикой. Сам сервис определяет ключевые данные для HTTP-запроса; дальнейшие конфигурации, такие как колбэк успеха или ошибки, могут быть переданы вызывающей стороной через частичный объект опций.

Тип HttpMutationOptions<Flight, Flight> указывает, что мутация получает _Flight_ и возвращает другой _Flight_. Последний соответствует сохранённому рейсу, который также содержит значения, назначенные сервером, такие как id.

При желании вы также можете указать, какая семантика должна использоваться для перекрывающихся вызовов, чтобы предотвратить состояния гонки (race conditions). Свойство _operator_ принимает для этой цели соответствующую стратегию. NgRx Toolkit предоставляет стратегии _switchOp_, _mergeOp_, _concatOp_ и _exhaustOp_, которые соответствуют хорошо известным flattening-операторам в RxJS. Если вызывающая сторона не указывает это свойство, используется _concatOp_. Это приводит к тому, что мутация запускает перекрывающиеся запросы один за другим.

Возвращённую мутацию можно вызвать как функцию, и она также предоставляет информацию о состоянии в виде сигналов (Signals). Следующий листинг иллюстрирует это на примере компонента, который использует мутацию напрямую.

```typescript
export class HomeComponent {
  private flightService = inject(FlightService);

  private saveFlight = this.flightService.createSaveMutation({ … })

  private saveFlightIsPending = this.saveFlight.isPending;
  private saveFlightError = this.saveFlight.error;
  private saveFlightValue = this.saveFlight.value;
  private saveFlightParams = this.saveFlight.isSuccess;

  save(): void {
    this.saveFlight({
      id: 0,
      from: 'Graz',
      to: 'Hamburg',
      date: new Date().toISOString(),
      delayed: false,
    });
  }
}
```

## Предоставление ресурсов и мутаций через Signal Store

Чтобы управлять состоянием и сохранять реактивный поток управляемым, показанный здесь пример инкапсулирует ресурс и мутацию в NgRx Signal Store. Чтобы навести мост к стору, NgRx Toolkit предоставляет фичи _withMutations_ и _withResource_:

```typescript
import {
  withMutations,
  withResource,
} from '@angular-architects/ngrx-toolkit';

[…]

export const FlightDetailStore = signalStore(
  { providedIn: 'root' },

  withState({
    filter: {
      id: 0,
    },
  }),

  withProps(() => ({
    _flightService: inject(FlightService),
    _snackBar: inject(MatSnackBar),
  })),

  withResource((store) => ({
    flight: store._flightService.findResourceById(store.filter.id),
  })),

  withMutations((store) => ({
    saveFlight: store._flightService.createSaveMutation({
      onSuccess(flight: Flight) {
        patchState(store, { flightValue: flight });
        store._snackBar.open('Flight saved', 'OK');
      },
      onError(error: unknown) {
        store._snackBar.open('Error saving flight!', 'OK');
        console.error(error);
      },
    }),
  })),

  withMethods((store) => ({
    updateFilter: signalMethod((id: number) => {
      if (id !== store.filter.id()) {
        patchState(store, {
          filter: {
            id,
          },
        });
      }
    }),
  }))
);
```

Фича _withResource_ сопоставляет имя, используемое стором, с ресурсом, возвращаемым показанным выше сервисом доступа к данным. Позвольте обратить ваше внимание на следующую строку:

```
patchState(store, { flightValue: flight });
```

Здесь возникает вопрос, откуда берётся значение _flightValue_. На самом деле это одно из свойств, введённых _withResource_. По соглашению это свойство начинается с имени, указанного в _withResource_, и возвращает полученное значение. Как мы увидим ниже, стору даются дополнительные свойства, следующие этому шаблону, включая _flightIsLoading_ и _flightError_.

Фича _withMutations_ действует аналогично и дополнительно определяет обработчик успеха и ошибки. Она настраивает такие свойства, как _saveFlightIsPending_ и _saveFlightError_.

Метод _updateFilter_ устанавливает id, которым управляет стор, и таким образом запускает ресурс. Что интересно, стор конфигурирует этот метод как _signalMethod_. В результате id можно передавать не только как _number_, но и как Signal<number>, и даже как Observable<number>. В двух последних случаях _signalMethod_ снова вызывает логику, как только становится доступно новое значение.

## Стор и Signal Forms

Потребляющий компонент получает стор через внедрение зависимостей (dependency injection) и предоставляет Signal Form для редактирования загруженного рейса:

```typescript
import { Control, form, required, submit } from '@angular/forms/signals';

[…]

export class FlightEditComponent {
  private store = inject(FlightDetailStore);

  id = input.required({
    transform: numberAttribute,
  });

  isPending = this.store.saveFlightIsPending;
  error = this.store.saveFlightError;

  flight = linkedSignal(() => normalize(this.store.flightValue()));

  flightForm = form(this.flight, (schema) => { 
    required(schema.from);
    required(schema.to);
    required(schema.date);
  });

  constructor() {
    this.store.updateFilter(this.id);
  }

  save(): void {
    submit(this.flightForm, async (form) => {
      const result = await this.store.saveFlight(form().value());

      if (result.status === 'error') {
        return {
          kind: 'processing_error',
            // ^^^ постарайтесь быть более конкретным
          error: result.error,
        }
      }
      return null;
    });
  }
}
```

Взаимодействие между стором и формой сопряжено с небольшой сложностью: в то время как форма отвечает за изменение загруженных данных, стор публикует эти данные только для чтения (read-only), чтобы обеспечить согласованность. Стор допускает изменения только через предоставленные методы.

Это означает, что данные, полученные из стора, должны быть перенесены в локальную рабочую копию. Именно об этом здесь и заботится _linkedSignal_. Его можно представить как _computed_ с рабочей копией, в которую может выполнять обратную запись привязка данных (data binding) формы. Кроме того, показанный вызов _linkedSignal_ делегирует вспомогательной функции _normalize_ (здесь подробно не рассматриваемой), которая форматирует дату рейса для использования с <input type="datatime-local">.

Чтобы преобразовать получившееся свойство _flight_ в Signal Form, компонент передаёт его в функцию _form_, предоставляемую Angular. Во второй параметр передаётся схема с правилами валидации. В результате получается свойство _flightForm_, которое позволяет привязывать отдельные свойства рейса — например, _from_ и _to_ — к полям формы.

Чтобы сохранить, компонент вызывает функцию _submit_, предоставляемую Signal Forms. Переданное лямбда-выражение делегирует мутации _saveFlight_ и возвращает результат валидации в случае ошибки на стороне сервера. Signal Form обрабатывает этот результат валидации так же, как и те, что получаются из правил валидации в указанной схеме.

Помимо состояния _error_, мутация может приводить к двум другим состояниям. Очевидное — это _success_, но также возможно, что мутация прерывает операцию, используя семантику _switch_ или _exhaust_, чтобы предотвратить состояния гонки. Состояние _aborted_ указывает на этот случай.

Следующий листинг привязывает отдельные поля _flightForm_ к полям ввода и отображает ошибки валидации:

```html

<pre>Form-level errors: {{ flightForm().errors() | json }}</pre>
<form class="flight-form" (ngSubmit)="save()">

  <div class="form-group">
    <label for="flight-from">From</label>

      class="form-control"

      id="flight-from"
      name="from"
    />
    <pre>Field-level Errors: {{ flightForm.from().errors | json }}</pre>
  </div>

  […]
```

## Заключение

С Resource API, мутациями (Mutations) и Signal Forms Angular впервые предоставляет согласованный реактивный поток данных — от загрузки через редактирование к сохранению. NgRx Signal Store служит каркасом, чтобы последовательно свести эти строительные блоки воедино. Это замыкает реактивный цикл и создаёт чётко структурированную модель для управления состоянием и формами, которая одновременно надёжна и расширяема.
