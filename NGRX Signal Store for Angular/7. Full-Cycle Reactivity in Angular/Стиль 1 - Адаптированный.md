## Загрузка данных с помощью Resource API

За создание ресурсов и мутаций отвечает сервис доступа к данным. Для загрузки он опирается на _httpResource_:

```typescript
@Injectable({
  providedIn: 'root',
})
export class FlightService {
  […]

  findResourceById(id: Signal<number>) {
    return httpResource<Flight>(
      () =>
        !id()
          ? undefined
          : {
              url: 'https://demo.angulararchitects.io/api/flight',
              params: {
                id: id(),
              },
            },  
      {
        defaultValue: initFlight,
      }
    );
  }
}
```

Как только id в переданном сигнале меняется, ресурс пересчитывает объект запроса и затем подгружает по нему нужный рейс. Есть, правда, одно исключение: если id оказывается _falsy_ (например, _0_), объект запроса получает значение _undefined_. По соглашению ресурс в этом случае ничего не загружает. Такой приём откладывает загрузку до того момента, когда появится «настоящий» id.

Чтобы ресурс не вернул _undefined_ ещё до того, как отработает первый вызов, в примере задано значение по умолчанию (_defaultValue_). В роли Null-объекта здесь выступает _initFlight_.

## Сохранение данных с помощью Mutation API

Mutation API, который предоставляет NgRx Toolkit, — это аналог Angular Resource API, только для изменения данных. Помимо _rxMutation_ для произвольных изменений, возвращающих свой результат в виде Observable, есть _httpMutation_ — он подходит для изменений, которые напрямую выражаются HTTP-сообщениями:

```typescript
import { httpMutation } from '@angular-architects/ngrx-toolkit';

[…]

@Injectable({
  providedIn: 'root',
})
export class FlightService {
  […]

  createSaveMutation(options: Partial<HttpMutationOptions<Flight, Flight>>) {
    return httpMutation({
      ...options,
      request: (flight) => ({
        url: 'https://demo.angulararchitects.io/api/flight',
        method: 'POST',
        body: flight,
      }),
      operator: concatOp
    });
  }

  […]
}
```

И здесь сервис доступа к данным играет роль фабрики. Сам сервис задаёт ключевые параметры HTTP-запроса; дополнительные настройки — например, колбэк успеха или ошибки — вызывающая сторона может передать через частичный объект опций.

Тип HttpMutationOptions<Flight, Flight> говорит о том, что мутация принимает _Flight_ и возвращает другой _Flight_. Последний соответствует сохранённому рейсу, в котором уже есть значения, проставленные сервером, — например, id.

При желании можно указать, какая семантика должна применяться к перекрывающимся вызовам, чтобы предотвратить состояния гонки (race conditions). Для этого свойство _operator_ принимает соответствующую стратегию. NgRx Toolkit предлагает стратегии _switchOp_, _mergeOp_, _concatOp_ и _exhaustOp_, которые соответствуют хорошо знакомым flattening-операторам в RxJS. Если вызывающая сторона это свойство не указывает, используется _concatOp_. В этом случае мутация выполняет перекрывающиеся запросы один за другим.

Возвращённую мутацию можно вызвать как функцию, и она также отдаёт информацию о состоянии в виде сигналов (Signals). В следующем листинге это показано на примере компонента, который использует мутацию напрямую.

```typescript
export class HomeComponent {
  private flightService = inject(FlightService);

  private saveFlight = this.flightService.createSaveMutation({ … })

  private saveFlightIsPending = this.saveFlight.isPending;
  private saveFlightError = this.saveFlight.error;
  private saveFlightValue = this.saveFlight.value;
  private saveFlightParams = this.saveFlight.isSuccess;

  save(): void {
    this.saveFlight({
      id: 0,
      from: 'Graz',
      to: 'Hamburg',
      date: new Date().toISOString(),
      delayed: false,
    });
  }
}
```

## Предоставляем ресурсы и мутации через Signal Store

Чтобы управлять состоянием и держать реактивный поток под контролем, в нашем примере ресурс и мутация инкапсулируются в NgRx Signal Store. Чтобы связать их со стором, NgRx Toolkit даёт фичи _withMutations_ и _withResource_:

```typescript
import {
  withMutations,
  withResource,
} from '@angular-architects/ngrx-toolkit';

[…]

export const FlightDetailStore = signalStore(
  { providedIn: 'root' },

  withState({
    filter: {
      id: 0,
    },
  }),

  withProps(() => ({
    _flightService: inject(FlightService),
    _snackBar: inject(MatSnackBar),
  })),

  withResource((store) => ({
    flight: store._flightService.findResourceById(store.filter.id),
  })),

  withMutations((store) => ({
    saveFlight: store._flightService.createSaveMutation({
      onSuccess(flight: Flight) {
        patchState(store, { flightValue: flight });
        store._snackBar.open('Flight saved', 'OK');
      },
      onError(error: unknown) {
        store._snackBar.open('Error saving flight!', 'OK');
        console.error(error);
      },
    }),
  })),

  withMethods((store) => ({
    updateFilter: signalMethod((id: number) => {
      if (id !== store.filter.id()) {
        patchState(store, {
          filter: {
            id,
          },
        });
      }
    }),
  }))
);
```

Фича _withResource_ сопоставляет имя, под которым стор будет к нему обращаться, с ресурсом, который возвращает показанный выше сервис доступа к данным. Обратите внимание вот на эту строку:

```
patchState(store, { flightValue: flight });
```

Тут возникает вопрос: откуда берётся значение _flightValue_? На самом деле это одно из свойств, которые добавляет _withResource_. По соглашению такое свойство начинается с имени, указанного в _withResource_, и возвращает полученное значение. Как мы увидим ниже, по этому же шаблону стор получает и другие свойства, в том числе _flightIsLoading_ и _flightError_.

Фича _withMutations_ работает аналогично, но дополнительно задаёт обработчики успеха и ошибки. Она заводит такие свойства, как _saveFlightIsPending_ и _saveFlightError_.

Метод _updateFilter_ выставляет id, которым управляет стор, и тем самым запускает ресурс. Любопытно, что стор оформляет этот метод как _signalMethod_. Благодаря этому id можно передавать не только как _number_, но и как Signal<number> и даже как Observable<number>. В двух последних случаях _signalMethod_ повторно вызывает логику, как только появляется новое значение.

## Стор и Signal Forms

Компонент-потребитель получает стор через внедрение зависимостей (dependency injection) и предоставляет Signal Form для редактирования загруженного рейса:

```typescript
import { Control, form, required, submit } from '@angular/forms/signals';

[…]

export class FlightEditComponent {
  private store = inject(FlightDetailStore);

  id = input.required({
    transform: numberAttribute,
  });

  isPending = this.store.saveFlightIsPending;
  error = this.store.saveFlightError;

  flight = linkedSignal(() => normalize(this.store.flightValue()));

  flightForm = form(this.flight, (schema) => { 
    required(schema.from);
    required(schema.to);
    required(schema.date);
  });

  constructor() {
    this.store.updateFilter(this.id);
  }

  save(): void {
    submit(this.flightForm, async (form) => {
      const result = await this.store.saveFlight(form().value());

      if (result.status === 'error') {
        return {
          kind: 'processing_error',
            // ^^^ постарайтесь дать более конкретное значение
          error: result.error,
        }
      }
      return null;
    });
  }
}
```

Взаимодействие стора и формы связано с одной небольшой тонкостью: за изменение загруженных данных отвечает форма, тогда как стор публикует эти данные только для чтения (read-only) — чтобы сохранить согласованность. Стор допускает изменения исключительно через предоставленные им методы.

Значит, полученные из стора данные нужно перенести в локальную рабочую копию. Именно этим здесь и занимается _linkedSignal_. Можно представлять его как _computed_ с рабочей копией, в которую может писать обратно привязка данных (data binding) формы. Кроме того, показанный вызов _linkedSignal_ делегирует вспомогательной функции _normalize_ (её мы подробно не разбираем), которая форматирует дату рейса так, чтобы её можно было использовать с <input type="datatime-local">.

Чтобы превратить получившееся свойство _flight_ в Signal Form, компонент передаёт его в функцию _form_, предоставляемую Angular. Во второй параметр передаётся схема с правилами валидации. В результате получается свойство _flightForm_, которое позволяет привязывать отдельные свойства рейса — например, _from_ и _to_ — к полям формы.

Для сохранения компонент вызывает функцию _submit_, которую дают Signal Forms. Переданное лямбда-выражение делегирует мутации _saveFlight_ и в случае ошибки на стороне сервера возвращает результат валидации. Signal Form обрабатывает этот результат валидации точно так же, как и те, что получаются из правил валидации в заданной схеме.

Помимо состояния _error_, мутация может приводить к ещё двум состояниям. Очевидное — это _success_. Но возможно и так, что мутация прерывает операцию по семантике _switch_ или _exhaust_, чтобы предотвратить состояние гонки. Этот случай обозначается состоянием _aborted_.

Следующий листинг привязывает отдельные поля _flightForm_ к полям ввода и выводит ошибки валидации:

```html
@if (flightForm.id().value() !== 0) {
<pre>Form-level errors: {{ flightForm().errors() | json }}</pre>
<form class="flight-form" (ngSubmit)="save()">

  <div class="form-group">
    <label for="flight-from">From</label>
    <input
      class="form-control"
      [control]="flightForm.from"
      id="flight-from"
      name="from"
    />
    <pre>Field-level Errors: {{ flightForm.from().errors | json }}</pre>
  </div>

  […]

  <div class="mt-20">
    <button 
        class="btn btn-default" 
        type="button" 
        (click)="save()">Save</button>
  </div>

</form>
}
```

Для простоты компонент выводит ошибки валидации в виде JSON-документа. Здесь пригодился бы компонент, который соответствующим образом обрабатывал бы обнаруженные ошибки.

## Заключение

С Resource API, мутациями (Mutations) и Signal Forms Angular впервые предлагает целостный реактивный поток данных — от загрузки через редактирование к сохранению. NgRx Signal Store выступает каркасом, который последовательно сводит эти строительные блоки воедино. Так замыкается реактивный цикл и возникает чётко структурированная модель управления состоянием и формами — одновременно надёжная и расширяемая.
