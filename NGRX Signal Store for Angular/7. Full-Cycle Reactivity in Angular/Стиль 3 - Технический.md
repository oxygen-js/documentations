И ресурс (resource), и мутация (mutation) предоставляют сигналы (Signals), которые стор (store) компонента экспонирует наружу — потребляющему компоненту. К ним относятся полученные данные, флаги состояния, такие как _isLoading_ и _isProcessing_, а также любая информация об ошибках. Signal Form отображает загруженные данные и одновременно служит источником для обратной записи изменений через мутацию.

## Загрузка данных с помощью Resource API

Создание ресурсов и мутаций возложено на сервис доступа к данным. Для загрузки он использует _httpResource_:

```typescript
@Injectable({
  providedIn: 'root',
})
export class FlightService {
  […]

  findResourceById(id: Signal<number>) {
    return httpResource<Flight>(
      () =>
        !id()
          ? undefined
          : {
              url: 'https://demo.angulararchitects.io/api/flight',
              params: {
                id: id(),
              },
            },  
      {
        defaultValue: initFlight,
      }
    );
  }
}
```

Всякий раз, когда значение id в переданном сигнале изменяется, ресурс пересчитывает объект запроса (request object) и затем по нему загружает запрошенный рейс. Есть, однако, одно исключение: если id является _falsy_ (например, _0_), объект запроса получает значение _undefined_. По соглашению ресурс в этом случае не выполняет загрузку. Такой приём откладывает загрузку до момента, когда фактически станет доступен «реальный» id.

Чтобы ресурс не возвращал результат _undefined_ до завершения первого вызова, в примере задаётся значение по умолчанию (_defaultValue_). _initFlight_ выступает здесь в роли Null-объекта (Null Object).

## Сохранение данных с помощью Mutation API

Mutation API, предоставляемый NgRx Toolkit, является аналогом Angular Resource API для модификации данных. Помимо _rxMutation_, предназначенного для произвольных изменений, возвращающих результат в виде Observable, имеется _httpMutation_, который подходит для изменений, напрямую выразимых через HTTP-сообщения:

```typescript
import { httpMutation } from '@angular-architects/ngrx-toolkit';

[…]

@Injectable({
  providedIn: 'root',
})
export class FlightService {
  […]

  createSaveMutation(options: Partial<HttpMutationOptions<Flight, Flight>>) {
    return httpMutation({
      ...options,
      request: (flight) => ({
        url: 'https://demo.angulararchitects.io/api/flight',
        method: 'POST',
        body: flight,
      }),
      operator: concatOp
    });
  }

  […]
}
```

И здесь сервис доступа к данным выступает в роли фабрики (factory). Сам сервис определяет ключевые параметры HTTP-запроса; дополнительные настройки — например, колбэк успеха или ошибки — вызывающая сторона передаёт через частичный объект опций (partial options object).

Тип HttpMutationOptions<Flight, Flight> сигнализирует, что мутация принимает _Flight_ и возвращает другой _Flight_. Последний соответствует сохранённому рейсу, который вдобавок содержит значения, присвоенные сервером, например id.

При необходимости можно также указать, какая семантика должна применяться к перекрывающимся вызовам (overlapping calls), чтобы предотвратить состояния гонки (race conditions). Для этого свойство _operator_ принимает соответствующую стратегию. NgRx Toolkit предоставляет стратегии _switchOp_, _mergeOp_, _concatOp_ и _exhaustOp_, соответствующие хорошо известным flattening-операторам (операторам уплощения вложенных потоков) в RxJS. Если вызывающая сторона это свойство не задаёт, применяется _concatOp_. В результате мутация выполняет перекрывающиеся запросы последовательно, один за другим.

Возвращённая мутация вызывается как функция и, кроме того, предоставляет информацию о состоянии в виде сигналов (Signals). Следующий листинг иллюстрирует это на примере компонента, использующего мутацию напрямую.

```typescript
export class HomeComponent {
  private flightService = inject(FlightService);

  private saveFlight = this.flightService.createSaveMutation({ … })

  private saveFlightIsPending = this.saveFlight.isPending;
  private saveFlightError = this.saveFlight.error;
  private saveFlightValue = this.saveFlight.value;
  private saveFlightParams = this.saveFlight.isSuccess;

  save(): void {
    this.saveFlight({
      id: 0,
      from: 'Graz',
      to: 'Hamburg',
      date: new Date().toISOString(),
      delayed: false,
    });
  }
}
```

## Предоставление ресурсов и мутаций через Signal Store

Чтобы управлять состоянием и сохранять управляемость реактивного потока, рассматриваемый пример инкапсулирует ресурс и мутацию в NgRx Signal Store. Для интеграции со стором NgRx Toolkit предоставляет фичи (features) _withMutations_ и _withResource_:

```typescript
import {
  withMutations,
  withResource,
} from '@angular-architects/ngrx-toolkit';

[…]

export const FlightDetailStore = signalStore(
  { providedIn: 'root' },

  withState({
    filter: {
      id: 0,
    },
  }),

  withProps(() => ({
    _flightService: inject(FlightService),
    _snackBar: inject(MatSnackBar),
  })),

  withResource((store) => ({
    flight: store._flightService.findResourceById(store.filter.id),
  })),

  withMutations((store) => ({
    saveFlight: store._flightService.createSaveMutation({
      onSuccess(flight: Flight) {
        patchState(store, { flightValue: flight });
        store._snackBar.open('Flight saved', 'OK');
      },
      onError(error: unknown) {
        store._snackBar.open('Error saving flight!', 'OK');
        console.error(error);
      },
    }),
  })),

  withMethods((store) => ({
    updateFilter: signalMethod((id: number) => {
      if (id !== store.filter.id()) {
        patchState(store, {
          filter: {
            id,
          },
        });
      }
    }),
  }))
);
```

Фича _withResource_ сопоставляет имя, под которым к ресурсу обращается стор, с ресурсом, возвращаемым показанным выше сервисом доступа к данным. Обратим внимание на следующую строку:

```
patchState(store, { flightValue: flight });
```

Здесь возникает вопрос: откуда берётся значение _flightValue_? На деле это одно из свойств, привносимых фичей _withResource_. По соглашению имя такого свойства начинается с имени, указанного в _withResource_, и возвращает полученное значение. Как будет показано ниже, по этому же шаблону стор получает и другие свойства, в том числе _flightIsLoading_ и _flightError_.

Фича _withMutations_ работает аналогично и дополнительно определяет обработчики успеха (success handler) и ошибки (error handler). Она заводит такие свойства, как _saveFlightIsPending_ и _saveFlightError_.

Метод _updateFilter_ устанавливает id, которым управляет стор, и тем самым триггерит ресурс. Примечательно, что стор конфигурирует этот метод как _signalMethod_. Благодаря этому id допускается передавать не только как _number_, но и как Signal<number> и даже как Observable<number>. В двух последних случаях _signalMethod_ повторно вызывает логику, как только становится доступно новое значение (то есть подписывается на источник и реагирует на каждое новое значение).

## Стор и Signal Forms

Потребляющий компонент получает стор через внедрение зависимостей (dependency injection) и предоставляет Signal Form для редактирования загруженного рейса:

```typescript
import { Control, form, required, submit } from '@angular/forms/signals';

[…]

export class FlightEditComponent {
  private store = inject(FlightDetailStore);

  id = input.required({
    transform: numberAttribute,
  });

  isPending = this.store.saveFlightIsPending;
  error = this.store.saveFlightError;

  flight = linkedSignal(() => normalize(this.store.flightValue()));

  flightForm = form(this.flight, (schema) => { 
    required(schema.from);
    required(schema.to);
    required(schema.date);
  });

  constructor() {
    this.store.updateFilter(this.id);
  }

  save(): void {
    submit(this.flightForm, async (form) => {
      const result = await this.store.saveFlight(form().value());

      if (result.status === 'error') {
        return {
          kind: 'processing_error',
            // ^^^ постарайтесь задать более конкретное значение
          error: result.error,
        }
      }
      return null;
    });
  }
}
```

Взаимодействие между стором и формой сопряжено с одной небольшой сложностью: за модификацию загруженных данных отвечает форма, тогда как стор публикует эти данные только для чтения (read-only) ради обеспечения согласованности (consistency). Стор допускает изменения исключительно через предоставленные им методы.

Это означает, что данные, полученные из стора, необходимо перенести в локальную рабочую копию (working copy). Именно эту задачу здесь решает _linkedSignal_. Его можно мыслить как _computed_ с рабочей копией, в которую может выполнять обратную запись привязка данных (data binding) формы (то есть значение производно от источника, но при этом локально перезаписываемо). Кроме того, показанный вызов _linkedSignal_ делегирует вспомогательной функции _normalize_ (подробно здесь не рассматриваемой), которая форматирует дату рейса для использования с <input type="datatime-local">.

Чтобы преобразовать получившееся свойство _flight_ в Signal Form, компонент передаёт его в функцию _form_, предоставляемую Angular. Во второй параметр передаётся схема (schema) с правилами валидации. В результате формируется свойство _flightForm_, позволяющее привязывать отдельные свойства рейса — например, _from_ и _to_ — к полям формы.

Для сохранения компонент вызывает функцию _submit_, предоставляемую Signal Forms. Переданное лямбда-выражение делегирует мутации _saveFlight_ и в случае ошибки на стороне сервера возвращает результат валидации. Signal Form обрабатывает этот результат валидации идентично тем, что порождаются правилами валидации в указанной схеме.

Помимо состояния _error_, мутация может приводить к двум другим состояниям. Очевидное — _success_. Но возможен и вариант, когда мутация прерывает операцию по семантике _switch_ или _exhaust_ для предотвращения состояний гонки. На этот случай указывает состояние _aborted_.

Следующий листинг привязывает отдельные поля _flightForm_ к полям ввода (input) и отображает ошибки валидации:

```html
@if (flightForm.id().value() !== 0) {
<pre>Form-level errors: {{ flightForm().errors() | json }}</pre>
<form class="flight-form" (ngSubmit)="save()">

  <div class="form-group">
    <label for="flight-from">From</label>
    <input
      class="form-control"
      [control]="flightForm.from"
      id="flight-from"
      name="from"
    />
    <pre>Field-level Errors: {{ flightForm.from().errors | json }}</pre>
  </div>

  […]

  <div class="mt-20">
    <button 
        class="btn btn-default" 
        type="button" 
        (click)="save()">Save</button>
  </div>

</form>
}
```

Для простоты компонент выводит ошибки валидации в виде JSON-документа. Здесь был бы уместен компонент, который обрабатывал бы обнаруженные ошибки соответствующим образом.

## Заключение

С Resource API, мутациями (Mutations) и Signal Forms Angular впервые предоставляет целостный реактивный поток данных — от загрузки через редактирование к сохранению. NgRx Signal Store выступает каркасом, последовательно сводящим эти строительные блоки воедино. Это замыкает реактивный цикл и формирует чётко структурированную модель управления состоянием и формами — одновременно надёжную и расширяемую.
