NGRX Signal Store, выпущенный вскоре после Angular 17, предлагает крайне лёгковесное решение для управления состоянием (state management). Не имея прямых зависимостей от RxJS, он целиком опирается на сигналы (Signals). Однако его главное преимущество — несомненно, высокая степень расширяемости. С помощью так называемых пользовательских фич (custom features) повторяющиеся задачи можно очень легко реализовать централизованно.

Первый рассматриваемый здесь пример пользовательской фичи довольно прямолинеен. Дальше задача усложняется: потребитель (consumer) фичи должен иметь возможность определять имена сигналов и методов, которые эта фича настраивает. Иначе быстро возникнут конфликты имён (naming conflicts). Как показывают приведённые далее примеры, это не противоречит строгой типизации в TypeScript.

Показанные здесь примеры вдохновлены примером от [Марко Станимировича (Marko Stanimirović)](https://twitter.com/MarkoStDev) — участника основной команды NGRX, стоящего за Signal Store, — и решением для управления сущностями (Entity management) `@ngrx/signals/entity`, которое поставляется вместе с Signal Store.

## Signal Store 101

Прежде чем приступить, я кратко введу всех в курс дела и резюмирую важнейшие аспекты NGRX Signal Store. [Подробный обзор](https://www.angulararchitects.io/blog/the-new-ngrx-signal-store-for-angular-2-1-flavors/) можно найти в [моей статье здесь](https://www.angulararchitects.io/blog/the-new-ngrx-signal-store-for-angular-2-1-flavors/).

Signal Store входит в состав NGRX начиная с версии 17 и находится в npm-пакете `@ngrx/signals`. В центре — функция `signalStore`, создающая новый Signal Store. Технически это сервис Angular, получающий сигналы, производные (вычисляемые) сигналы и методы для управления состоянием:

```typescript
import {
  patchState,
  signalStore,
  withHooks,
  withMethods,
  withComputed,
  withState,
} from '@ngrx/signals';

[…]

export const FlightBookingStore = signalStore(

  { providedIn: 'root' },

  // Свойства состояния (State Properties)
  withState({
    from: 'Paris',
    to: 'London',
    initialized: false,
    flights: [] as Flight[],
    basket: {} as Record<number, boolean>,
  }),

  // Вычисляемое состояние (Calculated State)
  withComputed(({ flights, basket, from, to }) => ({
    selected: computed(() => flights().filter((f) => basket()[f.id])),
    criteria: computed(() => ({ from: from(), to: to() })),
  })),

  // Расширение (Extension)
  withCallState(),

  // Методы (Methods)
  withMethods((state) => {
    const { basket, flights, from, to, initialized } = state;
    const flightService = inject(FlightService);

    return {
      load: async () => {
        if (!from() || !to()) return;

        // Обновление состояния расширения (callState)
        patchState(state, setLoading());

        const flights = await flightService.findPromise(from(), to());
        patchState(state, { flights });

        // Обновление состояния расширения (callState)
        patchState(state, setLoaded());

      },
    };

  })
);
```

Примечательно, что Signal Store уже на этой первой ступени расширения целиком построен на так называемых фичах: `withState`, `withComputed` и `withMethods` — это три фичи, которые обогащают хранилище (store) свойствами (сигналами и вычисляемыми сигналами) и методами.

Функция `withCallState` — это первая простая пользовательская фича, вводящая сигнал `callState`. Этот сигнал, среди прочего, информирует, идёт ли в данный момент загрузка данных или процесс загрузки уже завершён. Она вводит следующие сигналы:

* `callState`: может принимать значения `init`, `loading`, `loaded` либо объект с сообщением об ошибке.
* `loading`: вычисляемый сигнал (computed signal); возвращает `true`, если `callState` имеет значение `loading`.
* `loaded`: вычисляемый сигнал; возвращает `true`, если `callState` имеет значение `loaded`.
* `error`: вычисляемый сигнал; возвращает сообщение об ошибке, которое может храниться в `callState`, либо `null`.

Фича `CallState` также предоставляет так называемые апдейтеры (Updaters), такие как `setLoading` или `setLoaded`. Это функции, используемые потребителями (как в листинге 1) для изменения значения `callState`. В общем случае апдейтеры позволяют потребителям изменять управляемое состояние, не зная точно, как оно структурировано внутри (то есть инкапсулируют форму состояния).

В следующем листинге показан простой компонент, использующий `FlightBookingStore`. Он делегирует отдельным сигналам и собственному методу `load`. Он также обращается к сигналу `loading`, который настраивает фича `CallState`:

```typescript
@Component({ … })
export class FlightSearchComponent {
  private store = inject(FlightBookingStore);

  // Получение сигналов из хранилища (Store)
  from = this.store.from;
  to = this.store.to;
  basket = this.store.basket;
  flights = this.store.flights;
  selected = this.store.selected;

  // Получение сигнала из расширения (Extension)
  loading = this.store.loading;

  async search() {
    this.store.load();
  }
}
```

## Простое первое расширение

Вот первая, очень простая версия фичи `CallState`, использованной в предыдущем разделе:

```typescript
import {
  SignalStoreFeature,
  signalStoreFeature,
  withComputed,
  withState,
} from '@ngrx/signals';

[…]

export type CallState = 'init' | 'loading' | 'loaded' | { error: string };

export function withCallState() {
  return signalStoreFeature(
    withState<{ callState: CallState }>({ callState: 'init' }),
    withComputed(({ callState }) => ({
      loading: computed(() => callState() === 'loading'),
      loaded: computed(() => callState() === 'loaded'),
      error: computed(() => {
        const state = callState();
        return typeof state === 'object' ? state.error : null
      }),
    }))
  );
}
```

Это функция, возвращающая результат `signalStoreFeature`. Функция `signalStoreFeature`, в свою очередь, просто группирует существующие фичи: `withState` вводит свойство `callState`, а `withComputed` определяет на его основе обсуждавшиеся ранее вычисляемые сигналы.

Апдейтеры, предоставляемые фичей, возвращают лишь частичный объект состояния (partial state object) с тем свойством, которое подлежит обновлению:

```typescript
export function setLoading(): { callState: CallState } {
  return { callState: 'loading' };
}

export function setLoaded(): { callState: CallState } {
  return { callState: 'loaded' };
}   

export function setError(error: string): { callState: CallState } {
  return { callState: { error } };
}
```

## Теперь начинается настоящее: типизация

Реализация `CallState` из предыдущего раздела кратко резюмирует решение одного повторяющегося требования. Будучи реализованной, фича может интегрироваться в хранилища отдельных приложений.

Однако недостаток данной реализации в том, что введённые сигналы имеют фиксированные имена: `callState`, `loading`, `loaded` и `error`. Это быстро приводит к конфликтам имён, особенно если одному и тому же хранилищу фича требуется более одного раза. Пример — хранилище, которому нужно управлять несколькими `callState` для разных сущностей, например для рейсов (flights) и пассажиров (passengers).

В этом случае потребитель должен иметь возможность указывать имена вводимых сигналов. Именно этим мы займёмся ниже. Чтобы сделать данное расширение типобезопасным (type-safe), сначала нужно немного поразмыслить о типизации функции `withCallState`.

Сейчас наша функция `withCallState` не имеет явного типа возвращаемого значения. Поэтому TypeScript выводит (infers) этот тип, анализируя возвращаемое в функции значение. Компилятор распознаёт, что доступно свойство `callState`.

Тип, определённый здесь путём вывода (inference), — это `SignalStoreFeature<Input, Output>`. Параметр типа `Input` задаёт, какие сигналы и методы фича ожидает от хранилища, а `Output` указывает, какие дополнительные сигналы и методы фича предоставляет. Наша фича не предъявляет никаких требований к хранилищу, но предоставляет сигнал `callState`, а также несколько вычисляемых сигналов, таких как `loading`. Соответственно, наши типы `Input` и `Output` выглядят так:

![Пользовательские расширения с точки зрения системы типов](https://www.angulararchitects.io/wp-content/uploads/2023/11/extensions-type-system-01.png)

Следует отметить, что `state` описывает вводимый сигнал, а свойство `signals` представляет сигналы, вычисленные на его основе. Это представление соответствует по крайней мере упрощённому внешнему виду (external view).

Внутреннее представление (internal view) несколько сложнее, в частности потому, что `withState` сначала вводит сигнал `callState`, и лишь затем `withComputed` добавляет вычисляемые сигналы. Поэтому внутреннее представление имеет два выхода (output), объединяемых посредством вспомогательного типа (helper type).

![Пользовательские расширения с точки зрения системы типов](https://www.angulararchitects.io/wp-content/uploads/2023/11/extensions-type-system-02.png)

Для простоты на предыдущем изображении вспомогательный тип назван `Merged Result`. Однако в действительности у Signal Store для этого есть несколько внутренних типов.

На логическом уровне внутреннее и внешнее представления эквивалентны. TypeScript может потребоваться небольшой «толчок» в виде утверждения типа (type assertion), чтобы это распознать. Однако явно определять внутреннее представление несколько хлопотно и в настоящее время не вполне возможно, поскольку требуемые вспомогательные типы не входят в публичный API Signal Store. Поэтому здесь я применяю паттерн, неоднократно встречающийся и в коде самого Signal Store: сочетание перегрузки функции (function overload) с внешним представлением и реализации функции (function implementation), использующей `SignalStoreFeature` вместо `SignalStoreFeature<Input, Output>` для внутреннего представления:

```typescript
// Перегрузка с внешним представлением (External View)
export function withCallState()
  : SignalStoreFeature<
    {
      state: {},
      signals: {},
      methods: {}
    },
    {
      state: {
        callState: CallState
      },
      signals: {
        loading: Signal<boolean>,
        loaded: Signal<boolean>,
        error: Signal<{ error: string } | null>
      },
      methods: {}
    }>;
// Реализация с внутренним представлением (Internal View)
export function withCallState(): SignalStoreFeature {
  return signalStoreFeature(
    withState<{ callState: CallState }>({ callState: 'init' }),
    withComputed(({ callState }) => ({
      loading: computed(() => callState() === 'loading'),
      loaded: computed(() => callState() === 'loaded'),
      error: computed(() => {
        const state = callState();
        return typeof state === 'object' ? state.error : null
      }),
    }))
  );
}
```

Тип `SignalStoreFeature` без параметров типа использует для `Input` и `Output` более общие типы, не предполагающие конкретных имён или типов данных.

## Типизация и динамические свойства — как они сочетаются?

Теперь, когда базовая структура типизации готова, мы можем расширить её настраиваемыми именами свойств. По образцу `@ngrx/signals/entity` потребители должны иметь возможность задать префикс при активации фичи:

```typescript
export const FlightBookingStore = signalStore(
  { providedIn: 'root' },

  withState({ … }),
  withComputed(( … ) => ({ … })),

  withCallState({ prop: 'flights' }), 
  withCallState({ prop: 'passengers'}),

  […]
);
```

Этот префикс должен теперь включаться в имена свойств, определяемых фичей. Например, первый вызов `withCallState` должен породить следующие свойства:

* `flightsCallState` (state)
* `flightsLoading` (computed)
* `flightsLoaded` (computed)
* `flightsError` (computed)

Второй вызов аналогично приводит к этим свойствам:

* `passengersCallState` (state)
* `passengersLoading` (computed)
* `passengersLoaded` (computed)
* `passengersError` (computed)

Настройка этих свойств во время выполнения (at runtime) не является большой проблемой в мире TypeScript, тем более что лежащий в основе JavaScript в любом случае динамический язык. Сложность же в том, чтобы также сообщить об этих свойствах системе типов.

Для этой задачи сначала нужно найти способ выразить префикс в объявлении типа. На этом этапе нам на пользу тот факт, что литералы тоже можно использовать в качестве типов данных:

```typescript
export type BoxStatus = 'open' | 'closed';
const candyBox: BoxStatus = 'open';
```

Такие объединения строковых литералов (String Literal Union Types) часто применяются в TypeScript-приложениях для выражения перечислений (enum). Это даже ближе к EcmaScript, чем использование ключевого слова `enum` из TypeScript. Любопытно, что никто не обязывает нас предлагать несколько вариантов. Поэтому такой вариант вполне корректен:

```typescript
export type BoxStatusAfterHolidays = 'closed';
```

Итак, здесь у нас есть тип, способный хранить ровно одно строковое значение. Точно этот же паттерн мы используем, чтобы сообщить системе типов о нашем префиксе. Сначала создадим тип, который на основе префикса определяет имя вводимого сигнала:

```typescript
export type NamedCallState<Prop extends string> = {
  [K in Prop as `${K}CallState`]: CallState;
};
```

Это так называемый отображаемый тип (mapped type), который отображает один тип в новый. Параметр типа `Prop extends string` описывает исходный тип. Это может быть любая строка, используемая как тип. При этом `string` записывается в нижнем регистре, поскольку здесь мы ссылаемся на конкретную строку, а не на тип объекта `String`. Запись `K in Prop` также сводится к этой строке. В более сложных случаях ключевое слово `in` можно было бы использовать, например, для перебора (циклического обхода) свойств исходного типа.

Аналогично можно поступить с вычисляемыми сигналами, которые требуется ввести:

```typescript
export type NamedCallStateComputed<Prop extends string> = {
  [K in Prop as `${K}Loading`]: Signal<boolean>;
} & {
    [K in Prop as `${K}Loaded`]: Signal<boolean>;
} & {
    [K in Prop as `${K}Error`]: Signal<string | null>;
};
```

Поскольку отображаемый тип может содержать только одно отображение, здесь используется несколько отображаемых типов. Они объединяются оператором `&` (оператор пересечения, intersection operator). С помощью этих двух типов мы теперь можем задать типизацию нашей функции `withCallState`:

```typescript
export function withCallState<Prop extends string>(config: {
  prop: Prop;
}): SignalStoreFeature<
  { state: {}, signals: {}, methods: {} },
  {
    state: NamedCallState<Prop>,
    signals: NamedCallStateComputed<Prop>,
    methods: {}
  }
>;
export function withCallState<Prop extends string>(config: {
  prop: Prop;
}): SignalStoreFeature {
[…]
}
```

Теперь система типов знает о наших настраиваемых свойствах. Кроме того, важно создать эти свойства во время выполнения. Для этой цели используется вспомогательная функция `getCallStateKeys`:

```typescript
function getCallStateKeys(config: { prop: string }) {
  return {
    callStateKey: `${config.prop}CallState`,
    loadingKey: `${config.prop}Loading`,
    loadedKey: `${config.prop}Loaded`,
    errorKey: `${config.prop}Error`,
  };
}
```

Эта вспомогательная функция возвращает во время выполнения (runtime) те же сопоставления (mappings), что и введённые ранее типы во время компиляции (compile time). Обновлённая реализация `withCallState` подхватывает эти имена и настраивает соответствующие свойства:

```typescript
[…]
export function withCallState<Prop extends string>(config: {
  prop: Prop;
}): SignalStoreFeature {
  const { callStateKey, errorKey, loadedKey, loadingKey } =
    getCallStateKeys(config);

  return signalStoreFeature(
    withState({ [callStateKey]: 'init' }),
    withComputed((state: Record<string, Signal<unknown>>) => {

      const callState = state[callStateKey] as Signal<CallState>;

      return {
        [loadingKey]: computed(() => callState() === 'loading'),
        [loadedKey]: computed(() => callState() === 'loaded'),
        [errorKey]: computed(() => {
          const v = callState();
          return typeof v === 'object' ? v.error : null;
        })
      }
    })
  );
}
```

Чтобы апдейтеры могли работать с динамическими свойствами, они также получают соответствующий параметр:

```typescript
export function setLoading<Prop extends string>(
  prop: Prop
): NamedCallState<Prop> {
  return { [`${prop}CallState`]: 'loading' } as NamedCallState<Prop>;
}
```

Эту же идею можно встретить и в `@ngrx/signals/entity`. Апдейтер затем используется следующим образом:

```typescript
load: async () => {
  patchState(state, setLoading('flights'));
  […]
}
```

## Заключение

Команда NGRX известна тем, что исключительно умело задействует возможности системы типов TypeScript. Результат — крайне простой в использовании и при этом типобезопасный API.

В этой статье мы сменили перспективу и обсудили, как можно задействовать паттерны, применяемые командой NGRX, для своих пользовательских фич Signal Store. Это даёт возможность настраивать имена свойств и тем самым избегать конфликтов имён без ущерба для типобезопасности.

Для этого нам приходится иметь дело с аспектами TypeScript, с которыми разработчики приложений обычно сталкиваются не так часто. Поэтому используемые паттерны порой могут казаться несколько сложными. Хорошая новость в том, что эти паттерны нужны нам, только когда мы разрабатываем высоко переиспользуемые решения. Как только мы возвращаемся к роли разработчика приложения, в нашем распоряжении оказывается типобезопасное решение, удобное в использовании.
