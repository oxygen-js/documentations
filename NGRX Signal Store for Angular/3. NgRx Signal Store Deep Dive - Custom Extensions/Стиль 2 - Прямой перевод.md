NGRX Signal Store, выпущенный вскоре после Angular 17, предлагает очень лёгкое решение для управления состоянием. Не имея прямых зависимостей от RxJS, он полностью полагается на сигналы (Signals). Однако его наибольшая сила — без сомнения, высокая степень расширяемости. С помощью так называемых пользовательских фич (custom features) повторяющиеся задачи могут быть очень легко реализованы централизованным образом.

Первый пример пользовательских фич, представленный здесь, очень прост. После этого всё становится немного сложнее: потребитель фичи должен иметь возможность определять имена сигналов и методов, создаваемых фичей. Иначе быстро возникнут конфликты имён. Как показывают следующие примеры, это не противоречит строгой типизации в TypeScript.

Показанные здесь примеры вдохновлены примером от [Марко Станимировича (Marko Stanimirović)](https://twitter.com/MarkoStDev) — участника основной команды NGRX, стоящего за Signal Store, — и решением для управления сущностями (Entity) `@ngrx/signals/entity`, поставляемым вместе с Signal Store.

## Signal Store 101

Прежде чем мы начнём, я хотел бы быстро ввести всех в курс дела и резюмировать наиболее важные аспекты NGRX Signal Store. [Подробный обзор](https://www.angulararchitects.io/blog/the-new-ngrx-signal-store-for-angular-2-1-flavors/) можно найти в [моей статье здесь](https://www.angulararchitects.io/blog/the-new-ngrx-signal-store-for-angular-2-1-flavors/).

Signal Store входит в состав NGRX начиная с версии 17 и находится в npm-пакете `@ngrx/signals`. В центре находится функция `signalStore`, которая создаёт новый Signal Store. Технически это сервис Angular, который получает сигналы, производные (вычисляемые) сигналы и методы для управления состоянием:

```typescript
import {
  patchState,
  signalStore,
  withHooks,
  withMethods,
  withComputed,
  withState,
} from '@ngrx/signals';

[…]

export const FlightBookingStore = signalStore(

  { providedIn: 'root' },

  // Свойства состояния
  withState({
    from: 'Paris',
    to: 'London',
    initialized: false,
    flights: [] as Flight[],
    basket: {} as Record<number, boolean>,
  }),

  // Вычисляемое состояние
  withComputed(({ flights, basket, from, to }) => ({
    selected: computed(() => flights().filter((f) => basket()[f.id])),
    criteria: computed(() => ({ from: from(), to: to() })),
  })),

  // Расширение
  withCallState(),

  // Методы
  withMethods((state) => {
    const { basket, flights, from, to, initialized } = state;
    const flightService = inject(FlightService);

    return {
      load: async () => {
        if (!from() || !to()) return;

        // Обновление состояния расширения (callState)
        patchState(state, setLoading());

        const flights = await flightService.findPromise(from(), to());
        patchState(state, { flights });

        // Обновление состояния расширения (callState)
        patchState(state, setLoaded());

      },
    };

  })
);
```

Что интересно, Signal Store уже на этой первой ступени расширения полностью основан на так называемых фичах: `withState`, `withComputed` или `withMethods` — это три фичи, которые обогащают хранилище свойствами (сигналами и вычисляемыми сигналами) и методами.

Функция `withCallState` — это первая простая пользовательская фича, которая вводит сигнал `callState`. Этот сигнал, среди прочего, сообщает, загружаются ли данные в данный момент или процесс загрузки уже завершён. Она вводит следующие сигналы:

* `callState`: может содержать значения `init`, `loading`, `loaded` или объект с сообщением об ошибке.
* `loading`: вычисляемый сигнал; возвращает `true`, если `callState` имеет значение `loading`.
* `loaded`: вычисляемый сигнал; возвращает `true`, если `callState` имеет значение `loaded`.
* `error`: вычисляемый сигнал; возвращает сообщение об ошибке, которое может храниться в `callState`, или `null`.

Фича `CallState` также предлагает так называемые апдейтеры (Updaters), такие как `setLoading` или `setLoaded`. Это функции, используемые потребителями, как в листинге 1, для изменения значения `callState`. В целом апдейтеры позволяют потребителям изменять управляемое состояние, не зная точно, как оно структурировано.

Следующий листинг показывает простой компонент, использующий `FlightBookingStore`. Он делегирует отдельным сигналам и своему методу `load`. Он также обращается к сигналу `loading`, созданному фичей `CallState`:

```typescript
@Component({ … })
export class FlightSearchComponent {
  private store = inject(FlightBookingStore);

  // Получение сигналов из хранилища
  from = this.store.from;
  to = this.store.to;
  basket = this.store.basket;
  flights = this.store.flights;
  selected = this.store.selected;

  // Получение сигнала из расширения
  loading = this.store.loading;

  async search() {
    this.store.load();
  }
}
```

## Простое первое расширение

Вот первая, очень простая версия фичи `CallState`, использованной в предыдущем разделе:

```typescript
import {
  SignalStoreFeature,
  signalStoreFeature,
  withComputed,
  withState,
} from '@ngrx/signals';

[…]

export type CallState = 'init' | 'loading' | 'loaded' | { error: string };

export function withCallState() {
  return signalStoreFeature(
    withState<{ callState: CallState }>({ callState: 'init' }),
    withComputed(({ callState }) => ({
      loading: computed(() => callState() === 'loading'),
      loaded: computed(() => callState() === 'loaded'),
      error: computed(() => {
        const state = callState();
        return typeof state === 'object' ? state.error : null
      }),
    }))
  );
}
```

Это функция, которая возвращает результат `signalStoreFeature`. Функция `signalStoreFeature`, в свою очередь, просто группирует существующие фичи: `withState` вводит свойство `callState`, а `withComputed` определяет на его основе обсуждавшиеся ранее вычисляемые сигналы.

Апдейтеры, предоставляемые фичей, возвращают лишь частичный объект состояния со свойством, которое нужно обновить:

```typescript
export function setLoading(): { callState: CallState } {
  return { callState: 'loading' };
}

export function setLoaded(): { callState: CallState } {
  return { callState: 'loaded' };
}   

export function setError(error: string): { callState: CallState } {
  return { callState: { error } };
}
```

## Теперь действительно начинается: типизация

Реализация `CallState` из предыдущего раздела кратко резюмирует решение повторяющегося требования. Будучи однажды реализованной, отдельные приложения могут интегрировать эту фичу в свои хранилища.

Однако недостаток этой реализации заключается в том, что введённые сигналы имеют фиксированные имена: `callState`, `loading`, `loaded` и `error`. Это быстро приводит к конфликтам имён, особенно если одному и тому же хранилищу фича требуется более одного раза. Примером этого является хранилище, которое хочет управлять несколькими `callState` для разных сущностей, например для рейсов (flights) и пассажиров (passengers).

В этом случае потребитель должен иметь возможность указывать имена вводимых сигналов. Именно об этом мы и позаботимся ниже. Чтобы сделать это расширение типобезопасным, нам сначала нужно немного подумать о типизации функции `withCallState`.

Наша функция `withCallState` в настоящее время не имеет явного типа возвращаемого значения. Поэтому TypeScript выводит этот тип, рассматривая возвращаемое значение в функции. Компилятор понимает, что доступно свойство `callState`.

Определённый здесь путём вывода тип — это `SignalStoreFeature<Input, Output>`. Параметр типа `Input` определяет, какие сигналы и методы фича ожидает от хранилища, а `Output` указывает, какие дополнительные сигналы и методы фича предоставляет. Наша фича не предъявляет никаких ожиданий к хранилищу, но предоставляет сигнал `callState`, а также несколько вычисляемых сигналов, таких как `loading`. Соответственно, наши типы `Input` и `Output` выглядят следующим образом:

![Пользовательские расширения с точки зрения системы типов](https://www.angulararchitects.io/wp-content/uploads/2023/11/extensions-type-system-01.png)

Следует отметить, что `state` описывает сигнал, который нужно ввести, а свойство `signals` представляет сигналы, вычисленные на его основе. Это представление соответствует, по крайней мере, упрощённому внешнему виду.

Внутренний вид немного сложнее, особенно потому, что `withState` сначала вводит сигнал `callState` и только затем `withComputed` добавляет вычисляемые сигналы. Поэтому внутренний вид имеет два выхода (output), которые объединяются с помощью вспомогательного типа.

![Пользовательские расширения с точки зрения системы типов](https://www.angulararchitects.io/wp-content/uploads/2023/11/extensions-type-system-02.png)

Для простоты на предыдущем изображении вспомогательный тип назван `Merged Result`. Однако истина в том, что Signal Store имеет для этого несколько внутренних типов.

На логическом уровне внутренний вид и внешний эквивалентны. TypeScript может потребоваться небольшой толчок в виде утверждения типа (type assertion), чтобы это распознать. Однако явное определение внутреннего вида немного раздражает и в настоящее время не вполне возможно, поскольку требуемые вспомогательные типы не являются частью публичного API Signal Store. Поэтому я использую здесь паттерн, который можно также неоднократно встретить в коде Signal Store: сочетание перегрузки функции с внешним видом и реализации функции, использующей `SignalStoreFeature` вместо `SignalStoreFeature<Input, Output>` для внутреннего вида:

```typescript
// Перегрузка с внешним видом
export function withCallState()
  : SignalStoreFeature<
    {
      state: {},
      signals: {},
      methods: {}
    },
    {
      state: {
        callState: CallState
      },
      signals: {
        loading: Signal<boolean>,
        loaded: Signal<boolean>,
        error: Signal<{ error: string } | null>
      },
      methods: {}
    }>;
// Реализация с внутренним видом
export function withCallState(): SignalStoreFeature {
  return signalStoreFeature(
    withState<{ callState: CallState }>({ callState: 'init' }),
    withComputed(({ callState }) => ({
      loading: computed(() => callState() === 'loading'),
      loaded: computed(() => callState() === 'loaded'),
      error: computed(() => {
        const state = callState();
        return typeof state === 'object' ? state.error : null
      }),
    }))
  );
}
```

Тип `SignalStoreFeature` без параметров типа использует для `Input` и `Output` более общие типы, которые не предполагают конкретных имён или типов данных.

## Типизация и динамические свойства — как они работают вместе?

Теперь, когда базовая структура типизации на месте, мы можем расширить её настраиваемыми именами свойств. Следуя примеру `@ngrx/signals/entity`, потребители должны иметь возможность определять префикс при активации фичи:

```typescript
export const FlightBookingStore = signalStore(
  { providedIn: 'root' },

  withState({ … }),
  withComputed(( … ) => ({ … })),

  withCallState({ prop: 'flights' }), 
  withCallState({ prop: 'passengers'}),

  […]
);
```

Этот префикс должен теперь быть включён в имена свойств, определяемых фичей. Например, первый вызов `withCallState` должен создать следующие свойства:

* `flightsCallState` (состояние)
* `flightsLoading` (вычисляемое)
* `flightsLoaded` (вычисляемое)
* `flightsError` (вычисляемое)

Второй вызов аналогично приводит к этим свойствам:

* `passengersCallState` (состояние)
* `passengersLoading` (вычисляемое)
* `passengersLoaded` (вычисляемое)
* `passengersError` (вычисляемое)

Создание этих свойств во время выполнения — не большая проблема в мире TypeScript, особенно потому, что лежащий в основе JavaScript в любом случае является динамическим языком. Однако сложность заключается в том, чтобы также сообщить о этих свойствах системе типов.

Для этой задачи сначала нужно найти способ выразить префикс в объявлении типа. На этом этапе нам идёт на пользу тот факт, что литералы также могут использоваться в качестве типов данных:

```typescript
export type BoxStatus = 'open' | 'closed';
const candyBox: BoxStatus = 'open';
```

Такие объединённые типы строковых литералов (String Literal Union Types) часто используются в TypeScript-приложениях для выражения перечислений (enum). Это даже ближе к EcmaScript, чем использование ключевого слова `enum` из TypeScript. Забавно, что никто не заставляет нас предлагать несколько вариантов. Поэтому такой вариант полностью допустим:

```typescript
export type BoxStatusAfterHolidays = 'closed';
```

Итак, здесь у нас есть тип, который может содержать ровно одно строковое значение. Мы используем именно этот паттерн, чтобы сообщить системе типов о нашем префиксе. Сначала мы создаём тип, который определяет имя вводимого сигнала на основе префикса:

```typescript
export type NamedCallState<Prop extends string> = {
  [K in Prop as `${K}CallState`]: CallState;
};
```

Это так называемый отображаемый тип (mapped type), который отображает один тип в новый. Параметр типа `Prop extends string` описывает исходный тип. Это может быть любая строка, используемая как тип. `string` также должен быть записан в нижнем регистре, потому что в этой точке мы ссылаемся на конкретную строку, а не на тип объекта `String`. Запись `K in Prop` также сводится к этой строке. В более сложных случаях можно было бы использовать ключевое слово `in`, например, чтобы пройтись циклом по свойствам исходного типа.

Аналогично мы можем поступить с вычисляемыми сигналами, которые нужно ввести:

```typescript
export type NamedCallStateComputed<Prop extends string> = {
  [K in Prop as `${K}Loading`]: Signal<boolean>;
} & {
    [K in Prop as `${K}Loaded`]: Signal<boolean>;
} & {
    [K in Prop as `${K}Error`]: Signal<string | null>;
};
```

Поскольку отображаемый тип может иметь только одно отображение, здесь используется несколько отображаемых типов. Они объединяются с помощью оператора `&` (оператор пересечения, intersection). С помощью этих двух типов мы теперь можем задать типизацию нашей функции `withCallState`:

```typescript
export function withCallState<Prop extends string>(config: {
  prop: Prop;
}): SignalStoreFeature<
  { state: {}, signals: {}, methods: {} },
  {
    state: NamedCallState<Prop>,
    signals: NamedCallStateComputed<Prop>,
    methods: {}
  }
>;
export function withCallState<Prop extends string>(config: {
  prop: Prop;
}): SignalStoreFeature {
[…]
}
```

Теперь система типов знает о наших настроенных свойствах. Кроме того, теперь важно создать эти свойства во время выполнения. Для этой цели используется вспомогательная функция `getCallStateKeys`:

```typescript
function getCallStateKeys(config: { prop: string }) {
  return {
    callStateKey: `${config.prop}CallState`,
    loadingKey: `${config.prop}Loading`,
    loadedKey: `${config.prop}Loaded`,
    errorKey: `${config.prop}Error`,
  };
}
```

Эта вспомогательная функция возвращает во время выполнения те же сопоставления, что и введённые ранее типы во время компиляции. Обновлённая реализация `withCallState` подхватывает эти имена и создаёт соответствующие свойства:

```typescript
[…]
export function withCallState<Prop extends string>(config: {
  prop: Prop;
}): SignalStoreFeature {
  const { callStateKey, errorKey, loadedKey, loadingKey } =
    getCallStateKeys(config);

  return signalStoreFeature(
    withState({ [callStateKey]: 'init' }),
    withComputed((state: Record<string, Signal<unknown>>) => {

      const callState = state[callStateKey] as Signal<CallState>;

      return {
        [loadingKey]: computed(() => callState() === 'loading'),
        [loadedKey]: computed(() => callState() === 'loaded'),
        [errorKey]: computed(() => {
          const v = callState();
          return typeof v === 'object' ? v.error : null;
        })
      }
    })
  );
}
```

Чтобы апдейтеры могли справляться с динамическими свойствами, они также получают соответствующий параметр:

```typescript
export function setLoading<Prop extends string>(
  prop: Prop
): NamedCallState<Prop> {
  return { [`${prop}CallState`]: 'loading' } as NamedCallState<Prop>;
}
```

Эту идею также можно найти в `@ngrx/signals/entity`. Затем апдейтер используется следующим образом:

```typescript
load: async () => {
  patchState(state, setLoading('flights'));
  […]
}
```

## Заключение

Команда NGRX известна тем, что исключительно умело использует возможности системы типов TypeScript. Результатом является чрезвычайно простой в использовании и типобезопасный API.

В этой статье мы сменили перспективу и обсудили, как вы можете использовать паттерны, применяемые командой NGRX, для своих пользовательских фич Signal Store. Это позволяет настраивать имена свойств и тем самым избегать конфликтов имён без ущерба для типобезопасности.

Чтобы сделать это, нам приходится иметь дело с аспектами TypeScript, с которыми разработчики приложений обычно сталкиваются не так часто. Поэтому используемые паттерны иногда могут казаться немного сложными. Хорошая новость в том, что эти паттерны нужны нам, только если мы разрабатываем высоко переиспользуемые решения. Как только мы возвращаемся к роли разработчика приложений, у нас есть типобезопасное решение, которым удобно пользоваться.
