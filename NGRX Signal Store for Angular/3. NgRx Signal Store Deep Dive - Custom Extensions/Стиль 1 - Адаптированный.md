NGRX Signal Store, вышедший вскоре после Angular 17, предлагает очень лёгкое решение для управления состоянием. Он не зависит напрямую от RxJS и целиком построен на сигналах (Signals). Но его главная сила — без сомнения, высокая степень расширяемости. С помощью так называемых пользовательских фич (custom features) повторяющиеся задачи можно очень легко реализовать централизованно.

Первый пример пользовательской фичи, который мы здесь рассмотрим, совсем простой. А вот дальше станет интереснее: потребитель фичи должен иметь возможность сам задавать имена сигналов и методов, которые эта фича создаёт. Иначе конфликты имён появятся очень быстро. И, как покажут дальнейшие примеры, это вовсе не противоречит строгой типизации в TypeScript.

Приведённые здесь примеры вдохновлены примером от [Марко Станимировича (Marko Stanimirović)](https://twitter.com/MarkoStDev) — участника основной команды NGRX, стоящего за Signal Store, — а также решением для управления сущностями `@ngrx/signals/entity`, которое поставляется вместе с Signal Store.

## Signal Store 101

Прежде чем начать, давайте быстро введём всех в курс дела и подытожим самое важное про NGRX Signal Store. [Подробный обзор](https://www.angulararchitects.io/blog/the-new-ngrx-signal-store-for-angular-2-1-flavors/) есть в [моей статье вот тут](https://www.angulararchitects.io/blog/the-new-ngrx-signal-store-for-angular-2-1-flavors/).

Signal Store входит в состав NGRX начиная с версии 17 и находится в npm-пакете `@ngrx/signals`. В его центре — функция `signalStore`, создающая новый Signal Store. Технически это сервис Angular, который получает сигналы, производные (вычисляемые) сигналы и методы для управления состоянием:

```typescript
import {
  patchState,
  signalStore,
  withHooks,
  withMethods,
  withComputed,
  withState,
} from '@ngrx/signals';

[…]

export const FlightBookingStore = signalStore(

  { providedIn: 'root' },

  // Свойства состояния
  withState({
    from: 'Paris',
    to: 'London',
    initialized: false,
    flights: [] as Flight[],
    basket: {} as Record<number, boolean>,
  }),

  // Вычисляемое состояние
  withComputed(({ flights, basket, from, to }) => ({
    selected: computed(() => flights().filter((f) => basket()[f.id])),
    criteria: computed(() => ({ from: from(), to: to() })),
  })),

  // Расширение
  withCallState(),

  // Методы
  withMethods((state) => {
    const { basket, flights, from, to, initialized } = state;
    const flightService = inject(FlightService);

    return {
      load: async () => {
        if (!from() || !to()) return;

        // Обновляем состояние расширения (callState)
        patchState(state, setLoading());

        const flights = await flightService.findPromise(from(), to());
        patchState(state, { flights });

        // Обновляем состояние расширения (callState)
        patchState(state, setLoaded());

      },
    };

  })
);
```

Что интересно, Signal Store уже на этой первой ступени целиком построен на так называемых фичах: `withState`, `withComputed` и `withMethods` — это три фичи, которые наполняют хранилище свойствами (сигналами и вычисляемыми сигналами) и методами.

Функция `withCallState` — это первая простая пользовательская фича, которая вводит сигнал `callState`. Этот сигнал, среди прочего, сообщает, идёт ли сейчас загрузка данных или процесс загрузки уже завершён. Он вводит следующие сигналы:

* `callState`: может принимать значения `init`, `loading`, `loaded` или объект с сообщением об ошибке.
* `loading`: вычисляемый сигнал; возвращает `true`, если у `callState` значение `loading`.
* `loaded`: вычисляемый сигнал; возвращает `true`, если у `callState` значение `loaded`.
* `error`: вычисляемый сигнал; возвращает сообщение об ошибке, которое может храниться в `callState`, либо `null`.

Фича `CallState` также предлагает так называемые апдейтеры (Updaters), такие как `setLoading` или `setLoaded`. Это функции, которые потребители (как в листинге 1) используют для изменения значения `callState`. В целом апдейтеры позволяют потребителям менять управляемое состояние, не зная в точности, как оно устроено внутри.

В следующем листинге показан простой компонент, который использует `FlightBookingStore`. Он делегирует работу отдельным сигналам и методу `load`. Кроме того, он обращается к сигналу `loading`, который создаёт фича `CallState`:

```typescript
@Component({ … })
export class FlightSearchComponent {
  private store = inject(FlightBookingStore);

  // Получаем сигналы из хранилища
  from = this.store.from;
  to = this.store.to;
  basket = this.store.basket;
  flights = this.store.flights;
  selected = this.store.selected;

  // Получаем сигнал из расширения
  loading = this.store.loading;

  async search() {
    this.store.load();
  }
}
```

## Первое, совсем простое расширение

Вот первая, очень простая версия фичи `CallState`, которую мы использовали в предыдущем разделе:

```typescript
import {
  SignalStoreFeature,
  signalStoreFeature,
  withComputed,
  withState,
} from '@ngrx/signals';

[…]

export type CallState = 'init' | 'loading' | 'loaded' | { error: string };

export function withCallState() {
  return signalStoreFeature(
    withState<{ callState: CallState }>({ callState: 'init' }),
    withComputed(({ callState }) => ({
      loading: computed(() => callState() === 'loading'),
      loaded: computed(() => callState() === 'loaded'),
      error: computed(() => {
        const state = callState();
        return typeof state === 'object' ? state.error : null
      }),
    }))
  );
}
```

Это функция, которая возвращает результат `signalStoreFeature`. А функция `signalStoreFeature`, в свою очередь, просто группирует уже существующие фичи: `withState` вводит свойство `callState`, а `withComputed` определяет на его основе обсуждавшиеся ранее вычисляемые сигналы.

Апдейтеры, которые предоставляет фича, лишь возвращают частичный объект состояния с тем свойством, которое нужно обновить:

```typescript
export function setLoading(): { callState: CallState } {
  return { callState: 'loading' };
}

export function setLoaded(): { callState: CallState } {
  return { callState: 'loaded' };
}   

export function setError(error: string): { callState: CallState } {
  return { callState: { error } };
}
```

## Вот теперь начинается самое интересное: типизация

Реализация `CallState` из предыдущего раздела кратко описывает решение одной повторяющейся задачи. Будучи однажды написанной, фича может встраиваться в хранилища отдельных приложений.

Однако у этой реализации есть недостаток: введённые сигналы имеют фиксированные имена — `callState`, `loading`, `loaded` и `error`. Это быстро приводит к конфликтам имён, особенно если одному и тому же хранилищу фича нужна больше одного раза. Пример — хранилище, которому нужно управлять несколькими `callState` для разных сущностей, например для рейсов (flights) и пассажиров (passengers).

В этом случае потребитель должен иметь возможность сам задать имена вводимых сигналов. Именно этим мы и займёмся дальше. Чтобы сделать такое расширение типобезопасным, нам сначала придётся немного поразмышлять о типизации функции `withCallState`.

Сейчас у нашей функции `withCallState` нет явного типа возвращаемого значения. Поэтому TypeScript выводит (инферит) этот тип, глядя на то, что функция возвращает. Компилятор понимает, что доступно свойство `callState`.

Выведенный здесь тип — это `SignalStoreFeature<Input, Output>`. Параметр типа `Input` определяет, какие сигналы и методы фича ожидает от хранилища, а `Output` указывает, какие дополнительные сигналы и методы она предоставляет. Наша фича ничего не ожидает от хранилища, но предоставляет сигнал `callState`, а также несколько вычисляемых сигналов вроде `loading`. Соответственно, наши типы `Input` и `Output` выглядят так:

![Пользовательские расширения с точки зрения системы типов](https://www.angulararchitects.io/wp-content/uploads/2023/11/extensions-type-system-01.png)

Стоит отметить, что `state` описывает сигнал, который нужно ввести, а свойство `signals` представляет сигналы, вычисленные на его основе. Это представление соответствует, по крайней мере, упрощённому взгляду снаружи.

Внутреннее устройство немного сложнее, прежде всего потому, что `withState` сначала вводит сигнал `callState`, и только потом `withComputed` добавляет вычисляемые сигналы. Поэтому внутренний взгляд содержит два выхода (output), которые объединяются с помощью вспомогательного типа.

![Пользовательские расширения с точки зрения системы типов](https://www.angulararchitects.io/wp-content/uploads/2023/11/extensions-type-system-02.png)

Для простоты на предыдущей картинке вспомогательный тип назван `Merged Result`. Но на деле у Signal Store для этого есть несколько внутренних типов.

Логически внутренний и внешний взгляды эквивалентны. Возможно, TypeScript потребуется лёгкий «толчок» в виде утверждения типа (type assertion), чтобы он это распознал. Однако явно описывать внутренний взгляд немного утомительно и сейчас не вполне возможно, поскольку нужные вспомогательные типы не входят в публичный API Signal Store. Поэтому я применяю здесь приём, который не раз встречается и в коде самого Signal Store: сочетание перегрузки функции с внешним взглядом и реализации функции, использующей `SignalStoreFeature` вместо `SignalStoreFeature<Input, Output>` для внутреннего взгляда:

```typescript
// Перегрузка с внешним взглядом
export function withCallState()
  : SignalStoreFeature<
    {
      state: {},
      signals: {},
      methods: {}
    },
    {
      state: {
        callState: CallState
      },
      signals: {
        loading: Signal<boolean>,
        loaded: Signal<boolean>,
        error: Signal<{ error: string } | null>
      },
      methods: {}
    }>;
// Реализация с внутренним взглядом
export function withCallState(): SignalStoreFeature {
  return signalStoreFeature(
    withState<{ callState: CallState }>({ callState: 'init' }),
    withComputed(({ callState }) => ({
      loading: computed(() => callState() === 'loading'),
      loaded: computed(() => callState() === 'loaded'),
      error: computed(() => {
        const state = callState();
        return typeof state === 'object' ? state.error : null
      }),
    }))
  );
}
```

Тип `SignalStoreFeature` без параметров типа использует для `Input` и `Output` более общие типы, которые не предполагают конкретных имён или типов данных.

## Типизация и динамические свойства — как они уживаются вместе?

Теперь, когда базовая структура типизации готова, мы можем расширить её настраиваемыми именами свойств. По примеру `@ngrx/signals/entity` потребители должны иметь возможность задать префикс при активации фичи:

```typescript
export const FlightBookingStore = signalStore(
  { providedIn: 'root' },

  withState({ … }),
  withComputed(( … ) => ({ … })),

  withCallState({ prop: 'flights' }), 
  withCallState({ prop: 'passengers'}),

  […]
);
```

Этот префикс должен теперь войти в имена свойств, которые определяет фича. Например, первый вызов `withCallState` должен породить такие свойства:

* `flightsCallState` (состояние)
* `flightsLoading` (вычисляемое)
* `flightsLoaded` (вычисляемое)
* `flightsError` (вычисляемое)

Второй вызов аналогично приводит к таким свойствам:

* `passengersCallState` (состояние)
* `passengersLoading` (вычисляемое)
* `passengersLoaded` (вычисляемое)
* `passengersError` (вычисляемое)

Создать эти свойства во время выполнения — задача несложная в мире TypeScript, тем более что лежащий в основе JavaScript и так является динамическим языком. А вот сложность в другом: нужно ещё и рассказать о этих свойствах системе типов.

Для этого сначала нужно найти способ выразить префикс в объявлении типа. Здесь нам на руку играет тот факт, что литералы тоже можно использовать в качестве типов данных:

```typescript
export type BoxStatus = 'open' | 'closed';
const candyBox: BoxStatus = 'open';
```

Такие объединённые типы из строковых литералов (String Literal Union Types) часто применяются в TypeScript-приложениях, чтобы выразить перечисления (enum). Это даже ближе к EcmaScript, чем использование ключевого слова `enum` из TypeScript. Забавно, что никто не заставляет нас предлагать несколько вариантов. Поэтому вот такой вариант тоже вполне допустим:

```typescript
export type BoxStatusAfterHolidays = 'closed';
```

Итак, у нас есть тип, который может хранить ровно одно строковое значение. Точно такой же приём мы используем, чтобы рассказать системе типов о нашем префиксе. Сначала создадим тип, который на основе префикса задаёт имя вводимого сигнала:

```typescript
export type NamedCallState<Prop extends string> = {
  [K in Prop as `${K}CallState`]: CallState;
};
```

Это так называемый отображаемый тип (mapped type), который отображает один тип в новый. Параметр типа `Prop extends string` описывает исходный тип. Это может быть любая строка, используемая как тип. При этом `string` должен быть в нижнем регистре, ведь здесь мы имеем в виду конкретную строку, а не тип объекта `String`. Запись `K in Prop` тоже сводится к этой строке. В более сложных случаях ключевое слово `in` можно использовать, например, чтобы пройтись по свойствам исходного типа.

Точно так же можно поступить и с вычисляемыми сигналами, которые мы хотим ввести:

```typescript
export type NamedCallStateComputed<Prop extends string> = {
  [K in Prop as `${K}Loading`]: Signal<boolean>;
} & {
    [K in Prop as `${K}Loaded`]: Signal<boolean>;
} & {
    [K in Prop as `${K}Error`]: Signal<string | null>;
};
```

Поскольку отображаемый тип может содержать только одно отображение, здесь используются несколько отображаемых типов. Они объединяются оператором `&` (оператор пересечения, intersection). С помощью этих двух типов мы теперь можем задать типизацию нашей функции `withCallState`:

```typescript
export function withCallState<Prop extends string>(config: {
  prop: Prop;
}): SignalStoreFeature<
  { state: {}, signals: {}, methods: {} },
  {
    state: NamedCallState<Prop>,
    signals: NamedCallStateComputed<Prop>,
    methods: {}
  }
>;
export function withCallState<Prop extends string>(config: {
  prop: Prop;
}): SignalStoreFeature {
[…]
}
```

Теперь система типов знает о наших настраиваемых свойствах. Кроме того, важно создать эти свойства во время выполнения. Для этого используется вспомогательная функция `getCallStateKeys`:

```typescript
function getCallStateKeys(config: { prop: string }) {
  return {
    callStateKey: `${config.prop}CallState`,
    loadingKey: `${config.prop}Loading`,
    loadedKey: `${config.prop}Loaded`,
    errorKey: `${config.prop}Error`,
  };
}
```

Эта вспомогательная функция возвращает во время выполнения те же сопоставления, что и введённые ранее типы во время компиляции. Обновлённая реализация `withCallState` берёт эти имена и создаёт соответствующие свойства:

```typescript
[…]
export function withCallState<Prop extends string>(config: {
  prop: Prop;
}): SignalStoreFeature {
  const { callStateKey, errorKey, loadedKey, loadingKey } =
    getCallStateKeys(config);

  return signalStoreFeature(
    withState({ [callStateKey]: 'init' }),
    withComputed((state: Record<string, Signal<unknown>>) => {

      const callState = state[callStateKey] as Signal<CallState>;

      return {
        [loadingKey]: computed(() => callState() === 'loading'),
        [loadedKey]: computed(() => callState() === 'loaded'),
        [errorKey]: computed(() => {
          const v = callState();
          return typeof v === 'object' ? v.error : null;
        })
      }
    })
  );
}
```

Чтобы апдейтеры тоже справлялись с динамическими свойствами, они получают соответствующий параметр:

```typescript
export function setLoading<Prop extends string>(
  prop: Prop
): NamedCallState<Prop> {
  return { [`${prop}CallState`]: 'loading' } as NamedCallState<Prop>;
}
```

Эту же идею можно встретить и в `@ngrx/signals/entity`. Используется такой апдейтер вот так:

```typescript
load: async () => {
  patchState(state, setLoading('flights'));
  […]
}
```

## Заключение

Команда NGRX известна тем, что исключительно умело выжимает максимум из возможностей системы типов TypeScript. Результат — крайне удобный в использовании и при этом типобезопасный API.

В этой статье мы сменили угол зрения и обсудили, как вы можете применить приёмы команды NGRX в своих собственных фичах Signal Store. Это позволяет настраивать имена свойств и тем самым избегать конфликтов имён, не жертвуя типобезопасностью.

Для этого нам пришлось коснуться тех аспектов TypeScript, с которыми разработчики приложений обычно сталкиваются не так уж часто. Поэтому используемые приёмы порой могут казаться немного запутанными. Хорошая новость в том, что эти приёмы нужны нам, только когда мы разрабатываем сильно переиспользуемые решения. А как только мы возвращаемся в роль разработчика приложения, у нас на руках типобезопасное решение, которым удобно пользоваться.
