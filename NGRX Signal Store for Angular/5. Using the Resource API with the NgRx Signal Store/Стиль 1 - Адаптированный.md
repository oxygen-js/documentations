В версии 19 NgRx появилась фича `withProps` — она позволяет добавлять в Store произвольные свойства. Благодаря ей Store может, например, держать у себя сервисы, Observable-объекты или ресурсы (resources).

В этой статье я покажу, как с помощью `withProps` задействовать новый Resource API прямо внутри Signal Store. Это даёт Store привычное реактивное поведение и защищает от состояний гонки (race conditions) — причём без RxJS. А чтобы связать со Store шаблонную форму (template-driven form), в примере используется `linkedSignal` в паре с новым помощником `signalMethod` из Signal Store.

📂 [Исходный код](https://github.com/manfredsteyer/desserts.git) (см. ветку _08b-details_)

## Реактивный поток

Чтобы показать, как Resource API работает внутри Signal Store, я возьму приложение с десертами — оно уже знакомо вам по другим статьям блога:

![Пример приложения](https://www.angulararchitects.io/wp-content/uploads/2024/12/example-app.jpg)

Фильтры — Original Name и English Name — запускают реактивный поток, в результате чего на экране появляются десерты с рейтингами. А по нажатию на кнопку Store загружает рейтинги для тех десертов, что отображаются прямо сейчас.

Прежде чем начать, давайте изобразим этот поток в виде простого реактивного графа:

![](https://www.angulararchitects.io/wp-content/uploads/2024/12/reactive-graph.png)

Как видно, Store принимает намерения потребителя (например, компонента) через три метода, показанных слева. Сигнал `filter` запускает `dessertsResource`: каждый раз, когда фильтр меняется, ресурс подгружает десерты.

`ratingsResource` напрямую запускается методом `loadRatings`, а метод `updateRatings` обновляет локальную рабочую копию (working copy) этого ресурса. Ещё один метод, не показанный на схеме, мог бы записывать эту рабочую копию обратно в backend.

## Взгляд со стороны потребителя

Теперь посмотрим на Store «снаружи» — изучим компонент, который его использует:

```typescript
@Component([...])
export class DessertsComponent {
  #store = inject(DessertStore);

  originalName = linkedSignal(() => this.#store.filter.originalName());
  englishName = linkedSignal(() => this.#store.filter.englishName());

  ratedDesserts = this.#store.ratedDesserts;
  loading = this.#store.loading;

  #linkedFilter = computed(() => ({
    originalName: this.originalName(),
    englishName: this.englishName()
  }));

  constructor() {
    this.#store.updateFilter(this.#linkedFilter)
  }

  loadRatings(): void {
    this.#store.loadRatings();
  }

  updateRating(id: number, rating: number): void {
    this.#store.updateRating(id, rating);
  }
}
```

Поскольку в примере фильтр (`originalName` и `englishName`) привязан к шаблонной форме, компонент оборачивает эти поля в Linked Signals. Это позволяет использовать `ngModel` с двусторонним связыванием:

```html
<input [(ngModel)]="originalName" name="originalName" />
<input [(ngModel)]="englishName" name="englishName" />
```

Здесь двустороннее связывание обновляет локальную рабочую копию, которую предоставляет Linked Signal. А чтобы эта копия попала обратно в Store, конструктор передаёт её в метод `updateFilter`.

Тут важно заметить: `updateFilter` получает не просто текущий фильтр, а вычисляемый (computed) сигнал `linkedFilter`. Чуть ниже мы увидим, что этот метод — так называемый Signal Method, который срабатывает каждый раз, когда переданный сигнал меняется. Именно так в примере фильтр в Store остаётся актуальным.

Признаю, с реактивной формой синхронизировать форму со Store было бы проще.

## Сам Store

Если заглянуть внутрь Store, мы увидим, что его состояние (state) — это и есть фильтр:

```typescript
export type Requested = undefined | true;

export const DessertStore = signalStore(
  { providedIn: 'root' },
  withState({
    filter: {
      originalName: '',
      englishName: 'Cake',
    },
    ratingsRequested: undefined as Requested
  }),
 [...],
);
```

Свойств для загруженных десертов и рейтингов здесь нет. Их дадут ресурсы, о которых речь пойдёт ниже. Состояние загрузки (loading state) тоже можно вывести из них. А так как для загрузки рейтингов параметры не нужны, мы используем просто флаг `ratingsRequested`.

## Получаем сервисы через withProps

Следующая фича, которую мы добавляем в Store, — это `withProps`. С её помощью Store внедряет сервисы, которые нам понадобятся дальше для настройки ресурсов и методов:

```typescript
export const DessertStore = signalStore(
  [...],
  withProps(() => ({
    _dessertService: inject(DessertService),
    _ratingService: inject(RatingService),
    _toastService: inject(ToastService),
  })),
  [...],
);
```

Это слегка упрощает реализацию Store. До появления `withProps` такие сервисы приходилось внедрять через параметры, добавляемые к отдельным фичам. А поскольку это приводило к длинным спискам параметров и требовало повторений для каждой фичи, где использовался один и тот же сервис, применять для этой задачи `withProps` куда удобнее.

Обратите также внимание, что имена свойств здесь начинаются с **подчёркивания.** По соглашению Signal Store считает такие свойства **приватными** и не показывает их потребителю.

## Настраиваем ресурсы

Для настройки ресурсов Store использует ещё один блок `withProps`:

```typescript
export const DessertStore = signalStore(
  [...],
  withProps((store) => ({
    _dessertsResource: resource({
      params: store.filter,
      loader: (loaderParams) => {
        const filter = loaderParams.params;
        const abortSignal = params.abortSignal;
        return store._dessertService.findPromise(filter, abortSignal);
      },
    }),
    _ratingsResource: resource({
      params: store.ratingsRequested
      loader: (loaderParams) => {
        const abortSignal = loaderParams.abortSignal;
        return store._ratingService.loadExpertRatingsPromise(abortSignal);
      }
    })
  })),
  [...],
);
```

Так как блок `withProps` с ресурсами объявлен после блока с сервисами, ресурсы могут к этим сервисам обращаться. `_dessertsResource` берёт фильтр из состояния Store в качестве своих `params`. А значит, любое изменение фильтра запускает загрузку десертов.

## Опционально: отдаём ресурсы только для чтения

Как и внедрённые сервисы, ресурсы из предыдущего раздела помечены как приватные. В нашем примере они — всего лишь деталь реализации: потребителю всё равно, как именно загружаются данные — через Resource API, через RxJS или ещё как-то.

Но если бы мы всё же захотели отдать ресурсы потребителю, можно было бы добавить ещё один блок `withProps` и выставить наружу их версии только для чтения (read-only):

```typescript
export const DessertStore = signalStore(
  [...],
  withProps((store) => ({
    dessertsResource: store._dessertsResource.asReadonly(),
    ratingsResource: store._ratingsResource.asReadonly(),
  })),
  [...],
);
```

Ресурс только для чтения даёт загруженные данные, состояние загрузки и информацию об ошибках, но не позволяет менять локальную рабочую копию. Эту идею — определять приватные ресурсы и выставлять их наружу как публичные read-only — можно автоматизировать пользовательской фичей вроде `withResource`.

## Выводим вью-модели

Чтобы получить вью-модели (view models), которые отображаются через связывание данных (data binding), Store вводит несколько вычисляемых сигналов:

```typescript
export const DessertStore = signalStore(
  [...],
  withComputed((store) => ({
    ratedDesserts: computed(() => toRated(
      store._dessertsResource.value(),
      store._ratingsResource.value()
    )),
    loading: computed(() =>
      store._dessertsResource.isLoading()
        || store._dessertsResource.isLoading())
  })),
  [...]
);
```

Вспомогательная функция `toRated` (здесь не показана) объединяет десерты с загруженными рейтингами. Сигнал `loading` возвращает `true`, когда хотя бы один из ресурсов находится в состоянии загрузки.

## Объявляем методы

Поскольку Resource API даёт нам реактивный (а значит, декларативный) способ загрузки данных, для методов остаётся совсем немного императивного кода:

```typescript
export const DessertStore = signalStore(
  [...],
  withMethods((store) => ({
    updateFilter: signalMethod<DessertFilter>((filter) => {
      patchState(store, { filter });
    }),
    loadRatings: () => {
        patchState(store, { ratingsRequested: true });
        store._ratingsResource.reload();
    },
    updateRating: (id: number, rating: number) => {
      store._ratingsResource.update(ratings => ({
        ...ratings,
        [id]: rating,
      }));
    },
  })),
  [...],
);
```

В основном эти методы просто записывают переданные значения в Store. Метод `updateFilter` собран на основе нового помощника `signalMethod`. Так как в примере используется `signalMethod<DessertFilter>`, метод принимает либо `DessertFilter`, либо `Signal<DessertFilter>`. Во втором случае метод выполняется каждый раз, когда переданный сигнал меняется.

При этом переданный сигнал отслеживается, а вот **реализация метода — нет (!)**, по соглашению она untracked. Поэтому метод не перезапускается, когда меняется какой-нибудь сигнал внутри вызываемых им методов. Обычно такое поведение и нужно: с одного взгляда видно, когда именно отрабатывает реализация метода.

Помощник `signalMethod` похож на `rxMethod`, который появился ещё в первой версии Signal Store. Но `signalMethod` не зависит от RxJS. А значит, заботиться о защите от состояний гонки нам придётся самим. В нашем примере эту задачу берёт на себя Resource API.

Метод `loadRatings` запускает `_ratingsResource`: он ставит флаг `ratingsRequested` в `true` и вызывает `reload`. Такой подход уже обсуждался [здесь](https://www.angulararchitects.io/en/blog/asynchronous-resources-with-angulars-new-resource-api/). Метод `updateRating` обновляет локальную рабочую копию ресурса. Признаю, выглядит это немного странно, ведь обычно store не меняет writable-сигналы напрямую. Однако это вписывается в саму идею локальных рабочих копий, которую дают `linkedSignal` и Resource API: напрямую обновлять временные данные прямо внутри реактивного потока. Возможно, со временем мы найдём более изящный способ решить эту задачу.

## Обработка ошибок с помощью хуков и эффектов

Если загрузить данные не получилось, ресурс сообщает о происходящем через свой сигнал `error`. Чтобы показать соответствующее toast-уведомление, Store настраивает хук `onInit` с эффектом (Effect), который отслеживает `error`:

```typescript
export const DessertStore = signalStore(
 [...],
  withHooks({
    onInit(store) {
      const toastService = store._toastService;
      const dessertsError = store._dessertsResource.error;

      effect(() => {
        const error = store._dessertsResource.error;
        if (error) {
          store._toastService.show('Error: ' + getMessage(error));
        }
      });
    }
  })
);
```

Метод `getMessage` возвращает сообщение об ошибке, полученное из объекта `error`:

```typescript
export function getMessage(error: unknown) {
  if (error && typeof error === "object" && "message" in error) {
    return error.message;
  }
  return String(error);
}
```

Чтобы было проще повторять эту операцию для других сигналов `error`, эффект можно вынести во вспомогательную функцию:

```typescript
export const DessertStore = signalStore(
  [...],
  withHooks({
    onInit(store) {
      const toastService = store._toastService;
      const dessertsError = store._dessertsResource.error;
      const ratingsError = store._ratingsResource.error;

      displayErrorEffect(dessertsError, toastService);
      displayErrorEffect(ratingsError, toastService);
    }
  })
);
```

Эта вспомогательная функция принимает `errorSignal` и `toastService` и настраивает тот самый эффект, что мы видели выше:

```typescript
export function displayErrorEffect(
  errorSignal: Signal<unknown>,
  toastService: ToastService
) {
  effect(() => {
    const error = errorSignal();
    if (error) {
      toastService.show("Error: " + getMessage(error));
    }
  });
}
```

## Собираем всё вместе

Теперь, когда мы разобрали все части Signal Store, использующего Resource API, давайте взглянем на полную реализацию store:

```typescript
export const DessertStore = signalStore(
  { providedIn: "root" },
  withState({
    filter: {
      originalName: "",
      englishName: "Cake",
    },
  }),
  withProps(() => ({
    _dessertService: inject(DessertService),
    _ratingService: inject(RatingService),
    _toastService: inject(ToastService),
  })),
  withProps((store) => ({
    _dessertsResource: resource({
      request: Store.filter,
      loader: (params) => {
        const filter = params.request;
        const abortSignal = params.abortSignal;
        return store._dessertService.findPromise(filter, abortSignal);
      },
    }),
    _ratingsResource: resource({
      loader: (params) => {
        const abortSignal = params.abortSignal;
        return store._ratingService.loadExpertRatingsPromise(abortSignal);
      },
    }),
  })),
  withProps((store) => ({
    dessertsResource: store._dessertsResource.asReadonly(),
    ratingsResource: store._ratingsResource.asReadonly(),
  })),
  withComputed((store) => ({
    ratedDesserts: computed(() =>
      toRated(store._dessertsResource.value(), store._ratingsResource.value())
    ),
    loading: computed(
      () =>
        store._dessertsResource.isLoading() ||
        store._dessertsResource.isLoading()
    ),
  })),
  withMethods((store) => ({
    updateFilter: signalMethod<DessertFilter>((filter) => {
      patchState(store, { filter });
    }),
    loadRatings: () => {
      store._ratingsResource.reload();
    },
    updateRating: (id: number, rating: number) => {
      store._ratingsResource.update((ratings) => ({
        ...ratings,
        [id]: rating,
      }));
    },
  })),
  withHooks({
    onInit(store) {
      const toastService = store._toastService;
      const dessertsError = store._dessertsResource.error;
      const ratingsError = store._ratingsResource.error;

      displayErrorEffect(dessertsError, toastService);
      displayErrorEffect(ratingsError, toastService);
    },
  })
);
```

## Обсуждение и взгляд в будущее

Благодаря `withProps`, появившейся в NgRx 19, Signal Store может объявлять произвольные свойства. Это упрощает работу с внедрёнными сервисами и позволяет настраивать ресурсы. Store может использовать такие ресурсы как деталь реализации, а может и отдавать их потребителю. Во втором случае ресурс стоит сделать доступным для внешнего мира только для чтения.

Использование Resource API прямо внутри Signal Store позволяет описать реактивный поток данных целиком на Signal API от Angular. А новый помощник `signalMethod` пригодится, чтобы связывать сигналы со Store.

Поскольку ресурс предоставляет локальную рабочую копию загруженных данных, её можно перезаписать пользовательскими правками, прежде чем отправить обратно на сервер. Напрямую обновлять эту рабочую копию ощущается немного странно, ведь обычно store не меняет writable-сигналы напрямую. Но, как мы уже обсуждали, временное обновление данных внутри реактивного графа — это часть идеи, заложенной в такие рабочие копии. Возможно, сообщество со временем придумает способ получше. Кроме того, может оказаться полезной фича `withResource` — для объявления приватного ресурса, выставляемого наружу только для чтения.
