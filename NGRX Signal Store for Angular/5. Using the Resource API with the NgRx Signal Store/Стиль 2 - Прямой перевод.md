В версии 19 NgRx представил фичу `withProps`, которая позволяет добавлять в Store произвольные свойства. С помощью этой фичи Store может, например, определять центральные сервисы, Observable-объекты или ресурсы (Resources).

Эта статья показывает, как использовать `withProps`, чтобы задействовать новый Resource API внутри Signal Store. Это позволяет Store предоставлять привычное реактивное поведение и предотвращать состояния гонки (race conditions) без RxJS. Для подключения шаблонной формы (template-driven form) к Store в примере используется `linkedSignal` вместе с новым помощником `signalMethod` из Signal Store.

## Реактивный поток

Чтобы продемонстрировать использование Resource API внутри Signal Store, я использую приложение с десертами, известное по другим постам блога:

![Пример приложения](https://www.angulararchitects.io/wp-content/uploads/2024/12/example-app.jpg)

Фильтры — Original Name и English Name — запускают реактивный поток, в результате чего отображаются десерты с рейтингами. Также нажатие на показанную кнопку заставляет store загружать рейтинги для отображаемых в данный момент десертов.

Прежде чем начать, я хотел бы визуализировать этот поток в виде простого реактивного графа:

![](https://www.angulararchitects.io/wp-content/uploads/2024/12/reactive-graph.png)

Как мы видим здесь, Store может получать намерение потребителя (например, компонента) через три метода, показанные слева. Сигнал `filter` запускает `dessertsResource`: каждый раз, когда он изменяется, ресурс загружает десерты.

`ratingsResource` запускается напрямую методом `loadRatings`, а метод `updateRatings` обновляет локальную рабочую копию (working copy) этого ресурса. Другой метод, не показанный здесь, мог бы записывать эту рабочую копию обратно в backend.

## Перспектива потребителя

Теперь давайте взглянем на внешнее представление Store, исследовав компонент, который его потребляет:

```typescript
@Component([...])
export class DessertsComponent {
  #store = inject(DessertStore);

  originalName = linkedSignal(() => this.#store.filter.originalName());
  englishName = linkedSignal(() => this.#store.filter.englishName());

  ratedDesserts = this.#store.ratedDesserts;
  loading = this.#store.loading;

  #linkedFilter = computed(() => ({
    originalName: this.originalName(),
    englishName: this.englishName()
  }));

  constructor() {
    this.#store.updateFilter(this.#linkedFilter)
  }

  loadRatings(): void {
    this.#store.loadRatings();
  }

  updateRating(id: number, rating: number): void {
    this.#store.updateRating(id, rating);
  }
}
```

Поскольку в примере фильтр, `originalName` и `englishName`, привязан к шаблонной форме, компонент оборачивает их в Linked Signals. Это позволяет использовать `ngModel` с двусторонним связыванием:

```html
<input [(ngModel)]="originalName" name="originalName" />
<input [(ngModel)]="englishName" name="englishName" />
```

В этом случае двустороннее связывание обновляет локальную рабочую копию, предоставляемую Linked Signal. Чтобы эта рабочая копия отправлялась обратно в Store, конструктор передаёт её в метод `updateFilter` Store.

Важно видеть, что `updateFilter` получает не просто текущий фильтр, а вычисляемый (computed) сигнал `linkedFilter`. Как мы увидим ниже, этот метод является так называемым Signal Method, который выполняется каждый раз, когда переданный сигнал изменяется. Это позволяет в примере держать фильтр в store актуальным.

Признаться, реактивная форма упростила бы синхронизацию формы со Store.

## Store

Если мы посмотрим на внутреннее представление Store, мы увидим, что состояние (state) представляет фильтр:

```typescript
export type Requested = undefined | true;

export const DessertStore = signalStore(
  { providedIn: 'root' },
  withState({
    filter: {
      originalName: '',
      englishName: 'Cake',
    },
    ratingsRequested: undefined as Requested
  }),
 [...],
);
```

Свойства для загруженных десертов и рейтингов здесь отсутствуют. Их предоставляют ресурсы, обсуждаемые ниже. Также состояние загрузки (loading state) может быть выведено из них. Поскольку у нас нет параметров для загрузки рейтингов, используется просто флаг `ratingsRequested`.

## Получение сервисов через withProps

Следующая фича, добавляемая в Store, — это `withProps`. Store использует её, чтобы внедрить сервисы, которые понадобятся нам позже для настройки ресурсов и методов:

```typescript
export const DessertStore = signalStore(
  [...],
  withProps(() => ({
    _dessertService: inject(DessertService),
    _ratingService: inject(RatingService),
    _toastService: inject(ToastService),
  })),
  [...],
);
```

Это немного упрощает реализацию Store. До появления `withProps` такие сервисы приходилось внедрять в параметры, добавляемые к отдельным фичам. Поскольку это приводило к длинным спискам параметров и должно было повторяться для разных фич, в которых использовался один и тот же сервис, использование `withProps` для этой задачи кажется более подходящим.

Обратите также внимание, что имена свойств, введённые здесь, начинаются с **подчёркивания.** По соглашению Signal Store рассматривает такие свойства как **приватные** и не предоставляет их потребителю.

## Настройка ресурсов

Для настройки ресурсов Store использует ещё один блок `withProps`:

```typescript
export const DessertStore = signalStore(
  [...],
  withProps((store) => ({
    _dessertsResource: resource({
      params: store.filter,
      loader: (loaderParams) => {
        const filter = loaderParams.params;
        const abortSignal = params.abortSignal;
        return store._dessertService.findPromise(filter, abortSignal);
      },
    }),
    _ratingsResource: resource({
      params: store.ratingsRequested
      loader: (loaderParams) => {
        const abortSignal = loaderParams.abortSignal;
        return store._ratingService.loadExpertRatingsPromise(abortSignal);
      }
    })
  })),
  [...],
);
```

Поскольку блок `withProps` с ресурсами определён после блока с сервисами, ресурсы могут обращаться к этим сервисам. `_dessertsResource` использует фильтр из состояния Store в качестве своих `params`. Следовательно, изменение фильтра запускает загрузку десертов.

## Опционально: предоставление ресурсов только для чтения

Как и внедрённые сервисы, ресурсы, обсуждавшиеся в предыдущем разделе, помечены как приватные. В этом примере они являются лишь деталью реализации: потребителю всё равно, загружаются ли данные через Resource API, через RxJS или другим способом.

Однако если бы мы захотели предоставить ресурсы потребителю, мы могли бы добавить ещё один блок `withProps`, выставляющий наружу их версии только для чтения (read-only):

```typescript
export const DessertStore = signalStore(
  [...],
  withProps((store) => ({
    dessertsResource: store._dessertsResource.asReadonly(),
    ratingsResource: store._ratingsResource.asReadonly(),
  })),
  [...],
);
```

Ресурс только для чтения предоставляет загруженные данные, состояние загрузки и информацию об ошибках, но не позволяет обновлять локальную рабочую копию. Эта идея определения приватных ресурсов, выставляемых наружу как публичные ресурсы только для чтения, могла бы быть автоматизирована пользовательской фичей вроде `withResource`.

## Вывод вью-моделей

Чтобы вывести вью-модели (view models), отображаемые через связывание данных (data binding), Store вводит несколько вычисляемых сигналов:

```typescript
export const DessertStore = signalStore(
  [...],
  withComputed((store) => ({
    ratedDesserts: computed(() => toRated(
      store._dessertsResource.value(),
      store._ratingsResource.value()
    )),
    loading: computed(() =>
      store._dessertsResource.isLoading()
        || store._dessertsResource.isLoading())
  })),
  [...]
);
```

Вспомогательная функция `toRated` (здесь не показана) объединяет десерты с загруженными рейтингами. Сигнал `loading` возвращает `true`, когда один из ресурсов находится в состоянии загрузки.

## Предоставление методов

Поскольку Resource API даёт нам реактивный, а значит, декларативный способ загрузки данных, для методов остаётся не так много императивного кода:

```typescript
export const DessertStore = signalStore(
  [...],
  withMethods((store) => ({
    updateFilter: signalMethod<DessertFilter>((filter) => {
      patchState(store, { filter });
    }),
    loadRatings: () => {
        patchState(store, { ratingsRequested: true });
        store._ratingsResource.reload();
    },
    updateRating: (id: number, rating: number) => {
      store._ratingsResource.update(ratings => ({
        ...ratings,
        [id]: rating,
      }));
    },
  })),
  [...],
);
```

Эти методы в основном записывают переданные значения в Store. Метод `updateFilter` настроен с помощью нового помощника `signalMethod`. Поскольку в примере используется `signalMethod<DessertFilter>`, метод принимает `DessertFilter` или `Signal<DessertFilter>`. В последнем случае определённый метод выполняется каждый раз, когда переданный сигнал изменяется.

Хотя переданный сигнал отслеживается, **реализация метода не отслеживается** (!) по соглашению. Следовательно, она не выполняется повторно, когда изменяется сигнал в одном из вызванных методов. Обычно такое поведение желательно, поскольку оно позволяет с первого взгляда увидеть, когда выполняется реализация метода.

Помощник `signalMethod` похож на `rxMethod`, уже представленный в первой версии Signal Store. Однако `signalMethod` не зависит от RxJS. Это также означает, что мы должны сами заботиться о предотвращении состояний гонки. В показанном здесь примере эту задачу берёт на себя Resource API.

Метод `loadRatings` запускает `_ratingsResource`, устанавливая флаг `ratingsRequested` в true и вызывая `reload`. Этот подход уже обсуждался [здесь](https://www.angulararchitects.io/en/blog/asynchronous-resources-with-angulars-new-resource-api/). Метод `updateRating` обновляет локальную рабочую копию ресурса. Признаю, это ощущается немного странно, поскольку store обычно не изменяет writable-сигналы напрямую. Однако это согласуется с идеей локальных рабочих копий, предоставляемых `linkedSignal` и Resource API: прямое обновление временных данных внутри реактивного потока. Возможно, мы найдём более рациональный способ выполнить эту задачу.

## Обработка ошибок с помощью хуков и эффектов

Если загрузка данных не срабатывает, ресурс информирует о текущей ситуации через свой сигнал `error`. Чтобы отобразить соответствующее toast-сообщение, Store настраивает хук `onInit` с эффектом (Effect), отслеживающим `error`:

```typescript
export const DessertStore = signalStore(
 [...],
  withHooks({
    onInit(store) {
      const toastService = store._toastService;
      const dessertsError = store._dessertsResource.error;

      effect(() => {
        const error = store._dessertsResource.error;
        if (error) {
          store._toastService.show('Error: ' + getMessage(error));
        }
      });
    }
  })
);
```

Метод `getMessage` возвращает сообщение об ошибке, выведенное из объекта `error`:

```typescript
export function getMessage(error: unknown) {
  if (error && typeof error === "object" && "message" in error) {
    return error.message;
  }
  return String(error);
}
```

Чтобы упростить повторение этой задачи для других сигналов `error`, мы могли бы вынести эффект во вспомогательную функцию:

```typescript
export const DessertStore = signalStore(
  [...],
  withHooks({
    onInit(store) {
      const toastService = store._toastService;
      const dessertsError = store._dessertsResource.error;
      const ratingsError = store._ratingsResource.error;

      displayErrorEffect(dessertsError, toastService);
      displayErrorEffect(ratingsError, toastService);
    }
  })
);
```

Эта вспомогательная функция принимает `errorSignal` и `toastService` и настраивает показанный ранее эффект:

```typescript
export function displayErrorEffect(
  errorSignal: Signal<unknown>,
  toastService: ToastService
) {
  effect(() => {
    const error = errorSignal();
    if (error) {
      toastService.show("Error: " + getMessage(error));
    }
  });
}
```

## Всё вместе

Теперь, когда мы обсудили все секции Signal Store, использующего Resource API, давайте взглянем на полную реализацию store:

```typescript
export const DessertStore = signalStore(
  { providedIn: "root" },
  withState({
    filter: {
      originalName: "",
      englishName: "Cake",
    },
  }),
  withProps(() => ({
    _dessertService: inject(DessertService),
    _ratingService: inject(RatingService),
    _toastService: inject(ToastService),
  })),
  withProps((store) => ({
    _dessertsResource: resource({
      request: Store.filter,
      loader: (params) => {
        const filter = params.request;
        const abortSignal = params.abortSignal;
        return store._dessertService.findPromise(filter, abortSignal);
      },
    }),
    _ratingsResource: resource({
      loader: (params) => {
        const abortSignal = params.abortSignal;
        return store._ratingService.loadExpertRatingsPromise(abortSignal);
      },
    }),
  })),
  withProps((store) => ({
    dessertsResource: store._dessertsResource.asReadonly(),
    ratingsResource: store._ratingsResource.asReadonly(),
  })),
  withComputed((store) => ({
    ratedDesserts: computed(() =>
      toRated(store._dessertsResource.value(), store._ratingsResource.value())
    ),
    loading: computed(
      () =>
        store._dessertsResource.isLoading() ||
        store._dessertsResource.isLoading()
    ),
  })),
  withMethods((store) => ({
    updateFilter: signalMethod<DessertFilter>((filter) => {
      patchState(store, { filter });
    }),
    loadRatings: () => {
      store._ratingsResource.reload();
    },
    updateRating: (id: number, rating: number) => {
      store._ratingsResource.update((ratings) => ({
        ...ratings,
        [id]: rating,
      }));
    },
  })),
  withHooks({
    onInit(store) {
      const toastService = store._toastService;
      const dessertsError = store._dessertsResource.error;
      const ratingsError = store._ratingsResource.error;

      displayErrorEffect(dessertsError, toastService);
      displayErrorEffect(ratingsError, toastService);
    },
  })
);
```

## Обсуждение и перспективы

Благодаря `withProps`, представленной в NgRx 19, Signal Store может определять произвольные свойства. Это упрощает работу с внедрёнными сервисами и позволяет настраивать ресурсы. Store может использовать такие ресурсы как детали реализации или предоставлять их потребителю. В последнем случае ресурс следует сделать доступным только для чтения для внешнего мира.

Прямое использование Resource API внутри Signal Store позволяет нам определить реактивный поток данных полностью с использованием Signal API от Angular. Новый помощник `signalMethod` может использоваться для подключения сигналов к Store.

Поскольку ресурс предоставляет локальную рабочую копию загруженных данных, её можно перезаписать обновлениями, предоставленными пользователем, перед отправкой обратно на сервер. Прямое обновление этой рабочей копии ощущается немного странно, поскольку обычно Store не изменяет writable-сигналы напрямую. Однако, как обсуждалось выше, временное обновление данных внутри реактивного графа является частью идеи, стоящей за такими рабочими копиями. Возможно, сообщество в итоге найдёт способ получше для выполнения этой задачи. Также может быть некоторая ценность в фиче `withResource` для определения приватного ресурса, выставляемого наружу только для чтения.
