В версии 19 NgRx появилась фича `withProps`, которая позволяет добавлять в Store произвольные свойства. С её помощью Store может, например, определять общие сервисы, Observable-объекты или ресурсы (Resources).

В этой статье показано, как с помощью `withProps` задействовать новый Resource API внутри Signal Store. Такой подход обеспечивает Store привычное реактивное поведение и предотвращает состояния гонки (race conditions) без RxJS. Для подключения шаблонной формы (template-driven form) к Store в примере используется `linkedSignal` совместно с новым помощником `signalMethod` из Signal Store.

## Реактивный поток

Чтобы продемонстрировать использование Resource API внутри Signal Store, я задействую приложение с десертами, знакомое по другим публикациям блога:

![Пример приложения](https://www.angulararchitects.io/wp-content/uploads/2024/12/example-app.jpg)

Фильтры — Original Name и English Name — запускают реактивный поток, результатом которого становится отображение десертов с рейтингами. Кроме того, нажатие показанной кнопки заставляет store загрузить рейтинги для отображаемых в данный момент десертов.

Прежде чем начать, изобразим этот поток в виде простого реактивного графа (reactive graph):

![](https://www.angulararchitects.io/wp-content/uploads/2024/12/reactive-graph.png)

Как видно, Store получает намерение потребителя (например, компонента) через три метода, показанные слева. Сигнал (signal) `filter` запускает `dessertsResource`: при каждом его изменении ресурс загружает десерты.

`ratingsResource` запускается напрямую методом `loadRatings`, а метод `updateRatings` обновляет локальную рабочую копию (working copy) этого ресурса. Другой метод, не показанный на схеме, мог бы записывать эту рабочую копию обратно в backend.

## Перспектива потребителя

Теперь рассмотрим внешнее представление (external view) Store, изучив потребляющий его компонент:

```typescript
@Component([...])
export class DessertsComponent {
  #store = inject(DessertStore);

  originalName = linkedSignal(() => this.#store.filter.originalName());
  englishName = linkedSignal(() => this.#store.filter.englishName());

  ratedDesserts = this.#store.ratedDesserts;
  loading = this.#store.loading;

  #linkedFilter = computed(() => ({
    originalName: this.originalName(),
    englishName: this.englishName()
  }));

  constructor() {
    this.#store.updateFilter(this.#linkedFilter)
  }

  loadRatings(): void {
    this.#store.loadRatings();
  }

  updateRating(id: number, rating: number): void {
    this.#store.updateRating(id, rating);
  }
}
```

Так как в примере поля фильтра (`originalName` и `englishName`) привязаны к шаблонной форме, компонент оборачивает их в Linked Signals (сигналы, производные от других сигналов, но допускающие локальную перезапись). Это даёт возможность использовать `ngModel` с двусторонним связыванием (two-way binding):

```html
<input [(ngModel)]="originalName" name="originalName" />
<input [(ngModel)]="englishName" name="englishName" />
```

В данном случае двустороннее связывание обновляет локальную рабочую копию, предоставляемую Linked Signal. Чтобы эта рабочая копия отправлялась обратно в Store, конструктор передаёт её в метод `updateFilter` Store.

Принципиально важно: `updateFilter` получает не текущее значение фильтра, а вычисляемый (computed) сигнал `linkedFilter`. Как будет показано ниже, этот метод представляет собой так называемый Signal Method, выполняющийся при каждом изменении переданного сигнала. Благодаря этому фильтр в store поддерживается в актуальном состоянии.

Признаю, реактивная форма (reactive form) упростила бы синхронизацию формы со Store.

## Store

Если обратиться к внутреннему представлению (internal view) Store, видно, что его состояние (state) представляет фильтр:

```typescript
export type Requested = undefined | true;

export const DessertStore = signalStore(
  { providedIn: 'root' },
  withState({
    filter: {
      originalName: '',
      englishName: 'Cake',
    },
    ratingsRequested: undefined as Requested
  }),
 [...],
);
```

Свойства для загруженных десертов и рейтингов здесь отсутствуют — их предоставляют ресурсы, рассматриваемые далее. Состояние загрузки (loading state) также выводится из них. Поскольку параметров для загрузки рейтингов нет, используется лишь флаг `ratingsRequested`.

## Получение сервисов через withProps

Следующая фича, добавляемая в Store, — `withProps`. Store использует её для внедрения (inject) сервисов, которые понадобятся далее при настройке ресурсов и методов:

```typescript
export const DessertStore = signalStore(
  [...],
  withProps(() => ({
    _dessertService: inject(DessertService),
    _ratingService: inject(RatingService),
    _toastService: inject(ToastService),
  })),
  [...],
);
```

Это несколько упрощает реализацию Store. До появления `withProps` такие сервисы приходилось внедрять через параметры, добавляемые к отдельным фичам. Поскольку это вело к длинным спискам параметров и требовало повторения для разных фич, использующих один и тот же сервис, применение `withProps` для этой задачи представляется более уместным.

Обратите внимание: имена введённых здесь свойств начинаются с **подчёркивания.** По соглашению (convention) Signal Store трактует такие свойства как **приватные** и не выставляет их потребителю.

## Настройка ресурсов

Для настройки ресурсов Store использует ещё один блок `withProps`:

```typescript
export const DessertStore = signalStore(
  [...],
  withProps((store) => ({
    _dessertsResource: resource({
      params: store.filter,
      loader: (loaderParams) => {
        const filter = loaderParams.params;
        const abortSignal = params.abortSignal;
        return store._dessertService.findPromise(filter, abortSignal);
      },
    }),
    _ratingsResource: resource({
      params: store.ratingsRequested
      loader: (loaderParams) => {
        const abortSignal = loaderParams.abortSignal;
        return store._ratingService.loadExpertRatingsPromise(abortSignal);
      }
    })
  })),
  [...],
);
```

Так как блок `withProps` с ресурсами определён после блока с сервисами, ресурсы имеют доступ к этим сервисам. `_dessertsResource` использует фильтр из состояния Store в качестве `params`. Следовательно, изменение фильтра запускает загрузку десертов (поскольку `params` реактивны, при их изменении `loader` перезапускается).

## Опционально: выставление наружу ресурсов только для чтения

Как и внедрённые сервисы, ресурсы из предыдущего раздела помечены как приватные. В этом примере они являются лишь деталью реализации: потребителю безразлично, загружаются ли данные через Resource API, через RxJS или иным способом.

Однако если требуется предоставить ресурсы потребителю, можно добавить ещё один блок `withProps`, выставляющий наружу их версии только для чтения (read-only):

```typescript
export const DessertStore = signalStore(
  [...],
  withProps((store) => ({
    dessertsResource: store._dessertsResource.asReadonly(),
    ratingsResource: store._ratingsResource.asReadonly(),
  })),
  [...],
);
```

Ресурс только для чтения предоставляет загруженные данные, состояние загрузки и информацию об ошибках, но не допускает обновления локальной рабочей копии. Эту идею — определять приватные ресурсы и выставлять их наружу как публичные read-only — можно автоматизировать пользовательской фичей наподобие `withResource`.

## Вывод вью-моделей

Чтобы получить вью-модели (view models), отображаемые через связывание данных (data binding), Store вводит ряд вычисляемых сигналов:

```typescript
export const DessertStore = signalStore(
  [...],
  withComputed((store) => ({
    ratedDesserts: computed(() => toRated(
      store._dessertsResource.value(),
      store._ratingsResource.value()
    )),
    loading: computed(() =>
      store._dessertsResource.isLoading()
        || store._dessertsResource.isLoading())
  })),
  [...]
);
```

Вспомогательная функция `toRated` (здесь не приведена) объединяет десерты с загруженными рейтингами. Сигнал `loading` возвращает `true`, когда один из ресурсов находится в состоянии загрузки.

## Предоставление методов

Поскольку Resource API даёт реактивный, а значит, декларативный механизм загрузки данных, императивного кода в методах остаётся немного:

```typescript
export const DessertStore = signalStore(
  [...],
  withMethods((store) => ({
    updateFilter: signalMethod<DessertFilter>((filter) => {
      patchState(store, { filter });
    }),
    loadRatings: () => {
        patchState(store, { ratingsRequested: true });
        store._ratingsResource.reload();
    },
    updateRating: (id: number, rating: number) => {
      store._ratingsResource.update(ratings => ({
        ...ratings,
        [id]: rating,
      }));
    },
  })),
  [...],
);
```

Эти методы преимущественно записывают переданные значения в Store. Метод `updateFilter` построен на новом помощнике `signalMethod`. Так как в примере используется `signalMethod<DessertFilter>`, метод принимает `DessertFilter` либо `Signal<DessertFilter>`. Во втором случае определённый метод выполняется при каждом изменении переданного сигнала.

При этом переданный сигнал отслеживается (tracked), а **реализация метода — нет (untracked)** (!) по соглашению. Следовательно, она не перевыполняется при изменении какого-либо сигнала внутри вызванных методов (то есть зависимости от сигналов, прочитанных в теле, не образуются). Обычно такое поведение желательно, поскольку позволяет с первого взгляда определить, когда именно отрабатывает реализация метода.

Помощник `signalMethod` схож с `rxMethod`, представленным ещё в первой версии Signal Store. Однако `signalMethod` не зависит от RxJS. Это также означает, что о предотвращении состояний гонки нужно заботиться самостоятельно. В приведённом примере эту задачу берёт на себя Resource API.

Метод `loadRatings` запускает `_ratingsResource`, устанавливая флаг `ratingsRequested` в `true` и вызывая `reload`. Этот подход уже рассматривался [здесь](https://www.angulararchitects.io/en/blog/asynchronous-resources-with-angulars-new-resource-api/). Метод `updateRating` обновляет локальную рабочую копию ресурса. Признаю, это выглядит несколько необычно, поскольку store обычно не модифицирует writable-сигналы напрямую. Тем не менее это согласуется с идеей локальных рабочих копий, предоставляемых `linkedSignal` и Resource API: прямое обновление временных данных внутри реактивного потока. Возможно, со временем найдётся более рациональный способ решения этой задачи.

## Обработка ошибок с помощью хуков и эффектов

Если загрузка данных завершается неудачей, ресурс сообщает о текущей ситуации через сигнал `error`. Чтобы отобразить соответствующее toast-уведомление, Store настраивает хук `onInit` с эффектом (Effect), отслеживающим `error`:

```typescript
export const DessertStore = signalStore(
 [...],
  withHooks({
    onInit(store) {
      const toastService = store._toastService;
      const dessertsError = store._dessertsResource.error;

      effect(() => {
        const error = store._dessertsResource.error;
        if (error) {
          store._toastService.show('Error: ' + getMessage(error));
        }
      });
    }
  })
);
```

Метод `getMessage` возвращает сообщение об ошибке, выведенное из объекта `error`:

```typescript
export function getMessage(error: unknown) {
  if (error && typeof error === "object" && "message" in error) {
    return error.message;
  }
  return String(error);
}
```

Чтобы упростить повторение этой операции для других сигналов `error`, эффект можно вынести во вспомогательную функцию:

```typescript
export const DessertStore = signalStore(
  [...],
  withHooks({
    onInit(store) {
      const toastService = store._toastService;
      const dessertsError = store._dessertsResource.error;
      const ratingsError = store._ratingsResource.error;

      displayErrorEffect(dessertsError, toastService);
      displayErrorEffect(ratingsError, toastService);
    }
  })
);
```

Эта вспомогательная функция принимает `errorSignal` и `toastService` и настраивает показанный ранее эффект:

```typescript
export function displayErrorEffect(
  errorSignal: Signal<unknown>,
  toastService: ToastService
) {
  effect(() => {
    const error = errorSignal();
    if (error) {
      toastService.show("Error: " + getMessage(error));
    }
  });
}
```

## Всё вместе

Теперь, когда мы рассмотрели все секции Signal Store, использующего Resource API, приведём полную реализацию store:

```typescript
export const DessertStore = signalStore(
  { providedIn: "root" },
  withState({
    filter: {
      originalName: "",
      englishName: "Cake",
    },
  }),
  withProps(() => ({
    _dessertService: inject(DessertService),
    _ratingService: inject(RatingService),
    _toastService: inject(ToastService),
  })),
  withProps((store) => ({
    _dessertsResource: resource({
      request: Store.filter,
      loader: (params) => {
        const filter = params.request;
        const abortSignal = params.abortSignal;
        return store._dessertService.findPromise(filter, abortSignal);
      },
    }),
    _ratingsResource: resource({
      loader: (params) => {
        const abortSignal = params.abortSignal;
        return store._ratingService.loadExpertRatingsPromise(abortSignal);
      },
    }),
  })),
  withProps((store) => ({
    dessertsResource: store._dessertsResource.asReadonly(),
    ratingsResource: store._ratingsResource.asReadonly(),
  })),
  withComputed((store) => ({
    ratedDesserts: computed(() =>
      toRated(store._dessertsResource.value(), store._ratingsResource.value())
    ),
    loading: computed(
      () =>
        store._dessertsResource.isLoading() ||
        store._dessertsResource.isLoading()
    ),
  })),
  withMethods((store) => ({
    updateFilter: signalMethod<DessertFilter>((filter) => {
      patchState(store, { filter });
    }),
    loadRatings: () => {
      store._ratingsResource.reload();
    },
    updateRating: (id: number, rating: number) => {
      store._ratingsResource.update((ratings) => ({
        ...ratings,
        [id]: rating,
      }));
    },
  })),
  withHooks({
    onInit(store) {
      const toastService = store._toastService;
      const dessertsError = store._dessertsResource.error;
      const ratingsError = store._ratingsResource.error;

      displayErrorEffect(dessertsError, toastService);
      displayErrorEffect(ratingsError, toastService);
    },
  })
);
```

## Обсуждение и перспективы

Благодаря `withProps`, представленной в NgRx 19, Signal Store может определять произвольные свойства. Это упрощает работу с внедрёнными сервисами и позволяет настраивать ресурсы. Store может использовать такие ресурсы как детали реализации либо предоставлять их потребителю. В последнем случае ресурс следует сделать доступным наружу только для чтения.

Прямое использование Resource API внутри Signal Store позволяет описать реактивный поток данных целиком средствами Signal API от Angular. Новый помощник `signalMethod` применяется для подключения сигналов к Store.

Поскольку ресурс предоставляет локальную рабочую копию загруженных данных, её можно перезаписать пользовательскими обновлениями до отправки обратно на сервер. Прямое обновление этой рабочей копии выглядит несколько необычно, поскольку обычно Store не модифицирует writable-сигналы напрямую. Однако, как обсуждалось выше, временное обновление данных внутри реактивного графа является частью замысла, заложенного в такие рабочие копии. Возможно, сообщество в итоге найдёт более удачный способ решения этой задачи. Также может оказаться полезной фича `withResource` для определения приватного ресурса, выставляемого наружу только для чтения.
