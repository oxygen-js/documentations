## Цель

Цель этой статьи — показать, как реализовать пользовательские фичи для Signal Store, которые позволяют делать следующее:

* Искать сущности (entities)
* Выбирать несколько сущностей
* Отображать выбранные сущности
* Undo/Redo (отмена/повтор)

А вот как выглядит демо-приложение, которое я построил поверх этих пользовательских фич:

![Демо-приложение](https://www.angulararchitects.io/wp-content/uploads/2023/11/result.png)

И вот весь код, который нужен, чтобы настроить store, включая Undo/Redo и подключение к Data Service, который загружает сущности с бэкенда:

```typescript
export const FlightBookingStore = signalStore(
  { providedIn: 'root' },
  withEntities<Flight>(),
  withCallState(),
  withDataService(FlightService, { from: 'Graz', to: 'Hamburg'} ),
  withUndoRedo(),
);
```

Как видите, для управления сущностями я использую пакет `@ngrx/signals/entities`. Помимо этого, я вынес оставшуюся логику в три переиспользуемые пользовательские фичи: про `withCallState` я уже рассказывал в предыдущей статье. В этой статье речь пойдёт о `withDataService`, а также будет приведён код для `withUndoRedo`.

## Пользовательская фича DataService

Идея фичи `DataService` в том, чтобы завести состояние (state) для фильтра поиска и подключить Angular-сервис, который использует этот фильтр для поиска сущностей. На более продвинутом этапе развития фича могла бы также вызывать `DataService` для сохранения и удаления сущностей. Однако, поскольку такая реализация не дала бы здесь дополнительных полезных идей, я решил её пропустить ради краткости.

Чтобы сделать фичу `DataService` универсальной (generic), нам понадобятся несколько общих типов, описывающих всё, с чем эта фича взаимодействует:

```typescript
import { EntityId } from "@ngrx/signals/entities";
[...]

export type Filter = Record<string, unknown>;
export type Entity = { id: EntityId };

export interface DataService<E extends Entity, F extends Filter> {
    load(filter: F): Promise<E[]>;
}
```

Эти типы описывают, как устроен наш фильтр поиска, что мы понимаем под сущностью и как должен выглядеть `DataService`. Тип `EntityId` приходит из `@ngrx/signals/entities` и принимает `string` или `number`.

Допущение о том, что сущность — это произвольный объект со свойством `id`, — одно из соглашений `@ngrx/signals/entities`, призванных сократить ваш код. Если ваш первичный ключ называется иначе, вы можете сообщить об этом `@ngrx/signals/entities`. Но чтобы пример оставался компактным, я решил придерживаться этого соглашения.

## Реализуем универсальную пользовательскую фичу

Функция `withDataService` возвращает фичу `DataService`:

```typescript
export function withDataService<E extends Entity, F extends Filter, S extends DataService<E, F>>(dataServiceType: Type<S>, filter: F) {
    [...]
}
```

Её параметры типа описывают сущность, которой нужно управлять, соответствующий фильтр поиска и сам `DataService`. При вызове этого универсального метода нам достаточно передать `DataService` и начальный фильтр. Остальное TypeScript выводит сам:

```typescript
withDataService(FlightService, { from: 'Graz', to: 'Hamburg'} ),
```

Функция `withDataService` вызывает `signalStoreFeature`, чтобы настроить нашу пользовательскую фичу:

```typescript
export function withDataService<E extends Entity, F extends Filter, S extends DataService<E, F>>(dataServiceType: Type<S>, filter: F) {
    return signalStoreFeature(
        // Наши ожидания к store:
        {
            state: type<{
                callState: CallState,
                entityMap: Record<EntityId, E>,
                ids: EntityId[]
            }>(),
            signals: type<{
                entities: Signal<Entity[]>
            }>(),
            methods: type<{}>()
        },

        // Композиция нескольких фич:
        withState( [...] ),
        withComputed( [...] ),
        withMethods( [...] )
    );
}
```

Как уже показывалось в первой статье серии, функция `signalStoreFeature` по сути собирает уже существующие фичи в одну новую. Например, с помощью `withState` можно добавить новые свойства состояния, с помощью `withComputed` — вычисляемые сигналы (computed Signals), а с помощью `withMethods` — методы.

Однако в этот раз есть одна небольшая особенность: у нашей фичи есть некоторые **ожидания** к тому Signal Store, с которым она будет использоваться. Она ожидает, что на месте уже будут фичи `callState` и `entity`. Первая заводит нужное нам свойство `callState`; вторая — свойства `entityMap` и `ids`, а также вычисляемый сигнал `entities`.

Эти ожидания задаются первым параметром, который передаётся в `signalStoreFeature`. Он описывает ожидаемые свойства состояния (`state`), вычисляемые сигналы (`signals`) и методы. Поскольку методов мы не ожидаем, ключ `methods` можно вообще опустить вместо того, чтобы указывать `type<{}>()`.

Чтобы избежать конфликтов имён, фича `entity` позволяет использовать разные имена свойств. Для простоты я придерживаюсь имён по умолчанию. Но в одной из следующих статей серии вы узнаете, как типобезопасно работать с динамическими именами свойств.

Оставшаяся часть этой пользовательской фичи — это всего лишь добавление свойств состояния, вычисляемых сигналов и методов поверх ожидаемых фич:

```typescript
export function withDataService<E extends Entity, F extends Filter, S extends DataService<E, F>>(dataServiceType: Type<S>, filter: F) {
    return signalStoreFeature(
        // Первый параметр содержит
        // наши ожидания к store:
        // Если они не выполнены, TypeScript
        // не даст добавить эту фичу!
        {
            state: type<{
                callState: CallState,
                entityMap: Record<EntityId, E>,
                ids: EntityId[]
            }>(),
            signals: type<{
                entities: Signal<Entity[]>
            }>(),
            methods: type<{}>()
        },
        withState({
            filter,
            selectedIds: {} as Record<EntityId, boolean>,
        }),
        withComputed(({ selectedIds, entities }) => ({
            selectedEntities: computed(() => entities().filter(e => selectedIds()[e.id]))
        })),
        withMethods((store) => {
            const dataService = inject(dataServiceType)
            return {
                updateFilter(filter: F): void {
                    patchState(store, { filter });
                },
                updateSelected(id: EntityId, selected: boolean): void {
                    patchState(store, ({ selectedIds }) => ({
                        selectedIds: {
                            ...selectedIds,
                            [id]: selected,
                        }
                    }));
                },
                async load(): Promise<void> {
                    patchState(store, setLoading());
                    const result = await dataService.load(store.filter());
                    patchState(store, setAllEntities(result));
                    patchState(store, setLoaded());
                }
            };
        })
    );
}
```

## Фича Undo/Redo

Фича Undo/Redo реализована очень похожим образом. Внутри она управляет двумя стеками: стеком отмены (undo) и стеком повтора (redo). По сути это массивы из элементов `StackItem`:

```typescript
export type StackItem = {
    filter: Filter;
    entityMap: Record<EntityId, Entity>,
    ids: EntityId[]
};
```

Каждый `StackItem` представляет собой снимок (snapshot) текущего фильтра поиска и информации, которую использует фича entity (`entityMap`, `ids`).

Для конфигурации фичи используется тип `UndoRedoOptions`:

```typescript
export type UndoRedoOptions = {
    maxStackSize: number;
}

export const defaultUndoRedoOptions: UndoRedoOptions = {
    maxStackSize: 100
}
```

Объект опций позволяет ограничить размер стека. Если стек разрастается слишком сильно, старые элементы удаляются по правилу «первым пришёл — первым ушёл» (First In, First Out).

Фичу добавляет функция `withUndoRedo`. Устроена она так:

```typescript
export function withUndoRedo(options = defaultUndoRedoOptions) {

    let previous: StackItem | null = null;
    let skipOnce = false;

    const undoStack: StackItem[] = [];
    const redoStack: StackItem[] = [];

    [...]

    return signalStoreFeature(
        // Ожидания к store:
        {
            state: type<{
                filter: Filter,
                entityMap: Record<EntityId, Entity>,
                ids: EntityId[]
            }>(),
        },
        [...]
        withMethods((store) => ({
            undo(): void { [...] },
            redo(): void { [...] }
        })),
        withHooks({
            onInit(store) {
                effect(() => {
                    const filter = store.filter();
                    const entityMap = store.entityMap();
                    const ids = store.ids();

                    [...]
                });
            }
        })

    )
}
```

Подобно рассмотренной выше функции `withDataService`, она вызывает `signalStoreFeature` и задаёт свои ожидания к store через первый аргумент. Она вводит методы `undo` и `redo`, которые восстанавливают состояние из соответствующих стеков. Чтобы наблюдать за состоянием, хук `onInit` в конце создаёт эффект (effect). После каждого изменения этот эффект сохраняет исходное состояние в стеке отмены.

Одна вещь в этой реализации Undo/Redo немного особенная: сама фича хранит часть внутреннего состояния — например, `undoStack` и `redoStack` — которое не является частью Signal Store.

Полную реализацию этой фичи смотрите в моём 📂 [GitHub-репозитории](https://github.com/manfredsteyer/standalone-example-cli/tree/arc-signal-store-custom-examples) (🔀 Ветка: arc-signal-store-custom-examples). Если хотите увидеть другую реализацию, которая хранит внутреннее состояние фичи прямо в Signal Store, загляните в ветку 🔀 `arc-signal-custom-examples-undoredo-alternative`.

## Использование store в компоненте

Чтобы использовать наш signal store «в 7 строках кода» в компоненте, просто внедрите его и делегируйте обращения к его сигналам и методам:

```typescript
@Component( [...] )
export class FlightSearchComponent {
  private store = inject(FlightBookingStore);

  // Делегируем сигналам
  from = this.store.filter.from;
  to = this.store.filter.to;
  flights = this.store.entities;
  selected = this.store.selectedEntities;
  selectedIds = this.store.selectedIds;

  // Делегируем методам
  async search() {
    this.store.load();
  }

  undo(): void {
    this.store.undo();
  }

  redo(): void {
    this.store.redo();
  }

  updateCriteria(from: string, to: string): void {
    this.store.updateFilter({ from, to });
  }

  updateBasket(id: number, selected: boolean): void {
    this.store.updateSelected(id, selected);
  }

}
```

## Заключение и что дальше

Реализация повторяющихся задач через универсальные пользовательские фичи позволяет драматически сократить исходный код. В этой статье мы реализовали Signal Store для простого сценария всего в 7 строках кода. И хотя реализация таких фич в универсальном виде поначалу добавляет немного накладных расходов, эти усилия точно окупятся, как только у вас наберётся несколько сценариев, устроенных подобным образом.

Чтобы переиспользовать существующие пользовательские фичи, наша фича делегирует им работу. API, предоставляемый NGRX Signal Store, позволяет пользовательской фиче убедиться, что другие фичи уже настроены. Она объявляет, какие свойства состояния, вычисляемые сигналы и методы ожидает. Если их нет, TypeScript выдаёт ошибку компиляции.

Ради простоты мы ограничились именами свойств по умолчанию, которые вводят оркестрируемые фичи. Однако, чтобы избежать конфликтов имён, эти имена тоже можно настраивать. Например, фича `entity`, поставляемая вместе с Signal Store, поддерживает такие динамические свойства без ущерба для типобезопасности. В следующей статье я покажу, как применить эту же идею и к нашим пользовательским фичам.
