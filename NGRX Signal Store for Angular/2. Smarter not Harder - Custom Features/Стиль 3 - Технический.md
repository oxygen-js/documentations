## Цель

Цель этой статьи — показать, как реализовать пользовательские фичи для Signal Store, обеспечивающие следующее:

* Поиск сущностей (entities)
* Выбор нескольких сущностей
* Отображение выбранных сущностей
* Undo/Redo (отмена/повтор операций)

Так выглядит демо-приложение, которое я построил поверх этих пользовательских фич:

![Демо-приложение](https://www.angulararchitects.io/wp-content/uploads/2023/11/result.png)

И вот весь код, необходимый для настройки store, включая Undo/Redo и подключение его к Data Service, который загружает сущности с бэкенда:

```typescript
export const FlightBookingStore = signalStore(
  { providedIn: 'root' },
  withEntities<Flight>(),
  withCallState(),
  withDataService(FlightService, { from: 'Graz', to: 'Hamburg'} ),
  withUndoRedo(),
);
```

Как видно, для управления сущностями я использую пакет `@ngrx/signals/entities`. Помимо этого, я вынес оставшуюся логику в три переиспользуемые пользовательские фичи: `withCallState` уже разбиралась в предыдущей статье. Данная статья разбирает `withDataService` и приводит код для `withUndoRedo`.

## Пользовательская фича DataService

Идея фичи `DataService` состоит в том, чтобы завести состояние (state) для фильтра поиска и подключить Angular-сервис, который использует этот фильтр для поиска сущностей. На дальнейшем этапе разработки фича могла бы также вызывать `DataService` для сохранения и удаления сущностей. Однако, поскольку эти реализации не добавили бы здесь дополнительного понимания, я решил опустить их ради краткости.

Чтобы сделать фичу `DataService` обобщённой (generic), необходимо несколько общих типов, описывающих всё, с чем фича взаимодействует:

```typescript
import { EntityId } from "@ngrx/signals/entities";
[...]

export type Filter = Record<string, unknown>;
export type Entity = { id: EntityId };

export interface DataService<E extends Entity, F extends Filter> {
    load(filter: F): Promise<E[]>;
}
```

Эти типы описывают структуру нашего фильтра поиска, то, что подразумевается под сущностью, и то, как должен выглядеть `DataService`. Тип `EntityId` импортируется из `@ngrx/signals/entities` и принимает `string` либо `number`.

Допущение о том, что сущность — это произвольный объект со свойством `id`, является одним из соглашений `@ngrx/signals/entities`, предназначенных для сокращения вашего кода. Если ваш первичный ключ называется иначе, об этом можно сообщить `@ngrx/signals/entities` соответствующим образом (через конфигурацию имени идентификатора). Однако, чтобы пример оставался компактным, я решил придерживаться этого соглашения.

## Реализация обобщённой пользовательской фичи

Функция `withDataService` возвращает фичу `DataService`:

```typescript
export function withDataService<E extends Entity, F extends Filter, S extends DataService<E, F>>(dataServiceType: Type<S>, filter: F) {
    [...]
}
```

Её параметры типа описывают управляемую сущность, соответствующий фильтр поиска и `DataService`. При вызове этого обобщённого метода достаточно передать `DataService` и начальный фильтр — остальное TypeScript выводит автоматически (через вывод типов, type inference):

```typescript
withDataService(FlightService, { from: 'Graz', to: 'Hamburg'} ),
```

Функция `withDataService` вызывает `signalStoreFeature` для настройки нашей пользовательской фичи:

```typescript
export function withDataService<E extends Entity, F extends Filter, S extends DataService<E, F>>(dataServiceType: Type<S>, filter: F) {
    return signalStoreFeature(
        // Наши ожидания к store:
        {
            state: type<{
                callState: CallState,
                entityMap: Record<EntityId, E>,
                ids: EntityId[]
            }>(),
            signals: type<{
                entities: Signal<Entity[]>
            }>(),
            methods: type<{}>()
        },

        // Композиция нескольких фич:
        withState( [...] ),
        withComputed( [...] ),
        withMethods( [...] )
    );
}
```

Как было показано в первой статье серии, функция `signalStoreFeature` по сути компонует существующие фичи в новую. Например, мы можем ввести новые свойства состояния через `withState`, вычисляемые сигналы (computed Signals) через `withComputed` или методы через `withMethods`.

Однако в этот раз есть одна небольшая особенность: у нашей фичи есть **ожидания** (expectations) к тому Signal Store, с которым она используется. Она ожидает наличия фич `callState` и `entity`. Первая заводит нужное нам свойство `callState`; вторая заводит свойства `entityMap` и `ids`, а также вычисляемый сигнал `entities`.

Эти ожидания определяются первым параметром, передаваемым в `signalStoreFeature`. Он описывает ожидаемые свойства состояния (`state`), вычисляемые сигналы (`signals`) и методы. Поскольку методов мы не ожидаем, ключ `methods` можно опустить вместо указания `type<{}>()`.

Чтобы избежать конфликтов имён, фича `entity` допускает использование различных имён свойств. Для простоты здесь я придерживаюсь имён по умолчанию. Однако в одной из следующих статей серии вы узнаете, как типобезопасно работать с динамическими именами свойств.

Оставшаяся часть этой пользовательской фичи — это лишь добавление свойств состояния, вычисляемых сигналов и методов поверх ожидаемых фич:

```typescript
export function withDataService<E extends Entity, F extends Filter, S extends DataService<E, F>>(dataServiceType: Type<S>, filter: F) {
    return signalStoreFeature(
        // Первый параметр содержит
        // наши ожидания к store:
        // Если они не выполнены, TypeScript
        // не позволит добавить эту фичу!
        {
            state: type<{
                callState: CallState,
                entityMap: Record<EntityId, E>,
                ids: EntityId[]
            }>(),
            signals: type<{
                entities: Signal<Entity[]>
            }>(),
            methods: type<{}>()
        },
        withState({
            filter,
            selectedIds: {} as Record<EntityId, boolean>,
        }),
        withComputed(({ selectedIds, entities }) => ({
            selectedEntities: computed(() => entities().filter(e => selectedIds()[e.id]))
        })),
        withMethods((store) => {
            const dataService = inject(dataServiceType)
            return {
                updateFilter(filter: F): void {
                    patchState(store, { filter });
                },
                updateSelected(id: EntityId, selected: boolean): void {
                    patchState(store, ({ selectedIds }) => ({
                        selectedIds: {
                            ...selectedIds,
                            [id]: selected,
                        }
                    }));
                },
                async load(): Promise<void> {
                    patchState(store, setLoading());
                    const result = await dataService.load(store.filter());
                    patchState(store, setAllEntities(result));
                    patchState(store, setLoaded());
                }
            };
        })
    );
}
```

## Фича Undo/Redo

Фича Undo/Redo реализована весьма схожим образом. Внутренне она управляет двумя стеками: стеком отмены (undo stack) и стеком повтора (redo stack). По сути это массивы элементов `StackItem`:

```typescript
export type StackItem = {
    filter: Filter;
    entityMap: Record<EntityId, Entity>,
    ids: EntityId[]
};
```

Каждый `StackItem` представляет собой снимок (snapshot) текущего фильтра поиска и информации, используемой фичей entity (`entityMap`, `ids`).

Для конфигурирования фичи используется тип `UndoRedoOptions`:

```typescript
export type UndoRedoOptions = {
    maxStackSize: number;
}

export const defaultUndoRedoOptions: UndoRedoOptions = {
    maxStackSize: 100
}
```

Объект опций позволяет ограничить размер стека. При чрезмерном росте стека более старые элементы удаляются согласно правилу «первым пришёл — первым вышел» (First In, First Out, FIFO).

Функцию добавляет `withUndoRedo`. Её структура такова:

```typescript
export function withUndoRedo(options = defaultUndoRedoOptions) {

    let previous: StackItem | null = null;
    let skipOnce = false;

    const undoStack: StackItem[] = [];
    const redoStack: StackItem[] = [];

    [...]

    return signalStoreFeature(
        // Ожидания к store:
        {
            state: type<{
                filter: Filter,
                entityMap: Record<EntityId, Entity>,
                ids: EntityId[]
            }>(),
        },
        [...]
        withMethods((store) => ({
            undo(): void { [...] },
            redo(): void { [...] }
        })),
        withHooks({
            onInit(store) {
                effect(() => {
                    const filter = store.filter();
                    const entityMap = store.entityMap();
                    const ids = store.ids();

                    [...]
                });
            }
        })

    )
}
```

Аналогично рассмотренной выше функции `withDataService`, она вызывает `signalStoreFeature` и определяет свои ожидания к store через первый аргумент. Она вводит методы `undo` и `redo`, восстанавливающие состояние из соответствующих стеков. Для наблюдения за состоянием хук `onInit` в конце создаёт эффект (effect — реактивный побочный эффект, реагирующий на изменения сигналов). После каждого изменения этот эффект сохраняет исходное состояние в стеке отмены (undo stack).

Одна деталь в этой реализации фичи Undo/Redo несколько особенна: сама фича хранит часть внутреннего состояния — например, `undoStack` и `redoStack` — которое не входит в Signal Store (это замыкания на уровне фабричной функции, а не свойства state).

Полную реализацию этой фичи смотрите в моём 📂 [GitHub-репозитории](https://github.com/manfredsteyer/standalone-example-cli/tree/arc-signal-store-custom-examples) (🔀 Ветка: arc-signal-store-custom-examples). Если вы хотите увидеть альтернативную реализацию, которая хранит внутреннее состояние фичи непосредственно в Signal Store, посмотрите ветку 🔀 `arc-signal-custom-examples-undoredo-alternative`.

## Использование store в компоненте

Чтобы использовать наш signal store «в 7 строк кода» в компоненте, достаточно внедрить его (inject) и делегировать обращения к его сигналам и методам:

```typescript
@Component( [...] )
export class FlightSearchComponent {
  private store = inject(FlightBookingStore);

  // Делегирование сигналам
  from = this.store.filter.from;
  to = this.store.filter.to;
  flights = this.store.entities;
  selected = this.store.selectedEntities;
  selectedIds = this.store.selectedIds;

  // Делегирование методам
  async search() {
    this.store.load();
  }

  undo(): void {
    this.store.undo();
  }

  redo(): void {
    this.store.redo();
  }

  updateCriteria(from: string, to: string): void {
    this.store.updateFilter({ from, to });
  }

  updateBasket(id: number, selected: boolean): void {
    this.store.updateSelected(id, selected);
  }

}
```

## Заключение и перспективы

Реализация повторяющихся задач посредством обобщённых пользовательских фич позволяет кардинально сократить объём исходного кода. В этой статье мы реализовали Signal Store для простого сценария всего в 7 строках кода. Хотя реализация таких фич в обобщённом виде поначалу добавляет некоторые накладные расходы (overhead), эти затраты определённо окупаются, как только у вас появляется несколько сценариев, структурированных подобным образом.

Для переиспользования существующих пользовательских фич наша фича делегирует им работу. API, предоставляемый NGRX Signal Store, позволяет пользовательской фиче гарантировать, что другие фичи были сконфигурированы. Она декларирует, какие свойства состояния, вычисляемые сигналы и методы она ожидает. При их отсутствии TypeScript выдаёт ошибку компиляции.

Ради простоты мы ограничились именами свойств по умолчанию, введёнными оркестрируемыми фичами. Однако, чтобы избежать конфликтов имён, эти имена также можно сконфигурировать. Например, фича `entity`, поставляемая вместе с Signal Store, поддерживает такие динамические свойства без ущерба для типобезопасности. В следующей статье я покажу, как применить эту же идею и к нашим пользовательским фичам.
