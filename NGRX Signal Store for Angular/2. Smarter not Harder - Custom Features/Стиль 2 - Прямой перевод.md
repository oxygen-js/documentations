## Цель

Цель этой статьи — показать, как реализовать пользовательские фичи для Signal Store, которые позволяют сделать следующее:

* Поиск сущностей (entities)
* Выбор нескольких сущностей
* Отображение выбранных сущностей
* Undo/Redo

Вот как выглядит демо-приложение, которое я построил поверх этих пользовательских фич:

![Демо-приложение](https://www.angulararchitects.io/wp-content/uploads/2023/11/result.png)

И вот весь код, который нам нужен для настройки store, включая Undo/Redo и подключение его к Data Service, загружающему сущности с бэкенда:

```typescript
export const FlightBookingStore = signalStore(
  { providedIn: 'root' },
  withEntities<Flight>(),
  withCallState(),
  withDataService(FlightService, { from: 'Graz', to: 'Hamburg'} ),
  withUndoRedo(),
);
```

Как видите, для управления сущностями я использую пакет `@ngrx/signals/entities`. Помимо этого, я перенёс оставшуюся логику в три переиспользуемые пользовательские фичи: `withCallState` уже обсуждалась в предыдущей статье. Эта статья обсуждает `withDataService` и предоставляет код для `withUndoRedo`.

## Пользовательская фича DataService

Идея, лежащая в основе фичи `DataService`, состоит в том, чтобы настроить состояние (state) для фильтра поиска и подключить Angular-сервис, который использует этот фильтр для поиска сущностей. На дальнейшем этапе разработки фича могла бы также вызывать `DataService` для сохранения и удаления сущностей. Однако, поскольку эти реализации не добавили бы здесь дополнительного понимания, я решил их пропустить ради краткости.

Чтобы сделать фичу `DataService` универсальной (generic), нам нужны некоторые общие типы, описывающие всё, с чем эта фича взаимодействует:

```typescript
import { EntityId } from "@ngrx/signals/entities";
[...]

export type Filter = Record<string, unknown>;
export type Entity = { id: EntityId };

export interface DataService<E extends Entity, F extends Filter> {
    load(filter: F): Promise<E[]>;
}
```

Эти типы описывают, как устроен наш фильтр поиска, что мы имеем в виду, когда говорим о сущности, и как должен выглядеть `DataService`. Тип `EntityId` приходит из `@ngrx/signals/entities` и принимает `string` или `number`.

Ожидание того, что сущность — это произвольный объект со свойством `id`, является одним из соглашений, которые предоставляет `@ngrx/signals/entities` для сокращения вашего кода. Если ваш первичный ключ называется иначе, вы можете сообщить об этом `@ngrx/signals/entities` соответствующим образом. Однако, чтобы представленный пример оставался небольшим, я решил придерживаться этого соглашения.

## Реализация универсальной пользовательской фичи

Функция `withDataService` возвращает фичу `DataService`:

```typescript
export function withDataService<E extends Entity, F extends Filter, S extends DataService<E, F>>(dataServiceType: Type<S>, filter: F) {
    [...]
}
```

Её параметры типа описывают сущность, которой нужно управлять, соответствующий фильтр поиска и `DataService`. При вызове этого универсального метода нам нужно лишь передать `DataService` и начальный фильтр. Остальное TypeScript выводит сам:

```typescript
withDataService(FlightService, { from: 'Graz', to: 'Hamburg'} ),
```

Функция `withDataService` вызывает `signalStoreFeature`, чтобы настроить нашу пользовательскую фичу:

```typescript
export function withDataService<E extends Entity, F extends Filter, S extends DataService<E, F>>(dataServiceType: Type<S>, filter: F) {
    return signalStoreFeature(
        // Наши ожидания к store:
        {
            state: type<{
                callState: CallState,
                entityMap: Record<EntityId, E>,
                ids: EntityId[]
            }>(),
            signals: type<{
                entities: Signal<Entity[]>
            }>(),
            methods: type<{}>()
        },

        // Композиция нескольких фич:
        withState( [...] ),
        withComputed( [...] ),
        withMethods( [...] )
    );
}
```

Как было показано в первой статье этой серии, функция `signalStoreFeature` по сути компонует существующие фичи в новую. Например, мы можем ввести новые свойства состояния с помощью `withState`, вычисляемые сигналы (computed Signals) с помощью `withComputed` или методы с помощью `withMethods`.

Однако в этот раз одна небольшая деталь немного отличается: у нашей фичи есть некоторые **ожидания** к Signal Store, с которым она используется. Она ожидает, что фичи `callState` и `entity` будут на месте. Первая настраивает свойство `callState`, которое нам нужно; вторая настраивает свойства `entityMap` и `ids`, а также вычисляемый сигнал `entities`.

Эти ожидания определяются первым параметром, передаваемым в `signalStoreFeature`. Он описывает ожидаемые свойства состояния (`state`), вычисляемые сигналы (`signals`) и методы. Поскольку мы не ожидаем никаких методов, мы также можем опустить ключ `methods` вместо того, чтобы указывать `type<{}>()`.

Чтобы избежать конфликтов имён, фича `entity` позволяет использовать разные имена свойств. Чтобы не усложнять, я придерживаюсь здесь имён по умолчанию. Однако в одной из следующих статей этой серии вы узнаете, как типобезопасно работать с динамическими именами свойств.

Оставшиеся части этой пользовательской фичи — это лишь добавление свойств состояния, вычисляемых сигналов и методов поверх ожидаемых фич:

```typescript
export function withDataService<E extends Entity, F extends Filter, S extends DataService<E, F>>(dataServiceType: Type<S>, filter: F) {
    return signalStoreFeature(
        // Первый параметр содержит
        // наши ожидания к store:
        // Если они не выполнены, TypeScript
        // не даст добавить эту фичу!
        {
            state: type<{
                callState: CallState,
                entityMap: Record<EntityId, E>,
                ids: EntityId[]
            }>(),
            signals: type<{
                entities: Signal<Entity[]>
            }>(),
            methods: type<{}>()
        },
        withState({
            filter,
            selectedIds: {} as Record<EntityId, boolean>,
        }),
        withComputed(({ selectedIds, entities }) => ({
            selectedEntities: computed(() => entities().filter(e => selectedIds()[e.id]))
        })),
        withMethods((store) => {
            const dataService = inject(dataServiceType)
            return {
                updateFilter(filter: F): void {
                    patchState(store, { filter });
                },
                updateSelected(id: EntityId, selected: boolean): void {
                    patchState(store, ({ selectedIds }) => ({
                        selectedIds: {
                            ...selectedIds,
                            [id]: selected,
                        }
                    }));
                },
                async load(): Promise<void> {
                    patchState(store, setLoading());
                    const result = await dataService.load(store.filter());
                    patchState(store, setAllEntities(result));
                    patchState(store, setLoaded());
                }
            };
        })
    );
}
```

## Фича Undo/Redo

Фича Undo/Redo реализована очень похожим образом. Внутри она управляет двумя стеками: стеком undo и стеком redo. По сути эти стеки — это массивы с элементами `StackItem`:

```typescript
export type StackItem = {
    filter: Filter;
    entityMap: Record<EntityId, Entity>,
    ids: EntityId[]
};
```

Каждый `StackItem` представляет собой снимок (snapshot) текущего фильтра поиска и информации, которую использует фича entity (`entityMap`, `ids`).

Для конфигурирования фичи используется тип `UndoRedoOptions`:

```typescript
export type UndoRedoOptions = {
    maxStackSize: number;
}

export const defaultUndoRedoOptions: UndoRedoOptions = {
    maxStackSize: 100
}
```

Объект опций позволяет нам ограничить размер стека. Более старые элементы удаляются согласно правилу «первым пришёл — первым ушёл» (First In, First Out), если стек становится слишком большим.

Функция `withUndoRedo` добавляет фичу. Она устроена следующим образом:

```typescript
export function withUndoRedo(options = defaultUndoRedoOptions) {

    let previous: StackItem | null = null;
    let skipOnce = false;

    const undoStack: StackItem[] = [];
    const redoStack: StackItem[] = [];

    [...]

    return signalStoreFeature(
        // Ожидания к store:
        {
            state: type<{
                filter: Filter,
                entityMap: Record<EntityId, Entity>,
                ids: EntityId[]
            }>(),
        },
        [...]
        withMethods((store) => ({
            undo(): void { [...] },
            redo(): void { [...] }
        })),
        withHooks({
            onInit(store) {
                effect(() => {
                    const filter = store.filter();
                    const entityMap = store.entityMap();
                    const ids = store.ids();

                    [...]
                });
            }
        })

    )
}
```

Подобно рассмотренной выше функции `withDataService`, она вызывает `signalStoreFeature` и определяет свои ожидания к store, используя первый аргумент. Она вводит методы `undo` и `redo`, восстанавливающие состояние из соответствующих стеков. Чтобы наблюдать за состоянием, хук `onInit` в конце создаёт эффект (effect). После каждого изменения этот эффект сохраняет исходное состояние в стеке undo.

Одна вещь немного особенная в этой реализации фичи Undo/Redo: сама фича хранит некоторое внутреннее состояние — например, `undoStack` и `redoStack` — которое не является частью Signal Store.

Полную реализацию этой фичи смотрите в моём 📂 [GitHub-репозитории](https://github.com/manfredsteyer/standalone-example-cli/tree/arc-signal-store-custom-examples) (🔀 Ветка: arc-signal-store-custom-examples). Если вы хотите увидеть другую реализацию, которая также хранит внутреннее состояние фичи в Signal Store, посмотрите ветку 🔀 `arc-signal-custom-examples-undoredo-alternative`.

## Использование store в компоненте

Чтобы использовать наш signal store «в 7 строк кода» в компоненте, просто внедрите его и делегируйте обращения к его сигналам и методам:

```typescript
@Component( [...] )
export class FlightSearchComponent {
  private store = inject(FlightBookingStore);

  // Делегируем сигналам
  from = this.store.filter.from;
  to = this.store.filter.to;
  flights = this.store.entities;
  selected = this.store.selectedEntities;
  selectedIds = this.store.selectedIds;

  // Делегируем методам
  async search() {
    this.store.load();
  }

  undo(): void {
    this.store.undo();
  }

  redo(): void {
    this.store.redo();
  }

  updateCriteria(from: string, to: string): void {
    this.store.updateFilter({ from, to });
  }

  updateBasket(id: number, selected: boolean): void {
    this.store.updateSelected(id, selected);
  }

}
```

## Заключение и перспективы

Реализация повторяющихся задач с помощью универсальных пользовательских фич позволяет радикально сократить ваш исходный код. В этой статье мы реализовали Signal Store для простого сценария всего в 7 строках кода. Хотя реализация таких фич в универсальном виде поначалу добавляет некоторые накладные расходы, эти усилия наверняка окупятся, как только у вас будет несколько сценариев, структурированных подобным образом.

Чтобы переиспользовать существующие пользовательские фичи, наша пользовательская фича делегирует им работу. API, предоставляемый NGRX Signal Store, позволяет пользовательской фиче убедиться, что другие фичи были настроены. Она определяет, какие свойства состояния, вычисляемые сигналы и методы она ожидает. Если они отсутствуют, TypeScript выдаёт ошибку компиляции.

Ради простоты мы ограничились именами свойств по умолчанию, введёнными оркестрируемыми фичами. Однако, чтобы избежать конфликтов имён, эти имена тоже можно настроить. Например, фича `entity`, поставляемая с Signal Store, поддерживает такие динамические свойства без ущерба для типобезопасности. В следующей статье я покажу, как использовать эту идею и для наших пользовательских фич.
