Новый Event API привносит в мир Signal Store функциональность, знакомую по основанному на Redux «глобальному» (Global) хранилищу NgRx. Поскольку Redux подразумевает единое центральное хранилище, команда NgRx использует более общий термин — Flux.

Одна из больших сильных сторон этого нового API заключается в том, что он может использоваться конкретно для выбранных случаев. Это позволяет нам начать в облегчённом виде и добавлять паттерн Flux к отдельным хранилищам (store) по мере необходимости.

В этой статье я покажу вам, как использовать этот новый API.

## События между хранилищами

Чтобы проиллюстрировать потенциал для целенаправленного использования Event API, я сначала продемонстрирую установление слабой связанности (loose coupling) между двумя хранилищами. Здесь акцент делается на самой работе с событиями (eventing), а не на реализации Flux. После этого я обсужу, как реализовать Flux для выбранного хранилища.

Используемый пример состоит из трёх хранилищ: _DessertStore_ управляет загруженным списком десертов, а _DessertDetailStore_ используется для обработки выбранного десерта. Кроме того, рейтинги отдельных десертов находятся в _RatingStore_:

![Signal Store-ы в примере](https://www.angulararchitects.io/wp-content/uploads/2025/05/stores.png)

Такое разделение не является чем-то необычным для облегчённых хранилищ вроде Signal Store. В отличие от Redux, разделение основано не на технических аспектах, таких как reducer-ы, эффекты или селекторы. Вместо этого небольшая часть состояния инкапсулируется вместе с вычислениями и операциями, которые на неё опираются.

Управляемое состояние часто соответствует сущности (entity) с точки зрения некоторой (под)фичи. В рассматриваемом здесь примере это деление означает, что десерт, обновлённый _DessertDetailStore_, может также потребовать обновления в _DessertStore_, чтобы загруженный обзор оставался актуальным.

Чтобы предотвратить связывание обоих хранилищ друг с другом, хорошим вариантом является событие (event), указывающее на обновление десерта. Такие события можно определить с помощью функции _event_ из Event API. Однако чаще встречается предоставление нескольких связанных событий внутри группы событий (Event Group):

```javascript
import { type } from '@ngrx/signals';
import { eventGroup } from '@ngrx/signals/events'; 
import { Dessert } from './dessert';

export const dessertDetailStoreEvents = eventGroup({
  source: 'Dessert Detail Store',
  events: {
    dessertUpdated: type<{
        dessert: Dessert
    }>(),
  },
});
```

Все события в группе имеют один и тот же источник (source). Источник информирует о той части приложения, которая инициирует событие. Это облегчает понимание потоков сообщений при отладке. Каждое событие определяется именем (здесь: _dessertUpdate_) и типом (_type_) для его полезной нагрузки (_payload_). Эта _payload_ содержит данные, которые дополнительно описывают событие.

_DessertDetailStore_ инициирует событие _dessertUpdate_ после сохранения десерта. Для этого он использует сервис _Dispatcher_, предоставляемый Eventing API:

```javascript
import { Dispatcher } from '@ngrx/signals/events';

[...]

export const DessertDetailStore = signalStore(
  [...]
  withProps(() => ({
    [...],
    _dispatcher: inject(Dispatcher)
  })),
  withMethods((store) => ({
    [...],
    save(id: number, dessert: Partial<Dessert>): void {

      [...]
      // Инициировать событие
      const event = dessertDetailStoreEvents.dessertUpdated({
        dessert: savedDessert
      });
      store._dispatcher.dispatch(event);

    },
  })),
);
```

Reducer выступает в роли получателя в _DessertStore_. Он получает payload события и возвращает Updater:

```javascript
import { on, withReducer } from '@ngrx/signals/events';

[…]

export const DessertStore = signalStore(
  { providedIn: 'root' },
  withState({
    […]
    desserts: [] as Dessert[],
  }),
  withReducer(
    on(dessertDetailStoreEvents.dessertUpdated, ({ payload }) => {
      const updated = payload.dessert;
      return (store) => ({
        desserts: store.desserts.map((d) =>
          d.id === updated.id ? updated : d,
        ),
      });
    }),
  ),
  […]
)
```

Updater заменяет изменённый десерт в списке всех полученных десертов. Возвращаемое состояние является частичным (partial), что означает, что оно должно содержать только изменённые свойства. Как это типично для Signal Store, Updater-ы также могут быть вынесены в собственные функции:

```javascript
export type DessertSlice = {
  desserts: Dessert[];
};

function updateDessert(updated: Dessert) {
  return (store: DessertSlice) => ({
    desserts: store.desserts.map((d) => 
          (d.id === updated.id ? updated : d)),
  });
}
```

Такая функция-Updater позволяет сократить reducer:

```javascript
withReducer(
  on(dessertDetailStoreEvents.dessertUpdated, ({ payload }) => {
    return updateDessert(updated);
  }),
),
```

Эта возможность также используется для делегирования Updater-ам, предоставляемым пользовательскими фичами вроде _withEntity_. Если текущее состояние не требуется, reducer может напрямую предоставить обновлённое частичное состояние. Этот вариант я обсужу ниже.

## Flux/«Redux» для отдельных хранилищ

На следующем шаге я хотел бы показать, как Event API может использоваться для реализации паттерна Flux для отдельных хранилищ. Это похоже на использование основанного на Redux «глобального» (Global) хранилища NgRx:

* Потребитель инициирует событие (event) (в «глобальном» хранилище это называется действием, Action).
* Эффекты (Effects) получают события и запускают асинхронные side-эффекты, публикуя результаты как дальнейшие события.
* Reducer-ы в хранилище получают события и обновляют управляемое состояние.

И снова: Event API и Flux могут применяться выборочно и не обязательно навязываться всем хранилищам. Это особенно интересно, когда несколько компонентов потребляют и изменяют одно и то же состояние.

Как и раньше, реализация начинается с группы событий. Она содержит событие _loadDesserts_, которое вызывает Effect для загрузки десертов. Затем этот эффект сообщает об исходе этой операции событием _loadDessertsSuccess_ или, в случае ошибки, событием _loadDessertsError_:

```javascript
import { eventGroup } from '@ngrx/signals/events'; 
import { type } from '@ngrx/signals';
import { Dessert } from './dessert';

export const dessertEvents = eventGroup({
  source: 'Dessert Feature',
  events: {
    loadDesserts: type<{
        originalName: string,
        englishName: string,
    }>(),
    loadDessertsSuccess: type<{
        desserts: Dessert[]
    }>(),
    loadDessertsError: type<{
        error: string
    }>(),
  },
});
```

### Reducer

Reducer-ы для загрузки десертов несколько проще, чем в первом случае. Вместо updater-а, который отображает текущее состояние в новое, они просто возвращают новые значения:

```javascript
import { on, withReducer } from '@ngrx/signals/events';
[…]

export const DessertStore = signalStore(
  { providedIn: 'root' },
  withState({
    filter: {
      originalName: '',
      englishName: '',
    },
    loading: false,
    desserts: [] as Dessert[],
    error: '',
  }),
  withReducer(
    […],
    on(dessertEvents.loadDesserts, ({ payload }) => {
      return {
        filter: payload,
        loading: true,
      };
    }),
    on(dessertEvents.loadDessertsSuccess, ({ payload }) => {
      return {
        desserts: payload.desserts,
        loading: false,
      };
    }),
    on(dessertEvents.loadDessertsError, ({ payload }) => {
      return {
        error: payload.error,
        loading: false,
      };
    }),
  ),
  […]
);
```

### Effects

Эффекты (Effects) в Event API, которые не имеют ничего общего с одноимённым понятием из мира Signals, технически представляют собой основанные на RxJS пайпы, которые отображают входящие события в исходящие. Для этого они обычно включают асинхронную операцию через «уплощающий» (flattening) оператор, такой как _switchMap_ или _mergeMap_:

```javascript
import { Events, withEffects } from '@ngrx/signals/events';
import { mapResponse } from '@ngrx/operators';

[…]

export const DessertStore = signalStore(
  […],
  withProps(() => ({
    _dessertService: inject(DessertService),
    _toastService: inject(ToastService),
    _events: inject(Events),
  })),
  […]
  withEffects((store) => ({
    loadDesserts$: store._events.on(dessertEvents.loadDesserts).pipe(
      switchMap((e) =>
        store._dessertService.find(e.payload).pipe(
          mapResponse({
            next: (desserts) => dessertEvents.loadDessertsSuccess({ desserts }),
            error: (error) =>
              dessertEvents.loadDessertsError({ error: String(error) }),
          }),
        ),
      ),
    ),
  })),
);
```

Хотя описательное имя, такое как _loadDesserts$_ в показанном примере, важно для читаемости, технически оно не имеет значения, потому что Event API запускает Effect не по его имени, а через события. Хранилище создаёт эффекты через сервис _Event_. Его метод _on_ фильтрует полученные события и возвращает их через _Observable_. Если хранилище не передаёт в _on_ никаких параметров, оно получает все события. Если хранилище передаёт один или несколько типов событий, _on_ возвращает только совпадающие события.

Чтобы отобразить результат запущенного side-эффекта в исходящее событие, полезен оператор _mapResponse_, поставляемый с NgRx. Это комбинация _map_ и _catchError_, которая позволяет преобразовывать как результаты, так и ошибки.

Как и в ранние дни NgRx Global Store, важно, чтобы ошибки обрабатывались, например, с помощью _catchError_ или, как показано, с помощью _mapResponse_. В противном случае RxJS закрывает observable, и дальнейшие события больше не обрабатываются Effect. Как обычно, эти ошибки должны обрабатываться на том уровне, где они произошли. В показанном случае это внутри _switchMap_, что приводит к дальнейшей вложенности.

Опции вроде _resubscribeOnError_ из Global Store, которая автоматически компенсирует ошибки, в настоящее время не существует. Однако команда NgRx рассматривает её.

### Потребление хранилища

Чтобы потреблять хранилище, отдельные компоненты могут получить Dispatcher через инъекцию (injection) и использовать его для инициирования нужного события:

```javascript
this.#dispatcher.dispatch(
  dessertEvents.loadDesserts({
    originalName: this.originalName(),
    englishName: this.englishName(),
  }),
);
```

## Бонус: Redux DevTools

Signal Store не поставляется с официальной интеграцией с [Redux Dev Tools](https://chromewebstore.google.com/detail/redux-devtools/lmhkpmbekcpmknklioeibfkpmmfibljd), которые позволяют заглянуть внутрь хранилища во время отладки, а также предоставляют информацию об истории отдельных изменений состояния. Однако эту возможность можно добавить, используя механизм расширяемости Signal Store.

Реализацию этой идеи можно найти в community-пакете @angular-architects/ngrx-toolkit. Он предоставляет фичу _withDevtools_, которую можно добавить в хранилище:

```javascript
import { withDevtools } from '@angular-architects/ngrx-toolkit';
[…]

export const DessertStore = signalStore(
  […]
  withDevtools('DessertStore')
);
```

Переданная строка используется как имя для узла, который представляет соответствующее хранилище в Dev Tools. Если Dev Tools были установлены как браузерное расширение, данные, хранящиеся в store, и их историю можно просматривать в Developer Console:

## Резюме

Новый Event API для NgRx Signal Store позволяет целенаправленно реализовывать паттерны Flux по мере необходимости. С дополнительными инструментами, такими как Redux DevTools, он упрощает отладку. Event API предоставляет гибкий мост между облегчённым использованием Signal Store и проверенными практиками управления состоянием, известными из мира Redux.
