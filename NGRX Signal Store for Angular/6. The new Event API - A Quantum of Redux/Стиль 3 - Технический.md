Новый Event API переносит в мир Signal Store функциональность, знакомую по основанному на Redux «глобальному» (Global) хранилищу NgRx. Так как Redux подразумевает единственное центральное хранилище (single central store), команда NgRx использует более общий термин — Flux.

Одно из существенных преимуществ нового API в том, что его можно задействовать точечно, только для отдельных сценариев. Это позволяет начать с минимальными издержками (lightweight) и добавлять паттерн Flux к конкретным хранилищам (store) по мере необходимости.

В этой статье я покажу, как пользоваться этим новым API.

## События между хранилищами (Eventing Between Stores)

Чтобы продемонстрировать потенциал точечного применения Event API, я сначала покажу, как организовать слабую связанность (loose coupling) между двумя хранилищами. Здесь акцент сделан на самом механизме событий (eventing), а не на реализации Flux. После этого я рассмотрю реализацию Flux для отдельно взятого хранилища.

Рассматриваемый пример состоит из трёх хранилищ: _DessertStore_ управляет загруженным списком десертов, а _DessertDetailStore_ служит для обработки выбранного десерта. Кроме того, рейтинги отдельных десертов находятся в _RatingStore_:

![Signal Store-ы в примере](https://www.angulararchitects.io/wp-content/uploads/2025/05/stores.png)

Такое разбиение типично для легковесных хранилищ вроде Signal Store. В отличие от Redux, разбиение производится не по техническим аспектам — reducer-ам, эффектам или селекторам, — а путём инкапсуляции небольшого фрагмента состояния вместе с вычислениями и операциями, которые на него опираются.

С точки зрения некоторой (под)фичи управляемое состояние часто соответствует одной сущности (entity). В рассматриваемом примере такое деление приводит к тому, что десерт, обновлённый в _DessertDetailStore_, может потребоваться обновить и в _DessertStore_, чтобы загруженный обзорный список оставался актуальным.

Чтобы не вводить прямую связанность между этими двумя хранилищами, удачным решением будет событие (event), сигнализирующее об обновлении десерта. Такие события определяются функцией _event_ из Event API. Тем не менее на практике чаще объявляют несколько связанных событий в составе группы событий (Event Group):

```javascript
import { type } from '@ngrx/signals';
import { eventGroup } from '@ngrx/signals/events'; 
import { Dessert } from './dessert';

export const dessertDetailStoreEvents = eventGroup({
  source: 'Dessert Detail Store',
  events: {
    dessertUpdated: type<{
        dessert: Dessert
    }>(),
  },
});
```

Все события в группе имеют один и тот же источник (source). Источник указывает на ту часть приложения, которая инициирует событие, что упрощает анализ потоков сообщений при отладке. Каждое событие задаётся именем (здесь — _dessertUpdate_) и типом (_type_) полезной нагрузки (_payload_). Эта _payload_ содержит данные, дополнительно описывающие событие.

После сохранения десерта _DessertDetailStore_ инициирует (dispatch) событие _dessertUpdate_. Для этого он использует сервис _Dispatcher_, предоставляемый Eventing API:

```javascript
import { Dispatcher } from '@ngrx/signals/events';

[...]

export const DessertDetailStore = signalStore(
  [...]
  withProps(() => ({
    [...],
    _dispatcher: inject(Dispatcher)
  })),
  withMethods((store) => ({
    [...],
    save(id: number, dessert: Partial<Dessert>): void {

      [...]
      // Инициировать (dispatch) событие
      const event = dessertDetailStoreEvents.dessertUpdated({
        dessert: savedDessert
      });
      store._dispatcher.dispatch(event);

    },
  })),
);
```

В роли получателя в _DessertStore_ выступает Reducer. Он принимает payload события и возвращает Updater (функцию-обновитель состояния):

```javascript
import { on, withReducer } from '@ngrx/signals/events';

[…]

export const DessertStore = signalStore(
  { providedIn: 'root' },
  withState({
    […]
    desserts: [] as Dessert[],
  }),
  withReducer(
    on(dessertDetailStoreEvents.dessertUpdated, ({ payload }) => {
      const updated = payload.dessert;
      return (store) => ({
        desserts: store.desserts.map((d) =>
          d.id === updated.id ? updated : d,
        ),
      });
    }),
  ),
  […]
)
```

Updater заменяет изменённый десерт в списке всех загруженных десертов. Возвращаемое состояние является частичным (partial state) — то есть достаточно вернуть только изменившиеся свойства, остальные сохранятся без изменений. Как обычно для Signal Store, Updater-ы можно вынести в отдельные функции:

```javascript
export type DessertSlice = {
  desserts: Dessert[];
};

function updateDessert(updated: Dessert) {
  return (store: DessertSlice) => ({
    desserts: store.desserts.map((d) => 
          (d.id === updated.id ? updated : d)),
  });
}
```

Такая функция-Updater позволяет сократить reducer:

```javascript
withReducer(
  on(dessertDetailStoreEvents.dessertUpdated, ({ payload }) => {
    return updateDessert(updated);
  }),
),
```

Эта же возможность используется для делегирования Updater-ам, предоставляемым пользовательскими фичами (custom features) вроде _withEntity_. Если текущее состояние reducer-у не требуется, он может вернуть обновлённое частичное состояние напрямую (без функции-обновителя, замыкающейся на текущий store). Этот вариант рассмотрен ниже.

## Flux/«Redux» для отдельных хранилищ

На следующем шаге я покажу, как с помощью Event API реализовать паттерн Flux для отдельных хранилищ. Это аналогично использованию основанного на Redux «глобального» (Global) хранилища NgRx:

* Потребитель инициирует (dispatch) событие (event); в «глобальном» хранилище оно называется действием (Action).
* Эффекты (Effects) принимают события, запускают асинхронные побочные эффекты (side effects) и публикуют результаты в виде последующих событий.
* Reducer-ы внутри хранилища принимают события и обновляют управляемое состояние.

И здесь Event API и Flux применяются выборочно — их не требуется навязывать всем хранилищам. Это особенно полезно, когда одно и то же состояние потребляют и изменяют несколько компонентов.

Как и ранее, реализация начинается с группы событий. Она содержит событие _loadDesserts_, которое вызывает Effect для загрузки десертов. Затем этот эффект сообщает об итоге операции событием _loadDessertsSuccess_, а в случае ошибки — событием _loadDessertsError_:

```javascript
import { eventGroup } from '@ngrx/signals/events'; 
import { type } from '@ngrx/signals';
import { Dessert } from './dessert';

export const dessertEvents = eventGroup({
  source: 'Dessert Feature',
  events: {
    loadDesserts: type<{
        originalName: string,
        englishName: string,
    }>(),
    loadDessertsSuccess: type<{
        desserts: Dessert[]
    }>(),
    loadDessertsError: type<{
        error: string
    }>(),
  },
});
```

### Reducer

Reducer-ы для загрузки десертов несколько проще, чем в первом случае. Вместо updater-а, отображающего текущее состояние в новое, они просто возвращают новые значения:

```javascript
import { on, withReducer } from '@ngrx/signals/events';
[…]

export const DessertStore = signalStore(
  { providedIn: 'root' },
  withState({
    filter: {
      originalName: '',
      englishName: '',
    },
    loading: false,
    desserts: [] as Dessert[],
    error: '',
  }),
  withReducer(
    […],
    on(dessertEvents.loadDesserts, ({ payload }) => {
      return {
        filter: payload,
        loading: true,
      };
    }),
    on(dessertEvents.loadDessertsSuccess, ({ payload }) => {
      return {
        desserts: payload.desserts,
        loading: false,
      };
    }),
    on(dessertEvents.loadDessertsError, ({ payload }) => {
      return {
        error: payload.error,
        loading: false,
      };
    }),
  ),
  […]
);
```

### Effects

Эффекты (Effects) в Event API не имеют ничего общего с одноимённым понятием из мира Signals. Технически это пайпы (pipes) на базе RxJS, отображающие входящие события в исходящие. Для этого они, как правило, выполняют асинхронную операцию через flattening-оператор (уплощающий поток вложенных Observable) — например, _switchMap_ или _mergeMap_:

```javascript
import { Events, withEffects } from '@ngrx/signals/events';
import { mapResponse } from '@ngrx/operators';

[…]

export const DessertStore = signalStore(
  […],
  withProps(() => ({
    _dessertService: inject(DessertService),
    _toastService: inject(ToastService),
    _events: inject(Events),
  })),
  […]
  withEffects((store) => ({
    loadDesserts$: store._events.on(dessertEvents.loadDesserts).pipe(
      switchMap((e) =>
        store._dessertService.find(e.payload).pipe(
          mapResponse({
            next: (desserts) => dessertEvents.loadDessertsSuccess({ desserts }),
            error: (error) =>
              dessertEvents.loadDessertsError({ error: String(error) }),
          }),
        ),
      ),
    ),
  })),
);
```

Хотя описательное имя — например, _loadDesserts$_ в показанном примере — важно для читаемости, технически оно несущественно: Event API запускает Effect не по имени, а по событиям. Хранилище создаёт эффекты через сервис _Event_. Его метод _on_ фильтрует поступающие события и возвращает их через _Observable_. Если в _on_ не переданы параметры, хранилище получает все события; если же переданы один или несколько типов событий, _on_ возвращает только совпадающие.

Чтобы отобразить результат запущенного побочного эффекта в исходящее событие, удобно применить оператор _mapResponse_, поставляемый с NgRx. Он представляет собой комбинацию _map_ и _catchError_ и позволяет преобразовывать как успешные результаты, так и ошибки.

Как и в ранние дни NgRx Global Store, критически важно обрабатывать ошибки — например, через _catchError_ или, как показано, через _mapResponse_. Иначе RxJS завершит (закроет) observable, и последующие события Effect обрабатывать перестанет. Как обычно, ошибки нужно обрабатывать на том уровне, где они возникли. В показанном случае это внутри _switchMap_, что приводит к дополнительной вложенности.

Опции наподобие _resubscribeOnError_ из Global Store, которая автоматически компенсирует ошибки (переподписываясь на поток), на данный момент нет. Однако команда NgRx рассматривает её добавление.

### Потребление хранилища

Чтобы работать с хранилищем, отдельные компоненты могут получить Dispatcher через инъекцию зависимостей (injection) и инициировать (dispatch) нужное событие:

```javascript
this.#dispatcher.dispatch(
  dessertEvents.loadDesserts({
    originalName: this.originalName(),
    englishName: this.englishName(),
  }),
);
```

## Бонус: Redux DevTools

Signal Store не включает официальную интеграцию с [Redux Dev Tools](https://chromewebstore.google.com/detail/redux-devtools/lmhkpmbekcpmknklioeibfkpmmfibljd), которые позволяют наблюдать содержимое хранилища при отладке и дают информацию об истории отдельных изменений состояния. Тем не менее эту возможность можно добавить через механизм расширяемости (extensibility) Signal Store.

Реализация этой идеи доступна в community-пакете @angular-architects/ngrx-toolkit. Он предоставляет фичу _withDevtools_, добавляемую в хранилище:

```javascript
import { withDevtools } from '@angular-architects/ngrx-toolkit';
[…]

export const DessertStore = signalStore(
  […]
  withDevtools('DessertStore')
);
```

Переданная строка используется как имя узла, представляющего соответствующее хранилище в Dev Tools. Если Dev Tools установлены как браузерное расширение, данные хранилища (store) и их историю можно просматривать в Developer Console:

## Итоги

Новый Event API для NgRx Signal Store обеспечивает точечную (по требованию) реализацию паттернов Flux. В сочетании с дополнительными инструментами вроде Redux DevTools он упрощает отладку. Event API служит гибким мостом между легковесным использованием Signal Store и проверенными практиками управления состоянием, известными из мира Redux.
