Новый Event API приносит в мир Signal Store ту же функциональность, которая нам уже знакома по «глобальному» (Global) хранилищу NgRx на базе Redux. Поскольку Redux подразумевает единственное центральное хранилище, команда NgRx предпочитает более общий термин — Flux.

Одна из главных сильных сторон нового API в том, что его можно применять точечно — только там, где он действительно нужен. Благодаря этому можно начать легко и без лишнего веса, а паттерн Flux добавлять к отдельным хранилищам (store) по мере необходимости.

В этой статье я покажу, как пользоваться новым API.

## События между хранилищами

Чтобы наглядно показать, как можно применять Event API точечно, я сначала продемонстрирую, как наладить слабую связанность между двумя хранилищами. Здесь нас интересует именно работа с событиями (eventing), а не реализация Flux как такового. После этого я расскажу, как реализовать Flux для конкретно выбранного хранилища.

В нашем примере три хранилища. _DessertStore_ управляет загруженным списком десертов, а _DessertDetailStore_ используется для работы с выбранным десертом. Кроме того, рейтинги отдельных десертов лежат в _RatingStore_:

![Signal Store-ы в примере](https://www.angulararchitects.io/wp-content/uploads/2025/05/stores.png)

Такое разделение — обычное дело для легковесных хранилищ вроде Signal Store. В отличие от Redux, разбивка здесь идёт не по техническим признакам — таким как reducer-ы, эффекты или селекторы. Вместо этого мы инкапсулируем небольшой кусочек состояния вместе с вычислениями и операциями, которые на него опираются.

Управляемое состояние часто соответствует одной сущности (entity) с точки зрения некоторой (под)фичи. В нашем примере это разделение приводит к тому, что десерт, обновлённый в _DessertDetailStore_, возможно, придётся обновить и в _DessertStore_ — чтобы загруженный список оставался актуальным.

Чтобы не связывать оба хранилища напрямую друг с другом, отличным решением будет событие (event), сигнализирующее об обновлении десерта. Такие события можно определять с помощью функции _event_ из Event API. Однако чаще удобнее объявить сразу несколько связанных событий внутри группы событий (Event Group):

```javascript
import { type } from '@ngrx/signals';
import { eventGroup } from '@ngrx/signals/events'; 
import { Dessert } from './dessert';

export const dessertDetailStoreEvents = eventGroup({
  source: 'Dessert Detail Store',
  events: {
    dessertUpdated: type<{
        dessert: Dessert
    }>(),
  },
});
```

У всех событий в группе один и тот же источник (source). Источник сообщает, какая часть приложения породила событие. Благодаря этому потоки сообщений проще отслеживать при отладке. Каждое событие задаётся именем (здесь — _dessertUpdate_) и типом (_type_) его полезной нагрузки (_payload_). Эта _payload_ содержит данные, которые подробнее описывают событие.

После сохранения десерта _DessertDetailStore_ инициирует (триггерит) событие _dessertUpdate_. Для этого он использует сервис _Dispatcher_, который предоставляет Eventing API:

```javascript
import { Dispatcher } from '@ngrx/signals/events';

[...]

export const DessertDetailStore = signalStore(
  [...]
  withProps(() => ({
    [...],
    _dispatcher: inject(Dispatcher)
  })),
  withMethods((store) => ({
    [...],
    save(id: number, dessert: Partial<Dessert>): void {

      [...]
      // Инициируем событие
      const event = dessertDetailStoreEvents.dessertUpdated({
        dessert: savedDessert
      });
      store._dispatcher.dispatch(event);

    },
  })),
);
```

Получателем в _DessertStore_ выступает Reducer. Он принимает payload события и возвращает Updater:

```javascript
import { on, withReducer } from '@ngrx/signals/events';

[…]

export const DessertStore = signalStore(
  { providedIn: 'root' },
  withState({
    […]
    desserts: [] as Dessert[],
  }),
  withReducer(
    on(dessertDetailStoreEvents.dessertUpdated, ({ payload }) => {
      const updated = payload.dessert;
      return (store) => ({
        desserts: store.desserts.map((d) =>
          d.id === updated.id ? updated : d,
        ),
      });
    }),
  ),
  […]
)
```

Updater подменяет в списке всех загруженных десертов тот единственный, который был изменён. Возвращаемое состояние частичное (partial) — то есть в нём достаточно указать только изменившиеся свойства. Как это обычно бывает в Signal Store, Updater-ы тоже можно вынести в отдельные функции:

```javascript
export type DessertSlice = {
  desserts: Dessert[];
};

function updateDessert(updated: Dessert) {
  return (store: DessertSlice) => ({
    desserts: store.desserts.map((d) => 
          (d.id === updated.id ? updated : d)),
  });
}
```

Такая функция-Updater позволяет сократить reducer:

```javascript
withReducer(
  on(dessertDetailStoreEvents.dessertUpdated, ({ payload }) => {
    return updateDessert(updated);
  }),
),
```

Эту же возможность используют, чтобы делегировать работу Updater-ам, которые предоставляют собственные фичи вроде _withEntity_. А если текущее состояние и вовсе не нужно, reducer может напрямую вернуть обновлённое частичное состояние. Об этом варианте я расскажу чуть ниже.

## Flux/«Redux» для отдельных хранилищ

На следующем шаге я хочу показать, как с помощью Event API реализовать паттерн Flux для отдельных хранилищ. Это очень похоже на использование «глобального» (Global) хранилища NgRx на базе Redux:

* Потребитель инициирует событие (event); в «глобальном» хранилище оно называется действием (Action).
* Эффекты (Effects) принимают события, запускают асинхронные side-эффекты и публикуют результаты как новые события.
* Reducer-ы внутри хранилища принимают события и обновляют управляемое состояние.

И снова: Event API и Flux можно применять выборочно — их вовсе не обязательно навязывать всем хранилищам подряд. Особенно это удобно, когда одно и то же состояние читают и меняют сразу несколько компонентов.

Как и раньше, реализация начинается с группы событий. В ней есть событие _loadDesserts_, которое вызывает Effect для загрузки десертов. Этот эффект затем сообщает об итогах операции событием _loadDessertsSuccess_, а в случае ошибки — событием _loadDessertsError_:

```javascript
import { eventGroup } from '@ngrx/signals/events'; 
import { type } from '@ngrx/signals';
import { Dessert } from './dessert';

export const dessertEvents = eventGroup({
  source: 'Dessert Feature',
  events: {
    loadDesserts: type<{
        originalName: string,
        englishName: string,
    }>(),
    loadDessertsSuccess: type<{
        desserts: Dessert[]
    }>(),
    loadDessertsError: type<{
        error: string
    }>(),
  },
});
```

### Reducer

Reducer-ы для загрузки десертов здесь чуть проще, чем в первом случае. Вместо updater-а, который отображает текущее состояние в новое, они просто возвращают новые значения:

```javascript
import { on, withReducer } from '@ngrx/signals/events';
[…]

export const DessertStore = signalStore(
  { providedIn: 'root' },
  withState({
    filter: {
      originalName: '',
      englishName: '',
    },
    loading: false,
    desserts: [] as Dessert[],
    error: '',
  }),
  withReducer(
    […],
    on(dessertEvents.loadDesserts, ({ payload }) => {
      return {
        filter: payload,
        loading: true,
      };
    }),
    on(dessertEvents.loadDessertsSuccess, ({ payload }) => {
      return {
        desserts: payload.desserts,
        loading: false,
      };
    }),
    on(dessertEvents.loadDessertsError, ({ payload }) => {
      return {
        error: payload.error,
        loading: false,
      };
    }),
  ),
  […]
);
```

### Effects

Эффекты (Effects) в Event API — это не то же самое, что одноимённое понятие из мира Signals. Технически это пайпы на базе RxJS, которые отображают входящие события в исходящие. Для этого они обычно выполняют асинхронную операцию через какой-нибудь «уплощающий» (flattening) оператор — например, _switchMap_ или _mergeMap_:

```javascript
import { Events, withEffects } from '@ngrx/signals/events';
import { mapResponse } from '@ngrx/operators';

[…]

export const DessertStore = signalStore(
  […],
  withProps(() => ({
    _dessertService: inject(DessertService),
    _toastService: inject(ToastService),
    _events: inject(Events),
  })),
  […]
  withEffects((store) => ({
    loadDesserts$: store._events.on(dessertEvents.loadDesserts).pipe(
      switchMap((e) =>
        store._dessertService.find(e.payload).pipe(
          mapResponse({
            next: (desserts) => dessertEvents.loadDessertsSuccess({ desserts }),
            error: (error) =>
              dessertEvents.loadDessertsError({ error: String(error) }),
          }),
        ),
      ),
    ),
  })),
);
```

Хотя осмысленное имя — например, _loadDesserts$_ в нашем примере — важно для читаемости, технически оно ни на что не влияет: Event API запускает Effect не по имени, а по событиям. Хранилище создаёт эффекты через сервис _Event_. Его метод _on_ фильтрует полученные события и возвращает их через _Observable_. Если хранилище не передаёт в _on_ никаких параметров, оно получает все события. Если же передать один или несколько типов событий, _on_ вернёт только подходящие.

Чтобы превратить результат запущенного side-эффекта в исходящее событие, удобно воспользоваться оператором _mapResponse_, который поставляется вместе с NgRx. По сути это комбинация _map_ и _catchError_: он позволяет преобразовывать как результаты, так и ошибки.

Как и в ранние дни NgRx Global Store, здесь критически важно обрабатывать ошибки — например, через _catchError_ или, как показано, через _mapResponse_. Иначе RxJS закроет observable, и дальнейшие события Effect обрабатывать уже не будет. Как обычно, ошибки нужно обрабатывать на том уровне, где они возникли. В нашем случае это внутри _switchMap_, что приводит к дополнительной вложенности.

Опции вроде _resubscribeOnError_ из Global Store, которая автоматически компенсирует ошибки, на данный момент нет. Но команда NgRx рассматривает её добавление.

### Потребление хранилища

Чтобы работать с хранилищем, отдельные компоненты могут получить Dispatcher через инъекцию (injection) и с его помощью инициировать нужное событие:

```javascript
this.#dispatcher.dispatch(
  dessertEvents.loadDesserts({
    originalName: this.originalName(),
    englishName: this.englishName(),
  }),
);
```

## Бонус: Redux DevTools

Signal Store не идёт в комплекте с официальной интеграцией с [Redux Dev Tools](https://chromewebstore.google.com/detail/redux-devtools/lmhkpmbekcpmknklioeibfkpmmfibljd), которые позволяют заглянуть внутрь хранилища во время отладки и заодно показывают историю отдельных изменений состояния. Тем не менее эту возможность можно добавить, воспользовавшись механизмом расширяемости Signal Store.

Реализацию этой идеи можно найти в community-пакете @angular-architects/ngrx-toolkit. Он предоставляет фичу _withDevtools_, которую можно добавить в хранилище:

```javascript
import { withDevtools } from '@angular-architects/ngrx-toolkit';
[…]

export const DessertStore = signalStore(
  […]
  withDevtools('DessertStore')
);
```

Переданная строка используется как имя узла, который представляет соответствующее хранилище в Dev Tools. Если установить Dev Tools как браузерное расширение, то данные, хранящиеся в store, и их историю можно будет смотреть в Developer Console:

## Итоги

Новый Event API для NgRx Signal Store позволяет точечно реализовывать паттерны Flux там, где это нужно. А вместе с дополнительными инструментами вроде Redux DevTools он заметно упрощает отладку. Event API становится гибким мостиком между легковесным использованием Signal Store и проверенными практиками управления состоянием, знакомыми по миру Redux.
