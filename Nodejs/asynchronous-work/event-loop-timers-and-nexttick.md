# Цикл событий Node.js

## Что такое цикл событий?

Цикл событий — это то, что позволяет Node.js выполнять неблокирующие операции ввода-вывода (I/O), несмотря на то, что по умолчанию используется единственный поток выполнения JavaScript, — за счёт передачи операций системному ядру всякий раз, когда это возможно.

Поскольку большинство современных ядер многопоточны, они могут обрабатывать несколько операций, выполняющихся в фоновом режиме. Когда одна из этих операций завершается, ядро сообщает об этом Node.js, чтобы соответствующий колбэк можно было добавить в очередь **poll** и в конечном итоге выполнить. Мы объясним это подробнее далее в этом разделе.

## Цикл событий: объяснение

Когда Node.js запускается, он инициализирует цикл событий, обрабатывает предоставленный входной скрипт (или переходит в [REPL](https://nodejs.org/api/repl.html#repl_repl), который в этом документе не рассматривается), который может выполнять асинхронные вызовы API, планировать таймеры или вызывать `process.nextTick()`, после чего начинает обработку цикла событий.

На следующей схеме показан упрощённый обзор порядка операций цикла событий.
```
   ┌───────────────────────────┐
   │           timers          │
   └─────────────┬─────────────┘
                 │
                 v
   ┌───────────────────────────┐
┌─>│     pending callbacks     │
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │       idle, prepare       │
│  └─────────────┬─────────────┘      ┌───────────────┐
│  ┌─────────────┴─────────────┐      │   incoming:   │
│  │           poll            │<─────┤  connections, │
│  └─────────────┬─────────────┘      │   data, etc.  │
│  ┌─────────────┴─────────────┐      └───────────────┘
│  │           check           │
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │      close callbacks      │
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
└──┤           timers          │
   └───────────────────────────┘
```

> Каждый блок мы будем называть «фазой» цикла событий.

У каждой фазы есть FIFO-очередь колбэков, которые нужно выполнить. Хотя каждая фаза по-своему особенна, в общем случае, когда цикл событий входит в определённую фазу, он выполняет любые операции, специфичные для этой фазы, а затем выполняет колбэки из очереди этой фазы до тех пор, пока очередь не будет исчерпана или пока не будет выполнено максимальное число колбэков. Когда очередь исчерпана или достигнут лимит колбэков, цикл событий переходит к следующей фазе, и так далее.

Поскольку любая из этих операций может запланировать _ещё_ операции, а новые события, обрабатываемые в фазе **poll**, ставятся в очередь ядром, события poll могут добавляться в очередь во время обработки уже имеющихся событий poll. В результате долго выполняющиеся колбэки могут заставить фазу poll работать значительно дольше, чем порог таймера. Подробности смотрите в разделах о фазах **timers** и **poll**.

> Между реализациями для Windows и Unix/Linux есть небольшое расхождение, но для данной демонстрации это несущественно. Самые важные части приведены здесь. На самом деле шагов семь или восемь, но те, что нас интересуют — те, которые Node.js действительно использует, — перечислены выше.

## Обзор фаз

  * **pending callbacks** : выполняет колбэки ввода-вывода, отложенные на следующую итерацию цикла.
  * **idle, prepare** : используются только внутренне.
  * **poll** : получает новые события ввода-вывода; выполняет связанные с вводом-выводом колбэки (почти все, за исключением колбэков закрытия, тех, что запланированы таймерами, и `setImmediate()`); здесь node будет блокироваться, когда это уместно.
  * **check** : здесь вызываются колбэки `setImmediate()`.
  * **close callbacks** : некоторые колбэки закрытия, например `socket.on('close', ...)`.
  * **timers** : эта фаза выполняет колбэки, запланированные с помощью `setTimeout()` и `setInterval()`. Кроме того, эти колбэки могут выполниться перед входом в цикл событий. Иногда это происходит для setTimeout(() => { ... }, 0) вне цикла ввода-вывода.

Между каждым проходом цикла событий Node.js проверяет, ожидает ли он какого-либо асинхронного ввода-вывода или таймеров, и корректно завершает работу, если таковых нет.

Начиная с libuv 1.45.0 (Node.js 20) поведение цикла событий изменилось: таймеры теперь выполняются только после фазы **poll**, а не до и после неё, как в более ранних версиях. Это изменение может влиять на моменты выполнения колбэков `setImmediate()` и на то, как они взаимодействуют с таймерами в определённых сценариях.

## Фазы подробно

### pending callbacks

Эта фаза выполняет колбэки для некоторых системных операций, таких как определённые типы ошибок TCP. Например, если TCP-сокет получает `ECONNREFUSED` при попытке подключения, некоторые *nix-системы предпочитают подождать перед тем, как сообщить об ошибке. Это будет поставлено в очередь для выполнения в фазе **pending callbacks**.

### poll

У фазы **poll** есть две основные функции:

  1. Вычисление того, как долго ей следует блокироваться и опрашивать ввод-вывод, а затем
  2. Обработка событий в очереди **poll**.

Когда цикл событий входит в фазу **poll** _и нет запланированных таймеров_, произойдёт одно из двух:

  * _Если очередь**poll** **не пуста**_, цикл событий будет перебирать свою очередь колбэков, синхронно выполняя их, пока либо очередь не будет исчерпана, либо не будет достигнут зависящий от системы жёсткий лимит.

  * _Если очередь**poll** **пуста**_, произойдёт одно из ещё двух событий:

    * Если были запланированы скрипты с помощью `setImmediate()`, цикл событий завершит фазу **poll** и продолжит работу в фазе **check**, чтобы выполнить эти запланированные скрипты.

    * Если скрипты **не** были запланированы с помощью `setImmediate()`, цикл событий будет ждать, пока колбэки не будут добавлены в очередь, а затем немедленно выполнит их.

Как только очередь **poll** опустеет, цикл событий проверит таймеры, _чьи временные пороги были достигнуты_. Если один или несколько таймеров готовы, цикл событий вернётся к фазе **timers**, чтобы выполнить колбэки этих таймеров.

### check

Эта фаза позволяет циклу событий выполнять колбэки сразу после завершения фазы **poll**. Если фаза **poll** становится простаивающей и были поставлены в очередь скрипты с помощью `setImmediate()`, цикл событий может перейти к фазе **check**, вместо того чтобы ждать.

`setImmediate()` на самом деле является особым таймером, работающим в отдельной фазе цикла событий. Он использует API libuv, который планирует выполнение колбэков после завершения фазы **poll**.

Как правило, по мере выполнения кода цикл событий в конечном итоге доходит до фазы **poll**, где он ждёт входящего подключения, запроса и т. п. Однако если колбэк был запланирован с помощью `setImmediate()`, а фаза **poll** становится простаивающей, она завершится и продолжит работу в фазе **check**, вместо того чтобы ждать событий **poll**.

### close callbacks

Если сокет или дескриптор закрывается внезапно (например, `socket.destroy()`), событие `'close'` будет сгенерировано в этой фазе. В противном случае оно будет сгенерировано через `process.nextTick()`.

### timers

Таймер задаёт **порог**, _по истечении которого_ предоставленный колбэк _может быть выполнен_, а не **точное** время, в которое человек _хочет, чтобы он был выполнен_. Колбэки таймеров будут выполнены настолько рано, насколько их можно запланировать после того, как прошёл заданный промежуток времени; однако планировщик операционной системы или выполнение других колбэков могут их задержать.

> Технически именно фаза **poll** управляет тем, когда выполняются таймеры.

Например, предположим, вы запланировали тайм-аут для выполнения по истечении порога в 100 мс, после чего ваш скрипт начинает асинхронно читать файл, что занимает 95 мс:
```js
const fs = require('node:fs');

function someAsyncOperation(callback) {
  // Предположим, что это занимает 95 мс
  fs.readFile('/path/to/file', callback);
}

const timeoutScheduled = Date.now();

setTimeout(() => {
  const delay = Date.now() - timeoutScheduled;

  console.log(`${delay}ms have passed since I was scheduled`);
}, 100);

// выполняем someAsyncOperation, что занимает 95 мс
someAsyncOperation(() => {
  const startCallback = Date.now();

  // делаем что-то, что займёт 10 мс...
  while (Date.now() - startCallback < 10) {
    // ничего не делаем
  }
});
```

Когда цикл событий входит в фазу **poll**, его очередь пуста (`fs.readFile()` ещё не завершился), поэтому он будет ждать столько миллисекунд, сколько осталось до достижения порога ближайшего таймера. Пока он ждёт, проходит 95 мс, `fs.readFile()` завершает чтение файла, и его колбэк, выполнение которого занимает 10 мс, добавляется в очередь **poll** и выполняется. Когда колбэк завершится, в очереди больше не будет колбэков, поэтому цикл событий увидит, что порог ближайшего таймера был достигнут, и вернётся к фазе **timers**, чтобы выполнить колбэк таймера. В этом примере вы увидите, что общая задержка между планированием таймера и выполнением его колбэка составит 105 мс.

> Чтобы фаза **poll** не «морила голодом» цикл событий, [libuv](https://libuv.org/) (библиотека на C, реализующая цикл событий Node.js и всё асинхронное поведение платформы) также имеет жёсткий максимум (зависящий от системы), по достижении которого она прекращает опрашивать новые события.

## `setImmediate()` против `setTimeout()`

`setImmediate()` и `setTimeout()` похожи, но ведут себя по-разному в зависимости от того, когда они вызываются.

  * `setImmediate()` предназначен для выполнения скрипта после завершения текущей фазы **poll**.
  * `setTimeout()` планирует запуск скрипта после того, как истечёт минимальный порог в миллисекундах.

Порядок, в котором выполняются таймеры, будет варьироваться в зависимости от контекста, в котором они вызываются. Если оба вызываются из главного модуля, то тайминг будет ограничен производительностью процесса (на которую могут влиять другие приложения, работающие на машине).

Например, если мы запустим следующий скрипт, который находится не внутри цикла ввода-вывода (то есть в главном модуле), порядок, в котором выполняются два таймера, недетерминирован, так как он ограничен производительностью процесса:
```js
// timeout_vs_immediate.js
setTimeout(() => {
  console.log('timeout');
}, 0);

setImmediate(() => {
  console.log('immediate');
});
```

Однако если вы переместите эти два вызова внутрь цикла ввода-вывода, колбэк immediate всегда выполняется первым:
```js
// timeout_vs_immediate.js
const fs = require('node:fs');

fs.readFile(__filename, () => {
  setTimeout(() => {
    console.log('timeout');
  }, 0);
  setImmediate(() => {
    console.log('immediate');
  });
});
```

Главное преимущество использования `setImmediate()` вместо `setTimeout()` в том, что `setImmediate()` всегда будет выполнен раньше любых таймеров, если он запланирован внутри цикла ввода-вывода, независимо от того, сколько таймеров присутствует.

### Понимание `process.nextTick()`

Возможно, вы заметили, что `process.nextTick()` не отображён на схеме, хотя он является частью асинхронного API. Это потому, что `process.nextTick()` технически не является частью цикла событий. Вместо этого `nextTickQueue` будет обработана после завершения текущей операции, независимо от текущей фазы цикла событий. Здесь под _операцией_ понимается переход от нижележащего обработчика на C/C++ к обработке JavaScript, который должен быть выполнен.

Возвращаясь к нашей схеме: каждый раз, когда вы вызываете `process.nextTick()` в данной фазе, все колбэки, переданные в `process.nextTick()`, будут разрешены прежде, чем цикл событий продолжит работу. Это может создать некоторые нехорошие ситуации, потому что **это позволяет вам «уморить голодом» ваш ввод-вывод, делая рекурсивные вызовы`process.nextTick()`**, что не даёт циклу событий добраться до фазы **poll**.

### Почему это вообще разрешено?

Зачем что-то подобное было включено в Node.js? Отчасти это философия проектирования, согласно которой API всегда должен быть асинхронным, даже там, где он не обязан быть таким. Возьмём для примера такой фрагмент кода:
```js
function apiCall(arg, callback) {
  if (typeof arg !== 'string') {
    return process.nextTick(
      callback,
      new TypeError('argument should be string')
    );
  }
}
```

Этот фрагмент проверяет аргумент, и, если он некорректен, передаёт ошибку в колбэк. API был обновлён относительно недавно, чтобы можно было передавать аргументы в `process.nextTick()`: теперь он может принимать любые аргументы, переданные после колбэка, и передавать их как аргументы колбэку, чтобы вам не приходилось вкладывать функции друг в друга.

Мы передаём ошибку обратно пользователю, но только _после_ того, как позволили выполниться остальному коду пользователя. Используя `process.nextTick()`, мы гарантируем, что `apiCall()` всегда запускает свой колбэк _после_ остального кода пользователя и _до_ того, как циклу событий будет позволено продолжить работу. Чтобы этого добиться, стеку вызовов JS позволяется размотаться, а затем немедленно выполнить предоставленный колбэк, что даёт возможность делать рекурсивные вызовы `process.nextTick()`, не достигая `RangeError: Maximum call stack size exceeded` от v8.

Эта философия может приводить к потенциально проблемным ситуациям. Возьмём для примера такой фрагмент:
```js
let bar = null;

// у этого асинхронная сигнатура, но колбэк вызывается синхронно
function someAsyncApiCall(callback) {
  callback();
}

// колбэк вызывается до того, как `someAsyncApiCall` завершится.
someAsyncApiCall(() => {
  // поскольку someAsyncApiCall ещё не завершился, bar не присвоено никакого значения
  console.log('bar', bar); // null
});

bar = 1;
```

Пользователь определяет `someAsyncApiCall()` с асинхронной сигнатурой, но на самом деле функция работает синхронно. Когда она вызывается, колбэк, переданный в `someAsyncApiCall()`, вызывается в той же фазе цикла событий, потому что `someAsyncApiCall()` на самом деле не делает ничего асинхронного. В результате колбэк пытается обратиться к `bar`, хотя этой переменной, возможно, ещё нет в области видимости, потому что скрипт не смог отработать до конца.

Поместив колбэк в `process.nextTick()`, мы даём скрипту возможность отработать до конца, что позволяет инициализировать все переменные, функции и т. д. прежде, чем будет вызван колбэк. У этого также есть преимущество: цикл событий не сможет продолжить работу. Возможно, пользователю будет полезно узнать об ошибке до того, как циклу событий позволят продолжить. Вот предыдущий пример с использованием `process.nextTick()`:
```js
let bar = null;

function someAsyncApiCall(callback) {
  process.nextTick(callback);
}

someAsyncApiCall(() => {
  console.log('bar', bar); // 1
});

bar = 1;
```

Вот ещё один пример из реальной жизни:
```js
const server = net.createServer(() => {}).listen(8080);

server.on('listening', () => {});
```

Когда передаётся только порт, привязка к порту происходит немедленно. Так что колбэк `'listening'` мог бы быть вызван немедленно. Проблема в том, что к этому моменту колбэк `.on('listening')` ещё не будет установлен.

Чтобы обойти это, событие `'listening'` ставится в очередь через `nextTick()`, чтобы дать скрипту отработать до конца. Это позволяет пользователю установить любые обработчики событий, какие он захочет.

## `process.nextTick()` против `setImmediate()`

У нас есть два вызова, которые похожи с точки зрения пользователей, но их имена сбивают с толку.

  * `process.nextTick()` срабатывает немедленно в той же фазе
  * `setImmediate()` срабатывает на следующей итерации, или «тике», цикла событий

По сути, имена следовало бы поменять местами. `process.nextTick()` срабатывает более немедленно, чем `setImmediate()`, но это артефакт прошлого, который вряд ли изменится. Такая замена сломала бы значительный процент пакетов на npm. Каждый день добавляется всё больше новых модулей, а значит, каждый день ожидания увеличивает число потенциальных поломок. Хотя имена и сбивают с толку, сами имена меняться не будут.

> Мы рекомендуем разработчикам использовать `setImmediate()` во всех случаях, потому что о нём проще рассуждать.

## Зачем использовать `process.nextTick()`?

Есть две основные причины:

  1. Позволить пользователям обработать ошибки, освободить ставшие ненужными ресурсы или, возможно, повторить запрос до того, как цикл событий продолжит работу.

  2. Иногда необходимо позволить колбэку выполниться после того, как стек вызовов размотался, но до того, как цикл событий продолжит работу.

Один из примеров — соответствие ожиданиям пользователя. Простой пример:
```js
const server = net.createServer();
server.on('connection', conn => {});

server.listen(8080);
server.on('listening', () => {});
```

Предположим, что `listen()` запускается в начале цикла событий, но колбэк listening помещён в `setImmediate()`. Если не передано имя хоста, привязка к порту произойдёт немедленно. Чтобы цикл событий продолжил работу, он должен дойти до фазы **poll**, а значит, существует ненулевая вероятность, что соединение могло быть принято, что позволит событию connection сработать раньше события listening.

Ещё один пример — расширение `EventEmitter` и генерация события изнутри конструктора:
```js
const EventEmitter = require('node:events');

class MyEmitter extends EventEmitter {
  constructor() {
    super();
    this.emit('event');
  }
}

const myEmitter = new MyEmitter();
myEmitter.on('event', () => {
  console.log('an event occurred!');
});
```

Вы не можете сгенерировать событие из конструктора немедленно,

потому что скрипт ещё не дойдёт до того момента, где пользователь назначит колбэк этому событию. Поэтому внутри самого конструктора вы можете использовать `process.nextTick()`, чтобы установить колбэк, который сгенерирует событие после того, как конструктор завершится, что даёт ожидаемый результат:
```js
const EventEmitter = require('node:events');

class MyEmitter extends EventEmitter {
  constructor() {
    super();

    // используем nextTick, чтобы сгенерировать событие после того, как назначен обработчик
    process.nextTick(() => {
      this.emit('event');
    });
  }
}

const myEmitter = new MyEmitter();
myEmitter.on('event', () => {
  console.log('an event occurred!');
});
```
