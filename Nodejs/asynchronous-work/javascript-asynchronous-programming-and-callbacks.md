# Асинхронное программирование на JavaScript и колбэки

## Асинхронность в языках программирования

Компьютеры асинхронны по своей природе.

Асинхронность означает, что события могут происходить независимо от основного потока выполнения программы.

В современных пользовательских компьютерах каждая программа выполняется в течение определённого промежутка времени, после чего останавливает своё выполнение, чтобы дать возможность продолжить работу другой программе. Этот процесс происходит по кругу настолько быстро, что заметить его невозможно. Нам кажется, что наши компьютеры выполняют множество программ одновременно, но это иллюзия (за исключением многопроцессорных машин).

Внутри программы используют _прерывания (interrupts)_ — сигнал, который посылается процессору, чтобы привлечь внимание системы.

Не будем сейчас углубляться в детали этого механизма, просто держите в уме, что для программ нормально быть асинхронными и приостанавливать своё выполнение до тех пор, пока им не потребуется внимание, позволяя компьютеру в это время выполнять другие задачи. Когда программа ожидает ответа от сети, она не может заблокировать процессор, пока запрос не завершится.

Обычно языки программирования синхронны, и некоторые из них предоставляют способ управлять асинхронностью средствами языка или через библиотеки. C, Java, C#, PHP, Go, Ruby, Swift и Python по умолчанию синхронны. Некоторые из них обрабатывают асинхронные операции с помощью потоков выполнения, порождая новый процесс.

## JavaScript

JavaScript по умолчанию **синхронен** и однопоточен. Это означает, что код не может создавать новые потоки выполнения и работать параллельно.

Строки кода выполняются последовательно, одна за другой, например:
```js
const a = 1;
const b = 2;
const c = a * b;
console.log(c);
doSomething();
```

Но JavaScript родился внутри браузера, и его главной задачей в самом начале было реагировать на действия пользователя, такие как `onClick`, `onMouseOver`, `onChange`, `onSubmit` и так далее. Как же он мог делать это при синхронной модели программирования?

Ответ кроется в его окружении. **Браузер** предоставляет способ сделать это, предоставляя набор API, способных обрабатывать такого рода функциональность.

Позднее Node.js представил неблокирующее окружение ввода-вывода (I/O), чтобы распространить эту концепцию на доступ к файлам, сетевые вызовы и так далее.

## Колбэки

Вы не можете знать, когда пользователь собирается нажать кнопку. Поэтому вы **определяете обработчик события для события click**. Этот обработчик события принимает функцию, которая будет вызвана, когда событие произойдёт:
```js
document.getElementById('button').addEventListener('click', () => {
  // элемент нажат
});
```

Это так называемый **колбэк**.

Колбэк — это обычная функция, которая передаётся как значение в другую функцию и будет выполнена только тогда, когда произойдёт событие. Мы можем делать это, потому что в JavaScript функции являются объектами первого класса: их можно присваивать переменным и передавать в другие функции (называемые **функциями высшего порядка**).

Часто весь клиентский код оборачивают в слушатель события `load` на объекте `window`, который запускает функцию-колбэк только тогда, когда страница готова:
```js
window.addEventListener('load', () => {
  // окно загружено
  // делайте что угодно
});
```

Колбэки используются повсюду, не только в событиях DOM.

Один из распространённых примеров — использование таймеров:
```js
setTimeout(() => {
  // выполнится через 2 секунды
}, 2000);
```

XHR-запросы также принимают колбэк; в этом примере — путём присваивания функции свойству, которое будет вызвано, когда произойдёт определённое событие (в данном случае — изменение состояния запроса):
```js
const xhr = new XMLHttpRequest();
xhr.onreadystatechange = () => {
  if (xhr.readyState === 4) {
    if (xhr.status === 200) {
      console.log(xhr.responseText);
    } else {
      console.error('error');
    }
  }
};
xhr.open('GET', 'https://yoursite.com');
xhr.send();
```

### Обработка ошибок в колбэках

Как обрабатывать ошибки при использовании колбэков? Одна очень распространённая стратегия — та, которую взял на вооружение Node.js: первым параметром любой функции-колбэка является объект ошибки — **колбэки с ошибкой на первом месте (error-first callbacks)**.

Если ошибки нет, этот объект равен `null`. Если ошибка есть, он содержит некоторое описание ошибки и другую информацию.
```js
const fs = require('node:fs');

fs.readFile('/file.json', (err, data) => {
  if (err) {
    // обработка ошибки
    console.log(err);
    return;
  }

  // ошибок нет, обрабатываем данные
  console.log(data);
});
```

### Проблема с колбэками

Колбэки прекрасно подходят для простых случаев!

Однако каждый колбэк добавляет уровень вложенности, и когда у вас много колбэков, код очень быстро начинает усложняться:
```js
window.addEventListener('load', () => {
  document.getElementById('button').addEventListener('click', () => {
    setTimeout(() => {
      items.forEach(item => {
        // ваш код здесь
      });
    }, 2000);
  });
});
```

Это всего лишь простой код с 4 уровнями вложенности, но мне доводилось видеть гораздо больше уровней вложенности, и это не доставляет удовольствия.
