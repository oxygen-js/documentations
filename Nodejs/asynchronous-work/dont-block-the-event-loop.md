# Не блокируйте цикл событий (и пул рабочих потоков)

## Стоит ли вам читать это руководство?

Если вы пишете что-то более сложное, чем небольшой сценарий для командной строки, это руководство поможет вам создавать более производительные и более безопасные приложения.

Этот документ написан с прицелом на серверы Node.js, но изложенные концепции применимы и к сложным приложениям Node.js в целом. Там, где детали зависят от операционной системы, изложение ориентировано на Linux.

## Кратко

Node.js выполняет код на JavaScript в цикле событий (инициализация и колбэки) и предоставляет пул рабочих потоков (Worker Pool) для обработки затратных задач вроде файлового I/O. Node.js хорошо масштабируется — порой лучше, чем более тяжеловесные подходы наподобие Apache. Секрет масштабируемости Node.js в том, что она использует небольшое число потоков выполнения для обслуживания множества клиентов. Если Node.js может обойтись меньшим числом потоков, то она тратит больше времени и памяти вашей системы на работу с клиентами, а не на накладные расходы по времени и памяти для потоков (память, переключение контекста). Но поскольку у Node.js всего несколько потоков, вы должны выстраивать приложение так, чтобы использовать их разумно.

Вот хорошее эмпирическое правило, помогающее держать ваш сервер Node.js быстрым: _Node.js работает быстро, когда объём работы, связанной с каждым клиентом в любой момент времени, «мал»_.

Это относится и к колбэкам в цикле событий, и к задачам в пуле рабочих потоков.

## Почему мне стоит избегать блокировки цикла событий и пула рабочих потоков?

Node.js использует небольшое число потоков выполнения для обслуживания множества клиентов. В Node.js есть два типа потоков: один цикл событий (он же главный цикл, главный поток, поток событий и т. д.) и пул из `k` рабочих потоков (Worker) в пуле рабочих потоков (он же threadpool).

Если поток выполнения тратит много времени на выполнение колбэка (цикл событий) или задачи (рабочий поток), мы говорим, что он «заблокирован». Пока поток заблокирован работой в интересах одного клиента, он не может обрабатывать запросы других клиентов. Отсюда две причины не блокировать ни цикл событий, ни пул рабочих потоков:

  1. Производительность. Если вы регулярно выполняете тяжеловесную работу на потоке любого из двух типов, _пропускная способность_ (запросов в секунду) вашего сервера пострадает.
  2. Безопасность. Если при определённых входных данных один из ваших потоков может заблокироваться, злоумышленник способен подать такой «злонамеренный ввод», заставить ваши потоки заблокироваться и не давать им работать с другими клиентами. Это была бы атака типа [отказ в обслуживании](https://en.wikipedia.org/wiki/Denial-of-service_attack) (Denial of Service).

## Краткий обзор Node

Node.js использует событийно-ориентированную архитектуру: у неё есть цикл событий для координации и пул рабочих потоков для затратных задач.

### Какой код выполняется в цикле событий?

При запуске приложения Node.js сначала проходят фазу инициализации: подключают модули через `require` и регистрируют колбэки для событий. Затем приложения Node.js входят в цикл событий, отвечая на поступающие клиентские запросы выполнением соответствующего колбэка. Этот колбэк выполняется синхронно и может зарегистрировать асинхронные запросы, чтобы продолжить обработку после своего завершения. Колбэки этих асинхронных запросов также будут выполнены в цикле событий.

Цикл событий также обслуживает неблокирующие асинхронные запросы, сделанные его колбэками, например сетевой I/O.

Итак, цикл событий выполняет колбэки на JavaScript, зарегистрированные для событий, а также отвечает за обслуживание неблокирующих асинхронных запросов вроде сетевого I/O.

### Какой код выполняется в пуле рабочих потоков?

Пул рабочих потоков Node.js реализован в libuv ([документация](http://docs.libuv.org/en/v1.x/threadpool.html)), который предоставляет универсальный API для отправки задач.

Node.js использует пул рабочих потоков для обработки «затратных» задач. Сюда входит I/O, для которого операционная система не предоставляет неблокирующего варианта, а также особенно ресурсоёмкие для CPU задачи.

Вот те API-модули Node.js, которые задействуют этот пул рабочих потоков:

  1. Интенсивные по I/O
    1. [DNS](https://nodejs.org/api/dns.html): `dns.lookup()`, `dns.lookupService()`.
    2. [Файловая система](https://nodejs.org/api/fs.html#fs_threadpool_usage): все API файловой системы, кроме `fs.FSWatcher()` и тех, что явно синхронны, используют пул потоков libuv.
  2. Интенсивные по CPU
    1. [Crypto](https://nodejs.org/api/crypto.html): `crypto.pbkdf2()`, `crypto.scrypt()`, `crypto.randomBytes()`, `crypto.randomFill()`, `crypto.generateKeyPair()`.
    2. [Zlib](https://nodejs.org/api/zlib.html#zlib_threadpool_usage): все API zlib, кроме явно синхронных, используют пул потоков libuv.

Во многих приложениях Node.js эти API — единственный источник задач для пула рабочих потоков. Приложения и модули, использующие [аддон на C++](https://nodejs.org/api/addons.html), могут отправлять в пул рабочих потоков и другие задачи.

Для полноты отметим: когда вы вызываете один из этих API из колбэка в цикле событий, цикл событий несёт небольшие затраты на подготовку, входя в C++-привязки Node.js для этого API и отправляя задачу в пул рабочих потоков. Эти затраты ничтожны по сравнению с общей стоимостью задачи, ради чего цикл событий её и делегирует. Отправляя такую задачу в пул рабочих потоков, Node.js передаёт указатель на соответствующую C++-функцию в C++-привязках Node.js.

### Как Node.js решает, какой код выполнять следующим?

Абстрактно можно считать, что цикл событий и пул рабочих потоков ведут очереди — соответственно для ожидающих событий и ожидающих задач.

На самом деле цикл событий не ведёт настоящую очередь. Вместо этого у него есть набор файловых дескрипторов, за которыми он просит следить операционную систему при помощи механизма вроде [epoll](http://man7.org/linux/man-pages/man7/epoll.7.html) (Linux), [kqueue](https://developer.apple.com/library/content/documentation/Darwin/Conceptual/FSEvents_ProgGuide/KernelQueues/KernelQueues.html) (OSX), event ports (Solaris) или [IOCP](https://msdn.microsoft.com/en-us/library/windows/desktop/aa365198.aspx) (Windows). Эти файловые дескрипторы соответствуют сетевым сокетам, любым отслеживаемым файлам и так далее. Когда операционная система сообщает, что один из этих файловых дескрипторов готов, цикл событий преобразует это в соответствующее событие и вызывает связанный с ним колбэк (или колбэки). Подробнее об этом процессе можно узнать [здесь](https://www.youtube.com/watch?v=P9csgxBgaZ8).

В отличие от этого, пул рабочих потоков использует настоящую очередь, элементы которой — задачи для обработки. Рабочий поток извлекает задачу из этой очереди и работает над ней, а по завершении поднимает для цикла событий событие «Как минимум одна задача завершена».

### Что это значит для проектирования приложения?

В системе с одним потоком на клиента, как Apache, каждому ожидающему клиенту назначается собственный поток. Если поток, обслуживающий одного клиента, блокируется, операционная система прерывает его и передаёт очередь другому клиенту. Тем самым операционная система гарантирует, что клиенты, требующие небольшого объёма работы, не будут наказаны из-за клиентов, которым нужно больше работы.

Поскольку Node.js обслуживает множество клиентов немногими потоками, то, если поток блокируется при обработке запроса одного клиента, ожидающие клиентские запросы могут не получить очереди, пока поток не завершит свой колбэк или задачу. _Таким образом, справедливое отношение к клиентам — ответственность вашего приложения_. Это значит, что вам не следует выполнять слишком много работы для какого-либо клиента в рамках одного колбэка или задачи.

Отчасти именно поэтому Node.js хорошо масштабируется, но это также означает, что за справедливое планирование отвечаете вы. В следующих разделах рассказывается, как обеспечить справедливое планирование для цикла событий и для пула рабочих потоков.

## Не блокируйте цикл событий

Цикл событий замечает каждое новое клиентское подключение и координирует формирование ответа. Все входящие запросы и исходящие ответы проходят через цикл событий. Это значит, что если цикл событий где-либо задерживается слишком надолго, ни текущие, ни новые клиенты не получат очереди.

Вы должны убедиться, что никогда не блокируете цикл событий. Иными словами, каждый ваш колбэк на JavaScript должен завершаться быстро. Это, разумеется, относится и к вашим `await`, вашим `Promise.then` и так далее.

Хороший способ этого добиться — рассуждать о [«вычислительной сложности»](https://en.wikipedia.org/wiki/Time_complexity) ваших колбэков. Если ваш колбэк выполняет постоянное число шагов независимо от аргументов, то вы всегда предоставите каждому ожидающему клиенту справедливую очередь. Если же ваш колбэк выполняет разное число шагов в зависимости от аргументов, то стоит задуматься, насколько длинными могут быть эти аргументы.

Пример 1. Колбэк с постоянным временем.
```js
app.get('/constant-time', (req, res) => {
  res.sendStatus(200);
});
```

Пример 2. Колбэк со сложностью `O(n)`. Этот колбэк выполнится быстро при малых `n` и медленнее при больших `n`.
```js
app.get('/countToN', (req, res) => {
  const n = req.query.n;

  // n итераций, прежде чем передать очередь кому-то другому
  for (let i = 0; i < n; i++) {
    console.log(`Iter ${i}`);
  }

  res.sendStatus(200);
});
```

Пример 3. Колбэк со сложностью `O(n^2)`. Этот колбэк всё ещё выполнится быстро при малых `n`, но при больших `n` он будет работать намного медленнее, чем предыдущий пример с `O(n)`.
```js
app.get('/countToN2', (req, res) => {
  const n = req.query.n;

  // n^2 итераций, прежде чем передать очередь кому-то другому
  for (let i = 0; i < n; i++) {
    for (let j = 0; j < n; j++) {
      console.log(`Iter ${i}.${j}`);
    }
  }

  res.sendStatus(200);
});
```

### Насколько осторожным нужно быть?

Node.js использует для JavaScript движок Google V8, который весьма быстр для многих распространённых операций. Исключения из этого правила — регулярные выражения и операции с JSON, о которых пойдёт речь ниже.

Тем не менее для сложных задач стоит подумать об ограничении входных данных и отклонении слишком длинного ввода. Так, даже если у вашего колбэка большая сложность, ограничивая ввод, вы гарантируете, что колбэк не займёт больше времени наихудшего случая на самом длинном допустимом вводе. После этого вы можете оценить стоимость наихудшего случая для этого колбэка и определить, приемлемо ли его время выполнения в вашем контексте.

### Блокировка цикла событий: REDOS

Один распространённый способ катастрофически заблокировать цикл событий — использование «уязвимого» [регулярного выражения](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Regular_Expressions).

#### Как избегать уязвимых регулярных выражений

Регулярное выражение (regexp) сопоставляет входную строку с шаблоном. Обычно мы думаем, что сопоставление regexp требует одного прохода по входной строке — время `O(n)`, где `n` — длина входной строки. Во многих случаях одного прохода действительно достаточно. К сожалению, в некоторых случаях сопоставление regexp может потребовать экспоненциального числа проходов по входной строке — время `O(2^n)`. Экспоненциальное число проходов означает, что если движку нужно `x` проходов, чтобы определить совпадение, то ему понадобится `2*x` проходов, если мы добавим ко входной строке всего один символ. Поскольку число проходов линейно связано с необходимым временем, результатом такого вычисления станет блокировка цикла событий.

_Уязвимое регулярное выражение_ — это выражение, на котором ваш движок регулярных выражений может потратить экспоненциальное время, подвергая вас риску [REDOS](https://owasp.org/www-community/attacks/Regular_expression_Denial_of_Service_-_ReDoS) на «злонамеренном вводе». Вопрос о том, уязвим ли ваш шаблон регулярного выражения (то есть может ли движок regexp потратить на нём экспоненциальное время), на самом деле сложен, и ответ различается в зависимости от того, используете ли вы Perl, Python, Ruby, Java, JavaScript и т. д. Однако вот несколько эмпирических правил, применимых ко всем этим языкам:

  1. Избегайте вложенных квантификаторов вроде `(a+)*`. Движок regexp в V8 некоторые из них обрабатывает быстро, но другие уязвимы.
  2. Избегайте OR с перекрывающимися ветвями, например `(a|a)*`. Опять же, они лишь иногда быстры.
  3. Избегайте использования обратных ссылок, например `(a.*) \1`. Ни один движок regexp не может гарантировать вычисление этого за линейное время.
  4. Если вам нужно простое сопоставление строк, используйте `indexOf` или локальный эквивалент. Это дешевле и никогда не займёт больше `O(n)`.

Если вы не уверены, уязвимо ли ваше регулярное выражение, помните, что Node.js обычно без труда сообщает о _совпадении_ даже для уязвимого regexp и длинной входной строки. Экспоненциальное поведение срабатывает, когда совпадения нет, но Node.js не может быть в этом уверена, пока не перепробует множество путей по входной строке.

#### Пример REDOS

Вот пример уязвимого regexp, подвергающего свой сервер риску REDOS:
```js
app.get('/redos-me', (req, res) => {
  const filePath = req.query.filePath;

  // REDOS
  if (filePath.match(/(\/.+)+$/)) {
    console.log('valid path');
  } else {
    console.log('invalid path');
  }

  res.sendStatus(200);
});
```

Уязвимое regexp в этом примере — это (плохой!) способ проверить корректность пути в Linux. Оно сопоставляет строки, представляющие собой последовательность разделённых «/» имён, например «/a/b/c». Оно опасно, потому что нарушает правило 1: содержит дважды вложенный квантификатор.

Если клиент сделает запрос с `filePath`, равным `///.../\n` (100 символов «/», за которыми следует символ перевода строки, не совпадающий с «.» из regexp), то цикл событий фактически зависнет навсегда, заблокировав цикл событий. Эта REDOS-атака клиента приводит к тому, что все остальные клиенты не получат очереди, пока сопоставление regexp не завершится.

По этой причине вам стоит с опаской относиться к использованию сложных регулярных выражений для валидации пользовательского ввода.

#### Ресурсы для защиты от REDOS

Есть инструменты для проверки безопасности ваших regexp, например:

  * [safe-regex](https://github.com/davisjam/safe-regex)
  * [rxxr2](https://github.com/superhuman/rxxr2).

Однако ни один из них не выявит все уязвимые regexp.

Другой подход — использовать другой движок регулярных выражений. Вы можете взять модуль [node-re2](https://github.com/uhop/node-re2), который использует молниеносно быстрый движок regexp от Google — [RE2](https://github.com/google/re2). Но учтите: RE2 не на 100% совместим с regexp движка V8, поэтому, подставляя модуль node-re2 для обработки ваших regexp, проверяйте наличие регрессий. К тому же особенно сложные regexp модуль node-re2 не поддерживает.

Если вы пытаетесь сопоставить что-то «очевидное», вроде URL или пути к файлу, найдите пример в [библиотеке regexp](http://www.regexlib.com) или используйте npm-модуль, например [ip-regex](https://www.npmjs.com/package/ip-regex).

### Блокировка цикла событий: основные модули Node.js

Несколько основных модулей Node.js имеют синхронные затратные API, в том числе:

  * [Шифрование](https://nodejs.org/api/crypto.html)
  * [Сжатие](https://nodejs.org/api/zlib.html)
  * [Файловая система](https://nodejs.org/api/fs.html)
  * [Дочерний процесс](https://nodejs.org/api/child_process.html)

Эти API затратны, потому что связаны со значительными вычислениями (шифрование, сжатие), требуют I/O (файловый I/O) или, возможно, и того и другого (дочерний процесс). Эти API предназначены для удобства при написании сценариев, но не для использования в контексте сервера. Если вы выполняете их в цикле событий, они завершатся намного дольше типичной инструкции JavaScript, заблокировав цикл событий.

На сервере _вам не следует использовать следующие синхронные API этих модулей_:

  * Шифрование:
    * `crypto.randomBytes` (синхронная версия)
    * `crypto.randomFillSync`
    * `crypto.pbkdf2Sync`
    * Также будьте осторожны, передавая большой объём входных данных процедурам шифрования и расшифровки.
  * Сжатие:
    * `zlib.inflateSync`
    * `zlib.deflateSync`
  * Файловая система:
    * Не используйте синхронные API файловой системы. Например, если файл, к которому вы обращаетесь, находится в [распределённой файловой системе](https://en.wikipedia.org/wiki/Clustered_file_system#Distributed_file_systems) вроде [NFS](https://en.wikipedia.org/wiki/Network_File_System), время доступа может сильно варьироваться.
  * Дочерний процесс:
    * `child_process.spawnSync`
    * `child_process.execSync`
    * `child_process.execFileSync`

Этот список достаточно полон по состоянию на Node.js v9.

### Блокировка цикла событий: JSON DOS

`JSON.parse` и `JSON.stringify` — ещё одни потенциально затратные операции. Хотя их сложность равна `O(n)` от длины ввода, при больших `n` они могут занимать на удивление много времени.

Если ваш сервер работает с объектами JSON, особенно поступающими от клиента, стоит с осторожностью относиться к размеру объектов или строк, с которыми вы работаете в цикле событий.

Пример: блокировка на JSON. Мы создаём объект `obj` размером 2^21 и вызываем для него `JSON.stringify`, затем запускаем `indexOf` по строке, а потом делаем JSON.parse. Строка после `JSON.stringify` занимает 50 МБ. Сериализация объекта занимает 0,7 секунды, `indexOf` по строке в 50 МБ — 0,03 секунды, а разбор строки — 1,3 секунды.
```js
let obj = { a: 1 };
const iterations = 20;

// Экспоненциально увеличиваем объект, вкладывая его в самого себя
for (let i = 0; i < iterations; i++) {
  obj = { obj1: obj, obj2: obj };
}

// Измеряем время сериализации объекта
let start = process.hrtime();
const jsonString = JSON.stringify(obj);
let duration = process.hrtime(start);
console.log('JSON.stringify took', duration);

// Измеряем время поиска строки внутри JSON
start = process.hrtime();
const index = jsonString.indexOf('nomatch'); // Всегда -1
duration = process.hrtime(start);
console.log('String.indexOf took', duration);

// Измеряем время обратного разбора JSON в объект
start = process.hrtime();
const parsed = JSON.parse(jsonString);
duration = process.hrtime(start);
console.log('JSON.parse took', duration);
```

Существуют npm-модули, предлагающие асинхронные API для JSON. См., например:

  * [JSONStream](https://www.npmjs.com/package/JSONStream), у которого есть API на основе потоков.
  * [Big-Friendly JSON](https://www.npmjs.com/package/bfj), у которого есть как API на основе потоков, так и асинхронные версии стандартных API для JSON, использующие описанную ниже парадигму разбиения на цикле событий.

### Сложные вычисления без блокировки цикла событий

Предположим, вы хотите выполнить сложные вычисления на JavaScript, не блокируя цикл событий. У вас есть два варианта: разбиение (partitioning) или вынос (offloading).

#### Разбиение

Вы можете _разбить_ ваши вычисления так, чтобы каждая часть выполнялась в цикле событий, но регулярно уступала (передавала очередь) другим ожидающим событиям. В JavaScript легко сохранить состояние выполняемой задачи в замыкании, как показано в примере 2 ниже.

В качестве простого примера предположим, что вы хотите вычислить среднее чисел от `1` до `n`.

Пример 1. Неразбитое среднее, стоимость `O(n)`.
```js
for (let i = 0; i < n; i++) {
  sum += i;
}

const avg = sum / n;
console.log('avg: ' + avg);
```

Пример 2. Разбитое среднее, каждый из `n` асинхронных шагов стоит `O(1)`.
```js
function asyncAvg(n, avgCB) {
  // Сохраняем текущую сумму в JS-замыкании.
  let sum = 0;
  function help(i, cb) {
    sum += i;
    if (i == n) {
      cb(sum);
      return;
    }

    // «Асинхронная рекурсия».
    // Асинхронно планируем следующую операцию.
    setImmediate(help.bind(null, i + 1, cb));
  }

  // Запускаем вспомогательную функцию с колбэком, который вызовет avgCB.
  help(1, function (sum) {
    const avg = sum / n;
    avgCB(avg);
  });
}

asyncAvg(n, function (avg) {
  console.log('avg of 1-n: ' + avg);
});
```

Этот принцип можно применить к перебору массивов и тому подобному.

#### Вынос

Если вам нужно сделать что-то более сложное, разбиение — не лучший вариант. Это потому, что разбиение использует только цикл событий, и вы не получите пользы от нескольких ядер, почти наверняка имеющихся на вашей машине. _Помните: цикл событий должен координировать клиентские запросы, а не выполнять их сам_. Для сложной задачи перенесите работу с цикла событий в пул рабочих потоков.

##### Как выполнить вынос

У вас есть два варианта пула рабочих потоков — назначения, куда можно вынести работу.

  1. Вы можете использовать встроенный пул рабочих потоков Node.js, разработав [аддон на C++](https://nodejs.org/api/addons.html). На старых версиях Node собирайте свой C++-аддон с помощью [NAN](https://github.com/nodejs/nan), а на новых версиях используйте [N-API](https://nodejs.org/api/n-api.html). [node-webworker-threads](https://www.npmjs.com/package/webworker-threads) предлагает способ доступа к пулу рабочих потоков Node.js только средствами JavaScript.
  2. Вы можете создать и обслуживать собственный пул рабочих потоков, выделенный под вычисления, а не под ориентированный на I/O пул рабочих потоков Node.js. Самые прямолинейные способы сделать это — использовать [дочерний процесс (Child Process)](https://nodejs.org/api/child_process.html) или [Cluster](https://nodejs.org/api/cluster.html).

Вам _не_ следует просто создавать [дочерний процесс](https://nodejs.org/api/child_process.html) для каждого клиента. Вы можете получать клиентские запросы быстрее, чем успеваете создавать и обслуживать дочерние процессы, и ваш сервер может превратиться в [форк-бомбу](https://en.wikipedia.org/wiki/Fork_bomb).

##### Недостаток выноса

Недостаток подхода с выносом в том, что он влечёт накладные расходы в виде _затрат на коммуникацию_. Только циклу событий разрешено видеть «пространство имён» (состояние JavaScript) вашего приложения. Из рабочего потока вы не можете манипулировать объектом JavaScript в пространстве имён цикла событий. Вместо этого вам приходится сериализовать и десериализовать любые объекты, которыми вы хотите поделиться. Затем рабочий поток может работать со своей собственной копией этих объектов и вернуть изменённый объект (или «патч») циклу событий.

По вопросам сериализации см. раздел о JSON DOS.

##### Несколько советов по выносу

Возможно, вы захотите различать интенсивные по CPU и интенсивные по I/O задачи, потому что у них заметно разные характеристики.

Интенсивная по CPU задача продвигается только тогда, когда её рабочий поток запланирован, а рабочий поток должен быть запланирован на одно из [логических ядер](https://nodejs.org/api/os.html#os_os_cpus) вашей машины. Если у вас 4 логических ядра и 5 рабочих потоков, один из этих потоков не сможет продвигаться. В результате вы платите накладные расходы (память и планирование) за этот поток и не получаете от него отдачи.

Интенсивные по I/O задачи связаны с обращением к внешнему поставщику услуг (DNS, файловая система и т. д.) и ожиданием его ответа. Пока рабочий поток с интенсивной по I/O задачей ждёт ответа, ему нечем заняться, и операционная система может снять его с планирования, дав другому рабочему потоку возможность отправить свой запрос. Таким образом, _интенсивные по I/O задачи будут продвигаться, даже когда связанный с ними поток не выполняется_. Внешние поставщики услуг вроде баз данных и файловых систем сильно оптимизированы под конкурентную обработку множества ожидающих запросов. Например, файловая система изучает большой набор ожидающих запросов на запись и чтение, чтобы объединить конфликтующие обновления и извлечь файлы в оптимальном порядке.

Если вы полагаетесь лишь на один пул рабочих потоков, например на пул Node.js, то различающиеся характеристики работы, привязанной к CPU и к I/O, могут навредить производительности вашего приложения.

По этой причине вы, возможно, захотите держать отдельный вычислительный пул рабочих потоков.

#### Вынос: выводы

Для простых задач, например для перебора элементов сколь угодно длинного массива, разбиение может быть хорошим вариантом. Если ваши вычисления сложнее, лучше подходит вынос: затраты на коммуникацию, то есть накладные расходы на передачу сериализованных объектов между циклом событий и пулом рабочих потоков, компенсируются выгодой от использования нескольких ядер.

Однако если ваш сервер сильно полагается на сложные вычисления, стоит задуматься, действительно ли Node.js — хороший выбор. Node.js превосходна для работы, привязанной к I/O, но для затратных вычислений она может оказаться не лучшим вариантом.

Если вы выбираете подход с выносом, см. раздел о том, как не блокировать пул рабочих потоков.

## Не блокируйте пул рабочих потоков

У Node.js есть пул рабочих потоков, состоящий из `k` рабочих потоков. Если вы используете описанную выше парадигму выноса, у вас может быть отдельный вычислительный пул рабочих потоков, к которому применимы те же принципы. В любом случае предположим, что `k` намного меньше числа клиентов, которых вы можете обслуживать одновременно. Это соответствует философии Node.js «один поток на многих клиентов» — секрету её масштабируемости.

Как обсуждалось выше, каждый рабочий поток завершает свою текущую задачу, прежде чем перейти к следующей в очереди пула рабочих потоков.

Так вот, стоимость задач, необходимых для обработки запросов ваших клиентов, будет разной. Одни задачи можно выполнить быстро (например, чтение коротких или кэшированных файлов либо генерация небольшого числа случайных байтов), а другие займут больше времени (например, чтение больших или некэшированных файлов либо генерация большего числа случайных байтов). Ваша цель — _минимизировать разброс времени выполнения задач_, и для этого следует использовать _разбиение задач_.

### Минимизация разброса времени выполнения задач

Если текущая задача рабочего потока намного затратнее других задач, он будет недоступен для работы над остальными ожидающими задачами. Иными словами, _каждая относительно длинная задача фактически уменьшает размер пула рабочих потоков на единицу, пока не завершится_. Это нежелательно, потому что — до определённого предела — чем больше рабочих потоков в пуле, тем выше пропускная способность пула (задач в секунду), а значит, и выше пропускная способность сервера (клиентских запросов в секунду). Один клиент с относительно затратной задачей снизит пропускную способность пула рабочих потоков, что в свою очередь снизит пропускную способность сервера.

Чтобы избежать этого, старайтесь минимизировать разброс длительности задач, отправляемых в пул рабочих потоков. Хотя внешние системы, к которым обращаются ваши I/O-запросы (БД, файловая система и т. д.), уместно рассматривать как «чёрные ящики», вам следует представлять относительную стоимость этих I/O-запросов и избегать отправки запросов, которые заведомо окажутся особенно долгими.

Два примера должны проиллюстрировать возможный разброс времени выполнения задач.

#### Пример разброса: долгие чтения из файловой системы

Предположим, вашему серверу необходимо читать файлы для обработки некоторых клиентских запросов. Изучив API [файловой системы](https://nodejs.org/api/fs.html) Node.js, вы решили ради простоты использовать `fs.readFile()`. Однако до версии v10 функция `fs.readFile()` не была разбитой: она отправляла единственную задачу `fs.read()`, охватывающую весь файл. Если для одних пользователей вы читаете короткие файлы, а для других — длинные, то `fs.readFile()` может внести значительный разброс в длительность задач в ущерб пропускной способности пула рабочих потоков.

В качестве наихудшего сценария предположим, что злоумышленник может убедить ваш сервер прочитать _произвольный_ файл (это [уязвимость обхода каталога](https://www.owasp.org/index.php/Path_Traversal)). Если ваш сервер работает под Linux, злоумышленник может указать чрезвычайно медленный файл: [`/dev/random`](http://man7.org/linux/man-pages/man4/random.4.html). Для всех практических целей `/dev/random` бесконечно медленный, и любой рабочий поток, которому поручено читать из `/dev/random`, никогда не завершит эту задачу. Затем злоумышленник отправляет `k` запросов, по одному на каждый рабочий поток, и никакие другие клиентские запросы, использующие пул рабочих потоков, не будут продвигаться.

#### Пример разброса: долгие криптографические операции

Предположим, ваш сервер генерирует криптографически стойкие случайные байты с помощью [`crypto.randomBytes()`](https://nodejs.org/api/crypto.html#crypto_crypto_randombytes_size_callback). Функция `crypto.randomBytes()` не разбита: она создаёт единственную задачу `randomBytes()`, чтобы сгенерировать столько байтов, сколько вы запросили. Если для одних пользователей вы создаёте меньше байтов, а для других — больше, то `crypto.randomBytes()` — ещё один источник разброса в длительности задач.

### Разбиение задач

Задачи с переменной стоимостью по времени могут навредить пропускной способности пула рабочих потоков. Чтобы минимизировать разброс времени выполнения задач, по возможности следует _разбивать_ каждую задачу на подзадачи сопоставимой стоимости. Когда каждая подзадача завершается, она должна отправлять следующую подзадачу, а когда завершается финальная подзадача, она должна уведомить того, кто отправил задачу.

Продолжая пример с `fs.readFile()`, вам вместо этого следует использовать `fs.read()` (ручное разбиение) или `ReadStream` (автоматически разбитый вариант).

Тот же принцип применим к задачам, привязанным к CPU: пример `asyncAvg` может быть неуместен для цикла событий, но хорошо подходит для пула рабочих потоков.

Когда вы разбиваете задачу на подзадачи, более короткие задачи разворачиваются в небольшое число подзадач, а более длинные — в большее число подзадач. Между подзадачами более длинной задачи рабочий поток, которому она назначена, может работать над подзадачей из другой, более короткой задачи, тем самым улучшая общую пропускную способность пула рабочих потоков по задачам.

Обратите внимание, что число завершённых подзадач — бесполезная метрика для пропускной способности пула рабочих потоков. Вместо этого следите за числом завершённых _задач_.

### Отказ от разбиения задач

Напомним, что цель разбиения задач — минимизировать разброс времени их выполнения. Если вы можете различать более короткие и более длинные задачи (например, суммирование массива против его сортировки), то можно создать по одному пулу рабочих потоков для каждого класса задач. Маршрутизация более коротких и более длинных задач в отдельные пулы рабочих потоков — ещё один способ минимизировать разброс времени выполнения задач.

В пользу этого подхода: разбиение задач влечёт накладные расходы (затраты на создание представления задачи в пуле рабочих потоков и на манипулирование очередью пула), и отказ от разбиения избавляет вас от затрат на дополнительные обращения к пулу рабочих потоков. Он также уберегает вас от ошибок при разбиении задач.

Недостаток этого подхода в том, что рабочие потоки во всех этих пулах будут нести накладные расходы по памяти и времени и конкурировать друг с другом за процессорное время. Помните, что каждая привязанная к CPU задача продвигается только пока она запланирована. Поэтому рассматривать этот подход стоит лишь после тщательного анализа.

### Пул рабочих потоков: выводы

Используете ли вы только пул рабочих потоков Node.js или держите отдельные пулы, вам следует оптимизировать пропускную способность ваших пулов по задачам.

Для этого минимизируйте разброс времени выполнения задач с помощью разбиения задач.

## Риски npm-модулей

Хотя основные модули Node.js предлагают строительные блоки для широкого спектра приложений, иногда нужно нечто большее. Разработчики Node.js колоссально выигрывают от [экосистемы npm](https://www.npmjs.com/) с сотнями тысяч модулей, предлагающих функциональность для ускорения вашего процесса разработки.

Помните, однако, что большинство этих модулей написаны сторонними разработчиками и, как правило, выпускаются лишь с гарантиями «по мере возможностей» (best-effort). Разработчику, использующему npm-модуль, следует беспокоиться о двух вещах, хотя о второй часто забывают.

  1. Соблюдает ли он свои API?
  2. Могут ли его API заблокировать цикл событий или рабочий поток? Многие модули не прилагают усилий, чтобы обозначить стоимость своих API, в ущерб сообществу.

Для простых API стоимость можно оценить: стоимость строковых операций несложно понять. Но во многих случаях неясно, сколько может стоить тот или иной API.

_Если вы вызываете API, который может сделать что-то затратное, перепроверьте стоимость. Попросите разработчиков задокументировать её или изучите исходный код самостоятельно (и отправьте PR с документацией стоимости)._

Помните: даже если API асинхронный, вы не знаете, сколько времени он может проводить в рабочем потоке или в цикле событий в каждой из своих частей. Например, предположим, что в приведённом выше примере `asyncAvg` каждый вызов вспомогательной функции суммировал бы _половину_ чисел, а не одно из них. Тогда эта функция всё ещё была бы асинхронной, но стоимость каждой части составляла бы `O(n)`, а не `O(1)`, что делало бы её куда менее безопасной для произвольных значений `n`.

## Заключение

У Node.js есть два типа потоков выполнения: один цикл событий и `k` рабочих потоков. Цикл событий отвечает за колбэки JavaScript и неблокирующий I/O, а рабочий поток выполняет задачи, соответствующие коду на C++, который завершает асинхронный запрос, включая блокирующий I/O и интенсивную по CPU работу. Оба типа потоков работают не более чем над одной активностью за раз. Если какой-либо колбэк или задача занимает много времени, выполняющий его поток становится _заблокированным_. Если ваше приложение делает блокирующие колбэки или задачи, это может привести к деградации пропускной способности (клиентов в секунду) в лучшем случае и к полному отказу в обслуживании в худшем.

Чтобы написать высокопроизводительный, более защищённый от DoS веб-сервер, вы должны обеспечить, чтобы ни на безобидном, ни на злонамеренном вводе ни ваш цикл событий, ни ваши рабочие потоки не блокировались.
