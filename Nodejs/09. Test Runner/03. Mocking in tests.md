# Мокирование в тестах

Мокирование — это способ создания подобия, марионетки. Обычно это делается в манере кукловодства «когда 'a', делай 'b'». Идея в том, чтобы ограничить число подвижных частей и контролировать вещи, которые «не имеют значения». «Моки» (mocks) и «заглушки» (stubs) — технически разные виды «тестовых дублёров» (test doubles). Для любознательных: заглушка — это замена, которая ничего не делает (no-op), но отслеживает своё вызов. Мок — это заглушка, у которой к тому же есть фиктивная реализация (то самое «когда 'a', делай 'b'»). В рамках этого документа разница не важна, и заглушки называются моками.

Тесты должны быть детерминированными: их можно запускать в любом порядке, любое число раз, и они всегда должны выдавать один и тот же результат. Правильная подготовка (setup) и мокирование делают это возможным.

Node.js предоставляет множество способов мокировать различные фрагменты кода.

Эта статья посвящена следующим типам тестов:

тип| описание| пример| кандидаты на мокирование| модульный (unit)| наименьший фрагмент кода, который можно изолировать| `const sum = (a, b) => a + b`| собственный код, внешний код, внешняя система| компонентный (component)| модуль + зависимости| `const arithmetic = (op = sum, a, b) => ops[op](a, b)`| внешний код, внешняя система| интеграционный (integration)| компоненты, работающие вместе| -| внешний код, внешняя система| сквозной (end-to-end, e2e)| приложение + внешние хранилища данных, доставка и т.д.| Фиктивный пользователь (например, агент Playwright), буквально использующий приложение, подключённое к реальным внешним системам.| нет (не мокировать)  
---|---|---|---  
  
Существуют разные точки зрения на то, когда стоит мокировать, а когда нет; их основные положения изложены ниже.

## Когда мокировать, а когда нет

Есть 3 основных кандидата на мокирование:

  * Собственный код
  * Внешний код
  * Внешняя система

### Собственный код

Это то, что контролирует ваш проект.
```js
import foo from './foo.mjs';

export function main() {
  const f = foo();
}
```

Здесь `foo` — это зависимость «собственного кода» для `main`.

#### Зачем

Для настоящего модульного теста `main` функцию `foo` следует мокировать: вы тестируете, что работает `main`, а не что работают `main` \+ `foo` (это уже другой тест).

#### Почему нет

Мокирование `foo` может доставить больше хлопот, чем пользы, особенно когда `foo` проста, хорошо протестирована и редко обновляется.

Не мокировать `foo` может быть лучше, потому что это более аутентично и увеличивает покрытие `foo` (поскольку тесты `main` также будут проверять `foo`). Однако это может создавать шум: когда `foo` ломается, сломается и куча других тестов, поэтому отследить проблему становится более утомительно: если провален лишь 1 тест для элемента, в конечном счёте ответственного за проблему, это очень легко заметить; тогда как 100 проваленных тестов создают ситуацию «иголки в стоге сена» при поиске настоящей проблемы.

### Внешний код

Это то, что ваш проект не контролирует.
```js
import bar from 'bar';

export function main() {
  const f = bar();
}
```

Здесь `bar` — это внешний пакет, например npm-зависимость.

Бесспорно, для модульных тестов это следует всегда мокировать. Для компонентных и интеграционных тестов необходимость мокирования зависит от того, что это такое.

#### Зачем

Проверка того, что работает код, который ваш проект не сопровождает, не является целью модульного теста (и у такого кода должны быть свои собственные тесты).

#### Почему нет

Иногда мокировать просто нереалистично. Например, вы почти никогда не станете мокировать крупный фреймворк, такой как react или angular (лекарство оказалось бы хуже болезни).

### Внешняя система

Это такие вещи, как базы данных, окружения (Chromium или Firefox для веб-приложения, операционная система для node-приложения и т.д.), файловые системы, хранилища в памяти и т.д.

В идеале мокировать их было бы не нужно. Помимо создания тем или иным способом изолированных копий для каждого случая (обычно очень непрактично из-за затрат, дополнительного времени выполнения и т.д.), следующий по предпочтительности вариант — мокирование. Без мокирования тесты саботируют друг друга:
```js
import { db } from 'db';

export function read(key, all = false) {
  validate(key);

  if (all) {
    return db.getAll(key);
  }

  return db.getOne(key);
}

export function save(key, val) {
  validate(key, val);

  return db.upsert(key, val);
}
```

В приведённом выше коде первый и второй случаи (операторы `it()`) могут саботировать друг друга, потому что они выполняются конкурентно и мутируют одно и то же хранилище (состояние гонки): вставка в `save()` может привести к тому, что во всём остальном корректный тест `read()` провалит своё утверждение о найденных элементах (а `read()` может сделать то же самое с `save()`).

## Что мокировать

### Модули + единицы кода

Здесь используется [`mock`](https://nodejs.org/api/test.html#class-mocktracker) из средства запуска тестов Node.js.
```js
import assert from 'node:assert/strict';
import { before, describe, it, mock } from 'node:test';

describe('foo', { concurrency: true }, () => {
  const barMock = mock.fn();
  let foo;

  before(async () => {
    const barNamedExports = await import('./bar.mjs')
      // отбрасываем исходный экспорт по умолчанию
      .then(({ default: _, ...rest }) => rest);

    // Обычно нет необходимости вручную вызывать restore() после каждого теста
    // и reset() после всех (node делает это автоматически).
    mock.module('./bar.mjs', {
      defaultExport: barMock,
      // Сохраняем остальные экспорты, которые вы не хотите мокировать.
      namedExports: barNamedExports,
    });

    // Это ДОЛЖЕН быть динамический импорт, потому что это единственный способ гарантировать,
    // что импорт начнётся после того, как мок будет настроен.
    ({ foo } = await import('./foo.mjs'));
  });

  it('should do the thing', () => {
    barMock.mock.mockImplementationOnce(function bar_mock() {
      /* … */
    });

    assert.equal(foo(), 42);
  });
});
```

### API

Малоизвестный факт: существует встроенный способ мокировать `fetch`. [`undici`](https://github.com/nodejs/undici) — это реализация `fetch` в Node.js. Она поставляется вместе с `node`, но в настоящее время не предоставляется самим `node`, поэтому её необходимо установить (например, `npm install undici`).
```js
import assert from 'node:assert/strict';
import { beforeEach, describe, it } from 'node:test';

import { MockAgent, setGlobalDispatcher } from 'undici';

import endpoints from './endpoints.mjs';

describe('endpoints', { concurrency: true }, () => {
  let agent;
  beforeEach(() => {
    agent = new MockAgent();
    setGlobalDispatcher(agent);
  });

  it('should retrieve data', async () => {
    const endpoint = 'foo';
    const code = 200;
    const data = {
      key: 'good',
      val: 'item',
    };

    agent
      .get('https://example.com')
      .intercept({
        path: endpoint,
        method: 'GET',
      })
      .reply(code, data);

    assert.deepEqual(await endpoints.get(endpoint), {
      code,
      data,
    });
  });

  it('should save data', async () => {
    const endpoint = 'foo/1';
    const code = 201;
    const data = {
      key: 'good',
      val: 'item',
    };

    agent
      .get('https://example.com')
      .intercept({
        path: endpoint,
        method: 'PUT',
      })
      .reply(code, data);

    assert.deepEqual(await endpoints.save(endpoint), {
      code,
      data,
    });
  });
});
```

### Время

Как и Доктор Стрэндж, вы тоже можете управлять временем. Обычно вы делаете это просто для удобства, чтобы избежать искусственно затянутых прогонов тестов (вы действительно хотите ждать 3 минуты, пока сработает этот `setTimeout()`?). Возможно, вы также захотите путешествовать во времени. Здесь используется [`mock.timers`](https://nodejs.org/api/test.html#class-mocktimers) из средства запуска тестов Node.js.

Обратите внимание на использование часового пояса здесь (`Z` в метках времени). Если пренебречь указанием единообразного часового пояса, это, вероятно, приведёт к неожиданным результатам.
```js
import assert from 'node:assert/strict';
import { describe, it, mock } from 'node:test';

import ago from './ago.mjs';

describe('whatever', { concurrency: true }, () => {
  it('should choose "minutes" when that\'s the closet unit', () => {
    mock.timers.enable({ now: new Date('2000-01-01T00:02:02Z') });

    const t = ago('1999-12-01T23:59:59Z');

    assert.equal(t, '2 minutes ago');
  });
});
```

Это особенно полезно при сравнении со статической фикстурой (которая зафиксирована в репозитории), например при [снимковом тестировании (snapshot testing)](https://nodejs.org/api/test.html#snapshot-testing).
