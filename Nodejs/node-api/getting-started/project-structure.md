# Анатомия проекта Node-API

Прежде чем погружаться в код, полезно понять файлы и структуру, которые есть у каждого проекта `node-addon-api`. Все примеры N-API в этих руководствах следуют одной и той же структуре, поэтому на этой странице она объясняется один раз, чтобы те руководства могли сосредоточиться на том, что делает каждое из них уникальным.

## Выбор уровня API

Node-API работает на двух уровнях:

  * **C API** встроен непосредственно в Node.js и полностью задокументирован на [страницах API Node.js](https://nodejs.org/api/n-api.html). Даёт вам максимальный контроль без дополнительных зависимостей.
  * **Обёртка на C++ (`node-addon-api`)** — это пакет npm, оборачивающий C API в идиоматичную объектную модель C++. Рекомендуется для большинства проектов: она устраняет значительное количество шаблонного кода, сохраняя при этом полную гарантию стабильности ABI, которую даёт Node-API.

В руководствах этого раздела используется `node-addon-api`.

> Node-API стабилен во всех поддерживаемых на данный момент выпусках Node.js. Для наилучшего опыта используйте [выпуск Active LTS или Maintenance LTS](https://nodejs.org/en/about/releases/). Проверить версию Node.js, которую вы используете, можно командой `node -v`.

## Раскладка каталогов
```
.
├── binding.gyp        # сообщает node-gyp, как компилировать исходники на C/C++
├── build/             # скомпилированный вывод (генерируется)
├── lib/
│   └── binding.js     # слой JavaScript, загружающий скомпилированный бинарник
├── node_modules/
├── src/
│   └── *.cc / *.h     # ваша реализация на C/C++
├── test/
│   └── *.js           # код тестов
├── package.json
└── package-lock.json
```

## package.json

Две записи в `package.json` специфичны для нативных аддонов.

### Зависимость `node-addon-api`
```json
"dependencies": {
  "node-addon-api": "^8.0.0"
}
```

[`node-addon-api`](https://github.com/nodejs/node-addon-api) добавляет обёртку на C++ к C API, встроенному в Node.js. Она делает создание и манипулирование объектами JavaScript в C++ простым и полезна даже тогда, когда нижележащая библиотека, которую вы оборачиваете, написана на C.

### `"gypfile": true`
```
"gypfile": true
```

Это сообщает npm, что пакет требует этапа нативной компиляции. Когда npm видит эту запись, он автоматически вызывает свою встроенную копию `node-gyp`, которая читает `binding.gyp`, чтобы собрать бинарник.

## binding.gyp

`binding.gyp` — это [GYP](https://gyp.gsrc.io/)-файл, описывающий, как компилировать и линковать ваш код на C/C++. Он должен называться в точности `binding.gyp`.

[GYP](https://gyp.gsrc.io/) (Generate Your Projects) позволяет вам написать единое описание сборки, работающее в Windows, macOS и Linux. [`node-gyp`](https://github.com/nodejs/node-gyp) читает этот файл и производит подходящие для платформы файлы сборки (проект MSVC в Windows, Makefile в Linux, проект Xcode в macOS), после чего вызывает компилятор.

Минимальный `binding.gyp` для проекта `node-addon-api` выглядит так:
```json
{
  "targets": [
    {
      "target_name": "my_addon",
      "sources": ["src/my_addon.cc"],
      "include_dirs": ["<!@(node -p \"require('node-addon-api').include\")"],
      "dependencies": ["<!(node -p \"require('node-addon-api').gyp\")"]
    }
  ]
}
```

Полный формат GYP задокументирован в [пользовательской документации GYP](https://gyp.gsrc.io/docs/UserDocumentation.md).

## lib/binding.js

Каталог `lib/` содержит тонкую обёртку на JavaScript, которая загружает скомпилированный бинарник и реэкспортирует его. Этот слой держит логику загрузки бинарника в одном месте и даёт вам естественное место для добавления валидации на стороне JavaScript или удобных методов.

Типичный `binding.js` использует пакет [`bindings`](https://www.npmjs.com/package/bindings), чтобы разрешить путь к файлу `.node` независимо от платформы:
```js
'use strict';
const addon = require('bindings')('my_addon');
module.exports = addon;
```

> Если вы устанавливаете пакеты глобально с помощью npm и сталкиваетесь с `Error: EACCES: permission denied`, используйте [`nvm`](https://github.com/nvm-sh/nvm) для управления вашей установкой Node.js. С nvm глобальные установки попадают в ваш домашний каталог и никогда не требуют `sudo`.
