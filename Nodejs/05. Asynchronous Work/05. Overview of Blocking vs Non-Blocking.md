# Обзор блокирующего и неблокирующего кода

Этот обзор охватывает разницу между **блокирующими** и **неблокирующими** вызовами в Node.js. В обзоре будут упоминаться цикл событий и libuv, но предварительного знания этих тем не требуется. Предполагается, что читатели обладают базовым пониманием языка JavaScript и [паттерна колбэков](<01. JavaScript Asynchronous Programming and Callbacks.md>) в Node.js.

> Под «I/O» здесь в первую очередь подразумевается взаимодействие с системным диском и сетью, поддерживаемое [libuv](https://libuv.org/).

## Блокировка

**Блокировка** — это когда выполнению дополнительного JavaScript в процессе Node.js приходится ждать завершения операции, не относящейся к JavaScript. Это происходит потому, что цикл событий не может продолжать выполнять JavaScript, пока идёт **блокирующая** операция.

В Node.js JavaScript, демонстрирующий низкую производительность из-за интенсивной нагрузки на CPU, а не из-за ожидания операции, не относящейся к JavaScript (такой как ввод-вывод), обычно не называют **блокирующим**. Синхронные методы стандартной библиотеки Node.js, использующие libuv, — это наиболее часто применяемые **блокирующие** операции. У нативных модулей также могут быть **блокирующие** методы.

Все методы ввода-вывода в стандартной библиотеке Node.js предоставляют асинхронные версии, которые являются **неблокирующими** и принимают функции-колбэки. У некоторых методов есть и **блокирующие** аналоги, имена которых оканчиваются на `Sync`.

## Сравнение кода

**Блокирующие** методы выполняются **синхронно**, а **неблокирующие** методы выполняются **асинхронно**.

На примере модуля файловой системы вот **синхронное** чтение файла:
```js
const fs = require('node:fs');

const data = fs.readFileSync('/file.md'); // блокирует здесь, пока файл не будет прочитан
```

А вот эквивалентный **асинхронный** пример:
```js
const fs = require('node:fs');

fs.readFile('/file.md', (err, data) => {
  if (err) {
    throw err;
  }
});
```

Первый пример выглядит проще второго, но имеет недостаток: вторая строка **блокирует** выполнение любого дополнительного JavaScript до тех пор, пока весь файл не будет прочитан. Обратите внимание, что в синхронной версии, если выбрасывается ошибка, её нужно будет перехватить, иначе процесс аварийно завершится. В асинхронной версии автор сам решает, должна ли ошибка выбрасываться, как показано.

Давайте немного расширим наш пример:
```js
const fs = require('node:fs');

const data = fs.readFileSync('/file.md'); // блокирует здесь, пока файл не будет прочитан
console.log(data);
moreWork(); // выполнится после console.log
```

А вот похожий, но не эквивалентный асинхронный пример:
```js
const fs = require('node:fs');

fs.readFile('/file.md', (err, data) => {
  if (err) {
    throw err;
  }

  console.log(data);
});
moreWork(); // выполнится до console.log
```

В первом примере выше `console.log` будет вызван до `moreWork()`. Во втором примере `fs.readFile()` является **неблокирующим**, поэтому выполнение JavaScript может продолжиться, и `moreWork()` будет вызван первым. Возможность запустить `moreWork()`, не дожидаясь завершения чтения файла, — это ключевое проектное решение, позволяющее добиться более высокой пропускной способности.

## Конкурентность и пропускная способность

Выполнение JavaScript в Node.js однопоточно, поэтому конкурентность относится к способности цикла событий выполнять функции-колбэки JavaScript после завершения другой работы. Любой код, который предполагается выполнять конкурентно, должен позволять циклу событий продолжать работу, пока идут операции, не относящиеся к JavaScript, такие как ввод-вывод.

В качестве примера рассмотрим случай, когда каждый запрос к веб-серверу занимает 50 мс, и 45 мс из этих 50 мс — это ввод-вывод базы данных, который можно выполнить асинхронно. Выбор **неблокирующих** асинхронных операций высвобождает эти 45 мс на каждый запрос для обработки других запросов. Это существенная разница в пропускной способности лишь за счёт выбора **неблокирующих** методов вместо **блокирующих**.

Цикл событий отличается от моделей во многих других языках, где для обработки конкурентной работы могут создаваться дополнительные потоки выполнения.

## Опасности смешивания блокирующего и неблокирующего кода

Есть некоторые паттерны, которых следует избегать при работе с вводом-выводом. Давайте рассмотрим пример:
```js
const fs = require('node:fs');

fs.readFile('/file.md', (err, data) => {
  if (err) {
    throw err;
  }

  console.log(data);
});
fs.unlinkSync('/file.md');
```

В примере выше `fs.unlinkSync()` с большой вероятностью будет выполнен до `fs.readFile()`, что удалит `file.md` до того, как он будет фактически прочитан. Лучший способ написать это — полностью **неблокирующий** и гарантированно выполняющийся в правильном порядке:
```js
const fs = require('node:fs');

fs.readFile('/file.md', (readFileErr, data) => {
  if (readFileErr) {
    throw readFileErr;
  }

  console.log(data);

  fs.unlink('/file.md', unlinkErr => {
    if (unlinkErr) {
      throw unlinkErr;
    }
  });
});
```

Приведённый выше код помещает **неблокирующий** вызов `fs.unlink()` внутрь колбэка `fs.readFile()`, что гарантирует правильный порядок операций.
