# Публикация пакета

Все приведённые конфигурации `package.json` (кроме специально помеченных «не работает») работают в Node.js 12.22.x (последняя из v12, самая старая поддерживаемая линейка) и 17.2.0 — актуальной последней на тот момент1, а также, забавы ради, с webpack 5.53.0 и 5.63.0 соответственно. Они доступны здесь: [JakobJingleheimer/nodejs-module-config-examples](https://github.com/JakobJingleheimer/nodejs-module-config-examples).

Для любопытных: разделы «Как мы сюда попали» и «В кроличью нору» дают предысторию и более глубокие объяснения.

## Выберите своё решение

Есть 2 основных варианта, которые покрывают почти все сценарии использования:

  * Писать исходный код и публиковать в CJS (вы используете `require()`); CJS потребляем как из CJS, так и из ESM (во всех версиях node). Перейдите к разделу «Исходный код и дистрибутив на CJS».
  * Писать исходный код и публиковать в ESM (вы используете `import` и не используете top-level `await`); ESM потребляем как из ESM, так и из CJS (в node 22.x и 23.x; см. [`require()` для ES Module](https://nodejs.org/api/modules.html#loading-ecmascript-modules-using-require)). Перейдите к разделу «Исходный код и дистрибутив на ESM».

Как правило, лучше публиковать только 1 формат — либо CJS, _либо_ ESM. Не оба. Публикация нескольких форматов может привести к опасности двойного пакета (dual-package hazard), а также к другим недостаткам.

Есть и другие доступные варианты, в основном для исторических целей.

Вы как автор пакета пишете| Потребители вашего пакета пишут свой код на| Ваши варианты| Исходный код на CJS с использованием `require()`| ESM: потребители делают `import` вашего пакета| Исходный код на CJS и дистрибутив только на ESM| Исходный код на CJS с использованием `require()`| CJS и ESM: потребители делают либо `require()`, либо `import` вашего пакета| Исходный код на CJS и дистрибутив на обоих форматах CJS и ESM| Исходный код на ESM с использованием `import`| CJS: потребители делают `require()` вашего пакета (и вы используете top-level `await`)| Исходный код на ESM и дистрибутив только на CJS| Исходный код на ESM с использованием `import`| CJS и ESM: потребители делают либо `require()`, либо `import` вашего пакета| Исходный код на ESM и дистрибутив на обоих форматах CJS и ESM  
---|---|---  
  
### Исходный код и дистрибутив на CJS

Самая минимальная конфигурация может состоять лишь из [`"name"`](https://nodejs.org/api/packages.html#name). Но чем меньше загадочности, тем лучше: по сути, просто объявите экспорты пакета через поле (или набор полей) `"exports"`.

**Рабочий пример**: [cjs-with-cjs-distro](https://github.com/JakobJingleheimer/nodejs-module-config-examples/tree/main/packages/cjs/cjs-distro)
```json
{
  "name": "cjs-source-and-distribution"
  // "main": "./index.js"
}
```

Обратите внимание, что `packageJson.exports["."] = filepath` — это сокращённая запись для `packageJson.exports["."].default = filepath`

### Исходный код и дистрибутив на ESM

Просто, проверено и надёжно.

Обратите внимание, что начиная с Node.js v23.0.0 можно делать `require` статического ESM (кода, который не использует top-level `await`). Подробнее см. [Загрузка модулей ECMAScript с помощью `require()`](https://nodejs.org/api/modules.html#loading-ecmascript-modules-using-require).

Это почти то же самое, что и конфигурация CJS-CJS выше, с 1 небольшим отличием: поле [`"type"`](https://nodejs.org/api/packages.html#type).

**Рабочий пример**: [esm-with-esm-distro](https://github.com/JakobJingleheimer/nodejs-module-config-examples/tree/main/packages/esm/esm-distro)
```json
{
  "name": "esm-source-and-distribution",
  "type": "module"
  // "main": "./index.js"
}
```

Обратите внимание, что ESM теперь _является_ «обратно» совместимым с CJS: модуль CJS теперь _может_ делать [`require()` для ES Module](https://nodejs.org/api/modules.html#loading-ecmascript-modules-using-require) без флага, начиная с 23.0.0 и 22.12.0.

### Исходный код на CJS и дистрибутив только на ESM

Это требует небольшой ловкости, но тоже довольно просто. Это может быть предпочтительным выбором для более старых проектов, ориентирующихся на более новые стандарты, или для авторов, которые просто предпочитают CJS, но публикуют для другого окружения.

**Рабочий пример**: [cjs-with-esm-distro](https://github.com/JakobJingleheimer/nodejs-module-config-examples/tree/main/packages/cjs/esm-distro)
```json
{
  "name": "cjs-source-with-esm-distribution",
  "main": "./dist/index.mjs"
}
```

Расширение файла [`.mjs`](https://nodejs.org/api/esm.html#enabling) — это козырь: оно переопределит **любую** другую конфигурацию, и файл будет трактоваться как ESM. Использование этого расширения файла необходимо, потому что `packageJson.exports.import` **НЕ** означает, что файл является ESM (вопреки распространённому, если не всеобщему, заблуждению), а лишь то, что это файл, который будет использоваться при импорте пакета (ESM _может_ импортировать CJS. См. раздел «Подводные камни» ниже).

### Исходный код на CJS и дистрибутив на обоих форматах CJS и ESM

Чтобы _напрямую_ обслужить обе аудитории (так чтобы ваш дистрибутив работал «нативно» в любом случае), у вас есть несколько вариантов:

#### Присоединяйте именованные экспорты напрямую к `exports`

Классический, но требует некоторой изощрённости и ловкости. Это означает добавление свойств к существующему `module.exports` (вместо переприсваивания `module.exports` целиком).

**Рабочий пример**: [cjs-with-dual-distro (properties)](https://github.com/JakobJingleheimer/nodejs-module-config-examples/tree/main/packages/cjs/dual/property-distro)
```json
{
  "name": "cjs-source-with-esm-via-properties-distribution",
  "main": "./dist/cjs/index.js"
}
```

Плюсы:

  * Меньший вес пакета
  * Легко и просто (вероятно, наименьшие усилия, если вы не против придерживаться небольшого синтаксического ограничения)
  * Исключает опасность двойного пакета (Dual-Package Hazard)

Минусы:

  * Требует очень специфичного синтаксиса (либо в исходном коде, и/или трюков с бандлером).

Иногда модуль CJS может переприсваивать `module.exports` чему-то другому (будь то объект или функция) вот так:
```js
const someObject = {
  foo() {},
  bar() {},
  qux() {},
};

module.exports = someObject;
```

Node.js обнаруживает именованные экспорты в CJS с помощью [статического анализа, который ищет определённые паттерны](https://github.com/nodejs/cjs-module-lexer/tree/main?tab=readme-ov-file#parsing-examples), а приведённый выше пример его обходит. Чтобы сделать именованные экспорты обнаруживаемыми, сделайте так:
```js
module.exports.foo = function foo() {};
module.exports.bar = function bar() {};
module.exports.qux = function qux() {};
```

#### Используйте простую ESM-обёртку

Сложная настройка, и трудно добиться нужного баланса.

**Рабочий пример**: [cjs-with-dual-distro (wrapper)](https://github.com/JakobJingleheimer/nodejs-module-config-examples/tree/main/packages/cjs/dual/wrapper-distro)
```json
{
  "name": "cjs-with-wrapper-dual-distro",
  "exports": {
    ".": {
      "import": "./dist/esm/wrapper.mjs",
      "require": "./dist/cjs/index.js",
      "default": "./dist/cjs/index.js"
    }
  }
}
```

Плюсы:

  * Меньший вес пакета

Минусы:

  * Вероятно, потребует сложных трюков с бандлером (мы не смогли найти ни одного готового варианта для автоматизации этого в Webpack).

Когда вывод CJS из бандлера обходит обнаружение именованных экспортов в Node.js, для явного повторного экспорта известных именованных экспортов для потребителей ESM можно использовать ESM-обёртку.

Когда CJS экспортирует объект (который сопоставляется с `default` в ESM), вы можете сохранить ссылки на все члены объекта локально в обёртке, а затем повторно экспортировать их, чтобы потребитель ESM мог обращаться ко всем им по имени.
```js
import cjs from '../cjs/index.js';

const { a, b, c /* … */ } = cjs;

export { a, b, c /* … */ };
```

**Однако** это ломает живые привязки (live bindings): переприсваивание `cjs.a` не отразится в `esmWrapper.a`.

#### Два полных дистрибутива

Накидайте кучу всего и надейтесь на лучшее. Это, вероятно, самый распространённый и самый простой из вариантов «CJS в CJS и ESM», но вы за это заплатите. Это редко бывает хорошей идеей.

**Рабочий пример**: [cjs-with-dual-distro (double)](https://github.com/JakobJingleheimer/nodejs-module-config-examples/tree/main/packages/cjs/dual/double-distro)
```json
{
  "name": "cjs-with-full-dual-distro",
  "exports": {
    ".": {
      "import": "./dist/esm/index.mjs",
      "require": "./dist/cjs/index.js",
      "default": "./dist/cjs/index.js"
    }
  }
}
```

Плюсы:

  * Простая конфигурация бандлера

Минусы:

  * Больший вес пакета (по сути, двойной)
  * Уязвим к опасности двойного пакета (Dual-Package Hazard)

Как вариант, вы можете использовать ключи `"default"` и `"node"`, которые менее контринтуитивны: Node.js всегда выберет вариант `"node"` (который всегда работает), а инструменты, не относящиеся к Node.js, выберут `"default"`, когда настроены на цель, отличную от node. **Это исключает опасность двойного пакета.**
```json
{
  "name": "cjs-with-alt-full-dual-distro",
  "exports": {
    ".": {
      "node": "./dist/cjs/index.js",
      "default": "./dist/esm/index.mjs"
    }
  }
}
```

### Исходный код на ESM и дистрибутив только на CJS

Мы больше не в Канзасе, Тото.

Конфигурации (есть 2 варианта) почти такие же, как в разделе «Исходный код на ESM и дистрибутив на обоих форматах CJS и ESM», просто исключите `packageJson.exports.import`.

💡 Использование `"type": "module"`2 в паре с расширением файла `.cjs` (для файлов commonjs) даёт наилучшие результаты. Подробнее о том, почему, см. в разделах «В кроличью нору» и «Подводные камни» ниже.

**Рабочий пример**: [esm-with-cjs-distro](https://github.com/JakobJingleheimer/nodejs-module-config-examples/tree/main/packages/esm/cjs-distro)

### Исходный код на ESM и дистрибутив на обоих форматах CJS и ESM

Когда исходный код написан не на JavaScript (например, на TypeScript), варианты могут быть ограничены из-за необходимости использовать расширение(-я) файлов, специфичное для этого языка (например, `.ts`), и может не быть эквивалента `.mjs`.

Аналогично разделу «Исходный код на CJS и дистрибутив на обоих форматах CJS и ESM», у вас есть те же варианты.

#### Публикация только CJS-дистрибутива с экспортом через свойства

Хитро в изготовлении и требует хороших ингредиентов.

Этот вариант почти идентичен экспортам через свойства из раздела «Исходный код на CJS с дистрибутивом на CJS и ESM» выше. Единственное отличие — в package.json: `"type": "module"`.

Лишь некоторые инструменты сборки поддерживают генерацию такого вывода. [Rollup](https://www.rollupjs.org/) производит совместимый вывод «из коробки» при целевом commonjs. Webpack начиная с [v5.66.0+](https://github.com/webpack/webpack/releases/tag/v5.66.0) делает это с новым типом вывода [`commonjs-static`](https://webpack.js.org/configuration/output/#type-commonjs-static) (до этого ни один из вариантов commonjs не производил совместимый вывод). В настоящее время это невозможно с [esbuild](https://esbuild.github.io/) (который производит нестатический `exports`).

Приведённый ниже рабочий пример был создан до недавнего релиза Webpack, поэтому он использует Rollup (я как-нибудь доберусь и до добавления варианта с Webpack).

Эти примеры предполагают, что файлы javascript внутри используют расширение `.js`; `"type"` в `package.json` управляет тем, как они интерпретируются:

`"type":"commonjs"` \+ `.js` → `cjs`  
`"type":"module"` \+ `.js` → `mjs`

Если ваши файлы явно _все_ используют расширения файлов `.cjs` и/или `.mjs` (ни один не использует `.js`), `"type"` избыточен.

**Рабочий пример**: [esm-with-cjs-distro](https://github.com/JakobJingleheimer/nodejs-module-config-examples/tree/main/packages/esm/dual/property-distro)
```json
{
  "name": "esm-with-cjs-distribution",
  "type": "module",
  "main": "./dist/index.cjs"
}
```

💡 Использование `"type": "module"`2 в паре с расширением файла `.cjs` (для файлов commonjs) даёт наилучшие результаты. Подробнее о том, почему, см. в разделах «В кроличью нору» и «Подводные камни» ниже.

#### Публикация CJS-дистрибутива с ESM-обёрткой

Здесь много всего происходит, и обычно это не лучший вариант.

Это тоже почти идентично разделу «Исходный код на CJS и двойной дистрибутив с использованием ESM-обёртки», но с тонкими отличиями: `"type": "module"` и некоторые расширения файлов `.cjs` в package.json.

**Рабочий пример**: [esm-with-dual-distro (wrapper)](https://github.com/JakobJingleheimer/nodejs-module-config-examples/tree/main/packages/esm/dual/wrapper-distro)
```json
{
  "name": "esm-with-cjs-and-esm-wrapper-distribution",
  "type": "module",
  "exports": {
    ".": {
      "import": "./dist/esm/wrapper.js",
      "require": "./dist/cjs/index.cjs",
      "default": "./dist/cjs/index.cjs"
    }
  }
}
```

💡 Использование `"type": "module"`2 в паре с расширением файла `.cjs` (для файлов commonjs) даёт наилучшие результаты. Подробнее о том, почему, см. в разделах «В кроличью нору» и «Подводные камни» ниже.

#### Публикация обоих полных дистрибутивов CJS и ESM

Накидайте кучу всего (с сюрпризом) и надейтесь на лучшее. Это, вероятно, самый распространённый и самый простой из вариантов «ESM в CJS и ESM», но вы за это заплатите. Это редко бывает хорошей идеей.

С точки зрения конфигурации пакета есть несколько вариантов, которые различаются в основном личными предпочтениями.

##### Пометить весь пакет как ESM и специально пометить экспорты CJS как CJS через расширение файла `.cjs`

Этот вариант несёт наименьшую нагрузку на разработку и удобство разработчика.

Это также означает, что тому инструменту сборки, каким бы он ни был, придётся производить файл дистрибутива с расширением `.cjs`. Это может потребовать связывания нескольких инструментов сборки в цепочку или добавления последующего шага, чтобы переместить/переименовать файл, чтобы он имел расширение `.cjs` (например, `mv ./dist/index.js ./dist/index.cjs`). Это можно обойти, добавив последующий шаг для перемещения/переименования этих выходных файлов (например, [Rollup](https://rollupjs.org/) или [простой shell-скрипт](https://stackoverflow.com/q/21985492)).

Поддержка расширения файла `.cjs` была добавлена в 12.0.0, и его использование заставит ESM правильно распознавать файл как commonjs (`import { foo } from './foo.cjs'` работает). Однако `require()` не разрешает `.cjs` автоматически, как это происходит для `.js`, поэтому расширение файла нельзя опустить, как это принято в commonjs: `require('./foo')` не сработает, а `require('./foo.cjs')` работает. Использование его в экспортах вашего пакета не имеет недостатков: `packageJson.exports` (и `packageJson.main`) в любом случае требуют расширение файла, а потребители ссылаются на ваш пакет по полю `"name"` вашего package.json (так что они об этом блаженно не подозревают).

**Рабочий пример**: [esm-with-dual-distro](https://github.com/JakobJingleheimer/nodejs-module-config-examples/tree/main/packages/esm/dual/double-distro)
```json
{
  "type": "module",
  "exports": {
    ".": {
      "import": "./dist/esm/index.js",
      "require": "./dist/index.cjs"
    }
  }
}
```

Как вариант, вы можете использовать ключи `"default"` и `"node"`, которые менее контринтуитивны: Node.js всегда выберет вариант `"node"` (который всегда работает), а инструменты, не относящиеся к Node.js, выберут `"default"`, когда настроены на цель, отличную от node. **Это исключает опасность двойного пакета.**
```json
{
  "type": "module",
  "exports": {
    ".": {
      "node": "./dist/index.cjs",
      "default": "./dist/esm/index.js"
    }
  }
}
```

💡 Использование `"type": "module"`2 в паре с расширением файла `.cjs` (для файлов commonjs) даёт наилучшие результаты. Подробнее о том, почему, см. в разделах «В кроличью нору» и «Подводные камни» ниже.

##### Используйте расширение файла `.mjs` (или эквивалентное) для всех файлов исходного кода

Конфигурация для этого такая же, как в разделе «Исходный код на CJS и дистрибутив на обоих форматах CJS и ESM».

**Исходный код не на JavaScript**: собственная конфигурация языка, отличного от JavaScript, должна распознавать/указывать, что входные файлы являются ESM.

#### Node.js до 12.22.x

🛑 Вам не следует так делать: версии Node.js до 12.x достигли конца срока эксплуатации (End of Life) и теперь уязвимы для серьёзных эксплойтов безопасности.

Если вы исследователь безопасности, которому нужно изучить Node.js до v12.22.x, свяжитесь с нами за помощью в настройке.

## Общие замечания

[Определение синтаксиса (Syntax detection)](https://nodejs.org/api/packages.html#syntax-detection) — это _**не**_ замена надлежащей конфигурации пакета; определение синтаксиса не безошибочно и имеет [значительные затраты на производительность](https://github.com/nodejs/node/pull/55238).

При использовании [`"exports"`](https://nodejs.org/api/packages.html#conditional-exports) в package.json обычно рекомендуется включать `"./package.json": "./package.json"`, чтобы его можно было импортировать ([`module.findPackageJSON`](https://nodejs.org/api/module.html#modulefindpackagejsonspecifier-base) не подвержен этому ограничению, но `import` может быть удобнее).

`"exports"` может быть предпочтительнее [`"main"`](https://nodejs.org/api/packages.html#main), потому что предотвращает внешний доступ к внутреннему коду (так что вы можете быть относительно уверены, что пользователи не полагаются на то, на что не должны). Если вам это не нужно, `"main"` проще и может быть для вас лучшим вариантом.

Поле `"engines"` даёт как удобное для человека, так и удобное для машины указание того, с какой версией(-ями) Node.js совместим пакет. В зависимости от используемого менеджера пакетов может быть выброшено исключение, приводящее к сбою установки, когда потребитель использует несовместимую версию Node.js (что может быть очень полезно для потребителей). Включение этого поля избавит от множества головных болей потребителей со старой версией Node.js, которые не могут использовать пакет.

## В кроличью нору

Конкретно в отношении Node.js есть 4 проблемы, которые нужно решить:

  * Определение формата файлов исходного кода (автор запускает свой собственный код)

  * Определение формата файлов дистрибутива (код, который получат потребители)

  * Оповещение о коде дистрибутива для случая, когда его делают `require()` (потребитель ожидает CJS)

  * Оповещение о коде дистрибутива для случая, когда его делают `import` (потребитель, вероятно, хочет ESM)

⚠️ Первые 2 **независимы** от последних 2.

Способ загрузки НЕ определяет формат, в котором интерпретируется файл:

  * **`exports.require`** **в package.json** **≠** **`CJS`**. `require()` НЕ и не может слепо интерпретировать файл как CJS; например, `require('foo.json')` правильно интерпретирует файл как JSON, а не CJS. Модуль, содержащий вызов `require()`, разумеется, должен быть CJS, но то, что он загружает, не обязательно тоже CJS.
  * **`exports.import`** **в package.json** **≠** **`ESM`**. `import` аналогично НЕ и не может слепо интерпретировать файл как ESM; `import` может загружать CJS, JSON и WASM, а также ESM. Модуль, содержащий инструкцию `import`, разумеется, должен быть ESM, но то, что он загружает, не обязательно тоже ESM.

Поэтому когда вы видите опции конфигурации, ссылающиеся на `require` или `import` или названные так, устоите перед соблазном предположить, что они предназначены для _определения_ CJS против ES Modules.

⚠️ Добавление поля (или набора полей) `"exports"` в конфигурацию пакета фактически [блокирует глубокие пути в пакет](https://nodejs.org/api/packages.html#package-entry-points) для всего, что явно не перечислено в подпутях экспортов. Это означает, что это может быть ломающим изменением.

⚠️ Тщательно взвесьте, распространять ли и CJS, и ESM: это создаёт потенциал для опасности двойного пакета (Dual Package Hazard) (особенно при неправильной настройке и если потребитель попытается схитрить). Это может привести к чрезвычайно запутанному багу в потребляющих проектах, особенно когда ваш пакет настроен неидеально. Потребители могут даже быть застигнуты врасплох промежуточным пакетом, который использует «другой» формат вашего пакета (например, потребитель использует дистрибутив ESM, а какой-то другой пакет, который потребитель также использует, сам использует дистрибутив CJS). Если ваш пакет в каком-либо смысле хранит состояние, потребление и CJS-, и ESM-дистрибутивов приведёт к параллельным состояниям (что почти наверняка непреднамеренно).

### Опасность двойного пакета

Когда приложение использует пакет, который предоставляет исходники и в CommonJS, и в ES-модулях, существует риск определённых багов, если загружаются оба экземпляра пакета. Этот потенциал исходит из того факта, что `pkgInstance`, созданный через `const pkgInstance = require('pkg')`, — это не тот же самый `pkgInstance`, созданный через `import pkgInstance from 'pkg'` (или альтернативный основной путь вроде `'pkg/module'`). Это и есть «опасность двойного пакета», когда два экземпляра одного и того же пакета могут быть загружены в рамках одной среды выполнения. Хотя маловероятно, что приложение или пакет намеренно загрузит оба экземпляра напрямую, часто бывает так, что приложение загружает одну копию, а зависимость приложения загружает другую копию. Эта опасность может возникнуть, потому что Node.js поддерживает смешивание модулей CommonJS и ES, и это может привести к неожиданному и запутанному поведению.

Если основной экспорт пакета — это конструктор, сравнение `instanceof` экземпляров, созданных двумя копиями, возвращает `false`, а если экспорт — это объект, свойства, добавленные к одному (например, `pkgInstance.foo = 3`), отсутствуют в другом. Это отличается от того, как работают инструкции `import` и `require` в окружениях полностью на CommonJS или полностью на ES-модулях соответственно, и поэтому удивляет пользователей. Это также отличается от поведения, к которому пользователи привыкли при использовании транспиляции с помощью инструментов вроде [Babel](https://babeljs.io/) или [`esm`](https://github.com/standard-things/esm#readme).

### Как мы сюда попали

[CommonJS (CJS)](https://wiki.commonjs.org/wiki/Modules) был создан _задолго_ до ECMAScript Modules (ESM), ещё когда JavaScript был подростком — CJS и jQuery появились с разницей всего в 3 года. CJS не является официальным ([TC39](https://tc39.es)) стандартом и поддерживается лишь ограниченным числом платформ (в первую очередь Node.js). ESM как стандарт был на подходе несколько лет; сейчас он поддерживается всеми основными платформами (браузеры, Deno, Node.js и т. д.), а значит, будет работать практически везде. Когда стало ясно, что ESM фактически заменит CJS (который всё ещё очень популярен и широко распространён), многие пытались перейти на него рано, зачастую до того, как тот или иной аспект спецификации ESM был окончательно закреплён. Из-за этого те решения менялись со временем по мере появления более точной информации (часто на основе выводов/опыта тех нетерпеливых энтузиастов), переходя от догадок к соответствию спецификации.

Дополнительное усложнение — бандлеры, которые исторически управляли значительной частью этой территории. Однако многое из того, для управления чем нам раньше требовались бандлеры, теперь является нативной функциональностью; тем не менее бандлеры всё ещё (и, вероятно, всегда будут) необходимы для некоторых вещей. К сожалению, функциональность, которую бандлерам больше не нужно предоставлять, глубоко укоренена в реализациях более старых бандлеров, поэтому они порой могут быть слишком услужливыми, а в некоторых случаях — антипаттерном (авторы бандлеров сами часто не рекомендуют упаковывать библиотеку в бандл). Как и почему — это тема для отдельной статьи.

## Подводные камни

Поле `"type"` в `package.json` меняет значение расширения файла `.js` на `commonjs` или ES `module` соответственно. В двойных/смешанных пакетах (содержащих и CJS, и ESM) очень часто это поле используют неправильно.
```json
{
  "type": "module",
  "main": "./dist/CJS/index.js",
  "exports": {
    ".": {
      "import": "./dist/esm/index.js",
      "require": "./dist/cjs/index.js",
      "default": "./dist/cjs/index.js"
    },
    "./package.json": "./package.json"
  }
}
```

Это не работает, потому что `"type": "module"` заставляет `packageJson.main`, `packageJson.exports["."].require` и `packageJson.exports["."].default` интерпретироваться как ESM (хотя на самом деле они CJS).

Исключение `"type": "module"` создаёт противоположную проблему:
```json
{
  "main": "./dist/CJS/index.js",
  "exports": {
    ".": {
      "import": "./dist/esm/index.js",
      "require": "./dist/cjs/index.js",
      "default": "./dist/cjs/index.js"
    },
    "./package.json": "./package.json"
  }
}
```

Это не работает, потому что `packageJson.exports["."].import` будет интерпретироваться как CJS (хотя на самом деле это ESM).

## Сноски

  1. В Node.js v13.0–13.6 был баг, из-за которого `packageJson.exports["."]` должен был быть массивом, где подробные опции конфигурации шли первым элементом (в виде объекта), а «default» — вторым элементом (в виде строки). См. [nodejs/modules#446](https://github.com/nodejs/modules/issues/446). ↩

  2. Поле `"type"` в package.json меняет то, что означает расширение файла `.js`, аналогично [атрибуту type HTML-элемента script](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/script#attr-type). ↩ ↩2 ↩3 ↩4
