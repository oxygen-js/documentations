# Введение в менеджер пакетов npm

## Знакомство с npm

`npm` — это стандартный менеджер пакетов для Node.js.

По данным на сентябрь 2022 года в реестре npm было зарегистрировано более 2,1 миллиона пакетов, что делает его крупнейшим на Земле репозиторием кода для одного языка, и можно быть уверенным, что найдётся пакет практически для всего.

Он появился как способ загружать зависимости пакетов Node.js и управлять ими, но с тех пор превратился в инструмент, используемый также и во фронтенд-разработке на JavaScript.

> [**Yarn**](https://yarnpkg.com) и [**pnpm**](https://pnpm.io) — альтернативы npm cli. Вы можете ознакомиться и с ними.

## Пакеты

`npm` устанавливает, обновляет и управляет загрузкой зависимостей вашего проекта. Зависимости — это заранее собранные фрагменты кода, такие как библиотеки и пакеты, которые нужны вашему приложению Node.js для работы.

### Установка всех зависимостей

Если в проекте есть файл `package.json`, то, запустив
```
npm install
```

вы установите всё, что нужно проекту, в папку `node_modules`, создав её, если она ещё не существует.

### Установка отдельного пакета

Вы также можете установить конкретный пакет, запустив
```
npm install <package-name>
```

Более того, начиная с npm 5, эта команда добавляет `<package-name>` в раздел _dependencies_ файла `package.json`. До версии 5 для этого нужно было указывать флаг `--save`.

Часто вы будете встречать эту команду с дополнительными флагами:

  * `--save-dev` устанавливает пакет и добавляет запись в раздел _devDependencies_ файла `package.json`
  * `--no-save` устанавливает пакет, но не добавляет запись в раздел _dependencies_ файла `package.json`
  * `--save-optional` устанавливает пакет и добавляет запись в раздел _optionalDependencies_ файла `package.json`
  * `--no-optional` предотвращает установку опциональных зависимостей

Можно использовать и сокращённые формы флагов:

  * -S: `--save`
  * -D: `--save-dev`
  * -O: `--save-optional`

Разница между _devDependencies_ и _dependencies_ в том, что первый раздел содержит инструменты разработки, например библиотеку для тестирования, тогда как второй поставляется вместе с приложением в продакшене.

Что касается _optionalDependencies_, разница в том, что сбой сборки такой зависимости не приведёт к сбою установки. Но именно ваша программа отвечает за обработку отсутствия зависимости. Подробнее об [опциональных зависимостях](https://docs.npmjs.com/cli/configuring-npm/package-json#optionaldependencies).

### Обновление пакетов

Обновление также сделано простым — достаточно запустить
```
npm update
```

`npm` проверит все пакеты на наличие более новой версии, которая удовлетворяет вашим ограничениям версий.

Вы также можете указать для обновления отдельный пакет:
```
npm update <package-name>
```

## Версионирование

Помимо простой загрузки, `npm` также управляет **версионированием**, поэтому вы можете указать любую конкретную версию пакета либо потребовать версию выше или ниже той, что вам нужна.

Часто вы будете сталкиваться с тем, что библиотека совместима только с определённым мажорным релизом другой библиотеки.

Или же баг в последнем релизе библиотеки, всё ещё не исправленный, вызывает проблему.

Указание явной версии библиотеки также помогает держать всех на одной и той же точной версии пакета, чтобы вся команда работала с одинаковой версией, пока файл `package.json` не будет обновлён.

Во всех этих случаях версионирование очень помогает, и `npm` следует стандарту семантического версионирования (semver).

Вы можете установить конкретную версию пакета, запустив
```
npm install <package-name>@<version>
```

## Запуск задач

Файл package.json поддерживает формат для указания задач командной строки, которые можно запускать с помощью
```
npm run <task-name>
```

Например:
```json
{
  "scripts": {
    "start-dev": "node lib/server-development",
    "start": "node lib/server-production"
  }
}
```

Очень часто эту возможность используют для запуска Webpack:
```json
{
  "scripts": {
    "watch": "webpack --watch --progress --colors --config webpack.conf.js",
    "dev": "webpack --progress --colors --config webpack.conf.js",
    "prod": "NODE_ENV=production webpack -p --config webpack.conf.js"
  }
}
```

Так что вместо того чтобы набирать эти длинные команды, которые легко забыть или ошибиться в них, вы можете запустить
```
$ npm run watch
$ npm run dev
$ npm run prod
```
