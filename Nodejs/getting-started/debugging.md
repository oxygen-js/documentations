# Отладка Node.js

Это руководство поможет вам приступить к отладке ваших приложений и скриптов Node.js.

## Включение инспектора

При запуске с ключом `--inspect` процесс Node.js прослушивает подключение отлаживающего клиента. По умолчанию он слушает на хосте и порту 127.0.0.1:9229. Каждому процессу также назначается уникальный [UUID](https://tools.ietf.org/html/rfc4122).

Клиенты инспектора должны знать и указывать адрес хоста, порт и UUID для подключения. Полный URL будет выглядеть примерно так: `ws://127.0.0.1:9229/0f2c936f-b1cd-4ac9-aab3-f63b0f33d55e`.

Node.js также начнёт прослушивать сообщения отладки, если получит сигнал `SIGUSR1`. (`SIGUSR1` недоступен в Windows.) В Node.js 7 и более ранних версиях это активирует устаревший Debugger API. В Node.js 8 и более поздних версиях это активирует Inspector API.

## Последствия для безопасности

Поскольку отладчик имеет полный доступ к среде выполнения Node.js, злоумышленник, способный подключиться к этому порту, может выполнить произвольный код от имени процесса Node.js. Важно понимать последствия открытия порта отладчика в публичных и частных сетях для безопасности.

### Открывать порт отладки публично небезопасно

Если отладчик привязан к публичному IP-адресу или к 0.0.0.0, любые клиенты, способные достичь вашего IP-адреса, смогут подключиться к отладчику без каких-либо ограничений и выполнить произвольный код.

По умолчанию `node --inspect` привязывается к 127.0.0.1. Вам нужно явно указать публичный IP-адрес, 0.0.0.0 и т. п., если вы намерены разрешить внешние подключения к отладчику. Это может подвергнуть вас потенциально значительной угрозе безопасности. Мы рекомендуем убедиться, что настроены соответствующие межсетевые экраны и средства контроля доступа, чтобы предотвратить угрозу безопасности.

См. раздел «Включение сценариев удалённой отладки», где приведены советы о том, как безопасно разрешить подключение удалённых клиентов отладчика.

### Локальные приложения имеют полный доступ к инспектору

Даже если вы привяжете порт инспектора к 127.0.0.1 (по умолчанию), любые приложения, работающие локально на вашей машине, будут иметь неограниченный доступ. Это сделано намеренно, чтобы локальные отладчики могли удобно подключаться.

### Браузеры, WebSocket и политика одного источника

Веб-сайты, открытые в веб-браузере, могут отправлять запросы WebSocket и HTTP в рамках модели безопасности браузера. Начальное HTTP-подключение необходимо для получения уникального идентификатора сеанса отладчика. Политика одного источника (same-origin policy) не позволяет веб-сайтам устанавливать это HTTP-подключение. Для дополнительной защиты от [атак типа DNS rebinding](https://en.wikipedia.org/wiki/DNS_rebinding) Node.js проверяет, что заголовки 'Host' для подключения указывают либо IP-адрес, либо в точности `localhost`.

Эти политики безопасности запрещают подключение к удалённому серверу отладки путём указания имени хоста. Вы можете обойти это ограничение, указав либо IP-адрес, либо используя ssh-туннели, как описано ниже.

## Клиенты инспектора

Минимальный CLI-отладчик доступен с помощью `node inspect myscript.js`. К инспектору Node.js могут подключаться и несколько коммерческих инструментов и инструментов с открытым исходным кодом.

### Chrome DevTools 55+, Microsoft Edge

#### Вариант 1: использовать встроенный интерфейс DevTools

  * Откройте `chrome://inspect` (`edge://inspect` в Microsoft Edge) в своём браузере.
  * Нажмите кнопку Configure и убедитесь, что целевой хост и порт указаны в списке.
  * Ваше приложение Node.js должно появиться в списке Remote Target.

#### Вариант 2: подключиться вручную

  * Перейдите на `http://localhost:<inspect-port>/json/list`. Он должен вернуть объект JSON, содержащий `devtoolsFrontendUrl`.
  * Скопируйте значение `devtoolsFrontendUrl` из ответа и вставьте его в адресную строку браузера.

Дополнительную информацию см. в [Chrome DevTools Frontend](https://github.com/ChromeDevTools/devtools-frontend) и [руководстве по Microsoft Edge DevTools](https://learn.microsoft.com/microsoft-edge/devtools-guide-chromium/).

### Visual Studio Code 1.10+

  * На панели Debug нажмите значок настроек, чтобы открыть `.vscode/launch.json`. Выберите «Node.js» для начальной настройки.

Дополнительную информацию см. на <https://code.visualstudio.com/docs/nodejs/nodejs-debugging>.

### Visual Studio 2017+

  * Выберите «Debug > Start Debugging» в меню или нажмите F5.
  * [Подробные инструкции](https://github.com/Microsoft/nodejstools/wiki/Debugging).

### JetBrains WebStorm и другие IDE от JetBrains

  * Создайте новую конфигурацию отладки Node.js и нажмите Debug. Для Node.js 7+ по умолчанию будет использоваться `--inspect`. Чтобы отключить это, снимите флажок `js.debugger.node.use.inspect` в реестре IDE (IDE Registry). Чтобы узнать больше о запуске и отладке Node.js в WebStorm и других IDE от JetBrains, ознакомьтесь с [онлайн-справкой WebStorm](https://www.jetbrains.com/help/webstorm/running-and-debugging-node-js.html).

### chrome-remote-interface

  * Библиотека для упрощения подключений к конечным точкам [Inspector Protocol](https://chromedevtools.github.io/debugger-protocol-viewer/v8/).

Дополнительную информацию см. на <https://github.com/cyrus-and/chrome-remote-interface>.

### Eclipse IDE с расширением Eclipse Wild Web Developer

  * Из файла .js выберите «Debug As... > Node program», или
  * Создайте конфигурацию отладки для подключения отладчика к работающему приложению Node.js (уже запущенному с `--inspect`).

Дополнительную информацию см. на <https://eclipseide.org/>.

## Включение сценариев удалённой отладки

Мы рекомендуем никогда не позволять отладчику слушать на публичном IP-адресе. Если вам нужно разрешить удалённые подключения для отладки, мы рекомендуем вместо этого использовать ssh-туннели. Мы приводим следующий пример исключительно в иллюстративных целях. Пожалуйста, осознайте риск для безопасности, связанный с разрешением удалённого доступа к привилегированному сервису, прежде чем продолжить.

Допустим, вы запускаете Node.js на удалённой машине remote.example.com, которую хотите иметь возможность отлаживать. На этой машине вам следует запустить процесс node так, чтобы инспектор слушал только localhost (по умолчанию).
```
node --inspect server.js
```

Теперь на вашей локальной машине, с которой вы хотите инициировать подключение клиента отладки, вы можете настроить ssh-туннель:
```
ssh -L 9221:localhost:9229 [[email protected]](/cdn-cgi/l/email-protection)
```

Это запускает сеанс ssh-туннеля, при котором подключение к порту 9221 на вашей локальной машине будет перенаправлено на порт 9229 на remote.example.com. Теперь вы можете подключить отладчик, такой как Chrome DevTools или Visual Studio Code, к localhost:9221, который должен иметь возможность отлаживать так, как если бы приложение Node.js выполнялось локально.

## Устаревший отладчик

**Устаревший отладчик признан устаревшим начиная с Node.js 7.7.0. Пожалуйста, используйте вместо него`--inspect` и Inspector.**

При запуске с ключами **\--debug** или **\--debug-brk** в версии 7 и более ранних Node.js прослушивает команды отладки, определённые прекращённым протоколом отладки V8 (V8 Debugging Protocol), на TCP-порту, по умолчанию `5858`. Любой клиент отладчика, который «говорит» на этом протоколе, может подключиться к работающему процессу и отлаживать его; пара популярных из них перечислены ниже.

V8 Debugging Protocol больше не поддерживается и не документируется.

### Встроенный отладчик

Запустите `node debug script_name.js`, чтобы запустить ваш скрипт под встроенным отладчиком командной строки. Ваш скрипт запускается в другом процессе Node.js, запущенном с опцией `--debug-brk`, а исходный процесс Node.js выполняет скрипт `_debugger.js` и подключается к вашей цели. Дополнительную информацию см. в [документации](https://nodejs.org/dist/latest/docs/api/debugger.html).

### node-inspector

Отлаживайте своё приложение Node.js с помощью Chrome DevTools, используя промежуточный процесс, который переводит [Inspector Protocol](https://chromedevtools.github.io/debugger-protocol-viewer/v8/), используемый в Chromium, в протокол отладчика V8, используемый в Node.js. Дополнительную информацию см. на <https://github.com/node-inspector/node-inspector>.
