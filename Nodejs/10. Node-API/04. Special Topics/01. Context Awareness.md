# Учёт контекста

Исторически Node.js работал как однопоточный процесс. Всё изменилось с появлением [рабочих потоков (Worker Threads)](https://nodejs.org/api/worker_threads.html#worker_threads_worker_threads) в Node 10. Рабочие потоки добавляют дружественную к JavaScript абстракцию конкурентности, о которой должны знать разработчики нативных аддонов. На практике это означает, что ваш нативный аддон может загружаться и выгружаться более одного раза, а его код может выполняться конкурентно в нескольких потоках выполнения. Есть определённые шаги, которые вы должны предпринять, чтобы гарантировать корректную работу кода вашего нативного аддона.

Модель рабочих потоков предписывает, что каждый Worker работает полностью независимо от других и взаимодействует с родительским Worker через объект MessagePort, предоставляемый родителем. Это делает рабочие потоки по сути изолированными друг от друга. То же самое верно и для вашего нативного аддона.

Каждый рабочий поток работает в своей собственной среде, которую также называют контекстом. Контекст доступен каждой функции Node-API как значение [`napi_env`](https://nodejs.org/api/n-api.html#napi_env).

## Множественная загрузка и выгрузка

Если вашему нативному аддону требуется постоянная память, выделение этой памяти в статическом глобальном пространстве — это верный путь к катастрофе. Вместо этого _необходимо_, чтобы эта память выделялась каждый раз внутри контекста, в котором инициализируется нативный аддон. Обычно эта память выделяется в методе `Init` вашего нативного аддона. Но в некоторых случаях она также может выделяться во время работы вашего нативного аддона.

Помимо описанной выше множественной загрузки, ваш нативный аддон также подвержен автоматической выгрузке сборщиком мусора движка JavaScript-среды выполнения, когда ваш нативный аддон больше не используется. Чтобы предотвратить утечки памяти, любая память, выделенная вашим нативным аддоном, _должна_ быть освобождена при выгрузке вашего нативного аддона.

В следующих разделах описаны две разные техники, которые вы можете использовать для выделения и освобождения постоянной памяти, связанной с вашим нативным аддоном. Эти техники можно использовать по отдельности или вместе в вашем нативном аддоне.

## Данные экземпляра

Node-API даёт вам возможность связать один фрагмент памяти, выделяемый вашим нативным аддоном, с контекстом, в котором он работает. Эта техника называется «данными экземпляра» (instance data) и полезна, когда ваш нативный аддон выделяет один фрагмент данных при загрузке.

`napi_set_instance_data` позволяет вашему нативному аддону связать один выделенный фрагмент памяти с контекстом, в котором загружен ваш нативный аддон. Затем в любом месте вашего нативного аддона можно вызвать `napi_get_instance_data`, чтобы получить расположение выделенной памяти.

В вызове `napi_set_instance_data` вы указываете колбэк-финализатор. Колбэк-финализатор вызывается, когда ваш нативный аддон освобождается из памяти, и именно в нём вы должны освободить память, связанную с этим контекстом.

### Ресурсы

[Environment Life Cycle APIs](https://nodejs.org/api/n-api.html#environment-life-cycle-apis) — документация Node.js по `napi_set_instance_data` и `napi_get_instance_data`.

### Пример

В этом примере создаётся несколько рабочих потоков. Каждый рабочий поток создаёт структуру `AddonData`, которая привязывается к контексту рабочего потока с помощью вызова `napi_set_instance_data`. Со временем значение, хранящееся в структуре, увеличивается и уменьшается с помощью вычислительно затратной операции.

Спустя время рабочие потоки завершают свои операции, после чего выделенная структура освобождается в функции `DeleteAddonData`.

#### binding.c
```c
#include <assert.h>
#include <math.h>
#include <stdlib.h>

#define NAPI_EXPERIMENTAL
#include <node_api.h>

// Структура, содержащая информацию, необходимую всё время существования аддона.
// Она заменяет использование глобальных статических данных данными,
// привязанными к экземпляру аддона, связывая экземпляр этой структуры с каждым
// экземпляром этого аддона во время инициализации аддона. Затем экземпляр этой
// структуры передаётся каждой привязке, предоставляемой аддоном. Таким образом,
// данные, хранящиеся в экземпляре этой структуры, доступны каждой привязке,
// как были бы доступны глобальные статические данные.
typedef struct {
  double value;
} AddonData;

// Это и есть настоящая, полезная выполняемая работа: увеличение или уменьшение
// значения, хранящегося для каждого экземпляра аддона, после пропускания его
// через затратное по CPU, но в остальном бесполезное вычисление.
static int ModifyAddonData(AddonData* data, double offset) {
    // Затратно увеличить или уменьшить значение.
    data->value = tan(atan(exp(log(sqrt(data->value * data->value))))) + offset;

    // Округлить значение до ближайшего целого.
    data->value =
        (double)(((int)data->value) +
        (data->value - ((double)(int)data->value) > 0.5 ? 1 : 0));

    // Вернуть значение как целое число.
    return (int)(data->value);
}

// Это шаблонный код. Экземпляр структуры `AddonData`, созданный во время
// инициализации аддона, должен быть уничтожен при выгрузке аддона. Эта функция
// будет вызвана, когда объект `exports` аддона попадёт под сборку мусора.
static void DeleteAddonData(napi_env env, void* data, void* hint) {
  // Избежать предупреждений о неиспользуемых параметрах.
  (void) env;
  (void) hint;

  // Освободить данные, привязанные к экземпляру аддона.
  free(data);
}

// Это тоже шаблонный код. Он создаёт и инициализирует экземпляр структуры
// `AddonData` и привязывает его жизненный цикл к жизненному циклу объекта
// `exports` экземпляра аддона. Это означает, что данные будут доступны этому
// экземпляру аддона до тех пор, пока движок JavaScript поддерживает его живым.
static AddonData* CreateAddonData(napi_env env, napi_value exports) {
  AddonData* result = malloc(sizeof(*result));
  result->value = 0.0;
  assert(napi_set_instance_data(env, result, DeleteAddonData, NULL) == napi_ok);
  return result;
}

// Эта функция вызывается из JavaScript. Она использует затратную операцию для
// увеличения на единицу значения, хранящегося внутри структуры `AddonData`.
static napi_value Increment(napi_env env, napi_callback_info info) {
  // Получить данные, привязанные к экземпляру аддона.
  AddonData* addon_data = NULL;
  assert(napi_get_instance_data(env, ((void**)&addon_data)) == napi_ok);

  // Увеличить значение, привязанное к экземпляру аддона, и создать из него
  // новое целое число JavaScript.
  napi_value result;
  assert(napi_create_int32(env,
                           ModifyAddonData(addon_data, 1.0),
                           &result) == napi_ok);

  // Вернуть целое число JavaScript обратно в JavaScript.
  return result;
}

// Эта функция вызывается из JavaScript. Она использует затратную операцию для
// уменьшения на единицу значения, хранящегося внутри структуры `AddonData`.
static napi_value Decrement(napi_env env, napi_callback_info info) {
  // Получить данные, привязанные к экземпляру аддона.
  AddonData* addon_data = NULL;
  assert(napi_get_instance_data(env, ((void**)&addon_data)) == napi_ok);

  // Уменьшить значение, привязанное к экземпляру аддона, и создать из него
  // новое целое число JavaScript.
  napi_value result;
  assert(napi_create_int32(env,
                           ModifyAddonData(addon_data, -1.0),
                           &result) == napi_ok);

  // Вернуть целое число JavaScript обратно в JavaScript.
  return result;
}

// Инициализировать аддон таким образом, чтобы он мог быть инициализирован
// несколько раз за процесс. Телу функции, следующему за этим макросом,
// предоставляется значение `env`, имеющее тип `napi_env`, и значение `exports`,
// имеющее тип `napi_value` и ссылающееся на JavaScript-объект, который в
// конечном счёте содержит функции, которые этот аддон хочет предоставить.
// В конце функция должна вернуть `napi_value`. Она может вернуть `exports`,
// либо может создать новое `napi_value` и вернуть его.
NAPI_MODULE_INIT(/*env, exports*/) {
  // Создать новый экземпляр данных для экземпляра, который будет связан с
  // инициализируемым здесь экземпляром аддона и который будет уничтожен вместе
  // с экземпляром аддона.
  AddonData* addon_data = CreateAddonData(env, exports);

  // Объявить привязки, предоставляемые этим аддоном. Созданные выше данные
  // передаются как последний параметр-инициализатор и будут переданы привязке
  // при её вызове.
  napi_property_descriptor bindings[] = {
    {"increment", NULL, Increment, NULL, NULL, NULL, napi_enumerable, addon_data},
    {"decrement", NULL, Decrement, NULL, NULL, NULL, napi_enumerable, addon_data}
  };

  // Предоставить JavaScript две объявленные выше привязки.
  assert(napi_define_properties(env,
                                exports,
                                sizeof(bindings) / sizeof(bindings[0]),
                                bindings) == napi_ok);

  // Вернуть предоставленный объект `exports`. Теперь у него два новых свойства,
  // которые являются функциями, которые мы хотим предоставить JavaScript.
  return exports;
}
```

#### index.js
```js
// Пример, иллюстрирующий случай, когда нативный аддон загружается несколько раз.
// Весь этот файл выполняется дважды, конкурентно — один раз в главном потоке
// выполнения и один раз в потоке, запущенном из главного потока.

// Мы загружаем модуль worker threads, который позволяет нам запускать несколько
// сред Node.js, каждую в своём собственном потоке выполнения.
const { Worker, isMainThread } = require('worker_threads');

// Мы загружаем нативный аддон.
const addon = require('bindings')('multiple_load');

// Количество итераций можно подстроить так, чтобы вывод от двух потоков
// чередовался. Слишком мало итераций — и вывод одного потока следует за выводом
// другого, что не очень хорошо иллюстрирует конкурентность.
const iterations = 1000;

// Эта функция — холостой цикл, выполняющий случайное блуждание от 0 путём
// вызова нативного аддона для увеличения или уменьшения начального значения.
function useAddon(addon, prefix, iterations) {
  if (iterations >= 0) {
    if (Math.random() < 0.5) {
      console.log(prefix + ': new value (decremented): ' + addon.decrement());
    } else {
      console.log(prefix + ': new value (incremented): ' + addon.increment());
    }
    setImmediate(() => useAddon(addon, prefix, --iterations));
  }
}

if (isMainThread) {
  // В главном потоке мы запускаем worker и ждём, пока он выйдет в онлайн. Затем
  // мы запускаем цикл.
  new Worker(__filename).on('online', () =>
    useAddon(addon, 'Main thread', iterations)
  );
} else {
  // Во вторичном потоке мы немедленно запускаем цикл.
  useAddon(addon, 'Worker thread', iterations);
}
```

## Хуки очистки

Ваш нативный аддон может получить одно или несколько уведомлений от движка среды выполнения Node.js, когда контекст, в котором работал ваш нативный аддон, уничтожается. Это даёт вашему нативному аддону возможность освободить любую выделенную память до того, как контекст будет уничтожен движком среды выполнения Node.js.

Преимущество этой техники в том, что ваш нативный аддон может выделить несколько фрагментов памяти для привязки к контексту, в котором работает ваш нативный аддон. Это может быть полезно, если вам нужно выделить несколько буферов памяти из разных участков кода во время работы вашего нативного аддона.

Недостаток в том, что если вам нужно обращаться к этим выделенным буферам, вы сами отвечаете за отслеживание указателей внутри контекста, в котором работает ваш нативный аддон. В зависимости от архитектуры вашего нативного аддона это может быть или не быть проблемой.

### Ресурсы

[Cleanup on exit of the current Node.js instance](https://nodejs.org/api/n-api.html#cleanup-on-exit-of-the-current-nodejs-instance) — документация Node.js по `napi_add_env_cleanup_hook` и `napi_remove_env_cleanup_hook`.

### Пример

Поскольку отслеживание выделенных буферов зависит от архитектуры нативного аддона, это тривиальный пример, показывающий, как буферы могут выделяться и освобождаться.

#### binding.cc
```cpp
#include <stdlib.h>
#include <stdio.h>
#include "node_api.h"

namespace {

void CleanupHook (void* arg) {
  printf("cleanup(%d)\n", *static_cast<int*>(arg));
  free(arg);
}

napi_value Init(napi_env env, napi_value exports) {
  for (int i = 1; i < 5; i++) {
    int* value = (int*)malloc(sizeof(*value));
    *value = i;
    napi_add_env_cleanup_hook(env, CleanupHook, value);
  }
  return exports;
}

}  // анонимное пространство имён

NAPI_MODULE(NODE_GYP_MODULE_NAME, Init)
```

#### index.js
```js
'use strict';

// Мы загружаем нативный аддон.
const addon = require('bindings')('multiple_load');
const assert = require('assert');
const child_process = require('child_process');

assert.ok(addon);

if (process.argv[2] === 'child') {
  const childAddon = require('bindings')('multiple_load');
  assert.ok(childAddon);
  process.exit(0);
}

const child = child_process.fork(__filename, ['child'], {
  stdio: 'inherit',
});

child.on('exit', code => {
  assert.strictEqual(code, 0);
});
```
