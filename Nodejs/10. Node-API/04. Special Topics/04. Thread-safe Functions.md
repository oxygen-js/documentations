# Потокобезопасные функции

JavaScript-функции обычно можно вызывать только из главного потока выполнения нативного аддона. Если аддон создаёт дополнительные потоки выполнения, то функции node-addon-api, которым требуется `Napi::Env`, `Napi::Value` или `Napi::Reference`, не должны вызываться из этих потоков.

Когда у аддона есть дополнительные потоки выполнения и JavaScript-функции необходимо вызывать на основе обработки, завершённой этими потоками, эти потоки должны взаимодействовать с главным потоком аддона, чтобы главный поток мог вызвать JavaScript-функцию от их имени. API потокобезопасных функций предоставляют простой способ сделать это.

Потокобезопасная функция создаётся в главном потоке выполнения через [ThreadSafeFunction::New](https://github.com/nodejs/node-addon-api/blob/main/doc/threadsafe_function.md#new):
```cpp
New(napi_env env,
    const Function& callback,
    const Object& resource,
    ResourceString resourceName,
    size_t maxQueueSize,
    size_t initialThreadCount,
    ContextType* context,
    Finalizer finalizeCallback,
    FinalizerDataType* data);
```

Потокобезопасная функция инкапсулирует:

  * Очередь сообщений: запросы на запуск JavaScript-функции помещаются в очередь и обрабатываются асинхронно главным потоком выполнения. Количество записей, разрешённых в очереди, прежде чем `NonBlockingCall()` вернёт ошибку «очередь заполнена», управляется параметром `maxQueueSize` (укажите `0` для неограниченной очереди).
  * JavaScript-функция: колбэк для запуска (параметр `callback`). Эта функция либо (a) автоматически выполняется без аргументов при вызове через перегрузки `[Non]BlockingCall()` без аргументов, либо (b) передаётся как аргумент в функцию колбэка, предоставленную в перегрузках `[Non]BlockingCall(DataType* data, Callback callback)`.
  * Контекст: необязательные, произвольные данные (параметр `context`), связываемые с потокобезопасной функцией.
  * Финализатор: необязательный колбэк (параметр `finalizeCallback`), выполняемый при уничтожении потокобезопасной функции, когда все потоки выполнения закончили её использование.
  * Данные финализатора: необязательные данные (параметр `data`), предоставляемые колбэку-финализатору.

## Вызов потокобезопасной функции

Потоки выполнения могут вызывать JavaScript через [`[Non]BlockingCall`](https://github.com/nodejs/node-addon-api/blob/main/doc/threadsafe_function.md#blockingcall--nonblockingcall). Это добавит запись в очередь соответствующей потокобезопасной функции, которая будет обработана асинхронно в главном потоке выполнения во время его обработки цикла событий.

## Управление потоками выполнения

Несколько потоков выполнения могут одновременно использовать потокобезопасную функцию. Потокобезопасная функция управляет своим жизненным циклом, подсчитывая число потоков, активно её использующих. Это число начинается со значения параметра начального количества потоков в `New()`, увеличивается через `Acquire()` и уменьшается через `Release()`. Как только число активных потоков достигает нуля, потокобезопасная функция уничтожается, выполняя колбэк-финализатор в главном потоке выполнения, если он был предоставлен.

Вот два общих подхода к использованию потокобезопасных функций в приложениях:

### Известное число потоков выполнения

Если число потоков выполнения известно на момент создания потокобезопасной функции, задайте параметру `initial_thread_count` это число в вызове `New()`. Каждый поток будет иметь собственный доступ к потокобезопасной функции, пока не вызовет `Release()`. Как только все потоки вызвали `Release()`, потокобезопасная функция уничтожается.

### Создание потоков выполнения

Ещё один распространённый вариант использования — динамическое создание и уничтожение потоков выполнения на основе различной логики во время выполнения. Один из способов справиться с этим сценарием — предоставить несколько нативных JavaScript-функций, которые взаимодействуют с API потокобезопасных функций, следующим образом:

  1. Создание потокобезопасной функции через `New()` с начальным количеством потоков, равным `1`.
  2. Вызов `Acquire()` и создание нового нативного потока выполнения. Новый поток теперь может использовать `[Non]BlockingCall()`.
  3. Инициирование очистки/уничтожения, например, следующим образом ...

  * вызывая `Abort()` и заставляя каждый поток вызвать либо `[Non]BlockingCall()`, либо `Release()`
  * используя собственную логику с другими потокобезопасными API, чтобы гарантировать, что все потоки вызовут `Release()` для уменьшения числа активных потоков до `0`.

## Часто задаваемые вопросы

### В: Моё приложение не завершается корректно. Оно просто зависает.

По умолчанию Node будет ждать финализации потокобезопасной функции, прежде чем выполнить очистку и завершиться. См. раздел «Управление потоками выполнения». Это поведение можно изменить вызовом `Unref()`, что позволит Node выполнить очистку, не дожидаясь, пока число потоков достигнет нуля. Вызов `Ref()` вернёт потокобезопасную функцию к прежнему поведению при завершении, требуя, чтобы все использующие её потоки выполнили `Release()` и/или `Abort()`.

### В: Если поток получает `napi_closing` от вызова `[Non]BlockingCall()`, нужно ли ему всё равно вызывать `Release()`?

Нет. Возвращаемое значение `napi_closing` должно означать для потока, что потокобезопасную функцию больше нельзя использовать. Это _включает_ вызов `Release()`.
