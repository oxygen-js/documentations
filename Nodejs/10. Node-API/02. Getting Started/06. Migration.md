# Миграция на Node-API

Цель этого руководства — дать вам хорошее представление о необходимых шагах и доступных инструментах для миграции существующего нативного аддон-модуля Node на базе [NAN](https://github.com/nodejs/nan) на [Node-API](https://nodejs.org/api/n-api.html) с использованием пакета [node-addon-api](https://github.com/nodejs/node-addon-api).

В этом руководстве используется инструмент конвертации, поставляемый вместе с Node-API, чтобы дать вам фору в миграции. Однако инструмент конвертации сделает лишь часть работы. По-прежнему требуется дополнительная ручная доработка, как описано ниже.

Чтобы несколько ограничить масштаб, в этом руководстве используется [node-microtime](https://github.com/wadey/node-microtime) — простой нативный аддон на базе NAN. Этот аддон делает системные вызовы для определения текущего времени с точностью до микросекунд, если это поддерживается операционной системой.

Прежде чем начать, убедитесь, что у вас установлены все необходимые [предварительные требования](<01. Prerequisites.md>) и [инструменты](<02. Tools.md>).

> Node-API стабилен во всех поддерживаемых на данный момент выпусках Node.js. Используйте [выпуск Active LTS или Maintenance LTS](https://nodejs.org/en/about/releases/). Проверить версию Node.js, которую вы используете, можно с помощью `node -v`.

## Клонирование `node-microtime`

В качестве первого шага клонируйте GitHub-репозиторий [node-microtime](https://github.com/wadey/node-microtime) на свою систему:
```
git clone https://github.com/wadey/node-microtime.git
```

Прежде чем вносить изменения, стоит сначала собрать и протестировать `node-microtime`, чтобы убедиться, что необходимые инструменты разработки установлены и настроены правильно.

> Поскольку `node-microtime` уже мигрировал на `node-addon-api`, вам нужно переключиться на тег `v2.1.9`, чтобы следовать этому руководству.
```bash
cd node-microtime
git checkout tags/v2.1.9
npm install
npm test
```

Команда `npm install` запускает процесс сборки, а `npm test` выполняет код. Вы можете увидеть предупреждения компилятора, которые не влияют на возможность запуска кода. При успешной сборке и запуске вы должны увидеть вывод, который выглядит примерно так:
```
microtime.now() = 1526334357974754
microtime.nowDouble() = 1526334357.976626
microtime.nowStruct() = [ 1526334357, 976748 ]

Guessing clock resolution...
Clock resolution observed: 1us
```

## Запуск инструмента конвертации

После того как базовая работа кода проверена, следующий шаг — запустить [инструмент конвертации Node-API (Node-API Conversion Tool)](https://github.com/nodejs/node-addon-api/blob/main/doc/conversion-tool.md). Учтите, что инструмент конвертации _заменяет файлы на месте_. Никогда не запускайте инструмент конвертации на единственной копии своего проекта. И, очевидно, запускать его нужно только _один раз_.
```bash
npm install --save node-addon-api
node ./node_modules/node-addon-api/tools/conversion.js ./
```

Для этого небольшого проекта инструмент конвертации отрабатывает очень быстро. На данном этапе инструмент конвертации изменил следующие файлы проекта:

  * `binding.gyp`
  * `package.json`
  * `src/microtime.cc`

Теперь пересоберите сконвертированный код:
```
npm install
```

Как вы увидите, есть одна или несколько ошибок компиляции, которые нужно устранить. Иногда их бывает довольно много, но ничего непреодолимого.

## Доработка

Инструмент конвертации не может предусмотреть каждую ситуацию в коде. Поэтому обычно возникают проблемы, которые нужно устранять вручную. Ниже приведены проблемы, с которыми вы, скорее всего, столкнётесь в этом проекте. Лучший подход — устранять каждую проблему по очереди и пытаться выполнить `npm install` после устранения каждой из них, пока ошибок не останется.

### Cannot find module 'nan'

Эта ошибка, а также её аналог, когда не удаётся найти `napi.h`, возникает из-за отсутствия кода в файле `binding.gyp`. Для этого проекта вы увидите в binding.gyp такой код:
```
'include_dirs' : [ '<!(node -e "require(\'nan\')")' ]
```

Каталоги включаемых файлов C/C++ по-прежнему указывают на NAN. Вместо этого они должны указывать на Node-API. Приведённую выше строку следует заменить на такую:
```
'include_dirs' : [ "<!@(node -p \"require('node-addon-api').include\")" ]
```

В других проектах вы можете получить ошибку, когда не удаётся найти `napi.h`. Причина та же, что и здесь. Свойство `include_dirs` каждой цели (target) должно включать ссылку на `node-addon-api`, как показано выше.

### Use of undeclared identifier 'env'

Три C++-функции — `Now`, `NowDouble` и `NowStruct` — каждая ссылается на переменную `env`, которая не определена. `env` предназначена для хранения переменной [окружения Node-API (Node-API Environment)](https://github.com/nodejs/node-addon-api/blob/main/doc/env.md), которая необходима почти для всех вызовов Node-API. Значение для `env` можно легко получить из аргумента `Napi::CallbackInfo`, передаваемого в каждую C++-функцию. Один из способов исправить эту ошибку — добавить следующий код в качестве первой строки в теле каждой функции:
```
Napi::Env env = info.Env();
```

Альтернативой было бы заменить каждое вхождение `env` в этих трёх функциях на `info.Env()`. Выбор за вами.

### Использование `void` в качестве возвращаемого значения

Каждая из трёх C++-функций — `Now`, `NowDouble` и `NowStruct` — определена так, что возвращает значение `void`. На самом деле каждая из них должна возвращать значение JavaScript. Лучший способ добиться этого — заменить `void` на `Napi::Value`. Это позволяет каждой из функций возвращать значение JavaScript неопределённого типа. Это может быть любое значение JavaScript, включая String, Number, Boolean, Array и т. д. Вот как они должны выглядеть:
```cpp
Napi::Value Now(const Napi::CallbackInfo&info) {
Napi::Value NowDouble(const Napi::CallbackInfo&info) {
Napi::Value NowStruct(const Napi::CallbackInfo&info) {
```

### Новый способ определения `exports`

Node-API использует другую технику для определения объекта `exports`.

Код:
```cpp
Nan::Export(target, "now", Now);
Nan::Export(target, "nowDouble", NowDouble);
Nan::Export(target, "nowStruct", NowStruct);
```

нужно заменить на:
```cpp
exports.Set(Napi::String::New(env,"now"), Napi::Function::New(env, Now));
exports.Set(Napi::String::New(env,"nowDouble"), Napi::Function::New(env, NowDouble));
exports.Set(Napi::String::New(env,"nowStruct"), Napi::Function::New(env, NowStruct));
```

`exports` — это `Napi::Object`, представляющий объект JavaScript. Метод `Set` задаёт значение свойств объекта и принимает два аргумента: имя свойства и его значение. Оба этих аргумента должны быть значениями JavaScript.

Ещё одно изменение критически важно для работы Node-API. Функция `InitAll` _должна_ возвращать переменную `exports`. Эта строка должна быть последней строкой в теле функции:
```
return exports;
```

Отсутствие этой строки, скорее всего, приведёт к ошибке segfault во время выполнения.

### Замена для `ErrnoException`

Объект NAN `ErrnoException` не существует в Node-API. Существующий код выглядит так:
```
Napi::Error::New(env, Napi::ErrnoException(errno, "gettimeofday")).ThrowAsJavaScriptException();
```

Но его можно легко заменить кодом, который выглядит так:
```cpp
std::string msg =  "gettimeofday: " + std::string(strerror(errno));
Napi::Error::New(env, msg).ThrowAsJavaScriptException();
```

## Финальная проверка

Как только код скомпилируется без ошибок, протестируйте внесённые изменения:
```
npm test
```

Вы должны увидеть результаты, похожие на те, что были до миграции.

Поздравляем! Вы только что сконвертировали свой первый NAN-модуль в Node-API.
