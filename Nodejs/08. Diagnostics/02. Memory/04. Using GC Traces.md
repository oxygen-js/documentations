# Трассировка сборки мусора

Это руководство разбирает основы трассировки сборки мусора.

К концу этого руководства вы сможете:

  * Включать трассировку в вашем приложении Node.js
  * Интерпретировать трассировки
  * Выявлять потенциальные проблемы с памятью в вашем приложении Node.js

О работе сборщика мусора можно узнать многое, но если запомнить лишь одну вещь, то вот она: пока работает GC, ваш код не работает.

Возможно, вам захочется узнать, как часто и как долго выполняется сборка мусора и каков её результат.

## Подготовка

Для целей этого руководства мы будем использовать такой скрипт:
```js
// script.mjs

import os from 'node:os';

let len = 1_000_000;
const entries = new Set();

function addEntry() {
  const entry = {
    timestamp: Date.now(),
    memory: os.freemem(),
    totalMemory: os.totalmem(),
    uptime: os.uptime(),
  };

  entries.add(entry);
}

function summary() {
  console.log(`Total: ${entries.size} entries`);
}

// выполнение
(() => {
  while (len > 0) {
    addEntry();
    process.stdout.write(`~~> ${len} entries to record\r`);
    len--;
  }

  summary();
})();
```

> Хотя утечка здесь очевидна, в контексте реального приложения найти источник утечки может быть непросто.

## Запуск с трассировкой сборки мусора

Вы можете увидеть трассировки сборки мусора в выводе консоли вашего процесса с помощью флага `--trace-gc`.
```
$ node --trace-gc script.mjs
```

### Разбор трассировки с `--trace-gc`

Флаг `--trace-gc` (или `--trace_gc`, годится любой) выводит в консоль все события сборки мусора. Состав каждой строки можно описать так:
```
[13973:0x110008000]       44 ms: Scavenge 2.4 (3.2) -> 2.0 (4.2) MB, 0.5 / 0.0 ms  (average mu = 1.000, current mu = 1.000) allocation failure
```

Значение токена| Интерпретация| 13973| PID работающего процесса| 0x110008000| Isolate (экземпляр кучи JS)| 44 ms| Время с момента запуска процесса в мс| Scavenge| Тип / фаза GC| 2.4| Использованная куча до GC в МБ| (3.2)| Общая куча до GC в МБ| 2.0| Использованная куча после GC в МБ| (4.2)| Общая куча после GC в МБ| 0.5 / 0.0 ms (average mu = 1.000, current mu = 1.000)| Время, потраченное на GC, в мс| allocation failure| Причина GC  
---|---  
  
Здесь мы сосредоточимся только на двух событиях:

  * Scavenge
  * Mark-sweep

Куча делится на _пространства_. Среди них есть пространство, называемое «новым» (new), и другое, называемое «старым» (old).

> 👉 На самом деле структура кучи немного отличается, но для этой статьи мы будем придерживаться упрощённой версии. Если вам нужны подробности, рекомендуем посмотреть этот [доклад Питера Маршалла](https://v8.dev/blog/trash-talk) об Orinoco.

### Scavenge

Scavenge — это название алгоритма, который выполняет сборку мусора в новом пространстве. Новое пространство — это место, где создаются объекты. Новое пространство спроектировано так, чтобы быть небольшим и быстрым для сборки мусора.

Давайте представим сценарий Scavenge:

  * мы разместили `A`, `B`, `C` и `D`. 
```    | A | B | C | D | <unallocated> |
```

  * мы хотим разместить `E`
  * места недостаточно, память исчерпана
  * тогда запускается сборка (мусора)
  * мёртвые объекты собираются
  * живой объект остаётся
  * предположим, что `B` и `D` были мёртвыми 
```    | A | C | <unallocated> |
```

  * теперь мы можем разместить `E`
```     | A | C | E | <unallocated> |
```

v8 перемещает объекты, не собранные после двух операций Scavenge, в старое пространство.

> 👉 Полный [сценарий Scavenge](https://github.com/thlorenz/v8-perf/blob/master/gc.md#sample-scavenge-scenario)

### Mark-sweep

Mark-sweep используется для сбора объектов из старого пространства. Старое пространство — это место, где живут объекты, пережившие новое пространство.

Этот алгоритм состоит из двух фаз:

  * **Mark** : помечает ещё живые объекты чёрным, а остальные — белым.
  * **Sweep** : сканирует белые объекты и преобразует их в свободное пространство.

> 👉 На самом деле шаги Mark и Sweep немного сложнее. Пожалуйста, прочитайте этот [документ](https://github.com/thlorenz/v8-perf/blob/master/gc.md#marking-state) для получения дополнительных сведений.

![алгоритм mark and sweep](https://upload.wikimedia.org/wikipedia/commons/4/4a/Animation_of_the_Naive_Mark_and_Sweep_Garbage_Collector_Algorithm.gif)

## `--trace-gc` в действии

### Утечка памяти

Теперь, если вы быстро вернётесь к предыдущему окну терминала, вы увидите в консоли множество событий `Mark-sweep`. Мы также видим, что объём памяти, освобождённый после события, незначителен.

Теперь, когда мы стали экспертами по сборке мусора! Что мы можем из этого заключить?

Вероятно, у нас утечка памяти! Но как в этом убедиться? (Напоминание: в этом примере это довольно очевидно, но что насчёт реального приложения?)

Но как обнаружить контекст?

### Как получить контекст плохих размещений

  1. Предположим, мы наблюдаем, что старое пространство непрерывно растёт.
  2. Уменьшите [`--max-old-space-size`](https://nodejs.org/api/cli.html#--max-old-space-sizesize-in-megabytes) так, чтобы общий размер кучи был ближе к пределу
  3. Запускайте программу, пока не столкнётесь с нехваткой памяти.
  4. Полученный лог показывает контекст сбоя.
  5. Если возникает OOM (нехватка памяти), увеличьте размер кучи примерно на 10 % и повторите несколько раз. Если наблюдается тот же паттерн, это указывает на утечку памяти.
  6. Если OOM не возникает, зафиксируйте размер кучи на этом значении — плотно упакованная куча уменьшает потребление памяти и задержку вычислений.

Например, попробуйте запустить `script.mjs` следующей командой:
```
node --trace-gc --max-old-space-size=50 script.mjs
```

Вы должны столкнуться с OOM:

Теперь попробуйте с 100 МБ:

### Медлительность

Как определить, происходит ли слишком много сборок мусора и не вызывают ли они накладных расходов?

  1. Проанализируйте данные трассировки, а именно время между последовательными сборками.
  2. Проанализируйте данные трассировки, конкретно время, потраченное на GC.
  3. Если время между двумя GC меньше времени, потраченного на GC, приложение серьёзно голодает.
  4. Если и время между двумя GC, и время, потраченное на GC, очень велики, вероятно, приложение может обойтись меньшей кучей.
  5. Если время между двумя GC намного больше времени, потраченного на GC, приложение относительно здорово.

## Устранение утечки

Теперь давайте устраним утечку. Вместо использования объекта для хранения наших записей мы могли бы использовать файл.

Давайте немного изменим наш скрипт:
```js
// script-fix.mjs
import fs from 'node:fs/promises';
import os from 'node:os';

let len = 1_000_000;
const fileName = `entries-${Date.now()}`;

async function addEntry() {
  const entry = {
    timestamp: Date.now(),
    memory: os.freemem(),
    totalMemory: os.totalmem(),
    uptime: os.uptime(),
  };
  await fs.appendFile(fileName, JSON.stringify(entry) + '\n');
}

async function summary() {
  const stats = await fs.lstat(fileName);
  console.log(`File size ${stats.size} bytes`);
}

// выполнение
(async () => {
  await fs.writeFile(fileName, '----START---\n');
  while (len > 0) {
    await addEntry();
    process.stdout.write(`~~> ${len} entries to record\r`);
    len--;
  }

  await summary();
})();
```

Использование `Set` для хранения данных — вовсе не плохая практика; вам просто нужно заботиться о потреблении памяти вашей программой.

> Примечание: исходный код этого [упражнения](https://github.com/nodejs/diagnostics/tree/main/documentation/memory/step3/exercise) можно найти в репозитории Node.js Diagnostics.

Теперь давайте выполним этот скрипт.
```
node --trace-gc script-fix.mjs
```

Вы должны заметить две вещи:

  * События Mark-sweep появляются реже
  * потребление памяти не превышает 25 МБ против более чем 130 МБ в первом скрипте.

Это вполне логично, поскольку новая версия нагружает память меньше, чем первая.

**Вывод** : что вы думаете об улучшении этого скрипта? Вы, вероятно, замечаете, что новая версия скрипта работает медленно. Что, если снова использовать `Set` и записывать его содержимое в файл только тогда, когда память достигает определённого размера?

> API [`getheapstatistics`](https://nodejs.org/api/v8.html#v8getheapstatistics) может вам помочь.

## Бонус: трассировка сборки мусора программно

### Использование модуля `v8`

Возможно, вы захотите не собирать трассировки на протяжении всего времени жизни вашего процесса. В этом случае установите флаг из самого процесса. Модуль `v8` предоставляет API для установки флагов на лету.
```js
import v8 from 'v8';

// включение trace-gc
v8.setFlagsFromString('--trace-gc');

// отключение trace-gc
v8.setFlagsFromString('--notrace-gc');
```

### Использование хуков производительности

В Node.js вы можете использовать [хуки производительности](https://nodejs.org/api/perf_hooks.html) для трассировки сборки мусора.
```js
const { PerformanceObserver } = require('node:perf_hooks');

// Создаём наблюдатель производительности
const obs = new PerformanceObserver(list => {
  const entry = list.getEntries()[0];
  /*
  Запись является экземпляром PerformanceEntry, содержащим
  метрики одного события сборки мусора.
  Например:
  PerformanceEntry {
    name: 'gc',
    entryType: 'gc',
    startTime: 2820.567669,
    duration: 1.315709,
    kind: 1
  }
  */
});

// Подписываемся на уведомления о сборках мусора
obs.observe({ entryTypes: ['gc'] });

// Останавливаем подписку
obs.disconnect();
```

### Разбор трассировки с помощью хуков производительности

Вы можете получить статистику GC в виде [PerformanceEntry](https://nodejs.org/api/perf_hooks.html#perf_hooks_class_performanceentry) из колбэка в [PerformanceObserver](https://nodejs.org/api/perf_hooks.html#perf_hooks_class_performanceobserver).

Например:
```json
{
  "name": "gc",
  "entryType": "gc",
  "startTime": 2820.567669,
  "duration": 1.315709,
  "kind": 1
}
```

Свойство| Интерпретация| name| Имя записи производительности.| entryType| Тип записи производительности.| startTime| Миллисекундная временная метка высокого разрешения, отмечающая время начала записи производительности.| duration| Общее количество миллисекунд, прошедшее для этой записи.| kind| Тип выполненной операции сборки мусора.| flags| Дополнительная информация о GC.  
---|---  
  
Для получения дополнительной информации вы можете обратиться к [документации о хуках производительности](https://nodejs.org/api/perf_hooks.html).
